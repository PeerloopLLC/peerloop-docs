# Session Decisions - 2026-01-09_13-08-40

## Decision: Use Vite's loadEnv Instead of dotenv

### Context
The D1 REST API fallback was failing on MBA-2017 because `require('dotenv')` doesn't work in Vite's ESM environment.

### Options Considered
1. **Use `createRequire` from `node:module`** - Would work in Node.js ESM but adds complexity
2. **Use Vite's built-in `loadEnv`** - Native solution, already bundled
3. **Dynamic `import('dotenv')`** - Async, complicates synchronous code

### Decision
Use Vite's `loadEnv()` in `astro.config.mjs` to populate `process.env` at config time.

### Rationale
- Vite already bundles dotenv internally
- Native integration with the build system
- Loads at config time, before any SSR code runs
- Allows removal of dotenv as a direct dependency
- The `.env -> .dev.vars` symlink continues to work

### Implementation
```javascript
import { loadEnv } from 'vite';
const env = loadEnv(mode, process.cwd(), '');
Object.assign(process.env, env);
```

### Consequences
- Removed dotenv from package.json (one less dependency)
- Simplified d1-rest.ts (no more loadDevVars function)
- Environment loading is now centralized in astro.config.mjs
- Works correctly on both MBA-2017 (Node adapter) and MacMini (Cloudflare adapter)
