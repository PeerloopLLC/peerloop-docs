# Session Decisions - 2026-01-24

## 1. D1 Reset Strategy - Schema-Driven Drop Order
**Type:** Implementation
**Topics:** d1, cloudflare, workflow

**Trigger:** Needed to reset D1 databases (local, staging, production) after consolidating 19 migration files into a two-file system (0001_schema.sql + 0002_seed.sql). Wrangler tracks migrations by filename, so modified files aren't re-applied without a full reset.

**Options Considered:**
1. `PRAGMA foreign_keys = OFF` before DROP statements - **Doesn't work** (D1 enforces at platform level)
2. `PRAGMA defer_foreign_keys = ON` before DROP statements - **Doesn't help** (only affects data ops, not DROP TABLE)
3. Delete all data first, then drop empty tables - Possible but complex
4. Drop tables in FK dependency order (children before parents) ← **Chosen**

**Decision:** Implement schema-driven reset that parses `0001_schema.sql` to determine FK relationships, then drops tables in reverse dependency order.

**Rationale:**
- D1's FK enforcement is immutable - must work within its constraints
- Parsing the schema makes `0001_schema.sql` the single source of truth for both creation AND destruction
- Self-maintains as schema evolves - no hardcoded table lists to update
- Works for remote D1 where file deletion isn't possible

**Consequences:**
- Created `scripts/reset-d1.js` with two strategies:
  - Local: Delete SQLite files (simpler, faster)
  - Remote: Parse schema and drop in dependency order
- Added npm scripts: `db:reset:local`, `db:reset:staging`, `db:reset:remote`
- Documented in CLAUDE.md, BEST-PRACTICES.md, and DECISIONS.md

## 2. Leave Production D1 Reset Untested
**Type:** Strategic
**Topics:** d1, deployment

**Trigger:** After successfully testing reset on local and staging D1, user asked about testing production.

**Options Considered:**
1. Test production reset now ← Not chosen
2. Leave production untested, trust it works like staging ← **Chosen**

**Decision:** Do not test the production D1 reset at this time. The staging test confirmed the approach works on remote D1.

**Rationale:**
- Staging and production use identical D1 infrastructure
- The 5-second safety delay provides time to abort
- Production reset is a high-risk operation with no immediate need
- Can test production when there's an actual requirement

**Consequences:**
- Production reset capability exists but remains untested
- Added caution note in documentation about production usage
