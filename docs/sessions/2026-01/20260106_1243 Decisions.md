# Session Decisions - 2026-01-06

## Decision 1: Use D1 REST API for MBA-2017 Development

**Type**: Strategic
**Driver**: Claude-driven (user approved)
**Working On**: Enabling development on MBA-2017 without workerd support
**Trigger**: Login returning 500 error because D1 binding unavailable on Node adapter

**Options Considered**:
1. Accept limitation - develop only on MacMini for features needing D1
2. Use mock data fallback when D1 unavailable
3. **Use Cloudflare D1 REST API** <- **Chosen**
4. Deploy to Cloudflare for all testing

**Decision**: Implement a D1 REST API client that mimics the D1Database interface, allowing MBA-2017 to query remote D1 via HTTP.

**Rationale**:
- REST API doesn't require workerd runtime
- Provides real database access (not mock data)
- ~50-200ms latency acceptable for development
- Keeps development workflow consistent between machines

**Consequences**:
- Created `src/lib/db/d1-rest.ts`
- Added `dotenv` dependency for loading `.dev.vars`
- MBA-2017 can now run full dev workflow with real D1 data

## Decision 2: Require Explicit Opt-In for Dev Fallbacks

**Type**: Strategic
**Driver**: User-driven
**Working On**: Implementing fallback for JWT_SECRET
**Trigger**: User concern about fallback code potentially running in production

**Options Considered**:
1. **Automatic fallback** - Always fall back to process.env when binding unavailable
2. **Explicit opt-in** <- **Chosen** - Require `ENABLE_D1_REST_FALLBACK=true` flag

**Decision**: Dev fallbacks (D1 REST API, process.env for secrets) only activate when:
1. `ENABLE_D1_REST_FALLBACK=true` is set in `.dev.vars`
2. `NODE_ENV !== 'production'`

**Rationale** (user's words):
> "I don't think we want that code in production to fallback. This provision of API access is simply to allow me to continue working while I am not on the MacMini. As such, I think we should be deliberate in our use of the API and to be surgical in checking that we are in development and on MBA-2017 when we use it"

**Consequences**:
- Created `src/lib/env.ts` with `isDevFallbackEnabled()` check
- Updated `getDB()` to check flag before using REST fallback
- Production deployments ignore the flag entirely
- More verbose but safer configuration

## Decision 3: Machine-Aware Test Strategy

**Type**: Implementation
**Driver**: User-driven
**Working On**: Testing the D1 REST API fallback
**Trigger**: User requested tests that detect machine and report accordingly

**Options Considered**:
1. Skip tests on MBA-2017 entirely
2. Mock D1 for all tests
3. **Machine detection with conditional tests** <- **Chosen**

**Decision**: Create tests that detect macOS version and run appropriate tests for each machine type using Vitest's `it.runIf()`.

**Rationale**:
- Tests validate actual behavior on each machine
- MBA-2017 tests confirm REST API fallback works
- MacMini tests confirm native binding used
- Single test file works on both machines

**Consequences**:
- Created `src/__tests__/d1-rest-fallback.test.ts`
- Tests report machine type at runtime
- 11 tests pass on MBA-2017, 1 skipped (MacMini-specific)
