## Decision 1: Environment Variable Loading for Astro Dev

**Type**: Implementation
**Driver**: Claude-driven
**Working On**: SPAY page verification - Stripe Connect not working
**Trigger**: `STRIPE_SECRET_KEY` was in `.dev.vars` but not loaded by `astro dev`

**Options Considered**:
1. Duplicate environment variables in both `.dev.vars` and `.env` files
2. **Create symlink: `ln -s .dev.vars .env`** <- **Chosen**
3. Run via `wrangler pages dev` instead of `astro dev`

**Decision**: Create a symlink from `.env` to `.dev.vars` so both Astro and wrangler can use the same file.

**Rationale**: Symlink avoids duplication and ensures both tools always use identical values. Running via wrangler changes the dev workflow unnecessarily.

**Consequences**:
- Added symlink step to MBA-2017 setup in `docs/tech/tech-013-devcomputers.md`
- Updated `.dev.vars.example` with symlink instruction

## Decision 3: D1 REST Fallback Pattern

**Type**: Implementation
**Driver**: Claude-driven
**Working On**: CCNT page returning 503 error
**Trigger**: `learn.astro` checked for native D1 binding directly instead of using REST fallback

**Options Considered**:
1. **Use `getDB(Astro.locals)` which handles REST fallback** <- **Chosen**
2. Add native D1 check with separate REST fallback code
3. Require all development on Mac Mini (which supports local D1)

**Decision**: Always use `getDB()` helper function in page files, never direct `Astro.locals.runtime?.env?.DB` access.

**Rationale**: `getDB()` abstracts the D1 access pattern and automatically handles MBA-2017's REST API fallback. Direct binding checks fail silently on machines without local D1 support.

**Consequences**:
- Fixed `src/pages/courses/[slug]/learn.astro`
- Established pattern: never access D1 binding directly in page files
