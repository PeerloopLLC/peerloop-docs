# Session Learnings - 2026-01-07

## Technical

### D1 REST API Fallback Pattern

**Issue**: API endpoints using direct `locals.runtime?.env?.DB` access fail on MBA-2017 because the Node adapter (used on macOS < 13.5) doesn't provide Cloudflare bindings.

**Solution**: Always use `getDB(locals)` helper from `@lib/db`:

```typescript
// BAD - bypasses REST fallback
const db = locals.runtime?.env?.DB;
if (!db) {
  return Response.json({ error: 'Database not available' }, { status: 503 });
}

// GOOD - enables REST fallback on MBA-2017
let db: D1Database;
try {
  db = getDB(locals);
} catch {
  return Response.json({ error: 'Database not available' }, { status: 503 });
}
```

**Also Required**: Add `process.env` fallback for secrets:
```typescript
// BAD
const jwtSecret = locals.runtime?.env?.JWT_SECRET;

// GOOD
const jwtSecret = locals.runtime?.env?.JWT_SECRET || process.env.JWT_SECRET;
```

**Files Affected**: 18 API endpoints needed this fix - all in `src/pages/api/`

### Node vs Cloudflare Adapter Behavior

| Environment | Adapter | `locals.runtime?.env` | D1 Access |
|-------------|---------|----------------------|-----------|
| Mac Mini (macOS 13.5+) | Cloudflare | Available | Native binding |
| MBA-2017 (macOS 12) | Node | Undefined | REST API fallback |
| Production | Cloudflare | Available | Native binding |

The `getDB()` helper in `src/lib/db/index.ts` handles this automatically when `ENABLE_D1_REST_FALLBACK=true` is set in `.dev.vars`.

## Workflow

### Page Verification Process

**Pattern Established**:
1. Read page spec from `docs/pages/{CODE}/{CODE}.md`
2. Read component files and check for TODOs
3. Check `incoming/` for screenshots
4. Analyze screenshot against spec
5. Add **Interactive Elements** section documenting:
   - Buttons (onClick handlers)
   - Links (internal, external)
   - Form inputs
   - API calls triggered
   - Target page status (implemented vs PageSpecView)
6. Move screenshot to `docs/pages/{CODE}/` with timestamp naming
7. Update Verification Notes section

**Interactive Elements Section** - new standard for verified pages:
```markdown
## Interactive Elements

### Buttons (with onClick handlers)
| Element | Component | Action | Status |

### Links - Navigation
| Element | Target | Status |

### API Calls (triggered by interactions)
| Trigger | Endpoint | Method | Status |

### Target Pages Status
| Target | Page Code | Implemented |
```

- Moves to `docs/pages/{CODE}/` folder
- Multiple screenshots numbered: `-1.png`, `-2.png`

## Architecture

### Error Propagation Pattern

When D1 is unavailable:
1. API returns `503 Service Unavailable` with `{ error: 'Database not available' }`
2. React component shows error state with "Retry" button
3. User can retry after fixing the underlying issue

This is the correct pattern - fail fast, show actionable error, allow retry.
