# Session Learnings - 2026-01-17

## 1. Timestamps Are Git Artifacts
**Topics:** workflow, npm

**Context:** The `check-env.sh` startup script compared `package.json` timestamp to `node_modules` to detect when `npm install` was needed.

**Learning:** Git operations (checkout, commit) touch file timestamps without changing content. Timestamp comparisons produce false positives after normal git workflows.

**Pattern:** Compare content hashes instead of timestamps. Store hash after install, compare at startup.

## 2. Hash Storage in Gitignored Locations
**Topics:** workflow, npm

**Context:** Needed a way to store "last known state" of package-lock.json per machine, surviving git operations.

**Learning:** `node_modules/` is gitignored, so files stored there persist locally per machine and are untouched by git fetch/pull. This makes it ideal for machine-local state like dependency hashes.

**Pattern:**
```bash
# postinstall hook stores hash
shasum -a 256 package-lock.json | cut -d' ' -f1 > node_modules/.package-lock-hash

# startup check compares
CURRENT=$(shasum -a 256 package-lock.json | cut -d' ' -f1)
STORED=$(cat node_modules/.package-lock-hash)
```

## 3. Global/Local Skill Architecture
**Topics:** workflow

**Context:** `/q-codecheck` needed both generic checks (TypeScript, ESLint, Tailwind) that work across projects and project-specific checks (Astro).

**Learning:** Split skills into global (`~/.claude/commands/`) and local (`.claude/commands/`) components. Global contains full implementation, checks for local hook, calls it if present, aggregates results.

**Pattern:**
- Global: `~/.claude/commands/q-skillname.md` - full implementation
- Local: `.claude/commands/q-skillname-local.md` - project additions
- Global calls local if it exists, local is optional

## 4. Tags Handle Cross-Cutting Concerns
**Topics:** workflow

**Context:** Initial `/q-learnings` design used context as file-level organization (separate files for deployment, stripe, etc.). This broke down when a learning involved both Cloudflare AND Stripe.

**Learning:** Use topic tags within sections rather than context-based file separation. One file per session, multiple tagged sections. A single learning can have multiple topics.

**Pattern:**
```markdown
## 1. Learning Title
**Topics:** cloudflare, stripe

**Context:** ...
```

## 5. Capture Everything, Listed Topics Are Priorities
**Topics:** workflow

**Context:** Defining topic lists for learnings/decisions raised the question: what about items that don't fit listed topics?

**Learning:** All learnings and decisions are important. Listed topics are priorities to scan for, not filters. For unlisted items, create an appropriate topic name. This signals to the user that this area might warrant being added to the local config if it recurs.
