## Learning 1: PageSpec v1.1 Schema Structure

**Category:** Schema Design
**Confidence:** High

The PageSpec JSON schema v1.1 has specific structures that must be followed exactly:

### Features Structure
```json
"features": [
  {
    "title": "Section Name",
    "items": [
      { "description": "...", "implemented": true, "storyRefs": ["US-A045"] }
    ]
  }
]
```

### Interactive Elements Structure
```json
"interactiveElements": {
  "buttons": [
    { "element": "Button Name", "component": "ComponentName", "action": "API endpoint or action", "status": "active|inactive" }
  ],
  "links": [
    {
      "section": "Section Name",
      "items": [
        { "element": "Link Name", "target": "/route", "status": "active|inactive" }
      ]
    }
  ],
  "targetPages": [
    { "target": "/route", "pageCode": "CODE", "implemented": "yes|no" }
  ],
  "summary": {
    "buttons": { "total": N, "active": N, "inactive": N },
    "links": { "total": N, "active": N, "inactive": N },
    "analytics": { "total": N, "active": N, "inactive": N }
  }
}
```

**Why it matters:** Incorrect schema structure causes Astro build failures with PageSpecValidationError.

## Learning 2: Access Level Values

**Category:** Schema Design
**Confidence:** High

The `access` field in PageSpec must be one of the exact enum values:
- `Public` - No authentication required
- `Authenticated` - Any logged-in user
- `Admin` - Admin role required
- `Creator` - Creator role required
- `StudentTeacher` - Student-Teacher role required

**Common mistake:** Using descriptive text like "Authenticated (admin)" instead of just "Admin".

**Why it matters:** Zod validation fails on non-enum values.

## Learning 3: Verification and Consolidation Workflow

**Category:** Process
**Confidence:** High

The `/q-verify-page` and `/q-consolidate-page` skills work together:

1. **Verify**: Analyze screenshot + code → Create/update `docs/pages/CODE/CODE.md`
2. **Consolidate**: Sync `docs/pages/CODE/CODE.md` → `src/data/pages/.../code.json`

Key outputs from consolidation:
- `verificationNotes` section with date, components, screenshots
- `_discrepancies` section documenting gaps between spec and reality
- Updated `features` with implementation status
- Updated `interactiveElements` with active/inactive status

## Learning 4: Course-Scoped Roles

**Category:** Architecture
**Confidence:** High

Roles in Peerloop can be:
- **Global**: Admin, Moderator (platform-wide)
- **Course-scoped**: Creator, StudentTeacher, Student, Graduate, Guest

A user's role relative to a course depends on their relationship:
- **Guest**: Not enrolled, can view public info
- **Student**: Enrolled, in progress
- **Graduate**: Completed course, has certificate
- **StudentTeacher**: Certified to teach (requires Creator approval)
- **Creator**: Course author

This affects page access - same user may have different access to same page depending on course context.

## Patterns Identified

1. **Always validate JSON specs after editing** - Run `npx tsx scripts/validate-page-spec.ts <file>` to catch schema errors before Astro build
2. **Check enum values in schema** - When validation fails, check `src/lib/schemas/page-spec.ts` for allowed values
3. **Document discrepancies explicitly** - The `_discrepancies` section makes gaps visible for future implementation
