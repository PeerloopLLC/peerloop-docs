# Session Learnings - 2026-01-06

## Technical

### D1 REST API Fallback for MBA-2017

**Learning:** Cloudflare D1 can be accessed via REST API when the workerd runtime isn't available (macOS < 13.5).

**Key Implementation Details:**
- Cloudflare's D1 REST API endpoint: `POST https://api.cloudflare.com/client/v4/accounts/{account_id}/d1/database/{database_id}/query`
- The REST client must mimic the D1Database interface (`prepare()`, `bind()`, `first()`, `all()`, `run()`, `batch()`)
- `batch()` via REST is sequential, not truly atomic like native D1
- Performance: ~50-200ms per query (acceptable for dev, not production)

**Files Created:**
- `src/lib/db/d1-rest.ts` - REST API client
- `src/lib/env.ts` - Environment variable helper with dev fallback

### Environment Variable Access in Astro

**Learning:** Astro's Node adapter doesn't have access to `locals.runtime.env` (Cloudflare-specific). Need fallback pattern.

**Pattern:**
```typescript
// In production/MacMini: locals.runtime.env.VAR
// On MBA-2017 dev: process.env.VAR (loaded via dotenv)
const value = locals.runtime?.env?.VAR || process.env.VAR;
```

**Better Pattern (what we implemented):**
```typescript
// src/lib/env.ts
export function getEnv(locals: App.Locals, key: string): string | undefined {
  return locals.runtime?.env?.[key] ?? (isDevFallbackEnabled() ? process.env[key] : undefined);
}
```

### dotenv for .dev.vars Loading

**Learning:** Astro doesn't automatically load `.dev.vars` (wrangler-specific). Need `dotenv` package.

**Implementation:**
```typescript
import dotenv from 'dotenv';
dotenv.config({ path: resolve(process.cwd(), '.dev.vars') });
```

This runs at module import time and populates `process.env`.

## Workflow

### Explicit Opt-In for Dev Fallbacks

**Learning:** Dev-only fallbacks should be explicit, not automatic.

**Why:**
- Production should never silently fall back
- Makes debugging easier (know which path you're on)
- Prevents accidental REST API usage in production

**Pattern:**
```bash
# .dev.vars (MBA-2017 only)
ENABLE_D1_REST_FALLBACK=true
```

```typescript
function isRestFallbackEnabled(): boolean {
  return process.env.ENABLE_D1_REST_FALLBACK === 'true'
    && process.env.NODE_ENV !== 'production';
}
```

### Database IDs Can Be Stale

**Learning:** D1 database IDs in `wrangler.toml` can become stale if database is recreated.

**Symptom:** `404: The database {id} could not be found`

**Fix:** Check Cloudflare dashboard for actual database ID and update:
- `wrangler.toml`
- Any hardcoded IDs in code (like `d1-rest.ts`)

## Architecture

### Machine-Aware Testing

**Learning:** Tests can detect the machine type and run appropriate tests.

**Implementation:**
```typescript
function detectMachine() {
  const version = execSync('sw_vers -productVersion').trim();
  const [major, minor] = version.split('.').map(Number);
  return {
    name: major >= 13 && minor >= 5 ? 'MacMini' : 'MBA-2017',
    supportsWorkerd: major >= 13 && minor >= 5
  };
}

// Then use vitest's runIf:
it.runIf(!machine.supportsWorkerd)('MBA-2017 specific test', ...)
it.runIf(machine.supportsWorkerd)('MacMini specific test', ...)
```
