# Session Learnings - 2026-01-23

## 1. Astro's getViteConfig Causes Vitest Hanging Process
**Topics:** testing, astro, workflow

**Context:** After tests completed successfully, Vitest would hang with "close timed out after 5000ms" and "Tests closed successfully but something prevents Vite server from exiting" messages. The hanging-process reporter showed 16-43 FILEHANDLE entries with unknown stack traces.

**Learning:** Astro's `getViteConfig()` wrapper starts Vite with file watchers enabled by default, even in single-run test mode (`--run`). These file handles aren't properly closed when tests complete. The `server.watch: null` setting (added in Vite 5.3+) should disable this, but Astro's wrapper may override it.

**Pattern:**
```typescript
// Don't use Astro's wrapper for tests:
// import { getViteConfig } from 'astro/config';

// Use standard Vitest config instead:
import { defineConfig } from 'vitest/config';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  server: { watch: null },  // Disable file watcher
  resolve: {
    alias: {
      '@lib': resolve(__dirname, 'src/lib'),
      // ... other aliases from tsconfig.json
    },
  },
  test: { ... },
});
```

## 2. Path Aliases Must Be Duplicated in Vitest Config
**Topics:** testing, astro, workflow

**Context:** After switching from `getViteConfig` to `defineConfig`, tests failed with "Failed to resolve import @lib/db/d1-rest". The path aliases defined in `tsconfig.json` were no longer being picked up.

**Learning:** Astro's `getViteConfig()` automatically applies tsconfig path aliases to Vite's resolve configuration. When using standard `defineConfig`, you must manually duplicate all path aliases from `tsconfig.json` to `vitest.config.ts` in the `resolve.alias` section.

**Pattern:**
```typescript
// tsconfig.json has:
// "paths": { "@lib/*": ["src/lib/*"], "@components/*": ["src/components/*"] }

// vitest.config.ts needs:
resolve: {
  alias: {
    '@lib': resolve(__dirname, 'src/lib'),
    '@components': resolve(__dirname, 'src/components'),
  },
},
```

## 3. Page Test Template: 6-Category Structure
**Topics:** testing, workflow

**Context:** Creating comprehensive page tests for LoginForm, SignupForm, and CourseBrowse. Needed a consistent structure that could be reused across different page families.

**Learning:** Page component tests can be organized into 6 consistent categories that adapt to the page type:

| Category | Auth Forms | List/Browse Pages |
|----------|-----------|-------------------|
| 1. RENDERING | Form elements, links | Headers, filters, grid |
| 2. INTERACTION | Typing in fields | Search input, clicks |
| 3. VALIDATION | Client-side errors | Filter logic |
| 4. SUBMISSION | API calls, loading | Sorting behavior |
| 5. NAVIGATION | Redirects, returnUrl | Pagination |
| 6. ERROR HANDLING | API/network errors | Empty state |

## 4. HTML5 Email Validation Affects Test Behavior
**Topics:** testing

**Context:** Testing SignupForm email validation. The test used `"notanemail"` as invalid input, but the error message wasn't appearing.

**Learning:** Input fields with `type="email"` have browser-level HTML5 validation that may prevent form submission before JavaScript validation runs. In jsdom, this works differently than real browsers. To test JS validation, use an email that passes HTML5 but fails the custom regex (e.g., `"john@example"` which lacks a TLD).

**Pattern:**
```typescript
// This may not trigger JS validation:
await user.type(emailInput, 'notanemail');

// This passes HTML5 but fails regex /^[^\s@]+@[^\s@]+\.[^\s@]+$/:
await user.type(emailInput, 'john@example');
```

## 5. Testing Components with Multiple Matching Elements
**Topics:** testing

**Context:** CourseBrowse tests failed with "Found multiple elements with the text: React Basics" because CourseCard renders the title in multiple places (e.g., as heading and in link).

**Learning:** When a component renders the same text in multiple places, use `getAllByText()` with length check instead of `getByText()`. For specific element assertions, use `within()` to scope queries to a container.

**Pattern:**
```typescript
// Instead of:
expect(screen.getByText('Course Title')).toBeInTheDocument();

// Use:
expect(screen.getAllByText('Course Title').length).toBeGreaterThan(0);

// Or scope to a container:
const card = screen.getByTestId('course-card-1');
expect(within(card).getByRole('heading')).toHaveTextContent('Course Title');
```

## 6. Debounced Search Requires Real Timers in Tests
**Topics:** testing

**Context:** CourseBrowse has a 300ms debounced search. Initial tests using fake timers with `vi.useFakeTimers()` were timing out at 5 seconds.

**Learning:** Fake timers with userEvent can be tricky. For simple debounce tests, using real timers with `waitFor` and a longer timeout is more reliable than coordinating fake timers with userEvent's internal timing.

**Pattern:**
```typescript
// Simple approach - real timers with waitFor:
it('filters after debounce', async () => {
  const user = userEvent.setup();
  render(<Component />);

  await user.type(searchInput, 'query');

  await waitFor(
    () => expect(screen.getByText('1 result')).toBeInTheDocument(),
    { timeout: 1000 }  // Longer than debounce + render
  );
});
```
