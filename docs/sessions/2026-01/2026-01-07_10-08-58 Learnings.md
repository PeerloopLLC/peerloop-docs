# Session Learnings - 2026-01-07

## Technical

### Page JSON Schema Consistency
**Issue:** SPAY page threw error "Cannot read properties of undefined (reading 'replace')" in PageSpecView.

**Root Cause:** The `outgoing` connections in the JSON used `source/sourceRoute` (correct for incoming) instead of `target/targetRoute` (correct for outgoing).

**Fix:** Changed the third outgoing connection from:
```json
{ "source": "SETT", "sourceRoute": "/settings" }
```
to:
```json
{ "target": "SETT", "targetRoute": "/settings" }
```

**Prevention:** When creating page JSON specs manually (not from research folder templates), verify:
- `incoming[]` uses `source`, `sourceRoute`
- `outgoing[]` uses `target`, `targetRoute`

### Layout Toggle Pattern
**Discovery:** LandingLayout has a spec/content toggle pattern that AppLayout and AdminLayout were missing.

**Pattern:**
1. Body has `show-spec` class when `pageCode` is provided
2. Content wrapped in `.spec-content` (PageSpecView) and `.real-content` (production)
3. `SpecPanelToggle` component shows button to switch views
4. CSS controls visibility based on `show-spec` class

**Applied To:** AppLayout (22 pages) and AdminLayout (8 pages) now support optional `pageCode` prop. Backward compatible - pages without prop work unchanged.

## Architecture

### Moderator vs Admin Role Separation
**Insight:** The Moderator role is distinct from Admin:

| Role | Focus | Pages |
|------|-------|-------|
| Admin | Platform management (users, courses, payouts) | `/admin/*` (AdminLayout) |
| Moderator | Content moderation (flagged posts, reviews) | `/mod/*` (AppLayout) |

**Current Gap:**
- Admin can manage moderators (invite, remove role)
- Moderators need their own dashboard for doing moderation work
- No Moderator-specific navigation in AppLayout

**Added to PLAN:** Block 8.10 documents this gap with proposed pages (MDSH, MHST, MSET for moderators; AMOD for admin).

## Workflow

### Page Verification Dependencies
**Lesson:** Some pages can't be verified in isolation due to data dependencies:

| Page | Dependency |
|------|------------|
| CSUC (Checkout Success) | Requires creator to have Stripe connected + student to complete enrollment |
| SPAY (Payment Settings) | Requires creator login |
| CCNT (Course Content) | Requires enrolled student |

**Approach:** Verify in order of dependency chain, or use test data to simulate states.

### Checking All Layout Files
**Process:** When adding a feature to one layout, check if other layouts need the same:
1. Found 3 layouts: LandingLayout, AppLayout, AdminLayout
2. LandingLayout had toggle, others didn't
3. Applied same pattern to AppLayout and AdminLayout

This prevents feature drift between layouts.
