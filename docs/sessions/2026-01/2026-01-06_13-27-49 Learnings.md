# Session Learnings - 2026-01-06

## Technical

### D1 REST API Fallback Architecture

**Pattern for site-wide database access:**
- Use `getDB(locals)` instead of directly accessing `locals.runtime?.env?.DB`
- The `getDB()` function encapsulates fallback logic: native binding first, REST API if enabled
- Environment variables should use `getEnv(locals, 'KEY')` for the same fallback pattern

**Why the native binding check isn't a network call:**
- `locals.runtime?.env?.DB` is just checking if an object exists in memory
- On machines where workerd fails (macOS < 13.5), this object is simply undefined
- No network call happens until you actually use the database object (`.prepare().all()`)

**Safety in production:**
- Native D1 binding is ALWAYS present on Cloudflare (production/preview)
- The fallback code path is never reached in production
- Double protection: `isRestFallbackEnabled()` also checks `NODE_ENV !== 'production'`

### Configuration Design

**Single control variable is cleaner:**
- Previously had `D1_MODE=local|remote` which was confusing
- Now just `ENABLE_D1_REST_FALLBACK=true` - binary, explicit
- Credentials can stay uncommented on all machines - ignored when not needed

**Why credentials don't need conditional commenting:**
- MacMini: Native binding takes priority, credentials never read
- MBA-2017: Fallback enabled, credentials used
- Production: Native binding + production check = credentials ignored
- Leaving them set but unused is harmless

## Workflow

### API Route Update Pattern

When adding a new database access pattern, the update scope is larger than expected:
- 22 API route files needed updating
- Each had the same pattern: `const db = locals.runtime?.env?.DB`
- Systematic find-and-replace approach worked well

### Documentation Should Match Implementation

- Had `D1_MODE` documented in `.dev.vars.example` but never implemented
- Created confusion when actual implementation used different variable
- Lesson: Implement what you document, or update docs when implementation differs

## Architecture

### Fallback Logic Design

Good fallback pattern:
```typescript
// 1. Try preferred method
if (preferredMethod()) {
  return preferredResult;
}

// 2. Check if fallback is explicitly enabled
if (fallbackEnabled()) {
  return fallbackResult;
}

// 3. Fail explicitly
throw new Error('Descriptive message');
```

This pattern:
- Doesn't silently degrade
- Requires explicit opt-in for fallback
- Provides clear error messages
