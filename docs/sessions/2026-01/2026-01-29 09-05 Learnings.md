## 1. CurrentUser Singleton Pattern with Stale-While-Revalidate
**Topics:** astro, auth

**Context:** Implementing global user state management to replace deprecated `is_*` role flags with course-aware role checking.

**Learning:** The stale-while-revalidate pattern works well for user state in client-side React islands:
1. Check localStorage cache first for instant hydration
2. Render immediately with cached data
3. Fetch fresh data in background
4. Update UI if data changed

This provides perceived instant load while ensuring fresh data.

**Pattern:**
```typescript
export async function initializeCurrentUser(): Promise<CurrentUser | null> {
  // 1. Instant hydration from cache
  const cached = loadFromCache();
  if (cached) {
    setCurrentUser(new CurrentUser(cached));
  }

  // 2. Fetch fresh in background
  const response = await fetch('/api/me/full');
  const data = await response.json();

  // 3. Update singleton and cache
  setCurrentUser(new CurrentUser(data));
  saveToCache(data);

  return getCurrentUser();
}
```

## 2. Capability-Based vs Role-Based Authorization
**Topics:** auth, astro

**Context:** The old approach used boolean flags (`is_student`, `is_creator`) but users can have different roles per course.

**Learning:** Capability-based authorization (`canTakeCourses`, `canTeachCourses`, `canCreateCourses`) is more flexible than role-based. Combined with course-specific methods (`isStudentFor(courseId)`, `isStudentTeacherFor(courseId)`), it supports the complex Peerloop permission model where a user can be a student for Course A but a Student-Teacher for Course B.

**Pattern:**
```typescript
// Menu visibility using capabilities
const menuItems = [
  { id: 'workspace', capabilities: ['canTeachCourses', 'canCreateCourses'] },
  { id: 'courses', requiresAuth: true }, // any authenticated user
];

// Course-specific checks
if (user.isStudentTeacherFor(courseId)) {
  // Show teaching controls
}
```

## 3. Tailwind v4 outline-none Behavior Change
**Topics:** astro, workflow

**Context:** Running `/q-codecheck` found `outline-none` flagged in ErrorPage.tsx.

**Learning:** In Tailwind v4, `outline-none` was renamed to `outline-hidden` to clarify its purpose. The old class name still works but the check script catches it as a v4 compatibility issue. Always use `focus:outline-hidden` for hiding focus outlines.
