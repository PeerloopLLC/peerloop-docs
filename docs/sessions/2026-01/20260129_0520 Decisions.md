# Session Decisions - 2026-01-29

## 1. New `/dash/*` Routes Separate from Existing `/dashboard/*`
**Type:** Strategic
**Topics:** astro, workflow

**Trigger:** Need to implement an experimental dashboard with persistent sidebar navigation without disrupting existing dashboard pages.

**Options Considered:**
1. Modify existing `/dashboard/*` routes in place
2. Create new `/dash/*` routes for experimental layout ← Chosen
3. Use feature flag to switch between layouts

**Decision:** Create new `/dash/*` routes with DashLayout, keeping existing `/dashboard/*` routes unchanged.

**Rationale:** Allows iterative development of new dashboard experience without risk to existing functionality. Users can compare both. Easy rollback if needed.

**Consequences:** 7 new routes created under `/dash/`, new DashLayout.astro, all existing dashboard pages continue working.

## 2. Visitor-Friendly Dashboard (No Auth Redirect)
**Type:** Implementation
**Topics:** auth, astro

**Trigger:** Should unauthenticated visitors see the dashboard layout or be redirected to login?

**Options Considered:**
1. Redirect to login like existing AppLayout ← Not chosen
2. Show dashboard shell with marketing CTAs ← Chosen
3. Show completely different landing page

**Decision:** Dashboard is accessible to visitors with limited menu items and "Sign Up" CTAs. No automatic redirect to login.

**Rationale:** Better SEO (pages are SSR-rendered), allows visitors to explore the platform structure, reduces friction in onboarding funnel.

**Consequences:** DashNavbar checks `isAuthenticated` and conditionally shows menu items. FeedSlidePanel shows "Sign up to join communities" for visitors.

## 3. Two Distinct Slide-Out Panel Types
**Type:** Implementation
**Topics:** astro

**Trigger:** "My Feeds" and "More" actions need slide-out panels but with different behaviors.

**Options Considered:**
1. Single panel component with size prop
2. Two separate panel components with different sizing ← Chosen
3. Modal dialogs instead of slide-out panels

**Decision:** Create FeedSlidePanel (full viewport height) and MoreSlidePanel (smaller popup) as separate components.

**Rationale:** Different use cases warrant different UX. Feed selection needs more space for search + community lists. More options is a quick menu. Separate components are easier to maintain and customize.

**Consequences:** Two panel components created. Both use Escape key to close. Both have click-outside-to-close behavior.
