# Session Learnings - 2026-01-05

## Technical

### Cloudflare D1 Setup

1. **Multi-account wrangler requires explicit account ID**
   - When logged into multiple Cloudflare accounts, wrangler CLI fails in non-interactive mode
   - Solution: Use `CLOUDFLARE_ACCOUNT_ID=xxx` environment variable prefix for commands
   - Pages projects don't support `account_id` in wrangler.toml - must use env var

2. **Pages vs Workers wrangler.toml differences**
   - `account_id` in wrangler.toml causes Pages deployment to fail
   - Error: "Configuration file for Pages projects do not support account_id"
   - Solution: Remove from wrangler.toml, use env var for CLI commands only

3. **Migration file naming matters**
   - D1 applies migrations in alphabetical order
   - Main branch had `0001_schema.sql` and `0002_seed.sql`
   - jfg-dev had `0001_initial_schema.sql` and `0002_seed_data.sql`
   - Different names = different migrations = conflicts when merging

4. **Database ID changes require local D1 reset**
   - When creating new D1 database (after deleting old one), local D1 state becomes stale
   - Must run `rm -rf .wrangler/state/v3/d1` before `npm run db:migrate:local`
   - Otherwise migrations fail with schema conflicts

### React 19 + Cloudflare Workers

5. **MessageChannel error is real but misleading in local dev**
   - Local `npm run cf:dev` showed MessageChannel error
   - But actual deployment failure was different (account_id issue)
   - Main branch already had the `react-dom/server.edge` fix
   - Always check deployment logs, not just local errors

### Branch Management

6. **Check branch divergence before making fixes**
   - jfg-dev was behind main by 5 commits
   - Main already had fixes we were trying to apply
   - `git log main..branch` and `git merge-base main branch` reveal relationship
   - Creating fresh branch from main avoided conflict resolution

## Workflow

### Session Documentation Consolidation

7. **Extracting patterns from session files is valuable**
   - 18 decision files + 25 learnings files contained 72 distinct decisions
   - Many decisions were scattered, some conflicting
   - Consolidating into DECISIONS.md and BEST-PRACTICES.md creates actionable reference
   - "Latest wins" principle resolves conflicts automatically

8. **Organize decisions by impact level**
   - Architecture decisions affect everything - list first
   - Database/data model decisions have cascading effects
   - UI/testing decisions are more localized
   - Makes document scannable for relevant guidance

## Documentation

### Reference Documents Structure

9. **DECISIONS.md vs BEST-PRACTICES.md serve different purposes**
   - DECISIONS.md: What we chose and why (audit trail)
   - BEST-PRACTICES.md: How to do things (actionable patterns)
   - Both reference session files for historical context
   - Both use "latest wins" to stay current

10. **Include code examples in best practices**
    - Patterns without examples require re-discovery
    - Examples like SSR data fetching pattern, image fallback, pagination
    - Makes best practices immediately usable, not just advisory
