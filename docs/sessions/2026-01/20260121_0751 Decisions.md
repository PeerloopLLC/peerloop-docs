# Session Decisions - 2026-01-21

## 1. Include Flag Submission in ADMIN.MODERATION Scope
**Type:** Implementation
**Topics:** workflow, d1

**Trigger:** ADMIN.MODERATION scope in PLAN.md included the queue, actions, and logging - but not the flag submission endpoint. Without it, the queue would be empty.

**Options Considered:**
1. Create seed data for testing, defer flag submission to COMMUNITY block
2. Include flag submission endpoint (POST /api/flags) in this work ← Chosen

**Decision:** Include the flag submission endpoint as part of ADMIN.MODERATION.

**Rationale:**
- Enables end-to-end testing of the moderation workflow
- Provides a complete feature without dependencies on other blocks
- Flag submission is simple (~200 lines) and logically belongs with moderation

**Consequences:** Added `src/pages/api/flags.ts` - users can now report posts, comments, or profiles from any feed.

## 2. Cache Content Snapshot at Flag Time
**Type:** Implementation
**Topics:** stream, d1

**Trigger:** Content lives in Stream.io, but flagged content might be deleted before review.

**Options Considered:**
1. Fetch content from Stream.io when admin views the flag
2. Cache content snapshot in content_flags table at flag time ← Chosen
3. Store duplicate copy in separate D1 posts table

**Decision:** Cache content as JSON in `content_snapshot` column at flag time.

**Rationale:**
- Preserves evidence even if original content is deleted
- Reduces Stream.io API calls during moderation
- Single table is simpler than maintaining separate posts table

**Consequences:** `content_flags.content_snapshot` stores JSON with text, author info, timestamp, and optional course context.

## 3. Separate Tables for Flags, Actions, and Warnings
**Type:** Architecture
**Topics:** d1

**Trigger:** Need to track moderation decisions for audit trail and user history.

**Options Considered:**
1. Add status/action columns to content_flags table
2. Use separate tables for actions and warnings ← Chosen

**Decision:** Three-table design:
- `content_flags` - The flagged content and its review status
- `moderation_actions` - Log of every action taken (dismiss, remove, warn, suspend)
- `user_warnings` - Warning records linked to users for escalation tracking

**Rationale:**
- Multiple actions can occur per flag (e.g., warning then suspension)
- Warning history needed for escalation decisions
- Clear separation of concerns aids querying and maintenance

**Consequences:** Migration 0013 creates all three tables with appropriate foreign keys and indexes.
