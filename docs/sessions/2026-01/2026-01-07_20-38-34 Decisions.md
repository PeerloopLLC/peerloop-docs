# Session Decisions - 2026-01-07

## Decision 1: Proper Payment Tracking Architecture

**Context:** Initial refund implementation searched Stripe API for payments by transfer_group, which is not a supported search field.

**Decision:** Implement proper `transactions` and `payment_splits` tables per DB-SCHEMA.md design.

**Rationale:**
- Stripe doesn't support searching charges/payment_intents by transfer_group
- Storing payment data in database is the recommended pattern
- Enables quick lookups for refunds without API calls
- Provides transaction history for auditing

**Implementation:**
- Created migration `0005_transactions.sql`
- Webhook stores transaction on checkout.session.completed
- Refund endpoint looks up from database, not Stripe API

## Decision 2: PageSpecView Toggle Required for All Pages

**Context:** Implemented admin pages (AUSR, ACRS, ACAT) lost PageSpecView capability when real components were added.

**Decision:** All implemented pages must retain PageSpecView toggle capability.

**Rationale:**
- Developers need to reference specs while implementing
- QA can compare implementation against spec
- Consistent pattern across all pages

**Pattern:**
```astro
<Layout pageCode="CODE">
  <PageSpecView pageData={data} />           <!-- default slot -->
  <RealComponent client:load slot="content" /> <!-- content slot -->
</Layout>
```

## Decision 3: Defer Refund Email Notifications

**Context:** Refund flow works but doesn't notify the student.

**Decision:** Add email notifications as future feature after Resend integration.

**Rationale:**
- Core refund functionality works end-to-end
- Email integration (Resend) is a separate block
- Documented as pending feature in AENR.md

## Decision 4: Refund UX Needs Business Rules

**Context:** Current UI has separate "Full Refund" and "Partial Refund" buttons with manual amount entry.

**Decision:** Note for future - implement single "Refund" button with calculated amount modal.

**Rationale:**
- Admin shouldn't have to calculate refund amounts
- Business rules needed: sessions attended, time elapsed, modules completed
- Admin can override calculated amount for edge cases
- Requires business decision before implementation

**Status:** Documented in AENR.md as consideration for future session.
