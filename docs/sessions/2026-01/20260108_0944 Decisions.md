# Session Decisions - 2026-01-08

## Decision 1: Adopt Zod for PageSpec Validation

**Context:** Had manual validation code (~300 lines) separate from TypeScript types. Maintenance burden when schema changes.

**Decision:** Migrate to Zod for PageSpec validation.

**Rationale:**
- Single source of truth: Schema defines both types AND validation
- Types derived via `z.infer<>` - no manual type definitions
- Better error messages out of the box
- Reduced code: ~300 lines → ~80 lines
- Already installed in project (not actively used)

**Implementation:**
- `src/lib/schemas/page-spec.ts` - Zod schema
- `src/lib/types/page-spec.ts` - Re-exports types
- `src/lib/validation/page-spec-validator.ts` - Uses Zod

## Decision 2: Move Skills to Project Level

**Context:** Skills in `~/.claude/commands/` are global and not version controlled.

**Decision:** All skills should be in `.claude/commands/` (project level).

**Rationale:**
- Skills are project-specific
- Version control tracks changes
- Team members get same skills from repo
- Changes visible in git status

**Action:** Moved `q-consolidate-page.md` to project level.

## Decision 3: Replace (Not Append) Consolidation Notes

**Context:** Running `/q-consolidate-page` multiple times could accumulate duplicate notes.

**Decision:** Update skill to replace existing consolidation note rather than append.

**Rationale:**
- Cleaner .md files
- Idempotent behavior
- Latest consolidation date always visible

## Decision 4: Add Runtime Validation to PageSpecView

**Context:** TypeScript doesn't validate JSON imports at compile time. Invalid JSON could cause runtime errors.

**Decision:** Validate JSON at render time in PageSpecView.astro.

**Rationale:**
- Catches malformed JSON early (build time for SSG, request time for SSR)
- Descriptive error messages with context
- Consistent with CLI validation approach

**Implementation:**
```astro
const pageData = parsePageSpec(rawData, context);
```

## Decision 5: Defer Broader Zod Adoption

**Context:** Zod worked well for PageSpec. Could expand to API validation, forms, etc.

**Decision:** Add as Block 9.5 (Candidate) for future evaluation.

**Rationale:**
- Current implementation proves the pattern
- Need to evaluate bundle size impact
- Should assess API validation needs first
- Not blocking for MVP

**Evaluation criteria added to PLAN.md:**
- API validation reduces boilerplate?
- Bundle size acceptable?
- Team finds patterns intuitive?
