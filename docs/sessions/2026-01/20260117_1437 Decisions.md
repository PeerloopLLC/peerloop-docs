# Session Decisions - 2026-01-17

## 1. Hash-Based Sync Detection Over Timestamps
**Type:** Implementation
**Topics:** workflow, npm

**Trigger:** check-env.sh was producing false positives after git operations

**Options Considered:**
1. Timestamp comparison (package.json vs node_modules)
2. Timestamp comparison (package-lock.json vs node_modules)
3. Content hash comparison ← Chosen
4. npm ls to check for missing packages (slow)

**Decision:** Store SHA-256 hash of package-lock.json after npm install, compare current hash to stored hash at startup.

**Rationale:** Timestamps are artifacts of git operations. Content hashes reflect actual file changes. Hash storage in node_modules/ survives git operations since it's gitignored.

**Consequences:** Added `postinstall` hook to package.json, updated check-env.sh to use hash comparison.

## 2. Global/Local Skill Architecture
**Type:** Strategic
**Topics:** workflow

**Trigger:** /q-codecheck needed both generic checks and project-specific checks (Astro)

**Options Considered:**
1. Everything in local project file
2. Everything in global file with conditionals
3. Global + local hook pattern ← Chosen

**Decision:** Split skills into global (full implementation) and local (project-specific hook). Global calls local if it exists.

**Rationale:** Keeps generic functionality portable across projects while allowing project-specific customization. Local hook is optional - global works standalone.

**Consequences:** Created pattern used for /q-codecheck and /q-learn-decide. Global files in ~/.claude/commands/, local hooks in .claude/commands/.

## 3. Tags Within Sections Over Context-Based Files
**Type:** Strategic
**Topics:** workflow

**Trigger:** Cross-cutting learnings (involving both Cloudflare and Stripe) didn't fit single-context file organization

**Options Considered:**
1. Separate files per context (deployment.md, stripe.md)
2. Pick one context, cross-reference from others
3. Duplicate content in multiple files
4. Tags within sections, one file per session ← Chosen

**Decision:** Use topic tags within each section. One file per session with all learnings/decisions, each tagged with relevant topics.

**Rationale:** Handles cross-cutting concerns naturally. Keeps related items together. Searchable via grep on topic names.

**Consequences:** Simplified file structure. Added **Topics:** field to each section.

## 4. Separate Files for Learnings and Decisions
**Type:** Strategic
**Topics:** workflow

**Trigger:** Considered merging /q-learnings and /q-decisions into single output file

**Options Considered:**
1. Merge into single file with "Category" field
2. Keep completely separate skills and files
3. Single skill, separate output files ← Chosen

**Decision:** One skill (/q-learn-decide) produces two files: Learnings.md and Decisions.md with same timestamp.

**Rationale:** Different structures serve different purposes. Decisions need Options/Rationale/Consequences. Learnings need Context/Learning/Pattern. Separate files preserve appropriate structure while single skill reduces user effort.

**Consequences:** Created /q-learn-decide skill. Removed separate /q-learnings and /q-decisions.

## 5. Shared Topics List for Learnings and Decisions
**Type:** Implementation
**Topics:** workflow

**Trigger:** Question of whether learnings and decisions should have separate topic lists

**Options Considered:**
1. Separate topic lists for each
2. Shared topic list ← Chosen

**Decision:** Use single shared topics list in local hook. Both learnings and decisions use the same topics.

**Rationale:** Topics describe the domain (stripe, d1, cloudflare), not the item type. Decisions already have Type field (Strategic/Implementation) for categorization. Shared list simplifies maintenance and enables consistent grep/search.

**Consequences:** Single topics table in q-learn-decide-local.md serves both file types.
