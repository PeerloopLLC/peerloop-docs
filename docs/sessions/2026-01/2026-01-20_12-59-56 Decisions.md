# Session Decisions - 2026-01-20

## 1. Use Path-Based Endpoints for Stream Reaction Filtering
**Type:** Implementation
**Topics:** stream

**Trigger:** Comments were appearing under all posts instead of being filtered per activity.

**Options Considered:**
1. Query parameter filtering `/reaction/?activity_id=xxx` - Not working
2. Path-based endpoints `/reaction/activity_id/{id}/{kind}/` ← Chosen
3. Client-side filtering after fetching all reactions - Inefficient

**Decision:** Use Stream's path-based endpoint format for filtering reactions by activity or parent reaction.

**Rationale:** The query parameter approach silently fails to filter. Path-based endpoints are the correct REST API format for Stream.io and properly filter at the API level.

**Consequences:** Updated `StreamReactions.filter()` method in `src/lib/stream.ts` to construct path-based URLs based on filter options.

## 2. Accept Top-Level Comment Count Display
**Type:** Implementation
**Topics:** stream

**Trigger:** User asked about showing total comment counts (including nested replies) vs top-level only.

**Options Considered:**
1. Show top-level comment count only (what Stream provides) ← Chosen
2. Calculate total count server-side by traversing all reactions
3. Store our own count in the database

**Decision:** Use Stream's `reaction_counts.comment` as-is, which shows top-level comments only.

**Rationale:** This matches the pattern used by YouTube, Facebook, and other platforms. It's simpler, avoids extra API calls, and users understand "3 comments" means 3 top-level threads.

**Consequences:** Comment button shows count like "Comment 2" reflecting top-level comments. Nested reply counts are visible when expanding threads.
