## Zod for Schema Validation

### Single Source of Truth Pattern
- Define schema with Zod (`z.object()`, `z.enum()`, etc.)
- Export types via `z.infer<typeof Schema>`
- No manual type definitions to maintain - types derived automatically
- Reduced ~300 lines of manual validation to ~80 lines with Zod

### Implementation Structure
```
src/lib/schemas/page-spec.ts      ← Zod schema (defines structure + exports types)
src/lib/types/page-spec.ts        ← Re-exports types (backward compatibility)
src/lib/validation/page-spec-validator.ts ← Validation functions
```

### Zod Error Handling
- `safeParse()` returns `{ success, data }` or `{ success: false, error }`
- `ZodError.issues` contains array of `{ path, message }`
- Wrap in custom error class for cleaner messages:
```typescript
static fromZodError(error: ZodError, context?: string): PageSpecValidationError {
  const issues = error.issues.map((issue) => {
    const path = issue.path.join('.');
    return path ? `${path}: ${issue.message}` : issue.message;
  });
  return new PageSpecValidationError(`${prefix}Validation failed:\n  ${issues.join('\n  ')}`);
}
```

## Tailwind CSS v4 Syntax

### Gradient Classes Changed
- Tailwind v4 uses `bg-linear-to-b` (CSS native) instead of `bg-gradient-to-b`
- VS Code diagnostics may show suggestion but original v4 syntax is correct
- Don't "fix" `bg-linear-to-b` to `bg-gradient-to-b` - the former is the modern syntax

## Claude Code 2.1.1 Behavior

### Skill File Location
- Global skills: `~/.claude/commands/` (NOT in git)
- Project skills: `<project>/.claude/commands/` (tracked by git)
- Skills should default to project-level for version control
- Move global skills to project with: `mv ~/.claude/commands/*.md <project>/.claude/commands/`

### In-Place Skill Updates
- Changes to skill files are immediately available (no restart needed)
- Skills are read from disk when invoked

## Astro Component Validation

### Runtime Validation Pattern
```astro
---
import { parsePageSpec } from '@lib/validation/page-spec-validator';

interface Props {
  pageData: unknown;  // Accept unknown
  context?: string;
}

const { pageData: rawData, context } = Astro.props;
const pageData = parsePageSpec(rawData, context);  // Validate at runtime
---
```

- Accept `unknown` in props, validate before use
- Throws descriptive error if JSON is malformed
- Catches issues at build time (SSG) or request time (SSR)

## TypeScript and JSON Files

### TypeScript Doesn't Validate JSON
- `tsc --noEmit` only checks `.ts/.tsx` files
- JSON imports are not validated against TypeScript interfaces at compile time
- Need explicit runtime validation for JSON files
- CLI script can provide validation: `npx tsx scripts/validate-page-spec.ts <file>`
