# Session Decisions - 2026-01-23

## 1. Add SEEDDATA Phase for Database Seeding Strategy
**Type:** Strategic
**Topics:** d1, workflow, testing

**Trigger:** Discussion about MBA-2017 vs MacMini having inconsistent data for manual testing.

**Options Considered:**
1. Accept the discrepancy as a workaround ← Current state
2. Create seed data tooling to ensure consistent data across environments ← Chosen

**Decision:** Add a new SEEDDATA phase to PLAN.md between TESTING and POLISH.

**Rationale:** Consistent seed data across environments enables:
- Reproducible manual testing on both machines
- Testing empty state scenarios systematically
- Easier onboarding of new environments

**Consequences:** PLAN.md updated with SEEDDATA phase containing:
- SEEDDATA.STRATEGY - Create seed data organically, export to SQL migration
- SEEDDATA.TOOLING - Scripts for seed export/apply
- SEEDDATA.EMPTY_STATE - Test app with empty database

## 2. Decompose CourseDetail Before Testing
**Type:** Implementation
**Topics:** testing, workflow

**Trigger:** CourseDetail.tsx is ~490 lines with 15+ distinct sections, making direct testing unwieldy.

**Options Considered:**
1. Test the monolithic component as-is
2. Decompose into subcomponents first, then test ← Chosen

**Decision:** Break CourseDetail.tsx into subcomponents before writing tests. Establish this as the process for all large components.

**Rationale:**
- Smaller components are easier to test in isolation
- Subcomponents can be reused
- Reduces test complexity and maintenance burden

**Consequences:**
- PLAN.md updated with decomposition process (>200 lines or 5+ sections → decompose first)
- CDET decomposition task added as next step before CDET tests
- Proposed subcomponents: CourseHero, CoursePriceCard, CourseObjectives, CourseCurriculum, CoursePrerequisites, CourseTestimonials, CourseSidebar, CourseSTList

## 3. D1_DATABASE_ID Not Needed on MacMini
**Type:** Implementation
**Topics:** d1, cloudflare

**Trigger:** After adding D1_DATABASE_ID to .dev.vars for MBA-2017, questioned whether MacMini also needs it.

**Options Considered:**
1. Add D1_DATABASE_ID to both machines for consistency
2. Only add to MBA-2017 where it's actually needed ← Chosen

**Decision:** D1_DATABASE_ID is only required on MBA-2017. MacMini uses native D1 binding which reads database from `wrangler.toml`.

**Rationale:** The D1 REST fallback code only runs when `ENABLE_D1_REST_FALLBACK=true`, which is only set on MBA-2017. Adding unnecessary env vars to MacMini would be confusing.

**Consequences:**
- `.dev.vars.example` updated to show D1_DATABASE_ID as commented out (MBA-2017 only)
- PLAN.md clarifies this is MBA-2017 specific
