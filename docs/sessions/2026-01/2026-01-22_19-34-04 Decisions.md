# Session Decisions - 2026-01-22

## 1. No Admin Messaging Feature
**Type:** Strategic
**Topics:** workflow

**Trigger:** Discussion about whether to add "Contact Admin" messaging capability.

**Options Considered:**
1. Add admin messaging via `/messages?support=true` parameter
2. Use designated support user account
3. Don't add admin messaging - use existing contact form ← Chosen

**Decision:** Admin/support messaging will not be part of the Messages feature. The existing contact form at `/contact` handles support requests.

**Rationale:** For an MVP-scale app, support tickets are a different workflow than peer-to-peer messaging. The contact form already exists and is appropriate for the app's size. Real support would use external tools (Zendesk, Intercom) when scaling.

**Consequences:** Messages feature remains focused on peer-to-peer communication: students ↔ teachers ↔ creators.

## 2. Polling Over WebSockets for Message Updates
**Type:** Implementation
**Topics:** astro

**Trigger:** Need real-time message updates in the Messages component.

**Options Considered:**
1. WebSocket real-time connection
2. Server-Sent Events (SSE)
3. Polling at 10-second interval ← Chosen

**Decision:** Use polling at 10-second intervals for message updates in MVP.

**Rationale:** WebSockets add significant complexity (connection management, reconnection logic, server infrastructure). Polling is simple, works with Cloudflare Pages, and is acceptable for MVP. WebSockets can be added later when real-time requirements grow.

**Consequences:**
- Simpler implementation
- Messages appear within 10 seconds (not instant)
- Lower infrastructure requirements
- Out of scope: WebSocket upgrades, typing indicators, read receipts
