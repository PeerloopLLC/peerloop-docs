# Session Learnings - 2026-01-09

## RFC System Design

### Insight: Separating Raw Input from Actionable Work

When processing client change requests after initial implementation has begun, a structured approach is essential:

1. **Preserve the original** - Save raw client input as `original.txt` before any processing
2. **Document formally** - Create a formatted CD-XXX.md with structured sections
3. **Make actionable** - Create RFC.md with checkbox items for tracking
4. **Enable feedback** - Include Comments section for dated follow-up notes

### Rationale for RFC vs Direct Implementation

The first 34 client documents (CD-001 to CD-034) were processed during the design phase before implementation. New documents (CD-035+) arrive when code already exists, requiring:
- Careful impact analysis
- Grouped review before implementation
- Tracking of what changes are pending

### File Organization Pattern

```
RFC/
├── INDEX.md           # Lookup table (status, counts)
└── CD-XXX/
    ├── original.txt   # Raw client input
    ├── CD-XXX.md      # Formatted source + Comments section
    └── RFC.md         # Actionable checklist
```

## Skill Design

### `/q-add-client-note` Workflow

Two-phase approach works well:
1. **Analysis Phase** - Present grouped changes, questions, priorities
2. **Creation Phase** - Only create RFC folder after user approval

This prevents creating artifacts for notes that may need discussion first.

## Documentation Updates

### CLAUDE.md Integration

When adding new systems (like RFC), update:
1. Project Structure section (add folder)
2. Research Reference tables (add lookup guidance)
3. Dedicated section explaining the system

This ensures Claude knows where to look and how to use the system.
