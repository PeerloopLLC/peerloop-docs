# Session Decisions - 2026-01-29

## 1. Capability-Based Menu Visibility Over Role-Based
**Type:** Implementation
**Topics:** auth, astro

**Trigger:** Refactoring DashNavbar to use new CurrentUser system.

**Options Considered:**
1. Keep old `roles: ['student', 'creator']` array system ← Deprecated
2. Use capability flags like `requiresAuth: true` and `capabilities: ['canTeachCourses']` ← Chosen
3. Inline all visibility logic in filter function

**Decision:** Menu items now use `requiresAuth: boolean` for any-authenticated-user items and `capabilities: string[]` for specific capability requirements.

**Rationale:** Capabilities map directly to the new schema columns (`can_take_courses`, `can_teach_courses`, `can_create_courses`). This is more maintainable than mapping old role names and avoids confusion about deprecated flags.

**Consequences:**
- DashNavbar MenuItem interface updated
- `getVisibleItems()` callback simplified
- Menu items are capability-based, not role-based

## 2. Initialize CurrentUser in DashNavbar, Not DashLayout
**Type:** Implementation
**Topics:** astro, auth

**Trigger:** Deciding where to call `initializeCurrentUser()`.

**Options Considered:**
1. Initialize in Astro layout (.astro file) and pass as prop
2. Initialize in DashNavbar React component ← Chosen
3. Initialize in a separate React context provider

**Decision:** Call `initializeCurrentUser()` in DashNavbar's `useEffect`.

**Rationale:** DashNavbar is already a React island that needs the user state. Astro layouts can't easily manage client-side async initialization. A separate context provider adds complexity for minimal benefit since DashNavbar is always present on dashboard pages.

**Consequences:**
- DashNavbar is self-contained for user state
- No need for React Context provider
- MoreSlidePanel receives user as prop from DashNavbar
