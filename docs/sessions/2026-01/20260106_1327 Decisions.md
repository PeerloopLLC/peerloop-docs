# Session Decisions - 2026-01-06

## Decision 1: Use ENABLE_D1_REST_FALLBACK Instead of D1_MODE

**Type**: Strategic
**Driver**: User-driven
**Working On**: D1 REST API fallback site-wide implementation
**Trigger**: Discovered overlap between documented `D1_MODE=local|remote` and implemented `ENABLE_D1_REST_FALLBACK=true`

**Options Considered**:
1. Refactor to use `D1_MODE=local|remote` as originally documented
2. Keep `ENABLE_D1_REST_FALLBACK=true` and remove D1_MODE <- **Chosen**
3. Keep both variables (confusing)

**Decision**: Keep the implemented `ENABLE_D1_REST_FALLBACK` variable and remove all references to `D1_MODE`.

**Rationale**:
- `ENABLE_D1_REST_FALLBACK` is already implemented and working
- Binary flag (true/false) is clearer than mode switching (local/remote)
- The variable name explicitly describes what it enables
- Avoids refactoring working code

**Consequences**:
- Updated `docs/tech/tech-013-devcomputers.md` to remove D1_MODE references
- Updated `.dev.vars.example` to use new variable
- Updated `.dev.vars` to remove D1_MODE

## Decision 2: Keep Cloudflare Credentials Uncommented on All Machines

**Type**: Implementation
**Driver**: User-driven
**Working On**: Configuration file cleanup
**Trigger**: User asked if credentials need to be commented/uncommented like the old D1_MODE pattern required

**Options Considered**:
1. Comment credentials on MacMini, uncomment on MBA-2017 (old pattern)
2. Keep credentials uncommented everywhere <- **Chosen**
3. Move credentials to machine-specific config files

**Decision**: Cloudflare credentials (`CLOUDFLARE_ACCOUNT_ID`, `CLOUDFLARE_API_TOKEN`) stay uncommented in `.dev.vars` on all machines. Only `ENABLE_D1_REST_FALLBACK` needs to be toggled.

**Rationale**:
- On MacMini: Native D1 binding takes priority, credentials are never read (harmless)
- On MBA-2017: Fallback enabled, credentials are used
- In Production: Native binding + production check = credentials ignored even if present
- Simplifies configuration - one variable to toggle instead of three
- Reduces chance of configuration errors

**Consequences**:
- Updated `.dev.vars` and `.dev.vars.example` with clear documentation
- Added notes explaining that credentials are ignored when native binding is available
- Only `ENABLE_D1_REST_FALLBACK` line needs to be commented/uncommented per machine

## Decision 3: Configuration-Based Rather Than Detection-Based Fallback

**Type**: Strategic
**Driver**: Claude-driven (from previous session, confirmed this session)
**Working On**: D1 REST API fallback architecture
**Trigger**: User asked why there's no actual machine detection or D1 testing

**Options Considered**:
1. Auto-detect machine and enable fallback automatically
2. Test D1 connection and fall back on failure
3. Require explicit opt-in via environment variable <- **Chosen**

**Decision**: Keep the explicit opt-in pattern where developer must set `ENABLE_D1_REST_FALLBACK=true`.

**Rationale**:
- Production should never silently fall back to REST API
- Makes it clear when you're using fallback vs native binding
- Prevents accidental REST API usage on MacMini or production
- Developer is aware they're in a degraded mode
- Startup hook already displays machine info for awareness

**Consequences**:
- Developers must manually configure their `.dev.vars` for MBA-2017
- Console log `[MBA-2017 Dev] Using D1 REST API fallback` confirms when active
- No risk of production accidentally using REST API
