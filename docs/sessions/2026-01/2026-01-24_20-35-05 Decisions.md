# Session Decisions - 2026-01-24

## 1. Extend Categories Table Schema
**Type:** Implementation
**Topics:** d1, testing

**Trigger:** API test expected `description`, `icon`, `is_active` columns on categories table, but schema only had 4 columns.

**Options Considered:**
1. Adjust test to match current minimal schema
2. Add all planned fields (description, icon, is_active) ← Chosen
3. Add only description and icon

**Decision:** Extended the categories table to include `description`, `icon`, and `is_active` columns.

**Rationale:** The page spec (`src/data/pages/admin/categories.json`) explicitly listed "Description field" and "Icon field" as "Planned, not implemented." The test was anticipating what the design said SHOULD exist.

**Consequences:**
- Updated `0001_schema.sql` with new columns
- Updated `0002_seed.sql` with category descriptions, emoji icons, and is_active flags
- Categories now support soft-disable and richer metadata for UI display

## 2. Adopt Two-File Migration System
**Type:** Strategic
**Topics:** d1, workflow, testing

**Trigger:** 19 migration files with scattered ALTER TABLE statements caused test failures (duplicate column errors) when applying to fresh databases.

**Options Considered:**
1. Two-file system (0001_schema + 0002_seed) ← Chosen
2. Separate directories (schema/, seed/, upgrades/)
3. Current + Archive pattern

**Decision:** Consolidated all migrations into two files:
- `0001_schema.sql` - Complete schema (47 tables, all indexes)
- `0002_seed.sql` - All seed data

**Rationale:**
- Simplest approach for pre-production development
- Single source of truth for schema
- No ALTER TABLE failures on fresh DBs
- Easy to modify until production launch
- Can add incremental migrations (0003+) when production has real data

**Consequences:**
- Deleted 17 migration files (0003-0019)
- All tests pass (2458 tests)
- Added documentation to CLAUDE.md about the migration strategy
- Schema changes now edit 0001 directly instead of creating new ALTER TABLE migrations
