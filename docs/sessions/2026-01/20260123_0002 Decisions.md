# Session Decisions - 2026-01-23

## 1. AANA Component Structure
**Type:** Implementation
**Topics:** architecture, analytics

**Trigger:** Implementing Admin Analytics (AANA) page following the plan.

**Options Considered:**
1. Single monolithic AdminAnalytics component ← Not chosen
2. Separate section components in `analytics/admin/` folder ← Chosen
3. Reuse CANA sub-components with props

**Decision:** Create dedicated sub-components for AANA in `src/components/analytics/admin/`:
- `AdminKPICards.tsx` - 5 executive KPI cards
- `RevenueSection.tsx` - Revenue charts and metrics table
- `UsersSection.tsx` - User growth and funnel
- `CoursesSection.tsx` - Course/creator performance
- `STSection.tsx` - S-T pipeline and flywheel
- `EngagementSection.tsx` - Session and engagement metrics

**Rationale:** AANA has different data requirements and metrics than CANA. While chart components (AreaChart, PieChart, etc.) are shared, the section components have unique layouts and data shapes. Dedicated components maintain clarity and allow independent evolution.

**Consequences:** Created 7 new files in `src/components/analytics/admin/` plus the main orchestrator.

## 2. TANA API Structure
**Type:** Implementation
**Topics:** api, analytics

**Trigger:** Implementing S-T Analytics (TANA) page.

**Options Considered:**
1. Single endpoint returning all data ← Not chosen
2. Four focused endpoints matching data needs ← Chosen
3. Reuse CANA endpoints with role filtering

**Decision:** Create four dedicated endpoints for TANA:
- `GET /api/me/st-analytics` - Summary KPIs
- `GET /api/me/st-analytics/earnings` - Earnings time series
- `GET /api/me/st-analytics/sessions` - Session metrics and patterns
- `GET /api/me/st-analytics/students` - Progress distribution

**Rationale:** Follows the established pattern from CANA. Separate endpoints allow:
- Parallel fetching for faster initial load
- Independent caching potential
- Clearer responsibility per endpoint
- Easier testing and debugging

**Consequences:** Created 4 API files in `src/pages/api/me/st-analytics/`.

## 3. Fix CANA to Match PageSpecView Pattern
**Type:** Implementation
**Topics:** astro, workflow

**Trigger:** User pointed out AANA was incorrectly removing PageSpecView instead of using dual-view pattern.

**Options Considered:**
1. Only fix AANA ← Not chosen
2. Fix both AANA and CANA to match STUD pattern ← Chosen

**Decision:** Updated both CANA and AANA (and TANA) to use the dual-view pattern:
- PageSpecView in default slot
- Real component in `content` slot
- `pageCode` prop on AppLayout

**Rationale:** Consistency across all analytics pages. The STUD page demonstrated the correct pattern that was being followed elsewhere in the codebase.

**Consequences:** All three analytics pages (AANA, CANA, TANA) now support both spec view and production view toggling.
