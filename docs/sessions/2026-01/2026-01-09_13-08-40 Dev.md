# Development Session Log - 2026-01-09_13-08-40

## Session Focus
Fix D1 REST API fallback not working on MBA-2017 after the dotenv import fix for Cloudflare Pages.

## Problem Statement
The HOME page was not getting data from remote D1 on MBA-2017, even though:
- Code worked on MacMini with local D1
- Code worked on Cloudflare preview deploy with remote D1
- `ENABLE_D1_REST_FALLBACK=true` was set in `.dev.vars`

## Root Cause Analysis

### Initial Investigation
Added diagnostic logging to trace the flow:
- `d1-rest.ts`: `loadDevVars()` function
- `db/index.ts`: `isRestFallbackEnabled()` and `getDB()` functions

### Discovery
Diagnostic logs revealed:
```
[D1-REST] loadDevVars failed: require is not defined
[DB] isRestFallbackEnabled check: {
  ENABLE_D1_REST_FALLBACK: undefined,
  ...
}
```

The `require('dotenv')` call in `loadDevVars()` was failing silently because Vite's ESM environment doesn't provide CommonJS `require()`.

## Solution Implemented

### 1. astro.config.mjs - Use Vite's loadEnv
```javascript
import { loadEnv } from 'vite';

const mode = process.env.NODE_ENV || 'development';
const env = loadEnv(mode, process.cwd(), '');
Object.assign(process.env, env);
```

### 2. src/lib/db/d1-rest.ts - Remove dotenv loading
Removed the entire `loadDevVars()` function and its invocation. Added comment explaining env vars are now loaded via Vite.

### 3. Removed dotenv dependency
```bash
npm uninstall dotenv
```

## Technical Notes
- Vite bundles dotenv internally for its `loadEnv()` function
- The third parameter `''` to `loadEnv()` loads ALL env vars, not just `VITE_` prefixed ones
- The `.env -> .dev.vars` symlink works seamlessly with this approach
- Environment vars are loaded at config time, before any SSR code runs
