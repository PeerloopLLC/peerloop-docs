# Session Learnings - 2026-01-20

## 1. BBB Uses Checksum-Based API Authentication
**Topics:** video, auth

**Context:** Implementing BigBlueButton adapter for VideoProvider interface

**Learning:** BBB authenticates API calls using SHA1 checksum. The formula is: `SHA1(apiName + queryParams + secret)`. All requests are GET with query parameters, unlike PlugNmeet which uses POST with JSON and HMAC headers.

**Pattern:**
```typescript
async function createChecksum(apiName: string, queryString: string, secret: string): Promise<string> {
  const data = `${apiName}${queryString}${secret}`;
  const encoder = new TextEncoder();
  const dataBuffer = encoder.encode(data);
  const hashBuffer = await crypto.subtle.digest('SHA-1', dataBuffer);
  const hashArray = Array.from(new Uint8Array(hashBuffer));
  return hashArray.map((b) => b.toString(16).padStart(2, '0')).join('');
}
```

## 2. BBB Returns XML, Not JSON
**Topics:** video, astro

**Context:** Processing BBB API responses in Cloudflare Workers environment

**Learning:** BBB API returns XML responses, requiring parsing. For simple responses, regex-based extraction works well. For complex nested structures (like recordings), iterative parsing is needed. Consider using a proper XML parser for production.

## 3. VideoProvider Abstraction Enables Platform Swapping
**Topics:** video, workflow

**Context:** Client changed from PlugNmeet to BigBlueButton mid-implementation

**Learning:** The VideoProvider interface abstraction paid off immediately. When Brian switched from PlugNmeet to BBB, only the adapter needed implementing - all other code (APIs, pages, components) remained unchanged. The interface defines: createRoom, endRoom, getJoinUrl, getRoomInfo, isRoomActive, getRecordings, deleteRecording, parseWebhook.

## 4. BBB Join URLs Require Password Parameter
**Topics:** video

**Context:** Building session join flow

**Learning:** BBB requires either attendeePW or moderatorPW in the join URL. These passwords are returned when creating a meeting, so they need to be stored (in session DB or in-memory cache) to generate join URLs later without re-querying BBB. For our use case, we store them in the sessions table (bbb_attendee_pw, bbb_moderator_pw).

## 5. Session Booking Requires Conflict Detection
**Topics:** d1, video

**Context:** Creating session booking API

**Learning:** Double-booking prevention requires checking both teacher and student schedules. The SQL overlap check pattern: `(scheduled_start < ? AND scheduled_end > ?) OR (scheduled_start >= ? AND scheduled_start < ?)` handles all overlap cases (partial overlaps, contained sessions, spanning sessions).
