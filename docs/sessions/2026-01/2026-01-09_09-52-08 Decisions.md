# Session Decisions - 2026-01-09

## Decision 1: RFC System for Post-Implementation Changes

**Context:** Client documents CD-001 through CD-034 were processed during design phase. CD-035+ arrive after implementation has begun.

**Decision:** Create RFC (Request for Changes) system separate from original client-docs workflow.

**Rationale:**
- Changes to existing code require impact analysis
- Multiple RFCs should be reviewed together before implementation
- Need tracking mechanism (checkboxes) for implementation progress

**Structure:**
```
RFC/
├── INDEX.md           # Lookup table
└── CD-XXX/
    ├── original.txt   # Raw input
    ├── CD-XXX.md      # Formatted document
    └── RFC.md         # Actionable checklist
```

---

## Decision 2: INDEX.md vs README.md

**Context:** RFC folder needs a central file for looking up RFCs.

**Decision:** Use `INDEX.md` instead of `README.md`.

**Rationale:**
- File serves as lookup table, not general documentation
- Claude references it to find open RFCs
- Clearer naming for the file's purpose

---

## Decision 3: Preserve Original Input

**Context:** Client notes arrive as pasted text or files.

**Decision:** Save raw input as `original.txt` before any processing.

**Rationale:**
- Preserves exact wording from client
- CD-XXX.md is formatted/interpreted version
- Can reference original if questions arise

---

## Decision 4: Comments Section in CD Documents

**Context:** After initial RFC creation, client/team may have follow-up feedback.

**Decision:** Add "## Comments" section at bottom of each CD-XXX.md with dated entries.

**Format:**
```markdown
## Comments

**YYYY-MM-DD (Name):** Comment text
```

**Rationale:**
- Keeps feedback with source document
- Dated entries show evolution of thinking
- Can add corresponding tasks to RFC.md

---

## Decision 5: RFC Review Timing

**Context:** When should RFCs be implemented?

**Decision:** Complete base implementation from original 34 documents first, then review all RFCs together.

**Rationale:**
- Avoids churn during initial build
- Allows batching related changes
- Better impact assessment when base is stable
