# Session Decisions - 2026-01-22

## 1. Use Recharts with Wrapper Pattern for Charts
**Type:** Implementation
**Topics:** astro, workflow

**Trigger:** Need charting library for analytics pages (CANA, TANA, AANA).

**Options Considered:**
1. Recharts with wrapper components ← Chosen
2. Tremor (pre-built dashboard components)
3. Nivo (declarative charts)
4. Chart.js with react-chartjs-2

**Decision:** Use Recharts wrapped in abstraction components (`src/components/charts/`) for future migration flexibility.

**Rationale:** Recharts is mature, React-native, and well-documented. Wrappers isolate library-specific code so we can swap to Tremor/Nivo later if needed without changing consumers.

**Consequences:** Created `types.ts`, `AreaChart.tsx`, `BarChart.tsx`, `PieChart.tsx`, `FunnelChart.tsx`, `index.ts` in `src/components/charts/`.

## 2. Separate Analytics Pages per Role (CANA/TANA/AANA)
**Type:** Strategic
**Topics:** workflow

**Trigger:** Need analytics for creators, student-teachers, and admins with different scopes.

**Options Considered:**
1. Separate pages per role (CANA, TANA, AANA) ← Chosen
2. Single unified analytics page with role-based sections
3. Add analytics tabs to existing dashboards

**Decision:** Create three distinct analytics pages, each optimized for their role's perspective and metrics.

**Rationale:**
- Each role has different metrics needs and scope
- Admin sees platform-wide, Creator sees their courses, S-T sees personal teaching
- Shared chart components reduce duplication
- Cleaner navigation and mental model

**Consequences:**
- CANA implemented at `/dashboard/creator/analytics`
- TANA spec'd at `/dashboard/teaching/analytics`
- AANA spec'd at `/admin/analytics`

## 3. API Endpoint Pattern: Main + Sub-endpoints
**Type:** Implementation
**Topics:** d1, workflow

**Trigger:** Analytics pages need multiple data types (summary, trends, tables).

**Options Considered:**
1. Single endpoint returning all data
2. Main endpoint + sub-endpoints ← Chosen
3. GraphQL-style single query

**Decision:** Use main endpoint for summary KPIs plus sub-endpoints for specific data (e.g., `/api/me/creator-analytics` + `/api/me/creator-analytics/enrollments`).

**Rationale:**
- Parallel fetching improves perceived performance
- Smaller payloads per request
- Can load summary quickly, charts progressively
- Matches existing API patterns

**Consequences:** CANA has 7 endpoints, TANA planned for 4, AANA planned for 6.

## 4. Page Views in Funnel Shows "Coming Soon" Placeholder
**Type:** Implementation
**Topics:** workflow

**Trigger:** Funnel analysis needs page view data for complete conversion picture.

**Options Considered:**
1. Show placeholder "Coming soon" ← Chosen
2. Omit page views step entirely
3. Implement tracking immediately

**Decision:** Show "Page Views" as first funnel step with "Coming soon" indicator rather than omitting it.

**Rationale:**
- Indicates the metric exists and will be tracked
- Better UX than confusing partial funnel
- Tracking implementation deferred (needs PostHog/analytics setup)
- Easy to fill in later

**Consequences:** Funnel shows 4 steps with first grayed out. No tracking infrastructure needed yet.
