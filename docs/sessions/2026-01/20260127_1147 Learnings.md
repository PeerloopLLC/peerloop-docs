# Session Learnings - 2026-01-27

## 1. SQLite Boolean Trap in D1 API Responses
**Topics:** d1, testing

**Context:** The CEAR (Earnings Detail) API test for `stripe_status.payouts_enabled` expected `true` but received `1`.

**Learning:** D1/SQLite stores booleans as integers (0/1). When building API response objects, every boolean field from D1 must be coerced with `!!`. The `connected` field in the same endpoint already used `!!user.stripe_account_id`, but `payouts_enabled` was passed through raw. This is a per-field bug, not a systemic fix — each endpoint must audit its boolean coercions individually.

**Pattern:**
```typescript
// BAD: passes SQLite integer through
payouts_enabled: user.stripe_payouts_enabled,  // returns 1, not true

// GOOD: coerce to boolean
payouts_enabled: !!user.stripe_payouts_enabled, // returns true
```

## 2. Child Component Fetch Mocks in Dashboard Tests
**Topics:** testing

**Context:** CreatorDashboard and TeacherDashboard both render sub-components that call `fetch()` on mount — `InstructorFeedToggle` and `AvailabilityQuickView` respectively. These consumed mock fetch responses meant for the dashboard data, causing test failures.

**Learning:** When testing a parent component that renders children with their own `useEffect` + `fetch` calls, the `setupFetchSuccess` helper must provide mock responses for ALL fetches that fire during render — not just the parent's. The fix is to include extra `mockResolvedValueOnce` calls in the setup helper so child components get their own responses.

**Pattern:**
```typescript
function setupFetchSuccess(data = mockDashboardData) {
  mockFetch.mockResolvedValueOnce({ ok: true, json: () => Promise.resolve(data) });
  // AvailabilityQuickView also fetches on mount
  mockFetch.mockResolvedValueOnce({ ok: true, json: () => Promise.resolve({ slots: [] }) });
}
```

## 3. Duplicate Text in React Testing Library
**Topics:** testing

**Context:** `getByText('React Fundamentals')` and `getByText('My Students')` threw "found multiple elements" errors because the text appeared in both a heading/card and an image alt attribute or link.

**Learning:** Dashboard layouts frequently duplicate text across headings, card titles, link labels, and image alt texts. Use `getAllByText().length` assertions instead of `getByText()` when the text could reasonably appear in multiple DOM locations.

## 4. Population Script: Shell Script as Source of Truth
**Topics:** testing, workflow

**Context:** Built a Node.js script to parse `test-{CODE}.sh` shell scripts and populate `testCoverage` in page JSON files. The script extracts `COMPONENT_TEST` and `API_TESTS` variables plus endpoint comments from headers.

**Learning:** The shell scripts serve dual purpose — they're both executable test orchestrators AND structured data sources. Parsing variables (`COMPONENT_TEST="..."`) and header comments (`# API Endpoints: - GET /api/...`) from shell scripts is reliable because the format is standardized across all 59 scripts. The population script can be re-run after adding new test files to keep JSONs current.

## 5. Disambiguating Spec Data from Reality with Naming
**Topics:** workflow, testing

**Context:** The `apiCalls` field in page JSON contained original design endpoints (e.g., `/api/creators/me/analytics/export`) that were never implemented. This mixed aspirational and actual data, making test coverage assessment unreliable.

**Learning:** Renaming `apiCalls` → `plannedApiCalls` across schema, component, and all 67 JSON files cleanly separates design intent from implementation reality. The `testCoverage.apiTests` section now serves as ground truth for what's actually built and tested, while `plannedApiCalls` preserves the original spec without polluting the test baseline.
