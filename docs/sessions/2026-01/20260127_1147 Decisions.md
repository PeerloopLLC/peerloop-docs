# Session Decisions - 2026-01-27

## 1. Rename apiCalls to plannedApiCalls
**Type:** Strategic
**Topics:** workflow, testing

**Trigger:** Discovered that `apiCalls` in page JSON contained a mix of implemented endpoints and aspirational design endpoints (e.g., `/export`), plus a bogus `granularity` row. The `testCoverage.apiTests` section (populated from shell scripts) reflected reality, creating confusion about which source of truth to trust.

**Options Considered:**
1. Strip unimplemented endpoints from `apiCalls` to match reality
2. Rename `apiCalls` → `plannedApiCalls` to clarify it's design intent ← Chosen
3. Leave as-is and document the difference

**Decision:** Rename `apiCalls` to `plannedApiCalls` across the Zod schema, PageSpecView component, all 67 JSON files, and helper scripts.

**Rationale:** Preserves all original spec data without losing information, while eliminating ambiguity. The `testCoverage.apiTests` section serves as the "what's actually built" layer, and `plannedApiCalls` is the "what was designed" layer. No data loss, clear semantics.

**Consequences:** Changed `PageSpecSchema`, `PageSpecView.astro`, `page-spec.ts` types, all JSON files, `populate-test-coverage.ts`, and `parse-page-md.ts`. Section heading in rendered PageSpecView now reads "Planned API Calls".

## 2. Delete Orphan settings.json
**Type:** Implementation
**Topics:** workflow

**Trigger:** Two JSON files (`settings.json` and `settings/index.json`) both used code `SETT` with the same route `/settings`. Needed to determine which was canonical.

**Options Considered:**
1. Delete `settings.json` (the older monolithic design) ← Chosen
2. Delete `settings/index.json` (the hub page)
3. Give one a new code

**Decision:** Delete `settings.json`. It was a pre-split monolithic design (status "In Scope") superseded by the hub + sub-pages architecture (`settings/index.json` with status "Implemented" plus dedicated `profile.json`, `payments.json`, `notifications.json`, `security.json`).

**Rationale:** `settings.json` was not in PAGES-MAP.md, had status "In Scope" (never built), and its detailed data requirements/error handling were already distributed across the sub-page JSONs. The hub page (`settings/index.json`) is the implemented reality.

**Consequences:** Down from 67 to 66 JSON files. No duplicate SETT code.

## 3. Fix discuss.json Code from CHAT to CDIS
**Type:** Implementation
**Topics:** workflow

**Trigger:** Two JSON files used code `CHAT` — `courses/[slug]/chat.json` (Course Chat) and `courses/[slug]/discuss.json` (Course Discussion). The test script `test-CDIS.sh` existed but couldn't match to any JSON.

**Options Considered:**
1. Change discuss.json code to CDIS to match PAGES-MAP.md and test script ← Chosen
2. Keep both as CHAT
3. Use a different code

**Decision:** Change `courses/[slug]/discuss.json` code from `CHAT` to `CDIS`, matching PAGES-MAP.md's canonical assignment and the existing `test-CDIS.sh` script.

**Rationale:** PAGES-MAP.md (single source of truth for page codes) already had CDIS for Course Discussion. The JSON was simply misnamed.

**Consequences:** No more duplicate CHAT codes. `test-CDIS.sh` now correctly matches and populates `testCoverage` for the discuss page.
