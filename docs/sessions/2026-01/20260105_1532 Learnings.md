# Session Learnings - 2026-01-05

## Technical

### Stripe Connect Integration
- **Stripe API versions use codenames**: The API version format is `YYYY-MM-DD.codename` (e.g., `2025-12-15.clover`). TypeScript will error if you use an outdated version string.
- **Express accounts are simplest for marketplace payouts**: Express accounts handle KYC/compliance through Stripe's hosted onboarding, reducing platform liability.
- **Separate Charges and Transfers model**: For multi-party splits (Creator + S-T + Platform), use `transfer_group` to link a charge to multiple transfers rather than destination charges.
- **Webhook signature verification is critical**: Always verify `stripe-signature` header using `stripe.webhooks.constructEvent()` before processing events.
- **Test mode is completely isolated**: Test accounts (`pk_test_`, `sk_test_`) exist in a separate sandbox. No interference with live mode accounts, even for the same email.

### bcrypt Password Hashing
- **bcrypt versions matter**: `$2a$` and `$2b$` are different bcrypt versions. A hash generated with one version may not verify correctly if the stored hash uses a different prefix. Always regenerate hashes if verification fails unexpectedly.
- **Seed data password hashes can drift**: If seed files contain static password hashes, they may become invalid if generated incorrectly. Always test login after applying migrations.

### Astro Routing
- **Route collision with index.astro**: Having both `settings.astro` and `settings/index.astro` causes Astro warnings and will be a hard error in future versions. Use only one pattern (prefer `settings/index.astro` for expandability).

## Workflow

### Stripe CLI for Local Development
- **`stripe listen` creates temporary webhook secrets**: The `whsec_...` from `stripe listen` is session-specific. For production, create permanent webhook endpoints in Stripe Dashboard.
- **Two terminals needed for full testing**: Terminal 1 for `npm run dev`, Terminal 2 for `stripe listen --forward-to localhost:4321/api/webhooks/stripe`.

### Testing Payment Flows
- **Creator must have Stripe Connect first**: Can't test checkout until a course creator has connected their Stripe account (has `stripe_account_id` in DB).
- **Test card for Stripe**: `4242 4242 4242 4242` with any future date and any CVC.

## Documentation

### API Endpoint Organization
- **`/api/me/*` pattern for current user endpoints**: Cleaner than passing user ID in query params. Auth is implicit from session cookie.
- **Stripe endpoints grouped under `/api/stripe/*`**: `connect`, `connect-link`, `connect-status` for account management; `/api/checkout/create-session` for payments; `/api/webhooks/stripe` for events.

## Architecture

### Component State Patterns
- **EnrollButton multi-state pattern**: Loading → check auth → check enrollment → show appropriate UI. This CSR personalization pattern (Layer 2 from tech-014) works well for user-specific states on SSR pages.
- **StripeConnectSettings status-driven UI**: Map account status (pending/active/restricted) to config objects with color, icon, title, description, action. Cleaner than multiple conditionals.

### Payment Split Implementation
- **Calculate splits from gross amount**: Platform fee comes from gross, transfers to recipients use remaining amounts. Don't double-deduct.
- **Store `transfer_group` with enrollment ID**: Makes it easy to find related transfers for refunds using `enrollment_${id}` pattern.
