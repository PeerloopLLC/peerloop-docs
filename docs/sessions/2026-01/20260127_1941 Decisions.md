# Session Decisions - 2026-01-27 (Session 130)

## 1. JSON Files Stay Under src/, Not docs/
**Type:** Strategic
**Topics:** workflow, astro

**Trigger:** User wanted to reduce cognitive load — 47 files for one page spread across 6 directory trees.

**Options Considered:**
1. Move JSON to `docs/pages/` structure — rejected, 55+ Astro import changes needed
2. Keep JSON in `src/data/pages/` and enrich metadata ← Chosen

**Decision:** JSON stays under `src/data/pages/` (runtime-accessible). Instead of moving files, enrich each JSON's metadata to serve as a single-lookup index for all related file paths.

**Rationale:** Moving would break 55 Astro imports and build config. Enriching metadata achieves the same cognitive benefit without refactoring.

**Consequences:** Created `tsxComponents[]`, `astroComponents[]`, `docs{}` in metadata.

---

## 2. Replace component with tsxComponents Array
**Type:** Implementation
**Topics:** astro, workflow

**Trigger:** Some pages import multiple React components (HOME has 7, CDET has 2). Single `component` string was insufficient.

**Options Considered:**
1. Keep `component` and add `additionalComponents` — awkward
2. Replace with `tsxComponents[]` array ← Chosen
3. Single `components` array mixing tsx and astro — loses type distinction

**Decision:** Remove `component` field, add `tsxComponents[]` and `astroComponents[]` as separate arrays. Only components directly imported in the `.astro` file are listed (not child components).

**Rationale:** Clear separation of component types; future-proofs for Astro component islands.

**Consequences:** Schema v1.3, all 66 JSONs updated, PageSpecView renders array.

---

## 3. Drop block and status From Schema
**Type:** Strategic
**Topics:** workflow

**Trigger:** Analyzing PAGES-MAP.md data for migration. User decided these fields are no longer needed.

**Options Considered:**
1. Keep and backfill missing block values from PAGES-MAP ← Rejected
2. Drop both fields entirely ← Chosen

**Decision:** Remove `block` (number) and `status` (enum) from PageMetadataSchema. Delete `PageStatusSchema` entirely.

**Rationale:** Block numbers are historical (build sequence) and no longer relevant. Status was inconsistent between PAGES-MAP and JSON (different classification systems). Neither provides value going forward.

**Consequences:** Stripped from all 66 JSONs and schema. PAGES-MAP.md will be repurposed as a simple code→JSON/route→JSON index.

---

## 4. PAGES-MAP.md Becomes a Lightweight Index
**Type:** Strategic
**Topics:** workflow

**Trigger:** All data in PAGES-MAP.md is now either in JSON files or being dropped (block, status, user journeys).

**Options Considered:**
1. Delete PAGES-MAP.md entirely — loses quick lookup convenience
2. Repurpose as code→JSON and route→JSON index ← Chosen
3. Auto-generate from JSON files — chosen approach for the rewrite

**Decision:** Rewrite PAGES-MAP.md as a two-column lookup: page code → JSON path, and route → JSON path. Content will be auto-generated from the 66 JSON files.

**Rationale:** Keeps the convenience of a single lookup file while eliminating data duplication. JSON is the source of truth.

**Consequences:** Not yet implemented — next session work.
