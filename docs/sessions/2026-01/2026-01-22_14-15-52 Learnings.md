# Session Learnings - 2026-01-22

## 1. JavaScript URL() Constructor Accepts Any Protocol
**Topics:** validation, astro

**Context:** Implementing URL validation for social link fields in profile settings.

**Learning:** The JavaScript `new URL()` constructor accepts any protocol scheme - `httpx://`, `ftp://`, `custom://` are all valid URL structures. To validate HTTP/HTTPS only, you must explicitly check `parsed.protocol === 'http:' || parsed.protocol === 'https:'`.

**Pattern:**
```typescript
function isValidUrl(url: string): boolean {
  try {
    const parsed = new URL(url);
    return parsed.protocol === 'http:' || parsed.protocol === 'https:';
  } catch {
    return false;
  }
}
```

## 2. On-Blur Validation Doesn't Affect Dirty State
**Topics:** astro, workflow

**Context:** Implementing on-blur handle validation while maintaining the "hasChanges" dirty state for the Save button.

**Learning:** On-blur validation and dirty state tracking are independent concerns. Field errors (`fieldErrors`) and data changes (`formData` vs `savedData`) use separate state variables, so validation can run without affecting the Save button's enabled/disabled state.

**Pattern:**
```
validateHandleOnBlur() → setFieldError('handle', ...) → fieldErrors state
                                                        ↓
                                            hasChanges is NOT recalculated
                                            (it only watches formData vs savedData)
```

## 3. SQLite INTEGER for Booleans Requires Conversion
**Topics:** d1, astro

**Context:** Notification preferences stored as INTEGER 0/1 in D1, but API should return proper booleans.

**Learning:** When bridging SQLite/D1 with TypeScript, boolean columns stored as INTEGER need explicit conversion: `Boolean(user.email_session_reminder)` when reading, and `value ? 1 : 0` when writing.

## 4. Structured Field Errors for Inline Validation
**Topics:** astro, workflow

**Context:** Moving from string array errors to inline field-level error display.

**Learning:** API returning `{ field: string; message: string }[]` instead of `string[]` enables precise inline error display. Each input can receive its specific error, and errors clear when that specific field is edited.

**Pattern:**
```typescript
// API returns
{ fieldErrors: [{ field: 'handle', message: 'Already taken' }] }

// Component uses
const getFieldError = (field: string) => fieldErrors.find(e => e.field === field)?.message;
<Input error={getFieldError('handle')} />
```
