# Session Decisions - 2026-01-21

## 1. Share Admin Moderation Endpoints with Moderators
**Type:** Implementation
**Topics:** api, auth

**Trigger:** Building the moderator queue (MODQ) required moderation API access for moderators.

**Options Considered:**
1. Create duplicate `/api/mod/*` endpoints with moderator auth ← Rejected (code duplication)
2. Update existing `/api/admin/moderation/*` endpoints to allow both roles ← Chosen
3. Create generic `/api/moderation/*` endpoints used by both interfaces

**Decision:** Update existing admin moderation endpoints to accept `['admin', 'moderator']` roles, with inline restrictions for admin-only actions.

**Rationale:** Avoids code duplication while maintaining role-based restrictions. The admin interface at `/admin/moderation` remains admin-only (via AdminLayout), while moderators access the same APIs through their own interface at `/mod`.

**Consequences:**
- Updated 6 API endpoints: index.ts, [id].ts, dismiss.ts, warn.ts, suspend.ts, remove.ts
- Added permanent suspension admin-only restriction in suspend.ts

## 2. Moderators Cannot Issue Permanent Suspensions
**Type:** Implementation
**Topics:** auth, moderation

**Trigger:** Defining moderator capabilities when building ADMIN.MODERATORSYSTEM.

**Options Considered:**
1. Full moderation parity with admins
2. No suspension capability for moderators
3. Temporary suspensions only (1d, 7d, 30d) for moderators ← Chosen

**Decision:** Moderators can issue 1d, 7d, or 30d suspensions. Permanent suspensions require admin role.

**Rationale:** Moderators need enough authority to address violations promptly, but permanent bans are high-impact decisions that should require admin oversight.

**Consequences:**
- `suspend.ts` checks `session.roles.includes('admin')` before allowing `duration: 'permanent'`
- ModeratorQueue component prompts only offer temp ban options

## 3. Context Actions as Floating Action Button (FAB)
**Type:** Implementation
**Topics:** ui, workflow

**Trigger:** Implementing ADMIN.CONTEXTACTIONS - role-aware quick actions on public pages.

**Options Considered:**
1. Inline action bar in page header ← Rejected (clutters UI for regular users)
2. Floating action button (FAB) in bottom-right ← Chosen
3. Context menu on right-click ← Rejected (not discoverable)
4. Actions panel in sidebar ← Rejected (takes space, not visible by default)

**Decision:** Implement as a FAB that expands to show available actions. Hidden when no actions available.

**Rationale:** FAB is non-intrusive for regular users (shows nothing if not admin/owner/ST) but provides quick access for privileged users. Standard mobile pattern, easily discoverable.

**Consequences:**
- Created ContextActionsPanel component with expand/collapse state
- Shows badge count when collapsed
- Integrated into Course Detail and Creator Profile pages
