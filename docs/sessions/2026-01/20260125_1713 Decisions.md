# Session Decisions - 2026-01-25

## 1. Split Test Documentation into Multiple Files
**Type:** Implementation
**Topics:** documentation, workflow, testing

**Trigger:** TEST-COVERAGE.md grew to 379 lines. User asked if it would benefit from functional grouping like API docs.

**Options Considered:**
1. Keep as single file with better organization
2. Split by test type (API, Components, Pages, Unit) ← Chosen
3. Split by feature domain (Auth, Admin, Courses, etc.)

**Decision:** Split into 4 sub-documents mirroring test directory structure: TEST-API.md, TEST-COMPONENTS.md, TEST-PAGES.md, TEST-UNIT.md. TEST-COVERAGE.md becomes the index.

**Rationale:** Test directory structure is `tests/api/`, `tests/components/`, etc. Documentation that mirrors this structure is intuitive to maintain. When adding a test to `tests/api/admin/`, you know to update TEST-API.md. Feature-domain split would mix API tests and component tests together.

**Consequences:** Created 4 new files, refactored TEST-COVERAGE.md from 379 lines to ~70 lines (index only).

## 2. Add testCoverage Section to PageSpec Schema
**Type:** Strategic
**Topics:** testing, workflow, documentation

**Trigger:** User wants to verify a page and its API dependencies are fully tested before working on it. Asked "How can I know page CANA is ready to work on?"

**Options Considered:**
1. Add test coverage info to PAGES-MAP.md
2. Create separate TEST-MATRIX.md document
3. Add testCoverage section to PageSpec JSON ← Chosen

**Decision:** Extend PageSpec Zod schema (v1.2) with `testCoverage` section containing page test, component test, API tests array, and a summary with `readyToWork` flag.

**Rationale:** PageSpec already has `apiCalls` declaring what APIs the page uses. Adding test coverage there creates a single source of truth for each page. The computed `readyToWork` flag answers the workflow question directly.

**Consequences:**
- Updated `src/lib/schemas/page-spec.ts` (v1.1 → v1.2)
- Added testCoverage to AUSR and CANA PageSpecs as examples
- Created `scripts/populate-test-coverage.ts` for auto-population
- Added npm scripts: `test:coverage`, `test:coverage:write`

## 3. Track Coverage Ratios Instead of Just File Counts
**Type:** Implementation
**Topics:** testing, documentation

**Trigger:** Initial doc sync showed "49 API tests" which seemed complete, but user pointed out this doesn't answer "how much of the codebase is tested?"

**Options Considered:**
1. Keep counting test files only
2. Add codebase counts for context ← Chosen
3. Calculate actual line/branch coverage

**Decision:** TEST-COVERAGE.md summary now shows: Codebase count | Test count | Coverage percentage for APIs and Pages.

**Rationale:** "49 API tests" sounds good until you know there are 168 endpoints (29% coverage). The ratio makes coverage gaps actionable and sets expectations for test completion work.

**Consequences:**
- Summary table now shows: API Endpoints (168) → 49 tests (29%)
- Added detailed Coverage Gaps section listing untested route groups
- TEST-API.md and TEST-PAGES.md headers include coverage ratios
