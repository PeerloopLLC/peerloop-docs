# Session Decisions - 2026-01-20

## 1. Dispute Resolution Model
**Type:** Implementation
**Topics:** d1, workflow

**Trigger:** Need to track session disputes and their resolutions for admin oversight.

**Options Considered:**
1. Simple status field on sessions table
2. Separate disputes table with foreign key to sessions ← Considered
3. Inline dispute fields on sessions table ← Chosen

**Decision:** Add dispute tracking fields directly to the sessions table rather than creating a separate disputes table.

**Rationale:** Disputes are always 1:1 with sessions, so a separate table adds complexity without benefit. The sessions table now includes: `dispute_status`, `dispute_reported_by`, `dispute_reported_at`, `dispute_reason`, `resolution_type`, `resolution_notes`, `resolved_by`, `resolved_at`.

**Consequences:** Migration 0012 added these fields. Simpler queries but sessions table is now wider.

## 2. Recording Fetch Strategy
**Type:** Implementation
**Topics:** video, cloudflare

**Trigger:** Admin needs to access session recordings stored in BBB.

**Options Considered:**
1. Pre-fetch and cache all recording URLs in sessions table ← Partial
2. Fetch from BBB on-demand only
3. Hybrid: cache URL after first fetch ← Chosen

**Decision:** Use hybrid approach - fetch recording URL from BBB on first request, cache in `sessions.recording_url` for subsequent requests.

**Rationale:** Avoids unnecessary BBB API calls while ensuring recordings are always accessible. The BBB webhook already populates `recording_url` when recordings become available, so the on-demand fetch is a fallback for missed webhooks.

**Consequences:** Recording endpoint checks cached URL first, then queries BBB if needed, and caches the result.

## 3. Session Credits as Transaction Records
**Type:** Implementation
**Topics:** stripe, d1

**Trigger:** Dispute resolution can grant free session credits to students or teachers.

**Options Considered:**
1. Dedicated session_credits table
2. Boolean flag on user (has_credit)
3. Transaction record with type='session_credit' ← Chosen

**Decision:** Store session credits as transaction records with `type='session_credit'` and `amount=0`.

**Rationale:** Reuses existing transactions infrastructure, maintains audit trail, and doesn't require new tables. The metadata field stores source session ID for traceability.

**Consequences:** Query for credits: `WHERE type='session_credit' AND status='completed'`. Credit redemption logic TBD (not implemented this session).
