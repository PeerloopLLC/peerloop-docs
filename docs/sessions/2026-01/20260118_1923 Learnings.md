# Session Learnings - 2026-01-18

## 1. Global/Local Skill Pattern for Portability
**Topics:** workflow

**Context:** Needed to make Claude Code skills reusable across multiple projects while allowing project-specific customization.

**Learning:** Split skills into global (portable) and local (project-specific) versions. Global skills in `~/.claude/commands/` handle the workflow, read `config.json` for settings, and call local extensions. Local skills in `.claude/commands/` with `-local` suffix add project-specific logic.

**Pattern:**
- Global: `~/.claude/commands/q-skill.md` - portable workflow
- Local: `.claude/commands/q-skill-local.md` - project extensions
- Config: `.claude/config.json` - project settings (terminology, paths, features)

---

## 2. Naming Convention for Skill Scope
**Topics:** workflow

**Context:** Needed clear distinction between skills that extend globals vs skills that are purely local.

**Learning:** Use prefix conventions to indicate skill scope:
- `q-*` in global location: Portable skills
- `q-*-local`: Local extensions of global skills
- `L-*`: Local-only skills with no global equivalent (e.g., L-add-client-note, L-verify-page)

---

## 3. Config.json as Central Configuration
**Topics:** workflow

**Context:** Global skills need project-specific settings (terminology, paths, features) without hardcoding.

**Learning:** Use `.claude/config.json` with standardized sections:
- `terminology`: stage/subStage names (Block/Section vs Phase/Step)
- `features`: enabled capabilities (APIs, tests, Tailwind, database)
- `paths`: file locations (sessions, reference docs, tech docs)
- `thresholds`: numeric limits (CLAUDE.md line counts)
- Framework settings: astro, cloudflare, github, editor

Global skills read config with fallback defaults if missing.

---

## 4. Skill Naming Should Match Counterparts
**Topics:** workflow

**Context:** Had `/q-resume` as counterpart to `/q-end-session` but naming was inconsistent.

**Learning:** When skills are logical pairs (start/end, begin/resume), names should follow same pattern. Renamed `q-resume` to `q-resume-session` to match `q-end-session`.

---

## 5. Speed vs Completeness in Force-End
**Topics:** workflow

**Context:** Assumed `/q-force-end` should skip local extensions to save tokens.

**Learning:** Speed savings come from removing gates and questions, NOT from skipping local skills. If a global skill runs, its local extension must also run. The user decides what to skip, not the skill.
