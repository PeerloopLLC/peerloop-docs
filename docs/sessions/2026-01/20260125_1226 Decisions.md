## testing: Fix Endpoint Bug vs Adjust Test

**Decision:** When endpoint uses wrong column name (`scheduled_at` instead of `scheduled_start`), fix the endpoint rather than work around it in tests.

**Rationale:**
- Schema is source of truth during development
- Endpoint had a bug (referencing non-existent column)
- Using alias (`scheduled_start as scheduled_at`) preserves API response format while fixing the query

**Applied:** `src/pages/api/admin/student-teachers/[id].ts`

## testing: Resend Email Mock Approach

**Decision:** Mock Resend with class-based mock for moderator invite tests.

**Implementation:**
```typescript
vi.mock('resend', () => ({
  Resend: class MockResend {
    constructor() {}
    emails = {
      send: vi.fn().mockResolvedValue({ error: null }),
    };
  },
}));
```

**Rationale:**
- Matches the constructor pattern used by Resend SDK
- Allows testing both success and failure email scenarios
- Keeps tests fast (no actual email sending)

## testing: Test File Organization for Admin Endpoints

**Decision:** Continue using three-file structure for admin endpoint tests:
- `list.test.ts` - GET collection endpoints (pagination, filtering, sorting)
- `detail.test.ts` - GET/DELETE single resource endpoints
- `actions.test.ts` - POST/PATCH action endpoints (activate, deactivate, approve, etc.)

**Rationale:**
- Consistent with previous admin endpoint tests
- Logical grouping by operation type
- Files stay manageable size (25-35 tests each)
- Easy to find tests for specific functionality
