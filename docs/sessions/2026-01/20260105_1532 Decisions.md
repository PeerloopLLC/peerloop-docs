## Decision 2: Stripe Hosted Checkout (vs Embedded/Custom)

**Type**: Implementation
**Driver**: Claude-driven (following tech-003-stripe.md)
**Working On**: Block 2.1 - Stripe Integration
**Trigger**: Implementing payment flow for course enrollment

**Options Considered**:
1. **Stripe Checkout (hosted)** - Redirect to Stripe's payment page <- **Chosen**
2. **Stripe Elements (embedded)** - Build custom payment form with Stripe Elements
3. **Custom form + PaymentIntents** - Fully custom UI with direct API calls

**Decision**: Use Stripe Checkout Sessions with redirect flow.

**Rationale**:
- Stripe handles PCI compliance entirely
- Automatic mobile optimization
- Apple Pay / Google Pay built-in
- Less code to maintain
- Matches the marketplace pattern in tech-003-stripe.md

**Consequences**:
- `createCheckoutSession()` in lib/stripe.ts
- Success/cancel URLs point back to course pages
- All sensitive payment UI handled by Stripe

## Decision 3: `/api/me/*` Pattern for User-Specific Endpoints

**Type**: Implementation
**Driver**: Claude-driven
**Working On**: Block 2.4 - Student Dashboard
**Trigger**: Need endpoint for current user's enrollments

**Options Considered**:
1. **`/api/me/enrollments`** - Implicit user from session <- **Chosen**
2. **`/api/enrollments?student_id=`** - Explicit user ID in query params
3. **`/api/users/[id]/enrollments`** - User ID in path

**Decision**: Use `/api/me/*` pattern for current-user endpoints.

**Rationale**:
- Cleaner API surface - no need to pass user ID
- Auth is implicit from session cookie
- Prevents accidental data leakage (can't query other users' data)
- Common pattern (GitHub, Stripe, etc. use `/me` or `/user`)

**Consequences**:
- Created `/api/me/enrollments.ts`
- Pattern can be extended: `/api/me/sessions`, `/api/me/certificates`, etc.
