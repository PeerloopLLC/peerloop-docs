## Vite's Built-in Environment Variable Loading

### Discovery
When using `require('dotenv')` in Vite's ESM environment, the call fails silently because `require is not defined`. This was causing the D1 REST API fallback to fail on MBA-2017 since `ENABLE_D1_REST_FALLBACK` remained undefined.

### Solution
Vite provides `loadEnv()` which:
1. Loads `.env` files using dotenv internally (bundled, not a peer dependency)
2. Can load ALL env vars (not just `VITE_` prefixed) by passing `''` as the third parameter
3. Works reliably in both ESM and CommonJS contexts

### Implementation Pattern
```javascript
// In astro.config.mjs
import { loadEnv } from 'vite';

const mode = process.env.NODE_ENV || 'development';
const env = loadEnv(mode, process.cwd(), '');
Object.assign(process.env, env);
```

### Key Insight
- Vite bundles dotenv internally for `loadEnv()`
- No need for dotenv as a direct dependency
- This populates `process.env` for SSR code at config load time
- The `.env -> .dev.vars` symlink works seamlessly with this approach

## Diagnostic Logging Strategy

When debugging silent failures in module loading:
1. Add explicit `console.log` statements at each decision point
2. Log both the condition check and the actual values being checked
3. The diagnostic logs revealed `require is not defined` immediately

## Environment-Specific Module Loading

ESM environments (Vite, modern Node with `type: module`) don't have `require()` by default. Options:
1. Use `createRequire` from `node:module` (works in Node.js ESM)
2. Use the framework's built-in env loading (preferred - Vite's `loadEnv`)
3. Dynamic `import()` for async loading
