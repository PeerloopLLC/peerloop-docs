# Session Learnings - 2026-01-19

## 1. Stream.io v3 vs v2 SDK Incompatibility
**Topics:** stream, astro

**Context:** Installed Stream.io v3 SDK packages (`@stream-io/node-sdk`, `@stream-io/feeds-react-sdk`) but the Stream Dashboard only shows v2 feed groups configuration.

**Learning:** Stream's v3 SDK and v2 dashboard are incompatible. The dashboard banner explicitly states "Feeds v3 data and configuration is not yet integrated into the dashboard." When using the v2 dashboard for feed management, use the `getstream` npm package (v2 SDK) instead of the v3 packages.

**Pattern:**
```bash
# v3 SDK (don't use with v2 dashboard)
npm install @stream-io/node-sdk @stream-io/feeds-react-sdk

# v2 SDK (compatible with current dashboard)
npm install getstream
```

## 2. getstream Package Requires Node.js Polyfills in Browser
**Topics:** stream, astro, cloudflare

**Context:** After installing `getstream` v2 SDK, browser threw errors: "global is not defined" and "process is not defined"

**Learning:** The `getstream` package uses Node.js globals (`global`, `process`) that don't exist in the browser. These need to be polyfilled via Vite config.

**Pattern:**
```javascript
// astro.config.mjs
export default defineConfig({
  vite: {
    define: {
      // Polyfills for Node.js packages that use Node globals
      global: 'globalThis',
      'process.env': '{}',
    },
  },
});
```

## 3. Stream Client-Side Tokens Have Limited Permissions
**Topics:** stream, auth

**Context:** After connecting with a client-side user token, attempts to read/write the `townhall:main` feed failed with 403 "no policies allowing this request"

**Learning:** Stream's client-side user tokens have restrictive default permissions. Users can typically only access feeds they "own" (e.g., `user:123` feed for user 123). Shared feeds like `townhall:main` require either:
1. Custom feed policies configured via Stream API (not exposed in v2 dashboard UI)
2. Server-side API proxy using the API secret (which has full permissions)

The server-side proxy approach is more secure and reliable.

## 4. Server-Side Feed Operations Pattern for Stream
**Topics:** stream, astro, auth

**Context:** Needed to allow all authenticated users to read/write a shared community feed without complex permission configuration.

**Learning:** Proxy all feed operations through your own API endpoints. The server uses the API secret (full permissions) while your endpoints enforce your own auth and business logic.

**Pattern:**
```typescript
// src/pages/api/feeds/townhall.ts
import { getStreamClient } from '@lib/stream';

export const GET: APIRoute = async ({ cookies, locals }) => {
  // 1. Verify user auth (your JWT)
  const session = await getSession(cookies, jwtSecret);
  if (!session) return Response.json({ error: 'Auth required' }, { status: 401 });

  // 2. Server-side Stream client has full permissions
  const client = getStreamClient(locals);  // uses API_SECRET
  const feed = client.feed('townhall', 'main');
  const response = await feed.get({ limit: 25 });

  return Response.json({ activities: response.results });
};
```

## 5. Graceful Error Handling with Actionable Messages
**Topics:** astro, workflow

**Context:** Stream permission errors showed raw JSON in the UI, which was confusing for users and didn't help admins fix the issue.

**Learning:** Parse API errors to detect specific error codes/types, then show user-friendly messages with actionable guidance for admins. Keep technical details in console for debugging.

**Pattern:**
```typescript
interface ErrorInfo {
  userMessage: string;
  isConfigError: boolean;
  configAction?: string;
}

function parseError(err: unknown, response?: Response): ErrorInfo {
  // 403 = permission error
  if (response?.status === 403) {
    return {
      userMessage: 'Feed permissions need to be configured',
      isConfigError: true,
      configAction: 'Configure feed policies in Stream Dashboard',
    };
  }
  // ... handle other specific errors
}
```

## 6. Stream v2 SDK Client Initialization
**Topics:** stream

**Context:** Setting up the Stream v2 SDK for both server and client-side use.

**Learning:** The v2 `getstream` package uses the `connect()` function. Server-side uses API key + secret, client-side uses API key + user token + app ID.

**Pattern:**
```typescript
// Server-side (full permissions)
import { connect } from 'getstream';
const client = connect(apiKey, apiSecret);
const token = client.createUserToken(userId);

// Client-side (via user token)
const client = connect(apiKey, userToken, appId);
```
