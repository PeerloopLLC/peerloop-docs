# Session Learnings - 2026-01-22

## 1. MSGS Enables Profile Messaging But Doesn't Unblock Other Pages
**Topics:** architecture, messaging

**Context:** After implementing MSGS (Direct Messages), analyzed which other pages were now unblocked.

**Learning:** MSGS provides 1-on-1 async messaging infrastructure, but CHAT (Course Chat) and HELP (Summon Help) use completely different real-time WebSocket/Durable Objects patterns. They're not blocked on MSGS - they're separate communication paradigms (async DM vs. real-time group/help chat).

**Pattern:** When analyzing feature dependencies, distinguish between:
- Same data/table dependencies (true blockers)
- Similar UI patterns (not blockers, just similar code)
- Different communication paradigms (completely independent)

## 2. URL Query Parameters for Deep Linking to Conversations
**Topics:** messaging, ux

**Context:** MSGS needed to support "Message" buttons from STPR, CPRO, SDSH, CMST pages, plus notification clicks.

**Learning:** Two query parameters handle all entry points:
- `?to=<userId>` - Opens or creates conversation with a user (from profile pages)
- `?conversation=<id>` - Opens specific conversation directly (from notifications)

**Pattern:**
```tsx
// Parse both parameters
const toUserId = params.get('to');
const conversationId = params.get('conversation');

// Handle ?to= - find existing or open new modal
if (toUserId) {
  const existing = conversations.find(c => c.participants.some(p => p.id === toUserId));
  if (existing) setSelectedId(existing.id);
  else setIsNewModalOpen(true);
}

// Handle ?conversation= - open directly
if (conversationId) {
  const exists = conversations.some(c => c.id === conversationId);
  if (exists) setSelectedId(conversationId);
}
```

## 3. Building Leaderboards Without Full Gamification System
**Topics:** architecture, d1

**Context:** LEAD spec called for goodwill points leaderboard, but CD-023 (goodwill system) isn't implemented yet.

**Learning:** Leaderboards can be built incrementally using existing data:
- `user_stats.students_taught` → "Top Teachers"
- `user_stats.courses_completed` → "Top Learners"
- `user_stats.average_rating` → "Highest Rated"

The UI (tabs, ranking display, user cards, percentiles) is 80-90% reusable. When goodwill_points is added later, it's just a new category and different SQL queries.

**Pattern:** Build features with existing data when possible, designing UI to be data-source agnostic. Add categories/data sources later without rewriting components.

## 4. Calculating User Rank and Percentile Efficiently
**Topics:** d1, performance

**Context:** Leaderboard needed to show user's own rank ("You are #47, Top 15%").

**Learning:** Calculate rank by counting users with higher scores, rather than fetching and sorting all users:

```sql
-- Get user's score
SELECT students_taught as score FROM user_stats WHERE user_id = ?

-- Count users with higher scores (this IS the rank - 1)
SELECT COUNT(*) as count FROM user_stats WHERE students_taught > ?

-- Calculate: rank = higher_count + 1, percentile = ((total - rank + 1) / total) * 100
```

This scales regardless of user count - no need to fetch/sort the entire leaderboard.

## 5. Page Spec Connections Track Implementation Status
**Topics:** workflow, documentation

**Context:** Updated page specs to track which page connections (MSGS ← STPR, CPRO, etc.) are implemented.

**Learning:** Adding ✅ markers to connection tables in page specs provides quick visibility into:
- Which integrations are complete
- Which pages still need work
- Feature completeness at a glance

**Pattern:**
```markdown
| Target | Trigger | Notes |
|--------|---------|-------|
| MSGS | "Message" button | DM the ST ✅ |
| SBOK | "Book Session" | Pre-select this ST |
```
