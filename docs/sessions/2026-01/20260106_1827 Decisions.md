## Decision 1: PAGES-MAP Refactoring to Individual Page Files

**Type**: Strategic
**Driver**: User-driven
**Working On**: Documentation structure for feature-to-story traceability
**Trigger**: Concern that PAGES-MAP.md would grow too large with user story mappings

1. Keep everything in PAGES-MAP.md (simpler, but grows unwieldy)
2. Split into individual `docs/pages/CODE.md` files <- **Chosen**

**Decision**: Refactor PAGES-MAP.md into a slim lookup table (~140 lines) with individual page detail files in `docs/pages/CODE.md`.

**Consequences**:

- PAGES-MAP.md reduced from 820 → 140 lines
- Added "Details" column in lookup table linking to individual files

## Decision 3: Document Features As-Built vs Snapshot

**Type**: Strategic
**Driver**: User-driven (asked question, Claude recommended)
**Working On**: Feature traceability approach
**Trigger**: User asked how to track user story coverage

**Options Considered**:
1. Snapshot on demand (read code, extract features later)
2. Document as built (add features during implementation) <- **Chosen**

**Decision**: Document features in page files as they are implemented, not via later code analysis.

**Rationale**:
- During implementation, developer knows intent, edge cases, and story context
- Reading code later requires inferring features from structure
- Risk of missing subtle behaviors or over-counting implementation details
- Consistent granularity when documented at implementation time

**Consequences**:
- Features should be added to `docs/pages/CODE.md` during implementation
- Each feature tagged with `[US-XXXX]` story code
- Gap Analysis section captures features without formal stories
