# Session Decisions

**Date:** 2026-01-08
**Session Focus:** Creator Dashboard Implementation

---

## Decisions Made

### 1. Component-First Dashboard Architecture

**Decision:** Build role-specific dashboards as standalone React components that can later be composed into a unified multi-role dashboard.

**Context:** User raised concern about Block 9.6 (Role-Based Page Access Model) and whether dashboards should handle multi-role users now.

**Options Considered:**
1. Wait for Block 9.6 to design unified dashboard
2. Build separate pages that duplicate some functionality
3. Build components now, unify later (chosen)

**Rationale:**
- Doesn't block current work on 9.6 design decisions
- Individual dashboard components become "sections" in unified dashboard later
- Multi-role users eventually see everything on one page
- Single-role users get same experience either way
- Components are reusable

**Impact:** CDSH built as `CreatorDashboard` component, same pattern should be used for TDSH.

---

### 2. API Path Pattern for Dashboard

**Decision:** Use `/api/me/creator-dashboard` instead of `/api/creators/me/dashboard`.

**Rationale:** Follows existing pattern of `/api/me/enrollments` for user-specific data. All "me" endpoints are under `/api/me/`.

**Impact:** Consistent API structure, easier to discover related endpoints.

---

### 3. Defer Pending Approvals Detail

**Decision:** Implement pending approvals as placeholder section, not full functionality.

**Rationale:**
- ST applications depend on Block 7 (S-T Management)
- Certification requests depend on Block 6 (Certifications)
- Homework reviews could be added but aren't priority
- Badge count and empty state provide foundation

**Impact:** Section ready for future blocks, no wasted work on unavailable features.

---

### 4. Request Payout UI Only

**Decision:** Implement Request Payout button UI without wiring to backend API.

**Rationale:**
- POST `/api/payouts/request` endpoint doesn't exist yet
- Admin payouts (APAY) handle actual payout processing
- Creator-initiated payout request is separate workflow
- Better to show intent than hide feature

**Impact:** Users see button, know feature exists. Can be wired when payout request workflow is designed.
