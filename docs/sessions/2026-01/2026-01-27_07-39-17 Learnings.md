# Session Learnings - 2026-01-27

## 1. API Test Helper Import Aliases Are Distinct
**Topics:** testing

**Context:** Running LGIN and SGUP test scripts failed — login.test.ts had `bcrypt.hash(undefined, number)` and register.test.ts had `createAPIContext is not a function`.

**Learning:** The project has two separate test helper aliases: `@test-helpers` (resolves to `tests/helpers/index.ts` for DB utilities like `describeWithTestDB`, `getTestDB`) and `@api-helpers` (resolves to `tests/api/helpers/api-test-helper.ts` for API utilities like `createAPIContext`, `createMockRequest`, `VALID_PASSWORD`). Importing API-specific helpers from `@test-helpers` silently returns `undefined` rather than throwing, which surfaces as confusing runtime errors downstream.

**Pattern:** Always use `@api-helpers` for: `createAPIContext`, `createMockRequest`, `VALID_PASSWORD`, `INVALID_PASSWORDS`, `uniqueEmail`, `uniqueHandle`, `validRegistrationData`. Use `@test-helpers` for: `describeWithTestDB`, `getTestDB`, `resetTestDB`, `closeTestDB`, `DB_TEST`.

## 2. PaginatedResult Uses camelCase (`totalPages` not `total_pages`)
**Topics:** testing, d1

**Context:** Two notifications API tests failed expecting `body.total_pages` but received `undefined`.

**Learning:** The `createPaginatedResult()` helper in `src/lib/db/index.ts` returns camelCase keys (`totalPages`, `hasMore`), not snake_case. Tests must match the actual response shape. The `PaginatedResult<T>` interface defines: `items`, `total`, `page`, `limit`, `totalPages`, `hasMore`.

## 3. Enrollments Table Uses `enrolled_at` Not `created_at`
**Topics:** d1, testing

**Context:** Adding test seed data for the courses `[id]` detail test failed with `SqliteError: table enrollments has no column named created_at`.

**Learning:** The `enrollments` table schema uses `enrolled_at` (with default `datetime('now')`) instead of the `created_at`/`updated_at` pattern used by most other tables. When seeding enrollment test data, use `enrolled_at` for the timestamp column.

---

## 4. React Duplicate Key Warnings in Tests Come from Mock Data Reuse
**Topics:** testing

**Context:** NotificationsList pagination test produced React "duplicate key" stderr warnings for `notif-1` through `notif-4`.

**Learning:** When testing "load more" pagination, the mock for page 2 must return items with different IDs than page 1. Using `mockResolvedValue` (not `mockResolvedValueOnce`) causes both pages to return identical data, producing duplicate React keys when items are appended. Fix: use `mockResolvedValueOnce` for page 1, then `mockResolvedValueOnce` for page 2 with distinct IDs.

## 5. Categories Seed Data Creates 15 Records, Not 3
**Topics:** testing, d1

**Context:** `categories.test.ts` expected `toBe(3)` categories (matching its own `beforeAll` inserts) but got 18 because the migration seed (`0002_seed.sql`) inserts 15 categories that persist across test runs.

**Learning:** Tests using `describeWithTestDB` get a fresh DB with both migration files applied. The seed data from `0002_seed.sql` is always present. Tests that count records should use `toBeGreaterThanOrEqual()` or account for seed data, not hardcode counts based only on their own inserts.

---

## 6. Gap Analysis Must Read Test Content, Not Just Count Tests
**Topics:** workflow, testing

**Context:** The `/L-test-plan` gap analysis was flagging endpoints as ⚠️ (thin coverage) purely based on test count thresholds (e.g., 5 tests = ⚠️) without reading what those tests actually covered.

**Learning:** A meaningful gap analysis requires reading the actual `it()` descriptions to verify that auth (401/403), success cases, validation, parameter handling, and error cases (503) are covered. A 5-test file covering all those areas for a simple read-only endpoint has no gaps, while a 10-test file missing auth tests does. Count-based thresholds are misleading proxies.
