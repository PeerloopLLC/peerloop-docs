# Session Decisions - 2026-01-22

## 1. Use Move Up/Down Buttons Instead of Drag-Drop for Curriculum
**Type:** Implementation
**Topics:** astro, workflow

**Trigger:** The plan mentioned curriculum reordering for modules. Drag-drop would require adding a new dependency and more complex implementation.

**Options Considered:**
1. Drag-and-drop with library (react-beautiful-dnd, dnd-kit)
2. Move up/down buttons following CategoriesAdmin pattern ← Chosen

**Decision:** Use move up/down buttons for module reordering, following the existing CategoriesAdmin pattern.

**Rationale:** MVP scope should avoid new dependencies where possible. The CategoriesAdmin component already demonstrates this pattern works well. For a typical course with 5-10 modules, up/down buttons are sufficient and maintain consistency with the admin UI.

**Consequences:**
- No new npm dependencies
- Consistent UX with admin category management
- May add drag-drop later if user feedback requests it

## 2. Multi-Tab Course Editor vs Single Long Form
**Type:** Implementation
**Topics:** astro

**Trigger:** Course editing has many fields across different concerns (basic info, curriculum, publishing). Need to organize the UI.

**Options Considered:**
1. Single long scrollable form
2. Multi-tab interface (Basic Info, Curriculum, Publishing) ← Chosen
3. Wizard/stepper flow

**Decision:** Use a three-tab interface: Basic Info, Curriculum, Publishing.

**Rationale:** Tabs group related fields logically and reduce cognitive load. Creators can focus on one aspect at a time. The publishing tab can show a checklist and validation status clearly. This matches patterns seen in other course platforms (Teachable, Thinkific).

**Consequences:**
- Clean separation of concerns in UI
- Each tab can have its own save action if needed
- Curriculum becomes its own dedicated component

## 3. Textarea for Description Instead of Rich Text Editor
**Type:** Implementation
**Topics:** astro

**Trigger:** Course description field needs text input. Rich text editing adds complexity.

**Options Considered:**
1. Rich text editor (TipTap, Slate, etc.)
2. Plain textarea with markdown support ← Chosen
3. Markdown editor with preview

**Decision:** Use plain textarea for MVP. Can display as markdown if needed.

**Rationale:** Rich text editors add significant complexity (library deps, sanitization, storage format). For MVP, a simple textarea is sufficient. The course detail page can render markdown if descriptions use it. Can upgrade to rich text editor in Phase 2 based on creator feedback.

**Consequences:**
- Simpler implementation
- No new editor dependencies
- May need to upgrade later based on feedback
