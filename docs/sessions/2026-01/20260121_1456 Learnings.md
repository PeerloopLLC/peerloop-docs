# Session Learnings - 2026-01-21

## 1. Vite `define` Replaces process.env With Empty Object
**Topics:** Vite, Astro, Environment Variables, MBA-2017

**Context:** MBA-2017 D1 REST fallback was returning "Database not available" despite correct `.dev.vars` configuration.

**Learning:** The `'process.env': '{}'` define in `astro.config.mjs` replaces ALL `process.env` references with an empty object `{}` at bundle time - including server-side code. This was added for getstream compatibility but breaks any code relying on `process.env` at runtime.

**Pattern:**
```javascript
// In astro.config.mjs
define: {
  'process.env': '{}',  // This breaks runtime process.env access
}
```

Debug revealed: `process.env keys: []` (empty array) at runtime.

## 2. Vite Converts String "true" to Boolean true
**Topics:** Vite, Environment Variables

**Context:** Even after switching to `import.meta.env`, the fallback check `env.ENABLE_D1_REST_FALLBACK === 'true'` was returning false.

**Learning:** Vite's `loadEnv` and env injection can convert the string `"true"` to boolean `true`. When checking env vars, check for both: `value === true || value === 'true'`.

**Pattern:**
```typescript
// Wrong - may fail if Vite converted to boolean
const enabled = env.SOME_FLAG === 'true';

// Right - handles both string and boolean
const enabled = env.SOME_FLAG === true || env.SOME_FLAG === 'true';
```

## 3. MBA-2017 Development Adds Unsustainable Complexity
**Topics:** Development Environment, MBA-2017, Architecture

**Context:** Spent significant session time attempting to fix MBA-2017 D1 REST fallback, which required working around the `'process.env': '{}'` define.

**Learning:** The fix required explicitly injecting each env var via Vite define, which:
1. Is hacky and error-prone
2. Requires updating astro.config.mjs for every new env var needed
3. May have unintended consequences for other environments

MBA-2017 development should be avoided when MacMini is available. The complexity cost exceeds the mobility benefit.

## 4. Resend Email Integration is Straightforward
**Topics:** Resend, Email, React Email

**Context:** Implemented first transactional email feature (moderator invites).

**Learning:** Resend + React Email integration is clean:
1. Install: `npm install resend @react-email/components`
2. Create React component in `src/emails/`
3. Import and pass to `resend.emails.send({ react: <Component /> })`
4. Use separate API keys for dev vs production
5. Production keys go via `wrangler secret put`

The email sends even without domain verification (during testing), though deliverability is better with verified domain.
