# Session Decisions - 2026-01-21

## 1. Polling vs Real-Time for Notification Badge
**Type:** Implementation
**Topics:** astro, workflow

**Trigger:** AppHeader needed to show dynamic notification count, but real-time updates would require WebSocket/SSE infrastructure.

**Options Considered:**
1. Polling on page load only ← Chosen
2. Polling every 60 seconds
3. WebSocket/SSE real-time updates
4. Server-Sent Events with Cloudflare Durable Objects

**Decision:** Fetch notification count once on component mount via `/api/me/notifications/count`. No periodic polling or real-time push.

**Rationale:** For MVP, users will see updated counts on page navigation which is sufficient. Real-time adds complexity (Durable Objects, connection management) with minimal UX benefit for a notification badge. Can upgrade later if needed.

**Consequences:** Badge updates only on page load/navigation, not in real-time while viewing a page.

## 2. Session Reminder Emails Deferred to Cloudflare Cron
**Type:** Implementation
**Topics:** cloudflare, workflow

**Trigger:** The plan called for session reminder emails (24h and 1h before sessions), but these require scheduled execution.

**Options Considered:**
1. Implement with Cloudflare Cron Workers now
2. Defer session reminders to post-MVP ← Chosen
3. Client-side reminder with service worker

**Decision:** Create the email template (`SessionReminderEmail.tsx`) but don't wire it up. Defer the cron worker implementation.

**Rationale:** Cloudflare scheduled workers require separate configuration and deployment. The template is ready for when cron is implemented. Other notification types (booking confirmation, certificate, payout) work without scheduling.

**Consequences:** Session reminders won't work until a future cron implementation. Template is ready and waiting.

## 3. Email Preferences in Users Table vs Separate Table
**Type:** Implementation
**Topics:** d1

**Trigger:** Needed to store user email opt-in/opt-out preferences for 7 notification types.

**Options Considered:**
1. Add columns to users table ← Chosen
2. Create separate user_preferences table
3. Create notification_preferences table with rows per type

**Decision:** Add 7 boolean columns directly to the users table via migration 0017.

**Rationale:** Simpler queries (single SELECT with column name), no JOINs needed, clear defaults in schema. For 7 preferences, the denormalized approach is cleaner than maintaining a separate table. Can always migrate to a separate table if preference count grows significantly.

**Consequences:** User table is slightly wider. Preference queries are single-column reads.
