# Session Learnings - 2026-01-23

## 1. Analytics Page Dual-View Pattern
**Topics:** astro, workflow

**Context:** Implemented AANA and TANA analytics pages following the CANA pattern. Initially replaced PageSpecView with the real component, but the correct pattern requires both views.

**Learning:** PeerLoop pages use a dual-view pattern where both PageSpecView (development) and the real component (production) coexist:
- PageSpecView goes in the default slot (shown in `.spec-content`)
- Real component goes in the named `content` slot (shown in `.real-content`)
- The SpecPanelToggle component controls which view is visible
- The `pageCode` prop on AppLayout enables the toggle

**Pattern:**
```astro
<AppLayout pageCode={pageData.metadata.code}>
  <!-- Spec View (Development Mode) -->
  <div class="container-main py-8">
    <PageSpecView pageData={pageData} />
  </div>

  <!-- Sidepanel Navigation -->
  <Fragment slot="sidepanel">...</Fragment>

  <!-- Real Content (Production Mode) -->
  <Fragment slot="content">
    <RealComponent client:load />
  </Fragment>
</AppLayout>
```

## 2. TimeSeriesData Index Signature Requirement
**Topics:** typescript, recharts

**Context:** STAnalytics component had a type error when passing earnings data to AreaChart.

**Learning:** The `TimeSeriesData` type from `@components/charts/types.ts` requires an index signature `[key: string]: string | number` because Recharts accesses data dynamically by key. When defining custom interfaces for chart data, include this index signature for compatibility:

**Pattern:**
```typescript
interface EarningsData {
  date: string;
  earnings: number;
  [key: string]: string | number; // Required for TimeSeriesData compatibility
}
```

## 3. PAGES-MAP.md as Source of Truth for Page Locations
**Topics:** workflow, documentation

**Context:** When looking for the CANA and TANA pages, initially tried guessing paths like `/creator/analytics` which didn't exist.

**Learning:** Always consult `PAGES-MAP.md` to find page locations. The quick lookup table maps page codes to their actual routes and file paths. This avoids wasted time searching incorrect directories.

**Pattern:**
```bash
# Find page by code
grep "CANA" PAGES-MAP.md
# Result: | CANA | Creator Analytics | `/dashboard/creator/analytics` | ...
```

## 4. Analytics API Pattern: Period Helpers
**Topics:** d1, api

**Context:** Both AANA and TANA needed consistent period handling across multiple endpoints.

**Learning:** Create reusable period helper functions for analytics APIs:
- `getPeriodDateSQL(period)` - Returns SQL date expression for period start
- `getPreviousPeriodDateSQL(period)` - Returns start/end for comparison period
- `getGroupingSQL(period)` - Returns SQL for date grouping (daily/weekly/monthly)
- `fillMissingDates(data, period)` - Fills gaps in time series for continuous charts

These helpers ensure consistent behavior across all analytics endpoints and reduce duplication.

## 5. S-T Analytics Simpler Than Creator Analytics
**Topics:** architecture, analytics

**Context:** Implementing TANA after AANA and CANA.

**Learning:** Student-Teacher analytics (TANA) is intentionally simpler than Creator analytics (CANA):
- S-Ts have a single dimension (their teaching) vs Creators managing multiple courses/S-Ts
- No S-T leaderboard needed (that's in CANA for creators to view their S-Ts)
- No funnel analysis (that's platform-level in AANA)
- Focus on personal metrics: earnings, sessions, assigned students, patterns

This keeps TANA focused and avoids feature bloat.
