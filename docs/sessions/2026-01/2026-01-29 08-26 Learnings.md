# Session Learnings - 2026-01-29

## 1. Slide Panel CSS Positioning Math
**Topics:** astro, tailwind

**Context:** FeedSlidePanel was partially visible when closed despite having `-translate-x-full`. The panel had `left-64` (256px) with `marginLeft: -64px`, creating an effective position of 192px. Translating by -100% of its 320px width still left 128px visible.

**Learning:** When positioning slide-out panels, ensure the math works: `effective_position - translate_amount = final_position`. If final_position > -width, the panel is still visible. Use conditional `left` classes instead of combining static positioning with negative margins.

**Pattern:**
```tsx
// Bad: Math doesn't hide the panel
className={`fixed left-64 ... ${isOpen ? 'translate-x-0' : '-translate-x-full'}`}
style={{ marginLeft: '-64px' }}  // 256 - 64 = 192; 192 - 320 = -128 (still visible!)

// Good: Conditional positioning
className={`fixed ... ${isOpen ? 'left-64 translate-x-0' : 'left-0 -translate-x-full'}`}
```

## 2. PageSpec JSON Validation Requirements
**Topics:** astro, workflow

**Context:** New dash page specs were failing validation with missing required fields (plannedApiCalls, interactiveElements.summary.analytics, testCoverage.apiTests).

**Learning:** The PageSpec schema has many required fields even for simple/placeholder pages. When creating new page specs, copy from a fully-formed example (like login.json) rather than minimal specs. Required fields include: dataRequirements[], userStories[], statesVariations[], mobileConsiderations[], errorHandling[], analyticsEvents[], plannedApiCalls[], notes[].

## 3. Permission Flags vs Role Flags
**Topics:** auth, d1

**Context:** The existing `is_student`, `is_student_teacher`, `is_creator` flags are global but some roles are course-specific. A user can be a student for Course A and a Student-Teacher for Course B.

**Learning:** Separate "capabilities" (what you CAN do) from "roles" (what you ARE for a specific course):
- `can_create_courses` - permission to author courses (admin grants)
- `can_teach_courses` - permission to be certified as ST (admin grants)
- `isStudentTeacherFor(courseId)` - actual role checked against student_teachers table

The old flags are deprecated but kept for backwards compatibility.

## 4. CurrentUser Singleton Pattern for Astro Islands
**Topics:** astro, auth

**Context:** Need shared user state across multiple React islands that bundle separately.

**Learning:** Use a class singleton stored on `window.__peerloop.currentUser` with a typed wrapper module. Each island imports `getCurrentUser()` and gets the same instance. This works because:
1. Window is shared across all islands
2. Typed wrapper provides IDE support
3. Class encapsulates role-checking logic

**Pattern:**
```typescript
// src/lib/current-user.ts
declare global {
  interface Window {
    __peerloop?: { currentUser: CurrentUser | null };
  }
}

export function getCurrentUser(): CurrentUser | null {
  return window.__peerloop?.currentUser ?? null;
}
```

## 5. Stale-While-Revalidate for User Data
**Topics:** auth, workflow

**Context:** Need fast page loads while keeping user data fresh.

**Learning:** The stale-while-revalidate pattern works well for user state:
1. On page load, immediately hydrate from localStorage (instant UI)
2. Fetch fresh from `/api/me/full` in background
3. If data differs, update UI and cache
4. No hard expiration - always use cached data while refreshing

User rarely notices the background refresh, and data is usually unchanged anyway.
