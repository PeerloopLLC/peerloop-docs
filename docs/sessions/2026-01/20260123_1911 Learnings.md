# Session Learnings - 2026-01-23

## 1. D1 REST Fallback Requires D1_DATABASE_ID
**Topics:** d1, cloudflare, workflow

**Context:** D1 REST fallback test was failing on MBA-2017 with `getD1RestConfig()` returning null.

**Learning:** The D1 REST API fallback requires three environment variables, not two:
- `CLOUDFLARE_ACCOUNT_ID`
- `CLOUDFLARE_API_TOKEN`
- `D1_DATABASE_ID` (was missing)

On MBA-2017, `D1_DATABASE_ID` must point to the staging database since there's no local D1. MacMini doesn't need this because it uses native D1 binding from `wrangler.toml`.

**Pattern:**
```bash
# MBA-2017 .dev.vars
ENABLE_D1_REST_FALLBACK=true
D1_DATABASE_ID=605f1ab8-62cc-4934-a2fd-d828e188f50e  # staging only!

# MacMini .dev.vars - D1_DATABASE_ID not needed (native binding)
```

## 2. MBA-2017 and MacMini Have Different Manual Testing Data Sources
**Topics:** d1, workflow, testing

**Context:** Discussing consistency of manual testing across development machines.

**Learning:** There's an inherent discrepancy in manual testing environments:

| Machine | Manual Testing DB | Shared With |
|---------|-------------------|-------------|
| MBA-2017 | Remote staging D1 | Preview deployments |
| MacMini | Local D1 (wrangler) | Nothing (isolated) |

This is a workaround because MBA-2017 can't run local D1 (macOS 12 < 13.5). The consequence is different data states during manual testing. Automated tests are consistent because both machines use better-sqlite3 in-memory.

## 3. Decompose Large Components Before Testing
**Topics:** testing, workflow

**Context:** Attempting to write tests for CourseDetail.tsx (~490 lines, 15+ sections).

**Learning:** Large monolithic components make testing difficult. Better to decompose first:
- Enables testing each subcomponent in isolation
- Improves maintainability and reuse
- Reduces cognitive load

**Pattern:** Process for each page test:
1. Review component size/complexity
2. If >200 lines or 5+ sections → decompose into subcomponents first
3. Test subcomponents individually, then integration

## 4. List/Browse Family Test Pattern
**Topics:** testing

**Context:** Creating tests for CRLS (CreatorBrowse) and STDR (STDirectory).

**Learning:** List/Browse pages share a consistent test structure:
1. RENDERING - Page elements appear
2. SEARCH - Search filters results
3. FILTERING - Filter controls work
4. SORTING - Sort options reorder results
5. NAVIGATION - Links work correctly
6. EMPTY STATE - Message when no results

These pages typically have: search input, filter controls (pills, dropdowns, toggles), sort dropdown, grid of cards, empty state, CTA section.
