# Session Decisions - 2026-01-31

## 1. Schema TSV Format: Row per Column with Data as Columns
**Type:** Implementation
**Topics:** d1, workflow

**Trigger:** User asked about including schema information in TSV exports.

**Options Considered:**
1. Row 2 as schema metadata (type info below headers)
2. Separate schema file with traditional column layout ← Chosen
3. Column headers include type info (e.g., "email:TEXT:NOT NULL")

**Decision:** Use a separate schema file with schema metadata as columns (column_name, data_type, nullable, etc.) and each data record as an additional column.

**Rationale:** This format allows reading across a row to see a field's definition and all values, or reading down a user column to see their complete record. More flexible than embedding metadata in headers.

**Consequences:** Created 48 `*_schema.tsv` files with this format.

## 2. Organize Schema TSVs in Dated Subfolder
**Type:** Implementation
**Topics:** workflow

**Trigger:** User requested files be moved to project directory in a dated subfolder.

**Options Considered:**
1. Project root (cluttered)
2. `data/schema/` folder (no date)
3. `data/schema/YYYY-MM-DD/` folder ← Chosen

**Decision:** Store schema TSV exports in `data/schema/2026-01-31/`.

**Rationale:** Dated folders allow keeping multiple snapshots for comparison as schema evolves during development.

**Consequences:** Created `data/schema/2026-01-31/` with all 48 schema files.

## 3. Include Schema-Only Files for Empty Tables
**Type:** Implementation
**Topics:** d1

**Trigger:** 20 tables have no seed data.

**Options Considered:**
1. Skip tables without seed data
2. Create schema-only TSV files ← Chosen

**Decision:** Create TSV files for all 48 tables, even those without seed data.

**Rationale:** Complete documentation of all tables. Schema-only files still document columns, types, constraints, and foreign keys - useful for understanding the full data model.

**Consequences:** All tables documented consistently.
