# Session Learnings - 2026-01-21

## 1. Moderators Can Share Admin API Endpoints with Role Restrictions
**Topics:** auth, api

**Context:** Building the moderator system required deciding whether to create duplicate API endpoints for moderators or share the existing admin moderation endpoints.

**Learning:** Rather than duplicating APIs, existing admin endpoints can accept multiple roles via `requireRole(cookies, jwtSecret, ['admin', 'moderator'])`. Role-specific restrictions (like "permanent suspension requires admin") can be added inline with `session.roles.includes('admin')` checks.

**Pattern:**
```typescript
// Allow both admin and moderator access
const session = await requireRole(cookies, jwtSecret, ['admin', 'moderator']);

// Add role-specific restriction inline
if (body.duration === 'permanent' && !session.roles.includes('admin')) {
  return Response.json({ error: 'Permanent suspensions require admin privileges' }, { status: 403 });
}
```

## 2. Context-Aware UI Actions with Registry Pattern
**Topics:** astro, workflow

**Context:** Different users (admin, creator owner, student-teacher) need different quick actions when viewing the same page (course detail, creator profile).

**Learning:** A registry pattern cleanly maps page types + roles to available actions. The component fetches session, determines ownership, then queries the registry for applicable actions. Actions are defined once and filtered by context.

**Pattern:**
```typescript
// registry.ts - Define actions once per page type + role
export const actionRegistry: ActionRegistry = {
  course: {
    admin: [{ id: 'feature', type: 'api', endpoint: '/api/admin/courses/{id}/feature' }],
    creator_owner: [{ id: 'edit', type: 'navigate', href: '/dashboard/creator/courses/{slug}/edit' }],
  }
};

// Component - Query registry based on user context
const actions = getActionsForContext(pageType, user, { isOwner });
```

## 3. Ownership Detection in Client Components
**Topics:** auth, astro

**Context:** The ContextActionsPanel needs to show owner-specific actions (like "Edit Profile") when viewing your own profile or course.

**Learning:** Pass entity IDs (creatorId, userId) from SSR-rendered Astro pages to client components. The client fetches session via `/api/auth/session`, then compares `user.id === entityData.creatorId` to determine ownership. This avoids duplicating ownership logic in the API.

**Pattern:**
```tsx
// Astro page passes creatorId from SSR data
<ContextActionsPanel context={{
  pageType: 'course',
  entityId: course.slug,
  entityData: { creatorId: course.creator.id }
}} />

// Component computes ownership client-side
const isOwner = user && context.entityData?.creatorId
  ? user.id === context.entityData.creatorId
  : false;
```
