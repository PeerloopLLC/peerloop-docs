# Session Decisions - 2026-01-20

## 1. CDIS Course Feed Access Model
**Type:** Strategic
**Topics:** stream, auth

**Trigger:** Defining requirements for the Course Discuss feed (CDIS) before ending session.

**Options Considered:**
1. Authenticated-only view and post
2. Public view, enrolled-only post ← Chosen
3. Enrolled-only view and post (like IFED)

**Decision:** CDIS will allow public viewing but restrict posting to enrolled/completed students, course creator, and student teachers.

**Rationale:** Public viewing allows potential students to see community activity as social proof. Restricting posting to enrolled users maintains quality and relevance.

**Consequences:** Access model documented in PLAN.md under COMMUNITY.COURSEFEED:
| Role | View | Post |
|------|------|------|
| Course Creator | Yes | Yes |
| Student Teachers (this course) | Yes | Yes |
| Enrolled/Completed Students | Yes | Yes |
| Non-enrolled visitors | Yes | No |

## 2. Client-Side Filtering for MVP Feeds
**Type:** Implementation
**Topics:** stream

**Trigger:** Need to filter townhall activities by instructor or course without separate Stream feed groups.

**Options Considered:**
1. Create separate Stream feed groups (course, instructor) - proper architecture
2. Filter townhall client-side by custom fields ← Chosen
3. Duplicate posts to multiple feeds

**Decision:** For MVP, filter the townhall feed client-side rather than implementing separate Stream feed groups.

**Rationale:** Simpler implementation, faster to ship. Can migrate to proper feed groups later when scaling requires it. The over-fetch (3x limit) is acceptable for MVP volume.

**Consequences:**
- InstructorFeed fetches 3x requested limit, filters by courseId/actor
- CourseFeed will follow same pattern
- May need to revisit when feed volume grows
