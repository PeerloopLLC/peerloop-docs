# Session Decisions - 2026-01-19

## 1. Use Stream v2 SDK Instead of v3
**Type:** Implementation
**Topics:** stream

**Trigger:** v3 SDK installed but getting "Feed group with ID 'townhall' not found" errors despite the feed group existing in the dashboard.

**Options Considered:**
1. Continue with v3 SDK and figure out separate configuration
2. Switch to v2 SDK (`getstream` package) ← Chosen
3. Wait for Stream to update their dashboard for v3

**Decision:** Switch to the v2 `getstream` package for Stream Activity Feeds integration.

**Rationale:** The Stream dashboard explicitly states v3 is not yet integrated. The v2 SDK has full dashboard support, more documentation, and is battle-tested. Development velocity is more important than using the newest SDK version.

**Consequences:**
- Uninstalled `@stream-io/node-sdk` and `@stream-io/feeds-react-sdk`
- Installed `getstream` package
- Updated `src/lib/stream.ts` to use v2 API (`connect()` instead of `new StreamClient()`)
- Added Vite polyfills for `global` and `process.env`

## 2. Server-Side Feed Operations (Proxy Pattern)
**Type:** Implementation
**Topics:** stream, auth, astro

**Trigger:** Client-side tokens getting 403 "no policies allowing this request" when accessing shared `townhall:main` feed.

**Options Considered:**
1. Configure feed policies in Stream Dashboard (if available)
2. Disable auth checks in Stream (development only)
3. Proxy all feed operations through server API ← Chosen
4. Use server-generated tokens with elevated permissions

**Decision:** Implement server-side API endpoints (`/api/feeds/townhall`) that proxy feed operations using the API secret.

**Rationale:**
- Feed policies UI not available in Stream v2 dashboard
- Server-side approach is more secure (API secret never exposed to client)
- Allows enforcing our own auth and business logic
- Simplifies client code (just fetch from our API)
- Pattern scales well to other feeds (course, instructor)

**Consequences:**
- Created `src/pages/api/feeds/townhall.ts` (GET and POST)
- Simplified `TownHallFeed.tsx` to use our API instead of Stream SDK
- Removed need for `StreamProvider` context component
- Client no longer needs Stream SDK or user tokens

## 3. Remove StreamProvider Component
**Type:** Implementation
**Topics:** stream, astro

**Trigger:** After implementing server-side feed operations, the StreamProvider (which managed client-side Stream connection) became unnecessary.

**Options Considered:**
1. Keep StreamProvider for potential future client-side features
2. Remove StreamProvider since server-side pattern is preferred ← Chosen

**Decision:** Remove `StreamProvider.tsx` and simplify the component structure.

**Rationale:**
- Server-side operations are more secure and simpler
- StreamProvider added complexity (token fetching, connection state, error handling)
- YAGNI - if we need client-side Stream features later, we can add them back
- Cleaner codebase with fewer dependencies

**Consequences:**
- Deleted `src/components/community/StreamProvider.tsx`
- Updated `src/components/community/index.ts` to only export `TownHallFeed`
- Simplified `src/pages/community/index.astro` (no longer wraps with StreamProvider)
- `TownHallFeed.tsx` is now a standalone component using fetch API
