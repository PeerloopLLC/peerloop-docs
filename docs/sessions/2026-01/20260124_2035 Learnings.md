# Session Learnings - 2026-01-24

## 1. Migration Ordering with Foreign Keys
**Topics:** d1, testing

**Context:** When consolidating schema migrations into a single file, the `payment_splits` table referenced `payouts(id)` but `payouts` was defined later in the file.

**Learning:** SQLite requires tables to exist before they can be referenced by foreign keys. When writing consolidated schema files, order tables so that referenced tables are created before referencing tables.

**Pattern:**
```sql
-- ✓ Correct order: referenced table first
CREATE TABLE payouts (...);
CREATE TABLE payment_splits (
  payout_id TEXT REFERENCES payouts(id)
);

-- ✗ Wrong order: will fail
CREATE TABLE payment_splits (
  payout_id TEXT REFERENCES payouts(id)  -- payouts doesn't exist yet!
);
CREATE TABLE payouts (...);
```

## 2. Two-File Migration Strategy for Pre-Production
**Topics:** d1, workflow, testing

**Context:** The project had 19 migration files with ALTER TABLE statements scattered across them, causing failures when applying to fresh test databases (duplicate column errors).

**Learning:** For pre-production applications, a two-file migration system works better than incremental migrations:
- `0001_schema.sql` - Complete current schema (single source of truth)
- `0002_seed.sql` - All seed data

This approach is:
- **Idempotent** - Fresh DBs get full schema, no ALTER TABLE failures
- **Easy to evolve** - Change schema freely during development
- **Testable** - Tests always work against consistent schema
- **Future-proof** - Add `0003+` migrations when production goes live

## 3. Schema Mismatch Detection in Tests
**Topics:** testing, d1

**Context:** When writing API tests, test data INSERTs often assumed columns existed that didn't (or vice versa), causing test failures.

**Learning:** Before writing INSERT statements in tests:
1. Read the actual migration file to verify column names
2. Check which columns are NOT NULL vs nullable
3. Present schema mismatches to the user as design decisions, not just test fixes

The schema may be wrong, not the test! During early development, tests can reveal missing columns that SHOULD exist.

## 4. Test Database Uses better-sqlite3, Not D1
**Topics:** testing, d1

**Context:** The test infrastructure uses better-sqlite3 (in-memory SQLite) which applies all migrations fresh each run, unlike D1 which tracks applied migrations.

**Learning:** The test database:
- Creates fresh in-memory DB on each test run
- Applies ALL `.sql` files from `migrations/` in alphabetical order
- Has no migration tracking (unlike wrangler's D1)
- Works identically on all development machines (no macOS version requirement)

This means ALTER TABLE migrations that worked fine on D1 will fail on test DB if the column already exists from the CREATE TABLE.
