# Session Learnings - 2026-01-21

## 1. SQLite ALTER TABLE Cannot Use Non-Constant Defaults
**Topics:** d1, database

**Context:** When adding columns to an existing `certificates` table, the migration used `DEFAULT (datetime('now'))` for the `created_at` column.

**Learning:** SQLite does not allow non-constant default values (like function calls) in ALTER TABLE statements. The error message is "Cannot add a column with non-constant default". This differs from CREATE TABLE which allows such defaults.

**Pattern:**
```sql
-- FAILS in ALTER TABLE:
ALTER TABLE certificates ADD COLUMN created_at TEXT NOT NULL DEFAULT (datetime('now'));

-- WORKS in ALTER TABLE (use NULL or literal):
ALTER TABLE certificates ADD COLUMN created_at TEXT;
-- Then handle datetime in application code when inserting
```

## 2. Existing Table Schema Requires ALTER Not CREATE
**Topics:** d1, database

**Context:** The migration initially used `CREATE TABLE IF NOT EXISTS certificates` but the table already existed with a different schema (missing columns like `status`, `revoked_at`).

**Learning:** `CREATE TABLE IF NOT EXISTS` silently succeeds when the table exists, regardless of schema differences. To add columns to an existing table, use `ALTER TABLE ADD COLUMN` statements instead. Always check the current table schema before writing migrations.

**Pattern:**
```bash
# Check existing table schema before migration
npx wrangler d1 execute peerloop-db --remote \
  --command "PRAGMA table_info(certificates)"
```

## 3. Astro Files Use `class` Not `className`
**Topics:** astro

**Context:** IDE diagnostics showed TypeScript errors about `class` property not existing on JSX elements in `.astro` files.

**Learning:** Astro template syntax uses standard HTML `class` attribute, not React's `className`. IDE/TypeScript diagnostics for `.astro` files may show false positives about this. The CLI checks (`tsc --noEmit`, `astro check`) are the authoritative source for Astro file validity.

## 4. TypeScript Array Type Inference With Optional Properties
**Topics:** typescript

**Context:** An array of action objects was being built conditionally with `.push()`, but some objects had `variant: 'danger'` while initial objects did not.

**Learning:** TypeScript infers array element types from the initial values. If initial objects lack a property that later pushed objects have, TypeScript will error. Solution: explicitly type the array or include all possible properties in initial objects.

**Pattern:**
```typescript
// Type the array explicitly to allow optional properties
const actions: ActionItem[] = [
  { label: 'View', icon: Icon, onClick: () => {} },
];

// Now this works even though variant wasn't in initial items
actions.push({ label: 'Delete', icon: Icon, onClick: () => {}, variant: 'danger' });
```

## 5. Teaching Certificate Approval Requires S-T Record Management
**Topics:** workflow, auth

**Context:** When approving a teaching certificate, the system needs to also create/update the student_teachers record and set the user's `is_student_teacher` flag.

**Learning:** Teaching certificates are tightly coupled with student_teacher records. Approval must: 1) Update certificate status, 2) Create/reactivate student_teachers record, 3) Set `users.is_student_teacher = 1`. Revocation must reverse steps 2-3 (deactivate S-T record, clear flag if no other active S-T certifications).

**Pattern:**
```typescript
// On teaching cert approval:
if (certificate.type === 'teaching') {
  // Create or reactivate ST record
  await db.prepare(`INSERT OR UPDATE student_teachers...`);
  // Update user flag
  await db.prepare('UPDATE users SET is_student_teacher = 1 WHERE id = ?');
}
```
