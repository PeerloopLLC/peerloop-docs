# Session Decisions - 2026-01-18

## 1. Config File Format and Location
**Type:** Strategic
**Topics:** workflow

**Trigger:** Needed centralized configuration for global skills to read project-specific settings.

**Options Considered:**
1. Markdown file in project root
2. JSON file in project root
3. JSON file in `.claude/` folder ← Chosen

**Decision:** Use `.claude/config.json` for project configuration.

**Rationale:** JSON is explicit and parseable with consistent field names. Location in `.claude/` groups it with other Claude Code files.

**Consequences:** Created config.json with sections for terminology, features, paths, thresholds, and framework settings. All global skills updated to read from this file with fallback defaults.

## 2. Skill Naming Convention
**Type:** Strategic
**Topics:** workflow

**Trigger:** Need to distinguish between portable global skills, local extensions, and local-only skills.

**Options Considered:**
1. All skills use `q-` prefix regardless of scope
2. Use `q-` for global, `q-*-local` for extensions, different prefix for local-only ← Chosen
3. Use completely different prefixes for each type

**Decision:** Use `L-` prefix for local-only skills (no global equivalent), keep `q-*-local` for extensions of global skills.

**Rationale:** Clear visual distinction. `L-` immediately signals "this is Peerloop-specific with no portable version."

**Consequences:** Renamed `q-add-client-note`, `q-consolidate-page`, `q-verify-page` to `L-add-client-note`, `L-consolidate-page`, `L-verify-page`.

## 3. Resume Skill Naming
**Type:** Implementation
**Topics:** workflow

**Trigger:** `/q-resume` didn't match naming pattern of its counterpart `/q-end-session`.

**Options Considered:**
1. `q-resume-session` - mirrors q-end-session ← Chosen
2. `q-start-session` - clearer opposite of "end"
3. `q-session-start` / `q-session-end` - noun-first pattern

**Decision:** Rename to `q-resume-session` for consistency with `q-end-session`.

**Rationale:** Logical pairs should follow same naming pattern. Resume/end are the session bookends.

**Consequences:** Renamed skill and all references to it.

## 4. Force-End Must Call Local Extensions
**Type:** Strategic
**Topics:** workflow

**Trigger:** Initially assumed force-end should skip local extensions for speed.

**Options Considered:**
1. Skip local extensions in force-end for speed
2. Always call local extensions when global runs ← Chosen

**Decision:** Force-end calls local extensions. Speed savings come from removing gates, not skipping locals.

**Rationale:** User made clear that if a global skill runs, local must run too. The user decides what to skip, not the skill.

**Consequences:** Updated `/q-force-end` to include Step 4 calling `/q-force-end-local`.

## 5. Git History Split into Global/Local
**Type:** Implementation
**Topics:** workflow

**Trigger:** `/q-git-history` had generic git extraction plus project-specific "client perspective" guidance.

**Options Considered:**
1. Keep as local-only skill
2. Split into global (extraction) + local (perspective guidance) ← Chosen

**Decision:** Create global `/q-git-history` for extraction, local `/q-git-history-local` for client perspective categories.

**Rationale:** Git extraction is portable. Client perspective categories are project-specific (what to highlight for users/admins).

**Consequences:** Created global skill, local extension, and added `editor` setting to config.json.
