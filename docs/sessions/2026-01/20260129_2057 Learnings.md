# Session Learnings - 2026-01-29

## 1. MBA-2017 Uses Different D1 Database Than Wrangler CLI
**Topics:** d1, cloudflare, deployment

**Context:** Console logs weren't appearing on `/` route. After navigating to correct page, got 500 error: "no such column: can_create_courses". We had just run migrations but the error persisted.

**Learning:** MBA-2017's REST API client (`d1-rest.ts`) reads `D1_DATABASE_ID` from `.dev.vars`, which points to the **staging** database (`605f1ab8-...`). Meanwhile, the wrangler CLI defaults to **production** (`21c0179a-...`). Running `npm run db:migrate:remote` migrates production, not staging.

**Pattern:**
```
# What we ran (migrates PRODUCTION):
npm run db:migrate:remote

# What we needed (migrates STAGING that MBA-2017 uses):
wrangler d1 migrations apply peerloop-db-staging --remote --env preview
```

| Database | ID | Used By |
|----------|-----|---------|
| Production | `21c0179a-9e7d-4cc1-91d5-ea402717fce8` | wrangler CLI default, Cloudflare Pages main branch |
| Staging | `605f1ab8-62cc-4934-a2fd-d828e188f50e` | MBA-2017 REST API, preview deployments |

## 2. Wrangler Environment Flag for Staging is `--env preview`
**Topics:** d1, cloudflare, workflow

**Context:** Tried using `--env staging` to target the staging database, but it still hit production.

**Learning:** In `wrangler.toml`, the staging database is configured under `[env.preview]`, not `[env.staging]`. The environment name in wrangler must match the TOML section name exactly.

**Pattern:**
```bash
# Wrong - no [env.staging] section exists
wrangler d1 migrations apply peerloop-db --remote --env staging

# Correct - matches [env.preview] section in wrangler.toml
wrangler d1 migrations apply peerloop-db-staging --remote --env preview
```

Also note: the database name changes (`peerloop-db` → `peerloop-db-staging`) depending on environment.

## 3. DashLayout vs MainLayout Page Identification
**Topics:** astro, workflow

**Context:** User expected console.logs from DashNavbar on what they thought was the `/` route, but they weren't appearing.

**Learning:** The Creator Dashboard page uses `AppHeader` (top navigation bar), not `DashNavbar` (persistent left sidebar). Quick indicators:
- **DashLayout**: 256px left sidebar, no top header
- **MainLayout/AppHeader**: Hamburger menu, "Peerloop" logo in top bar, user dropdown in upper right

Network panel helps identify: requests from `AppHeader.tsx` indicate MainLayout, not DashLayout.
