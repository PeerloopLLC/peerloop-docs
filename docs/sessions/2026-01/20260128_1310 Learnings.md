# Session Learnings - 2026-01-28

## 1. Distinguishing Legitimate vs Seed-Compensation Assertions
**Topics:** testing

**Context:** After excluding seed data from test databases, needed to identify which `toBeGreaterThanOrEqual(N)` assertions should be tightened to exact counts vs which are legitimate.

**Learning:** Not all `>=` assertions are seed-compensation. The key question is: "Does the test setup create a specific number of items?" If yes, assert the exact count.

**Pattern:**
| Pattern | Example | Keep `>=`? |
|---------|---------|------------|
| **Ordering check** | `prev.score >= current.score` | ✅ Yes |
| **Position check** | `indexOf >= 0` | ✅ Yes |
| **Technical check** | `latency_ms >= 0` | ✅ Yes |
| **Count assertion** | `items.length >= 4` | ❌ → `.toBe(N)` |
| **Has "seed" comment** | `// could include seed data` | ❌ → fix + remove comment |

**Quick identification:**
1. Grep for `toBeGreaterThanOrEqual` in test files
2. Grep for `seed` comments (explicit compensation)
3. For each hit, check if it's in a loop comparing consecutive items (ordering) or asserting a count
4. Only tighten count assertions where the test setup creates a known quantity

## 2. Test Assertion Tightening Scope
**Topics:** testing

**Context:** After tightening 44 assertions across 12 files, identified which test files have the most seed-compensation patterns.

**Learning:** Admin dashboard and admin listing endpoints tend to have the most loose assertions because they aggregate multiple entity counts. These are the highest-priority files to review when tightening assertions.

**High-density files (most assertions tightened):**
- `tests/api/admin/dashboard.test.ts` - 14 assertions (stats for users, courses, enrollments, STs, revenue)
- `tests/api/admin/courses/index.test.ts` - 7 assertions (various filter counts)
- `tests/api/admin/users/index.test.ts` - 6 assertions (role/status filter counts)
