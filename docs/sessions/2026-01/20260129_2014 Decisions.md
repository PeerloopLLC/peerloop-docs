# Session Decisions - 2026-01-29 (Continued)

## 1. Two-Global Architecture for State Management
**Type:** Strategic
**Topics:** auth, astro

**Trigger:** Initial implementation mixed network status into InitUserResult return type, which felt hacky for the currentUser global.

**Options Considered:**
1. Single global with all state mixed together
2. Two globals: currentUser (user data) + networkState (API status) ← Chosen
3. Separate file/module for network state

**Decision:** Implement two separate globals on `window.__peerloop`: `currentUser` for pure user data, `networkState` for auth/API status.

**Rationale:** Clean separation of concerns. User data shouldn't be polluted with error strings. networkState is extensible for future needs (apiHealth, isOnline). Clear mental model.

**Consequences:**
- Created NetworkState type with authStatus, authError, lastAuthAttempt
- Added getNetworkState() accessor
- DashNavbar reads from both globals
- Documented in docs/tech/tech-017-state-management.md

## 2. Page Area Organization for CURRENTUSER
**Type:** Strategic
**Topics:** workflow, astro

**Trigger:** Needed to understand how currentUser and networkState apply across different parts of the app.

**Options Considered:**
1. One global initialization for entire app
2. Area-specific initialization (APP, ADMIN, PUBLIC) ← Chosen
3. Lazy initialization in accessors

**Decision:** Each page area has its own navbar component that initializes globals:
- APP pages: DashNavbar
- ADMIN pages: AdminNavbar (to be created)
- PUBLIC pages: No navbar (handle individually, WELC is exception)

**Rationale:** Astro pages reset window on navigation anyway, so no benefit to global initialization. Each area can have area-specific menu items and behavior. Simpler mental model.

**Consequences:**
- Added CURRENTUSER.APP, .ADMIN, .PUBLIC sections to PLAN.md
- AdminNavbar component to be created
- PUBLIC pages assessed individually for needs
