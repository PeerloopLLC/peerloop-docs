# Session Decisions - 2026-01-29

## 1. Dashboard as Homepage
**Type:** Strategic
**Topics:** astro, workflow

**Trigger:** Client wanted visitors to see the app immediately, not a marketing page.

**Options Considered:**
1. Marketing landing → Dashboard redirect for logged-in users
2. Dashboard for everyone, marketing at `/welcome` ← Chosen
3. Hybrid minimal marketing + quick login

**Decision:** The homepage (`/`) now shows DashLayout for all users (visitors and authenticated). Marketing content moved to `/welcome` (WELC).

**Rationale:** Dashboard-first experience shows the product immediately. Marketing content preserved at `/welcome` for campaigns and links.

**Consequences:**
- `src/pages/index.astro` now uses DashLayout
- `src/pages/welcome.astro` created with old HOME content
- `src/data/pages/index.json` updated for dashboard
- `src/data/pages/welcome.json` created
- `/dash` route deleted (was redundant)
- `/dash/*` sub-routes remain

## 2. CurrentUser Storage Location
**Type:** Implementation
**Topics:** astro, auth

**Trigger:** Need shared user state across Astro islands that bundle separately.

**Options Considered:**
1. React Context - only works in React tree
2. Window global - universal but untyped
3. Module singleton - may create multiple instances
4. nanostores - another dependency ← Rejected
5. Class singleton on window with typed wrapper ← Chosen

**Decision:** Use a `CurrentUser` class singleton stored on `window.__peerloop.currentUser` with typed access functions.

**Rationale:** Window is shared across all islands. Typed wrapper provides IDE support. Class encapsulates role-checking logic. No extra dependencies.

**Consequences:** Created `src/lib/current-user.ts` with `getCurrentUser()`, `isVisitor()`, `initializeCurrentUser()`.

## 3. Single API Endpoint for User State
**Type:** Implementation
**Topics:** auth, d1

**Trigger:** Need to load user identity + all course relationships at page load.

**Options Considered:**
1. Multiple parallel API calls (session, enrollments, certifications) ← Rejected
2. Single `/api/me/full` endpoint ← Chosen

**Decision:** Create single `GET /api/me/full` endpoint that returns user identity, capabilities, enrollments, ST certifications, and created courses in one response.

**Rationale:** Less complexity than parallel calls. Single point of failure is easier to handle. Response is cached in localStorage anyway.

**Consequences:** Created `src/pages/api/me/full.ts` with parallel DB queries internally.

## 4. Explicit Course Context (Not URL-Implicit)
**Type:** Implementation
**Topics:** astro, auth

**Trigger:** Role methods need to know which course to check.

**Options Considered:**
1. Implicit from URL - CurrentUser parses URL for courseId ← Rejected
2. Explicit parameter - components pass courseId ← Chosen

**Decision:** Role methods take explicit courseId: `isStudentTeacherFor(courseId)`, `isCreatorFor(courseId)`.

**Rationale:** Explicit is clearer, works on multi-course pages (browse, dashboard), no URL parsing magic, easier to test.

**Consequences:** All role-checking methods require courseId parameter. Components that need course context must have access to it.

---

## 5. New Permission Flags in User Schema
**Type:** Strategic
**Topics:** d1, auth

**Trigger:** Old `is_*` role flags are global but some roles are course-specific.

**Options Considered:**
1. Keep using old flags, add course-specific checks elsewhere ← Rejected
2. Add new capability flags, deprecate old flags ← Chosen

**Decision:** Added four new permission fields to users table:
- `can_create_courses` (default: 0)
- `can_take_courses` (default: 1)
- `can_teach_courses` (default: 0)
- `can_moderate_courses` (default: 0)

Old flags marked deprecated but kept for backwards compatibility.

**Rationale:** Separates "what you CAN do" (capabilities, admin-controlled) from "what you ARE" (roles, course-specific). Cleaner mental model.

**Consequences:**
- Updated `migrations/0001_schema.sql` with new columns
- Updated `migrations/0002_seed.sql` with values
- Updated `src/lib/db/types.ts` with new fields
- `/api/me/full` returns new fields
- `CurrentUser` class has `canCreateCourses`, etc. properties
