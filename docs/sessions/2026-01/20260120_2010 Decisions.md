## 1. BigBlueButton Over PlugNmeet for Video Platform
**Type:** Strategic
**Topics:** video

**Trigger:** Mid-session client direction change

**Options Considered:**
1. PlugNmeet - Modern UI, microservices architecture, lower cost
2. BigBlueButton ← Chosen - Established, education-focused, client preference

**Decision:** Use BigBlueButton as the video conferencing platform for PeerLoop.

**Rationale:** Brian (client) made the final call to use BBB instead of PlugNmeet. The VideoProvider interface abstraction means we can swap if needed later.

**Consequences:**
- Created BBB adapter implementing VideoProvider interface
- Updated sessions table schema from plugnmeet_* to bbb_* columns
- Updated tech docs (tech-001 as SELECTED, tech-006 as NOT SELECTED)
- Migration 0011 handles column rename

## 2. Store BBB Passwords in Sessions Table
**Type:** Implementation
**Topics:** video, d1

**Trigger:** BBB requires passwords for join URLs, need to avoid re-querying

**Options Considered:**
1. Re-query BBB getMeetingInfo each time ← More API calls
2. Store in-memory cache ← Lost on worker restart
3. Store in sessions table ← Chosen - Persistent, simple

**Decision:** Add bbb_attendee_pw and bbb_moderator_pw columns to sessions table.

**Rationale:** Storing passwords in DB means any worker instance can generate join URLs without additional BBB API calls. Passwords are only meaningful for the session duration anyway.

**Consequences:**
- Added two columns to migration 0011
- Updated Session type in db/types.ts
- BBB adapter caches passwords in-memory during room creation
- Join endpoint can retrieve from DB if needed
