# Session Learnings - 2026-01-26

## 1. Test File Organization Should Match API Route Structure
**Topics:** testing, workflow

**Context:** Started with 189 test files for 166 API endpoints. Files were inconsistently organized with separate `list.test.ts`, `create.test.ts`, `detail.test.ts`, and `actions.test.ts` files.

**Learning:** Test files in `[id]/` subdirectories mirror Astro's file-based routing, making the test-to-endpoint mapping immediately obvious. The pattern:
- `index.test.ts` → collection endpoints (GET list, POST create)
- `[id]/index.test.ts` → item endpoints (GET detail, PATCH, DELETE)
- `[id]/action.test.ts` → action endpoints (approve, suspend, etc.)

## 2. Path Aliases Eliminate Deep Relative Import Errors
**Topics:** testing, workflow

**Context:** Moving test files between directories required updating deep relative imports like `../../../../../src/pages/api/...`. Counting `../` levels was error-prone.

**Learning:** Path aliases (`@/`, `@test-helpers`, `@api-helpers`) make imports location-independent. When reorganizing files, only the file moves—no import paths need changing.

**Pattern:**
```typescript
// Before (fragile)
import { POST } from '../../../../../src/pages/api/admin/users/[id]/suspend';
import { createAPIContext } from '../../../helpers';
import { describeWithTestDB } from '../../../../helpers';

// After (stable)
import { POST } from '@/pages/api/admin/users/[id]/suspend';
import { createAPIContext } from '@api-helpers';
import { describeWithTestDB } from '@test-helpers';
```

Both `tsconfig.json` and `vitest.config.ts` need the same alias definitions.

## 3. BSD sed Has Regex Limitations - Use perl Instead
**Topics:** workflow

**Context:** Attempted complex find-and-replace with nested regex groups in sed on macOS.

**Learning:** BSD sed (macOS default) has limited regex support compared to GNU sed. For complex patterns with groups and alternations, use perl:

**Pattern:**
```bash
# BSD sed fails on complex patterns
sed -i '' "/pattern/s|old|new|g"  # Limited

# perl handles complex regex reliably
perl -i -pe "s/pattern/replacement/g"
```

## 4. Vitest Cannot Target [id] Directories via CLI Filter
**Topics:** testing

**Context:** Running `npm test -- --run 'tests/api/admin/users/[id]/suspend.test.ts'` failed because `[id]` was interpreted as a regex character class.

**Learning:** Vitest runs tests in bracket-named directories fine when included via broader patterns, but the CLI `--run` filter interprets `[` as regex. Tests in these directories work when running the parent directory or all tests.

**Workaround:** Run the parent directory instead:
```bash
npm test -- --run tests/api/admin/users/  # Works
```

## 5. Simple Count Comparison Detects Duplicate Tests
**Topics:** testing, workflow

**Context:** Suspected test duplication but wasn't sure of the scope.

**Learning:** Comparing file counts quickly reveals mismatches:

**Pattern:**
```bash
# Count test files vs endpoint files
echo "Tests:" && find tests/api -name '*.test.ts' | wc -l
echo "Endpoints:" && find src/pages/api -name '*.ts' | wc -l

# Find specific differences
diff <(find tests/api -name '*.test.ts' | sed 's|tests/||' | sort) \
     <(find src/pages/api -name '*.ts' | sed 's|src/pages/||' | sort)
```

A 1:1 ratio indicates proper coverage without duplication.
