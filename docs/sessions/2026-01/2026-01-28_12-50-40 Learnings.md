# Session Learnings - 2026-01-28

## 1. Seed Data in Test DB Causes Test Pollution
**Topics:** testing, d1

**Context:** Tests were failing because they expected specific counts (e.g., 4 enrollments) but got more (10) due to seed data being applied to the test database.

**Learning:** The `resetTestDB()` function was applying all migration files including `0002_seed.sql`. Tests should have isolated, controlled data - seed data pollutes test state and causes unpredictable assertions.

**Pattern:**
```typescript
// In test-db.ts - skip seed files for test isolation
const files = fs.readdirSync(migrationsDir)
  .filter((f) => f.endsWith('.sql') && !f.includes('seed'))
  .sort();
```

## 2. Vitest vi.mock Hoisting Requires vi.hoisted() for Variables
**Topics:** testing

**Context:** Stream and Stripe tests were failing with "Error when mocking a module" because mock variables were defined after `vi.mock()` calls, but `vi.mock` is hoisted to the top of the file.

**Learning:** When using variables inside `vi.mock()` factory functions, those variables must also be hoisted using `vi.hoisted()`. Otherwise, they're undefined when the mock runs.

**Pattern:**
```typescript
// WRONG - variables don't exist when vi.mock runs
const mockFn = vi.fn();
vi.mock('@/lib/module', () => ({ fn: mockFn }));

// CORRECT - use vi.hoisted to ensure variables exist
const { mockFn } = vi.hoisted(() => ({
  mockFn: vi.fn(),
}));
vi.mock('@/lib/module', () => ({ fn: mockFn }));
```

## 3. R2 Mocks Must Capture Written Content for Read-Back Verification
**Topics:** testing, cloudflare

**Context:** R2 health check tests were failing because the mock returned hardcoded content that didn't match what was written. The endpoint writes content, reads it back, and verifies they match.

**Learning:** When mocking R2 (or any storage), if the code writes then reads and compares, the mock's `get()` must return exactly what was passed to `put()`. Use a closure to capture the written content.

**Pattern:**
```typescript
let writtenContent = '';
const mockR2 = {
  put: vi.fn().mockImplementation((_key, content) => {
    writtenContent = content;
    return Promise.resolve(undefined);
  }),
  get: vi.fn().mockImplementation(() => Promise.resolve({
    text: () => Promise.resolve(writtenContent), // Return what was written
  })),
  delete: vi.fn().mockResolvedValue(undefined),
};
```

## 4. Check Auth/Validation Before External Service Availability
**Topics:** testing, auth, cloudflare

**Context:** Resource download tests expected 401/403 for auth failures but got 503 because the endpoint checked R2 availability before checking authentication.

**Learning:** Endpoints should check auth and validation (resource exists, user enrolled, etc.) BEFORE checking if external services (R2, Stripe) are available. This provides meaningful errors and avoids 503s when auth would fail anyway.

**Pattern:**
```typescript
// WRONG order - 503 hides auth errors in tests
const r2 = locals.runtime?.env?.STORAGE;
if (!r2) return Response.json({ error: 'Storage not available' }, { status: 503 });
// ... then auth check

// CORRECT order - auth errors returned first
const session = await getSession(cookies, jwtSecret);
if (!session) return Response.json({ error: 'Auth required' }, { status: 401 });
// ... then R2 check (only when we actually need it)
```

## 5. OAuth Test Mocking Strategy
**Topics:** testing, auth

**Context:** OAuth tests were failing because they mocked `generateState()` and `generateCodeVerifier()` from arctic, but the actual `Google`/`GitHub` provider classes were still instantiated and failed without real credentials.

**Learning:** Mock the provider factory functions (`createGoogleProvider`, `createGitHubProvider`) to return mock objects with `createAuthorizationURL()`, rather than trying to mock individual arctic utilities.

**Pattern:**
```typescript
vi.mock('@lib/auth', async (importOriginal) => {
  const actual = await importOriginal<typeof import('@lib/auth')>();
  return {
    ...actual,
    createGoogleProvider: vi.fn(() => ({
      createAuthorizationURL: vi.fn((state, verifier, scopes) => {
        const url = new URL('https://accounts.google.com/o/oauth2/v2/auth');
        url.searchParams.set('state', state);
        return url;
      }),
    })),
  };
});
```

## 6. Response.redirect() Requires Absolute URLs
**Topics:** astro, auth

**Context:** OAuth callback endpoints were failing because `Response.redirect('/login?error=...')` was passed a relative path, which isn't a valid absolute URL.

**Learning:** `Response.redirect()` requires an absolute URL. Build it using the request's `url.origin`.

**Pattern:**
```typescript
// Inside the handler where `url` is available from the request
const redirectWithError = (path: string, error: string): Response => {
  const errorUrl = `${url.origin}${path}?error=${encodeURIComponent(error)}`;
  return Response.redirect(errorUrl, 302);
};
```

## 7. "Failed Files" vs "Failed Tests" in Vitest
**Topics:** testing

**Context:** Vitest showed "6 failed files" but "0 failed tests", which was confusing.

**Learning:** Vitest marks a file as "failed" in two cases:
1. Import/setup errors (test file can't even run)
2. All tests in the file are skipped

This is different from individual test failures. When `beforeAll` throws an error, all tests in that describe block are skipped, and the file shows as failed.

## 8. Schema Evolution Creates Test Data Debt
**Topics:** testing, d1

**Context:** Many tests failed with NOT NULL constraint violations because test data INSERTs were written for an older schema version and missing required columns like `handle`, `created_by`, `duration`, `label`, `recipient_type`.

**Learning:** When schema evolves (columns added, made NOT NULL), test data fixtures often lag behind. This creates "test data debt" that surfaces as constraint violations. Consider:
- Centralizing test data creation helpers
- Reviewing test data when schema changes
- Using factory functions that always include all required fields

## 9. user_stats Uses user_id as Primary Key (No Separate ID)
**Topics:** d1, testing

**Context:** Tests were inserting `(id, user_id, ...)` into `user_stats` but the table uses `user_id` as the primary key with no separate `id` column.

**Learning:** Some tables use a foreign key as the primary key (1:1 relationships). Always check the schema before writing INSERT statements - don't assume every table has a separate `id` column.

## 10. user_qualifications Uses Simplified Schema
**Topics:** d1

**Context:** Tests tried to insert structured qualification data (`type`, `title`, `institution`, `year`) but the schema uses a single `sentence` field.

**Learning:** The `user_qualifications` table was simplified to just `sentence` (full text) instead of structured fields. The endpoint returns this as-is. Test assertions should check `sentence` field, not `title`.
