# Session Learnings - 2026-01-25

## 1. Vitest Module Mocking: Mock at the Boundary Function
**Topics:** testing, auth

**Context:** Writing tests for admin API endpoints that use `requireRole()` for authentication. Attempted to mock `getSession()` (which `requireRole` calls internally) but admin endpoints still returned 302 redirects.

**Learning:** When using `vi.mock()`, mocking an internal function doesn't affect callers within the same module. If `requireRole()` calls `getSession()` internally, mocking `getSession()` only works for external callers. For admin endpoints that use `requireRole()`, mock `requireRole()` directly.

**Pattern:**
```typescript
// WRONG - mocking internal function doesn't affect requireRole
vi.mock('../../../../src/lib/auth', async (importOriginal) => {
  const actual = await importOriginal();
  return {
    ...actual,
    getSession: vi.fn().mockImplementation(() => Promise.resolve(mockSessionResult)),
  };
});

// CORRECT - mock the boundary function the endpoint actually calls
vi.mock('../../../../src/lib/auth', async (importOriginal) => {
  const actual = await importOriginal();
  return {
    ...actual,
    requireRole: vi.fn().mockImplementation(async (_cookies, _jwtSecret, requiredRoles) => {
      if (!mockSessionResult) {
        throw new Response(null, { status: 302, headers: { Location: '/login' } });
      }
      const hasRole = requiredRoles.some((role) => mockSessionResult.roles.includes(role));
      if (!hasRole) {
        throw new Response('Forbidden', { status: 403 });
      }
      return mockSessionResult;
    }),
  };
});
```

## 2. Test Data Setup Reveals Schema Mismatches
**Topics:** testing, d1

**Context:** Admin API tests failed during `beforeAll` setup with SQLite errors about missing columns or constraint failures.

**Learning:** Test data setup acts as schema validation. When writing INSERT statements in tests, column mismatches surface immediately. This session caught several:

| Table | Test Used | Schema Has |
|-------|-----------|------------|
| categories | `sort_order` | `display_order` |
| enrollments | `created_at, updated_at` | `enrolled_at` only |
| transactions | `user_id, type` | Stripe-specific fields |
| student_teachers | `created_at, updated_at` | `certified_date, approved_by` |

**Pattern:** When test setup fails with constraint errors, check the actual schema in `migrations/0001_schema.sql` rather than assuming the column names.

## 3. Enrollment Status Enum Values
**Topics:** d1, testing

**Context:** DELETE user endpoint checked `status = 'active'` for active enrollments, but this always returned 0 because 'active' isn't a valid status.

**Learning:** The enrollments table uses this status enum:
```sql
status TEXT NOT NULL CHECK (status IN ('enrolled', 'in_progress', 'completed', 'cancelled'))
```

"Active" enrollments (blocking user deletion) means `status IN ('enrolled', 'in_progress')`, not `status = 'active'`.

## 4. Foreign Key Reference Patterns in Test Data
**Topics:** d1, testing

**Context:** Enrollment insert failed with FK constraint error when using `student_teacher_id = 'st-active-1'` (the student_teachers record ID).

**Learning:** The `enrollments.student_teacher_id` references `users(id)`, not `student_teachers(id)`. When an enrollment is assigned to a student-teacher, it stores the user ID of the S-T, not the student_teachers record ID. The student_teachers table is a certification record (user + course), not the identity.

**Pattern:**
```sql
-- WRONG: using student_teachers.id
INSERT INTO enrollments (student_teacher_id) VALUES ('st-active-1')

-- CORRECT: using the user_id of the student-teacher
INSERT INTO enrollments (student_teacher_id) VALUES ('usr-regular-1')
```
