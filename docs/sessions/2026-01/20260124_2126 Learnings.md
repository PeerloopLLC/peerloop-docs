# Session Learnings - 2026-01-24

## 1. D1 Platform-Level Foreign Key Enforcement
**Topics:** d1, cloudflare

**Context:** Attempted to reset a D1 database by dropping all tables. First tried `PRAGMA foreign_keys = OFF` before DROP statements, then tried `PRAGMA defer_foreign_keys = ON`.

**Learning:** Cloudflare D1 enforces `foreign_keys=ON` at the platform level - this cannot be disabled. The `PRAGMA foreign_keys=OFF` directive silently fails. While `PRAGMA defer_foreign_keys=ON` is supported, it only defers constraint checking for data operations (INSERT/UPDATE/DELETE) to the end of the transaction. It does NOT help with DROP TABLE - SQLite still prevents dropping a table that is referenced by other tables' foreign keys.

**Pattern:**
```sql
-- THIS DOES NOT WORK in D1:
PRAGMA foreign_keys = OFF;  -- Silently ignored
DROP TABLE parent_table;    -- Fails if child_table references it

-- THIS ALSO DOESN'T HELP for DROP:
PRAGMA defer_foreign_keys = ON;  -- Only affects data operations
DROP TABLE parent_table;         -- Still fails
```

**Reference:** [Cloudflare D1 Foreign Keys docs](https://developers.cloudflare.com/d1/sql-api/foreign-keys/)

## 2. Local D1 Reset via SQLite File Deletion
**Topics:** d1, cloudflare, workflow

**Context:** Needed a reliable way to reset local D1 database after consolidating migrations.

**Learning:** For local D1 development, the simplest and most reliable reset approach is to delete the SQLite files directly. Wrangler stores local D1 data at `.wrangler/state/v3/d1/miniflare-D1DatabaseObject/`. Deleting the `*.sqlite`, `*.sqlite-shm`, and `*.sqlite-wal` files causes wrangler to create a fresh database on next access.

**Pattern:**
```bash
# Reset local D1 by deleting SQLite files
rm -f .wrangler/state/v3/d1/miniflare-D1DatabaseObject/*.sqlite*

# Then re-apply migrations
npm run db:migrate:local
```

## 3. Schema-Driven FK Dependency Resolution
**Topics:** d1, workflow

**Context:** Remote D1 doesn't allow file deletion - need to drop tables via SQL. But D1's immutable FK enforcement means tables must be dropped in dependency order.

**Learning:** Parse the schema file (`0001_schema.sql`) to extract FK relationships, build a dependency graph using topological sort, then drop tables in reverse order (children before parents). This makes the schema file the single source of truth for both table creation AND destruction order.

**Pattern:**
```javascript
// Extract FK dependencies from CREATE TABLE statements
const tableRegex = /CREATE TABLE.*?"?(\w+)"?\s*\(([\s\S]*?)\);/gi;
const refRegex = /REFERENCES\s+"?(\w+)"?/gi;

// Build dependency map: table -> [tables it references]
// Topological sort: visit referenced tables first
// Reverse the sorted order for DROP (children before parents)
```

The script parses the schema once and works for any number of tables with any FK structure. Self-maintains as schema evolves.
