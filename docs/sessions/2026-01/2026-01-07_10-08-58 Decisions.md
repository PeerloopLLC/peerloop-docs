# Session Decisions - 2026-01-07

## Decision 1: Apply Spec/Content Toggle to All Layouts

**Type**: Implementation
**Driver**: User-driven
**Working On**: Page verification - SPAY page showing PageSpecView instead of real content
**Trigger**: SPAY page had real StripeConnectSettings component but it wasn't rendering because AppLayout lacked the toggle mechanism

**Options Considered**:
1. Replace PageSpecView with real content directly (one-off fix)
2. Add toggle pattern to AppLayout matching LandingLayout <- **Chosen**
3. Create a shared layout component for the toggle

**Decision**: Apply the same spec/content toggle pattern from LandingLayout to both AppLayout and AdminLayout, with backward compatibility.

**Rationale**:
- Consistent pattern across all layouts
- Pages can opt-in with `pageCode` prop
- Existing pages without pageCode continue to work unchanged
- Enables development workflow where spec and production views coexist

**Consequences**:
- AppLayout updated with `.spec-content`, `.real-content` wrappers and SpecPanelToggle
- AdminLayout updated with same pattern
- SPAY page updated to pass `pageCode` prop
- 22 AppLayout pages and 8 AdminLayout pages can now use the toggle

## Decision 3: Moderation Links in Admin Sidebar

**Type**: Strategic
**Driver**: User-driven
**Working On**: Admin sidebar audit - evaluating moderation section
**Trigger**: Found `/admin/moderation` and `/admin/moderators` links pointing to non-existent pages

**Options Considered**:
1. Remove both moderation links from Admin sidebar
2. Keep `/admin/moderators` (for admin to manage mods), remove `/mod` link
3. Keep both links (admin can access mod queue to supervise + manage moderators) <- **Chosen**

**Decision**: Keep both moderation-related links in the Admin sidebar. Update the "Moderation Queue" link to point to `/mod` (where the actual page exists).

**Rationale**:
- Admins should be able to supervise moderation activity (access mod queue)
- Admins need to manage moderators (invite, remove role)
- Both functions are legitimate admin concerns

**Consequences**:
- "Moderation Queue" link now points to `/mod` (working page)
- "Moderators" link points to `/admin/moderators` (will 404 until page built)
- Block 8.10 added to PLAN.md to track this work

## Decision 4: Document Moderator System Gap in PLAN.md

**Type**: Strategic
**Driver**: User-driven
**Working On**: Understanding the Moderator vs Admin role separation
**Trigger**: Confusion about whether moderation links belong in Admin menu

**Options Considered**:
1. Create the Moderator pages immediately
2. Document the gap in PLAN.md for future work <- **Chosen**
3. Remove Moderator-related items from scope

**Decision**: Add Block 8.10 "Moderator System" to PLAN.md documenting:
- Moderator Interface (AppLayout) - pages moderators use to do their work
- Admin Moderator Management (AdminLayout) - admin pages to manage moderators
- Open questions about navigation patterns and escalation workflow

**Rationale**:
- The gap is real and needs tracking
- Design decisions needed before implementation
- Follows existing pattern (Student-Teacher has their own dashboard section)
- Clear separation of concerns between roles

**Consequences**:
- Block 8.10 added to PLAN.md with detailed breakdown
- Page codes proposed: MDSH, MHST, MSET (Moderator), AMOD (Admin)
- User stories US-A034 through US-A039 referenced
- Work deferred pending design decisions
