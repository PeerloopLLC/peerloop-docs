# Session Decisions - 2026-01-22

## 1. Build LEAD with user_stats Instead of Waiting for Goodwill System
**Type:** Implementation
**Topics:** architecture, d1

**Trigger:** LEAD spec called for goodwill_points leaderboard, but CD-023 (goodwill system) isn't implemented. User asked "What if anything would be kept in a full goodwill/leaderboard system?"

**Options Considered:**
1. Wait for CD-023 goodwill implementation before building LEAD
2. Create throwaway goodwill schema just for leaderboard ← Rejected
3. Build LEAD using existing user_stats data, extend later ← Chosen

**Decision:** Build LEAD using user_stats (students_taught, courses_completed, average_rating) now. When CD-023 is implemented, add "Goodwill Points" as a new category.

**Rationale:**
- 80-90% of UI/component work is reusable regardless of data source
- user_stats data already exists and provides meaningful rankings
- Avoids creating throwaway schema that conflicts with proper CD-023 design
- Only ~20 lines of SQL queries would need replacing later

**Consequences:**
- LEAD implemented with 3 categories: Top Teachers, Top Learners, Highest Rated
- Clear path to extend when goodwill system exists
- No schema debt created

## 2. Remove PROF from MSGS Incoming Connections
**Type:** Implementation
**Topics:** messaging, architecture

**Trigger:** Analyzing which pages should have Message buttons leading to MSGS.

**Options Considered:**
1. Keep PROF → MSGS (message any user from their profile) ← Rejected
2. Remove PROF, rely on STPR/CPRO for profile messaging ← Chosen

**Decision:** Remove PROF from MSGS connections. STPR (ST Profile) and CPRO (Creator Profile) already provide Message buttons for the relevant user types.

**Rationale:**
- PROF (`/profile`) is typically the user's own profile, not others' profiles
- STPR and CPRO already handle public-facing profiles that need messaging
- Avoids confusing UX of "message yourself"

**Consequences:**
- MSGS spec updated to remove PROF from incoming connections
- Clearer mental model: message people from their public profiles (STPR/CPRO)

## 3. Move "Message Student" from TDSH to CMST
**Type:** Implementation
**Topics:** messaging, ux

**Trigger:** Determining where ST should be able to message their students.

**Options Considered:**
1. Add Message action to TDSH (Teaching Dashboard) ← Rejected
2. Add Message action to CMST (My Students) page ← Chosen

**Decision:** Message button belongs on CMST where the full student list lives, not on the dashboard overview.

**Rationale:**
- CMST shows full student details with context (course, progress, sessions)
- Dashboard is overview, not action-heavy
- Natural workflow: go to students page → find student → message them

**Consequences:**
- Message icon added to each student row in MyStudents.tsx
- CMST spec updated with MSGS connection
- MSGS spec updated: TDSH → CMST for student messaging
