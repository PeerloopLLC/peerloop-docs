# Session Learnings - 2026-01-24

## 1. Astro API Routes Require Database at `locals.runtime.env.DB`
**Topics:** astro, d1, testing

**Context:** Writing API tests for auth endpoints, we needed to mock the Astro APIContext with a test database.

**Learning:** The `getDB(locals)` function looks for the database at `locals.runtime.env.DB`, not `locals.db`. This is because in production, Cloudflare bindings are accessed via `locals.runtime.env`. Tests must place the mock database at this path for API routes to find it.

**Pattern:**
```typescript
const locals = {
  runtime: {
    env: {
      DB: getTestDB(),
      JWT_SECRET: 'test-secret',
    },
  },
};
```

## 2. Jose JWT Library Requires Special Handling in Node.js Tests
**Topics:** auth, testing

**Context:** API tests for register/login endpoints were failing with `TypeError: payload must be an instance of Uint8Array` when trying to create JWT tokens.

**Learning:** The `jose` library's JWT signing works differently in Node.js test environment vs browser/Cloudflare Workers. The solution is to mock `setAuthCookies` in tests rather than trying to make real JWT signing work. This is appropriate since we're testing API logic, not the JWT library.

**Pattern:**
```typescript
vi.mock('../../../src/lib/auth', async (importOriginal) => {
  const actual = await importOriginal<typeof import('../../../src/lib/auth')>();
  return {
    ...actual,
    setAuthCookies: vi.fn().mockResolvedValue(undefined),
  };
});
```

## 3. Always Verify Database Schema Before Writing Test INSERT Statements
**Topics:** d1, testing

**Context:** Course endpoint tests failed with various `NOT NULL constraint` and `no column named` errors.

**Learning:** The actual database schema often differs from assumptions based on documentation or API responses. Before writing INSERT statements in tests, always check `migrations/*.sql` for exact column names and NOT NULL constraints. Key differences discovered:
- `categories` table has no `description`, `icon`, or `is_active` columns
- `course_includes` uses `item` not `include_item`
- `course_prerequisites` requires `type` and `content` not just `prerequisite`
- `course_curriculum` requires `module_order` (NOT NULL)

## 4. Vitest Uses Different CLI Syntax Than Jest
**Topics:** testing, workflow

**Context:** The npm scripts `test:unit`, `test:api`, `test:integration` were using `--testPathPattern` which is Jest syntax.

**Learning:** Vitest doesn't support `--testPathPattern`. Instead, pass the path directly as an argument: `vitest run tests/api` instead of `vitest run --testPathPattern='tests/api'`.

**Pattern:**
```json
{
  "test:unit": "vitest run tests/unit",
  "test:api": "vitest run tests/api",
  "test:integration": "vitest run tests/integration"
}
```

## 5. API Test Helper Pattern for Astro Routes
**Topics:** testing, astro

**Context:** Needed a reusable way to test Astro API routes with proper context simulation.

**Learning:** A well-designed API test helper should provide: (1) `createAPIContext()` that builds the full Astro context with `locals`, `cookies`, and environment, (2) `createMockRequest()` for building Request objects with JSON bodies, (3) `createMockCookies()` that tracks set/deleted cookies for assertions, and (4) test data helpers like `uniqueEmail()`, `validRegistrationData()`.

**Pattern:** See `tests/api/helpers/api-test-helper.ts` for the full implementation that can be reused across all API endpoint tests.
