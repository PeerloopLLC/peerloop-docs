# Session Learnings - 2026-01-29

## 1. Astro Islands for Dashboard Layout with React Interactivity
**Topics:** astro, workflow

**Context:** Implementing a new persistent sidebar dashboard layout that needs to be SEO-friendly (SSR) while supporting interactive features like slide-out panels and session-aware menus.

**Learning:** The Astro Islands architecture works well for dashboard layouts. The static shell (DashLayout.astro) handles SSR for the HTML structure, while React components with `client:load` handle all interactive elements (navbar, panels, user menu). Session fetching happens inside the React component via API calls, not in Astro frontmatter.

**Pattern:**
```astro
// DashLayout.astro - Static shell
<DashNavbar client:load currentPath={Astro.url.pathname} />
<main class="lg:ml-64">
  <slot />
</main>
```

```tsx
// DashNavbar.tsx - React island
useEffect(() => {
  fetch('/api/auth/session').then(...)
}, []);
```

## 2. Role-Based Menu Filtering with Array Roles
**Topics:** auth, astro

**Context:** The dashboard navbar needs to show different menu items based on user roles (visitor, student, student_teacher, creator, admin).

**Learning:** Define menu items with optional `roles` array. Filter items based on user state at render time. Items without `roles` are shown to everyone. Use a `getVisibleItems` callback that checks role flags.

**Pattern:**
```tsx
const menuItems: MenuItem[] = [
  { id: 'discover', label: 'Discover', href: '/dash/discover' }, // No roles = public
  { id: 'workspace', label: 'Workspace', href: '/dash/workspace',
    roles: ['student_teacher', 'creator'] }, // Role-restricted
];

const getVisibleItems = useCallback(() => {
  return menuItems.filter((item) => {
    if (!item.roles) return true;
    if (!isAuthenticated) return false;
    return item.roles.some((role) => userHasRole(role));
  });
}, [isAuthenticated, ...roleFlags]);
```

## 3. Slide-Out Panel Positioning with Transform Transitions
**Topics:** astro, workflow

**Context:** Need two different slide-out panels - one full-height for feed selection, one smaller for "More" options.

**Learning:** For panels that slide out from behind a fixed sidebar, use `position: fixed` with `left` offset at sidebar width, then apply `transform: translateX(-100%)` when closed. The transform creates smooth animation. Different panel sizes can use different positioning strategies (fixed full-height vs fixed-bottom popup).

**Pattern:**
```tsx
// Full-height panel sliding from behind sidebar
<div className={`fixed left-64 top-0 z-50 h-screen w-80 transform transition-transform
  ${isOpen ? 'translate-x-0' : '-translate-x-full'}`}
  style={{ marginLeft: '-64px' }}  // Slides from behind navbar
>

// Smaller popup panel
<div className={`fixed bottom-4 left-64 z-50 w-64 transform transition-all
  ${isOpen ? 'translate-x-4 opacity-100' : '-translate-x-full opacity-0'}`}
>
```

## 4. Tailwind v4 Class Migration: outline-none → outline-hidden
**Topics:** workflow, testing

**Context:** Running `/q-codecheck` flagged `outline-none` usage in newly created components.

**Learning:** In Tailwind v4, `outline-none` behavior changed. Use `outline-hidden` when you want to visually hide the outline but maintain accessibility. The check script catches this pattern and prompts for updates.
