# Session Decisions - 2026-01-27

## 1. Auto-Generate PAGES-MAP.md from Page JSONs
**Type:** Implementation
**Topics:** workflow

**Trigger:** Session 130 upgraded PageSpec schema to v1.3 with file lookup metadata in JSON files. PAGES-MAP.md had redundant columns (Status, Block) now superseded by JSON metadata.

**Options Considered:**
1. Keep hand-maintained PAGES-MAP.md with simplified columns
2. Auto-generate PAGES-MAP.md from page JSONs via script <- Chosen
3. Delete PAGES-MAP.md entirely (use JSONs directly)

**Decision:** Created `scripts/generate-pages-map.ts` that reads all 66 page JSONs and generates a two-lookup index (by code and by route). Run via `npm run pages-map:write`.

**Rationale:** Auto-generation ensures the index stays in sync with the JSON source of truth. The script reuses classification logic (sections, on-hold codes) that would otherwise need manual maintenance. Option 3 was rejected because a flat lookup file is faster to scan than opening individual JSONs.

**Consequences:** PAGES-MAP.md is now generated, not edited. Removed Status, Block, and "Implemented Pages by Block" sections. Added npm scripts `pages-map` and `pages-map:write`.

## 2. Mermaid Flowchart for Site Interconnection Visualization
**Type:** Implementation
**Topics:** workflow

**Trigger:** Need to visualize how pages interconnect before routing changes. Considered SEO sitemap.xml but user needed a developer-facing interconnection view, not crawler-facing.

**Options Considered:**
1. Markdown report (text tables)
2. HTML network graph visualization
3. Terminal output (disposable)
4. Mermaid diagram in markdown <- Chosen

**Decision:** Mermaid flowchart embedded in SITE-MAP.md, auto-generated from page JSON `connections.outgoing` data.

**Rationale:** Mermaid renders in VS Code preview and GitHub without external tools. Git-trackable. Balances visual clarity with maintainability. The page JSONs already have rich connection data (target codes, trigger descriptions) making generation straightforward.

**Consequences:** Created `scripts/generate-site-map.ts` with `npm run site-map:write`.

## 3. Focus-Area Diagrams with Outbound-Only Edges
**Type:** Implementation
**Topics:** workflow

**Trigger:** Full 60-page Mermaid diagram with 202 edges was too dense to be useful.

**Options Considered:**
1. Single large diagram (status quo) — too dense
2. Focus areas with all touching edges — boundary set too large (26 nodes)
3. Focus areas with outbound-only edges <- Chosen

**Decision:** Added `--focus <area>` flag (public, student, teacher, creator, admin). Each focus shows only edges originating FROM primary pages, keeping boundary nodes to 6-11 per diagram.

**Rationale:** Outbound edges answer "where can you navigate from this area?" which is the useful question for understanding a role's journey. Inbound edges ("what links to this area?") pull in nearly every other page and add noise. The boundary nodes (dashed style) still show exit points from the focus area.

**Consequences:** Without `--focus`, generates all 5 diagrams in one file. Each diagram is 12-24 primary + 6-11 boundary pages — readable and focused.
