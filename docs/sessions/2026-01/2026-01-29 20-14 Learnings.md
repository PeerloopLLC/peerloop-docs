# Session Learnings - 2026-01-29 (Continued)

## 1. Astro Page Loads Reset Window Globals
**Topics:** astro, workflow

**Context:** Discussing how to share auth state across APP, ADMIN, and PUBLIC page areas.

**Learning:** In Astro (non-SPA), each page navigation is a full page load. The `window.__peerloop` globals are wiped on every navigation. This means:
- Globals only exist for a single page's lifetime
- Cross-page "memory" comes from localStorage cache and cookies
- Each page's main component (DashNavbar, AdminNavbar) re-initializes the globals

**Pattern:**
```
Page Load → Globals reset → Navbar mounts → initializeCurrentUser() →
  1. Read localStorage (instant)
  2. Fetch API (verify)
  3. Set globals
→ Other islands read from globals
```

## 2. Separation of Concerns: User Data vs Network Status
**Topics:** auth, workflow

**Context:** Initially mixed network status into the currentUser return value, which felt hacky.

**Learning:** Separating concerns into two globals is cleaner:
- `currentUser` - Pure user data (who is this person, what can they do)
- `networkState` - Network/auth status (did the API call succeed, what went wrong)

This allows:
- User data not polluted with error strings
- networkState extensible for future needs (apiHealth, isOnline)
- Clear mental model for component authors

## 3. Session Cookie vs Window Globals
**Topics:** auth, astro

**Context:** Worried about "losing login state" when navigating between page areas.

**Learning:** "Logged in" state is NOT stored in window globals. It's stored in:
1. HTTP-only session cookie (server knows you're authenticated)
2. localStorage cache (instant hydration of user data)
3. localStorage flag (was_logged_in - distinguishes visitor vs expired)

Window globals are just an in-memory convenience for React islands on the SAME page to share state. They're rebuilt from the persistence layer on each page load.

## 4. Consulting Before Pervasive Changes
**Topics:** workflow

**Context:** Made significant architecture changes without discussing design options first.

**Learning:** For pervasive changes that affect multiple files or establish new patterns, discuss the design approach with the user BEFORE implementing. Present options, trade-offs, and get alignment. This prevents rework and ensures the solution matches the user's mental model.
