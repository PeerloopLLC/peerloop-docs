# Session Decisions - 2026-01-06

## Decision 1: Consolidate Page Documentation into Single Structure

**Type**: Strategic
**Driver**: Claude-driven (with user approval)
**Working On**: Block 8 - Admin Tools / PAGES-MAP refactoring
**Trigger**: PAGES-MAP.md had grown to 820+ lines; page specs existed in two separate locations (`research/run-001/pages/` for planning, `docs/pages/` for implementation)

**Options Considered**:
1. Keep separate files - Planning specs in `research/run-001/pages/`, implementation status in `PAGES-MAP.md`
2. **Merge into `docs/pages/CODE.md`** - Single source of truth per page combining plan + implementation <- **Chosen**
3. Keep everything in PAGES-MAP.md - One massive file with all details

**Decision**: Merge planning specs from `research/run-001/pages/page-*.md` into implementation docs at `docs/pages/CODE.md`, creating a single authoritative file per page.

**Rationale**:
- **Single source of truth**: One file per page contains everything (purpose, user stories, connections, features, API endpoints, implementation status)
- **Reduced duplication**: No need to maintain parallel files that could drift
- **Better traceability**: User story tags `[US-XXXX]` on feature items link requirements to implementation
- **Cleaner PAGES-MAP.md**: Slim lookup table (~140 lines) with pointers to detail files
- **Easier maintenance**: Update one file when either planning or implementation changes

**Consequences**:
- Created 40 new `docs/pages/CODE.md` files for unimplemented pages
- PAGES-MAP.md reduced from 820 to ~140 lines (lookup table only)
- Original PAGES-MAP.md preserved at `.claude/PAGES-MAP-original.md`
- Merged 9 pages (Admin + Creator) with full specs this session
- 29 pages remain to be merged in future sessions
- `research/run-001/pages/` files become reference-only (may be archived later)

**Format Established** (from ACRS.md/ACAT.md templates):
1. Metadata table (Route, Access, Priority, Status, Block, files)
2. Purpose
3. User Stories (table with IDs, story, priority, status)
4. Connections (Incoming/Outgoing navigation)
5. Data Requirements (entities and fields)
6. Features (grouped, with `[US-XXXX]` tags)
7. Sections (UI details from plan)
8. API Endpoints (with request/response examples)
9. States & Variations
10. Error Handling
11. Implementation Notes
