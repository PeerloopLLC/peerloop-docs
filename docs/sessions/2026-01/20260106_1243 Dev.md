# Development Session Log - 2026-01-06

## Session Focus
Implementing D1 REST API fallback for MBA-2017 development environment.

---

## Chronological Log

### Initial Problem Report
**User:** Reported login returning 500 error on MBA-2017 with dev server running on port 4321.

**Action:** Investigated the login endpoint at `src/pages/api/auth/login.ts`. Found it was returning "Database not available" because `locals.runtime?.env?.DB` is undefined when using the Node adapter on MBA-2017.

---

### Root Cause Analysis
**User:** Asked to check tech docs for existing solutions.

**Action:** Read `docs/tech/tech-013-devcomputers.md`. Confirmed MBA-2017 limitations:
- macOS 12.x cannot run workerd (requires 13.5+)
- Local D1 not supported
- APIs return 503 when D1 unavailable (by design)

---

### Exploring Solutions
**User:** Asked if there's another option - API access to remote D1.

**Action:** Identified Cloudflare's D1 REST API as a viable solution. Created `src/lib/db/d1-rest.ts` implementing a D1Database-compatible interface using REST API calls.

---

### Initial Implementation
**Action:**
1. Created D1 REST client that mimics D1Database interface
2. Updated `getDB()` in `src/lib/db/index.ts` to fall back to REST client when binding unavailable
3. Attempted to load `.dev.vars` manually (initially failed)

---

### Environment Variable Loading Issues
**User:** Asked about dotenv utility after several failed attempts.

**Action:** Installed `dotenv` package and configured it to load `.dev.vars` at module import time. This populated `process.env` with credentials.

---

### API Token Permission Error
**Problem:** 403 error - "The given account is not valid or is not authorized"

**User:** Confirmed they had created API token using "Edit Cloudflare Workers" template.

**Action:** Explained that template doesn't include D1 permissions. Guided user to add `Account > D1 > Edit` permission to their token.

---

### Database ID Mismatch
**Problem:** 404 error - database ID not found.

**User:** Provided correct database ID from Cloudflare dashboard: `21c0179a-9e7d-4cc1-91d5-ea402717fce8`

**Action:** Updated database ID in both `wrangler.toml` and `src/lib/db/d1-rest.ts`.

---

### Password Verification
**Problem:** "Invalid email or password" after D1 connection working.

**Action:** Added debug logging. Discovered user exists with proper bcrypt hash, but `dev123` wasn't the correct password.

**User:** Reset all users in remote D1 to use `dev123` password.

---

### JWT Secret Issue
**Problem:** "JWT secret not configured" - code was only checking Cloudflare binding.

**User:** Asked "Why?" when proposed adding process.env fallback.

**Action:** Explained that `locals.runtime.env` doesn't exist with Node adapter. The JWT_SECRET is in `.dev.vars` but code wasn't checking `process.env`.

---

### Refactoring for Explicit Opt-In
**User:** Raised concern about fallback code running in production. Wanted deliberate, surgical approach.

**Action:** Agreed and proposed explicit flag approach:
1. Added `ENABLE_D1_REST_FALLBACK=true` to `.dev.vars`
2. Created `isRestFallbackEnabled()` check requiring both flag AND not-production
3. Created `src/lib/env.ts` with `getEnv()` helper for environment variables
4. Updated `getDB()` to only use REST fallback when explicitly enabled

---

### Successful Login Test
**Action:** Tested login after refactoring. Received successful response with user data.
