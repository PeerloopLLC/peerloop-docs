## testing: Stripe Constructor Mock Pattern

**Context:** Testing payout endpoints that use `new Stripe(apiKey)` for transfers

**Learning:** When mocking a module that uses constructor syntax (`new Stripe()`), you must use class syntax in the mock, not a function:

```typescript
// WRONG - "is not a constructor" error
vi.mock('stripe', () => ({
  default: vi.fn().mockReturnValue({
    transfers: { create: mockFn }
  })
}));

// CORRECT - class syntax works with `new`
vi.mock('stripe', () => ({
  default: class MockStripe {
    constructor() {}
    transfers = { create: mockStripeTransfersCreate };
  },
}));
```

**Why:** Vitest/Jest mocks need to be callable with `new`. A function mock returns a value; a class mock creates an instance with the defined properties.

**See:** `tests/api/admin/payouts/actions.test.ts`

## d1: Sessions Table Column Names

**Context:** Writing tests for student-teacher endpoint that queries sessions

**Learning:** The sessions table uses `scheduled_start` and `scheduled_end` columns, NOT `scheduled_at`. This is consistent with how sessions have both a start and end time.

**Schema columns:**
- `scheduled_start` - When session begins
- `scheduled_end` - When session ends

**Endpoint alias:** The `[id].ts` endpoint aliases `scheduled_start as scheduled_at` for backward compatibility in the response, but the DB column is `scheduled_start`.

**Fixed:** `src/pages/api/admin/student-teachers/[id].ts:142`

## testing: Auth Endpoints May Throw Responses

**Context:** Testing moderator invite endpoint auth behavior

**Learning:** Some Astro API endpoints throw `Response` objects instead of returning them for authentication/authorization failures. Tests must use try/catch:

```typescript
// WRONG - expects return value
const response = await POST({ ...ctx, request });
expect(response.status).toBe(302);

// CORRECT - handle thrown Response
try {
  await POST({ ...ctx, request });
  expect.fail('Expected thrown Response');
} catch (response) {
  expect(response).toBeInstanceOf(Response);
  expect((response as Response).status).toBe(302);
}
```

**Pattern:** The `requireRole()` helper throws a Response on auth failure. Endpoints that call it directly (not wrapped in try/catch) propagate the throw.

**See:** `tests/api/admin/moderators/invite.test.ts`

## d1: Transactions Table Schema

**Context:** Setting up test data for payout detail tests

**Learning:** The `transactions` table does NOT have a `user_id` column. It links to users indirectly through `enrollment_id`:

```
transactions → enrollment_id → enrollments → user_id → users
```

**Correct INSERT:**
```sql
INSERT INTO transactions (id, enrollment_id, amount, status, ...)
```

**See:** `tests/api/admin/payouts/detail.test.ts`

## testing: Stripe Environment Variable Required

**Context:** Payout action tests failing with undefined API key

**Learning:** Even with mocked Stripe, endpoints may still check for `STRIPE_SECRET_KEY` before instantiating. Always provide it in test context:

```typescript
const ctx = createAPIContext({
  db,
  env: { STRIPE_SECRET_KEY: 'sk_test_mock' }
});
```

**See:** `tests/api/admin/payouts/actions.test.ts`
