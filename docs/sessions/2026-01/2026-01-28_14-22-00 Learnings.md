## 1. Astro APIContext Has ~20 Required Properties
**Topics:** astro, testing

**Context:** Tests were failing TypeScript strict mode because our `createAPIContext()` mock only returned 9 properties, but Astro's `APIContext` type requires ~20+.

**Learning:** When mocking Astro's `APIContext`, you need to include all these properties: `request`, `url`, `params`, `props`, `cookies`, `locals`, `redirect`, `rewrite`, `site`, `generator`, `clientAddress`, `originPathname`, `isPrerendered`, `routePattern`, `preferredLocale`, `preferredLocaleList`, `currentLocale`, `session`, `getActionResult`, `callAction`, and `csp`.

**Pattern:**
```typescript
// Full APIContext mock with all required properties
const context: TestAPIContext = {
  cookies, locals: locals as App.Locals, url, params: {}, props: {},
  request: new Request(url),
  site: undefined, generator: 'Astro v5.x (Test)',
  clientAddress: '127.0.0.1', originPathname: url.pathname,
  isPrerendered: false, routePattern: '/api/test',
  preferredLocale: undefined, preferredLocaleList: undefined, currentLocale: undefined,
  session: undefined,
  getActionResult: () => undefined,
  callAction: async () => { throw new Error('Not implemented'); },
  redirect: (path, status = 302) => new Response(null, { status, headers: { Location: path } }),
  rewrite: async () => { throw new Error('Not implemented'); },
  csp: { insertDirective: () => {}, insertStyleResource: () => {}, insertStyleHash: () => {}, insertScriptResource: () => {}, insertScriptHash: () => {} },
};
```

## 2. TypeScript Errors Cascade from Incomplete Mock Types
**Topics:** testing, workflow

**Context:** We had 4,581 TypeScript errors in test files, but fixing the APIContext mock alone eliminated 2,114 errors (46%).

**Learning:** TypeScript errors cascade - when a foundational type is incomplete, every place that spreads, casts, or uses that type generates additional errors. Fixing the root cause fixes many downstream errors automatically.

**Pattern:** Always check foundational test helpers first when facing large TypeScript error counts.

## 3. Response.json() Returns Unknown in Strict Mode
**Topics:** testing, astro

**Context:** The native `Response.json()` method returns `Promise<unknown>` in TypeScript strict mode, causing thousands of TS18046 errors when accessing properties on the result.

**Learning:** In test files, the simplest fix is to cast the result: `await response.json() as any`. While `any` isn't ideal for production code, it's acceptable for tests where the goal is verifying behavior, not enforcing types on mock data.

**Pattern:**
```typescript
// Before: TS18046 error
const body = await response.json();
expect(body.error).toBe('...');  // Error: 'body' is of type 'unknown'

// After: Works
const body = await response.json() as any;
expect(body.error).toBe('...');
```

## 4. D1 first() Returns Empty Object Type
**Topics:** d1, testing

**Context:** When using D1's `.first()` method in tests, the return type is `Promise<{} | null>`, causing TS2339 errors when accessing properties.

**Learning:** Cast `.first()` results to `any` in tests: `.first() as any`

**Pattern:**
```typescript
// Before: TS2339 error
const user = await db.prepare('SELECT * FROM users WHERE id = ?').bind(id).first();
expect(user?.name).toBe('Test');  // Error: Property 'name' does not exist on type '{}'

// After: Works
const user = await db.prepare('SELECT * FROM users WHERE id = ?').bind(id).first() as any;
expect(user?.name).toBe('Test');
```

## 5. HTMLElement Needs Double Cast for Specific Element Types
**Topics:** testing

**Context:** Testing Library's `getByRole` returns `HTMLElement`, but we sometimes need specific element types like `HTMLButtonElement` or `HTMLSelectElement`.

**Learning:** When direct casting fails with "neither type sufficiently overlaps", use double cast through `unknown`: `as unknown as HTMLSelectElement`

**Pattern:**
```typescript
// Direct cast fails
const select = screen.getByLabelText(/what can we help/i) as HTMLSelectElement;  // Error

// Double cast works
const select = screen.getByLabelText(/what can we help/i) as unknown as HTMLSelectElement;
```

## 6. Vitest Mock lastCall May Be Undefined
**Topics:** testing

**Context:** When asserting on mock function calls, `mockFetch.mock.lastCall[0]` throws TS18048 because `lastCall` may be undefined.

**Learning:** Use non-null assertion when you've verified the mock was called: `mockFetch.mock.lastCall![0]`

**Pattern:**
```typescript
// Before: TS18048 error
expect(mockFetch.mock.lastCall[0]).toContain('/api/');

// After: Works (use when you know mock was called)
expect(mockFetch.mock.lastCall![0]).toContain('/api/');
```
