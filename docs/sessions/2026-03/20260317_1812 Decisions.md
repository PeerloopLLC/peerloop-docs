# Conv Decisions - 2026-03-17

## 1. Session Invite Model for Instant Booking
**Type:** Strategic
**Topics:** d1, astro

**Trigger:** Client requested "Book Now" capability — students and teachers starting a session immediately rather than picking a future time slot.

**Options Considered:**
1. "Book Now" as a variant of existing booking flow (student unilaterally books now+5min)
2. Mutual "Start Now" with real-time acceptance (WebSocket/polling)
3. Session Invite model — teacher sends short-lived invite via notifications ← Chosen

**Decision:** Adopted the Session Invite model. Teacher sends a 30-minute-expiry invite scoped to an enrollment, delivered via the existing notification system. Student accepts with one click, session created at now+5min.

**Rationale:** Option 1 lacks teacher authorization (client requirement). Option 2 requires real-time signaling infrastructure we don't have. Option 3 reuses the existing notification system, requires no new real-time infrastructure, and naturally enforces teacher-initiated authorization.

**Consequences:** New `session_invites` table, 3 new API endpoints, notification helpers, teacher UI button, student booking page invite flow. RFC CD-037 created.

## 2. Two Entry Points: New Booking and Reschedule
**Type:** Implementation
**Topics:** d1

**Trigger:** Need to handle both "student has unbooled modules" and "all modules already booked" cases without allowing overbooking.

**Options Considered:**
1. Only allow "Book Now" when modules are available (block when all booked)
2. Allow both new booking and reschedule via the same invite flow ← Chosen

**Decision:** The accept endpoint checks `getBookingEligibility()`. If `can_book=true`, creates a new session. If `can_book=false` but scheduled sessions exist, auto-cancels the next upcoming session and creates a new one (reschedule). No user picker needed — always reschedule the chronologically next session.

**Rationale:** Auto-selecting the next upcoming session preserves sequential learning order and avoids a confusing picker UI. The session count invariant is maintained because reschedule cancels before creating.

**Consequences:** Accept endpoint has two code paths. Reschedule must cancel BEFORE conflict checks to avoid false positives.

## 3. Lazy Expiry Over Background Jobs
**Type:** Implementation
**Topics:** d1, cloudflare

**Trigger:** Session invites expire after 30 minutes. Need a mechanism to enforce expiry.

**Options Considered:**
1. Background cron job to expire stale invites
2. Lazy expiry — check on access, update status inline ← Chosen

**Decision:** No background job. Expiry is checked at two points: (a) GET endpoint lazily updates stale pending invites to 'expired', (b) accept endpoint checks `expires_at` and returns 410 if expired.

**Rationale:** Cloudflare Workers don't have persistent background processes. A cron trigger would add complexity for minimal benefit — stale invites are harmless until someone tries to use them.

**Consequences:** An expired invite may briefly appear as "pending" in the DB until the next read. The notification action button still works — it just shows an expiry message when clicked.

## 4. One Pending Invite Per Enrollment
**Type:** Implementation
**Topics:** d1

**Trigger:** Should teachers be able to send multiple concurrent invites to the same student?

**Decision:** No — reject with 409 if a pending invite already exists for the enrollment. Teacher must wait for expiry/decline before sending another.

**Rationale:** Prevents notification spam. One invite is sufficient — if the student doesn't respond within 30 minutes, the teacher can send a fresh one.

## 5. Unmocked Notification Tests as Option A, E2E as Deferred
**Type:** Strategic
**Topics:** testing

**Trigger:** After manual testing confirmed the feature works, discussed how to capture this as automated tests.

**Options Considered:**
1. Unmocked integration test verifying notification DB rows ← Chosen (now)
2. Full Playwright E2E two-browser test ← Chosen (deferred to E2E-GAPS block)

**Decision:** Write unmocked notification integration tests now (3 tests verifying real notification rows). Defer Playwright E2E tests to new `E2E-GAPS` block in PLAN.md.

**Rationale:** Option A catches the highest-value gap (notification content correctness) with minimal effort. E2E tests require Playwright infrastructure setup and are better batched with other multi-user flows.

## 6. Every PLAN Block Must Include a Cleanup Phase
**Type:** Strategic
**Topics:** workflow

**Trigger:** After completing SESSION-INVITE and verifying tests, discovered RFC checklist items still unchecked and CURRENT-BLOCK-PLAN.md still populated with completed block data. Required a separate cleanup pass.

**Decision:** Every PLAN block and multi-step endeavour must include an explicit Cleanup phase as the final step, run immediately after verification passes.

**Rationale:** Cleanup is easiest while context is fresh. Deferring it creates ambiguity in future convs — unchecked RFC items look like incomplete work, stale CURRENT-BLOCK-PLAN.md confuses the resume flow.

**Consequences:** Added to memory as a feedback rule. Future PLAN blocks will include a Cleanup section covering: check off RFC items, reset CURRENT-BLOCK-PLAN.md, remove temporary scaffolding, verify no stale references.
