# Conv Learnings - 2026-03-19

## 1. MyFeeds renders null when currentUser is null — protects unmocked tests
**Topics:** testing, astro

**Context:** Adding `<MyFeeds />` to Teacher/Creator dashboards caused no test failures despite those tests not mocking `useCurrentUser()`.

**Learning:** When a component guards with `if (!currentUser) return null`, it silently no-ops in tests where `useCurrentUser()` returns `null` (the default when unmocked). Only tests that explicitly mock `useCurrentUser()` with a partial object (like StudentDashboard) break — because the mock object is missing `getFeeds()`. The fix is adding the method to the mock, not adding mocks to every test file.

## 2. Stream.io flat feeds have no unread/unseen tracking
**Topics:** stream

**Context:** Needed badge counts ("3 new posts") for the /feeds hub page. Expected Stream to provide this.

**Learning:** Stream.io flat feeds (community, course, townhall) are a chronological append log — they store posts and support reactions/comments, but have zero server-side support for: unread/unseen counts per user, activity count since a timestamp, cross-feed aggregation, or date-range filtering on ranked feeds. Only notification feeds support unseen/unread. Any "new since last visit" indicator must be built externally.

## 3. CQRS pattern: Stream as write model, D1 as read model
**Topics:** stream, d1

**Context:** Stream can't answer "how many new posts since this user last visited?" and fetching N feeds' worth of activities to count client-side is expensive.

**Learning:** The hybrid approach works: Stream stores content durably (posts, reactions, comments, threading), D1 stores a thin metadata index (~150 bytes/row) that answers navigation questions. Write-time indexing (INSERT into D1 alongside stream.addActivity()) avoids polling entirely. Badge counts are a single D1 query across all feeds — zero Stream API calls.

**Pattern:** Dual-write at post creation, upsert visit timestamp on feed page load, query badge counts via LEFT JOIN between feed_activities and feed_visits.

## 4. FeedSlidePanel replacement — slideout vs. full page for primary navigation
**Topics:** astro

**Context:** The navbar "My Feeds" triggered a 320px slideout panel. The client sees feeds as 50% of learning.

**Learning:** A slideout panel communicates "quick utility" — fine for secondary navigation. When the client says something is a primary learning surface, it needs a full page with prominent styling, search, and room to grow (activity indicators, cross-feed digest). The slideout was replaced with a full-page `/feeds` route. The transition was clean — the FeedSlidePanel was already refactored to use `getFeeds()` in Phase 5, so the /feeds page used the same data source.

## 5. D1 FK constraints in tests require seed users
**Topics:** testing, d1

**Context:** Feed activity integration tests failed with `FOREIGN KEY constraint failed` because `feed_activities.actor_id` and `feed_visits.user_id` reference `users(id)`.

**Learning:** When testing tables with FK constraints, insert minimal user rows in `beforeEach`. Pattern: `INSERT OR IGNORE INTO users (id, email, password_hash, name, handle, can_take_courses, created_at, updated_at) VALUES (?, ?, 'hash', ?, ?, 1, datetime('now'), datetime('now'))`. The `OR IGNORE` handles repeat runs.

## 6. Comment endpoints lack feed context — dual-write gap
**Topics:** stream, d1

**Context:** Tried to add dual-write (D1 indexing) to comment endpoints but they don't carry the feed slug — they operate on Stream activity IDs directly.

**Learning:** Comment endpoints (`community/[slug]/comments.ts`, `townhall/comments.ts`) receive an `activityId` or `parentId` but not the feed slug. Without the slug, we can't write to `feed_activities` (which keys on feed_type + feed_id). Options for future: pass slug through request body, look up feed from activity ID, or accept that only top-level posts generate badges. For Genesis scale, the gap is acceptable.
