### Dependency Graph Discussion

**User:** "We should add a PLAN Block to assess current unit and API testing for 'stunbles' in user choices... PLATO is a keeper. BUt I want to talk about how to end up with the data from a heppy path actually residing in the database"

**Claude:** Discussion evolved the design from monolithic runs to composable segments with dependency declarations.

### Leaf-Driven Design

**User:** "Maybe I mis-stated. Is the 'bottom' of the tree one of the leafs? or is it most base dependency (the root)"

**Claude:** Clarified: user means start design from leaves (most dependent — `become-teacher`), traverse back to root (`register`). The traversal IS the run. Each node is a composable segment.

### Composability

**User:** "OPtion B, definitely. SOme goals are achievable from the user;s point of view without the creations of a community. They might want to just become a creator. So the leafs can be any node or leaf. COmposable should allow us to take advantage of common ancestors in sister branches"

**Claude:** Designed segment model: `PlatoSegment` with `requires[]` for dependency declaration. Runner resolves graph via topological sort. Common ancestors execute once, carry state shared.

### Route and Data Separation

**User:** "What still needs discussing is data... the browser path to get to these user actions is not declared, and there can be more than one path"

**Claude:** Separated segment into 3 layers: intent (what), surface (route), mechanism (API call). Segments declare the route WHERE the action happens, not HOW the user navigates there. Browser tests verify the route has the UI, not the navigation path.
