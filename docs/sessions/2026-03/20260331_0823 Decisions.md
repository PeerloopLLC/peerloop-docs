# Conv Decisions - 2026-03-31

## 1. PLATO Model B: Sequential DB-Accumulation
**Type:** Strategic
**Topics:** testing, plato, architecture

**Trigger:** User proposed page-centric model during Conv 061 discussion; existing composable segment model (Model A) used artificial carry state

**Options Considered:**
1. Model A — Composable segments with dependency graph, topological sort, explicit carry state
2. Model B — Sequential DB-accumulation, fixed run order, DB is the truth ← Chosen
3. Hybrid — composable segments with DB state (rejected as contradictory)

**Decision:** Adopted Model B. Runs execute in fixed order. Each deposits data into the DB. No cross-run carry state. Breaks fast and loudly.

**Rationale:** Model A's carry state hides integration gaps — if data is passed via named maps, you never discover that the page doesn't provide it. Model B removes the safety net: if the DB doesn't have the data, the test fails. User: "I prefer Model B because it breaks fast and loudly."

**Consequences:** Runs are not independently executable. Run N requires Runs 1..(N-1) to have completed. The rigidity is the feature, not a limitation.

## 2. API Emulation over Playwright for PLATO
**Type:** Strategic
**Topics:** testing, plato, e2e

**Trigger:** User asked "Why can't we just have PeerLoop be controlled by our PLATO runs and execute the UI to achieve the goals?"

**Options Considered:**
1. API emulation — call endpoints directly, fast, deterministic ← Chosen
2. Playwright-driven — open browser, click through UI, comprehensive but flaky

**Decision:** PLATO runs use API emulation. Playwright is reserved for targeted E2E tests (multi-browser flows).

**Rationale:** User: "The playwright isn't giving us fail fast and loud, it is giving us fail fast and garbled muttering." Flakiness (hydration timing, animation delays, selector brittleness) means failures don't reliably indicate real bugs. API emulation is 150-300x faster and produces actionable error messages.

**Consequences:** The developer must manually walk each run in the browser to verify the UI can trigger the API calls. Run definitions serve as manual test scripts.

## 3. Intra-Run Context Survives, Cross-Run Carry Eliminated
**Type:** Implementation
**Topics:** testing, plato

**Trigger:** Plan Mode stress test revealed that eliminating ALL carry state is too aggressive

**Options Considered:**
1. Eliminate all carry state (pure DB) — breaks within-run data flow
2. Keep intra-run context (`$context`), eliminate cross-run carry (`$chainCarry`) ← Chosen
3. Keep both (no change from Model A)

**Decision:** Renamed `carry` → `provides`/`context` on actions. `$context.actionName.key` for within-run references. `$chainCarry` removed entirely.

**Rationale:** Within a page visit, a GET response provides data for a POST. This isn't carry state — it's "what the page showed the user." Cross-run data should come from the DB via discovery GETs.

**Consequences:** The runner clears context between runs. Only the DB persists.

## 4. Actor Identity via fromDB Source
**Type:** Implementation
**Topics:** testing, plato

**Trigger:** Stress test found that actor sessions don't persist across runs without chainCarry

**Options Considered:**
1. Keep `fromCarry` — requires chainCarry (contradicts Model B)
2. `fromDB` — runner queries users table by persona email ← Chosen
3. Login step at start of each run — realistic but adds unnecessary API calls

**Decision:** New `fromDB` actor source. Runner does `SELECT id, email, is_admin, can_create_courses FROM users WHERE email = ?`.

**Rationale:** Simple, realistic. The actor exists in the DB because a prior run registered them. If they don't exist, the test fails loudly — which is the point.

**Consequences:** Actors are defined once in the persona set (not per-run). Any run can use any actor via `fromDB`.
