# Conv Learnings - 2026-03-28

## 1. Taxonomy Simplification Through Usage Analysis
**Topics:** schema-design, taxonomy

**Context:** User questioned the complexity of 5 tables (categories, topics, user_topic_interests, user_interests, course_tags) all serving related purposes

**Learning:** Tracing actual downstream consumption revealed that everything collapsed to category-level matching. `experience_level` was captured but never read. `user_interests` was a sync copy consumed by one endpoint's 20-point tag bonus. `course_tags` had no production data. The real useful granularity was at what we called "topics" (subtopics), which act as tags connecting users to courses.

**Pattern:** When evaluating schema complexity, map each table/column to its actual readers (not just writers). Columns that are written but never read are dead weight.

---

## 2. Naming Matters — User Mental Model Should Drive Schema Names
**Topics:** schema-design, ux

**Context:** The original names (categories, topics, interests) were all near-synonyms causing confusion

**Learning:** The user's mental model was: "tags are things you attach to stuff, topics are groups you browse by." Renaming to match this model (`categories` → `topics`, `topics` → `tags`) made the system immediately understandable. The schema should use the same vocabulary as the user and UI, not database-design jargon.

---

## 3. Multi-Tag Course Model Unlocks Cross-Topic Discovery
**Topics:** schema-design, taxonomy, feeds

**Context:** Old model had `courses.category_id` as a single FK — one category per course

**Learning:** Replacing with `course_tags` (many-to-many FK to master tags table) means a course can appear under multiple topics. A "Vibe Coding 101" course tagged with "React & Frontend" + "AI-Assisted Development" appears under both Full-Stack Development and AI Coding topics. This also makes user↔course matching more precise — score by tag overlap count rather than binary category match.
