# Conv Decisions - 2026-03-28

## 1. Clean Break on All Taxonomy Naming — No Aliases or Backward Compatibility
**Type:** Strategic
**Topics:** api, tag-taxonomy, naming

**Trigger:** User's response to design question about `?category=` → `?topic=` parameter rename

**Options Considered:**
1. Keep `?category` as an alias alongside `?topic` for backward compat
2. Clean break — rename to `?topic=`, remove all old naming ← Chosen

**Decision:** All API parameters, response fields, and variable names use new taxonomy terms (topics, tags, topicId, tagId). No aliases, no backward-compat shims.

**Rationale:** User directive: "stay consistent on actively remove conveniences like aliases unless they have strategic benefit." No external consumers depend on these API shapes yet (pre-launch).

**Consequences:** Every frontend component that fetches from these endpoints will need updating (Phase 5). No ambiguity about which naming convention is "current."

## 2. Remove category_id Entirely from Course Responses
**Type:** Implementation
**Topics:** api, schema, tag-taxonomy

**Trigger:** User confirmation of design question #3

**Options Considered:**
1. Keep `category_id` in responses derived from first tag's topic
2. Remove `category_id` entirely. Detail views return `tags: {id, name, topic_id}[]` ← Chosen
3. Replace with `topicId` (derived from primary tag)

**Decision:** Course list responses have no taxonomy field at all. Course detail responses include full tag objects with topic_id for grouping.

**Rationale:** A course can have tags across multiple topics — there's no single "category" to report. Tags are a detail-view concern; list views don't need them.

**Consequences:** Frontend list components (CourseCard, etc.) can no longer filter/group by category client-side. All topic filtering happens server-side via the `?topic=` parameter.

## 3. Testimonials Group by Topic Instead of Category
**Type:** Implementation
**Topics:** api, testimonials, tag-taxonomy

**Trigger:** Design question #4 — testimonials page previously grouped by `categories`

**Options Considered:**
1. Group testimonials by topic (via course_tags → tags → topics) ← Chosen
2. Group testimonials by course only (drop topic grouping)

**Decision:** Testimonials derive their topic from the course's tags, using LEFT JOINs through `course_tags → tags → topics`. GROUP BY t.id prevents duplicates from multi-tag courses.

**Rationale:** Consistent with the rest of the taxonomy — users filter/browse by topic.

**Consequences:** Testimonials without tagged courses show `topic_id: null, topic_name: null`. The `?topic=` filter parameter replaces `?category=`.
