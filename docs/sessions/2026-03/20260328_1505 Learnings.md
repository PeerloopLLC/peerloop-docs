# Conv Learnings - 2026-03-28

## 1. Balanced Brace Parsing for JSX Expression Extraction
**Topics:** route-matrix, regex, jsx-parsing

**Context:** The route-matrix scanner used simple regex patterns for `href={...}` that missed ternary expressions and conditional links.

**Learning:** Regex-only extraction of JSX attribute values is inherently limited — ternaries, logical operators, and nested template literals all break simple patterns. A lightweight balanced-brace parser that tracks string/template literal context handles all cases without requiring a full AST parser. The key insight: extract the balanced expression content first, THEN regex out string/template values from within it.

**Pattern:** `extractBalancedExpr(content, startIdx)` — tracks depth for `{}`, skips string literals (`'`/`"`), handles template literal `${}` nesting by incrementing depth on `${` and decrementing on `}` when depth > 1.

## 2. Structural Route Matching Resolves Generic Param Names
**Topics:** route-matrix, routing, param-resolution

**Context:** `normalizeDynamic` replaced `${courseIdParam}` with `[id]` (matching substring "id"), causing `/teaching/courses/[id]` to appear as a broken target when the actual route is `/teaching/courses/[courseId]`.

**Learning:** Dynamic param names in code variables rarely match Astro file-based route param names. Rather than trying to make `normalizeDynamic` smarter (which would require understanding all variable naming conventions), a post-processing step that structurally matches targets against known routes is more robust. Convert each segment to a wildcard pattern (`[anything]` → `[*]`), compare segment-by-segment, and when structure matches, adopt the actual route's param names.

## 3. Component Folder Naming Should Match Route Namespace
**Topics:** project-structure, naming-conventions

**Context:** `src/components/explore/` contained 37 files used exclusively by `/discover/*` pages — a naming mismatch from the development history where `/explore` routes were temporary.

**Learning:** When temporary development routes get renamed/replaced, the component folder should be renamed too. The cognitive overhead of "explore components serve discover pages" compounds as the team grows. The rename was trivial (folder move + import path updates) but gets harder over time as more code references accumulate.
