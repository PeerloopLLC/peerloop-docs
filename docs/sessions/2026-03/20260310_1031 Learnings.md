# Session Learnings - 2026-03-10

## 1. `HEAD~5` Was Arbitrary and `--since` Was Dead Code
**Topics:** cc-workflow

**Context:** Investigating why `/q-docs` showed 260+ changed files for a session that only touched 5 docs-repo files. The `detect-changes.sh` script accepted a `--since` parameter (defaulting to "24 hours ago") but completely ignored it — lines 19 and 32 hardcoded `HEAD~5` instead.

**Learning:** The `5` in `HEAD~5` had no documented rationale — it was chosen during the rapid initial build. The `--since` parameter interface was wired up but never connected to the actual implementation. The result was an unpredictable detection window: during high-commit-frequency work (like TERMINOLOGY's 10 sessions), 5 commits might span hours; during slow weeks, 5 commits could span days. Always verify that parameter interfaces are actually connected to implementation, especially in scripts built quickly.

## 2. Time-Based Detection Still Wastes Tokens on Multi-Session Days
**Topics:** cc-workflow

**Context:** After fixing `detect-changes.sh` to use `--since "24 hours ago"`, realized that a day with 3 sessions would still show all changes from sessions 1 and 2 when running `/q-docs` in session 3 — even though those were already documented.

**Learning:** The fundamental question isn't "what changed recently" but "what changed since the last `/q-docs` run." Time-based windows are a less-bad proxy for this, but the right approach is a **marker/cursor pattern** — record a checkpoint after each run, diff from that checkpoint forward next time. This is the same pattern as database migration cursors or Kafka consumer offsets: don't re-process what you've already handled.

**Pattern:** Record HEAD SHAs of both repos in a marker file after each `/q-docs` run. Next run diffs from marker to current HEAD + uncommitted changes.

## 3. Marker Files Should Be Committed for Multi-Machine Workflows
**Topics:** cc-workflow, dual-repo

**Context:** Initially added `.last-qdocs-run` to `.gitignore` as "machine-local state." User pointed out they work on two Mac Minis throughout the day, both using the same dual-repo setup.

**Learning:** When the same repos are used across multiple machines, state files that track "where we left off" must be committed — they're shared coordination state, not machine-local. The marker travels with the repo via git pull, so Mac A's `/q-docs` run sets the anchor and Mac B continues from there. If Mac B hasn't pulled the code repo (marker SHA ahead of local), the script's SHA validation fails gracefully and falls back to time-based detection.
