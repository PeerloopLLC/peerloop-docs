# Session Learnings - 2026-03-16

## 1. replace_all Safety Depends on Identifier Uniqueness
**Topics:** testing, workflow

**Context:** The TERMINOLOGY-CLEANUP block required renaming ~75 identifiers across 38 files. Some were globally unique (`AdminSTData`, `STGrowth`), others were short common substrings (`st` as a loop variable).

**Learning:** `replace_all` is safe for globally unique identifiers — there's only one thing in the codebase with that name, so a global replace can't hit unintended targets. For short identifiers like `st` that also appear as SQL table aliases (`teacher_certifications st`) or URL query params (`?st=alexteacher`), use word-boundary-aware `perl -pi -e 's/\bst\./teacher./g'` or targeted Edit calls instead.

**Pattern:** Use `Grep` first to audit all occurrences of the target string. If every match is something you want renamed, `replace_all` is safe. If there are false-positive matches, use targeted replacements.

## 2. SQL Table Aliases vs Column Aliases — Only Column Aliases Need Renaming
**Topics:** d1, testing

**Context:** The `st_id`/`st_name` rename touched SQL queries where `st` was both a table alias (`LEFT JOIN users st ON ...`) and a column alias (`st.name as st_name`).

**Learning:** SQL table aliases are query-internal shorthand — `st` as a table alias for `teacher_certifications` is fine and doesn't leak into application code. SQL column aliases (`AS st_name`) DO leak because they become TypeScript interface property names and JSON response keys. Only column aliases need renaming; table aliases can stay short.

## 3. PostToolUse Hooks for Behavioral Nudges at Point of Discovery
**Topics:** cc-workflow

**Context:** Despite feedback memories saying "always TodoWrite check findings," the pattern of dismissing issues as "pre-existing" kept recurring. The problem was attention narrowing during complex work — feedback memories aren't weighted strongly enough during execution.

**Learning:** A PostToolUse hook that fires after check commands (npm test, tsc, astro check, etc.) and injects a reminder when issues are found is more reliable than relying on memory. The hook fires at exactly the right moment: after the output, before the response. Two-layer design: Layer 1 (specific check commands) parses detailed counts and uses strong "TodoWrite NOW" language; Layer 2 (general failures) uses softer "Diagnose the failure" language since not all non-zero exits are bugs.

## 4. Claude Code Settings File Layering and Mid-Session Persistence
**Topics:** cc-workflow

**Context:** User kept getting prompted for `printf` permissions despite approving "always allow" each time.

**Learning:** Claude Code has 3 settings layers: `~/.claude/settings.json` (global), `.claude/settings.json` (project, git-tracked), `.claude/settings.local.json` (project-local, gitignored). Permission approvals from the prompt save to `settings.local.json`. Newly-added patterns may not take effect until the next session — mid-session additions can still re-prompt.

## 5. Deprecated Function Tests — Convert Rather Than Suppress
**Topics:** testing

**Context:** Astro check reported 3 hints — 1 unused import and 2 deprecation warnings from a test exercising `notifySessionLateCancelled` (a deprecated wrapper).

**Learning:** Rather than suppressing deprecation hints with `// @ts-ignore` or `eslint-disable`, convert the test to use the replacement function directly (`notifySessionCancelled`). This eliminates hints AND improves coverage — the replacement function gets direct test coverage. In this case, also added an additional test case (on-time cancellation) that the deprecated wrapper couldn't easily test.
