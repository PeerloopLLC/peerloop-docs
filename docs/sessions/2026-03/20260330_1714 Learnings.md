# Conv Learnings - 2026-03-30

## 1. Seed Data Masks Missing Creation Paths
**Topics:** testing, plato, seed-data

**Context:** Building PLATO's first run revealed that `autoJoinTheCommons` would fail in production because the `joined_via` CHECK constraint didn't include `'registration'`. 6356 existing tests missed this because they insert data via SQL.

**Learning:** Tests that insert prerequisite data directly can never catch bugs in the data creation path. PLATO's approach of creating all data through API calls surfaces these bugs naturally.

**Pattern:** If PLATO can't create data it needs through an API, that's a finding — not a reason for a workaround.

---

## 2. Vitest Dual Alias Mocking
**Topics:** testing, vitest, mocking

**Context:** Peerloop uses `@/lib/auth` and `@lib/auth` as aliases for the same module. `vi.mock()` treats these as separate module specifiers.

**Learning:** When a codebase uses multiple aliases for the same module, ALL aliases must be mocked with the SAME function instances. Use `vi.hoisted()` to create shared mock functions that both `vi.mock()` calls reference. Additionally, `requireRole` must be mocked directly (ESM internal calls bypass export-level mocks).

**Pattern:** `const mockAuth = vi.hoisted(() => { ... }); vi.mock('@/lib/auth', ...mockAuth); vi.mock('@lib/auth', ...mockAuth);`

## 3. Handler Import Resolution for Dynamic Routes
**Topics:** testing, plato, astro

**Context:** PLATO steps have URL paths with real values (`/admin/users/abc123`) but Astro's file system uses bracket params (`/admin/users/[id].ts`).

**Learning:** The runner must reconstruct file-system paths from the step's `params` keys, replacing resolved values with `[paramName]` brackets before importing.

---

## 4. Composable Segments vs Monolithic Runs
**Topics:** plato, architecture, testing

**Context:** Discussion evolved PLATO from monolithic runs (entire journeys) to composable segments (atomic goals with dependency declarations).

**Learning:** User goals are atomic ("create a course"), not journeys ("register, get access, create community, build course, publish"). The dependency graph between goals is the platform's capability map. Leaf-driven design (start from most dependent goal, traverse to root) discovers all prerequisites by necessity.

---

## 5. Segment Three-Layer Model
**Topics:** plato, architecture, testing

**Context:** Discussion about how PLATO knows the steps are achievable by the user and that the API calls match what the UI presents.

**Learning:** A segment declares three things separately: intent (what the user wants), surface (the route where it happens), and mechanism (the API calls). The segment doesn't care how the user navigated to the route. Browser tests verify the route has UI that triggers the API call. Navigation path testing is a separate concern.
