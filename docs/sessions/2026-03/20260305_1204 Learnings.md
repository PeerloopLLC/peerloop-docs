# Session Learnings - 2026-03-05

## 1. ISO 8601 vs SQLite datetime() Format Mismatch Breaks String Comparisons
**Topics:** d1, testing

**Context:** Integration tests for the message unread count endpoint used SQLite's `datetime('now')` for test data setup while the API endpoints use JavaScript's `new Date().toISOString()` (via the `now()` helper). Tests failed intermittently when comparing timestamps with `>`.

**Learning:** `now()` returns ISO 8601 format (`2026-03-05T11:45:50.123Z`) while SQLite's `datetime('now')` returns `2026-03-05 11:45:50`. The `T` character (0x54) sorts AFTER space (0x20) in string comparison, so an ISO timestamp at the same clock time appears "greater than" a SQLite timestamp. This breaks `created_at > last_read_at` comparisons when the two columns use different formats.

**Pattern:** Never mix formats in the same column. All Peerloop API code uses `now()` (ISO) consistently. Test data setup should also use ISO format (via JS Date) rather than SQLite's `datetime()` function.

## 2. NULL last_read_at as a Reliable Test Strategy
**Topics:** testing, d1

**Context:** Integration tests for message counting needed to verify "all unread" as a precondition. Using timestamp manipulation (setting `last_read_at` to past or future times) was fragile because tests execute within milliseconds of each other.

**Learning:** Setting `last_read_at = NULL` is immune to all timestamp precision issues because the count query treats NULL as "never read, count all messages from others." This is far more reliable than trying to engineer timestamp ordering within sub-millisecond test execution windows.

**Pattern:**
```sql
-- Clean slate for tests: reset to NULL (counts all from others)
UPDATE conversation_participants SET last_read_at = NULL WHERE conversation_id = ?;
DELETE FROM messages WHERE conversation_id = ?;
-- Then insert fresh messages — they'll always be counted as unread
```

## 3. CustomEvent for React Sibling Communication Without Prop Drilling
**Topics:** astro, messaging

**Context:** The "Mark all read" button lives in `ConversationList` (child), but the conversation data lives in `Messages` (parent). Needed to trigger a data refresh in the parent after the child calls the mark-all-read API.

**Learning:** `window.dispatchEvent(new CustomEvent('messages:mark-all-read'))` + `window.addEventListener` is a clean pattern for child-to-parent communication when prop drilling would add unnecessary complexity. The child fires the event, the parent listens and refreshes. No shared state, no callbacks threaded through multiple component layers.

**Pattern:**
```typescript
// Child: dispatch after API call
window.dispatchEvent(new CustomEvent('messages:mark-all-read'));

// Parent: listen and refresh
useEffect(() => {
  const handler = () => fetchConversations();
  window.addEventListener('messages:mark-all-read', handler);
  return () => window.removeEventListener('messages:mark-all-read', handler);
}, [fetchConversations]);
```

## 4. Correlated Subquery for Efficient Aggregate Unread Count
**Topics:** d1, messaging

**Context:** The `/api/me/messages/count` endpoint needs to return total unread messages across ALL conversations. The existing `GET /api/conversations` endpoint calculates unread per-conversation in an N+1 loop (one query per conversation).

**Learning:** A single SQL query with a correlated subquery and `COALESCE(SUM(...), 0)` computes the total in one round-trip. This avoids fetching participant lists, last messages, and other data the badge doesn't need. The query pattern: outer query selects all conversation_participants for the user, inner correlated subquery counts unread messages per conversation, outer SUM aggregates.

**Pattern:**
```sql
SELECT COALESCE(SUM(unread), 0) as count FROM (
  SELECT (
    SELECT COUNT(*) FROM messages m
    WHERE m.conversation_id = cp.conversation_id
      AND m.sender_id != ?
      AND (cp.last_read_at IS NULL OR m.created_at > cp.last_read_at)
  ) as unread
  FROM conversation_participants cp
  WHERE cp.user_id = ?
)
```
