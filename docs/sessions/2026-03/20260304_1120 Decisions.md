# Session Decisions - 2026-03-04 (Session 325)

## 1. Remove Sub-Ratings from Enrollment Reviews Endpoint
**Type:** Implementation
**Topics:** d1, astro

**Trigger:** D1_ERROR on `/api/student-teachers/:id/reviews` — queried non-existent columns.

**Options Considered:**
1. Fix the endpoint — remove sub-rating columns from the query ← Chosen
2. Fix the schema — add sub-rating columns to enrollment_reviews
3. Discuss design intent first

**Decision:** Remove `teacher_rating`, `interaction_rating`, `materials_rating` from the ST reviews endpoint. These columns belong on `session_assessments`, not `enrollment_reviews`. Enrollment reviews are simple rating + comment by design (per tech-022-ratings-feedback.md).

**Rationale:** The three-tier rating system has clear separation: session assessments have sub-ratings, enrollment reviews don't.

**Consequences:** Fixed the 500 error on ST profile pages. Added 13 tests for the endpoint.

## 2. Enforce Teacher-Enrollment Match on Session Booking
**Type:** Implementation
**Topics:** auth, d1

**Trigger:** Discussion about what prevents a student from booking sessions with a different teacher or for a different course.

**Options Considered:**
1. No guard — trust the UI to send correct data
2. Validate `teacher_id` matches `enrollment.student_teacher_id` ← Chosen

**Decision:** Add a 403 guard in `POST /api/sessions` that rejects bookings where the teacher doesn't match the enrollment's assigned ST.

**Rationale:** The enrollment records which ST the student is paired with. Downstream actions must validate against this relationship. A student enrolled with Marcus can only book Marcus.

**Consequences:** Updated the API and one existing test. Closes a security gap.

## 3. Sessions Must Link to Modules (Schema Change — Future)
**Type:** Strategic
**Topics:** d1, video

**Trigger:** Discussion about gating Mark Complete button on session completion, and realizing there's no way to know which module a session covers.

**Options Considered:**
1. Count-based — enable Mark Complete for module N when completed sessions >= N
2. Add `module_id` to sessions table — explicit link ← Chosen
3. Ordinal matching — match by chronological order

**Decision:** Add `module_id TEXT REFERENCES course_curriculum(id)` to the sessions table. Each booked session will be explicitly tied to a module.

**Rationale:** Without this link, students don't know what they're booking, teachers don't know what to teach, and progress tracking is disconnected from actual tutoring. The implicit ordinal assumption is fragile.

**Consequences:** Documented in tech-032-session-booking.md as open design question #2. Impacts: schema, booking wizard, booking API, session room, teacher dashboard, learn page, progress sidebar. Deferred to implementation phase.

## 4. Booking Flow — Target Design
**Type:** Strategic
**Topics:** astro, video

**Trigger:** User testing the booking flow and identifying UX gaps (teacher re-selection, no module context, no multi-session flow).

**Decision:** Defined the target booking flow:
- **Single session:** Course selected → teacher known from enrollment → skip teacher selection → go directly to availability for next unbooked module
- **Multi-session:** After confirming, success screen offers "Book Next Session" → advances to next unbooked module
- **Module ordering:** Presented in `module_order` sequence, only unbooked modules offered
- **Cancellation/refunds:** Future feature, partial refund for undelivered sessions (per CD-033 policy)

**Rationale:** The current wizard requires teacher selection every time and has no module context. The enrollment already establishes the teacher, and modules define the session content.

**Consequences:** Documented in tech-032-session-booking.md under open question #1. Requires #2 (module_id on sessions) to implement.

## 5. Module Tab UX — Add Section Label
**Type:** Implementation
**Topics:** astro

**Trigger:** User confused by "Mark Complete" and "Next" buttons lacking context on the Modules tab of the Learn page.

**Options Considered:**
1. Add "Module N of M" section label above the action bar ← Chosen
2. Rename buttons to "Next Module" / "Mark Module Complete"
3. Both label and rename

**Decision:** Add a "Module N of M" label above the navigation/complete action bar. Keep button labels as-is.

**Rationale:** The label frames the entire action area and makes clear that Next/Previous refer to modules and Mark Complete applies to the current module. Simpler than renaming buttons.

**Consequences:** Added `totalModules` prop to ModuleContent component. Label renders above the button row.
