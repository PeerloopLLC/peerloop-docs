# Session Learnings - 2026-03-17

## 1. $CLAUDE_PROJECT_DIR Is Not Available in Skill `!` Backtick Pre-computation
**Topics:** cc-workflow, dual-repo

**Context:** Replaced all hardcoded `/Users/jamesfraser/projects/peerloop-docs` paths in 11 r-* skill files with `$CLAUDE_PROJECT_DIR`. When `/r-learn-decide` was invoked, the `!` backtick command failed with `no such file or directory: /.claude/scripts/conv-files-learn-decide.sh` — the env var expanded to an empty string.

**Learning:** `$CLAUDE_PROJECT_DIR` works in hook `command` fields (configured in `settings.json`) but does NOT expand in `!` backtick pre-computation commands within SKILL.md files. The `!` backtick execution environment apparently does not inherit the same env vars that hooks receive. This means the path abstraction approach needs a different mechanism for `!` backtick lines — possibly a wrapper script that resolves the path via `git rev-parse --show-toplevel` or a known anchor file.

**Pattern:** The hooks in `settings.json` use `$CLAUDE_PROJECT_DIR` successfully:
```json
"command": "$CLAUDE_PROJECT_DIR/.claude/hooks/check-output-reminder.sh"
```
But in SKILL.md, this fails:
```
!`$CLAUDE_PROJECT_DIR/.claude/scripts/conv-files-learn-decide.sh`
```

## 2. Global q-* Commands Serve Multiple Projects
**Topics:** cc-workflow

**Context:** Initial cleanup plan included deleting 19 q-* command files from `~/.claude/commands/`. User corrected that these are global commands used by older projects (e.g., CCO Obsidian vault work) that haven't been converted to the r-* skill format.

**Learning:** The `~/.claude/commands/` directory is global — commands there are available to ALL Claude Code projects on the machine. Project-specific skills in `.claude/skills/` can shadow them, but deleting globals affects every project. Always ask before removing global commands.

## 3. settings.local.json Is Gitignored and Machine-Specific
**Topics:** cc-workflow, dual-repo

**Context:** The `settings.local.json` permission allow-list had stale `Skill(q-*)` and `Skill(w-*)` entries. User chose to strip all skill permissions and rebuild organically.

**Learning:** `settings.local.json` is gitignored — it exists independently on each machine. Permission allow-lists don't sync between MacMiniM4 and MacMiniM4-Pro. Each machine accumulates its own permissions as the user approves skills during use. This is by design — it prevents one machine's permission grants from affecting another.

## 4. Dual-Repo Skill Portability Requires Path Abstraction
**Topics:** cc-workflow, dual-repo

**Context:** All 11 r-* skills were authored on MacMiniM4-Pro (home dir `/Users/jamesfraser/`) and committed to git. On MacMiniM4 (home dir `/Users/livingroom/`), every `!` backtick invocation and `git -C` instruction pointed to a nonexistent path.

**Learning:** When skills are shared via git between machines with different home directories, any absolute path in a SKILL.md file becomes machine-specific. The correct abstraction is environment variables, but the specific mechanism depends on the execution context (`!` backtick vs hooks vs prose instructions). Need to find an approach that works across all three contexts.
