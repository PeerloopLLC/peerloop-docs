# Session Learnings - 2026-03-05

## 1. Positional Assignment Avoids Stale Denormalized Data
**Topics:** d1, architecture

**Context:** Designing how sessions link to course modules. Original plan was to store `module_id` directly on each session row at booking time.

**Learning:** When a relationship is derived from ordering (Nth session teaches Nth module), storing it at write time creates a denormalization that goes stale on every insert/delete. Computing it at read time (late binding) keeps the data always consistent. The trade-off — a small computation per read — is negligible for the expected data volumes (3-10 sessions per enrollment).

**Pattern:** Filter-then-zip: fetch completed sessions (frozen module_ids), fetch scheduled sessions (no module_id), fetch curriculum, skip frozen modules, zip remaining curriculum onto scheduled sessions positionally.

## 2. Resource Limits Surface as Test Failures in Unrelated Tests
**Topics:** testing, d1

**Context:** Adding session limit enforcement (422 when all modules booked) caused 16 test failures in files that never tested limits — they simply didn't create any `course_curriculum` entries.

**Learning:** When you add a resource limit (bookable sessions <= module count), every test that creates sessions against a course needs curriculum entries. Without them, `0 modules - 0 completed - 1 scheduled` yields `bookable_count = 0`, hitting the 422 before reaching any other validation. This is a form of test coupling through shared business constraints — the new rule surfaces as failures in previously independent tests.

**Pattern:** When adding limit enforcement, grep test suites for `INSERT INTO sessions` to find all tests that need curriculum data added.

## 3. Completed Sessions Freeze Module Assignment as Historical Record
**Topics:** d1, architecture

**Context:** The positional module assignment model means scheduled sessions reflow when sessions are added/cancelled. But completed sessions need a stable historical record.

**Learning:** The hybrid approach works well: `module_id` is nullable on the `sessions` table, NULL while scheduled (computed at read time), and set permanently when the BBB webhook marks the session as `completed`. Sequential completion (session N can't complete before session N-1) guarantees no gaps in the frozen assignments, making the reflow algorithm simple — just skip frozen modules and assign the rest positionally.
