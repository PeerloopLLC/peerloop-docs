# Session Learnings - 2026-03-01 (Session 320)

## 1. vi.fn() in vi.mock Factories Enables Per-Test Overrides
**Topics:** testing

**Context:** After adding `useCreatorGate` to 5 components, all 4 existing component test files needed to mock the hook. The default behavior (return `{ status: 'creator', hasCourses: true }`) works for most tests, but gate-state tests need to override it to `not-creator` or `loading`.

**Learning:** Using `vi.fn(() => defaultValue)` inside a `vi.mock` factory (instead of a plain function) allows per-test overrides via `vi.mocked(fn).mockReturnValue(...)`. The global `beforeEach(() => vi.clearAllMocks())` automatically resets the mock to the factory default for subsequent tests — no explicit `afterEach` cleanup needed.

**Pattern:**
```ts
// Module-level mock with vi.fn()
vi.mock('@components/auth/useCreatorGate', () => ({
  useCreatorGate: vi.fn(() => ({ status: 'creator' as const, hasCourses: true })),
}));

// Per-test override (auto-resets via vi.clearAllMocks)
it('shows gate when not-creator', () => {
  vi.mocked(useCreatorGate).mockReturnValue({ status: 'not-creator', hasCourses: false });
  render(<Component />);
  expect(screen.getByText('Creator Access Required')).toBeInTheDocument();
});
```

## 2. Shared Test DB State Can Cause Cross-Test Interference
**Topics:** testing, d1

**Context:** Added a Pattern C test to `courses/index.test.ts` by inserting a revoked creator's course into the shared `progressionId` at `progression_position: 2`. This shifted the auto-assigned position for a subsequent POST test, which expected position 2 as the next available slot.

**Learning:** When API tests share a test database (sequential, no parallelism), inserting data in one test block can shift expectations in later tests. For gate-check tests that only need a course to *exist* (not be in a specific progression), insert the course without a progression to avoid contaminating shared fixtures.

## 3. Pre-Existing Test Bugs Can Be Masked by Unrelated Test State
**Topics:** testing

**Context:** CreatorStudio had a test "navigates to course editor on edit click" that expected `mockLocation.href` to be `/creating/studio`. The actual component sets it to `/creating/studio?course=course-1`. This test was passing because a prior test (the 401 redirect test) had set `mockLocation.href` to `/login`, and the 401 test ran first, but the edit test was reading stale state from the previous test's `mockLocation` setup.

**Learning:** Removing tests that mask bugs in other tests (by changing shared mutable state like `mockLocation`) can expose latent assertion errors. When a test suddenly fails after unrelated changes, check if it was only passing due to side effects from a sibling test.

## 4. Client-Side Gate as UX Layer Over Server-Side Security
**Topics:** auth, astro

**Context:** The `useCreatorGate` hook reads from `CurrentUser` global state cached in `window.__peerloop`. It gates API calls and shows a friendly "Creator Access Required" message. But the server-side API endpoints (Pattern C gates) remain the authoritative security enforcement.

**Learning:** Client-side access gates serve a different purpose than server-side gates. The client gate prevents unnecessary network requests and provides instant UX feedback. It can be slightly less strict — showing a brief flash of content before the gate activates is acceptable because the API will reject unauthorized requests regardless. The stale-cache refresh pattern (call `refreshCurrentUser()` before denying) is appropriate at the client level but would be over-engineering at the API level.

## 5. Stale-While-Revalidate Applies to Permission Checks Too
**Topics:** auth

**Context:** When a user navigates to `/creating/*`, the cached `CurrentUser` might say "not a creator" if admin just granted permission while the tab was open. The hook calls `refreshCurrentUser()` before showing the gate, handling this case.

**Learning:** The stale-while-revalidate pattern used for `CurrentUser` data loading also applies to permission-dependent UX decisions. If cached state would deny access, refresh once before showing a gate — this covers the common admin-just-approved scenario without requiring real-time push notifications.
