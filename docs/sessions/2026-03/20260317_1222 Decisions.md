# Conv Decisions - 2026-03-17

## 1. Store All Session Times as UTC ISO 8601 with Z Suffix
**Type:** Implementation
**Topics:** d1, cloudflare, astro

**Trigger:** Client-reported bug: both student and teacher got "Session time has passed" when arriving on time. Root cause: bare datetime strings parsed as different timezones on Workers (UTC) vs browser (local).

**Options Considered:**
1. Store times with UTC Z suffix everywhere ← Chosen
2. Store times with timezone offset (e.g., `+05:00`)
3. Keep bare strings but normalize parsing with a wrapper function

**Decision:** All `scheduled_start` and `scheduled_end` values are stored as UTC ISO 8601 strings with Z suffix (e.g., `2026-03-20T18:00:00.000Z`). The POST endpoint validates and rejects bare datetime strings.

**Rationale:** Z-suffixed strings are unambiguous — `new Date()` parses them identically on Workers and in browsers. SQLite TEXT comparison works lexicographically. No wrapper functions needed on the read path. Option 2 adds complexity for no benefit (UTC is the universal standard). Option 3 is fragile and requires touching every consumer.

**Consequences:** Migration `0003_fix_session_times.sql` appends Z to existing bare values. Availability API now emits full UTC ISO strings instead of bare HH:MM. Booking component passes UTC strings directly instead of constructing bare datetimes.

> **Insight:** This is a textbook example of the "ambient timezone" antipatent — when code implicitly relies on the runtime's default timezone rather than explicitly specifying one. In serverless/edge environments where the runtime is always UTC, this creates a systematic mismatch with browser-side code. The fix isn't just appending Z — it's establishing the convention that all timestamps crossing the client/server boundary must be explicitly UTC.

## 2. Use Intl.DateTimeFormat sv-SE Trick for Timezone Conversion
**Type:** Implementation
**Topics:** cloudflare

**Trigger:** Need to convert teacher-local times (e.g., 9:00 AM America/New_York) to UTC on Cloudflare Workers without external dependencies.

**Options Considered:**
1. External library (luxon, date-fns-tz) — adds bundle size
2. Manual UTC offset table — fragile, doesn't handle DST
3. Intl.DateTimeFormat with sv-SE locale trick ← Chosen

**Decision:** Use the sv-SE locale trick: format a UTC guess into the target timezone, compute the offset, adjust. Single iteration, no library, works on Workers.

**Rationale:** Cloudflare Workers support `Intl.DateTimeFormat` with full ICU timezone data. The sv-SE locale produces `YYYY-MM-DD HH:MM:SS` format which is easy to parse back. No external dependency needed. Handles DST transitions correctly because it uses the same tz database as the platform.

**Consequences:** New utility `src/lib/timezone.ts` with `localToUTC()` and `formatLocalTime()`. 15 unit tests covering EDT, EST, UTC, Asia/Tokyo, DST edge cases, date boundary crossings.

## 3. Email Times Displayed in UTC with Label (Temporary)
**Type:** Implementation
**Topics:** astro

**Trigger:** Notification emails format session times for display. With UTC storage, `toLocaleTimeString()` on Workers produces UTC times.

**Options Considered:**
1. Display in UTC with "UTC" label ← Chosen (temporary)
2. Query each user's timezone and format per-recipient
3. Display in teacher's timezone for all parties

**Decision:** For now, format email times as UTC with explicit "UTC" label (e.g., "1:00 PM UTC"). Per-user timezone formatting is deferred.

**Rationale:** This is correct and unambiguous. Per-user timezone formatting requires a `timezone` column on the users table (doesn't exist yet) and recipient-specific formatting logic. The UTC approach ships immediately and unblocks the client. A follow-up task for per-user timezone formatting can be added to PLAN.md.

**Consequences:** All session-related emails (booking, reschedule, cancellation) now show times in UTC. Users may need to mentally convert, but times are never wrong.
