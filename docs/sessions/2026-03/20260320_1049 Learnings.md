# Conv Learnings - 2026-03-20

## 1. Skill Text Is More Reliable Than Memory for In-Chain Behavior
**Topics:** cc-workflow

**Context:** The recurring problem of mentioning documentation gaps as "pre-existing" during `/r-end` → `/r-eos` → `/r-docs` without TodoWriting them. Feedback memories (`feedback_surface_and_track_all_issues.md`, `feedback_codecheck_todos.md`) already documented this rule but it kept being violated.

**Learning:** Memory entries are loaded at conversation start but can be compressed away during long skill chain executions. Skill SKILL.md files are loaded fresh each time the Skill tool invokes them, guaranteeing the instruction is in context at the critical moment. For behavior that must hold during a specific skill's execution, embed the rule in the skill text itself rather than relying solely on memory.

**Pattern:** Defense-in-depth — place the rule at both the orchestrator level (`/r-eos` Rules section) and the executor level (`/r-docs` Action Plan callout). The orchestrator sets the expectation before the executor is invoked; the executor reinforces it at the point where gaps are actually discovered.

## 2. Trigger-Word Self-Monitoring in Skill Instructions
**Topics:** cc-workflow

**Context:** The gap-tracking failures had a common pattern: Claude would use words like "pre-existing", "missing", "stale", "gap" in output text without creating a TodoWrite task. The existing memory entry already listed these trigger words but the behavior persisted.

**Learning:** Listing trigger words directly in the skill text where the behavior occurs (not just in memory) creates an immediate self-check loop. The instruction "if you're saying it but not fixing it, TodoWrite it" is most effective when it appears in the same context window as the actual gap discovery, not loaded earlier and potentially evicted.
