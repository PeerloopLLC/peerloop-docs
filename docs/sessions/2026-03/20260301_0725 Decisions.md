# Session Decisions - 2026-03-01

## 1. URL Query Parameter for Sub-View Navigation in Astro Pages
**Type:** Implementation
**Topics:** astro

**Trigger:** The Edit button on `/creating/studio` was broken — `handleEditCourse` navigated to the same URL (`/creating/studio`) without the `courseId`, showing the course list again instead of the CourseEditor.

**Options Considered:**
1. React state (`editingCourseId` via `useState`) — instant switch, no page reload
2. URL query parameter (`?course=<id>`) ← Chosen

**Decision:** Use URL query parameters (`/creating/studio?course=<id>`) for sub-view navigation within a single Astro route. The component reads the param on mount via `useEffect`, and the Astro page reads it server-side via `Astro.url.searchParams` for breadcrumbs/title.

**Rationale:** URL params provide refresh-safety (editing survives F5), browser back button support (returns to course list), and bookmark-ability. The approach is consistent with existing patterns in the codebase (`Messages.tsx` reads `?to=` and `?conversation=`, `StripeConnectSettings.tsx` reads `?success=`). The tradeoff — a full page reload on edit click instead of an instant state switch — is consistent with Astro's MPA navigation model. Both enter and exit are symmetric (URL navigations).

**Consequences:**
- `handleEditCourse` and `handleCourseCreated` navigate via `window.location.href`
- `CreatorStudio` reads `?course=` in a mount `useEffect` (SSR-safe)
- `studio.astro` reads `Astro.url.searchParams.has('course')` for conditional breadcrumbs/title
- `CourseEditor`'s existing back link (`href="/creating/studio"`) works without changes
