# Conv Learnings - 2026-03-29

## 1. Hydration-Safe React Hooks for Astro SSR
**Topics:** astro, react, hydration, currentUser

**Context:** Multiple `client:load` React islands showed hydration mismatches because `useCurrentUser()` returned a cached user instantly from localStorage on the client but `null` during SSR.

**Learning:** React hooks used in Astro `client:load` islands must initialize with SSR-compatible values (null/'loading'), then populate from browser APIs (localStorage, fetch) in useEffect. This ensures the first client render matches server HTML. The fix belongs in the hook itself, not in per-component workarounds — 2 lines in one file vs ~15 lines across 4 files.

**Pattern:** `useState(null)` + `useEffect(() => { setState(getFromCache()); }, [])` instead of `useState(() => getFromCache())`.

## 2. Audit for Systemic Patterns Before Per-Component Fixes
**Topics:** debugging, architecture

**Context:** Initial fix was per-component (hasMounted/effectiveUser pattern). After fixing 3 components, user identified the pattern was brittle and would dissipate.

**Learning:** When the same workaround appears in 3+ places, it's a signal that the fix belongs at a lower level. Per-component fixes are fine for one-offs; systemic issues need systemic solutions.

## 3. OnboardingNudgeBanner Reuse Pattern
**Topics:** react, components, onboarding

**Context:** Onboarding nudge was inline in DiscoverFeedsGrid only. Needed on 4 surfaces.

**Learning:** Created reusable component with context prop for contextual messaging, compact/full variants, per-context localStorage dismiss. Returns null for visitors and onboarded users — safe to drop anywhere without conditionals in the parent.

## 4. interestTopicIds on CurrentUser Enables Synchronous Filter
**Topics:** currentUser, filtering, tags

**Context:** "My Interests" filter needed user's topic preferences. Without CurrentUser data, would require async fetch + loading state.

**Learning:** Adding `interestTopicIds` (derived from user_tags → tags → topics) to the /api/me/full payload makes interest-based filtering synchronous. Tiny payload (array of ~8-12 topic ID strings), single JOIN query, slots into existing Promise.all.

---

## 5. MY_INTERESTS_TOPIC Sentinel Pattern
**Topics:** filtering, UX, react

**Context:** Topic filter is single-select radio buttons. "My Interests" means multiple topics. Converting to multi-select would change existing UX.

**Learning:** A sentinel string constant (`'__interests__'`) as a pseudo-radio option avoids changing the single-select model. The filter logic checks for the sentinel and does a Set-based multi-topic match. Can't collide with real UUIDs.
