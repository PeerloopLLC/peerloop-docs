# Session Decisions - 2026-03-01

## 1. Map Studio Detail Route to List View (No Detail Page)
**Type:** Implementation
**Topics:** astro

**Trigger:** Three code locations navigated to `/dashboard/creator/studio/${courseId}` which had no corresponding page. Needed to decide: create a new `/creating/studio/[id].astro` detail page, or redirect to the list view.

**Options Considered:**
1. Create `/creating/studio/[id].astro` detail page
2. Map all studio detail links to `/creating/studio` (list view) ← Chosen

**Decision:** Map `/creating/studio/${id}` references to `/creating/studio` (the list view). No detail route page created.

**Rationale:** The plan explicitly noted "(list view, no detail route)" — the course editor is rendered as a state within the `CreatorStudio` React component, not as a separate Astro page. Creating a detail page would require architectural changes beyond the broken-links scope.

**Consequences:** CreatorStudio's `handleCourseCreated` and `handleEditCourse` now navigate to the studio list. The scanner reports 4 broken targets (down from 26), not 5.

## 2. Split /workspace into Role-Aware Routes
**Type:** Implementation
**Topics:** astro

**Trigger:** Two files used `/workspace` as a navigation target. No `/workspace` page exists — it was a remnant of an earlier design.

**Options Considered:**
1. Map `/workspace` to `/learning` for all users
2. Create a `/workspace` page that redirects based on role
3. Split inline: creators → `/creating`, teachers → `/teaching`, students → `/learning` ← Chosen

**Decision:** Replace `/workspace` with role-aware routing inline. In `AppNavbar.tsx`, check `canCreateCourses` first (→ `/creating`), then `canTeachCourses` (→ `/teaching`). In `success.astro` (enrollment context), use `/learning` since only students enroll.

**Rationale:** Creating a redirect page adds a hop and a file for no benefit. Inline role-checking already existed in the same components (they already knew the user's role). Matching each context to the right dashboard is clearer than a generic fallback.

**Consequences:** No `/workspace` references remain. Each navigation target is explicit and unambiguous.

## 3. Use LandingLayout for Public Pages (404, Verify, Placeholders)
**Type:** Implementation
**Topics:** astro

**Trigger:** Needed to choose a layout for 20 new pages: custom 404, certificate verification, and 17 placeholder pages.

**Options Considered:**
1. AppLayout (sidebar-based, requires auth context)
2. LandingLayout (header + footer, works for authenticated and unauthenticated) ← Chosen

**Decision:** All new public-facing pages use `LandingLayout`. This includes the 404 page, verify/[id] page, and all 17 placeholder pages (legal, marketing, support, browse).

**Rationale:** These pages must be accessible without login. `LandingLayout` provides the auth-aware header (shows login/signup for anonymous, user menu for authenticated) plus the full marketing footer. `AppLayout` assumes an authenticated context with a persistent sidebar, which is wrong for public pages.

**Consequences:** Consistent public page experience. The 404 page works for both authenticated and unauthenticated visitors.

## 4. Notification Actions with No Sub-Route Map to Parent Dashboard
**Type:** Implementation
**Topics:** astro

**Trigger:** Two notification helpers pointed to routes that don't exist: `/dashboard/creator/applications/${id}` and `/dashboard/teaching/cert-requests/${id}`.

**Options Considered:**
1. Create new pages for these routes
2. Map to parent dashboard route ← Chosen
3. Remove the actionUrl entirely

**Decision:** Map `notifySTApplication` actionUrl to `/creating` and `notifyCertRequest` actionUrl to `/teaching`. Users clicking the notification land on their dashboard where they can find the relevant item.

**Rationale:** Creating dedicated pages is out of scope for BROKENLINKS. The parent dashboard is the correct landing zone — both the creator and teaching dashboards show pending actions. Removing the URL would make notifications non-clickable, which is worse UX.

**Consequences:** Notifications land on the right dashboard but not directly on the specific item. When dedicated sub-routes are eventually built, the actionUrls can be updated.
