# Conv Decisions - 2026-03-30

## 1. PLATO Test Framework — Adopt for Platform Testing
**Type:** Strategic
**Topics:** testing, plato, architecture

**Trigger:** User concern about seed data masking missing creation paths

**Options Considered:**
1. E2E Playwright tests for all write paths — realistic but slow, fragile
2. API flow tests (PLATO) — tests server-side chain, fast, composable ← Chosen
3. Hybrid — both layers per goal

**Decision:** Create PLATO as a new test layer using composable segments with dependency resolution. API tests prove the server chain works; browser tests (future) prove the UI triggers the correct calls.

**Rationale:** Most bugs live in the write path. API flow tests are fast (~200ms per run), catch data-chain bugs, and can generate verified seed data. Browser layer adds UI verification without duplicating API logic.

**Consequences:** New test infrastructure in `tests/plato/`. Design doc at `docs/as-designed/plato.md`. PLAN block tracks implementation.

## 2. Composable Segments over Monolithic Runs
**Type:** Strategic
**Topics:** plato, architecture

**Trigger:** Discussion about reusability, dependency trees, and common ancestors

**Options Considered:**
1. Monolithic runs (self-contained journeys) — simple but duplicate steps
2. Composable segments with dependency resolution ← Chosen

**Decision:** Each segment is an atomic user goal with declared prerequisites. The runner resolves the dependency graph via topological sort. Common ancestors execute once.

**Rationale:** User goals are atomic. The dependency graph IS the platform capability map. Composability enables testing any node as a leaf without duplicating ancestor steps.

**Consequences:** Phase 2's monolithic run needs refactoring into segments. Runner needs dependency resolver.

## 3. No Direct DB Inserts in PLATO
**Type:** Strategic
**Topics:** plato, testing, data-integrity

**Trigger:** User pushed back on `setup` function for creating communities via SQL

**Options Considered:**
1. Allow direct DB inserts for prerequisites — pragmatic but hides gaps ← Rejected
2. Require all data through API calls — strict, surfaces real gaps ← Chosen

**Decision:** If PLATO can't create data through an API, that's a finding to report, not a reason for a workaround.

**Rationale:** The whole point of PLATO is proving data can be created through the app. Direct inserts reproduce the exact problem PLATO exists to solve.

**Consequences:** Found that `POST /api/me/communities` exists and works. No community creation gap.

## 4. Segment Route Declaration (Not Navigation Path)
**Type:** Implementation
**Topics:** plato, browser-testing

**Trigger:** Discussion about how users reach pages and multiple paths to same action

**Options Considered:**
1. Test full navigation paths — comprehensive but fragile, many paths per action
2. Declare route only, verify UI at route ← Chosen

**Decision:** Segments declare WHERE the action happens (route), not HOW the user gets there. Browser tests verify the route has the UI elements. Navigation testing is a separate concern.

**Rationale:** A user can reach `/creating/courses/new` from dashboard, community page, or direct link. The segment cares that the form exists and triggers the right API call, not how the user arrived.

**Consequences:** Browser tests are focused and stable. Navigation coverage handled separately.

## 5. Fix `joined_via` CHECK Constraint
**Type:** Implementation
**Topics:** schema, d1, bug-fix

**Trigger:** PLATO's first run failed because `autoJoinTheCommons` uses `joined_via = 'registration'` which wasn't in the CHECK constraint

**Options Considered:**
1. Change `autoJoinTheCommons` to use `'system'` — semantically wrong
2. Add `'registration'` to CHECK constraint ← Chosen

**Decision:** Added `'registration'` to `community_members.joined_via` CHECK constraint in `0001_schema.sql`.

**Rationale:** `'registration'` is semantically correct — the membership was created as a side effect of user registration, not by manual action or system provisioning.

**Consequences:** Schema change in `migrations/0001_schema.sql`. Needs to be applied to staging/production.

## 6. Happy Path Only for PLATO; Stumbles at Unit/API Layer
**Type:** Strategic
**Topics:** plato, testing, stumble-audit

**Trigger:** User question about whether PLATO should cover error paths

**Options Considered:**
1. Include stumble-and-recover steps in PLATO runs
2. Keep PLATO happy-path only, add STUMBLE-AUDIT block ← Chosen

**Decision:** PLATO tests goal completion. Stumbles (bad input, duplicate emails, wrong passwords) are single-step concerns tested at the unit/API layer. Added STUMBLE-AUDIT PLAN block.

**Rationale:** A stumble happens at one step and recovers at one step — no flow-level concern. The right altitude for testing stumbles is the individual endpoint.

**Consequences:** STUMBLE-AUDIT block added to PLAN.md.
