# Conv Learnings - 2026-03-17

## 1. Bare Datetime Strings Are Ambiguous Across Runtimes
**Topics:** cloudflare, d1, astro

**Context:** Client and teacher both arrived within 2 minutes of a scheduled session and both got "Session time has passed." Investigated the full session join chain.

**Learning:** `new Date("2026-03-20T14:00:00")` (no Z suffix) is parsed as **local time** by the runtime — UTC on Cloudflare Workers, but the user's local timezone in the browser. This means the same bare string represents different absolute instants depending on where it's parsed. The fix is to always store and transmit times as UTC ISO 8601 with explicit Z suffix (`2026-03-20T18:00:00.000Z`).

**Pattern:** Never construct bare datetime strings like `` `${date}T${time}:00` ``. Always use `.toISOString()` (which appends Z) or `localToUTC()` for timezone-aware conversion.

## 2. Teacher Availability Times Need Timezone Conversion at Generation Time
**Topics:** d1, video

**Context:** The availability table stores teacher times as `HH:MM` with an IANA `timezone` column (e.g., `America/New_York`). The `generateSlots()` function used `setHours()` + `toTimeString().slice(0,5)` which operates in the Worker's local time (UTC), ignoring the teacher's timezone entirely.

**Learning:** Availability slot generation must convert teacher-local times to UTC using the stored timezone. The `toTimeString()` approach is broken on Workers because it produces UTC-based HH:MM strings that the browser interprets as local time. The fix uses `Intl.DateTimeFormat` with the `sv-SE` locale trick to compute the UTC offset for any IANA timezone on any date (handles DST correctly).

**Pattern:**
```typescript
// src/lib/timezone.ts — localToUTC()
const guess = new Date(`${dateStr}T${timeStr}:00Z`);
const localStr = guess.toLocaleString('sv-SE', { timeZone: tz });
const localAsUtc = new Date(localStr.replace(' ', 'T') + 'Z');
const offsetMs = localAsUtc.getTime() - guess.getTime();
return new Date(guess.getTime() - offsetMs).toISOString();
```

## 3. SQLite datetime() and strftime() Produce Different Formats
**Topics:** d1

**Context:** Session listing query used `s.scheduled_start > datetime('now')` to filter upcoming sessions. After migrating stored times to Z-suffixed ISO strings, this comparison became fragile.

**Learning:** SQLite's `datetime('now')` produces `2026-03-17 12:00:00` (space separator, no T, no Z). But `.toISOString()` produces `2026-03-17T12:00:00.000Z`. For lexicographic TEXT comparisons to work reliably, both sides must use the same format. Fix: use `strftime('%Y-%m-%dT%H:%M:%SZ', 'now')` which produces ISO 8601 with T and Z.

## 4. Integration Tests Should Cover the Full Timezone Chain
**Topics:** testing

**Context:** After fixing the timezone bug, unit tests and existing integration tests all passed — but none of them actually verified that the full chain (availability → booking → DB → join window) produced correct timezone-aware results.

**Learning:** When fixing a cross-layer bug, write an integration test that exercises the entire data flow end-to-end. The `session-timezone.test.ts` test verifies: (1) NY teacher's 9:00 AM Eastern produces the correct UTC slot, (2) LA teacher's 9:00 AM Pacific produces a different UTC value (exactly 3 hours later), (3) booking stores the UTC values unchanged, (4) the frontend `getInitialState()` logic produces correct view states at various simulated times, (5) bare datetime strings are rejected.

## 5. Shell CWD Drift Breaks Relative-Path Hook Scripts
**Topics:** cc-workflow, dual-repo

**Context:** After running `npm test` (which uses `cd ../Peerloop && ...`), the shell cwd stayed in the code repo. When `/r-end` tried to run, its pre-computed context scripts (`.claude/scripts/*.sh`) failed because they use relative paths from the docs repo root.

**Learning:** The `!` backtick pattern in skill pre-computed context runs from the shell's current working directory, not from the skill's base directory. In a dual-repo setup where commands frequently `cd` to the other repo, the cwd can drift. Skills that use relative-path scripts in pre-computed context are vulnerable to this.
