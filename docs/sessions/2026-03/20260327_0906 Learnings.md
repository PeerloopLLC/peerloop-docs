# Conv Learnings - 2026-03-27

## 1. Peerloop r-end2 was already the evolved version
**Topics:** cc-workflow, dual-repo

**Context:** When comparing spt-docs r-end with peerloop r-end2, expected significant porting work.

**Learning:** The peerloop r-end2 already had all Peerloop-specific adaptations (Block terminology, correct topic routing, Peerloop repo paths). The spt-docs version was structurally identical but with spt references. The "port" was really just path updates (r-end2→r-end, r-docs/scripts→r-end/scripts, /r-resume→/r-start in resume command).

---

## 2. r-end-agent-failed was non-functional
**Topics:** cc-workflow

**Context:** The spt-docs r-end-agent-failed references `subagent_type` for custom agent names (`end-learn-decide`, etc.) that require `.claude/agents/` definitions.

**Learning:** No `.claude/agents/` directory exists in spt-docs, making r-end-agent-failed a dead skeleton. The primary r-end (with inline agent prompts) is the only working variant. Custom `subagent_type` values require agent definition files that were never created.

---

## 3. Consolidation through agent delegation
**Topics:** cc-workflow

**Context:** The core architectural change in the skill migration.

**Learning:** Instead of 8 separate skills called sequentially (r-eos → r-learn-decide → r-dump → ...), the evolved pattern puts all logic into a single r-end that dispatches 3 parallel agents. The refs/ directory lets agents read format instructions without bloating the main SKILL.md. This is a fan-out/fan-in pattern where the Extract file serves as the shared message bus.
