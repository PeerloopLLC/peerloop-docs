## 1. Persist Implementation Plans to Committed Files
**Type:** Strategic
**Topics:** cc-workflow, plan-persistence

## 2. Comprehensive API, Lean Rendering
**Type:** Implementation
**Topics:** api-design, admin, architecture

**Trigger:** Choosing between lean endpoints (evolved iteratively) vs comprehensive endpoints for admin intel.

**Options Considered:**
1. Comprehensive endpoints returning everything, components pick what to display <- Chosen
2. Lean endpoints evolved per-phase as page needs emerge

**Decision:** Intel API endpoints return everything known about an entity from an admin perspective. Components decide what to render. "Leaner" is an optimization task for later.

**Rationale:** User directive: "LEANER IS AN OPTIMIZATION TASK." Starting comprehensive avoids API changes as each phase discovers its needs. Components are the variable — they pick from the full payload.

**Consequences:** Intel type definitions include all available fields upfront. No API changes needed during Phases 2-6.

## 5. AdminBadge Is Round Badge with Count
**Type:** Implementation
**Topics:** ui, admin, badges

**Trigger:** Defining AdminBadge visual design.

**Options Considered:**
1. Simple red dot (present/absent)
2. Round badge like role badges, with count <- Chosen

**Decision:** AdminBadge matches the size/shape of existing role badges (see RoleBadge.tsx), uses red color, displays count. Count provides urgency guidance.

**Rationale:** User: "If you have counts, then counts could be a guide to urgency, and we can upgrade later. So it's less of a dot and more of a round badge like the roles have in size."

**Consequences:** AdminBadge component takes `count` prop, renders round badge matching role badge styling.

## 6. Find Members Page Specification
**Type:** Implementation
**Topics:** admin, discover, members

**Trigger:** Fleshing out /discover/members admin-only page.

**Decision:** Find Members page shows all members by default, has role checkboxes (Student/Teacher/Creator/Moderator/Admin), search by name. Admin-only — client directive prohibits member-to-member search but admin has no such restriction.

**Rationale:** User specified these requirements. Client has said members should not be able to search for other members at launch, but admin needs full visibility.

**Consequences:** DiscoverMembers component uses existing /api/admin/users endpoint with filters. DiscoverSlidePanel gets useCurrentUser() for conditional "Find Members" entry.
