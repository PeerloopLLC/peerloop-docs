# Conv Learnings - 2026-03-19

## 1. Dashboard Endpoints Are the Freshness Mechanism
**Topics:** astro, auth

**Context:** Proposed eliminating redundant dashboard API calls (`/api/me/enrollments`, `/api/me/teacher-dashboard`, etc.) since CurrentUser already carried 60-95% of the same data.

**Learning:** The dashboard endpoints aren't redundant by accident — they're the freshness mechanism. CurrentUser only refreshes on page navigation or tab focus. Without the dashboard's own `fetch()` call, the UI would show stale cached data. Before eliminating any dashboard API call, CurrentUser must be able to detect server-side changes independently.

---

## 2. Version Polling — Practical Staleness Detection Without Server Push
**Topics:** d1, auth

**Context:** Explored how to give CurrentUser a way to know when server data changed, inspired by Meteor's DDP/oplog-tailing pattern.

**Learning:** A monotonic `data_version` counter on the users table + a lightweight polling endpoint (`GET /api/me/version`, ~20 bytes, ~1ms on D1) is sufficient. The client polls every 30s; version mismatch triggers `refreshCurrentUser()`. This closes the staleness gap without WebSockets, SSE, or Durable Objects. The `data_version` column is compatible with future transport upgrades — only the client-side transport changes, the counter stays.

**Pattern:** `bumpUserDataVersion(db, userId)` called in 31 mutation endpoints. Client-side `startVersionPolling()` in `initializeCurrentUser()`, `stopVersionPolling()` in `clearCurrentUser()`.

---

## 3. Scope Boundary for Version Bumps
**Topics:** d1, auth

**Context:** Needed to decide which mutations should bump `data_version` vs which should not.

**Learning:** Only bump for data CurrentUser actually carries: identity, capabilities, enrollments, teacher certs, created courses, stats, community moderations, unread counts. Do NOT bump for: course content, session schedules, earnings, feed activity, other users' data. These are fetched by page-level dashboard APIs at navigation time.

## 4. E2E Seed Data Verification via page.evaluate()
**Topics:** testing

**Context:** Needed to verify that seed data correctly flows through the pipeline: seed SQL → D1 → `/api/me/full` → CurrentUser.

**Learning:** Playwright's `page.evaluate()` can call CurrentUser methods inside the browser context and return a plain summary object. Since CurrentUser is a class with private Maps, you can't just serialize it — instead, call its public methods (`getEnrollments()`, `getActiveTeacherCertifications()`, etc.) inside `evaluate()` and map to IDs for easy assertion. The `.first()` selector is needed for course titles that appear in multiple page sections.

**Pattern:** `snapshotCurrentUser(page)` helper + `navigateAndSnapshot(page, route)` for navigating with feed mocking + waiting for currentUser hydration.

---

## 5. Redundancy Analysis Methodology
**Topics:** astro, testing

**Context:** Needed to determine whether dashboard API calls were truly redundant or serving a purpose.

**Learning:** Field-by-field comparison (API returns vs component uses vs CurrentUser has) revealed three categories: (1) fully redundant (CurrentUser has it, component uses it), (2) genuinely unique (earnings, session lists, pending counts), (3) enrichable (3 fields missing from UserEnrollment: `hasReview`, `courseDuration`, `courseSessionCount`). The `/api/me/enrollments` endpoint was ~95% redundant, while `/api/me/teacher-dashboard` was only ~60% redundant because it serves operational data CurrentUser shouldn't carry.
