# Conv Learnings - 2026-03-28

## 1. EXISTS Subquery Pattern for Taxonomy Filtering
**Topics:** sql, d1, tag-taxonomy

**Context:** With `category_id` removed from courses, filtering by topic requires traversing `course_tags → tags → topics`. A JOIN would produce duplicate rows when a course has multiple tags under one topic.

**Learning:** Use `EXISTS (SELECT 1 FROM course_tags ct JOIN tags tg ON ct.tag_id = tg.id JOIN topics tp ON tg.topic_id = tp.id WHERE ct.course_id = c.id AND tp.slug = ?)` to filter without duplicates. This is both correct and efficient — SQLite optimizes EXISTS as a semi-join.

**Pattern:** All topic-based course filtering uses this EXISTS pattern consistently across 5 endpoints.

## 2. Related Courses via Shared Tags is Better Than Category Match
**Topics:** sql, recommendations, tag-taxonomy

**Context:** The old "related courses" query used `WHERE c.category_id = ?` — a single FK match. With multi-tag courses, the replacement uses a self-join on course_tags.

**Learning:** `EXISTS (SELECT 1 FROM course_tags ct1 JOIN course_tags ct2 ON ct1.tag_id = ct2.tag_id WHERE ct1.course_id = c.id AND ct2.course_id = ?)` finds courses sharing any tag with the current course. This is semantically richer than category matching — a course tagged "Python" and "Machine Learning" will match courses tagged with either, not just courses in the same broad category.

## 3. Onboarding Profile Simplified Dramatically
**Topics:** api, onboarding, tag-taxonomy

**Context:** The old onboarding POST handler managed 3 tables: `member_profiles`, `user_topic_interests` (with experience levels), and `user_interests` (backward-compat sync). The new system uses just 2: `member_profiles` and `user_tags`.

**Learning:** The `experience_level` per topic concept and the `user_interests` backward-compat sync were both dropped in the TAG-TAXONOMY refactor. The POST handler went from ~100 lines of insert logic to ~15 lines. Simpler data model = simpler code.

---

## 4. Recommendation Scoring Collapsed to Single Dimension
**Topics:** recommendations, tag-taxonomy

**Context:** Old scoring: 80pts for category match + up to 20pts for tag overlap (via `user_interests.tag ↔ course_tags.tag` string matching). New scoring: up to 100pts pure tag overlap (`min(overlap, 5) × 20`).

**Learning:** With FK-based course_tags and user_tags pointing to the same `tags` table, there's no need for two scoring dimensions. The category match was just a coarse proxy for tag overlap. Phase 4 will add graduated scoring (topic-level weighting), but the single-dimension approach is already more precise than the old dual system.
