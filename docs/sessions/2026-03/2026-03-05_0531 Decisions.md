# Session Decisions - 2026-03-05

## 1. Positional Module Assignment (Replaces Stored module_id)
**Type:** Architecture
**Topics:** d1, architecture

**Trigger:** Session 325 planned adding `module_id` as a NOT NULL column stored at booking time. During Session 331 design discussion, realized that cancellations cause the stored module_id to go stale — remaining sessions shift which module they cover.

**Options Considered:**
1. Store `module_id` at booking time (Session 325 plan) — goes stale on cancel/rebook
2. Positional computation at read time — always correct, modules reflow automatically ← Chosen
3. Store `module_id` and update all sessions on every cancel — complex, error-prone

**Decision:** Module assignment is computed positionally at read time. Sort an enrollment's non-cancelled sessions by `scheduled_start ASC`; the Nth session teaches the Nth module (by `module_order`). Completed sessions have `module_id` frozen (written to DB on completion). Scheduled sessions reflow dynamically.

**Rationale:** Cancelling session 2 of 3 should shift session 3 to cover module 2, not leave module 2 orphaned. Late binding keeps module order always sequential without any data migration on cancellation. The computation cost (3 queries + zip) is negligible for enrollment-scoped data (3-10 sessions).

**Consequences:** Created `src/lib/booking.ts` with `resolveModuleAssignments()` as single source of truth. `module_id` column added as nullable (NULL while scheduled, set on completion). Session limit enforcement: 422 when `completed + scheduled >= module count`. All API responses (GET list, GET detail, POST create) include computed module info.

## 2. Session Limit Enforcement via Module Count
**Type:** Implementation
**Topics:** d1, api

**Trigger:** `courses.session_count` existed but was never enforced in the booking API.

**Options Considered:**
1. Enforce against `courses.session_count` field
2. Enforce against actual curriculum module count ← Chosen
3. No enforcement (status quo)

**Decision:** Booking API returns 422 when `completed_sessions + scheduled_sessions >= curriculum_module_count`. Cancelled sessions free the slot (don't count toward limit).

**Rationale:** Module count is the authoritative source — it reflects actual course structure. `session_count` is informational/display-only. Counting only `completed + scheduled` (excluding `cancelled`) means cancellations naturally free slots for rebooking.

**Consequences:** `POST /api/sessions` now checks eligibility before conflict detection. Tests that create sessions need `course_curriculum` entries to avoid hitting the limit.
