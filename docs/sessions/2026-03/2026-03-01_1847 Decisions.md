# Session Decisions - 2026-03-01 (Session 320)

## 1. Centralized useCreatorGate Hook for Client-Side Access
**Type:** Implementation
**Topics:** auth, astro

**Trigger:** 5 creator page components each handled creator access differently — some checked 403, one pre-fetched `/api/me/courses` just for a count, two had no check at all. Inconsistent UX and wasted network requests.

**Options Considered:**
1. Fix each component individually (keep scattered patterns)
2. Create a shared `useCreatorGate` hook that reads `CurrentUser` global state ← Chosen
3. Add a route-level middleware/guard in Astro

**Decision:** All `/creating/*` page components use `useCreatorGate()` for client-side access gating. The hook reads from the already-cached `CurrentUser` instance (zero network cost) and returns `{ status, hasCourses }`.

**Rationale:** The `CurrentUser` global already has `canCreateCourses` and `hasCreatedCourses()` — making a separate API call for this information is redundant. A hook centralizes the gate logic, ensures consistent "Creator Access Required" UX across all pages, and eliminates the extra `/api/me/courses` round-trip from CreatorAnalytics.

**Consequence:** 5 components updated, 1 redundant API call eliminated, consistent gate UI across all creator pages. Server-side gates remain unchanged as the security enforcement layer.

## 2. vi.fn() Pattern for Testable Module Mocks
**Type:** Implementation
**Topics:** testing

**Trigger:** Component tests needed to mock `useCreatorGate` with a default return value for most tests, but override it per-test for gate-state tests.

**Options Considered:**
1. Plain function in `vi.mock` factory — no per-test override possible
2. Mutable variable pattern (like in `useCreatorGate.test.ts`) — verbose, needs explicit reset
3. `vi.fn()` in factory with `vi.mocked().mockReturnValue()` per-test ← Chosen

**Decision:** Use `vi.fn(() => defaultValue)` in `vi.mock` factories for hooks/functions that need per-test behavior variation. Override via `vi.mocked(fn).mockReturnValue(...)`. Rely on `vi.clearAllMocks()` in global `beforeEach` for automatic reset.

**Rationale:** Cleanest pattern — one line to declare default, one line to override per-test, zero cleanup code. Works with the existing `vi.clearAllMocks()` convention already used in all component test files.

**Consequence:** Adopted in 4 component test files. Established as the recommended pattern for future hook mocking.
