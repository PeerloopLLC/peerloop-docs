# Conv Decisions - 2026-03-19

## 1. Replace FeedSlidePanel with /feeds full page
**Type:** Strategic
**Topics:** astro, stream

**Trigger:** Client directive: feeds = 50% of learning. A 320px slideout doesn't convey that importance.

**Options Considered:**
1. Keep slideout as primary, add /feeds as secondary ← Rejected
2. Replace slideout with /feeds page link ← Chosen
3. Keep both with keyboard shortcut for slideout ← Rejected (complexity)

**Decision:** The navbar "My Feeds" now links to `/feeds` (full page). FeedSlidePanel removed from AppNavbar. File retained but orphaned — nothing imports it.

**Rationale:** A primary learning surface needs a full page with prominent styling, search, and room for activity indicators. The slideout was appropriate when feeds were a navigation utility, not a learning destination.

**Consequences:** FeedSlidePanel.tsx is orphaned (pending deletion). Home action for authenticated students changed from opening slideout to navigating to `/feeds`.

## 2. FEED-INTEL: D1 activity index alongside Stream.io (CQRS hybrid)
**Type:** Strategic
**Topics:** stream, d1

**Trigger:** Stream.io flat feeds can't provide unread counts, cross-feed aggregation, or "new since last visit" queries. Blocking the feeds-as-learning-surface vision.

**Options Considered:**
1. Pure Stream — accept no badges ← Rejected (blocks client vision)
2. Pure custom — rebuild feed storage ← Rejected (massive undertaking)
3. Hybrid: Stream stores content, D1 stores metadata index ← Chosen

**Decision:** Implemented CQRS pattern: Stream remains durable content store (posts, reactions, comments). D1 gets two new tables (`feed_visits`, `feed_activities`) as a thin metadata index (~150 bytes/row). Write-time indexing (no polling). Badge counts via single D1 query.

**Rationale:** Each system does what it's good at. Stream handles fan-out, reactions, threading. D1 handles cross-feed queries, unread counts, and (future) smart surfacing. The D1 index is rebuildable from Stream — it's not a new source of truth.

**Consequences:** New schema migration (0004_feed_activity_index.sql). Dual-write in 3 post endpoints. Visit recording in 3 feed GET endpoints. New API endpoint `/api/me/feed-badges`. New FEED-INTEL block in PLAN.md for future phases.

> **Insight:** This is a textbook CQRS pattern — the write model (Stream) optimizes for durability and fan-out, the read model (D1) optimizes for the specific queries the UI needs. The key insight is that the D1 index is *expendable* — if it's lost, it can be rebuilt from Stream. This makes the dual-write safe: a failed D1 insert means a badge count is off by one, not data loss.

## 3. Visit-based badge clearing (not per-post read tracking)
**Type:** Implementation
**Topics:** stream, d1

**Trigger:** How to clear the "N new" badge when a user checks a feed.

**Options Considered:**
1. Per-post read tracking (mark each post individually) ← Rejected (complex, unnecessary)
2. Visit-based: visiting feed sets `last_visited_at = NOW()`, all posts become "seen" ← Chosen

**Decision:** Visiting a feed page (offset=0, i.e., not pagination) upserts `feed_visits.last_visited_at`. All posts before that timestamp are considered "seen." Badge clears entirely on visit.

**Rationale:** Standard pattern (Slack channels, Reddit subreddits, Discord). Per-post tracking adds complexity with no user benefit at Genesis scale. Users want to know "something happened" — not to mark individual posts as read.

**Consequences:** Simple upsert on feed GET handlers. Badge count is always "new since last visit" — no partial read states.

## 4. Comment dual-write deferred — posts only for Genesis
**Type:** Implementation
**Topics:** stream, d1

**Trigger:** Comment endpoints don't carry feed slug context, making dual-write impossible without refactoring.

**Options Considered:**
1. Refactor comment endpoints to pass feed slug ← Deferred
2. Look up feed from Stream activity ID ← Complex, extra API call
3. Skip comment indexing for Genesis ← Chosen

**Decision:** Only top-level posts are dual-written to `feed_activities`. Comments/replies don't generate badge counts. Acceptable at Genesis scale (low volume).

**Rationale:** Comment endpoints operate on activity IDs, not feed slugs. Adding slug context requires changing the request contract. At Genesis scale (~80 users), the badge count being off by reply count is not noticeable. Tracked as follow-up task.

**Consequences:** A reply to a post won't bump the badge for that feed. Only new top-level posts trigger badges. Follow-up task created for future conv.
