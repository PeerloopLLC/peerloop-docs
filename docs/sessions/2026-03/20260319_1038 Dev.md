### Staleness & Version Polling Discussion

**User:** "Before you start how does the hook useCurrentUser() know it needs to refresh currentUser global?"

**Claude:** Traced the full pub/sub chain: `setCurrentUser()` → `notifyListeners()` → `useCurrentUser()` hook's `setUser` setState. Explained this is a manual pub/sub pattern for Astro islands.

**User:** "What triggers the pib to message the sub that a change it cares about has happened on the server?"

**Claude:** Revealed the key gap: nothing does. No server-push mechanism. CurrentUser only refreshes on page navigation, window focus, or explicit `refreshCurrentUser()` call. No mutation triggers a refresh today.

**User:** "We keep learning new things to adjust based on my questions (which is fine) but it leads me to feel we have a half-baked solution that more (unknown) questions can only fix. From what I can see we are no longer saving any server api calls, we have the same or about the same as before..."

**Claude:** Acknowledged the reality: the proposed refactor wouldn't reduce API calls — it was shuffling which fields come from where. Dashboard endpoints are the freshness mechanism. Revised recommendations: strip down to what delivers value, drop dashboard refactoring.

**User:** "THis is a good discussion, we won't decide yet as I have some other ideas. I used to work with MeteorJS where the pub/sub was truly coming from the server. Meteor tailed the Mongo database for changes and passed those chnages to the client via a separate push process... Can we build some sort of polling into the currentUser/networkStatus golbal mechanism that would trigger updating currentUser?"

**Claude:** Proposed version polling: `data_version` counter on users table, `GET /api/me/version` endpoint (~20 bytes), client polls every 30s. Explained this is Meteor's pattern minus the push transport.

**User:** "It does feel effective at solving our current issues. Can we eliminate the noise of changes to the db that don;t affect the user?... I guess what I am saying is that if we use this 30 second polling to keep currentUser less stale and leave the rest alone we could have a good system"

**Claude:** Defined the scope boundary: only bump for data CurrentUser carries. Operational data (sessions, earnings, feeds) stays with page-level APIs. Accepted unread count noise.

**User:** "THis does feel right. and accept the noise for unread messages only. Review the other api changes you were going to make when we were first looking at this this conv and streeamline them if that still has value"

**Claude:** Streamlined to 5 phases: version polling foundation, StudentDashboard enrollment enrichment (-1 API call), community count fold (-1 API call), community memberships + feed index, My Feeds component. Dropped TeacherDashboard/CreatorDashboard refactoring entirely.
