# Conv Decisions - 2026-03-18

## 1. Enrollment Guards — Block vs Warn Strategy
**Type:** Strategic
**Topics:** d1, astro

**Trigger:** Need to handle multiple enrollment edge cases: creator self-enrollment, active teacher self-enrollment, no teachers for course, no availability in time window.

**Options Considered:**
1. Block all problematic cases with 400 errors ← Partially chosen
2. Warn for all cases but allow enrollment
3. Mixed approach: hard block some, warn others ← Chosen

**Decision:** Creator self-enrollment, active teacher self-enrollment, duplicate active enrollment, and zero teachers for a course are HARD BLOCKS (400 error). Teachers exist but no availability in the configurable window is a WARN ONLY — the student can still enroll because teachers may update their schedules.

**Rationale:** Hard blocks prevent situations that break business logic (creator paying themselves, duplicate enrollments). The "no availability" case is soft because availability is dynamic — a teacher might add slots tomorrow. Zero teachers is different: there's literally nobody to teach the course, so enrollment would be pointless.

**Consequences:** `enrollment-guards.ts` returns a typed result with `reason` for blocks. The `no_teachers` block also triggers notifications to the creator and all admins.

## 2. Re-Enrollment as New Row with Partial Unique Index
**Type:** Implementation
**Topics:** d1

**Trigger:** User wants students to be able to retake completed courses, while preventing duplicate active enrollments.

**Options Considered:**
1. Reset existing enrollment row (status back to 'enrolled', clear progress)
2. New enrollment row per retake, unique constraint on (student_id, course_id) ← Rejected
3. New enrollment row per retake, PARTIAL unique index on active statuses only ← Chosen

**Decision:** Allow multiple enrollment rows for the same student+course. Completed/cancelled rows are retained as history. A partial unique index (`WHERE status IN ('enrolled', 'in_progress')`) enforces at most one active enrollment at the database level.

**Rationale:** Option 1 loses completion history. Option 2 blocks re-enrollment entirely. The partial unique index is the cleanest D1/SQLite solution — it enforces the business rule at the DB level while preserving all enrollment history.

**Consequences:** Changed duplicate checks in checkout and admin endpoints from `status != 'cancelled'` to `status IN ('enrolled', 'in_progress')`. Updated `CurrentUser` enrollment Map to prefer active enrollments with a priority function. Updated 4 test files for index compatibility.

> **Insight:** SQLite's partial indexes (`CREATE UNIQUE INDEX ... WHERE condition`) are underused but powerful for "at most one active" patterns. They're more reliable than application-level checks because they survive race conditions and direct DB access.

## 3. Availability Summary Endpoint — Public with Optional Auth
**Type:** Implementation
**Topics:** auth, astro

**Trigger:** Need to show teacher availability on the course page before enrollment. Should this require authentication?

**Options Considered:**
1. Require auth — only logged-in users see availability
2. Public endpoint, no auth at all
3. Public endpoint with optional auth to exclude self from teacher list ← Chosen

**Decision:** `GET /api/courses/:id/availability-summary` is public (no auth required). If the requesting user happens to be authenticated (cookie present), they are excluded from the teacher list so a teacher doesn't see themselves.

**Rationale:** Teacher names, ratings, and student counts are already public on the course page. Slot counts and next-available dates add minimal privacy risk. Requiring auth would prevent anonymous visitors from making informed enrollment decisions. Optional auth gives the best of both worlds.

**Consequences:** Endpoint reads session cookie opportunistically in a try/catch — auth failure doesn't block the response. The filtering (`teachers.filter(t => t.user_id !== currentUserId)`) only activates when a user is identified.

## 4. Availability Window Configurable via platform_stats KV Table
**Type:** Implementation
**Topics:** d1

**Trigger:** User wants the 30-day availability preview window to be admin-configurable.

**Options Considered:**
1. Hardcode 30 days
2. Add a dedicated `platform_settings` table
3. Use existing `platform_stats` KV table ← Chosen

**Decision:** Store `availability_window_days` as a row in the existing `platform_stats` table (key-value store). Default value: '30'. The availability summary endpoint reads it at runtime.

**Rationale:** `platform_stats` already exists as a generic KV store with the right schema (key, value, label, display_order). No new table needed. Admin UI for editing this value is deferred to a future block.

**Consequences:** Added seed row `('stat-006', 'availability_window_days', '30', 'Availability preview window (days)', 6)` to `0002_seed_core.sql`. Endpoint falls back to 30 if the setting is missing.

## 5. Inactive Teacher Treated as Regular Student for Enrollment
**Type:** Strategic
**Topics:** d1

**Trigger:** Should a teacher who has been deactivated (`is_active=0`) or paused teaching (`teaching_active=0`) be blocked from enrolling as a student?

**Options Considered:**
1. Block all teachers (active or not) from enrolling
2. Allow inactive/paused teachers to enroll as students ← Chosen

**Decision:** Only teachers with BOTH `is_active=1` AND `teaching_active=1` are blocked from self-enrolling. Deactivated or paused teachers are treated as regular students and can enroll.

**Rationale:** Per the user: "An inactive teacher is the same as a student from the POV of the course." A teacher who has stopped teaching might want to retake the course as a refresher. Only actively-teaching teachers are blocked because they already have access to course materials and booking themselves would break the payment model.

**Consequences:** Added `teachingActive` field to `UserTeacherCertification` and `isActivelyTeachingFor()` method to `CurrentUser`. The enrollment guard checks both flags. Updated `/api/me/full` to include `teaching_active` in the teacher certifications query.
