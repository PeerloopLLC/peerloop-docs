# Conv Decisions - 2026-03-19

## 1. Deprecate `now()` in Favor of `toUTCISOString()`
**Type:** Implementation
**Topics:** d1

**Trigger:** `src/lib/db/index.ts` exports `now()` which does `new Date().toISOString()` — identical to `toUTCISOString()` in `timezone.ts`. Having two functions for the same operation creates cognitive load and import confusion.

**Options Considered:**
1. Keep both, document the equivalence
2. Deprecate `now()`, migrate all 17 callers to `toUTCISOString()` ← Chosen
3. Delete `now()` immediately

**Decision:** Deprecate `now()` with `@deprecated` JSDoc tag, migrate all 17 importing files to `toUTCISOString()` from `timezone.ts`. Function body kept for safety during transition.

**Rationale:** One canonical function, zero cognitive load. `toUTCISOString()` lives in `timezone.ts` alongside all other date utilities, making it the natural home. Keeping the deprecated function body prevents breakage if any code was missed.

**Consequences:** 17 files migrated. `now()` still exports but is marked deprecated. Future code should only use `toUTCISOString()`.

## 2. Use `%f` (Milliseconds) in Schema Defaults
**Type:** Implementation
**Topics:** d1

**Trigger:** Schema defaults using `strftime('%Y-%m-%dT%H:%M:%SZ', 'now')` produce `2026-03-18T23:41:25Z` (no ms), while JS `toISOString()` produces `2026-03-18T23:41:25.000Z` (with `.000`). Both are valid ISO 8601 but differ.

**Options Considered:**
1. Use `%S` (no milliseconds) — accept minor format difference ← Rejected
2. Use `%f` (SS.SSS) — exact match with JS output ← Chosen

**Decision:** Use `strftime('%Y-%m-%dT%H:%M:%fZ', 'now')` for all 66 schema defaults.

**Rationale:** Exact format match between SQL defaults and JS-generated timestamps eliminates any edge cases in string comparison or display. `parseUTCDate()` handles both anyway, but consistency is free.

**Consequences:** All schema defaults now produce `.SSSZ` suffix matching JavaScript exactly.
