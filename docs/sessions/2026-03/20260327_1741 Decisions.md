# Conv Decisions - 2026-03-27

## 1. New Routes for Side-by-Side Comparison
**Type:** Strategic
**Topics:** url-routing, architecture

**Trigger:** User wanted to build role-aware course pages without modifying existing pages

**Options Considered:**
1. Modify existing `/discover/courses` and `/course/[slug]` in-place
2. Create new routes `/explore/courses` and `/explore/course/[slug]` ← Chosen

**Decision:** Build on new `/explore/` routes to compare side-by-side with existing pages before deciding which to keep

**Rationale:** Avoids risk to existing pages; allows A/B comparison; easy rollback (delete new files)

**Consequences:** 25 new files created; no existing files modified (except 1 accessor method on current-user.ts)

## 2. Hybrid Tab+Pill Filter Model for Listing Page
**Type:** Strategic
**Topics:** ux-design, course-listing

**Trigger:** User wanted role-relative filtering but also focused role perspectives

**Options Considered:**
1. Pills only (multi-select across all views)
2. Tabs only (mutually exclusive role views)
3. Hybrid: Tabs for role perspectives + pills inside "All" tab only ← Chosen

**Decision:** "All" tab has multi-select pill filters for combinatorial role filtering; role tabs (Student, Teaching, Created, Moderating) show focused perspectives with role-optimized card layouts

**Rationale:** User's insight: "the job of the All course view is to filter in combinations, but the roles of the other tabs is to show in the perspective of the tab's role"

**Consequences:** Each role tab has its own card layout and sub-filters (active/inactive states); "All" tab reuses standard browse grid with role badge overlay

## 3. Detail Page: Approach 1 — Role Section Divider
**Type:** Strategic
**Topics:** ux-design, course-detail

**Trigger:** Need to add role-specific tabs without crowding the existing tab bar

**Options Considered:**
1. Role section divider — universal tabs left, role tabs right with separator ← Chosen
2. Overflow dropdown — role tabs hidden in "More" menu
3. Two-row tabs — universal on top, role bar below
4. Hero buttons — no extra tabs, just action buttons in hero

**Decision:** Universal tabs stay as-is; role-specific tabs appear in a separate section below with colored divider and role label

**Rationale:** Preserves existing tab bar; role section is visually distinct; implemented by wrapping CourseTabs (no copy needed)

**Consequences:** Role tabs lazy-load data from existing API endpoints; section only appears for users with course relationships

## 4. No New API Endpoints — Client-Side Annotation
**Type:** Implementation
**Topics:** api-design, data-strategy, currentUser

**Trigger:** Planning the data strategy for the listing page

**Options Considered:**
1. New `/api/explore/courses` endpoint that returns courses with role annotations
2. Client-side annotation using CurrentUser data already in memory ← Chosen

**Decision:** The listing page SSR-fetches the public course catalog (same query as `/discover/courses`), then annotates each course with role badges on the client using CurrentUser's O(1) Map lookups

**Rationale:** CurrentUser already has all enrollment, certification, created course, and moderation data. Adding a new API would duplicate data and add a round-trip.

**Consequences:** Zero new API endpoints; role tabs (Student, Teaching, Created) pull directly from CurrentUser getters

## 5. Active/Inactive State Distinction Per Role
**Type:** Implementation
**Topics:** ux-design, data-model

**Trigger:** User pointed out that teachers/students can be active or past

**Options Considered:**
1. Show only active roles
2. Show all roles with active/inactive state ← Chosen

**Decision:** Each role badge and role tab distinguishes active vs inactive states: Student (Enrolled/Completed), Teacher (Teaching/Paused/Revoked), Creator (Published/Draft/Retired)

**Rationale:** A completed student and an in-progress student have very different needs; same for active vs paused teachers

**Consequences:** `getRoleBadges()` returns `isActive` flag per badge; role tabs show different sub-filter options and different tab content based on state
