# Session Learnings - 2026-03-04

## 1. Nullable FK Columns Don't Break Existing Tests
**Topics:** d1, testing

**Context:** Added `st_certification_id` column to enrollments table. Expected to need test updates across many files that INSERT into enrollments.

**Learning:** Because the column is `TEXT REFERENCES student_teachers(id) ON DELETE SET NULL` (nullable, no NOT NULL constraint), all existing INSERT statements that don't include it simply get NULL. All 5465 tests passed without any test file modifications.

**Pattern:** When adding audit/tracking columns to existing tables, make them nullable. This avoids a migration-style "update all consumers" burden and lets you incrementally populate the new column where it matters.

## 2. Stripe Metadata as Transient vs Persistent Data
**Topics:** stripe, d1

**Context:** The checkout flow already carried both `student_teacher_id` (st-xxx, the certification record) and `student_teacher_user_id` (usr-xxx, the user FK) in Stripe metadata. But only the user FK was persisted to the database.

**Learning:** Stripe metadata is transient — it exists on the checkout session and webhook payload, but once consumed, it's gone from the application's perspective. Any metadata field needed for future queries or audit must be explicitly persisted to the database at consumption time. The CERT-AUDIT block formalized this by storing `st-xxx` alongside `usr-xxx`.
