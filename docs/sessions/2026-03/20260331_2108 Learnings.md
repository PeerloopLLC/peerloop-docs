## 1. STUMBLE-AUDIT as Manual QA Walkthrough
**Topics:** testing, plato, ux

**Context:** User reframed STUMBLE-AUDIT from "audit test coverage of error paths" to "structured manual walkthrough of PLATO runs in the browser"

**Learning:** PLATO runs serve as a natural checklist for manual QA. Each run defines the journey; the browser walk reveals UX stumbles that automated tests can't catch (confusing labels, missing feedback, dead-end navigation). Fix-as-you-find with clean-room restart is the chosen workflow.

**Pattern:** Walk run → find issue → fix it → reset DB → restart dev server → walk from top. Don't sign off until clean end-to-end.

## 2. Entry Point Mapping Anchors PLATO Runs
**Topics:** plato, ux, testing

**Context:** PLATO run says "visit /signup" but a real user doesn't type URLs — they click something somewhere in the app

**Learning:** Before walking a PLATO run, map every entry point in the codebase that leads to the run's target route. This anchors the automated run to real user navigation paths and surfaces dead params, wrong redirects, and missing links.

## 3. Client-Side Import Side Effects in Astro
**Topics:** astro, bundling, auth

**Context:** Tried to import `validatePassword` from `@lib/auth` in SignupForm (client component)

**Learning:** Importing from a module with top-level side effects (e.g., `import bcrypt from 'bcryptjs'` in `password.ts`) pulls the entire module into the client bundle, even if the specific function doesn't use bcrypt. In Astro's multi-root architecture, duplicate pure validation logic in client components rather than importing from server modules with heavy dependencies. Add a comment explaining why to prevent future "cleanup" that reintroduces the import.

## 4. Three Handle Validators — Unification Needed
**Topics:** auth, validation

**Context:** Found 3 different handle validation patterns across SignupForm (client), auth/index.ts (register), and api/me/profile.ts (update)

**Learning:** Validation rules for the same field must have a single source of truth. When multiple validators exist with different rules (max length 20 vs 30, hyphens allowed vs not, leading digit rules differ), users encounter inconsistent behavior — they can set a handle in profile that they couldn't have used at registration. Task #8 tracks unification.

## 5. SSR vs React Island Decision for Auth-Dependent Content
**Topics:** astro, auth, performance

**Context:** Needed to differentiate home page content for visitors vs new members vs established members

**Learning:** With `output: 'server'` in Astro config, all pages are SSR — no static caching concern. Server-side auth checks in frontmatter are zero-cost compared to React islands. However, `CurrentUser` is already loaded client-side by AppNavbar, so auth-dependent content can also be handled via React islands without additional API calls. The choice depends on whether you need the differentiation before first paint (SSR) or can accept a flash (client-side).
