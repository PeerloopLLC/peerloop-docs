# Conv Decisions - 2026-03-18

## 1. Static SVG Placeholder for User Avatars
**Type:** Implementation
**Topics:** d1, astro

**Trigger:** All 10 seed users had NULL `avatar_url`, breaking visual testing of profiles, cards, and avatar components.

**Options Considered:**
1. UI Avatars service (external URL, letter-based, deterministic)
2. DiceBear SVG avatars (external URL, cartoon-style)
3. Static placeholder SVG in `public/images/` ← Chosen

**Decision:** Use a single static `/images/default-avatar.svg` checked into the repo for all seed users.

**Rationale:** No external dependency, works offline, renders everywhere, zero build cost. Real avatar upload is tracked as a future feature in FILE-UPLOADS block.

**Consequences:** Created `public/images/default-avatar.svg` (simple silhouette). All 10 dev seed users reference it. Production avatar upload will replace this per-user via R2.

## 2. Seed Data Gaps Routed to Existing vs New PLAN Blocks
**Type:** Strategic
**Topics:** workflow

**Trigger:** Seed data audit surfaced 3 feature gaps that needed tracking: avatar upload, recording persistence, certificate PDF generation.

**Options Considered:**
1. Create 3 new standalone deferred blocks
2. Fit items into existing blocks where possible ← Chosen

**Decision:** Avatar upload → FILE-UPLOADS (#23, existing). Certificate PDF → CERT-APPROVAL (#22, existing). Recording persistence → RECORDING-PERSIST (#31, new block).

**Rationale:** Avatar upload is literally "profile photos" which FILE-UPLOADS already covers. Certificate PDF is the missing piece of the CERT-APPROVAL pipeline. Only recording persistence had no existing block — BBB webhook handling is its own concern, distinct from video session management.

**Consequences:** Updated descriptions for FILE-UPLOADS and CERT-APPROVAL. Added subtask sections for both. Created new RECORDING-PERSIST block with VERIFY and R2-STORAGE subtasks.

## 3. Gabriel's Availability Uses Asia/Jerusalem Timezone
**Type:** Implementation
**Topics:** d1

**Trigger:** Gabriel Rymberg (creator) had no availability data despite being a teacher, preventing booking tests.

**Options Considered:**
1. Copy an existing teacher's US timezone slots
2. Use Israel timezone matching Gabriel's profile location (Tel Aviv) ← Chosen

**Decision:** Added Tue/Thu 10:00-18:00 in `Asia/Jerusalem` timezone.

**Rationale:** Gabriel's profile has `location: Tel Aviv, Israel`. Using a non-US timezone adds diversity to the seed data, which helps test timezone-aware scheduling features.

**Consequences:** Seed data now has 3 timezones: America/New_York (Sarah), America/Chicago (Marcus), Asia/Jerusalem (Gabriel).
