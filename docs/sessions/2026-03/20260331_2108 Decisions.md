# Conv Decisions - 2026-03-31

## 1. Remove Username Field from Signup Form
**Type:** Strategic
**Topics:** auth, ux, signup

**Trigger:** User observed handle field caused confusion — auto-generation was unclear, collision UX was poor, field felt optional but was required, 3 different validators existed

**Options Considered:**
1. Keep field, improve helper text and validation
2. Remove field, auto-generate server-side with collision suffix ← Chosen
3. Remove field, use random suffix (e.g., testwalker_8f3k)

**Decision:** Username/handle field removed from signup. Server auto-generates from name with incrementing collision suffix. Users change handle later in profile settings.

**Rationale:** Reduces signup friction (5 fields → 4), eliminates 4 open issues (handle helper text, uniqueness check, collision UX, validation mismatch at signup), follows Twitter/Discord pattern. The handle field had the most stumble potential of any signup field.

**Consequences:** 10 files changed. PLATO runs updated. API test suite rewritten for auto-generation tests. Client no longer sends handle to register endpoint. Profile settings becomes the only place to customize handle.

> **Insight:** When a form field generates multiple UX issues (confusion, validation mismatch, collision handling), removing it entirely is often better than fixing each issue individually. Auto-generation with later customization follows established patterns (Twitter, Discord) and reduces signup abandonment risk.

## 2. Fix-As-You-Find Workflow for STUMBLE-AUDIT
**Type:** Strategic
**Topics:** testing, workflow, plato

**Trigger:** User debated whether to fix issues during walkthrough or defer them

**Options Considered:**
1. Fix immediately, restart walkthrough ← Chosen
2. Defer to after walkthrough (risk losing context)
3. Log with fix context, batch after each run

**Decision:** Fix issues as discovered, then restart the walkthrough from scratch to get a clean run.

**Rationale:** STUMBLE-AUDIT is a one-time deep pass per PLATO run. Once walked clean, it's trusted until the API-level PLATO test breaks. Restarting is cheap. Context is valuable.

**Consequences:** Each PLATO run walkthrough may take multiple iterations. The "clean room" sign-off means completed runs are genuine quality gates.

## 3. Post-Signup Redirect to /onboarding
**Type:** Implementation
**Topics:** auth, ux, onboarding

**Trigger:** New members landed on the same generic home page as visitors after signup

**Options Considered:**
1. Redirect to /onboarding after signup ← Chosen
2. Show different home page content for new members (deferred — needs client input)
3. No redirect, rely on sidebar "Complete Your Profile" link

**Decision:** `handleAuthSuccess()` now checks `wasSignup` flag. Signup without a pending action redirects to `/onboarding`. Login behavior unchanged.

**Rationale:** The onboarding page already exists and captures interests/preferences. Directing new users there immediately is the lowest-friction way to improve the post-signup experience without client input on home page redesign.

**Consequences:** `auth-modal.ts` modified. Login vs signup now have different default destinations. If a pending action exists (e.g., returnUrl from ModeratorInvite), it takes priority over the onboarding redirect.
