# Conv Learnings - 2026-03-20

## 1. Stream.io Activities Store User Data at Write Time
**Topics:** stream, astro

**Context:** Feed posts showed inconsistent user avatars — some posts by the same user had an image, others didn't. Older posts (created before `userImage` was added to the POST payload) had `null` avatars stored in Stream.

**Learning:** Stream.io stores activity metadata at write time. There's no automatic denormalization or re-enrichment when user data changes. Posts created before a field was added will never have that field unless you enrich on read.

**Pattern:** Server-side enrichment on GET — batch-fetch fresh user data from D1 for all unique actors in the page, then patch activities before returning them. Single query regardless of post count, swallowed on failure so feeds never break.

```typescript
// Collect unique user IDs, batch query, patch activities
const userIds = new Set(activities.map(a => a.actor.replace('user:', '')));
const rows = await queryAll(db, `SELECT id, avatar_url, name, handle FROM users WHERE id IN (...)`, [...userIds]);
for (const a of activities) { a.userImage = userMap.get(actorId)?.avatar_url; }
```

## 2. Most Entity Components Already Showed Images
**Topics:** astro, testing

**Context:** Initial assumption was that images were missing across dozens of components. Thorough codebase exploration in Plan Mode revealed most components (`CourseCard`, `CreatorCard`, `TeacherCard`, `UserAvatar`, dashboards, `CourseTabs`) already rendered images.

**Learning:** Always explore before estimating scope. The actual gaps were much narrower: (1) community `cover_image_url` fetched but never rendered, (2) FeedsHub showing only icons, (3) inconsistent fallback patterns. What seemed like a massive project was really ~20 targeted file edits.

## 3. The Commons Has Special Rendering in Multiple Places
**Topics:** astro

**Context:** Fixed community cover images on the joined communities grid but The Commons still showed no image on `/community`. The Commons card is rendered separately from the grid (pinned, prominent styling) with its own distinct HTML — not shared with other community cards.

**Learning:** The Commons appears in at least 4 separate rendering contexts: (1) `/community` featured card, (2) `/feeds` FeedsHub townhall card, (3) community detail page header, (4) `MyFeeds` dashboard widget. Each has its own rendering code. When adding a visual feature to communities, check all 4 places.

## 4. INSERT-then-UPDATE is a Seed Data Anti-Pattern
**Topics:** d1, testing

**Context:** Dev seed data inserted users with `/images/default-avatar.svg`, then immediately overwrote `avatar_url` via UPDATE statements. Same pattern for community `cover_image_url` and course `thumbnail_url`.

**Learning:** Put final values directly in the INSERT statement. INSERT-then-UPDATE obscures the actual data shape (must read two places), makes SQL longer, and can silently fail if the UPDATE WHERE clause doesn't match. The only valid exception is patching data created by a different migration file (e.g., dev seed updating The Commons from core seed).

---

## 5. placehold.co Creates an External Service Dependency
**Topics:** astro

**Context:** 15 components used `placehold.co` URLs as avatar fallbacks — an external service that generates placeholder images. This means the UI breaks (shows broken images) if the service is down or slow.

**Learning:** For fallback avatars, inline gradient+initial divs are superior to external placeholder services: no network dependency, faster rendering, consistent with the design system, and they look better (branded gradient vs generic grey). The `UserAvatar` component already had the right pattern — it just wasn't used everywhere.
