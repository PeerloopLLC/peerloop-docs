# Conv Decisions - 2026-03-29

## 1. Use community_moderators instead of moderator_invites for community intel
**Type:** Implementation
**Topics:** admin-intel, schema

**Trigger:** Plan specified `moderator_invites.community_id` but that column doesn't exist — moderator_invites is global (email-based), not community-scoped.

**Options Considered:**
1. Skip pending moderator metric entirely
2. Use `community_moderators WHERE is_active = 0` for inactive/revoked moderators ← Chosen
3. Add community_id to moderator_invites (schema change)

**Decision:** Query `community_moderators WHERE is_active = 0` to count inactive moderators per community. Field name `pendingModeratorInvites` retained in the type for API compatibility.

**Rationale:** `community_moderators` is the community-scoped table with the right FK. Inactive moderators (revoked or pending reactivation) are the relevant admin signal. Schema changes for this metric aren't justified.

**Consequences:** The metric shows inactive/revoked moderators, not pending invites. The field name is slightly misleading but the data is more useful for admin attention.
