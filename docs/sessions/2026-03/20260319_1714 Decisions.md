# Conv Decisions - 2026-03-19

## 1. Defer Feed Activity Pruning Cron
**Type:** Strategic
**Topics:** d1, stream

**Trigger:** FEEDS-HUB block had a subtask for a 90-day pruning cron on `feed_activities`. User wanted to observe real D1 data growth before committing to a retention period.

**Options Considered:**
1. Implement 90-day cron now as originally planned
2. Defer until real data accumulates ← Chosen
3. Skip pruning entirely (rely on D1 storage being cheap)

**Decision:** Defer the pruning cron. Observe real D1 data growth at Genesis scale before deciding the retention period and trigger mechanism.

**Rationale:** At Genesis scale (~80 users × ~10 posts/day × 150 bytes), data growth is ~45KB/month — negligible. The right retention period depends on actual usage patterns we haven't observed yet. Premature optimization would add complexity (cron trigger, wrangler.toml config) for a problem that may not materialize for months.

**Consequences:** FEEDS-HUB block marked complete without the cron. Deferred item tracked in PLAN.md under FEED-INTEL.

## 2. Auto-Derive Feed API Base Path From Activity Metadata
**Type:** Implementation
**Topics:** stream, astro

**Trigger:** Two gaps discovered: (1) course feed comments routed to townhall endpoint (wrong dual-write target), (2) home timeline comments hit a nonexistent endpoint (404). Both caused by parent components not passing `feedApiBasePath` to `FeedActivityCard`.

**Options Considered:**
1. Have each parent component pass the correct `feedApiBasePath` explicitly — fragile, every new surface needs to know routing
2. Create a unified `/api/feeds/comments` endpoint that accepts feedType/feedId in the body — changes API contract
3. Derive the base path from activity metadata in `FeedActivityCard` ← Chosen

**Decision:** `FeedActivityCard` reads `communitySlug` and `courseSlug` from the activity object to derive the correct API base path. Explicit `feedApiBasePath` prop is still accepted as an override but no longer required.

**Rationale:** Stream activities carry their source context from creation time. Reading it in the component is the simplest approach — zero new endpoints, zero changes to existing callers, and every future surface (including the planned smart feed) gets correct routing automatically.

**Consequences:** Created `deriveFeedApiBasePath()` helper. Created course comments + reactions endpoints. All feed surfaces now route correctly. Both original gaps closed.

> **Insight:** Self-routing components that derive behavior from their data rather than from parent configuration are more resilient to new contexts. The smart feed will work without any routing code because the activities themselves know where they came from.

## 3. SMART-FEED Concept: Ranked Personalized Pseudo-Feed
**Type:** Strategic
**Topics:** stream, d1

**Trigger:** Client asked for a feed that surfaces "important" posts from all member feeds — a personalized, ranked view for engagement.

**Options Considered:**
1. Build a new Stream feed type with server-side ranking — Stream flat feeds don't support custom ranking
2. Build entirely in D1 — would duplicate Stream's content storage
3. Use D1 for selection/ranking, Stream for content fetch, existing FeedActivityCard for display ← Chosen direction

**Decision:** Capture SMART-FEED as a new PLAN.md block. The CQRS architecture (D1 index for queries, Stream for content) and the auto-routing FeedActivityCard are the foundation. FEED-INTEL Phases 2-3 are the technical implementation path. Detailed planning in next conv.

**Rationale:** The architecture already supports this — reactions are global (activity ID-keyed), activities carry source metadata, and the D1 index enables cross-feed queries. No fundamental new infrastructure needed, just selection logic, ranking algorithms, and a new page component.

**Consequences:** SMART-FEED block added to PLAN.md with concept capture. Topic matching (simple keyword vs embedding-based) flagged as a design decision for future conv.
