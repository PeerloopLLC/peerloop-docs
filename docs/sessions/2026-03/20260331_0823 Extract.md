### PLATO Model B Design Evolution

**User:** "Here's the way I look at this work: When you decide to do PLATO run, you know all the component sequences that it depends upon..."

**Claude:** User proposed a fundamentally different model: page-centric instead of API-centric. Each run models page visits with button presses that trigger API calls. The DB provides the data the page needs, not explicit carry state.

**User:** "I have morphed this whole block into a couple of directions that are quite different from each other. I think we need to agree on whether using the DB as the state of the system during PLATO testing is a good idea."

**Claude:** Presented two models clearly: Model A (composable segments with carry state) vs Model B (sequential DB-accumulation). Key tradeoff: Model A is flexible but artificial, Model B is rigid but realistic.

**User:** "I prefer Model B because it breaks fast and loudly"

**Claude:** Adopted Model B. Updated design doc (`docs/as-designed/plato.md`) and PLAN.md PLATO block to reflect the new architecture.

### Why Not Playwright

**User:** "do we have to emulate all of these PLATO runs? Why can't we just have PeerLoop be controlled by our PLATO runs and execute the UI to achieve the goals"

**Claude:** Presented comprehensive pros/cons of API emulation vs Playwright-driven runs. API emulation: fast (~200ms), deterministic, debuggable. Playwright: tests the real thing but flaky, slow (30-60s), hard to debug.

**User:** "Definitely NOT. The playwright isn't giving us fail fast and loud, it is giving us fail fast and garbled muttering."

**Claude:** Confirmed API emulation as the right foundation. Run definitions serve double duty: executable test AND manual test script for the developer to walk the UI.

### Plan Mode Stress Test

**Claude:** Entered Plan Mode. Launched Explore agent to read all Phase 2 code, then Plan agent to stress-test Model B against the existing codebase.

**Key findings from stress test:**
1. Intra-run data flow MUST survive (renamed to `$context` — "what the page showed the user")
2. Cross-run carry (`$chainCarry`) eliminated entirely
3. Actor sessions resolved via DB lookup by persona email (`fromDB` source)
4. Discovery GET pattern needed for cross-run entity discovery
5. Webhook payloads use intra-run context, not carry state
6. DB verifications use persona-based joins instead of carried IDs
