# Conv Decisions - 2026-03-19

## 1. Version Polling Over Server-Push for CurrentUser Freshness
**Type:** Strategic
**Topics:** d1, auth, cloudflare

**Trigger:** CurrentUser had no mechanism to detect server-side data changes. Dashboard API calls were serving as the de facto freshness mechanism.

**Options Considered:**
1. Version polling with `data_version` counter (30s interval) ← Chosen
2. Server-Sent Events (SSE) — true push, but requires Durable Objects or long-lived connections
3. WebSocket via Durable Objects — bidirectional, most complex
4. Blind periodic refresh (no version check) — wasteful

**Decision:** Use a monotonic `data_version` counter on the users table with a lightweight polling endpoint (`GET /api/me/version`).

**Rationale:** Simplest approach that works. One column, one endpoint, one `setInterval`. The polling endpoint is ~1ms on D1, ~20 bytes response. Compatible with future SSE/WebSocket upgrade — only the transport changes, the counter stays. Sufficient for Genesis cohort scale (60-80 students).

**Consequences:** 31 mutation endpoints now call `bumpUserDataVersion()`. Client polls every 30s. Staleness window reduced from "until next page navigation" to "≤30 seconds" for external changes, ~300ms for own mutations.

## 2. Keep TeacherDashboard and CreatorDashboard Endpoints As-Is
**Type:** Implementation
**Topics:** astro, auth

**Trigger:** Initial analysis showed 60-65% redundancy in these endpoints vs CurrentUser.

**Options Considered:**
1. Create lighter "operational" endpoints, consume CurrentUser for relationship data
2. Keep existing endpoints untouched ← Chosen

**Decision:** Do not refactor TeacherDashboard or CreatorDashboard endpoints. They serve operational/transactional data (earnings, pending counts, session lists, student lists) that CurrentUser shouldn't carry.

**Rationale:** Creating lighter endpoints would be a lateral move — same number of API calls, slightly smaller payloads, significant refactoring effort. The overlap (cert data, course data) is a few KB at most. Not worth the engineering cost.

**Consequences:** Only StudentDashboard will be refactored (Phase 2) because `/api/me/enrollments` is ~95% redundant. The other two dashboards keep their existing endpoints.

## 3. "Consume What's Loaded" Replaces "Summary vs. Detail" Principle
**Type:** Strategic
**Topics:** auth, astro

**Trigger:** The original CurrentUser design rule was: "If data answers a yes/no question across multiple pages, cache it in CurrentUser. If it's only needed on one page, fetch it there." This created a blind spot where dashboards re-fetched data CurrentUser already had.

**Options Considered:**
1. Keep the original principle, accept the redundancy
2. Update to "consume what's loaded" ← Chosen

**Decision:** New principle: "If CurrentUser already loads the data (for any reason), consume it from CurrentUser rather than re-fetching. Dashboard endpoints remain for operational/transactional data that CurrentUser doesn't carry."

**Rationale:** CurrentUser loads enrollment, cert, and course data for permission-checking methods (`isStudentFor()`, `isTeacherFor()`, `isCreatorFor()`). That same data is exactly what dashboards render. The old principle focused on "who needs it" (page count); the new one focuses on "who already has it."

**Consequences:** Updated `docs/architecture/state-management.md`. Phase 2 will refactor StudentDashboard to consume CurrentUser for enrollments.

## 4. Accept Unread Count Noise in Version Polling
**Type:** Implementation
**Topics:** d1, auth

**Trigger:** Unread notification/message counts are in CurrentUser (drive navbar badges). Every incoming message bumps `data_version`, potentially triggering frequent full refreshes.

**Options Considered:**
1. Accept the noise — every message/notification bumps version ← Chosen
2. Separate unread counts into their own lightweight polling endpoint

**Decision:** Accept the noise. Every new message/notification bumps `data_version` and may trigger a full `/api/me/full` refresh.

**Rationale:** Simpler (one counter, one endpoint). `/api/me/full` is fast (~100-300ms). Badge accuracy matters for UX. Acceptable at Genesis cohort scale. If it becomes a performance concern later, unread counts can be separated into their own polling endpoint.

## 5. The Commons as Townhall Feed, Always Pinned First
**Type:** Implementation
**Topics:** stream

**Trigger:** User requested a "My Feeds" component with a common feed for all members.

**Decision:** The Commons (`is_system=1` community) serves as the townhall feed. In the feed index (`getFeeds()`), it's always pinned at the top, followed by community feeds, then course feeds.

**Rationale:** The Commons is already the system community that all users join on signup. It has `is_system=1` in the schema and is created in core seed data. Natural fit for a townhall/common-feed role.

**Consequences:** Phase 4 will add `getTownhall()` method to CurrentUser. Phase 5's MyFeeds component will pin it first.
