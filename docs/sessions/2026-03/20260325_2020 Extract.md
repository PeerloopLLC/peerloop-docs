## Learnings

### 1. Component Reuse for Cross-Role Dashboards
**Topics:** react, architecture, dashboard
**Context:** Building a unified dashboard that shows content from 3 separate role dashboards
**Learning:** The existing dashboard components (DashboardStatCard, EarningsOverview, QuickActionButton, MyFeeds, CertificatesSection, AvailabilityQuickView, TeacherCertifications, CreatorCourseCard) were all composable enough to use unchanged inside new wrapper components. Only EnrollmentCard needed extraction from its parent (it was defined inline in StudentDashboard). This validates the original component design as reuse-friendly.

### 2. Activity-First vs Role-First Information Architecture
**Topics:** ux, dashboard, information-architecture
**Context:** Designing a unified dashboard for users who can hold multiple roles simultaneously
**Learning:** Grouping by activity type (Schedule, Needs Attention, Courses, People, Earnings) rather than by role (Learning tab, Teaching tab, Creating tab) enables cross-role insights that tabs can't provide. The key example: "Needs Your Attention" merges pending actions from teacher AND creator dashboards — homework reviews appear in both, and without a unified view a creator-who-teaches must check two separate dashboards. The trade-off is needing role badges on every item to maintain context.

### 3. Asymmetric Data Loading Patterns Enable Efficient Unified Views
**Topics:** performance, api, currentuser
**Context:** The three dashboards have fundamentally different data sources — student uses CurrentUser cache (zero API cost), teacher and creator each have composite API endpoints
**Learning:** This asymmetry actually helps unified loading. The student role's data is already in the CurrentUser singleton (loaded by AppNavbar on every page), so a unified dashboard only needs to fire 0-2 additional API calls for teacher/creator data. Conditional Promise.all based on role means a student-only user pays zero extra API cost.

## Decisions

### 1. Unified Dashboard as Additive Page (Not Replacement)
**Type:** Strategic
**Topics:** dashboard, architecture, ux, navigation
**Trigger:** User rejected B+ (tabbed) approach as merely moving the nav entry point, and chose C (true unified) with a button row linking to original pages
**Options Considered:**
1. B+ Tabbed — single /dashboard page with role tabs embedding existing dashboards. Low risk but doesn't provide cross-role insights.
2. C True Unified — single page with activity-first sections and role sub-columns. Higher value but more complex. ← Chosen
3. C as replacement — redirect /learning, /teaching, /creating to /dashboard. Maximum commitment.
**Decision:** Build Approach C as an additive /dashboard page. Original role-specific pages (/learning, /teaching, /creating) remain unchanged and accessible via a Dashboard Links button row at the top.
**Rationale:** Makes the unified dashboard zero-risk — it's a new page that composes existing components. Users can always fall back to full role-specific dashboards. Route redirect decision deferred to after Phase 1 evaluation.
**Consequences:** AppNavbar now has 4 dashboard-related items (Dashboard, Learning, Teaching, Creating). Route consolidation deferred.

### 2. All Sections Collapsible with localStorage Persistence
**Type:** Implementation
**Topics:** ux, dashboard, components
**Trigger:** User preference — asked whether sections should be collapsible
**Options Considered:**
1. No collapsibility — simpler, rely on progressive disclosure within sections
2. All sections collapsible ← Chosen — user controls information density
**Decision:** Every section wrapped in CollapsibleSection with expand/collapse via header click, state persisted in localStorage per section ID.
**Rationale:** Tri-role users could see 8+ sections. Collapsibility gives users control over page length without removing content.
**Consequences:** New shared CollapsibleSection component; all unified dashboard sections use it.

### 3. Include Student Stats Cards (New Feature)
**Type:** Implementation
**Topics:** dashboard, student, stats
**Trigger:** Student dashboard had no stat cards while teacher and creator had 3-4 each, creating visual imbalance in unified view
**Options Considered:**
1. Skip student stats — only teacher/creator stats appear
2. Add student stats ← Chosen — Courses Enrolled, Modules Completed, Completion Rate
**Decision:** Add 3 student stat cards computed from CurrentUser data (getEnrollments().length, sum of modulesCompleted, completion percentage).
**Rationale:** Creates visual balance in the Stats Overview section and provides students with at-a-glance progress metrics that don't exist on the original /learning page.
**Consequences:** New feature unique to unified dashboard — student stats are not on /learning.
