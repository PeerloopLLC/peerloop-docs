# Session Decisions - 2026-03-11

## 1. Auto-Advance Booking Wizard When Teacher Pre-Assigned
**Type:** Implementation
**Topics:** booking, astro

**Trigger:** Students who already selected a teacher at checkout were shown a teacher selection step that was effectively cosmetic — only the assigned teacher would work at booking time.

**Options Considered:**
1. Keep step 1 as-is, let it fail at confirm if wrong teacher selected
2. Filter teacher list to only the assigned teacher
3. Auto-advance to step 2, show teacher as context banner ← Chosen

**Decision:** Auto-advance to step 2 (date) when `assigned_teacher_id` exists. Show compact teacher card inline with the date heading. Allow clicking step 1 in the stepper to view all teachers (future: allow switching).

**Rationale:** Respects the constraint without hiding information. The student sees who they're booking with, doesn't waste a click, and can still browse other teachers. Sets up clean path for future "change teacher" feature.

**Consequence:** Added `assignedTeacherId` prop, teacher banner in date step, "Your teacher" badge, and "Continue with [name]" button on step 1 revisit.

## 2. Show Unavailable Teachers with Visual Distinction Rather Than Filtering
**Type:** Strategic
**Topics:** booking

**Trigger:** The booking page showed teachers with `teaching_active = 1` but didn't check if they had availability rows. Guy Rymberg (course creator) appeared as selectable but had zero available slots.

**Options Considered:**
1. Filter out teachers with no availability from the query ← Not chosen
2. Show all teachers, badge unavailable ones as greyed out ← Chosen

**Decision:** Show all certified teachers but visually distinguish those without availability (dashed border, reduced opacity, "No availability" badge, disabled click). Sort available teachers first.

**Rationale:** The client wants to eventually allow students to message teachers/creators to request teaching. Hiding unavailable teachers would prevent this. Showing them with clear visual state communicates "exists but not bookable now."

**Consequence:** Added `has_availability` EXISTS subquery to both teacher queries in `book.astro`, `hasAvailability` field on Teacher interface, conditional styling in teacher cards.

## 3. Reschedule Slot Release via exclude_session_id Parameter
**Type:** Implementation
**Topics:** booking, d1

**Trigger:** When rescheduling a session to the same day, the original time slot appeared unavailable because the availability endpoint included the session being rescheduled in its conflict check.

**Options Considered:**
1. Add `exclude_session_id` query param to availability endpoint ← Chosen
2. Change session status during reschedule flow (e.g., to 'rescheduling')
3. Have the frontend filter out the known session client-side

**Decision:** Add optional `exclude_session_id` param to `/api/teachers/:id/availability`. The booking page reads the `?reschedule=` URL param and passes it through.

**Rationale:** Cleanest approach — the endpoint's default behavior is unchanged, and the exclusion is explicit. Option 2 would create a transient status that complicates other queries. Option 3 would require the client to know session timing details.

**Consequence:** Three-file change: availability endpoint (param + query), book.astro (read param, pass prop), SessionBooking.tsx (include in fetch URL).
