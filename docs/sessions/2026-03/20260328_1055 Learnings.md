# Conv Learnings - 2026-03-28

## 1. ExploreTabBar Generalization Was a One-Line Change
**Topics:** component-architecture, reusability

**Context:** ExploreTabBar was tightly coupled to course tab types via `TAB_LABELS` and `ExploreTab`, but already received `label` on each config — it just ignored it.

**Learning:** The component's actual rendering logic was already domain-agnostic. The coupling was purely at the type/constant level. Replacing `TAB_LABELS[tab.id]` with `tab.label` (line 56) made it reusable for any entity type. This is a common pattern: components that appear tightly coupled may only need type widening, not structural changes.

## 2. Community Member→CourseRole Color Mapping
**Topics:** design-system, role-model

**Context:** Communities have different role terminology (member/teacher/creator/moderator) from courses (student/teacher/creator/moderator). The RoleBadge color system is keyed on CourseRole.

**Learning:** Rather than creating a parallel CommunityRole color scheme, map community `member` to CourseRole `student` (blue). This works because both are the "default participant" role. The mapping is explicit in `community-role-utils.ts` via `MEMBER_ROLE_MAP`. Same 4 colors, no new CSS classes needed.

## 3. getFeeds() Deduplication Lost Role Information
**Topics:** current-user, data-model, feeds

**Context:** The original `addCourseFeed` helper used a `seenCourseIds` Set to deduplicate — "first writer wins." A user who is both enrolled and teaching a course only got the enrollment role in the feed entry, losing the teacher role entirely.

**Learning:** Replaced with `courseFeedMap` that collects ALL roles per courseId across enrollments, teacherCertifications, and createdCourses before emitting one feed entry. This is backward-compatible since `roles` is a new field. The semantic improvement: feed cards can now show all of a user's relationships to a course.

## 4. Shared FeedRoleTab vs Separate Files
**Topics:** component-architecture, code-organization

**Context:** Phase 1 created 4 separate community role tab files (MemberTab, TeachingTab, CreatedTab, ModerationTab) because they had different data sources. Phase 2 feed role tabs all take the same input (pre-filtered feeds).

**Learning:** When role tabs differ only in filter logic and empty-state copy — not in data source or structure — a single shared `FeedRoleTab` component with props is cleaner than 4 duplicate files. The orchestrator pre-filters and passes per-role empty messages.

---

## 5. Coexisting Pages for Client Comparison
**Topics:** deployment-strategy, client-workflow

**Context:** User wanted `/feeds` (original FeedsHub) to stay while `/discover/feeds` (new role-aware) was added, so the client can compare both and decide which becomes canonical.

**Learning:** When building a replacement page, keep the original accessible until the client approves the new version. Add the new page to navigation (DiscoverSlidePanel) without replacing existing nav links. This avoids premature redirect decisions and gives the client a side-by-side comparison.
