# Conv Learnings - 2026-03-19

## 1. Hybrid Query Strategy for Complex Scoring
**Topics:** d1, stream

**Context:** The SMART-FEED design initially proposed a large CTE with 7+ JOINs in D1 to score candidates in a single SQL query — joining feed_activities with community_members, enrollments, teacher_certifications, feed_visits, and user_topic_interests.

**Learning:** Complex multi-table scoring queries are fragile and hard to test in D1. The proven pattern from `getFeedBadgeCounts()` (simple tuple-matching with minimal JOINs) works because it keeps D1 queries simple and does relationship resolution in app code. The hybrid approach — D1 selects raw candidates, app loads relationship metadata via parallel queries, TypeScript scores candidates — is more testable, debuggable, and matches existing architecture.

**Pattern:** D1 for indexed lookups → app for scoring logic → Stream for content enrichment. Each phase is independently testable.

## 2. feed_id Stores Slugs, Not IDs
**Topics:** d1

**Context:** When designing the SMART-FEED candidate query, the assumption was that `feed_activities.feed_id` stored database IDs, enabling direct JOINs to `communities.id` or `courses.id`.

**Learning:** `feed_id` stores slugs (e.g., "python-devs", "react-101"), not database IDs. This is by design — it matches how `getFeedBadgeCounts()` and all feed endpoints work. Attempting to JOIN through slug→ID lookups in SQL adds subqueries to every ON clause. The correct approach is to accept slug-based feed_id and do any ID-based lookups (for relationship scoring) in app code after fetching candidates.

## 3. Missing Index Discovered via Verification
**Topics:** d1, testing

**Context:** Plan Mode verification identified that feed affinity scoring (counting how often a user posts in each feed via `GROUP BY actor_id`) would require a full table scan on `feed_activities` without a dedicated index.

**Learning:** The existing indexes (`idx_feed_activities_feed` on `(feed_type, feed_id, created_at)` and `idx_feed_activities_created` on `(created_at)`) covered badge counting but not actor-based queries. Added `idx_feed_activities_actor ON (actor_id, created_at)` to support feed affinity scoring.

## 4. Discovery Cards Need Server-Side Dismiss Tracking
**Topics:** stream, d1

**Context:** The existing recommendation components (`RecommendedCommunities`, `RecommendedCourses`) use localStorage for dismiss tracking — a single key that hides the entire recommendation band.

**Learning:** Per-item dismissal for discovery cards requires server-side tracking (new `smart_feed_dismissals` table) because: (a) localStorage doesn't persist across devices, (b) the dismissal must exclude the feed from future D1 queries, not just hide a UI element, (c) individual feeds need independent dismiss tracking, not band-level toggling.

---

## 5. Existing Recommendation Queries Are Reusable for Discovery
**Topics:** d1

**Context:** SMART-FEED discovery needs to find public feeds matching the user's interests that they haven't joined.

**Learning:** `/api/recommendations/communities.ts` already implements the exact CTE chain needed: `user_topic_interests → topics.category_id → courses.category_id → progressions.community_id → communities`. The discovery candidate query adapts this chain rather than reinventing it, adding vitality scoring (recent activity count) and feed_activities fetching on top.

## 6. Cursor Pagination with Interleaved Discovery
**Topics:** stream, astro

**Context:** SMART-FEED uses cursor-based pagination (`before` timestamp) instead of offset-based. Discovery cards are interleaved among member posts.

**Learning:** Discovery cards must NOT be cursor-tracked. The `before` cursor applies only to member candidates' `created_at` in the D1 query. Discovery cards are computed fresh per page and injected by position (1 per N member posts). The `nextCursor` in the response is the `created_at` of the last member post, not discovery. This means discovery cards may repeat across pages at Genesis scale — acceptable, and a `seen_discovery` cache can be added later if needed.
