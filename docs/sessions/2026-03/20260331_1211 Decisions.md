# Conv Decisions - 2026-03-31

## 1. Name the Group Concept "Scenario"
**Type:** Strategic
**Topics:** plato, testing

**Trigger:** User needed a name for independent groups of personas with their own goals

**Options Considered:**
1. Scenario — goal-oriented, self-contained
2. Cast / Troupe — theatrical (fits PLATO theme)
3. Arc — story arc metaphor

**Decision:** Use "Scenario" as the name ← Chosen by user

**Rationale:** Clearest, most intuitive for ad-hoc creation. "Generate a new scenario to test..." reads naturally.

**Consequences:** PlatoScenario interface, scenarios/ directory, ScenarioCategory type (test/seed/repro)

## 2. Multi-Course Discovery via findBy Pattern
**Type:** Implementation
**Topics:** plato, testing

**Trigger:** With multiple courses, courses[0].id picks the wrong course (API returns created_at DESC)

**Options Considered:**
1. findBy(field,value) in extractPath — search array for matching element ← Chosen
2. $scenario namespace with cross-run context persistence
3. Title-based searchParams filtering on discovery GETs

**Decision:** Add findBy() to extractPath so provides can declare: courses.findBy(title,$persona.courseTitle).id

**Rationale:** Clean, declarative, no conditional logic in runs, backward compatible (courses[0] still works). Single small change to the runner.

**Consequences:** All discovery GETs updated. Custom parseDotPath() needed for paren-aware dot splitting.

## 3. Actor Bindings for Multi-Student Scenarios
**Type:** Implementation
**Topics:** plato, testing

**Trigger:** Need same run (e.g., enroll-student) to work with different students

**Options Considered:**
1. actorBindings on RunRef — maps run actor slots to persona keys ← Chosen
2. Parameterized runs — runs accept actor name as parameter
3. Swap entire personaData before each run

**Decision:** RunRef.actorBindings remaps persona keys to run actor slots transparently

**Rationale:** Runs don't change at all. The scenario chain declares which persona fills each actor slot per invocation. Combined with course flattening (runtimeOverrides.courseIndex), this enables full multi-course/multi-student composition.

**Consequences:** applyActorBindings() + flattenCourseData() in runner. Cookie stores rekeyed by persona email.

## 4. Separate Run for Per-Course Teacher Certification
**Type:** Implementation
**Topics:** plato, testing

**Trigger:** self-certify-creator fails on second invocation (Stripe already connected)

**Options Considered:**
1. New add-teacher-cert run (certification only, no Stripe) ← Chosen
2. Make self-certify-creator idempotent (skip Stripe if already connected)
3. Conditional logic in the run

**Decision:** Created add-teacher-cert.run.ts — teacher certification without Stripe Connect

**Rationale:** Runs should be atomic and composable. The original self-certify-creator stays for first-time setup; add-teacher-cert handles additional courses. No conditional logic needed in either run.

**Consequences:** New run registered in loader. Ecosystem scenario uses self-certify-creator for first course, add-teacher-cert for subsequent.
