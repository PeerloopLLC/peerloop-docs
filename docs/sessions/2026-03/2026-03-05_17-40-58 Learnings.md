# Session Learnings - 2026-03-05

## 1. React Hooks Can't Be Called in Loops — Extract Sub-Components
**Topics:** astro, messaging

**Context:** Phase 3 of MSG-ACCESS required adding `useCanMessage` checks to list items in CourseSTList (multiple S-Ts) and CommunityTabs (multiple members). React hooks must be called at the top level of a component — you can't call them inside `.map()`.

**Learning:** The standard pattern is to extract each list item into its own sub-component (`STCard`, `MemberRow`) where each instance independently calls the hook. This means N parallel API calls for N items, but browsers handle this efficiently via HTTP/2 multiplexing. For small lists (2-10 items) this is perfectly acceptable.

**Pattern:**
```tsx
// Wrong — hook in a loop
{items.map(item => {
  const { canMessage } = useCanMessage(item.id); // ERROR
  return <div>{canMessage && <MessageBtn />}</div>;
})}

// Right — extract sub-component
function ItemCard({ item }) {
  const { canMessage } = useCanMessage(item.id); // OK
  return <div>{canMessage && <MessageBtn />}</div>;
}
{items.map(item => <ItemCard key={item.id} item={item} />)}
```

## 2. "Inherently Valid" vs "Conditional" Is About Context, Not Code
**Topics:** messaging, astro

**Context:** SessionBooking was listed as Phase 3 (conditional) but receives `enrollmentId` as a required prop — the user is enrolled by definition if they see this component. The messaging relationship is guaranteed by the component's existence.

**Learning:** The distinction between inherently valid and conditional surfaces is about the *context that renders the component*, not the component itself. If a prop's presence proves the relationship (like `enrollmentId`), no API check is needed — just add a direct message link. Save `useCanMessage` for surfaces where the viewer's relationship is genuinely unknown.

## 3. Permission Flags vs Role Identity in SQL Queries
**Topics:** d1, astro

**Context:** The `/discover/students` page filtered by `can_take_courses = 1`, which returns every user on the platform because it's a permission flag, not a role indicator. Guy Rymberg (a Creator) appeared as a "student" with 0 enrollments.

**Learning:** `can_take_courses` means "this user is allowed to enroll" — it's a capability, not an identity. On Peerloop, everyone can take courses, so this filter is effectively `WHERE true`. When building directory pages, filter by *activity* (e.g., `courses_enrolled > 0`) or use `EXISTS` subqueries for role detection rather than relying on permission flags as role indicators.

## 4. Discovery Pages Need Actionable Intent
**Topics:** messaging, astro

**Context:** User questioned why `/discover/students` exists when student-to-student messaging is blocked. The page shows people you can't interact with.

**Learning:** Every discovery/browse page should have a clear user intent that leads to action. Courses → enroll. Teachers → book. Creators → browse courses. Students → ??? (can't message for MVP). Pages without actionable outcomes are dead ends. Better to hide them until the enabling feature ships than to set expectations the platform can't meet. The description "connect with peers" promises what the policy blocks.
