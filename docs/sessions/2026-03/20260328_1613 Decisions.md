# Conv Decisions - 2026-03-28

## 1. TAG-TAXONOMY: Rename categories→topics, topics→tags, drop category_id
**Type:** Strategic
**Topics:** schema-design, taxonomy, naming

**Trigger:** User frustration with confusing similar names and question about whether all tables are needed

**Options Considered:**
1. Keep current 5-table system — rejected (confusing, redundant)
2. Collapse to 2 tables (categories + user_categories) — rejected (loses subtopic granularity valuable for onboarding UX and multi-tag courses)
3. Rename to topics/tags, drop redundant tables, make course_tags FK-based ← Chosen

**Decision:** Rename `categories` → `topics` (15 display groups), `topics` → `tags` (55 atomic items). Drop `user_interests`, `user_topic_interests`, `experience_level`. Repurpose `course_tags` from free-text to FK-based many-to-many. Drop `courses.category_id` entirely — derive from `course_tags → tags → topics`.

**Rationale:** Matches user mental model (tags/topics), eliminates redundant tables, enables multi-tag courses, simplifies all downstream matching to direct tag overlap

**Consequences:** ~30 source files + ~150 test files need updating across 6 phases. Smart feed scoring changes from binary category match to graduated tag overlap. Multi-conv effort.

> **Insight:** Schema naming that mirrors the user's vocabulary rather than database-design conventions reduces cognitive load across the entire team. When stakeholders, UI, and schema all use the same terms, misunderstandings drop and design discussions become productive immediately.

## 2. Drop courses.category_id — Derive Topics from course_tags
**Type:** Implementation
**Topics:** schema-design, taxonomy

**Trigger:** Two options presented: keep `category_id` as denormalized primary, or derive everything from `course_tags`

**Options Considered:**
1. Keep `courses.topic_id` as denormalized primary topic — more convenient, one less join
2. Drop entirely, derive from `course_tags → tags.topic_id` ← Chosen

**Decision:** Option 2 — no denormalized primary topic on courses

**Rationale:** User chose "Simplify" — one source of truth for course-topic relationships. Avoids maintaining two paths (direct FK + course_tags).

**Consequences:** Every query that previously used `courses.category_id` must join through `course_tags`. Course filter sidebar derives topics from course tag data. More joins but cleaner data model.

## 3. Smart Feed Scoring: Binary → Graduated Tag Overlap
**Type:** Implementation
**Topics:** smart-feed, scoring, taxonomy

**Trigger:** Changing from single `category_id` per course to multiple tags per course

**Options Considered:**
1. Keep binary matching (any tag overlap = 1.0) — loses information
2. Graduated scoring: `intersectionCount(feedTags, userTagIds) / feedTags.size` ← Chosen

**Decision:** Topic match score becomes graduated — a course feed with 3 of user's 5 tags scores higher than one with 1 match

**Rationale:** Multi-tag model provides richer signal; graduated scoring exploits it

**Consequences:** `feedCategoryMap: Map<string, string>` → `feedTagsMap: Map<string, Set<string>>`. `platform_stats` key renamed `smart_feed_weight_category_affinity` → `smart_feed_weight_tag_affinity`.
