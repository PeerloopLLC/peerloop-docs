# Conv Decisions - 2026-03-29

## 1. Hydration Safety at Hook Level (Option 1)
**Type:** Implementation
**Topics:** react, astro, hydration, currentUser

**Trigger:** Per-component hydration fixes (hasMounted/effectiveUser) were brittle and would dissipate over time

**Options Considered:**
1. Hydration-safe useCurrentUser() by default — return null on first render ← Chosen
2. Separate useHydratedCurrentUser() hook — two hooks to choose from
3. Use client:only="react" for all authenticated islands — no SEO

**Decision:** Make useCurrentUser() and useAuthStatus() hydration-safe by default. Initialize with null/'loading', populate from cache in useEffect.

**Rationale:** 2 lines in one file vs ~15 lines across every consumer. Any future component using the hook gets hydration safety for free. Flash impact limited to brief additive role UI on discover pages (imperceptible).

**Consequences:** Removed hasMounted/effectiveUser from ExploreCourses, ExploreCommunities, OnboardingNudgeBanner. Kept UnifiedDashboard merged skeleton (correct UX regardless).

> **Insight:** When an SSR framework hydrates client components, any hook that reads browser-only state (localStorage, cookies, window) must defer that read to useEffect. Initializing with the browser value creates a hydration mismatch on every page load — the fix must live in the hook, not in consumers, because the invariant (SSR = null) is universal.

## 2. Reusable OnboardingNudgeBanner Over Inline Nudges (Option 2)
**Type:** Implementation
**Topics:** react, components, onboarding

**Trigger:** UNIFIED-DASHBOARD followup item: component-level onboarding nudge pattern

**Options Considered:**
1. Inline nudge in each component — 3+ copies of same logic
2. Reusable OnboardingNudgeBanner component ← Chosen
3. Shared render function — collapses to Option 2 (needs hooks)

**Decision:** Single reusable component with context prop, compact/full variants, per-context dismiss.

**Rationale:** Single source of truth. Consistent UX. Easy to extend to new pages.

**Consequences:** Created OnboardingNudgeBanner.tsx + 14 tests. Refactored DiscoverFeedsGrid to use it. Integrated into 3 Astro pages.

## 3. interestTopicIds on CurrentUser
**Type:** Implementation
**Topics:** currentUser, tags, filtering

**Trigger:** "My Interests" filter needed user's topic preferences for course filtering

**Options Considered:**
1. Fetch from /api/me/onboarding-profile on demand — async, loading state needed
2. Add to CurrentUser via /api/me/full ← Chosen

**Decision:** Add interestTopicIds to MeFullResponse and CurrentUser class.

**Rationale:** Enables synchronous filtering. Tiny payload. Single query in existing Promise.all. Natural complement to hasCompletedOnboarding().

**Consequences:** New field on MeFullResponse, Set in CurrentUser, getInterestTopicIds() + hasInterests() methods, fetchInterestTopicIds() in /api/me/full.ts.

## 4. MY_INTERESTS_TOPIC Pseudo-Radio Over Multi-Select
**Type:** Implementation
**Topics:** UX, filtering, courses

**Trigger:** "My Interests" needs multi-topic matching but filter sidebar is single-select

**Options Considered:**
1. Special pseudo-topic radio option ← Chosen
2. Convert to multi-select checkboxes — changes existing UX

**Decision:** Sentinel string constant as special radio option. Filter logic handles it as multi-topic match.

**Rationale:** Works within existing single-select model. No UX change for existing users.

**Consequences:** MY_INTERESTS_TOPIC exported from CourseFilterSidebar. ExploreAllTab handles sentinel in filter logic.
