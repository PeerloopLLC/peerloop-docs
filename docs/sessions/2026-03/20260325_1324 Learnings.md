# Conv Learnings - 2026-03-25

## 1. E2E Tests Drift Silently When Components Change
**Topics:** testing, astro

**Context:** 8 E2E tests were failing because component headings/text had changed (e.g., "Home Feed" → "Smart Feed", "Earnings Overview" → "Creator Earnings") but the Playwright tests still referenced the old strings.

**Learning:** E2E tests that assert on specific UI text are fragile to rename/rebrand changes. When renaming headings or restructuring components, a grep for the old text across `e2e/` should be part of the change. The test suite won't catch this until someone runs it — and with 126 tests already failing (for a different reason), these 8 drifted tests were invisible.

## 2. SmartFeed API Shape Differs From Old HomeFeed
**Topics:** testing, stream

**Context:** The `home-feed.spec.ts` used mock data shaped for the old HomeFeed (`{ activities: [{id, actor, userName, text, ...}], next }`) but the page now uses SmartFeed which expects `{ activities: SmartFeedItem[], nextCursor }` where each item wraps an `activity` inside `{ activity, smartScore, surfaceReason, isDiscovery, feedType, feedId }`.

**Learning:** When a component is replaced (HomeFeed → SmartFeed), the mock data fixtures must be updated to match the new API contract. Created separate `mockSmartFeedResponse`/`emptySmartFeedResponse` exports so community/course feed tests can continue using the old format while home-feed tests use the SmartFeed shape.

## 3. Playwright Treats opacity-0 Elements as Not Visible
**Topics:** testing

**Context:** Notification delete buttons have `opacity-0 group-hover:opacity-100` — hidden until hover. `getByTitle('Delete notification')` finds them in the DOM, but `toHaveCount()` was returning 0 because the elements weren't attached yet (race with rendering), not because of opacity.

**Learning:** For elements that are in the DOM but CSS-hidden, use `toBeAttached()` to confirm DOM presence before calling `.count()`. The `opacity-0` class doesn't prevent Playwright from finding elements — the real issue was a timing race where the notification items hadn't finished rendering when the count was checked.

## 4. E2E Mutation Tests Permanently Alter Dev DB State
**Topics:** testing, d1

**Context:** The notification mutation tests (delete, mark-all-read) run against the live dev D1 database. After a test run, `notif-david-001` is deleted and `notif-david-002` is marked as read. Subsequent runs fail because the seed data assumptions no longer hold. The booking flow test also creates extra notifications.

**Learning:** E2E tests that mutate shared dev DB state must either: (a) restore state after themselves, (b) not depend on exact counts/specific records, or (c) use a test-specific DB. We chose (b) — notification tests now use general assertions (`/\d+ notification/`, `toBeGreaterThanOrEqual(1)`) instead of exact seed data expectations. A full DB reset (`npm run db:reset:local && npm run db:setup:local:dev`) is needed between runs that include mutation tests.

## 5. toLocaleTimeString with hour:'numeric' Produces Locale-Dependent Format
**Topics:** testing, astro

**Context:** The booking wizard displays time slots using `toLocaleTimeString([], { hour: 'numeric', minute: '2-digit' })`, which produces "10:00 AM - 11:00 AM" in en-US locale. The E2E test regex `/^\d{2}:\d{2} - \d{2}:\d{2}$/` expected 24-hour format without AM/PM.

**Learning:** Never assume `toLocaleTimeString()` output format in tests. Use a flexible regex that handles both 12h/24h, AM/PM, and locale-specific separators (hyphen vs en-dash): `/\d{1,2}:\d{2}\s*(?:AM|PM)?\s*[-–]\s*\d{1,2}:\d{2}\s*(?:AM|PM)?/i`

## 6. Login Redirect Races With Subsequent page.goto()
**Topics:** testing, auth

**Context:** After adding a post-login redirect (`window.location.href = '/dashboard'`), E2E tests that called `login()` then immediately `page.goto('/admin')` got `net::ERR_ABORTED` — the redirect was still in-flight when the next navigation started.

**Learning:** When the login flow triggers a redirect, the E2E login helper must wait for the redirect to complete. Changed from `await expect(emailInput).not.toBeVisible()` (modal closed) to `await page.waitForURL('**/dashboard')` (redirect landed). Any flow that adds navigation side-effects must update the test helpers that trigger that flow.

## 7. Query Param Name Mismatch Between Callers and Reader
**Topics:** auth, astro

**Context:** `login.astro` read `?returnUrl=` but all callers (feed.astro, session pages, course booking) sent `?redirect=`. The return URL was never picked up, so even when a redirect param was provided, login success stayed on `/login`.

**Learning:** When adding a query param contract, grep for all callers to verify they use the same param name. Fixed by reading both (`redirect` preferred, `returnUrl` as fallback) and applied the same fix to `signup.astro`.
