# Session Learnings - 2026-03-05

## 1. getCurrentUser() Cannot Determine Cross-User Relationships
**Topics:** auth, messaging, astro

**Context:** Evaluating whether `getCurrentUser()` (client-side global) could replace an API call for messaging permission checks on profile pages.

**Learning:** `getCurrentUser()` knows about the *current user's* capabilities (isAdmin, hasCreatedCourses, isActiveStudentTeacher) but has no information about *relationships with other users*. It can answer "am I a student?" but not "am I *this person's* student?" — that requires a DB query joining enrollments, student_teachers, and courses tables. For visitor detection and self-profile checks it's sufficient, but relationship-gated UI needs a server round trip.

## 2. SessionHistory Used Wrong URL Parameter for Messages
**Topics:** messaging, astro

**Context:** Phase 2 audit of messaging surfaces found SessionHistory already had a message button, but with `?user=` instead of `?to=`.

**Learning:** The Messages page reads `?to=` for the recipient user ID. SessionHistory was using `?user=` — a silent bug where clicking the message icon would open the messages page but not preselect the recipient. When multiple surfaces independently implement the same URL pattern, inconsistencies creep in. The `?to=` parameter is now the canonical pattern across all surfaces.

---

## 3. API Response Missing Data That SQL Already Joins
**Topics:** astro, testing

**Context:** TeacherUpcomingSessions component needed `student_id` for the message link, but the API response didn't include it.

**Learning:** The `/api/me/teacher-dashboard` SQL query already joined `sessions s JOIN users u ON s.student_id = u.id` but only selected `u.name` and `u.avatar_url` — not `s.student_id`. The data was one SELECT column away but invisible to the frontend. When extending component capabilities, check the API's SQL first — the join may already exist, making the change trivial (one-line SELECT addition + interface update).
