# Conv Decisions - 2026-03-18

## 1. DATE-FORMAT: Canonical UTC ISO 8601 with Z for All Timestamps
**Type:** Strategic
**Topics:** d1, astro

**Trigger:** Past student completion dates on the Teacher dashboard displayed one day early due to format mismatch between seed data (ISO 8601 with Z) and SQLite `datetime('now')` output (no timezone marker). JavaScript's `new Date()` parses these differently, causing inconsistent timezone behavior.

**Options Considered:**
1. Fix at render layer only — add `{ timeZone: 'UTC' }` to `toLocaleDateString()` calls
2. Fix at data layer only — normalize seed data and schema defaults ← Partial
3. Canonical format everywhere + standardized formatters ← Chosen

**Decision:** All timestamps stored as UTC ISO 8601 with Z suffix (`2026-03-18T23:41:25Z`). All date display uses centralized formatting functions from `src/lib/timezone.ts`. Never use raw `toLocaleDateString()`.

**Rationale:** A single canonical format eliminates the ambiguity between SQLite's `datetime('now')` format and ISO 8601. Centralized formatters (`formatDateUTC`, `formatDateTimeUTC`, `formatLocalTime`) ensure consistent rendering regardless of which format the database returns during the migration period. `parseUTCDate()` handles both formats as a transition bridge.

**Consequences:**
- Created 4 new functions in `src/lib/timezone.ts`
- 68 files use raw `toLocaleDateString()` and need migration (tracked in PLAN.md DATE-FORMAT block)
- Schema defaults (`datetime('now')`) need migration to `strftime('%Y-%m-%dT%H:%M:%SZ', 'now')`
- Added to docs/DECISIONS.md as DATE-FORMAT entry

## 2. Role-Specific Earnings Labels on Dashboards
**Type:** Implementation
**Topics:** astro

**Trigger:** Guy Rymberg's Teaching dashboard showed $0.00 earnings. Investigation revealed the query correctly filters `recipient_type = 'teacher'`, but Guy's payment splits are `creator_as_instructor` / `creator_as_author`. The generic "Earnings Overview" label didn't clarify which revenue stream was shown.

**Options Considered:**
1. Widen the earnings query to include all split types on the Teaching dashboard
2. Keep query focused on teacher splits, clarify with role-specific labels ← Chosen

**Decision:** Show only the teacher portion of splits on `/teaching` (labeled "Teaching Earnings") and only creator portions on `/creating` (labeled "Creator Earnings"). The `EarningsOverview` component accepts an optional `title` prop.

**Rationale:** A Creator-as-Teacher like Guy earns differently in each role. Mixing split types would make earnings confusing. Clear labels tell the user exactly which revenue stream they're viewing.

**Consequences:** `EarningsOverview` now has `title` prop (default: "Earnings Overview" for backward compatibility). Both dashboards pass role-specific titles.

## 3. Current/Past Tabbed Lists for Students and Teachers
**Type:** Implementation
**Topics:** astro, d1

**Trigger:** Guy's "My Students" section was empty because all his students had `status = 'completed'` and the query only showed `enrolled`/`in_progress`. Similarly, creators had no visibility into inactive teachers.

**Options Considered:**
1. Widen existing query to include completed records
2. Add a second query and tabbed UI with Current/Past separation ← Chosen

**Decision:** Both the Teacher dashboard (students) and Creator dashboard (teachers) use a tabbed interface with "Current" and "Past" tabs. Separate API queries for each. Default tab is Current if it has data, otherwise Past.

**Rationale:** Mixing active and completed records in one list conflates different UX needs. Current students need progress bars; past students need completion dates. Current teachers need active status; past teachers are informational. Tabs let users see both without clutter.

**Consequences:**
- Teacher dashboard: `students` + `past_students` in API response, `TeacherStudentList` component with tabs
- Creator dashboard: `teachers` + `past_teachers` in API response, new `CreatorTeacherList` component with tabs
- `total_students` stat includes both current + past count
