# Session Decisions - 2026-03-05

## 1. Establish Official Platform Glossary (GLOSSARY.md)
**Type:** Strategic
**Topics:** naming, workflow, docs-infra

**Trigger:** Cascading naming inconsistencies — "Student-Teacher" vs "Teacher," "user" vs "member" vs "visitor," ambiguous schema column names causing bugs. The `/discover/teachers` duplication bug was the immediate trigger, but the underlying problem was systemic terminology drift.

**Options Considered:**
1. Fix issues ad-hoc as they arise
2. Add naming conventions to DEVELOPMENT-GUIDE.md
3. Create a dedicated GLOSSARY.md as the single source of truth for all platform terminology ← Chosen

**Decision:** Created `GLOSSARY.md` at docs repo root. It covers identity hierarchy, core domain terms, user story code mapping, DB table naming targets, component naming conventions, URL route targets, and ambiguous terms to avoid. It is prescriptive — code that contradicts the glossary is the bug.

**Rationale:** A dedicated glossary is the only approach that works across schema, code, UI text, and documentation. Embedding it in DEVELOPMENT-GUIDE.md would bury it; ad-hoc fixes don't prevent recurrence. The glossary must exist before any renames begin so all changes derive from one source of truth.

**Consequences:** GLOSSARY.md created (draft). TERMINOLOGY block added to PLAN.md with 4 phases. CURRENT-BLOCK-PLAN.md created for cross-session persistence.

## 2. "Teacher" Replaces "Student-Teacher" Platform-Wide
**Type:** Strategic
**Topics:** naming, astro, d1

**Trigger:** User identified "Student-Teacher" as confusing — it could mean "a student who teaches" or "a teacher of students." The abbreviations (S-T, ST) further muddied code readability. Since all teachers on Peerloop are former students (that's the flywheel), the "student" prefix adds no information.

**Options Considered:**
1. Keep "Student-Teacher" (status quo)
2. Abbreviate consistently to "ST" everywhere
3. Rename to "Teacher" everywhere ← Chosen
4. Rename to "Tutor" or "Peer Teacher"

**Decision:** "Teacher" replaces "Student-Teacher," "S-T," and "ST" across the entire platform — schema, code, components, API routes, UI text, and documentation. User story IDs (US-T###) remain frozen.

**Rationale:** Every teacher was once a student — that's the core flywheel. The prefix is redundant and causes confusion in naming (`STCard`, `student_teachers` table, `student_teacher_id` column). "Teacher" is clearer and shorter. Pre-beta is the cheapest time.

**Consequences:** `student_teachers` table → `teacher_certifications`. Component directory `student-teachers/` → `teachers/`. All `ST*` component prefixes → `Teacher*`. API routes `/api/student-teachers/*` → `/api/teachers/*`, `/api/me/st-*` → `/api/me/teacher-*`. ~309 occurrences across ~131 files for table rename alone.

## 3. Identity Hierarchy: Visitor → Member → Student → Teacher → Creator
**Type:** Strategic
**Topics:** naming, auth

**Trigger:** User identified that "visitor," "user," and "member" were used interchangeably across the platform, leading to confusion about what each term means.

**Options Considered:**
1. Use "user" for everyone (status quo)
2. Formal hierarchy with distinct terms ← Chosen
3. Just distinguish "logged in" vs "not logged in"

**Decision:** Established a formal identity hierarchy: Visitor (unauthenticated) → Member (has account) → Student (enrolled) → Teacher (certified) → Creator (creates courses). These are progressive and non-exclusive (one person can hold multiple roles). In code, `users` table and `user`/`userId` variables stay (universal convention). In UI text, "member" for authenticated, "visitor" for unauthenticated.

**Rationale:** The hierarchy maps directly to the Peerloop flywheel — each role is an evolution of the previous. Clear definitions prevent code that conflates "user" (everyone) with "member" (authenticated) or "student" (enrolled). The code/UI split respects that `users` is the universal auth convention while "member" is the better UI term.

**Consequences:** Documented in GLOSSARY.md §1. Living documentation will be updated in Phase 4 of TERMINOLOGY block. Historical session docs exempt.

## 4. Schema Column Naming Convention
**Type:** Implementation
**Topics:** d1, naming

**Trigger:** The `st.id` bug and broader analysis of ambiguous foreign keys across the schema (e.g., `student_teacher_id` referencing `users.id` but reading like it references `student_teachers.id`).

**Options Considered:**
1. Rename only the columns that caused bugs
2. Adopt a comprehensive convention and rename all non-conforming columns ← Chosen
3. Use SQL aliases only (no schema changes)

**Decision:** Adopted conventions: entity PKs keep `id`; FKs use `{role}_id` for user references; actor columns use `{action}_by_user_id`; booleans use `is_`/`has_` prefix; scoped roles prefix with domain (`member_role`). 22 `_by` columns will be renamed, 3 high-ambiguity FKs, plus 1 minor rename.

**Rationale:** A consistent convention makes wrong usage visible at the point of writing, not after a bug is deployed. The `_by` → `_by_user_id` convention is the most aggressive change (~460 occurrences) but ensures every FK is unambiguously typed.

**Consequences:** ~926 occurrences across schema, src, and tests for column renames. Organized as Phase 3 of TERMINOLOGY block (after table renames). SQL sweep included to catch latent bugs exposed by the new names.

## 5. Four-Phase Execution Order for Terminology Work
**Type:** Strategic
**Topics:** workflow, naming

**Trigger:** The terminology standardization spans schema, code, components, routes, UI text, and documentation — too much for one session and too interdependent to do in arbitrary order.

**Options Considered:**
1. Do everything in one pass
2. Schema first, then code, then docs
3. Glossary → Tables → Schema → Surfaces (4 phases) ← Chosen

**Decision:** Execute in strict order: Phase 1 (GLOSSARY — establish the rules), Phase 2 (TABLES — rename `student_teachers` + components/routes), Phase 3 (SCHEMA — rename columns + SQL sweep), Phase 4 (SURFACES — UI text + living docs). Each phase is a separate commit. CURRENT-BLOCK-PLAN.md provides cross-session persistence.

**Rationale:** Glossary must exist before any renames (source of truth). Table names affect column naming decisions. Column renames are mechanical once tables are settled. UI text and docs come last because they reference everything else. Separate commits enable bisection if something breaks.

**Consequences:** CURRENT-BLOCK-PLAN.md created with full progress tracking. Estimated ~1,300+ total occurrences across all phases. Will span multiple sessions.
