## Learnings

### 1. Skills Are Prompt Injections, Not Functions
**Topics:** cc-workflow
**Context:** Debugging recurring r-end failures (stopping after step 2, missing TodoWrite items)
**Learning:** Skills don't create stack frames — they inject prompts into the same context window. Nesting 3 levels deep asks Claude to mentally track a return path across multiple prompt injections, which it does unreliably. The deeper the nesting and the more context each inner skill consumes, the more likely the outer instructions are lost.

### 2. Collector Pattern Eliminates Redundant Conversation Scans
**Topics:** cc-workflow
**Context:** Designing how agents would get conversation data without each copying the full history
**Learning:** Instead of 4 agents each re-scanning the full conversation (expensive, redundant), scan once in the main context and write a structured extract file. Agents then read only their relevant sections — a materialized view pattern. This also creates a debugging artifact (the extract file) that the nested-skill approach completely lacked.

### 3. Two-Level Prefix Matching Pattern in sync-gaps.sh
**Topics:** docs-infra
**Context:** Fixing false-positive "undocumented route" flags for webhooks/bbb-analytics
**Learning:** When a shell script matches route prefixes, generic prefixes (e.g., `webhooks`) shadow specific ones (e.g., `webhooks/bbb-analytics`) if the script takes the first grep match. The fix is to check multi-level prefixes first — same pattern already existed for `me/*` routes, just needed extending to `webhooks/*`. Any future prefix namespace with sub-prefixes can be added to the same condition.

## Decisions

### 1. Replace Nested Skill Calls with Collector + Agent Dispatch
**Type:** Strategic
**Topics:** cc-workflow, dual-repo
**Trigger:** Recurring r-end failures: stopping after r-eos completes (forgets steps 3-8), missing TodoWrite items from r-docs
**Options Considered:**
1. Flatten to monolith — single 800+ line skill with all logic inline ← trades return-stack problem for "forgets later steps in long prompt"
2. Inlined checklist driver — keep standalone skills but r-end reimplements critical parts ← worst of both: duplication AND long prompt
3. Agent-based parallelism with collector ← Chosen
**Decision:** Build r-end2 using collector + agent dispatch. Main context scans conversation once into a structured extract file, dispatches 4 agents in parallel to process it, then runs save-state/commit/push inline.
**Rationale:** Solves both problems: (1) main context stays lean so it reliably reaches tail steps, (2) each agent gets isolated context so no bleed-through. Collector eliminates redundant conversation scans. Extract file doubles as debugging artifact.
**Consequences:** Created new r-end2 skill with 5 files (SKILL.md + 4 refs). Standalone skills (r-learn-decide, r-dump, etc.) remain for independent use. Old r-end preserved for comparison.

### 2. Extract Files Preserved in docs/sessions/ (Temporary Decision)
**Type:** Implementation
**Topics:** cc-workflow, docs-infra
**Trigger:** Deciding where to store the conv extract — /tmp/ (ephemeral) vs docs/sessions/ (preserved)
**Options Considered:**
1. /tmp/conv-extract-NNN.md — ephemeral, doesn't pollute repo
2. docs/sessions/{MONTH}/ — preserved alongside Learnings/Decisions/Dev files ← Chosen
**Decision:** Preserve extract files in docs/sessions/ for now. Review after one week of use to decide if they should be kept permanently or made ephemeral.
**Rationale:** Extract files are valuable for debugging agent behavior and calibrating collector quality during the initial rollout period.
**Consequences:** Extract files will accumulate in docs/sessions/. One-week review needed.

### 3. Shared Reference Files for Agent Format Rules
**Type:** Implementation
**Topics:** cc-workflow
**Trigger:** How to give agents the formatting rules from standalone skills
**Options Considered:**
1. Inline rules in agent prompts within SKILL.md — makes SKILL.md very large
2. Have agents read standalone skill SKILL.md files — couples r-end2 to internal structure of other skills
3. Extract format rules into shared refs/ files ← Chosen
**Decision:** Create `refs/fmt-{name}.md` files that agents read for format rules. Standalone skills remain the source of truth; refs files are curated extracts of format rules only.
**Rationale:** Decouples agent format rules from standalone skill execution flow. Agents get focused instructions without pre-computed context or confirmation templates they don't need.
**Consequences:** Created 4 refs files. Potential for drift between refs and standalone skills — but refs are small and focused, making drift easy to spot.

### 4. Uncategorized Section as Teaching Signal
**Type:** Implementation
**Topics:** cc-workflow
**Trigger:** User asked about catch-all for unusual observations
**Options Considered:**
1. No catch-all — strict categorization only
2. Uncategorized section for unusual observations ← Chosen
**Decision:** Add an "Uncategorized" section to the extract for anything that doesn't fit the standard categories. Use it to identify patterns that suggest new categories or improved collector logic.
**Rationale:** The collector is the single point of failure. This section catches what it might otherwise miss, and over time reveals whether the extract format needs expanding.
**Consequences:** Extract files have an §Uncategorized section. Review periodically to see if patterns emerge.
