# Session Decisions - 2026-03-10

## 1. Separate Skill for Dual-Repo Timecards
**Type:** Strategic
**Topics:** cc-workflow

**Trigger:** User manually runs `/q-timecard` twice (once per repo) and merges the output by hand every session.

**Options Considered:**
1. Extend `/q-timecard` with a `repo=both` mode
2. Create a new `/q-timecard-dual` skill ← Chosen
3. Have `/q-timecard` chain to itself for the second repo

**Decision:** Created `/q-timecard-dual` as a standalone skill rather than extending `/q-timecard`.

**Rationale:** The single-repo skill is clean and composable with simple arguments. The merge logic (unified header, combined sections, two Git History blocks) is a different concern. Keeping them separate means both remain simple and avoids argument complexity. A dual skill calling the single skill was rejected because `/q-timecard` opens the editor as a side effect.

**Consequences:** 11 skills total. Both timecard skills share the same formatting rules (git history formatting, Dataview compliance) but are maintained independently.

## 2. Use Fixed /tmp/timecard.md Instead of Session-Numbered Files
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** Found 11 accumulated `/tmp/timecard-session-*.md` files — disposable output that was never cleaned up.

**Options Considered:**
1. Keep session-numbered names, add cleanup logic ← Rejected
2. Use fixed `/tmp/timecard.md` that overwrites each run ← Chosen

**Decision:** Both `/q-timecard` and `/q-timecard-dual` now write to `/tmp/timecard.md`.

**Rationale:** Timecards are copy-once-and-discard. Session-numbered names are useful when comparing outputs side-by-side, but that never happens with timecards. Fixed filename = self-cleaning via overwrite.

**Consequences:** Removed session number pre-computation from q-timecard-dual. Cleaned up 11 old files.

## 3. Reject Marker-Based "latest" Auto-Detection for Timecards
**Type:** Strategic
**Topics:** cc-workflow

**Trigger:** Considered adding `/q-timecard-dual latest` to auto-detect uncommitted-to-timecard commits using a marker file (similar to `.last-qdocs-run`).

**Options Considered:**
1. Add marker tracking with `latest` argument ← Rejected
2. Keep explicit `c2d3` count syntax ← Chosen

**Decision:** Do not implement marker-based auto-detection for timecards.

**Rationale:** Five edge cases identified — the most critical being premature marker advance (marker updates at generation time, but user might not copy the output, losing those commits from future `latest` runs). Also: no sane first-run fallback, cross-skill coordination needed between single and dual timecard skills, and confirmation workflow adds friction. The explicit `c2d3` syntax costs ~5 seconds but eliminates an entire class of state-tracking bugs.

**Consequences:** Argument syntax stays simple. No shared state file between timecard skills.
