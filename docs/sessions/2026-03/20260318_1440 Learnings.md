# Conv Learnings - 2026-03-18

## 1. SQLite Partial Unique Index Cascades Through Test Suite
**Topics:** d1, testing

**Context:** Added `CREATE UNIQUE INDEX ... WHERE status IN ('enrolled', 'in_progress')` to enforce one active enrollment per student+course. Multiple test files broke because they inserted multiple active enrollments for the same (student_id, course_id) using `INSERT OR IGNORE`, expecting those to be independent.

**Learning:** Partial unique indexes in SQLite/D1 are powerful but have blast radius across the test suite. Tests that use `INSERT OR IGNORE` for the same composite key will silently fail when a partial unique index blocks them — the insert is ignored, leaving stale data from a prior test. This makes failures cascade and hard to diagnose.

**Pattern:** When adding a unique index, grep the test suite for all INSERT statements touching the constrained columns. For each test file, either: (1) add `beforeEach` cleanup that deletes dependent rows (payment_splits, transactions, notifications) before the enrollment, or (2) use unique student/course IDs per test block.

## 2. CurrentUser Map Keyed by courseId Needs Priority Logic for Re-Enrollment
**Topics:** astro, d1

**Context:** The `CurrentUser.enrollments` Map was built with `new Map(data.enrollments.map(e => [e.courseId, e]))` — last entry wins. With re-enrollment allowing multiple rows per student+course, the Map needed to prefer active enrollments over completed ones.

**Learning:** When a Map is keyed by a business identifier (courseId) rather than a row ID, and the underlying data can have multiple rows per key, you need explicit priority logic during Map construction. A simple `enrollmentPriority()` function (enrolled=4, in_progress=3, completed=2, cancelled=1) with a "keep higher priority" check during iteration solved this cleanly.

## 3. Unauthenticated Availability Summary — Slot Counts Are Low-Sensitivity
**Topics:** auth, astro

**Context:** Needed to decide whether the course availability preview endpoint should require authentication. Teacher names and ratings are already public on the course page.

**Learning:** Showing slot COUNTS and next-available DATES (not exact times) is low-sensitivity information that doesn't meaningfully expose teacher schedules beyond what's already public. The exact booking slots (specific times) remain behind the authenticated booking flow. This two-tier approach — public summary for browsing, authenticated details for booking — is a clean privacy boundary.

## 4. Enrollment Guards as a Shared Module Simplifies Multi-Consumer Validation
**Topics:** d1, astro

**Context:** Both the checkout endpoint and admin enrollment endpoint needed the same set of enrollment validations (creator check, active teacher check, duplicate check, no-teachers check). Previously each had its own inline duplicate check with slightly different SQL.

**Learning:** Extracting guards into `src/lib/enrollment-guards.ts` with a single `checkEnrollmentGuards()` function that returns a typed result object (`EnrollmentGuardResult`) made it trivial to add the same validation to both consumers. The result object pattern (allowed + reason + metadata) is cleaner than throwing errors because it lets the caller decide the HTTP status and message.

## 5. Test Cleanup for FK-Constrained Tables Needs Reverse Dependency Order
**Topics:** d1, testing

**Context:** Adding `beforeEach` cleanup for enrollments in the Stripe webhook tests failed with FK constraint errors — `payment_splits` and `transactions` reference `enrollments`.

**Learning:** When cleaning up test data in tables with foreign keys, delete in reverse dependency order: payment_splits → transactions → notifications → enrollments. D1/SQLite enforces `foreign_keys=ON` by default, so you can't just delete the parent row. The `beforeEach` cleanup pattern should mirror the migration's DROP TABLE order (children first).
