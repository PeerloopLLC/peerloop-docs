# Session Decisions - 2026-03-12

## 1. No Code-Level Tab Isolation — Use Two Browser Vendors Instead
**Type:** Strategic
**Topics:** testing, auth, workflow

**Trigger:** User wanted multiple browser tabs to have independent sessions for faster multi-user testing (student + teacher in same BBB session, testing both sides of a transaction without logging out/in).

**Options Considered:**
1. Tab-scoped auth tokens — login returns JWT in response body, stored in sessionStorage, sent via Authorization header. Server prefers header over cookie. Medium-large effort, minor SSR flash.
2. Dev-mode impersonation — `?_as=userId` param, server middleware impersonates in DEV mode. Small-medium effort, requires admin login first.
3. Hybrid — option 1 for client-side, option 2 for SSR pages.
4. Two browser vendors (Chrome + Safari) — zero code changes ← Chosen

**Decision:** Use two different browser vendors for multi-user manual testing. No code changes to the production codebase.

**Rationale:** User explicitly did not want to burden the production codebase with testing artifacts. Two browsers provide genuinely independent cookie jars and localStorage with zero implementation cost. Covers the primary use case (two simultaneous users) completely.

**Consequences:**
- No LOCALSTORAGE plan block created
- Documented the practice in BEST-PRACTICES.md §8, CLI-TESTING.md, and dev-setup.md
- Saved as feedback memory for future sessions
