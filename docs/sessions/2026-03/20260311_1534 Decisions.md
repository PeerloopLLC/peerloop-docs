# Session Decisions - 2026-03-11

## 1. Reschedule vs Cancellation: 30-Minute Threshold
**Type:** Strategic
**Topics:** booking, notifications

**Trigger:** When a student reschedules, both a cancel and a new booking occur. Should both parties receive "cancelled" + "booked" notifications, or a single "rescheduled" notification?

**Options Considered:**
1. Always send separate cancel + book notifications
2. Use a `reschedule: true` flag from the UI, always treat as reschedule ← Partial
3. Time-based: if rebook within 30 minutes of cancel, it's a reschedule; otherwise it's a cancellation ← Chosen

**Decision:** If a new session is booked within 30 minutes of cancelling the old one (detected via `cancelled_at` timestamp), send "Session Rescheduled" notifications. Beyond 30 minutes, send separate "Session Cancelled" and "Session Booked" notifications.

**Rationale:** Covers both the explicit reschedule flow (cancel + rebook in same action, milliseconds apart) and the edge case where someone cancels manually then quickly rebooks. The 30-minute window is generous enough for manual rebooking but short enough to distinguish from a genuine cancel-then-later-rebook.

**Consequences:** POST `/api/sessions` reads `cancelled_at` from the old session to calculate elapsed time. DELETE must happen before POST in the client flow. The `reschedule: true` flag on DELETE suppresses standalone cancel in-app notifications (emails still sent).

## 2. Reschedule Still Subject to Late Cancellation Penalty
**Type:** Strategic
**Topics:** booking, policies

**Trigger:** Should rescheduling bypass the late-cancellation penalty (< 24h before session)?

**Options Considered:**
1. Reschedules bypass late-cancel rules entirely
2. Reschedules are still subject to late-cancel rules ← Chosen

**Decision:** The unbooking part of a reschedule is still subject to the late cancellation penalty. The `is_late_cancel` flag is set on the cancelled session regardless of whether it's a standalone cancel or part of a reschedule.

**Rationale:** From the teacher's perspective, a late reschedule has the same impact as a late cancel — they lose a slot they could have filled. The policy protects teacher availability.

## 3. All Parties Receive All Booking/Cancel Notifications
**Type:** Implementation
**Topics:** notifications, booking

**Trigger:** Previously only the teacher received late-cancel in-app notifications. Student received no cancel notifications. Regular cancellations sent no in-app notifications at all.

**Options Considered:**
1. Keep selective notifications (teacher-only for late cancel)
2. Notify all involved parties for every booking event ← Chosen

**Decision:** Both student and teacher receive in-app notifications for all booking events: new booking, cancellation, and reschedule.

**Rationale:** Both parties need to know about schedule changes. The previous teacher-only notification was an incomplete implementation, not a deliberate policy choice.

**Consequences:** New `notifySessionCancelled()` and `notifySessionRescheduled()` helpers added to `notifications.ts`. Old `notifySessionLateCancelled()` deprecated (redirects to new function). DELETE endpoint sends cancel notifications to both parties. POST sends reschedule notifications to both parties when rescheduling.

## 4. Defer Full UTC Timezone Refactor
**Type:** Strategic
**Topics:** booking, architecture

**Trigger:** Date display bug (March 16 → March 15) revealed that the entire timezone chain is broken: teacher availability stored in local time, slots generated in server timezone, sessions stored as naive datetimes.

**Options Considered:**
1. Fix the display bug minimally and refactor timezone handling later ← Chosen
2. Do the full UTC-everywhere refactor now

**Decision:** Apply minimal `parseLocalDate()` fix for the date display bug. Plan the full UTC refactor as a separate PLAN block: convert availability to UTC server-side, return UTC slots, store sessions as UTC, convert to local on client display.

**Rationale:** The timezone issue doesn't affect local dev (server TZ matches test data). The refactor touches the availability API, booking component, session storage, and all display code — a cross-cutting change best done as a focused block rather than mid-session.

**Consequences:** Added UTC-TIMES block to PLAN.md. Current slot system (1-hour on-the-hour) stays as-is — only the timezone handling changes.
