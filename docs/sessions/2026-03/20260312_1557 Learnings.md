# Session Learnings - 2026-03-12

## 1. HTTP-Only Cookies Are the Real Constraint for Multi-Tab Sessions
**Topics:** auth, testing

**Context:** User wanted per-tab localStorage isolation so different browser tabs could be logged in as different users simultaneously (e.g., student + teacher testing BBB sessions).

**Learning:** Scoping localStorage alone is insufficient for multi-user tabs. The real constraint is HTTP-only auth cookies (`peerloop_access`, `peerloop_refresh`) which are shared across all tabs in the same browser and cannot be scoped per-tab. Any tab-isolation solution must address the cookie layer, not just localStorage. The `window.__peerloop` in-memory global and `sessionStorage` are already naturally per-tab, but the server-side auth (which drives SSR rendering and API responses) always sees the same cookie.

## 2. Two Browser Vendors Give Free Session Isolation
**Topics:** testing, workflow

**Context:** After analyzing three code-level approaches (tab-scoped auth tokens, dev-mode impersonation, hybrid), the user asked whether two different browser vendors would work.

**Learning:** Different browser vendors (Chrome, Safari, Firefox) have completely independent cookie jars and localStorage. This gives full session isolation with zero code changes — each browser is a genuinely independent client. For two-user testing (the 90% case), this is the simplest and most robust approach. The limitation is scaling: 3+ simultaneous users requires incognito windows or additional browsers.

**Pattern:**
| Browser | User | Use Case |
|---------|------|----------|
| Chrome | Primary test user (student) | Enroll, book, join BBB |
| Safari | Counterparty (teacher) | View booking, join same BBB |

## 3. localStorage Usage Is Concentrated in 7 Files with 5 Keys
**Topics:** auth, astro

**Context:** Audited all localStorage usage to scope a potential LOCALSTORAGE plan block.

**Learning:** Peerloop's localStorage footprint is small and well-categorized:
- **User auth** (2 keys): `peerloop_user_cache`, `peerloop_was_logged_in` — in `current-user.ts`
- **Stripe recovery** (1 key): `pl_pending_sessions` — in `EnrollButton.tsx` + `MyCourses.tsx`
- **Theme** (1 key): `theme` — in `UserAccountDropdown.tsx` + `MoreSlidePanel.tsx`
- **UI dismissals** (2 keys): `peerloop:recs:*:dismissed` — in recommendation components

No utility wrapper exists — all calls are direct `localStorage.getItem/setItem`. No `sessionStorage` usage anywhere. The `window.__peerloop` singleton pattern handles in-memory cross-island state.
