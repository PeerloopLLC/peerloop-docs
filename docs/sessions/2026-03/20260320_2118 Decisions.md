# Conv Decisions - 2026-03-20

## 1. Unified Gradient+Initial Avatar Fallback Pattern
**Type:** Implementation
**Topics:** astro

**Trigger:** 15 components used `placehold.co` external service for avatar fallbacks. Others used `UserAvatar` with gradient+initial. Inconsistent visual treatment and external dependency.

**Options Considered:**
1. Keep placehold.co (status quo)
2. Use `UserAvatar` component everywhere ← Chosen (with `xs` size addition)
3. Create a utility function returning gradient classes

**Decision:** Eliminate all `placehold.co` references. Use `UserAvatar` component where sizes match (xs/sm/md/lg/xl), inline the same gradient+initial pattern for custom sizes. Added `xs` (h-6 w-6) size to `UserAvatar`.

**Rationale:** No external dependency, faster rendering, consistent branded look, works offline. The gradient pattern (`bg-linear-to-br from-primary-400 to-primary-600`) is the universal fallback.

**Consequences:** 14 source files updated, 3 test files updated. Zero `placehold.co` references remain in codebase.

## 2. Community Cover Images as Icon Replacement (Not Banner)
**Type:** Implementation
**Topics:** astro

**Trigger:** Communities have `cover_image_url` in the schema and API responses, but it was never rendered in the UI. User requested adding images to community displays.

**Options Considered:**
1. Full-width banner above community name
2. Replace icon square with cover image ← Chosen
3. Both — banner with icon overlay

**Decision:** Replace the existing icon/emoji square (w-12/w-14 rounded-lg) with the cover image when available. Fallback chain: cover image → emoji icon → gradient.

**Rationale:** Minimal layout change, consistent with existing card patterns, works in all contexts (cards, headers, feed directory). Banner would require layout restructuring across 6+ pages.

**Consequences:** Updated 6 community-related pages/components. Community cover images now visible on detail pages, index page, discover page, and creator dashboard.

## 3. Server-Side Feed Activity Avatar Enrichment
**Type:** Implementation
**Topics:** stream, d1, astro

**Trigger:** Feed posts showed inconsistent user avatars. Older Stream.io activities stored `null` for `userImage` because the field wasn't added to the POST payload until recently.

**Options Considered:**
1. Backfill Stream activities (destructive, complex API)
2. Enrich on read from D1 ← Chosen
3. Accept stale avatars

**Decision:** Add `enrichActivitiesWithAvatars()` to `feed-activity.ts`. All 3 feed GET endpoints (townhall, community, course) call it before returning activities. Single batch query for all unique actors per page.

**Rationale:** Solves both historical data and future avatar changes. One DB query per page load. Failure is swallowed — stale avatars are better than broken feeds. No Stream API changes needed.

**Consequences:** New function in `feed-activity.ts`, 3 feed API endpoints updated. All feed posts now show fresh user avatars regardless of when they were created.

> **Insight:** This is a common pattern in activity feed systems — "denormalize at write, enrich at read." Stream stores the activity's core content, D1 provides fresh user metadata. It's the same approach Twitter uses for timeline rendering.

## 4. The Commons Cover Image in Core Seed
**Type:** Implementation
**Topics:** d1

**Trigger:** The Commons cover image was in the dev seed as an UPDATE, but The Commons is a production entity created in `0002_seed_core.sql`.

**Options Considered:**
1. Keep as dev-seed UPDATE (status quo)
2. Move to core seed INSERT ← Chosen

**Decision:** Add `cover_image_url` to The Commons INSERT in `0002_seed_core.sql` using a stable Unsplash URL. Remove the dev-seed UPDATE for The Commons.

**Rationale:** Production entities should have their data in the production seed. The Commons will always have a cover image — it's not test data. Uses Unsplash (stable, content-appropriate) not picsum (random).

**Consequences:** Core seed updated, dev-seed UPDATE removed. Dev communities still use picsum in their INSERTs.
