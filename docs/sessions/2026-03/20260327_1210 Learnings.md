# Conv Learnings - 2026-03-27

## 1. BBB Recording Downloads Require Cookie-Based Two-Step Auth
**Topics:** bbb, video, recordings, blindside

**Context:** Fred Dixon (Blindside Networks CEO) explained the recording download mechanism in response to Fraser's email

**Learning:** Blindside Networks recordings are `.m4v` (MPEG-4) format. Download requires two HTTP requests: (1) GET the capture page URL to receive a session cookie, (2) GET the `.m4v` file URL with that cookie. Direct download without the cookie fails. The `record_id` in the URL corresponds to BBB's `internal_meeting_id`.

**Pattern:** `fetchWithCookieAuth()` in `r2.ts` — fetch with `redirect: 'manual'`, extract `Set-Cookie`, use cookie in second request.

**Topics:** bbb, analytics, architecture

**Learning:** The codebase was more prepared than PLAN.md indicated: `meta_analytics-callback-url` was already passed in `createRoom()` (join.ts:135), the endpoint existed (bbb-analytics.ts), JWT verification was implemented, and `session_analytics` table was ready. The only missing piece was deployment and vendor confirmation.

## 3. BBB Webhooks Are Self-Configuring Per-Meeting
**Topics:** bbb, webhooks, staging, architecture

**Context:** While designing the staging webhook strategy, analyzed how each vendor's webhooks are configured

**Learning:** BBB callback URLs (both `meta_endCallbackUrl` and `meta_analytics-callback-url`) are set per-meeting dynamically from `request.url.origin` in `join.ts`. This means they automatically point to whatever domain served the page — no vendor-side URL configuration needed per environment. This is fundamentally different from Stripe, where webhook endpoints are hardcoded in the Dashboard.

## 4. R2 Bucket Isolation Was Missing Between Production and Preview
**Topics:** cloudflare, r2, staging, infrastructure

**Context:** While designing the staging webhook strategy, discovered that `wrangler.toml` had the same R2 bucket (`peerloop-storage`) for both production and preview environments

**Learning:** D1 and KV were already properly separated between environments, but R2 was shared. This meant any test recording replication on preview would write to production R2. Created `peerloop-storage-staging` and updated preview config to use it.

## 5. Webhook Payload Capture Strategy for Test Fixtures
**Topics:** testing, webhooks, fixtures

**Context:** User asked how to catch real webhook payloads on staging for examining variability

**Learning:** A `webhook_log` table with fire-and-forget INSERTs at the top of each webhook handler provides durable payload capture without blocking processing. Payloads can later be extracted via `wrangler d1 execute` for fixture generation. Auth headers are redacted (logged as `<redacted>`) for security.
