# Session Learnings - 2026-03-01

## 1. Route Scanner as Before/After Verification Tool
**Topics:** astro, testing, workflow

**Context:** Session 315 built a route adjacency scanner (`scripts/route-matrix.mjs`) that reported 26 broken link targets. This session used it as a verification tool after fixing stale routes.

**Learning:** Running the scanner before and after changes gives measurable proof of fix completeness (26 → 4). The remaining 4 were all expected: 1 dynamic route that works at runtime, 3 scanner false positives (phantom references from comments or regex mismatches). This "measure, fix, re-measure" approach is much more reliable than manual inspection.

## 2. Astro Dynamic Routes Create Scanner False Positives
**Topics:** astro, testing

**Context:** The scanner reported `/community/the-commons` as broken because no literal `.astro` file exists at that path, even though `community/[slug]/index.astro` serves it at runtime.

**Learning:** Static analysis of Astro routes can't resolve dynamic `[slug]` parameters. Any scanner that maps file-system `.astro` files to routes will report false positives for dynamic routes used with specific slugs. These should be documented as expected exceptions rather than treated as bugs.

## 3. Tests That Assert href Values Serve as Route Contracts
**Topics:** testing, astro

**Context:** After updating 42 stale `/dashboard/*` routes across 15 source files, the test suite caught exactly 9 test assertions still using old URLs. All were in test files asserting `toHaveAttribute('href', '/dashboard/...')`.

**Learning:** Tests that verify link `href` values act as an inadvertent "route contract" — they enforce that navigation targets match expectations. When routes are refactored, these tests immediately surface the mismatch. This is a useful pattern: even simple assertions like `expect(link).toHaveAttribute('href', '/learning')` catch stale routes that grep might miss in complex template expressions.

## 4. Role-Aware Route Splitting is Cleaner Than Generic Workspace URLs
**Topics:** astro, workflow

**Context:** Two files used `/workspace` as a catch-all post-action URL. The refactor split it: creators → `/creating`, teachers → `/teaching`, students → `/learning`.

**Learning:** Generic "workspace" URLs that try to serve multiple roles create ambiguity — where does a user who is both a creator and teacher land? Explicit role-based routes eliminate this ambiguity and make the navigation intent clear. In `AppNavbar.tsx`, the split also allowed checking `canCreateCourses` before `canTeachCourses`, giving creators priority.
