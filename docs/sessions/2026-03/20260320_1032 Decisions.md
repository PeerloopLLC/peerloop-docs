# Conv Decisions - 2026-03-20

## 1. Smart Feed Card Visual System — Source-Type Color Differentiation
**Type:** Implementation
**Topics:** astro, stream

**Trigger:** All Smart Feed cards looked identical — white background, gray border. No visual way to tell if a post is from The Commons, a community, or a course feed.

**Options Considered:**
1. Color-code cards by source type with left accent bar + background tint ← Chosen
2. Add source type label/icon to header only
3. Use different card layouts per source type

**Decision:** Cards are visually differentiated by source type using a `border-l-4` colored accent bar and background tint: townhall=primary (sky-blue), community=secondary (neutral/white), course=indigo. Right-side image strip shows course thumbnail or community cover when available.

**Rationale:** Left accent bar is scannable at a glance in mixed feeds, takes zero horizontal space, and works alongside the existing highlight ring system. Follows established color conventions from FeedsHub and MyFeeds components.

**Consequences:** FeedActivityCard now self-determines its color from activity data (no new props needed). Single-feed views (CommunityFeed, CourseFeed) show uniform color — consistent and fine.

> **Insight:** Indigo was used instead of blue for courses because the project's custom `primary` is sky-blue — standard Tailwind blue is visually indistinguishable at light tint levels.

## 2. Smart Feed Navigation: MyFeeds Widget as Hub Connector
**Type:** Implementation
**Topics:** astro

**Trigger:** `/feed` (Smart Feed) was only reachable from `/feeds` (FeedsHub) — one click removed from primary navigation. Dashboards had no direct link.

**Options Considered:**
1. Add `/feed` to AppNavbar directly
2. Add "View Smart Feed" link to MyFeeds widget on all dashboards ← Chosen
3. Replace `/feeds` nav link with `/feed`

**Decision:** Added "View Smart Feed" link at top of MyFeeds widget. Since MyFeeds already appears on `/learning`, `/teaching`, and `/creating`, this gives all 3 dashboards a direct path to the Smart Feed without changing the navbar.

**Rationale:** MyFeeds is the natural hub connector — it already shows feed-related content with badge counts. Adding the link there contextualizes it alongside the user's feed list.

## 3. client:only="react" for useCurrentUser() Components
**Type:** Implementation
**Topics:** astro, auth

**Trigger:** Hydration mismatch errors on `/feeds` and `/learning` pages.

**Decision:** Use `client:only="react"` instead of `client:load` for any component that branches on `useCurrentUser()` in its first render. This skips SSR entirely, avoiding the skeleton-vs-content mismatch.

**Rationale:** `useCurrentUser()` reads from `window.__peerloop` which doesn't exist during SSR. Server renders loading skeleton, client renders populated state — HTML trees don't match. Since these are authenticated-only pages, there's no SEO benefit to SSR.

## 4. Smart Feed Page Size as Admin-Tunable Parameter
**Type:** Implementation
**Topics:** stream

**Trigger:** PAGE_SIZE was hardcoded to 20 in SmartFeed.tsx. All other Smart Feed parameters (diversity cap, discovery frequency, scoring weights) were already admin-tunable via platform_stats.

**Decision:** Added `smart_feed_page_size` to platform_stats (default 20). API returns it in the response; component syncs from server on first load.

**Rationale:** Consistency with existing parameter model. Admin can tune feed density without code deploy.

## 5. Picsum.photos for Seed Data Images
**Type:** Implementation
**Topics:** testing

**Trigger:** Seed data had no images for users, courses, or communities. Cards with source image strip had nothing to display.

**Options Considered:**
1. Picsum.photos deterministic URLs ← Chosen
2. Local image files in public/images/seed/
3. SVG color placeholders

**Decision:** Use `https://picsum.photos/seed/{slug}/WxH` for all seed images. Deterministic based on slug, stable across reseeds, zero files to commit.

**Rationale:** Zero maintenance, works immediately, gives distinct images per entity. Only requires network access (acceptable for dev).

## 6. IMAGE-MGMT Deferred to PLAN Block #35
**Type:** Strategic
**Topics:** astro, cloudflare

**Trigger:** Schema supports images for users, courses, and communities, but no upload endpoints exist. The Commons needs an admin-configurable cover image.

**Decision:** Image upload/management tracked as PLAN.md block #35 (IMAGE-MGMT), extending the existing FILE-UPLOADS block. Covers: user avatar upload, course thumbnail upload (creator), community cover upload (creator), admin override for The Commons image.

**Rationale:** Image upload is a distinct feature requiring R2 integration, upload endpoints, and UI — not something to add mid-feed-card work. Seed data with picsum URLs covers immediate testing needs.
