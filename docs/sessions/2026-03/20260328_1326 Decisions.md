## 2. Admin Tab Pattern Over FAB for Admin Actions
**Type:** Strategic
**Topics:** admin, ux, architecture

**Trigger:** User observed that admin role has no presence in regular page UIs and proposed admin-as-tab pattern

**Options Considered:**
1. Admin actions via FAB (existing ContextActionsPanel approach)
2. Admin tab alongside existing role tabs ← Chosen (user's idea)
3. Admin toolbar/banner at top of page

**Decision:** Created deferred block ADMIN-PAGE-ROLE (#38) in PLAN.md. When admin visits a page, an "Admin" tab appears (like existing role tabs) showing admin-specific content.

**Rationale:** Consistent with discover pages pattern (role tabs appear based on your roles). More discoverable than floating button. Handles the complexity of admin also being student/teacher/creator — each role gets its own tab.

**Consequences:** Admin actions in page UIs deferred to ADMIN-PAGE-ROLE block. FAB system remains available as complementary option.

> **Insight:** The admin-as-tab pattern avoids a common trap in role-based UIs: privileged users seeing fewer capabilities than they have. When admin actions live only in a separate `/admin` section, admins must context-switch between "browsing the platform" and "administering the platform." Surfacing admin capabilities inline via tabs keeps the admin in-flow while maintaining clean separation from regular user UI.

## 3. Inline Buttons Over Component System for Owner Shortcuts
**Type:** Implementation
**Topics:** role-aware, ux

**Trigger:** Need to add owner/self-view shortcuts to 4 pages

**Options Considered:**
1. Use existing ContextActionsPanel FAB system
2. Create new shared RoleActions component (per PLAN.md original design)
3. Add inline buttons to each page's existing header/hero ← Chosen

**Decision:** Add buttons directly into existing components (CourseHero, CreatorProfileHeader, TeacherProfileHeader, community Astro template) using getCurrentUser() pattern.

**Rationale:** Each page has different header structures and button styles. A shared component would need extensive props/configuration for minimal code savings. Inline approach matches what PublicProfile.tsx already does for own-profile detection.

**Consequences:** 4 focused file changes, no new components, consistent with existing patterns.
