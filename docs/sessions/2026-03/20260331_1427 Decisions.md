# Conv Decisions - 2026-03-31

## 1. Add create-homework as 7th run (not in Phase 3 spec)
**Type:** Implementation
**Topics:** plato

**Trigger:** submit-homework needs a homework assignment to exist, but no create-homework run was listed in Phase 3

**Options Considered:**
1. Include homework creation inside submit-homework run (less composable)
2. Add create-homework as separate run ← Chosen
3. Use SqlTopUp to insert assignment (breaks API-driven pattern)

**Decision:** Added create-homework as a separate atomic run

**Rationale:** More composable — scenarios can create assignments independently of submissions

**Consequences:** 7 runs instead of 6 in Phase 3

## 2. Propagate courseTitle to $runtime for cross-actor access
**Type:** Implementation
**Topics:** plato, runner

**Trigger:** enroll-student's findBy needs courseTitle but student actor can't see creator's persona

**Options Considered:**
1. Add courses[] to student personas (data duplication)
2. Propagate courseTitle to $runtime after flattenCourseData ← Chosen
3. Use $context from a creator discovery action (adds complexity to every run)

**Decision:** Runner auto-sets `$runtime.courseTitle` from any persona that has courseTitle, after every run step

**Rationale:** Zero-cost for runs that don't need it; transparently available for those that do

**Consequences:** Modified api-runner.ts executeScenario loop

## 4. Use $actor.student.userId directly in certify-teacher
**Type:** Implementation
**Topics:** plato, testing

**Trigger:** eligible_students[0] non-deterministic when multiple students complete same course

**Options Considered:**
1. Keep eligible_students[0] and control ordering (fragile)
2. Use $actor.student.userId directly ← Chosen

**Decision:** Changed certify-teacher body from `$context.viewTeachers.eligibleStudentId` to `$actor.student.userId`

**Rationale:** The run knows which student to certify — no need to discover from the list

**Consequences:** certify-teacher now works reliably with multiple completions of the same course
