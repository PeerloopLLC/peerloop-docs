# Session Learnings - 2026-03-05

## 1. SQLite ORDER BY with Identical Timestamps Is Non-Deterministic
**Topics:** d1, testing

**Context:** Integration test created 3 notifications in rapid succession using `now()`, then asserted the first item was the "newest" by type.

**Learning:** When multiple rows share the same `created_at` value (as happens when INSERT calls run within the same millisecond), SQLite does not guarantee a stable ordering with `ORDER BY created_at DESC`. The test failed because it assumed the last INSERT would sort first. The fix was to assert on the *set* of types present rather than positional ordering.

**Pattern:**
```typescript
// Fragile — depends on insert order matching timestamp order
expect(body.items[0].type).toBe('system');

// Robust — validates presence without order dependency
const types = body.items.map((n: { type: string }) => n.type).sort();
expect(types).toEqual(['new_follower', 'session_booked', 'system']);
```

## 2. Playwright Strict Mode and Partial Button Name Matching
**Topics:** testing, astro

**Context:** E2E test for filter tabs used `getByRole('button', { name: 'All' })` which matched both the "All" filter tab and the "Mark **all** as read" button.

**Learning:** Playwright's `getByRole` does substring matching by default. Short labels like "All" are especially prone to colliding with other elements containing the same word. Always use `{ exact: true }` for common/short button labels.

**Pattern:**
```typescript
// Fails: matches "Mark all as read" too
page.getByRole('button', { name: 'All' })

// Works: exact match only
page.getByRole('button', { name: 'All', exact: true })
```

## 3. Duplicate Headings from Astro Template + React Island
**Topics:** astro, testing

**Context:** The `/notifications` page has `<h1>Notifications</h1>` in both the Astro template (`notifications.astro`) and the `NotificationsList` React component. Playwright strict mode threw an error when targeting `getByRole('heading', { name: 'Notifications', level: 1 })`.

**Learning:** When Astro pages contain both a static heading in the `.astro` template and a heading in a `client:load` React island, E2E tests must disambiguate. Options: target by content unique to one (e.g., notification item text), use `.first()`, or scope the locator to a specific container.

## 4. E2E Mutations Require DB Reseed or Serial Isolation
**Topics:** testing

**Context:** E2E tests that mark notifications as read or delete them modify the shared dev DB, causing later tests to fail when they expect the original seed data.

**Learning:** Two approaches work: (1) Group mutation tests in a separate `test.describe` with `{ mode: 'serial' }` so they run in predictable order after read-only tests, or (2) reseed before each mutation test. The serial approach is simpler and faster for this use case.

## 5. AppHeader vs AppNavbar — Desktop Layout Has No Notification Badge
**Topics:** astro, testing

**Context:** E2E test tried to find the notification count badge via `AppHeader`'s `a[aria-label*="Notifications"]` selector, but it wasn't visible.

**Learning:** On desktop viewport (Playwright's default), `AppLayout` renders `AppNavbar` (left sidebar) without a notification count badge. The `AppHeader` with its badge is mobile-only. E2E tests must target what's actually visible at the test viewport. The notification count badge only appears in the `AppHeader` component which is hidden on desktop.

## 6. Notification System Is Pure D1 — No Stream.io Dependency
**Topics:** d1, stream

**Context:** User asked to confirm whether the notification system uses GetStream.

**Learning:** The notification system is entirely D1/database-driven — direct SQL INSERTs to a `notifications` table. Stream.io is used exclusively for activity feeds (Block 5 COMMUNITY). This keeps notifications simple, cost-free (no external API calls), and fully under our control. The `notifications` table schema has a CHECK constraint limiting types to 12 values.
