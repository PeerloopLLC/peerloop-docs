# Conv Learnings - 2026-03-28

## 1. Tag-Overlap EXISTS Subquery as Universal "Similar Items" Pattern
**Topics:** d1, sql, architecture

**Context:** Replacing `WHERE c.category_id = ?` (single-dimension similarity) with multi-tag overlap

**Learning:** The `EXISTS (SELECT 1 FROM course_tags ct1 JOIN course_tags ct2 ON ct1.tag_id = ct2.tag_id WHERE ct1.course_id = c.id AND ct2.course_id = ?)` pattern is a universal "similar items" query. SQLite can short-circuit after the first matching tag, making it efficient. This same pattern works for community matching, recommendation scoring, and feed discovery.

**Pattern:** Used consistently in 8 SSR pages and 5+ API endpoints after this conv.

## 2. Graduated Scoring: Weighted Overlap with Topic Bonus
**Topics:** smart-feed, scoring, architecture

**Context:** Upgrading from binary topic match (0 or 1) to continuous tag overlap

**Learning:** The formula `weight = 1.0 + topicAffinity` for each tag ensures: (a) base case with no posting history gives equal weight (simple intersection ratio), (b) active topics get proportionally more weight. Normalizing by `totalWeight` rather than `feedTags.size` accounts for the variable weights. This is a one-query, two-map pattern: `SELECT slug, tag_id, topic_id` builds both `feedTagsMap` and `tagTopicMap`.

## 3. primary_topic_id as Computed Property for Client-Side Filtering
**Topics:** architecture, ssr, d1

**Context:** Courses no longer have `category_id` but client-side filters need a topic dimension

**Learning:** Rather than denormalizing a `primary_topic_id` column onto courses, a correlated subquery `(SELECT tg.topic_id FROM course_tags ct JOIN tags tg ... ORDER BY tp.display_order LIMIT 1)` computes it at query time. Trade-off: ~1 subquery per course in result set (acceptable for SSR with LIMIT 50), avoids schema denormalization and stale data. The `primary_topic_id` only exists in the response/type layer, not in the database.

## 4. Mechanical Test Sweep at Scale
**Topics:** testing, tooling

**Context:** 157 test files, 266+ occurrences of `category_id` in test data fixtures

**Learning:** The dominant pattern was `category_id: 'cat-XXX'` as a property in inline course object literals. This is highly regular and automatable. However, edge cases required manual attention: SQL INSERT column lists (removing both column and corresponding `?` in VALUES), dangling commas from removed properties, and type collisions between mock-data and db/types `CourseTag`.
