# Conv Decisions - 2026-03-19

## 1. RESUME-STATE.md Becomes Transparent TodoWrite Persistence
**Type:** Strategic
**Topics:** cc-workflow, dual-repo

**Trigger:** User observed that with `/r-end` auto-saving state and `/r-start` loading it, RESUME-STATE.md is just a persistence layer — the user shouldn't have to think about it separately from TodoWrite.

**Options Considered:**
1. Keep RESUME-STATE.md as a user-visible artifact alongside TodoWrite
2. Make RESUME-STATE.md invisible — TodoWrite is the only interface ← Chosen

**Decision:** RESUME-STATE.md is now purely a serialization mechanism for TodoWrite tasks across conversation boundaries. Users interact only with TodoWrite. `/r-start` Step 7 reads RESUME-STATE.md, creates TaskCreate entries, then deletes the file. `/r-end` serializes pending tasks to RESUME-STATE.md before committing.

**Rationale:** One interface (TodoWrite) is simpler than two (TodoWrite + RESUME-STATE.md). The file is an implementation detail — git history preserves it for audit.

**Consequences:**
- Added Step 7 to `/r-start` skill (transfer + delete)
- Old Step 7 (resume) renumbered to Step 8
- Added `TaskCreate` to `/r-start` allowed-tools
- Updated rules to reference "Steps 1–7"
- Feedback memory saved for future conversations

> **Insight:** This follows the "transparent persistence" pattern common in frontend state management (Redux Persist, localStorage-backed hooks). The abstraction boundary is clear: TodoWrite owns the runtime state, RESUME-STATE.md owns the durable state, and the skills handle the serialization/deserialization at conversation boundaries.
