## 2. Manifest-based pruning over controller-side section removal
**Type:** Implementation
**Topics:** cc-workflow, r-end2, concurrency

**Trigger:** User identified that controller-side section removal is blunt — agents should signal what they consumed

**Options Considered:**
1. Controller removes entire sections by heading name — blunt, doesn't know if agent succeeded ← Initially implemented
2. Each agent prunes its own section from Extract — race condition on parallel writes ← Rejected
3. Agents append consumed line numbers to shared manifest, controller prunes by manifest ← Chosen

**Decision:** Agents append consumed line numbers to `/tmp/extract-manifest.txt` via atomic `echo >>`. Controller reads manifest after all agents complete, sorts descending, deletes listed lines, inserts pointer lines for orphaned headings.

**Rationale:** Agents know exactly which lines they cherry-picked. Manifest is append-only (no race conditions). Controller handles deletion at the natural synchronization point. Failed agents don't append, so their sections stay in the Extract as fallback.

**Consequences:** Updated agent 1 and 2 prompts with manifest instructions. Replaced Step 4b. Agent 3 and 4 don't need manifest (they don't produce session files that duplicate Extract content).

## 3. /api/me/dashboard-summary endpoint not needed
**Type:** Implementation
**Topics:** api-design, dashboard

**Trigger:** UNIFIED-DASHBOARD.FOLLOWUP item asking whether a single summary endpoint would be more efficient

**Options Considered:**
1. Add /api/me/dashboard-summary replacing 3 calls — duplicates logic from 3 existing endpoints
2. Add as supplementary 4th call for PriorityHeader — net worse (4 calls vs 3)
3. Keep current 3-parallel-call pattern (Promise.all) ← Chosen

**Decision:** No new endpoint. Current pattern is already efficient.

**Rationale:** Promise.all means wall-clock time = slowest of 3 calls, not the sum. A summary endpoint would either duplicate existing logic or add complexity without meaningful improvement.

**Consequences:** Marked PLAN.md follow-up item as done with rationale.
