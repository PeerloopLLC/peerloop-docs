# Conv Decisions - 2026-03-29

## 1. ADMIN-INTEL Block Structure — 6-Phase Approach
**Type:** Strategic
**Topics:** admin, architecture, planning

**Trigger:** User requested admin capabilities on member-facing pages; existing deferred block #38 ADMIN-PAGE-ROLE was insufficient

**Options Considered:**
1. Simple admin tab on course/community pages only (minimal scope)
2. 6-phase approach covering foundation, tabs, profiles, discover/members, dashboard, bidirectional links ← Chosen
3. Full admin page redesign (too large)

**Decision:** Adopt 6-phase ADMIN-INTEL block with foundation-first dependency model. All phases depend on Phase 1, then Phases 2-6 can proceed in parallel.

**Rationale:** The user's vision spans every entity type and multiple page surfaces. Phased approach allows incremental delivery while the single-source component pattern prevents duplication. Phase 1 foundation (color, API, badge, links) is reused by all subsequent phases.

**Consequences:** New PLAN block created with 40+ subtasks across 6 phases. Deferred #38 ADMIN-PAGE-ROLE superseded. Branch `jfg-dev-9` will be created for implementation. 12 new files, 14 modified files planned.

## 2. Admin Color: Red
**Type:** Implementation
**Topics:** admin, ui, role-tabs

**Trigger:** Need to visually distinguish admin tabs from existing role tabs

**Options Considered:**
1. Red (bg-red-500) ← Chosen
2. Rose (bg-rose-500)
3. Slate (bg-slate-500)

**Decision:** Use red for admin color — `bg-red-500`, `text-red-700`, with dark mode variants `dark:border-red-400 dark:text-red-400`.

**Rationale:** Red is the standard "admin" color, visually distinct from all existing role colors (blue, green, purple, amber), and carries the right connotation of authority/attention.

**Consequences:** Add `red` entry to `roleColorClasses` in CourseTabs.tsx and CommunityTabs.tsx. No change to ROLE_COLORS (typed Record<CourseRole, ...>, admin not a CourseRole).

## 3. Do NOT Add 'admin' to CourseRole Type
**Type:** Implementation
**Topics:** type-safety, role-tabs, admin

**Trigger:** Design question about whether to extend CourseRole union type

**Options Considered:**
1. Add 'admin' to CourseRole union — cleanest extension but ripples through types used across many files
2. Keep CourseRole as-is, handle admin as a special case in ExploreCourseTabs ← Chosen

**Decision:** Admin bypasses the CourseRole type system entirely. ExploreCourseTabs checks `currentUser.isAdmin` and directly constructs an ExtraTabConfig, skipping computeRoleTabs().

**Rationale:** Admin is orthogonal to course roles (an admin can simultaneously be student+teacher+creator for the same course). CourseRole is used across badge rendering, listing filters, and color lookups. Adding 'admin' would require changes in places that have nothing to do with admin functionality.

**Consequences:** No type system changes needed. Admin tab added via direct ExtraTabConfig construction in ExploreCourseTabs and ExploreCommunityTabs.

## 4. New /api/admin/intel/* Endpoints
**Type:** Implementation
**Topics:** api-design, admin, d1

**Trigger:** Deciding how to serve admin intel data to member-side components

**Options Considered:**
1. Client-side aggregation of existing admin list endpoints (reuses code but multiple fetches, heavy payloads)
2. New lightweight intel endpoints per entity type ← Chosen

**Decision:** Create new `/api/admin/intel/course/[id].ts`, `/api/admin/intel/user/[id].ts`, `/api/admin/intel/community/[id].ts`, and a batch `/api/admin/intel/courses.ts` endpoint.

**Rationale:** Existing admin endpoints are paginated list endpoints with heavy SQL joins. AdminIntel needs aggregated counts + last-5-items for a single entity — 3-4 simple indexed COUNT queries. New endpoints are lightweight, server-side auth-gated, and purpose-built.

**Consequences:** 4 new API files. Follow existing admin endpoint patterns (requireRole, getDB, queryFirst/queryAll). Batch endpoint uses `WHERE course_id IN (...)` for list badge overlay.

## 5. /discover/members as Admin-Only Page
**Type:** Strategic
**Topics:** admin, discover, navigation

**Trigger:** User wants unified member browser for admins while keeping Find Teachers / Find Creators for all users

**Options Considered:**
1. Replace Find Teachers / Find Creators with unified /discover/members for admins
2. Add /discover/members as admin-only alongside existing Find Teachers / Find Creators ← Chosen

**Decision:** Create new admin-only `/discover/members` page. Keep `/discover/teachers` and `/discover/creators` as-is for all users.

**Rationale:** User's explicit rationale: "admins typically stay inside their admin area and don't know their apps well. We are encouraging admins to be students, etc and to see how members in various roles work with the system." The existing discover pages serve different purposes than admin member browsing.

**Consequences:** New Astro page + React component. Add "Members" entry to DiscoverSlidePanel (admin-only, conditionally rendered). Add card to discover hub page (admin-only). Update url-routing.md.
