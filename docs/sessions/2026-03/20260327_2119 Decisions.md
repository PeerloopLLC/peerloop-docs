# Conv Decisions - 2026-03-27

## 1. Course Listing Card Content Per Role Tab
**Type:** Strategic
**Topics:** ui-design, explore-courses

**Trigger:** Visual review of all tab cards revealed different roles need different card information.

**Options Considered:**
1. Same card for all tabs -- Rejected (roles have different needs)
2. Different card content per tab <- Chosen

**Decision:** Each role tab gets tailored card content:
- Teaching: rating, teacher count, next session, detail page link
- Created: teacher count, View/Edit buttons, no Discussion toggle
- Student: progress bar, assigned teacher, action CTAs
- Moderating: no price/sessions/level; flagged count, feed link, last activity
- All: standard browse card with role badges

**Rationale:** Roles have fundamentally different relationships to courses -- teachers manage delivery, creators manage content, moderators manage community, students track progress.

**Consequences:** 4 distinct card designs across tabs. Moderating tab got a custom `ModerationCourseCard` component.

## 2. Revenue and Discussion Toggle Removed from Both Tabs
**Type:** Implementation
**Topics:** ui-design, explore-courses

**Trigger:** User noticed asymmetry -- revenue and Discussion toggle were only on Created tab, not Teaching.

**Options Considered:**
1. Add to both tabs
2. Remove from both tabs <- Chosen
3. Keep on Created only

**Decision:** Revenue left off both tabs (save for detail pages). Discussion toggle removed from Created tab cards (manage from dashboard detail page). View/Edit buttons kept on Created tab.

**Rationale:** Keeps cards focused on course health metrics, avoids sensitive financial data on listing cards. Discussion toggle is a management action better suited to the detail page.

**Consequences:** `CreatorCourseCard` got `showDiscussionToggle` prop (default true for Dashboard backward compat).

## 3. Visitor and No-Role Member Get Explicit Card CTAs
**Type:** Strategic
**Topics:** ui-design, conversion, explore-courses

**Trigger:** Visual review of visitor/member views showed no explicit action on cards.

**Options Considered:**
1. Card click-through only (no CTA) -- Rejected
2. Explicit CTA buttons <- Chosen

**Decision:** Visitor cards show "Join to Enroll" (-> `/signup`), no-role member cards show "Enroll" (-> course detail page). Users with any role on a course get no CTA (they already have context).

**Rationale:** These users are the conversion target -- explicit CTAs make the action obvious rather than relying on implicit card clickability.

**Consequences:** `ExploreCard` got a `cta` prop; `ExploreAllTab` computes the appropriate CTA based on `currentUser` and `roleBadges.length`.

## 4. Popular Courses Carousel at 75% Height
**Type:** Implementation
**Topics:** ui-design, explore-courses

**Trigger:** Old /discover/courses had a Popular Courses carousel that was a good discovery feature, missing from new page.

**Options Considered:**
1. Don't bring it back
2. Same size as old page
3. 75% height to de-emphasize <- Chosen

**Decision:** Bring back `RecommendedCourses` carousel on `/explore/courses` with `compact` prop -- card width reduced from `w-72` to `w-56`, maintaining aspect-video ratio for proportional height reduction.

**Rationale:** Carousel aids discovery but shouldn't overshadow the main grid. De-emphasis lets it serve as a quick-glance recommendation without dominating above the fold.

**Consequences:** `RecommendedCourses` got `compact?: boolean` prop. Old /discover/courses usage unaffected.

## 5. "Student . Completed" Badge Fix
**Type:** Implementation
**Topics:** ui-design, role-badges

**Trigger:** Badge showed "Completed . Completed" -- label and state were identical for completed students.

**Options Considered:**
1. "Enrolled . Completed"
2. "Student . Completed" <- Chosen

**Decision:** Changed non-active student badge label from "Completed" to "Student" to distinguish role from status.

**Rationale:** The badge format is `label . state`. When both are "Completed", the role context is lost. "Student" identifies the role, "Completed" identifies the state.

**Consequences:** `role-utils.ts` line 140: `isActive ? 'Enrolled' : 'Student'`. Two test assertions updated.
