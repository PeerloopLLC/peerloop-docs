# Conv Learnings - 2026-03-29

## 1. Admin Tab Bypasses CourseRole Type System
**Topics:** role-tabs, type-safety, admin

**Context:** Planning how to add admin tab to existing role-tab pages

**Learning:** Admin is a platform-level role, not a course relationship. Adding 'admin' to `CourseRole = 'student' | 'teacher' | 'creator' | 'moderator'` would ripple through badge rendering, listing filters, and color lookups across many files. Instead, bypass `computeRoleTabs()` entirely — ExploreCourseTabs checks `currentUser.isAdmin` directly and appends an `ExtraTabConfig` with `roleColor: 'red'`. The `roleColorClasses` map in CourseTabs/CommunityTabs is string-keyed (not CourseRole-keyed), so adding `'red'` there is a one-line change with no type system ripple.

**Pattern:** When extending a tab system with a role that's orthogonal to the existing role hierarchy, construct ExtraTabConfig directly rather than modifying the role computation function.

## 2. Single Source Component Pattern for Admin Content
**Topics:** architecture, admin, component-design

**Context:** Risk of duplicating admin content across 14+ surfaces (detail pages, list pages, profiles, dashboard)

**Learning:** Each entity type (course, community, member) should have ONE admin component that renders in full or compact mode. Other surfaces embed it — detail pages show full admin panel, list pages show a badge, dashboard shows summary. This mirrors the existing OnboardingNudgeBanner pattern (compact/full variants with context prop). The alternative — building separate admin views for each surface — would create maintenance nightmares as admin capabilities evolve.

## 3. Situation-Centric vs Entity-Centric Admin Perspectives
**Topics:** admin, ux-design, architecture

**Context:** User articulated the fundamental distinction between `/admin` pages and admin content on member pages

**Learning:** Two complementary admin perspectives: (1) Situation-centric (`/admin` pages) — "I have a situation, find who's involved" — tables, filters, queues, bulk actions. (2) Entity-centric (member-side admin content) — "I'm looking at THIS entity, what needs attention here?" — summary + links to /admin for deep dives. Bidirectional navigation between both worlds is essential. This pattern encourages admins to use the app as members, developing empathy for the user experience while retaining admin superpowers.

## 4. New Admin Intel API Endpoints Preferred Over Client-Side Aggregation
**Topics:** api-design, admin, performance

**Context:** Deciding whether to reuse existing admin list endpoints or create new lightweight intel endpoints

**Learning:** Existing admin endpoints are paginated list endpoints with heavy SQL joins returning full table data. AdminIntel needs lightweight aggregated summaries (counts + most-recent items) for a single entity. New `/api/admin/intel/*` endpoints are 2-3 simple files with targeted COUNT queries on D1 — much lighter than calling 3-5 existing endpoints and extracting counts client-side. Also keeps `requireRole(['admin'])` server-side, preventing non-admin clients from even attempting the calls.
