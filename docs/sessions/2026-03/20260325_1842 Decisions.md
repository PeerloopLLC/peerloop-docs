# Conv Decisions - 2026-03-25

## 1. Replace Nested Skill Calls with Collector + Agent Dispatch
**Type:** Strategic
**Topics:** cc-workflow, dual-repo

**Trigger:** Recurring r-end failures: stopping after r-eos completes (forgets steps 3-8), missing TodoWrite items from r-docs. Root cause: skills are prompt injections, not functions — nesting 3 levels deep loses the outer return path.

**Options Considered:**
1. Flatten to monolith — single 800+ line skill with all logic inline ← trades return-stack problem for "forgets later steps in long prompt"
2. Inlined checklist driver — keep standalone skills but r-end reimplements critical parts ← worst of both: duplication AND long prompt
3. Agent-based parallelism with collector ← Chosen

**Decision:** Build r-end2 using collector + agent dispatch. Main context scans conversation once into a structured extract file, dispatches 4 agents in parallel to process it, then runs save-state/commit/push inline.

**Rationale:** Solves both failure modes: (1) main context stays lean so it reliably reaches tail steps, (2) each agent gets isolated context so no bleed-through. Collector eliminates redundant conversation scans. Extract file doubles as debugging artifact.

**Consequences:** Created new r-end2 skill with 5 files (SKILL.md + 4 refs). Standalone skills (r-learn-decide, r-dump, etc.) remain for independent use. Old r-end preserved for comparison during one-week evaluation.

> **Insight:** The instinct to compose skills like functions comes naturally from a programming background, but LLM context windows don't have call stacks. The "materialized view" pattern — pre-compute shared data to disk, then dispatch isolated agents — is the LLM-native equivalent of function composition.

## 3. Shared Reference Files for Agent Format Rules
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** How to give agents the formatting rules from standalone skills without coupling to their internal structure.

**Options Considered:**
1. Inline rules in agent prompts within SKILL.md — makes SKILL.md very large
2. Have agents read standalone skill SKILL.md files — couples r-end2 to internal structure of other skills
3. Extract format rules into shared refs/ files ← Chosen

**Decision:** Create `refs/fmt-{name}.md` files that agents read for format rules. Standalone skills remain the source of truth; refs files are curated extracts of format rules only.

**Rationale:** Decouples agent format rules from standalone skill execution flow. Agents get focused instructions without pre-computed context or confirmation templates they don't need.

**Consequences:** Created 4 refs files (fmt-learn-decide.md, fmt-dump.md, fmt-update-plan.md, fmt-docs.md). Potential for drift between refs and standalone skills — but refs are small and focused, making drift easy to spot.

## 4. Uncategorized Section as Teaching Signal
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** Need a catch-all for unusual observations that don't fit standard extract categories.

**Options Considered:**
1. No catch-all — strict categorization only
2. Uncategorized section for unusual observations ← Chosen

**Decision:** Add an "Uncategorized" section to the extract for anything that doesn't fit the standard categories. Use it to identify patterns that suggest new categories or improved collector logic.

**Rationale:** The collector is the single point of failure for r-end2. This section catches what it might otherwise miss, and over time reveals whether the extract format needs expanding.

**Consequences:** Extract files have an §Uncategorized section. Review periodically to see if patterns emerge that warrant new categories.
