# Conv Learnings - 2026-03-18

## 1. DB Setup Script Naming Confusion
**Topics:** d1, workflow

**Context:** User asked to see all scripts that modify D1, then noted the naming was confusing.

**Learning:** The original DB setup combo scripts had several usability problems:
- `fullsetup` wasn't actually "full" (missing booking seed)
- `setup:clean` was ambiguous ("clean" could mean many things)
- `db:setup:booking` broke the `:{target}` suffix pattern (no `:local`)
- No script gave all seeds in one command
- The default (`db:setup:local`) included dev seed, requiring `:clean` to get minimal — backwards from intuition

**Pattern:** Name combo scripts so the base is minimal and suffixes add layers. Users shouldn't have to "subtract" from a default — they should "add" to a base.

---

## 2. Additive Seed Chaining Eliminates Ambiguity
**Topics:** d1, workflow

**Context:** After renaming, user refined the pattern to be strictly linear: each level chains through the previous one.

**Learning:** When seed files have a natural dependency order (dev data must exist before Stripe accounts can be linked, Stripe accounts must exist before booking scenarios make sense), the script chain should mirror that dependency. `db:setup:local:booking` calls `db:setup:local:stripe` which calls `db:setup:local:dev` which calls `db:setup:local`. Reading the name tells you exactly what you get — everything up to and including that level.

---

## 3. Speculative Scripts Violate YAGNI
**Topics:** workflow

**Context:** I added `:all` variants for both local and staging that combined all seeds. User asked where they were used — nowhere.

**Learning:** Don't add convenience scripts "because the gap exists." If no code, test, or workflow references it, it's speculative. The `:booking` variant already gives you everything (since it chains through `:stripe` → `:dev`), making `:all` redundant by design.
