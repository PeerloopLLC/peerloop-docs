# Session Learnings - 2026-03-09

## 1. Stale-While-Revalidate Cache Can Flash Broken UI After Deploys
**Topics:** astro, auth

**Context:** After deploying to staging, the AppNavbar content disappeared for users who had existing localStorage data. The navbar flashed briefly then vanished. Clearing two localStorage keys (`peerloop_user_cache`, `peerloop_was_logged_in`) restored it.

**Learning:** The stale-while-revalidate pattern loads cached `MeFullResponse` from localStorage and renders instantly (~0ms), then fetches fresh data from the API (~200-500ms). When a deploy changes the API response shape, the cached data is structurally incompatible — `JSON.parse()` succeeds but the component reads `undefined` for renamed/missing fields. React silently renders nothing (conditional blocks like `{user.displayName && <span>}` fail open), causing the flash-then-disappear behavior.

**Pattern:** The destructive case is only structural mismatch — semantic staleness (correct shape, stale values) is already handled by the background API refresh. The fix is a structural guard on cache read, not a full cache invalidation strategy.

## 2. Structural Guard Is More Surgical Than Build Version Key
**Topics:** astro, auth

**Context:** Evaluated three approaches to prevent stale cache flashes: (1) build version key, (2) schema validation, (3) structural guard.

**Learning:** A build-version cache key forces every user to see a loading flash on every deploy, even when the API shape didn't change. A structural guard only triggers when there's an actual mismatch — so users get snappy loads on most deploys. The guard checks only the minimum fields that the `CurrentUser` constructor would crash on (6 fields), not the entire `MeFullResponse` shape. This avoids maintenance burden — new optional fields don't need guard updates.

## 3. Parallel Agents Miss Scattered References in Large Files
**Topics:** cc-workflow

**Context:** During the docs folder reorganization, three parallel agents updated references across ~50 files. But DECISIONS.md (27 stale refs), BEST-PRACTICES.md (7), and POLICIES.md (3) still had stale references after the first pass.

**Learning:** Large files with many scattered references (especially in `**See:**` annotations throughout a 1800-line file) get partially processed by agents that have context limits. A dedicated cleanup pass after the initial parallel wave is necessary. Grep verification before committing is essential — don't trust agent completion reports alone.

**Pattern:** Run verification greps after every batch of reference updates: `grep -r "old_pattern" --include="*.md" . | grep -v "docs/sessions/"` to exclude historical files.

## 4. Docs Reorganization Follows Same Cognitive Threshold Pattern
**Topics:** docs-infra

**Context:** The `docs/tech/` folder had drifted to contain both vendor evaluations (Stripe, Cloudflare) and internal architecture docs (state management, URL routing). Deciding where new docs should go required guessing.

**Learning:** Three reorganizations in this project followed the same pattern: incremental drift → cognitive threshold → systematic reorganization. Terminology block (~960 files, ~5000 replacements), page specs deletion (Sessions 307+311), and now docs folders (~65 files, ~200 replacements). The cost of reorganizing is proportional to the drift duration. The mapping document (`DOCS-REORG-MAP.md`) is the critical artifact for preserving session log interpretability.

## 5. TypeScript Type Predicates Provide Type Safety for Cache Validation
**Topics:** testing

**Context:** The `isValidCachedData()` function uses `data is MeFullResponse` return type.

**Learning:** A TypeScript type predicate function lets you do runtime validation that narrows the type for the compiler. After `isValidCachedData(parsed.data)` returns true, the compiler knows `parsed.data` is `MeFullResponse` — no casts needed downstream. This is the right pattern for validating data from external sources (localStorage, API responses) where the shape isn't guaranteed at compile time.
