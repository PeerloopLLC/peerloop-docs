# Session Decisions - 2026-03-01 (Session 319)

## 1. Creator Access Gate Policy: Permission vs State
**Type:** Strategic
**Topics:** auth, architecture

**Trigger:** New creator approved by admin was blocked from `/creating` because the access check used course count (state) instead of `can_create_courses` flag (permission).

**Options Considered:**
1. Standardize all gates to use `can_create_courses` only (Pattern A)
2. Standardize all gates to use course count only (Pattern B)
3. Use different patterns for different actions ← Chosen

**Decision:** Create actions (new courses, new communities) gate on permission only (`can_create_courses = 1`). View/manage actions (dashboard, course list, earnings, communities, analytics) gate on permission OR state (`can_create_courses = 1 OR has_courses`). This ensures new creators can access the dashboard and revoked creators retain access to existing content.

**Rationale:** Permission and state answer different questions. "May this user create?" is a permission. "Does this user have content to manage?" is state. View/manage actions need both paths because: (a) new creators have permission but no courses yet, and (b) revoked creators have courses but no permission. Both must access the dashboard.

**Consequences:** Updated 4 API endpoints to Pattern C. Established the policy in `POLICIES.md`. 8 analytics endpoints remain Pattern B, gated client-side by `useCreatorGate` hook (pending implementation).

## 2. Creator Revocation: Existing Content Stays Accessible
**Type:** Strategic
**Topics:** auth, architecture

**Trigger:** While fixing the access gates, discovered that the `OR` logic would allow revoked creators (permission=0, courses>0) to still access the dashboard. Needed a policy decision.

**Options Considered:**
1. Lock revoked creators out entirely
2. Allow read-only access to existing content
3. Retain full access to existing content, only block creation ← Chosen

**Decision:** When admin revokes `can_create_courses`, only creation of new courses and communities is blocked. Everything else (dashboard, editing existing courses, managing communities, viewing analytics, managing earnings) stays accessible.

**Rationale:** Revoking creation permission shouldn't orphan existing content. The creator may need to wind down, manage existing students, or respond to community activity. Existing course/community endpoints use ownership checks (Pattern D: `creator_id = userId`), which are unaffected by the permission flag.

## 3. POLICIES.md for Platform Behavior Rules
**Type:** Strategic
**Topics:** docs-infra

**Trigger:** Creator access control policies didn't fit cleanly into DECISIONS.md (architectural/implementation) or PLAYBOOK.md (docs-repo conventions). Needed a home for prescriptive platform behavior rules.

**Options Considered:**
1. Add to DECISIONS.md under a new section
2. Create a separate POLICIES.md ← Chosen

**Decision:** Created `POLICIES.md` for platform behavior policies — access control, business logic, user capabilities. It is prescriptive: if code contradicts a policy, the code is the bug.

**Rationale:** DECISIONS.md records *how* we build (architecture). POLICIES.md defines *what* the platform does (behavior). The distinction matters because policies guide implementation across multiple sessions and are the authority when code is inconsistent — exactly the situation that revealed this session's bugs.

**Consequences:** Created `POLICIES.md`, added references in `CLAUDE.md` (project structure + research reference sections).

## 4. Analytics Empty State: Client-Side Course Count Gate
**Type:** Implementation
**Topics:** astro, architecture

**Trigger:** New creator visiting `/creating/analytics` fired 7 parallel API calls that all returned 403, showing a red error box with "Retry" button.

**Options Considered:**
1. Change 8 analytics API endpoints to accept new creators (return empty data) ← Not chosen
2. Gate in the component: check course count first, show empty state if zero ← Chosen

**Decision:** `CreatorAnalytics.tsx` checks course count before firing analytics requests. If zero courses, renders a friendly empty state with "Create Your First Course" link. Follow-up: replace the course count fetch with `useCreatorGate` hook using `CurrentUser` global.

**Rationale:** Avoids 7 wasted API calls. Single place for the empty state message. Doesn't require auditing all 7 analytics sub-components for empty-array handling. The analytics APIs remain gated (Pattern B) as a defense-in-depth measure.

## 5. Admin Cannot Override Creator Permission for Course Creation
**Type:** Implementation
**Topics:** auth

**Trigger:** Reviewed whether `POST /api/admin/courses` should bypass the target user's `can_create_courses` flag.

**Decision:** Admin must first grant `can_create_courses` permission before creating a course for a user. The two-step flow is an intentional safeguard against accidentally creating content for revoked creators.

**Rationale:** If admin could bypass the flag, it would put the system in a contradictory state: user has a course but can't access the dashboard to manage it. The explicit two-step process forces the admin to consciously re-enable a revoked creator.
