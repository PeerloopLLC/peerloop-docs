## 1. Feature Tracking Rule: All Features Must Be in PLAN.md
**Type:** Strategic
**Topics:** docs-infra, workflow

**Trigger:** Scanned all 31 tech docs and found 4 blocks worth of untracked features. Tech docs had "Future Features" and "Planned" sections with no corresponding PLAN.md entries.

**Options Considered:**
1. Leave features in tech docs and periodically sweep ← Not sufficient alone
2. Add a CLAUDE.md rule requiring all features to land in PLAN.md with cross-references in tech docs ← Chosen
3. Move all feature discussion out of tech docs into PLAN.md ← Over-centralizes; tech docs should retain the "how" and "why"

**Decision:** Any time a feature is mentioned — in a tech doc, session discussion, RFC, or code comment — it must be added to PLAN.md. Where it goes is situational (active block, deferred, sub-task). If the feature originated from a tech doc, add a cross-reference noting the PLAN block. Tech docs describe how and why; PLAN.md is the single source of truth for what needs to be done.

**Rationale:** PLAN.md is the execution authority. Without this rule, features silently accumulate in tech docs where they're invisible to planning. The cross-reference closes the loop in both directions.

**Consequences:** Added rule to CLAUDE.md under "Feature Tracking Rule." Extracted 4 new deferred blocks (POSTHOG, MOCK-DATA-MIGRATION, RATINGS-EXT, CURRENTUSER-REFRESH) and 1 missing item (SessionHistory in MSG-ACCESS.PHASE2).

## 2. Custom Calendar Over react-big-calendar
**Type:** Strategic
**Topics:** astro, workflow

**Trigger:** Audited calendar views across the platform. Found 3 custom month grids, no week/day views, and react-big-calendar installed but never used. User wants year/month/week/day views with role-specific filtering, clickable items, and an admin oversight calendar.

**Options Considered:**
1. Adopt react-big-calendar for week/day views, keep custom month grid ← Rejected
2. Build a fully custom `PeerloopCalendar` component system ← Chosen
3. Use a different calendar library (FullCalendar, etc.) ← Not evaluated; custom chosen proactively

**Decision:** Build a custom multi-view calendar component (`PeerloopCalendar`) with year, month, week, and day views. The calendar accepts typed `CalendarItem[]` arrays and renders them — it doesn't know what the items are. Role-specific pages fetch and combine data layers based on active filters.

**Rationale:** The platform needs cell-level control that libraries don't provide — availability multi-select, heat-map year views, togglable data layers with role-specific filtering, and consistent Tailwind styling. A library would fight every customization. Building custom means the calendar grows with the platform.

**Consequences:** Added CALENDAR as active block in PLAN.md with 5 phases (CORE, STUDENT, ST, ADMIN, MIGRATE). react-big-calendar will be removed from package.json after migration.

## 3. Branching Workflow Test Architecture
**Type:** Strategic
**Topics:** testing

**Trigger:** Discussion about testing multi-step platform flows (booking, completion, payment, messaging) where different user actions at decision points produce different downstream state.

**Options Considered:**
1. Independent tests each rebuilding the world from scratch ← Expensive, slow
2. Shared setup + branching `describe` blocks at decision points ← Chosen
3. Parameterized test matrices ← Too rigid for workflows with different assertion shapes

**Decision:** Workflow tests use a shared expensive setup (users, courses, enrollments, sessions) and branch into separate `describe` blocks at decision points. Shared helper functions like `setupCompletedSession(db)` get to the decision point cheaply. Only branches where the decision changes downstream state are worth testing.

**Rationale:** The expensive part is building the world; the branches are cheap once at the decision point. This gives N test variants for roughly the cost of 1.5 full tests. Integration tests cover branching logic (ms per branch); E2E tests cover only the 2-3 most critical happy paths.

**Consequences:** Added WORKFLOW-TESTS as deferred block in PLAN.md with 4 workflow groups (BOOKING, COMPLETION, PAYMENT, MESSAGING) prioritized by branch count and downstream impact.

**Type:** Implementation
**Topics:** testing, stripe, video

**Trigger:** Discussion about reliably testing Stripe and BBB webhooks in the dev environment. Currently requires 3+ terminal windows and tribal knowledge.

**Options Considered:**
1. Defer webhook dev setup until needed ← Blocks WORKFLOW-TESTS.PAYMENT
2. Make it an active block with scripted automation ← Chosen

**Decision:** DEV-WEBHOOKS is an active PLAN.md block, not deferred. It includes an orchestrator script (`dev-webhooks.sh`), a trigger helper (`trigger-webhook.sh`), npm scripts, and Stripe data alignment via fixture overrides.

**Rationale:** This is infrastructure that unblocks WORKFLOW-TESTS.PAYMENT and is also useful for manual QA. One command should start the full webhook dev environment.

**Consequences:** Added as active block in PLAN.md with SCRIPTS, DATA-ALIGNMENT, and DOCS sub-sections.
