# Conv Decisions - 2026-03-19

## 1. Live Stream API Seeding for Smart Feed E2E Tests
**Type:** Implementation
**Topics:** stream, testing

**Trigger:** Smart feed E2E tests need real Stream.io activities that match D1 `feed_activities` rows. Without matching data in both layers, the smart feed pipeline can't enrich candidates.

**Options Considered:**
1. D1-only + mock Stream responses — fast, no external dependency, but doesn't test real enrichment
2. Live Stream API seeding ← Chosen — real data in both layers, tests the full pipeline
3. Hybrid (SQL seed D1, one-time Stream setup, capture IDs back) — complex setup, IDs change on app reset

**Decision:** Use a standalone Node.js script (`scripts/seed-feeds.mjs`) that creates activities via Stream REST API and dual-writes to D1 with the returned `stream_activity_id`.

**Rationale:** The smart feed's value is its ranking algorithm, which depends on real engagement signals from Stream (`reaction_counts`). Mocking would test UI rendering but not scoring. Stream credentials already exist in `.dev.vars`, the `StreamClient` wrapper supports `addActivity()`, and the script runs in ~5 seconds.

**Consequences:** New script `scripts/seed-feeds.mjs`, npm scripts `db:seed:feeds:local`/`staging`, pipeline level `db:setup:local:feeds`. Feed seed is the only setup step that makes external API calls.

## 2. Serial E2E Tests with Single User Due to Dev Server Concurrency
**Type:** Implementation
**Topics:** testing, astro

**Trigger:** Playwright tests with 5 parallel workers caused the Astro dev server to hang at 99.7% CPU. David Rodriguez's smart feed request consistently hung even in isolation via Playwright (but worked via curl).

**Options Considered:**
1. Mock Stream API in E2E tests — defeats the purpose of real-data testing
2. Serial tests with single user (Sarah Miller) ← Chosen — reliable, tests full pipeline
3. Increase dev server timeout/workers — doesn't fix the root cause

**Decision:** Use `test.describe.configure({ mode: 'serial' })` and Sarah Miller for all smart feed E2E tests. Content-based waiting (`expect(text).toBeVisible()`) instead of API response interception.

**Rationale:** Sarah's feed loads reliably in Playwright (1-2s). The Astro dev server's single-threaded Node.js event loop can't handle concurrent smart feed requests (D1 + Stream). Serial execution with content-based waits is the pragmatic solution for local E2E testing.

**Consequences:** All 6 smart feed E2E tests pass in ~10s. David's feed works fine via curl but has a Playwright-specific timing issue with the dev server — not a code bug, just a dev-mode limitation.

## 3. Feed Seed as Top-Level Setup Pipeline Step
**Type:** Implementation
**Topics:** testing, workflow

**Trigger:** Feed seed data requires dev seed users/communities AND makes external API calls to Stream.io. It needs a clear position in the progressive setup chain.

**Options Considered:**
1. Bundle into `db:setup:local:dev` — would force Stream API calls on all dev setups
2. Keep as standalone `db:seed:feeds:local` only — users must remember to run it separately
3. Add as new pipeline level `db:setup:local:feeds` ← Chosen — follows existing progression

**Decision:** Add `db:setup:local:feeds` as level 5 in the setup chain: `local` → `dev` → `stripe` → `booking` → `feeds`. It includes everything from booking plus feed seed.

**Rationale:** Follows the existing progressive pattern where each level adds opt-in data. Feed seed is the only level that makes external API calls (to Stream.io), so keeping it last and opt-in avoids surprises. The individual `db:seed:feeds:local` script remains available for re-seeding without a full reset.

**Consequences:** Updated `package.json` with 4 new scripts (`db:setup:local:feeds`, `db:setup:staging:feeds`, `db:seed:feeds:local`, `db:seed:feeds:staging`). Docs updated in CLI-QUICKREF, CLI-REFERENCE, CLI-TESTING.
