# Conv Learnings - 2026-03-29

## 1. Plan SQL queries must be verified against actual schema
**Topics:** schema, testing, admin-intel

**Context:** The admin-intel plan specified SQL queries referencing columns that don't exist (e.g., `moderator_invites.community_id`, `communities.created_by_user_id`, `creator_applications.pitch`).

**Learning:** Plans written from design discussion are not schema-verified. Always grep the migration file for actual column names before writing endpoint code or test INSERT statements. Four mismatches were caught during test execution — the production endpoint code would have failed at runtime.

**Pattern:** Before writing API endpoint SQL: `grep "CREATE TABLE.*tablename" migrations/0001_schema.sql` and read the column list.

---

## 2. ExtraTabConfig early return blocks admin tab for non-role users
**Topics:** admin-intel, react, course-tabs

**Context:** ExploreCourseTabs and ExploreCommunityTabs had `if (roleTabs.length === 0) return [];` in the extraTabs useMemo. An admin browsing a course they have no role in would get zero role tabs, and the early return would prevent the admin tab from appearing.

**Learning:** When adding cross-cutting concerns (like admin tabs) to role-based tab systems, the early return pattern that was safe for role-only tabs becomes a blocker. Must restructure to build array unconditionally, append admin tab, then return combined result.

---

## 3. Three profile pages have different user ID property names
**Topics:** admin-intel, profile, architecture

**Context:** Mounting AdminMemberSummary on profile pages required finding the user ID in three different data shapes: `profile.id` (PublicProfile), `creator.id` (CreatorProfile from User interface), `teacher.user_id` (TeacherProfile from TeacherProfileData).

**Learning:** The profile system has no unified interface — each profile type has its own data shape from different SSR loaders. Any component that needs to mount across all profile types must account for these differences.

---

## 4. SSR + API = two-layer admin protection
**Topics:** security, admin-intel, astro

**Context:** The /discover/members page needs admin-only access.

**Learning:** Two complementary protection layers: SSR frontmatter (`getSession()` → redirect) prevents page render for non-admins; API endpoint (`requireRole`) prevents data access even if SSR is bypassed. The DiscoverSlidePanel uses client-side check (`useCurrentUser()?.isAdmin`) since it's a hydrated React component — entry simply doesn't appear for non-admins, and clicking the URL hits the SSR redirect.
