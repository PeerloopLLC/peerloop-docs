# Session Decisions - 2026-03-04

## 1. Store user_id (Not student_teachers.id) in Enrollment FK
**Type:** Implementation
**Topics:** d1, stripe

**Trigger:** `enrollments.student_teacher_id` FK references `users(id)` but the checkout pipeline was inserting `student_teachers.id` — causing FK violations.

**Options Considered:**
1. Change the FK to reference `student_teachers(id)` instead of `users(id)`, and update all JOIN consumers
2. Keep the FK as `users(id)` and fix the pipeline to pass `user_id` through Stripe metadata ← Chosen

**Decision:** Keep `enrollments.student_teacher_id` referencing `users(id)`. Fix the checkout pipeline to resolve `student_teachers.id` → `user_id` at the earliest point (`create-session.ts`) and pass both IDs through Stripe metadata.

**Rationale:** The majority of the codebase (admin endpoints, analytics, reviews, reassign-st) already correctly treats `enrollments.student_teacher_id` as a `users.id`. Changing the FK to `student_teachers(id)` would require updating ~10+ consumers. Fixing the pipeline at the source (checkout creation) is a 3-file change.

**Consequences:** Stripe metadata now carries both `student_teacher_id` (st-xxx, for webhook JOINs) and `student_teacher_user_id` (usr-xxx, for enrollment FK). Backward-compatible: in-flight sessions with empty `student_teacher_user_id` fall back to null.

## 2. Self-Healing on Both Success Page and /courses Dashboard
**Type:** Implementation
**Topics:** stripe, astro

**Trigger:** User requested self-healing fallback when Stripe webhooks are missed. Two surfaces considered.

**Options Considered:**
1. Success page only — it has the Stripe `session_id` from the redirect URL
2. Success page + /courses dashboard — adds localStorage bridge for safety net ← Chosen

**Decision:** Implement self-healing on both surfaces. Success page calls `createEnrollmentFromCheckout` directly in SSR. /courses page uses a localStorage bridge: `EnrollButton` stores the Stripe session_id before redirect, `MyCourses` checks it on mount and calls `POST /api/stripe/verify-checkout`.

**Rationale:** The success page covers 99% of cases, but if a user closes the tab during the Stripe redirect or navigates directly to /courses, the localStorage fallback catches the gap. The verify-checkout endpoint is reusable by both surfaces and any future self-healing needs.

**Consequences:** New files: `src/lib/enrollment.ts` (shared logic), `src/pages/api/stripe/verify-checkout.ts` (endpoint). Webhook handler now delegates to shared function. localStorage key `pl_pending_sessions` stores pending session IDs with 1-hour expiry.

## 3. Extract Enrollment Creation into Shared Module
**Type:** Implementation
**Topics:** stripe

**Trigger:** Enrollment creation logic needed by both the webhook and the self-healing endpoint.

**Options Considered:**
1. Duplicate the logic in both the webhook and verify-checkout endpoint
2. Extract into a shared `src/lib/enrollment.ts` module ← Chosen

**Decision:** Extract the full `handleCheckoutCompleted` body into `createEnrollmentFromCheckout()` in `src/lib/enrollment.ts`. Webhook becomes a thin wrapper. Verify-checkout and success page SSR call the same function.

**Rationale:** Single source of truth for enrollment creation. Already idempotent (existing enrollment check). Any future enrollment creation path (e.g., Cron Trigger polling from PLAN.md) can reuse the same function.

**Consequences:** `src/pages/api/webhooks/stripe.ts` shrank significantly. Removed unused imports (`getEnv`, `sendEmail`, `getUserEmail`, `EnrollmentConfirmationEmail`) from webhook — they're now in `enrollment.ts`.
