# Conv Decisions - 2026-03-17

## 1. Create Clock Abstraction for Time-Sensitive Endpoints
**Type:** Implementation
**Topics:** testing

**Trigger:** Test failures caused by `new Date()` in handlers being untestable and timing-fragile.

**Options Considered:**
1. Keep using `new Date()` with large time margins in tests
2. Create `getNow()` abstraction in `src/lib/clock.ts` ← Chosen
3. Use Vitest's `vi.useFakeTimers()` globally

**Decision:** Created `src/lib/clock.ts` with `getNow()` function. Updated session cancel, session join, and availability endpoints to use it.

**Rationale:** Option 1 is fragile (margins can still be crossed). Option 3 is heavy-handed and can break async code. A simple function wrapper is minimal overhead and gives tests a clean mock point.

**Consequences:** Three endpoints updated. Tests can now freeze time for exact boundary testing. No runtime cost — `getNow()` just returns `new Date()` in production.

## 2. Create UTC Test Date Helper Library
**Type:** Implementation
**Topics:** testing

**Trigger:** Multiple test files constructing dates with timezone-unsafe patterns, causing CI failures.

**Options Considered:**
1. Fix each test file individually with inline `Date.UTC()` calls
2. Create shared helper library in `tests/helpers/dates.ts` ← Chosen

**Decision:** Created `tests/helpers/dates.ts` with `utcDate()`, `futureUTC()`, `pastUTC()`, `nextDayOfWeekUTC()`, `toDateStringUTC()`. Exported from `@test-helpers`.

**Rationale:** Centralized helpers enforce the correct pattern and reduce cognitive load. Helpers also serve as documentation of the right way to construct test dates.

**Consequences:** Available to all test files via `@test-helpers` import. Documented in BEST-PRACTICES.md §8.

## 3. Add CI Timezone Lint Script
**Type:** Implementation
**Topics:** testing

**Trigger:** Timezone-unsafe patterns are invisible until CI fails. Need proactive detection.

**Options Considered:**
1. ESLint rule (custom plugin)
2. Shell script with grep patterns ← Chosen
3. Rely on code review

**Decision:** Created `scripts/lint-timezone.sh` with `npm run lint:tz`. Flags `setHours()`, `new Date(y,m,d)`, `getDate()`/`getDay()` in test files. Hard-fails on `setHours` and multi-arg Date; warns on `getDate`/`getDay`.

**Rationale:** Shell script is simple, zero-dependency, and catches the patterns that caused actual CI failures. ESLint plugin would be more thorough but heavier than needed.

**Consequences:** New npm script `lint:tz`. Can be added to CI pipeline when ready.

## 4. Use Absolute Paths in Skill Script References
**Type:** Implementation
**Topics:** cc-workflow, dual-repo

**Trigger:** `/r-end` failed because `!` backtick pre-computed context used relative paths that broke when cwd drifted to the code repo.

**Options Considered:**
1. Always `cd` back to docs repo before running skills
2. Use absolute paths in all skill script references ← Chosen
3. Make scripts detect their own location

**Decision:** Changed all 18 `!`.claude/scripts/...`` references across 10 skills from relative to absolute paths (`/Users/jamesfraser/projects/peerloop-docs/.claude/scripts/...`).

**Rationale:** Absolute paths are immune to cwd drift. Option 1 requires discipline (easy to forget). Option 3 adds complexity to every script.

**Consequences:** All r-* and w-post-fix skills now work regardless of cwd. Paths are machine-specific but both dev machines use the same home directory structure.
