# Conv Decisions - 2026-03-28

## 1. Feeds Empty State: Onboarding CTA over Generic Browse
**Type:** Implementation
**Topics:** ux, feeds, onboarding

**Trigger:** Phase 7 — feeds empty state feature

**Options Considered:**
1. Keep generic "browse courses/communities" CTA for all empty states
2. Detect incomplete onboarding and show targeted "Select Your Interests" CTA <- Chosen
3. Always show both CTAs regardless of onboarding status

**Decision:** Show onboarding CTA for authenticated users who haven't completed tag selection; show generic browse CTAs for visitors and completed-onboarding users.

**Rationale:** The smart-feed discovery pipeline depends on user tags for personalization. Nudging tagless users to complete onboarding directly feeds the recommendation engine.

**Consequences:** `DiscoverFeedsGrid` now imports `useCurrentUser` and checks `hasCompletedOnboarding()`.

## 2. Settings Tag Editor as Dedicated Page
**Type:** Implementation
**Topics:** ux, settings, onboarding

**Trigger:** Phase 7 — settings editor for post-onboarding tag editing

**Options Considered:**
1. Link to full onboarding wizard from settings hub (existing behavior)
2. Create dedicated `/settings/interests` page reusing TopicPicker <- Chosen
3. Add tag editing inline to profile settings page

**Decision:** Create `/settings/interests` as a standalone page with `InterestsSettings` component that loads/saves via the existing onboarding-profile API.

**Rationale:** The full onboarding wizard includes steps (role selection, experience survey) that don't belong in a quick "update my interests" flow. A dedicated page focuses on the one thing the user wants to change.

**Consequences:** New files: `InterestsSettings.tsx`, `interests.astro`. Settings hub link changed from `/onboarding` to `/settings/interests`.
