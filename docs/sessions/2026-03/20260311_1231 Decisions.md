# Session Decisions - 2026-03-11

## 1. Add action_label Column to Notifications Schema
**Type:** Implementation
**Topics:** d1, notifications

**Trigger:** Notification cards had a generic "View →" button that gave no indication of destination. Different notification types link to sessions, certificates, profiles, admin pages, etc.

**Options Considered:**
1. Derive label from notification `type` field in the UI component (mapping table)
2. Add `action_label TEXT` column to notifications table ← Chosen
3. Embed label in `action_url` as a query param

**Decision:** Add nullable `action_label` column to the `notifications` table. Each notification helper sets a contextual label at creation time. Component falls back to "View" for notifications without a label.

**Rationale:** The notification creator has the best context for what the label should say. Client-side derivation would require maintaining a parallel mapping and couldn't handle the `notifySystem` generic helper which takes arbitrary URLs.

**Consequences:** Schema change to `0001_schema.sql`, all 12 notification helpers updated with labels, seed data updated, API endpoint returns `action_label`, component renders it.

## 2. Separate Card Click (Mark Read) from Navigation (Explicit Button)
**Type:** Implementation
**Topics:** notifications, astro

**Trigger:** User reported notification UX issues: 1) not obvious cards are clickable, 2) no indication of destination, 3) click-to-navigate collides with click-to-mark-read.

**Options Considered:**
1. Keep stretch-link pattern, add hover styling ← Rejected (still no mobile affordance)
2. Card click = mark read, explicit labeled button = navigate ← Chosen
3. Make only the title a visible link

**Decision:** Card click marks notification as read. A visible "Go to Session →" / "Browse Courses →" button (using `action_label`) handles navigation. `e.stopPropagation()` on the button prevents double-marking.

**Rationale:** Cleanly separates two distinct user intents. The button is discoverable on both desktop and mobile, and the label tells the user exactly where they'll go.

**Consequences:** Removed stretch-link `<a>` overlay, added explicit `<a>` button with action_label text, card `onClick` handles mark-as-read.

## 3. Send Welcome Notifications on Registration
**Type:** Implementation
**Topics:** auth, notifications

**Trigger:** Seed data had "Welcome to PeerLoop!" and "Discover Courses" notifications for Alex, but no code path created these for real users on registration.

**Options Considered:**
1. Keep as seed-only test data ← Rejected (real users get no onboarding notifications)
2. Add to registration endpoint as fire-and-forget ← Chosen

**Decision:** POST `/api/auth/register` now creates two welcome notifications via `notifySystem` after user creation: "Welcome to PeerLoop!" → `/how-it-works` and "Discover Courses" → `/discover/courses`. Both are non-blocking (`Promise.allSettled`).

**Rationale:** Every new user should get a guided first experience. The notification bell becomes immediately useful rather than empty.

**Consequences:** `register.ts` imports `notifySystem`, creates 2 notifications. Test added to verify. Seed data updated to match the same URLs and labels.

## 4. Welcome Notification Points to /how-it-works, Discover Points to /discover/courses
**Type:** Implementation
**Topics:** astro, notifications

**Trigger:** Original seed data pointed "Welcome" to `/courses` (empty for new user) and "Discover" to `/courses/ai-tools-overview` (wrong route format, 404).

**Options Considered:**
1. `/courses` for welcome ← Rejected (shows empty "my courses" page)
2. `/how-it-works` for welcome, `/discover/courses` for discover ← Chosen
3. `/` (homepage) for welcome

**Decision:** Welcome → `/how-it-works` ("See How It Works →"), Discover → `/discover/courses` ("Browse Courses →").

**Rationale:** `/how-it-works` explains the learn-teach-earn flywheel — exactly what a new user needs. `/discover/courses` is the public catalog, not the user's (empty) enrolled courses list.

**Consequences:** Updated in `register.ts`, seed data, and test.
