# Session Learnings - 2026-03-12

## 1. Time-Sensitive Tests Must Use Fake Timers
**Topics:** testing

**Context:** `TeacherDetailContent.test.tsx` had tests that calculated "15 days ago" and "3 months ago" using `new Date()` at runtime. As real time passed since the tests were written, the date math drifted and assertions like `/15 days/` stopped matching.

**Learning:** Any test that asserts on relative time calculations (days ago, months ago, years ago) must freeze time with `vi.useFakeTimers()` and `vi.setSystemTime()`. Use fixed ISO dates rather than relative `setDate(getDate() - N)` to make assertions deterministic and immune to calendar drift.

**Pattern:**
```ts
const NOW = new Date('2026-03-12T12:00:00Z');
beforeEach(() => { vi.useFakeTimers(); vi.setSystemTime(NOW); });
afterEach(() => { vi.useRealTimers(); });

// Then use fixed dates in fixtures:
certified_date: '2026-02-25T12:00:00Z', // exactly 15 days before NOW
```

## 2. Mock Modules Must Export Every Function the Code Under Test Calls
**Topics:** testing

**Context:** `session-lifecycle.test.ts` mocked `@/lib/notifications` with `notifySessionBooked`, `notifySessionConfirmed`, and `notifySessionLateCancelled`, but the DELETE handler for sessions had been updated to call `notifySessionCancelled` (the replacement function). The mock didn't include it, causing a "No export is defined on the mock" error.

**Learning:** When a source module adds or renames exports, all mocks of that module must be updated in lockstep. Vitest's mock system throws at runtime if code calls an export that isn't in the mock object. After renaming/adding functions, grep for `vi.mock('...')` of that module across the test suite.

## 3. Duplicate Type Definitions Cause "Two Types with Same Name" Errors
**Topics:** testing, astro

**Context:** `STData` was defined identically in 4 files (`TeacherPerformanceTable.tsx`, `CreatorAnalytics.tsx`, `AdminAnalytics.tsx`, `admin/TeacherSection.tsx`). When `CreatorAnalytics` passed its `STData[]` to `TeacherPerformanceTable`, TypeScript rejected it because they were structurally identical but nominally different types.

**Learning:** Export shared interfaces from the component that owns them, and import everywhere else. Even if two interfaces are structurally identical, TypeScript treats separately-declared interfaces as incompatible when comparing branded/nominal types in certain contexts (like component props).

## 4. Astro Check Reports Hints at Different Severity Levels
**Topics:** astro, testing

**Context:** Running `npx astro check` showed "0 errors, 0 warnings, 16 hints" but didn't display the hint details at default verbosity. Had to use `--minimumSeverity hint` to see the actual issues.

**Learning:** Use `npx astro check --minimumSeverity hint` to see all diagnostics including unused imports in test files. The default output only shows the count. Hints are mostly `ts(6133)` unused variable warnings in test files — low severity but worth cleaning up to prevent accumulation.

## 5. Test Import Starter Kit Creates Persistent Noise
**Topics:** testing, cc-workflow

**Context:** When writing test files, I was importing a full starter kit (`describe`, `it`, `expect`, `vi`, `beforeEach`, `afterEach`, `beforeAll`, `afterAll`, `waitFor`, `within`, etc.) then only using a subset. This left unused imports that surfaced as Astro hints.

**Learning:** Draft with the starter kit for speed, but do a cleanup pass immediately after finishing the test file. This takes seconds when the file is fresh but creates persistent lint noise if left. Added to BEST-PRACTICES.md and CLI-TESTING.md as project convention.
