# Conv Decisions - 2026-03-28

## 1. Community "All" Tab Shows Full Catalog (Not Just Undiscovered)
**Type:** Implementation
**Topics:** ux, communities, explore-pattern

**Trigger:** Designing community listing page tabs — should "All" show ALL public communities or only those not joined?

**Options Considered:**
1. Only undiscovered communities (current `/discover/communities` behavior)
2. ALL public communities with role badges on joined ones ← Chosen

**Decision:** "All" tab shows the full catalog, consistent with how `/discover/courses` works.

**Rationale:** Consistency with the courses pattern. Users see role badges on communities they've joined alongside ones they haven't. The "Member" tab provides the filtered "my communities" view.

**Consequences:** SSR fetch changed from `filter=discover` to `filter=all`. Each community card annotated with role badges client-side.

## 2. UserFeedLink Extended With Primitives, Not RoleBadgeConfig
**Type:** Implementation
**Topics:** current-user, data-model, separation-of-concerns

**Trigger:** Needed role data on feed links — should CurrentUser return UI-ready RoleBadgeConfig or raw role strings?

**Options Considered:**
1. Add `RoleBadgeConfig[]` directly to UserFeedLink (circular dependency risk: CurrentUser → explore types)
2. Add `parentId` only, let explore layer derive roles (loses info from getFeeds() dedup)
3. Add `parentId: string` + `roles: string[]` as primitives ← Chosen

**Decision:** UserFeedLink carries `parentId` and `roles` as simple strings. The explore layer converts to RoleBadgeConfig for rendering via `feed-role-utils.ts`.

**Rationale:** Avoids circular dependency between core `current-user.ts` and UI `explore/types.ts`. All consumers get role data without importing UI types. Backward-compatible — existing FeedsHub/MyFeeds ignore the new fields.

**Consequences:** Two-step conversion: getFeeds() populates string roles, feed-role-utils converts to badges. Clean separation of data and presentation.

## 3. Keep /feeds and /discover/feeds Coexisting (Client Decision Pending)
**Type:** Strategic
**Topics:** routes, client-workflow, migration

**Trigger:** User corrected premature redirect of /feeds → /discover/feeds

**Options Considered:**
1. Redirect /feeds → /discover/feeds immediately
2. Keep both pages, let client compare and decide ← Chosen
3. Replace /feeds content with ExploreFeeds

**Decision:** Both pages coexist. `/feeds` stays as original FeedsHub (navbar "My Feeds" links here). `/discover/feeds` is the new role-aware version (accessible from Discover slideout).

**Rationale:** User explicitly wanted client to have the option. No links to the original were changed.

**Consequences:** Two feed directory pages exist simultaneously. Client will decide which becomes canonical. Nav entry added to DiscoverSlidePanel for `/discover/feeds`.

## 4. CERT-APPROVAL Block Expanded to 4 Phases
**Type:** Strategic
**Topics:** certificates, planning, lifecycle

**Trigger:** CompletedTabContent dead link to non-existent `/course/[slug]/certificate` page

**Options Considered:**
1. Hide/disable the link until the page exists
2. Create a stub page
3. Expand CERT-APPROVAL to document the full lifecycle gap ← Chosen

**Decision:** Expanded CERT-APPROVAL from 3 sparse items to 4 detailed phases with a complete "What Exists" / "What's Missing" inventory.

**Rationale:** The certificate feature spans a multi-step lifecycle (teacher recommend → creator approve → PDF generate → student view). A stub page would sit in front of 3 missing upstream steps. Better to plan the full chain.

**Consequences:** PLAN.md CERT-APPROVAL block now has Phase 1 (student page + dead link fix), Phase 2 (creator approval), Phase 3 (PDF + R2), Phase 4 (public page). Deferred block priority unchanged.
