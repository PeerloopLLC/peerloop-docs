# Session Learnings - 2026-03-05

## 1. Missing GROUP BY on Join Tables Produces Silent Duplicates
**Topics:** d1, astro

**Context:** `/discover/teachers` showed Guy Rymberg 4 times with different student counts (67, 52, 38, 15). The `student_teachers` table has one row per (user, course) pair.

**Learning:** When querying a join/relationship table for a directory listing, always GROUP BY the entity you're displaying (the user), not the relationship (the certification). The telltale sign is different aggregate values across duplicated cards — each row was showing per-course `students_taught` instead of the SUM. The fix was adding `GROUP BY u.id` and `SUM(st.students_taught)`.

**Pattern:**
```sql
-- WRONG: one card per certification
SELECT st.id as st_id, u.name, st.students_taught FROM student_teachers st JOIN users u ...

-- RIGHT: one card per teacher, aggregated stats
SELECT u.id as user_id, u.name, SUM(st.students_taught) as students_taught FROM student_teachers st JOIN users u ... GROUP BY u.id
```

## 2. Ambiguous Table Names Mask Semantic Errors
**Topics:** d1, naming

**Context:** The column `st.id` was used as the React key and card identifier, but it's the per-course certification record ID, not the teacher's user ID. The name "student_teachers" made it sound like the table contains teachers (people), when it actually contains certifications (per-course records).

**Learning:** When a table name sounds like it contains entities (people, things) but actually contains relationships (certifications, memberships, enrollments), the ambiguity propagates into every query. Developers assume `st.id` means "the student-teacher's ID" when it really means "this particular certification record's ID." Renaming the table to describe what rows actually are (e.g., `teacher_certifications`) makes wrong usage immediately obvious.

## 3. Terminology Debt Compounds Like Technical Debt
**Topics:** naming, workflow

**Context:** The user identified that "Student-Teacher" was a mouthful, "user/member/visitor" were used interchangeably, and these inconsistencies had propagated through schema, code, components, API routes, UI text, and documentation.

**Learning:** Inconsistent terminology isn't just a documentation problem — it's a code quality problem. When the same concept has multiple names (S-T, ST, Student-Teacher, student_teacher, teacher), every new developer (or AI) must learn all the synonyms. Worse, the ambiguity creates real bugs (like the `st.id` issue). Establishing a glossary early and enforcing it across all layers prevents the compound interest of naming debt. Pre-production is the cheapest time to fix this.
