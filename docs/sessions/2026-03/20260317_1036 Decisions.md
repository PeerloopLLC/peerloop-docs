# Session Decisions - 2026-03-17

## 1. Adopt "Conv" (Conversation) to Replace "Session" for Work Unit Tracking
**Type:** Strategic
**Topics:** cc-workflow, docs-infra

**Trigger:** "Session" has meaning in Claude Code (a CC process invocation) and was overloaded as the project's numbered work unit. User wanted separate terms.

**Options Considered:**
1. Keep "Session" for everything — live with the ambiguity
2. Rename to "Conv" (Conversation) for project work units ← Chosen
3. Use a completely new term (e.g., "Sprint", "Block")

**Decision:** Adopt "Conv" as the numbered work unit. Conv 001 starts fresh (not continuing from Session 393). Both terms coexist during transition; Session numbering ends at 393.

**Rationale:** "Conversation" maps naturally to what the unit actually is — a Claude Code conversation. It avoids collision with CC's internal "session" concept. Starting fresh at 001 makes the break clean.

**Consequences:** CONV-COUNTER file created, SESSION-INDEX.md marked as ended, session-tracker.sh hook disabled, 11 r-* skills use Conv terminology throughout.

## 2. Dual-Repo as Paired Unit for Conv Lifecycle
**Type:** Strategic
**Topics:** dual-repo, cc-workflow

**Trigger:** Designing how /r-start, /r-end, /r-commit interact with two repos (peerloop-docs + Peerloop).

**Options Considered:**
1. Conv lifecycle touches docs repo only — code commits separate ← Rejected
2. Both repos treated as paired unit — same Conv number, always commit both ← Chosen
3. Docs always + code optionally (ask each time) ← Rejected

**Decision:** Both repos act as a paired unit. /r-start checks both clean + pulls both. /r-commit always commits both. /r-end commits + pushes both. CONV-COUNTER lives in docs repo only (process concern), but Conv number appears in both repos' commit messages.

**Rationale:** The user thinks of them as one project separated only by documentation size. The same Conv number in both repos' commits creates a traceable link. Always committing both eliminates the "forgot to commit code" failure mode.

**Consequences:** /r-start halts if either repo is dirty. /r-commit uses `git -C` for both repos. Conv number in commit titles (`Conv NNN: description`) for easy scanning.

## 3. Conv Number in Commit Title (Not Just Body)
**Type:** Implementation
**Topics:** cc-workflow, dual-repo

**Trigger:** User scans git log to count commits for timecard billing. Having the Conv number visible in the title line makes this faster.

**Options Considered:**
1. Conv only in commit body metadata ← Initially chosen, then reversed
2. Conv in title as `Conv NNN:` prefix ← Chosen

**Decision:** Commit titles start with `Conv NNN:` followed by the description. Conv also appears in the body metadata.

**Rationale:** Scannable in `git log --oneline`. User's existing workflow depends on seeing the session/conv number at a glance.

**Consequences:** Updated r-commit SKILL.md to use `Conv NNN:` title format.

## 4. Disable Session Hooks Immediately (Not Gradually)
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** session-tracker.sh hook and /r-start both modify tracked files on startup, creating a conflict.

**Options Considered:**
1. Disable session-tracker.sh hook immediately — clean break ← Chosen
2. Have /r-start expect and stage SESSION-INDEX.md changes
3. Move session tracking into /r-start (track both counters)

**Decision:** Disable both session-tracker.sh and check-resume-state.sh hooks before next launch. Session numbering ends at 393.

**Rationale:** Running both systems simultaneously creates state conflicts (hook dirties repo before /r-start can check clean state). The Conv system fully replaces both hooks' functionality (/r-start handles counter, /r-resume handles resume state).

**Consequences:** settings.json updated to remove both hooks. dual-repo-info.sh and check-output-reminder.sh hooks remain (no conflict).

## 5. Coexistence Strategy: r-* and w-* Skills Side by Side
**Type:** Strategic
**Topics:** cc-workflow, docs-infra

**Trigger:** Adopting new skills from prod-helpers while existing w-* skills are proven and in use.

**Options Considered:**
1. Replace w-* skills immediately ← Too risky
2. Run both r-* and w-* side by side, migrate when confident ← Chosen
3. Fork w-* skills and rename to r-* ← Messy

**Decision:** Both skill sets coexist. r-* skills are the Conv-aware versions; w-* skills remain for fallback. Peerloop-specific w-* skills (timecard, schema-dump, codecheck, etc.) keep their names. r-docs symlinks to w-docs/scripts/ for shared infrastructure.

**Rationale:** The r-* system is new and untested in the Peerloop context. Having w-* as fallback means no risk of losing the proven workflow. The symlink avoids script duplication during coexistence.

**Consequences:** 11 new r-* skill directories + 11 helper scripts created. w-* skills untouched. Future cleanup task (#5) to remove w-* equivalents after r-* is proven.

## 6. /r-start Halts on Dirty Repos
**Type:** Implementation
**Topics:** dual-repo, cc-workflow

**Trigger:** Designing what /r-start does when repos have uncommitted changes.

**Options Considered:**
1. Pull anyway and hope for the best ← Rejected
2. Pull docs only, report code status ← Rejected
3. Check both clean first, HALT if dirty ← Chosen

**Decision:** /r-start checks both repos for uncommitted changes before doing anything. If either is dirty, it halts and presents options (commit, stash, or discard).

**Rationale:** If the Conv lifecycle is followed correctly (/r-end commits+pushes both), repos should always be clean at /r-start. Dirty state means something went wrong — surface it immediately rather than creating a harder-to-debug state later.

**Consequences:** dual-repo-status.sh script created for pre-computed status check in /r-start.
