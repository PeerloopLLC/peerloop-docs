# Conv Decisions - 2026-03-27

## 1. Instructor-Only Webcam Storage
**Type:** Strategic
**Topics:** bbb, privacy, recordings, policy

**Trigger:** Blindside Networks offered the option to store all webcams or instructor-only

**Options Considered:**
1. Store all webcams (full session replay)
2. Store instructor webcam only <- Chosen

**Decision:** Only the instructor's (Teacher's) webcam is stored in session recordings

**Rationale:** Student privacy — students may include minors. Also reduces file size.

**Consequences:** Must contact Blindside Networks to set this on the Peerloop account. Documented in POLICIES.md §6.

## 2. No Recording Size Limit
**Type:** Implementation
**Topics:** r2, recordings, storage

**Trigger:** Discussion about whether to cap recording file sizes

**Options Considered:**
1. No limit (best-effort download, BBB URL as fallback) <- Chosen
2. Soft limit with warning (e.g. 500MB)
3. Hard limit (skip R2 for large files)

**Decision:** No max file size enforced for recording R2 replication

**Rationale:** Instructor-only webcam keeps files small (~50-200MB/hour). R2 has no per-object limit. Replication is already best-effort. `recording_size_bytes` column added for admin monitoring over time.

**Consequences:** Added `recording_size_bytes` to sessions schema. Admin can query total storage usage.

## 3. Separate R2 Bucket for Staging/Preview
**Type:** Implementation
**Topics:** cloudflare, r2, staging, infrastructure

**Trigger:** Discovered production and preview shared the same R2 bucket

**Options Considered:**
1. Separate staging R2 bucket (`peerloop-storage-staging`) <- Chosen
2. Key-prefix isolation (prefix all staging keys with `staging/`)
3. Skip R2 in preview (can't test recording replication)

**Decision:** Created `peerloop-storage-staging` R2 bucket, bound to preview environment

**Rationale:** Matches existing D1/KV separation pattern. Free. Clean isolation.

**Consequences:** Updated `wrangler.toml`, `env-vars-secrets.md`.

## 4. Staging Webhook Testing Strategy
**Type:** Strategic
**Topics:** webhooks, staging, stripe, bbb, infrastructure

**Trigger:** User asked how to test webhooks across production vs staging

**Options Considered:**
1. Branch alias `staging.peerloop.pages.dev` <- Chosen
2. Custom domain (e.g., `staging.peerloop.com`)
3. Webhook proxy Worker

**Decision:** Use `staging.peerloop.pages.dev` (already exists) as stable webhook target. BBB webhooks auto-configure per-meeting. Stripe needs a second Dashboard endpoint. Stream.io webhooks not used.

**Rationale:** Simplest approach. Staging branch already exists and is used for client approvals. No new infrastructure needed.

**Consequences:** Created `docs/guides/STAGING-WEBHOOKS-SETUP.md` checklist. User action items: CF Dashboard secrets, Stripe endpoint, Blindside email.

## 5. webhook_log Table for Payload Capture
**Type:** Implementation
**Topics:** webhooks, testing, debugging

**Trigger:** User wanted to examine webhook payload variability across real sessions

**Options Considered:**
1. Log tailing (`wrangler pages deployment tail`) — ephemeral
2. webhook_log table in D1 <- Chosen
3. Standalone capture endpoint

**Decision:** Added `webhook_log` table with fire-and-forget INSERTs in all 3 webhook handlers

**Rationale:** Durable, queryable, in-context (knows which handler received it). Useful beyond fixtures — debugging, audit. May be replaced by proper API logging later.

**Consequences:** Schema change (new table + 2 indexes). 3 webhook handlers modified. Auth headers redacted.
