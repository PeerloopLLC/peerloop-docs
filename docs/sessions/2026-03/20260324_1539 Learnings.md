## 1. createAPIContext Test Helper Supports Runtime Env Vars
**Topics:** testing, video

**Context:** BBB-CLEANUP tests needed `BBB_URL` and `BBB_SECRET` in the runtime environment for `createBBBProvider()` to be called. The initial test failed because `endRoom` was never invoked — the code path checked for env vars and silently skipped.

**Learning:** `createAPIContext()` in `tests/api/helpers/api-test-helper.ts` accepts an `env` option that merges into `locals.runtime.env`. Pass `{ env: { BBB_URL: '...', BBB_SECRET: '...' } }` to test code paths that depend on runtime environment variables.

**Pattern:**
```typescript
const ctx = createAPIContext({ db: getTestDB(), env: { BBB_URL: 'https://bbb.test', BBB_SECRET: 'test-secret' } });
```

## 2. Non-Blocking Async Pattern for Post-Completion Actions
**Topics:** astro, d1

**Context:** `triggerPostSessionActions()` needed to run after `completeSession()` but failures must not break the completion response. Using `await` would make failures propagate.

**Learning:** Use `.catch()` on the promise (not `await` with try/catch) to fire-and-forget. This ensures the function runs but its failure doesn't affect the caller's return value. The caller remains synchronous from its perspective.

**Pattern:**
```typescript
triggerPostSessionActions(db, sessionId, ...)
  .catch((err) => console.error('Post-session actions failed (non-critical):', err));

return { status: 'completed', ... }; // Always returns, regardless of action failures
```

## 3. Module Backfill Requires Terminal-State Check on Earlier Sessions
**Topics:** d1

**Context:** When backfilling NULL `module_id` on later completed sessions, we can't just check "are earlier sessions completed?" — cancelled and no-show sessions should also count as resolved.

**Learning:** The backfill guard checks `status != 'completed' AND status != 'cancelled' AND status != 'no_show'`. Any of these three terminal states means the earlier session won't change position, so the later session's module can be safely resolved.

## 4. Enrollment Completion Uses DISTINCT Module Count
**Topics:** d1

**Context:** Need to determine "all modules covered" for auto-completing enrollment. Can't just count completed sessions — a module could theoretically have multiple sessions (reschedule → original + replacement both completed).

**Learning:** `COUNT(DISTINCT module_id)` against completed sessions with non-NULL `module_id` compared to `COUNT(*)` of `course_curriculum` rows gives a reliable "all modules covered" check. This handles edge cases where duplicate sessions exist for the same module.

## 5. user_stats Upsert Pattern for Counter Increments
**Topics:** d1

**Context:** `user_stats` rows may not exist for a user when their first session completes. Need to handle both insert and increment atomically.

**Learning:** D1/SQLite `INSERT ... ON CONFLICT DO UPDATE` handles the upsert cleanly. The initial INSERT provides defaults for a new row, and the ON CONFLICT clause increments the existing value.

**Pattern:**
```sql
INSERT INTO user_stats (user_id, sessions_completed) VALUES (?, 1)
ON CONFLICT(user_id) DO UPDATE SET sessions_completed = sessions_completed + 1,
  updated_at = strftime('%Y-%m-%dT%H:%M:%fZ', 'now')
```
