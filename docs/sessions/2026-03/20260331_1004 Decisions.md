# Conv Decisions - 2026-03-31

## 1. PLATO Flywheel Run Structure
**Type:** Implementation
**Topics:** plato, testing

**Trigger:** Original plan had register-student, enroll-student, book-session, complete-session, certify-teacher. Endpoint analysis revealed prerequisite gaps.

**Options Considered:**
1. Original 5-run plan (register, enroll, book, complete, certify)
2. Revised plan: register-student, self-certify-creator, enroll-student, complete-course, certify-teacher ← Chosen

**Decision:** Added self-certify-creator run (Stripe Connect + teacher cert) before enrollment. Merged book+complete into complete-course (handles all 3 sessions). Kept 5 total runs.

**Rationale:** Enrollment guards require teacher_certifications count > 0. Booking requires teacher cert. 3 modules need 3 session cycles. The revised structure satisfies all prerequisites.

**Consequences:** Full flywheel now closes in 11 runs total (6 existing + 5 new). 6367 tests passing.

## 2. Persona Field Organization: DB-REQUIRED vs SITE-NECESSARY
**Type:** Strategic
**Topics:** plato, testing-philosophy, data-model

**Trigger:** User observed that PLATO course data was minimal — About tab empty during manual testing. Asked how to make personas richer while keeping the system extensible.

**Options Considered:**
1. Option A: Flat fields with section comments, no conditionals ← Chosen
2. Option B: Nested entity bags (course: { ... }, community: { ... })
3. Option C: Entity templates with spread interpolation

**Decision:** Keep flat persona structure (Option A). Organize fields into DB-REQUIRED and SITE-NECESSARY sections using comments. No runtime conditionals — if persona has the field and run references it, it's sent.

**Rationale:** Simplest approach. Self-documenting via comments. To create a second creator, copy the block and change values. No structural changes to runner or interpolation. User explicitly chose simplicity over flexibility.

**Consequences:** Persona file now has all schema fields for courses, communities, and modules. Runs updated to send site-necessary fields. Both PLATO docs updated with the pattern.

> **Insight:** The publish gate (minimum validation) and "complete data" (what a real user fills out) are fundamentally different concerns. Test frameworks that only satisfy validation gates miss the user experience surface area — the same gap that causes empty product pages in production despite all tests passing.

## 3. Skip Session Join in PLATO — Fire BBB Webhook Directly
**Type:** Implementation
**Topics:** plato, bbb, testing

**Trigger:** Session join endpoint has a 15-minute join window validation that would require time manipulation in tests.

**Options Considered:**
1. Mock the clock to manipulate time for join window validation
2. Skip join, fire BBB room_ended webhook directly on scheduled sessions ← Chosen

**Decision:** Skip the join endpoint entirely. Fire BBB `room_ended` webhook directly — `completeSession()` accepts both `scheduled` and `in_progress` sessions.

**Rationale:** The join endpoint's time validation adds complexity without testing the data chain PLATO cares about. The join→BBB→complete path is tested by unit/integration tests. PLATO only needs to prove the booking→completion→enrollment-auto-complete chain works.

**Consequences:** Run 10 (complete-course) is simpler: book + BBB webhook per session. No clock mocking needed. Sessions go from 'scheduled' directly to 'completed'.
