# Session Learnings - 2026-03-12

## 1. Components Bypassing Global State Create Hidden API Costs
**Topics:** astro, auth

**Context:** Auditing currentUser usage found 5 components making their own `/api/auth/session` calls despite running on pages where `AppNavbar` already initializes the full `getCurrentUser()` global.

**Learning:** When a global state system exists (like `window.__peerloop.currentUser`), components written before or independently of that system tend to create their own lightweight auth path. This creates hidden per-page API costs — `EnrollButton` was making 2 network requests (session + enrollment check) that could be answered in O(1) from the already-loaded `CurrentUser` Map. The cost is invisible because each component works fine in isolation.

**Pattern:** Periodically audit for `fetch('/api/auth/session')` calls in components that live on AppLayout pages — they should use `getCurrentUser()` instead.

## 2. Dual SessionUser Types Diverge Silently
**Topics:** auth, testing

**Context:** Found two independent `SessionUser` interfaces — one in `messages/types.ts` (4 fields) and one in `context-actions/types.ts` (11 fields with role booleans) — neither matching `CurrentUser`'s capability-based model.

**Learning:** When multiple parts of a codebase define their own "who is the user" type, the types diverge silently. The context-actions `SessionUser` used flat `is_teacher`/`is_creator` booleans from the DB, while `CurrentUser` derives these from course relationships (`isActiveTeacher()`, `hasCreatedCourses()`). The flat booleans can be wrong — a user with `can_create_courses=true` but zero courses would show "Creator Dashboard" incorrectly. Consolidating to a single source of truth fixes the semantic gap.

## 3. React Hooks in Non-React Files Need Import Awareness
**Topics:** astro, testing

**Context:** Added `useCurrentUser()` hook directly to `current-user.ts` (a lib file, not a component). This required adding `import { useState, useEffect } from 'react'` at the top of a file that was previously framework-agnostic.

**Learning:** Adding React hooks to a lib file couples it to React's module system. This works fine in Astro's React islands architecture because the file is only ever imported by React components (which are bundled with React). But it means the file can no longer be imported in pure server-side contexts without React being available. For `current-user.ts` this is acceptable since it's inherently client-side (`window.__peerloop`), but be aware of the coupling.

## 4. Listener Pattern for Global Singletons Enables React Reactivity
**Topics:** astro, auth

**Context:** `getCurrentUser()` reads from `window.__peerloop` — a raw singleton. React components calling it directly get the value at render time but don't re-render when background stale-while-revalidate updates the user.

**Learning:** A simple `Set<Listener>` + `notifyListeners()` in `setCurrentUser()` bridges the gap between a non-React singleton and React's reactivity model. The `useCurrentUser()` hook subscribes on mount, unsubscribes on unmount, and calls `setUser()` on changes — standard pattern but easy to forget when the global was designed before React hooks were needed. The subscription mechanism (`subscribeToUserChange`) is also useful for non-React consumers (analytics, logging).

## 5. Unread Counts Fit Naturally in /api/me/full
**Topics:** auth, d1

**Context:** `AppNavbar` was making a separate `/api/me/notifications/count` call on every page load, in addition to the `/api/me/full` call from `initializeCurrentUser()`.

**Learning:** Lightweight aggregate counts (unread notifications, unread messages) are ideal candidates for inclusion in the "bootstrap" API response. They add negligible query cost (simple COUNT with indexed WHERE) but eliminate an entire HTTP round-trip per page load. The `?? 0` fallback in the constructor handles backward compatibility with older cached data that lacks the new fields — important for the stale-while-revalidate pattern where localStorage may hold pre-upgrade data.
