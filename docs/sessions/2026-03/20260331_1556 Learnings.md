# Conv Learnings - 2026-03-31

## 1. CTE Cross-Reference Limitation in D1 INSERT Context
**Topics:** d1, testing, plato

**Context:** Building SqlTopUp enrichment steps that need enrollment IDs (which depend on user IDs and course IDs from earlier CTEs)

**Learning:** In the D1/better-sqlite3 test environment, CTEs that reference other CTEs via scalar subqueries (e.g., `_e` referencing `_u` and `_c`) return NULL when the CTE is part of an INSERT...SELECT statement. Single-level CTEs (`_u`, `_c`, `_comm`) work fine. The workaround is to use explicit JOINs on the base tables instead of CTE references for enrollment lookups.

**Pattern:** Use `FROM enrollments e JOIN users u ON e.student_id = u.id JOIN courses c ON e.course_id = c.id WHERE u.email = ? AND c.title = ?` instead of referencing enrollment CTEs.

## 2. flattenCourseData Iteration Order Bug
**Topics:** plato, testing

**Context:** Multi-creator seed scenario where Gabriel's courses were being ignored in favor of Guy's

**Learning:** `Object.entries()` on a spread object preserves insertion order from the original persona set. When `flattenCourseData` finds the "first persona with courses," it always finds Guy (defined first in seed-full.ts). The fix is to prefer the explicitly bound `'creator'` actor slot, which correctly points to whichever creator persona is active for that chain step.

## 3. Core Seed vs Dev Seed Admin Identity
**Topics:** plato, migrations

**Context:** SqlTopUp steps referencing the admin user for payouts, moderation, and notifications

**Learning:** The core seed (`0002_seed_core.sql`) creates admin as `admin@peerloop.com` with ID `usr-admin`. The dev seed creates a separate "Brian" admin at `brian@peerloop.com`. Since PLATO replaces the dev seed, only `admin@peerloop.com` exists. The seed-full persona set's admin entry still references `brian@peerloop.com` but isn't used in chain steps.

## 4. UNION ALL with JOINs in D1 INSERT Context
**Topics:** d1, testing, plato

**Context:** Attempting to INSERT multiple enrollment-dependent rows using UNION ALL with different JOIN conditions

**Learning:** INSERT...SELECT with UNION ALL where each SELECT has its own JOIN pattern works in SQLite but reports success even when 0 rows are inserted (no error on empty result set). `prepare().run()` doesn't fail -- it just inserts nothing. For seed data, splitting into individual INSERT steps per enrollment is more reliable and debuggable than UNION ALL.
