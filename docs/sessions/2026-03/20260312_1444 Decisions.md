# Session Decisions - 2026-03-12

## 1. Merge /learn Into Course Detail Page as Learn Tab
**Type:** Strategic
**Topics:** astro, workflow

**Trigger:** After adding the Sessions tab to the course detail page, it became clear that `/course/{slug}/learn` and `/course/{slug}` had significant overlap (both show curriculum, resources, progress, sessions). Students navigated between two pages for one course.

**Options Considered:**
1. Full merge — absorb /learn into the detail page as a Learn tab ← Chosen
2. Keep both but auto-redirect enrolled students from detail page to /learn
3. Keep both but remove overlapping tabs (Sessions, Curriculum) from detail page

**Decision:** One unified course page with role-adaptive tabs. Non-enrolled visitors see marketing tabs (About, Teachers, Resources, Feed). Enrolled students additionally see Learn, Sessions, Curriculum. The standalone `/learn` page will be retired.

**Rationale:** Option C (keep both, clean overlap) reduces confusion but sacrifices convenience — enrolled students still bounce between pages. Option B (redirect) hides the marketing page from enrolled users who might want it. Option A gives one URL, one page, one mental model. The `/learn` content (sidebar + sub-tabs) gets simplified into an accordion within the tab, removing ~690 lines of orchestration code.

**Consequences:** COURSE-PAGE-MERGE block added to PLAN.md with 4 sub-blocks. Estimated ~300 lines new code, ~690 lines retired. ModuleContent, HomeworkTab, CourseReviewModal reused as-is.

## 2. Accordion Modules (Option C) Over Strip Navigation (Option B)
**Type:** Implementation
**Topics:** astro, workflow

**Trigger:** Two viable designs for the Learn tab — needed to choose one before planning implementation.

**Options Considered:**
1. Option B: Horizontal numbered pill strip + single content area below
2. Option C: Vertical accordion cards, current auto-expanded, others collapsed ← Chosen

**Decision:** Accordion cards (Option C) for the Learn tab.

**Rationale:** Decided by evaluating from a new student's perspective. C shows module titles in collapsed cards — acts as a table of contents confirming what they paid for. B hides titles behind numbered pills requiring click-through. Additionally: C shows completed module dates/teachers (progress timeline), scales naturally to 10+ modules, and Peerloop courses have descriptive module titles (confirmed) which make collapsed cards informative.

**Consequences:** ASCII mockups for 3 lifecycle states (new, mid-course, complete) captured in PLAN.md. Implementation will reuse ModuleContent for expanded state and build a simple collapsed card for the other states.

## 3. Surface Sessions on My Courses Cards Instead of Adding Nav Item
**Type:** Strategic
**Topics:** astro

**Trigger:** Students couldn't find their next session without navigating to the dashboard, then finding the "View All" link. The `/learning/sessions` page existed but was buried.

**Options Considered:**
1. Add "Sessions" to the AppNavbar sidebar ← Rejected (client pushing to reduce nav items)
2. Add session banners to My Courses (`/courses`) cards + Sessions tab on course page ← Chosen
3. Both A and B

**Decision:** Surface session info contextually on My Courses cards and the course page Sessions tab. No new nav item.

**Rationale:** Client is exerting pressure to reduce AppNavbar items. The student's natural instinct is "My Courses" to find their sessions. Putting session info directly on course cards (1 click from nav) and providing a Sessions tab on each course page covers the need without nav bloat.

**Consequences:** MyCourses cards now show "Next: date with teacher" banners. Cards link to `/course/{slug}/sessions` (or directly to `/session/{id}` when joinable). "Book Session" button conditionally hidden when a session exists. Completed courses show "Sessions" button linking to history.

## 4. Content Never Locked — Only Session Completion Order Gated
**Type:** Strategic
**Topics:** workflow

**Trigger:** Initial mockups showed future modules as "locked", implying inaccessible content. Client corrected this.

**Options Considered:**
1. Lock future module content until previous session completed ← Rejected
2. All content always accessible, only session completion order gated ← Chosen

**Decision:** Students can browse all module content (video, objectives, exercises, resources) freely at any time. The only constraint: a module's session cannot be marked complete before the previous module's session is done.

**Rationale:** Students learn at different paces and may want to preview upcoming material. Locking content creates frustration without educational benefit. The real dependency is on the 1-on-1 tutoring session sequence, not on content access.

**Consequences:** No lock icons or greyed-out UI. Future modules are clickable/expandable in the accordion. Only the [Mark Complete] action is gated on session order.
