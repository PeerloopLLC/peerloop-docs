# Conv Learnings - 2026-03-19

## 1. SQLite `%f` Format Specifier Produces Exact JS Match
**Topics:** d1

**Context:** During DATE-FORMAT schema migration, needed to decide whether SQL defaults should produce milliseconds (`.000Z`) matching JavaScript's `toISOString()`.

**Learning:** SQLite's `strftime('%Y-%m-%dT%H:%M:%fZ', 'now')` uses `%f` for `SS.SSS` (seconds with 3-digit milliseconds). This produces `2026-03-19T00:58:54.067Z` — an exact format match with JavaScript's `new Date().toISOString()`. No ambiguity, no bridge logic needed for newly-written data.

## 2. Parallel Agent Strategy for Large Mechanical Migrations
**Topics:** workflow

**Context:** DATE-FORMAT migration touched ~130 files across 5 phases. Phase 3 (app writes, ~70 files) and Phase 4 (display, ~58 files) were the largest.

**Learning:** Splitting mechanical migrations into parallel agents by file area (admin components, dashboard/teacher/course, remaining) dramatically reduces wall-clock time. Each agent can work independently because the edits are file-local (adding an import, replacing a function call). The key prerequisite: edits must not overlap between agents.

**Pattern:** Phase 3 split into 3A (SQL `datetime('now')`) and 3B (`now()` deprecation) — ran in parallel because file overlap was minimal and the edits target different code patterns. Phase 4 split into 3 batches by component area.

## 3. `parseUTCDate()` Bridge Enables Zero-Regression Migration
**Topics:** d1, astro

**Context:** Migrating 130+ files from old format (`2026-03-18 23:41:25`) to new format (`2026-03-18T23:41:25.000Z`) could break tests or display if any code expects the old format.

**Learning:** The `parseUTCDate()` function in `timezone.ts` normalizes both old and new formats to a `Date` object by detecting the presence/absence of `Z` or `T`. Because ALL display functions route through `parseUTCDate()`, the migration could proceed phase-by-phase without breakage. The full test suite (5901 tests) passed after each phase with only 1 trivial test regex update needed.

## 4. SQL String Comparison Risk with Mixed Formats
**Topics:** d1

**Context:** `datetime('now')` in WHERE clauses produces `2026-03-18 23:41:25`. If stored values use `2026-03-18T23:41:25.000Z`, the `T` (ASCII 84) > space (ASCII 32) causes comparison bugs for same-instant values.

**Learning:** When migrating timestamp formats, WHERE/CASE clauses must be migrated alongside INSERT/UPDATE — not deferred. SQL string comparison is lexicographic, so format consistency between stored values and comparison values is critical. The fix: parameterize all `datetime('now')` with `toUTCISOString()` bound via `?`.

## 5. Number Formatting `toLocaleString()` Must Be Excluded
**Topics:** astro

**Context:** Grep for `toLocaleString` found ~81 files, but ~12 of those use it for number formatting (currency, counts like `amount.toLocaleString()`), not dates.

**Learning:** When doing a broad display migration targeting `toLocaleDateString()`/`toLocaleString()`, always distinguish date formatting from number formatting. A mechanical replace-all would break currency displays. The agents correctly identified and excluded these.
