# Session Learnings - 2026-03-01

## 1. Astro SSR: `window` Not Available in Component Body
**Topics:** astro

**Context:** Added `new URLSearchParams(window.location.search)` directly in a React component's function body to read URL query params.

**Learning:** In Astro's `client:load` directive, the component renders on the server first (SSR), then hydrates on the client. `window` doesn't exist during SSR. Browser APIs must be accessed inside `useEffect`, which only runs client-side. The error message ("Browser APIs are not available on the server") is clear but easy to forget when writing what feels like client-only code.

---

## 2. Astro.url.searchParams Available Server-Side for Conditional Page Rendering
**Topics:** astro

**Context:** Needed the Astro page (`studio.astro`) to render different breadcrumbs and page titles depending on whether `?course=<id>` was in the URL.

**Learning:** `Astro.url.searchParams` is available in the frontmatter because the full URL is known at request time. This allows server-rendered HTML to adapt to query params without any client-side JavaScript — breadcrumbs, `<title>`, and conditional sections can all be SSR'd correctly on first paint.

## 3. Combine URL-Reading and Data-Fetching into a Single Mount Effect
**Topics:** astro

**Context:** Had two separate `useEffect(() => {}, [])` hooks — one reading `?course=` from the URL, the other fetching course data. The fetch effect fired before the URL effect set state, causing an unnecessary API call.

**Learning:** Two `useEffect`s with empty deps both fire on the same render tick. The second effect sees the initial state, not state set by the first. When one effect's result determines whether another should run, combine them into a single effect: check URL first, then conditionally fetch. This avoids the race and the wasted network request.

**Pattern:**
```tsx
useEffect(() => {
  const courseId = new URLSearchParams(window.location.search).get('course');
  if (courseId) {
    setEditingCourseId(courseId);
  } else {
    fetchCourses();
  }
}, [fetchCourses]);
```

---

## 4. Render Order: Sync Guards Before Async Guards
**Topics:** astro

**Context:** The `editingCourseId` check (synchronous, from URL) was placed after the `loading` check (async, from fetch). When skipping the fetch in editor mode, `loading` stayed `true` forever, showing a stuck spinner.

**Learning:** When mixing synchronous and asynchronous render guards, check the synchronous condition first. If `editingCourseId` is known immediately (from URL/state), it should short-circuit the render before the loading spinner gets a chance to show.
