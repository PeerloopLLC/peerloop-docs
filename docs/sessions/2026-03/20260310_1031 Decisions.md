# Session Decisions - 2026-03-10

## 1. Use Marker-Anchored Detection in detect-changes.sh
**Type:** Implementation
**Topics:** cc-workflow, docs-infra

**Trigger:** `/q-docs` Live Context showed 260+ changed files for a session that only modified 5 docs files. The `HEAD~5` strategy captured weeks of prior sessions' TERMINOLOGY work, wasting tokens on triage.

**Options Considered:**
1. Keep `HEAD~5` (status quo) — arbitrary commit count, unpredictable window
2. Use `--since "24 hours ago"` — time-based, better but still shows prior sessions' work on multi-session days
3. Marker-anchored: record HEAD SHAs after each run, diff from marker forward ← Chosen

**Decision:** Replace `HEAD~5` with a marker file (`.last-qdocs-run`) that records both repos' HEAD SHAs after each `/q-docs` run. The next run diffs from the marker to current HEAD plus uncommitted changes. Falls back to `--since "24 hours ago"` when no marker exists.

**Rationale:** The marker pattern answers the precise question — "what changed since the last time we documented?" — rather than using time or commit count as proxies. Eliminates duplicate work across multi-session days. Same pattern as database migration cursors.

**Consequences:** `detect-changes.sh` rewritten with `collect_changes()` function supporting both marker and time-based modes. Marker file is committed to git (not gitignored) for cross-machine continuity. `--reset` flag added as escape hatch.

## 2. Single Architecture Doc for Skills System
**Type:** Implementation
**Topics:** docs-infra

**Trigger:** User asked where to document the new Skills 2 system — one doc per skill or a single doc for all?

**Options Considered:**
1. One doc per skill (parallels skill directories) — many thin files with redundant system context
2. Single doc for the system ← Chosen

**Decision:** Create `docs/architecture/skills-system.md` covering the Skills 2 format, skills-canon drift model, per-skill reference table, and how-to guides.

**Rationale:** The system is the interesting thing — backtick injection, drift tools, marker strategy — not individual skill internals. Individual skill behavior is already documented in their SKILL.md files. A reader trying to understand "how do our skills work" needs the full picture in one place. The per-skill reference table grows as skills are migrated but the system docs stay stable.

**Consequences:** Created 185-line architecture doc. Added reference in CLAUDE.md's Research Reference section.
