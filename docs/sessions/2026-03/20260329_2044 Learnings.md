# Conv Learnings - 2026-03-29

## 1. CC Plan Mode Files Are Ephemeral
**Topics:** cc-workflow, plan-persistence

**Context:** Conv 055 created a detailed ADMIN-INTEL plan in Plan Mode. User was told it was "saved." After /r-end, /clear, /r-start, the plan file was gone.

**Learning:** `.claude/plans/` is gitignored — plan files exist only during the conversation that created them. "Save" in Plan Mode means "write to the plans directory" which is ephemeral. For cross-session persistence, plans must be written to a committed location (e.g., `docs/as-designed/`).

**Pattern:** Before /r-end, check if any plan files exist in `.claude/plans/` and offer to persist them to a durable location. When user says "save the plan" — they mean persist permanently.

## 2. Context-Driven Admin Content Design
**Topics:** admin, architecture, ux-design

**Context:** Designing what goes into admin tabs/sections on member-facing pages.

**Learning:** Admin content on member pages should NOT be pre-specified per-page. Instead, each page asks: "What admin knowledge or actions would help an admin looking at THIS entity right now?" The answer emerges during implementation per-page. The API returns comprehensive data; the component decides what to show. This avoids premature specification and allows the content to evolve naturally.

## 3. ExtraTabConfig String Typing Enables Clean Admin Injection
**Topics:** role-tabs, type-safety, admin

**Context:** Planning how to add admin tab without modifying CourseRole type system.

**Learning:** `ExtraTabConfig.id` is `string` and `ExtraTabConfig.roleColor` is `string` — not union-typed like `DetailTabId` or `CourseRole`. This means admin tab injection requires zero type system changes: just push an ExtraTabConfig with `id: 'admin-intel'` and `roleColor: 'red'` directly in ExploreCourseTabs. The `DetailTabId` union is only for `computeRoleTabs()` return type. The existing tab system was already designed to accept arbitrary extensions.

## 4. Course-to-Community Join Path via Progressions
**Topics:** schema, d1, admin

**Context:** Writing SQL for course intel flagged content count.

**Learning:** Courses don't directly own communities. The FK chain is: `courses.progression_id` -> `progressions.community_id` -> `communities.id` -> `content_flags.community_id`. This multi-hop join is needed for any query that relates course-level data to community-level data (e.g., flagged content for a course). The progressions table is the bridge — communities are the social layer, progressions group courses within them.
