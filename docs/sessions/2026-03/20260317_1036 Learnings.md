# Session Learnings - 2026-03-17

## 1. Session vs Conversation Terminology Collision
**Topics:** cc-workflow

**Context:** The project used "Session" for both Claude Code process invocations AND numbered work units tracked in SESSION-INDEX.md. This created ambiguity — "Session 393" could mean the CC process or the numbered work block.

**Learning:** When adopting tooling that has its own terminology (Claude Code uses "session" internally), choose a distinct term for project-level concepts that avoids collision. "Conv" (Conversation) cleanly separates the project's numbered work unit from CC's internal session concept. The distinction matters most during `/clear` cycles — one CC session can span multiple Convs (via `/r-pre-clear`), and one Conv can span multiple CC sessions (across machines).

## 2. Session Hook vs Skill Startup Conflict
**Topics:** cc-workflow, dual-repo

**Context:** The existing `session-tracker.sh` hook auto-incremented session numbers on every CC launch. The new `/r-start` skill also increments a counter. Running both would create a sequencing problem — the hook modifies `SESSION-INDEX.md` before `/r-start` checks for clean repos, causing a HALT.

**Learning:** When replacing a hook-based system with a skill-based one, the transition must be atomic — disable the old hook before the first invocation of the new skill. Having both active simultaneously creates state conflicts that are hard to debug. The cleanest approach is: commit everything with the old system, disable the hook, quit, then start fresh with the new system.

## 3. Dual-Repo Pull Safety
**Topics:** dual-repo, cc-workflow

**Context:** Designing `/r-start` for dual-repo — should it pull both repos or just docs?

**Learning:** The concern about pulling a code repo with uncommitted changes is valid but self-resolving if the lifecycle is followed correctly. `/r-end` commits+pushes both repos, so the next `/r-start` finds both clean. The real risk is work done *outside* the Conv lifecycle (IDE edits between sessions). The right answer is: check both repos for dirty state first, HALT if dirty, then pull both. This surfaces problems immediately rather than hiding them.

## 4. Wrapper Scripts for ! Backtick Pre-computation
**Topics:** cc-workflow

**Context:** Porting skills from prod-helpers that use `!` backtick pre-computed context with shell pipes.

**Learning:** The permission checker matches full command strings — piped commands trigger "multiple operations" failures. Extracting piped commands into named scripts in `.claude/scripts/` solves this cleanly. One blanket permission rule `Bash(.claude/scripts/*)` covers all scripts, and the script filenames serve as self-documentation in the skill's `!` backtick lines. All scripts should use `$CLAUDE_PROJECT_DIR` for path resolution to handle cwd drift.

---

## 5. Symlinks for Shared Skill Infrastructure
**Topics:** cc-workflow, docs-infra

**Context:** Both w-docs and r-docs need the same detection scripts (detect-changes.sh, sync-gaps.sh, tech-doc-sweep.sh, dev-env-scan.sh, route-mapping.txt). Duplicating would create drift risk.

**Learning:** Symlinking `r-docs/scripts → w-docs/scripts` keeps a single source of truth during the coexistence period. The `${CLAUDE_SKILL_DIR}` variable resolves correctly through the symlink. When the w-* system is eventually removed, the scripts move to r-docs/scripts/ as actual files.
