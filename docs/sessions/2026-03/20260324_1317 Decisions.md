# Conv Decisions - 2026-03-24

## 1. Students Cannot Complete Sessions — Teacher/Creator Only
**Type:** Strategic
**Topics:** auth, video

**Trigger:** Initially added student permission to the `POST /api/sessions/:id/complete` endpoint so students could fix stuck sessions. User flagged the abuse risk.

**Options Considered:**
1. Allow all participants (teacher + student) to complete ← Initially implemented
2. Teacher and course creator only ← Chosen
3. Teacher, creator, and admin only

**Decision:** Only the session teacher or course creator can mark a session complete. Students get an inline message form to notify the teacher instead.

**Rationale:** Completion triggers `module_id` freezing and progress tracking. A student who no-showed could click "Complete Session Now" to get `completed` status (with module credit) instead of `no_show`. The teacher is the authority on whether the session actually happened.

**Consequences:** Students see a "Message teacher" form instead of the complete button. Added inline message sending via `POST /api/conversations` with pre-populated text. Added linkified URLs in MessageThread so session links in messages are clickable.

> **Insight:** Auth expansion decisions should always include abuse-case analysis. The question isn't "can this user do this action?" but "what would a bad-faith user gain from this permission?"

## 2. Client-Side Completion Fallback Now, Cron Trigger Deferred
**Type:** Strategic
**Topics:** video, cloudflare

**Trigger:** Auto-complete via admin cleanup endpoint requires manual admin action. Need automatic recovery for stuck sessions.

**Options Considered:**
1. Cloudflare Cron Triggers — `scheduled()` handler runs hourly ← Deferred (CRON-CLEANUP block)
2. Client-side polling fallback — "Complete Session Now" button after `scheduledEnd` ← Chosen for now
3. Piggyback on admin dashboard page load — run cleanup during SSR

**Decision:** Implement client-side fallback now (Option 2), defer Cron Triggers to pre-launch (Option 1). Option 3 rejected — side effects during page rendering is architecturally messy.

**Rationale:** Client-side is zero infrastructure and covers the most common case (participant is still on the page). Cron needs Astro + Cloudflare adapter dual-export investigation and notification batching design — better as its own block.

**Consequences:** Created CRON-CLEANUP deferred block in PLAN.md with investigation items. Defense-in-depth chain: BBB webhook → empty room → client-side → admin cleanup → future cron.

## 3. Solution Quality Override in CLAUDE.md
**Type:** Strategic
**Topics:** cc-workflow

**Trigger:** User noticed a pattern of defaulting to "simplest" solutions without presenting alternatives. Multiple instances in this conv: generic message link instead of inline form, silent error swallowing, not considering auth implications.

**Options Considered:**
1. Memory note only
2. CLAUDE.md override of system prompt defaults ← Chosen
3. Both ← Chosen

**Decision:** Added CLAUDE.md section that explicitly overrides the system prompt's "avoid over-engineering" and "try the simplest approach first" directives. Must always present quick, durable, and middle-ground options and let the user choose.

**Rationale:** The system prompt's default bias toward simplicity is appropriate for general use but wrong for this production project. Accumulated quick fixes create long-term debt. The user decides what constitutes over-engineering.

**Consequences:** CLAUDE.md override + memory file `feedback_no_simplest_fix.md`. Applies to all future conversations in this project.

## 4. Build Admin Cleanup Endpoint for STALE-CLEANUP From the Start
**Type:** Implementation
**Topics:** video

**Trigger:** NO-SHOW PLAN item specified `POST /api/admin/sessions/no-shows`. AUTO-COMPLETE specified renaming it to `/cleanup`. STALE-CLEANUP specified consolidating both.

**Decision:** Built `POST /api/admin/sessions/cleanup` from the start with both `detectNoShows()` and a placeholder for `detectStaleInProgress()`, then wired in auto-complete in the same conv. Avoided creating and then renaming/consolidating endpoints.

**Rationale:** Three PLAN items shared the same endpoint. Building the consolidated version upfront avoided a rename and API change mid-block.
