# Conv Learnings - 2026-03-31

## 1. Race Condition in Agent-Written Files
**Topics:** cc-workflow, dual-repo

**Context:** `.last-qdocs-run` marker file was tracked by git. The docs agent writes it during `/r-end`, but if the write completes after the Step 6 commit, the file is left dirty — blocking the next `/r-start`.

**Learning:** Files written by `/r-end` agents as side effects should be gitignored, not tracked. The intermittent nature (doesn't happen every time) is classic race behavior — agent completion timing vs `git add .` timing.

**Pattern:** When a script writes a file that's only meaningful locally (anchors, markers, caches), add it to `.gitignore` and `git rm --cached`.

## 2. Discovery GET Pattern for Cross-Run Data
**Topics:** testing, plato

**Context:** In Model B, runs can't carry IDs from prior runs. Run 5 (add-modules) needs the courseId created by Run 4.

**Learning:** Each run that operates on a prior run's data starts with a "discovery GET" — the same API call the page would make to load its data. This is realistic (users browse their dashboard to find things) and validates list/detail endpoints simultaneously.

**Pattern:** `GET /api/me/courses` → extract `courses[0].id` → use in subsequent POST/PUT calls within the same run.

## 3. Actor Resolution via DB Lookup
**Topics:** testing, plato

**Context:** Without cross-run carry, how does Run 2 know the creator's userId registered in Run 1?

**Learning:** The runner queries the DB by persona email: `SELECT id, email FROM users WHERE email = ?`. This is infrastructure setup (like `seedCoreTestDB()`), not a test assertion. Simple, realistic, no carry state needed.

---

## 4. Intra-Run Context vs Cross-Run Carry
**Topics:** testing, plato

**Context:** Model B eliminates cross-run carry but needed a mechanism for within-run data flow (GET response feeds POST request on the same page).

**Learning:** `$context.actionName.key` replaces `$carry.stepName.key` for intra-run references. Semantically: "what the page showed the user." Context is cleared between runs — only the DB persists. This distinction (intra-run context vs cross-run carry) is the key design insight.

---

## 5. API Response Shapes Require Verification
**Topics:** testing, plato

**Context:** Run definitions assumed `results[0].id` but actual endpoints return `courses[0].id`, `communities[0].slug`, etc.

**Learning:** When writing discovery GET actions, always check the actual endpoint's `Response.json(...)` call to get the correct response key. Don't assume a generic `results` wrapper.
