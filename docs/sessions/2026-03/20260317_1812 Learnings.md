# Conv Learnings - 2026-03-17

## 1. SQLite datetime() vs JavaScript toISOString() Format Mismatch
**Topics:** d1, testing

**Context:** Session invite endpoints used `datetime('now')` in SQL for expiry comparisons, but stored `expires_at` using JavaScript's `new Date().toISOString()`.

**Learning:** SQLite's `datetime('now')` returns `YYYY-MM-DD HH:MM:SS` format, while JS `.toISOString()` returns `YYYY-MM-DDTHH:MM:SS.sssZ`. String comparison between these formats fails silently — the `T` separator and `Z` suffix cause lexicographic mismatch. Always use parameterized JS ISO strings instead of `datetime('now')` when the stored values were written by JS.

**Pattern:**
```sql
-- BAD: mixes SQLite datetime format with JS ISO strings
WHERE expires_at <= datetime('now')

-- GOOD: pass JS Date as parameter
const nowISO = new Date().toISOString();
WHERE expires_at <= ?  -- bind nowISO
```

## 2. FK Ordering in Test Cleanup
**Topics:** d1, testing

**Context:** Integration tests for session invites had `session_invites.notification_id → notifications(id)` and `session_invites.accepted_session_id → sessions(id)` foreign keys. Cleanup DELETE statements failed with FK constraint errors.

**Learning:** D1/SQLite enforces FK constraints on DELETE. When cleaning up test data with FK relationships, always delete child tables first. For session invites: delete `session_invites` before `notifications` (notification_id FK) and before `sessions` (accepted_session_id FK).

## 3. Component Interface vs API Response Casing Mismatch
**Topics:** astro, testing

**Context:** `MyStudents.tsx` defined its `Student` interface with snake_case fields (`enrollment_id`, `course_slug`, etc.) but the API endpoint `/api/me/teacher-students` returned camelCase (`enrollmentId`, `courseSlug`). The component rendered without TypeScript errors because the interface was manually typed, not generated from the API response.

**Learning:** When a React component consumes an API response, verify the interface field names match the actual JSON keys. TypeScript won't catch this at compile time if the interface is declared separately from the fetch call — the mismatch only surfaces at runtime as `undefined` values. The "Book Now" button sent `enrollment_id: undefined` because `student.enrollment_id` didn't exist on the actual response object.

## 4. Conflict Checks Must Run After Reschedule Cancellation
**Topics:** d1

**Context:** The session invite accept endpoint initially ran conflict checks before cancelling the session being rescheduled. This caused the existing sessions to register as conflicts with the new `now+5min` session.

**Learning:** In any reschedule flow, cancel the old session BEFORE running conflict checks on the new time. Otherwise the old session's time range may overlap with the new session and trigger a false 409 conflict.

## 5. Positional Module Assignment Works Naturally with Instant Booking
**Topics:** d1

**Context:** Concerned that booking a session at `now+5min` might break the positional module assignment model (which sorts sessions by `scheduled_start ASC` to assign modules).

**Learning:** The positional model handles instant booking seamlessly — a session at `now+5min` just becomes the earliest session chronologically, and module assignment reflows correctly. No changes to `booking.ts` were needed. The overbooking guard (`bookable_count`) and conflict checks also work unchanged.

## 6. Join Window Opens Automatically for Instant Sessions
**Topics:** video

**Context:** The session join endpoint opens a join window 15 minutes before `scheduled_start`.

**Learning:** With `scheduled_start = now + 5min`, the join window starts at `now - 10min` — already in the past. Both parties can join immediately after session creation without any changes to the join endpoint logic.

## 7. Every Multi-Step Block Needs a Cleanup Phase
**Topics:** workflow

**Context:** After completing the SESSION-INVITE block and verifying all tests pass, discovered RFC checklist items were still unchecked and CURRENT-BLOCK-PLAN.md still had the completed block's data. Required a separate cleanup pass.

**Learning:** Cleanup is easiest immediately after verification while context is fresh. Every PLAN block should include an explicit Cleanup phase as the final step — check off RFC items, reset CURRENT-BLOCK-PLAN.md to template, remove temporary scaffolding, verify no stale references. Deferring cleanup creates ambiguity in future convs.
