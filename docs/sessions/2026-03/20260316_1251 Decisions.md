# Session Decisions - 2026-03-16

## 1. cert_id vs teacher_id for Certification Record IDs
**Type:** Implementation
**Topics:** d1, testing

**Trigger:** The `st_id` SQL alias appeared in two different contexts: SSR loaders (where it was `teacher_certifications.id` — a certification record PK) and enrollment APIs (where it was `users.id` — the teacher's user ID). Both needed renaming but to different targets.

**Options Considered:**
1. Rename both to `teacher_id` — simpler but ambiguous (is it user ID or cert ID?)
2. Rename SSR loader version to `cert_id`, enrollment API version to `teacher_id` ← Chosen
3. Rename SSR loader version to `teacher_cert_id` — verbose

**Decision:** Use `cert_id` for teacher certification record PKs in SSR loaders, `teacher_id` for teacher user IDs in enrollment/earnings APIs.

**Rationale:** The names should reflect what the value actually represents. In SSR loaders, the ID is the certification record's primary key (used as the TeacherProfileData.id). In enrollment APIs, it's the teacher's user ID. Using distinct names prevents confusion about which entity the ID references.

**Consequences:** Two different rename targets for `st_id` depending on context. SSR loaders and teacher/[handle] astro page use `cert_id`; enrollment, earnings, and full.ts APIs use `teacher_id`.

## 2. PostToolUse Hook Over Review Skill for Check Output Tracking
**Type:** Strategic
**Topics:** cc-workflow

**Trigger:** User observed that Claude Code rarely creates TodoWrite items proactively during work — issues found by checks were dismissed as "pre-existing" rather than tracked. Asked whether a periodic review skill would help.

**Options Considered:**
1. Create a review skill that re-evaluates recent interactions for missed TodoWrite items
2. Create a PostToolUse hook that fires after check commands and injects a reminder ← Chosen
3. Rely on feedback memories alone (status quo, proven insufficient)

**Decision:** Build a PostToolUse hook (`check-output-reminder.sh`) registered in `.claude/settings.json` with a Bash matcher. Two-layer design: specific check command parsers + general failure catch-all.

**Rationale:** A hook fires automatically at the exact moment of discovery (zero friction), while a skill requires manual invocation (reactive). The behavioral problem is attention narrowing during complex work — intervention must happen at the right moment, not after the fact.

**Consequences:** Hook registered in project settings, fires on every Bash tool call but exits immediately for non-check commands. Outputs a reminder only when issues are detected. Takes effect from next session.

## 3. Widen Hook to Catch All Non-Zero Exit Codes
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** User pointed out that failures from commands like `npm install` or `git push` also go untracked, not just check commands.

**Options Considered:**
1. Keep hook scoped to check commands only
2. Add a general catch-all for any non-zero exit code ← Chosen

**Decision:** Layer 2 of the hook catches any Bash command with non-zero exit code that isn't already handled by the specific check parsers.

**Rationale:** A failed `npm install` or `npm run build` is just as actionable as a failed `npm test`. The softer wording ("Diagnose the failure. If it reveals an issue, TodoWrite it") accounts for cases where a non-zero exit is expected or just needs a retry.

**Consequences:** Every Bash failure now surfaces a reminder. Low risk of noise since successful commands produce no output.
