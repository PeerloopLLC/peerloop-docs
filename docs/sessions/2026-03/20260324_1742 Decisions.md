# Conv Decisions - 2026-03-24

## 1. INVITE-UX: Component-Local Polling Over CurrentUser State
**Type:** Implementation
**Topics:** astro, auth

**Trigger:** INVITE-UX needed to show real-time invite status on MyStudents. Three approaches considered.

**Options Considered:**
1. Add `pendingInvites` to CurrentUser — reactive via `useCurrentUser()`, no polling needed. Expands CurrentUser scope with ephemeral data.
2. Component-local polling + version bump — MyStudents polls invite status every 10s; acceptance triggers `bumpUserDataVersion` for notification badge refresh. ← Chosen
3. Full global notification listener — reduce notification poll interval when pending invite exists, plus banner/modal outside MyStudents.

**Decision:** Option 2 — component-local invite status polling in MyStudents, plus `bumpUserDataVersion` in accept.ts for the notification badge signal.

**Rationale:** Invite status is ephemeral page-local data that doesn't belong in global state. CurrentUser should carry durable identity/capability data. The version bump leverages the existing 30-second poll infrastructure without new mechanisms.

**Consequences:** MyStudents has its own 10-second poll loop (auto-stops on acceptance/expiry). The notification badge updates within 30s of acceptance. No CurrentUser schema change needed.

## 2. DATETIME-FIX: strftime Over datetime for ISO String Comparisons
**Type:** Implementation
**Topics:** d1

**Trigger:** Test failure during INVITE-UX revealed that `datetime('now', '-5 minutes')` produces space-separated format that compares incorrectly against ISO timestamps stored with T separator.

**Options Considered:**
1. Convert stored values to space-separated format at write time — breaks ISO convention, affects 130+ files migrated in Conv 011.
2. Replace `datetime()` with `strftime('%Y-%m-%dT%H:%M:%fZ', ...)` in comparisons — matches stored format, mechanical fix. ← Chosen
3. Use `julianday()` for all comparisons — correct but less readable and harder to debug.

**Decision:** Replace all `datetime('now', ...)` comparison uses with `strftime('%Y-%m-%dT%H:%M:%fZ', 'now', ...)`. Leave `date('now', ...)` as-is (safe — 10-char prefix sorts correctly).

**Rationale:** The strftime approach produces the same T-separator format as stored values, making lexicographic comparison correct. It's a mechanical find-and-replace with no semantic change. The Conv 011 migration fixed writes but missed read-side comparisons — this completes it.

**Consequences:** Fixed 6 broken queries across 5 files. Added caveat to DB-GUIDE.md and DECISIONS.md addendum.

## 3. RECORDING-R2: Partial Implementation With Vendor Inquiry
**Type:** Strategic
**Topics:** video, cloudflare

**Trigger:** RECORDING-R2 aimed to replicate BBB recordings to R2. Research revealed BBB's default format is an HTML playback page, not a downloadable video file.

**Options Considered:**
1. Defer entirely — wait for vendor confirmation before writing any code.
2. Partial implementation — wire up the code path, fix the parser, add schema column, make it activate when a download URL becomes available. ← Chosen
3. Attempt to scrape/download from the playback HTML page — fragile and vendor-dependent.

**Decision:** Implement the full R2 replication pipeline (parser fix, schema, helper, webhook wiring) but have it activate only when a `downloadUrl` is available in the recording data. Create a separate action item for the user to contact Blindside Networks about enabling the video format.

**Rationale:** The code is ready for the moment the video format is enabled — zero additional dev work needed. The parser fix (handling multiple BBB formats) was valuable regardless. No wasted effort since the schema column and helper are lightweight.

**Consequences:** `recording_r2_key` column exists but will be NULL until BBB video format is enabled. `replicateRecordingToR2()` exists but won't fire without a download URL. User needs to contact Blindside Networks.
