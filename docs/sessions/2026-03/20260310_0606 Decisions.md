# Session Decisions - 2026-03-10

## 1. Adopt Skills 2 Merged Single-File Pattern
**Type:** Strategic
**Topics:** cc-workflow, docs-infra

**Trigger:** Converting `/q-docs` from old command format to Skills 2. Initially created a two-file approach (SKILL.md + project.md) mirroring the old global/local split. Realized the file boundary was solving a deployment problem that no longer exists.

**Options Considered:**
1. Two files: SKILL.md (portable) + project.md (config) — preserves old separation
2. Single SKILL.md with all content merged ← Chosen
3. Symlinked SKILL.md + local project.md

**Decision:** Merge global and local content into a single SKILL.md per project. The canonical template in `skills-canon` uses `<!-- PROJECT -->` comment markers to indicate where project-specific content goes.

**Rationale:** Eliminates the handoff reliability problem ("REQUIRED: read project.md"). Claude processes one linear document without branching. The old file boundary was solving a deployment concern (separate repos) that's irrelevant now that both are in the same skill directory. Section-level drift reporting handles portability concerns adequately.

**Consequences:** The SKILL.md for Peerloop is 200 lines (vs 625 for the old two-file system). The `project.md` file was deleted. The canonical template has placeholder comments for customization points.

## 2. Use Real Copies with Drift Tools, Not Symlinks
**Type:** Strategic
**Topics:** cc-workflow, docs-infra

**Trigger:** Designing how canonical skill templates relate to project-specific copies. Symlinks would prevent drift but also prevent project-specific evolution.

**Options Considered:**
1. Symlinks to canonical repo (prevents drift)
2. Real copies with drift detection tools ← Chosen
3. Git subtree embedding

**Decision:** Each project gets real copies of skill files. A `skills-canon` repo holds canonical templates. `drift-report.sh` detects divergence, `skill-patch.sh` enables surgical sync.

**Rationale:** Drift is inevitable — projects evolve at different rates and need project-specific modifications. The system should make drift visible and manageable, not prevent it. Section-level diff reporting makes triage actionable.

**Consequences:** Created `~/skills-canon/` repo with tools (`drift-report.sh`, `skill-patch.sh`, `install.sh`). Each project's skill directory includes a `.sync` metadata file tracking last reconciliation.

## 3. Use `!` Backtick Injection for Deterministic Skill Execution
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** The old `/q-docs` relied on Claude running `find`, `grep`, and comparison logic at runtime — non-deterministic and expensive in tool calls.

**Options Considered:**
1. Keep instructions for Claude to run commands at runtime (status quo)
2. Pre-compute data via `!` backtick injection with helper scripts ← Chosen

**Decision:** Write helper scripts (`detect-changes.sh`, `sync-gaps.sh`, `tech-doc-sweep.sh`, `dev-env-scan.sh`) that run at skill invocation time. Their output is injected into the prompt via `!` backtick syntax. Claude sees data, not instructions to gather data.

**Rationale:** Pre-computed data in the prompt is cheaper than runtime tool calls. Scripts do mechanical comparison work (diffing file lists, matching patterns) deterministically. Claude's role shifts to judgment and writing — what it's good at.

**Consequences:** Instructions shrank 68% (629 → 200 lines). Total prompt grew (adds ~1000 lines of injected data) but eliminates dozens of runtime tool calls. Scripts live in `${CLAUDE_SKILL_DIR}/scripts/` and are shared via `route-mapping.txt` data file.

## 4. Create skills-canon Repository for Skill Template Management
**Type:** Strategic
**Topics:** cc-workflow, docs-infra

**Trigger:** Need to backup, version, and share skill templates across projects while allowing per-project customization.

**Options Considered:**
1. Global `~/.claude/commands/` only (status quo)
2. Monorepo with project overlays
3. Standalone canonical repo with drift tools ← Chosen

**Decision:** Created `~/skills-canon/` as a git repo containing canonical skill templates and management tools.

**Rationale:** Centralized versioning without enforcing uniformity. Tools enable bidirectional sync — innovations flow from any project to canonical and back. `CUSTOMIZATION-GUIDE.md` documents what to change per project.

**Consequences:** Repo structure: `q-docs/SKILL.md` (template), `q-docs/CUSTOMIZATION-GUIDE.md`, `tools/drift-report.sh`, `tools/skill-patch.sh`, `tools/install.sh`. Ready for GitHub backup.

## 5. Extract Shared Configuration to Data Files
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** API route mapping table was duplicated in SKILL.md (for Claude to read) and sync-gaps.sh (for script logic). Changes to one required updating both.

**Options Considered:**
1. Keep duplicated in both files
2. Extract to shared data file read by both ← Chosen

**Decision:** Created `route-mapping.txt` as a simple `prefix|doc_file` format read by both SKILL.md (as a reference) and sync-gaps.sh (for programmatic matching).

**Rationale:** Single source of truth, two consumers. The data file format is simple enough for both humans and scripts to parse.

**Consequences:** SKILL.md notes the authoritative source is `${CLAUDE_SKILL_DIR}/scripts/route-mapping.txt`. sync-gaps.sh reads it instead of a hardcoded case statement.
