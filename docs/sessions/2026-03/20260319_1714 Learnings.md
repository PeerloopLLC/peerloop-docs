# Conv Learnings - 2026-03-19

## 1. D1 Blocks sqlite_version() But VALUES Syntax Works
**Topics:** d1, cloudflare

**Context:** FEED-INTEL badge query uses `(col1, col2) IN (VALUES (?, ?), ...)` syntax requiring SQLite 3.38+. Unit tests pass on better-sqlite3 but D1 might use an older version. Cloudflare blocks `SELECT sqlite_version()` as a disallowed function.

**Learning:** Test the actual syntax directly rather than checking the version number. Running `SELECT 1 WHERE ('a', 'b') IN (VALUES ('a', 'b'), ('c', 'd'))` on D1 staging returned successfully, proving the syntax is supported without needing to know the exact version.

**Pattern:** When a platform blocks introspection functions, test the capability empirically with a minimal query instead.

## 2. Integration Tests Catch Schema Mismatches That Unit Tests Miss
**Topics:** testing, d1

**Context:** The `GET /api/me/feed-badges` endpoint queried `enrollments.user_id` — but the column is `student_id`. Unit tests for `getFeedBadgeCounts()` passed because they test the lib function (which receives a pre-built feed list), not the SQL that constructs the feed list from user relationships.

**Learning:** HTTP-level integration tests exercise the full query chain — auth → relationship queries → badge calculation → response. The bug lived in the relationship query layer, invisible to unit tests that only test the badge calculation layer.

## 3. Comment Endpoints Had Slug Context All Along
**Topics:** stream, astro

**Context:** Conv 015 skipped dual-write in comment endpoints because "comment endpoints lack feed slug context." Conv 016 discovered the slug IS available: community comments live at `/api/feeds/community/[slug]/comments` (slug in URL params), townhall comments are always `townhall:the-commons`.

**Learning:** Before deferring work based on an assumption about what data is available, check the actual route parameters. Astro's `[slug]` dynamic routes make the slug available in `params.slug` on every handler in that directory.

## 4. Stream Reactions Are Global — Enabling Pseudo-Feed Interactions
**Topics:** stream

**Context:** Exploring whether a "smart feed" (ranked, personalized view of posts from multiple feeds) could support interactions (comments, likes) that flow back to the originating feeds.

**Learning:** Stream.io reactions (comments, likes, replies) attach to activity IDs globally, not to feeds. A post from `community:python-devs` displayed on any surface can receive reactions that automatically appear everywhere that activity is shown. No "routing back" is needed — reactions were never feed-local in the first place. This is the key architectural property that makes the SMART-FEED concept viable.

## 5. Activity Metadata Enables Self-Routing Components
**Topics:** stream, astro

**Context:** FeedActivityCard defaulted `feedApiBasePath` to `/api/feeds/townhall`, causing course feed and home timeline comments/reactions to route to the wrong endpoint (or 404).

**Learning:** Stream activities carry source metadata set at creation time (`communitySlug`, `courseSlug`). By reading these fields, `FeedActivityCard` can derive the correct API base path without any parent component passing it. This makes the component work correctly on any surface — dedicated feed pages, home timeline, and future surfaces like the smart feed — without each parent needing to know the routing rules.

## 6. Architecture Doc Naming Convention Drift
**Topics:** docs-infra

**Context:** Session 362 decided to drop `tech-NNN:` numbering prefixes from architecture docs. But 7 files still had them in their H1 headings — the filenames were renamed but the headings weren't updated.

**Learning:** When a rename convention is established, grep for ALL instances — filenames, H1 headings, references in other docs. A partial migration leaves confusing inconsistency (new docs follow the convention, old ones don't).
