## Development Transcript

### Per-Tab localStorage Isolation Investigation

**User:** "We currently use localstorage for the global vars including currentUser. I would like to make a new PLAN block LOCALSTORAGE where we allow the same browser to have multiple tab sessions of this app and each having its own localStogae entries. That woud allow me to test much more quickly (not havng to log off then on to test two sides of a user transaction and also allow me to test BBB sessions"

**Claude:** Launched an explore agent to audit all localStorage usage across the codebase. Found 7 files using 5 localStorage keys (user cache, was-logged-in flag, Stripe pending sessions, theme, recommendation dismissals). Identified the core constraint: auth uses HTTP-only cookies (`peerloop_access`, `peerloop_refresh`) shared across all tabs — scoping localStorage alone won't give independent sessions. Presented three approaches with tradeoffs: (A) tab-scoped auth tokens via sessionStorage + Authorization header, (B) dev-mode impersonation with `?_as=userId`, (C) hybrid. Recommended Option B as fastest path.

**User:** "I do not understand. Dev mode is a good idea to restrict this (the existing code works well with localSTorage as is). But I need to be AlexChen in one tabe and Sarah Miles in another. Is that accomplished properly or is this a hacky way of achieveng the goals?"

**Claude:** Clarified that impersonation is the standard pattern used by admin tools (Stripe Dashboard, Shopify, etc.). The server fully loads the impersonated user's record — SSR rendering, API responses, BBB join URLs all use the correct identity. It's a single middleware check, not scattered patches, and production builds ignore it entirely via `import.meta.env.DEV` guard. localStorage scoping becomes trivial since each tab's sessionStorage holds a different `_as` value used as the key prefix.

**User:** "Not yet. Would I obviate the need for any of this by using two different vendor browsers?"

**Claude:** Confirmed that two browsers (Chrome + Safari) gives completely independent cookies, localStorage, SSR rendering, and BBB sessions — zero code changes needed. Only limitation: 3+ simultaneous users needs incognito or additional browsers.

**User:** "Yes. I do not want to burder the productin codebase with testing artefacts, so 2 browser vendor is a good solution for now"

**Claude:** Saved feedback memory about no testing artifacts in production code. No PLAN block needed.
