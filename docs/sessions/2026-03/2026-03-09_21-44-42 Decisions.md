# Session Decisions - 2026-03-09

## 1. Structural Guard Over Build Version Key for Cache Protection
**Type:** Implementation
**Topics:** astro, auth

**Trigger:** AppNavbar disappeared after staging deploy due to stale localStorage cache with incompatible `MeFullResponse` shape.

**Options Considered:**
1. Build version key — invalidate cache on every deploy via `Date.now()` injected at build time
2. Full schema validation — validate every field of cached data before use
3. Structural guard — check only the minimum fields the constructor needs ← Chosen

**Decision:** Added `isValidCachedData()` type guard to `loadFromCache()` in `current-user.ts`. Checks 6 critical fields: `user.id`, `user.name`, `user.handle`, `enrollments` (array), `teacherCertifications` (array), `createdCourses` (array).

**Rationale:** Build version key penalizes all users on every deploy (loading flash even when API shape unchanged). Schema validation creates maintenance burden (every new field needs a guard update). Structural guard is self-healing, zero-config, and only triggers on actual structural mismatch. The API refresh already handles semantic staleness within milliseconds.

**Consequences:** Added `isValidCachedData()` to `current-user.ts`, 17 new tests in `tests/lib/current-user-cache.test.ts`, cache structural guard section merged into `docs/architecture/state-management.md`.

## 2. Split docs/tech/ Into docs/vendors/ and docs/architecture/
**Type:** Strategic
**Topics:** docs-infra

**Trigger:** Cognitive dissonance when deciding where new docs should go — `docs/tech/` contained both vendor evaluations and internal architecture docs.

**Options Considered:**
1. Rename folders to match actual content (vendors/ + architecture/) ← Chosen
2. Keep tech/ but add architecture/ alongside
3. Keep everything in tech/ with prefixes (vendor-*, arch-*)
4. Do nothing

**Decision:** Split `docs/tech/` into `docs/vendors/` (19 files — external services) and `docs/architecture/` (13 files — internal design). Dropped `tech-0XX-` numbering, using descriptive names (e.g., `stripe.md`, `state-management.md`).

**Rationale:** Option A is the cleanest long-term. The folder name now answers "where does my new doc go?" unambiguously. Numbering provided no value — files are found by content, not sequence.

**Consequences:** 42 files moved, ~200 references updated across ~65 files. `docs/DOCS-REORG-MAP.md` created as Rosetta Stone for historical session log references.

## 4. Session Logs Immutable — Use Mapping Document Instead
**Type:** Strategic
**Topics:** docs-infra

**Trigger:** ~160 session log files contain old `docs/tech/` paths. Should they be updated?

**Options Considered:**
1. Update all session logs (like the terminology block did)
2. Leave session logs unchanged, create mapping document ← Chosen

**Decision:** Session logs are historical records — they reference the path that existed at the time. `docs/DOCS-REORG-MAP.md` provides old→new lookup for every moved file.

**Rationale:** Session logs document what happened in a specific session. Changing paths retroactively makes them less accurate as historical records. The mapping document costs almost nothing to create and serves the same navigational purpose.

**Consequences:** Created `docs/DOCS-REORG-MAP.md` with 41 entries (32 tech files + 9 root files).

## 5. Merge CACHE.md Into state-management.md
**Type:** Implementation
**Topics:** docs-infra

**Trigger:** Initially created standalone `docs/tech/CACHE.md` for cache documentation. Then discovered `tech-020-state-management.md` already covered localStorage keys, stale-while-revalidate pattern, and `was_logged_in` behavior.

**Options Considered:**
1. Keep CACHE.md as standalone reference with cross-reference from state-management.md

**Decision:** Merged the structural guard section, design decisions (why not build version key, why not schema validation, why no TTL), and maintenance notes into `docs/architecture/state-management.md`. Deleted CACHE.md.

**Consequences:** `docs/architecture/state-management.md` now has a "Cache Structural Guard" section at the end.
