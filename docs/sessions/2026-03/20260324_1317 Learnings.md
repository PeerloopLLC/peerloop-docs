# Conv Learnings - 2026-03-24

## 1. Missing Status Filters Cause Silent Data Loss in UI
**Topics:** astro, testing

**Context:** Sessions created via instant booking (session invites) weren't appearing on the `/course/{slug}/sessions` page. The API returned the data correctly (200, 0.6kB response), but the UI showed nothing.

**Learning:** The `renderSessionsTab()` function in CourseTabs filtered sessions into `scheduled`, `completed`, and `cancelled` buckets — but the schema defines 5 statuses (`in_progress` and `no_show` were unhandled). Sessions in unhandled statuses silently disappeared from the UI while the API returned them correctly. The root cause was a UI rendering gap, not a query bug.

**Pattern:** When a schema defines an enum of valid values (like session status), every UI component that branches on those values must handle ALL of them. Add a catch-all or log unrecognized values rather than silently dropping them.

## 2. BBB "End Meeting" Is Hidden From Students
**Topics:** video

**Context:** During live testing, both participants clicked "Leave" in BBB but nobody clicked "End Meeting". The session stayed `in_progress` forever.

**Learning:** BBB only shows the "End Meeting" option to moderators (teachers) — and even then, it's buried in a three-dot menu (⋯) in the top-right corner. Students only see "Leave". This means the `meeting-ended` webhook will never fire if the teacher doesn't know to use the hidden menu. The fix required multiple layers: empty-room detection in the webhook handler, UX guidance banners, and client-side fallback completion.

## 3. Prop-Based State vs DB State Creates Race Conditions
**Topics:** astro, testing

**Context:** After completing a stuck session, the SessionRoom view flickered between "completed" and "in_session" every 15 seconds.

**Learning:** SessionRoom has two effects: a 1-second timer calling `getInitialState()` (reads the static `session.status` prop from server render) and a 15-second poller (reads live DB status). When the poller detected `completed`, the timer immediately overwrote it using the stale prop. Fix: guard the timer against overriding `completed` state. More generally, any React component with both prop-derived state and polling-derived state needs explicit guards to prevent the stale prop from overriding the live data.

## 4. Silent Error Swallowing Masks Real Bugs
**Topics:** astro, testing

**Context:** CourseTabs `fetchCourseSessions` had a `catch { /* Silently fail */ }` block and an `if (res.ok)` gate that ignored non-200 responses. When debugging the sessions bug, this made it appear like no sessions existed rather than surfacing the actual error.

**Learning:** Silent error handling in fetch calls makes debugging much harder. At minimum, log non-200 responses with the status code and endpoint URL. The "silently fail — tab will show empty state" pattern is appropriate for non-critical data, but critical data fetches (like session lists) should surface errors.

## 5. Auth Decisions Require Abuse-Case Thinking
**Topics:** auth

**Context:** Initially allowed students to complete stuck sessions via the "Complete Session Now" button. The user caught that this would let a student who no-showed mark the session as `completed` (instead of `no_show`), getting module credit they didn't earn.

**Learning:** When expanding auth permissions on an endpoint, always think through the abuse cases — especially when the action triggers side effects (module_id freezing, progress tracking). The principle: the person who has authority over the activity should be the one who can confirm it. For sessions, that's the teacher.

## 6. Empty Room Detection Needs Minimum Participant Threshold
**Topics:** video

**Context:** The empty-room auto-complete logic could theoretically fire when only one participant joined and then left (teacher opens room, student never shows).

**Learning:** Requiring at least 2 distinct users in attendance records before auto-completing prevents false positives. If only one person showed, the session should be handled by the no-show detection system instead, which correctly marks it as `no_show` rather than `completed`.
