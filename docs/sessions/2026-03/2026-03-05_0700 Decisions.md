# Session Decisions - 2026-03-05

## 1. Use module_id Nullability (Not Status) for Frozen/Computable Split
**Type:** Implementation
**Topics:** d1, video

**Trigger:** BBB webhook tests for module_id-on-completion failed — `resolveModuleAssignments()` treated `in_progress` sessions as frozen, but they don't have `module_id` set yet.

**Options Considered:**
1. Split by status (`completed`/`in_progress` = frozen, `scheduled` = computable)
2. Split by `module_id != null` (frozen) vs `module_id == null` (computable) ← Chosen
3. Add special-case handling for `in_progress` sessions

**Decision:** Sessions are classified as "frozen" when they have a non-null `module_id`, regardless of status. All other sessions are positionally computed.

**Rationale:** Option 2 is the simplest and most correct — `module_id` is the actual indicator of whether a module assignment has been permanently recorded. Status is a lifecycle field that doesn't reliably correlate with when `module_id` gets written (the webhook writes it as part of the completion transition).

**Consequences:** Changed one filter in `src/lib/booking.ts`. All 27 booking + webhook tests pass. Updated DECISIONS.md with the refinement.

## 2. Out-of-Order Completion: Mark Completed but Skip module_id
**Type:** Implementation
**Topics:** video, d1

**Trigger:** Block plan required "validate sequential completion" in the BBB webhook, but webhooks fire for irreversible events.

**Options Considered:**
1. Reject the webhook entirely (don't mark completed) — breaks reality
2. Mark completed AND write module_id anyway — breaks positional invariant
3. Mark completed but skip module_id write (NULL) ← Chosen
4. Queue for later resolution

**Decision:** When a session completes out of order (earlier sessions still `scheduled`), mark it `completed` with `module_id = NULL`. Log a warning.

**Rationale:** The BBB meeting already ended — we can't pretend it didn't. But writing a potentially incorrect `module_id` would break the positional algorithm's invariant that frozen module_ids are sequential. Leaving it NULL preserves data integrity and the module can be resolved later when earlier sessions catch up.

**Consequences:** Added sequential check query in `handleRoomEnded`. 4 new tests cover in-order, second-session, out-of-order, and no-curriculum scenarios.

## 3. Booking Eligibility via Server-Side Prop (Not Client API)
**Type:** Implementation
**Topics:** astro, video

**Trigger:** Booking wizard needed module context (next module, remaining count, can-book flag) before the user starts the flow.

**Options Considered:**
1. Create a new `GET /api/enrollments/:id/booking-eligibility` endpoint ← Not chosen
2. Fetch server-side in `book.astro` and pass as prop ← Chosen
3. Fetch client-side on mount in the React component

**Decision:** Call `getBookingEligibility()` in the Astro page's frontmatter and pass the result as a `moduleInfo` prop to `SessionBooking`.

**Rationale:** The page already has DB access and enrollment context from its auth checks. Adding one more query costs nothing. The data is available immediately on hydration — no loading spinner, no extra round-trip. Creating a new API endpoint would add surface area for a single consumer.

**Consequences:** `book.astro` imports `getBookingEligibility`, passes prop. `SessionBooking.tsx` gains `ModuleInfo` type, module banner, and fully-booked guard. Gracefully degrades when `moduleInfo` is null (no curriculum).
