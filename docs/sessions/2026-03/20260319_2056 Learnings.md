# Conv Learnings - 2026-03-19

## 1. Astro Dev Server Concurrency Limits with Stream API Calls
**Topics:** astro, testing, stream

**Context:** Smart feed E2E tests make real Stream API calls during enrichment. Running 5 Playwright workers in parallel caused the dev server to hang at 99.7% CPU — requests would go out but never return.

**Learning:** Astro's local dev server (single-threaded Node.js) can't handle concurrent smart feed requests because each one does D1 queries + a Stream API call. When the browser fires 5+ requests during hydration (`/api/me/full`, notifications, messages, AND `/api/feeds/smart`), the server gets blocked. The same API call works fine via `curl` in isolation (218ms).

**Pattern:** For E2E tests that hit external APIs through the dev server, use `test.describe.configure({ mode: 'serial' })` and `--workers=1` to prevent concurrent requests.

## 2. Content-Based Waits Beat API Response Interception in Astro Dev Mode
**Topics:** testing, astro

**Context:** Initial E2E approach used `page.waitForResponse()` to catch the smart feed API response. This failed because Astro's `client:load` components fire API calls during hydration — the response arrives before the Playwright listener is attached.

**Learning:** In Astro dev mode with React islands (`client:load`), API calls fire during component hydration, which overlaps with `page.goto()` resolution. Using `page.waitForResponse()` in `Promise.all` with `page.goto()` worked for some users but not others (dev server concurrency). The most reliable approach: wait for visible content with `expect(locator).toBeVisible({ timeout })`.

**Pattern:**
```typescript
// Reliable: wait for content
await page.goto('/feed', { waitUntil: 'domcontentloaded' });
await expect(page.getByText(/known-content/).first()).toBeVisible({ timeout: 30000 });

// Unreliable in Astro dev: waitForResponse
const resp = page.waitForResponse(r => r.url().includes('/api/...'));
await page.goto('/feed');
await resp; // May miss the response or hang
```

## 3. Stream.io foreign_id for Seed Data Management
**Topics:** stream

**Context:** The feed seed script creates activities in Stream.io via the REST API. Re-running the script creates new activities each time.

**Learning:** Stream's `foreign_id` field provides deduplication identity. Each seed activity gets a deterministic `foreign_id` like `seed-community-ai-for-you-3`. Stream allows multiple activities with the same `foreign_id` if they have different `time` values, so re-running creates duplicates rather than updates. The `--clean` flag clears D1 first, and new Stream activities are created with fresh IDs. Old Stream activities become orphaned but don't affect functionality.

## 4. Discovery Card Detection Needs Timeout in Playwright
**Topics:** testing

**Context:** E2E tests checking for discovery cards used `isVisible()` without a timeout, which returned `false` even though cards existed — they hadn't rendered yet when checked.

**Learning:** `locator.isVisible()` is an instant check (no waiting). For elements that render asynchronously (like discovery cards interleaved with member posts after a React state update), use `expect(locator).toBeVisible({ timeout: 5000 })` in a try/catch to allow rendering time before deciding the element doesn't exist.
