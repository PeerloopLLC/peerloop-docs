# Conv Learnings - 2026-03-20

## 1. Astro Hydration Mismatch with useCurrentUser()
**Topics:** astro, auth

**Context:** `/feeds` and `/learning` pages showed hydration mismatch errors in the console. Pages loaded but React regenerated the tree from scratch.

**Learning:** Any `client:load` component that branches on `useCurrentUser()` (or any `window`-dependent state) in its first render will mismatch. Server renders loading skeleton (no `window.__peerloop`), client has cached user immediately and renders the populated/empty state. Use `client:only="react"` for components that depend on browser-only state.

**Pattern:** Components that start in a uniform loading state (spinner/skeleton) regardless of environment can safely use `client:load`. Components that immediately consume `window.__peerloop` must use `client:only="react"`.

## 2. Infinite Loop in Smart Feed Discovery Interleaving
**Topics:** stream, testing

**Context:** David Rodriguez's `/feed` page hung permanently — no error, no timeout, had to kill the terminal. Debug logging showed all 5 pipeline steps completed, so the hang was in the synchronous `interleaveDiscovery()` function.

**Learning:** The interleave algorithm had a latent infinite loop: when member posts are exhausted AND the discovery cap is reached, remaining unused discovery cards keep the outer `while` condition true but neither index advances. Only triggers when discovery candidates outnumber member posts — which is the exact scenario for new/sparse users (the users who need discovery most).

**Pattern:** Any interleaving loop with two independent cursors and a cap on one of them needs a "no progress" break condition. Added: `if (memberIdx >= memberPosts.length && discoveryInserted >= maxDiscovery) break;`

## 3. Stream Feed ID vs D1 Feed ID Mismatch (Townhall)
**Topics:** stream, d1

**Context:** Guy's Smart Feed showed no townhall posts despite 2 existing in D1. The `getUserFeedList()` returns `feedId: 'the-commons'` (community slug) but the seed script stored `feed_id: 'main'` (Stream routing address).

**Learning:** Stream uses `feedGroup:feedId` as its routing address (e.g., `townhall:main`), but the D1 `feed_activities` table uses the community/course slug as `feed_id` for JOIN compatibility with the smart feed candidate query. These are different namespaces. The real townhall endpoint already used `'the-commons'` for D1 — only the seed script was wrong.

**Pattern:** When dual-writing to Stream + D1, the Stream feed address and D1 feed_id are NOT the same value. D1 uses slugs; Stream uses its own feed IDs.

## 4. Primary and Blue Are Visually Indistinguishable
**Topics:** astro

**Context:** Designed a 3-color card system (townhall=primary, community=secondary, course=blue). All cards looked identical because the project's custom `primary` is sky-blue (`#0ea5e9`) and Tailwind's `blue` at the 50/200/400 levels is nearly identical (`#3b82f6`).

**Learning:** When the project's primary color IS a blue variant, you can't use standard Tailwind `blue-*` as a contrasting color. Switched courses to `indigo-*` which provides clear visual distinction from sky-blue primary.

## 5. Stream Activities Are Immutable for Field Additions
**Topics:** stream

**Context:** Added `userHandle`, `communityIcon`, `courseThumbnailUrl`, `isSystemCommunity`, `activityType` to Stream activities. Existing posts don't have these fields.

**Learning:** Stream activities store whatever fields are passed at creation time. You can't backfill fields on existing activities. Components must gracefully degrade when new optional fields are missing — use the same detection logic that falls back to slug-based inference for older posts.

## 6. No Timeout on Stream API Fetch Causes Unrecoverable Hangs
**Topics:** stream, cloudflare

**Context:** When investigating David's feed hang, discovered that `streamFetch()` had no timeout. A hanging Stream response would hang the entire Cloudflare Worker request indefinitely — no error, no fallback, not even Ctrl+C could stop it in dev.

**Learning:** Always add `AbortController` with a timeout to external API calls, especially in edge/worker environments where there's no global request timeout. Added 10s timeout to `streamFetch()`. The existing try/catch in enrichment catches the abort error and falls back to D1-only data.

---

## 7. Seed Script Needs Separate Stream and D1 Feed IDs
**Topics:** testing, stream

**Context:** The `seed-feeds.mjs` script used a single `feedId` field for both Stream API calls and D1 inserts. Townhall's Stream address is `main` but D1 uses `the-commons`.

**Learning:** Added `streamFeedId` override field to the seed post data. The script uses `post.streamFeedId || post.feedId` for Stream API calls and always uses `post.feedId` for D1 inserts.
