# Conv Learnings - 2026-03-24

## 1. BBB "Leave" vs "End Meeting" Are Completely Different Operations
**Topics:** video

**Context:** User ran a live session between teacher (Sarah) and student (Alex). After the session, the teacher ended up on a BBB Learning Analytics Dashboard instead of back in Peerloop.

**Learning:** BBB has two distinct exit paths with very different consequences:
- **"Leave"** (all participants) — fires `participant_left` webhook, updates attendance, but session stays `in_progress`. The BBB room keeps running empty.
- **"End Meeting"** (moderator/teacher only) — fires `room_ended` webhook → `completeSession()` runs → status becomes `completed`.

If both participants click "Leave" but nobody clicks "End Meeting", the session is stuck `in_progress` forever with no auto-recovery. This is a critical UX gap — teachers need clear guidance to use "End Meeting", and the system needs a fallback for empty rooms.

## 2. Teacher Gets No Immediate Guidance After Sending Instant Session Invite
**Topics:** video, astro

**Context:** User tested the instant session invite flow. After clicking the bolt icon on `/teaching/students`, the button showed a checkmark for 3 seconds and reset. Teacher had to hunt to find and join the created session.

**Learning:** The invite flow has the right plumbing (teacher gets a notification when student accepts with a "Join Session" link), but the teacher UX after sending the invite is poor:
1. No confirmation modal or status indicator
2. Teacher stays on `/teaching/students` with no "what next" guidance
3. Must notice the notification bell to learn the student accepted
4. No prominent join prompt — just a regular notification item

## 3. Instant-Booking Sessions May Not Appear on Course Sessions Page
**Topics:** d1, astro

**Context:** After completing an instant-booking session, the user navigated to `/course/ai-tools-overview/sessions` and didn't see the session that was created via the invite flow.

**Learning:** The sessions query joins `sessions → enrollments` filtered by `course_id + student_id`. Invite-created sessions have the correct `enrollment_id`, so they should appear. This needs investigation — could be a status filter issue, caching, or a query bug. Tracked as SESSION-FIX.SESSIONS-BUG.

## 4. BBB Provides Learning Analytics Dashboard to Moderators
**Topics:** video

**Context:** Teacher (Sarah) was redirected to a BBB Learning Analytics Dashboard URL after the meeting ended (`/learning-analytics-dashboard/?meeting=...`).

**Learning:** BBB natively provides session analytics (attendance duration, engagement metrics) via a dashboard URL to moderators. Peerloop currently has no integration with this data — it's only accessible via the BBB redirect. BBB may have APIs to fetch this data programmatically, which would allow storing and displaying it within Peerloop's session detail views.

## 5. Students Have Rich Course UI, Teachers Have Nothing Equivalent
**Topics:** astro

**Context:** User observed that students have a 6-tab course detail page (`/course/{slug}`) with About, Teachers, Resources, Feed, Sessions, and Learn tabs. Teachers have no equivalent — just summary cards on `/teaching` and grouped sessions on `/teaching/sessions`.

**Learning:** This is a significant feature gap. Teachers need a course-specific view to manage their teaching activity per course — seeing their students, sessions, resources, and reviews for a specific course. Created as standalone TEACHER-COURSE-VIEW block (separate from SESSION-FIX).
