### FEEDS-HUB.ACTIVITY — Stream.io Limitations Discussion

**User:** "I need to understand this. Are we trying to work within the restrictions of GetSTreams API by snapshotting counts periodically to report the diffs as new? If so how are we deciding if they have seen the new? Probably seeing one removes the entire badge."

**Claude:** Explained the mechanics honestly: Stream has no unread tracking for flat feeds. Proposed approach: store `last_visited_at` per user+feed in D1, fetch small batch from Stream to count new posts, visit = clear entire badge. Acknowledged it's visit-based not per-post, and count is approximate.

**User:** "I am quite disappointed in GetSTream if I am being frandk. I don't see how I am going to surface ppsts from different feeds a member is connected to in any meaningful way. ... Let's discusss"

**Claude:** Laid out 3 options (pure Stream, pure custom, hybrid). Proposed the middle ground: Stream as durable content store, D1 as metadata index (CQRS pattern). Explained write-time indexing avoids polling, single D1 query for badge counts across all feeds, zero Stream API calls for badges.

**User:** "Yes! This sounds eminently doable! The 2 issues I see are how much data stays in D1 and what to do if it get's pruned but is part of a conversation thread. There is also a need to keep polling for new posts across all feeds from the system's point of view. Can we consider this briefly now?"

**Claude:** Addressed all 3 concerns: D1 index is ~150 bytes/row (~45KB/month at Genesis scale), 90-day TTL with weekly pruning. Pruning is safe because D1 never stores content — threads live in Stream forever. Write-time indexing avoids polling entirely for Genesis (all posts flow through our API). Client-side freshness via existing version polling.

**User:** "This idea has great merit and needs a proper plan. SO lets create a new block in PLAN around this work, with enough of our discussion to be picked up and properly planned. Then we can get back to the inciting question: using delta counts of feeds as badges"

**Claude:** Created FEED-INTEL block in PLAN.md with full schema design, SQL examples, data lifecycle, and implementation phases. Updated FEEDS-HUB.ACTIVITY to reference FEED-INTEL as approach.
