# Conv Learnings - 2026-03-31

## 1. maybeUpdateActorSession Auto-Detection Can Corrupt Actor Sessions
**Topics:** plato, testing

**Context:** The `loadEnrollments` discovery action provided `{ teacherId: creatorUserId }`. The runner's `maybeUpdateActorSession` checks `provided.teacherId` and overwrote the student actor's session with the creator's userId.

**Learning:** The auto-detection in `maybeUpdateActorSession` matches on key names (userId, studentId, creatorId, teacherId). Any discovery action that provides data under those key names will corrupt the current actor's session. Work around by using non-matching key names (e.g., `assignedTeacherUserId`).

**Pattern:** Avoid `provides` keys named exactly `userId`, `studentId`, `creatorId`, or `teacherId` in discovery actions where the provided value belongs to a different actor than the one executing the action.

## 2. Astro Handlers Read URL Params from ctx.url, Not request.url
**Topics:** plato, astro, testing

**Context:** PLATO runner was setting searchParams on the Request object, but Astro API route handlers destructure `{ url }` from the context object, not the request.

**Learning:** When building test infrastructure for Astro API routes, search parameters must be set on the APIContext's `url` property, not just on the Request URL. The runner now creates a new URL with searchParams and assigns it to `ctx.url`.

## 3. Publish Gate Is Not a Good Proxy for "Complete Data"
**Topics:** plato, testing-philosophy

**Context:** PLATO runs 1-6 only sent fields needed to pass the publish gate (title, description, thumbnail, price, modules, duration). The About tab fields (objectives, includes, prerequisites, target_audience, peerloop_features) were not tested.

**Learning:** The publish gate tests minimum validity, not user experience completeness. PLATO should model what a diligent user does, not what the minimum validation allows. The fix: organize persona fields into DB-REQUIRED (publish gate) and SITE-NECESSARY (complete experience) sections, and have runs send both.

## 4. Stripe Webhook Requires stripe-signature Header Before Mock Executes
**Topics:** plato, stripe, testing

**Context:** The Stripe webhook endpoint checks `request.headers.get('stripe-signature')` and returns 400 before ever calling the mocked `constructWebhookEvent`. Without the header, the mock is never reached.

**Learning:** When mocking webhook endpoints, the real validation code still runs before the mock intercepts. Added `headers` field to PageAction type and passed `{ 'stripe-signature': 'plato-test-signature' }` in the webhook action.

---

## 5. Enrollment Guards Require Active Teachers Before Checkout
**Topics:** plato, enrollment, business-logic

**Context:** The checkout endpoint calls `checkEnrollmentGuards` which counts active `teacher_certifications` for the course. If zero, enrollment is blocked with "no_teachers" error.

**Learning:** The creator must self-certify (or have another certified teacher) BEFORE any student can enroll. This is enforced at the checkout level, not the booking level. Run order: self-certify-creator must precede enroll-student.
