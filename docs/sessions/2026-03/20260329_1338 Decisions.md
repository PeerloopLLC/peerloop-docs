# Conv Decisions - 2026-03-29

## 1. Astro Middleware for Authentication Guards
**Type:** Strategic
**Topics:** middleware, auth, astro

**Trigger:** `/dashboard` and `/learning` had no auth guards — visitors could access directly

**Options Considered:**
1. SSR guards in individual .astro files — simple but scattered
2. SSR guards + OAuth callback redirect — covers both tasks but still per-page
3. Astro middleware with route matcher ← Chosen

**Decision:** Centralize authentication in `src/middleware.ts` with route classification (protected prefixes + exact matches).

**Rationale:** Single point of enforcement, easy to extend, no DB queries needed (JWT only). Pages retain authorization logic.

**Consequences:** Created middleware.ts, 86 tests. All member-only routes now protected.

## 2. No Onboarding Gate in Middleware
**Type:** Strategic
**Topics:** onboarding, middleware, ux

**Trigger:** Initial implementation enforced onboarding completion in middleware — user questioned whether this was appropriate

**Options Considered:**
1. Strict gate — trap users on /onboarding until complete
2. Soft nudge — only block member-only pages ← Initially chosen
3. No middleware enforcement — OAuth redirect only + component nudges ← Final choice

**Decision:** Remove onboarding enforcement from middleware entirely. Keep OAuth callback redirect for first-touch UX. Component-level nudges for pages that use interests.

**Rationale:** Onboarding is nice-to-have, not required. Users who skip get degraded experience. Middleware should only handle authentication.

**Consequences:** Simplified middleware by ~80 lines. Removed `peerloop_onboarded` cookie infrastructure.

## 3. Two-Set Route Classification (Prefix + Exact)
**Type:** Implementation
**Topics:** middleware, routing

**Trigger:** `/community` (My Communities) is member-only but `/community/[slug]` is public — prefix matching blocks both

**Options Considered:**
1. Restructure URLs (e.g., `/my/communities`) — invasive, breaks convention
2. Exact-match set for mixed-access base paths ← Chosen
3. Inline SSR guard on `/community` only — inconsistent with middleware approach

**Decision:** Split protected routes into `PROTECTED_PREFIXES` (match path + sub-paths) and `PROTECTED_EXACT` (match bare path only).

**Rationale:** Handles the "bare = my" URL convention (`/courses` = My Courses, `/community` = My Communities) without restructuring URLs.

**Consequences:** 8 exact-match paths, 7 prefix-match paths. Clean separation tested with 86 cases.
