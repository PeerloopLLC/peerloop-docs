# Session Learnings - 2026-03-09

## 1. Adjacency Matrix Has One Footer Row But Two Footer Variants Exist
**Topics:** docs-infra, astro

**Context:** When building the route grid map from the adjacency matrix, assumed Footer was a single pseudo-page. User noticed marketing pages and app pages have different footers.

**Learning:** The `Footer.astro` component renders two variants via a `variant` prop: "full" (5-column marketing footer in LandingLayout) and "compact" (single-line with Help/Privacy/Terms in AppLayout/AdminLayout). The adjacency matrix's `[Footer]` row captures only the full variant's links. The compact footer's links (M09, M14, M15) had to be manually specified since the matrix doesn't distinguish variants.

**Pattern:** When a shared component has variants that produce different navigation, each variant needs its own pseudo-page in the grid map.

## 2. Marketing Placeholder Pages Appear "Bare" Because They Lack Chrome
**Topics:** docs-infra, astro

**Context:** 21 pages flagged "no footer / no navbar" in the grid — all the marketing placeholder pages created during BROKENLINKS block (Session 317).

**Learning:** These pages are minimal "Coming Soon" placeholders that don't use the full `LandingLayout` with Header + Footer Full. They only link back to `/` via a simple `link`. This is correct for the current implementation state, but when these pages are properly built out, they should gain `PublicHeader` (N07) + `Footer Full` (N03). The grid accurately reflects current state, making it useful as a test artifact — it shows what's missing.

## 3. Chrome Stripping Makes Page-Specific Connections Visible
**Topics:** docs-infra

**Context:** The raw adjacency matrix has 84×84 cells where most connections are chrome (navbar, footer, panels). This makes every page look like it connects to everything.

**Learning:** Stripping chrome connections (foot, nav, nav:disc, nav:feed, nav:user) from page cells and delegating them to pseudo-pages dramatically clarifies each page's unique navigation. The homepage went from ~60 connections to just 3 page-specific exits (btn: P04, lnk: D01 P02). This separation between "chrome provides" and "page provides" is the key insight that makes the grid usable.

---

## 4. Group-Letter + 2-Digit Code System Is Self-Documenting
**Topics:** docs-infra

**Context:** Needed a compact code system for 90+ routes that would work in small grid cells and be immediately understandable.

**Learning:** Using a group letter (M=Marketing, C=Course, T=Teaching, etc.) as the first character means you can read any code and instantly know which sub-grid to look in. Combined with 2-digit sequential numbering, this guarantees uniqueness while keeping codes to 3 characters. Much better than arbitrary 3-letter acronyms (CDET, SBOK) which require a lookup table to decode.

---

## 5. PublicHeader Was an Undocumented Navigation Source
**Topics:** docs-infra, astro

**Context:** The adjacency matrix had rows for AppNavbar, AdminNavbar, Footer, AuthModals — but no row for the public/marketing header component.

**Learning:** `Header.tsx` (used in LandingLayout) provides navigation to Courses, How It Works, Pricing, For Creators, plus Sign In/Get Started buttons. This was never captured as a pseudo-page in the adjacency matrix. The grid map exercise surfaced this gap. Any future adjacency matrix regeneration should include a `[PublicHeader]` row.
