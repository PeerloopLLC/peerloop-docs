### Date Display Timezone Bug

**User:** "The date Selected on Step 2 (March 16) shows as March 15 on Step 3"

**Claude:** Identified as timezone bug: `new Date("2026-03-16")` parses as UTC midnight, displays as March 15 in US timezones. Found 4 instances of `new Date(selectedDate)` in SessionBooking.tsx. Added `parseLocalDate()` helper that appends `T00:00:00` for local timezone parsing. Fixed all 4 instances.

**User:** "Is this the normall way of handling this? I would have thought using UTC and displaying in local units would be more resilient?"

**Claude:** Agreed that UTC-everywhere is the standard approach. Suggested `timeZone: 'UTC'` option on `toLocaleDateString()` as a simpler alternative for date-only strings.

**User:** "Please think it through soe more, from the teaher showing availability to the student booking. They each need local relevant times but they must be saving their selections in some standard time-zone neutral date time format"

**Claude:** Traced the full timezone chain: (1) Teacher availability stored with `timezone` column in teacher-local times — correct. (2) Availability API generates slots using server's local timezone via `setHours()` — broken in production (CF Workers = UTC). (3) Slots returned as naive HH:MM strings — no timezone context. (4) Sessions stored as naive datetimes — no Z suffix or offset. (5) Display uses `new Date()` which interprets differently per environment. Mapped the correct approach: convert teacher availability to UTC server-side, return UTC slots, store as UTC, convert to local on display. Noted the slot concept (1-hour, on-the-hour) is only in 2 places and doesn't need changing.
