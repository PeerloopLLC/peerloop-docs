# Conv Learnings - 2026-03-24

## 1. SQLite datetime() vs ISO String Comparison Is Silently Broken
**Topics:** d1, testing

**Context:** While implementing the INVITE-UX feature, the GET `/api/session-invites` endpoint was extended to return recently-expired invites using `expires_at > datetime('now', '-5 minutes')`. A test inserting an invite expired 1 hour ago unexpectedly matched the query.

**Learning:** SQLite's `datetime('now', ...)` outputs `YYYY-MM-DD HH:MM:SS` (space at position 10), but all stored timestamps use JavaScript `.toISOString()` format `YYYY-MM-DDTHH:MM:SS.sssZ` (T at position 10). Since SQLite compares TEXT values lexicographically, `T` (ASCII 84) always beats ` ` (ASCII 32), making **any** ISO string sort as "greater than" any `datetime()` output — regardless of actual time values.

This means `column > datetime('now', '-7 days')` is always TRUE. Six queries across the codebase had this latent bug — admin dashboard stats, teacher session counts, and smart feed activity filters were all silently returning all rows instead of the intended time window. The data was usually recent enough that results appeared correct.

**Pattern:** Always use `strftime('%Y-%m-%dT%H:%M:%fZ', 'now', '-X days')` instead of `datetime('now', '-X days')` when comparing against ISO-formatted TEXT columns. `date('now', ...)` is safe because its 10-character output (`YYYY-MM-DD`) is a prefix of the ISO format and sorts correctly.

## 2. BBB Recording Format Limitation
**Topics:** video

**Context:** RECORDING-R2 sub-block aimed to replicate BBB recordings to Cloudflare R2 for long-term storage. Investigation revealed a fundamental platform limitation.

**Learning:** BBB's default recording format is "presentation" — an interactive HTML5 playback page, not a downloadable video file. The `getRecordings` API returns a playback URL pointing to this HTML page. A separate "video" workflow that produces downloadable MP4/webm exists but is **not enabled by default** and requires server-side configuration. On Blindside Networks' managed service, this may require a vendor request.

Additionally, our `BBBRecording` interface modeled `playback.format` as a single object, but BBB actually returns an array when multiple formats exist. The parser was silently dropping non-first formats.

**Pattern:** When integrating with external APIs, verify the actual response format with real data or official docs — don't assume the interface model is complete. Also: "URL" doesn't always mean "downloadable file" — HTML playback pages and binary video files are fundamentally different.

## 3. bumpUserDataVersion as a Cross-Component Signal
**Topics:** auth, astro

**Context:** When a student accepts a session invite, the teacher needs to know. The existing notification poll runs every 60 seconds, but `bumpUserDataVersion()` triggers a CurrentUser refresh within 30 seconds.

**Learning:** The `data_version` + version polling pattern (Conv 013) is a lightweight cross-component signal mechanism. Adding `bumpUserDataVersion(db, userId)` to any mutation endpoint makes the affected user's client refresh within 30s — without new polling loops, WebSocket connections, or state management. For the invite flow, this was more appropriate than adding invite-specific state to CurrentUser (which would have been hacky — ephemeral invite status doesn't belong in global user state).

**Pattern:** For "user X needs to know something changed" scenarios, bump their data_version. Let the existing polling infrastructure propagate it. Reserve CurrentUser for durable identity/capability data.

## 4. Component-Local Polling for Ephemeral State
**Topics:** astro, testing

**Context:** MyStudents needed to show real-time invite status (pending → accepted → expired) on the student row after a teacher sends an invite.

**Learning:** For page-local ephemeral state like invite status, a component-level poll (10-second interval while pending, auto-stop on terminal state) is simpler and more appropriate than wiring into global state. The ExpiryCountdown component uses a 1-second `setInterval` for the live countdown, while the status poll uses a 10-second interval that self-cleans on unmount or state change. Keeping the poll in a `useRef` map keyed by enrollmentId means multiple concurrent invites can poll independently.
