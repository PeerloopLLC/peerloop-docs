# Conv Decisions - 2026-03-19

## 1. Feeds Strategy: Build Phase 5 First, Then Composite Page (Option 1)
**Type:** Strategic
**Topics:** astro, stream

**Trigger:** Client directive that feeds provide ~50% of platform learning. Students ask questions in feeds. Discussion on whether to build a MyFeeds dashboard card (Phase 5) or jump to a composite `/feeds` page immediately.

**Options Considered:**
1. Build Phase 5 as-scoped (MyFeeds card + FeedSlidePanel refactor), then create FEEDS-HUB block for composite page ← Chosen
2. Expand Phase 5 to include composite `/feeds` page now — bigger scope but delivers client vision sooner
3. Pause Phase 5, design composite page architecture first

**Decision:** Option 1 — build Phase 5 as-scoped, then open FEEDS-HUB block. Phase 5 doesn't create student navigation ambiguity because FeedSlidePanel + dashboard card + `/feed` aggregated timeline cover the journey adequately for Genesis cohort.

**Rationale:** The current feed surfaces (FeedSlidePanel, `/feed`, `/community/[slug]`, `/course/[slug]/feed`) already give students a complete path from "I have a question" to posting in the right feed. The slideout panel IS the navigation layer. A composite page improves the experience but isn't blocking. Phase 5 also refactors FeedSlidePanel to use `getFeeds()`, eliminating 2 more redundant API calls.

**Consequences:** FEEDS-HUB block added to PLAN.md with comprehensive scope: composite `/feeds` page, navigation changes (AppNavbar "My Feeds" link update), activity tracking (last-visited timestamps, "X new posts" badges).

## 2. FeedSlidePanel Refactor Bundled Into Phase 5
**Type:** Implementation
**Topics:** astro

**Trigger:** FeedSlidePanel currently makes its own API calls (`/api/communities?filter=mine`, `/api/me/enrollments`) to build the feed list — same data CurrentUser now carries via `getFeeds()`.

**Options Considered:**
1. Refactor FeedSlidePanel as a separate task
2. Bundle refactor into Phase 5 alongside MyFeeds card ← Chosen

**Decision:** Bundle the FeedSlidePanel refactor into Phase 5. Both consume `getFeeds()`, and doing them together ensures consistent behavior and eliminates 2 more redundant API calls in one pass.

**Rationale:** The FeedSlidePanel and MyFeeds dashboard card render the same data — one as a slideout, one as a card. Building them from the same `getFeeds()` source in the same phase prevents divergence.

**Consequences:** Phase 5 updated in PLAN.md to include FeedSlidePanel refactor alongside MyFeeds card component.
