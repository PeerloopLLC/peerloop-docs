# Session Decisions - 2026-03-05

## 1. Dedicated Endpoint for canMessage Over Piggybacking
**Type:** Implementation
**Topics:** messaging, astro

**Trigger:** Phase 1 messaging UX needed to conditionally show/hide Message buttons on 3 profile surfaces. Each surface has a different data-fetching pattern (PublicProfile fetches from API, Creator/ST pages use SSR props).

**Options Considered:**
1. Add `canMessage` field to existing `/api/users/:handle` response (requires adding optional auth to a public endpoint) + different approach for SSR pages
2. Have each component call the user search endpoint to check if target appears in results (hacky)
3. New dedicated `GET /api/me/can-message/:userId` endpoint + `useCanMessage` hook ← Chosen

**Decision:** Created a single-purpose endpoint that wraps the existing `canMessage()` library function, plus a reusable React hook. All 3 surfaces use the same hook regardless of how they load their profile data.

**Rationale:** One place to build, one hook to reuse, works identically across all surfaces. Avoids changing the auth contract of the public user profile API. The extra round trip is trivial — it's a single boolean check. The hook also handles visitor (no API call) and self-profile (returns false) short-circuits client-side.

**Consequences:** New files: `src/pages/api/me/can-message/[userId].ts`, `src/lib/useCanMessage.ts`. Phase 3 conditional surfaces will reuse the same hook.

## 2. Phase 2 Admin Panels: Direct Links Without Permission Check
**Type:** Implementation
**Topics:** messaging, admin

**Trigger:** 6 admin detail panels needed message buttons. Question: should they use `useCanMessage` hook like Phase 1 profile pages?

**Options Considered:**
1. Use `useCanMessage` hook for consistency ← Not chosen
2. Direct `/messages?to=` links without any check ← Chosen

**Decision:** Admin panels render direct message links without calling `useCanMessage`. No hook, no API call.

**Rationale:** POLICIES.md rule 1: "Admin or global moderator can message anyone." The admin context is itself the proof of permission — checking would always return true. Skipping the check avoids 6 unnecessary API calls per admin page load.

**Consequences:** Admin message buttons appear immediately (no loading state). If an admin's role were ever revoked while they have the panel open, the API gate on `POST /api/conversations` would still block the message — the defense is at the write layer, not the UI layer.
