# Session Learnings - 2026-03-04

## 1. FK Constraints in D1 Cause Silent Enrollment Failures
**Topics:** d1, stripe

**Context:** Testers enrolled in a course but the course didn't appear on their `/courses` dashboard. The Stripe webhook handler was inserting a `student_teachers.id` (`st-xxx`) into `enrollments.student_teacher_id` which has `REFERENCES users(id)`.

**Learning:** D1 enforces `foreign_keys=ON` at the platform level. When a FK violation occurs inside a webhook handler wrapped in try/catch, the INSERT fails silently — the webhook returns 500 to Stripe, Stripe retries, same failure repeats. The enrollment is never created and there's no user-visible error. Always verify that the ID type being inserted matches the FK target table.

**Pattern:** When a column references `users(id)`, trace the full data flow from client → API → Stripe metadata → webhook INSERT to ensure the ID is resolved to the correct type at each hop. In this case, `create-session.ts` fetched `user_id` from the ST lookup but discarded it, passing the raw `st.id` into Stripe metadata instead.

## 2. Stripe Metadata Carries Both Record ID and User ID
**Topics:** stripe

**Context:** The webhook handler needs two different IDs for the same student-teacher: the `student_teachers.id` (`st-xxx`) for JOINs to look up email/Stripe account info, and the `users.id` (`usr-xxx`) for the enrollment FK.

**Learning:** When Stripe metadata needs to serve multiple downstream consumers with different ID needs, store both IDs explicitly rather than trying to resolve one from the other in the webhook. Metadata is cheap (key/value strings); the alternative — a DB lookup to resolve `st-xxx` → `usr-xxx` inside the webhook — adds fragility and a DB round-trip for every checkout.

## 3. Self-Healing Pattern Extends to Any Webhook-Dependent Creation
**Topics:** stripe, astro

**Context:** Session 223 established the self-healing pattern for Stripe Connect status: fetch live data from Stripe API on page load and sync the DB if it drifted. The same pattern was needed for enrollment creation when webhooks are missed.

**Learning:** The self-healing pattern has two natural surfaces: (1) SSR page load (success page has the `session_id` from Stripe redirect), and (2) client-side mount with localStorage bridge (for pages that don't have the session_id). The SSR path is simpler and covers 99% of cases; the localStorage bridge is a safety net for edge cases like users who close the success page before it loads.

**Pattern:** Extract webhook logic into a shared function (`src/lib/enrollment.ts`), then call it from both the webhook handler and the self-healing endpoint. The function must be idempotent (check-before-create) so it's safe to call from multiple paths.

## 4. Variable Scoping Across if-Blocks in TypeScript
**Topics:** testing

**Context:** Added a `studentTeacher` variable inside an `if (student_teacher_id)` block, then referenced it outside the block via `studentTeacher?.user_id`. TypeScript compiled it but it was a runtime scoping error — `studentTeacher` was `const` inside the block and undefined outside.

**Learning:** When extracting a value from inside a conditional block for use outside it, declare the variable (with `let`) before the block and assign inside. Don't rely on optional chaining (`?.`) to handle an out-of-scope `const` — it will throw a ReferenceError, not return undefined.
