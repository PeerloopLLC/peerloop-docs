# Session Decisions - 2026-03-12

## 1. Leave Public/Legacy Headers Using /api/auth/session
**Type:** Implementation
**Topics:** auth, astro

**Trigger:** Audit found `Header.tsx` (LandingLayout) and `AppHeader.tsx` (LegacyAppLayout) making `/api/auth/session` calls. Question: should these also use `getCurrentUser()`?

**Options Considered:**
1. Migrate Header.tsx to use `getCurrentUser()` / `initializeCurrentUser()` on public pages
2. Create `getCurrentUserIfCached()` for lightweight cache-only reads ← Future (tracked in PUBLIC-PAGES)
3. Leave as-is — `/api/auth/session` is appropriate for public pages ← Chosen

**Decision:** Public page headers continue using the lightweight `/api/auth/session` endpoint. No changes to `Header.tsx` or `AppHeader.tsx`.

**Rationale:** Public pages don't need course relationships, stats, or Stripe data. The `/api/me/full` payload (~2-5KB) is overkill for rendering an avatar + dashboard link. Non-logged-in visitors would trigger a wasted 401. The real wins are on AppLayout pages where `currentUser` is already initialized.

**Consequences:** Created PUBLIC-PAGES plan block (deferred #29) to track future unification. The block includes a `getCurrentUserIfCached()` helper for when public pages need personalization.

## 2. Map CurrentUser Capabilities to ContextActions Registry Interface
**Type:** Implementation
**Topics:** auth, astro

**Trigger:** ContextActionsPanel's `getActionsForContext()` expects `{ is_admin, is_moderator, is_creator, is_teacher }` booleans, but `CurrentUser` uses capability-based fields.

**Options Considered:**
1. Refactor `getActionsForContext()` to accept `CurrentUser` directly
2. Create an adapter/mapper at the call site ← Chosen
3. Add `is_teacher`/`is_creator` convenience booleans to `CurrentUser` class

**Decision:** Map at the call site: `isAdmin` → `is_admin`, `canModerateCourses` → `is_moderator`, `canCreateCourses` → `is_creator`, `isActiveTeacher()` → `is_teacher`.

**Rationale:** The registry's role-based interface is intentionally simple — it doesn't need course-specific role checking. Changing the registry signature would require updating the type system across multiple files for no functional benefit. The 4-line mapping is clear and keeps the registry decoupled from `CurrentUser`'s richer model.

## 3. Add Pub/Sub Listener System to current-user.ts
**Type:** Implementation
**Topics:** auth, astro

**Trigger:** Need for `useCurrentUser()` hook that auto-updates when background stale-while-revalidate completes.

**Options Considered:**
1. Use a CustomEvent on `window` (e.g., `peerloop:user-changed`)
2. Add a `Set<Listener>` inside `current-user.ts` with `subscribeToUserChange()` ← Chosen
3. Use a React context provider wrapping the app

**Decision:** Module-level `Set<UserChangeListener>` with `subscribeToUserChange()` returning an unsubscribe function. `setCurrentUser()` calls `notifyListeners()` on every update.

**Rationale:** Keeps the subscription mechanism co-located with the state it watches. No DOM dependency (unlike CustomEvent). No React wrapper needed (unlike Context). The Set-based pattern is standard, lightweight, and works for both React hooks and non-React consumers. The unsubscribe-via-closure pattern matches React's `useEffect` cleanup.

**Consequences:** `useCurrentUser()` hook added as the React-facing API. All `setCurrentUser()` calls (init, refresh, clear) now trigger listeners.
