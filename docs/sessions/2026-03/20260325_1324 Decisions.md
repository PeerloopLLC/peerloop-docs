# Conv Decisions - 2026-03-25

## 1. Login Redirect: Scoped to Auth Pages, Not Global
**Type:** Implementation
**Topics:** auth, astro

**Trigger:** `/login` page didn't redirect after successful login — URL stayed at `/login`. Three approaches considered.

**Options Considered:**
1. Quick fix — fix param name only, add default redirect in AutoOpenAuthModal
2. Durable fix — fix param name + handle redirect in `handleAuthSuccess()` as fallback ← Chosen
3. Middle ground — fix param name + add redirect only in `/login` page's AutoOpenAuthModal

**Decision:** Redirect logic lives in `handleAuthSuccess()` in `auth-modal.ts`, scoped to auth pages only. When `window.location.pathname` is `/login` or `/signup` and there's no explicit `pendingAction`, redirect to `/dashboard`. On other pages (modal opened from navbar or `useRequireAuth`), no redirect — user stays on current page.

**Rationale:** Option B is most durable because it handles the redirect regardless of which component opens the modal on an auth page. The path check prevents unwanted redirects when the modal is opened from non-auth pages (e.g., navbar login button while browsing a course).

**Consequences:**
- E2E login helper updated to `waitForURL('**/dashboard')` instead of checking modal visibility
- `login.astro` and `signup.astro` now read both `?redirect=` and `?returnUrl=` params
- All 135 E2E tests pass with the new behavior

## 2. E2E Notification Tests: Resilient to DB State
**Type:** Implementation
**Topics:** testing

**Trigger:** Notification E2E tests broke on re-runs because mutation tests permanently altered seed data, and other tests (booking flow) created additional notifications.

**Options Considered:**
1. Reset DB before notification tests (slow, complex)
2. Make tests resilient to state — no exact counts, general assertions ← Chosen
3. Add API-based setup/teardown for each test (heavyweight)

**Decision:** Notification tests use general assertions instead of exact seed data expectations. Wait for any notification to appear (`/\d+ notification/`), check `count >= 1` instead of `count === 2`, use `toBeAttached()` for CSS-hidden elements.

**Rationale:** The cheapest, most maintainable approach. Tests verify component behavior (rendering, filtering, deletion) without coupling to specific seed data. DB reset is still needed periodically but not between every test run.

**Consequences:** Notification tests pass regardless of DB state (as long as at least one notification exists for the test user).
