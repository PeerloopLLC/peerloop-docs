# Conv Decisions - 2026-03-28

## 1. Community Role Tabs: Conservative Scaffolding Over Full Features
**Type:** Implementation
**Topics:** explore, communities, ux

**Trigger:** Phase 3 audit showed communities have 3 permission checks vs courses' 15+ tabs

**Options Considered:**
1. Skip Phase 3 entirely — communities don't need role tabs yet
2. Build minimal role tabs with placeholder content ← Chosen
3. Build full role tabs + first real feature (moderation queue)

**Decision:** Build conservative scaffolding (Option 2) so the client can see where future functionality will reside

**Rationale:** Client needs visibility into the platform's structure. Placeholder tabs are trivially upgradeable when real features (settings, moderation queue) are built. 2-3 hour build vs 0 (skip) or 8-12 hours (full).

**Consequences:** Creator sees Settings + Manage Members tabs; Moderator sees Moderation tab. All show placeholder content with "coming soon" messaging.

## 2. /discover/feeds Reframed as Feed Discovery, Not Feed Inventory
**Type:** Strategic
**Topics:** ux, architecture, feeds, discover

**Trigger:** User observed "My Feeds" label drift and questioned the premise of feed discovery

**Options Considered:**
1. Kill `/discover/feeds`, keep `/feeds` only
2. Reframe as "My Feeds" under different nav location
3. Transform into "Conversations Worth Joining" with parent-entity CTAs ← Chosen

**Decision:** Transform `/discover/feeds` into a visitor-accessible discovery page surfacing active public feeds with clear "Join Community →" / "View Course →" CTAs

**Rationale:** Feeds are derivative (can't join directly), but showing active discussions with CTAs to join the parent entity is genuine discovery value. Reuses existing Smart Feed discovery pipeline (~70% overlap).

**Consequences:** `/discover/feeds` is now visitor-accessible (no auth wall). Discovery section shows feed-level cards ranked by vitality (topic-matched for auth users, popular for visitors). "Your Feeds" role-aware directory shown below for authenticated users only. DiscoverSlidePanel label changed from "My Feeds" to "Feeds".

## 3. Feed-Level Discovery (C1) Before Thread-Level (C2)
**Type:** Strategic
**Topics:** feeds, ux, smart-feed

**Trigger:** User asked about C1 vs C2 visual difference

**Options Considered:**
1. Feed-level discovery (C1): card per feed, one preview post, vitality stats ← Chosen
2. Thread-level discovery (C2): specific conversation threads with multiple participants

**Decision:** Ship C1 now, defer C2 pending user behavior validation

**Rationale:** C1 reuses proven infrastructure (2-hour build). C2 requires new schema work (thread metadata), conversation scoring signals (~12-16 hours). If C1 discovery cards aren't getting clicks, C2 won't help. Upgradeable: same candidate pipeline, only enrichment layer changes.

**Consequences:** Discovery cards show feed-level info (name, vitality, latest post author). Thread-level previews deferred until C1 validates the concept.
