# Session Decisions - 2026-03-05 (Session 341)

## 1. Three-Function Messaging Relationship Architecture
**Type:** Implementation
**Topics:** messaging, api-design, d1

**Trigger:** Session 338 defined the messaging access control policy (POLICIES.md §4) but left implementation pending. Need a shared function for relationship checks that works across 3 API endpoints and future UI surfaces.

**Options Considered:**
1. Single `canMessage(senderId, recipientId)` function called per-user ← Simple but O(N) queries for lists
2. Two functions: single check + batch check ← Good for API + lists, but search needs SQL integration
3. Three functions: single check, batch check, SQL clause generator ← Chosen

**Decision:** Implement three exported functions in `src/lib/messaging.ts`:
- `canMessage()` — single-pair boolean (delegates to batch internally)
- `getMessageableFlags()` — batch check returning `Record<string, boolean>` for annotating lists
- `messageableContactsSQL()` — returns SQL WHERE clause + params for search endpoint integration

**Rationale:** Each function serves a different use case with different performance needs. The batch function avoids N+1 queries for list views. The SQL function ensures LIMIT works correctly with filtered search results. The single function delegates to batch for DRY relationship logic.

**Consequences:**
- `src/lib/messaging.ts` created with all relationship rules in one place
- 3 API endpoints gated: `POST /api/conversations`, `POST /api/conversations/:id/messages`, `GET /api/users/search`
- 20 new tests covering all relationship permutations
- Phase 1 of tech-018 implementation plan marked complete

## 2. Batch Check Returns Annotation Flags, Not Filtered List
**Type:** Implementation
**Topics:** messaging, api-design

**Trigger:** When showing Message buttons on list views (S-T list, community members), should the helper return a filtered subset of users or boolean flags for all users?

**Options Considered:**
1. Return filtered array of messageable user IDs ← Caller can't distinguish "not messageable" from "not in list"
2. Return `Record<string, boolean>` mapping each ID to its status ← Chosen

**Decision:** `getMessageableFlags()` returns `Record<string, boolean>` — every input ID gets a true/false flag.

**Rationale:** The caller needs to show all users in the list (e.g., all S-Ts for a course) but conditionally render the Message button. A filtered list would conflate "remove from view" with "hide message button." The annotation pattern keeps display logic and messaging logic separate.
