# Conv Decisions - 2026-03-25

## 1. Separate session_analytics Table for BBB Learning Analytics
**Type:** Implementation
**Topics:** d1, video

**Trigger:** BBB analytics callback delivers a rich JSON payload per session. Need to decide storage strategy.

**Options Considered:**
1. Separate `session_analytics` table — one row per session, full JSON blob + summary columns ← Chosen
2. JSON column on `sessions` table — simpler, no JOIN, but widens sessions table
3. Normalized columns on `sessions` — individual columns for each metric, most queryable but rigid

**Decision:** Option 1 — separate `session_analytics` table with `session_id` (UNIQUE FK), `analytics_json` (TEXT), `duration_seconds`, `attendee_count`.

**Rationale:** Clean separation from session lifecycle data. Full JSON blob preserved for debugging and future use. Summary columns enable quick queries without JSON parsing. Upsert on `session_id` handles BBB retries. Doesn't bloat the sessions table which is queried frequently.

**Consequences:** Added table to schema. Session detail API JOINs for analytics only on completed sessions. No migration needed (schema is pre-launch).

## 2. BBB Reconciliation via Admin Cleanup Endpoint (Now) + Cron (Deferred)
**Type:** Strategic
**Topics:** video, cloudflare

**Trigger:** BBB webhooks and analytics callbacks are fire-and-forget. If our endpoint is down when BBB fires them, the data is lost (analytics) or the session state is stale (completion, recordings).

**Options Considered:**
1. Option A: Extend existing admin cleanup endpoint with `reconcileBBBSessions()` ← Chosen (now)
2. Option B: Cron-triggered reconciliation via Cloudflare Cron Trigger ← Deferred to CRON-CLEANUP
3. Option C: Retry queue with `pending_analytics` flag — analytics not recoverable, so only partial help

**Decision:** Option A now (manual trigger via admin cleanup), Option B deferred to CRON-CLEANUP block (pre-launch).

**Rationale:** Option A adds reconciliation with zero new infrastructure — just a few extra BBB API calls per cleanup run. Option B is the production answer but requires Astro + Cloudflare adapter work for `scheduled()` handler. Analytics callbacks are acknowledged as best-effort (not recoverable).

**Consequences:** `reconcileBBBSessions()` added to `booking.ts`, wired into cleanup endpoint. Catches missed `meeting-ended` (via `getRoomInfo`) and missed `recording_ready` (via `getRecordings`). CRON-CLEANUP block updated with reconciliation scope.

## 3. Dual Defense Against SQLite datetime() Misuse
**Type:** Strategic
**Topics:** d1, testing, workflow

**Trigger:** Discovered that `datetime()` in SQL comparisons produces silent bugs due to format mismatch with ISO timestamps. Need to prevent recurrence.

**Options Considered:**
1. Option A: CLAUDE.md rule only — catches AI-authored code
2. Option B: Lint check in `/w-codecheck` only — catches all code
3. Option C: Both A + B ← Chosen

**Decision:** Both defenses: CLAUDE.md rule (primary, read every conversation) + `/w-codecheck` check #5 (safety net for manual edits).

**Rationale:** CLAUDE.md rule costs nothing and is the primary defense for the majority of code written in this project. The lint check catches edge cases (copy-paste from SQLite docs, manual edits). Both took minutes to implement.

**Consequences:** CLAUDE.md updated with "SQLite Datetime Rule" section including wrong/correct examples. `/w-codecheck` SKILL.md updated with check #5 (grep-based scan, zero false positives on current codebase).

## 4. Dispute Warning as In-App Notification (Not Email)
**Type:** Implementation
**Topics:** d1

**Trigger:** Two TODOs in `resolve.ts` for sending warning notifications to student/teacher on dispute resolution. Originally noted as "pending NOTIFY block."

**Options Considered:**
1. In-app notification only (via existing `createNotification` pattern) ← Chosen
2. Email notification (via Resend)
3. Both in-app + email

**Decision:** In-app notification using new `dispute_warning` notification type. Non-blocking, fire-and-forget.

**Rationale:** Consistent with how all other session notifications work (session_completed, session_no_show, etc.). Email can be added later when the email notification pipeline is more mature. The in-app notification includes course name and resolution notes, with a link to the session.

**Consequences:** Added `dispute_warning` to schema CHECK constraint, `NotificationType` union, notification icon map, and `notifyDisputeWarning()` helper. Removed both TODOs from resolve.ts.
