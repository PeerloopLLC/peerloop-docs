# Session Learnings - 2026-03-12

## 1. Route-Matrix Scanner False Positives on Hardcoded Slugs
**Topics:** astro, docs-infra

**Context:** The route-matrix scanner flagged `/community/the-commons` as a broken target because it's a hardcoded literal string in the Discover page, not a template variable like `${slug}`.

**Learning:** Hardcoded literal paths that are concrete instances of dynamic routes (e.g., `/community/the-commons` â†’ `/community/[slug]`) need normalization in the scanner. The `normalizeDynamic()` function only handled template literals (`${slug}`), not static strings. Added a `LITERAL_SLUG_ROUTES` array with regex patterns that normalize known concrete slugs to their dynamic equivalents.

**Pattern:**
```js
const LITERAL_SLUG_ROUTES = [
  [/^\/community\/(?!$|\[)[^/]+$/, '/community/[slug]'],
];
```

## 2. TeacherPerformanceTable Linked by UUID Instead of Slug
**Topics:** astro

**Context:** The `TeacherPerformanceTable` component used `course_id` (a UUID) in its href, producing links like `/course/abc-123-uuid` which would 404. The API query already joined the `courses` table but only selected `course_id` and `course_title`, not `slug`.

**Learning:** When building navigation links, always use the slug (human-readable URL segment), not the internal ID. The fix required changes in three places: the API SQL query (add `c.slug as course_slug`), the API response transform, and the component's interface + href.

## 3. Profile Route is `/@[handle]` Not `/profile/[handle]`
**Topics:** astro

**Context:** `MessageThread.tsx` and `Leaderboard.tsx` linked to `/profile/${handle}`, but no `/profile/[handle]` page exists. The actual profile page is `/@[handle]` (via `src/pages/@[handle].astro`), and `/profile.astro` just redirects to it.

**Learning:** The public profile route uses the Twitter-style `/@handle` convention. Components that need to link to user profiles should use `/@${handle}`, not `/profile/${handle}`.

## 4. Dead Links to Unbuilt Pages Accumulate Silently
**Topics:** astro

**Context:** Two planned-but-unbuilt pages (`/course/[slug]/certificate` and `/course/[slug]/review`) had working links pointing to them from multiple components. The certificate link appeared in 6 course tab pages via LearnTab; the review link appeared in MyCourses.

**Learning:** The route-matrix scanner is effective at catching these. For unbuilt pages, either remove the link with a TODO comment, or redirect to an existing page that handles the same functionality (e.g., the review modal on the learn tab). Don't leave dead links to planned pages â€” they create user-facing 404s.
