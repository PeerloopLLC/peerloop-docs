# Conv Decisions - 2026-03-19

## 1. SMART-FEED: Hybrid Query Strategy (D1 Simple Select + App-Side Scoring)
**Type:** Implementation
**Topics:** d1, stream

**Trigger:** Plan Mode verification found the original 7-JOIN CTE design fragile and hard to test.

**Options Considered:**
1. Large CTE with all scoring in SQL — single D1 query does selection, relationship JOINs, and scoring
2. Hybrid: D1 selects raw candidates, app loads metadata in parallel, TypeScript scores ← Chosen
3. Stream-only: use Stream's timeline feed with client-side filtering — limited by Stream's lack of cross-feed queries

**Decision:** D1 does simple candidate selection using the proven tuple-matching pattern from `getFeedBadgeCounts()`. App code loads relationship metadata (teacher certs, creators, communities, topic interests, feed affinity) via 4-5 parallel D1 queries and scores candidates in TypeScript. Stream enriches with engagement data.

**Rationale:** Matches the existing architecture (feed-badges.ts pre-computes feed lists in app code). Each phase is independently testable. The scoring module (`scoring.ts`) has a stable interface and swappable internals.

**Consequences:** Created `src/lib/smart-feed/` with 5 modules: candidates, scoring, enrichment, discovery, orchestrator.

## 2. SMART-FEED: Discovery Cards from Day One (Not Deferred)
**Type:** Strategic
**Topics:** stream, d1

**Trigger:** Client's flywheel model requires discoverability for the Genesis group — students need to find communities and courses they didn't know about.

**Options Considered:**
1. Members-only pool first, discovery later — safe but limits flywheel testing
2. Discovery from day one with access control ← Chosen
3. Hybrid (members first, discovery Phase 2) — initially recommended but client overruled

**Decision:** Discovery cards are included in the initial SMART-FEED implementation. Public communities and public course feeds are surfaced to non-members with preview text, engagement social proof, and CTA to join/enroll.

**Rationale:** The Genesis group (60-80 students, 4-5 courses) is the proving ground for the flywheel. Discovery can't wait — visibility drives enrollment drives teaching.

**Consequences:** `DiscoveryCard` component, `smart_feed_dismissals` table, discovery candidate selection adapted from recommendation queries, access control matrix (public only, no private feeds).

## 3. SMART-FEED: Scoring Params in platform_stats (Not Dedicated Table)
**Type:** Implementation
**Topics:** d1

**Trigger:** Need tunable scoring weights without code deploys. Under 15 parameters total.

**Options Considered:**
1. Dedicated `smart_feed_config` table — more structured but overkill for <15 params
2. `platform_stats` key-value rows ← Chosen
3. Hardcoded constants in code — not tunable without deploys

**Decision:** Scoring weights and parameters stored as `platform_stats` rows with `smart_feed_*` key prefix. 12 rows total covering weights, decay, diversity cap, discovery frequency.

**Rationale:** Under 15 params fits perfectly as key-value rows. Tunable via direct D1 update now, admin UI later (ADMIN-SETTINGS-UI block). No schema complexity for a small parameter set.

**Consequences:** Added 12 `platform_stats` rows to `0002_seed_core.sql`. `loadScoringParams()` reads them with fallback defaults.

## 4. SMART-FEED: Algorithm Isolation Pattern
**Type:** Implementation
**Topics:** d1, stream

**Trigger:** Client requirement that the scoring algorithm be swappable as the platform grows.

**Decision:** Scoring algorithm isolated in `src/lib/smart-feed/scoring.ts` behind a stable interface: `scoreCandidates(candidates, context, params) → ScoredCandidate[]`. The candidates module, enrichment module, and orchestrator are unaffected by algorithm changes.

**Rationale:** The interface is the contract. Candidates provide raw data + relationship metadata. The algorithm consumes it and produces scores. Replacing the algorithm (e.g., from weighted signals to a trained model) only changes `scoring.ts` internals.

**Consequences:** `ScoringContext` type captures all relationship metadata. `ScoringParams` loads from `platform_stats`. Discovery posts use hardcoded weights (engagement-heavy) vs member posts using tunable weights.

## 5. Community & Course Feed Privacy: Schema-First, Toggle UI Deferred
**Type:** Implementation
**Topics:** d1

**Trigger:** Discussion about what happens when SMART-FEED discovers feeds the user can't access.

**Decision:** `communities.is_public` (existing, default 1) controls community discovery. New `courses.feed_public` column (default 1) controls course feed discovery. No toggle UI exists yet — all feeds are public at Genesis. Toggle UI tracked in FEED-PRIVACY deferred block.

**Rationale:** SMART-FEED queries filter on these flags now, so when privacy toggles ship, discovery respects them automatically. Zero changes needed to SMART-FEED code.

**Consequences:** Added `feed_public INTEGER NOT NULL DEFAULT 1` to courses table. FEED-PRIVACY block (#34) added to PLAN.md deferred list.

## 6. Topic Matching: Static + Feed Affinity (No Semantic/NLP)
**Type:** Implementation
**Topics:** d1, stream

**Trigger:** Determining which signals constitute "importance" for the smart feed.

**Options Considered:**
1. Static topics only (`user_topic_interests` → category match)
2. Static topics + feed affinity + category affinity ← Chosen
3. Add keyword extraction from post text (NLP) — high build cost, high brittleness
4. Semantic embedding matching — requires vector store, overkill

**Decision:** Three topic signals included: (a) static topic match via `user_topic_interests → topics → categories`, (b) feed affinity from `feed_activities` GROUP BY actor, (c) category affinity mapped through course/community categories. No NLP or semantic matching.

**Rationale:** All three are SQL joins on indexed tables with no external dependencies. Feed affinity (posting frequency) IS the interest signal at Genesis scale — a user who posts 10x in Python Devs clearly cares about Python. Low build cost, low brittleness.

**Consequences:** `idx_feed_activities_actor` index added for affinity queries. `loadScoringContext()` runs 6 parallel queries including affinity computation.
