# Conv Learnings - 2026-03-18

## 1. Test Mock Data Must Match Component Interface Casing
**Topics:** testing

**Context:** MyStudents.test.tsx had 5 pre-existing failures. The component's `Student` TypeScript interface uses camelCase (`enrollmentId`, `avatarUrl`, `courseSlug`) but the test mock data used snake_case (`enrollment_id`, `avatar_url`, `course_slug`).

**Learning:** When mock data property names don't match the component's TypeScript interface, all properties resolve to `undefined` at render time. This silently breaks rendering — the component doesn't crash, it just takes every falsy code path. The failure manifests as "element not found" in tests rather than a type error, making the root cause non-obvious.

**Pattern:** Always cross-reference mock data keys against the component's interface definition. TypeScript won't catch this at test time because `mockFetch` returns untyped JSON.

## 2. Seed Data Review Surfaces Feature Gaps, Not Just Data Gaps
**Topics:** d1, testing

**Context:** A seed data completeness audit for manual testing revealed three categories of issues: (1) actual missing seed data (avatars, Gabriel's Stripe/availability), (2) features not yet built (certificate PDF generation, avatar upload), and (3) architecture that doesn't need seeding (BBB join URLs are generated on-demand, not stored).

**Learning:** Seed data audits are an effective discovery mechanism for feature gaps. Walking each user through all related tables forces you to confront what the system actually does vs. what it should do. The audit surfaced 3 new PLAN.md blocks (CERT-APPROVAL PDF generation, FILE-UPLOADS avatar, RECORDING-PERSIST) that wouldn't have been found through code review alone.

## 3. BBB Video Join URLs Are Ephemeral By Design
**Topics:** video

**Context:** Initial seed data review flagged "no video room URLs" as a Tier 1 gap. Investigation revealed that BBB join URLs are generated on-demand via `POST /api/sessions/{id}/join` — the server creates the room if it doesn't exist, signs the join URL with checksum auth, and returns it. The only persistent URL is `recording_url`, populated by webhook after session ends.

**Learning:** The `sessions` table stores `bbb_meeting_id` and passwords but intentionally does not store join URLs. This is correct architecture — BBB join URLs are signed with timestamps and participant info, making them inherently ephemeral. Seeding fake URLs would create broken links, not useful test data.

## 4. UPDATE Statements Are Cleaner Than Modifying Wide INSERT Rows
**Topics:** d1

**Context:** Needed to add `last_login` and social URL values to seed users. The users INSERT is a single massive statement with 30+ columns per row — modifying it risks breaking the SQL.

**Learning:** Using `UPDATE users SET last_login = '...' WHERE id = '...';` after the INSERT block is cleaner and more maintainable than trying to add columns to wide INSERT statements. It's also self-documenting — the UPDATE section has its own comment block explaining what it does.
