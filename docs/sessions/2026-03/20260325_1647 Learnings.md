# Conv Learnings - 2026-03-25

## 1. CourseFeed Component Takes slug, Not courseId
**Topics:** astro, testing

**Context:** Building the Feed tab in TeacherCourseView, initially passed `courseId` and `canPost` props to CourseFeed.

**Learning:** `CourseFeed` accepts a single `slug` prop, not `courseId`/`canPost`. The component fetches its own data internally using the slug. Always check component interfaces before wiring them up.

---

## 2. Existing Resources API Requires Enrollment — Teachers Get 403
**Topics:** d1, auth

**Context:** Considered reusing `/api/courses/[id]/resources` for the teacher Resources tab. That endpoint checks `enrollments` for `student_id` match — teachers aren't enrolled, so they'd get a 403.

**Learning:** Access control is baked into each API endpoint individually. When a new role needs access to the same data, evaluate whether to widen the existing gate or create a dedicated endpoint. For the teacher resources case, a dedicated endpoint was chosen to allow independent evolution.

---

## 3. Breadcrumb Context via Query Param Pattern Is Extensible
**Topics:** astro

**Context:** The public course page (`/course/[slug]`) already supports `?via=discover-courses` and `?via=community-courses` for breadcrumb context. Adding `?via=teaching&cid={courseId}` followed the same pattern with zero new infrastructure.

**Learning:** The `via` query param on course pages is the established mechanism for breadcrumb context. Any new navigation source to a course page should use `via=source` to get appropriate breadcrumbs. The `cid` param carries the course ID for back-links when the source uses IDs (like teaching routes) rather than slugs.

---

## 4. Public Course Page Is Not a "Student Page"
**Topics:** astro

**Context:** Initially labeled the link "View Student Page" on the teacher course view. User pointed out that `/course/[slug]` is a public page visible to everyone — not student-specific.

**Learning:** `/course/[slug]` is the public course page, not a role-specific page. Labels should reflect what the page *is* ("Course Page"), not who *typically* visits it. This distinction matters as role-aware features blur the lines between who visits which pages.
