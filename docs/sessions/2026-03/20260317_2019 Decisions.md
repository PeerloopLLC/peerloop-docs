# Session Decisions - 2026-03-17

## 1. Use $CLAUDE_PROJECT_DIR for Skill Path Abstraction
**Type:** Strategic
**Topics:** cc-workflow, dual-repo

**Trigger:** All 11 r-* skill files had hardcoded paths to `/Users/jamesfraser/projects/peerloop-docs` (MacMiniM4-Pro), breaking them on MacMiniM4 (`/Users/livingroom/`).

**Options Considered:**
1. Replace with `$CLAUDE_PROJECT_DIR` env var ← Chosen
2. Use `git rev-parse --show-toplevel` in each script
3. Create a resolver script that detects the docs repo root
4. Use relative paths from cwd

**Decision:** Replace all hardcoded paths with `$CLAUDE_PROJECT_DIR` for the docs repo and `$CLAUDE_PROJECT_DIR/../Peerloop` for the code repo.

**Rationale:** `$CLAUDE_PROJECT_DIR` is already proven in hooks and scripts. It's set by Claude Code to the primary working directory, making it drift-proof. Options 2-3 add complexity; option 4 is fragile when cwd drifts.

**Consequences:** 41 replacements across 11 files. However, discovered that `!` backtick pre-computation does NOT expand this env var — the prose `git -C` instructions work (Claude resolves them at runtime), but the `!` backtick commands need a different fix. This is a known open issue to resolve.

## 2. Keep Global q-* Commands, Don't Delete
**Type:** Strategic
**Topics:** cc-workflow

**Trigger:** Cleanup plan initially included removing all 19 q-* commands from `~/.claude/commands/`.

**Options Considered:**
1. Delete all q-* commands (they're replaced by r-* skills in Peerloop)
2. Keep all q-* commands (they serve other projects) ← Chosen
3. Selectively delete only Peerloop-specific ones

**Decision:** Keep all 19 q-* global commands. They are used by older projects that haven't adopted the r-* skill format.

**Rationale:** Global commands affect all projects on the machine. The q-* commands are harmless in Peerloop (the r-* skills take precedence), and deleting them would break other projects.

**Consequences:** Both q-* commands and r-* skills coexist. In Peerloop, the r-* skills are invoked directly. In other projects, the q-* commands remain available.

## 4. Strip All Skill Permissions from settings.local.json
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** settings.local.json had stale `Skill(q-*)` and `Skill(w-*)` permission entries that no longer matched active skill names.

**Options Considered:**
1. Replace old names with new r-* names
2. Strip all skill permissions and rebuild organically ← Chosen

**Decision:** Remove all `Skill(*)` entries from the allow-list. Permissions will be re-added as the user approves skills during normal use.

**Rationale:** User preferred a clean slate. Since this file is gitignored and machine-specific, it's easy to rebuild. Avoids guessing which skills will actually be used.

**Consequences:** First invocation of each r-* skill will prompt for permission. Once approved, it's remembered for future use.

## 5. Rename session-files-*.sh to conv-files-*.sh
**Type:** Implementation
**Topics:** cc-workflow, docs-infra

**Trigger:** Consistency with Conv terminology. Scripts named `session-files-dev.sh` and `session-files-learn-decide.sh` were the only "session" references remaining in the scripts directory.

**Options Considered:**
1. Leave as-is (cosmetic, low priority)
2. Rename to conv-files-*.sh and update skill references ← Chosen

**Decision:** Rename both scripts and update the r-dump and r-learn-decide skill files that reference them.

**Rationale:** These scripts are committed to git (not gitignored), so both machines will see the rename on next pull. Consistent naming prevents confusion about whether something is part of the old or new system.

**Consequences:** Old scripts deleted, new scripts created with identical logic (only comment text changed). Skill references updated from `session-files-dev.sh` to `conv-files-dev.sh` and `session-files-learn-decide.sh` to `conv-files-learn-decide.sh`.
