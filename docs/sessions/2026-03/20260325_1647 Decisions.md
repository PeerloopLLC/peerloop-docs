# Conv Decisions - 2026-03-25

## 1. Teacher Course Route: Activity Namespace
**Type:** Implementation
**Topics:** astro

**Trigger:** TEACHER-COURSE-VIEW block requires a route for per-course teacher detail page.

**Options Considered:**
1. `/teaching/courses/[courseId]` — activity namespace, consistent with `/teaching/*` routes ← Chosen
2. `/course/[slug]/teach` — resource namespace, hangs off the public course URL

**Decision:** Use `/teaching/courses/[courseId]` in the activity namespace.

**Rationale:** Keeps the teacher in their workspace. All other teacher routes use `/teaching/*`. Auth model is simpler (everything under `/teaching/` requires teacher auth). Uses courseId directly (no slug→id lookup). Breadcrumb is natural: Teaching → Course Name.

**Consequences:** New directory `src/pages/teaching/courses/` with `[courseId].astro`. API under `/api/teaching/courses/[courseId]`.

## 2. Dedicated Teacher Resources API (Not Shared)
**Type:** Implementation
**Topics:** d1, auth

**Trigger:** Teacher Resources tab needs access to course materials, but existing `/api/courses/[id]/resources` requires student enrollment.

**Options Considered:**
1. Link-through — Resources tab just links to the student page (zero code)
2. Extend existing API — add teacher cert check alongside enrollment check
3. Dedicated endpoint — `GET /api/teaching/courses/[courseId]/resources` with its own auth ← Chosen

**Decision:** Create a dedicated teacher resources endpoint.

**Rationale:** User expects the teacher resources interface to diverge from the student view over time. A dedicated endpoint allows independent evolution (e.g., download counts visible to teachers, future resource management). Isolates teacher auth from student auth.

**Consequences:** New file `src/pages/api/teaching/courses/[courseId]/resources.ts`. Includes module grouping with `module_order` sorting and `download_count` field not present in student API.

## 3. ROLE-AWARE-PAGE-FEATURES Block
**Type:** Strategic
**Topics:** astro, workflow

**Trigger:** While building Course Links section on teacher course view, realized that contextual role-based links could apply to every page — not just the teaching view.

**Options Considered:**
1. Add ad-hoc links per page as needed
2. Create a dedicated PLAN block for systematic role-aware features across all pages ← Chosen

**Decision:** Created ROLE-AWARE-PAGE-FEATURES block in PLAN.md with audit, component design, per-page implementation, and cleanup sub-blocks.

**Rationale:** Systematic approach avoids inconsistency. Every page should be evaluated for what each role (teacher, creator, admin, student, visitor) might want to do when viewing it. A shared component pattern prevents per-page bespoke solutions.

**Consequences:** New block in PLAN.md (PENDING). Will be tackled after TEACHER-COURSE-VIEW is complete.

## 4. Remove Redundant Course Links, Keep Only "View Course Page"
**Type:** Implementation
**Topics:** astro

**Trigger:** Course Links section originally had three buttons: View Student Page, Course Feed, Course Resources. Feed and Resources are now native tabs on the teacher course view.

**Decision:** Remove Course Feed and Course Resources buttons. Keep only "View Course Page" (renamed from "View Student Page") with `?via=teaching&cid={courseId}` for breadcrumb context.

**Rationale:** The tabs already provide Feed and Resources in-context. The Resources link would 403 anyway (enrollment-gated). "View Course Page" is the only genuinely useful cross-link — lets the teacher preview what students/visitors see.

**Consequences:** Single link in Course Links section. Breadcrumb on public course page reads Teaching → Course Name → Course Page.
