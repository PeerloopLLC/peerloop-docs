# Conv Learnings - 2026-03-25

## 1. Component Reuse for Cross-Role Dashboards
**Topics:** react, architecture, dashboard

**Context:** Building a unified dashboard that shows content from 3 separate role dashboards

**Learning:** The existing dashboard components (DashboardStatCard, EarningsOverview, QuickActionButton, MyFeeds, CertificatesSection, AvailabilityQuickView, TeacherCertifications, CreatorCourseCard) were all composable enough to use unchanged inside new wrapper components. Only EnrollmentCard needed extraction from its parent (it was defined inline in StudentDashboard). This validates the original component design as reuse-friendly.

---

## 2. Activity-First vs Role-First Information Architecture
**Topics:** ux, dashboard, information-architecture

**Context:** Designing a unified dashboard for users who can hold multiple roles simultaneously

**Learning:** Grouping by activity type (Schedule, Needs Attention, Courses, People, Earnings) rather than by role (Learning tab, Teaching tab, Creating tab) enables cross-role insights that tabs can't provide. The key example: "Needs Your Attention" merges pending actions from teacher AND creator dashboards — homework reviews appear in both, and without a unified view a creator-who-teaches must check two separate dashboards. The trade-off is needing role badges on every item to maintain context.

---

## 3. Asymmetric Data Loading Patterns Enable Efficient Unified Views
**Topics:** performance, api, currentuser

**Context:** The three dashboards have fundamentally different data sources — student uses CurrentUser cache (zero API cost), teacher and creator each have composite API endpoints

**Learning:** This asymmetry actually helps unified loading. The student role's data is already in the CurrentUser singleton (loaded by AppNavbar on every page), so a unified dashboard only needs to fire 0-2 additional API calls for teacher/creator data. Conditional Promise.all based on role means a student-only user pays zero extra API cost.
