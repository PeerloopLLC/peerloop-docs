# Conv Learnings - 2026-03-17

## 1. Test Date Construction Must Be Explicitly UTC
**Topics:** testing

**Context:** 5 test failures surfaced in the GitHub PR merge check for jfg-dev-7-fix → main. Tests passed locally but failed on CI (GitHub Actions, UTC timezone).

**Learning:** `setHours()`, `new Date(year, month, day)`, and `getDate()` silently use the machine's local timezone. On a dev machine in US Eastern, 10:00 local falls within a teacher's 09:00-17:00 ET availability window. On CI (UTC), 10:00 local = 10:00 UTC, which is before the window starts at 13:00 UTC. The test and endpoint were "wrong in the same direction" locally, masking both bugs.

**Pattern:**
```typescript
// BAD — timezone-dependent
const d = new Date();
d.setHours(10, 0, 0, 0);

// GOOD — explicitly UTC
const d = new Date();
d.setUTCHours(10, 0, 0, 0);

// BEST — use test helper
import { nextDayOfWeekUTC } from '@test-helpers';
const nextMonday = nextDayOfWeekUTC(1, 15); // Monday 15:00 UTC
```

## 2. Boundary Values in Time-Dependent Tests Are Fragile
**Topics:** testing

**Context:** Session cancel tests seeded data at `datetime('now', '+1 day')` — exactly 24h from now. The late-cancel guard triggers at < 24h. By the time the DELETE handler ran (milliseconds later on CI's slower runner), the session was 23.999h away, tripping the guard.

**Learning:** Never seed test data at the exact boundary of a time-based guard. Use a comfortable margin (e.g., +2 days for a 24h boundary). Execution time, CI speed, and system clock granularity all contribute drift that can cross tight boundaries.

## 3. Availability Endpoint Date Iteration Must Be Fully UTC-Consistent
**Topics:** d1, testing

**Context:** Changing `setHours(0,0,0,0)` to `setUTCHours(0,0,0,0)` in the availability endpoint fixed the range boundaries but broke the date iteration loop. `getDay()` (local) and `toISOString()` (UTC) disagreed at UTC midnight in negative-offset timezones — `getDay()` returned Sunday while `dateStr` said Monday.

**Learning:** When iterating dates in server-side code, ALL operations must use the same timezone. Mixing `getDay()` (local) with `toISOString()` (UTC) causes off-by-one day errors near midnight. Use `getUTCDay()`, `setUTCDate()`, `getUTCDate()` consistently. Also: `new Date(dateStr + 'T00:00:00')` parses as local time; use `'T00:00:00Z'` for UTC.

## 4. Clock Injection Pattern for Testable Time Logic
**Topics:** testing

**Context:** Endpoints using `new Date()` for time-sensitive decisions (join windows, late-cancel checks) are untestable without controlling the clock. Tests had to use margin-based workarounds (+2 days) instead of testing exact boundary behavior.

**Learning:** A simple `getNow()` function in `src/lib/clock.ts` that wraps `new Date()` gives tests a clean mock point. Tests can freeze time with `vi.mock('@/lib/clock', () => ({ getNow: () => new Date('2026-03-20T12:00:00Z') }))` to test exact boundary conditions without timing fragility.

## 5. Test State Leakage Across Describe Blocks
**Topics:** testing

**Context:** message-lifecycle.test.ts had 2 failing tests. The "Cross-User Isolation" block assumed `conversations[0]` was Alice-Bob, but the previous "Multi-Conversation" block had created Alice-Carol, which sorted first by `updated_at DESC`.

**Learning:** Never assume array ordering from API responses when tests share state across describe blocks. Filter by a stable identifier (participant ID, known field) rather than relying on array index. This is especially fragile when earlier test blocks create additional data that changes sort order.

## 6. Skill Pre-Computed Context Breaks When CWD Drifts
**Topics:** cc-workflow, dual-repo

**Context:** `/r-end` failed because `!`.claude/scripts/dual-repo-status.sh`` resolved against cwd, which had drifted to `../Peerloop` after `cd ../Peerloop && npm test` commands. The script exists in the docs repo but the relative path couldn't find it.

**Learning:** In a dual-repo setup where skills run shell commands via `!` backtick syntax, relative paths in pre-computed context are unreliable. CWD drifts throughout a conversation as commands run in the code repo. All 18 script references across 10 skills were affected. Fix: use absolute paths (`/Users/jamesfraser/projects/peerloop-docs/.claude/scripts/...`).
