# Conv Learnings - 2026-03-31

## 1. D1/better-sqlite3 One-Statement-Per-Execution Rule
**Topics:** d1, testing, plato

**Context:** Brian admin INSERT had 2 statements (users + user_stats) in one SqlTopUp step

**Learning:** D1 and better-sqlite3 reject multi-statement SQL strings. SqlTopUp steps must contain exactly one statement each.

**Pattern:** Split compound INSERTs into separate SqlTopUp steps with distinct labels.

---

## 2. Verify Queries Should Use Semantic Predicates, Not ID Exclusions
**Topics:** testing, plato

**Context:** Creator count verify used `id != 'usr-admin'` which broke when Brian admin was added (he also has `can_create_courses = 1`)

**Learning:** Verify queries that exclude specific IDs are fragile. Use semantic predicates like `is_admin = 0` that capture the intent rather than listing exclusions.

---

## 3. Schema Column Name Discovery During SqlTopUp
**Topics:** d1, schema, plato

**Context:** Marcus n8n data required 3 column name corrections: `sort_order` → `module_order`, `stripe_checkout_session_id` → `transfer_group`, `approved_by_user_id` → `recommended_by_user_id`

**Learning:** Always verify column names against `0001_schema.sql` before writing SqlTopUp INSERT statements. The schema is the source of truth — don't assume column names from the SQL seed file, which may use different conventions.

---

## 4. Enrollment JOIN Pattern for SqlTopUp
**Topics:** d1, plato

**Context:** Enrollment-dependent SqlTopUp INSERTs need enrollment_id + assigned_teacher_id

**Learning:** The `DAVID_N8N` / `MARCUS_N8N` JOIN constants provide `e.id`, `e.student_id`, `e.assigned_teacher_id`, and `e.course_id` — all the enrollment fields needed for sessions, transactions, and reviews. Define new constants for new enrollment combinations.
