# Session Decisions - 2026-01-28

## 1. SSR Testing: Extract to Testable Functions
**Type:** Strategic
**Topics:** testing, astro, d1

**Trigger:** 17 .astro pages query D1 directly in frontmatter. This layer is untested and errors silently return empty pages.

**Options Considered:**
1. Extract data fetching to `src/lib/ssr/` functions ← Chosen
2. Use E2E tests to implicitly cover SSR layer
3. Accept the gap for MVP, document for later

**Decision:** Extract SSR data fetching logic to testable functions in `src/lib/ssr/`, then unit test those functions.

**Rationale:** Option A enables fast, isolated unit tests without Astro rendering complexity. Each page's data fetching becomes a pure function: `(db) → data | error`. E2E alone would be slow and wouldn't isolate SSR issues.

**Consequences:** 17 pages need refactoring. New `src/lib/ssr/` directory created. SSR tests added to test suite.

## 2. SSR Errors: Show Error Page, Not Empty Content
**Type:** Strategic
**Topics:** astro, d1

**Trigger:** Current SSR catch blocks silently return empty arrays, causing users to see empty pages with no indication of failure.

**Options Considered:**
1. Keep as-is (empty page on error)
2. Add error boundaries showing error page ← Chosen
3. Discuss later

**Decision:** When D1 is unavailable or queries throw, show an error page instead of empty content.

**Rationale:** Empty pages are confusing UX. Users can't tell if there's no data or if something broke. Error pages are honest and actionable.

**Consequences:** Layouts need error boundary support. SSR tests must verify error handling behavior.

## 3. E2E Scope: 3 Critical Flows Only
**Type:** Strategic
**Topics:** testing

**Trigger:** Defining E2E test scope for MVP.

**Options Considered:**
1. Critical 2-3 flows only ← Chosen
2. One flow per user role (student, teacher, creator, admin)
3. Defer E2E entirely to post-MVP

**Decision:** Write 3 critical E2E flows: (1) Browse→Enroll, (2) Auth→Dashboard, (3) Admin Overview.

**Rationale:** E2E tests are expensive to write and maintain. Focus on flows that touch multiple layers and provide confidence the stack works. Comprehensive E2E coverage would be maintenance burden without proportional value.

**Consequences:** Teaching, creator, and moderator flows deferred. E2E remains lightweight.
