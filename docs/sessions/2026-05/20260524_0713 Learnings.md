# Conv Learnings - 2026-05-24

## 1. Figma `<instance>` nodes are the design system's import boundaries
**Topics:** matt-design-system, design-translation

**Context:** Conv 184 audit showed SocialPost.tsx and other Conv 175 components inlined duplicates of components that Matt drew as separate primitives. The audit also revealed CourseAnchor — built earlier same conv — had inlined the Button styling instead of importing Button.tsx.

**Learning:** Matt's Figma `<instance>` nodes are explicit reuse boundaries — he chose to compose, not inline. When translating any frame/page/component into code, every `<instance>` in the metadata should map to an imported code component, not inline markup. If the matching code component doesn't exist yet, stop and surface the gap — don't proceed with an inline duplicate "for now." This is now a standing rule (`feedback_reuse_existing_components.md`).

**Pattern:** Before building any Figma render, call `get_metadata` and audit every `<instance name="…">`. Each name maps to an existing component or a missing primitive that blocks the build.

## 2. Astro can import React but not the reverse
**Topics:** astro, react, architecture

**Context:** Conv 184 initially built leaf primitives as Astro (.astro), but SocialPost.tsx (React) needed to import them — and React can't import Astro. The asymmetry forced a conversion of all 6 Conv 184 primitives + MattIcon to React (.tsx).

**Learning:** Astro renders imported React components as static HTML by default (no hydration cost unless `client:*` directive is added). React, however, has no native way to render Astro components. So when a primitive may be consumed from BOTH Astro and React contexts (typical for design-system leaf primitives), build it in React. Reserve Astro for page-wrappers and shells where Astro's slot composition + static-first rendering shines. Pattern: **primitives in React (broadest reusability), page-wrappers in Astro**.

**Pattern:** `<Button variant="course" href={href}>{children}</Button>` works identically in an `.astro` file and a `.tsx` file. The Astro caller passes `className` (React prop name), not Astro's `class`.

## 3. Strict-B mirroring catches silent bugs as a side-effect
**Topics:** matt-design-system, strict-b

**Context:** The Conv 175 `SubNav.astro` stub had `active?: boolean` per-item, but neither caller passed it — the active-tab indicator never fired. Bug was latent for ~9 convs because no `/matt/` page exercised a selected-tab baseline.

**Learning:** Refactoring to a strict-B-mirrored API (e.g., switching from caller-precomputed `active` to container-derived `currentPath`) surfaces latent bugs that were invisible until the API changed. Patterns that "look like they should work" but never get exercised tend to ossify. Forcing a Figma-grounded rewrite re-tests every assumption.

## 4. Figma frame-name patterns are evidence even when not named as components
**Topics:** matt-design-system, design-translation

**Context:** Matt's 9 Post Anchor rows have IDENTICAL inner frame names (`Frame 153/161/152/150`) across content types, but he didn't extract them as a named `AnchorRow` component. The user observation: "the 9 shapes will render different things" — i.e., shared shape, divergent content.

**Learning:** Frame-name reuse is structural evidence of shared shape, but Matt's component-naming convention is the load-bearing signal for "what should be a primitive in code." Per `feedback_tokenize_only_matt_variables.md`: extract what Matt named, don't extract what he didn't. The IconLabelChip exception (extracted despite Matt not naming it) is justified ONLY by the user's explicit "make these reusable" directive — that's the override condition.

## 5. UserIcon is initials, not role-icon
**Topics:** matt-design-system, figma-mcp

**Context:** Conv 184 probe of `1:35` (UserIcon) found it renders 2-letter initials ("GR") inside the 40×40 frame, NOT a role icon. My pre-probe mental model (formed from the Components page screenshot) had assumed it was a role-icon container.

**Learning:** Visual inspection from inventory screenshots is unreliable. Even with a "this is obviously what it is" assumption, probe via `get_design_context` before building. Conv 178 made the same mistake with Button's "3-orthogonal-dimension" claim; Conv 184 just made it with UserIcon. Probe first.

## 6. Matt's `data-name` is the translation key for instance-named primitives
**Topics:** figma-mcp, matt-design-system

**Context:** Conv 184 Social Post probe showed `<div data-node-id="516:15831" data-name="User Icon">` — Matt's component instances expose their library name via `data-name`. Same for `data-name="Entity Pill"`, `data-name="Analytic Count"` in other probes.

**Learning:** `data-name` is Figma's reliable component-instance label even when the implementation's rendered markup is a generic `<div>`. Future Figma renders should grep the design context for `data-name="…"` to identify reuse boundaries. The icon registries (`creator`, `course`, etc.) use `data-name` too, which is why `data-name` (not the visible label or filename) is the canonical translation key per `reference_figma_mcp_behavior.md`.
