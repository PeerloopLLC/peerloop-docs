# Conv Decisions - 2026-05-23

## 1. Matt-namespaced icon registry uses SVG files as source of truth (Vite ?raw glob), not path-extracted TS registry
**Type:** Implementation
**Topics:** matt-design-system, icon-system, astro

**Trigger:** Starting MMP-PH2 with 39 Matt-canonical SVGs ready to port from `.scratch/matt-main/components/icons/`. Needed to choose storage shape — the existing codebase has only one icon-registry precedent (`src/lib/icon-paths.ts` — flat record of inner-path strings).

**Options Considered:**
1. Path-extracted TS registry — hand-transcribe `<path>` content into a `Record<string, IconEntry>` mirroring `src/lib/icon-paths.ts`. Type-safe via `MattIconName` union; matches established codebase pattern. Edit cost: 39 hand transcriptions; re-exports require re-port.
2. SVG-files + Vite `?raw` glob — keep .svg files in `src/components/matt/icons/svg/`, consume via `import.meta.glob('./svg/*.svg', { query: '?raw', ...eager })` ← Chosen. Edit cost: bulk perl-pi on fill, drop new SVGs into dir, build picks up. No compile-time IconName union — runtime lookup + dev-only warn. Future re-exports trivial.
3. Per-file Astro components (`Home.astro`, `Course.astro`, etc.) — 39 explicit components. Verbose; rejected without further consideration.
4. Astro asset import (`import HomeSvg from './svg/home.svg'`) — produces `<img>` or asset URL, loses `currentColor` recoloring. Rejected as architecturally wrong for icons.

**Decision:** Option 2 (B in the question presented to user). File location: `src/components/matt/icons/`.

**Rationale:** Aligned with the Conv 181 tokenize-only-matt-variables / honest-orphan principle — keep Matt's exact artifact (the SVG file) as the canonical thing, don't translate into our format unless we have to. Re-exports from Figma drop straight into `svg/`. Trade-off accepted: no compile-time `MattIconName` union (runtime lookup + dev-only warn) — registry shape is provisional and PH4 empirical re-render will determine whether type-safety is justified.

**Consequences:** New `src/components/matt/icons/` tree under code repo. Vite glob is new for this codebase (existing icons use path-string registry). All future Matt icon drops follow this pattern. Fill normalization is required before commit (Conv 182 perl-pi pattern: `s/fill="#414141"/fill="currentColor"/g` plus a distinct-fill audit step for outliers like `info.svg`'s `#1C1B1F`).

## 2. [CASCADE-BROKEN] closed — Matt's button CSS uses variant-scoped variables, not entity-cascade variables (Conv 178 validation)
**Type:** Implementation
**Topics:** matt-design-system, css-variables, tailwind

**Trigger:** Tying up doc items surfaced #16 [MDS-CASCADE-VALIDATED]. matt-design-system.md §5 Entity still carried Conv 176 caveat with "Root cause unconfirmed" wording. Conv 178 found the root cause.

**Options Considered:**
1. Keep the Conv 176 caveat unchanged; reflect Conv 178 finding only in PLAN.md (avoids large doc edit) — rejected; doc would lie about validation status.
2. Reframe the caveat as a validated finding: cascade is NOT broken; Matt's button CSS just doesn't use the entity-cascade variables ← Chosen.
3. Delete the caveat entirely now that root cause is known (loses historical record) — rejected; the Conv 176 wording is the symptom report and remains accurate.

**Decision:** Keep the original Conv 176 paragraph (historical symptom record) and add a "Conv 178 validation" follow-up paragraph stating the root cause: Matt's button CSS uses `var(--Background)` and `var(--Border)` (variant-scoped variables tied to the button's `variant` prop), NOT `var(--Entity-Background)` / `var(--Entity-Primary)` (the cascading entity variables). The cascade isn't broken — it was never wired into Matt's button design. Retire the "until [CASCADE-BROKEN] resolves" wording; [CASCADE-BROKEN] is closed.

**Rationale:** The Conv 178 finding is architecturally meaningful: the entity cascade is intended only for components Matt explicitly wired with `--Entity-*` variables (entity headers, route-level entity color hints). Multi-variant primitives like Buttons use a parallel-but-independent variant-prop mechanism. This shapes how MMP-PH3 component primitives are built — variant-prop primitives don't need to consume `--Entity-*`, they need their own `--{Primitive}-{Background,Color,Border}` variable cluster scoped to each variant class.

**Consequences:** matt-design-system.md §5 Entity updated. [CASCADE-BROKEN] removed from open-issues mental ledger. MMP-PH3 [MATT-EXEC-CMP-BTN] can proceed with confidence on the variant-prop pattern (matches existing `Button.tsx`).
