## Uncategorized
- **Pre-sync M4 was missing entire `§Dual-Repo Shell Discipline` section, not just stale on it.** When the user asked "did you update it to jamesfraser?", the natural assumption was a content-flip (livingroom → jamesfraser). Reading the backup tarball revealed M4 had no such section at all — M4Pro added it during merge/streamline work. This generalizes: cross-machine drift includes "section absent on machine A" as a failure mode beyond just "section stale on machine A". A sync mechanism that only diffs existing sections would miss new-section additions on the other side.
    → Suggestion: when designing the durable [CMS] solution, account for "additive drift" (new sections / new files) not just "edit drift" (changed lines in existing files). Manual rsync handles both; some cloud-sync mechanisms only handle the latter cleanly.
- **L2 forward-direction script bug surfaced during verification.** Initial bash `for f in $LINKED` loop didn't word-split the multi-line variable correctly — single iteration consumed the entire newline-separated filename list as one `$f`, producing a misleading "all 50 missing" output that contradicted L1's "51 files exist". The reverse-direction (orphan) check passed cleanly via `xargs basename`, which surfaced the contradiction. Fixed with `while IFS= read -r f` from process substitution.
    → Suggestion: not a bug worth fixing forever — but a reminder that bilateral verification (forward AND reverse) catches each direction's bugs. Single-direction checks can be misleading if the script has its own bug.
