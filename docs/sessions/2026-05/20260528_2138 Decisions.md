## 1. /profile structure: account hub with settings flattened (A)
**Type:** Strategic
**Topics:** astro, cc-workflow

**Trigger:** "the slideout options turn into subNavBar menu items" — but the slideout's "Settings" item opens a 5-page hub (nested), and the public profile is outward-facing.

**Options Considered:**
1. Literal slideout 1:1 (keeps Settings as a nested hub-tab)
2. Account hub, settings flattened ← Chosen
3. Profile-forward merge (public profile as default tab)

**Decision:** `/profile` becomes a flat 6-tab account area (Account / Edit Profile / Interests / Payments / Notifications / Security); the `/settings` hub is flattened into siblings; public `/@handle` stays separate; Help/Dark-mode/Sign-out are Account-tab actions.

**Rationale:** Matches the course flat-tab model (no nested menus), separates outward (`/@handle`) from inward (account mgmt), kills the dead SubNav (#29).

**Consequences:** New `profile/[...tab].astro` + `_profile-tabs.ts`; `/settings/*` islands reused; per-tab faithful redesign deferred to #33.

## 4. npm: do not upgrade standalone (keep 10.9.3)
**Type:** Strategic
**Topics:** cc-workflow, dual-repo

**Trigger:** "should we upgrade npm itself on this computer?"

**Options Considered:**
1. Keep 10.9.3 ← Chosen
2. `npm i -g npm@11` on M4
3. Coordinated node bump on both machines (the right path IF upgrading)

**Decision:** No standalone upgrade. 10.9.3 is current, `engines` allows it, and `.nvmrc`-pinned node bundles it on both machines.

**Rationale:** A one-machine global npm upgrade splits the toolchain and risks author-side lockfile churn (the thing `npm ci` just fixed); nvm-global npm upgrades are non-durable.

**Consequences:** No change made; documented the node-bump path for any future upgrade.
