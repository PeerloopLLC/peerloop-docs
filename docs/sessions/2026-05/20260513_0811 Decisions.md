# Conv Decisions - 2026-05-13

## 1. Default `autoStartRecording=true` rather than make caller-opt-in
**Type:** Implementation
**Topics:** bbb, video

**Trigger:** Discovered we never send autoStartRecording; recording gap likely caused by moderator forgetting to click button.

**Options Considered:**
1. Make caller pass `autoStartRecording: true` explicitly per-session
2. Default to `true` in provider + caller; allow opt-out ← Chosen
3. Don't change behavior, just document the gotcha

**Decision:** Three-layer fallback (`options ?? defaults ?? true`) all defaulting to `true`. Provider default `autoStartRecording: true`. Caller `join.ts` explicitly passes `true` for clarity even though defaults would suffice.

**Rationale:** Peerloop sessions are 1-on-1 tutoring; policy is "every session recorded". Relying on moderator click leaks recordings whenever someone forgets. The two-parameter BBB design fits universities (per-session opt-in by instructor); for our use case auto-start matches policy.

**Consequences:** All future sessions auto-start recording. No UI option for moderator to disable auto-start (could be added later via `enableRecording=false` toggle in session-creation form). If a user wants an unrecorded session, they'd need a separate flow — none requested.

## 2. Build account-wide admin endpoint + UI rather than just the per-session diagnostic
**Type:** Implementation
**Topics:** bbb, admin, observability

**Trigger:** Investigation needed a "does Blindside have ANY recordings on our account" answer; existing per-session endpoint can't give that without iterating every session.

**Options Considered:**
1. Just run the diagnostic script ad-hoc when needed
2. Build admin endpoint + UI (count + latest 20) ← Chosen
3. Defer to `[BR-STATUS]` (recording_status column for richer per-session UI)

**Decision:** Built `/api/admin/bbb/recordings` endpoint, `/admin/recordings` page, `RecordingsAdmin.tsx` component, and AdminNavbar entry. The script (`scripts/bbb-list-recordings.mjs`) also kept for command-line use.

**Rationale:** Recording issues will recur (vendor configuration drift, webhook regressions, server outages). Having admin UI lets non-CC users (Brian, future admins) diagnose recording state without running scripts or asking engineering. Durable infrastructure. Cost was ~250 LOC across 4 new files; well-bounded.

**Consequences:** New admin nav entry visible to all admins. Endpoint queries BBB live on every request — could add caching later if used heavily but currently fine. Latest-20 cap is arbitrary; can be widened if needed.

## 3. Establish `.scratch/` as a persistent gitignored workspace convention
**Type:** Strategic
**Topics:** docs-infra, cc-workflow

**Trigger:** User explicitly requested a place to park files (draft emails, etc.) that survive past conversation scrollback but aren't worth committing.

**Options Considered:**
1. Use `/tmp/` for scratch (ephemeral; lost on reboot)
2. Use `.scratch/` gitignored folder at docs-repo root ← Chosen
3. Use Obsidian vault for scratch

**Decision:** `.scratch/` at docs-repo root, gitignored. CC may write freely without further approval. Documented in CLAUDE.md (short pointer) + `.scratch/README.md` (detailed conventions).

**Rationale:** User noted this works well in other CC projects. Persistence across conversations is the key property — scrollback lost vs scratch file re-readable. Gitignored prevents accidentally committing drafts/notes/dumps. Local-only matches the "scratch" semantics; mirror sync would clutter both machines.

**Consequences:** New project convention. Future CC has clear pre-authorization to write here without checking. Repo root has one more entry. `.scratch/` doesn't sync between machines (intentional).
