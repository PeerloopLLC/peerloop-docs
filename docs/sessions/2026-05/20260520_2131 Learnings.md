# Conv Learnings - 2026-05-20

## 1. Astro template literals using `${Astro.url.search}` confuse route-extraction regexes
**Topics:** astro, tooling

**Context:** Conv 166 added `src/pages/course/[slug]/[tab].astro:83` with `Astro.redirect(\`/course/${slug}${Astro.url.search}\`)` to preserve query string on role-gate redirect. `scripts/route-matrix.mjs` extracted this as a link target, but its generic `${...}` → `[param]` fallback in `normalizeDynamic` produced `/course/[slug][param]` — a broken-looking pseudo-route (no slash separator) that flagged a phantom broken target.

**Learning:** Template-literal route extractors must distinguish path-segment params (`${slug}`, `${course.slug}`, `${otherParticipant.handle}`) from runtime appenders (`${Astro.url.search}`, `${searchParams.toString()}`) that aren't route segments at all. A blanket "strip on dot" heuristic would break legitimate property-access params. Targeted name-based exclusion (`url.search`, `searchParams`) is more precise.

**Pattern:** When scanning JSX/Astro templates for route literals, blacklist known query-string sources before applying generic param normalization:
```js
function normalizeDynamic(path) {
  return path
    // Drop query-string appenders — not route segments.
    .replace(/\$\{[^}]*url\.search[^}]*\}/gi, '')
    .replace(/\$\{[^}]*searchParams[^}]*\}/gi, '')
    // ... then existing slug/handle/id/generic rules
}
```

## 2. Heuristic-only static checks for "wrong column on wrong table" are very prone to false positives
**Topics:** tooling, testing

**Context:** /w-codecheck Check 8 (schema-aware deleted_at) was added per Conv 117 to catch SQL blocks applying `deleted_at IS NULL` to tables that don't have the column. The Conv 167 baseline run surfaced 90+ findings across 30+ files. Spot-checks confirmed ALL false positives: every flagged block correctly applies the filter to a table that DOES have `deleted_at` (typically `courses`, `enrollments`, `users`), and merely mentions other tables in joins/subqueries that use their own filters (`is_active = 1`, etc.).

**Learning:** Static checks based on "is this table name present in the same SQL block as this column name?" are too coarse for SQL — schema and grammar interact. The valid check is "is the column being applied to a table that doesn't have it?", which requires either: (a) qualified-column matching (`<table>.deleted_at`), (b) parsing the SQL grammar to associate column refs with their owning table, or (c) requiring the codebase convention to always qualify column references with table aliases. Per `memory/feedback_heuristic_calibration.md`: any new detection heuristic MUST be re-validated against the motivating case before commit. Check 8 was never validated against a real motivating Conv 117 sample with both qualifying and non-qualifying tables in the same block.

**Pattern:** When adding a static-check heuristic, write a fixture of canonical positives and negatives BEFORE shipping. Run the check against the fixture and confirm the expected hit pattern. A check that fires 90 times in a clean codebase is not flagging real issues — it's flagging itself as broken.

## 3. ESLint `ignores: ['.astro/**']` does NOT match nested `.astro/` dirs without `**/` prefix
**Topics:** tooling

**Context:** [CCK-LINT] fix discovered the existing ignore pattern `.astro/**` was matching only root-level `.astro/`, not the auto-generated `src/pages/api/communities/.astro/content.d.ts` that Astro's content sync produces nested under API routes. ESLint glob ignores require explicit `**/` prefix to recurse.

**Learning:** ESLint flat-config `ignores` patterns are NOT auto-recursive. To exclude a directory at any depth, prefix with `**/` (e.g., `**/.astro/**`). The same rule applies to dist outputs that appear at multiple depths.
