# Conv Decisions - 2026-05-20

## 1. CRT-4: single shared AllSessionsTabContent component, not three role-specific variants
**Type:** Implementation
**Topics:** frontend, architecture

**Trigger:** Three role groups (CREATOR / ADMIN / MODERATOR) all consume identical `scope=all` data and need the same UI affordances. The CRT-3 pattern (TeacherSessionsTabContent — scope=teacher) was one component per scope, but extending that for 3 more roles where the SCOPE is identical (only the group label differs) would be 3 nearly-identical files.

**Options Considered:**
1. Three components — CreatorSessionsTabContent / AdminSessionsTabContent / ModeratorSessionsTabContent
2. One shared AllSessionsTabContent rendered in 3 different group labels ← Chosen
3. Single tab placed under highest-precedence role (creator > admin > moderator)

**Decision:** Single shared component, rendered in 3 separate groups via the role flags. Distinct tab IDs (`creator-sessions` / `admin-sessions` / `moderator-sessions`) preserve URL routing per role.

**Rationale:** Data is identical for all three (`scope=all`); UI affordances (sections, row design with both student + teacher) are identical. Three components would diverge over time for cosmetic reasons. Multi-role users (rare: e.g., creator who is also admin) see multiple groups with the same "All Sessions" tab — acceptable cost for one-component upkeep.

**Consequences:** Multi-role users see redundant tabs (CREATOR: All Sessions, ADMIN: All Sessions) — not ideal but rare in practice (verified seed data has Brian as admin, Guy as creator — disjoint).

## 2. CRT-DEDICATED-PAGES: single dynamic `[tab].astro` catch-all, not four cloned files
**Type:** Strategic
**Topics:** astro, routing, architecture

**Trigger:** Manual refresh of `/course/<slug>/teaching-sessions` (or the new creator/admin/moderator URLs) 404s because no `.astro` page exists for those routes. Two routing patterns considered.

**Options Considered:**
1. Clone `sessions.astro` per extra tab — 4 new files × ~170 lines each = ~680 LOC duplication, matches existing codebase pattern of one-file-per-route
2. Single dynamic `[tab].astro` catch-all with role-flag access gate ← Chosen

**Decision:** Single catch-all. Whitelist maps tab ID → required role flag; access gate redirects to /404 (unknown tab) or /course/<slug> (lacks role).

**Rationale:** Durable per CLAUDE.md §Solution Quality — adding future role tabs is a 1-line mapping entry, not a new file. Static-route precedence means existing 7 static .astro files are untouched. The Critical Rule check noted this is a novel routing pattern for the codebase (no existing `[param].astro` under `/course/[slug]`), but the trade-off is clear and the user invoked CRT-DEDICATED-PAGES knowing both options were on the table from CRT-5 deferral.

**Consequences:** All 4 role-tab URLs now load correctly on manual refresh / shared bookmark. Future role tabs require only `roleTabMap` + `tabLabels` entries.

## 3. CRT-STUDENT-EXPLICIT-SCOPE: extended fix scope beyond named site
**Type:** Implementation
**Topics:** frontend

**Trigger:** Task wording named only `SessionsTabContent` fetch, but `ResourcesTabContent.tsx:71` had the identical dual-role-bug shape (`isEnrolled`-gated student-scope fetch that defaults to teacher precedence for dual-role users).

**Options Considered:**
1. Fix only the named site (CourseTabs.tsx:131) — leaves the same bug on the student Resources tab
2. Fix both sites (CourseTabs + ResourcesTabContent) ← Chosen

**Decision:** Both sites updated.

**Rationale:** Identical bug shape, identical fix, no design ambiguity. Leaving one site unfixed would just re-surface as a follow-up.

**Consequences:** Dual-role bug closed on both student-scope tabs (Sessions + Resources past-sessions section). Existing TeacherSessionsTabContent precedent (`scope=teacher`) confirms the pattern.
