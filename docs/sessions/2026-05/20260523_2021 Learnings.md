# Conv Learnings - 2026-05-23

## 1. Hex fill audit before bulk-rewrite catches Material-Icons paint defaults
**Topics:** matt-design-system, icon-system, figma-export

**Context:** Bulk perl-pi rewrite of `fill="#414141"` → `fill="currentColor"` left 38/39 SVGs normalized; one outlier (info.svg) had `fill="#1C1B1F"` (Material Design 3 `on-surface` default).

**Learning:** When normalizing fills across an icon set exported from Figma, run a distinct-fill audit before assuming a single hex value covers all icons. Direct Material Icons exports may carry `#1C1B1F` (MD3 on-surface) instead of the designer's chosen color. The audit pattern: `for f in *.svg; do for fill in $(grep -oE 'fill="[^"]*"' "$f" | sort -u); do case "$fill" in 'fill="currentColor"'|'fill="#D9D9D9"'|'fill="none"') ;; *) echo "$f: $fill" ;; esac; done; done`.

**Pattern:** For each new Figma export drop into `src/components/matt/icons/svg/`, run the distinct-fill audit before commit. If any non-`currentColor`/`#D9D9D9`/`none` fills surface, normalize them OR file a question for the designer if the variant fill might be intentional.

## 2. Vite `import.meta.glob` with `?raw` query produces typed `Record<string, string>` for inline-SVG patterns
**Topics:** astro, vite, icon-system

**Context:** Chose SVG-files-as-source-of-truth (option B) over path-extracted TS registry (option A) for the Matt-namespaced icon registry. Need to inline SVGs at build time with zero client JS.

**Learning:** `import.meta.glob<string>('./svg/*.svg', { query: '?raw', import: 'default', eager: true })` returns a `Record<string, string>` (path → raw text) at build time. Combined with a regex to strip the outer `<svg>` tag (`source.match(/<svg[^>]*>([\s\S]*)<\/svg>\s*$/)?.[1]`), this gives you inline-able body content that the Astro component can wrap with new `<svg class={...} viewBox={...} set:html={inner} />`. Build resolves the glob; no runtime cost.

**Pattern:** For future re-exports, drop the SVG file into `src/components/matt/icons/svg/` and run the fill-normalization step. Build picks it up automatically — no registry edits required.

## 3. Astro `interface Props` triggers ts(6196) "declared but never used" when there's no type-export anchoring it
**Topics:** astro, typescript

**Context:** First astro-check after writing MattIcon.astro produced 1 hint: `'Props' is declared but never used` (ts(6196)). Existing `Icon.astro` uses the same `interface Props { ... }` pattern but doesn't trigger the hint — diagnosed why.

**Learning:** Astro's special-casing of `interface Props` for `Astro.props` typing isn't automatic — tsc still needs to see Props referenced somewhere. The existing `Icon.astro` has `name: IconName | string` referencing the `IconName` exported type, which keeps Props "alive" through type composition. My first version had `export type MattIconName = keyof typeof innerByName` BEFORE the Props interface, but the export was useless (`innerByName` is `Record<string, string>` so `keyof` resolves to `string`) AND it confused tsc's usage analysis. Fix: removed the useless export type; added `Astro.props as Props` cast to satisfy the usage check.

**Pattern:** When writing a new `.astro` component, if `Props` isn't referenced through type composition elsewhere, add `as Props` to the `Astro.props` destructuring line.

## 4. Figma MCP `get_variable_defs` is selection-bound — implication for design-system probing
**Topics:** figma-mcp, design-tokens

**Context:** Conv 181 discovered the two-tool-class distinction. While graduating the MCP doc, surfaced the consequence: token-verification ("does Variable X exist?") has no direct MCP call.

**Learning:** Because `get_variable_defs` returns only Variables consumed by the selected node, AND `get_libraries`/`search_design_system` enumerate subscribed libraries (not local file Variables), there is NO MCP tool that enumerates a file's full local Variable collection. The workaround is the **visual-absence heuristic**: if no layer in any page uses red/pink, `--carmine-red` is effectively absent regardless of whether the Variable record technically exists. This is the basis for `feedback_tokenize_only_matt_variables.md` — don't invent tokens for values that don't probe positive in `get_variable_defs`.

**Pattern:** When asked "does Matt have a token for X?" the answer must come from `get_variable_defs(<node that uses X>)`, not from `get_libraries` or `search_design_system`. If no node uses X, the answer is "effectively absent" — file the question for the designer rather than inventing a speculative token.

## 5. Doc graduation pattern: scratch (staging) → memory (recall-shorthand) → docs/reference (canonical)
**Topics:** docs-infra, knowledge-management

**Context:** [MCP-DOC] graduated `.scratch/figma-mcp-setup.md` + `memory/reference_figma_mcp_behavior.md` content into `docs/reference/figma-mcp.md`.

**Learning:** Empirical discoveries about external tools (vendors, MCPs, SDKs) flow through three stages: (1) Scratch — walkthrough or investigation file during the discovery conv (2) Memory — rule-shorthand encoded for fast cross-conv recall during work (3) Docs/Reference — canonical reference doc once the empirical findings stabilize. The memory file stays after graduation (it's the recall-shorthand version); the scratch source goes (its content is now in git via the reference doc). Both memory and docs/reference can survive in parallel — they serve different reader contexts (Claude during work vs. human reading reference).

**Pattern:** When empirical findings about an external system stabilize (after 2-3 convs of discovery), graduate them: write the docs/reference doc, delete the scratch source, keep the memory file. Cross-link the memory file at the bottom of the docs/reference doc under "Related".
