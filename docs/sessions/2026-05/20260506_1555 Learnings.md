## 1. CLAUDE.md asymmetric rule placement was best resolved by promoting, not by inversion
**Topics:** cc-workflow, docs-infra

**Context:** [ARP] watch task tracked the asymmetry that 🔴/🟠 Issue Surfacing lived in CLAUDE.md while 👉 pointing-emoji + A) B) C) option-phrasing lived in memory with one CLAUDE.md cross-reference. The original revisit trigger ("cross-ref cost grows") had not fired — only one CLAUDE.md site referenced both memory rules together.

**Learning:** When the trigger hasn't fired, three options open up — (A) leave as-is, (B) consolidate to CLAUDE.md, (C) move CLAUDE.md rule out to memory. Option B was chosen because the §Issue Surfacing pattern was already proven (Conv 150 baselines consolidation) and the cost was modest (37 lines). Option C would have been load-bearing-rule-friendly but risky — §Issue Surfacing is referenced by skills, agents, and `/r-end` in critical paths; making it grep-recall-only would degrade reliability for marginal aesthetic gain. The pattern: when consolidating output-formatting rules, promote into CLAUDE.md (memory becomes stub), do not invert.

**Pattern:** New CLAUDE.md section §User-Facing Questions placed adjacent to §Issue Surfacing; both memory files reduced to stub form using the proven `feedback_visual_issue_alerts.md` template; the still-memory-resident `feedback_pause_on_pointing_questions.md` paired sibling stays in memory because it's *behavior* (sequencing) not *format* — that distinction held up under the consolidation.

## 2. MEMORY.md index lines have a second failure mode beyond softness — drift
**Topics:** cc-workflow, docs-infra

**Context:** [ILS] codified the rule that index lines must surface load-bearing markers/triggers/anti-patterns rather than topic labels. Audit pass found `e2e-testing-patterns.md` whose index line claimed three sub-rules — "BBB webhook HMAC token; filter-dropdown caveats; `waitForLoadState('networkidle')`" — but the file body only had two ("React Hydration" + "Cross-Test DB Contamination"). The drift was severe enough that any future Claude grepping MEMORY.md for "HMAC" or "filter-dropdown" would see hits, click through, and find nothing.

**Learning:** Soft summaries are one failure mode (Conv 151 ILS canonical case: "Bold questions" eliding 👉). Drift is a separate, more damaging failure mode where the index claims rules that aren't (or no longer are) in the body. The fix is the same discipline applied at write-time — after every memory file edit, re-read MEMORY.md and reconcile — but the prevention story differs: soft summaries come from laxity at the index, drift comes from updating the body without updating the index.

**Pattern:** "Index-vs-body drift discipline" added to `feedback_memory_index_load_bearing.md` as a second section. Process rule: after Edit/Write to any `memory/` file, immediately Read MEMORY.md and check the index line; reconcile token additions, deletions, or rule removals. The check is short — read body headings, read index line, ask "does the line still describe what's in the file?"

## 3. Watch-tasks failed silently because their framing hid implicit preconditions
**Topics:** cc-workflow, dual-machine

**Context:** [WAS] codified the rule that watch-tasks must state assumed delivery/load preconditions in the subject. The Conv 149→150 [OPW] case took an entire conv plus an Explore agent to falsify "is the file even on this machine?" because the watch-task framing was just "watch strengthening over ~5 convs."

**Learning:** A watch-task is implicitly two assertions: the watched outcome AND the precondition that the rule/fix is in scope on the observing machine. Without the second part stated, the audit at watch-end debates content when it should be falsifying preconditions. The recurring failure mode: rule absence looks indistinguishable from rule weakness when the watcher can't see whether the rule is loaded. The fix is purely framing — add `— assumes Y is loaded on machine Z` to the task subject — but it's load-bearing because TodoWrite/RESUME-STATE entries are what reload across convs. If the assumption isn't there, future-Claude won't think to check it.

**Pattern:** Six common implicit-assumption categories captured: file presence on machine; MEMORY.md indexing (orphan files); fix-commit deployment; cache invalidation; session-start hook success; behavior-path coverage. The format `[CODE] Watch X over N — assumes Y` was chosen because the assumption clause is what survives across `/r-start` transfers.

## 4. Three meta-rules surfaced from one conv's memory work share a common root
**Topics:** cc-workflow, docs-infra

**Context:** This conv codified three meta-rules that had been queued as watch-tasks: index lines must be load-bearing ([ILS]), heuristics must be tested against motivating cases ([CMH]), watch-tasks must state preconditions ([WAS]). Each was independently surfaced in earlier convs and queued for capture.

**Learning:** All three rules share one structural feature — they're about **closing the loop between the rule and the case that motivated it**. Index lines must surface the marker that distinguishes the rule (the "case" being the rule's activation pattern). Heuristics must be calibrated against the case the memo cites (the "case" being the bug the heuristic catches). Watch-tasks must check the precondition (the "case" being whether the rule is even in play). The shared root: rules without case-anchoring degrade silently. The three memories together form a corpus for case-anchoring discipline at three layers of the cc-workflow stack.

**Pattern:** When proposing a new rule, ask three questions in order: (1) Does the rule have a named motivating case? (no → demand one). (2) Will the index expose the case's distinguishing marker? (no → load-bearing rewrite). (3) Could a watcher of the rule check whether the rule is even applicable on the machine in question? (no → state preconditions). Failing any of the three produces silent-decay rules.
