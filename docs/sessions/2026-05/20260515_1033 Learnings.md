# Conv Learnings - 2026-05-15

## 1. Blindside getRecordings requires `limit=N` parameter (≤100)
**Topics:** video, bbb

**Context:** Conv 159's [BR-DIAG] reported 0 recordings via `getRecordings` with no params — concluded BBB wasn't producing recordings. Conv 161 Blindside reply revealed unbounded requests are blocked (would otherwise return thousands). The 0-recordings response was a silent false negative.

**Learning:** `getRecordings` MUST be called with `limit=N` where N ≤ 100. Without it, response has empty `<recordings>` element with no error. With `limit=500`, returns `validationError: "limit parameter is out of range"`. The cap is between 100 and 500 (not probed precisely).

**Pattern:** Both `scripts/bbb-list-recordings.mjs` and `src/lib/video/bbb.ts:445 getRecordings()` now pass `limit=100`. Provider's per-room `getRecordings(roomId?)` still uses hardcoded 100 since per-room sessions never exceed 1-2 recordings. Account-wide admin uses the new `listAllRecordings({limit, offset})` method with caller-specified limit.

## 2. Blindside supports `offset` on getRecordings (non-standard BBB extension)
**Topics:** video, bbb

**Context:** Building admin pagination on `/admin/recordings`. Standard BBB getRecordings spec doesn't include offset — only `meetingID`, `recordID`, `state`, `meta_*`. Blindside has implemented their own pagination.

**Learning:** `offset=N` works alongside `limit=N` on Blindside's getRecordings. Probed at offsets 0/2/4 with limit=2 — each returned distinct recordIDs matching expected chronological slice. This is Blindside-specific, not portable to other BBB hosts.

---

## 3. Blindside doesn't return a total count — derivation requires a second call
**Topics:** video, bbb

**Context:** Implementing standard paginated response shape (`{items, total, page, limit, totalPages, hasMore}`).

**Learning:** `<recordings>` element in getRecordings response has no aggregate count attribute. The paginated endpoint at `/api/admin/bbb/recordings` makes two parallel BBB calls per request: one for the page (`limit=N&offset=N`) and one for total (`limit=100`). Acceptable cost since admin pagination is infrequent and ~14KB extra per request.

## 4. dist/server/wrangler.json is the source of truth for `wrangler deploy` targeting
**Topics:** cloudflare, astro, deployment

**Context:** `npm run deploy:staging` script reads `CLOUDFLARE_ENV=staging ENVIRONMENT=staging npm run build && wrangler deploy` — no `--env staging` flag on wrangler. The wrangler.toml comment suggested `--env staging` should be there.

**Learning:** `@astrojs/cloudflare` adapter generates `dist/server/wrangler.json` at build time, scoped to whichever env is named in `CLOUDFLARE_ENV`. `wrangler deploy` reads that generated file, not the project's `wrangler.toml`. So `--env` is applied at build time, not at the CLI step. To verify safety, inspect `dist/server/wrangler.json` for `targetEnvironment`, `name`, D1 `database_name`, R2 `bucket_name` — all should be staging-suffixed.

**Pattern:** When auditing any Astro + Cloudflare deploy script, check the generated wrangler.json over the source wrangler.toml.

## 5. Chrome MCP tab has separate cookie store from user's main Chrome
**Topics:** cc-workflow

**Context:** User authed as Brian in their main Chrome window; chrome MCP tab landed on staging login despite same Chrome.app instance. Single browser was connected via `list_connected_browsers`.

**Learning:** Even within the same Chrome.app process, the chrome MCP extension manages its own profile/cookie store that's isolated from the user's normal browsing context. For staging/prod URLs that need auth, the user must log in once per session in the MCP-driven tab. After that, the session persists for the conv.

## 6. /admin/sessions is a single page with click-to-expand side panel (no per-session URL)
**Topics:** astro, admin

**Context:** Built mental model of `/admin/sessions/<id>` route from grep results, then discovered 404 in the browser.

**Learning:** The Astro routing `/admin/sessions.astro` (singular file, no `[id]` dir) is the only route. Session detail is shown via a side-panel that opens when a row is clicked. `SessionsAdmin.tsx:108` maintains `selectedSession` state; row click → fetch `/api/admin/sessions/<id>` → set state → panel opens with `SessionDetailContent`. Recording URL is lazy-fetched via "View Recording" button (`fetchRecording(id)` → `/api/admin/sessions/<id>/recording`).

**Pattern:** Filled circle (●) glyph on the session row indicates session has recording. Hollow (○) means no recording. The marker is visible inline in the row text.

## 7. Empirical UI verification catches "JSX missing" bugs that data-path tracing misses
**Topics:** testing, video

**Context:** [BR-TRACE] code-side investigation showed `TeacherCourseView` declared `recording_url: string | null` in its `CourseSession` type (line 87) and the API confirmed it returns the field. Both data-path checks looked correct. But empirical visit of `/teaching/courses/<id>` Sessions sub-tab showed no recording link rendering — `SessionRow` (lines 729–753) reads avatar/name/module/date/status but never reads `recording_url`. TypeScript can't catch "you declared the type but forgot to use it in render."

**Learning:** Architecture-verified ≠ UI-verified. Type declaration only proves the data CAN be carried — not that the JSX renders it. For any "does this feature appear in the UI" verification, an empirical browser visit is required.

## 8. Course pages currently lack role-aware views
**Topics:** astro, ux

**Context:** Admin Brian visiting `/course/intro-to-n8n/sessions` saw "No sessions yet" empty state because the SSR loader filters by `cookies.user_id` with no role branching. Admin / creator / moderator viewers get the same empty-student view as an unenrolled visitor.

**Learning:** Course-tab pages need role detection + role-appropriate content (per-role tabs or filtered lists). Tracked as `[CRT]` for a future block. Investigation starting points: `fetchCourseTabData` loader (Conv 130 [RA-SSR]), `isUserAdmin`/`getUserPermissionFlags` helpers (Conv 123 [RA-ADM]).

## 9. Three different recording-link affordances across user-facing surfaces
**Topics:** ux, video

**Context:** Walking 6 user-facing surfaces, the recording-link UI varied:
- `StudentSessionsList`, `TeacherSessionsList`, fixed `TeacherCourseView` SessionRow → icon-only `<PlayCircleIcon>` + `title="View recording"` (no aria-label)
- `SessionsTabContent` (course Sessions tab) → text "Recording"
- `ResourcesTabContent` (course Resources tab) → text "Watch"

**Learning:** Each surface evolved its recording link independently. Pick one canonical affordance (likely icon + text + aria-label) and standardise. Tracked as `[REC-LABEL]`.
