# Conv Learnings - 2026-05-22

## 1. Decision walkthrough produces high-quality alignment when paced one-at-a-time
**Topics:** cc-workflow, plan

**Context:** Conv 173 surfaced 8 architectural decisions in §4 of `matt-pre-plan.md` simultaneously. User's first preference was "go through them one at a time" — not batch-approve. Single-shot A/B/C presentation per decision (with full pros/cons table + recommendation) let the user make each call without skimming bias.

**Learning:** When >3 novel architectural decisions surface in a single planning doc, default to one-at-a-time walkthrough over en-bloc approval. The doc still ships with all decisions visible; the walkthrough enforces deliberation. En-bloc approval is for low-novelty execution checkpoints, not foundational architectural picks. The 8-of-8 "recommendations stood — no overrides" outcome doesn't undermine the approach; it confirmed the recommendations were calibrated, but the user got to verify each one in isolation rather than skim a block.

**Pattern:** Pre-plan doc has §4 "Open Decisions" with each decision laid out fully (options, pros/cons, recommendation). At the end of the doc draft, ask `👉👉👉 Approve all en-bloc, or walk through one at a time?` rather than just asking for sign-off. Either path consumes the same content; the walkthrough is durable when the decisions are foundational.

## 2. Pre-plan should explicitly preserve "BLOCKING" status until resolved, then transition to "RESOLVED Conv N"
**Topics:** docs-infra, plan

**Context:** The Conv 173 pre-plan §4 was initially titled "Open Decisions (BLOCKING)" — a strong signal that nothing downstream can proceed. After all 8 decisions resolved with the user, the heading was changed to "Open Decisions (RESOLVED Conv 173)" with a top-of-section resolution-summary table. This makes the historical state visible (decisions WERE blocking, AND were resolved this conv) without losing the structural integrity of the doc.

**Learning:** Pre-plan docs are time-staged artifacts. State transitions (`BLOCKING` → `RESOLVED Conv N`) belong in the section heading itself so a reader skimming the doc 3 convs later sees the status at a glance. Don't delete the original "BLOCKING" framing — convert it (heading + summary table) so the path through the gates is preserved.

**Pattern:** When a pre-plan section gates execution, title it `## N. Open Decisions (BLOCKING)`. At resolution, edit to `## N. Open Decisions (RESOLVED Conv N)` and prepend a 1-row-per-decision resolution-summary table; leave individual decision analyses intact below for traceability.

## 3. Authority split (designer = visual; user = architect) compresses planning scope
**Topics:** plan, matt-design

**Context:** The Conv 173 pre-plan explicitly stated `/matt/*` is "visual re-skinning, not architecture work" — every `/matt/*` route corresponds to an existing page; we wrap its existing data/behavior in Matt's tokens + components. This framing meant route-map work was *mapping*, not *designing*; data-fetcher patterns were *reuse*, not *re-implementation*. Compressed an otherwise-massive scope into 7 execution phases (~8–11 convs) instead of a multi-quarter rebuild.

**Learning:** When a designer ships happy-path visual specs INTO an already-functional app, the planning unit isn't "design a system" — it's "map visual primitives onto existing IA + behavior, then enumerate gaps for unseen pages." The framing decision (visual-only vs full re-architecture) collapses the planning state space by an order of magnitude. Make this framing explicit in the pre-plan §1 before any deliverable enumeration.

**Pattern:** Open the pre-plan with an "Authority Split" sub-section — name what's *in scope* (visual, tokens, layout shell, page composition) and what's *out of scope* (routing, data fetching, role logic, permissions, schema, multi-role architecture). Reference the existing app components/utilities that supply the out-of-scope behaviors. The remaining work is then re-skin, not rebuild.
