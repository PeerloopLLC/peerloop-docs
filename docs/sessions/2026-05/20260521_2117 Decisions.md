# Conv Decisions - 2026-05-21

## 1. /matt/* scope = visual re-skinning of existing pages, NOT architecture work
**Type:** Strategic
**Topics:** matt-design, scope

**Trigger:** User clarified Matt is the visual designer; the user understands the app's multi-role complexity. Happy path was deliberately simplified for Matt. The pages Matt designed already exist in code.

**Options Considered:**
1. /matt/* = parallel architecture (build net-new pages with new behavior)
2. /matt/* = visual re-skin of existing pages (wrap existing behavior in Matt's tokens + components) ← Chosen
3. /matt/* = hybrid (some pages new, some re-skinned)

**Decision:** /matt/* is exclusively visual re-skin. Each /matt/* route mirrors an existing Peerloop route; we wrap existing data + role logic + permissions in Matt's design language.

**Rationale:** Existing app already handles all multi-role complexity, role-encoding, role-specific tabs (ExploreTabBar + dynamic [tab].astro catch-all). Matt deliberately worked from a simplified single-role view. Re-implementing behavior would be wasteful and would risk drift between /matt/* and the rest of the app.

**Consequences:** [MATT-PRE-PLAN] scope shrinks meaningfully — no schema work, no role-system design, no new architectural decisions. The work becomes: token files, layout shell visual re-skin, component primitive visual re-skin, page assembly using existing behavior. Documented in `docs/as-designed/matt-design-system.md` § "Authority Split".

## 2. CSS variable names match Matt's Figma Variable naming verbatim (Title-Case-Hyphenated)
**Type:** Implementation
**Topics:** matt-design, token-naming, css

**Trigger:** Header 1 Dev Mode screenshot revealed Matt has Figma Variables wired up; Dev Mode exports them as `var(--Text-Default, #414141)` style with hex fallback.

**Options Considered:**
1. Follow Tailwind's lowercase-hyphenated convention (`--color-text-default`)
2. Match Matt's Figma Variable naming verbatim (`--Text-Default`) ← Chosen

**Decision:** Use Matt's exact variable names in `tokens-primitives.css` and `tokens-semantic.css`.

**Rationale:** Future paste-back from Figma Dev Mode is one-to-one lossless. No translation layer. Reduces drift risk between Figma source and CSS implementation.

**Consequences:** Tailwind 4 `@theme` block must reference these custom property names directly. Standard Tailwind utility classes (e.g., `bg-primary`) will need explicit theme mappings rather than automatic naming. Documented in `docs/as-designed/matt-design-system.md` § "Color Primitives" → naming convention.

## 3. Visitor = unauthenticated UI state, not a new authenticated role
**Type:** Strategic
**Topics:** matt-design, role-system

**Trigger:** Matt's design enumerates 6 roles (Teacher, Student, Visitor, Creator, Admin, Moderator); existing schema/CLAUDE.md only lists 5 (Visitor absent).

**Options Considered:**
1. Visitor = new authenticated role requiring schema change
2. Visitor = the not-logged-in / browsing state (UX concept, not a DB role) ← Chosen

**Decision:** Visitor is the unauthenticated state. No schema change required.

**Rationale:** Existing app already handles unauthenticated users via middleware + per-page auth gating (POLICIES §7 — Public-browsable tier). Matt's "Visitor" role tab is a visual recognition of that existing tier, not a new identity concept.

**Consequences:** [MATT-PRE-PLAN] avoids any DB migration work. Visitor handling in components uses the same `currentUser === null` check the existing app uses. Documented in `docs/as-designed/matt-design-system.md` § "Roles in this design" + § "Schema role flags → Matt's role enumeration".

**Topics:** matt-design, docs-infra

4. Keep in .scratch indefinitely

**Consequences:** Doc is now committed, propagates to M4Pro on next sync, discoverable from `docs/INDEX.md`. .scratch/ original deleted. Both [MDM] and [MATT-PRE-PLAN] task descriptions updated to reference the new path. INDEX entry added under "How Should It Look/Work?" section.
