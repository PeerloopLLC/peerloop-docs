# Conv Learnings - 2026-05-06

## 1. Skill-bash gets `$HOME` and `$CLAUDE_PROJECT_DIR` for free — together they cover per-machine path derivation
**Topics:** cc-workflow, dual-repo

**Context:** Designing the cross-machine memory-sync system, the live memory dir path includes both a per-machine username (`/Users/livingroom` vs `/Users/jamesfraser`) AND a per-machine project slug (`-Users-livingroom-projects-peerloop-docs` vs `-Users-jamesfraser-projects-peerloop-docs`). Hardcoding either would have made the skill non-portable across machines.

**Learning:** Hook scripts and skill bash blocks inherit `$HOME` (per-machine username) and Claude Code provides `$CLAUDE_PROJECT_DIR` (per-machine project root path). Together they let any path that includes both ingredients be derived at runtime with zero hardcoding. The project-slug encoding Claude Code uses (`/Users/X/projects/foo` → `-Users-X-projects-foo`) is deterministic via `${CLAUDE_PROJECT_DIR//\//-}`, so the live memory dir path becomes one-line portable: `$HOME/.claude/projects/${CLAUDE_PROJECT_DIR//\//-}/memory`.

**Pattern:**
```bash
SLUG="${CLAUDE_PROJECT_DIR//\//-}"
LIVE="$HOME/.claude/projects/$SLUG/memory"
```

## 2. MEMORY.md auto-loads at SessionStart and does NOT refresh mid-session
**Topics:** cc-workflow, docs-infra

**Context:** Designing the memory-sync /r-start hook — would Claude's view of MEMORY.md update if /r-start writes a new MEMORY.md to disk mid-session?

**Learning:** Per `code.claude.com/docs/en/memory.md`: "The first 200 lines of MEMORY.md, or the first 25KB, whichever comes first, are loaded at the start of every conversation." This is the explicit doc. Mid-session refresh is not promised — the docs frame the entire memory system around session boundaries and use load-at-start (not "reload") phrasing. Sub-files (`feedback_*.md` etc.) are read on-demand by Claude when relevant, so they DO see freshly-synced content via the Read tool. Only MEMORY.md itself benefits from an explicit Read after a mid-session sync to push fresh content into the conversation context.

**Pattern:** When a skill modifies MEMORY.md mid-session, follow the modification with an explicit `Read` of MEMORY.md so the fresh content lands in the conversation as a tool result.

## 3. Git history IS the ledger — content-addressed storage subsumes a separate manifest
**Topics:** cc-workflow, dual-repo

**Context:** Initial memory-sync design proposed a `memory-index.md` manifest with checksums. Iteration revealed it was redundant.

**Learning:** When a directory is committed to git, every commit hash is itself a content-addressed snapshot of canonical state. SHAs are derivable from any working copy on demand (`shasum`, `cmp`, `diff -rq`). A separate manifest only adds value if you're detecting tampering; for plain synchronization, git itself answers "what is the canonical state" and "do two trees match" with built-in primitives. Checking out a commit IS applying the manifest. This generalizes: any time a design has "tracked file" + "manifest of tracked file" both committed, the manifest is probably redundant unless it carries information not derivable from the file (e.g., per-machine apply state, which a committed file cannot represent).

## 4. Dry-run rsync catches first-run path bugs before live execution
**Topics:** cc-workflow, testing

**Context:** Implementing memory-sync skill changes. The first /r-end of this conv would be the live first run, so any path-derivation bug would surface only when committing real state.

**Learning:** Before relying on a real execution to validate first-run bash logic, run `rsync -a --delete --dry-run --itemize-changes` (or equivalent) against the same source/destination paths the live skill will use. Validates: (a) path derivation resolves correctly, (b) source dir exists with expected contents, (c) destination is writable, (d) no rsync syntax errors in the skill body, (e) the change set looks like what you expect. The `>f+++++++` itemize notation explicitly shows "new file, all attributes new" — distinguishes "fresh bootstrap" from "would mutate existing data."

## 5. User's "all changes through CC" workflow discipline is doing real architectural work
**Topics:** cc-workflow

**Context:** Memory-sync design needed to decide how to handle "manual edits to live between /r-end and next /r-start." Most file-sync designs need defensive fail-stop logic to detect this case.

**Learning:** Because this user mediates ALL changes to project files through CC (including memory files), there's no scenario where live changes happen "outside the system." That invariant collapses an entire class of file-sync defensive logic — the simple "trust mirror at SessionStart" rule becomes safe by construction, no fail-stop disambiguation needed, no commit-message machine-name parsing needed. Designs that can trust a single mediated point of authorship are substantially simpler than designs that have to defend against arbitrary edits. The discipline pays for itself architecturally, on top of the observability benefits it already provides.
