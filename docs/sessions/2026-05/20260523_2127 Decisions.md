## 1. Button: strict-B single `property1` prop mirroring Matt's Figma enum
**Type:** Implementation
**Topics:** matt, design-system, ui

**Trigger:** Conv 183 MCP probe revealed Property 1 conflates State + Size; couldn't be encoded with React's typical orthogonal `size` + CSS `:hover` pattern without divergence from Matt's source.

**Options Considered:**
1. Orthogonal in React: `size` prop + CSS `:hover` pseudo-class (idiomatic web, but re-orthogonalizes Matt's source)
2. Single `property1` prop mirroring Matt's 5-value enum exactly ← Chosen
3. Three orthogonal props (the current Conv 178 doc framing — `size` × `state` × `variant`)

**Decision:** B — `property1` prop accepting `'Default' | 'Hover' | 'Large' | 'Small' | 'SmallHover'`. Hover and SmallHover are explicit caller-picked states, NOT CSS pseudo-class behavior.

**Rationale:** Source-of-truth fidelity over React idiom. The user has emphasized strict-B mirroring across multiple decisions. Per `feedback_tokenize_only_matt_variables.md`, "hardcode what Matt has hardcoded; tokenize what Matt has tokenized." Matt encoded State+Size as conflated; we mirror that conflation.

**Consequences:** Awkward in React (caller must manually swap `Default` → `Hover` on mouseover; wrapper component needed for typical interactivity). Documented in `matt-design-system.md §5` Button section. Non-Primary + Hover combos render Matt's literal Primary-darkened styling anyway (Phase 6 extrapolation needed for proper variant-aware hover).

## 2. MainNav D1: Route-driven Main pill
**Type:** Implementation
**Topics:** matt, navigation, ux

**Trigger:** Matt's Main Nav (108:4468) has a `Property 1=Main` variant on Nav Item that visually wraps active parent + active child in one white pill. Behavior interpretation needed (route-driven vs click-to-expand) before building.

**Options Considered:**
1. Route-driven (pill follows currentPath, at most one pill at a time, derived state) ← Chosen
2. Click-to-expand drawer (user toggles open; ALL children shown when expanded)

**Decision:** Route-driven. An item enters `Main` state if currentPath matches it OR any child AND it has children. The pill is a derived visual, not user-controlled state.

**Rationale:** Matt's design has no toggle indicator (▼/▶) — visual cues for click-to-expand are absent. Naming "Main" supports active-section-display interpretation. The assembled usage instance shows exactly ONE Main + N Default siblings, consistent with a single active section.

**Consequences:** MainNav.tsx implements an `isActiveItem` predicate. No expand/collapse state in MainNav. Sibling parents are always rendered Default unless their route is current. The architecture works without React state (pure function of currentPath + items[]).

## 3. MainNav D2: Keep 70px collapse mode as Peerloop extension
**Type:** Implementation
**Topics:** matt, navigation, ux, peerloop-extension

**Trigger:** Matt drew NO 70px collapsed sidebar — only the 220px expanded form. Conv 175/176 Sidebar.tsx had a collapse toggle. Strict-B would remove it.

**Options Considered:**
1. Remove (strict-B fidelity to Matt's source)
2. Keep as Peerloop extension (220 ↔ 70 toggleable) ← Chosen

**Decision:** Keep. Documented in matt-design-system.md as a Peerloop addition on top of Matt's Main Nav primitive.

**Rationale:** Sidebar collapse is a near-universal UX expectation; users with smaller screens benefit. The collapsed render does NOT use MainNav (which would awkwardly compress the 220px Figma layout) — Sidebar.tsx renders a separate icon-only nav when collapsed. This isolates the deviation from Matt's source to the Sidebar shell, leaving MainNav itself faithful.

**Consequences:** Sidebar.tsx remains a shell component with `expanded` state. MainNav is consumed only in expanded mode. Future Admin/Teacher sidebars can opt into or out of the collapse mode independently.

## 4. MainNav D3: Props-driven data model
**Type:** Implementation
**Topics:** matt, navigation, architecture

**Trigger:** MainNav needs nav data; should it be hardcoded inside (current Sidebar.tsx pattern) or passed as a prop (more reusable)?

**Options Considered:**
1. Props-driven: `MainNav` accepts `items: NavItemData[]` prop ← Chosen
2. Hardcoded inside MainNav

**Decision:** Props-driven. Caller passes the nav structure as data.

**Rationale:** Matches existing CourseTabs.tsx pattern. Enables Admin/Teacher/Student sidebar variants from one component. Lower-cost decision now than retrofitting later. The current Sidebar.tsx still hardcodes its NAV array internally, but the array passes through to MainNav — the primitive itself is reusable.

**Consequences:** `NavItemData` + `NavSubItemData` interfaces exported from MainNav.tsx. Sidebar.tsx defines its NAV array as a module constant. Future role-specific sidebars can supply their own NAV arrays.

## 7. Disabled state on Button: standard a11y orphan
**Type:** Implementation
**Topics:** matt, design-system, accessibility, ui

**Trigger:** Matt's Property 1 enum has no `Disabled` value. Native HTML `<button disabled>` accessibility is non-negotiable.

**Options Considered:**
1. Add `Disabled` as a 6th Property 1 value (diverges from Matt)
2. Standard CSS rule `[disabled]:opacity-50 disabled:cursor-not-allowed` (orphan but minimal-divergence) ← Chosen

**Decision:** Keep the existing Conv 175/176 disabled styling as a CSS-level orphan.

**Rationale:** Native HTML `<button disabled>` is non-negotiable for accessibility. The styling is a CSS rule on `[disabled]` attribute, NOT a 6th Property 1 value — preserves Matt's 5-value enum exactly while adding a11y coverage.

**Consequences:** Documented in Button.tsx comments. matt-design-system.md §5 notes "no Disabled state drawn — accessibility orphan via native HTML `<button disabled>` + CSS `[disabled]` rule."
