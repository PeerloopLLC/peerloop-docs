# Conv Learnings - 2026-05-25

## 1. Astro View-Transitions drop island-unique arbitrary Tailwind utilities
**Topics:** astro, cc-workflow

**Context:** AppNavbar is a `transition:persist` island. After / → /matt → / via client-side VT, `py-[10px]` had 0 stylesheet rules (padding collapsed to 0); `w-[220px]`/`gap-[12px]` survived.

**Learning:** When VT swaps in a sibling route's CSS bundle, arbitrary utilities used ONLY by the persisted legacy island are not in that bundle and get dropped. Survivors are classes also referenced by the destination route.

**Pattern:**
Use standard non-hijacked Tailwind steps (always globally generated) for persisted-island layout; for one-off px with no safe standard (e.g. 220px, 16px), use inline `style` (on-element, immune to CSS swaps).

---

## 2. Inline `<script>` in .astro doesn't re-run after View-Transition navigation
**Topics:** astro

**Context:** Home `#messages-card` (auth-only `hidden`) revealed by a plain inline `<script>`; vanished after a VT round-trip.

**Learning:** Bundled/inline scripts run once at parse, not after VT navigations. Bind one-time DOM init to `document.addEventListener('astro:page-load', …)` (fires on initial load AND every VT navigation).

---

## 3. Duplicate `style` attribute in JSX is silently dropped (build warning only)
**Topics:** astro, testing

**Context:** Added `style={{left:'220px'}}` to a `<div>` that already had `style={{top:'1rem'}}`; the fix never applied — only a Vite WARN "Duplicate style attribute in JSX element."

**Learning:** JSX keeps only one `style` attr; the symptom is "my inline style silently didn't apply." Always MERGE into the existing `style`. The dev-server log named the bug outright when screenshots couldn't.

## 4. The Tailwind-bridge `--spacing-*` override hijacks the whole app's spacing scale
**Topics:** astro, cc-workflow

**Context:** Legacy navbar `w-64` rendered at 64px not 256px.

**Learning:** `tokens-tailwind-bridge.css` `@theme` defines `--spacing-4..64: var(--space-N)` (Matt literal px). In Tailwind v4 that overrides the built-in numeric steps, so every legacy utility using {4,8,12,16,20,24,32,40,48,64} is ~4× too small app-wide. Steps NOT in that set (2, 2.5, 3, 6, …) fall back to `calc(--spacing * N)` and are unaffected — which is why most desktop content (p-6) looked fine and the navbar (w-64/h-4) broke. Imported globally since Conv 174.

## 5. `.scratch/` is machine-local — scratch→docs graduation must run on the authoring machine
**Topics:** cc-workflow, dual-repo

**Context:** MMP-PH5 graduation needs `.scratch/matt-figma-pages.md` etc., which live only on MacMiniM4 (gitignored, never synced).

**Learning:** Any "promote scratch to docs" task is pinned to whichever machine authored the scratch files; can't be done remotely.

---

## 6. DOM ground truth beats screenshots for dimensional/stacking assessment
**Topics:** cc-workflow, testing

**Context:** Twice this conv a layout state ("flush" / "not affected") was asserted from a screenshot that was actually wrong; an inline-style fix appeared applied in a screenshot but `getComputedStyle` showed `left:0px` (behind navbar). The dev-server log also named the duplicate-`style` bug outright.

**Learning:** For dimensional or stacking claims, query the DOM (`getComputedStyle`, `getBoundingClientRect`, `elementFromPoint`) and read the dev log before claiming — screenshots are unreliable for this class of assessment. Codified in `feedback_dom_truth_over_screenshots.md`.
