# Conv Decisions - 2026-05-29

## 1. Retire `data-matt` entirely (not just on stamped roots)
**Type:** Implementation
**Topics:** astro

**Trigger:** `data-prov` supersedes the unreferenced `data-matt` precursor; user leaned toward strict-C ("remove only on touched roots") and asked for downsides.

**Options Considered:**
1. Strict C — replace `data-matt` only on stamped root elements; leave 4 child/layout occurrences
2. B = C + sweep the 4 leftovers (UserIcon ×2 children, SocialPost ×1 child, AppLayout brand-mark) ← Chosen
3. A — replace everywhere (under-specified on children)

**Decision:** Fully retire `data-matt` (all root + child + the AppLayout `brand-mark`), keeping the distinct `data-matt-preview` in the dev showcase.

**Rationale:** `data-matt` is 100% unreferenced dead weight; leaving 4 child occurrences creates intra-file convention mixing (`data-prov` on root + `data-matt` on children) — the exact unsynced-encoding problem §12 exists to kill. Cost to close the gap was 4 deletions.

**Consequences:** Single clean provenance convention; `data-matt` gone from src/ (34 files touched for it across the stamp pass).

## 2. Composing-wrapper, demo-harness, and wrapper-mode stamping rules
**Type:** Implementation
**Topics:** astro

**Trigger:** 3 edge cases surfaced by the stamping subagents.

**Options Considered:**
1. Leave them unstamped / special-case the gate
2. Resolve each faithfully to §12b "outermost rendered element" ← Chosen

**Decision:** (a) Conditional-render branches stamp each branch root; (b) `UserIcon` roleDot wrapper `<span>` carries the stamp too (it's the outermost node in that mode); (c) `Input` forwards an optional `data-prov` override to its root so `PasswordInput`/`SearchInput` own the DOM-root identity; (d) `_SocialPostDemo` (a demo with no DOM of its own) wrapped in a stamped `<div>`.

**Rationale:** Keeps the 1:1 registry⟺stamp invariant true at both the source-gate and DOM levels with no special-casing in the gate.

**Consequences:** All 59 stamped; documented in `matt-provenance.md §12b` "Edge cases resolved at W3".

## 3. Page-conformity report is informational, not a hard gate
**Type:** Implementation
**Topics:** testing

**Trigger:** `legacy` stamps + uncovered interactives are expected until each page is re-skinned (W4).

**Decision:** `prov:page-report` always exits 0; the hard registry⟺marker⟺stamp gate stays in `prov:sweep`.
