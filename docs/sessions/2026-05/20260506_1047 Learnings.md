# Conv Learnings - 2026-05-06

## 1. eslint-disable-next-line directive must be on immediately-prior line
**Topics:** eslint, code-conventions

**Context:** Phase 4 attempted to suppress a react-hooks/exhaustive-deps warning on a useEffect. Wrote `// eslint-disable-next-line react-hooks/exhaustive-deps` 4 lines above the `useEffect(() => {`, separated by 3 lines of WHY-comment prose explaining the intentional omission.

**Learning:** ESLint's `eslint-disable-next-line` only suppresses lint on the literal next line. Intervening comments invalidate it — produces "Unused eslint-disable directive" warning AND the original warning still fires.

**Pattern:**
```tsx
// WHY comment (multi-line OK)
// can go ABOVE the useEffect freely
useEffect(() => {
  ...body...
  // eslint-disable-next-line react-hooks/exhaustive-deps  ← directive must be here
}, [explicit-deps]);
```
The directive can sit just inside the arrow body, immediately above the closing `}, [deps])`. WHY-comments go above the useEffect.

## 2. useCurrentUser was not memoizing — every focus event churned all consumers
**Topics:** react, performance, current-user

**Context:** [UCM] sanity check from Conv 148 §Uncategorized. Surfaced because Conv 149 just landed multiple `[currentUser]`/`[user]`-keyed `useMemo`/`useCallback` extractions in LE-P3/4/5.

**Learning:** `setCurrentUser` had no equality check; every `refreshCurrentUser()` (called on every window focus by AppNavbar's focus handler) created a new `CurrentUser` instance and notified all listeners with the new ref. The cure was a 3-line dedup guard using the already-existing `dataVersion` field. The code looked right at every site, but the singleton-update layer leaked instability that defeated downstream memos.

**Pattern:** When establishing memoization at the consumer (useMemo/useCallback deps), verify the *producer* of the dep value also memoizes. Local stability is necessary but not sufficient. The cheapest place to fix it is usually as close to the source as possible.

## 3. Server-truth fixtures need `dataVersion` bumps to model reality
**Topics:** testing, fixtures

**Context:** UCM dedup fix broke 3 existing tests that updated `name` between two `validMeFullResponse()` calls without bumping `dataVersion`. The dedup correctly skipped the listener notification because the fixture pretended the server returned a "changed" payload with the same version.

**Learning:** Test fixtures that simulate a server-driven update should bump the version field too, not just the payload field being asserted on. Otherwise the test models a state the server would never produce. The fix was 1-line per test: `validMeFullResponse({ dataVersion: 1 })`.

## 4. Hoist static config arrays to module scope when closed over by useCallback
**Topics:** react, performance, code-conventions

**Context:** AppNavbar's `getVisibleItems` useCallback was missing `menuItems` from its dep array. The `menuItems` 7-item array was defined inside the component body as a const literal, recreated every render. All entries were pure JSX/string literals with no closure dependencies.

**Learning:** Static config arrays declared inside a component but closed over by a `useCallback` will appear as a missing dep — but adding them to deps creates churn (new identity every render). Hoisting the array to module scope eliminates the false dep entirely, simplifies the dep array, and signals intent: "this is configuration, not state."

**Pattern:** If an array/object literal inside a component has no closure dependencies (no props, no state, no derived values), hoist it above the component function.
