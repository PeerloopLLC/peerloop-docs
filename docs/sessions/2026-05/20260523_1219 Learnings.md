## 2. CC's MCP server list is loaded once at session start
**Topics:** cc-workflow

**Context:** Ran `claude mcp add --transport http figma https://mcp.figma.com/mcp`, confirmed via `claude mcp list` that figma was registered with "Needs authentication" status, then tried `/mcp` in the running session — the figma entry was not visible.

**Learning:** `claude mcp add` writes to `~/.claude.json` but does not hot-reload into the running CC session's in-memory MCP list. New MCP entries require a fresh CC instance to surface. Opening another terminal session doesn't help the original session either — the in-memory list is per-process. Implication: when adding MCP servers mid-conv, plan for an `/r-end` → exit → `/r-start` bridge to activate.

## 3. `.claude.json` MCP entries are project-scoped automatically
**Topics:** cc-workflow

**Context:** `claude mcp add` output: "Added HTTP MCP server figma with URL: https://mcp.figma.com/mcp to local config / File modified: /Users/livingroom/.claude.json [project: /Users/livingroom/projects/peerloop-docs]".

**Learning:** `claude mcp add` defaults to writing to the user-level `~/.claude.json` file but tags entries with `[project: <cwd>]`, so the MCP activates only when CC is launched from that project directory. This is effectively B1 (project-scoped) even though the file lives at user level. Net consequence: M4Pro will need its own `claude mcp add` invocation since `.claude.json` is per-machine, not git-synced — but the project-tagging means the entry won't leak into other projects on that machine.

## 4. TodoWrite ↔ PLAN.md hygiene threshold
**Topics:** cc-workflow

**Context:** RESUME-STATE.md transferred 53 outstanding tasks to TodoWrite at /r-start; conv added one more (`[VDF]`) reaching 54. User flagged: "54 tasks in todoWrite is too many to ever do in a session."

**Learning:** TodoWrite is the conv's working set; PLAN.md is the cross-conv canonical task home. When tasks accumulate past ~30 and many belong to inactive Blocks (BBB-RECORDING, DEPLOYMENT.DB-SYNC, infra watch-tasks), they should be moved to PLAN.md and deleted from TodoWrite. The triage rules: (a) tasks already in PLAN.md → just delete TodoWrite copies, (b) tasks resolved by other work → delete, (c) net-new tasks → add to PLAN.md then delete TodoWrite copy, (d) memory captures → write the file then delete. The aim is a TodoWrite that reads as "what THIS conv (and the next 1-2) will actually touch."

## 5. Memory consolidation when multiple captures point at one pattern
**Topics:** docs-infra

**Context:** Conv 178 carryforward had three "save memory" tasks (`[MFM]` Matt extraction methodology, `[STOR]` source-of-truth before visual ID, `[DTU]` defer to user-supplied source); Conv 179 added a fourth (`[VDF]` vendor-docs-first). All four were context-specific instances of one broader rule.

**Learning:** Rather than four separate near-duplicate memory files (which dilute MEMORY.md index signal), consolidate into one file with named contexts. The MEMORY.md index entry exposes distinctive markers (`[VDF]`, `[MFM]`, `[STOR]`, `[DTU]`) so future grep-based recall still finds the rule under any of its triggers. This addresses the per-file vs per-rule tension: 1 rule, 1 file, but multiple grep-anchored entry points.

**Pattern:** Memory file structure used: Rule → Why → How to apply (with N numbered concrete contexts) → Anti-pattern → Related (with `[[...]]` links).
