# Conv Learnings - 2026-05-29

## 1. Terse mnemonic codes lose intent across carry-over; recover from committed history
**Topics:** cc-workflow, docs-infra

**Context:** ~10 of 31 carried TodoWrite tasks survived as codes only — descriptions didn't make it through the RESUME-STATE→TodoWrite chain across many convs.

**Learning:** The original definitions DO survive in committed history — PLAN.md block rows, `plan/matt/*.md` phase files, `docs/DECISIONS.md`, and `docs/sessions/**/*.md` extracts. A single grep of those per code recovers intent (and caught two mis-states: a mislabeled task — #11 RTB was "real task board" but is actually "Role Tab Bar" — and two already-done/superseded tasks). When carry-over text is thin, the fix is archaeology, not guessing. Two watch-tasks that predate the Conv 211 consolidation couldn't be reconstructed and were flagged honestly rather than invented.

**Pattern:**
```
grep -rn "CODE" PLAN.md plan/ docs/sessions/<month>/*.md
```
per opaque code → reconstruct from the richest hit.

## 2. Verify a hash-sentinel with the SAME algorithm the producer used
**Topics:** cc-workflow

**Context:** The drift sentinel check falsely reported 🔴 MISMATCH because verification compared `shasum` (sha1, 40 hex) against a stored sha256 (64 hex).

**Learning:** Hash length is the tell — 40 vs 64 hex = sha1 vs sha256. The postinstall hook uses `shasum -a 256`; verification must match. Always read the producer's algorithm before comparing.

## 3. "Auto-generated" docs can outlive their generator
**Topics:** docs-infra

**Context:** [OPM-REGEN] asked to regen `orig-pages-map.md`, but both the generator script (`scripts/generate-pages-map.ts`) and its `src/data/pages/` source were deleted in the migration.

**Learning:** A doc's `Regenerate: <cmd>` header is a claim, not a guarantee — verify the pipeline still exists before treating a regen task as actionable. When the pipeline is gone, the doc is a frozen historical artifact; mark it as such so the stale instruction stops misleading future readers.

## 4. Tailwind v4 `outline-none` ≠ v3 `outline-none`
**Topics:** astro, testing

**Context:** TW-V4-FLAGS `outline-none` flags across 6 form/feed components.

**Learning:** In v4, `outline-none` means `outline-style: none`; the v3 "hide visible outline but keep forced-colors a11y" behavior moved to `outline-hidden`. For `focus:outline-none` paired with an alternative focus indicator, `outline-hidden` is the intent-preserving v4 form. `bg-gradient-to-*`→`bg-linear-to-*` is an unambiguous rename (old form is a deprecated alias).
