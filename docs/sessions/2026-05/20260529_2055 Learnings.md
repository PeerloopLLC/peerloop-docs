# Conv Learnings - 2026-05-29

## 1. Never re-issue identical tool calls to "flush" a perceived buffer
**Topics:** cc-workflow, tooling

**Context:** A couple of early Bash/Read results came back empty during `/r-start`; the result was modeled as output buffering that would "flush" after N calls.

**Learning:** There is NO buffer and NO flush mechanism. Re-issuing an identical tool call does not flush — it re-executes and appends another full copy of the output into context. Spamming `Read RESUME-STATE.md` ~25× (plus SKILL.md regions, TaskList, greps) burned ~420K tokens / ~10 min on a near-empty interaction. The first result is authoritative; empty means empty; at most one retry with a concrete reason. Light interaction + climbing tokens → suspect a loop and halt.

**Pattern:** Persisted as a top-of-MEMORY.md **Tool-Call Discipline** section (loads every session) + `memory/feedback_no_tool_call_spam_loops.md`.

## 2. A skill's invocation-rendered body is a pre-pull snapshot
**Topics:** cc-workflow, dual-repo

**Context:** `/r-start` Step 7.5 (added Conv 215, arriving via this conv's own Step-2 pull) was silently skipped in Conv 218.

**Learning:** The SKILL.md body Claude executes is rendered at invocation; `/r-start` Step 2 then pulls a potentially newer copy of the same skill. Any conv where the other machine edited the running skill executes the STALE body. This is staleness, not truncation. Fix: after the self-pull, re-read the on-disk SKILL.md.

**Pattern:** Persisted as `memory/feedback_skill_body_stale_after_self_pull.md` + `r-start` Step 2.5 self-update detector (diffs `HEAD@{1}..HEAD` on its own SKILL.md after the pull; prints ⚠️ + "re-read on-disk" if it self-updated).

## 3. Re-derived hash checks must match the producer's algorithm
**Topics:** tooling

**Context:** An ad-hoc dependency-drift check threw a false `DEPS_DRIFT` during `/r-start`.

**Learning:** The postinstall sentinel stores `shasum -a 256` (64 hex); a plain `shasum` re-derivation is sha1 (40 hex) → guaranteed mismatch. Trust the authoritative check (`LOCKFILE_UNCHANGED` + precomputed `OK`), not a hand-rolled hash with a different algorithm.
