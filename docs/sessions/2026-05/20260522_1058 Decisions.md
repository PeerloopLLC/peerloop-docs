# Conv Decisions - 2026-05-22

## 1. CSS length units: Hybrid (rem for typography + spacing; px for borders + hairlines)
**Type:** Strategic
**Topics:** matt-design, tokens

**Trigger:** Spec doc `matt-design-system.md` §4 Q8 surfaced Conv 172 — "what units should the token system use?". Three options laid out in `matt-pre-plan.md` §4 Decision 1.

**Options Considered:**
1. Pure px — concrete, predictable, matches Matt's Figma 1:1; no accessibility scaling
2. Pure rem — accessibility-correct; pixel-named tokens get awkward (`--space-4: 0.25rem` reads inconsistently)
3. Hybrid — rem for typography + spacing (with pixel-named tokens); px for borders + 1px hairlines + radius ← Chosen

**Decision:** Hybrid. Typography emits rem (`--text-base: 1rem`); spacing emits rem with px-named tokens (`--space-4: 0.25rem`); borders + 1px hairlines + `--radius-*` stay as px.

**Rationale:** Standard design-system convergence — Tailwind 4, Material, IBM Carbon all use this pattern. Accessibility where it matters (text scales with user pref); predictability where it doesn't (a `1px` divider stays `1px`). Matches Tailwind 4's native convention so `@theme` integration is frictionless. Pure px would have foreclosed accessibility scaling permanently; pure rem would have introduced naming/value mismatch.

**Consequences:** All token files in Phase 1 (`[MATT-EXEC-TKN]`) emit hybrid units. Spacing token names stay pixel-anchored (`--space-4`, `--space-16`, etc.) but values are rem. Border-width tokens and `1px` UI hairlines stay px. `--radius-*` stays px (small absolute sizes, no accessibility concern). One-line cognitive overhead in the token file ("rem unless it's a border") is acceptable.

## 2. Primitive naming normalization: kebab-case for primitives; semantics keep Title/Slash form
**Type:** Strategic
**Topics:** matt-design, tokens

**Trigger:** Conv 172 found Matt's 15 color primitives use 3 different naming conventions in one Figma collection: kebab-case (`pastel-blue`, `dark-green`), Title Case with space (`Ashy Blue`), lowercase with space (`alert light`). CSS variables can't contain spaces.

**Options Considered:**
1. Match Matt's literal names with case-sensitive normalization (`--Ashy-Blue`, `--alert-light`) — maximum traceability but mixed-case visually noisy and grep-ambiguous
2. Force kebab-case for primitives (`--ashy-blue`, `--alert-light`); semantics keep Matt's `Title-Case/with-slash` form ← Chosen
3. Match each token's original case exactly — three styles in one file, encodes inconsistency

**Decision:** Force kebab-case for primitives; semantics unchanged. PLAN.md "Figma Variable names verbatim" directive remains true for SEMANTICS (`--Text-Default`, `--Course-Primary`) where Matt's naming is already consistent and CSS-compatible; primitives are normalized because Matt's own inconsistency would otherwise propagate into our code permanently.

**Rationale:** Consistent + ergonomic + conventional CSS. The two outliers (`Ashy Blue`, `alert light`) translate to kebab-case trivially — no real fidelity loss. Future-default identity: when Matt's design becomes THE design, kebab-case is the right shape. Flag the Figma inconsistency back to Matt for v2 (already in `[TSV]` task).

**Consequences:** `tokens-primitives.css` emits 15 kebab-case variables. `tokens-semantic.css` emits 14 Title/Slash-formatted variables (`--Text-Default: var(--gray-base);`). Two conventions in two files keeps each file consistent internally — the SPLIT is the design.

## 3. Tailwind 4 `@theme` integration: Hybrid bridge file
**Type:** Strategic
**Topics:** matt-design, tokens, infra

**Trigger:** Matt's semantic tokens (`--Text-Default`) don't fit Tailwind 4's expected `--color-foo` shape that auto-generates `bg-foo`, `text-foo` utility classes. PLAN.md DECISIONS §3 mandated "verbatim" Matt naming. How to reconcile?

**Options Considered:**
1. Matt's tokens stay OUTSIDE `@theme`; consume via `bg-[var(--Text-Default)]` arbitrary-value syntax — pristine separation but verbose at call sites
2. Mirror Matt's tokens INTO `@theme` with Tailwind-shaped names (`--color-text-default`) — utility-class ergonomic but conflicts with verbatim directive and creates drift risk
3. Hybrid `tokens-tailwind-bridge.css` — Matt's tokens canonical; bridge file imports them and re-exports as `--color-*` ← Chosen

**Decision:** Build `tokens-tailwind-bridge.css` (~30 lines) sitting next to canonical token files. Bridge imports primitives + semantics, then declares `@theme { --color-text-default: var(--Text-Default); ... }` for ~15 mappings. Tailwind utility classes (`bg-text-default`, `text-course-primary`) become available; canonical names stay pristine.

**Rationale:** Best-of-both — canonical Matt-shaped names ARE the source of truth (`--Text-Default` is what callsites reference directly); the bridge gives utility-class ergonomics for the cases (mostly colors) where they matter. Drift is grep-visible because the bridge file is small, in-tree, and updated alongside primary tokens. IDE autocomplete works (via the bridge entries).

**Consequences:** Phase 1 ships 3 token files instead of 2. Both forms work at call sites — `bg-[var(--Course-Primary)]` (canonical) and `bg-course-primary` (utility). Drift requires actively editing the bridge file — grep-visible if it happens.

## 4. Primary color coexistence: Coexist (existing `--color-primary-*` untouched)
**Type:** Strategic
**Topics:** matt-design, tokens

**Trigger:** Existing `global.css` `@theme` block defines `--color-primary-50` through `-950` as sky-blue. Matt's `americana-blue` (`#0777B6`) sits at roughly the `primary-700`/`primary-800` step. During the coexistence period (until the future flip `/matt/` → `/`), both color systems live in the codebase.

**Options Considered:**
1. Replace existing primary scale with Matt-derived — breaks every non-`/matt/*` page; visible color shift across ~70 pages
2. Coexist — existing scale untouched; `/matt/*` doesn't import old scale ← Chosen
3. Bridge — emit existing `--color-primary-*` names as aliases of Matt's `--Primary-Default` — same visible-shift risk as #1 hidden behind one indirection

**Decision:** Coexist. Old `--color-primary-*` sky-blue scale stays in `global.css` `@theme` unchanged. Matt's tokens (primitives + semantics + bridge) live in parallel files. `/matt/*` pages don't import the old scale; they consume Matt's tokens directly or via the bridge. Future flip block will consolidate by removing the old scale.

**Rationale:** Conv 169 directive ("coexist alongside existing") + zero risk to existing app. Replacement would have caused subtle blue shifts on every non-`/matt/*` page overnight — confounders the testing phase doesn't need. Two color systems in one codebase is a manageable mental tax until the flip; consolidation is the flip's job, not Phase 1's.

**Consequences:** `global.css` import chain includes both the existing `@theme` and the new Matt token files. Mental rule for devs: "which blue? which route?" — `/matt/*` uses Matt's blue; everywhere else uses existing primary. Future flip is a one-PR removal of the old scale.

## 5. `/matt/` index route: Member-only (analog of `/dashboard`)
**Type:** Strategic
**Topics:** matt-design, page-routing, auth

**Trigger:** Matt's `Home.svg` series (3 state variants — visitor / enrolled / completed) is community-feed style, closer to `/dashboard` than to existing `/` (marketing landing). Decision: what does `/matt/` resolve to?

**Options Considered:**
1. `/matt/` = `/dashboard` analog (member-only); visitors → `/matt/login` ← Chosen
2. `/matt/` = `/` analog (public landing); add `/matt/dashboard` separately — doubles home routes; Matt drew only one Home
3. `/matt/` is auth-tier-detecting (one route, three states) — confuses POLICIES §7 auth tiers

**Decision:** `/matt/` is member-only — `AppLayout` (Matt's), JWT-gated by middleware. Visitor flow redirects to `/matt/login` (which we'll build alongside; the existing `/login` page is the data shape, Matt-tokens applied). After the future flip, `/matt/` → `/dashboard` and existing `index.astro` → `/fraser/`.

**Rationale:** Matches Matt's design intent (community feed for authenticated user). Clear auth tier per POLICIES §7. Simple route logic — no auth-tier-detecting branch in one page file. Visitor handling adds one small route (`/matt/login`) instead of one complex SSR-branching root page.

**Consequences:** Phase 5 (`[MATT-EXEC-PG2]`) includes `/matt/login.astro` as an explicit deliverable alongside `/matt/index.astro`. The login flow re-skins the existing `/login` page's auth-modal trigger pattern; no new business logic.

2. Rebuild as new `Sidebar.tsx` — clean IA match; reuses hooks/utilities (CurrentUser, modal utils, UserAccountDropdown) ← Chosen

## 7. HeaderBar architecture: Slot-based (named slots)
**Type:** Implementation
**Topics:** matt-design, components, infra

**Trigger:** Matt's HeaderBar has 3 distinct content shapes by breakpoint per spec §2: Desktop = breadcrumb strip; Tablet Portrait = centered brand strip; Mobile = 3-slot row (Messages / brand / Notifications). DOM children differ by breakpoint, not just CSS.

**Options Considered:**
1. Single component with breakpoint-conditional internal JSX — all 3 shapes' DOM ships at every breakpoint; CSS hides what shouldn't show
2. Three separate components; layout chooses one via media queries — three imports per page; duplicates breakpoint logic at callsite
3. Slot-based — `<HeaderBar>` exposes named slots (`header-left` / `header-center` / `header-right`); pages pass content; layout owns breakpoint outer structure ← Chosen

**Decision:** Astro `<HeaderBar>` component with `<slot name="header-left" />`, `<slot name="header-center" />`, `<slot name="header-right" />`. Layout's breakpoint logic decides outer structure (mobile = 3-slot flex row; tablet = centered single slot; desktop = breadcrumb container). Pages opt into the shape they need.

**Rationale:** Astro's named-slot primitive was designed for layouts whose structure varies by content. Single import, breakpoint-driven outer container, page-driven inner content — best of options 1 and 2. Mobile-only icons (Messages, Notifications) live in the page that needs them, not bundled into a desktop DOM that hides them. Pattern scales to SubNav (which has its own breakpoint variance per spec §2.5) — same mental model, two applications.

**Consequences:** Phase 2 `HeaderBar.astro` exposes 3 named slots. SubNav follows the same slot-based pattern. Pages compose the shape they need by passing slot content. Adding a fourth breakpoint shape later doesn't require changing the component — just updating the layout's outer-container CSS.
