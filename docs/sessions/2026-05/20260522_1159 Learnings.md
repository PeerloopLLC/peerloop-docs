# Conv Learnings - 2026-05-22

## 1. Tailwind 4 emits @theme tokens on-demand, not eagerly
**Topics:** astro, cloudflare (build pipeline), matt-design-system

**Context:** After Phase 1 built clean, inspected `dist/client/_astro/global.*.css` and grep'd for Matt's bridge color tokens — `--color-text-default`, `--color-course-primary`, etc. — and found ZERO occurrences. The legacy `--color-primary-50…950` WAS present (existing app uses `bg-primary-*` utilities). Phase 2 components then USED `text-text-default`, `bg-entity-background`, etc., and the rebuilt CSS contained all those tokens.

**Learning:** Tailwind 4's `@theme` block REGISTERS tokens but only EMITS them to production CSS when at least one utility class consumes them. This is correct/expected behavior — keeps bundle size tight. Implication: don't panic if a freshly-authored bridge token isn't in `dist/` until a component exercises it. Verification protocol = "write component that consumes it, rebuild, re-grep."

**Pattern:** When adding new bridge tokens, immediate verification is build-and-grep with at least one consumer component in the same change.

## 2. `--spacing-N` overrides Tailwind utility scale globally, not additively
**Topics:** matt-design-system, astro

**Context:** Pre-plan §5 bridge spec included `--spacing-4: 0.25rem` through `--spacing-64` in @theme. Before authoring, ran `grep -rho 'p-[0-9]\{1,2\}'` and found 572 `p-4` callsites + similar `m-N`/`gap-N`/`px-N`/`py-N` across existing app. Built and inspected: `.p-4{padding:var(--spacing-4)}` confirms Tailwind 4 reroutes the utility to the explicit @theme key, overriding the default multiplicative `calc(var(--spacing) * 4) = 1rem`.

**Learning:** Setting `--spacing-N: VALUE` in @theme OVERRIDES the `p-N`/`m-N`/`gap-N`/etc. utility rule globally for that N. Non-overridden N values still use multiplicative. Creates a MIXED scale: `p-1=4px` (multiplicative), `p-4=4px` (overridden, was 16px), `p-5=20px` (multiplicative). After Conv 174 user choice B: pixel-named utilities (p-4, p-8, p-12, p-16, p-20, p-24, p-32, p-40, p-48, p-64) all mean their pixel value. Existing legacy `p-4` callsites shrink 4×.

**Pattern:** When bridging an existing scale into Tailwind 4, audit usage first (`grep -rho 'p-[0-9]+'`) and decide explicitly: override, namespace, or omit. Don't follow a spec doc's bridge sketch literally without checking utility-class collision.

## 3. Matt 2-layer + 3-layer cascade preserves correctly when authored as var() chains
**Topics:** matt-design-system

**Context:** Matt's design has `--Student-Primary` semantically referencing `--Primary-Default` (which references primitive `--americana-blue`). Authored `tokens-semantic.css` with `var()` chain (`--Student-Primary: var(--Primary-Default)`) instead of flat primitive ref (`--Student-Primary: var(--americana-blue)`). Inspected built CSS: chain preserved as-authored.

**Learning:** Browser resolves CSS variable chains lazily — `:root` declaration order doesn't matter; chains work as long as each link is defined somewhere on the matching cascade context. Critical for preserving Matt's design-system resilience: changing `--Primary-Default` propagates to `Student-Primary` + `Text-Primary` automatically.

**Pattern:** When extracting a design system that uses semantic-to-semantic refs, NEVER flatten. The cascade IS the value.

## 4. detect-machine.sh still mis-detects M4Pro despite Conv 168 fix
**Topics:** cc-workflow

**Context:** `/r-start`'s SessionStart hook output "Machine: Unknown (M4Pro.local)". Conv 168 [MND] claimed to fix this with case patterns `*M4Pro*` + `*M4-Pro*` in `~/.claude/hooks/detect-machine.sh`. Verified live: `hostname -s` returns `M4Pro`, `~/.claude/.machine-name` cached as `Unknown (M4Pro.local)`. Case patterns either don't match `M4Pro` (no hyphens) or the script reads hostname in a form with `.local` suffix that the glob misses.

**Learning:** The Conv 168 fix didn't fully land OR the case patterns have an unanticipated edge case on M4Pro. `.machine-name` cache file persists the stale value across sessions — must be refreshed to test.

**Pattern:** When fixing a hostname-detection bug, verify on the actual target machine, not just the script logic. And clear the cache file before re-running.

## 5. Tailwind `lg:` breakpoint is 1px off Matt's Desktop boundary
**Topics:** matt-design-system, astro

**Context:** Matt's spec: Desktop ≥1025px / Tablet Portrait ≤1024px. Tailwind 4 default `lg:` = `min-width: 1024px`. At exactly 1024px, Matt's intent says tablet shell but Tailwind says lg:. Used Tailwind's defaults for Phase 2 stub, flagged as [MSH-REFINE] task.

**Learning:** Off-the-shelf Tailwind breakpoints don't exactly align with Matt's pixel boundaries. Custom breakpoint config would fix; not worth the global change for a 1px shift until [TSV] reviews all breakpoint discrepancies together.
