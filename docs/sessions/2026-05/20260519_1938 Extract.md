**User:** "isn't the tilde itself equal to HOME portably?"

**Claude:** Right — `~` IS functionally equivalent to `$HOME`, both resolve to current user's home. The difference is expansion timing: `~` at parse time (only outside double quotes, at word start), `$HOME` at variable-expansion time (anywhere). Realized the fix was to drop the surrounding double quotes from path string assignments (paths have no spaces, so unquoted is safe). Did:
- `"$HOME/.claude/projects/..."` → `~/.claude/projects/...` (regex `s{"\$HOME/([^"]+)"}{~/$1}g`)
- `"$HOME/projects/peerloop-docs/..."` → `~/projects/peerloop-docs/...`
- `SLUG="${CLAUDE_PROJECT_DIR//\//-}"` → `SLUG=$(echo ~/projects/peerloop-docs | tr / -)` (subshell pipes tilde-expanded path through `tr` to produce slug)

Verified empirically: `SLUG=$(echo ~/projects/peerloop-docs | tr / -)` yields `-Users-jamesfraser-projects-peerloop-docs` on M4Pro and the LIVE path `~/.claude/projects/$SLUG/memory` resolves correctly.

**User:** "are we sure this will work on both M4Pro (here) and on M4? Is this the same technique used in ~/projects/spt-docs?"

**Claude:** Empirical verification of both points:

1. **Cross-machine portability** — confirmed via simulated M4 (`HOME=/Users/livingroom bash -c '...'`):

   - M4 (simulated): `~/projects/peerloop-docs` → `/Users/livingroom/projects/peerloop-docs`, SLUG `-Users-livingroom-projects-peerloop-docs`

2. **spt-docs comparison** — partial match:
