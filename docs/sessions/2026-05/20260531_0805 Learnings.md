# Conv Learnings - 2026-05-31

## 1. Re-skinning ≠ porting — the Conv 205/221 filter-only collapse dropped functionality
**Topics:** matt-design, migration, discover

**Context:** /courses (Conv 205) and /communities (Conv 221) "ports" replaced legacy's distinct per-role components (EnrollmentCard progress, teaching/created/moderation views) with a single filter-only browse catalog.

**Learning:** A port must transfer functionality + content faithfully AND apply Matt styling — both co-equal. The filter-only collapse kept (some) styling but silently dropped per-role rendering; it read as "done" and shipped. Captured as memory `feedback_port_functionality_and_styling`.

## 2. Read the legacy source-of-truth before concluding a comparison
**Topics:** process, verification

**Context:** Twice concluded "/courses & /communities role tabs are just filters, at parity" by reading only the NEW components, not the legacy `/old` views. User corrected both times.

**Learning:** In any review/compare/port/parity task, fully read BOTH sides (especially the legacy source) before surfacing conclusions. "Already in context" counts only as a genuine full read this conv. No directive permits skipping full checks. Captured as `feedback_read_legacy_source_before_conclusion`.

## 3. Host-aware shared card avoids the "modify-shipped-card" dilemma
**Topics:** matt-design, components

**Context:** The All card is the Conv-205 shared `CourseCatalogCard` (also serving recommendations + a communities mirror). Content parity needed more fields, but mutating it changes every caller.

**Learning:** A `context` prop ("catalog" vs default "recommendation") lets one host opt into richer content while other callers keep the trimmed design — enriches one surface without a fork or breaking symmetry.

**Pattern:**
```tsx
// CourseCatalogCard — context prop gates catalog-only fields
<CourseCatalogCard context="catalog" /* adds badge/sessions/duration/creator avatar+link */ />
<CourseCatalogCard /* default: trimmed for recommendations */ />
```
