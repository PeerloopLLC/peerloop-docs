# Conv Learnings - 2026-05-20

## 1. Astro `client:load` JSON-serializes prop boundaries — React element nodes break
**Topics:** astro, react

**Context:** Conv 165 [CRT-3] wired `extraTabs` from `sessions.astro` to `CourseTabs` (React island with `client:load`) by passing a `ReactNode` constructed via `createElement(TeacherSessionsTabContent, props)`. Browser threw "Objects are not valid as a React child (found: object with keys {$$typeof, type, key, props, _owner, _store})" at first render.

**Learning:** Astro hydrates `client:*` components by JSON-serializing their props server-side and re-parsing them in the browser. A `ReactElement` instance survives `JSON.stringify` (its fields are enumerable) but is rebuilt as a *plain object*, not a real React element — React then refuses to render it. The pattern of constructing JSX-like content in Astro frontmatter and passing it as a prop to a hydrated React component is broken by design, even though TypeScript accepts it (`ReactNode` is the right type, but the runtime contract fails).

**Pattern:** When a React island needs role-specific or page-specific child content, do NOT pass React nodes across the prop boundary. Either (a) pass primitive descriptors (booleans, strings, IDs) and let the island construct the JSX internally — the approach we landed on for [CRT-3] (`isTeacherOfCourse` flag → CourseTabs imports `TeacherSessionsTabContent` and renders it), or (b) use Astro `<slot/>` composition (changes the component's interface shape, heavier refactor). Option (a) is cheap when the set of possible contents is small and known to the island.

## 2. Bias the API contract toward caller-declared intent when role-based precedence creates dual-role ambiguity
**Topics:** d1, auth

**Context:** Conv 165 [CRT-2] added role-aware scoping to `GET /api/courses/[id]/sessions` (admin/creator/moderator → all, teacher → own teaching sessions, enrolled student → own student sessions). This introduced a latent regression for users with BOTH roles: a teacher who is also enrolled in the same course would see teaching-scoped sessions on the student-perspective Sessions tab (because precedence picked teacher). Caught while designing [CRT-3]'s "My Teaching Sessions" tab — both the existing student tab and the new teacher tab would hit the same endpoint and render the same (teacher-scoped) data, defeating the point.

**Learning:** Whenever a single endpoint serves multiple scopes that map to different UI surfaces, server-side precedence is the wrong default — the *caller* (the UI tab) knows which scope it wants; the server only knows what scopes are permitted. Add an explicit scope query param early, before the precedence default ossifies into call sites that depend on it. Default-without-scope can still pick highest-privilege for backwards-compat, but every new UI surface should declare its intent.

**Pattern:** API shape: `?scope=student|teacher|all` (optional). Server rejects (403) if caller lacks the role required for the requested scope. Omitted scope keeps the legacy "highest-privilege precedence" default. This catches dual-role bugs at the API boundary instead of letting them propagate into UI components.

## 3. Browser verification catches contract bugs that type-check + tests miss
**Topics:** testing, astro, react

**Context:** Conv 165 [CRT-3] passed 5/5 baseline gates (tsc, astro check, lint, 6438 tests, build) with `extraTabs.push({ content: createElement(...) })` in `sessions.astro`. Only browser-loading the page surfaced the Astro→React JSON serialization bug — the React Error appeared in the console at hydration time.

**Learning:** For changes that cross the Astro→React hydration boundary (props passed to `client:*` islands), unit tests and type checks are insufficient. The serialization contract is a runtime invariant not modeled in the type system. CLAUDE.md system reminder already mandates browser verification for UI changes, but it's worth reinforcing: "all gates green" does not equal "feature works."

**Pattern:** Any conv that introduces a new prop shape on a `client:load` / `client:visible` / `client:idle` component MUST include a browser load + console-error check before claiming the slice is done — even when types and tests pass.
