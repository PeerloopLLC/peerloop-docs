# Conv Learnings - 2026-05-28

## 1. Page-marker regex needs line-anchoring — the §9 grep-pollution gotcha generalizes
**Topics:** matt-provenance, prov-sweep, regex, heuristic-calibration

**Context:** While teaching `prov-sweep.ts` about `@matt-inspired`, the first regex re-used `MARKER_RE` to detect `@matt-source` on pages. 404.astro's own comment mentions `Button (`@matt-source 40:482`) primitive` in prose — that prose reference matched, misclassifying 404 as `@matt-source` instead of `@matt-inspired`.

**Learning:** `prov-sweep.ts:86-89` already noted "tightening the accept-condition is more robust than enumerating reject-conditions (the §9 grep-pollution gotcha)" — but only for the component case. Pages have the same shape problem when their headers cite child-component markers in prose. Solution: line-anchored regex with allowed leaders (`^[\s/*]*@marker\b`), not bare token-anywhere.

**Pattern:** When extending a marker-detection regex to a new file class, re-derive the accept-condition from scratch — don't assume the original regex's discipline carries over to the new context. Test against the file most likely to contain prose mentions of the marker (i.e., the just-retrofitted exemplar).

## 2. Marker flip ≠ retrofit
**Topics:** standin-matt, matt-design-system, scope-discipline

**Context:** Treated /profile retrofit as: marker flip @stand-in → @matt-inspired + 2 primitive swaps (EmptyState, Button). User rejected — the task was tagged "complex, deferred from Conv 205" anticipating substantive account-page design work, not markup tidying on what's already a placeholder stub.

**Learning:** A page that's *already* composed in Matt's design language can still need genuine design work to qualify as a real retrofit. "Already mostly Matt-styled" is not "retrofit complete." For [STANDIN-MATT]-class work, ask before flipping the marker when the task tags complex/Opus and the existing page is a stub — the marker is the visible signal that work is genuine, not just markup-clean.

**Pattern:** When a task tag says complex/Opus and the target file is a placeholder, the retrofit owes real content (or a deliberate "still stub but in proper marker class" surfaced as a question first).

## 3. Single-anti-pattern memory entries beat statement-style "stub pointers"
**Topics:** memory-discipline, option-phrasing, recurring-failures, cc-workflow

**Context:** `feedback_option_phrasing.md` was a "stub pointer" → CLAUDE.md. MEMORY.md indexed it bland-topic-style: "A) B) C) labels rule lives in CLAUDE.md". The rule has been in both files for many convs, and the anti-pattern still happens at least once per conv. User: *"I really, really wish your MEMORY.md and/or CLAUDE.md would give this more importance."*

**Learning:** Rule-as-statement loads but doesn't trigger in-the-moment. Rule-as-anti-pattern-verbatim ("NEVER ask 'X, or Y?' — ALWAYS labels above 👉") works as a self-check when drafting questions. Per `feedback_memory_index_load_bearing`, MEMORY.md lines should expose distinctive markers and anti-patterns, not topic labels. Stub pointers regress to topic-label framing — break that pattern when the rule keeps failing.

**Pattern:** When a directive keeps being violated despite being in always-loaded context, the fix isn't "louder doc placement" alone — it's reformulating from statement to vivid trigger (the verbatim anti-pattern + a self-check question to ask before sending).
