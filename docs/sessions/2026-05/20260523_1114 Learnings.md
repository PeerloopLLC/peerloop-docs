# Conv Learnings - 2026-05-23

## 1. `.scratch/matt-main/*.svg` files are reference-only Figma exports, not pluckable source assets
**Topics:** matt-design, scratch-conventions, dual-repo

**Context:** Mid-conv, while trying to extract individual icons from the 300 KB `Icons.svg`, I assumed the SVG might contain extractable per-glyph paths. User corrected: those SVGs were created several convs ago at CC's instruction as a hedge against not having Figma MCP access. They are catalogue exports from Figma — accurate visual references, but reference-only. The actual extractable source-of-truth Components live in Figma.

**Learning:** Any `.svg` file in `.scratch/matt-main/components/` (or nested screenshots/glyphs inside them) is a rendered reference, NOT a pluckable source asset. To bring components into code, the user re-exports them from Figma directly. Same applies to Module.svg, Note.svg, SocialPost.svg, Icons.svg.

**Pattern:** When designing extraction workflow, ask "what's the source of truth?" before assuming local files are it. Local SVG reference files might be the product of an earlier hedge, not the canonical asset.

## 2. Figma exports use source Material-Icon names, not Matt's catalogue labels
**Topics:** matt-design, figma-export, naming

**Context:** When extracting 39 icons, Figma named them after their Material-Icon source (`newspaper.svg`, `mail.svg`, `calendar_month.svg`, `chat_bubble.svg`, `present_to_all.svg`) — but Matt's catalogue labels are different (`Feed`, `Message`, `Calendar`, `Chat`, `Present`). Without consulting the catalogue, registering them in `icon-paths.ts` under Figma-export names would have broken discoverability — code looking up "the feed icon" would have to know to find `newspaper`.

**Learning:** The catalogue label (Matt's intent) is the authority for naming, NOT the file Figma writes to disk. When extracting from Figma, always cross-reference against the design's authoritative catalogue/index — filenames alone aren't enough.

**Pattern:** For future Figma extractions, build the rename map from a side-by-side: extracted-filename + catalogue-label + final-codebase-name. Catalogue label is the tie-breaker.

## 3. Figma "Include bounding box ✅" gives uniform viewBox — load-bearing for icon systems
**Topics:** figma-export, icon-system, tailwind

**Context:** When advising on Figma export settings, recommended turning ON "Include bounding box" specifically because without it, Figma exports SVGs tight to the visible geometry — a tall narrow icon gets `viewBox="0 0 18 22"` while a short wide one gets `viewBox="0 0 24 19"`. Sizing in code becomes inconsistent: `<Icon class="h-6 w-6" />` would render tall icons "bigger" than wide ones at the same Tailwind class. With bounding box on, every icon gets uniform `viewBox="0 0 24 24"` matching Matt's canvas. Verified: all 39 icons came out with `viewBox="0 0 24 24"`.

**Learning:** "Include bounding box" is the critical Figma export setting for any icon system where consistent sizing matters. Without it, you fight per-icon size adjustments in CSS forever. With it, sizing is uniform and Tailwind utility classes work as expected across the whole set.

**Pattern:** Default ON for icon batches; default OFF for illustrations / artwork where you want the SVG to crop to the actual content.

## 4. Matt's button architecture is 3 orthogonal dimensions, not a flat matrix
**Topics:** matt-design, design-system, figma-variants

**Context:** Spec doc `matt-design-system.md §6` framed Buttons as "6 variants × 3 sizes" — a flat 18-cell matrix. The actual Figma Component reveals 3 orthogonal dimensions: (1) Color via Variable Mode (Primary/Outlined/Course/Student/Creator/Default), (2) State via Property 1 (Default/Hover/possibly Active+Disabled), (3) Size via the 5 stacked frame cells (xs/sm/md/lg/xl). Captured properly, 6+4+5 ≈ 15 surgical captures cover the system, NOT 6×4×5 = 120.

**Learning:** When inspecting a Figma Component before extraction, identify the orthogonal property dimensions. If the spec doc framed the matrix as cross-product, the actual Component may factor independently — capturing independently means dramatically fewer captures.

**Pattern:** Before extraction, list the dimensions Matt's Component uses (Property 1 enum values, Variable Modes, frame-level sizes) and decide whether they're independent (factor) or cross-product (full matrix).

## 5. Matt's button CSS uses `var(--Background)` / `var(--Border)` — cascade IS the intended pattern
**Topics:** matt-design, cascade-broken, css-architecture

**Context:** Dev Mode CSS for Matt's Button Primary showed `background: var(--Background, #...)` and `border: 1px solid var(--Border, #...)`. This validates [CASCADE-BROKEN] as an implementation issue: Matt's design IS authored against the cascade. When our codebase's `.entity-*` class fails to propagate through Tailwind 4's `bg-entity-background`, that's our build/integration issue, not a design choice.

**Learning:** Inspecting a designer's authored CSS reveals architectural intent that visual inspection alone cannot. If you see `var(--Token)` in Dev Mode output, the designer is explicitly relying on cascade — implementation must support it.

**Pattern:** When debugging cascade issues, always check the designer's source CSS for `var(--*)` references before assuming the design uses hard-coded hex.

## 6. MCP servers load at session start — mid-conv configuration has no effect
**Topics:** mcp, cc-workflow, session-lifecycle

**Context:** When the user announced Pro account + asked if I could "see the files," I confirmed via ToolSearch that no Figma MCP was connected this session. Even if MCP config landed mid-conv, this session's tool list is frozen at start. To enable MCP for use, the sequence must be: config → /r-end current conv → /clear or /r-start fresh → new conv loads MCP at start.

**Learning:** MCP configuration is a session-boundary affair, not a runtime one. Plan MCP setup work around conv boundaries — not during active work.

**Pattern:** For MCP onboarding, schedule: (a) /r-end existing conv to commit work, (b) install/configure MCP between convs, (c) /r-start fresh to pick up new tools, (d) verify presence via ToolSearch in new conv before relying on it.

## 7. "Harvest while access is available" — time-bounded resource pattern
**Topics:** workflow, planning, solution-quality

**Context:** Mid-conv, user revealed Figma Dev Mode trial ends today. I'd been about to start narrowly on CMP-ICN. Pivoted to recommend full 8-subtask harvest pass while access was live. Per CLAUDE.md §Solution Quality default-durable: quick path (icons only) lost to durable path (all CMP source assets) because the access window itself was the limit. This pivot was retrospectively unnecessary (Pro account access came online same day) but the reasoning was right.

**Learning:** When a resource has a time deadline, the durable move is "harvest everything you'll need across the full block scope," not "consume just what you need today." Even if the deadline turns out to be soft, the pattern protects against cliff-edge access loss. Precedent: Conv 172 Figma Variable extraction harvested 35 variables in one pass during the same trial.

**Pattern:** Before draining a time-bounded resource, list every block-scope artifact that depends on it, harvest them in one pass into a structured drop-location, then build off the local cache.
