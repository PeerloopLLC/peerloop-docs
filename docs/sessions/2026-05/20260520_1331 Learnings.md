# Conv Learnings - 2026-05-20

## 1. "Wall of terminal errors + blank page on dev" = the existing [BR-NAVBAR-HYDRATE] symptom, broader than admin-only
**Topics:** astro, testing, dual-repo

**Context:** User reported "many loading errors" on local dev. After probing, no errors were found anywhere — the page rendered cleanly, zero console errors, dev-server log clean. User clarified: errors were in the *terminal* (not browser DevTools) and the page was *blank* on first load but recovered after a few seconds on retry. This matches the existing [BR-NAVBAR-HYDRATE] task signature exactly.

**Learning:** Vite/Astro hydration mismatches dump a wall of HMR errors into the dev-server terminal AND blank the page until client hydration completes. The symptom is invisible to anyone *not* looking at the terminal at the moment it happens (it self-heals). When a user reports "many loading errors", ask whether it's terminal output vs browser console — they're very different signals. Also: [BR-NAVBAR-HYDRATE] was originally diagnosed as admin-only (`AdminNavbar` `<div>` vs `<a>`), but user hit the same symptom on a non-admin page — scope is wider than diagnosed, likely a shared parent component.

## 2. Local dev seed never modeled a recording-having session — every test required hand-populating `recording_url`
**Topics:** testing, d1, video

**Context:** [REC-LABEL] needed local-dev validation of 10 recording-link surfaces. The seed had `recording_url TEXT` defined on `sessions` but the seed's INSERT statement omitted the column entirely — every seeded session shipped with NULL. To exercise any recording-link surface locally, you had to manually `UPDATE sessions SET recording_url=…` after seeding.

**Learning:** When a feature is gated on a nullable column being non-NULL, seed at least one row that exercises the gated state. Otherwise every developer testing that feature does the same hand-population dance. Fix landed Conv 163: `migrations-dev/0001_seed_dev.sql` now ends with an `UPDATE sessions SET recording_url = '<real Blindside URL>' WHERE id = 'ses-sarah-n8n-1'` so the n8n session always has a working recording in fresh local DBs.

## 3. Chrome MCP can have multiple browsers connected — `switch_browser` is the cleanest disambiguator
**Topics:** cc-workflow

**Context:** Two Claude Code sessions both had /chrome enabled. The Chrome MCP's `list_connected_browsers` returned 2 browsers; either ambiguity needed resolving before any browser action. Exiting the other session's terminal did NOT disconnect — Chrome extension holds the connection independently of the CC terminal.

**Learning:** `switch_browser` is the right disambiguator when multiple browsers are connected: it broadcasts a Connect prompt to every extension, and the user clicks in the one they want this session to drive. Telling the user to "exit /chrome in the other session" doesn't work alone — the extension persists past terminal exit. Either ask the user to click Disconnect in the extension popup, brute-force close the Chrome window, or use `switch_browser`.

## 4. Astro 6.1.5 → 6.3.6 has identical absolute-path-leaking line in `vite-plugin-astro/compile.js:50` — upstream fix not landed
**Topics:** astro, dual-repo

**Context:** Found Astro dev emitting `<script src="/Users/jamesfraser/projects/Peerloop/node_modules/...">` (absolute filesystem path) into HTML. Verified by `npm pack astro@6.3.6` + diff: the buggy line is byte-identical in latest stable. The naïve fix (relative path) doesn't work because Vite resolves relative imports against the importer, which is the same .astro file.

**Learning:** Cosmetic dev-only path leak; functionally a no-op (Vite serves 200 either way) but cross-machine portability hazard. Documented as [AAP] with verification probe (`curl http://localhost:4321/ | grep -oE 'src="[^"]*ClientRouter[^"]*"'`) for each future Astro upgrade. Don't burn time on workarounds — defer until upstream lands a fix.
