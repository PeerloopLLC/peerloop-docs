# Conv Learnings - 2026-05-28

## 1. Investigative framings require surfacing findings before acting
**Topics:** cc-workflow, output-formatting

**Context:** User asked "what could be done about" near-cap MEMORY.md, picked option C from a multiple-choice; CC interpreted that as authorization to execute end-to-end. User pushed back: structurally impossible for the user to preemptively scope an action whose findings they haven't seen.

**Learning:** §Solution Quality's "proceed without explicit approval" + §Critical Rule's "size ≠ novelty" carve-outs are calibrated for **directive** requests, not **investigative** ones. Verbs like *audit / review / investigate / examine / classify / analyze* trigger a fundamentally different mode: surface per-item dispositions, get 👉👉👉 confirmation, then execute. Picking an option within an investigative framing authorizes the *approach* (depth/scope), not the *execution*.

**Pattern:** New CLAUDE.md §Investigative Framings + memory/feedback_audit_surface_findings_first.md. The verb test: "tell me what's true / what's there / what's wrong" → surface first. "Make this change / build this thing / fix this bug" → proceed.

## 2. Skill `!`-backticks must use tilde-literal paths, not relative or `$VAR`
**Topics:** cc-workflow, skills

**Context:** /r-prune-claude initial run returned `CLAUDE.md line count: 12` (the SKILL.md itself, not project CLAUDE.md). `/r-optimize` had hardcoded `livingroom` username (would break on M4Pro `jamesfraser`).

**Learning:** The Conv 162 sweep convention from `feedback_git_dash_c_enforcement.md` applies to skill `!`-backticks identically: relative paths resolve from the skill's base directory (not project root); `$VAR` triggers `simple_expansion` permission prompt; hardcoded usernames break cross-machine portability. Tilde-literal (`~/projects/peerloop-docs/...`) is the only form that's both portable + prompt-free.

**Pattern:** All future skills' `!`-backticks should use `~/projects/peerloop-docs/...` and `~/projects/Peerloop/...`. Validated by fixing r-prune-claude and r-coherence-check during this conv.

## 3. Skill arguments reach the skill body text, NOT bash `!`-backticks
**Topics:** cc-workflow, skills

**Context:** P1 design for /r-coherence-check used `if echo "${ARGUMENTS:-}" | grep -qE '\-\-(deep|...)'` in `!`-backticks to conditionally load heavy content. On first `--deep` invocation, Mode detected as STRUCTURAL — `$ARGUMENTS` was empty in bash; the harness appended `ARGUMENTS: --deep` to the prompt body instead.

**Learning:** Skills can't conditionally do bash work based on args. Either always run the work (paying the cost every invocation) or move the conditional to the skill body which CAN read the prompt-bottom `ARGUMENTS:` line. The safe-fallback worked (default-to-structural) but the design's cost-savings assumption was incorrect.

**Pattern:** For arg-driven skill behavior, parse `ARGUMENTS:` in the skill body's Step 0, not in `!`-backticks. The `${ARGUMENTS:-}` default-to-empty trick still gives a sane fallback when args are unset, but won't actually fire on `--deep` etc.

## 4. Stub-pointer pattern: CLAUDE.md = rule, memory file = motivating case
**Topics:** cc-workflow, memory-system

**Context:** Multiple memory files (feedback_pointing_emoji_prefix, feedback_default_durable_no_ask, feedback_baseline_includes_astro_check, feedback_verify_baselines_in_conv, feedback_audit_surface_findings_first, etc.) follow this shape.

**Learning:** Behavioral rule lives in CLAUDE.md (always-loaded). Memory file body retains motivating-case archaeology + edge cases + Conv-N incident detail (read on-demand). MEMORY.md index line provides the distinctive-marker hook that triggers grep matches. The three layers are different cost/freshness profiles.

**Pattern:** When promoting a memory-only rule to CLAUDE.md (like multi-conv scope this conv), do the followup edit to the memory file: change "(not in CLAUDE.md)" → "(promoted Conv N to CLAUDE.md §X)" so future-CC sees the canonical home.

## 5. Tiered invocation (structural cheap default + deep heavy opt-in) for audit skills
**Topics:** cc-workflow, skills

**Context:** /r-coherence-check P1 improvement. Initial composite design loaded full CLAUDE.md + all memory files on every invocation (~170KB pre-computed). Tiered design lets structural mode run cheap while deep mode pays for semantic analysis only when explicitly invoked.

**Learning:** The cadence-cost asymmetry is real: reference rot checks fire daily; semantic deep review fires monthly. Bundling them under one entry point means the rare-but-expensive check determines the skill's invocation friction. Splitting via flag (default + `--deep`) keeps both checks discoverable through one name while letting each carry its own cost.

**Pattern:** For audit/lint skills with both deterministic and judgment-based checks, default to the cheap deterministic set and gate the heavy semantic set behind a `--deep` flag. The `argument-hint: "[--deep | --semantic | --all]"` frontmatter declares accepted variants.
