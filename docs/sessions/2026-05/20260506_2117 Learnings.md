# Conv Learnings - 2026-05-06

## 1. Output redirection scope traps in bash skill blocks
**Topics:** skill-authoring, bash, cc-workflow

**Context:** Conv 155 wrote a `{ echo header; echo body; echo "$DIFF_OUT"; } > "$LOG"` block intending the header to log only and the diff to also print to stdout. The braces redirected EVERYTHING to file; the user couldn't see what came in cross-machine. Conv 156 audit revealed the diff DID run correctly (1 modified MEMORY.md + 3 new files matching the Conv 155 payload) — only the display layer was broken.

**Learning:** When a bash author wants a structured stream to log AND a useful subset to display, the safer shape is two separate emit blocks (one to file, one to stdout) rather than a single `{...} > file` block. `tee` is a third option (same stream both places) but it sends the WHOLE stream to stdout, which may be too verbose. Choice between "two separate blocks" and `tee` depends on whether the displayed text is identical to the logged text.

**Pattern:**
```bash
# Log full forensics
{ echo "=== header ==="; ...verbose detail...; } > "$LOG"
# Display user-relevant subset
echo "Summary: ..."
echo "$IMPORTANT_PART"
```
Reviewers should mentally check every `{ ...; } > "$LOG"` block: "is the user supposed to see anything inside these braces?"

## 2. Special-case architecture pays off when the rule generalizes
**Topics:** software-design, skill-authoring

**Context:** Conv 155 implemented a halt-and-ask path for the specific data-loss case (`Only in $LIVE`). Conv 156 user asked to extend halt-and-ask to ALL non-empty diffs. The redesign was small because the halt-and-ask architecture already existed — the change was widening the predicate, not introducing the pattern.

**Learning:** When you build a halt-and-ask path for a special case, the structure is reusable. Future generalization is a predicate change, not an architecture change. "The architecture you build for the special case becomes the architecture you have for the general case." Implication for skill authoring: when adding a guard for a narrow risk window, prefer the structural shape that would generalize cleanly (visible bash branches, separate phases, named conditions) over an inline ad-hoc check.

## 3. Two-phase bash split is the canonical halt-and-ask shape for skills
**Topics:** skill-authoring, cc-workflow

**Context:** Implementing the always-pause Step 5.7 redesign needed a way to halt the skill mid-execution and ask the user, then conditionally run rsync. Bash can't prompt interactively from inside a skill — Claude is the executor, the user types responses to Claude.

**Learning:** Skill bash that needs user approval before a destructive operation must split into two `\`\`\`bash\`\`\`` blocks separated by Claude prompt prose: Phase 1 detects/displays/halts, Phase 2 runs the action. Inline `read` prompts don't work (Claude doesn't run interactive bash sessions). A three-phase split (forensics → ask → action) adds no value because the "ask" step is just Claude prose, not a separate bash. Step 5.5 (npm install drift) follows the same shape — this is now the canonical shape across the skill suite.
