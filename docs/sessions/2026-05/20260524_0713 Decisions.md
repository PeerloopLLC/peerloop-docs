# Conv Decisions - 2026-05-24

## 1. Standardize Conv 184 primitives + MattIcon on React (.tsx)
**Type:** Strategic
**Topics:** astro, react, architecture

**Trigger:** SocialPost.tsx (React) needed to import the Conv 184 primitives (built as .astro). React can't import Astro.

**Options Considered:**
1. Convert SocialPost.tsx → SocialPost.astro (one file conversion, but big refactor — slot conversion, embed-prop rework)
2. Build React-counterparts alongside .astro versions (duplication; defeats single-source-of-truth)
3. Convert all Conv 184 primitives + MattIcon to React (.tsx); Astro pages import the React versions as static SSR ← Chosen

**Decision:** All design-system leaf primitives are React (.tsx). Astro page-wrappers may import them and render as static HTML at SSR (no JS bundle cost). Files converted: MattIcon, UserIcon, EntityPill, EntityLink, IconLabelChip, CourseAnchor.

**Rationale:** React's broader compatibility — Astro can natively render React; the reverse is unsupported. Standardizing primitives on React removes the architectural seam.

**Consequences:** All current Astro callers (SubNavItem.astro, /matt/index.astro showcase, CourseHeader.astro residue) work transparently after import-path + `class`→`className` updates. Future Astro/React seam questions resolve to "primitives are React; pages are Astro."

## 2. 9 distinct anchor row components, no shared `AnchorRow` base
**Type:** Strategic
**Topics:** matt-design-system, component-architecture

**Trigger:** Restructure question for Entities + Post Anchors as collections — should each Post Anchor be a thin wrapper over a shared base, or fully independent components?

**Options Considered:**
1. 9 standalone components, no shared anything (Option A) — too much duplication of outer-flex shape
2. Generic `AnchorRow` primitive + 9 thin wrappers (Option B) — introduces an abstraction Matt didn't name; data-shape heterogeneity forces slot-heavy assembly
3. 9 distinct components each composing from leaf primitives (UserIcon / EntityPill / IconLabelChip etc.) but no shared row base (Option C) ← Chosen

**Decision:** Each anchor row is its own component. Reuse happens at the leaf-primitive level, not the row-shape level.

**Rationale:** Matt's component-naming convention is the load-bearing signal — he named by content type, not row shape. Per `feedback_tokenize_only_matt_variables.md`: extract what Matt named. The heterogeneous data per anchor type also resists clean abstraction.

**Consequences:** CourseAnchor built this conv; 8 remaining anchor types (Creator/Certification/Module/Resource/Review/Student-Teacher/Video Clip/Milestone) tracked as `[CMP-ANCH-REST]` for Phase 5.

## 4. New standing rule: scan `<instance>` children, import existing primitives, never inline duplicates
**Type:** Strategic
**Topics:** cc-workflow, design-translation, matt-design-system

**Trigger:** Audit revealed SocialPost inlines Avatar/ActionPill duplicates; CourseAnchor (built same conv) inlined the CTA button styling instead of importing Button.tsx. Both violations stemmed from "Astro/React boundary as friction" reasoning — friction that proved fictional.

**Options Considered:**
1. Fix the specific violations and move on (no codified rule)
2. Save as standing memory rule + retroactive application to CourseAnchor + commitment to scan `<instance>` nodes on every future Figma render ← Chosen

**Decision:** New rule `feedback_reuse_existing_components.md` requires `get_metadata` scan of `<instance>` children before any Figma render; each instance name must map to an existing imported component or surface as a missing-primitive gap.

**Rationale:** Inlined duplicates accumulate as design-system drift; refactors to consolidate are slow and skippable. Codifying the rule catches the next violation before it lands.

**Consequences:** CourseAnchor refactored to use Button.tsx. SocialPost refactored to use UserIcon + IconLabelChip + AnalyticCount + CourseAnchor (embed). _SocialPostDemo updated. All future component builds will scan `<instance>` nodes first.

## 5. UserIcon image-mode extrapolation pulled forward to Conv 184
**Type:** Implementation
**Topics:** matt-design-system, strict-b

**Trigger:** SocialPost.tsx needs `avatarUrl` support for production feeds (real user images). UserIcon as originally built was initials-only (Matt drew only initials). Three resolution paths surfaced.

**Options Considered:**
1. Extend UserIcon with `avatarUrl?` now — pragmatic strict-B extrapolation ← Chosen
2. Wrap UserIcon in SocialPost — initials for avatarUrl absent, inline `<img>` fallback
3. Defer SocialPost refactor until Phase 6

**Decision:** Option 1 — added `avatarUrl?` prop to UserIcon.tsx with explicit "strict-B extrapolation per `feedback_tokenize_only_matt_variables.md`" annotation in header.

**Rationale:** Real avatars are a production requirement Matt didn't draw. Extending a primitive with a documented extension is honest; building a wrapper or deferring violates the user's "use existing primitives" rule.

**Consequences:** UserIcon now supports both modes. SocialPost.tsx uses `<UserIcon initials avatarUrl label />` cleanly. Honest-orphan principle satisfied via header annotation.
