# Conv Decisions - 2026-05-06

## 1. Drop symlink approach for cross-machine memory sync
**Type:** Strategic
**Topics:** cc-workflow, dual-repo

**Trigger:** User proposed hook-based sync; Claude surfaced symlink as a simpler alternative.

**Options Considered:**
1. Hook + packet system (user's proposal)
2. Symlink the memory dir directly into the repo (simpler, eliminates dual-store)
3. Hybrid

**Decision:** Reject the symlink approach (Option 2). Proceed with the hook/skill-based mirror approach (eventually became skill-based, not hook-based — see Decision 2).

**Rationale:** User stated explicit preference: "I don't want to use symlinks." Hook/mirror approach also "documents the memories inside the repo" — visible in PR diffs/git log — which the user values for paper-trail reasons. The architectural simplicity of symlinks doesn't outweigh the visibility benefit of an explicit mirror.

**Consequences:** Took on the responsibility of maintaining a redundant copy of memory state inside the repo; offset by using checksums and (eventually) git itself as the manifest. [CMS] task replaced by [MSI] (Memory-Sync Integration).

## 2. Skill-based memory sync (no Claude Code hooks)
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** Original framing was "adapting one or more hooks." Design discussion revealed all sync points coincide with existing skills (/r-start, /r-commit, /r-end).

**Options Considered:**
1. PostToolUse + SessionStart/End hooks (continuous mid-conv sync)
2. Inline steps in /r-start, /r-commit, /r-end skill bodies (sync only at skill touchpoints)

**Decision:** Option 2 — skill-based only. No hook involvement. Mirror is allowed to lag live during the conv; sync happens at skill invocation points.

**Rationale:** Skill execution is explicit and user-invoked, with no event-firing edge cases. The "one machine at a time" + "always pull first" invariants make mid-conv continuous sync unnecessary — the other machine only sees state at /r-end push boundaries anyway. Lower complexity, lower failure surface, easier to reason about.

**Consequences:** Mirror reflects "last commit point" rather than "live state at every moment." Working tree may be momentarily inconsistent mid-conv (mirror lags live until next /r-commit or /r-end), but that inconsistency is invisible to the other machine.

## 3. No separate index file; no manifest.md; git is the manifest
**Type:** Implementation
**Topics:** cc-workflow, dual-repo

**Trigger:** Initial design proposed `memory-index.md` with checksums. User questioned whether it was needed.

**Options Considered:**
1. Mirror + checksum index file (committed)
2. Mirror + human-readable manifest.md (committed, regenerated)
3. Mirror only — git itself serves as the manifest

**Decision:** Option 3. Single committed `memory-sync/memories/` directory. No index, no manifest, no .gitignore additions for memory-sync.

**Rationale:** Comparing two trees is what `diff -rq mirror live` does in one call. SHAs are derivable on demand from filesystem state. A materialized index would only matter if we were detecting tampering; for synchronization it's redundant. Manifest.md would have been purely aesthetic and would risk drifting from the mirror unless deterministically regenerated.

**Consequences:** Smallest possible repo footprint for the feature. No file to keep in sync with the mirror. Hooks/skills only need to invoke `rsync` and `git add`, nothing more.

## 4. Bootstrap via first /r-end, not a separate init skill
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** Mirror dir doesn't exist in either machine's repo at the moment the design lands; need a one-time initialization.

**Options Considered:**
1. Standalone bootstrap skill (e.g., `/w-memory-sync-init`)
2. Treat the first /r-end (or /r-commit) after this lands as the bootstrap — `mkdir -p` + first rsync naturally seeds the mirror
3. Manual one-time script in `.claude/scripts/`

**Decision:** Option 2 — no separate bootstrap step.

**Rationale:** Self-bootstrapping system; the day the code lands, the first invocation of /r-end or /r-commit on either machine seeds the mirror. No ceremony, no remembering to run init, no separate skill to maintain. Idempotent — if mirror already exists, the rsync just refreshes it.

**Consequences:** Conv 154's /r-end IS the bootstrap (M4 side). M4Pro side will bootstrap-by-pull on its next /r-start.

## 5. Concurrent-edit conflicts surface via git's native merge-conflict UX
**Type:** Implementation
**Topics:** cc-workflow, dual-repo

**Trigger:** What happens if both machines edit the same memory file between syncs?

**Options Considered:**
1. Custom 3-way merge logic in the skill
2. Detect via per-machine ledger and fail-stop with diff
3. Rely on git's existing merge-conflict behavior on the in-repo mirror; /r-start already halts on diverged pull

**Decision:** Option 3 — git native conflict UX is acceptable.

**Rationale:** Under "one machine at a time + always pull first," concurrent edits should be rare. When they do happen, surfacing them through git's standard merge-conflict mechanism is loud and familiar; user resolves manually before /r-start can complete its pull step. No special handling needed in the sync logic itself.

**Consequences:** Sync logic stays minimal — no merge code, no per-machine ledger, no per-file mtime tracking. Conflict resolution is user-driven via standard git tools when (rarely) needed.

## 6. /r-start adds explicit Read of MEMORY.md after sync (Layer 2 mitigation for auto-load lag)
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** Verified that MEMORY.md is auto-loaded at SessionStart but not refreshed mid-session. /r-start runs AFTER SessionStart, so its mirror→live sync wouldn't otherwise reach Claude's auto-context this conv.

**Options Considered:**
1. Accept one-conv lag — Claude's auto-context shows pre-sync MEMORY.md until next session
2. Explicit Read(MEMORY.md) in /r-start after the rsync — fresh content lands as tool result, visible to Claude this conv

**Decision:** Option 2.

**Rationale:** Trivial cost (one Read call), restores parity with the user's current manual workflow (which has no lag because manual rsync happens before Claude starts). Preserves the cross-machine sync's intended responsiveness. Sub-files don't need this treatment because they're read on-demand by Claude as needed, and on-demand reads naturally see freshly-synced content.

**Consequences:** /r-start Step 5.7 explicitly performs the Read after the rsync, citing the documented 200-line/25KB SessionStart cap as supporting context.
