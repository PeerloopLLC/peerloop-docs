# Conv Decisions - 2026-05-22

## 1. Refactor ToDoItem from hybrid controlled+uncontrolled to fully controlled (no useState)
**Type:** Implementation
**Topics:** astro, cloudflare, react, frontend

**Trigger:** ToDoItem's `useState` triggered the [DSSR-SCOPE] React-hooks-null SSR crash, cascading to Sidebar.tsx and zero-ing the entire page body. Diagnostic test (removing ToDoItem from the showcase) confirmed it as the trigger.

**Options Considered:**
1. Keep `useState`, use `client:only="react"` to skip SSR on instances — adds boilerplate everywhere; doesn't fix the cascade-crash to Sidebar
2. Refactor ToDoItem to fully controlled — parent owns `checked` + `onChange`; primitive becomes stateless and SSR-safe ← **Chosen**
3. Try the upstream dedupe + noExternal workaround now — risky during active matt/* work; Conv 122 reportedly tried and failed

**Decision:** Refactor to fully controlled. Discipline rule for matt/* primitives until [DSSR-SCOPE] is upstream-fixed: NO `useState` / `useEffect` in primitive components.

**Rationale:** Controlled-only is also cleaner architecturally — parent state (e.g., the onboarding checklist DB record) is the single source of truth anyway. Stateless primitives align with the rest of the matt family (Module/Note/SocialPost/RoleTabBar are all hookless).

**Consequences:** Production callers must wire `checked` + `onChange` explicitly. Demos that want interactive toggles need a small wrapper React component owning state. Tracked as [DSSR-SCOPE] task #26 — when upstream fix lands, the discipline rule relaxes.

## 2. Use direct entity-specific Tailwind utilities, not the `.entity-*` cascade, inside matt/* primitives
**Type:** Implementation
**Topics:** tailwindcss, frontend

**Trigger:** User feedback that active Module + ToDoItem rendered grey instead of pastel-blue. Investigation showed the `.entity-student` → `--Entity-Background → bg-entity-background` cascade doesn't propagate in this Tailwind 4 + Astro setup.

**Options Considered:**
1. Debug the cascade until it works (root-cause Tailwind 4 / @theme behavior) — open-ended; no obvious lever
2. Use direct entity utilities per variant — matches `Button.tsx`, works empirically ← **Chosen**
3. Use arbitrary values (`bg-[var(--Student-Background)]`) — bypasses the bridge but loses Tailwind ergonomics

**Decision:** Direct utilities in primitives (`bg-student-background` per `entity="student"`, etc.). Retain `.entity-*` cascade for non-primitive components that work via `var(--Entity-Background)` directly.

**Rationale:** Reliability over elegance. The cascade pattern documented in matt-design-system.md §5 is broken in some way we don't understand yet; ship-ready primitives can't depend on it. The direct pattern is what `Button.tsx` already uses, so consistency wins.

**Consequences:** matt-design-system.md §5 needs an addendum or revision describing when the cascade works and when not. Tracked as [CASCADE-BROKEN] task #28.

## 3. Extract React component demos with rich JSX content into `_Demo.tsx` files; don't inline in .astro pages
**Type:** Implementation
**Topics:** astro, frontend

**Trigger:** Two distinct Astro parser crashes — inline `<svg viewBox=…/>` (Module/SocialPost icons) and inline `<div className=…>` (SocialPost Course-embed) — both inside `.astro` expression blocks. Web research confirmed this is Astro by design, not a bug.

**Options Considered:**
1. Reduce showcase content (no embed, no rich icons) — visible regression
2. Convert all matt-design-system showcase content to React `_Demo.tsx` files ← **Chosen**
3. Wait for Astro to support full JSX in expression blocks (roadmap #716) — open-ended timeline

**Decision:** When a React component prop needs rich JSX, write the consumer as a React component file (underscore-prefixed for internal/showcase usage) and reference it from `.astro` as a single `<ComponentReference />`.

**Rationale:** Astro's expression-block syntax restriction is documented and stable. Working with the grain instead of fighting it. The `_Demo.tsx` pattern is also a clean separation — showcase wrappers stay out of production page files.

**Consequences:** matt-pre-plan.md should document this as the canonical pattern (showcase / demo content goes in React, not in expression blocks).
