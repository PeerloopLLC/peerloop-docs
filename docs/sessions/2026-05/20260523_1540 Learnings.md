# Conv Learnings - 2026-05-23

## 1. Figma Remote MCP is link-based by design — don't enumerate
**Topics:** figma, mcp, design-tools, architecture

**Context:** Spent multiple turns trying to "list pages and drill down" — got incomplete results (2 of 9 pages listed). User pushed back when expected pages were missing. WebFetched Figma's docs and found explicit statement that the API expects user-supplied URLs.

**Learning:** The Remote MCP is designed for user-as-navigator: user copies a Figma link (right-click → "Copy link to selection"), pastes the URL, AI queries that specific node. Trying to crawl the file structure fights the API design. Page listing returns only currently-scoped pages; ANY node-id can be queried directly (invisible-but-accessible pattern).

**Pattern:** When using Remote MCP for a file with multiple pages, immediately ask the user for the page URLs as a lookup rather than attempting `get_metadata(fileKey)` discovery. Store as `.scratch/<project>-figma-pages.md` for reference.

## 2. MCP `get_design_context` output shape varies by node type
**Topics:** figma, mcp, code-generation

**Context:** Probed three different node types in this conv: section (Button Primary section node), component variant (Button Primary > Default), and page/card (Course In Feed). Each returned dramatically different content.

**Learning:** Section nodes return sparse metadata + "drill into sublayers" instruction (no code). Component variant nodes return typed React with INFERRED PROPS (`hasLabel`, `hasLeadingIcon`, etc.) — most useful for primitive building. Page/card nodes return inlined HTML+CSS with `data-name` attributes — NOT named component references. The output type determines whether you can build primitives, render visual layouts, or just navigate.

## 3. Translation is mandatory — there is no paste-in workflow
**Topics:** figma, mcp, implementation-workflow

**Context:** Was about to recommend rework on Conv 178 SVG harvest based on the assumption that MCP output would reference icons by name, enabling efficient paste-in. Probe revealed icons rendered as `<img>` with expiring asset URLs, components inlined as raw markup, and MCP itself explicitly warns: *"The generated React+Tailwind code MUST be converted to match the target project's technology stack"*.

**Learning:** Registry-key naming convention is DECOUPLED from MCP output. A translation table handles the mapping regardless of whether keys are Figma symbol names or designer labels. Code-readability dominates the choice. The `data-name` attribute is the translation key for substitution.

## 4. Variable Mode bakes into CSS-var fallback hex
**Topics:** figma, design-tokens, css-variables

**Context:** Course In Feed card (in Course context) had `var(--background,#e8f4df)` (pastel-green) and `var(--primary,#327d00)` (dark-green) inline. The Button Primary Default variant (in Auto Primary context) had `var(--background,#0777b6)` (americana-blue). Same `--background` variable name, different fallback hex per parent context.

**Learning:** Figma MCP renders CSS variables with the resolved Variable Mode's primitive baked into the fallback hex. Same variable reference means different things in different parent contexts. Implication: our codebase needs to define those CSS variables (--background, --primary, --color, --border, --text/default, --text/tertiary, --white) AND implement Variable Mode switching (likely `[data-variable-mode="course"]` selector pattern) for the cascade to work.

## 5. Verify external tool behavior empirically before recommending
**Topics:** workflow-discipline, recommendations, claude-behavior

**Context:** Made 4+ architectural recommendations about Figma Remote MCP this conv (page filtering being Dev-seat related, search_design_system being suitable for icon discovery, MCP returning name-referenced components, registry naming optimizing for paste-in efficiency). EVERY one was reversed by the next probe. User pushback: *"what you think is going on often is inaccurate or incompletely understood. I don't want recommendations prematurely."*

**Learning:** When a recommendation depends on how an external tool behaves, probe the tool on a real node BEFORE recommending. One tool call beats one rework cycle. This pattern got captured as the [EMP] context in feedback_external_source_of_truth_first.md.

## 6. Matt's "Ready for Dev" labels are human-visible, not API-readable
**Topics:** figma, design-handoff, matt-collaboration

**Context:** Found "Section Title" component set on Components page with WIP/Dev Ready/Archived variants. Hypothesized this was Figma metadata the MCP could filter on. User clarified: Matt erects a separate green-banner label frame above each finished frame (with title + "Last Touched" date) — purely visual convention, not Figma's built-in frame-level Dev-Ready flag.

**Learning:** Dev-Ready status is NOT API-detectable in this Figma file. To know what's Ready for Dev, ask the designer or visually scan the file. Future skill design should not try to programmatically detect Dev-Ready frames.
