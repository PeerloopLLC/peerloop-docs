# Conv Learnings - 2026-05-21

## 1. Figma MCP requires the viewer's own Dev seat, not just file-level Dev Mode access
**Topics:** figma, mcp, design-tooling

**Context:** Spent ~30 min troubleshooting why the "Enable local MCP Server" toggle was missing from the user's Figma desktop Preferences despite Dev Mode being active in Matt's file (3-day shared window) and the desktop app being on a recent version (126.3.12).

**Learning:** Dev Mode permissions on a shared file are not the same as Dev Mode capability on your own seat. The MCP server toggle only appears when YOUR account has a Dev-seat-eligible plan. Sharing Dev Mode access to a file lets you *inspect* code in that file but does not grant MCP server access from your local desktop app. Brian's paid Figma setup tonight is expected to unblock this.

**Pattern:** Before promising MCP-based workflows, verify the user's own account has the right seat — not just file-level access. Quick test: open Figma → Preferences → search for "MCP". If absent on a recent version, the seat is the gate.

## 2. Prod D1 migration drift can be invisible: tracker retains old filename after a seed-split rename
**Topics:** d1, deployment, migrations

**Context:** Audited prod D1 migration state during PROD-PW-APPLY prep. Local and staging both reported "No migrations to apply", but prod listed 3 pending. Investigation revealed prod's `d1_migrations` table had entries for `0001_schema.sql` and `0002_seed.sql` — the OLD filename from before the seed-split rename to `0002_seed_core.sql`. The current `0002_seed_core.sql` SQL had been applied (data present) but the tracker thought it hadn't been; `0003` and `0004` had never been applied (so `feed_visits` and `feed_activities` tables are missing on prod).

**Learning:** When migration files are renamed in a project's history, the `d1_migrations` table on environments that ran the migration under the old name retains the old name forever. `wrangler d1 migrations list` will then falsely report the renamed file as pending. Fix: `UPDATE d1_migrations SET name = '<new>' WHERE name = '<old>'` on the affected environment.

**Pattern:** When auditing migration state across environments, compare both (a) wrangler's "Migrations to be applied" list AND (b) actual schema state via `SELECT name FROM sqlite_master`. The tracker can lie about applied files when renames have happened; the schema can lie about which formal migration produced the current state.

## 3. Figma batch export auto-names files after frame names — replaces manual inventory
**Topics:** figma, design-tooling, workflow

**Context:** Originally instructed user to screenshot the Layers panel for the Happy Path page to get a frame-name list, then separately batch-export frames. User pointed out that expanding the tree manually would take 20 minutes and miss frames.

**Learning:** Figma's batch export (`⌘A` on canvas → right panel Export → `+` → format → click Export) produces one file per selected frame, each named after the Figma frame's display name. The directory listing after export IS the frame inventory — no separate screenshot needed.

**Pattern:** For inventory + export of a Figma page's frames, skip the layers-panel screenshot entirely. Use `⌘A` + batch export, then derive the frame list from filenames. Saves ~20 min on dense pages.

## 4. Figma exports preserve internal frame nesting as folder hierarchy — actual screens are often one or two levels deep
**Topics:** figma, design-tooling

**Context:** When the user exported the Happy Path page, the top level of `happy path/` had 113 files (mostly milestone "🏆 Purpose_" frames, design assets, and unnamed `Frame N` items). The actual 31 happy-path screens (Course About, Modules, Resources, Teachers, Certification, etc.) were nested at `happy path/Content/Happy/`.

**Learning:** Figma's batch export preserves the page's internal frame nesting structure as folder hierarchy. A frame named "Content" on the page becomes a folder, with its child frames as files inside. When triaging exports, the *actual* screens are often one or two folder levels deep, not at the export root.

**Pattern:** When inventorying a Figma export, recursively scan all subfolders. The most-deeply-nested SVGs with screen-like names (Course X, About Y) are usually the actual implementable screens; the top level is often milestone labels and supporting assets. Be aware: Figma also preserves user typos verbatim in filenames (e.g., `typograhy-overview.png` — missing second 'p') and Unicode page names (e.g., `α1 Happy Path` → `happy path/` with the α prefix surviving the export).
