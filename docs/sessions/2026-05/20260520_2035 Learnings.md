# Conv Learnings - 2026-05-20

## 1. Astro→React `client:load` cannot serialize React element nodes across the prop boundary
**Topics:** astro, frontend, react

**Context:** CRT-3 (Conv 165) hit a runtime error when an `.astro` file constructed JSX in frontmatter and passed it as a prop to a `client:load` React island. The element got JSON-serialized into a plain `{$$typeof, type, key, props, _owner, _store}` object that React rejected as invalid child. TypeScript accepted it (ReactNode is the right type) — only runtime catches it.

**Learning:** Pass primitive descriptors (booleans, strings, IDs) across `client:load` boundaries and let the island construct JSX internally. The chassis pattern locked in Conv 165 (CourseTabs builds extra-tab content from role flags) is the correct shape for any future role-based extra tabs — confirmed by the CRT-4 implementation following it cleanly.

**Pattern:** `<CourseTabs isTeacherOfCourse={isTeacherOfCourse} isCreatorOfCourse={isCreatorOfCourse} ...client:load />` — flags only; CourseTabs imports child components and renders `<TeacherSessionsTabContent />` / `<AllSessionsTabContent />` etc. internally.

## 2. Astro routes static paths over dynamic ones, enabling catch-all without shadowing
**Topics:** astro, routing

**Context:** CRT-DEDICATED-PAGES chose a single dynamic `[tab].astro` to handle 4 role-tab URLs without cloning `sessions.astro` four times. Initial concern: would `[tab].astro` shadow the 7 existing static routes (sessions.astro, feed.astro, etc.)?

**Learning:** Astro routing prefers static paths over dynamic ones — confirmed empirically via browser test (`/course/intro-to-n8n/book` still hits `book.astro` even with `[tab].astro` present). This means catch-all pages are safe to add alongside existing static routes; the catch-all only fires for unmatched segments. Adoption pattern: whitelist allowed tab IDs at the top of the catch-all and redirect to /404 for anything outside the whitelist.

## 3. tsc 6133 "unused variable" can fire on Astro frontmatter even when the variable IS used
**Topics:** astro, testing

**Context:** In `[tab].astro`, `const search = Astro.url.search;` followed by `return Astro.redirect(\`/course/${slug}${search}\`);` triggered ts 6133 "declared but never read" — even though the variable is consumed on the very next line inside a template literal.

**Learning:** Astro's TS analyzer doesn't always trace template-literal use through `Astro.redirect()`'s return-type signature. Workaround: inline the expression `Astro.redirect(\`/course/${slug}${Astro.url.search}\`)`. Not a real bug — just analyzer scope. If you hit this in another Astro page, inline rather than declaring intermediate variables.

## 4. Test mocks need real shape — async unhandled errors from rendered children can hide
**Topics:** testing, react

**Context:** CRT-6 click test rendered `CourseTabs` with `baseProps` (no role flags, no enrollment), defaulting to the About tab. The About tab includes `CourseAvailabilityPreview` which does its own fetch + `data.teachers.map(...)`. The generic mockFetch fallback returns `{}` — undefined `teachers` — and the component crashes asynchronously. The crash never failed any assertion (async unhandled exception, not sync), but vitest reported it as "1 unhandled error." Caused 1 of my 8 new tests to fail (the click never registered) but the crash itself was a pre-existing CourseAvailabilityPreview defensive-coding gap.

**Learning:** When testing component A that mounts component B as a side-effect of default tab routing, either (a) mock the fetch with B's expected shape, or (b) route initial state to avoid B. Used (b) — passed `initialTab="learn"` and a stub LearnTab mock to dodge the About tab entirely. The underlying CourseAvailabilityPreview crash deserves its own defensive check; tracked as [CAP-DEFEND].
