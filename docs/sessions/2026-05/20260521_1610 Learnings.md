# Conv Learnings - 2026-05-21

## 1. Inventories typed from screenshots can drift from disk reality
**Topics:** dual-repo, cc-workflow

**Context:** While curating `.scratch/matt-main`, the `.scratch/matt-figma/_INVENTORY.md` (authored at the end of Conv 169) listed `components/components-overview.png` as one of the items in `components/` — but only `components-layers.png` exists there. The inventory also under-counted top-level `happy path/` items significantly (described as "~113 items" but actual count is 100+ named files plus several subfolders, including unexpected ones like `Why we need it_ Complete the flywheel, back to start/` rendered as a folder rather than a file).

**Learning:** When acting on an inventory, walk the actual folders before trusting filename-level claims. The category-level guidance (tokens / layout / components / happy-path → buildable set) was reliable; the file-by-file listing was not. Inventories generated from screenshots or Figma exports are roadmaps, not contracts.

**Pattern:** For curation work, treat any pre-existing inventory as a *roadmap* (where to look, what categories matter) and reconcile against `ls` / `find` output before deciding what to copy. Fix typos and misplaced files at the copy step rather than carrying them forward (e.g., `typograhy-overview.png` → `typography-overview.png`, relocated from `tokens/color-guide/` to `tokens/typography/`).

## 2. Curation work needs a per-category exclusion table, not just the inclusion list
**Topics:** dual-repo, cc-workflow

**Context:** When isolating ~80 of 229 files from `.scratch/matt-figma` into `.scratch/matt-main`, listing only what was included would be insufficient — the absence of common patterns ("where are all the Prototype files?", "what about the 21 Section Title variants?", "where did the Why-we-need-it notes go?") could read as oversight rather than deliberate exclusion. The authored `_README.md` includes a 16-row exclusion table answering each "why is this missing" question explicitly, alongside the inclusion structure tree.

**Learning:** When curating from a large bag into a focused set, document the *exclusions* AND their rationale, not just the inclusions. Otherwise the next reader (a different machine, a colleague, future-you) re-asks the questions you already answered, and the curated folder loses its provenance.

**Pattern:** Pair every inclusion manifest with an exclusion table. The table prevents re-litigation and shortens the path back to source when an excluded file turns out to be needed later. Generalizable to any "isolate the useful subset" task — design exports, dependency audits, file-tree triages.
