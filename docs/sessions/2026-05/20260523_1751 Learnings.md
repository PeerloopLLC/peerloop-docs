# Conv Learnings - 2026-05-23

## 1. Figma MCP has two tool classes: selection-free vs selection-required
**Topics:** figma, mcp, design-system

**Context:** Conv 180 reference memory implied all Figma MCP tools work by node-id alone. Conv 181 [TSV] disproved this when `get_variable_defs(1:16)` failed with "select a layer first" despite passing a valid node-id.

**Learning:** The remote Figma MCP at `https://mcp.figma.com/mcp` has TWO classes of tools.
- **Selection-free** (any nodeId works): `get_metadata`, `get_libraries`, `search_design_system`, `get_screenshot`.
- **Selection-required** (user must click a specific layer in Figma desktop AND the passed nodeId must match the selected node): `get_design_context`, `get_variable_defs`.

Passing a parent or sibling nodeId fails even when there IS a selection — the nodeId must match the selected node exactly. Discovered when user had Body Default (36:256) selected but passing 36:255 (page root) errored.

**Pattern:** When the MCP returns "select a layer first", check (a) Figma desktop is the active window, (b) a specific layer is selected (not the canvas), AND (c) the passed nodeId matches the selected node. For batch-probing, ask the user to select a CONTAINER frame — `get_variable_defs` returns all Variables consumed by descendants in one call (Conv 181: Headers section → 10 Variables, Body section → 8 Variables).

## 2. Plugin-rendered visualizations are NOT authoritative Variable names
**Topics:** figma, design-system, mcp

**Context:** Conv 181 [TSV] flagged a false "naming drift" on `--Primary-Default` vs Figma's `Primary/Primary` based on the plugin-rendered Color Guide page. Direct probe via `get_variable_defs(477:8502)` revealed the actual Variable name IS `Primary/Default` — exactly matching local.

**Learning:** Matt's Color Guide page is generated by a "Color Variable Style-Guide Generator" plugin that renders labels concatenating group name + label text (e.g., group "Primary" + label "Primary" → visual "Primary/Primary"). The actual underlying Variable name uses `Primary/Default` (the variant suffix). Always verify Variable names via `get_variable_defs`, NEVER `get_metadata` text-label content. The plugin also omits Variables outside its filter scope — `--white` is a real Figma Variable but doesn't render in the Color Guide.

**Pattern:** When auditing tokens against Figma, the source of truth ordering is: (1) `get_variable_defs` on a consuming node (canonical), (2) `get_design_context` for hardcoded hex values (also canonical, surfaces non-Variable usage), (3) `get_metadata` labels (rendered display, NOT canonical naming). Never invert this ordering.

## 3. Local-file Variables are invisible to library-scoped MCP tools
**Topics:** figma, mcp, design-system

**Context:** Conv 181 [TSV] tested `get_libraries(fileKey)` and `search_design_system("color primary")` to find Matt's PeerLoop Variable collections.

**Learning:** `get_libraries` returns only subscribed external libraries (Material 3, iOS, macOS, etc.) — local Variables in the file itself don't appear because they aren't published as a library. `search_design_system` searches the same subscribed-library scope only. There is NO MCP tool that enumerates a file's full local Variable collection — Variable existence can only be verified by finding a consuming node and probing `get_variable_defs`. Variables defined in the file but unused anywhere are completely invisible to the MCP.

**Pattern:** "Does Matt have this Variable?" cannot be answered programmatically — only "Does this node consume this Variable?" can be. For comprehensive Variable audits, use visual sweep + probe each component/page in scope.

## 4. The "honest-orphan" principle for token systems
**Topics:** design-system, tokens

**Context:** Conv 181 [NOTE-YELLOW] probed for `--note-yellow` Variable. Found yellow IS used in Matt's Note (#FFF6B8 background) but as hardcoded hex, NOT a Figma Variable. Initial impulse was to add `--note-yellow` to local speculative primitives. User redirected with broader principle.

**Learning:** A hardcoded `#FFF6B8` in Note.tsx is honest if Matt later changes Note to a different yellow or removes Note entirely. A `--note-yellow` token implies a level of systematization Matt has NOT made; if Matt changes the value, the token name lies until updated. Hardcoded values are self-deprecating (breakage obvious in diff); named tokens accumulate stale meaning silently. The presence of a Figma Variable IS the signal to tokenize — not "looks like it should be a token". This collapses the entire "speculative token" category retroactively: Conv 172's `alert-light`/`carmine-red` additions would not have been added under this principle.

**Pattern:** Token-ify what Matt has tokenized; hardcode what Matt has hardcoded; scaffold what Matt hasn't categorized. Probe via `get_variable_defs` to determine; presence/absence in the response IS the decision criterion.

## 5. Variable Mode is observable at the Variable layer via MCP
**Topics:** figma, design-system, variable-mode

**Context:** Conv 180 reference memory documented Variable Mode bakes into CSS-var fallback hex (`var(--background,#e8f4df)` = Course context). Conv 181 [TSV] confirmed this at the `get_variable_defs` layer too.

**Learning:** Probing `get_variable_defs` on a node inside a Course-context container returns Course-mode values for context-bound Variables: `"Primary":"#327d00"` (dark-green, Course resolution), `"Background":"#e8f4df"` (pastel-green, Course resolution). Same Variable consumed by a node in different context resolves differently. The MCP gives you the resolved value for whatever context the selected node lives in — to probe a different mode, navigate to a node in that mode's context and re-select.

**Pattern:** For multi-mode Variable validation, identify nodes in each context (Course/Creator/Student/Default), select each, run `get_variable_defs`, compile a per-mode resolution table.

## 6. Tailwind 4 `--text-{name}--<modifier>` syntax consolidates typography
**Topics:** tailwind, typography

**Context:** Conv 181 [TSV] consolidated 18 typography Variables into a single set of utility classes that each carry size + weight + line-height + letter-spacing.

**Learning:** Tailwind 4 reads `--text-{name}--<modifier>` CSS variables and emits compound utility classes. Setting `--text-body-default: 0.875rem`, `--text-body-default--line-height: 1`, `--text-body-default--font-weight: 400`, `--text-body-default--letter-spacing: 0` emits a `text-body-default` utility that applies all four CSS properties in one class. Pre-Tailwind-4 this would have required a custom `@utility` block or `.text-body-default` rule in `@layer components`.

**Pattern:** When porting design-system text styles with multiple typographic properties, use the modifier-suffix pattern in `@theme` to avoid separate weight/lh utility classes.
