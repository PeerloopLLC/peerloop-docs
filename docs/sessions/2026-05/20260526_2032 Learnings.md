# Conv Learnings - 2026-05-26

## 1. Type-checking is blind to string-valued route references
**Topics:** astro, testing, cc-workflow

**Context:** ROUTE-FLIP moved pages to /old but left middleware prefixes, redirect targets (`/login`, `/dashboard`, `/onboarding`), and ~30 component hrefs pointing at old root paths.

**Learning:** Four distinct dead-route bugs (modal host unmounted, post-login `/dashboard`, `/login` redirect target, post-signup `/onboarding`) all passed `tsc` + `astro check` cleanly and only surfaced by running the flow in a browser. Route strings are opaque to the type system.

**Pattern:** A `goto()`-target / route-literal resolver that checks references against the actual Astro route tree is the cheap deterministic gate that would catch this class — prototype at `.scratch/e2e-route-map.mjs`.

## 2. The five-gate baseline excludes E2E by construction
**Topics:** testing, cc-workflow

**Context:** User asked why PLAN/plan-mode missed the E2E gap.

**Learning:** Gate #4 is `npm test` (vitest), not `test:all`; `/w-codecheck` + `/r-end` never invoke playwright. "Baseline green" is *defined* to exclude real-URL navigation — the only signal that catches a moved route. Plan mode inherits the blind spot by trusting baseline-green as a proxy for correctness. vitest integration tests run but are in-process `/api/*` tests, not page navigation — a route relocation is invisible to them.

## 3. A bulk file-relocation's real risk is the references that outlive the move
**Topics:** astro, cc-workflow

**Context:** ROUTE-FLIP was "a pure file-move + link-rewrite"; the link-rewrite missed the global concerns the legacy always-present AppNavbar carried (auth-modal host, logout action) plus several redirect-target literals.

**Learning:** When an always-present component (navbar) is replaced by a different shell, its *global* responsibilities silently vanish — they aren't a compile error, they're absent behavior. Re-homing them (AuthModalRenderer → AppLayout) is its own migration task, not part of the move.
