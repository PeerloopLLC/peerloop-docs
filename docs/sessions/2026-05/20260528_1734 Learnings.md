# Conv Learnings - 2026-05-28

## 1. PLAN.md narrative trail was redundant with session-file Extracts
**Topics:** docs-infra, cc-workflow

**Context:** Conv 211 questioned the value of the `*Last Updated: Conv N — [narrative]*` trail at the bottom of PLAN.md. Investigation found `/r-end` already writes 3 files per conv (Extract, Decisions, Learnings) under `docs/sessions/YYYY-MM/`. The Extract IS the canonical per-conv narrative.

**Learning:** Any docs structure that says "readable from X alone, without opening Y" should be sanity-checked against whether Y is the canonical home. PLAN.md's trail existed because someone wanted "what just happened" without opening a session file; the session-file Extract was always the canonical source, so the trail was pure redundancy. The structural fix: drop the duplicate and update the writing skill, don't try to preserve the duplicate "just in case."

## 2. Skill instructions can drift into doing more than their words say
**Topics:** cc-workflow

**Context:** `fmt-update-plan.md` step 5 said "Update Last Updated footer timestamp" — a simple instruction. In practice, the update-plan agent had drifted into writing a full narrative paragraph with chronological PREVIOUS chains. The audit of remaining drift surfaced action #4 ("Update Next Steps section") with the same pattern — the section literally doesn't exist in PLAN.md, so the agent must have been ignoring or reinterpreting it.

**Learning:** Dead-letter instructions (steps that point at structures that don't exist) are a drift vector — the agent either ignores them or reinterprets them creatively. When auditing a skill, grep for instructions that reference specific structures (sections, files, fields) and verify each structure actually exists. When removing such an instruction, add an explicit anti-pattern note ("Do NOT invent X at the bottom of Y") to prevent the next maintainer from re-adding it from intuition.

**Pattern:** When skill instructions and target-file structure disagree, the file is canonical — fix the skill, not the file.

## 3. Migrated-block awareness must propagate into agent prompts that read PLAN.md
**Topics:** cc-workflow, docs-infra

**Context:** Conv 210 + 211 migrated 8 blocks from inline-in-PLAN.md to `plan/<slug>/README.md` (or `plan/<family>/`). The update-plan agent prompt's READ + MODIFY lists predated this migration and only mentioned PLAN.md + plan/COMPLETED.md.

**Learning:** When introducing a new docs structure (per-block plan files in this case), every skill that reads or modifies the affected files needs its scope expanded. The `→ [plan/<slug>/README.md](...)` pointer in PLAN.md table rows is the discovery anchor — agents must look for it and follow it. Without this update, the canonical README would have gone stale because the agent kept editing PLAN.md table-row summaries instead of the README.

**Pattern:** Structure migration → audit every skill that touches the migrated content.
