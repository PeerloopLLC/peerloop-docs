### AAP — Astro absolute-filesystem path leak

**Claude:** Surfaced 🟠 finding from earlier probing: the dev HTML emits `<script type="module" src="/Users/jamesfraser/projects/Peerloop/node_modules/astro/components/ClientRouter.astro?astro&type=script&index=0&lang.ts">` — the user's absolute filesystem path leaked into the URL. Browser interprets it as `http://localhost:4321/Users/...` and Vite happens to serve it 200, so nothing breaks functionally. Same .astro file's style sub-module emits a correct relative path. Cross-machine portability hazard.

**User:** "we shouold fix that Astro absolute filepath issue you just surfaced, then move BR-HAVBAR-HYDRATE into the very next task after we start a new conv"

**Claude:** Investigated. Root cause located at `node_modules/astro/dist/vite-plugin-astro/compile.js:50` — emits `import "${compileProps.filename}?astro&type=script&index=${i}&lang.ts"` where `compileProps.filename` is the absolute path. Naïve relative-path fix doesn't work (Vite resolves relative imports against importer = same file = absolute). Confirmed Astro 6.3.6 (latest stable) has the identical buggy line via `npm pack` diff. Presented three options (A: defer + document / B: dev middleware HTML rewrite / C: patch-package monkey-patch).

**User:** "A. Make it clear that this is witing on an Astro fix post v6.3.6"

**Claude:** Created [AAP] task with full diagnosis (root cause file + line + version proof + recommended verification probe after each Astro upgrade). Saved BR-NAVBAR-HYDRATE next-conv-first directive for the close-out summary.
