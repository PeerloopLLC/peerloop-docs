# Conv Learnings - 2026-05-20

## 1. View Transitions persistence replays hydration errors across navigations
**Topics:** astro, frontend, debugging

**Context:** Conv 163 [DLE] reported a "wall of HMR errors + transient blank page" on a non-admin route after visiting `/admin/*`. Original Conv 161 diagnosis was AdminNavbar `<div>` vs `<a>` mismatch on `/admin/*` only. Conv 163 widened the scope to "likely a shared parent component on non-admin pages too."

**Learning:** When an Astro component carries `data-astro-transition-persist`, its DOM node — and its hydration error state — survives the navigation to the next page. The console replays the original mismatch error in the new page's context, looking like a new bug surface. Fresh hard-reload of the non-admin destination (no `/admin/*` in transit) showed zero errors, proving the scope was never widened — it was the same bug, replayed via the persistence mechanism. Conv 161's original diagnosis was correct; Conv 163's "scope widened" was a misread.

**Pattern:** When investigating cross-page hydration errors, always test (a) fresh hard reload of the suspected non-trigger page, isolated, and (b) the actual transit path. If (a) is clean and (b) reproduces, the error is travelling with a persisted component, not originating on the destination.

## 2. Codebase has an established SSR-safe pattern for current-user state (`isHydrated` flag)
**Topics:** auth, frontend, astro

**Context:** AdminNavbar.tsx:90 used `useState<CurrentUser|null>(getCurrentUser())` — a direct read of localStorage/window state in the useState initializer. AppNavbar.tsx (the sibling component, used everywhere else) does NOT do this — instead it uses `useState(null)` + an `isHydrated` flag + a useEffect that reads cache and sets `isHydrated = true`.

**Learning:** The `isHydrated` flag pattern in AppNavbar (lines 144-158) is the established codebase convention for SSR/CSR-safe rendering when the initial state would differ between server and client. AdminNavbar was the lone outlier. Repo grep `useState[<(].*getCurrentUser\(\)` returns exactly one hit, confirming this is a single isolated divergence.

**Pattern:**
```tsx
const [user, setUser] = useState<CurrentUser | null>(null);
const [isHydrated, setIsHydrated] = useState(false);

useEffect(() => {
  const cached = getCurrentUser();
  if (cached) setUser(cached);
  setIsHydrated(true);
  // ...further async refresh
}, []);

// guard role-dependent rendering:
{isHydrated && user && <UserChip user={user} />}
```

## 3. CourseTabs has full role-grouped-tabs infrastructure already wired — pages just don't use it
**Topics:** routing, frontend, role-aware

**Context:** Investigating [CRT] surfaced an expected diagnosis (no role-aware tab logic). What surprised: `CourseTabs.tsx` lines 237-318 already implement `extraTabs` with `groupLabel` and `roleColor` props, including the divider-separated rendering of role groups with small uppercase coloured group labels.

**Learning:** The chassis for role-aware course tabs exists; what's missing is the `.astro` pages computing role flags and passing `extraTabs` arrays. This reduces [CRT] from "design + build a tab grouping system" to "wire the existing system into the loader + pages". The "Course" left-side group + role-coloured right-side groups was probably anticipated when the component was originally written but never followed through.

## 4. Direct API auth is more reliable than form-button click for browser-driven role-switching tests
**Topics:** testing, browser-automation, chrome-mcp

**Context:** During the [RV] sweep, logging Guy in via clicking the form Sign In button (ref_44) did not navigate — the React island's submit handler didn't fire reliably. Direct `POST /api/auth/login` with credentials returned the user object cleanly.

**Learning:** For multi-role browser verification work, prefer driving the auth API directly (`fetch('/api/auth/login', ...)`) over clicking the login form button. Saves the flakiness of React-island form-submit timing and the need to re-find form refs after page changes. Pattern is also faster (no form-fill round-trip).

**Pattern:**
```js
fetch('/api/auth/logout', {method: 'POST', credentials: 'include'})
  .then(() => fetch('/api/auth/login', {
    method: 'POST',
    credentials: 'include',
    headers: {'Content-Type': 'application/json'},
    body: JSON.stringify({email: 'user@example.com', password: 'Password1'}),
  }));
```

## 5. ResetSpa-style "View Recording" buttons need a re-trigger when the panel state isn't yet bound
**Topics:** testing, react, browser-automation

**Context:** On Surface 7 verification, clicking the "View Recording" footer button (ref_261) once via `computer.left_click` didn't reveal the side panel's recording section, even after a 2-second wait. The API endpoint returned the cached URL correctly when queried separately. A second click via JS (`btn.click()`) made it appear.

**Learning:** When a React component lazy-fetches on a button click and immediately tries to render conditional content from the response, the first click may register before the useEffect hooks that bind setState handlers complete. In tests / browser automation, a single-click then read pattern can race; a re-trigger or short wait + observe loop is more reliable. Not a bug worth fixing in product code — pattern is just to be aware of in automation.
