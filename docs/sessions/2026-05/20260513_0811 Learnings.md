# Conv Learnings - 2026-05-13

## 1. Account-wide vs per-session getRecordings as diagnostic split
**Topics:** bbb, video, debugging

**Context:** Investigating why teacher-recorded session produced no recording artifact.

**Learning:** Calling BBB's `getRecordings` with **no `meetingID` parameter** returns all recordings on the server. This is diagnostically stronger than per-session checks because a single SUCCESS+0-result eliminates a whole class of hypotheses at once: webhook never fired, `BBB_SECRET` mismatch, `BBB_URL` misconfiguration, `bbb_meeting_id` mismatch — none of these could produce account-wide zero. Per-session "no recordings found" is consistent with several causes; account-wide zero narrows to server-side recording production.

**Pattern:** When debugging external-service integrations where multiple layers of indirection exist (auth, mapping, callbacks, processing), favor the broadest scope of query the API allows — it eliminates indirection-layer bugs as candidates in one shot.

## 2. The `record` vs `autoStartRecording` BBB footgun
**Topics:** bbb, video

**Context:** Code audit during recording-gap investigation revealed `record=true` always sent but `autoStartRecording` never sent.

**Learning:** BBB's `record=true` enables recording **capability** (the moderator's Record button appears) but does NOT auto-start recording. Without `autoStartRecording=true`, recording only begins if the moderator clicks the button during the session. The two-parameter design comes from BBB's lecture-hall heritage (instructor decides per-session). For 1-on-1 tutoring where every session should be recorded by policy, both flags must be set — relying on moderator click leaks recordings whenever someone forgets.

**Pattern:** When a vendor API has two flags that *together* gate a feature, audit defaults early. The "obvious" single flag isn't always sufficient.

## 3. Vendor escalation: do the obvious self-checks before asking
**Topics:** support, debugging, vendor-management

**Context:** Drafting reply to Fred at Blindside.

**Learning:** A reply that demonstrates "we've already done these self-checks, this is what only you can answer" gets faster and sharper vendor responses than a reply that's only questions. The Fred reply includes: confirmed URL we're hitting, account-wide getRecordings result, autoStartRecording fix already deployed, then three targeted questions naming specific BBB config knobs (`disableRecordingDefault`, `rap-process`, `rap-publish`).

**Pattern:** Vendor escalation message structure — (1) what you've already verified, (2) what only they can answer, (3) name the specific config knobs in your asks so they can answer concretely.

## 4. Diagnostic UIs need an as-of timestamp
**Topics:** ui-design, admin, debugging

**Context:** Designed RecordingsAdmin to show count + table + fetched_at card.

**Learning:** Surfacing the fetch wall-clock timestamp on a diagnostic page is more valuable than for normal app UIs. For external-service dashboards (Stripe balance, Resend send queue, BBB recordings), an admin needs to know **exactly when** the displayed state reflects the vendor's reality, not "now". Frozen-timestamp design + manual Refresh button > auto-refreshing display.

**Pattern:** External-service dashboards in admin: (1) stamp fetch time in a visible card, (2) require manual Refresh, (3) don't poll. The deliberate friction signals "this is a point-in-time snapshot".

## 5. Two-tier documentation for project conventions
**Topics:** docs-infra, cc-workflow

**Context:** Establishing `.scratch/` convention.

**Learning:** Putting a short pointer in CLAUDE.md plus a detailed reference in the artifact itself (here `.scratch/README.md`) mirrors how the project documents technology stacks. Short pointer is always-loaded; detailed reference is on-demand. Same logic applies to any project convention worth documenting durably without bloating CLAUDE.md.

**Pattern:** Two-tier: CLAUDE.md gets the index entry (always loaded), the artifact gets the detailed reference (on-demand).

## 6. /r-start re-invocation residue
**Topics:** cc-workflow

**Context:** User re-typed `/r-start` after it had already run in same session; pre-computed context showed `D RESUME-STATE.md` dirty state.

**Learning:** `/r-start` Step 7 deletes RESUME-STATE.md but doesn't auto-commit. The deletion sits in the working tree until next `/r-commit`/`/r-end` sweeps it. Re-invoking `/r-start` will hit the dirty-repo HALT guard because the guard can't distinguish self-inflicted Step 7 residue from real uncommitted work. Correct response: surface the residue's origin, present skip/commit/restart options, default-recommend skip.

**Pattern:** When a skill's own actions leave the repo dirty, future invocations of itself should surface "this is my residue" rather than treating it as anomalous.
