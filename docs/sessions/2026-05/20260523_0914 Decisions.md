# Conv Decisions - 2026-05-23

## 1. Pair wrangler upgrade with the Astro stack upgrade
**Type:** Implementation
**Topics:** packages, dependencies

**Trigger:** Initial `npm install astro@6.3.7 @astrojs/cloudflare@13.5.4 @astrojs/react@5.0.5` failed with ERESOLVE — `@astrojs/cloudflare@13.5.4` requires `wrangler@^4.83.0`, project had `wrangler@4.81.1`.

**Options Considered:**
1. `--legacy-peer-deps` flag — quick, bypasses the check
2. Add `wrangler@4.94.0` to the upgrade set ← Chosen
3. Defer the entire Astro upgrade until wrangler can be handled separately

**Decision:** Add wrangler@4.94.0 to the upgrade set as a coupled package change.

**Rationale:** `--legacy-peer-deps` papers over real version skew. Wrangler is the Cloudflare CLI; bumping it is a single-version-family change (4.x→4.x). The peer-dep declaration is correct upstream — the cleanest fix is to satisfy it. Per CLAUDE.md §Solution Quality: default to durable.

**Consequences:** 4-package upgrade instead of 3. No `--force` or `--legacy-peer-deps` in the lockfile. Wrangler 4.94.0 still satisfies `^4.67.0` semver in package.json, so the declared range didn't need changing.

## 2. Retire the stateless-primitives discipline
**Type:** Strategic
**Topics:** astro, react, design-system

**Trigger:** Conv 176 added "matt/* primitives must be stateless / no useState" as a hard rule, framed as an SSR-safety workaround. Conv 177's diagnosis showed the underlying bug was a Vite cold-start race, fixable with a 2-line config change. The discipline was responding to a misdiagnosed problem.

**Options Considered:**
1. Keep the discipline as defense-in-depth
2. Remove it entirely ← Chosen
3. Replace with a softer design preference (controlled where state is shared, stateful where local)

**Decision:** Remove the discipline from DEVELOPMENT-GUIDE.md. Replace the section with a new "Vite SSR Cold-Start Dep Discovery" section that documents the real bug class, the fix recipe, and the symptom signature to recognize if the cold-start crash ever returns.

**Rationale:** The discipline was over-cautious. Sidebar.tsx with useState works fine in dev (after cold-start) and in production. The new section is more useful than the discipline because it tells the next contributor what to do if the bug ever returns (add new Astro virtual modules to `optimizeDeps.include`).

**Consequences:** matt/* primitives can use hooks freely. ToDoItem.tsx was rewritten as a controlled-or-uncontrolled hybrid (proper React pattern) instead of the fully-controlled form Conv 176 mandated. The DEV-STAGING-SSR row in PLAN.md ON-HOLD was marked RESOLVED.

## 3. ToDoItem uses controlled-or-uncontrolled hybrid pattern
**Type:** Implementation
**Topics:** react, design-system, api-design

**Trigger:** User asked to redo ToDoItem.tsx to use useState. Three valid API shapes: fully controlled (Conv 176's existing), uncontrolled-only (`defaultChecked` + internal state), hybrid (both).

**Options Considered:**
1. Fully controlled (status quo from Conv 176) — parent owns state always
2. Uncontrolled only — `defaultChecked` initial value, internal state takes over
3. Hybrid — `checked` for controlled mode, `defaultChecked` for uncontrolled; component picks based on whether `checked` is undefined ← Chosen

**Decision:** Hybrid pattern matching the React standard idiom.

**Rationale:** Existing showcase callsites pass `checked={false}` and `checked={true}` as display-only static states — hybrid mode keeps these working (controlled, no onChange = no toggle). Future production callers can use either mode: controlled when state is parent-owned (database-backed enrolled lists), uncontrolled when the primitive is self-contained (one-off demos, prerequisites). Standard React pattern that any contributor will recognize.

**Consequences:** ToDoItem.tsx gained `defaultChecked?: boolean` prop. `checked` became optional. Component body uses `const isControlled = controlledChecked !== undefined` then derives `isChecked` from the appropriate source. Existing showcase usage works unchanged.

## 4. Don't downgrade React to 18.2 despite "stable" recommendation in some sources
**Type:** Strategic
**Topics:** packages, react

**Trigger:** The technet-experts article suggested downgrading React 19 → 18.2 + @astrojs/react 5 → 4 + Astro 5 as a "stable" fix for the invalid-hook-call class of bugs.

**Options Considered:**
1. Downgrade React + Astro stack to "stable" 18.2 / 5.x
2. Keep React 19 and find the actual fix ← Chosen

**Decision:** Stay on React 19. The technet article's recommendation is generic SEO content that misdiagnoses Vite cold-start cases as React-version incompatibilities.

**Rationale:** The Vite cold-start pattern affects React 18 projects too (Storybook #32049, Remix #10156 on Vite 5/6 — many of those reporters are on React 18). The crash isn't a React-version problem. Downgrading would lose React 19 features (Suspense improvements, useActionState, etc.) without fixing the actual bug.

**Consequences:** React stays at 19.2.3 (unchanged this conv). The actual fix landed in astro.config.mjs Vite config.
