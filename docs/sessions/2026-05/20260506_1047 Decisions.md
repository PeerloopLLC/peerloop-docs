# Conv Decisions - 2026-05-06

## 1. setCurrentUser dedup guard uses (id, dataVersion) tuple
**Type:** Implementation
**Topics:** react, current-user, performance

**Trigger:** [UCM] verified that `useCurrentUser()` returns a fresh ref on every refresh, defeating downstream `useMemo([currentUser])` work. AppNavbar's focus-event handler triggers `refreshCurrentUser()` on every tab-back.

**Options Considered:**
1. **Memoize at the hook level** — `useCurrentUser` does the equality check via a setState callback. Cons: every consumer of `subscribeToUserChange` would still receive the spurious fire — only the React hook would dedup.
2. **Add a client-side `if (deepEqual(prev, next))` guard in `refreshCurrentUser`** — wider check but expensive (deep-equal a complex object every refresh) and ad-hoc.
3. **Dedup at `setCurrentUser` using existing `dataVersion`** ← Chosen. The dataVersion field already exists for version polling — when it doesn't increment, the server is asserting the data is unchanged. Equality test is `prev.id === next.id && prev.dataVersion === next.dataVersion`, O(1).

**Decision:** Add a 6-line dedup guard at the top of `setCurrentUser` that drops the new instance and skips notification when both id and dataVersion are unchanged.

**Rationale:** Reuses an existing invariant (server is the source of truth for dataVersion) rather than inventing a new one. Fixes the issue at the singleton-update site so all consumers benefit (not just the React hook). Cheap. Backed by a regression test.

**Consequences:** All `subscribeToUserChange` consumers now only fire on real data changes. Backwards compat: any test fixtures or call sites that simulated a refresh-with-no-change must now bump `dataVersion` to model reality (3 test fixtures updated this conv). Real production behavior is unchanged because the server already bumps `dataVersion` on data change.

> **Insight:** Memoization established at the consumer is undermined when the producer churns. When investigating why `useMemo`/`useCallback` extractions don't deliver expected wins, audit the source of each dep value — singleton/global stores often leak ref-instability that local memoization cannot recover.

## 2. /r-start ends with bold Yes/No prompt to mark waiting state
**Type:** Implementation
**Topics:** cc-workflow, skills

**Trigger:** User wanted clearer signal at the end of /r-start that Claude is waiting for an answer; previously /r-start ended with "Recommended Action" prose followed by "Quick Context", with no explicit prompt.

**Options Considered:**
1. Add a separator/footer line after Quick Context.
2. **Reorder so Recommended Action is last, end with bold Yes/No prompt** ← Chosen. Combines best practice from `feedback_bold_questions.md` (bold + own line + clear question) with section ordering.
3. Replace recommended-action with a TaskList-style numbered options menu.

**Decision:** Reordered Step 8 Present Context template (Quick Context → Recommended Action → Yes/No prompt). Added new resume rule: "Recommended Action MUST be the last section — ends with `**Start [TASK-CODE] now? (yes / no)**` and HALT after asking."

**Rationale:** User-driven directive; one-liner addition with clear intent.

**Consequences:** Future `/r-start` runs always close on an actionable gate.
