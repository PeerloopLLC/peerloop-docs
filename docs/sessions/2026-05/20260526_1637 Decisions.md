# Conv Decisions - 2026-05-26

## 1. PROV-SWEEP scope — thin validator + candidate data, not the full §6 sweep
**Type:** Implementation
**Topics:** astro, cc-workflow

**Trigger:** User asked whether reusable tooling (option A) was justified.

**Options Considered:**
1. Full reusable §6 sweep (local + Figma automation)
2. Thin validator + committed candidate-data file; Figma matching stays a documented agent step ← Chosen
3. One-time manual sweep, no artifact
4. Committed candidate manifest doc only

**Decision:** Build only the part that is both recurring AND deterministic today: `scripts/prov-sweep.ts` (validator) + `scripts/prov-candidates.ts` (data). Figma matching deferred to when harvest infra lands.

**Rationale:** Recurrence justifies reusability, but the domain-delimitation gap means candidates can't fully auto-derive, the Figma half isn't scriptable (MCP agent step), and its integration target (#16/#17) doesn't exist — a full script risks rework against an absent interface.

**Consequences:** New `scripts/prov-sweep.ts` + `scripts/prov-candidates.ts` + `prov:sweep` npm script; `matt-provenance.md §6a`; `[PROV-MATCH]` split out as #25.

## 2. Both new files in scripts/ as TypeScript run via tsx
**Type:** Implementation
**Topics:** astro

**Trigger:** Where to place the sweep tool + its candidate registry.

**Options Considered:**
1. App `src/` (e.g. a provenance module)
2. `scripts/` TypeScript, run via tsx ← Chosen

**Decision:** `scripts/prov-sweep.ts` + `scripts/prov-candidates.ts`, run via tsx.

**Rationale:** Matches existing `tsx scripts/*.ts` convention; lets the validator type-safely `import { ICON_PROVENANCE }` (no regex-parsing TS); keeps sweep-only data out of app runtime (`ICON_PROVENANCE` has no app importers); follows the established "provenance data as TS const module" pattern.

**Consequences:** Files outside tsconfig `include` — don't touch the type-check gate.
