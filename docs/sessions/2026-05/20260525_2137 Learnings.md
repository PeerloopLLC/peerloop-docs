# Conv Learnings - 2026-05-25

## 1. Running the app catches bugs static reading can't
**Topics:** cc-workflow, astro, testing

**Context:** Two real bugs (#27 admin blank, #28 panel bleed) surfaced only by driving the live app in a browser at multiple widths/roles; neither was visible from code-reading. #28 in particular was a hidden CSS coupling.

**Learning:** For responsive/role-gated UI, browser verification at multiple widths + auth states is not optional polish — it's the only way to catch occlusion-dependent and redirect-path bugs.

---

## 2. Hidden-by-occlusion is a latent responsive bug
**Topics:** astro, cc-workflow

**Context:** DiscoverSlidePanel + UserAccountDropdown "closed" state (`-translate-x-full` from `left:220` = rest at -68→220) was never truly off-screen — only covered by the opaque 220px AppNavbar rail. Below `lg` the rail slides off-canvas, exposing the panels.

**Learning:** A "hidden" element whose concealment depends on another element stacked over it is a latent bug — the dependency breaks the moment the cover moves (responsive, z-index change, cover removal). Hide states should be self-sufficiently off-viewport.

**Pattern:**
Closed transform `translateX(calc(-100% - {anchorPx}px))` clears both own width AND the left anchor, viewport-independent; apply via inline style so it survives Astro View-Transition CSS swaps (arbitrary translate classes get dropped).

## 3. Falsify your own hypothesis before asserting a cause
**Topics:** cc-workflow

**Context:** For #27 (admin blank) I confidently attributed it to stale Vite optimize-deps; a clean dev restart falsified it. I then declined to guess a second mechanism.

**Learning:** A plausible mechanism is a hypothesis until tested. One cheap falsifying test (clean restart, unauth curl) beats a confident wrong explanation — and stopping at "symptom confirmed, mechanism unknown" is honest and correct when the test doesn't isolate the cause.

---

## 4. The user's "wrong" hypothesis can be the productive lead
**Topics:** cc-workflow

**Context:** User insisted the AppNavbar leaks onto `/matt/`. It doesn't (instrumented, 0 leak). But chasing that claim forced proving the rail/panels are removed on swap, which isolated the true culprit (panels bleeding on the legacy page, not `/matt/`).

**Learning:** When a user reports a specific symptom that you can't reproduce on the obvious path, treat the claim as a lead to triangulate from, not a thing to dismiss. The disproof of their framing often locates the real bug.
