# Conv Learnings - 2026-05-06

## 1. Content layer vs. transport layer in cross-machine sync
**Topics:** cc-workflow, dual-repo, cross-machine

**Context:** [CMS] manual sync via Desktop folder rsync. After sync, MEMORY.md contained `/Users/jamesfraser/...` paths that were correct on M4Pro (where the snapshot originated) but wrong on M4 (this machine). User initially read this as "did you update to jamesfraser?" — a sync action — when in fact the M4Pro snapshot brought a *new* section that hadn't existed on M4 before.

**Learning:** Memory file portability has two independent layers. **Transport** is which machine has which bytes. **Content** is whether the bytes are machine-specific. ALL sync mechanisms (manual / iCloud / git symlink / separate repo) propagate bytes verbatim — none rewrite usernames. So content portability must be solved separately. Manual sync without content portability *flips which machine is wrong* on every transfer rather than converging — the inconsistency oscillates.

**Pattern:** When designing any sync solution: (a) test that the bytes-to-be-synced produce correct behavior on the *destination*, not just the source; (b) prefer placeholders (`~`, `$CLAUDE_PROJECT_DIR`, `<user>`) over hardcoded paths so the file is content-portable independent of transport.

## 2. Four-layer verification ladder for cross-machine artifact sync
**Topics:** cc-workflow, testing, cross-machine

**Context:** User asked how to confirm M4 has "full access" to all 51 synced memory files. A single `ls` count would have been insufficient — files could exist but be wrong-sized, orphaned from MEMORY.md, missing frontmatter, or unloadable.

**Learning:** Each layer fails differently:
- **L1 Filesystem** — file exists, non-zero, readable (catches catastrophic copy failures)
- **L2 Index consistency** — MEMORY.md ↔ filesystem map is bilateral (every link resolves, every file is indexed) (catches drift between index and disk — exactly what `feedback_memory_index_load_bearing.md` is designed to prevent)
- **L2-byte** — sha-equal to source snapshot (catches right-size-wrong-content failures: partial writes, botched edits)
- **L3 Frontmatter** — files conform to expected structure (catches schema regressions)
- **L4 Functional** — Read a brand-new file and use its content (catches misconfigured load pipelines that pass byte-checks but fail in practice)

**Pattern:** Apply this ladder to any cross-machine artifact sync (memory files, secret rotation, build-artifact transfer). The same checks were embedded into the [MPS] task description so M4Pro must demonstrate convergence rather than assert it.

## 3. Self-falsifying task descriptions for manual procedures
**Topics:** cc-workflow, testing, work-tracking

**Context:** Wrote [MPS] task description for M4Pro to execute when next on that machine. Risk: "I think I did it" failure mode (the canonical Conv 149→150 [OPW] incident captured in `feedback_watch_task_assumptions.md`).

**Learning:** Task descriptions for manual procedures should embed their own falsifiability:
1. **Precondition check** at the top (idempotent — second invocation is a no-op rather than a re-application that double-edits)
2. **Stated invariants** describing what convergence looks like
3. **Verification commands** that produce a binary pass/fail signal

The task can't be marked completed on vibes — it gets marked completed because the verifier ran clean.

**Pattern:** Same shape as `feedback_heuristic_calibration.md` ("test the heuristic against its motivating case before commit") applied to manual cross-machine procedures.

## 4. Memory-file frontmatter convention for non-prefixed files
**Topics:** cc-workflow, memory-system

**Context:** L3 verification flagged `e2e-testing-patterns.md` as having no frontmatter — first line was `# E2E Testing Patterns (Peerloop)`. Needed to determine convention before adding frontmatter.

**Learning:** Non-prefixed memory files (those without `feedback_*` or `project_*` prefix — currently `plato-context.md`, `rename-lessons.md`, `e2e-testing-patterns.md`) follow a consistent shape: `type: feedback`, no H1 retained after frontmatter (content starts directly with `## H2` or prose). The `feedback_*` prefix is *omitted from the filename* for these because the topic is broad enough that the prefix would feel redundant (the file *is* its topic, not a sub-rule of one).

**Pattern:** When adding frontmatter to a previously bare memory file, match the established convention by examining 1-2 sibling files of the same naming style.
