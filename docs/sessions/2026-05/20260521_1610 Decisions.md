# Conv Decisions - 2026-05-21

## 1. Curation scope for matt-main (what to include from matt-figma)
**Type:** Implementation
**Topics:** dual-repo, cc-workflow

**Trigger:** User asked to isolate "the ones we will need to use to create the happy path pages using the style guide pages Matt created" — required a judgement call on which of 229 files cross the necessity threshold for the MacMiniM4 build session.

**Options Considered:**
1. Conservative: copy everything possibly relevant — tokens + layout + components + ALL happy-path subfolders including Prototype copies + Section Title-N variants
2. Aggressive: copy ONLY the 31 canonical `Content/Happy/` screens + minimum tokens (Generated palette + Body + Headers) — under 10 MB
3. Middle (chosen): Style-guide foundations (color + typography) + layout primitives + base components + the 31 canonical screens + 10 Purpose milestone frames + page-overview PNGs — 83 files, 85 MB

**Decision:** Option 3 — the middle path.

**Rationale:** The 31 canonical screens alone don't give MacMiniM4 a way to start building — you need the tokens (colors, fonts), layout primitives (page-structure SVGs across breakpoints), and a base component library. But including all Prototype copies + 21 Section Title variants + 10 Why-we-need-it justification frames is noise. The Purpose milestones group the screens into flow stages, which is structural context worth keeping. Page-overview PNGs let a designer scan the source canvases in one glance.

**Consequences:** `matt-main` is 38% of `matt-figma`'s file count and 62% of its size. If any excluded file is needed later, it remains in `matt-figma` one directory up; the exclusion table in `_README.md` documents where to find it and why it was left out.

## 2. Use Dropbox to transfer matt-main to MacMiniM4
**Type:** Implementation
**Topics:** dual-repo, cc-workflow, obsidian

**Trigger:** `.scratch/` is gitignored, so git-based transport (the dual-repo's normal cross-machine sync) does not move the curated folder. Asked the user to pick between rsync over LAN, tar+AirDrop, vault-copy, or other.

**Options Considered:**
1. rsync over LAN (requires SSH setup, both machines online simultaneously)
2. tar + AirDrop (one-shot, no setup)
3. Copy into the Obsidian vault (auto-syncs but pollutes the vault with non-knowledge content)
4. Dropbox (chosen by user)

**Decision:** Dropbox.

**Rationale:** User's call. Dropbox is already in place, handles binary files cleanly, propagates async (no requirement for both machines online), and keeps the design assets outside the Obsidian vault. The 85 MB curated folder is small enough that Dropbox transfer is fast.

**Consequences:** No action on the CC side — user moves it. The pattern (curate locally → user moves via existing cloud-sync) is reusable for future ad-hoc cross-machine transfers of gitignored content.
