# Conv Decisions - 2026-05-06

## 1. Always pause on non-empty mirror→live diff in /r-start Step 5.7
**Type:** Strategic
**Topics:** memory-sync, cc-workflow, skill-authoring

**Trigger:** User asked the skill to "always interrupt its flow and ask if it should continue and replace the live memories with the mirror as well as show that one line output of the net difference. That way we can deal with any diff that is unexpected."

**Options Considered:**
1. Keep Conv 155 design — halt only on `Only in $LIVE` (data-loss); display diff inline ([SDD] only)
2. Extend halt to all non-empty diffs ← Chosen
3. Halt only on diffs above some change-threshold (e.g., >= N files affected)

**Decision:** /r-start Step 5.7 always pauses on any non-empty `diff -rq` mirror vs live. Empty-diff is the only silent path. Two question shapes: yes/no for normal incoming changes; A/B/C + auto-backup for data-loss escalation.

**Rationale:** Visibility into every cross-machine state change is worth the per-conv friction (one yes/no when there are incoming changes). User is sole pilot per `user_hands_off_pilot_workflow.md` — they want full control of memory state, not just guard rails on the worst case. Threshold-based halts (option 3) introduce a tunable gate with no clean default; binary always-pause is simpler and more predictable.

**Consequences:** Every cross-machine sync now has a user checkpoint before destructive `rsync --delete` runs. The data-loss halt becomes an escalation sub-case (with auto-backup + A/B/C) of the broader always-pause rule. Memory entry `feedback_msi_first_sync_data_loss_window.md` rewritten to reflect this — its filename now lags the broadened content (small follow-up rename queued).

## 2. Two-phase bash split for Step 5.7
**Type:** Implementation
**Topics:** skill-authoring, memory-sync

**Trigger:** Implementing decision #1 needed a way to halt the skill mid-execution and ask the user, then conditionally run rsync.

**Options Considered:**
1. Single bash with all branches inline + interactive `read` prompt
2. Two-phase bash split — Phase 1 forensics+halt, Phase 2 rsync+cap-check ← Chosen
3. Three-phase split — Phase 1 forensics, Phase 2 ask, Phase 3 rsync

**Decision:** Two phases. Phase 1 always runs (forensics + display + log + decide whether to halt). Phase 2 runs immediately on empty diff (no question), or after user approval on non-empty diff.

**Rationale:** Bash can't prompt interactively from inside a skill — Claude is the executor, the user types responses to Claude. So the skill must split: bash → Claude prompt → bash. Three phases (option 3) doesn't add value because the "ask" step is just Claude prose, not a separate bash. Option 1 (`read` prompt) doesn't work because Claude doesn't execute interactive bash sessions. The two-phase split is the natural fit.

**Consequences:** Step 5.7 now has two `\`\`\`bash\`\`\`` blocks separated by branching prose. The MEMORY.md cap check moved to Phase 2 so it always runs on post-sync state (previously was inline with rsync, sometimes pre-sync if data-loss halt fired).

## 3. Empty-diff path is silent (no prompt on no-op)
**Type:** Implementation
**Topics:** memory-sync, ux

**Trigger:** Designing decision #1 — "always pause" had one ambiguity: what about empty diffs?

**Options Considered:**
1. Empty diff = silent, no prompt ← Chosen
2. Empty diff = always ask, even when nothing to apply

**Decision:** Empty diff is silent. Phase 2 runs immediately (no-op rsync + cap check).

**Rationale:** A prompt that has no decision behind it is friction without value. The phrase "always interrupt" was understood as "always interrupt when there's something to decide about." Mirror == live is byte-equal — there is no incoming change for the user to evaluate.

**Consequences:** Most /r-starts after a clean previous /r-end on the same machine will run silently through Step 5.7 (since live and mirror match). Only cross-machine syncs and out-of-band edits (theoretically excluded by `user_hands_off_pilot_workflow.md`) trigger the halt.

## 4. "no" path skips Phase 2 with warning, does not halt /r-start
**Type:** Implementation
**Topics:** memory-sync, ux

**Trigger:** Designing the yes/no question's "no" branch. What does "no" mean — skip-and-continue, or halt?

**Options Considered:**
1. "no" = skip Phase 2 and continue /r-start, with explicit warning that next /r-end will overwrite mirror with this machine's pre-sync state
2. "no" = HALT /r-start entirely (conv counter already pushed; user must manually resolve and re-run)

**Decision:** Option 1 — skip-with-warning, continue.

**Rationale:** The conv counter is already pushed by the time Step 5.7 runs. Halting /r-start would leave the counter incremented but no work tracked, which is awkward. Skip-with-warning lets the user inspect the diff in the docs repo (mirror is git-committed, so the diff is fully introspectable post-conv) and choose how to resolve before the next /r-end. The warning makes the consequence explicit.

**Consequences:** The "no" path is a soft escape valve, not a hard halt. User responsibility: don't run /r-end before resolving, or accept that mirror will reflect this machine's state. The trade-off favors low-friction continuation over strict consistency enforcement.
