# Conv Learnings - 2026-05-21

## 1. Validate detection heuristics against the actual motivating case from git, not the session-doc summary
**Topics:** d1, code-review, testing

**Context:** [CCK-DA] redesign required calibrating the v2 heuristic against the Conv 117 motivating case (per `memory/feedback_heuristic_calibration.md`). The session-doc Learnings file said the buggy code was `communities.deleted_at IS NULL` (qualified). Looking at the actual commit `7df6c02` showed the pre-fix SQL was `SELECT ... FROM communities WHERE slug = ? AND deleted_at IS NULL` — **unqualified** `deleted_at` against a single-table FROM where the table lacks the column. The session doc had loosely paraphrased it.

**Learning:** When designing a heuristic against a memo's "canonical case," check the **actual code at the time** (`git show <commit>`) before encoding the case's shape into your algorithm. Memos paraphrase; git is source of truth. A v2 algorithm that only handles qualified `<table>.deleted_at` (per the paraphrase) would have silently let the Conv 117 case through despite "passing" the calibration test on a synthetic-but-wrong fixture. The v2 design specifically handles both qualified and unqualified forms because the *actual* commit forced the design to.

**Pattern:** Calibration → fixture → grep the named commit → confirm fixture matches reality → only then encode the test case.

## 2. Naming-convention sweeps benefit from "code-truth" anchoring even when PLAN.md text disagrees
**Topics:** cc-workflow, dual-repo

**Context:** [MND] surfaced a hidden contradiction: PLAN.md said the canonical M4Pro name is `MacMiniM4Pro` (no hyphen), but the codebase canonical was `MacMiniM4-Pro` (hyphenated) — locked in via the TypeScript `MachineName` type in `tests/helpers/machine.ts`, propagated through `vitest.global-setup.ts`, `tests/README.md`, and 8 docs. Recent Conv 163–167 commits used the no-hyphen form as a workaround (since the hostname didn't match and the user manually typed the commit). PLAN.md text inherited that workaround as "canonical."

**Learning:** When PLAN.md text and code disagree on a name, treat it as a user-decision question (per CLAUDE.md §Critical Rule) rather than picking either silently. Code is more authoritative than PLAN.md (PLAN is editorial; code is enforceable), but the user may want to migrate code TO the PLAN form rather than back. Surfacing the conflict + offering both directions (with the scope of each) lets the user choose intentionally. Conv 168 chose to migrate code to PLAN's form, requiring 11-file sweep — but the choice was deliberate.

**Pattern:** Conflict detection during naming work → present both directions with explicit scope (`MacMiniM4-Pro: 11 files unchanged` vs `MacMiniM4Pro: 11 files touched + tsc verification`) → user picks.

## 3. Per-route opt-out annotation outperforms scanner-wide whitelist for "expected unreachable" routes
**Topics:** plato, code-review

**Context:** [RAM-NO-NAV] needed a way to suppress `⚠️ no discovered nav path` warnings for 20 legitimately no-nav routes (footers, 404, role-gated admin, role-conditional tabs). Options: (A) footer-scan in route-api-map.mjs (durable but partial — doesn't handle role-tabs or 404); (B) scanner-wide whitelist (low scope, but maintenance pain — every new no-nav route is a scanner edit); (C) per-route annotation in .astro frontmatter (`export const noNav = true`); (D) leave alone.

**Learning:** Per-route annotation (option C) wins on **declarative locality**: the file that knows it's no-nav declares it; the scanner reads the declaration; no central list to maintain. New no-nav routes self-document at creation time. Trade-off: requires touching each .astro file. Scanner implementation is ~10 lines (`parseNoNav` helper + field propagation + emit branching). Output distinguishes intentional (`ℹ️`) from accidental (`⚠️`) — preserves the warning's signal value.

**Pattern:** When suppressing a class of warnings, prefer per-target declarations over central exclusion lists — the declaration encodes intent at the point where intent is known.

## 4. HOME-simulation harness with structural-glob assertions catches dual-username bugs at script-author time
**Topics:** cc-workflow, dual-repo

**Context:** [XMV] was a recurring meta-task (4 convs in RESUME-STATE) framed as "front-load cross-machine verification before locking sweep rules." The Conv 162 tilde-everywhere sweep had landed without per-machine verification, and Conv 152/153 incidents showed how single-machine sweeps can silently break the other machine. User chose to build a harness over codifying discipline.

**Learning:** A test harness with named cases beats a prose rule for cross-machine concerns. The harness runs each canonical pattern via `HOME=/Users/livingroom bash -c "$expr"` and `HOME=/Users/jamesfraser bash -c "$expr"` then asserts against a structural glob (e.g., `/Users/*/projects/peerloop-docs`) — the glob accepts both HOMEs while rejecting structurally-different output. 9 baseline cases cover tilde expansion (quoted/unquoted), `$HOME` equivalence, slug derivation, memory-dir composition. A failing case = the pattern doesn't work the same under both HOMEs = a real bug to fix BEFORE commit. Bonus `--scan <file>` mode lists tilde/$HOME references in any target file for ad-hoc pre-commit review.

**Pattern:** When a discipline targets a binary precondition ("does X work on both machines?"), encode the check as a runnable harness with named test cases. Self-falsification beats self-discipline.
