# Conv Decisions - 2026-05-29

## 1. /profile is the permanent private account hub; public profile is a separate /@handle
**Type:** Strategic
**Topics:** matt, astro, cc-workflow

**Trigger:** User recollection that `/profile` redirects to `@me`; review found it doesn't.

**Options Considered:**
1. Honor the old plan: rename the hub, redirect `/profile`→`/@me`
2. Drop that plan: `/profile` stays the private hub, public profile = separate `/@handle` (+`/@me` alias) ← Chosen

**Decision:** `/profile` remains the private account/settings hub permanently; the public profile is the separate `/@handle` route (with `/@me` self-alias) migrated under RTMIG-4. The "redirect /profile→/@me" plan note is dropped.

**Rationale:** The pre-flip UserAccountDropdown linked "View Profile → /@handle" and "Settings → /settings" as distinct items — they were always separate. Redirecting the private hub to the public profile would destroy the hub.

**Consequences:** RTMIG-4 plan note corrected; the Account-tab "View public profile" link lights up when /@handle migrates to root.

## 2. No overlay/popover for the account surfaces; Visitor + Profile list links, not embedded forms
**Type:** Strategic
**Topics:** matt, cc-workflow

**Trigger:** "Popover" = the slide-out submenu the client rejected (the reason Conv 212 ported options to SubNav tabs).

**Options Considered:**
1. Unified popover for both auth states
2. Two components (legacy two-surface model)
3. Overlay-free: page-like surfaces holding links + a shared toggle ← Chosen

**Decision:** No overlays. `/login`+`/signup` stay standalone form pages (timeout/redirect targets); the Visitor (`/visitor`) and Profile (`/profile`) surfaces list Login/Sign-up/Sign-out as links/actions plus a shared `<ThemeToggle>` — not embedded forms.

**Rationale:** Honors the client's rejection of the slide-out submenu and matches the Conv-212 SubNav-tabs call; keeps the standalone auth pages reusable by timeout flows.

**Consequences:** New `/visitor` page; shared ThemeToggle on both surfaces; dark mode now reachable logged-out.

## 3. Two dedicated vetted-primitive registries + a 3-value data-prov runtime stamp (approach B)
**Type:** Strategic
**Topics:** matt, astro, cc-workflow, testing

**Trigger:** User wants a clear taxonomy (Matt-designed / legacy / Matt-inspired) and DOM-inspectable conformity; /profile/* has unvetted stand-in primitives.

**Options Considered:**
1. Formalize the existing mechanisms (@matt-source grep + prov-candidates.ts)
2. Build two new dedicated registry files + a `data-prov` stamp ← Chosen
3. Doc-only contract

**Decision:** Approach B — `matt-sourced-registry.generated.ts` (generated from markers) + `matt-inspired-registry.ts` (hand-maintained) + a 3-value `data-prov="matt-sourced|matt-inspired|legacy"` stamp (+ name, + node for sourced) on each primitive's outermost element, plus a conformity gate keeping marker ⟺ registry ⟺ stamp in sync.

**Rationale:** Makes "does this page contain unvetted UI?" a one-line DOM query; gives a browsable catalog; the `legacy` value makes unvetted UI explicit rather than inferred from absence.

**Consequences:** §12 spec written; registries built (W1+W2); stamping (W3 [PRIM-STAMP] #21) + /profile sweep (W4 [PROFILE-PRIM-SWEEP] #22) deferred.
