# Conv Learnings - 2026-05-07

## 1. Hyphen-sensitive regexes silently mis-capture parenthesized placeholder names
**Topics:** infra, scripts

**Context:** Probing /r-timecard-day output for `<!--BLOCK_PARAGRAPH:...-->` markers, used regex `<!--BLOCK_PARAGRAPH:([^-]+)-->`. The `[^-]+` negated class stopped at the first hyphen of `(no-block)`, capturing only `(no` and silently miscounting placeholders as 1 (instead of 2).

**Learning:** When matching a placeholder shape with literal parens, hyphens, and varying inner content, prefer `[^>]+` (or `.+?` lazy) over hyphen-sensitive negated classes. The character that closes the placeholder (here `>`) is the safer terminator.

**Pattern:** `/<!--PREFIX:([^>]+)-->/g` over `/<!--PREFIX:([^-]+)-->/g` — match the boundary char, not the smallest-distinct-non-hyphen-set.

## 2. loadCfg() returns synthesized object, not raw config
**Topics:** infra, scripts

**Context:** New vault-derivation block in timecard-day.js used `(loadCfg().rTimecardDay || {}).vaultPath` — silent failure (no error, just undefined). loadCfg() actually returns `{ full, rt, codeRepo, ... }` where `rt = full.rTimecardDay`. The script's existing convention is `(loadCfg().rt || {})`.

**Learning:** Before adding new code that consumes a long-lived helper, grep the file for existing usages to learn the convention. Conv 157 had two same-class mistakes (regex hyphen + this) — both rooted in "assumed shape vs. verified shape." Verification is cheap; the assume-the-shape habit is the actual cost.

## 3. Tilde-prefix paths in committed config are the right cross-machine portability primitive
**Topics:** infra, dual-repo, cc-workflow

**Context:** Designing `vaultPath` storage. Initial reflex: gitignored `.local.json` so each machine sets its own. User clarified path is identical when expressed as `~/...` (only `$HOME` differs). So one committed value works everywhere; runtime `~` expansion via `process.env.HOME` handles per-machine resolution.

**Learning:** "Per-machine paths require gitignored config" is wrong when `$HOME` is the only divergent piece. Tilde-prefix in committed config + `$HOME` runtime expansion = full portability with zero per-machine setup. Compounded with Obsidian Sync (or any cross-machine file sync) on the filesystem side, you get folder structure portability too — neither layer requires per-machine bootstrap.

**Pattern:** Committed config uses `~/path/to/thing`; consumer does `path.replace(/^~\//, process.env.HOME + '/')`.

## 4. The script-owns-everything rule extends to placeholder enumeration
**Topics:** infra, scripts

**Context:** /r-timecard-day SKILL.md said "scan renderedMarkdown for placeholders" — invited regex hot-path that bit me. The data needed (which blocks lacked Block-summary on every commit) was already in `blockProgress[].blockSummaries[]` from the script. Wrapping that as a top-level `placeholderNames[]` field eliminates the regex from every future executor.

**Learning:** When a skill executor has to re-derive data the script already knows, the right fix is to expose it from the script — not to make the executor smarter. The skill's "the script owns commit grouping, slot math, tag dedup, filter, routing, rendering" is a cross-cutting principle; placeholder enumeration was masquerading as executor work.

## 5. Refactor pure side-effect functions to also return summary data
**Topics:** infra, scripts

**Context:** `emitBlockBody(lines, bp, name)` was a pure side-effect function (pushed lines, returned nothing). To collect placeholder names alongside emission, refactored to return `placeholderName | null`. Caller can ignore the return value to preserve old behavior — the change is strictly additive.

**Learning:** When you need to track "did this side-effect produce X" across many call sites, prefer returning the summary value over passing in a shared mutable collector. Returns are functional, scope to the caller, and let the caller decide whether to collect. Mutable collectors couple every caller to the collection contract.

**Pattern:** Function pushes to an out-parameter array AND returns null/value. Caller `const ph = fn(out, ...); if (ph) collected.push(ph);`.

## 6. RESUME-STATE.md and committed code aren't in lockstep until /r-end
**Topics:** cc-workflow

**Context:** User asked "is /r-timecard-day written in such a way that it will work on M4Pro?" After a detailed yes-with-audit, I had to flag 🟠 that M4Pro is currently reading the pre-change skill code because /r-end hasn't run on M4 yet to commit + push. The portable design doesn't help if M4Pro is on the old version.

**Learning:** When validating "does X work on machine Y," distinguish "design works" from "machine Y has the design loaded." The two diverge during a conv where M4 has uncommitted changes. The distinction matters when answering "can I run this on M4Pro now?" — the answer can be "the design is portable AND you'd need to /r-commit + /r-start first for M4Pro to see it."
