# Conv Learnings - 2026-05-28

## 1. CC permission matcher is prefix-based on the literal command string
**Topics:** cc-workflow

**Context:** While auditing wildcards for danger in `.claude/settings.local.json`, had to reason about what `Bash(git:*)` actually matches vs `Bash(git -C <path> push --force)`.

**Learning:** CC's Bash permission matcher does prefix matching on the literal command string with `:*` as args-wildcard. `Bash(git push --force*)` likely does NOT match `git -C <path> push --force` because the prefixes differ. Deny patterns therefore cannot fully cover the `git -C` form via a single rule — would need separate deny entries for each `-C <repo-path>` variant, OR rely on system-prompt's "confirm before destructive git" guidance for those cases.

**Pattern:** When designing deny rules for a command with multiple invocation forms (positional vs `-C`, with/without env var prefix), enumerate the forms explicitly OR accept the limitation.

## 2. Broad command grants compose into arbitrary-code-execution surfaces
**Topics:** cc-workflow

**Context:** Removed `Bash(bash:*)` as the "allow-anything" escape hatch — only to realize `Bash(node:*)`, `Bash(python3:*)`, `Bash(npx tsx:*)`, and `Bash(xargs:*)` provide functionally equivalent escape hatches via `-e '<arbitrary>'` and command-pipe constructs.

**Learning:** Permission tightening must consider the WHOLE set of code-execution surfaces, not just the obvious `bash`. Path-scoping (`Bash(node .claude/scripts/*)`) is the cleaner fix than wildcard-banning, because it preserves the legitimate use case (running known scripts) while denying the `-e` escape.

**Pattern:** For interpreter binaries (node/python/tsx/perl/ruby), prefer path-scoped allows over `:*` wildcards.

---

## 3. Empty referenced directories are permission cruft
**Topics:** cc-workflow

**Context:** `.claude/settings.json` granted `Read(/RFC/**)` and `Read(/research/**)`, but both dirs existed empty. Real RFC content lives at `docs/requirements/rfc/`.

**Learning:** Permission entries pointing at empty/wrong directories are pure noise — they never match anything but clutter the allow list. Worth `ls`-checking referenced paths during audits.

---

## 4. Global vs project vs local settings: the right axis is "shared-ness"
**Topics:** cc-workflow

**Context:** User asked what's truly global. Forced ranking: UX/client-behavior preferences (statusLine, autoCompact, thinking, verbose) only make sense globally because they apply to every session in every project; permissions are inherently project-scoped because they describe a specific project's tool envelope.

**Learning:** The split is: **global** = CC-client behavior + cross-project hooks (machine detection); **project committed** = "what this project authorizes CC to do on any machine"; **project local** = destructive grants + per-machine paths + editor/OS-specific tools. The destructive cases stay local because they're a per-user trust decision, not a project-level fact.
