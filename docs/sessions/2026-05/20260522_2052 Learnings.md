# Conv Learnings - 2026-05-22

## 1. DEV-STAGING-SSR scope is broader than PLAN documented
**Topics:** astro, cloudflare, react, deployment

**Context:** PLAN.md entry for [DEV-STAGING-SSR] (Conv 122) describes the React-hooks-null SSR crash as affecting only `npm run dev:staging` (with `remoteBindings: true`), with the symptom being "HTTP 200 still returned, islands recover on client hydration." Conv 176 found this description incomplete.

**Learning:** The same `Cannot read properties of null (reading 'useState')` SSR error fires under plain `npm run dev` too (no `remoteBindings`) — at least on M4 in Conv 176. AND the symptom is NOT graceful island fallback; it's a complete page-body cutoff: the response stops after `<body class="…">` with zero further content. Any matt/* page that imports a stateful React component (one calling `useState`) crashes the entire body. The latent Sidebar.tsx `useState` only manifested once another stateful primitive (ToDoItem) was on the same page — so existing matt course pages worked because all their React components were stateless.

**Pattern:** Until the upstream fix lands or the canonical Vite dedupe workaround is reapplied (see [NPM-UP] task #29), matt/* primitives must be stateless / fully controlled. Internal state belongs in parent demo wrappers, not in primitives themselves.

## 2. Astro `.astro` expression-block parser only accepts component references, not inline HTML
**Topics:** astro, frontend

**Context:** Tried to pass `embed={<div className="…">…</div>}` to a React component from inside an `.astro` file. Got parse error `Expected ">" but found "className"`. Tried `class=` instead — same error (`Expected ">" but found "class"`). Initially read as a bug; web research confirmed it's documented Astro behavior.

**Learning:** Astro template-expression syntax in `.astro` files supports `{<Component />}` (capitalized component references) but NOT `{<div>…</div>}` (raw HTML elements). The Astro docs at docs.astro.build/en/reference/astro-syntax/ are explicit that kebab-case HTML attributes are required in template body, but the expression-block parser is more restrictive: it accepts only component references in `{…}` prop positions. This is by design — Astro keeps the JSX surface narrow to avoid conflating template-mode and expression-mode parsing.

**Pattern:** When a React component prop needs rich JSX content (like an embedded card), extract the JSX into a React `.tsx` file and reference it as a single component: `<SomeDemo />`. Underscore-prefix the file (`_SocialPostDemo.tsx`) to signal internal/showcase status. Pure JSX (with `className=`) is legal inside the .tsx file — only `.astro` expression blocks are restrictive.

## 3. Matt's `.entity-*` multi-mode cascade doesn't propagate through Tailwind 4's `@theme` intermediate
**Topics:** tailwindcss, frontend, astro

**Context:** matt-design-system.md §5 documents the Entity multi-mode cascade as a load-bearing mechanism: `:root` has `--Entity-Background: var(--gray-100)`; `.entity-student` overrides `--Entity-Background: var(--Student-Background)`; children consume `bg-entity-background` (which resolves to `var(--color-entity-background) → var(--Entity-Background)` via Tailwind 4 `@theme`). The CSS chain is logically correct per the spec — `var()` resolves at the using element, so overrides should propagate.

**Learning:** Empirically the cascade does NOT propagate when consumed through Tailwind 4's `--color-*` intermediate. The active background renders as the `:root` default (`--gray-100`, grey), not the overridden value (`--Student-Background`, pastel-blue), even with `.entity-student` and `bg-entity-background` on the same element. The exact failure mode is unknown — possibly Tailwind 4's `@theme`-generated CSS computes `--color-entity-background` once at `:root` time rather than re-resolving per-element, OR Vite's transform inlines the value statically. CourseHeader and similar Matt primitives that work visually may be using direct entity utilities (`bg-course-background`) under the hood, not the cascade.

**Pattern:** Use direct entity-specific Tailwind utilities (`bg-student-background`, `text-creator-primary`, etc.) — same pattern as `Button.tsx`'s six-variant collection. Reserve the `.entity-*` cascade for components that consume only canonical Matt CSS variables (`var(--Entity-Background)`) without going through the Tailwind bridge. Tracked as [CASCADE-BROKEN] task #28 — needs an isolated repro before either fixing the cascade or retiring the pattern.

## 4. SVG visual reference workflow: qlmanage SVG→PNG enables Read-tool inspection
**Topics:** cc-workflow, design

**Context:** Wanted to read Matt's Figma component SVGs (Module/Note/SocialPost/ToDoItem) visually before writing the primitives. Each SVG is 64–702 KB of XML — too dense to read via the Read tool (Module.svg returned a 34,129-token error at limit=50). The [MPV] workflow (symlink + Chrome view) requires a running browser the assistant can't drive when Chrome MCP is disconnected.

**Learning:** macOS `qlmanage -t -s 1200 -o /tmp file.svg` rasterizes SVG to PNG at the specified size. The Read tool then displays the PNG visually as an image, letting the assistant decode layout, colors, states. Total time per SVG: ~1 second. This bypasses both the text-too-dense issue AND the no-Chrome-driving issue, and is repeatable for any conv on macOS.

**Pattern:**
```bash
qlmanage -t -s 1200 -o /tmp file.svg
# produces /tmp/file.svg.png — read visually via the Read tool
```
Recommend adding this as a fallback step in the [MPV] workflow / matt-pre-plan documentation.

## 5. Astro inline `<svg>` in component-prop expressions crashes parser; emoji or React-IconComponent are the working alternatives
**Topics:** astro, frontend

**Context:** First attempt at Module/SocialPost showcase used inline SVG icons: `icon={<svg viewBox="0 0 24 24" stroke-width="1.5" .../>}` passed as ReactNode prop. Astro parser crashed at the attribute name: `Expected ">" but found "viewBox"`.

**Learning:** Same root cause as Learning #2 — Astro's `.astro` expression-block parser rejects HTML/SVG elements with attributes that don't pass strict checks. Even component-tag-name elements with all-lowercase like `<svg>` are problematic in expression contexts. Working alternatives: (a) pass an emoji string (`icon="⌘"`) — accepted as ReactNode by all current primitives, (b) reference a React Icon component (`icon={<MyIcon />}` if MyIcon is a `.tsx` component import), (c) extract the entire React-component-using-SVG into a `_Demo.tsx` file. Same fix family as the embed-card extraction.
