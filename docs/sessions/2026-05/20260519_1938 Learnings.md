# Conv Learnings - 2026-05-19

## 1. Bash permission gate flags `$VAR` but not `~`
**Topics:** cc-workflow, dual-repo

**Context:** User reported persistent `simple_expansion` permission prompts on commands like `git -C $CLAUDE_PROJECT_DIR push`. Investigation of `~/projects/spt-docs` showed `w-test-tilde/SKILL.md` empirically confirming "Form A removed (it hard-failed on simple_expansion)" while tilde form passes silently.

**Learning:** The Claude Code Bash tool's permission gate treats `$VAR` (any external env-var reference, including `$CLAUDE_PROJECT_DIR` and `$HOME`) as `simple_expansion` and prompts. Tilde (`~`) is functionally equivalent to `$HOME` for path resolution, but the gate treats `~` as a literal prefix — bash expands `~` at parse time before the gate sees it as a value. Net effect: use `~/projects/peerloop-docs` everywhere, never `$CLAUDE_PROJECT_DIR` or `$HOME`, to keep skill-issued commands fully prompt-free.

**Pattern:** Substitute `$VAR` → tilde-literal in all skill SKILL.md command snippets. Drop surrounding double quotes from path strings (paths have no spaces, unquoted is safe — tilde inside double quotes does NOT expand). For slug-style derivations, replace `${CLAUDE_PROJECT_DIR//\//-}` with `$(echo ~/projects/peerloop-docs | tr / -)`.

## 2. Cross-machine username asymmetry — M4 (`livingroom`) ≠ M4Pro (`jamesfraser`)
**Topics:** cc-workflow, dual-repo, infrastructure

**Context:** During CPD-SWEEP, an initial pass substituted `$HOME` with literal `/Users/jamesfraser` across 14 sites to eliminate the simple_expansion prompt. User corrected: M4 uses username `livingroom`, only M4Pro uses `jamesfraser`. The literal-path substitution would have silently broken /r-start, /r-commit, and /r-end on M4. Conv 152 [MPP] rewrote `livingroom`-bearing memory paths to portable forms specifically because of this asymmetry.

**Learning:** Both Macs do NOT share the same username. **M4Pro: `jamesfraser`, M4: `livingroom`.** Any "hardcode the username" shortcut breaks portability. The Conv 155 cross-machine sync worked because the original code used `${CLAUDE_PROJECT_DIR//\//-}` (which derives per-user at runtime); substituting it for a literal-jamesfraser slug undid that portability.

**Pattern:** Cross-machine code in this project must rely on either (a) tilde / `$HOME` semantics that resolve per-user at runtime, or (b) dynamic derivations like `$(echo ~/... | tr / -)`. Never hardcode `/Users/jamesfraser` or any username-bearing path in skill code.

## 3. Tilde inside double quotes does NOT expand
**Topics:** cc-workflow

**Context:** First pass of CPD-SWEEP attempted `MIRROR="~/projects/peerloop-docs/..."` to preserve a quoted assignment. The resulting path contained a literal `~` (mkdir/rsync would fail). bash tilde expansion only applies to a tilde at the **start of a word** (and special cases like after `=` in assignments and after `:` in PATH-style values), and crucially only **outside** quotes.

**Learning:** For path-string assignments where you want tilde expansion AND prompt-free single-line commands, use unquoted form: `LIVE=~/.claude/projects/$SLUG/memory`. Bash tilde-expands the `~` here (start of value after `=` in an assignment), and `$SLUG` interpolates at runtime as a local-scope variable (no gate prompt). The variable is then safe to use as `"$LIVE"` (quoted at use site).

**Pattern:** `VAR=~/path/$other_var` (unquoted assignment) is the prompt-free portable form. Use this whenever the path has no spaces. Local-scope script variables (`$SLUG`, `$LIVE`, etc.) defined and consumed inside the same bash block are NOT gated by simple_expansion — only references to external env vars are.
