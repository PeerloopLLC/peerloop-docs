# Conv Learnings - 2026-05-06

## 1. Cross-machine memory directories don't sync even though the project is shared
**Topics:** dual-repo, cc-workflow, dual-machine

**Context:** [OPW] investigation revealed `feedback_option_phrasing.md` (authored Conv 132, strengthened Conv 147) had never reached MacMiniM4-Pro. The memory-dir path includes the username (`/Users/<user>/.claude/projects/-Users-<user>-projects-peerloop-docs/memory/`), so memories written on one machine are invisible on the other even though the dual-repo project they describe is shared.

**Learning:** What looked like a Conv 148 rule violation was actually a delivery failure — Claude on this machine literally couldn't have read the rule. The "watch over ~5 convs" task ([OPW]) was actually a memory-delivery audit in disguise, surfacing the systemic gap by tripping over its own absence. Tracked as `[CMS]` for durable solution; for now manual sync via desktop-folder copy.

## 2. CLAUDE.md size accumulates as navigation + archaeology, not behavioral rules
**Topics:** cc-workflow, docs-infra

**Context:** A long-lived project's CLAUDE.md drifted to 677 lines / 22 sections. ~40% was three sections (Database Migrations 123, Research Reference 95, Project Structure 50), most of which was navigation indexes ("if you need X, look at Y") and Session-numbered archaeology — not behavioral rules.

**Learning:** Distinguish CLAUDE.md content by purpose: behavioral rules + essential project context belong here; navigation indexes belong in `docs/INDEX.md`; dated archaeology belongs in `TIMELINE.md`; run-scope context belongs with the run (`docs/as-designed/run-XXX/`). When a CLAUDE.md section grows past ~30 lines, it's worth checking whether some of it is navigation or archaeology that wants a different home. A delegated audit agent under-delivered for a long-lived project (11 findings vs. the ~25-30 a careful reader would see) — direct reading by the main context surfaced the structural issues the audit missed.

## 3. Memory ↔ CLAUDE.md coupling needs explicit canonical-home discipline
**Topics:** cc-workflow, docs-infra

**Context:** Multiple instances this conv where a rule was duplicated or partially split between CLAUDE.md and a memory file (Issue Surfacing × `feedback_visual_issue_alerts.md`; baselines × 2 memories; durable-vs-novel × `feedback_default_durable_no_ask.md`; Schema Discrepancy ↔ `feedback_option_phrasing.md` via cross-ref).

**Learning:** When a rule has both a CLAUDE.md home and a memory home, decide canonical home per rule, then make the non-canonical side either (a) a stub pointer with no rule duplication, or (b) a richer file holding incident context that CLAUDE.md doesn't carry. Either pattern is fine; what hurts is full duplication, because CLAUDE.md is loaded into every conv anyway — duplicating its rule body in a memory stub is dead weight. The reverse — promoting a memory rule into CLAUDE.md — is correct when the rule is high-frequency and load-bearing for behavior (e.g., baselines were promoted Conv 150).

## 4. The plan-mode + auto-mode pairing handles "thoroughly think it through, then execute" cleanly
**Topics:** cc-workflow

**Context:** User asked for plan mode for a big restructure. Dispatched 3 parallel Explore agents during planning to characterize external risk (very low), move-target state (docs/INDEX.md absent, TIMELINE.md a fit), and memory ripple (5 cross-refs all preserved). Plan written, approved, auto-mode entered, 10 steps executed without further confirmation.

**Learning:** Plan-mode is the right shape when (a) the user wants alignment on approach before any edits, (b) the change touches many files, (c) there are real risk decisions (what to keep vs. move). Auto-mode after approval lets execution proceed without interruption. Three parallel Explore agents during planning gave good coverage at low context cost — each with a focused remit, structured output, and a word cap.
