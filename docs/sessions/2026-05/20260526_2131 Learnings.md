# Conv Learnings - 2026-05-26

## 1. A git worktree at a pre-migration commit is a zero-debt reference environment
**Topics:** cc-workflow, dual-repo, astro

**Context:** Needed to inspect legacy `/old` look+behavior that 404s on the live branch (broken deep-nav). Considered three in-branch patches (client-side href rewrite, basePath prop, middleware 404-fallback).

**Learning:** A `git worktree add <path> <pre-flip-commit>` gives the genuine working app at that point in history, browsable on a second dev port, with zero modification to the working tree and nothing to remove later (vs. scaffolding that accrues removal-debt). When the artifact you'd patch is *being deleted anyway*, the durable choice is the most-contained/single-point-removable one — which here is "don't patch at all; reference history."

**Pattern:**
`git worktree add ~/projects/<repo>-ref <commit>` → `npm install` → seed own local D1 → `npm run dev -- --port <offset>`. Worktree is machine-local (not pushed); reproduce via an idempotent bootstrap script + the commit SHA in shared history.

## 2. /chrome bridge is port/URL-based, not directory-bound
**Topics:** cc-workflow, testing

**Context:** User worried the worktree (a separate directory) couldn't be inspected via /chrome.

**Learning:** The browser↔server boundary is HTTP. The bridge drives Chrome by URL/port and has no awareness of which filesystem/worktree served the page. Two dev servers (:4321 live + :4331 reference) coexist and are both fully drivable.

## 3. A silent route fallback would defeat RTMIG's verification signal
**Topics:** cc-workflow, astro, testing

**Context:** Middleware 404→/old rewrite was proposed as a convenience.

**Learning:** RTMIG-4's whole hazard is string-valued route refs invisible to tsc/astro-check — only a real 404 reveals a broken root href. A fallback that silently serves `/old/X` under a root URL would mask exactly those bugs (and re-add the redirect layer the user chose against). During this migration a visible 404 is a feature, not a defect.

---

## 4. RTMIG-4 is a legacy routing migration, not a build-to-Matt-design
**Topics:** cc-workflow, astro

**Context:** Assumed root pages might each need a Matt design.

**Learning:** Matt designed only 9 pages and they're already at root (moved by ROUTE-FLIP). The other ~90 `/old` pages have no Matt design, so RTMIG-4 cannot be "build to design" without blocking on 90 designs. It is fundamentally a routing migration of legacy pages; Matt redesign is a separate later track (MATT-EXEC-*/MMP-*). The open fork is which *shell* legacy bodies land in at root.
