# Conv Decisions - 2026-05-26

## 1. url-routing.md §§1–7 framing: design-canonical + status banner (Option B)
**Type:** Strategic
**Topics:** docs-infra, dual-repo

**Trigger:** Conv 197's §8 note + task #25 title both said "rewrite §§1–7 for /old layout", but §§1–7 also encode the enduring URL grammar Matt's root app inherits — literal /old-prefixing would invert the doc into a legacy reference.

**Options Considered:**
1. Literal /old rewrite — prefix every §§1–7 route with `/old/`, reframe as legacy reference
2. Design-canonical + status banner ← Chosen
3. Split structure — two top-level halves (Current Root App / Legacy /old Reference)

**Decision:** Keep §§1–7 as the canonical URL architecture (bare routes, no `/old/` prefix); add one status banner at top of § Route Categories stating which routes are built at root vs currently served under `/old/*`; rewrite only the file tree to the real post-flip layout. Applied the same confine-the-churn philosophy to the other three docs.

**Rationale:** The URL *grammar* is permanent; the *file location* under `/old/` is a transitional fact that reverses as Matt's pages roll forward. A durable doc separates "what URLs mean" (timeless) from "which app serves them" (a churning status table) so churn lives in one place.

**Consequences:** Banner + §8 table absorb future migration churn; "Why This Design" rationale untouched; the same banner technique seeded the matt-design-system/ INDEX banner and the matt-frames Status note.
