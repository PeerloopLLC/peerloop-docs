## 2. Use SVG (not PNG) for happy-path frame exports
**Type:** Implementation
**Topics:** figma, design-tooling

**Trigger:** User asked "did you mean SVG?" after I initially suggested PNG for batch-export of happy-path frames.

**Options Considered:**
1. PNG only — simpler, lossless visually, but no extractable values for Claude to read
2. SVG only — same visual fidelity, plus embedded styles (hex codes, font specs) Claude can extract directly from file content ← Chosen
3. Both PNG + SVG — strict superset but doubles export effort

**Decision:** SVG for all batch-exports going forward.

**Rationale:** Hedges against MCP failing again next conv. If MCP comes online cleanly, SVG is redundant but harmless. If MCP doesn't materialize after Brian's paid setup, SVG gives us extractable values (exact hex, font sizes, spacing) from the happy-path screens themselves — not just the dedicated token pages. Same visual fidelity as PNG. File sizes can balloon if Matt embeds bitmaps (some Course About SVGs hit 2-3 MB) but disk space isn't constrained.

**Consequences:** All 229 exports came in as SVG (where the user chose format) — gives us a complete extractable-values backup if MCP doesn't unblock next conv.

## 3. Tokens designed as future global default, consumed only by `/matt/` initially
**Type:** Strategic
**Topics:** matt-design, design-tokens, css-architecture

**Trigger:** User clarified style scope: "scoped to /matt only. We will be moving other pages into /matt so that they conform to the style guide. Eventually we will be flipping /matt into / and the current / tree into /fraser."

**Options Considered:**
1. Hard isolation — Matt-prefixed utility classes, separate CSS namespace, no shared surface
2. Future-global tokens consumed only by `/matt/` routes initially ← Chosen
3. Replace defaults immediately, re-skin existing pages as we go

**Decision:** Design tokens as the future global default (CSS custom properties + Tailwind theme extension), but only let `/matt/*` routes consume them initially via a dedicated `MattLayout.astro`.

**Rationale:** The eventual flip (`/matt/` → `/`, current `/` → `/fraser`) means tokens that look like "Matt's" today become the production default later. If we name/structure them as Matt-specific now, the flip requires renaming every reference. By treating tokens as the future-canonical layer from the start, the flip becomes a layout-import swap, not a token migration. Existing layouts stay untouched and keep using their current styles until they're individually migrated into the new system.

**Consequences:** Token file location/naming next conv should reflect future-default identity (e.g., `tokens.css`, `theme.config.ts`), not `matt-tokens.css`. The `/matt/` constraint lives in *which layouts opt in*, not in token naming or namespacing. Reaffirms before any actual implementation work.
