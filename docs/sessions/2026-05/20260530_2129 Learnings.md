# Conv Learnings - 2026-05-30

## 1. The route-flip never wrote `/old/` into hrefs — so grepping hrefs for it is tautological
**Topics:** cc-workflow, astro, dual-repo

**Context:** User asked to grep the /communities component tree for links containing `old/`; predicted none.

**Learning:** ROUTE-FLIP (Conv 197) relocated top-level page *files* into `src/pages/old/` but left every internal href and component untouched. The `/old` prefix is a filesystem/route-location artifact that never appears in a hardcoded href. So a "does any link contain old/" grep can only return zero — the real question is "does each bare-root href resolve to a built route yet, and is that target vetted?" — a route-resolution + human-vetting question, invisible to both grep and tsc.

---

## 2. 404-honesty is also the vetting gate (forward links self-seal)
**Topics:** cc-workflow, astro

**Context:** Worried that converting a page silently exposes unvetted pages reachable through it.

**Learning:** Because unbuilt routes 404 (no redirect layer, no resolving stubs), a new page's forward links can't surface unvetted content until that content's route is *deliberately built*. The community cards link to `/community/[slug]`, which has no root route → all 404. So the convert-vs-vetted boundary is self-enforcing for forward links; the only resolving links are pre-existing converted pages (`/`, `/onboarding`). Building a page top-down through the route hierarchy is safe by construction.

## 3. "Courses-parity" faithfully reproduces courses' OMISSIONS, not just its features
**Topics:** cc-workflow, testing

**Context:** Endpoint-parity check found the new page omits `/api/admin/intel/communities`; also the missing sort `<select>`.

**Learning:** Mirroring the `/courses` build (fidelity A) means the new page differs from its own legacy original exactly where `/courses` already diverged: admin-intel attention badges (dropped) and sort/pagination (intentionally omitted for small catalogs). "Parity" = match the new-design sibling, not the legacy source. When checking a port for "lost functionality", compare against the parity target, not only the legacy original — a gap vs legacy may be an intended gap inherited from the sibling.

---

## 4. tsc validates client-island API assumptions only when data flows through typed props
**Topics:** testing, astro

**Context:** CommunitiesCatalog relies on CurrentUser methods (getTeachingCommunityIds, isCommunityModeratorFor, getCommunityMemberships shape) invisible from the page.

**Learning:** Because those flow through typed props/imports, `tsc --noEmit` validated them before any browser run — the opposite of the route-flip's string-href breakages that only the browser caught. Typed data flow is the cheap gate; string-valued routes/behavior are the expensive (browser-only) gate.
