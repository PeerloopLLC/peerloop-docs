# Conv Learnings - 2026-05-25

## 1. MattIcon wrapper is `fill="none"` — harvested paths MUST carry `fill="currentColor"`
**Topics:** astro, matt-design-system, icons

**Context:** Harvesting Material SVGs (no fill attr) into the MattIcon registry for NAV-ICON-SWAP.

**Learning:** MattIcon's wrapper `<svg>` sets `fill="none"`, so any inner path without an explicit fill inherits "none" and renders INVISIBLE. Raw Material SVGs (marella mirror) have no fill on paths. Must inject `fill="currentColor"` into every path/circle/rect during harvest — same normalization Conv 182 did for Matt's own icons. Also: an unknown icon name → MattIcon renders a dashed-square placeholder (a useful "broken name" signal).

**Pattern:**
```
perl -pe 's/<(path|circle|rect|polygon|ellipse)(?![^>]*\bfill=)/<$1 fill="currentColor"/g'
```

## 2. VT-drop applies to island-unique ARBITRARY utilities, not within-route-family classes
**Topics:** astro, nav-retrofit

**Context:** Deciding whether AdminNavbar's `w-[220px]` etc. need inlining like the Conv 191 slideout offsets did.

**Learning:** The View-Transition CSS-drop bug (Conv 191) hits arbitrary utilities that are island-unique AND absent from the destination route's CSS bundle. AdminNavbar persists only across admin→admin VT, and every admin route generates its classes, so arbitrary classes (`w-[220px]`, `size-[32px]`) are safe there. Only the AppNavbar *slideout offsets* (left/bottom), which dropped when navigating to /matt, needed inline styles.

## 3. Matt's 43-icon set is product-oriented; nav-chrome & admin-tooling icons need Material harvest
**Topics:** matt-design-system, icons, nav-retrofit

**Context:** NAV-ICON-SWAP — mapping legacy nav icons to Matt glyphs.

**Learning:** Matt designed course/creator/student/feed/certification/ratings-type icons but no generic nav chrome (menu, search/discover, admin, chevron) or admin-tooling (users, tags, clipboard, moderation-warning, person-add). ~half of both navs' icons had no Matt equivalent. With Material Design icons as an allowed source, every gap fills — so the dashed-border fallback the user requested ended up unused. Probing the icon registry BEFORE swapping surfaced the gap (external-source-of-truth discipline).

## 4. cp-created files aren't harness-tracked — Edit fails until Read
**Topics:** cc-workflow

**Context:** Graduating matt-frames-ready-for-dev.md via `cp` + in-place Edit.

**Learning:** A file created by a Bash `cp` isn't registered as "Read" by the harness, so the first Edit on it errors ("File has not been read yet"). Read it once first. (cp+Edit is still preferable to re-Writing a 138-line table from memory — byte-fidelity beats transcription risk.)

## 5. Graduating a machine-pinned scratch file to committed docs removes the machine-pin permanently
**Topics:** matt-design-system, docs-infra, dual-repo

**Context:** MMP-PH5 + MFRD-GRADUATE — the gitignored `.scratch/matt-figma-*.md` files existed only on MacMiniM4, blocking graduation from MacMiniM4Pro (Conv 192).

**Learning:** The promotion was blocked not by logic but by machine: scratch is gitignored/un-synced, so it lived on one machine. Running on MacMiniM4 (the authoring machine) unblocked it; once in committed `docs/`, the lookup syncs to both machines and is no longer machine-pinned. Lesson: machine-pinned scratch work should graduate on the authoring machine at the first opportunity.
