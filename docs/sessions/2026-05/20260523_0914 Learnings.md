# Conv Learnings - 2026-05-23

## 1. Vite cold-start dep discovery is a documented industry-wide pattern
**Topics:** astro, cloudflare, vite, ssr, debugging

**Context:** Conv 122 → Conv 176 → Conv 177 chased the wrong diagnosis ("dual React copies via @astrojs/cloudflare") for months. Conv 177's web research found the same exact symptom (cold-start "Cannot read properties of null (reading 'useState')" originating in `.vite/deps_ssr/chunk-*.js`) is widely reported across Remix (#10156), TanStack/router (#4264), Storybook (#32049), and Vite itself (#17979). The pattern: Vite's lazy dep discovery finds a new import on the first request, re-optimizes the deps_ssr bundle, swaps the bundled React copy, and the in-flight render hits a null hooks dispatcher because it imported the old copy.

**Learning:** When SSR React hooks crash mid-render, "duplicate React copies" is the wrong first instinct unless the crash is persistent. If the crash is cold-start-only and self-heals on the next request, it's the Vite optimizeDeps race — a different bug class with a different fix. Both classes share the same error message, which is why this took 3 convs to pin down.

**Pattern:** Diagnostic checklist for SSR hook crashes:
1. Does the SAME request reliably reproduce after a fresh server start? If yes → cold-start race (this class).
2. Does the crash persist across multiple requests? If yes → duplicate React copies (#11825 class).
3. Does production build reproduce? If yes → fundamental config issue (neither class).

## 2. Diagnosis can survive long because misleading symptoms cluster
**Topics:** debugging, mental-models

**Context:** The "AppNavbar works but Sidebar crashes" mystery seemed like it required a complex per-component explanation. The real explanation was much simpler: request order. AppNavbar appeared to "work" because every test request that hit AppNavbar happened to land AFTER an earlier request had warmed the Vite cache. The /matttest probe revealed this when /matttest succeeded on the SECOND request to a Sidebar-using route, and then /matt/'s third request also succeeded.

**Learning:** When a hypothesis requires increasingly elaborate explanations to fit the data, test for order-dependence and cache state before adding complexity. A second probe of an earlier-failing condition can falsify a "structural" hypothesis cheaply.

**Pattern:** Before concluding "X is different from Y", swap their execution order and re-test. If the behavior flips, the difference was order/cache, not structure.

## 3. The Vite optimizeDeps.entries recipe doesn't reach Astro virtual modules
**Topics:** vite, astro, configuration

**Context:** The community-tested recipe `optimizeDeps.entries: ["src/**/*.tsx", "src/**/*.ts"]` works for Remix/React Router projects but didn't catch `astro/virtual-modules/transitions.js` for us. Reason: Astro virtual modules are imported by `.astro` files (which the esbuild scanner can't parse) AND through Astro's exports field, not visible as a direct `import` in src/.

**Learning:** For Astro projects, the Vite cold-start fix needs to combine `optimizeDeps.entries` (catch most app deps) with `optimizeDeps.include: ['astro/virtual-modules/<name>.js']` for each Astro virtual module the dev log mentions in `✨ new dependencies optimized: <name>`. New Astro features that introduce new virtual modules may need to be added to `include`.

**Pattern:** Watch the dev log for `✨ new dependencies optimized: <X>`. Any X listed there should go in `optimizeDeps.include` to prevent the cold-start crash.

## 4. Astro 6.3 added a `logger` field to APIContext
**Topics:** astro, testing, type-safety

**Context:** After upgrading to Astro 6.3.7, tsc flagged `tests/api/helpers/api-test-helper.ts:237` as missing the `logger` property — the test helper's `TestAPIContext` literal didn't include it. Fix was a no-op stub matching the existing `cache` stub pattern.

**Learning:** Astro minor versions can add fields to `APIContext` that require updating any TestAPIContext helper. The pattern that already exists for `cache` (no-op stub with comment explaining it's a test stub) is the template to follow when this happens again.
