# Conv Learnings - 2026-05-06

## 1. First cross-machine /r-start is the only data-loss window in [MSI]
**Topics:** msi, dual-repo, cc-workflow

**Context:** Conv 155 ran the first cross-machine /r-start (M4 → M4Pro). The pre-Conv-155 Step 5.7 went straight to `rsync -a --delete` mirror→live with no audit trail. User asked "what was different?" and the answer was unrecoverable.

**Learning:** The architectural property is precise: live can have content not in mirror only when (a) the user hasn't run /r-end on this machine yet, OR (b) this machine never ran /r-start with the existing mirror so live is from a pre-mirror state. After one successful /r-end on a machine, mirror reflects live, so subsequent /r-start halts cannot fire on `Only in $LIVE` (other machines' edits go to their own live, mirror via their own /r-end). The reverse syncs (live→mirror in /r-commit and /r-end) are safe because /r-start runs first per conv → live ⊇ mirror.

**Pattern:** When designing one-way sync mechanisms with `--delete`, the data-loss window is always at the bootstrap of the destination side. Detect it with a forward-direction-only check (`Only in $DEST` from `diff -rq`), halt before destructive action, snapshot the destination, present user with copy/proceed/inspect options.

## 2. `rsync -a` "transferred N bytes" doesn't imply content differed
**Topics:** msi, infra

**Context:** Conv 155's first rsync output showed `sent 100147 bytes, total size 94818, speedup 0.94` — full transfer. Initial assumption: content must have differed across all 51 files.

**Learning:** `rsync -a` defaults to size+mtime check. If mtimes differ (e.g., from a tarball extract on one machine and later edits on another), rsync transfers even when content is byte-identical. To distinguish content-vs-mtime drift, either use `rsync -c` (content checksum, more CPU) or run `diff -rq` separately.

**Pattern:** When rsync output suggests "everything changed" but the conv-history doesn't support that, suspect mtime drift before content drift. The forensics block in /r-start Step 5.7 now uses `diff -rq` directly, sidestepping this.

## 3. Hands-off pilot workflow is a load-bearing architectural invariant
**Topics:** dual-repo, cc-workflow

**Context:** Conv 155 wrote `user_hands_off_pilot_workflow.md` (type: user). The user mediates ALL project-file changes through Claude Code — never edits files directly.

**Learning:** This invariant is what makes several pieces of infrastructure substantially simpler: [MSI] trusts the mirror at SessionStart, skill state files (`CONV-COUNTER`, `.conv-current`, `RESUME-STATE.md`) rely on commit-then-push as the only valid mutation path, cross-machine concurrency is bounded by /r-end → push → /r-start → pull discipline. Designs that assume CC is sole author do NOT need to defend against arbitrary out-of-band edits.

**Pattern:** When designing infrastructure (skills, hooks, sync, drift detection) for this project, lean on the sole-author invariant. The override case (rare emergency manual edit) is the user's responsibility to flag and route through CC for review on the next conv.

## 4. Mirror folder project awareness via parent-repo nesting is sufficient
**Topics:** msi, dual-repo

**Context:** User initially worried mirror files lacked project awareness for cross-project use (peerloop + spt). Considered adding folder-rename, manifest file, or per-file frontmatter to encode project name.

**Learning:** Mirror IS already project-aware — it's at `<project>/.claude/memory-sync/memories/`, inside the repo. spt's mirror would be at `spt-docs/.claude/memory-sync/memories/` — completely separate filesystem location, separate git history. Cross-project leak would require manual file copy between two clones; that's an explicit cross-repo move, not an oversight. The shared user-level location (`~/.claude/projects/<slug>/`) is the LIVE side, where slug encoding already provides project awareness.

**Pattern:** Before adding identity markers (frontmatter, manifests, naming conventions), check whether the existing path/nesting already disambiguates. In this case, repo nesting was sufficient.
