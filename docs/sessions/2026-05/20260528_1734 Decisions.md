# Conv Decisions - 2026-05-28

## 1. Drop the PLAN.md narrative trail entirely
**Type:** Strategic
**Topics:** docs-infra, cc-workflow

**Trigger:** User asked what purpose the `*Last Updated:* Conv N — [narrative]*` trail at PLAN.md line 721 served after the Conv 210/211 migration left it stranded mid-file.

**Options Considered:**
1. Leave it in PLAN.md (accept placement awkwardness)
2. Move to `plan/conv-trail.md` (single file)
3. Split by block into `plan/<slug>/` files
4. Drop entirely; `/r-end` updated to stop appending ← **Chosen**

**Decision:** Delete the trail from PLAN.md and update `/r-end`'s `fmt-update-plan.md` to forbid future writes.

**Rationale:** `/r-end` already writes 3 files per conv under `docs/sessions/YYYY-MM/` (Extract, Decisions, Learnings). The Extract is the canonical conv narrative. The PLAN.md trail was a denormalized echo with zero unique value — even cross-cutting meta/infra changes are captured in the Extract. The "readable from PLAN.md alone" use case the trail served is itself broken now that PLAN.md is a thin index.

**Consequences:** PLAN.md drops ~100 lines per conv of redundant content; `/r-end` doc-bookkeeping load drops. Splitting by block (option 3) was actively argued against — entries are narrative-shaped and resist clean splits.

## 2. Consolidate cross-conv watch-tasks into a single section in PLAN.md, not TodoWrite
**Type:** Strategic
**Topics:** cc-workflow, docs-infra

**Trigger:** Audit found 8 unique pending watch-tasks scattered across 3 Conv-N drain sections (150-157 / 179 / 206), none of them in TodoWrite.

**Options Considered:**
1. A) PLAN.md `## Cross-Conv Watch Tasks` section / X) leave in PLAN.md only ← **Chosen**
2. A+Y) PLAN.md + also push 8 items to TodoWrite (duplication)
3. B+X) Standalone `plan/watch-tasks.md` file, no TodoWrite
4. B+Y) Standalone file + TodoWrite

**Decision:** Build `## Cross-Conv Watch Tasks` section in PLAN.md (between Block Sequence tables and BBB-RECORDING); do NOT migrate items into TodoWrite.

**Rationale:** PLAN.md keeps watch-tasks visible in the project's forward-looking surface. TodoWrite is for current-session work; watch-tasks span many sessions and the trigger conditions matter more than the per-session tracking. Standalone file (B) added a step (open a separate file) without solving a real problem.

**Consequences:** 3 sections collapse into 1; 4 sections delete entirely. PLAN.md gets a clean watch-task surface (8 items, each with explicit trigger condition). Items already in TodoWrite ([DTUNE-WATCH] #26, [SKILL-DISCOVERY-WATCH] #27) are NOT duplicated.

## 3. Conv-N drain sections with zero unique pending content get deleted, not archived
**Type:** Implementation
**Topics:** docs-infra, cc-workflow

**Trigger:** Conv 168 Items, Conv 169 Items, Conv 157 Timecard, Conv 200 Items all had 100% done bullets (or only pending items already in TodoWrite). The §Block-Completion-Rule from `fmt-update-plan.md` covers blocks but not chronological-drain sections.

**Options Considered:**
1. Archive each section in `plan/COMPLETED.md` as a per-Conv entry
2. Leave inline (status quo)
3. Delete entirely (git history preserves) ← **Chosen**

**Decision:** Sections with zero unique pending content get awk-deleted. The principle extends `fmt-update-plan.md`'s "PLAN.md contains only work that remains to be done" to Conv-N drain sections.

**Rationale:** Per-conv archival is what `docs/sessions/YYYY-MM/<timestamp> Extract.md` is for. plan/COMPLETED.md is for block-level closure entries, not chronological drains. Git log + grep for `[CODE]` recovers any granular detail if needed.

**Consequences:** 4 sections removed from PLAN.md (Conv 168, Conv 169, Conv 157 Timecard, Conv 200) without loss of recoverable information.
