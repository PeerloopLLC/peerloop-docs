## 1. Narrow [SEED-PW] to dev-only; surface prod-side as separate [PROD-PW] task
**Type:** Implementation
**Topics:** d1, deployment

**Trigger:** Mid-rotation, discovered the same bcrypt hash for "Password1" lives in `migrations/0002_seed_core.sql` (production-safe seed for `usr-admin` / admin@peerloop.com).

**Options Considered:**
1. Rotate prod + dev seeds to the same new password (Peerloop2) — uniform, but Peerloop2 becomes broadly known across dev README/scripts/test files, weakening prod admin secrecy
2. Rotate dev seed to Peerloop2; leave prod seed unchanged ← Chosen as narrow scope
3. Rotate dev seed to Peerloop2; rotate prod seed to a STRONG random password (and document in 1Password/etc.)

**Decision:** Apply Peerloop2 rotation only to migrations-dev + scripts + tests + e2e. Leave `migrations/0002_seed_core.sql` untouched. Surface prod-side as task #12 [PROD-PW] which captures: (a) prod password choice (Peerloop2 too well-known after dev rotation), (b) whether to also rotate EXISTING prod admin via `wrangler d1 execute` UPDATE (since seed re-runs hit PK collision and don't change existing rows).

**Rationale:** Task originally framed as "Rotate DEV seed-data passwords". Prod admin password has different blast radius and a separate decision space. Existing prod admin already has Password1 sitting in live D1 — fixing that is its own problem requiring an UPDATE migration, not a seed change. Better to make the prod-side decision explicit than bury it inside a "dev rotation" PR.

**Consequences:** Conv 167 closes the immediate Chrome-breach-warning symptom in dev. Prod admin still has Password1 hash in core seed AND in existing prod D1 — task #12 carries the open decision forward.

## 2. Targeted exclusion over blanket dot-strip in route-matrix normalizeDynamic
**Type:** Implementation
**Topics:** tooling, astro

**Trigger:** [RM-PARAM-BUG] root cause is `${Astro.url.search}` (query-string blob) being misclassified as a route param.

**Options Considered:**
1. Blanket: strip ALL `${...}` containing dots/parens/ternary chars — broad coverage but breaks legitimate property-access params (`${course.slug}`, `${otherParticipant.handle}`)
2. Targeted: drop only known query-string sources (`url.search`, `searchParams`) before existing keyword matching ← Chosen
3. Structural rewrite: parse SQL/template grammar and bind expressions to types — overkill for this scanner

**Decision:** Add two leading rules in `normalizeDynamic`:
```js
.replace(/\$\{[^}]*url\.search[^}]*\}/gi, '')
.replace(/\$\{[^}]*searchParams[^}]*\}/gi, '')
```

**Rationale:** The pathological pattern in this codebase is specifically `Astro.url.search` / `searchParams.toString()` appended to path templates. Other dot-containing expressions are legitimate route params. Targeted name-based exclusion is sound here; if a new appender pattern surfaces, add another rule (the script is small and accessible).

**Consequences:** Broken targets count 1 → 0. `/course/[slug]/[tab]` now correctly resolves as a real route across all 18 inbound references in `page-connections.md`. Future appenders need explicit rule additions (mild ongoing maintenance cost vs. fragility risk of grammar parsing).

## 3. Defensive null-check widening over root-cause investigation for CourseAvailabilityPreview
**Type:** Implementation
**Topics:** testing

**Trigger:** Conv 166 surfaced "1 unhandled error" in CourseTabs test pointing to `data.teachers.map(...)` crashing when fetch returned `{}` shape.

**Options Considered:**
1. Trace the mock fetch path and fix the upstream contract so `data.teachers` is always present
2. Widen the early-return guard to defensively handle malformed responses ← Chosen
3. Add an explicit empty-array fallback (`data.teachers ?? []`) and continue rendering with no teacher cards

**Decision:** Widened line 76 from `!data || data.teacherCount === 0` to `!data || !Array.isArray(data.teachers) || data.teachers.length === 0`.

**Rationale:** Type assertion at `await res.json()` is unchecked, so any fetch mock or future API contract drift can break this component asynchronously. A defensive guard is the durable fix even if the upstream contract is currently sound. The existing pattern was already defensive (checking `teacherCount === 0`); widening it costs zero readability. Tracing the upstream mock would solve this test but not future regressions.

**Consequences:** CourseTabs test 19/19 with 0 unhandled errors. The same guard now protects against any "data shaped but teachers absent" case, not just the original `{}` mock.
