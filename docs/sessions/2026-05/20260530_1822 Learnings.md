# Conv Learnings - 2026-05-30

## 1. Git is the authority over Edit "success" under degraded I/O
**Topics:** cc-workflow, testing

**Context:** The prior conv's [TERM-GARBLE] produced false-success Edit results on TopicPicker that never actually hit disk; this conv resumed after a deliberate `/quit` + terminal/CC restart.

**Learning:** When tool I/O is unreliable, an Edit tool's "updated successfully" string is not evidence — only `git diff` / the working tree is authoritative. Verify every Edit against `git diff` before trusting it, and check `git status` at resume to confirm prior phantom edits didn't land. (On resume, code repo was clean → confirmed the phantom Edits never landed.)

## 2. Worklist primitive *names* are hypotheses; verify against real APIs before swapping
**Topics:** cc-workflow, matt-provenance

**Context:** `/w-prim-candidates` flags interactive clusters deterministically but *names* the target primitive via agent reasoning (flagged in its own caveat, pending [PRIM-MATCH-INDEX]).

**Learning:** Always read the actual primitive `.tsx` API before a Tier-1 swap. 4 of the worklist's TopicPicker/Security guesses were semantic mismatches (accordion ≠ SelectableCard; text-link ≠ pill Button; no Badge/Checkbox/Textarea/Switch primitive exists). Don't force a mismatched primitive — that's the honest-orphan / Tier-2 boundary.

## 3. Pairing-consistency extends the half-swap rule up a level
**Topics:** matt-provenance

**Context:** SecuritySettings modal has Cancel+Delete as a visible `flex-1` button pair; Delete is danger-blocked (no Matt danger variant).

**Learning:** A visible set of sibling controls should convert all-or-nothing. Swapping only Cancel (pill) while Delete stays a rounded-rect looks worse than converting neither. The same logic scales to ProfileSettings' form: convert the 3 local wrappers together, not piecemeal.

---

## 4. The no-flush-exists rule is ambiguous under genuine I/O lag — wait, don't poke
**Topics:** cc-workflow

**Context:** Bursty/delayed tool-result delivery made "empty result" ambiguous (failed vs late); the no-flush-exists anti-pattern (Conv 218) was re-entered by re-issuing `git diff`/`echo flush` repeatedly.

**Learning:** Under suspected lag, the correct response is to wait one turn for the flush or do at most one retry-with-reason — never re-issue to "flush a buffer." Local-only command results arriving bursty + duplicated is a local-buffering signature, not server load (server load would be uniform latency, not bursty duplication).
