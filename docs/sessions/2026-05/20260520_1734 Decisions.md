# Conv Decisions - 2026-05-20

## 1. Add `scope=student|teacher|all` query param to `/api/courses/[id]/sessions` rather than flip precedence or accept dual-role limitation
**Type:** Implementation
**Topics:** d1, auth

**Trigger:** [CRT-2] introduced a regression for dual-role users (teacher-of-course + enrolled-in-course): server-side precedence picked teacher scope, so the standard student Sessions tab would now show teaching sessions. [CRT-3] would have hit the same endpoint from the new "My Teaching Sessions" tab, returning identical data on both tabs.

**Options Considered:**
1. Add `?scope=student|teacher|all` query param. Caller declares intent. Default-without-scope keeps the highest-privilege precedence for backwards compat. Tests stay clean (CRT-2 was young), endpoint stays singular. ← Chosen
2. Use existing `/api/teaching/courses/[courseId]` endpoint for the teacher tab, flip CRT-2 precedence so enrolled student wins over teacher. Two endpoints serving overlapping concerns; heavier payload than the teacher tab needs.
3. Leave precedence as-is, build teacher tab on same endpoint, accept dual-role users see identical data on both tabs as a known limitation, revisit later.

**Decision:** Option 1 — add an explicit `scope` query param now.

**Rationale:** Most durable; fixes the regression at the API boundary; single endpoint for all scopes; tests added at the right time (before call sites multiplied). Once the dual-role contract is locked at the API, [CRT-4] (creator/admin/moderator) and [CRT-5] (resources/learn/etc tabs) can rely on it without inheriting precedence ambiguities.

**Consequences:** API contract now has 3 explicit scope values + the default. `/api/courses/[id]/sessions` precedence is correctly described as "default behavior", not "the only behavior". 10 new tests added covering each scope branch + dual-role disambiguation + invalid-scope rejection. New behavior: `TeacherSessionsTabContent` passes `scope=teacher`; student Sessions tab will need to start passing `scope=student` explicitly in [CRT-4] or [CRT-5] when dual-role users surface in the wild. Existing student-tab callers still work because `SessionsTabContent` doesn't pass scope (default behavior preserved).

## 2. Pass role flags as primitives and build extra-tab content inside `CourseTabs` rather than constructing React nodes in `.astro` frontmatter
**Type:** Implementation
**Topics:** astro, react

**Trigger:** First wiring of the teacher tab used `extraTabs.push({ content: createElement(TeacherSessionsTabContent, { courseId, isActive: true }) })` in `sessions.astro`. Browser threw "Objects are not valid as a React child" at hydration; the React element instance was JSON-serialized into a plain object by Astro's `client:load` boundary.

**Options Considered:**
1. Component-descriptor pattern — `extraTabs` holds primitives (kind + props), `CourseTabs` has a switch to render the right component. Most flexible, supports caller-defined extra tabs from non-Astro callers.
2. Move tab construction into `CourseTabs.tsx` entirely — `.astro` pages pass primitive role flags (`isTeacherOfCourse`, etc.), `CourseTabs` imports the role components directly and builds the extraTabs array internally. Simplest, no new pattern, no type union. ← Chosen
3. Use Astro `<slot/>` composition — restructure `CourseTabs` to expose named slots for each role group. Cleanest separation but heavy refactor; changes CourseTabs's public shape and forces every consumer to migrate.

**Decision:** Option 2 — primitive role-flag props (`isTeacherOfCourse`, `isCreatorOfCourse`, `isAdmin`, `isModeratorOfCommunity` added to `CourseTabsProps`), `CourseTabs` constructs extra tabs internally. The original `extraTabs?` prop stays in the interface for future use but is no longer used from `.astro`.

**Rationale:** Lowest churn given only a small known set of role tabs (4) is needed for the CRT block. No type union to maintain. CourseTabs is the orchestrator anyway — owning the role→content mapping there is coherent. If a non-Astro caller ever needs to inject arbitrary extra tabs, the `extraTabs` prop still works because content stays inside React.

**Consequences:** Loader role flags from `fetchCourseTabData` (CRT-1) feed straight into `CourseTabs` as props. [CRT-4]/[CRT-5] will follow the same pattern: pass `isCreatorOfCourse` / `isAdmin` / `isModeratorOfCommunity` flags, `CourseTabs` builds CREATOR / ADMIN / MODERATOR extra-tab groups. The "build the tab content in Astro" pattern is now a known anti-pattern for this codebase.
