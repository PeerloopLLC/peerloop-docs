# Conv Learnings - 2026-05-23

## 1. Empirical MCP probe trumps inferred architectural documentation
**Topics:** matt, figma, design-system

**Context:** Conv 178 documented Matt's Button as a "3-orthogonal-dimension" component (Color × State × Size). Conv 183 [MATT-EXEC-CMP-BTN] probe of node `40:482` showed Property 1 is actually a 5-value enum conflating State AND Size (Default / Hover / Large / Small / Small Hover). The "3-orthogonal" framing was a logical projection of the Color collection's multi-mode structure onto State and Size — but Matt's actual Figma encodes them as a single property.

**Learning:** The [EMP] rule in `feedback_external_source_of_truth_first.md` is recurring — Conv 180 had 4 reversals, Conv 183 added 2 more (Button 3-orthogonal claim + the "5 size cells side-by-side" framing). The cost of one MCP probe is dramatically less than the cost of building against a misframed spec. **Probe THEN write the spec, never the inverse.**

**Pattern:** When transcribing Figma into docs, every claim in the doc should be tied to a specific probed node ID. If the doc says "N variants" or "X-orthogonal", the corresponding probe output should be a footnote.

## 2. Matt's "Main" pill is route-driven, not click-to-expand
**Topics:** matt, navigation, ux

**Context:** Inspecting the Main Nav usage example (513:15755) at first glance suggested the white pill might be a click-to-expand drawer. But the assembled instance shows only one Nav Item in Main state (with the active child inline), and the abstract Property 1=Main variant (150:8585) is sized 220×132 — fits exactly "parent row + divider + Sub Nav slot with N children." No toggle indicator (▼/▶) anywhere in Matt's design.

**Learning:** The Main pill represents the **currently-active section showing its currently-active child**. It's not user-controlled state — it's a route-derived visual. Naming "Main" (not "Expanded") supports this read. This is materially different from typical web sidebar patterns; if Matt had wanted click-to-expand, he'd have drawn ▼/▶ chevrons.

**Pattern:** Active-detection rule for sidebar nav: an item enters `Main` state if currentPath matches it OR any child AND it has children. Render the active path as one contiguous visual unit.

## 3. ASCII mockups via AskUserQuestion `preview` field as a decision aid
**Topics:** cc-workflow, ux, user-collaboration

**Context:** User pushed back on three architecture questions framed in text: "It is not possible for me to decide this with only text descriptions." Pure prose descriptions of UX behaviors and code-architecture trade-offs are hard to compare side-by-side.

**Learning:** `AskUserQuestion` with `preview` fields renders side-by-side monospace blocks — ideal for ASCII UI mockups, route-shift-shows-different-state comparisons, and code-snippet diffs. ASCII is genuinely better than prose for visual-decision questions even when Figma is the canonical source, because Figma can't show "what the React component will produce on screen" side-by-side with "what an alternative architecture would produce on screen." ASCII bridges this.

**Pattern:** When proposing architectural / UX decisions: (1) render Matt's Figma reference in ASCII as the baseline, (2) render each candidate option as ASCII showing how it differs, (3) use AskUserQuestion previews so the user picks visually rather than parsing text descriptions.

## 4. Matt's Hover variant is Primary-Variable-Mode-specific (partial coverage gap)
**Topics:** matt, design-system, figma

**Context:** The Button Hover variant (513:14820) uses a hardcoded gradient on `rgb(7, 119, 182)` (Primary blue) + hardcoded `#066ba4` border. NO `var(--background)` / `var(--border)` indirection. Default + Large + Small variants ARE Variable-Mode-aware (use `var(...)`). Switching Variable Mode (e.g., Course) leaves Hover rendering as darkened Primary-blue, NOT darkened Course-green.

**Learning:** Matt's design system has **partial coverage** for state-by-color cells. The cascade structure that works for Default doesn't propagate to Hover. Per `feedback_tokenize_only_matt_variables.md` ("hardcode what Matt has hardcoded"), our React Button mirrors this literally — non-Primary + Hover renders Primary-darkened anyway, with a code comment flagging this as a Phase 6 [MATT-EXEC-EXT] extrapolation candidate.

**Pattern:** When porting a multi-state × multi-color component from Figma, check each state for Variable-Mode-awareness independently. The mere presence of color tokens in the collection doesn't guarantee every state uses them.

## 5. `get_design_context` may not always require pre-selection
**Topics:** matt, figma, mcp

**Context:** Conv 180's `reference_figma_mcp_behavior.md` claimed `get_design_context` is selection-REQUIRED (user must click the layer in Figma desktop first). Conv 183 ran 8 successful `get_design_context` calls across Button + MainNav variants with no specific layer selection effort on the user's side. Each call returned full design context (CSS, typography, variables, asset URLs).

**Learning:** Either (a) the selection-REQUIRED claim was too strong / context-specific, (b) server behavior changed, or (c) "selection" is satisfied by the file being open + recently navigated. Logged via the watch task [MCP-SEL-MISFIRE]; if Conv 184+ also runs without pre-selection misfires, the memory file should be relaxed.

**Pattern:** When external-tool empirical memories conflict with current observations, surface the conflict explicitly via a watch task rather than silently retconning the memory. Multiple data points across multiple convs are required before updating the canonical claim.

## 6. ASCII Figma reference is reusable across decision rounds
**Topics:** matt, cc-workflow

**Context:** After rendering Matt's Main Nav usage example in ASCII as the "baseline," all subsequent decision questions could reference it implicitly. The user could see how each candidate architecture would diverge from the baseline at a glance.

**Learning:** Investing 60 seconds to render Matt's Figma source in ASCII pays off over the whole decision sequence. The same ASCII can anchor D1/D2/D3 + future variant questions. **Save the ASCII rendering in scratch alongside the screenshot** so future convs can re-use it.

**Pattern:** When starting on a new Matt primitive, render the Figma reference both as a PNG screenshot AND as an ASCII sketch in scratch. The PNG is for visual fidelity; the ASCII is for textual decision-aid.
