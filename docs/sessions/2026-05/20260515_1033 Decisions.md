## 3. Follow established admin pagination pattern with 2-call total derivation
**Type:** Implementation
**Topics:** video, bbb, astro

**Trigger:** Implementing `[BR-PAGE]` — admin /admin/recordings page should show 20-at-a-time with prev/next.

**Options Considered:**
1. Server-paginate, no total (Showing N-M + hasMore disables Next)
2. Server-paginate, 2 BBB calls per request to derive total ← Chosen
3. Client-paginate (fetch limit=100 once, slice in component)

**Decision:** Use `parsePagination` / `paginationOffset` / `createPaginatedResult` helpers from `src/lib/db/index.ts` (the existing admin pagination convention). Endpoint makes two parallel BBB calls: one for the requested page, one with `limit=100` to derive total. Response includes standard `{items, total, page, limit, totalPages, hasMore}` shape.

**Rationale:** Pattern alignment with 10+ other admin endpoints. Manual admin action is infrequent so the extra ~14KB call is acceptable. Preserves accurate total count (vs option 1 regression).

**Consequences:** Endpoint at `/api/admin/bbb/recordings?page=N&limit=N` returns canonical paginated shape + `fetched_at`. React component uses shared `AdminPagination` component (already provides chevrons, page-number list, items-per-page selector).

## 4. `listAllRecordings` as BBB-specific method (not in shared VideoProvider interface)
**Type:** Implementation
**Topics:** video, bbb

**Trigger:** Pagination capability needed for admin diagnostic. Question: extend the `VideoProvider` interface or add a BBB-only method.

**Options Considered:**
1. Widen `VideoProvider.getRecordings(roomId?: string)` signature to accept pagination opts — forces PlugNmeet and other providers to support BBB-specific pagination semantics
2. Add `getRecordings(opts?: {roomId, limit, offset})` overload — same interface widening
3. Add `listAllRecordings({limit, offset}): Promise<{recordings, total}>` only on BBB (not in interface) ← Chosen
4. Inline the 2-call logic in the endpoint, bypass provider abstraction

**Decision:** Add `listAllRecordings({limit, offset})` as a BBB-specific public method, not in the shared `VideoProvider` interface. Admin endpoint already creates a BBB-specific provider (`createBBBProvider`) so it can call the method directly. Interface unchanged.

**Rationale:** Admin diagnostic is inherently BBB-specific (account-wide listing). PlugNmeet and other video providers may have entirely different pagination semantics (or none). Forcing the interface to support BBB's `{limit, offset, total}` shape would overreach.

**Consequences:** Per-room `getRecordings(roomId?)` stays in interface unchanged. New BBB-only method coexists. Shared mapping logic extracted to private static `BBBProvider.extractRecordings(result)` helper used by both methods.
