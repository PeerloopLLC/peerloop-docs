### UCM — useCurrentUser memoization sanity check

**User:** "UCM is next"

**Claude:** Read `src/lib/current-user.ts` to verify whether `useCurrentUser()` returns a stable reference when `user.id` is unchanged.

**Verdict:** It does NOT. `setCurrentUser` (line 902) calls `notifyListeners(user)` unconditionally — every `refreshCurrentUser()` constructs `new CurrentUser(data)` and notifies listeners with the new instance. `AppNavbar.tsx:133` calls `refreshCurrentUser()` on every window focus → tab-back triggers a render cascade across every component using `[currentUser]` deps. Not a correctness bug, but real wasted-work bug — and it specifically degrades the LE-P3/4/5 useMemo/useCallback extractions just landed in this conv.

**Fix:** Discovered existing `dataVersion: number` field on CurrentUser (already used by version polling). Proposed dedup guard in `setCurrentUser`: if `prev.id === next.id && prev.dataVersion === next.dataVersion`, drop the new instance and skip notify. User approved.

Applied fix to `setCurrentUser` with explanatory comment block. Initial test run hit 3 failures in `current-user-listeners.test.ts` and `current-user-cache.test.ts` — fixtures used `validMeFullResponse()` for both initial and updated data without bumping `dataVersion`. Updated 3 tests to pass `{ dataVersion: 1 }` override on the "updated" call (matches real server behavior — server bumps version on data change). Added 1 new regression test "does NOT notify when refresh returns same id + dataVersion (dedup)" to lock in the behavior.

Final verification: 6415/6415 tests pass across 370 test files; tsc 0; astro 0/0/0; lint unchanged (4 unrelated `no-explicit-any`).
