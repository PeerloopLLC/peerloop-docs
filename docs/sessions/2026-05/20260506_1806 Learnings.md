# Conv Learnings - 2026-05-06

## 1. Tarball Desktop Convention Is Folder-Wrapped, Not Loose
**Topics:** cc-workflow, dual-repo

**Context:** [MPS] task description specified the tarball at `~/Desktop/from-M4-memory-conv152-after-MPP.tar.gz` (loose path). Actual location was `~/Desktop/from M4 - memory/from-M4-memory-conv152-after-MPP.tar.gz` — inside a `from M4 - memory/` folder. Initial `ls` of the loose path failed; had to scan Desktop to find it.

**Learning:** Future cross-machine transfer instructions should encode the folder-wrapping convention if it exists, or the receiving skill should accept a folder-wrapped tarball as a fallback. The filename was unique enough that a recursive-glob lookup (e.g., `find ~/Desktop -name 'from-M4-memory-*.tar.gz'`) would have been deterministic even without knowing the folder.

## 2. Bidirectional `diff -rq` Satisfies [BIV] in One Command
**Topics:** cc-workflow, testing

**Context:** [BIV] (Bilateral Verification) was codified as an open constraint Conv 152 because L2 forward-only checks had a bash-loop bug producing false-positive "all 50 missing" output, only caught by an ad-hoc reverse orphan check. During [MPS] verification, L3 used `diff -rq <M4-extract> <M4Pro>` which is *inherently* bidirectional — it reports `Only in dir1: X`, `Only in dir2: Y`, AND content diffs. A single L3 check covers what [BIV] flagged as needing two passes.

**Learning:** Where a verification can use `diff -rq` against a known-good source-of-truth tree, it implicitly satisfies the bilateral-check requirement. [BIV]'s actionability narrows to "scripted forward-only checks that walk one side's file list" — those still need an explicit reverse pass. Worth tightening [BIV]'s scope when next picked up.

**Pattern:**
```bash
# Tree-vs-tree byte equivalence + orphan detection in one command
diff -rq /tmp/source-of-truth-extract /Users/<user>/.claude/projects/.../memory/
# Reports: differing files, files only in source, files only in destination
```
