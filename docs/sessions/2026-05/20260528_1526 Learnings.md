# Conv Learnings - 2026-05-28

## 1. Reading huge PLAN.md files: chunk by line range, use sed to extract sections
**Topics:** cc-workflow, docs-infra

**Context:** PLAN.md at 2769 lines exceeded Read tool's 25k token cap. First Read with offset+limit failed with "exceeds maximum allowed tokens". The grep tool output also too-large at one point (44KB returned for the ACTIVE table).

**Learning:** When a file exceeds Read's token cap, use targeted Bash sed extraction (`sed -n 'START,ENDp' file > /tmp/excerpt.txt`) or chunked Reads with offset+limit values matched to expected section sizes. For bulk deletions across many ranges in one file, batch them into a single `sed -i -e '<range1>d' -e '<range2>d' ...` invocation in **reverse line order** so earlier line numbers stay valid as later sections are removed.

**Pattern:**
```bash
sed -i.bak -e '972,1020d' -e '953,971d' -e '933,952d' ... -e '139,144d' PLAN.md
# Reverse-ordered ranges; .bak preserves a recoverable original.
```

## 2. `awk` is the right tool for whole-row replacement in markdown tables
**Topics:** docs-infra

**Context:** The MATT-DESIGN-PUSH ACTIVE table row was a single 12KB line containing a massive narrative cell. Edit tool's exact-string match would have required the full 12KB string as `old_string`.

**Learning:** For known-line-number whole-line replacements, `awk 'NR==N {sub(/^| ROWPREFIX .*/, "REPLACEMENT")} {print}' file > /tmp/edited && mv /tmp/edited file` is cleaner than Edit. Bypasses the exact-match-key constraint and works on arbitrarily long lines.

## 3. Front-load design Q&A before multi-conv-scope work; resolve all unknowns up front when runway is abundant
**Topics:** cc-workflow

**Context:** User explicitly said "B then ask about doing the rest of A. We just started this conv and have lots of runway as long as the back-and-forth between us can be dealt with up front." This is the runway version of the multi-conv-scope carve-out.

**Learning:** When user signals "lots of runway, front-load the questions," present ALL design decisions in a single labeled Q1-Q4 block before executing. Pre-decide whatever doesn't need a vote (naming conventions, file format, etc.) and surface those decisions in a "Things I'm planning to decide without asking" block so the user can override if needed. This converts what would be 4+ back-and-forth turns into 1, which is what the user actually wants.

**Pattern:** `Q1 — A/B → Q2 — A/B → Q3 — A/B → Q4 — A/B → "Things I'm planning to decide without asking: ..." → 👉👉👉 **Q1–Q4: A or B each?**`

## 4. PLAN.md ACTIVE-table cells can be long, but migrated-block rows want a 3-line summary
**Topics:** docs-infra

**Context:** Existing PLAN.md table cells (MATT-DESIGN-PUSH at 12KB, NAV-RETROFIT at similar scale, ROUTE-MIGRATION) are massive single-cell narratives concatenated across many convs. They grew because every conv appended a "Conv X: ..." paragraph rather than rewriting to current state.

**Learning:** For migrated blocks, the ACTIVE row collapses to: "current focus + last conv landmark + next subtask + → [plan/.../README.md]." For non-migrated blocks, the long-cell pattern is acceptable until that block migrates. The 12KB pattern is unsustainable as a general rule but doesn't need cleanup until each block's migration moment.
