## 1. PG2 course tabs structure — per-tab components, shell duplicated
**Type:** Implementation
**Topics:** matt-design-system, astro

**Trigger:** Building 5 remaining course-tab bodies after the `[...tab]` routing skeleton landed.

**Options Considered:**
1. Per-tab components in `src/components/matt/course/`, shell duplicated across index.astro + [...tab].astro, index.astro untouched ← Chosen
2. Shared CourseTabShell extracted + About body extracted (refactors Conv 187 index.astro)

**Decision:** Option A — per-tab `.astro` body components rendered by a `tab ===` switch in `[...tab].astro`; index.astro (About) left as-is.

**Rationale:** Lowest risk to just-validated Conv 187 work; mirrors the legacy discover precedent (shell duplicated in 2 route files).

**Consequences:** ResourcesTab + TeachersTab built this way; shell duplication between index.astro and [...tab].astro accepted (noted for future dedup if PG2 churns tabs).

## 2. Modules data model — Session↔Module is 1:1; Matt's "Module" = Sub-Module
**Type:** Strategic
**Topics:** matt-design-system, d1

**Trigger:** Matt's Modules frame groups "N Modules" under "1-on-1 Sessions", conflicting with the 1:1 session-module design ([MOD-SCHEMA]).

**Options Considered:**
1. Add a session→many-modules grouping to the schema
2. Derive grouping from existing curriculum fields
3. Reinterpret Matt's nested "Module" as Sub-Module; session = Module (1:1) ← Chosen (user)

**Decision:** Every session has exactly one Module; Matt's/creators' nested "Modules" are Sub-Modules (a term misuse). No session→many-modules data model.

**Rationale:** User clarified the real product model; the subdivision a creator gave Matt still happens within the single session.

**Consequences:** Saved to memory `project_module_submodule_model.md`; [MOD-SCHEMA] resolved; [MODTAB] to build each card as one session/module with an inner "N Sub-Modules" count.

## 3. Role-color cue on photo avatars — opt-in corner dot
**Type:** Implementation
**Topics:** matt-design-system

**Trigger:** Creator-purple / teacher-blue distinction is lost when the avatar is a photo (initials bg not shown). User asked for a way to surface it.

**Options Considered:**
1. Colored halo ring
2. Corner role dot ← Chosen (user)
3. Colored role label only
4. Halo + colored label

**Decision:** Opt-in `roleDot` prop on UserIcon — a small corner dot in `--Entity-Primary`. Default off (existing avatars unchanged). Product extrapolation beyond Matt's drawn design.

**Rationale:** Reads the existing role cascade so it "just works"; opt-in avoids recoloring every avatar app-wide.

**Consequences:** Enabled in TeachersTab; surfaced + fixed the underlying [ENTITY-CASCADE-BUG].
