### Web research on the two Astro-Cloudflare landmines

**User:** "let's research these in more depth via the web and determine if upgrading any of our npm packageds would help"

**Claude:** Captured current versions (astro ^6.1.5, @astrojs/cloudflare ^13.1.8, @astrojs/react ^5.0.3, react/react-dom ^19.2.3), latest published (6.3.7, 13.5.4, 5.0.5), then ran 3 WebSearches + 2 WebFetches.

**Findings:**

- **Issue 1 (React invalid hook call):** Astro issue [#16529](https://github.com/withastro/astro/issues/16529) is open on Astro 6.2.0 + @astrojs/cloudflare 13.3.0 + @astrojs/react 5.0.4 — i.e., reproduced on versions NEWER than ours. Cloudflare workers-sdk [#11825](https://github.com/cloudflare/workers-sdk/issues/11825) is closed-with-workaround (opened Jan 2026). Workaround: `resolve.dedupe: ["react", "react-dom"]` + `ssr.noExternal: ["react", "react-dom"]` in Vite config. PLAN says Conv 122 tried this and it didn't work — but @astrojs/cloudflare 13.5.4 added "Forwards user-provided `optimizeDeps` settings to SSR/prerender environments" which may be the previously-missing piece. Worth retrying after the upgrade.
- **Issue 2 (JSX in Astro expression):** [Astro template-expressions docs](https://docs.astro.build/en/reference/astro-syntax/) explicitly require kebab-case HTML attributes (`class`, not `className`) but the expression-block parser only accepts component references, not arbitrary HTML elements. By design. Roadmap discussion [#716](https://github.com/withastro/roadmap/discussions/716) tracks broader JSX-syntax support but nothing has shipped. No upgrade fixes this — the `_Demo.tsx` extraction pattern IS the answer.
- **@astrojs/cloudflare 13.x changelog entries between 13.1.8 and 13.5.4** — none specifically mention React/hooks, but 13.5.2 fixed "cascade of reloads crashing first visit during Cloudflare SSR" and 13.5.4 added the optimizeDeps-forwarding which is the likely unblocker for the canonical workaround.
