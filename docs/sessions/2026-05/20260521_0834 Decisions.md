## 2. Per-route `export const noNav = true;` annotation chosen over scanner-wide whitelist
**Type:** Implementation
**Topics:** plato, code-review

**Trigger:** [RAM-NO-NAV] question about how to suppress 20 legitimate "no discovered nav path" тЪая╕П warnings from `route-api-map.mjs`.

**Options Considered:**
1. Footer-scan pass тАФ auto-discover footer-reachable routes (durable but doesn't cover role-tabs/404)
2. Explicit `noNavByDesign: string[]` whitelist in scanner тАФ quieter, low scope, central maintenance overhead
3. Per-route annotation in .astro frontmatter тЖР Chosen тАФ declarative locality
4. Leave alone тАФ accept benign warnings

**Decision:** Add `parseNoNav(content)` to scanner reading `export const noNav = true;` from .astro frontmatter; emit `тД╣я╕П no-nav by design` instead of `тЪая╕П no discovered path` for annotated routes. Applied to `/course/[slug]/[tab]`; broader sweep tracked as `[RAM-NONAV-SWEEP]`.

**Rationale:** Per-route annotation forces each future no-nav route to declare itself intentionally at creation time тАФ preserves the warning's signal value while making intentional cases self-documenting. Central whitelists drift as the codebase evolves; per-route declarations don't.

**Consequences:** New convention codified in `scripts/route-api-map.mjs:90-99` (`parseNoNav` helper). Existing 19 other legitimate no-nav routes still emit тЪая╕П тАФ intentional, per `[RAM-NONAV-SWEEP]` follow-up. Scanner output continues to distinguish `тЪая╕П` (real concern, route truly unreachable) from `тД╣я╕П` (designed unreachable from main nav).
