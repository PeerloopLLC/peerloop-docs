# Conv Learnings - 2026-05-29

## 1. Page-provenance markers do not propagate to embedded components
**Topics:** matt, astro, cc-workflow

**Context:** `/profile` is marked `@matt-inspired`, yet its 5 embedded settings islands are legacy UI (105 legacy-token usages, 0 Matt tokens).

**Learning:** A page-level provenance marker says nothing about the vetted-ness of the components rendered inside it. Conformity must be tracked at the component/DOM level, not inferred from the page marker. This is the structural gap [PRIM-REGISTRY] closes.

---

## 2. The pre-flip worktree is the authoritative record of legacy interaction behaviour
**Topics:** matt, cc-workflow

**Context:** The user's recollection of the profile chip (redirect to `@me`, special behaviour) was partly inaccurate; reading commit `608346a2` settled it definitively.

**Learning:** When a design decision hinges on "what did the legacy app actually do," read the pre-flip worktree source rather than reasoning from memory. It revealed the chip was a popover (not a link), dark-mode was login-only, and `/profile` (private) ≠ `/@handle` (public) — each of which redirected the design.

---

## 3. A "registry as single source of truth" can compete with an existing load-bearing marker
**Topics:** matt, cc-workflow

**Context:** User chose dedicated registry files (approach B). But the 41 `@matt-source` in-file markers are already the load-bearing, verifiable authorship SoT (§4 of matt-provenance.md).

**Learning:** For the Matt-sourced half, the registry file is best *generated* from the markers (a derived projection) rather than hand-maintained as a second SoT — avoiding drift and preserving the markers' re-probeable / git-audit virtues. Only the Matt-inspired half is a true hand-maintained SoT. Flagged to the user as a refinement within approach B.
