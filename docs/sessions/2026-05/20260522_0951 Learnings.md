# Conv Learnings - 2026-05-22

## 1. Matt's Multi-mode Figma Variables encode context-aware token resolution
**Topics:** matt-design, design-tokens, css-variables

**Context:** Inspected 5 Figma Variable collections. 3 of 5 use multi-mode: Button (3 vars × 6 modes), Entity (2 vars × 4 modes), Icon Size (1 var × 2 modes). Flat collections: Color Primitives (15), Color Semantics (14).

**Learning:** Multi-mode collections are Matt's mechanism for **context / variant / size switching**. Translates to CSS variable cascade scoped by class (Entity), discrete named variables (Icon Size), or component prop unions (Button). The pattern is system-wide architecture, not just a Button quirk.

**Pattern:**
```css
/* Multi-mode via parent-class CSS cascade (Entity pattern) */
:root { --Entity-Primary: var(--Text-Default); }
.entity-course { --Entity-Primary: var(--Course-Primary); }
.entity-creator { --Entity-Primary: var(--Creator-Primary); }
/* Children consume var(--Entity-Primary) blindly */
```

## 2. 2-layer / 3-layer cascade chains are the design system's resilience
**Topics:** matt-design, design-tokens, css-variables

**Context:** Color Semantics has 2-layer indirection: `Student/Primary → Primary/Default → americana-blue` (semantic references another semantic, which references the primitive). Entity layer adds a 3rd hop.

**Learning:** The cascade IS the value of the semantic layer; flattening it destroys design coherence. Changing the brand-primary value at `Primary/Default` propagates automatically to all downstream semantics without manual updates. **Implementation rule: `--Student-Primary: var(--Primary-Default);` NOT `var(--americana-blue);`**

## 3. Header Bar and Sub Nav are slots, not single components — content/mode differs by breakpoint
**Topics:** matt-design, layout, responsive-design

**Context:** Observed across Desktop / Tablet Portrait / Mobile page-structure frames.

**Learning:** Header Bar carries DIFFERENT CONTENT per breakpoint (Desktop = breadcrumb; Tablet Portrait = brand strip; Mobile = brand + Messages + Notifications icons). Sub Nav has DIFFERENT RENDERING per breakpoint (Desktop = vertical-left tab strip; Mobile = slide-in drawer from left). MattLayout.astro needs multi-component breakpoint-aware composition, NOT just CSS display:none tricks.

## 4. Matt composes pages from reusable Figma components (we mirror)
**Topics:** matt-design, component-architecture, react, astro

**Context:** Button collection's 6-mode multi-variant + `components/*.svg` inventory confirm Matt builds pages via instances + per-instance overrides.

**Learning:** Each Matt component maps to a parameterized React or Astro component. Use Astro for static structural shells; React for interactive UI. Slots/children for content composition. Variant props (literal union types) for multi-mode components. No one-off pages — pages are thin composition layers over primitive components.

## 5. Matt didn't formalize spacing tokens — extrapolated full scale wins
**Topics:** matt-design, design-tokens, tailwind

**Context:** Direct inspection of Figma Local Variables panel revealed 5 collections, none for spacing. Only Matt-confirmed values via observation: 16, 20, 24, 48 (all on standard 4-base scale).

**Learning:** When extrapolating to ~84 unseen pages, "ship only Matt's 4 values" forces ad-hoc value drift. Adopting full Tailwind-aligned 4-base scale (4/8/12/16/20/24/32/40/48/64) with Matt's confirmed values flagged inside is the durable choice. Token NAMES are stable interface; VALUES are tuneable.

## 6. Existing React icon default doesn't match Matt's primary size
**Topics:** matt-design, react, icons

**Context:** `({ className = 'h-5 w-5' }: IconProps)` resolves to 20px (Tailwind `h-5` = 1.25rem). Matt's Icon Size collection has Medium=24, Small=20.

**Learning:** Existing app defaults to Small icons. When re-skinning for `/matt/*`, explicit choice needed: Medium (24px) for prominent contexts (nav, action buttons), Small (20px) for inline. One-line fix at the icon default but system-wide consequence.

## 7. Seamless-edge pattern reveals two button archetypes
**Topics:** matt-design, components, button

**Context:** Button matrix Border row: themed modes (Course / Student / Creator / Default) use same value for Background AND Border → invisible border. Brand modes (Primary / Outlined) use `Primary/Default` → visible blue border.

**Learning:** Matt's Button is TWO archetypes hiding in one collection: definitive CTAs (filled or outlined blue, brand actions) vs soft pills (context-tinted, contextual actions). Re-skin must preserve this distinction, not flatten to "just pick a color."

## 8. Matt's design doesn't account for multi-role users
**Topics:** matt-design, roles, peerloop-extension

**Context:** Matt's brief was deliberately single-role. 31 happy-path screens show separate Teacher and Student pages. Initial Conv 171 attribution of "Control Bar = role-perspective switcher" was wrong.

**Learning:** Multi-role users (Creator + Teacher + Moderator on the same course) need their own Peerloop-extension component: **Role Tab Bar**. NOT in Matt's design. Extrapolate from his tokens + existing `ExploreTabBar` (Conv 042-044). The mis-attribution surfaced via direct evidence: Matt's Control Bar IS the bottom-nav primary-nav primitive on tablet/mobile, not a role switcher.

## 9. Matt's primitives have inconsistent naming conventions
**Topics:** matt-design, naming-conventions, css-variables

**Context:** Color Primitives uses three naming styles in one collection: kebab-case (`pastel-blue`), Title Case with space (`Ashy Blue`), lowercase with space (`alert light`).

**Learning:** Honor Matt's literal names where possible (per the doc's `--Text-Default` convention), but spaces in names require CSS-variable normalization. Decision needed during [MATT-PRE-PLAN]: normalize to kebab-case everywhere with a mapping note, OR preserve Matt's literal names where CSS permits. Color SEMANTICS use consistent Title Case + slash separator — no inconsistency there.
