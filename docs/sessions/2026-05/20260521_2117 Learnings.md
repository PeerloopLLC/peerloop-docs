# Conv Learnings - 2026-05-21

## 1. Plugin-generated SVGs are visually authoritative, not machine-parseable
**Topics:** matt-design, figma-export

**Context:** Inspected `tokens/color-guide/Generated with plugin_ Color Variable Style-Guide Generator.svg` expecting to extract structured token data; the SVG renders all text (including color names + hex codes) as `<path>` geometry rather than `<text>` elements.

**Learning:** Color/typography style-guide SVGs exported from Figma plugins are essentially flattened vector art. They're useful for visual reference (paste into Read tool to view as image), but NOT for programmatic token extraction. Use the overview PNGs and Figma Dev Mode CSS inspector for actual values; reserve the SVGs for renderable visual artifacts only.

**Pattern:** When extracting design tokens from a Figma export, look first for: (1) Figma Dev Mode access (best), (2) overview PNGs (renderable in Read tool, gives full visual context), (3) named primitive SVGs (read filename, NOT internal text), (4) Figma Variables panel if accessible.

## 2. Matt's design assumes "no global top header" — branding lives in the sidebar entirely
**Topics:** matt-design, layout-architecture

**Context:** Iterated through Course, Home, Teacher screenshots looking for a top header bar. None exists. PeerLoop branding + collapse toggle live in sidebar's top-left; user profile chip in sidebar's bottom-left; primary nav in sidebar middle; Earnings/Notifications also in sidebar.

**Learning:** Two-panel-only desktop architecture (Sidebar + Main Panel, 16px gutter). This is less common than the header+sidebar combination most web apps use. `MattLayout.astro` must NOT define a top-of-viewport header slot. What Matt calls "Header Bar" is a thin (~40–48px) per-page breadcrumb / back-nav strip at the TOP OF MAIN PANEL, not at the top of the viewport.

**Pattern:** When a designer's layout primitive named "X Bar" doesn't appear in their actual page mockups, look for it as a small annotation diagram (Matt's `Header Bar.svg` = 125×132 px) — these are spec labels, not page mockups.

## 3. Entity-color-coded headers are a load-bearing identity-signaling pattern, not decoration
**Topics:** matt-design, color-system, component-architecture

**Context:** Course pages render a dark hero header; Teacher/Creator pages render a purple-blue card (matches `creator-primary = #5840F4` exactly); Student pages (presumed) render in `americana-blue = #0777B6`.

**Learning:** Matt's semantic color tokens (Course, Creator, Student, Text) aren't just "themes" — they encode entity identity. Every entity-typed page header MUST flow its primary color from the matching semantic token. This is a contract for the component, not a styling choice. Implementations must wire entity → semantic color → header treatment.

**Pattern:** Token contract: `--{Entity}-Primary` (color) + `--{Entity}-Background` (color) + entity-header component takes the entity type as a prop and derives its color treatment from the matching tokens.

## 4. The Control Bar Matt designed already exists in code as `ExploreTabBar` (Conv 042–044)
**Topics:** matt-design, role-system, existing-architecture

**Context:** User explained that what Matt calls "Control Bar" = horizontal tab bar above main content, with tabs representing **roles** (Teacher/Student/Visitor/Creator/Admin/Moderator), shown only when current user has >1 role for the entity. Docs review found that `src/components/discover/ExploreTabBar` (built Conv 042-044) is exactly this pattern, plus `RoleBadge` (3-size pill with role-colored treatment), `ExploreCourseTabs` (injects role-specific tabs), `CurrentUser` singleton for O(1) role detection, and dynamic `src/pages/course/[slug]/[tab].astro` catch-all serving role-tab URLs (`teaching-sessions`, `creator-sessions`, etc., Conv 165-166).

**Learning:** /matt/* re-skin scope is **visual re-styling of existing role-aware infrastructure**, not architecture. The Control Bar component is a visual re-skin of `ExploreTabBar`. The Header Bar component is a visual re-skin of `Breadcrumbs.astro` (which already supports `?via=` query-param navigation context). We don't invent role-encoding — the existing app handles it.

**Pattern:** Before designing any "new" component from Matt's primitives, grep the existing codebase for the behavioral pattern. Matt's design is a visual *language*, not new system architecture.

## 5. The happy path is the calibration set; the rest of the app is the extrapolation test
**Topics:** matt-design, design-system, strategy

**Context:** Matt was hired ~5 months into the project, deliberately. The 31 happy-path screens are a curated subset of the ~84 existing pages, focused on the student → teacher flywheel. User's job is to (1) merge Matt's perspectives into the working app, and (2) extrapolate Matt's design language to pages Matt never drew.

**Learning:** This raises the bar on the design system being built: tokens, layout primitives, and component primitives must be complete and consistent enough that an unseen page (e.g. `/admin/payouts`, `/settings/security`) can be styled from tokens alone without inventing values. The token system is the *contract* for extrapolation. North Star tiebreaker for visual decisions: does this surface the flywheel? Does it lower student → teacher friction?

**Pattern:** When a design handoff covers a deliberate subset, explicitly enumerate what's MISSING from the set during planning — admin layouts, data tables, complex forms, empty/error/loading states, etc. Better to address extrapolation gaps proactively than piecemeal during implementation.

## 6. Matt's typography uses asymmetric line-height: `normal` for headers, explicit `150%` for body Medium/Large
**Topics:** matt-design, typography

**Context:** Header 1–5 (and their Bold variants) all use `line-height: normal` — i.e., browser default. Body Default and Body Small also use `normal`. But Body Medium and Body Large both use explicit `line-height: 150%` AND `letter-spacing: -0.352px`.

**Learning:** Tight leading for headings, looser leading for sustained body reading — a sound editorial choice, not an inconsistency. The negative letter-spacing (~ -0.022em) is subtle optical tightening at the larger body sizes to keep them from feeling "loose." Both values must carry through to the `tokens-typography.css` body tokens.

**Pattern:** Don't normalize a designer's apparent inconsistencies before confirming intent. The asymmetry usually encodes deliberate readability decisions.

## 7. Figma Dev Mode exports Figma Variables as real CSS custom properties — match the naming verbatim
**Topics:** matt-design, figma-export, token-naming

**Context:** Header 1 screenshot showed `color: var(--Text-Default, #414141);` — Matt has Figma Variables wired up, and Dev Mode emits them as `var()` with hex fallback.

**Learning:** Match Matt's Title-Case-Hyphenated Figma Variable names verbatim in `tokens-primitives.css` and `tokens-semantic.css` (e.g., `--Text-Default`, NOT `--color-text-default`). This makes future paste-back from Figma Dev Mode lossless — one-to-one variable name match, no translation layer.

**Pattern:** When a design tool exports CSS variables natively, the variable namespace IS the contract. Don't impose your own naming convention on top — adopt the source's.
