# Session Learnings - 2025-12-28_20-35-12

## Technical

### Mock Data Must Match DB Schema for Type Safety
When mock data types don't align with the database schema, you get cascading type errors across all consuming components. The approach of using mock data "for now" creates technical debt that compounds quickly.

**Key insight:** It's easier to conform mock data to the schema upfront than to retrofit it later when multiple components depend on the old structure.

### Nullable Fields Require Explicit Handling
When DB schema fields are nullable (`number | null`), TypeScript enforces handling at every usage site:
- Use `?? 0` for numbers used in calculations
- Use `?.toLowerCase() || ''` for strings in filters
- Use conditional rendering (`{field && ...}`) for display

### Foreign Keys vs Embedded Objects
Mock data embedded full objects (e.g., `category: Category`) while DB uses foreign keys (`category_id: string`). This required:
1. Changing the type to use FK
2. Adding lookup functions to join data at runtime
3. Updating all consuming components

### Array Properties Become Join Tables
Simple arrays like `objectives: string[]` become separate DB tables with structure:
```typescript
interface CourseObjective {
  id: string;
  course_id: string;
  objective: string;
  display_order: number;
}
```
Components iterating over these arrays must extract the actual value (e.g., `objective.objective` instead of just `objective`).

## Workflow

### Schema-First Development
Checking mock data against DB schema early prevents downstream refactoring. The comparison process:
1. Read both files in parallel
2. Create a field-by-field comparison table
3. Identify missing/extra/mismatched fields
4. Plan the migration systematically

### Component Updates Follow Data Changes
When data structure changes, affected components can be identified by:
1. Running TypeScript type check
2. Errors point directly to incompatible usage
3. Fix each error site systematically

## Architecture

### Alias Types for Backward Compatibility
Using type aliases (`export type Creator = User`) allows gradual migration:
- Existing code using `Creator` continues to work
- New code can use the proper `User` type
- Can be cleaned up later when migration is complete

### Denormalized Data in Mock Layer
The mock data layer keeps related data denormalized for convenience:
```typescript
// User includes stats, expertise, qualifications directly
// In DB these would be separate tables
stats: UserStats;
expertise: UserExpertise[];
qualifications: UserQualification[];
```
This mirrors how the API layer might return data after joining.
