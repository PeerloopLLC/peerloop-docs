# Session Decisions - 2025-12-28

## Decision 1: Standalone HTML vs Astro Page for Mermaid Viewer

**Type**: Implementation
**Driver**: Claude-driven (pragmatic response to build issues)
**Working On**: Creating Mermaid diagram viewer for mock data visualization
**Trigger**: Astro page at `/dev/diagrams` failed to build due to layout import issues and React/Astro type conflicts

**Options Considered**:
1. Fix Astro page build issues (debug layout imports, fix JSX type conflicts)
2. Create a standalone HTML file with Mermaid CDN <- **Chosen**
3. Use a third-party Mermaid preview tool

**Decision**: Generate a standalone HTML file (`docs/mock-data-diagram.html`) that uses the Mermaid CDN and can be opened directly in any browser.

**Rationale**:
- The Astro page was encountering multiple issues: layout module resolution, JSX class vs className conflicts, and would require significant debugging
- A standalone HTML file achieves the same goal with zero dependencies on the Astro build
- The HTML file is self-contained and works offline after initial load
- Adding `--html` flag to existing script was cleaner than maintaining a separate Astro component

**Consequences**:
- Added `--html` flag to `generate-mock-data-diagram.ts` script
- Created `npm run mock-diagram:html` command
- Output goes to `docs/mock-data-diagram.html`
- No Astro dev server required to view diagrams
- Removed problematic `/src/pages/dev/diagrams.astro` file

## Decision 2: Dev Login Password Strategy

**Type**: Implementation
**Driver**: Claude-driven (implementing user request for login capability)
**Working On**: Adding mock users and dev login system
**Trigger**: User requested ability to "log in as each one" for testing different roles

**Options Considered**:
1. Use user's handle as their password (e.g., `sarah-miller` / `sarah-miller`)
2. Use a single shared password for all dev accounts <- **Chosen**
3. Generate unique passwords and document them

**Decision**: Use a single shared password `DEV_PASSWORD = 'dev123'` for all mock accounts.

**Rationale**:
- Simplest approach for development - one password to remember
- Easy to document and share with team
- The `getDevAccounts()` function lists all available accounts, so no need to remember individual passwords
- Dev-only feature, security not a concern

**Consequences**:
- All mock users authenticate with password `dev123`
- Added `DEV_PASSWORD` constant and `devLogin()` function
- Login UI can use `getDevAccounts()` to show all available test accounts
