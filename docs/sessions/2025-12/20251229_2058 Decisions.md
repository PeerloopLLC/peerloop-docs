## Decision 2: Course Sub-Table Editing Pattern

**Type**: Implementation
**Driver**: User-driven
**Working On**: Admin page design
**Trigger**: Courses have 8+ sub-tables (objectives, curriculum, prerequisites, etc.)

**Options Considered**:
1. **Single tabbed form** <- **Chosen** - One `/admin/courses/[id]` page with tabs for each section
2. Separate pages - Individual edit pages for each sub-table

**Decision**: Use single tabbed form for course editing.

**Rationale**: Keeps related data together, reduces navigation, matches common admin UX patterns.

**Consequences**: Course admin will be more complex than other admin pages but provides better UX.

## Decision 3: Soft Delete Strategy

**Type**: Strategic
**Driver**: User-driven
**Working On**: Admin delete functionality
**Trigger**: Need to decide how to handle deletions in admin

**Options Considered**:
1. **Soft delete** <- **Chosen** - Add `deleted_at` flag, keep data for audit
2. Hard delete - Permanently remove records from database

**Decision**: Use soft delete with `deleted_at` timestamp for users, courses, and enrollments.

**Rationale**: Preserves data for audit trail, allows recovery from accidental deletion, maintains referential integrity.

**Consequences**: Created migration 0005 adding `deleted_at`, `suspended_at`, `suspended_reason` fields. All admin list queries need `WHERE deleted_at IS NULL`.

## Decision 5: Admin Auth Pattern

**Type**: Implementation
**Driver**: Claude-driven
**Working On**: Admin API authentication
**Trigger**: Need to protect admin APIs from non-admin users

**Options Considered**:
1. Create separate admin middleware file
2. **Use existing `requireRole` function directly** <- **Chosen**

**Decision**: Use existing `requireRole(cookies, jwtSecret, ['admin'])` directly in API handlers.

**Rationale**: Auth system already has role-based access control. `requireRole` in `src/lib/auth/session.ts` handles role checking and throws 403 on failure. No need for additional abstraction.

**Consequences**: Simpler implementation, consistent with existing auth patterns, no new files needed for auth.
