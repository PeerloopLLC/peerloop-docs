# Session Learnings - 2025-12-29

## Technical

### Cloudflare Pages Git Integration

1. **"Retry deployment" re-deploys the same commit** - When a deployment fails, clicking "Retry" in the CF Dashboard deploys the exact same commit that failed, not the latest. To deploy new code, you must push a new commit (even an empty one works: `git commit --allow-empty -m "message"`).

2. **wrangler.toml syncs to CF Dashboard** - For Git-connected Pages projects, settings in `wrangler.toml` (like `compatibility_flags`) automatically sync to the CF Dashboard. No manual dashboard updates needed.

3. **Compatibility date matters** - The `compatibility_date` in wrangler.toml controls which runtime features are available. Newer dates enable newer behaviors, but the `nodejs_compat_v2` flag should work regardless of date.

### React 19 + Cloudflare Workers SSR

1. **The Problem**: React 19's SSR uses `react-dom/server.browser` which requires `MessageChannel` API. Cloudflare Workers don't have this API, even with `nodejs_compat_v2`.

2. **The Solution**: Add a Vite alias to use `react-dom/server.edge` instead:
   ```javascript
   vite: {
     resolve: {
       alias: import.meta.env.PROD && {
         'react-dom/server': 'react-dom/server.edge',
       },
     },
   }
   ```

3. **Why it works**: The `.edge` version is designed for edge environments (Cloudflare Workers, Deno) that support Web Streams but not full Node.js APIs.

4. **Reference issues**:
   - [Astro #12824](https://github.com/withastro/astro/issues/12824)
   - [React #31827](https://github.com/facebook/react/issues/31827)

### Debugging CF Pages Builds

1. Check the commit hash in deploy logs (`HEAD is now at XXX`)
2. Compare against what's actually on the branch (`git log origin/main -1`)
3. If mismatch, the deployment was queued before push completed
