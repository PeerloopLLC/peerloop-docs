# Session Learnings - 2025-12-28

## Technical

### Mock Data Consolidation Strategy

When consolidating inline mock data to a centralized file:
1. Check DB-SCHEMA.md to ensure mock types align with actual schema
2. Create helper functions that transform between DB schema naming (e.g., `student_name`) and component naming (e.g., `name`)
3. The helper function layer allows components to remain simple while data stays schema-compliant

### User Role Modeling

PeerLoop uses additive boolean flags for roles (`is_student`, `is_student_teacher`, `is_creator`, `is_admin`, `is_moderator`) rather than a single role enum. This means:
- Users can hold multiple roles simultaneously
- Student-teachers are also students (`is_student: true, is_student_teacher: true`)
- Filtering by role requires considering overlaps (e.g., "pure students" are `is_student && !is_creator && !is_student_teacher`)

### Dev Login System Pattern

For development login:
- Use a single shared password (`DEV_PASSWORD = 'dev123'`) for all mock accounts
- Provide `getDevAccounts()` function for login UI to list all available accounts
- Keep password simple - it's dev-only and simplifies testing different roles

### Enrollment Data Model

Enrollments track:
- Student-teacher assignment (nullable - `null` means creator taught directly)
- Status progression: `enrolled` → `in_progress` → `completed` (or `cancelled`)
- Student-teachers must complete a course before they can teach it (chronologically consistent)

## Workflow

### Mermaid Diagram Generation

Two output options work better than trying to integrate into Astro:
1. **Markdown output** - Good for GitHub/docs, console preview
2. **HTML output** - Standalone file with Mermaid CDN, opens in any browser

The HTML approach avoids Astro build complexity while providing an interactive viewer.

### Mock Data Visualization

The diagram script pattern is useful:
- Shows relationships that are hard to see in raw data
- Catches inconsistencies (e.g., S-T not having completed the course they teach)
- Quick reference for which test accounts exist and their roles

## Documentation

### Platform Stats Table

DB-SCHEMA.md has a `platform_stats` table designed for marketing displays:
- Key/value pairs with display labels
- Used for homepage stats like "150+ students learning"
- Avoids hardcoding marketing numbers in components

### Schema Naming Conventions

DB schema uses explicit naming (`student_name`, `student_role`, `course_id`) while component props can use shorter names (`name`, `role`). The mock-data.ts helper functions bridge this gap.
