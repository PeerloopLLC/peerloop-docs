# Session Learnings - 2025-12-27

## Technical

### placehold.co Font Size Limitation
- **Learning:** placehold.co auto-sizes text to fit the image dimensions - no direct font size control
- **Solution:** For consistent text sizing across placeholder images, use CSS overlays instead of relying on the placeholder service
- **Pattern:** When `thumbnail_url` is null, render a div with Tailwind classes for full control over typography

### GitHub Actions E2E Tests with Cloudflare
- **Learning:** E2E tests that use `npm run dev` with Cloudflare adapter fail in GitHub Actions because Cloudflare bindings (D1, KV) aren't available
- **Workaround options:**
  1. Make E2E job `continue-on-error: true` (non-blocking)
  2. Use `npm run preview` instead of `npm run dev` for CI
  3. Skip E2E until Cloudflare Pages is configured
- **Blocker:** Need GitHub org Owner access to install Cloudflare Pages GitHub App

### Image Fallback Strategy
- **Approach:** CSS-based fallbacks are more flexible than placeholder services
- **Implementation:**
  ```tsx
  {thumbnail_url ? (
    <img src={thumbnail_url} ... />
  ) : (
    <div className="bg-primary-200 flex items-center justify-center">
      <span className="text-2xl font-semibold">{title}</span>
    </div>
  )}
  ```
- **Benefit:** Full control over font, color, layout; no external service dependency

## Workflow

### Spec Panel Toggle for Development
- **Pattern:** Keep page spec documentation visible during development via toggleable panel
- **Implementation:**
  - CSS classes `.spec-content` and `.real-content` with body class toggle
  - localStorage for persistence across page navigation
  - Bottom-right toggle button shows current mode
- **Benefit:** Easy reference to spec while building real content

### Block-Based Component Organization
- **Structure:** Group components by feature area (e.g., `src/components/home/`, `src/components/courses/`)
- **Reusability:** CourseCard and CreatorCard designed for use across multiple pages (homepage, browse, profiles)

## Architecture

### Static Data for Initial Development
- **Pattern:** Use hardcoded data in components initially, switch to API calls later
- **Example:** `FeaturedCourses.tsx` has static `featuredCourses` array with comment "will come from API later"
- **Benefit:** Faster iteration, no need for complete backend before UI work

### Seed Data SQL Organization
- **Pattern:** Separate seed data into its own migration file (`0002_seed_data.sql`)
- **Structure:** Group INSERTs by table type with clear section comments
- **Tip:** Use NULL for image URLs in seed data to test fallback behavior
