# Session Learnings - 2025-12-29

## Technical

### Mock Data Fallback is an Anti-Pattern

**Learning:** Silent fallback to mock data when the database is unavailable hides real problems.

**Why it matters:**
- Developers may not realize they're seeing mock data instead of real data
- Configuration issues get masked until production
- Tests can pass locally but fail in production
- Data inconsistency between mock and real database can cause subtle bugs

**Better approach:**
- APIs should return explicit errors (503 Service Unavailable) when DB is unavailable
- Create a `/api/health/db` endpoint for explicit connectivity checks
- Fail fast - it's better to see errors immediately than to have them masked

### SSR Data Fetching in Astro

**Learning:** For React components that need data in Astro, the cleanest pattern is:

1. Fetch in Astro frontmatter (SSR)
2. Pass data as props to React components
3. Components become "dumb" - they just render data, no data fetching imports

**Benefits:**
- Data fetching happens server-side (SEO, performance)
- Components are simpler and more testable
- Clear separation between data layer and presentation

**Example pattern:**
```astro
---
// Astro page frontmatter
const response = await fetch(`${origin}/api/courses`);
const data = await response.json();
---
<CourseBrowse client:load initialCourses={data.items} />
```

### Type Exports Between Files

**Learning:** When removing mock-data imports from API files, ensure types are exported from the correct location.

- `PlatformStat` was only in `mock-data.ts`, not in `db/types.ts`
- Had to add the type to `db/types.ts` to maintain type safety

**Rule:** Types used by database APIs should be in `db/types.ts`, even if they're also defined in mock-data.ts.

## Architecture

### Health Check Endpoints

**Pattern established:** `/api/health/db` endpoint for D1 connectivity checks.

**Response format:**
```json
// Success (200)
{ "status": "ok", "timestamp": "...", "latency_ms": 12, "tables": 25 }

// Failure (503)
{ "status": "error", "timestamp": "...", "error": "D1 database binding not available" }
```

**Use cases:**
- Startup verification before testing
- CI/CD health checks
- Debugging connectivity issues

### API Error Consistency

**Pattern established:** All data APIs return consistent error responses:

- `503 Service Unavailable` - Database not available
- `404 Not Found` - Resource doesn't exist
- `400 Bad Request` - Missing/invalid parameters
- `500 Internal Server Error` - Unexpected errors

## Workflow

### MBA-2017 Development Workflow Updated

**Key change:** Without D1 connection, pages now show errors (APIs return 503). This is intentional.

**Updated workflow for MBA-2017:**
1. Check D1 connectivity first: `curl http://localhost:4321/api/health/db`
2. Ensure remote D1 is connected before testing data-dependent pages
3. Validate SQL syntax locally without D1
4. Test full flows against remote D1
