# Session Decisions - 2025-12-29

## Decision 1: INSERT OR IGNORE for Idempotent Migrations

**Type**: Implementation
**Driver**: Claude-driven (in response to error)
**Working On**: Block 1.8 - Creating seed migration 0004
**Trigger**: Migration failed with UNIQUE constraint error on user_expertise table

**Options Considered**:
1. `INSERT INTO ... ON CONFLICT (id) DO NOTHING` - Original approach
2. `INSERT OR IGNORE INTO ...` <- **Chosen**
3. Wrap migration in transaction with rollback logic

**Decision**: Use `INSERT OR IGNORE INTO` for all seed data inserts in migration 0004.

**Rationale**:
- `ON CONFLICT (id) DO NOTHING` doesn't handle composite unique constraints like `(user_id, tag)`
- `INSERT OR IGNORE` is cleaner SQLite syntax that handles all constraint types
- Makes migration fully idempotent - safe to re-run if partial failure occurred
- Simpler than transaction-based rollback for seed data

**Consequences**:
- Migration 0004 successfully applied with 183 statements
- Pattern established for future seed migrations
- Documented in Learnings for future reference

## Decision 2: D1 as Canonical Source for Categories

**Type**: Strategic
**Driver**: Claude-driven
**Working On**: Block 1.8 - Syncing mock-data.ts with D1
**Trigger**: Discovered mismatch - D1 has 15 categories, mock-data.ts had 3

**Options Considered**:
1. Update D1 to match mock-data.ts (3 simplified categories)
2. Update mock-data.ts to match D1 (15 categories) <- **Chosen**
3. Keep both different and handle mapping at runtime

**Decision**: D1 schema is the canonical source of truth; mock-data.ts should sync to match D1.

**Rationale**:
- D1 categories were designed with full taxonomy in migration 0001
- Course category_id references must match D1 foreign keys
- Mock data is for development/testing, not production definitions
- Runtime mapping would add unnecessary complexity

**Consequences**:
- Updated mock-data.ts categories from 3 to 15 items
- Mock courses already use cat-001 and cat-002 which exist in both
- Established pattern: D1 migrations define canonical data, mock syncs to match
