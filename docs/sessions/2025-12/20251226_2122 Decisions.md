# Session Decisions - 2025-12-26

## Decision 1: User Role System - Boolean Flags Only

**Type**: Strategic
**Driver**: User-driven
**Working On**: DB-SCHEMA.md - users table
**Trigger**: User identified that having both a `role` enum AND boolean flags (`is_creator`, `is_student_teacher`, etc.) was redundant and confusing for a system where users hold multiple roles.

**Options Considered**:
1. Boolean flags only (`is_student`, `is_student_teacher`, `is_creator`, `is_admin`, `is_moderator`) <- **Chosen**
2. Roles array (JSON field with `['student', 'student_teacher', 'creator']`)
3. Junction table (many-to-many: users ↔ user_roles ↔ roles)

**Decision**: Use boolean flags only. Remove the `role` enum field. Add `is_student` (default: true) and keep existing flags with defaults of false.

**Rationale**:
- The role set is stable (5 roles, unlikely to change often)
- D1/SQLite queries stay simple (`WHERE is_creator = true`)
- Already partially implemented with existing boolean flags
- Roles are additive as users progress through the platform (student → student_teacher → creator)

**Consequences**:
- Removed `role` enum from users table in DB-SCHEMA.md
- Added `is_student` boolean field (was implicit before)
- Updated users table description to explain boolean flags approach
- Feature flags `allowed_roles` field still works - maps role names to boolean checks

## Decision 2: Three Separate Navigation Sets

**Type**: Strategic
**Driver**: User-driven
**Working On**: Navigation architecture discussion
**Trigger**: Discussion about what changes between Public and Gated areas led to question of whether headers/footers should differ at all.

**Options Considered**:
1. Single unified header/footer for all areas (adapts based on auth state)
2. Two sets: Public and Authenticated
3. Three sets: Public, Gated, Admin <- **Chosen**

**Decision**: Maintain three architecturally separate header/footer component sets, even if they contain similar items.

**Rationale**:
- User is not the designer - this gives designer flexibility to customize each independently
- Admin area uses a completely different pattern (SPA with left sidebar)
- Cleaner separation of concerns
- May share internal components (Logo, UserMenu) but wrapper components are distinct

**Consequences**:
- Updated COMPONENTS.md with: PublicHeader, GatedHeader, AdminSidebar, PublicFooter, GatedFooter, AdminFooter
- Added shared UserMenu component used by all three areas
- Admin navigation uses Twitter-style 3-column layout (primary nav | submenu | content)

## Decision 3: Role Switching via User Dropdown

**Type**: Strategic
**Driver**: User-driven
**Working On**: Header navigation UX
**Trigger**: User proposed showing role-based navigation as top-level header items (Student, Teacher, Creator menus) with sub-items for each.

**Options Considered**:
1. Top-level header items for each role with dropdown menus (hidden if user doesn't have role)
2. Role switching in user avatar dropdown <- **Chosen**
3. Defer to Novice/Expert UI toggle in future

**Decision**: For now, keep role navigation as submenu items in the user avatar dropdown. More prominent role-based navigation could be revisited with a Novice/Expert UI toggle.

**Rationale**:
- Top-level role menus have significant cons:
  - Header real estate issues (3 extra items for users with all roles)
  - Crowded on smaller screens
  - Complexity of naming (Student vs Learning vs My Learning)
- Simpler approach is sufficient for MVP
- Can evolve later without architectural changes

**Consequences**:
- UserMenu component includes role-based items filtered by `is_student`, `is_student_teacher`, `is_creator` flags
- Each role item links to its dashboard section (My Learning, My Teaching, Creator Studio)
- "Home" link included so users can return to public pages from gated area
