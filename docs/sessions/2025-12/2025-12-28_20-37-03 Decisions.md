# Session Decisions - 2025-12-28

## Decision 1: Mock Data Schema Conformance Strategy

**Type**: Implementation
**Driver**: User-driven
**Working On**: Mock data structure in `src/lib/mock-data.ts`
**Trigger**: Analysis revealed mock data types didn't match DB-SCHEMA.md, causing potential issues when migrating to D1

**Options Considered**:
1. Keep mock data as-is, fix during D1 migration (Block 1.6)
2. Conform mock data to DB schema now, using mock values for required fields <- **Chosen**
3. Create a separate adapter layer between mock data and components

**Decision**: Update mock data types and values to exactly match the database schema, using mock values (e.g., `$mock-hash$` for password_hash) and defaults where null isn't allowed.

**Rationale**:
- Mock data that doesn't match schema is "useless" for validating component behavior
- Fixing now catches type issues in components immediately
- Less refactoring during D1 migration since types already match
- Components consuming mock data will work unchanged with real API data

**Consequences**:
- All mock data types updated (Category, User, Course, and nested types)
- Data arrays restructured with proper IDs and foreign keys
- Helper functions updated to work with new structure
- Component updates required to handle object arrays instead of string arrays
- Tailwind classes updated (v3 to v4 syntax) as part of component fixes

## Decision 2: Keep Type Aliases for Backward Compatibility

**Type**: Implementation
**Driver**: Claude-driven
**Working On**: Type definitions in mock-data.ts
**Trigger**: Renaming `Creator` to `User` would break existing component imports

**Options Considered**:
1. Rename Creator to User everywhere, update all imports <- Not chosen
2. Keep Creator as type alias for User (`export type Creator = User`) <- **Chosen**

**Decision**: Keep `Creator` as an alias for `User` and `creators` as an alias for `users` array.

**Rationale**:
- Minimizes changes to consuming components
- Allows gradual migration
- Both names are semantically valid (creators are users with is_creator=true)

**Consequences**:
- Existing code using `Creator` type continues to work
- New code can use either `User` or `Creator`
- `creators` export points to `users` array

## Decision 3: Denormalize Related Data in Mock Layer

**Type**: Implementation
**Driver**: Claude-driven
**Working On**: User and Course type structure
**Trigger**: DB schema has separate tables for stats, expertise, qualifications, but mock data needs to be usable without joins

**Options Considered**:
1. Separate arrays mirroring DB tables, require joins in helper functions
2. Embed related data directly in parent objects <- **Chosen**

**Decision**: Keep stats, expertise, qualifications embedded in User objects; keep objectives, includes, etc. embedded in Course objects.

**Rationale**:
- Mock data layer simulates what API responses would look like after joining
- Simpler to work with in components
- Still uses proper typed objects (not raw strings)

**Consequences**:
- User type includes `stats: UserStats`, `expertise: UserExpertise[]`, `qualifications: UserQualification[]`
- Course type includes typed arrays for all related data
- When migrating to D1, API layer will perform joins and return same structure
