# Session Decisions - 2025-12-30

## Decision 1: Keep E2E Tests in CI with Local D1

**Type**: Strategic
**Driver**: User-driven
**Working On**: CI/CD pipeline fixes
**Trigger**: E2E tests failing in GitHub Actions due to missing D1 tables

**Options Considered**:
1. **Keep e2e in CI** - Add migration step, tests catch code regressions <- **Chosen**
2. Remove e2e from CI - Rely on local testing only
3. Make e2e optional - Manual workflow dispatch instead of every push
4. Staging deployment test - Deploy to preview environment and test against real D1

**Decision**: Keep e2e tests in CI and add `npm run db:migrate:local` step before running tests.

**Rationale**: E2E tests in CI catch regressions like broken pages, JavaScript errors, and API logic bugs before deployment. The local D1 (SQLite) validates code works together even if it doesn't test production infrastructure.

**Consequences**:
- Added migration step to `.github/workflows/ci.yml`
- CI now creates local D1 database before e2e tests
- Tests verify frontend + API integration on every push

## Decision 2: Migration Architecture Consolidation

**Type**: Strategic
**Driver**: User-driven
**Working On**: Database migration conflicts
**Trigger**: CI failing with "UNIQUE constraint failed: categories.id" due to duplicate seed data across migrations

**Options Considered**:
1. Add `ON CONFLICT DO NOTHING` to conflicting migrations
2. Remove duplicate data from 0002, keep 0001's categories
3. **Consolidate all migrations into clean structure** <- **Chosen**

**Decision**: Clear remote D1, make mock-data.ts the single source of truth, create new clean migrations from scratch.

**Rationale**: At this early stage of development (no production users), a clean slate is better than workarounds. Having seed data scattered across 5 migrations led to conflicts and made data drift inevitable.

**Consequences**:
- Remote D1 cleared and re-seeded
- Old migrations deleted (0001_initial_schema.sql, 0002-0005)
- New structure: `0001_schema.sql` + `0002_seed.sql`
- Categories now 15 (matching mock-data.ts) instead of 3

## Decision 3: Split Migration into Schema + Seed

**Type**: Implementation
**Driver**: Claude-driven (proposed), User-approved
**Working On**: Migration consolidation
**Trigger**: User asked if single large SQL file would be better to break up

**Options Considered**:
1. Single combined file (~600 lines) with schema + data
2. **Two files: schema-only + seed-only** <- **Chosen**

**Decision**: Split into `0001_schema.sql` (tables/indexes) and `0002_seed.sql` (data from mock-data.ts).

**Rationale**:
- Schema changes tracked separately from data changes
- Can re-seed by just re-running 0002
- Future migrations (0003+) are schema changes only
- Easier to review and maintain

**Consequences**:
- `0001_schema.sql` - ~400 lines, all tables and indexes
- `0002_seed.sql` - ~350 lines, all seed data
- Clear separation of concerns for future migrations

## Decision 4: Footer Version Display - Option C

**Type**: Implementation
**Driver**: User-driven
**Working On**: Version display feature
**Trigger**: Request to show version and environment (DEV/STG/PROD) in footer

**Options Considered**:
1. Option A: Inline with copyright - `© 2024 Peerloop LLC · v1.0.0 DEV`
2. Option B: Separate bottom line - version centered below copyright
3. **Option C: Corner badge with tooltip** <- **Chosen**

**Decision**: Display version after Privacy/Terms links with commit hash in tooltip. Show colored environment badge (DEV=blue, STG=orange) only in non-production environments.

**Rationale**: Cleanest approach - version is accessible but unobtrusive. Tooltip provides full commit hash for debugging. Environment badge helps developers know their context without cluttering production UI.

**Consequences**:
- Build-time injection via Vite's `define` option in astro.config.mjs
- Created `src/lib/version.ts` for environment detection
- Footer displays `v1.0.0` with hover tooltip for commit
- DEV/STG badge visible; PROD badge hidden
