# Session Learnings - 2025-12-29

## Technical

### D1 Database Access on macOS 12 (MBA-2017)
- Local D1 emulation requires macOS 13.5+ (workerd runtime limitation)
- MBA-2017 (macOS 12) must use `--remote` flag for all D1 operations
- `wrangler login` (OAuth) is sufficient for interactive CLI use
- API tokens only needed for CI/CD pipelines and non-interactive scenarios

### D1 Migration Conflicts
- Seed data migrations can fail with `UNIQUE constraint failed` if data already exists
- Solution: Clear existing data before re-applying migrations
- Consider using `INSERT OR REPLACE` in seed files for idempotency

### API Endpoint Pattern
- Astro API routes access D1 via `locals.runtime?.env?.DB`
- Good pattern: Fall back to mock data when D1 unavailable
- This enables development without remote DB access

## Architecture

### 4-Layer Data Fetching Architecture
Designed and documented in `docs/tech/tech-014-data-fetching.md`:
1. **Layer 1:** SSR + cache for public pages (fast, SEO optimized)
2. **Layer 2:** CSR overlay for personalization (non-blocking)
3. **Layer 3:** SSR + auth for user-specific pages
4. **Layer 4:** Hybrid admin (dedicated routes + Context Actions overlay)

### Context Actions Component Design
- Single component providing role-aware, page-aware actions
- Supports Admin, Creator, and Student-Teacher roles
- Three action types: navigate, quick action (with page refresh), planned (disabled)
- Security: Server-rendered only for privileged users (not in bundle for others)
- AI actions shown as "Coming soon" placeholder for post-MVP

### Admin Approach Decision
- Hybrid approach: dedicated `/admin/*` routes + inline Context Actions
- Admin pages should accept context params (e.g., `?id=` for deep linking)
- Quick actions like "Feature course" can be both in admin page and Context Actions
