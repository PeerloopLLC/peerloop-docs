# Session Learnings - 2025-12-29

## Technical

### SQLite INSERT OR IGNORE for Idempotent Migrations
- D1 migrations don't auto-rollback on failure - partial data may be inserted
- Use `INSERT OR IGNORE INTO` instead of `INSERT INTO ... ON CONFLICT (id) DO NOTHING`
- The `ON CONFLICT` syntax works for simple cases but fails with composite unique constraints
- `INSERT OR IGNORE` is cleaner and handles all constraint types (PRIMARY KEY, UNIQUE, etc.)

### D1 Migration Failure Behavior
- When a D1 migration fails midway, previously executed statements persist
- The migration is marked as "failed" in the migrations table
- Re-running requires fixing the SQL and ensuring idempotency (INSERT OR IGNORE)
- Always test complex migrations on local D1 first when possible (not MBA-2017)

### API Pattern for User-Related Endpoints
- Consistent pattern: queryAll/queryFirst + parsePagination + createPaginatedResult
- Join user_stats for aggregated metrics (students_taught, courses_created, etc.)
- Join user_expertise for tags (requires separate query + map aggregation)
- Use LEFT JOIN for optional related data (student_teacher_id, etc.)

## Workflow

### Page Coverage Audit Approach
1. Glob for page specs (JSON files in `src/data/pages/`)
2. Glob for Astro pages (`src/pages/**/*.astro`)
3. Cross-reference to find gaps
4. Check page content to distinguish real implementations vs PageSpecView placeholders
5. Real implementations have React component imports + D1 API fetching

### Mock Data to D1 Sync Workflow
1. Read mock-data.ts to identify all entities
2. Compare with D1 schema (migrations) for table coverage
3. Check existing API endpoints
4. Create seed migration with mock data
5. Update mock-data.ts to match D1 canonical values (e.g., categories)
6. Create missing API endpoints
