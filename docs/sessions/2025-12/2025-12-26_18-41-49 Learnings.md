# Session Learnings - 2025-12-26

## Technical

### Astro + Cloudflare Workers macOS Compatibility
- Cloudflare Workers runtime requires **macOS 13.5+**
- Older Macs (e.g., MBA-2017 on macOS 12.x) cannot run `npm run dev` with Cloudflare adapter
- **Solution:** Detect OS version in `astro.config.mjs` and fall back to Node adapter for local dev:
  ```javascript
  const version = execSync('sw_vers -productVersion').toString().trim();
  const [major, minor] = version.split('.').map(Number);
  const useNodeAdapter = major < 13 || (major === 13 && minor < 5);
  ```

### Tailwind v4 vs v3 Compatibility
- `@astrojs/tailwind` integration expects Tailwind v3
- Tailwind v4 moved PostCSS plugin to separate package `@tailwindcss/postcss`
- **Solution:** Use Tailwind v3.x for stability (as recommended in tech docs)

### react-markdown GFM Support
- `react-markdown` does NOT support tables by default
- Requires `remark-gfm` plugin for GitHub Flavored Markdown (tables, strikethrough, task lists)
- Requires `remark-breaks` plugin to treat single newlines as `<br>` elements
- Both are tiny packages from the unified.js ecosystem

### Astro URL Handling in Development
- `Astro.site` may be undefined in dev mode if not configured
- `Astro.url.origin` can fail during SSR
- **Solution:** Provide fallback: `const siteUrl = Astro.site ?? new URL('http://localhost:4321')`

### Symlinks in public/ Folder
- Symlinks work for local development
- Relative path must be calculated from symlink's parent directory, not from the symlink itself
- `public/content/pages -> ../../research/run-001/pages` (2 levels up from `public/content/`)

## Workflow

### Page Code Architecture
- 4-letter codes (HOME, CBRO, etc.) are used throughout the markdown documentation
- Codes appear in "Connections" tables linking pages together
- **Must preserve codes** - they're the stable identifier; routes/filenames can change

### Route Planning for SEO
- Folder hierarchy in `src/pages/` determines URL routes
- SEO-critical public pages need clean, descriptive URLs
- Dynamic routes use Astro's `[param]` syntax
- Decided on: `/teachers` (not `/tutors`), `/courses/[slug]/learn` (nested under course)

### Unified Dashboard Approach
- Single `/dashboard` entry point with role cards
- Sub-paths for each role: `/dashboard/learning`, `/dashboard/teaching`, `/dashboard/creator`
- Users can have multiple roles simultaneously

## Architecture

### JSON Extraction Pattern
- Parse markdown page specs into structured JSON
- Store JSON alongside Astro page files (e.g., `index.json` next to `index.astro`)
- JSON includes route information for all page connections
- Enables rendering from data rather than parsing markdown at runtime

### Route Mapping Centralization
- `scripts/page-routes.ts` is the single source of truth for all page routes
- Maps code → route, astroFile, jsonFile
- Used by parse script to auto-determine output paths and enrich connections

### Layout Strategy
- **LandingLayout:** Public pages, marketing header/footer, optional sidepanel
- **AppLayout:** Authenticated pages, sidebar navigation, breadcrumbs
- **AdminLayout:** SPA shell with dark sidebar for admin pages
