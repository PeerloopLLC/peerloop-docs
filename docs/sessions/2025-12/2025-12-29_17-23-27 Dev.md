## Chronological Log

### 1. User: Examine the homepage data source

User asked where the homepage gets its data from - D1 or mock-data.ts.

**Action:** Examined `src/pages/index.astro` and the home page components (`FeaturedCourses.tsx`, `FeaturedCreators.tsx`). Found they were importing directly from `@/lib/mock-data`.

**Finding:** Homepage was using mock-data.ts, not D1.

---

### 2. User: When in the PLAN do we switch to D1?

User asked when the transition from mock data to D1 was scheduled.

**Action:** Reviewed PLAN.md. Found that Block 1.6 API endpoints were complete (they query D1 with mock-data fallback), but there was no explicit task to update pages to USE these APIs.

**Finding:** Gap identified - APIs existed but pages weren't updated to consume them.

---

### 3. User: Can we do this now? Would it interfere with Block 2?

User asked if we could make the switch immediately.

**Action:** Analyzed Block 2 scope (Stripe, payments, enrollment). Confirmed no conflict - switching data source is orthogonal to payment flows.

**Decision:** Proceed with the switch now. It completes Block 1 work properly.

### 12. User: Question about mock-data fallback rationale

User challenged the decision to fall back to mock data when D1 is unavailable.

**Discussion:** User made valid points:
- Fallback hides real problems
- Data inconsistency between mock and D1
- False confidence in tests
- Production risk of showing stale data

**Decision:** Remove mock-data fallback entirely. APIs should return 503 when D1 is unavailable.

---

### 13. Implementation: Remove mock-data fallback

Updated all 9 API endpoints to return `503 Service Unavailable` instead of falling back to mock data:
- `/api/stats`
- `/api/categories`
- `/api/testimonials`
- `/api/courses`
- `/api/courses/[slug]`
- `/api/courses/[id]/reviews`
- `/api/creators`
- `/api/creators/[handle]`
- `/api/creators/[id]/courses`

---

### 14. Implementation: Create health check endpoint

User suggested adding a D1 connectivity check endpoint.

**Action:** Created `/api/health/db` endpoint that:
- Returns 200 with status, latency, and table count on success
- Returns 503 with error message on failure

Added `PlatformStat` type to `db/types.ts` (was only in mock-data.ts).
