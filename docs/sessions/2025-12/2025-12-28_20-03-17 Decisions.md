# Session Decisions - 2025-12-28

## Decision 1: Use grep-based Tailwind v4 Checker Instead of eslint-plugin

**Type**: Implementation
**Driver**: User-driven (requested investigation)
**Working On**: Tailwind v4 compatibility checking
**Trigger**: User asked to investigate eslint-plugin-tailwindcss and prettier-plugin-tailwindcss for Tailwind v4 linting

**Options Considered**:
1. **eslint-plugin-tailwindcss** - Beta v4 support, may have false positives, doesn't catch canonical class suggestions
2. **prettier-plugin-tailwindcss** - Already installed, but only sorts classes (no linting)
3. **VS Code IntelliSense (getDiagnostics)** - Catches canonical suggestions but only for open files
4. **Custom grep-based script** <- **Chosen** - Catches known v3→v4 renames project-wide

**Decision**: Create `scripts/check-tailwind-v4.sh` that greps for known v3→v4 class renames, added as `npm run check:tailwind`

**Rationale**:
- eslint-plugin doesn't catch canonical class suggestions (the main pain point)
- getDiagnostics only works on open files
- Grep-based approach covers entire project and catches known renames
- Can be enhanced as more renames are discovered

**Consequences**:
- Created `scripts/check-tailwind-v4.sh` (executable)
- Added `npm run check:tailwind` to package.json
- Updated `/q-codecheck` to use this script as primary Tailwind check
- Fixed 8 v3→v4 issues found by the script

## Decision 2: Update q-codecheck to Use getDiagnostics as Secondary Check

**Type**: Implementation
**Driver**: User-driven
**Working On**: Improving Tailwind linting in q-codecheck
**Trigger**: User wanted to catch Tailwind suggestions, not just errors

**Options Considered**:
1. **getDiagnostics only** - VS Code IntelliSense catches more but only for open files
2. **npm run build only** - Catches errors but not suggestions
3. **check:tailwind script as primary, getDiagnostics as secondary** <- **Chosen**

**Decision**: Use `npm run check:tailwind` as primary (project-wide grep), `getDiagnostics` as secondary (additional IDE hints when files are open)

**Rationale**: Layered approach provides best coverage - grep catches known issues everywhere, IDE catches edge cases in open files

**Consequences**: Updated `/q-codecheck.md` with new workflow and documented the limitation about IDE-only analysis

## Decision 3: Consolidate Mock Data into Single File

**Type**: Strategic
**Driver**: User-driven
**Working On**: Block 1.7 completion
**Trigger**: User wanted to add more mock data easily while waiting for Cloudflare setup

**Options Considered**:
1. **Keep data in components** - Status quo, data duplicated across 6 files
2. **Create `src/data/` directory** - Separate folder for data
3. **Create `src/lib/mock-data.ts`** <- **Chosen** - Single file with types and helper functions

**Decision**: Consolidate all course and creator mock data into `src/lib/mock-data.ts` with exported helper functions

**Rationale**:
- Single source of truth eliminates duplication
- Helper functions provide different views (list, detail, featured, related)
- Components import only what they need → smaller bundles
- Clear migration path when D1 API is ready (replace helpers with API calls)

**Consequences**:
- Created `src/lib/mock-data.ts` (632 lines with types, data, and helpers)
- Updated 6 components to import from mock-data
- Reduced component files significantly (e.g., CourseDetail -290 lines)
- New `mock-data.js` bundle chunk (17.44 kB)
