# Session Learnings - 2025-12-27

## Technical

### Cloudflare D1 & macOS Compatibility
- **Workerd runtime requires macOS 13.5+** - MBA-2017 (macOS 12.6) cannot run local D1 emulation
- **database_id in wrangler.toml is only used with `--remote` flag** - the command flag determines which database is used, not the presence of the ID
- **D1 limitations vs SQLite**:
  - D1 only supports `batch()` for atomic operations (no `BEGIN`/`COMMIT`)
  - Prepared statements are single-use in D1 (create new for each query)
  - Network latency 50-200ms when using `--remote`
- **Golden rule**: Code for D1's more restrictive behavior - it will work on both

### Multi-Machine Development Strategy
- Both machines share same `wrangler.toml` (checked into git)
- MacMini uses `--local`, MBA-2017 uses `--remote`
- Local and remote databases are completely separate - they don't sync
- Created `.dev.vars` (gitignored) for machine-specific configuration

### Arctic OAuth Library
- GitHub constructor takes 3 positional args: `(clientId, clientSecret, redirectUri)` - not an options object
- Google uses PKCE with `generateCodeVerifier()` and `generateState()` from arctic
- OAuth state must be stored in short-lived httpOnly cookies

### Cloudflare Multi-Account Access
- When user has access to multiple Cloudflare accounts, wrangler requires account ID
- Use `CLOUDFLARE_ACCOUNT_ID=xxx wrangler d1 create ...` to target specific account
- `wrangler whoami` shows all available accounts

## Architecture

### Auth API Structure
- `/api/auth/register` - POST, creates user + sets cookies
- `/api/auth/login` - POST, verifies password + sets cookies
- `/api/auth/logout` - POST, clears cookies
- `/api/auth/session` - GET, returns current user from cookie
- `/api/auth/google` and `/api/auth/github` - OAuth flows with callbacks
- OAuth creates users automatically if they don't exist (email verified)
- OAuth users have empty password_hash - can't login with password
