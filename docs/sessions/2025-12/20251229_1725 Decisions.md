# Session Decisions - 2025-12-29

## Decision 1: Remove Mock-Data Fallback from APIs

**Type**: Strategic
**Driver**: User-driven
**Working On**: Switching pages from mock-data to D1 APIs
**Trigger**: User questioned the rationale for falling back to mock data when D1 is unavailable

**Options Considered**:
1. Keep mock-data fallback for development convenience and graceful degradation
2. Remove fallback entirely - APIs return 503 when D1 unavailable <- **Chosen**
3. Make fallback configurable via environment variable

**Decision**: Remove all mock-data fallback logic from API endpoints. When D1 is unavailable, APIs return `503 Service Unavailable` with `{"error": "Database not available"}`.

**Rationale**:
- Fallback hides real configuration problems (you don't know D1 isn't working)
- Mock data can drift from D1 schema, causing subtle bugs
- False confidence - tests pass locally but fail in production
- Production should never silently serve stale/incorrect data
- Better to fail fast and know immediately when something is wrong

**Consequences**:
- All 9 data API endpoints updated to return 503 instead of fallback
- On MBA-2017 without D1, pages requiring data will show errors (expected behavior)
- Documentation updated to reflect new workflow
- Added health check endpoint to verify D1 connectivity

## Decision 2: Add D1 Health Check Endpoint

**Type**: Implementation
**Driver**: User-driven
**Working On**: Removing mock-data fallback
**Trigger**: User suggested "a good idea as an independent check, perhaps at startup or during testing"

**Options Considered**:
1. No health check - rely on page errors to indicate D1 problems
2. Add `/api/health/db` endpoint for explicit connectivity check <- **Chosen**
3. Add health check to all API responses as a header

**Decision**: Create dedicated `/api/health/db` endpoint that tests D1 connectivity and returns status, latency, and table count.

**Rationale**:
- Provides explicit way to verify D1 is working before testing
- Useful for CI/CD pipelines
- Helps debug whether errors are D1-related or application-related
- Lightweight check (counts tables) without querying business data

**Consequences**:
- New endpoint created at `src/pages/api/health/db.ts`
- Returns 200 with `{status, timestamp, latency_ms, tables}` on success
- Returns 503 with `{status, timestamp, error}` on failure
- Documented in tech-013-devcomputers.md
- Added to MBA-2017 workflow as first step

## Decision 3: SSR Pattern for Data Fetching

**Type**: Implementation
**Driver**: Claude-driven
**Working On**: Updating pages to use D1 APIs
**Trigger**: Needed to decide how React components should receive data from APIs

**Options Considered**:
1. Client-side fetching with useEffect in React components
2. SSR - fetch in Astro frontmatter, pass as props <- **Chosen**
3. Hybrid - SSR for initial data, client refetch for updates

**Decision**: Fetch data in Astro page frontmatter (server-side) and pass to React components as props. Components become "dumb" - they render data, don't fetch it.

**Rationale**:
- Follows Layer 1 pattern from tech-014-data-fetching.md (SSR + cache for public pages)
- Better SEO - data is in initial HTML
- Faster initial page load - no client-side fetch waterfall
- Simpler components - easier to test and reason about
- Parallel fetch in frontmatter for performance

**Consequences**:
- All 8 React components updated to accept data as props
- All 5 Astro pages updated with frontmatter data fetching
- Components no longer import from mock-data.ts
- Types imported from mock-data.ts for prop interfaces (still used for type definitions)
