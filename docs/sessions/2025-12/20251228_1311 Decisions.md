# Session Decisions - 2025-12-28

## Decision 1: Template-Based Project Initialization

**Type**: Strategic
**Driver**: User-driven
**Working On**: q-init command improvement
**Trigger**: Desire for consistent documentation structure across projects

**Options Considered**:
1. Inline templates - Put skeleton markdown directly in q-init.md
2. Template files - Create `.claude/templates/*.md` files that q-init copies <- **Chosen**
3. Structured instructions - Keep q-init lean with "must include" lists

**Decision**: Create template files in `.claude/templates/` that q-init copies to destinations

**Rationale**:
- Templates can be maintained independently
- Ensures consistency across projects
- Easy to update templates without changing q-init logic
- Visual reference for expected structure

**Consequences**:
- Created 10 template files
- q-init updated with copy instructions
- Template location documented

## Decision 2: Portable vs Project-Specific Documentation Commands

**Type**: Strategic
**Driver**: User-driven
**Working On**: Documentation update workflow
**Trigger**: Need for project-specific documentation that doesn't fit in portable q-docs

**Options Considered**:
1. Add project-specific items to q-docs (makes it non-portable)
2. Create separate /loc-docs command
3. Create /q-local-docs command <- **Chosen**

**Decision**: Create `/q-local-docs` for project-specific documentation, called by `/q-docs` at the end

**Rationale**:
- Keeps q-docs portable across projects
- Clear separation of concerns
- q-local-docs always exists (with empty-state design for new projects)
- Consistent naming with q-* family

**Consequences**:
- Created q-local-docs.md command
- Created q-local-docs.template.md for new projects
- q-docs updated to call q-local-docs at end
- CLAUDE.md updated to document the split

## Decision 3: Machine Name Standardization

**Type**: Implementation
**Driver**: User-driven
**Working On**: Development environment documentation
**Trigger**: Need to scan session files for machine-specific workarounds

**Options Considered**:
1. `Mac Mini M4` / `MacBook Air 2017` (full names)
2. `M4-Mini` / `Air-2017` (abbreviated)
3. `MacMini` / `MBA-2017` <- **Chosen**

**Decision**: Use `MacMini` and `MBA-2017` as standard machine names

**Rationale**:
- Short enough for inline use
- Unique enough for grep scanning
- MBA is common abbreviation for MacBook Air
- MacMini is recognizable single word

**Consequences**:
- Convention documented in DEVELOPMENT-GUIDE.md and CLAUDE.md
- q-local-docs scans session files for these names
- devcomputers.md updated to use standard names
- 8 existing session files identified for standardization (pending)
