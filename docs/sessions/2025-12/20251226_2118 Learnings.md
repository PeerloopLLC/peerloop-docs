# Session Learnings - 2025-12-26_21-18-37

## Architecture

### User Roles: Boolean Flags vs Enum

**Learning:** For systems where users can hold multiple roles simultaneously, boolean flags are cleaner than a single enum field.

**Previous approach:**
- `role` enum (student, student_teacher, creator, admin, moderator)
- Plus boolean flags for each role
- Redundant and confusing

**New approach:**
- Only boolean flags: `is_student`, `is_student_teacher`, `is_creator`, `is_admin`, `is_moderator`
- `is_student` defaults to true (all new users are students)
- Roles are additive as users progress

**Why this is better:**
- No redundancy between enum and booleans
- Simple SQL queries (`WHERE is_creator = true`)
- Native SQLite/D1 support
- Clear mental model: roles accumulate, not switch

### Three Navigation Sets

**Learning:** Public, Gated, and Admin areas should have separate header/footer components even if content overlaps.

**Rationale:**
- Gives designers flexibility to customize each independently
- Admin uses a completely different pattern (SPA with left sidebar)
- Cleaner separation of concerns
- May share internal components (Logo, UserMenu) but architecturally distinct

## Documentation

### Role References After Schema Change

**Learning:** When changing a fundamental field like `role`, grep the entire schema for related references.

**Found:** Several "role" references that didn't need changing:
- `student_role` in testimonials (display title, not system role)
- `role` in sub_community_members (community-specific role, different concept)
- `role` in team_members (job title for marketing)
- `allowed_roles` in features table (still uses role names, maps to boolean flags)

**Takeaway:** Same word can mean different things in different contexts. Analyze before changing.
