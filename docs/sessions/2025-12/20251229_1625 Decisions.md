## Decision 2: Data Fetching Architecture

**Type**: Strategic
**Driver**: User-driven (priorities) + Claude-driven (design)
**Working On**: Architecture planning for public vs authenticated data
**Trigger**: Components still import directly from mock-data.ts; need to design API integration

**Options Considered**:
1. All SSR - Server-render everything
2. All CSR - Client-side fetch everything
3. **Layered approach** - Different strategies for different data types <- **Chosen**

**Decision**: 4-layer data fetching architecture:
- Layer 1: SSR + 5min cache for public pages (fast, SEO)
- Layer 2: CSR overlay for personalization (non-blocking)
- Layer 3: SSR + auth for user-specific pages
- Layer 4: Hybrid admin (dedicated routes + Context Actions)

**Rationale**: User priorities were clear: "public-facing pages should be FAST for non-logged-in users and SEO optimized" while still supporting personalization

**Consequences**: Architecture documented in docs/tech/tech-014-data-fetching.md; guides all future data fetching decisions

## Decision 4: Context Actions - Navigation over Inline Forms

**Type**: Strategic
**Driver**: User-driven
**Working On**: Context Actions component design
**Trigger**: Concern about security (hidden controls) and complexity (forms in display pages)

**Options Considered**:
1. Inline forms - Edit directly on the page
2. **Navigate to admin page** - "Edit" button links to existing admin page <- **Chosen**
3. Drawer/modal - Form slides in from side

**Decision**: Context Actions menu items navigate to admin pages; no forms on public/user pages

**Rationale**:
- Security: Component not rendered for non-privileged users (not just hidden)
- Simplicity: Forms live only in admin pages
- Admin pages accept context params (e.g., `?id=`) for deep linking

**Consequences**: Admin pages designed to accept context; no form duplication

## Decision 5: Context Actions - Role Expansion

**Type**: Strategic
**Driver**: User-driven
**Working On**: Context Actions scope
**Trigger**: User insight: "this would be helpful for creators (for certain pages where they have jurisdiction), same probably for Student-teachers"

**Options Considered**:
1. Admin-only menu
2. **Role-aware menu for Admin, Creator, S-T** <- **Chosen**

**Decision**: Context Actions component is role-aware with multiple sections:
- Admin sees admin actions on all pages
- Creator sees creator actions on their own courses
- S-T sees S-T actions on courses they teach

**Rationale**: Same component, same pattern, extended to all privileged roles based on ownership/assignment

**Consequences**: Role-Action matrix defined; ownership logic required in component
