## Decision 1: Skip D1 Connectivity Tests

**Type**: Strategic
**Driver**: User-driven
**Working On**: Test infrastructure planning
**Trigger**: Discussion about testing D1 availability on different machines

**Options Considered**:
1. Create tests that verify D1 remote works on both machines
2. Create tests that verify D1 local works on MacMini only
3. **Skip D1 connectivity tests entirely** <- **Chosen**

**Decision**: Do not implement D1 connectivity tests.

**Rationale**: Tests use mock data, not D1 data. D1 availability doesn't gate test execution or affect test outcomes. The machine detection hook already informs developers of D1 capabilities at session start - this is an informational tool, not a test gate.

**Consequences**: No infrastructure tests added. Machine detection hook remains the primary way to communicate D1 constraints.

## Decision 3: Separate Testimonials from Stories

**Type**: Implementation
**Driver**: Claude-driven (user approved)
**Working On**: Block 1.8.1 and 1.8.2 planning
**Trigger**: User asked about testimonials page, Claude analyzed existing stories.json spec

**Options Considered**:
1. Use existing `course_testimonials` table for Stories page too
2. **Create separate `success_stories` table with richer data model** <- **Chosen**

**Decision**: Testimonials and Stories are distinct entities requiring different tables and APIs.

**Rationale**:
- `course_testimonials`: Short quotes (~200 chars), linked to course, for social proof
- `success_stories`: Full narratives with before/after stats, video, location, type (student/teacher/creator)

These serve different purposes: testimonials are quick validation, stories are in-depth case studies.

**Consequences**: Created both `/testimonials` page (uses existing table) and updated `/stories` page (new table + API).

## Decision 4: Dual-Mode API Design for Testimonials

**Type**: Implementation
**Driver**: Claude-driven
**Working On**: `/api/testimonials` endpoint enhancement
**Trigger**: Need to serve both homepage (featured only) and browse page (all with filters)

**Options Considered**:
1. Create separate endpoints: `/api/testimonials/featured` and `/api/testimonials/browse`
2. **Single endpoint with mode switch via query params** <- **Chosen**

**Decision**: Use `?featured=false` to switch between homepage mode (simple array) and browse mode (paginated with filters).

**Rationale**:
- Reduces API surface area
- Homepage callers don't need to change (featured=true is default)
- Browse page gets full pagination support
- Single source of truth for testimonial data

**Consequences**: API returns `{ testimonials: [] }` for homepage, `{ items: [], total, page, ... }` for browse.
