# Session Learnings - 2025-12-26

## Technical

### Authentication Decision: Custom JWT over Auth Services

**Learning:** Clerk has known compatibility issues with Astro + Cloudflare Pages (`node:async_hooks` error). Custom JWT using `jose` + `bcryptjs` + `arctic` is the safer choice for edge deployments.

**Key Libraries:**
- `jose` - JWT creation/verification (works on Cloudflare Workers)
- `bcryptjs` - Password hashing (pure JS, no native deps)
- `arctic` - Lightweight OAuth (edge-compatible)

### Two-Tier Feature Flags Pattern

**Learning:** Don't rely solely on a 3rd-party service for core feature flags. Use a two-tier approach:

| Tier | Tool | Purpose |
|------|------|---------|
| Core Features | Custom D1 | Feature enablement, role-based access - must always work |
| Experiments | PostHog | A/B tests, gradual rollouts - can gracefully fail |

**Rationale:** If PostHog is down, core features still work. Experiments just default to control variant.

### Calendar Component vs Platform

**Learning:** `react-big-calendar` (UI component) is preferable to `Cal.com` (full platform) when you only need calendar display + custom booking logic.

- Cal.com = overkill if you're not using their full booking system
- react-big-calendar + D1 = right level of abstraction, stores in your database

### PostHog Covers Multiple Needs

**Learning:** PostHog bundles analytics + session replays + feature flags + A/B testing. This eliminates need for:
- Separate analytics tool
- Separate session replay tool (like LogRocket)
- Separate A/B testing tool

Free tier: 1M events/month - covers MVP phase.

## Workflow

### Tech Documentation Pattern

**Learning:** Create dedicated tech docs (`research/tech/tech-NNN-*.md`) for each software/service decision. Include:
- Why chosen (decision rationale)
- Comparison with alternatives
- Integration code examples
- Pricing
- References

**Benefit:** Future sessions can reference these docs instead of re-researching. Also captures caveats discovered during implementation.

### Decision Investigation Before Implementation

**Learning:** Running a "Decisions to Investigate" phase before starting Block 0 ensures:
- Technology choices are documented with rationale
- Integration patterns are researched upfront
- Potential compatibility issues are identified early (e.g., Clerk + Cloudflare)

### CLAUDE.md as Navigation Guide

**Learning:** CLAUDE.md should include a comprehensive "Research Reference" section organized by what you're looking for:
- What are we building? → Goals, user stories
- How should it look? → Pages, components
- Data & APIs → Schema, endpoints
- Tech decisions → Tech docs

This helps new sessions (or compacted contexts) quickly find relevant docs.

## Architecture

### Complementary Tool Selection

**Learning:** Choose tools that complement rather than overlap:

| Function | Tool | Why |
|----------|------|-----|
| Error tracking | Sentry | Best-in-class for stack traces, grouping |
| Session replays | PostHog | Included with analytics |
| Analytics | PostHog | Unified platform |
| A/B testing | PostHog | Included, no separate tool |

Avoided: LogRocket (would duplicate PostHog's session replay).
