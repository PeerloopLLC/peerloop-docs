# Session Decisions - 2025-12-27

## Decision 1: Dual-Machine D1 Development Strategy

**Type**: Strategic
**Driver**: User-driven
**Working On**: Block 0.4 Database Schema / Development Environment
**Trigger**: MBA-2017 cannot run Cloudflare Workerd (requires macOS 13.5+)

**Options Considered**:
1. Node adapter only - No database access locally
2. Local SQLite file with abstraction layer
3. Remote D1 for MBA-2017, Local D1 for MacMini <- **Chosen**
4. DevContainer/Docker
5. Upgrade macOS (not possible on 2017 hardware)
6. Buy new MacBook (deferred)

**Decision**: Use remote D1 (`--remote` flag) on MBA-2017, local D1 (`--local` flag) on MacMini. Both machines share the same `wrangler.toml` but use different commands.

**Rationale**:
- MBA-2017 hardware cannot be upgraded
- Remote D1 works fine for development (just slower)
- MacMini has full Cloudflare support
- Both machines can share same codebase
- User rejected DevContainer approach

**Consequences**:
- Created `tech-013-devcomputers.md` documenting both machines
- Added npm scripts for both `--local` and `--remote` DB operations
- Created `.dev.vars` (gitignored) for machine-specific config
- Documented D1 vs SQLite behavioral differences to code for

## Decision 3: D1 Database in Client's Cloudflare Account

**Type**: Strategic
**Driver**: User-driven
**Working On**: Cloudflare D1 Setup
**Trigger**: User logged in as personal account but production will be on client's account

**Options Considered**:
1. Create D1 in personal account (user's own)
2. Log out and log in as client account
3. Use client's API token
4. Client adds user as team member <- **Chosen**
5. Client creates database and shares ID

**Decision**: Have client add user as team member with appropriate permissions, then use `CLOUDFLARE_ACCOUNT_ID` environment variable to target client's account.

**Rationale**:
- Keeps production resources in client's account
- User can still use their own authentication
- Proper access control for team development
- Future developers can follow same pattern

**Consequences**:
- D1 created in Brian@peerloop.com's account
- Database ID: `262985c0-3667-4f26-8dc5-0de789d50f33`
- wrangler.toml updated with real database_id

## Decision 4: Email Verification Deferred to Block 9

**Type**: Implementation
**Driver**: Claude-driven
**Working On**: Block 0.5 Authentication
**Trigger**: Email verification requires Resend integration (notifications)

**Options Considered**:
1. Implement email verification now with Resend
2. Defer to Block 9 (Notifications) <- **Chosen**

**Decision**: Mark email verification as "deferred to Block 9 - Notifications" in PLAN.md. Complete auth without email verification for now.

**Rationale**:
- Block 9 handles all notification infrastructure (Resend)

- OAuth users are already email-verified by Google/GitHub
- Keeps Block 0.5 focused on core auth

**Consequences**:

- OAuth users marked as verified (provider verified them)
- Email verification will be added in Block 9
- PLAN.md updated to reflect this
