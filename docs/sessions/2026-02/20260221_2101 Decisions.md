# Session Decisions - 2026-02-21

## 1. No UNIQUE Constraint on creator_applications.user_id
**Type:** Implementation
**Topics:** d1

**Trigger:** Designing the `creator_applications` table schema — should a user be limited to one application ever, or allowed to reapply after denial?

**Options Considered:**
1. UNIQUE constraint on `user_id` — one application per user ever
2. No UNIQUE constraint, enforce "one pending at a time" at API layer ← Chosen
3. UNIQUE constraint on `(user_id, status)` — one per status

**Decision:** No UNIQUE constraint on `user_id`. The API layer enforces that a user can have at most one pending application at a time, but denied users can submit new applications.

**Rationale:** Denied users should be able to reapply (e.g., after building more experience). A DB-level UNIQUE constraint would either prevent reapplication entirely or require deleting old records. API-layer enforcement is more flexible and provides clearer error messages ("You already have a pending application" vs a generic constraint violation).

**Consequences:** The `creator_applications` table can have multiple rows per user. The API checks for `status = 'pending'` before allowing a new submission.

## 2. excludeCapabilities Added to AppNavbar MenuItem Interface
**Type:** Implementation
**Topics:** astro

**Trigger:** Need to show "Become a Creator" menu item only to non-creators, while "Creating" shows only to creators. Existing `capabilities` filter only supports positive matching.

**Options Considered:**
1. Add a separate `hideFromCreators` boolean flag — too narrow/specific
2. Add `excludeCapabilities` array to MenuItem interface ← Chosen
3. Use a custom filter function per item — over-engineered

**Decision:** Added `excludeCapabilities` to the `MenuItem` interface as the inverse of `capabilities`. Items are hidden if the user has any of the excluded capabilities.

**Rationale:** Mirrors the existing `capabilities` pattern (same type, same semantics inverted), making it immediately understandable. Reusable for future cases where items should be hidden from certain roles. Extracted `hasCap()` helper to avoid duplicating the switch statement.

**Consequences:** `AppNavbar.tsx` modified with new interface field and updated `getVisibleItems` logic. "Become a Creator" item uses `excludeCapabilities: ['canCreateCourses']`.
