## 1. Enable Staging Webhook for Stripe (Reversing Session 223 Decision)
**Type:** Strategic
**Topics:** stripe, deployment, cloudflare

**Trigger:** User decided that the staging/preview environment should have a working Stripe webhook, contradicting the Session 223 decision to skip it.

**Options Considered:**
1. Keep Preview webhooks disabled (Session 223 decision) — test locally only

**Consequences:** Stripe Dashboard now has `peerloop-staging` endpoint (6 events, "Your account" context). CF Dashboard Preview secrets updated with new `STRIPE_WEBHOOK_SECRET`. Updated tech-003-stripe.md and DECISIONS.md.

## 2. Register Only "Your Account" Events (Not Connected Accounts)
**Type:** Implementation
**Topics:** stripe

**Trigger:** Stripe Dashboard webhook creation asks to choose between "Your account" and "Connected and v2 accounts" event delivery context.

**Options Considered:**
1. Register "Your account" only (covers 6 of 7 events) ← Chosen
2. Register both contexts (requires two webhook endpoints)
3. Register "Connected and v2 accounts" only

**Decision:** Use "Your account" context for the staging webhook. The only event requiring "Connected accounts" is `payout.failed`, which is a courtesy notification — Stripe emails creators directly as fallback.

**Rationale:** 6 of 7 events are platform-level. Adding a second webhook endpoint for a single courtesy notification adds complexity for minimal benefit pre-MVP.

**Consequences:** `payout.failed` handler works locally (via `stripe listen` which forwards everything) but won't fire on staging. Deferred to PLAN.md.

## 3. Add Dispute Handlers as Critical Webhook Events
**Type:** Strategic
**Topics:** stripe

**Trigger:** Systematic audit of all Stripe API calls in the codebase revealed `charge.dispute.created` and `charge.dispute.closed` as unhandled events with serious financial consequences.

**Options Considered:**
1. Add only the 4 core events to staging webhook
2. Add 4 core events + 2 dispute events ← Chosen
3. Add all events and filter in code

**Decision:** Add `charge.dispute.created` and `charge.dispute.closed` to the webhook registration (making 6 events total) and implement handlers immediately. Disputes are the most dangerous unhandled event because Stripe auto-rules against the platform if no evidence is submitted within 7-14 days.

**Rationale:** Unlike other missing events (session expired, transfer reversed), disputes have urgent financial and operational consequences. The platform needs to freeze enrollments, notify admin, and reverse transfers on loss.

**Consequences:** Schema updated (added 'disputed' to enrollments, 'disputed'/'dispute_lost' to transactions). Two new handlers with 3 tests (11 → 14 total). Transfer reversal added to dispute-loss handler.

## 4. Rename transfer.paid to transfer.created
**Type:** Implementation
**Topics:** stripe

**Trigger:** `transfer.paid` was not available in Stripe Dashboard event selector — it doesn't exist as a Stripe event.

**Options Considered:**
1. Use `transfer.created` (fires when funds move to connected account) ← Chosen
2. Use `transfer.updated` (fires on metadata changes)
3. Remove the handler entirely (checkout handler already marks splits as paid)

**Decision:** Use `transfer.created` as the correct event. Moved from the `default` block into a proper `case` statement since it's in Stripe's TypeScript union.

**Rationale:** `transfer.created` fires when `stripe.transfers.create()` completes and funds move — this IS the "paid" moment. Transfers don't have a pending → paid lifecycle like payouts.

**Consequences:** Handler renamed from `handleTransferPaid` to `handleTransferCreated`. Test updated. All references in docs corrected.

## 5. Update Schema CHECK Constraints for Dispute Statuses
**Type:** Implementation
**Topics:** d1, stripe

**Trigger:** Dispute handlers needed to set enrollment status to 'disputed' and transaction status to 'disputed'/'dispute_lost', which weren't in the CHECK constraints.

**Options Considered:**
1. Add new status values to schema CHECK constraints ← Chosen
2. Reuse existing statuses ('cancelled' with cancel_reason, 'failed')

**Decision:** Add 'disputed' to enrollments CHECK constraint and 'disputed'/'dispute_lost' to transactions CHECK constraint.

**Rationale:** 'Disputed' is semantically different from 'cancelled' — the enrollment may be restored if the platform wins. Using a distinct status enables clear querying and prevents conflating voluntary cancellations with dispute freezes.

**Consequences:** `0001_schema.sql` updated. Enrollments: `('enrolled', 'in_progress', 'completed', 'cancelled', 'disputed')`. Transactions: `('pending', 'completed', 'refunded', 'partially_refunded', 'failed', 'disputed', 'dispute_lost')`.
