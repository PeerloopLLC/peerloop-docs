# Session Learnings - 2026-02-21

## 1. CC SessionStart Hook Output Goes to Model Context, Not Terminal
**Topics:** cc-workflow

**Context:** User expected to see dual-repo info hook output in the terminal at session start. Output was not visible despite verbose mode being enabled (`True` in `/status`).

**Learning:** SessionStart hook stdout is injected into the model's context as `<system-reminder>` blocks. It is NOT displayed in the user's terminal. The hooks run correctly and produce output — the model (Claude) can see and act on it — but the user sees nothing. Toggling verbose mode (`Ctrl+O`) should control visibility, but there appears to be a regression where it doesn't. Additionally, `Ctrl+O` can only be pressed after the session starts, so there's no way to enable it in time for SessionStart hooks.

## 2. Bare console.error on Cloudflare Workers Is Ephemeral
**Topics:** cloudflare, deployment

**Context:** Surveyed API routes for Sentry integration planning. Found 176 API files with ~292 `console.error`/`console.warn` call sites.

**Learning:** `console.error` on Cloudflare Workers is ephemeral — output is gone the moment the request ends. There is no persistent log. Without an error tracking service (Sentry), production errors are invisible. This is fundamentally different from traditional servers where logs persist to disk or stdout is captured by a log aggregator.

## 3. Obsidian Sync Operates at Vault Level, Not Folder Level
**Topics:** obsidian

**Context:** Explored whether a gitignored `_private/` folder in the docs vault could be synced via Obsidian Sync while committed content uses git.

**Learning:** Obsidian Sync syncs the entire vault. You can *exclude* folders from sync, but you cannot sync *only* specific folders. This means a hybrid git+Sync approach on the same vault would have two sync mechanisms operating on the same files — a recipe for conflicts. The only clean separation is separate vaults with separate sync mechanisms.

## 4. useMarkdownLinks Essential for Multi-Context Rendering
**Topics:** obsidian, docs-infra

**Context:** Setting up Obsidian vault for a repo that's also viewed on GitHub and consumed by Claude Code.

**Learning:** Obsidian's default `[[wikilinks]]` only render inside Obsidian. Standard markdown links `[text](path)` render correctly on GitHub, in Cursor/VS Code, in Claude Code context, and in Obsidian. For any vault whose content is viewed outside Obsidian, `useMarkdownLinks: true` in `app.json` is essential. The trade-off is slightly more verbose link syntax.

## 5. file-recovery Plugin Provides Safety Net for Gitignored Content
**Topics:** obsidian

**Context:** Setting up Obsidian vault where `.obsidian/` is gitignored.

**Learning:** The `file-recovery` core plugin takes periodic snapshots of files edited in Obsidian. For gitignored content (like `.obsidian/` config or any future local-only files), this is the only undo mechanism — git doesn't protect what it doesn't track. Not a backup replacement, but catches accidental edits.

## 6. Content Access-Control Boundaries Drive System Architecture
**Topics:** docs-infra, obsidian

**Context:** Considered consolidating personal notes, timecards, and daily notes into the peerloop-docs vault to simplify workflow. Client visibility on GitHub prevents this.

**Learning:** Content with different audiences (client-visible vs personal) and different backup mechanisms (git vs Obsidian Sync) cannot be cleanly combined in one container without introducing layered privacy rules. The cleanest architecture assigns each content type to exactly one container with exactly one sync mechanism and one audience. Attempting to abstract over these boundaries (gitignored folders, selective sync) adds cognitive load without simplifying the workflow.

---

## 7. Sentry Already Evaluated — Tech Doc Exists but No Integration
**Topics:** workflow

**Context:** User asked about production API error tracking. Found `tech-008-sentry.md` with thorough evaluation from Dec 2025.

**Learning:** Before proposing new technology, always check existing tech docs (`docs/tech/tech-NNN-*.md`). The evaluation and selection may have already been done. In this case, Sentry was fully evaluated, compared against LogRocket and Highlight.io, and selected — but never integrated. The tech doc had integration examples ready to use.
