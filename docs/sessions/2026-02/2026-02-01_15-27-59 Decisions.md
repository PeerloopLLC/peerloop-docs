# Session Decisions - 2026-02-01

## 1. Single Icons File vs Folder Structure
**Type:** Implementation
**Topics:** astro, workflow

**Trigger:** ~30 inline SVG icons scattered across components, with CheckIcon duplicated 4 times.

**Options Considered:**
1. Single `icons.tsx` file with all icons ← Chosen
2. Icons folder with category files (`navigation.tsx`, `actions.tsx`)
3. One file per icon (`CheckIcon.tsx`, `FeedIcon.tsx`)

**Decision:** Create single `src/components/ui/icons.tsx` with all icons as named exports.

**Rationale:**
- Simplest starting point - can split later if file grows beyond ~50 icons
- One import path (`@components/ui/icons`)
- Easy to browse all available icons in one place
- Follows existing `ui/` folder pattern

**Consequences:**
- Created icons.tsx with 26 unique icons + 6 aliases
- Removed ~120 lines of inline SVGs from AppNavbar and MoreSlidePanel
- Marketing components still have inline icons (can migrate later)

## 2. Transparent Slide Panel Overlay
**Type:** Implementation
**Topics:** astro

**Trigger:** Grey overlay (`bg-black/30`) dimmed entire screen when slide panels opened.

**Options Considered:**
1. Keep grey overlay for modal-like focus
2. Remove overlay entirely (no click-to-close)
3. Transparent overlay for click-to-close without dimming ← Chosen

**Decision:** Use invisible overlay (`fixed inset-0 z-30` without background color).

**Rationale:**
- Maintains click-outside-to-close functionality
- Doesn't obscure main content
- Feels less intrusive/modal-like
- Consistent with "slide from beneath navbar" visual

**Consequences:**
- Click anywhere outside panel closes it
- Main content remains fully visible while panel is open
- User can read main content while browsing panel options

## 3. Role-Based Home Page Landing Destinations
**Type:** Strategic
**Topics:** astro, auth, workflow

**Trigger:** Home page (`/`) showed static content with no connection to navbar selection state.

**Options Considered:**
1. Add "Home" item to navbar
2. Logo = Home (current behavior, no selection state)
3. SPA-style panel switching
4. Role-based landing to existing navbar items ← Chosen

**Decision:** Different user roles land on different navbar items when visiting `/`:

| User Type | Landing Action |
|-----------|---------------|
| Visitor | Auto-open Discover slideout |
| Student (no roles) | Auto-open Feeds slideout |
| S-T / Creator | Redirect to /workspace |
| Admin | Redirect to /messages |

**Rationale:**
- Reuses existing navbar items (no new "Home" item needed)
- Tailors experience to user's primary action
- Visitors discover content; students see community; workers go to workspace
- Admin has "To Admin" link (/admin) in navbar

**Consequences:**
- Added "To Admin" navbar item for admins (after "More")
- Home page selection state now reflects user role
- /workspace and /messages pages needed for redirects to work (currently 404)
- Trigger-once pattern prevents re-navigation on re-renders
