# Session Learnings - 2026-02-04

## 1. Route Archaeology: Tracing Legacy Routes Through Session History
**Topics:** astro, workflow

**Context:** User asked why both `/course/[slug]/discuss` and `/course/[slug]/feed` existed in the routing documentation.

**Learning:** When encountering seemingly redundant routes, trace their origins through multiple documentation sources:
1. `tech-021-url-routing.md` - Current routing source of truth
2. `ORIG-PAGES-MAP.md` - Pre-refactor page inventory
3. `DECISIONS.md` - Session decisions explaining architectural evolution
4. `RFC/` folders - Client change requests that drove changes

In this case, `/discuss` was from the original CDIS page spec (forum-style Q&A), while `/feed` was added in Sessions 181-183 as part of the Stream.io feed architecture. The original CDIS spec even said "Consider Stream.io for feed infrastructure" - which became `/feed`.

**Pattern:** Check session references in routing docs (e.g., "Session 183") and grep for those sessions in `DECISIONS.md` to understand why changes were made.

## 2. Component Orphaning Risk with Route Changes
**Topics:** astro, workflow

**Context:** After updating `index.astro` to use `CourseTabs` instead of `CourseDetail`, the `CourseDetail.tsx` component became orphaned.

**Learning:** When switching a page to use a different component:
1. The old component may become completely unused
2. Content from the old component may need to be migrated to the new one
3. There's no automatic warning - you must manually check for orphaned components

**Pattern:** After switching components, run:
```bash
grep -r "CourseDetail" src/ --include="*.tsx" --include="*.astro"
```
If no imports found, the component is orphaned. Assess what content should be preserved.

## 3. Tab Consistency Pattern for Subroute Pages
**Topics:** astro, workflow

**Context:** `feed.astro` used `CourseTabs` with `initialTab="feed"`, but `index.astro` used the older `CourseDetail` component, causing inconsistent UX when navigating between tabs.

**Learning:** For URL-aware tabbed interfaces where each tab has its own route:
- All subroute pages (`/course/[slug]`, `/course/[slug]/feed`, etc.) should use the **same tab component**
- Only the `initialTab` prop should differ
- This ensures consistent UI and prevents "tab disappearing" when navigating

**Pattern:**
```astro
// index.astro
<CourseTabs initialTab="about" ... />

// feed.astro
<CourseTabs initialTab="feed" ... />

// curriculum.astro
<CourseTabs initialTab="curriculum" ... />
```

## 4. Hero Placement in Tabbed Layouts
**Topics:** astro, workflow

**Context:** When merging `CourseDetail` content into `CourseTabs`, we had to decide where `CourseHero` should live.

**Learning:** For tabbed layouts with a hero/header element, two patterns exist:

| Pattern | Hero Location | Pros | Cons |
|---------|--------------|------|------|
| **Above tabs** | Always visible | Context persists, like YouTube video player | Takes vertical space on all tabs |
| **Inside tab** | Only on first tab | More content space on other tabs | Context lost when switching tabs |

For course pages, "above tabs" is better because:
- Price/CTA should always be visible for conversion
- Course identity (thumbnail, title, creator) provides context on Feed/Curriculum tabs
- Matches YouTube pattern (video above comments/description tabs)

## 5. Redundancy Detection When Merging Components
**Topics:** astro, workflow

**Context:** After adding `CourseHero` above tabs, the "Course Info Bar" inside the About tab became redundant (both showed price, stats, CTA).

**Learning:** When merging components, check for content duplication:
1. List all sections in original component
2. List all sections in target component
3. Identify overlaps
4. After merge, check if any sections are now redundant

In this case, `CourseHero` (above tabs) and "Course Info Bar" (inside About tab) both displayed price, rating, student count, and CTA button.
