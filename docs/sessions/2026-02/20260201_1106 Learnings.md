## 1. React FormEvent Type is Deprecated in @types/react
**Topics:** astro, workflow

**Context:** When fixing TypeScript warnings, `React.FormEvent<HTMLFormElement>` was still flagged as deprecated even with the generic type parameter.

**Learning:** The `@types/react` package has marked `FormEvent` itself as deprecated with this note: "FormEvent doesn't actually exist. You probably meant to use ChangeEvent, InputEvent, SubmitEvent, or just SyntheticEvent instead." For form submission handlers, `React.SyntheticEvent<HTMLFormElement>` is the recommended replacement since `FormEvent` extends `SyntheticEvent` anyway.

**Pattern:**
```typescript
// Deprecated (triggers warning)
const handleSubmit = async (e: React.FormEvent<HTMLFormElement>) => {

// Recommended replacement
const handleSubmit = async (e: React.SyntheticEvent<HTMLFormElement>) => {
```

## 2. Recharts Cell Component Can Be Replaced with Data-Driven Colors
**Topics:** astro, workflow

**Context:** The `Cell` component from recharts was flagged as deprecated in type definitions, causing warnings when used to color individual bars or pie slices.

**Learning:** Instead of using `<Cell>` children inside `<Bar>` or `<Pie>` components, you can add a `fill` property directly to each data entry. Recharts automatically uses this property for coloring, eliminating the need for Cell components entirely.

**Pattern:**
```typescript
// Before (deprecated Cell usage)
<Bar dataKey="value">
  {data.map((entry, index) => (
    <Cell key={`cell-${index}`} fill={colors[index]} />
  ))}
</Bar>

// After (data-driven colors)
const coloredData = data.map((entry, index) => ({
  ...entry,
  fill: entry.color || colors[index % colors.length],
}));

<Bar dataKey="value" />  // Uses fill from coloredData automatically
```

## 3. TypeScript Underscore Prefix Only Works for Parameters, Not Variables
**Topics:** testing, workflow

**Context:** When fixing "declared but never read" warnings in test files, prefixing variable names with underscore (`_user`, `_boundValues`) didn't suppress the warnings.

**Learning:** The underscore prefix convention (`_varName`) only suppresses unused variable warnings for **function parameters**, not for regular variable declarations. For unused variables in a function body, you must either:
1. Remove the declaration entirely
2. Actually use the variable
3. Accept the warning

**Pattern:**
```typescript
// Works (parameter)
function handler(_event: Event) { }  // No warning

// Doesn't work (variable)
const _user = userEvent.setup();  // Still warns "never read"

// Fix: Remove the unused variable
render(<Component />);  // If user isn't needed, don't declare it
```

## 4. Bulk Sed Edits Are Unreliable for Code Changes
**Topics:** workflow

**Context:** Attempted to fix many test file warnings using batch sed commands, but several edits failed silently or produced unexpected results.

**Learning:** When making systematic changes across many files, using the Edit tool file-by-file is more reliable than sed in batch operations. Issues encountered:
- Pattern escaping problems
- Line number shifts after edits
- Silent failures when patterns don't match
- Difficulty verifying each change

**Pattern:** For bulk code changes:
1. Get the list of files needing changes
2. Read each file individually
3. Make targeted Edit tool changes
4. Verify the change before moving on

This takes longer but ensures accuracy.

## 5. Astro Named Slots Don't Affect Load Order
**Topics:** astro

**Context:** Question arose about whether the unnamed vs named slot in an Astro layout affected when content was loaded.

**Learning:** Astro slots determine **where** content appears in the DOM, not **when** it loads. Both slots are rendered at build/request time and delivered in the same HTML response. The order in the layout determines DOM order (affecting visual rendering and tab order), but both are part of the same HTML document. CSS then controls visibility based on dev/prod mode.
