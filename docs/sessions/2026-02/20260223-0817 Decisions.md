# Session Decisions - 2026-02-23 (Session 261)

## 1. canModerateFor() Checks canModerateCourses Capability
**Type:** Implementation
**Topics:** auth

**Trigger:** CURRENTUSER.REVIEW audit revealed `canModerateFor(courseId)` only checked `isAdmin` and `isCreatorFor(courseId)`, missing the `canModerateCourses` global capability flag.

**Options Considered:**
1. Keep as-is — admin + creator only
2. Add `canModerateCourses` check — three-tier moderation ← Chosen
3. Defer until `course_moderators` table exists

**Decision:** `canModerateFor(courseId)` now checks three conditions: `isAdmin`, `isCreatorFor(courseId)`, and `canModerateCourses`. Users with the global moderation capability can moderate any course.

**Rationale:** The `canModerateCourses` flag exists specifically for moderators. Without checking it, invited moderators (who receive this flag via `/api/moderator-invites/[token]/accept`) couldn't actually moderate — their flag was set but never read by `canModerateFor()`.

**Consequences:** Updated `src/lib/current-user.ts` line 364. When `course_moderators` table is added, a fourth check will enable per-course assignment.

**Options Considered:**

## 5. Roles Documentation Must Cover Platform vs Course Scope
**Type:** Strategic
**Topics:** auth, docs-infra

**Trigger:** User review of ROLES.md identified that the document didn't distinguish platform-level roles from course-scoped roles.

**Decision:** Added "Platform vs Course Scope" section to ROLES.md with a table mapping each role to its platform capability, course relationship, and scope layer (platform-only, course-scoped, or both).

**Rationale:** This distinction is fundamental to the permission model — `canTeachCourses` (platform) vs `isStudentTeacherFor(courseId)` (course-scoped) are different questions. Moderator is the hybrid case: global flag today, per-course assignment in future.
