# Session Learnings - 2026-02-18

## 1. Stripe Connect Status Needs Self-Healing Sync
**Topics:** stripe, auth

**Context:** After completing Stripe Express onboarding as Guy Rymberg, the Payment Settings page showed "Setup Incomplete" (pending) with all three indicators green (charges, payouts, details submitted). The `stripe listen` webhook forwarder wasn't running, so the `account.updated` webhook never reached the app.

**Learning:** The `connect-status` API endpoint fetched live data from Stripe (for the three boolean flags) but returned `status` from the database, which was still `'pending'` from account creation. When webhooks are missed, this creates a contradictory state. The fix: derive the correct status from Stripe's live data and sync the database when it drifts.

**Pattern:**
```typescript
// In connect-status.ts: derive status from live Stripe data, sync DB if stale
let derivedStatus = user.stripe_account_status;
if (account.details_submitted && account.charges_enabled && account.payouts_enabled) {
  derivedStatus = 'active';
}
if (derivedStatus !== user.stripe_account_status) {
  await db.prepare(`UPDATE users SET stripe_account_status = ? WHERE id = ?`)
    .bind(derivedStatus, user.id).run();
}
```

## 2. Stripe Test Mode Provides Bypass Shortcuts at Every Verification Step
**Topics:** stripe, testing

**Context:** Walking through Stripe Express onboarding in test mode to document the flow for a client-facing guide.

**Learning:** Stripe test mode displays a callout box at every step that requires external verification (SMS, bank login, etc.) with a bypass button. The pattern is consistent: "Use test phone number" → sends to `000-000-0000` → then "Use test code" with code `000000`. Bank details offer "Autofill" for test routing/account numbers. This means you see every screen a real user would see, but can skip external verification. Useful for documentation since you capture the real flow.

## 3. Stripe SDK TypeScript Types Don't Include All Webhook Event Names
**Topics:** stripe, testing

**Context:** Adding `transfer.paid` and `payout.failed` handlers to the webhook switch statement caused TypeScript errors — these event names aren't in the `Stripe.Event['type']` literal union in the SDK.

**Learning:** The Stripe Node SDK's TypeScript types for `event.type` use a fixed literal union that doesn't include every possible webhook event. Events like `transfer.paid` and `payout.failed` exist in Stripe but aren't in the type definition. Workaround: handle these in the `default` case with a string comparison (`event.type as string`), which avoids the type error while still processing the events correctly.

**Pattern:**
```typescript
default: {
  const eventType = event.type as string;
  if (eventType === 'transfer.paid') {
    await handleTransferPaid(db, event.data.object as Stripe.Transfer);
  }
}
```

## 4. Stripe CLI Webhook Secret Is Stable Across Sessions on Same Machine
**Topics:** stripe, workflow

**Context:** Started `stripe listen` and the webhook signing secret (`whsec_dc1e89...`) matched the value already in `.dev.vars` from a previous session.

**Learning:** The Stripe CLI produces a stable webhook signing secret per machine/account combination. You don't need to update `.dev.vars` each time you restart `stripe listen`. This simplifies the dev workflow — the secret is effectively a one-time setup.

## 5. CSP and Console Errors on Stripe Hosted Pages Are Harmless
**Topics:** stripe

**Context:** During Stripe Connect onboarding, the browser console showed CSP violations for `connect-js.stripe.com`, Sentry 429 errors, and iovation script blocked by ad blocker.

**Learning:** These errors are on Stripe's own `connect.stripe.com` domain, not the app. They're caused by: (1) Stripe's own CSP blocking source map requests, (2) Stripe's internal Sentry being rate-limited, (3) ad blockers blocking fraud detection scripts. None affect onboarding functionality. The app's terminal shows no errors because these are entirely client-side on Stripe's hosted page.

## 6. Cloudflare Pages Preview URLs Are Dynamic, Incompatible with Fixed Webhook Endpoints
**Topics:** stripe, cloudflare, deployment

**Context:** Investigating whether webhooks work in the Preview environment. Found `STRIPE_WEBHOOK_SECRET` configured in Cloudflare Dashboard but no corresponding webhook endpoint in Stripe.

**Learning:** Cloudflare Pages generates unique URLs per deployment (`<hash>.peerloop.pages.dev`). Stripe webhooks require a fixed, stable URL. This makes Preview webhook testing impractical without a stable staging domain alias. Most teams test payments locally (via `stripe listen` CLI) and only set up permanent webhook endpoints for production. The Preview `STRIPE_WEBHOOK_SECRET` is harmless but effectively unused until a stable staging URL is established.
