# Session Decisions - 2026-02-22

## 1. Experience Level Per-Category in UX, Per-Topic in API
**Type:** Implementation
**Topics:** ui-patterns, onboarding

**Trigger:** PLAN specifies experience level "per category" but the API stores it per-topic in `user_topic_interests`. Need to reconcile UX simplicity with data granularity.

**Options Considered:**
1. Per-topic experience selectors (55 individual dropdowns)
2. Per-category experience selector, mapped to each selected topic ← Chosen
3. Single global experience level for all topics

**Decision:** TopicPicker shows one experience level `<select>` per expanded category. When the user selects "Intermediate" for Technology, all selected topics in Technology get `experienceLevel: 'intermediate'` in the API payload. Reduces max 55 decisions to max 15.

**Rationale:** Per-topic granularity overwhelms users during onboarding — they haven't explored the platform yet. Category-level is the right fidelity for "getting started" preferences. The API still stores per-topic data, so future features can offer more granular editing.

**Consequences:** `TopicPicker.setCategoryExperience()` maps category-level selection to all selected topics. The data model doesn't change — only the UX simplifies.

## 2. hideWhenOnboarded as Generic MenuItem Property
**Type:** Implementation
**Topics:** astro, ui-patterns

**Trigger:** Need to hide "Complete Your Profile" navbar item once user has completed onboarding, without hardcoding item IDs in the filter logic.

**Options Considered:**
1. Special-case the `complete-profile` ID in `getVisibleItems()`
2. Add `hideWhenOnboarded?: boolean` to MenuItem interface ← Chosen
3. Use a capability like `excludeCapabilities`

**Decision:** Added `hideWhenOnboarded` boolean property to the `MenuItem` interface. The `getVisibleItems()` filter checks `if (item.hideWhenOnboarded && user?.hasCompletedOnboarding()) return false`.

**Rationale:** Generic and composable — any future menu item that should disappear post-onboarding can use the same property. Consistent with the existing `requiresAuth` and `capabilities` pattern on MenuItem. No special-casing of IDs.

**Consequences:** One new optional property on MenuItem interface, one new filter condition in `getVisibleItems()`.

## 3. Post-Save Behavior: First-Time Redirect, Returning Stay
**Type:** Implementation
**Topics:** onboarding, ui-patterns

**Trigger:** After saving the onboarding profile, should the user be redirected or stay on the page?

**Options Considered:**
1. Always redirect to homepage
2. Always stay on page
3. First-time → redirect to `/`, returning → stay on page ← Chosen

**Decision:** First-time users see "Profile complete! Redirecting you to the homepage..." and redirect after 2 seconds. Returning users (editing via dropdown/settings) see "Preferences saved successfully" and stay on the page.

**Rationale:** First-timers are in a setup flow and should continue their journey. Returning users intentionally navigated to edit preferences — redirecting them would be disruptive. The `isFirstTime` flag is derived from whether `onboardingCompletedAt` was null when the page loaded.

**Consequences:** `OnboardingProfile` tracks `isFirstTime` state. Button label changes ("Complete Setup" vs "Save Changes"). Success message differs. `setTimeout` redirect only for first-time.

## 4. Astro Page Tests Use File Content Assertions
**Type:** Implementation
**Topics:** testing, astro

**Trigger:** Need page-level tests for `/onboarding.astro`, but Astro pages are server-rendered and can't be tested with React Testing Library.

**Options Considered:**
1. Skip page tests entirely
2. Use `readFileSync` to assert on page file content ← Chosen
3. Spin up Astro dev server for integration tests

**Decision:** Page tests use `readFileSync` to read the `.astro` file and assert structural properties: uses AppLayout, has correct title, includes the React island component with `client:load`, has page code in comment.

**Rationale:** Lightweight, fast (1ms), and catches regressions in page structure (wrong layout, missing component, missing hydration directive). Integration-level testing is deferred to E2E (Playwright) in the TESTING block.

**Consequences:** Established pattern for Astro page tests across the project. 7 assertions covering layout, title, component inclusion, and page code.
