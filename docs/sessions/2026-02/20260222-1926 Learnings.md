# Session Learnings - 2026-02-22

## 1. Stripe Test Mode Magic Values for Identity Verification
**Topics:** stripe

**Context:** Setting up sarah.miller as a Stripe Connect Express account in test mode. SSN validation was failing with "invalid SSN."

**Learning:** Stripe test mode requires specific "magic" values for identity verification. For SSN, use `000-00-0000` (or last 4: `0000`) for successful verification. Random values trigger validation failures even in test mode. Other magic values: routing `110000000`, account `000123456789`.

## 2. CurrentUser Has No Stripe Data by Default
**Topics:** stripe, astro

**Context:** After completing Stripe Connect onboarding for sarah.miller, investigated whether the global `CurrentUser` should know about the user's Stripe status.

**Learning:** The `CurrentUser` class had zero Stripe-related fields — no `stripe_account_id`, no `stripe_account_status`, no `payouts_enabled`. The only way to check Stripe status was a separate `GET /api/stripe/connect-status` call. This meant components couldn't make instant UI decisions (show/hide payout buttons, Stripe prompts) without an extra API round-trip.

## 3. CurrentUser Is Architecturally a Read-Only Cache
**Topics:** astro, auth

**Context:** Before adding Stripe fields to CurrentUser, paused to audit whether CurrentUser was ever used as a source of truth anywhere in the codebase.

**Learning:** Thorough audit confirmed CurrentUser has clean separation of concerns:
- All `readonly` properties — class is immutable by design
- `setCurrentUser()` is private to the library — only `refreshCurrentUser()` can replace the instance
- No component ever sends CurrentUser data back to an API — endpoints always derive identity from JWT session
- Every update goes through `refreshCurrentUser()` → `GET /api/me/full`
- Server always validates permissions independently on every API call

This makes CurrentUser safe to extend with cached convenience data (like Stripe status) without security implications.

## 4. BBB and Stream IDs Are Not User-Level
**Topics:** d1, video, stream

**Context:** After adding Stripe data to CurrentUser, investigated whether BigBlueButton or Stream.io IDs should also be surfaced.

**Learning:** BBB fields (`bbb_meeting_id`, `bbb_internal_meeting_id`, `bbb_attendee_pw`, `bbb_moderator_pw`) live on the `sessions` table — they're per video session, not per user. Stream fields (`stream_activity_id`, `stream_reaction_id`) live on the `content_flags` table — they're per content item. Neither is user-level data appropriate for CurrentUser.

## 5. StripeConnectSettings Didn't Refresh CurrentUser on Return
**Topics:** stripe, astro

**Context:** After Stripe Connect onboarding redirects back to `/settings/payments?success=true`, examined what happens.

**Learning:** The component cleaned the URL query params and fetched Stripe status, but never called `refreshCurrentUser()`. This meant the global user state could be stale after a significant state change. The AppNavbar's window-focus refresh would eventually catch it, but only if the user navigated away and came back — a race condition if they stayed on the page.

## 6. ONBOARDING Feature Was "Part 1 of 3" — UI Not Yet Built
**Topics:** testing

**Context:** Wondered whether onboarding UI tests existed after schema and API work from a previous session.

**Learning:** The ONBOARDING feature in PLAN.md is structured as three phases: SCHEMA (done), API (done, tested), UI (not started). The API endpoint `POST /api/me/onboarding-profile` has full test coverage, but no UI component (`OnboardingProfile.tsx`, `TopicPicker.tsx`) exists yet. The `GET /api/topics` endpoint exists but has no test file. PLAN.md checkboxes were never marked after Phase 1 completion.
