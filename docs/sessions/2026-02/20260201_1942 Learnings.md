# Session Learnings - 2026-02-01

## 1. Astro Multi-Island Architecture Prevents React Context Sharing
**Topics:** astro, auth

**Context:** Implementing a login modal that could be triggered from AppNavbar and also from protected action buttons elsewhere in the app.

**Learning:** In Astro, each `client:load` React component creates its own React root. React Context cannot be shared between these islands - they're isolated. A login modal context provider in one island cannot be accessed by components in another island.

**Pattern:** Use global state on `window` with custom events for cross-island communication:
```typescript
// Global state module
const EVENT_NAME = 'peerloop:auth:change';

export function openLoginModal(onSuccess?: () => void): void {
  const state = getGlobalState();
  state.isOpen = true;
  state.pendingAction = onSuccess ?? null;
  window.dispatchEvent(new CustomEvent(EVENT_NAME));
}

export function subscribe(callback: () => void): () => void {
  window.addEventListener(EVENT_NAME, callback);
  return () => window.removeEventListener(EVENT_NAME, callback);
}
```

## 2. Modal Login Requires Explicit UI State Refresh
**Topics:** auth, astro

**Context:** After modal login succeeded, the page didn't show the user as logged in until manual refresh.

**Learning:** When using global auth state with React islands, successful login updates the global state but doesn't trigger re-renders in components with local state copies. Components must explicitly subscribe to auth change events and refresh their local state.

**Pattern:** Dispatch a specific event on login success; components subscribe and refresh:
```typescript
// In auth-modal.ts
export function handleLoginSuccess(): void {
  // ... update state ...
  window.dispatchEvent(new CustomEvent('peerloop:auth:loginsuccess'));
}

// In AppNavbar.tsx
useEffect(() => {
  const unsubscribe = subscribeToLoginSuccess(() => {
    setUser(getCurrentUser());
    forceUpdate((n) => n + 1);
  });
  return unsubscribe;
}, []);
```

## 3. TypeScript Nested Comments in JSDoc Cause Parse Errors
**Topics:** workflow

**Context:** Writing JSDoc with an example containing a comment caused cryptic TypeScript errors.

**Learning:** JSDoc block comments (`/** */`) cannot contain `/* */` inside them - even in example code. TypeScript parser treats the inner `*/` as closing the JSDoc block.

**Pattern:** Avoid comment syntax in JSDoc examples:
```typescript
// BAD - causes parse error
/**
 * @example
 * openModal(() => { /* callback */ });
 */

// GOOD
/**
 * @example
 * openModal(() => { doSomething(); });
 */
```

## 4. Global Window Type Extensions Must Not Conflict
**Topics:** astro, workflow

**Context:** Adding `authModal` to `window.__peerloop` caused TypeScript errors because the type was already declared in another file.

**Learning:** When multiple files declare `window.__peerloop`, TypeScript merges the declarations and requires them to be compatible. Redeclaring with a different shape causes "subsequent property declarations must have the same type" errors.

**Pattern:** Use type assertions to access properties without redeclaring the global:
```typescript
interface PeerloopGlobal {
  authModal?: AuthModalState;
  [key: string]: unknown;
}

function getGlobal(): PeerloopGlobal {
  const win = window as unknown as { __peerloop?: PeerloopGlobal };
  if (!win.__peerloop) win.__peerloop = {};
  return win.__peerloop;
}
```
