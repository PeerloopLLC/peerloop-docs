# Session Decisions - 2026-02-22 (Session 248)

## 1. PLAN.md Forward-Looking Philosophy
**Type:** Strategic
**Topics:** docs-infra, cc-workflow

**Trigger:** PLAN.md had grown to 1,260 lines with ~60% completed content. The old philosophy kept completed sub-blocks as one-liners until the entire block completed, causing steady bloat.

**Options Considered:**
1. Keep completed sub-blocks as one-liners (status quo)
2. Strip completed sub-blocks, only show remaining work ← Chosen

**Decision:** PLAN.md is forward-looking only. It contains only work that remains to be done. Completed content lives exclusively in COMPLETED_PLAN.md.

**Rationale:** A planning document's value is in showing what's next, not what's done. Mixing completed and pending work makes the file hard to scan and creates maintenance overhead.

**Consequences:**
- Active blocks show a "Completed:" summary line + only remaining sections
- No "Completed Blocks" section in PLAN.md
- When a block fully completes: terse entry to COMPLETED_PLAN.md, full removal from PLAN.md
- `/q-update-plan-local` rewritten to match

## 2. Block Completion Workflow Revised
**Type:** Strategic
**Topics:** docs-infra

**Trigger:** Old workflow described moving "full details" to COMPLETED_PLAN.md and leaving stubs in PLAN.md. But COMPLETED_PLAN.md was already restructured to terse format (Session 247), creating a mismatch.

**Options Considered:**
1. Keep full details in COMPLETED_PLAN.md (old approach)
2. Terse archive in COMPLETED_PLAN.md, clean removal from PLAN.md ← Chosen

**Decision:** When a block completes: (1) add terse entry to COMPLETED_PLAN.md (name + 1-line summary + session range), (2) fold any deferred items into related blocks, (3) remove entire block from PLAN.md with no stub or link.

**Rationale:** Session docs in `docs/sessions/` already preserve full detail. COMPLETED_PLAN.md serves as a quick index of what was done and when. PLAN.md stays clean.

## 3. PLAN.md Three-Tier Block Sequence
**Type:** Implementation
**Topics:** docs-infra

**Trigger:** Old Execution Sequence table mixed 11 completed rows with 9 active/pending/blocked rows.

**Decision:** Top of PLAN.md uses ACTIVE / ON-HOLD / DEFERRED structure. All completed rows removed.

**Rationale:** Clear separation between what's being worked on now, what's blocked, and what's queued. Priority numbers in DEFERRED table allow reordering.

## 5. Deferred Items Redistributed into Related Blocks
**Type:** Implementation
**Topics:** docs-infra

**Trigger:** Catch-all "Deferred Items" table had ~18 items from various completed blocks, disconnected from context.

**Decision:** Eliminated the table. Resolved items deleted (KV binding, Sharp config). Remaining items folded into related blocks: Stripe items → MVP-GOLIVE.STRIPE, email verification → MVP-GOLIVE.RESEND, real-time feeds → FEEDS.LIMITATIONS, session reminders → POLISH, Certificate PDF + "Schedule Later" → Post-MVP footer.
