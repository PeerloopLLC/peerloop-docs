# Session Decisions - 2026-02-20

## 1. Separate Documentation into Obsidian Vault Repo
**Type:** Strategic
**Topics:** workflow, deployment

**Trigger:** Code repo is cluttered with ~861 documentation files (~20.5M) that serve development planning but don't belong in a shipping codebase. Client needs a clean code repo.

**Options Considered:**
1. Keep everything in one repo (status quo)
2. Symlink repo folders INTO an Obsidian vault
3. Move docs OUT of repo into a vault, symlink back from repo ← Chosen
4. Copy/sync approach between two locations

**Decision:** Create a separate `peerloop-docs` git repo that serves as an Obsidian vault. Move all docs/research/RFC/planning files there. Code repo gets relative symlinks (`../peerloop-docs/...`) that are .gitignored. Both repos are shared with the client.

**Rationale:** Obsidian owns the real files (best plugin/indexing compatibility). Claude Code follows symlinks transparently. Code repo becomes clean. Relative symlinks work on any machine with both repos in the same parent directory. Client gets a browsable knowledge base as a separate deliverable.

**Consequences:** Detailed migration plan created at `.claude/plans/reflective-mixing-turing.md`. Not yet executed. Requires dual-repo commit workflow changes.

## 2. Vault Repo Hosted in Same GitHub Org
**Type:** Strategic
**Topics:** workflow, deployment

**Trigger:** Need to decide where to host the new peerloop-docs vault repo.

**Options Considered:**
1. Same GitHub organization as code repo ← Chosen
2. Separate standalone repo with collaborator invites
3. Decide later

**Decision:** Host `peerloop-docs` in the same GitHub organization as the Peerloop code repo.

**Rationale:** Keeps both repos discoverable together. Client already has org access. Simplest permission model.

**Consequences:** Will create repo in GitHub org during migration execution.

## 3. CLAUDE.md as Symlink Without Pre-Testing
**Type:** Implementation
**Topics:** workflow

**Trigger:** CLAUDE.md must be readable by Claude Code at the project root. If it moves to the vault, the code repo needs a symlink. Question: will Claude Code follow a symlinked CLAUDE.md?

**Options Considered:**
1. Test in a scratch directory first before committing to migration
2. Skip test, assume OS-level symlink resolution works ← Chosen

**Decision:** Proceed without pre-testing. Standard `open()`/`read()` syscalls follow symlinks on macOS. If issues arise during Phase 4 verification, fallback to keeping CLAUDE.md as a real file in the code repo.

**Rationale:** Symlink resolution is an OS-level guarantee, not application-specific. Low risk. The migration plan includes a verification step that catches any failure.

**Consequences:** Migration plan proceeds with CLAUDE.md as a symlink. Fallback documented in plan.

## 4. Use Relative Symlinks for Cross-Repo Portability
**Type:** Implementation
**Topics:** workflow

**Trigger:** Symlinks from code repo to vault repo must work on multiple development machines.

**Options Considered:**
1. Absolute paths (`~/projects/peerloop-docs/...`) — tied to specific home directory
2. Relative paths (`../peerloop-docs/...`) ← Chosen
3. Environment variable paths — adds complexity

**Decision:** Use relative symlinks (`../peerloop-docs/docs/`) so they work on any machine where both repos are cloned into the same parent directory.

**Rationale:** No hardcoded home directory paths. Works on MacMiniM4-Pro, MacMiniM4, and any client machine. Only requirement: both repos in the same parent folder.

**Consequences:** `scripts/link-docs.sh` creates relative symlinks. Setup instructions tell users to clone both repos into the same directory.
