# Session 171 Decisions - 2026-02-03

## 1. Universal Profile URL Pattern (/@handle)
**Type:** Architecture
**Topics:** routing, astro

**Trigger:** Implementing tech-021 URL routing scheme, specifically line 182: `/profile` → `/@me`

**Options Considered:**
1. Keep `/profile` as the profile page
2. Use `/@[handle]` for universal profiles with `/@me` redirect ← Chosen
3. Use `/users/[handle]` pattern

**Decision:** Implement `/@[handle]` as the universal profile URL pattern, where `/@me` redirects to the current user's actual handle URL.

**Rationale:**
- Standard pattern used by Twitter/X, Instagram - familiar to users
- Short, memorable, shareable URLs
- Adapts CTAs based on viewer (Edit vs Message)
- Works for all user types (students, S-Ts, creators)

**Consequences:**
- Created `src/pages/@[handle].astro`
- `/profile` now 301 redirects to `/@me`
- `/@me` client-side redirects to `/@{currentUser.handle}`

## 2. UserCard Component Architecture
**Type:** Implementation
**Topics:** ui, architecture

**Trigger:** User noted the profile header looked like a reusable card format for listings.

**Options Considered:**
1. Keep profile header embedded in PublicProfile ← Original
2. Extract to reusable UserCard with props-based API ← Chosen
3. Create page-specific cards (ProfileCard, ListingCard, etc.)

**Decision:** Extract profile header into reusable `UserCard` and `UserCardCompact` components that receive user data as props (purely presentational).

**Rationale:**
- Components can be reused on discover pages, search results, sidebars
- Props-based design allows any data source
- Parent components control ownership logic (`isOwnProfile`)
- Variants (full/compact) serve different contexts

**Consequences:**
- Created `src/components/users/` with UserAvatar, UserCard, UserCardCompact
- PublicProfile now imports and uses UserCard
- ~140 lines removed from PublicProfile.tsx

## 3. Comprehensive CurrentUser Global
**Type:** Strategic
**Topics:** architecture, auth, performance

**Trigger:** User asked why we weren't using the currentUser global for profile data. Discovered own profile still made redundant API calls.

**Options Considered:**
1. Keep currentUser "lean" (identity + capabilities only)
2. Fetch API for own profile (redundant but complete)
3. Augment currentUser to include stats, expertise, qualifications ← Chosen

**Decision:** Make CurrentUser comprehensive - load stats, expertise, and qualifications at login along with all other user data.

**Rationale:**
- Storage is abundant (localStorage ~5MB, user data <10KB)
- Network calls are expensive (latency, UX impact)
- User profile data is stable within a session
- Eliminates need for API calls when viewing own profile

**Consequences:**
- Updated `/api/me/full` to fetch stats, expertise, qualifications in parallel
- Added types and properties to CurrentUser class
- Own profile now renders instantly from global (no API call)

## 4. Own Profile Uses Global, Others Use API
**Type:** Implementation
**Topics:** architecture, performance

**Trigger:** Optimizing data flow after augmenting CurrentUser.

**Options Considered:**
1. Always fetch from API (consistent but wasteful)
2. Always use global (impossible for other users)
3. Hybrid: global for own profile, API for others ← Chosen

**Decision:** When viewing own profile (`/@{myHandle}`), use `getCurrentUser()` data directly. When viewing others' profiles, fetch from `/api/users/${handle}`.

**Rationale:**
- Own profile data already in memory - no network needed
- Other users' data must come from API
- Optimal balance of performance and freshness

**Consequences:**
- PublicProfile checks `isOwnProfile` before deciding data source
- Own profile: instant render from currentUser
- Others: fetch from API with loading state
