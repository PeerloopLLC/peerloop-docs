# Session Learnings - 2026-02-27

## 1. Playwright getByRole Name Matching is Substring-Based
**Topics:** testing

**Context:** The `/discover/teachers` page renders two h1 headings — "Teachers" (Astro template) and "Student-Teachers" (React component). `getByRole('heading', { name: 'Teachers', level: 1 })` matched both because "Teachers" is a substring of "Student-Teachers".

**Learning:** Playwright's accessible name matching is not exact by default. When a page has headings where one name is a substring of another (e.g., "Teachers" and "Student-Teachers"), use `{ exact: true }` to prevent ambiguous matches that cause strict mode failures.

**Pattern:**
```typescript
// BAD — matches both "Teachers" and "Student-Teachers"
page.getByRole('heading', { name: 'Teachers', level: 1 })

// GOOD — exact match only
page.getByRole('heading', { name: 'Teachers', level: 1, exact: true })
```

## 2. Sidebar Panel Overlay Blocks Element Visibility
**Topics:** testing, astro

**Context:** The AppLayout sidebar has auto-opening panels ("My Feeds", "Discover") that overlay main content. The course detail Teachers tab test failed because `getByRole('button', { name: 'Teachers' })` found the button but it was behind the sidebar panel.

**Learning:** When testing pages with AppLayout, sidebar panels can obscure main content. Scope interactive selectors to specific containers (`navigation`, `main`) or wait for a deeper element (like "What You'll Learn" heading) to confirm the full page has hydrated before interacting with tabs.

**Pattern:**
```typescript
// Scope to the specific navigation to avoid sidebar panel interference
const tabNav = page.getByRole('navigation', { name: 'Course sections' });
const teachersTab = tabNav.getByRole('button', { name: 'Teachers' });
```

## 3. Duplicate Elements Across Sidebar and Modal Cause Strict Mode Failures
**Topics:** testing

**Context:** The signup page has a "Create account" button both in the sidebar (for visitors) and inside the signup modal form. Using `page.getByRole('button', { name: /Create account/i })` matched both, causing the click to fail.

**Learning:** When modals or dialogs duplicate text/buttons from the page's base layout, scope to the dialog or a specific container. The pattern `page.getByRole('dialog').getByRole('button', ...)` reliably targets modal-only buttons.

---

## 4. Astro + React Island Pages Often Render Duplicate H1s
**Topics:** testing, astro

**Context:** Several pages (booking, teachers) have an H1 in the Astro template AND another H1 inside the React `client:load` component. Both render as level 1 headings, causing strict mode failures on `getByRole('heading', { name: '...', level: 1 })`.

**Learning:** In Astro island architecture, the static template and the React island can both render headings with the same text. Use `.first()` or scope to `main` when this pattern occurs. This is a recurring issue across the codebase — not a one-off.

---

## 5. Field Validation Errors Use Inline Text, Not role="alert"
**Topics:** testing

**Context:** Attempted to test signup form validation by looking for `role="alert"` after submitting with empty fields. The form uses inline `<p class="text-red-600">` for field-level validation errors, with `role="alert"` only for general API-level errors.

**Learning:** E2E validation testing is fragile when form components separate field-level errors (inline text) from API-level errors (`role="alert"`). For E2E, focus on form rendering and navigation rather than validation logic — leave validation to unit tests.
