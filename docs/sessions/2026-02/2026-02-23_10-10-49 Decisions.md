# Session Decisions - 2026-02-23

## 1. Two-Tier Moderator Model
**Type:** Strategic
**Topics:** auth, d1

**Trigger:** Client needs per-community moderation; single global `can_moderate_courses` flag is insufficient for day-to-day community feed oversight when Creators and S-Ts are away.

**Options Considered:**
1. Add `moderator` to `community_members.role` enum — Rejected (dual-role problem)
2. Separate `community_moderators` table ← Chosen (follows `student_teachers` pattern)
3. Per-course `course_moderators` table — Rejected (too granular; community is the right scope)

**Decision:** Two-tier model: Tier 1 (Global Moderator) uses existing `can_moderate_courses` flag for platform-wide policy enforcement. Tier 2 (Community Moderator) uses new `community_moderators` table for Creator-appointed, community-scoped oversight. Scope flows from community to course feeds via Community → Progression → Course chain.

**Rationale:** Community is the natural scope unit. Courses inherit moderation via the existing progression chain. Separate table avoids dual-role conflict and provides appointment metadata (appointed_by, revoked_by).

**Consequences:** Updated ROLES.md (full two-tier section), DB-SCHEMA.md (community_moderators table + moderation_actions/user_warnings doc gap fill), DECISIONS.md (§4), PLAN.md (MODERATION deferred block), stories-M-moderator.md (tier annotations).

## 2. Community Moderator Direct Appointment (Not Invite)
**Type:** Implementation
**Topics:** auth

**Trigger:** Deciding how Community Moderators are onboarded — reuse the email invite flow from Global Moderators, or a simpler path?

**Options Considered:**
1. Reuse moderator email invite flow — Rejected (unnecessary for known community members)
2. Direct appointment from member list ← Chosen

**Decision:** Community Moderators are appointed directly by the Creator from the community member list. No email invite flow. The user is already a known, registered community member.

**Rationale:** The invite flow exists for Global Moderators because the invitee might not have a Peerloop account. Community moderators are selected from people who are already active members — a direct appointment is simpler and friction-free.

**Consequences:** Future `POST /api/communities/:slug/moderators` endpoint creates row immediately (no pending/accept states). No email template needed for community moderator appointment.

## 3. Authority Map Lives in ROLES.md
**Type:** Strategic
**Topics:** docs-infra

**Trigger:** Need for a unified "who has authority at each level of the content hierarchy" reference. The Community → Progression → Course chain is documented across 4+ files, none of which show role annotations.

**Options Considered:**
1. New section in ROLES.md ← Chosen
2. Annotate RFC/CD-036 (historical RFC, not ideal for living reference)
3. New standalone `AUTHORITY-MAP.md` file (yet another file to maintain)

**Decision:** Added "Content Hierarchy & Authority Map" section to ROLES.md (between role definitions and Authorization Matrix). Includes hierarchy overview, authority-by-level table, per-feed moderation authority checklists, and scope inheritance SQL query.

**Rationale:** ROLES.md is already the role reference. Anyone looking at roles will also see the authority map. Avoids creating a separate file that duplicates role information.

**Consequences:** ROLES.md now serves three purposes: role definitions (§1-6), authority map (§7), and quick reference tables (§8-10).

**Options Considered:**

**Decision:** CREATOR-SETUP added to the ACTIVE table as next block after CURRENTUSER completes. Marked as "needs planning session" because it touches multiple areas (CreateCourseModal, Creator Studio, APIs, seed data). Five sub-sections: PLANNING, COMMUNITIES, PROGRESSIONS, COURSES-RETROFIT, STUDIO-UI.

**Rationale:** This is on the critical path for MVP — Genesis cohort Creators need to create communities and learning paths. A planning session is needed first because retrofitting course creation to require progression_id affects existing code across UI and API.

**Consequences:** PLAN.md updated with full CREATOR-SETUP block including open design questions. Will require a planning session before implementation begins.

## 5. BBB and S-T-CALENDAR Prioritized as Deferred 1-2
**Type:** Strategic
**Topics:** video

**Trigger:** Client request to track BBB session implementation and availability calendar enhancements in PLAN.md.

**Options Considered:**
1. Add to existing blocks (POLISH, MVP-GOLIVE) — Rejected (too significant)
2. New blocks at low priority — Rejected (these are critical path for Genesis cohort)
3. New blocks at deferred priority 1-2 ← Chosen

**Decision:** BBB at priority 1, S-T-CALENDAR at priority 2 in the deferred table. Previous blocks (RATINGS, FEEDS, etc.) renumbered 3-15. BBB has 5 sub-sections (ROUTING, RECORDINGS, ATTENDANCE, RESCHEDULING, TESTING). S-T-CALENDAR has 5 sub-sections (MONTH-VIEW, EXCEPTIONS, CREATOR-AS-ST, BOOKING-INTEGRATION, TESTING).

**Rationale:** A student who enrolls needs to book and attend a session — that's the core value proposition. Without BBB routing and S-T availability, enrollment is a dead end. The Creator-as-ST toggle is key to the flywheel: Creators teach initially, then step back as graduated S-Ts take over.

**Consequences:** PLAN.md deferred table now has 15 items. BBB and S-T-CALENDAR blocks are comprehensive with context audits showing what's already built vs what's missing.
