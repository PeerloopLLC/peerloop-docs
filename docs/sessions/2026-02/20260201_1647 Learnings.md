# Session Learnings - 2026-02-01

## 1. CurrentUser Instances Are Immutable
**Topics:** astro, auth

**Context:** User asked how the global `currentUser` would be updated if an admin changes a user's capabilities (e.g., `canTeachCourses`).

**Learning:** The `CurrentUser` class has all `readonly` properties. Once instantiated, individual properties cannot be modified. When external changes occur (admin grants capability, creator certifies ST), the client must fetch fresh data and create an entirely new `CurrentUser` instance to replace the old one.

**Pattern:**
```typescript
// CurrentUser properties are readonly - no setters
readonly canTeachCourses: boolean;
readonly canCreateCourses: boolean;

// To update, must create new instance
await refreshCurrentUser();  // Fetches from API, creates new instance
setUser(getCurrentUser());   // Get the new instance
```

---

## 2. Server-Side Validation is Security, Client State is UX
**Topics:** auth, astro

**Context:** Discussing implications of stale client state when capabilities change externally.

**Learning:** Stale `currentUser` data is a UX issue, not a security issue. The client state controls what UI elements are shown, but the server always validates permissions on API calls. A user with stale `canTeachCourses: true` might see teacher UI, but server-side checks will reject unauthorized actions.

**Pattern:** Design principle - client state is optimistic UI, server is authoritative. Document this clearly so developers don't rely solely on client-side checks for security.
