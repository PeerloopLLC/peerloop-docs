# Session Decisions - 2026-02-24

## 1. Open BBB in New Tab Instead of Same-Window Redirect
**Type:** Implementation
**Topics:** video, astro

**Trigger:** Blindside Networks blocks iframe embedding. Original code used `window.location.href` which navigated away entirely.

**Options Considered:**
1. `window.location.href` — navigate away to BBB ← Original
2. `window.open()` with new tab + polling ← Chosen
3. Iframe embed (blocked by Blindside)

**Decision:** Open BBB in a new browser tab via `window.open()`. Keep the PeerLoop session page open with an `in_session` state that polls `/api/sessions/:id` every 15 seconds for completion. When session ends (via webhook), transition to rating view automatically.

**Rationale:** This keeps the user's PeerLoop context intact. When they close the BBB tab, they return to our session page which already shows the post-session rating UI. Eliminates the "where do I go after the session?" problem.

**Consequences:** Added `in_session` ViewState to SessionRoom, added polling useEffect, added `renderInSessionScreen()` with rejoin button.

## 2. BBB_URL as Non-Secret in wrangler.toml, BBB_SECRET in CF Dashboard Only
**Type:** Implementation
**Topics:** video, cloudflare, deployment

**Trigger:** Needed to add BBB environment variables. Had to decide where each goes.

**Options Considered:**
1. Both in `.dev.vars` only — won't work for CF deployments
2. Both in wrangler.toml `[vars]` — exposes secret in git ← Rejected
3. BBB_URL in `[vars]`, BBB_SECRET in CF Dashboard ← Chosen

**Decision:** `BBB_URL` goes in `wrangler.toml` `[vars]` sections (top-level, production, preview) since the API endpoint URL is not a secret. `BBB_SECRET` (88-char shared secret) is set only in `.dev.vars` locally and CF Dashboard for deployed environments.

**Rationale:** Follows the existing pattern used by Stripe (publishable key in vars, secret key in dashboard) and Stream (API key in vars, API secret in dashboard). Keeps wrangler.toml committable.

**Consequences:** Added `BBB_URL` to all three `[vars]` sections in wrangler.toml. Added both to `.dev.vars` with placeholder secret.

## 3. Use CURRENT-BLOCK-PLAN.md for Multi-Session Block Tracking
**Type:** Strategic
**Topics:** workflow, cc-workflow

**Trigger:** BBB block spans 5 sub-blocks with ~25 items — too large for one session.

**Options Considered:**
1. Track entirely in PLAN.md ← Too noisy, mixes active work with deferred blocks
2. Use RESUME-STATE.md ← Session-specific, gets deleted on resume
3. Create `CURRENT-BLOCK-PLAN.md` at docs root ← Chosen

**Decision:** Create a `CURRENT-BLOCK-PLAN.md` file at the docs repo root for multi-session blocks. Contains all checkboxes, key files, progress summaries, and "next session" guidance. Deleted when block completes.

**Rationale:** PLAN.md is the master plan — good for overview but too long for detailed per-item tracking. RESUME-STATE.md is designed for single-session handoff. CURRENT-BLOCK-PLAN.md fills the gap: persistent across sessions, detailed enough to resume without re-reading code, deletable when done.

**Consequences:** Created the file with all 5 sub-blocks. Future sessions read it first and resume from the next unchecked item.
