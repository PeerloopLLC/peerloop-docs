# Session Decisions - 2026-02-28

## 1. Pseudo-Page Architecture for Shared Navigation
**Type:** Implementation
**Topics:** tooling, docs-infra

**Trigger:** Initial scanner hardcoded shared nav links directly into page rows and suppressed broken footer links. User pointed out this hides real broken links and proposed pseudo-pages as FROM-only rows.

**Options Considered:**
1. Hardcode shared nav sets, merge into page rows, suppress footer links (initial approach)
2. Model shared components as pseudo-pages with their own FROM rows ← Chosen

**Decision:** Shared navigation components (Footer, AppNavbar, AdminNavbar, DiscoverSlidePanel, FeedSlidePanel, UserAccountDropdown, AuthModals) are pseudo-pages — they get their own FROM rows in the TSV matrix but do not appear as TO columns. Page rows show inherited links with `nav`/`foot`/etc. labels at intersections.

**Rationale:** Pseudo-pages surface broken links in shared components (the Footer alone has 15 broken targets). They also make the data model honest — you can see exactly where a link comes from. The TSV becomes 73 rows (7 pseudo + 65 pages + header) × 66 columns (label + 65 page routes).

**Consequences:** The Broken Links table in PAGE-CONNECTIONS.md now shows all 26 broken targets with their sources (page or pseudo-page). Footer broken links are visible for the first time.

## 2. No Link Suppression — Surface Everything
**Type:** Implementation
**Topics:** tooling

**Trigger:** User noted that suppressing footer links hides real broken navigation that users would experience as Astro 404 pages.

**Options Considered:**
1. Suppress footer/marketing links and `/404` redirects (initial approach)
2. Only exclude truly non-navigational `<head>` metadata paths ← Chosen

**Decision:** Only `/favicon.svg` and `/og-image.png` are excluded (they're `<link rel="icon">` and `<meta>` tags, not navigation). Everything else — including footer links, `/404` redirects, and stale route references — is surfaced with ⚠️ markers.

**Rationale:** The scanner's value is finding connection problems. Hiding broken links defeats the purpose. The 26 broken targets discovered include 15 footer links, stale pre-refactor routes (`/dashboard/learning`, `/workspace`), and possible bugs (`/course/[id]` using id instead of slug).

**Consequences:** PAGE-CONNECTIONS.md now has a dedicated "Broken Links" section at the top showing all 26 broken targets with their referencing sources.
