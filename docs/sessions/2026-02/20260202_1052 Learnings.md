# Session Learnings - 2026-02-02

## 1. Astro ViewTransitions Renamed to ClientRouter in v5
**Topics:** astro

**Context:** Implemented SPA-like navigation with Astro View Transitions. During code quality checks, `npx astro check` reported deprecation warnings.

**Learning:** In Astro 5.x, the `<ViewTransitions />` component was renamed to `<ClientRouter />` to clarify its role. The functionality is identical - only the name changed. The old name still works but shows deprecation warnings.

**Pattern:**
```astro
// Astro 4.x (deprecated)
import { ViewTransitions } from 'astro:transitions';
<ViewTransitions />

// Astro 5.x (current)
import { ClientRouter } from 'astro:transitions';
<ClientRouter />
```

## 2. React Hydration Mismatch with Auth-Dependent UI
**Topics:** astro, auth

**Context:** AppNavbar shows different content based on user authentication state. Server can't access localStorage, so it renders a "loading" skeleton. Client has cached user data, so it tries to render the actual menu.

**Learning:** When server and client render different HTML due to auth state, React throws a hydration mismatch error. The solution is a two-phase approach: match server on first render (show skeleton), then update after hydration completes using `useEffect`.

**Pattern:**
```tsx
// Hydration-safe auth state
const [isHydrated, setIsHydrated] = useState(false);
const [user, setUser] = useState<User | null>(null);

// Match server on first render
const authStatus = isHydrated ? networkState.authStatus : 'loading';

useEffect(() => {
  // Load from cache after hydration
  setUser(getCurrentUser());
  setIsHydrated(true);
}, []);
```

## 3. Dev Mode vs Production Performance Gap
**Topics:** astro, workflow

**Context:** On MBA-2017, initial page load showed skeleton for 3-4 seconds. User concerned about app being slow.

**Learning:** Astro dev mode is significantly slower than production due to: no bundling, no minification, source maps, HMR overhead. Testing with `npm run preview` (production build served locally) showed much faster load times. Dev mode performance on older hardware is not representative of production.

**Pattern:**
```bash
# Dev mode (slow, for development)
npm run dev

# Production preview (fast, for testing)
npm run build && npm run preview
```

## 4. View Transitions transition:persist Requires Same-Layout Navigation
**Topics:** astro

**Context:** User tested View Transitions by navigating to 404 page (outside app) then hitting BACK.

**Learning:** `transition:persist` only works for navigations between pages using the same layout with View Transitions enabled. Navigating to a page without View Transitions (like Astro's default 404) and hitting BACK triggers a full page reload - the persisted component unmounts and remounts fresh.

## 5. getCurrentUser() Should Auto-Hydrate from Cache
**Topics:** auth

**Context:** Initial implementation had `getCurrentUser()` just reading from global, requiring explicit initialization. This caused delays in showing cached user data.

**Learning:** `getCurrentUser()` should proactively check localStorage and hydrate the global if not already set. This allows any component to call it and get cached data immediately, without waiting for an initialization flow.

**Pattern:**
```ts
export function getCurrentUser(): CurrentUser | null {
  if (typeof window === 'undefined') return null;

  // Return from global if exists
  if (window.__peerloop?.currentUser) {
    return window.__peerloop.currentUser;
  }

  // Auto-hydrate from localStorage
  const cached = loadFromCache();
  if (cached) {
    ensureGlobal();
    const user = new CurrentUser(cached);
    window.__peerloop!.currentUser = user;
    window.__peerloop!.networkState.authStatus = 'authenticated';
    return user;
  }

  return null;
}
```
