### `.env` Symlink and Local Dev Breakage

**User:** Pointed out `.env` symlink to `.dev.vars` might break since non-secrets were removed.

**Claude:** Investigated both code paths. Found all removed vars use `locals.runtime.env` (not `import.meta.env`). Tested with `wrangler pages dev` — confirmed wrangler.toml vars reach runtime, BUT top-level `[vars]` loaded production values (wrong Stream keys). Also discovered MBA-2017 would break since Node adapter can't read wrangler.toml. Identified these as regressions.

### Revert Decision

**User:** Asked if the reorganization needed to be reverted.

**Claude:** Honestly assessed that the original `.dev.vars` setup was not deficient — it worked on both machines and all environments. The wrangler.toml changes traded one form of cognitive load for another with more edge cases. Recommended reverting wrangler.toml and `.dev.vars` changes while keeping the valuable docs and `.prod.vars`.

### Revert Executed

**Claude:** Restored `wrangler.toml` to bindings-only (no `[vars]`). Restored `.dev.vars` with all variables. Updated tech-026, vendor docs, and `.dev.vars.example` to reflect the simpler setup.

### CF Dashboard Doesn't Allow Non-Secret Env Vars

**User:** Corrected that CF no longer allows plain-text env vars in the dashboard when wrangler.toml exists.

**Claude:** Verified via CF docs — confirmed the dashboard locks out plain-text vars when wrangler.toml is the source of truth. Only encrypted secrets can be added via dashboard.

### `.env.production` Considered and Rejected

**User:** Asked if `.env.production` (committed to git) could carry non-secrets to CF.

**Claude:** Explained the build-time vs runtime distinction. `.env.production` would be available during `astro build` via `import.meta.env`, but NOT at the Workers edge runtime via `locals.runtime.env` where the code actually reads these vars. Rejected as a solution.
