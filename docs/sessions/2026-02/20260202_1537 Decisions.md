# Session Decisions - 2026-02-02

## 1. Consolidate Bio Fields to bio_short and bio_full
**Type:** Implementation
**Topics:** d1, astro

**Trigger:** Discovered three bio fields in schema (`bio`, `bio_short`, `bio_full`) with different endpoints using different fields, causing data divergence.

**Options Considered:**
1. Keep single `bio` field, derive short version by truncation
2. Keep `bio_short` and `bio_full`, remove `bio` ← Chosen
3. Keep all three for backwards compatibility

**Decision:** Remove `bio` column entirely. Use `bio_short` (150 chars) for taglines/cards and `bio_full` (2000 chars) for profile pages.

**Rationale:**
- Explicit separation prevents confusion about which field to use
- Truncation creates poor UX (cut-off sentences)
- User can craft different messages for different contexts
- Matches common patterns (Twitter bio vs LinkedIn About)

**Consequences:**
- Schema updated to remove `bio` column
- `/api/me/full` returns both `bioShort` and `bioFull`
- `/api/me/profile` edits both fields separately
- ProfileSettings component now has "Tagline" and "Full Bio" fields

## 2. Remove Deprecated Role Flags from Schema
**Type:** Implementation
**Topics:** d1, auth

**Trigger:** Schema had DEPRECATED role flags (`is_student`, `is_student_teacher`, `is_creator`, `is_moderator`) alongside capability flags, causing confusion about which to use.

**Options Considered:**
1. Keep deprecated flags for backwards compatibility
2. Remove deprecated flags, keep only capabilities ← Chosen
3. Create migration path with deadline

**Decision:** Remove `is_student`, `is_student_teacher`, `is_creator`, `is_moderator` from schema entirely. Keep only `is_admin` (global admin is not course-specific).

**Rationale:**
- Role flags are derived from course relationships (e.g., being in `student_teachers` table makes you an ST for that course)
- Capability flags (`can_teach_courses`) represent admin-controlled permissions
- Having both creates confusion and code that uses the wrong one
- Comments saying "deprecated" don't prevent usage

**Consequences:**
- Schema and seed data updated
- `/api/me/profile` now returns capabilities instead of role flags
- Components must derive roles from course relationships, not user flags

## 3. Keep title Field as Job Title
**Type:** Implementation
**Topics:** d1

**Trigger:** Initially planned to remove `title` along with `bio`, but realized it serves a different purpose.

**Options Considered:**
1. Remove `title` (fold into bio_short)
2. Keep `title` as distinct job title field ← Chosen

**Decision:** Keep `title` field in schema. It represents the user's job title/profession, distinct from bio content.

**Rationale:**
- Job title is structured data (e.g., "Software Engineer", "Marketing Manager")
- Bio is narrative text ("I help people learn...")
- Title appears in different UI contexts (cards, search, profile header)
- Removing would lose semantic meaning

**Consequences:**
- `title` added to CurrentUser global
- ProfileSettings shows "Job Title" label (clarified from just "Title")

## 4. CurrentUser Global Should Include All User Profile Fields
**Type:** Strategic
**Topics:** astro, auth

**Trigger:** CurrentUser was missing `title`, `bioFull`, social links, and `privacyPublic` that were available in the profile endpoint, requiring extra API calls.

**Options Considered:**
1. Keep CurrentUser minimal, fetch profile data separately when needed
2. Include all user fields in CurrentUser ← Chosen
3. Create separate "profile" slice alongside CurrentUser

**Decision:** `/api/me/full` should return ALL user profile fields so CurrentUser is the single source of truth for the logged-in user's data.

**Rationale:**
- Avoids duplicate fetches when displaying user info
- Single source of truth reduces data sync issues
- Payload size increase is minimal (few extra text fields)
- Components shouldn't need to know which endpoint has which field

**Consequences:**
- `UserIdentity` type expanded with all profile fields
- CurrentUser class now has `title`, `bioFull`, `website`, `location`, social links, `privacyPublic`
- Any component can read user data from CurrentUser without additional fetches
