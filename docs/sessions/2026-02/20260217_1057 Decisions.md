# Session Decisions - 2026-02-17

## 1. Stretched-Link Pattern as Standard for Clickable Cards with Secondary Links
**Type:** Implementation
**Topics:** astro, react, ui

**Trigger:** CourseCard had nested `<a>` tags causing hydration errors. Needed a pattern for "whole card is clickable but has a secondary link inside."

**Options Considered:**
1. Inner `<a>` → `<span role="link">` with `onClick` + `window.location.href`
2. Stretched-link pattern: outer `<div>`, primary `<a>` with `after:absolute after:inset-0`, secondary `<a>` with `relative z-10` ← Chosen
3. Outer `<a>` with `onClick` on inner element using `stopPropagation()`

**Decision:** Use the stretched-link pattern for any card component that needs a primary clickable area with secondary links inside.

**Rationale:** Only option that preserves all native link behaviors (right-click, middle-click, status bar preview, crawlability, View Transitions) for both primary and secondary links. The `<span>` approach lost 5 key behaviors; the `stopPropagation` approach still has invalid nested `<a>` HTML.

**Consequences:** Applied to CourseCard and NotificationsList. Sets the pattern for any future card-with-secondary-link components.

## 2. Use `navigate()` for Post-API-Call Internal Redirects, `window.location.href` for Auth Boundaries
**Type:** Strategic
**Topics:** astro, auth

**Trigger:** Codebase audit found 15+ `window.location.href` instances. Needed a clear policy for when to use which navigation method.

**Options Considered:**
1. Replace all `window.location.href` with `navigate()` or `<a>` tags
2. Keep all as `window.location.href` (status quo)
3. Split by context: `<a>` for static links, `navigate()` for programmatic internal navigation, `window.location.href` only for auth boundaries and external URLs ← Chosen

**Decision:** Three-tier navigation strategy:
- **`<a href>`** — Default for all navigable UI elements (links, buttons that navigate, menu items)
- **`navigate()`** — For programmatic internal navigation after API calls (e.g., post-booking redirect, post-mutation page refresh)
- **`window.location.href`** — Reserved for auth boundary crossings (login/logout) and external URL redirects (Stripe, BBB)

**Rationale:** Auth boundaries intentionally need full page reload to clear stale React state. All other internal navigation should use View Transitions for smooth UX. External URLs can't use Astro's router.

**Consequences:** Fixed 12 files in this session. Remaining 8 `window.location.href` instances are all auth-boundary or external-URL — correctly left as-is.

## 3. Add `href` Support to AdminActionMenu for Navigation Actions
**Type:** Implementation
**Topics:** ui, admin

**Trigger:** Admin action menus used `<button onClick={() => window.location.href = ...}>` for navigation items, losing right-click and View Transitions.

**Options Considered:**
1. Convert each caller's `onClick` individually to use `navigate()` ← Rejected (still a `<button>`, no right-click)
2. Add optional `href` to `ActionItem` interface, render `<a>` when present ← Chosen
3. Create separate `AdminLinkMenu` component

**Decision:** Extended `ActionItem` with optional `href` property. When present, `AdminActionMenu` renders `<a>` instead of `<button>`. Made `onClick` optional since href-based actions don't need it.

**Rationale:** Single change in the shared component benefits all admin pages. Actions now express intent — "navigate here" (`href`) vs "do this" (`onClick`). No breaking changes since `onClick` was made optional, not removed.

**Consequences:** StudentTeachersAdmin, CertificatesAdmin, and any future admin components can use `href` for navigation menu items with zero extra effort.

## 4. Skip Fix 4 Initially, Then Implement After UX Justification
**Type:** Strategic
**Topics:** workflow

**Trigger:** When presenting the prioritized fix list, Fix 4 (admin navigation) was initially recommended to skip due to low impact (admin-only, small team).

**Options Considered:**
1. Skip Fix 4 entirely (low effort-to-impact ratio) ← Initially recommended
2. Implement Fix 4 for UX consistency (right-click in admin) ← Chosen after discussion
3. Defer to a future session

**Decision:** Implemented Fix 4 after user confirmed UX value of right-click/middle-click support even in admin tools.

**Rationale:** Even admin users benefit from opening detail pages in new tabs while keeping context. The implementation was low-effort since the AdminActionMenu pattern made it a centralized change.

**Consequences:** All admin navigation actions now use real `<a>` tags. No `window.location.href` remains for internal navigation outside of auth boundaries.
