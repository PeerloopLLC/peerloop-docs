# Session Learnings - 2026-02-25

## 1. react-day-picker v9 Is the Best React Calendar for Custom Cell Rendering
**Topics:** astro, ui

**Context:** Needed a month-view calendar component where each date cell shows multiple indicators (time ranges, recurring badges, override marks, booked status). Evaluated 5+ packages.

**Learning:** react-day-picker v9 is uniquely suited for calendars where you need full structural control over date cells. It allows replacing the entire `Day` component (not just injecting content), supports `mode="multiple"` for non-contiguous multi-select out of the box, is headless/unstyled (works natively with Tailwind), and weighs ~22 kB. Other packages either can't replace cell structure (react-calendar), lack non-contiguous multi-select (FullCalendar), are too heavy (react-big-calendar ~170 kB), or only customize event rendering not cells (@schedule-x). The only gap is drag-select for ranges, which is implementable via custom mouse events on the Day component.

**Pattern:**
```tsx
// Full Day component replacement in react-day-picker v9
function AvailabilityDay(props: DayProps) {
  const { day, modifiers, ...tdProps } = props;
  return (
    <td {...tdProps}>
      <div>{day.date.getDate()}</div>
      {/* Arbitrarily complex custom rendering here */}
    </td>
  );
}

<DayPicker
  mode="multiple"
  components={{ Day: AvailabilityDay }}
/>
```

## 2. CURRENT-BLOCK-PLAN.md Pattern for Cross-Session Plan Persistence
**Topics:** cc-workflow, docs-infra

**Context:** Detailed block plans in PLAN.md can be lost when sessions run out of tokens or context gets compacted. Previously used this pattern for the BBB block (Sessions 276-279).

**Learning:** Extracting the active block into a standalone `CURRENT-BLOCK-PLAN.md` at project root ensures the full plan (design decisions, schema, checklists) survives session boundaries. The key elements are: a progress table at the top for quick status, preserved design decisions so future sessions don't re-debate them, and checkable items with session annotations. The file is deleted when the block completes. This pattern is now formalized as the global `/q-make-block-persistent` skill.

## 3. Availability Calendar Data Model: Recurring Rules + Overrides
**Topics:** d1, architecture

**Context:** Designing storage for an availability system that supports both weekly recurring patterns and date-specific overrides (vacation days, extra hours, blocked dates).

**Learning:** When recurring rules and date-specific overrides have fundamentally different data shapes (day_of_week vs specific date), they belong in separate tables rather than being overloaded with flags on a single table. The render logic becomes clean: expand recurring rules for the visible month → overlay overrides → show merged result. This is the same pattern used by Google Calendar internally (repeating events table + exception instances table).
