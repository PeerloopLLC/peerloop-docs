# Session Learnings - 2026-02-03

## 1. D1/SQLite Silently Returns No Rows for Non-Existent Columns
**Topics:** d1, testing

**Context:** The `/discover/creators` page wasn't showing any creators despite Guy and Gabriel being in the database with `privacy_public = 1`.

**Learning:** D1/SQLite doesn't error when a WHERE clause references a non-existent column - it simply returns zero rows. The query used `WHERE u.is_creator = 1` but the actual column is `can_create_courses`. No console errors appeared, making this a silent failure.

**Pattern:** When debugging "no results" issues, verify column names against the schema before assuming data problems. Consider using TypeScript types derived from schema to catch mismatches at compile time.

## 2. PAGES-MAP.md is Historical Reference, Not Source of Truth
**Topics:** workflow, astro

**Context:** Noticed route mismatches between DiscoverSlidePanel links (`/discover/*`) and PAGES-MAP.md routes (`/teachers`, `/creators`).

**Learning:** PAGES-MAP.md represents the original app design before significant client changes. It's now a reference for extracting page components and features, but the new routing scheme is being built separately. The `/discover/*` routes in the Discover panel are part of the new architecture.

**Pattern:** When migrating pages, use PAGES-MAP.md to find the original implementation and components, but place them at new routes according to the evolving architecture.

## 3. LandingLayout vs AppLayout Page Structure
**Topics:** astro

**Context:** Converting pages from `_src/pages` (LandingLayout) to new `src/pages` (AppLayout).

**Learning:**
- **LandingLayout** (public pages): Uses named slots (`<Fragment slot="content">`, `<Fragment slot="sidepanel">`) for dev mode toggle between PageSpecView and real content
- **AppLayout** (app pages): Uses a simple default slot, includes persistent AppNavbar sidebar (256px), no dev mode toggle

**Pattern:** When converting, extract only the `<Fragment slot="content">` contents, wrap in AppLayout with a header, and include SSR data fetching from the frontmatter.

## 4. Schema Uses Capability-Based Column Names
**Topics:** d1

**Context:** Query used `is_creator` but column doesn't exist.

**Learning:** The Peerloop schema uses capability-based column names (`can_create_courses`, `can_take_courses`, `can_teach_courses`) rather than role-based names (`is_creator`, `is_student`). This aligns with the platform model where users can have multiple roles simultaneously.

**Pattern:** When writing queries about user roles:
- Creator: `can_create_courses = 1`
- Student: `can_take_courses = 1`
- Teacher (S-T): `can_teach_courses = 1` or check `student_teachers` table

## 5. Enrollment Count Requires Subquery
**Topics:** d1

**Context:** StudentDirectory needed to show courses enrolled count, but `user_stats` table only has `courses_completed`.

**Learning:** The `courses_enrolled` count must be derived via subquery on the `enrollments` table since it's not denormalized in `user_stats`. The subquery filters for active enrollments (not cancelled/deleted).

**Pattern:**
```sql
COALESCE(
  (SELECT COUNT(*) FROM enrollments e
   WHERE e.student_id = u.id
   AND e.status IN ('enrolled', 'in_progress', 'completed')
   AND e.deleted_at IS NULL), 0
) as courses_enrolled
```
