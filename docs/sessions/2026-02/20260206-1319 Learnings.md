# Session Learnings - 2026-02-06

## 1. Cloudflare Pages wrangler.toml Environments Are Non-Inheritable
**Topics:** cloudflare, deployment

**Context:** While documenting the full env var flow across MacMini, MBA-2017, Preview, and Production environments, traced how `wrangler.toml` sections map to each deployment target.

**Learning:** In Cloudflare Workers/Pages, `[vars]`, `[[d1_databases]]`, and `[[r2_buckets]]` are "non-inheritable" — child environments (`[env.preview]`, `[env.production]`) do NOT inherit from the top-level section. Every environment must redeclare all bindings explicitly. This is why `wrangler.toml` has the same D1/R2/vars blocks repeated three times (top-level, preview, production).

## 2. CF Pages Dashboard Only Allows Encrypted Secrets When wrangler.toml Exists
**Topics:** cloudflare, deployment

**Context:** Documenting why non-secret values like `STREAM_API_KEY` appear in both `.dev.vars` and `wrangler.toml`.

**Learning:** When a `wrangler.toml` file exists in the project, the Cloudflare Pages dashboard only allows adding encrypted secrets — you cannot add plain-text environment variables through the UI. Non-secrets must be in `wrangler.toml [vars]` to reach Cloudflare. Locally, `.dev.vars` overrides `wrangler.toml [vars]`, so the duplication doesn't cause conflicts.

## 3. `.dev.vars` Overrides `wrangler.toml [vars]` Locally
**Topics:** cloudflare, workflow

**Context:** Traced the override chain to understand why local dev uses dev Stream app (1457190) when `wrangler.toml` top-level has prod values (app 1456912).

**Learning:** When wrangler runs locally (`npm run dev`), it loads both `wrangler.toml [vars]` and `.dev.vars`, with `.dev.vars` values winning on conflict. This is how dev-specific values (test Stripe keys, dev Stream app) override production values that must exist in `wrangler.toml` for deployment. The `.env` symlink to `.dev.vars` provides the same values to the MBA-2017 Node adapter via `process.env`.

## 4. Stripe Sandbox Is the Successor to Test Mode (Same Key Prefixes)
**Topics:** stripe

**Context:** Noticed Stripe dashboard shows three modes (Live, Sandbox, Test) and investigated the difference between Sandbox and Test mode.

**Learning:** Stripe migrated all accounts from "test mode" to "sandboxes" transparently. The API key prefixes (`sk_test_`, `pk_test_`) are unchanged — no migration needed. Sandboxes add: (1) up to 5 isolated environments per account (vs one shared test environment), (2) role-based access control, and (3) per-sandbox settings/webhooks. For solo developers with one test environment, the practical difference is zero.

## 5. D1 REST API Has Payload Size Limits for Batch Operations
**Topics:** d1, cloudflare

**Context:** Running `npm run db:setup:staging` to reset the staging database on Cloudflare.

**Learning:** The D1 REST API rejects batch operations that exceed its payload size limit. A single API call containing 53 DROP TABLE statements failed with error 7403. The `reset-d1.js` script handles this gracefully with a fallback: when batch drop fails, it retries each table individually, successfully dropping all 53 tables. This is a known D1 limitation that the reset script was designed to handle.
