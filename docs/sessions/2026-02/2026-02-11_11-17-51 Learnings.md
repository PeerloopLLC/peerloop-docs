# Session Learnings - 2026-02-11

## 1. AdminNavbar is Much Simpler Than AppNavbar by Design
**Topics:** astro, architecture

**Context:** Building an admin sidebar to match the app's AppNavbar pattern.

**Learning:** The AppNavbar is 577 lines because it manages auth state machines, sliding panels, role-based filtering, stale-while-revalidate user loading, and login modals. The AdminNavbar achieves the same UX (persistent sidebar, View Transitions, mobile responsive) in ~170 lines by omitting all auth complexity — admin pages can assume the user is authenticated and authorized. This demonstrates that understanding *what to omit* is as important as knowing what to include.

---

## 2. transition:persist Keys Must Be Unique Across Layout Boundaries
**Topics:** astro

**Context:** Both AppNavbar and AdminNavbar use `transition:persist` for SPA-like persistence.

**Learning:** Using distinct persist keys (`"app-navbar"` vs `"admin-navbar"`) ensures Astro doesn't try to preserve one navbar when navigating to a page that uses a different layout. Navigating from `/admin` to `/` causes a full page load (swaps the entire shell), which is the desired behavior since it changes the entire chrome from dark to light theme. Shared keys would cause hydration mismatches.

---

## 3. Lazy Expiration Pattern for D1 (No Cron Available)
**Topics:** d1, cloudflare

**Context:** The moderator_invites table has an `expires_at` field, but D1/Workers don't have native cron-like scheduled tasks for cleanup.

**Learning:** The "lazy expiration" pattern — running `UPDATE ... SET status = 'expired' WHERE status = 'pending' AND expires_at < datetime('now')` at the start of every list query — is the practical approach for D1. Data gets cleaned up whenever someone looks at it, rather than on a schedule. This is a common pattern in serverless databases without background jobs.

---

## 4. Dual-View API Pattern With Shared Stats
**Topics:** d1, architecture

**Context:** The moderators page needs to show both active moderators (users table) and invitations (moderator_invites table) — fundamentally different data sources.

**Learning:** Rather than creating two separate API endpoints, a single `GET /api/admin/moderators?view=moderators|invites` endpoint with a `view` query param keeps the frontend simple (one `fetchData()`, one pagination state) while the backend switches between queries. Stats are always returned regardless of view, so stat cards stay populated when switching tabs. This is the first tab-based admin component in the project.

---

## 5. Admin Page Conversion Pattern is Highly Mechanical
**Topics:** astro, workflow

**Context:** Converting 11 admin pages from `_src/` staging to `src/` live.

**Learning:** Each conversion follows an identical pattern: remove `PageSpecView` import, remove `pageCode` prop, remove named `slot="content"`, use default `<slot />`. The old pages averaged ~18 lines with the dual-slot PageSpecView system; the new pages are ~11 lines each. This mechanical nature means future conversions (for non-admin pages) could potentially be scripted.
