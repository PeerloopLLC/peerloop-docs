# Session Decisions - 2026-02-20

## 1. Docs-Primary Claude Code Architecture
**Type:** Strategic
**Topics:** workflow

**Trigger:** Code repo had 861 documentation files (~20.5 MB) mixed with application code. CLAUDE.md, session logs, commands, hooks, and all CC artifacts cluttered the code repo.

**Options Considered:**
1. Keep everything in one repo (status quo)
2. Move docs to a companion repo, run CC from there ← Chosen
3. Use git submodules for docs

**Decision:** Create `peerloop-docs` as the CC home repo. Launch CC from there with `--add-dir ../Peerloop` to reach the code repo. Symlinks in the code repo bridge build-time dependencies (`docs/` and `research/`).

**Rationale:** Option 2 cleanly separates concerns — docs grow independently of code, the code repo stays lean for CI/CD, and the docs repo doubles as an Obsidian vault. Submodules (option 3) add complexity without benefit since the repos don't need to be version-locked.

**Consequences:**
- Code repo reduced from 27.6 MB to 8.6 MB (69% reduction)
- All CC sessions must launch from `peerloop-docs/`
- `link-docs.sh` must be run once per machine after cloning
- Commands updated with `../Peerloop/` prefix for code paths
- `/q-commit` redesigned for dual-repo git workflow

## 2. Symlinks Over Copies for Build Dependencies
**Type:** Implementation
**Topics:** workflow, astro

**Trigger:** Code repo build scripts (`populate-page-metadata.ts`, `parse-all-pages.ts`) reference `docs/` and `research/` via `path.resolve` from project root.

**Options Considered:**
1. Modify build scripts to accept configurable paths ← Rejected
2. Symlink docs/ and research/ back into code repo ← Chosen
3. Copy files as a build step

**Decision:** Use symlinks (`Peerloop/docs → ../peerloop-docs/docs`) created by `scripts/link-docs.sh`. Add symlink paths to `.gitignore`.

**Rationale:** Symlinks are transparent to Node.js/Vite/Astro with zero code changes. Modifying scripts would add complexity and create divergence between local dev and CI. Copying would create stale data risks.

**Consequences:**
- Build and all 4891 tests pass without any script changes
- `link-docs.sh` required as a one-time setup step per machine
- CI/CD doesn't need docs (build/test work without the vault content)

## 3. Temporary Symlink Bridge for Root Planning Files
**Type:** Implementation
**Topics:** workflow

**Trigger:** After migrating DECISIONS.md, PLAN.md, etc. to the docs repo, skills referencing them by traditional root paths would fail.

**Options Considered:**
1. Update all skills to use absolute paths to docs repo ← Rejected
2. Symlink root files from code repo to docs repo ← Chosen
3. Accept breakage during transition

**Decision:** Add root-level symlinks (e.g., `Peerloop/DECISIONS.md → ../peerloop-docs/DECISIONS.md`) for transitional sessions. Consider making permanent in `link-docs.sh`.

**Rationale:** Symlinks are gitignored and invisible. Skills work without modification. Once all sessions launch from docs repo, these become unnecessary — but they provide a safety net for any accidental code-repo CC launch.

**Consequences:**
- `/q-learn-decide` and other skills work in this transitional session
- Decision on whether to make permanent deferred to next session
