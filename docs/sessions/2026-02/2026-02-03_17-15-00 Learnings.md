# Session 171 Learnings - 2026-02-03

## 1. Astro Client-Side vs Server-Side Data Fetching
**Topics:** astro, architecture

**Context:** Attempted to use `getSessionUser()` in Astro frontmatter for the `/@[handle]` page, but the function didn't exist - got runtime error.

**Learning:** This project uses client-side data fetching via API calls and a `currentUser` global, not server-side in Astro frontmatter. The `currentUser` global is initialized by AppNavbar on page load. Components should use `getCurrentUser()` from `@lib/current-user` rather than expecting server-side user data.

**Pattern:**
```tsx
// ❌ Wrong - trying to get user server-side in Astro
const user = await getSessionUser(Astro); // Doesn't exist

// ✅ Correct - get from client-side global
import { getCurrentUser } from '@lib/current-user';
const currentUser = getCurrentUser(); // Returns from window.__peerloop
```

## 2. Component Extraction for Reusability
**Topics:** ui, architecture

**Context:** Profile header section looked like a reusable card format that could be used for user listings on discover pages.

**Learning:** When extracting reusable components:
1. Make them props-based (no internal data fetching)
2. Export types for the data shape (`UserCardData`)
3. Provide variants for different contexts (full vs compact)
4. Parent components handle auth/ownership logic and pass `isOwnProfile`

**Pattern:**
```
src/components/users/
├── UserAvatar.tsx       # Shared avatar with fallback initial
├── UserCard.tsx         # Full card (receives UserCardData props)
├── UserCardCompact.tsx  # Compact variant for grids
└── index.ts             # Barrel export
```

## 3. Comprehensive vs Lean User Objects
**Topics:** architecture, auth, performance

**Context:** Discovered that viewing own profile was still making API calls because `currentUser` didn't have stats, expertise, or qualifications.

**Learning:** The "lean session user" pattern originated from server-side sessions where memory was precious. For modern SPAs with localStorage caching:
- Storage is abundant (~5MB limit, user data is <10KB)
- Network calls are expensive (latency, loading spinners)
- User profile data is stable within a session
- **Conclusion:** Load everything once at login, use everywhere

**Pattern:**
```typescript
// /api/me/full now fetches ALL user data in parallel:
const [user, enrollments, stCerts, courses, stats, expertise, qualifications] =
  await Promise.all([...]);

// Components use instantly - no more API calls for own profile
const currentUser = getCurrentUser();
if (currentUser.stats) { /* render stats */ }
```

## 4. Seed Data Column Mismatch
**Topics:** d1, database

**Context:** Added `location`, `linkedin_url`, `twitter_url`, `youtube_url` to schema and API, but they weren't showing on profiles.

**Learning:** When schema adds new columns, seed INSERT statements using explicit column lists may not include them. The INSERT was:
```sql
INSERT INTO users (id, email, ..., website, can_create_courses, ...) VALUES
-- Missing: location, linkedin_url, twitter_url, youtube_url
```

Use UPDATE statements to add test data for new columns without restructuring all INSERTs:
```sql
UPDATE users SET
  location = 'Tel Aviv, Israel',
  linkedin_url = 'https://linkedin.com/in/guyrymberg'
WHERE id = 'usr-guy-rymberg';
```

## 5. Progressive Disclosure on Public Profiles
**Topics:** ui, ux

**Context:** User asked why social links weren't showing when they were null in the database.

**Learning:** Public profile pages typically hide empty fields rather than showing blank placeholders. This keeps the UI clean and focused on what the user actually wants to share. The settings page shows all fields (with "Public" badges) so users know what *could* be shown, but the profile only displays what *is* filled in.

**Pattern:**
```tsx
// Only render if value exists
{profile.location && (
  <span className="...">
    <LocationIcon /> {profile.location}
  </span>
)}
```
