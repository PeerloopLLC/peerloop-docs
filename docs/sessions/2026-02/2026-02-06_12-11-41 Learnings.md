# Session Learnings - 2026-02-06 (Session 204)

## 1. CF Pages Dashboard Locks Out Plain-Text Env Vars When wrangler.toml Exists
**Topics:** cloudflare, deployment

**Context:** Attempted to simplify env var management by removing non-secrets from `wrangler.toml [vars]` and relying on the CF dashboard for all env vars.

**Learning:** When a `wrangler.toml` file exists in a CF Pages project, the dashboard becomes read-only for non-secret configuration — it only allows adding **encrypted secrets**. Non-secret environment variables (publishable keys, app IDs, client IDs) **must** be defined in `wrangler.toml [vars]` to reach the Cloudflare runtime. There is no dashboard alternative.

---

## 2. CF Pages Non-Inheritable Key Rule Requires Vars in All Environment Sections
**Topics:** cloudflare, deployment

**Context:** Added `[vars]` to top-level `wrangler.toml` but not to `[env.production]` or `[env.preview]` sections.

**Learning:** `vars`, `d1_databases`, and `r2_buckets` are all non-inheritable in CF Pages. If any one non-inheritable key is overridden in an environment section (e.g. `[env.preview.d1_databases]`), ALL non-inheritable keys must also be specified in that section. Since Peerloop already has per-environment D1/R2 bindings, `[vars]` must appear in all three places: top-level, `[env.production.vars]`, and `[env.preview.vars]`.

---

## 3. `.dev.vars` Values Override `wrangler.toml [vars]` During Local Dev
**Topics:** cloudflare, deployment

**Context:** Worried that putting production values in top-level `wrangler.toml [vars]` would cause local dev to use wrong Stream/Stripe keys.

**Learning:** Wrangler merges both sources for local dev, with `.dev.vars` taking precedence. Verified via `wrangler pages dev` output which showed "Using vars defined in .dev.vars". So production values in `[vars]` are safely overridden by dev values in `.dev.vars` locally.

---

## 4. `.env` Files Don't Reach Cloudflare Runtime
**Topics:** cloudflare, deployment, astro

**Context:** Considered using `.env.production` (committed to git) to carry non-secrets to Cloudflare.

**Learning:** There are two distinct contexts in CF Pages: build-time (where `astro build` runs and Vite reads `.env` files) and runtime (Workers edge, where server-side code executes). `.env` files are only available at build-time via `import.meta.env`. The Workers runtime only has access to `wrangler.toml` bindings/vars and encrypted dashboard secrets. Since Peerloop accesses env vars at runtime via `locals.runtime.env`, `.env` files are irrelevant for Cloudflare.

---

## 5. `wrangler pages secret put` Has No `--env` Flag
**Topics:** cloudflare, deployment

**Context:** Documented deployment workflow for secrets.

**Learning:** Unlike `wrangler secret put` (Workers) which supports `--env preview`, the Pages variant `wrangler pages secret put` has no environment flag — it only sets production secrets. Preview secrets must be set via the CF dashboard. Simplest approach: use the dashboard for all secrets (both preview and production) to maintain one consistent workflow.

**See:** [Known Pages limitation - wrangler-action#354](https://github.com/cloudflare/wrangler-action/issues/354)

## 6. MBA-2017 Node Adapter Cannot Read wrangler.toml
**Topics:** cloudflare, deployment

**Context:** Explored moving non-secrets out of `.dev.vars` into `wrangler.toml [vars]` only.

**Learning:** The MBA-2017 uses the Node adapter (not Cloudflare adapter) because macOS 12 can't run workerd. The Node adapter doesn't use wrangler at all — it relies on `process.env` populated from `.env` (symlinked to `.dev.vars`) via `astro.config.mjs`'s `loadEnv()`. Any env var not in `.dev.vars` is invisible to MBA-2017. This means non-secrets must remain in `.dev.vars` even though they're also in `wrangler.toml`.

## 7. Vendor Naming Conventions Reveal Secret vs Non-Secret
**Topics:** stripe, stream, auth

**Context:** Auditing which env vars are actually secrets.

**Learning:** Every major vendor follows a consistent naming pattern that signals secret vs non-secret: Stripe uses `pk_` (publishable) vs `sk_` (secret). Stream uses `API_KEY` (public) vs `API_SECRET` (secret). OAuth uses `CLIENT_ID` (public, in browser redirect URLs) vs `CLIENT_SECRET`. Resend's `API_KEY` is server-only (secret) because the service has no client-side component. Rule of thumb: if the value appears in client-side JS or browser URLs, it's not a secret.
