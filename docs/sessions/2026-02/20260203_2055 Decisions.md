# Session Decisions - 2026-02-03

## 1. URL Routing Convention: Singular Resource Routes
**Type:** Architecture
**Topics:** astro, workflow

**Trigger:** Converting pages from LandingLayout to AppLayout raised the question of what URL pattern to use for individual resource pages.

**Options Considered:**
1. `/discover/creators/[handle]` - Nested under discover section
2. `/creators/[handle]` - Plural collection style
3. `/creator/[handle]` - Singular resource style ← Chosen

**Decision:** Use singular nouns for individual resource routes, keeping plural for collection/listing routes.

**Rationale:**
- REST convention: collection (plural) vs resource (singular)
- Shorter, more shareable URLs for profile pages
- Profile pages are "destinations" not part of discovery flow
- Pattern: `/discover/creators` (browse) → `/creator/brian` (view)

**Consequences:**
- Created pages at `/creator/[handle]`, `/teacher/[handle]`, `/course/[slug]`
- Updated 40+ component links across the codebase
- Consistent pattern for future resource pages

## 2. Update All Course Sub-Routes Under Singular Path
**Type:** Implementation
**Topics:** astro

**Trigger:** After creating `/course/[slug]`, needed to decide what to do with related routes like `/book` and `/learn`.

**Options Considered:**
1. Keep sub-routes at old location (`/courses/[slug]/book`)
2. Move all sub-routes to new location ← Chosen
3. Skip link updates for now

**Decision:** Move all course-related sub-routes to the singular `/course/[slug]/` namespace.

**Rationale:**
- Consistency - all course pages under same base path
- Avoids mixed routing confusion
- API routes stay at `/api/courses/` (separate namespace)

**Consequences:**
- `/course/[slug]` - course detail
- `/course/[slug]/book` - session booking
- `/course/[slug]/learn` - course content
- Updated 34 link occurrences across 19 files

## 3. AppLayout Max-Width Guidelines by Page Type
**Type:** Implementation
**Topics:** astro

**Trigger:** Converting multiple page types needed consistent container width decisions.

**Options Considered:**
1. Same width for all pages
2. Width based on content type ← Chosen
3. Let components decide their own width

**Decision:** Use content-based max-width classes:
- `max-w-6xl` - Grid/browse pages (creators, teachers, courses list)
- `max-w-5xl` - Content-heavy detail pages (course detail)
- `max-w-4xl` - Profile pages (creator, teacher profiles)
- `max-w-7xl` - Learning interface (needs full width for sidebar)

**Rationale:** Different content types have different optimal reading/viewing widths. Grids need more space, profiles are focused content.

**Consequences:** Established pattern for all future page conversions.
