## 2. Remove Mock Stripe IDs from Base Seed Data
**Type:** Implementation
**Topics:** stripe, d1

**Trigger:** Sarah Miller had `acct_mock_sarah` and Marcus Thompson had `acct_mock_marcus` in `0001_seed_dev.sql` — fake IDs that pass `IS NOT NULL` checks but fail against the real Stripe API.

**Options Considered:**
1. Keep mock IDs (works for unit tests) ← Status quo
2. Replace with NULL (honest "not onboarded" state) ← Chosen
3. Replace with real sandbox IDs inline

**Decision:** Changed Sarah and Marcus from `'acct_mock_sarah'/'acct_mock_marcus', 'active', 1` to `NULL, NULL, 0` in `0001_seed_dev.sql`.

**Rationale:** Mock IDs create a phantom state — code thinks Stripe is configured, but runtime calls fail. NULL honestly represents "not yet onboarded." Unit tests create their own test data and never depend on the dev seed (verified: all 56 Stripe tests pass after change).

**Consequences:** Base dev setup now has all users with NULL Stripe fields. Real IDs come from `0002_seed_stripe.sql` (opt-in).

## 3. Defer Automated Stripe Integration Tests
**Type:** Strategic
**Topics:** testing, stripe

**Trigger:** Explored `stripe trigger` and API-driven checkout sessions as automated integration tests.

**Options Considered:**
1. Build scripted integration test (`npm run test:stripe:integration`) that creates real checkout sessions and verifies webhook processing
2. Use `stripe trigger` with `--override` for simulated events
3. Rely on manual testing with seed data + `stripe listen` ← Chosen
4. Build `/api/dev/simulate-checkout` dev-only endpoint (deferred for later)

**Decision:** Do not build automated Stripe integration tests now. Manual testing with the new seed data infrastructure is more practical at current project scale.

**Rationale:** Automated integration tests require 4 running services (dev server, `stripe listen`, network, DB), are fragile to infrastructure state, and provide the same coverage as manual testing. The existing 56 unit tests (mocked Stripe) cover business logic. Automated integration tests make sense post-launch when CI/CD exists and multiple developers can break payment flows.

**Consequences:** `stripe trigger` CLI noted as a manual tool for edge-case testing (disputes, payout failures). `/api/dev/simulate-checkout` added to PLAN.md deferred items for future consideration.

## 5. Cross-Reference Pre-Launch Deferred Items in Go-Live Checklists
**Type:** Strategic
**Topics:** docs-infra

**Trigger:** Added Stripe Event Polling and Extended Self-Healing to PLAN.md deferred items table. These are pre-launch requirements, but the deferred items table and the MVP-GOLIVE checklist are in different sections of the same file — items could be forgotten when executing the go-live sequence.

**Options Considered:**
1. Add to deferred items table only (risk: overlooked at go-live)
2. Add to go-live checklist only (risk: no rationale captured)
3. Add to both with cross-references ← Chosen

**Decision:** When a deferred item is required before production launch, add it to BOTH the deferred items table (with rationale) AND the relevant MVP-GOLIVE subsection (as a checklist item with "see Deferred Items" reference).

**Rationale:** The deferred items table tracks *why* something was deferred. The go-live checklists track *what* to execute. Cross-referencing ensures nothing falls through the gap between "we decided to defer this" and "we need to do this before launch."

**Consequences:** Added to PLAYBOOK.md section 5 (Documentation Conventions). Applied immediately to MVP-GOLIVE.STRIPE with 2 pre-launch hardening checklist items.
