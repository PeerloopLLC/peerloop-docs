# Session Learnings - 2026-02-06

## 1. Astro `set:html` Enables Data-Driven SVG Without JS
**Topics:** astro

**Context:** Several Astro pages (discover/index, discover/communities, community/index) used a local `icons: Record<string, string>` object to store raw SVG path strings, then rendered them via `<svg set:html={icons[section.icon]} />`. This was a manual version of the pattern we formalized.

**Learning:** Astro's `set:html` directive is the key to building a zero-JS icon component. It injects raw HTML strings into the SVG element at build time, producing identical output to inline SVGs. The `Icon.astro` component leverages this: it looks up path data from a TypeScript registry and renders via `set:html={entry.paths}`. The compiled output is a plain `<svg>` with no client-side framework overhead.

**Pattern:**
```astro
<!-- Icon.astro compiles away entirely — output is just <svg>...</svg> -->
<svg class={className} fill={fill} stroke={stroke} viewBox={viewBox} set:html={entry.paths} />
```

## 2. Brand Icons Require Fill/Stroke Override in Registry
**Topics:** astro

**Context:** Most icons use `fill="none" stroke="currentColor"` (Heroicons outline style), but brand icons (Twitter bird, LinkedIn square, GitHub) use `fill="currentColor" stroke="none"`. When migrating Footer.astro's brand icons, they needed different SVG attributes than the default.

**Learning:** The icon path registry needs per-entry override capability for `fill` and `stroke`. The solution is a simple default-with-override pattern: entries only specify non-default values, and the Icon.astro component falls back to defaults (`fill="none"`, `stroke="currentColor"`). This keeps the common case clean while supporting fill-based brand icons.

## 3. Batch 8 Dynamic Icons Require Deleting Local Record Objects
**Topics:** astro

**Context:** Three files (discover/index.astro, discover/communities.astro, community/index.astro) had local `const icons: Record<string, string> = { ... }` objects in their frontmatter, used with `set:html={icons[variable]}`. Migrating these wasn't just SVG replacement — the entire local data structure had to be removed.

**Learning:** When migrating dynamic icon patterns, the cleanup involves: (1) deleting the local `icons` Record from frontmatter, (2) replacing `<svg ... set:html={icons[x]} />` with `<Icon name={x} />`, and (3) ensuring the variable passed to `name` matches the registry keys. The `Icon.astro` component's `name` prop accepts `string` (not just `IconName`) to support dynamic values from data (e.g., `community.icon`).

## 4. Replace-All with Exact Indentation Catches Most But Not All Instances
**Topics:** workflow

**Context:** When replacing identical SVG patterns (e.g., chevron-right breadcrumb separators), `replace_all: true` worked well for patterns with consistent indentation. However, in community/index.astro, one chevron-right had 8 spaces of indentation while others had 10 spaces, so the replace-all missed it.

**Learning:** After bulk replacements, always run `grep '<svg' src/**/*.astro` to catch stragglers. Indentation differences between seemingly identical SVG blocks are common — the same icon used in a navigation breadcrumb vs. a CTA button will have different surrounding indentation. A final grep verification is essential.
