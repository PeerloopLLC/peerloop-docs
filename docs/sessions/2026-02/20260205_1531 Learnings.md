# Session Learnings - 2026-02-05 (Session 192)

## 1. OAuth Callbacks Redirected to Non-Existent Route
**Topics:** auth, astro

**Context:** While auditing the redirect handling section of `tech-021-url-routing.md`, discovered that both GitHub and Google OAuth callback handlers redirected to `/dashboard` on success — a route that doesn't exist as a page.

**Learning:** When route conventions are refactored (e.g., `/dashboard/*` → bare routes like `/learning`, `/teaching`), OAuth callback redirect destinations can be overlooked since they're buried in API endpoint files, not page files. These should be audited whenever the routing architecture changes.

**Pattern:** OAuth callbacks now redirect to `/` (homepage), which adapts content based on auth state — the safest universal post-login landing.

## 2. Course Tab Pages Follow a Boilerplate Pattern
**Topics:** astro, d1

**Context:** When adding Teachers and Resources tabs to the course page, each required a new Astro page file (`teachers.astro`, `resources.astro`) that is nearly identical to `index.astro` and `feed.astro` — same data fetching, same component props, only `initialTab` differs.

**Learning:** The URL-aware tab pattern in Astro requires a dedicated `.astro` page file per tab route for SSR/bookmarkability, but the actual tab switching happens client-side via `history.pushState`. This means ~200 lines of duplicated data-fetching boilerplate per tab page. A future refactoring opportunity would be to extract the shared course data-fetching into a utility function that all tab pages call.

## 3. Server-Side vs Client-Side Data for Course Tabs
**Topics:** astro, d1, cloudflare

**Context:** The Teachers tab passes S-T data server-side from the Astro page (query runs at SSR time). The Resources tab fetches data client-side via `fetch('/api/courses/:id/resources')`.

**Learning:** Choose client-side fetch when the API already encapsulates complex access control (enrollment gating) and resource transformation (R2 download URL generation). Choose server-side when the data is simple joins with no special URL handling. The Resources API handles enrollment checks, R2 key → download URL mapping, and public/private filtering — duplicating all that in every Astro page would be error-prone.

## 4. Implementation Status Docs Drift Quickly
**Topics:** workflow

**Context:** The `tech-021-url-routing.md` Implementation Status table was significantly out of date — Discovery showed 6/7 when all 7 were done, Personal showed 2/6 when all 6 were done, and Auth/Marketing/Admin categories weren't tracked at all.

**Learning:** Implementation status tables in architecture docs should be verified against the actual filesystem whenever the doc is opened for editing. A quick `ls src/pages/` tree comparison catches drift that accumulates over many sessions. Consider adding a verification date ("Last verified: Session N") to make staleness visible.

## 5. The `is_public` Flag Enables Resource Preview for Non-Enrolled Visitors
**Topics:** d1, astro

**Context:** When designing the Resources tab, needed to decide what unauthenticated/non-enrolled visitors should see.

**Learning:** The `session_resources.is_public` column (already in the schema) enables a "preview" pattern where creators mark select resources as public teasers. The Resources tab leverages this: enrolled students see everything, visitors see only `is_public` resources plus an enrollment prompt. No schema changes were needed — the flag was designed for exactly this use case.
