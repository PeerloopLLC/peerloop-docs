# Session Decisions - 2026-02-04

## 1. ST Analytics Uses User IDs, Not ST Record IDs
**Type:** Implementation
**Topics:** d1, astro

**Trigger:** ST analytics API endpoints were failing with "no such column: st_id" errors.

**Options Considered:**
1. Fix the API code to match the schema (use userId) ← Chosen
2. Modify the schema to add st_id columns to sessions/enrollments
3. Discuss design intent first

**Decision:** Fix the API code to match the existing schema pattern, using userId directly through `teacher_id` and `student_teacher_id` columns.

**Rationale:** The schema design is sound - the course context is already captured via enrollment's course_id. This matches how Creator Analytics works (filters by `creator_id = userId`). Adding redundant st_id columns would complicate the schema without benefit.

**Consequences:** Updated all three ST analytics endpoints (`/api/me/st-analytics`, `/api/me/st-analytics/sessions`, `/api/me/st-analytics/students`) to filter by userId directly.

## 2. Completion Ratings Only Feed Into ST Rating
**Type:** Strategic
**Topics:** d1, workflow

**Trigger:** Needed to decide what ratings should appear on ST profiles.

**Options Considered:**
1. Session average (average of all session ratings for this ST+course)
2. Completion rating only (rating given when student finishes course) ← Chosen
3. Weighted blend (e.g., 70% completion + 30% session avg)

**Decision:** Only completion reviews feed into `student_teachers.rating`. Session ratings remain diagnostic-only and don't affect public ST ratings.

**Rationale:**
- Session ratings are noisy pulse checks, not overall evaluations
- Completion reviews reflect the full learning journey
- Matches how Airbnb/Coursera work (rate after experience ends)
- Provides clean, meaningful, comparable data

**Consequences:**
- Added `enrollment_reviews` table for completion ratings
- Added `rating` and `rating_count` columns to `student_teachers`
- Session ratings still go to `user_stats` (user-level aggregate)
- Clear separation between diagnostic and evaluative feedback

## 3. Two-Layer Review Capture: Immediate + Reminder
**Type:** Implementation
**Topics:** astro, workflow

**Trigger:** Needed to decide when/where to prompt students for completion reviews.

**Options Considered:**
1. Course Learn page (banner when completed)
2. Student Dashboard (reminder for unreviewed completions)
3. Dedicated review page (/course/[slug]/review)
4. Inline after last module (modal immediately) ← Chosen (Option D)
5. Dashboard reminder as backup ← Also Chosen (Option B)

**Decision:** Implement both Option D (immediate modal after last module completion) and Option B (amber reminder on dashboard for unreviewed completed courses).

**Rationale:**
- Option D captures reviews when experience is freshest
- Option B serves as gentle reminder for those who skipped
- Two-layer approach maximizes review capture without being annoying

**Consequences:**
- Created `CourseReviewModal` component
- Updated `CourseLearning` to show modal on course completion
- Updated `StudentDashboard` to show review CTA on completed courses without reviews
- API returns `has_review` boolean for completed enrollments

## 4. Required Comments for Completion Reviews
**Type:** Implementation
**Topics:** d1, workflow

**Trigger:** Determining validation rules for completion reviews.

**Options Considered:**
1. Optional comment (like session ratings)
2. Required comment (minimum 10 characters) ← Chosen

**Decision:** Completion reviews require a comment with minimum 10 characters.

**Rationale:**
- Completion reviews may be displayed publicly on ST profiles
- Required comments provide accountability and meaningful feedback
- Session ratings remain quick (optional comment) for different purpose

**Consequences:** API validates `comment.trim().length >= 10` on completion review submission.
