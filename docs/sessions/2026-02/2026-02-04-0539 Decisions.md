# Session Decisions - 2026-02-04

## 1. Activity vs Resource Route Namespace Convention
**Type:** Strategic
**Topics:** astro, workflow

**Trigger:** Inconsistency discovered between `/creator/earnings` (resource namespace) and `/creating/*` (activity namespace) for dashboard pages.

**Options Considered:**
1. Keep mixed: `/creator/earnings` alongside `/creating/studio`
2. Use activity namespace: All dashboard pages under `/creating/*` ŌåÉ Chosen
3. Use resource namespace: All pages under `/creator/*`

**Decision:** Dashboard and tool pages use activity namespaces (`/creating/*`, `/teaching/*`), while public profile pages use resource namespaces (`/creator/*`, `/teacher/*`).

**Rationale:** Activity namespaces (`/creating`, `/teaching`) communicate "what I'm doing" and group related tools together. Resource namespaces (`/creator/[handle]`) are for viewing others' public profiles. This creates intuitive, consistent navigation.

**Consequences:**
- Moved `/creator/earnings` ŌåÆ `/creating/earnings`
- Updated `CreatorDashboard.tsx` link
- Established pattern for future pages

## 2. Separate TEAR and CEAR Page Codes
**Type:** Implementation
**Topics:** workflow, stripe

**Trigger:** Discovered that page code `CEAR` was being used for ST earnings page, causing confusion.

**Options Considered:**
1. Keep CEAR for both ST and Creator earnings
2. Create separate codes: TEAR for ST, CEAR for Creator ŌåÉ Chosen
3. Use generic EARN code with role parameter

**Decision:** Use distinct page codes:
- `TEAR` = Teaching Earnings (`/teaching/earnings`) - ST's 70% share
- `CEAR` = Creator Earnings (`/creating/earnings`) - Creator's 15% royalty

**Rationale:** These are fundamentally different pages with different APIs, access controls, and data. Separate codes prevent confusion and enable accurate documentation.

**Consequences:**
- Created new `/api/me/creator-earnings` endpoint
- Created `CreatorEarningsDetail` component
- Renamed existing earnings page to TEAR at `/teaching/earnings`

## 3. Use Universal Profile URLs (/@handle)
**Type:** Implementation
**Topics:** astro, auth

**Trigger:** Teacher Dashboard "View Profile" link used `/teachers/me` instead of the universal `/@handle` format.

**Options Considered:**
1. Keep role-specific URLs: `/teachers/me`, `/creators/me`
2. Use universal URL with handle: `/@{handle}` ŌåÉ Chosen
3. Use generic `/profile/me` redirect

**Decision:** All "View Profile" links should use `/@{handle}` format, requiring the API to include the user's handle.

**Rationale:** The `/@handle` route is the canonical URL for viewing any user's public profile. Using it consistently ensures:
- Shareable URLs
- Consistent navigation
- Single source of truth for profile routing

**Consequences:**
- Added `handle` to teacher-dashboard API response
- Updated `TeacherDashboard.tsx` to use `/@${user.handle}`
- Pattern established for other dashboard profile links
