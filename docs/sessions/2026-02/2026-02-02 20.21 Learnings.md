# Session Learnings - 2026-02-02

## 1. Bulk sed Replacement Strategy for Large Migrations
**Topics:** testing, workflow

**Context:** Needed to update 128+ test files that used deprecated column names (`is_student`, `is_creator`, etc.) in SQL INSERT statements.

**Learning:** For large-scale migrations affecting many files, a staged sed approach works well:
1. First pass: Replace obvious patterns (column names in SQL)
2. Check TypeScript compilation to find missed cases
3. Manual fixes for edge cases (object property references)

**Pattern:**
```bash
# Replace column names in SQL INSERT statements
find tests/ -name "*.ts" | xargs sed -i '' 's/is_student,/can_take_courses,/g'
find tests/ -name "*.ts" | xargs sed -i '' 's/is_student)/can_take_courses)/g'
```

**Gotcha:** sed replacements can have unintended effects - the `bio` → `bio_full` replacement affected the `team_members` table which uses `bio` (not `bio_full`). Always verify with targeted greps after bulk replacements.

## 2. Test Data Objects vs SQL Column Names Are Separate Concerns
**Topics:** testing, d1

**Context:** After sed replaced SQL column names, TypeScript still failed because test data objects used old property names that were then referenced in `.bind()` calls.

**Learning:** When migrating database columns, there are three layers to update:
1. **SQL column names** in INSERT/SELECT statements
2. **Test data objects** (TypeScript interfaces with property names)
3. **`.bind()` calls** that reference the object properties

The sed approach handles #1 well, but #2 and #3 need manual updates when objects are defined separately from the SQL.

**Pattern:**
```typescript
// Test data object (needs property names matching what bind() expects)
const users = [
  { id: 'usr-1', can_take_courses: 1, can_create_courses: 0 }  // NOT is_student, is_creator
];

// bind() uses object properties
.bind(user.id, user.can_take_courses, user.can_create_courses)
```

## 3. Permission vs State Migration Has Three Phases
**Topics:** architecture, testing

**Context:** Migrating from cached role flags to capability + derived state architecture.

**Learning:** The Permission vs State migration naturally divides into three phases:
1. **Source code** (TypeScript interfaces, API endpoints) - Use derived state queries
2. **Test SQL** (INSERT statements) - Use capability column names
3. **Test assertions** - Match new API response shapes

Each phase can be completed and committed independently, allowing incremental progress on large migrations.

---

## 4. TypeScript Compilation as Migration Progress Indicator
**Topics:** testing, workflow

**Context:** During large test file migrations, tracking progress across 150+ files.

**Learning:** TypeScript compilation errors provide excellent progress tracking:
- Start: 25+ errors in test files
- After sed: 3 errors (targeted fixes needed)
- After manual fixes: 0 errors

Use `npx tsc --noEmit 2>&1 | wc -l` to track progress numerically. Zero lines = clean compilation.

---

## 5. API Response vs UI Mock Structures May Differ
**Topics:** testing, architecture

**Context:** Component tests like UserProfile.test.tsx use mock objects with `roles: { is_student, is_creator }` structure.

**Learning:** API endpoints return **derived state** (`is_creator`, `is_student_teacher`) computed from relationship tables, but UI components may structure this data differently. The component's internal interface doesn't need to match the API exactly - it's transformed at fetch time.

This means:
- API tests: Update to expect new response shapes
- Component tests: May keep existing mock structures if the component interface hasn't changed
