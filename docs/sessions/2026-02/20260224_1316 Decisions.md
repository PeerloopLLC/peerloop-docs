# Session Decisions - 2026-02-24

## 1. No-Show Detection at Read Time Rather Than Scheduled Job
**Type:** Implementation
**Topics:** video, d1, cloudflare

**Trigger:** Need to mark sessions as `no_show` when scheduled time passes and no one joins. Cloudflare Workers don't have built-in cron for frequent checks.

**Options Considered:**
1. Cron trigger (Cloudflare Workers scheduled handler) — runs periodically
2. BBB `room_ended` webhook — fires when room closes (but may not fire if room never created)
3. Read-time detection in st-sessions API ← Chosen

**Decision:** Detect no-shows as a side effect when S-T views their session history. The `st-sessions` API updates stale scheduled sessions (past end time, no attendance records) to `no_show` status before returning results.

**Rationale:** Simplest approach with zero infrastructure overhead. The S-T is the person who cares about no-shows, and they'll see the updated status the first time they view their history. No cron config, no webhook dependency. The trade-off is that no-show status isn't set until someone queries — acceptable for an MVP.

**Consequences:** No-show flagging works without any additional infrastructure. Status updates are scoped to the authenticated teacher's sessions only (safe). Detection delay is bounded by when the S-T next views their history.

## 2. Separate API Endpoint for Course Past Sessions
**Type:** Implementation
**Topics:** video, astro

**Trigger:** Resources tab needs to show completed sessions with recording links. Could embed this in the existing resources endpoint or create a new one.

**Options Considered:**
1. Extend `/api/courses/:id/resources` to include past sessions
2. Create new `/api/courses/:id/sessions?status=completed` endpoint ← Chosen
3. Fetch from generic `/api/sessions` with course filter

**Decision:** Create a dedicated `GET /api/courses/:id/sessions` endpoint that returns completed sessions for the authenticated enrolled student.

**Rationale:** Clean separation of concerns — resources are course materials (files, links), sessions are time-based events. The endpoint handles its own enrollment verification and can be extended later with other status filters. Avoids bloating the resources response with unrelated data.

**Consequences:** New endpoint created at `src/pages/api/courses/[id]/sessions.ts`. CourseTabs fetches past sessions alongside resources when the Resources tab is activated.
