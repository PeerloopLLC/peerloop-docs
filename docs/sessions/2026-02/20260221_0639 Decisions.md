## 2. Two-Vault Architecture: Personal vs Project
**Type:** Strategic
**Topics:** obsidian, docs-infra

**Trigger:** Considered consolidating personal Obsidian vault (timecards, notes, daily notes) with peerloop-docs vault to reduce friction.

**Options Considered:**
1. Single vault with gitignored `_private/` folder — no off-machine backup for private content
2. Single vault with Obsidian Sync + git — two sync mechanisms fighting over same files
3. Two separate vaults ← Chosen

**Decision:** Personal vault (Obsidian Sync, no GitHub) stays separate from project vault (GitHub, client-visible). No consolidation.

**Rationale:** Each vault has exactly one sync mechanism and one audience. Mixing creates either a backup gap (gitignored content has no off-machine backup) or sync conflicts (git + Obsidian Sync on same files). 44 years of client work confirms: content with different access profiles needs different containers.

**Consequences:** peerloop-docs vault is lightweight — for browsing project docs only, not journaling or personal notes. Timecards remain in personal vault. Cross-vault linking not possible; use searchable conventions.

## 3. Obsidian Vault: Minimal Core Plugins, No Daily Notes
**Type:** Implementation
**Topics:** obsidian

**Trigger:** Setting up `.obsidian/` in peerloop-docs for the first time.

**Options Considered:**
1. Full setup with daily notes, templates, community plugins
2. Minimal core plugins only ← Chosen
3. Copy settings from personal vault

**Decision:** 13 core plugins enabled (explorer, search, graph, backlinks, outline, bookmarks, file-recovery). No daily notes, no templates, no community plugins. Markdown links, not wikilinks.

**Rationale:** Daily notes belong in personal vault. Community plugins add overhead and can create unexpected files. The vault's primary purpose is browsing and searching 899 project docs, not note-taking.

**Consequences:** `.obsidian/` created with `app.json`, `appearance.json`, `core-plugins.json`, `community-plugins.json`, `hotkeys.json`. Already gitignored.

## 6. Durable Insight Capture Co-Located with Decisions
**Type:** Implementation
**Topics:** docs-infra, cc-workflow

**Trigger:** ★ Insight blocks generated during conversation contain durable knowledge that vanishes when the session ends. User wanted these captured alongside decisions.

**Options Considered:**
1. Separate INSIGHTS.md file
2. Insights as standalone learnings in session files
3. Insights co-located with the decision they illuminate ← Chosen

**Decision:** During `/q-learn-decide` processing, scan `★ Insight` blocks for durable, informative insights. Append qualifying insights to the relevant decision entry in DECISIONS.md or PLAYBOOK.md as `> **Insight:**` blockquotes.

**Rationale:** Co-location follows the same principle as co-locating tests with code — the context you need to understand *why* something exists should be close to the thing itself. A separate insights file creates cross-referencing overhead.

**Consequences:** `/q-learn-decide-local` updated with Insight Capture section defining qualification criteria and format.
