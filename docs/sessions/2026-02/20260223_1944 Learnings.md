# Session Learnings - 2026-02-23

## 1. Graceful Degradation for Non-Critical Parallel Fetches
**Topics:** astro

**Context:** CommunityManagement.tsx fetches communities, progressions, and members in parallel via `Promise.all`. The members list is informational — if it fails, the page should still render with progressions and settings working.

**Learning:** When adding a new parallel fetch for a non-critical section, parse the response conditionally rather than throwing on failure. This prevents a secondary endpoint failure from breaking the entire page.

**Pattern:**
```typescript
const [commRes, progRes, memRes] = await Promise.all([
  fetch('/api/me/communities'),
  fetch(`/api/me/communities/${slug}/progressions`),
  fetch(`/api/me/communities/${slug}/members`),
]);

// Critical fetches throw on failure
if (!commRes.ok) throw new Error('Failed to fetch communities');
if (!progRes.ok) throw new Error('Failed to fetch progressions');

// Non-critical fetch degrades gracefully
const memData = memRes.ok
  ? await memRes.json() as { members: Member[] }
  : { members: [] };
```

## 2. SQL CASE Expression for Role Priority Sorting
**Topics:** d1

**Context:** Community members need to be sorted by role importance (creator first, then student_teachers, then members), with joined_at as secondary sort within each role.

**Learning:** Use a SQL `CASE` expression in `ORDER BY` to define custom sort priority for enum-like text columns. This keeps the sorting logic in the query rather than requiring post-query sorting in application code.

**Pattern:**
```sql
ORDER BY
  CASE cm.role
    WHEN 'creator' THEN 0
    WHEN 'student_teacher' THEN 1
    WHEN 'member' THEN 2
  END,
  cm.joined_at ASC
```
