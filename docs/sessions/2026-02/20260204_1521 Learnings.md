# Session Learnings - 2026-02-04

## 1. 1:1 Community:Feed Simplifies Data Model
**Topics:** d1, architecture

**Context:** Designing community/feed architecture for CD-036. Original design had communities with multiple feeds inside (The Commons, Announcements, Help).

**Learning:** Mapping each community to exactly one feed (the community page IS the feed) dramatically simplifies both the data model and routing. Post categorization via tags provides filtering capability without the complexity of managing multiple feeds per community.

**Pattern:**
```
Community → 1 Feed (with tagged posts)
vs
Community → N Feeds (Announcements, Help, General...)
```

## 2. Tags vs Separate Feeds for Categorization
**Topics:** stream, architecture

**Context:** The Commons originally had three separate feeds: general, announcements, help & support.

**Learning:** Using post tags within a single feed provides the same UX benefit (filter to see just announcements) but with simpler data model, more flexibility (add tags without creating feeds), and query param filtering (`/community/the-commons?tag=announcements`).

## 3. Staged Schema Migration Reduces Risk
**Topics:** d1, workflow

**Context:** CD-036 requires communities, progressions, and course restructuring. Progressions require modifying the existing courses table (adding NOT NULL `progression_id`).

**Learning:** Splitting large schema changes into stages:
- **Stage 1:** Add new tables (communities, community_members, community_resources) - non-breaking
- **Stage 2:** Modify existing tables (courses.progression_id) - breaking

This allows shipping Stage 1 features immediately while Stage 2 gets dedicated attention.

## 4. Astro API Auth Pattern: getSession with Cookies
**Topics:** auth, astro

**Context:** Creating communities API endpoints needed optional authentication (visitors can see public communities, authenticated users see membership status).

**Learning:** The pattern for optional auth in Astro APIs:
```typescript
const jwtSecret = getEnv(locals, 'JWT_SECRET');
let userId: string | undefined;
if (jwtSecret) {
  const session = await getSession(cookies, jwtSecret);
  userId = session?.userId;
}
```

Note: Use `cookies` parameter (from Astro context), not `request`. The `getUser` helper doesn't exist - use `getSession` directly.

## 5. Community Membership Derived from Multiple Sources
**Topics:** d1, architecture

**Context:** S-T badges on community member listings. S-Ts are tied to courses, not communities, but courses belong to communities.

**Learning:** Community S-T status is derived via join:
```sql
community → progressions → courses → course_student_teachers → users
```

The `community_members.role` field stores the derived role ('creator', 'student_teacher', 'member') at join time, with `joined_via` tracking how they joined ('enrollment', 'invite', 'system', 'manual').
