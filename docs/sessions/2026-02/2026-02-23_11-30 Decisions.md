## 1. Upgrade Wrangler from 4.61 to 4.67
**Type:** Implementation
**Topics:** cloudflare, deployment

**Trigger:** Wrangler CLI showed "update available 4.67.0" at session start.

**Options Considered:**
1. Stay on 4.61.0 — no risk, but miss fixes
2. Upgrade to 4.67.0 ← Chosen

**Decision:** Upgrade wrangler to 4.67.0. No breaking changes affect Peerloop, and several bug fixes are directly relevant (Pages upload fix, CF_PAGES env var injection, D1 local dev ergonomics).

**Rationale:** 10 releases with zero breaking changes for our stack. Key fixes: .wrangler excluded from Pages uploads (silent bug), CF_PAGES_* auto-injected locally, spurious config diff warnings eliminated. Smoke tested with D1 query, tsc, and full build — all passed.

**Consequences:** package.json updated to `^4.67.0`, lockfile updated, committed to jfg-dev-6.

## 2. Document Stripe Seed as Separate Step in Migration Docs
**Type:** Implementation
**Topics:** stripe, d1, docs-infra

**Trigger:** Enrollment test on staging failed because `db:setup:staging` doesn't include Stripe account IDs — the Stripe seed is opt-in.

**Options Considered:**
1. Merge Stripe seed into the main dev seed ← Not chosen (breaks "test fresh install" use case)
2. Add quick reference table to migration docs ← Chosen
3. Leave as-is (rely on tribal knowledge)

**Decision:** Added a "Quick Reference: Full Database Setup" table to `docs/tech/tech-024-migrations.md` showing the 2x2 matrix of local/staging × with/without Stripe.

**Rationale:** Keeping Stripe seed separate is correct — not all dev workflows need Stripe accounts. But the extra step is easy to forget, so documenting it prominently prevents repeated confusion.

**Consequences:** `docs/tech/tech-024-migrations.md` updated with quick reference table.
