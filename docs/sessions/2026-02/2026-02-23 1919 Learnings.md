# Session Learnings - 2026-02-23

## 1. Fetch Mock Ordering Matters When URLs Are Substrings
**Topics:** testing

**Context:** After adding community/progression UI, the CreatorStudio tests started failing. The mock `fetch` handler checked `url.includes('/api/me/communities')` which matched both `/api/me/communities` and `/api/me/communities/test-community/progressions`. The more specific progressions URL was hitting the communities handler and returning the wrong response shape.

**Learning:** When mocking `fetch` with URL-based routing using `includes()`, always check the most specific URL patterns first. `/progressions` should be checked before `/api/me/communities` to prevent substring matches from swallowing nested routes.

**Pattern:**
```typescript
// WRONG: communities catches progressions too
if (url.includes('/api/me/communities')) { ... }

// RIGHT: check specific paths first
if (url.includes('/progressions')) { ... }
if (url.includes('/api/me/communities')) { ... }
```

## 2. Adding Parallel Fetches Breaks Existing Test Mocks
**Topics:** testing, astro

**Context:** When `CreatorStudio` and `CreatorDashboard` were updated to add a parallel `fetch('/api/me/communities')` call alongside existing fetches, all their existing tests broke. Tests using `mockResolvedValueOnce` only handled one fetch call, and tests using `mockImplementation` with URL routing didn't handle the new URL.

**Learning:** When adding a new parallel `fetch` to an existing component, every test that mocks `fetch` for that component needs updating. Tests using `mockResolvedValueOnce` are especially fragile — they need N mocked return values for N parallel fetches. Prefer `mockImplementation` with URL-based routing for components that make multiple API calls.

**Pattern:**
```typescript
// Fragile with parallel fetches:
mockFetch.mockResolvedValueOnce(response1); // only handles first call

// Robust with parallel fetches:
mockFetch.mockImplementation((url: string) => {
  if (url === '/api/me/communities') return Promise.resolve(commResponse);
  if (url === '/api/me/courses') return Promise.resolve(courseResponse);
  return Promise.resolve(defaultResponse);
});
```

## 3. Auto-Select UX for Single-Option Dropdowns
**Topics:** astro

**Context:** The CreateCourseModal needed community and progression pickers. When a creator has only one community (common case), showing a dropdown with one option is unnecessary friction.

**Learning:** For cascading dropdowns where most users will have 1-2 options, auto-select when there's only one option and show it as read-only text instead of a dropdown. This covers the 80% case (new creators with one community and one default progression) without any extra clicks while still supporting the multi-option case.

## 4. Community-First Empty State Cascading
**Topics:** astro

**Context:** CreatorStudio's empty state needed to handle two scenarios: (1) no courses but has communities, (2) no courses AND no communities. The existing empty state said "Create your first course" but that would fail if there were no communities since courses now require a progression_id.

**Learning:** When adding a prerequisite dependency (courses now require communities), the empty states and CTAs in dependent UIs need cascading logic: check the prerequisite first, then show the appropriate CTA. The "Create Course" button itself also needs the prerequisite check, not just the empty state.
