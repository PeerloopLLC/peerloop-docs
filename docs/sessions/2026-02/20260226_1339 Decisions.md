# Session Decisions - 2026-02-26

## 1. Rating Display Threshold at 3 Reviews
**Type:** Implementation
**Topics:** astro, d1

**Trigger:** Need to avoid misleading star ratings when a course or teacher has very few reviews.

**Options Considered:**
1. Always show stars, even with 1 review
2. Show "New" badge below 3 reviews, stars at 3+ ← Chosen
3. Show "Not enough reviews" text below threshold
4. Hide rating section entirely below threshold

**Decision:** Entities (courses and teachers) show a "New Course" or "New Teacher" green badge when below 3 reviews, and switch to star ratings at 3+ reviews.

**Rationale:** 3 is the minimum where a rating starts being statistically meaningful. The "New" badge is positive (not "insufficient data") and encourages early adoption. The constant `MIN_REVIEWS_FOR_RATING = 3` is centralized in `src/lib/ratings.ts` for easy adjustment.

**Consequences:** 6 components updated to use `getRatingDisplay()`. STListItem interface gained `total_reviews` field. Teachers query now pulls `us.total_reviews` from `user_stats`.

## 2. Single Review Response Per Review (No Back-and-Forth)
**Type:** Implementation
**Topics:** d1, astro

**Trigger:** Design of the review response system — how many responses should be allowed?

**Options Considered:**
1. Allow unlimited back-and-forth conversation ← Not chosen
2. Single response per review, enforced by UNIQUE constraint ← Chosen
3. Allow editing of response but no additional replies

**Decision:** One response per review, period. STs respond to enrollment reviews, Creators respond to course reviews. Enforced by `UNIQUE(review_type, review_id)` constraint in the `review_responses` table.

**Rationale:** Keeps reviews clean and professional. Matches patterns used by Google Reviews, Airbnb, etc. Prevents argument threads. The schema already had this constraint; the endpoint just needed to respect it.

**Consequences:** POST endpoint checks for existing response before insert (returns 409). No edit/delete endpoints needed.
