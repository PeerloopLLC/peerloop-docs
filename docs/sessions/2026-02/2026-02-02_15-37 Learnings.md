# Session Learnings - 2026-02-02

## 1. Schema Evolution Debt Creates Data Model Divergence
**Topics:** d1, workflow

**Context:** Investigating why `/profile` page showed different bio text than the CurrentUser global, discovered the database had THREE bio fields (`bio`, `bio_short`, `bio_full`) with different endpoints fetching different columns.

**Learning:** When a schema evolves (adding new columns like `bio_short` alongside `bio`), existing endpoints may continue using old columns while new endpoints use new columns. This creates silent data divergence where the same conceptual field has different values depending on which endpoint you query.

**Pattern:** When adding new schema columns that replace or supplement existing ones:
1. Audit ALL endpoints that touch the affected table
2. Create a migration plan to consolidate
3. Don't leave deprecated columns lingering - they accumulate confusion

## 2. Deprecated Role Flags vs Capability Flags Confusion
**Topics:** d1, auth

**Context:** The `users` table had both "role flags" (`is_student_teacher`, `is_creator`) marked as DEPRECATED and "capability flags" (`can_teach_courses`, `can_create_courses`). Different endpoints used different flags.

**Learning:** Having two parallel systems (old roles vs new capabilities) in the same table creates confusion about which to use. The schema comments said "deprecated" but the code still used them. The distinction matters: capabilities are admin-controlled permissions, while roles should be derived from course relationships.

**Pattern:** When deprecating fields:
- Remove them from the schema (don't just comment "deprecated")
- Or create a migration ticket and enforce a deadline
- Code naturally gravitates to what's in the schema

## 3. API Data Shape Should Match UI Needs, Not Database Structure
**Topics:** astro, workflow

**Context:** The `/api/me/full` endpoint for CurrentUser fetched only subset of user columns (capabilities, basic identity) while `/api/me/profile` fetched a different subset (editable fields, social links). This created a mismatch where CurrentUser didn't have `title` even though it was displayed in multiple places.

**Learning:** API response shapes should be driven by UI component needs, not by arbitrary endpoint boundaries. If the CurrentUser global is used to display user info across the app, it should contain ALL user display fields - not force components to make additional API calls.

**Pattern:** For "current user" type endpoints, include all fields the user might see about themselves anywhere in the app, even if it means a slightly larger payload.

## 4. Tracing Data Flow Reveals Hidden Dependencies
**Topics:** workflow, d1

**Context:** User asked "where does the data come from?" for three different views. By tracing each from component → API → SQL → schema, we discovered they were reading from different columns (`bio_short` vs `bio`) for what appeared to be the same field.

**Learning:** When debugging data inconsistencies, trace the full path: UI component → fetch call → API endpoint → SQL query → schema columns. Screenshots showing different values for "the same thing" often mean they're not actually the same thing at the database level.

## 5. Bio Field Split: Short vs Full for Different Contexts
**Topics:** astro, d1

**Context:** Decided to keep `bio_short` (tagline, 150 chars) and `bio_full` (detailed bio, 2000 chars) as distinct fields rather than one `bio` field.

**Learning:** Having separate short/full versions of text content is a common pattern (Twitter bio vs LinkedIn About, product title vs description). The short version is for cards, listings, and tight UI spaces; the full version is for dedicated profile/detail pages. Trying to use one field for both contexts leads to either truncation or text that's too long for cards.

**Pattern:**
- `bio_short`: Tagline, one-liner (150 chars) - displayed in cards, navbar, search results
- `bio_full`: Full bio (2000 chars) - displayed on profile page, about sections

## 6. Settings Page Migration Pattern: Extract Content Fragment
**Topics:** astro, workflow

**Context:** Migrating pages from old `LegacyAppLayout` (with spec view slots) to new `AppLayout`. The archived pages had `<Fragment slot="content">` containing the actual page content.

**Learning:** When migrating from a complex layout to a simpler one, extract just the "content" slot/fragment, discard the dev scaffolding (spec views, sidepanels), and keep the React component imports. Add dark mode variants while you're touching the file.

**Pattern:**
```astro
// Old (archived)
<LegacyAppLayout pageCode={...} showSidepanel={true}>
  <div class="spec-view">...</div>
  <Fragment slot="content">
    <ActualContent client:load />
  </Fragment>
</LegacyAppLayout>

// New
<AppLayout title="...">
  <ActualContent client:load />
</AppLayout>
```
