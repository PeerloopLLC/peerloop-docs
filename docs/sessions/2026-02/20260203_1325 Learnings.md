# Session 169 Learnings - 2026-02-03

## URL Routing Architecture

### "Bare = My" Principle
When designing authenticated app routes, the default context should be the user's personal content:
- `/courses` = MY enrolled courses (not the catalog)
- `/discover/courses` = Browse ALL courses (explicit discovery)

This is more intuitive than the reverse (`/my/courses` vs `/courses`) because users spend most time with their own content.

### Universal Profile URLs
The `/@handle` pattern (Twitter/X, Instagram) works well for user profiles:
- Short, memorable, shareable
- Works for all user types
- `/@me` can redirect to current user's profile

## Content Hierarchy Design

### Progressions as First-Class Entities
When courses can form sequences, the progression (learning path) should be the discovery unit, not individual courses:
- Users browse "journeys" not "units"
- Display with badges: "Learning Path" (2+ courses) or "Stand-alone" (1 course)
- Users still pay per course, but discover via progressions

### Course Copying vs Multi-Progression
If a course needs to appear in multiple learning paths, **copy it** rather than creating complex many-to-many relationships:
- Simpler data model (course belongs to exactly one progression)
- Shared resources via `source_course_id` reference
- Each copy can diverge independently

## Navigation Design

### Eliminating "More" Menus
"More" is an anti-pattern - it's a dumping ground for items that don't fit elsewhere. Better approach:
- Merge account-related items into user avatar dropdown
- Follow established patterns (GitHub, Google, LinkedIn)
- Everything has a logical home

### Role-Based Dashboard Links
Instead of generic "Workspace", show explicit role-based links:
- Learning → `/learning` (all students)
- Teaching → `/teaching` (S-Ts)
- Creating → `/creating` (Creators)

Users with multiple roles see multiple links.
