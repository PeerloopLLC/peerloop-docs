# Session Learnings - 2026-02-22

## 1. Stripe Sandbox Accounts Are Portable, DB Links Are Not
**Topics:** stripe, d1

**Context:** Needed to understand how Stripe test-mode connected accounts work across 2 dev machines and staging.

**Learning:** Stripe connected accounts (`acct_...`) live in Stripe's cloud and are accessible from any environment using the same `sk_test_` key. However, the `users.stripe_account_id` column that links a Peerloop user to their Stripe account is per-database. Each D1 instance (local, staging) maintains its own mapping independently. The Stripe account exists everywhere; the pointer to it doesn't.

**Pattern:** Solve with a separate seed file (`0002_seed_stripe.sql`) containing UPDATE statements that set `stripe_account_id` for test users. Apply after the main dev seed as an opt-in step.

## 2. Mock Stripe IDs in Seed Data Are a Hidden Trap
**Topics:** stripe, testing, d1

**Context:** Dev seed data had `acct_mock_sarah` and `acct_mock_marcus` as Stripe account IDs for student-teachers.

**Learning:** Mock IDs pass `IS NOT NULL` checks in application code, making it look like Stripe is configured. But any code path that actually calls the Stripe API with those IDs (e.g., `stripe.transfers.create({ destination: 'acct_mock_sarah' })`) would get a "No such account" error. This creates a phantom state — tests pass (because they mock Stripe), but manual testing fails at runtime.

**Pattern:** Use NULL for "not onboarded" (honest state) in the base seed. Use real Stripe sandbox IDs in an optional seed file for manual testing. Never use fake `acct_` IDs that look real but aren't.

## 3. Webhooks Are Unreliable by Design — Use Polling as Safety Net
**Topics:** stripe

**Context:** Discussed how to handle Stripe webhook events that fire asynchronously with no user action to trigger self-healing (e.g., `payout.failed`, `transfer.created`, `charge.dispute.created`).

**Learning:** Stripe's own documentation acknowledges webhooks may be missed or delayed. The industry standard pattern is: webhooks for real-time delivery + `stripe.events.list()` polling as a catch-up mechanism. Stripe stores events for 30 days. On Cloudflare, this maps to a Cron Trigger. The prerequisite is that all webhook handlers must be idempotent (already done in Session 223), so replaying events is safe.

## 4. Automated Stripe Integration Tests: High Friction, Low ROI Pre-Launch
**Topics:** testing, stripe

**Context:** Explored whether `stripe trigger` or API-driven checkout sessions should be wrapped in automated integration tests.

**Learning:** A real Stripe integration test requires 4 running services (dev server, `stripe listen`, network connectivity, seeded DB). The `stripe trigger` CLI sends generic payloads without custom metadata, so handlers reject them at the business logic layer. API-driven tests can customize payloads but add Stripe API latency and failure modes unrelated to application code. At pre-launch scale with one developer, manual testing with seed data + `stripe listen` + Stripe Dashboard provides the same coverage with less friction and better debugging visibility. Automated integration tests make sense when CI/CD exists and multiple developers can break payment flows.

## 5. Express Onboarding Is Stripe's KYC Responsibility
**Topics:** stripe

**Context:** Considered a helper script that calls `stripe.accounts.create()` to programmatically create test connected accounts.

**Learning:** `stripe.accounts.create()` creates the account object, but Express onboarding is where Stripe handles KYC/AML compliance — identity verification, bank account linking, tax form collection (W-9/1099). When using Express accounts, Stripe takes legal responsibility for this verification. Bypassing the onboarding UX with programmatic creation skips compliance steps that matter in production. In test mode the accounts auto-verify, but the pattern is wrong. Better to create accounts through the Stripe Dashboard (which mirrors the real flow) and reference the resulting `acct_` IDs in seed data.

## 6. Separate Seed Files for Layered Data Concerns
**Topics:** d1, testing

**Context:** User wanted Stripe test accounts in seed data but separable from the main dev seed to allow testing fresh-install flows.

**Learning:** Using a separate SQL file with only UPDATE statements (no INSERTs) creates a clean layered architecture: `0001_seed_dev.sql` provides users in their default state (no Stripe), `0002_seed_stripe.sql` optionally adds real Stripe account IDs on top. The UPDATE-only pattern means the file has a natural dependency — it must run after the INSERT file — and can be skipped entirely without breaking anything. This extends naturally to other optional test configurations (e.g., a future `0003_seed_enrollments.sql` for pre-populated enrollment states).
