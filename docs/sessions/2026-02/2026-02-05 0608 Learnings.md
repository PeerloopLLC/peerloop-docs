# Session Learnings - 2026-02-05

## 1. URL-Aware Tabs Pattern with Astro Subroutes
**Topics:** astro, architecture

**Context:** Converting community tabs from client-side state (`useState`) to URL-aware navigation that supports bookmarkable URLs.

**Learning:** The subroute pattern (folder with index.astro + siblings) enables URL-aware tabs where:
- Server renders the correct initial tab via `initialTab` prop
- Client hydrates and syncs with URL via `popstate` listener
- Tab clicks update URL via `history.pushState()` without page reload

**Pattern:**
```
src/pages/community/[slug]/
├── index.astro           # /community/[slug] → initialTab="feed"
├── courses.astro         # /community/[slug]/courses → initialTab="courses"
├── resources.astro       # /community/[slug]/resources → initialTab="resources"
└── members.astro         # /community/[slug]/members → initialTab="members"
```

Component receives `initialTab` prop and uses `handleTabChange()` that calls `window.history.pushState()` instead of just `setActiveTab()`.

## 2. Data Duplication in Subroute Pattern is Acceptable
**Topics:** astro, architecture

**Context:** All four subroute files share identical data fetching logic (API calls, transformations). This feels like code duplication.

**Learning:** Data duplication across subroutes is the established pattern in this codebase. The `/course/[slug]/` folder uses the same approach where `index.astro` and `feed.astro` duplicate data fetching. This is acceptable because:
1. Each file is a complete entry point for its URL
2. Extracting to shared function adds complexity without much benefit
3. Astro builds each route independently anyway

The trade-off is maintenance overhead vs. simplicity—for small teams, explicit duplication is often clearer than abstractions.

## 3. Handling Special Cases with Server-Side Redirects
**Topics:** astro

**Context:** The Commons (system community) shouldn't have a Courses tab, but users could manually navigate to `/community/the-commons/courses`.

**Learning:** Handle tab restrictions with server-side redirects in the subroute file:

```typescript
// In courses.astro
if (community.isSystem) {
  return Astro.redirect(`/community/${slug}`);
}
```

This ensures URLs that shouldn't exist redirect gracefully rather than showing an error or wrong content. The redirect happens at SSR time, so there's no client-side flash.
