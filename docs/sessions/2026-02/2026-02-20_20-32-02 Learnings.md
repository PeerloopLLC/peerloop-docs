# Session Learnings - 2026-02-20

## 1. Local D1 Database Requires Explicit Setup After Clone/Reset
**Topics:** d1, dev-environment

**Context:** After the docs-primary migration, starting `npm run dev` and attempting login produced `D1_ERROR: no such table: users: SQLITE_ERROR`.

**Learning:** Wrangler's local D1 database lives in `.wrangler/state/v3/d1/` as a SQLite file and starts empty. After any fresh clone, machine setup, or `db:reset:local`, you must run `npm run db:setup:local` (reset → migrate → seed) before the app can serve authenticated requests. This is not automatic.

## 2. Barrel File Exports Can Massively Exceed Actual Consumption
**Topics:** testing, workflow

**Context:** Reviewing `tests/helpers/index.ts` for MBA-2017 cleanup revealed it re-exported 75 symbols from 5 submodules.

**Learning:** Only 16 of 75 exports (21%) had any consumers in the codebase. The usage was extremely concentrated — 5 core exports (`DB_TEST`, `describeWithTestDB`, `getTestDB`, `resetTestDB`, `closeTestDB`) appeared in 179 files each, while the entire `mock-db.ts` module had zero consumers anywhere. Barrel files accumulate exports over time without natural pruning pressure. A periodic "grep for actual imports" audit catches this drift.

**Pattern:**
```bash
# Find which barrel exports are actually imported across the codebase
# Search for the barrel import path, extract named imports
grep -rh "from.*helpers" tests/ --include="*.ts" | sort | uniq -c | sort -rn
```

## 3. Deprecated Annotations Don't Prevent Build Dependencies
**Topics:** testing, workflow

**Context:** Session 227 retired the MBA-2017 and marked all related skip helpers with `@deprecated` JSDoc annotations, but left the code and exports in place.

**Learning:** `@deprecated` is a signal for humans and IDE warnings — it doesn't prevent code from compiling, being imported, or being bundled. True cleanup requires tracing all import paths and removing both the exports and their consumers. The gap between "deprecated" and "removed" is where dead code accumulates.

## 4. Vitest Setup Files Are Easy to Miss When Grepping for Consumers
**Topics:** testing

**Context:** After removing `getMachineCapabilities` from `machine.ts`, all 266 test files failed because `vitest.setup.ts` (at project root, not in `tests/`) imported it.

**Learning:** When auditing export consumers, search the entire project — not just the directory where most consumers live. Setup files (`vitest.setup.ts`, `vitest.config.ts`), root configs, and scripts can import from test helpers. The initial subagent search missed this because it focused on `tests/` and `scripts/`.

---

## 5. mock-db.ts Was a Machine-Capability Workaround, Not a Testing Pattern
**Topics:** testing, dev-environment

**Context:** The entire `mock-db.ts` module (226 lines, 11 exports) had zero consumers despite being prominently exported from the barrel.

**Learning:** `mock-db.ts` existed solely because the MBA-2017 couldn't run local D1 (wrangler required macOS 13.5+, MBA-2017 was on an older version). It provided fake D1 responses so tests could still run on that machine. Once both dev machines gained full D1 support via `better-sqlite3`, all 179 test files were written against real DB testing. The mock layer became invisible dead code — still exported but never imported. Machine-capability workarounds should be removed when the capability gap closes, not just deprecated.
