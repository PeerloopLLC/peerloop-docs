# Session Decisions - 2026-02-01

## 1. Full Component Folder Reorganization
**Type:** Strategic
**Topics:** workflow, astro

**Trigger:** Components folder had 34 subfolders with confusing duplicates (creator/ vs creators/, dash/ vs dashboard/, teachers/ vs teaching/) making it hard to know where to find or add components.

**Options Considered:**
1. Minimal cleanup - just rename confusing folders ← Not chosen
2. Full reorganization with new structure ← Chosen
3. Leave as-is and document conventions ← Not chosen

**Decision:** Perform full reorganization creating new ui/, layout/ folders and merging duplicate-ish folders into clear domain structures.

**Rationale:** The cognitive overhead of "which folder?" was slowing development. A one-time reorganization cost (updating imports) pays off in long-term discoverability. The 26% folder reduction (34→25) and clear naming makes the structure self-documenting.

**Consequences:**
- ~50 files moved
- All imports updated across source, tests, and page JSON files
- New barrel exports created for all reorganized folders
- Clear pattern established for future components

## 2. Separate ui/ and layout/ from Feature Folders
**Type:** Strategic
**Topics:** astro, workflow

**Trigger:** Charts were mixed with feature components, and app shell components (navbar, header, footer) were in a generic "navigation" folder.

**Options Considered:**
1. Keep charts in feature folders, navigation as-is ← Not chosen
2. Create ui/ for primitives, layout/ for shell ← Chosen
3. Create a single "shared/" folder ← Not chosen

**Decision:** Create `ui/` folder for design system primitives (currently charts, future: buttons, modals, cards) and `layout/` folder for app shell components (AppNavbar, Header, Footer, slide panels).

**Rationale:** Follows Atomic Design principles - separating primitives from features makes it clear what's reusable across the app vs. specific to a domain. The layout/ folder groups components that form the persistent app shell, separate from page content.

**Consequences:**
- `ui/charts/` created with barrel export
- `layout/` created with AppNavbar, Header, Footer, FeedSlidePanel, MoreSlidePanel
- Future design system components have a clear home
- Layout components can be imported from single barrel: `@components/layout`

## 3. Domain Folder Subfolder Convention: profiles/ vs studio/ vs workspace/
**Type:** Implementation
**Topics:** workflow

**Trigger:** When merging creator/ + creators/ and teachers/ + teaching/, needed a consistent way to separate public-facing vs internal-tool components.

**Options Considered:**
1. Flat structure with prefixes (CreatorProfile_, CreatorStudio_) ← Not chosen
2. Generic subfolders (public/, private/) ← Not chosen
3. Purpose-named subfolders (profiles/, studio/, workspace/) ← Chosen

**Decision:** Use semantic subfolder names:
- `profiles/` - Public-facing display (cards, browse pages, profile views)
- `studio/` - Content authoring (editors, creation tools)
- `workspace/` - Task management (availability, earnings, sessions)

**Rationale:** Names describe what the components do, not access level. "profiles" implies display, "studio" implies creation, "workspace" implies management. This is more discoverable than generic names.

**Consequences:**
- `creators/profiles/` and `creators/studio/` created
- `student-teachers/profiles/` and `student-teachers/workspace/` created
- Pattern documented for future domain folders
- Import paths are self-documenting: `@components/creators/studio/CourseEditor`
