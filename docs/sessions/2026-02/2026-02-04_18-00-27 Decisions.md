# Session Decisions - 2026-02-04

## 1. Progressions Architecture: Community → Progression → Course Chain
**Type:** Strategic
**Topics:** d1, astro

**Trigger:** Implementing CD-036 Stage 2 required defining how courses relate to communities.

**Options Considered:**
1. Direct community_id on courses (course belongs to community directly)
2. Community → Course (no intermediate layer)
3. Community → Progression → Course chain ← Chosen

**Decision:** Courses belong to progressions, progressions belong to communities. This creates a three-level hierarchy: Community → Progression → Course.

**Rationale:**
- Progressions enable "learning paths" (multi-course sequences)
- Single courses can be "standalone" progressions
- Enables course positioning ("Course 2 of 3")
- Allows copying courses between progressions without duplicating community membership

**Consequences:**
- Added `progressions` table to schema
- Added `progression_id` and `progression_position` to courses
- API returns progressions with nested courses
- CommunityTabs Courses tab shows courses grouped by progression

## 2. progression_id Nullable for Migration Flexibility
**Type:** Implementation
**Topics:** d1

**Trigger:** Existing courses in seed data don't have progressions assigned.

**Options Considered:**
1. Make progression_id NOT NULL, migrate all courses first ← Too restrictive
2. Make progression_id nullable, populate via seed data ← Chosen
3. Create default "Uncategorized" progression

**Decision:** Make `progression_id` nullable in the schema, but populate it for all courses in seed data.

**Rationale:**
- Allows schema changes to be applied without breaking existing data
- Seed data handles the migration by assigning progressions
- Application code can enforce the constraint logically
- Flexibility during development phase

**Consequences:**
- Schema allows NULL progression_id
- All seeded courses have progression_id populated
- Future courses should always have a progression

## 3. Course Feed Route as Redirect
**Type:** Implementation
**Topics:** astro

**Trigger:** Task to add `/course/[slug]/feed` but `/course/[slug]/discuss` already exists with full CourseFeed implementation.

**Options Considered:**
1. Duplicate the page at /feed ← Violates DRY
2. Rename /discuss to /feed ← Breaking change
3. Create /feed as 301 redirect to /discuss ← Chosen
4. Keep only /discuss, no /feed route

**Decision:** Create `/course/[slug]/feed.astro` as a 301 redirect to `/course/[slug]/discuss`.

**Rationale:**
- Both URLs work for users who might try either
- Single source of truth for implementation
- SEO-friendly (redirect, not duplicate content)
- No breaking changes to existing /discuss route

**Consequences:**
- Created `feed.astro` with simple redirect
- Users can access course discussions via either URL

## 4. Creator Feed as Stub Page
**Type:** Implementation
**Topics:** astro, stream

**Trigger:** Task to add `/creator/[handle]/feed` (IFED) but Stream.io integration not yet complete.

**Options Considered:**
1. Skip the page until Stream.io ready
2. Create full page with mock data
3. Create stub page with empty state UI ← Chosen

**Decision:** Create the feed page with proper layout, creator header, and empty state that explains what will appear.

**Rationale:**
- URL is available for navigation/linking now
- UI communicates feature intent to users
- Easy to wire up when Stream.io integration is ready
- Follows pattern of other stub pages in the project

**Consequences:**
- Created `/creator/[handle]/feed.astro`
- Shows empty state with "No posts yet" message
- Lists what content will appear (announcements, tips, etc.)
