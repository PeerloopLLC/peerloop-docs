# Session Learnings - 2026-02-22 (Session 248)

## 1. PLAN.md Bloat from Completed Content
**Topics:** docs-infra, cc-workflow

**Context:** PLAN.md had grown to 1,260 lines. Much of it was completed sub-blocks kept as "one-liners" per the old convention, plus resolved items (DEPLOY-WARNINGS), all-complete tables (Infrastructure Summary), and decision records that belonged elsewhere.

**Learning:** Keeping completed sub-blocks in PLAN.md — even as brief summaries — causes steady bloat. Over 248 sessions, the file became ~60% completed content and ~40% actual remaining work. Stripping to only remaining work reduced it to ~960 lines and made it scannable again.

**Pattern:** When a sub-block completes, add it to a single "Completed:" summary line at the top of the parent block. Don't keep the sub-block's section at all.

---

## 2. Catch-All Deferred Lists Become Dumping Grounds
**Topics:** docs-infra

**Context:** PLAN.md had a "Deferred Items" table with ~18 items from various completed blocks. Some were resolved, some overlapped with existing deferred blocks, and the rest were disconnected from any context.

**Learning:** A single deferred-items table becomes a disconnected dumping ground. Items lose their context and are easy to forget. Folding deferred items into their related blocks (e.g., Stripe items → MVP-GOLIVE.STRIPE) keeps them discoverable when that block is eventually worked on.

---

## 3. Skill Files Drift When Not Tied to Structure Changes
**Topics:** cc-workflow, docs-infra

**Context:** After restructuring PLAN.md, the `/q-update-plan-local` command still described the old structure — "Quick Status" section, "Completed Blocks" with links, session notes format, page counts — none of which existed anymore.

**Learning:** When a document's structure is significantly changed, the skills that manage it must be updated in the same session. Otherwise the next invocation of the skill will try to maintain a structure that no longer exists, creating confusion or reverting the changes.
