# Session Decisions - 2026-02-06

## 1. Separate icon-paths.ts Registry for Astro (Not Shared with icons.tsx)
**Type:** Implementation
**Topics:** astro

**Trigger:** Need to consolidate 61 inline SVGs across 14 Astro files. Could either: share path data between React and Astro, or maintain separate registries.

**Options Considered:**
1. Create `icon-paths.ts` as a standalone Astro registry, keep `icons.tsx` unchanged structurally ← Chosen
2. Refactor `icons.tsx` to import from `icon-paths.ts`, unifying both systems
3. Use `icons.tsx` directly in Astro files (would require JSX → HTML conversion)

**Decision:** Created `icon-paths.ts` as a standalone registry with raw HTML path strings, separate from the React `icons.tsx`. Added 6 new icon exports to `icons.tsx` for parity but left its structure untouched.

**Rationale:** Option 2 would touch every React component that imports icons — a massive blast radius for what should be an Astro-side cleanup. Option 3 is architecturally wrong (JSX attributes like `strokeLinecap` don't work in HTML templates). The separate registry approach is zero-risk to the working React system while giving Astro files the same centralized control.

**Consequences:** Two sources of truth for icon paths (HTML strings in `icon-paths.ts`, JSX in `icons.tsx`). Future work can unify them by having `icons.tsx` read from `icon-paths.ts` and convert to JSX attributes, but that's a separate PR.

## 2. Icon.astro Accepts `string` Name Prop (Not Just `IconName`)
**Type:** Implementation
**Topics:** astro

**Trigger:** Several pages pass dynamic icon names from data (e.g., `community.icon` from API response). A strict `IconName` type would require casting or runtime checks.

**Options Considered:**
1. Accept `IconName | string` with dev-mode console warning for unknown names ← Chosen
2. Accept only `IconName` (strict, but breaks dynamic usage without casting)
3. Accept `string` with no warning (permissive, but silent failures)

**Decision:** The `name` prop accepts `IconName | string`. Unknown names trigger a `console.warn` in dev mode and render a dashed-rectangle placeholder SVG.

**Rationale:** Data-driven icon names from the API (like `community.icon`) are plain strings at runtime. Requiring `as IconName` casts everywhere would be noisy. The dev-mode warning catches typos during development while the placeholder prevents broken layouts in production.

**Consequences:** Typos in static icon names won't be caught at compile time (they're valid strings), but the visual placeholder makes them immediately obvious during development.
