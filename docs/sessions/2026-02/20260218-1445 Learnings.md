# Session Learnings - 2026-02-18 (Session 224)

## 1. Stripe Connect Webhook Events Split by Account Context
**Topics:** stripe, deployment

**Context:** Setting up a Stripe webhook endpoint in the Dashboard for the staging environment, the first screen asks to choose between "Your account" and "Connected and v2 accounts" event delivery.

**Learning:** Stripe Connect splits webhook events into two delivery contexts. "Your account" events fire for resources owned by the platform (checkouts, transfers, refunds, account status notifications). "Connected and v2 accounts" events fire for resources inside connected accounts (creator bank payouts). Most marketplace events (4 of 5 in Peerloop) are "Your account" events. The exception is `payout.failed` — because the automatic bank payout happens entirely within the creator's connected account, not initiated by the platform.

**Pattern:** Map each `stripe.*.create()` API call in your codebase to the events it produces, then classify by which Stripe account context they fire in. This gives a systematic event inventory rather than guessing.

## 2. `transfer.paid` Is Not a Real Stripe Event
**Topics:** stripe, testing

**Context:** While selecting events in the Stripe Dashboard, `transfer.paid` was not listed. Only `transfer.created`, `transfer.reversed`, and `transfer.updated` exist.

**Learning:** Stripe transfers don't have a "pending → paid" lifecycle like payouts do. When `stripe.transfers.create()` is called with `source_transaction`, Stripe waits for the charge to settle then creates the transfer — `transfer.created` IS the "paid" moment. The Stripe SDK's TypeScript union also doesn't include `transfer.paid`, which we had worked around by putting it in a `default` block — but the real fix was using the correct event name.

## 3. `stripe listen` CLI Masks the Account Context Distinction
**Topics:** stripe, testing, deployment

**Context:** The `transfer.paid` handler worked locally during Session 223 testing but would never fire in production.

**Learning:** The Stripe CLI (`stripe listen`) forwards ALL events from ALL contexts (platform + connected accounts) without distinction. This masks the dashboard webhook configuration model, where you must explicitly choose "Your account" vs "Connected accounts." A handler that works locally may never receive events in staging/production if the event fires in the wrong context. Always verify event availability in the Stripe Dashboard before relying on local testing.

## 4. SQLite CHECK Constraints Silently Block Updates via INSERT OR IGNORE
**Topics:** d1, testing

**Context:** Adding `status = 'disputed'` to enrollments failed silently because the schema CHECK constraint only allowed `('enrolled', 'in_progress', 'completed', 'cancelled')`.

**Learning:** When a CHECK constraint blocks an operation, `INSERT OR IGNORE` silently skips the row. In test setup, this means data you think exists doesn't, causing cascading failures that look unrelated (e.g., "enrollment is null"). Always verify test data was actually inserted when debugging test failures. Also: schema CHECK constraints must be updated when adding new status values — easy to forget when the handler code compiles fine.

## 5. Cloudflare Pages Branch-Based URLs Are Stable Webhook Targets
**Topics:** cloudflare, stripe, deployment

**Context:** Session 223 decided to skip Preview webhooks because Cloudflare Pages URLs are dynamic. Session 224 revisited this.

**Learning:** While per-commit URLs (`<hash>.peerloop.pages.dev`) are dynamic, each branch gets a stable URL (`<branch>.peerloop.pages.dev`) that always resolves to the latest deployment on that branch. The `staging` branch at `staging.peerloop.pages.dev` is stable enough for Stripe webhook endpoints. The Session 223 decision was based on per-commit URLs — the branch-based URL was the missing piece.

## 6. Connected Accounts Are Per-Environment at the Database Level
**Topics:** stripe, d1, deployment

**Context:** After creating Guy Rymberg's Stripe Connect account locally, asked whether it would be available on the staging deployment.

**Learning:** Stripe Connect accounts exist in Stripe's cloud and are accessible with the same API keys from any environment. But the `stripe_account_id` linking a user to their account is stored in the local D1 database — each environment (local, staging, production) has its own D1 instance. So a creator must go through onboarding in each environment separately. Each onboarding creates a new `acct_...` ID; Stripe doesn't deduplicate by email. The accounts are independent and don't interfere with each other.

## 7. Dispute Handling Requires Transfer Reversal to Recover Platform Funds
**Topics:** stripe

**Context:** Writing the `charge.dispute.closed` handler for the "lost" case.

**Learning:** When Stripe processes a dispute, it claws back the full charge amount + $15 fee from the platform's balance immediately. But if the platform already transferred funds to connected accounts (creator's 85%), that money is still in the connected account. Without reversing those transfers on dispute loss, the platform absorbs the full loss while the creator keeps their share. The transfer reversal pattern is the same as the refund handler — `stripe.transfers.list()` by transfer_group, then `createReversal()` for each. If the creator already withdrew (via automatic payout), the reversal creates a negative balance deducted from their next payout.
