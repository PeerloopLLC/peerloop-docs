# Session Decisions - 2026-02-23

## 1. Community Management as Separate Page, Not Embedded in CreatorStudio
**Type:** Implementation
**Topics:** astro

**Trigger:** Need to add community CRUD UI — could embed in existing CreatorStudio or create a new page.

**Options Considered:**
1. Add community section within CreatorStudio.tsx (single page for all creator content)
2. Separate `/creating/communities` page with its own component tree ← Chosen

**Decision:** Communities get their own page at `/creating/communities` with a detail page at `/creating/communities/[slug]`.

**Rationale:** CreatorStudio is already 303 lines handling courses. Community management needs its own space for the community list, progression management, settings, and eventually member management. Matches the existing pattern where studio, earnings, and analytics are separate pages under `/creating/`.

**Consequences:** Created 2 Astro pages and 7 React components in a new `creators/communities/` directory. Navigation from dashboard and studio links to the new page.

## 2. Fetch Community List + Filter by Slug Instead of Dedicated Detail Endpoint
**Type:** Implementation
**Topics:** astro, d1

**Trigger:** CommunityManagement needs full community details for a single slug, but no `GET /api/me/communities/[slug]` endpoint exists (only PATCH and DELETE).

**Options Considered:**
1. Create a new `GET /api/me/communities/[slug]` endpoint
2. Fetch full list from `GET /api/me/communities` and filter client-side by slug ← Chosen

**Decision:** Fetch the full community list and find by slug in the component.

**Rationale:** Creators will have 1-5 communities typically. The list endpoint already returns all needed fields. Adding a new API endpoint adds code, tests, and maintenance for negligible benefit. The list is fetched in parallel with progressions, so no extra round trip.

**Consequences:** CommunityManagement calls `GET /api/me/communities` + `GET /api/me/communities/[slug]/progressions` in parallel, then filters the community list by slug.

## 3. Base Modal for New Modals, Inline Modal Pattern Preserved for CreateCourseModal
**Type:** Implementation
**Topics:** astro

**Trigger:** New modals needed for CreateCommunity and CreateProgression. CreateCourseModal uses an inline modal (no base component). Base Modal.tsx exists with focus trap, escape, and backdrop.

**Options Considered:**
1. Use base Modal.tsx for all new modals, leave CreateCourseModal as-is ← Chosen
2. Migrate CreateCourseModal to base Modal.tsx too
3. Use inline modal pattern for all new modals

**Decision:** New modals (CreateCommunityModal, CreateProgressionModal) use the base `Modal.tsx`. CreateCourseModal keeps its inline modal pattern since it was already working and got significant changes (community/progression picker added).

**Rationale:** Base Modal provides focus trap, escape key, and backdrop click out of the box. No reason to rewrite CreateCourseModal's working modal structure while also adding the new picker fields — that would risk more test breakage.

**Consequences:** Two modal patterns coexist. Not a problem since Modal.tsx is the standard going forward.
