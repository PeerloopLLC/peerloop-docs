# Session Learnings - 2026-02-25

## 1. D1 Seed File Size Limits Are Generous
**Topics:** d1, deployment

**Context:** After rewriting the dev seed file to cover all 58 schema tables (up from 18), the file grew to 782 lines / 80KB / 67 SQL statements. Question arose whether it should be split.

**Learning:** `wrangler d1 execute --file` handles multi-statement SQL files natively with no practical size concern at this scale. The production schema file (`0001_schema.sql`) is already larger at ~1,100 lines. Splitting would add complexity to the `db:seed:local` script for no benefit. All 67 statements executed in under a second.

## 2. External Service Data Cannot Be Seeded via SQL
**Topics:** d1, stream, video

**Context:** While populating all schema tables, some data inherently lives outside D1 — Stream.io feed content, BBB meeting rooms, R2 file uploads, Stripe payment objects.

**Learning:** The strategy is to seed the D1 metadata with mock/NULL values so dashboards render correctly, while accepting that external service features (community feeds, live video, actual file downloads) will show empty states until the services are running. Fake Stripe IDs (`pi_mock_*`, `acct_mock_*`) work for dashboard display but won't resolve against Stripe's API. BBB fields (`bbb_meeting_id`, etc.) are left NULL on seeded sessions.

## 3. Session Tracking Model: Conceptual vs Relational Links
**Topics:** d1

**Context:** Understanding how a student's progress through a multi-session course is represented across tables.

**Learning:** The `sessions` table (actual live meetings) and `course_curriculum.session_number` (which meeting covers which topic) are linked conceptually, not via FK. `module_progress` independently tracks which curriculum modules are complete. This means a student's state is reconstructed by joining: `enrollments` (overall status/percent) + `sessions` (meeting history) + `module_progress` (topic completion). The `progress_percent` on enrollments is a denormalized summary.

## 4. Mock-Data and Seed Data Must Stay in Sync
**Topics:** testing, d1

**Context:** Adding 2 Gabriel courses to the SQL seed required corresponding additions to `mock-data.ts` and updating `TEST_DATA_COUNTS` in `test-data.ts`. The counts are computed dynamically from mock-data arrays (e.g., `mockData.courses.length`).

**Learning:** Because `TEST_DATA_COUNTS` is derived from `mockData.courses.length` (not hardcoded), adding courses to the mock-data array automatically updates the count. The only manual sync needed was updating the comment from `// 4` to `// 6`. The integration tests compare DB row counts against these derived values, so both files must agree.
