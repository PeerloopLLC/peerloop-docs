# Session Decisions - 2026-02-04

## 1. Community Routes: `/community` as Primary (Not `/discover/community`)
**Type:** Strategic
**Topics:** astro, architecture

**Trigger:** Planning the community page routes. The Commons is auto-subscribed for all users, creating semantic tension between "my stuff" (bare routes) and "discovery" (discover/ prefix).

**Options Considered:**
1. `/discover/community` only - Consistent with public browsing pattern
2. `/community` only - Top-level concept like `/feed` ← Chosen
3. Both routes with different content - Only makes sense with multiple community types
4. `/community` primary with redirect from `/discover/community` - Hybrid approach

**Decision:** Use `/community` as the primary route. The Commons is semantically "my community" even though it's shared by everyone.

**Rationale:**
- Consistency with `/feed` (both are part of "My Feeds")
- Shorter, more memorable URL
- `/discover/communities` (plural) is for finding NEW communities to join

**Consequences:**
- Created `/community/index.astro` as hub page
- Created `/community/[slug].astro` for individual communities
- Updated `tech-021-url-routing.md` with Community & Feed Architecture section
- `/discover/communities` reserved for community discovery

## 2. 1:1 Community:Feed Mapping
**Type:** Strategic
**Topics:** stream, d1, architecture

**Trigger:** Clarifying how feeds relate to communities. Original CD-036 implied multiple feeds per community.

**Options Considered:**
1. Multiple feeds per community (The Commons has: general, announcements, help)
2. One feed per community, posts have tags ← Chosen
3. Separate feed tables with foreign keys to communities

**Decision:** Each community has exactly one feed. The community page IS the feed. Post categorization uses tags.

**Rationale:**
- Simpler data model (no feed entity needed)
- More flexible (tags can be added without schema changes)
- Routing simplification (`/community/[slug]` not `/community/[slug]/[feed]`)
- Matches common patterns (Discord channels, Slack channels)

**Consequences:**
- No separate `feeds` table needed
- Stream.io feed ID pattern: `community:{community_id}`
- Tag filtering via query params: `?tag=announcements`
- Updated CD-036 documentation with Session 181 clarifications

## 3. Tabbed UI for Community Pages (Not Sub-Routes)
**Type:** Implementation
**Topics:** astro, ui

**Trigger:** Designing community page structure with Feed, Courses, Resources, Members sections.

**Options Considered:**
1. Sub-routes (`/community/[slug]/feed`, `/community/[slug]/members`) - Better SEO, deep-linking
2. Tabbed single page with client-side switching ← Chosen - Cozy, focused feel

**Decision:** Use tabbed interface within a single page. All content pre-rendered, tabs switch visibility client-side.

**Rationale:**
- Client feedback: "cozy, focused group feel"
- Faster navigation between sections (no page loads)
- Content is cohesive (all about one community)
- Sub-routes would feel fragmented

**Consequences:**
- Created `CommunityTabs.tsx` component
- Tabs: Feed, Courses, Resources, Members
- The Commons hides Courses tab (platform-wide, not course-linked)
- All data passed as props (or fetched client-side)

## 4. Two-Stage CD-036 Implementation
**Type:** Strategic
**Topics:** d1, workflow

**Trigger:** CD-036 requires both new tables (communities, progressions) and modifications to existing tables (courses.progression_id NOT NULL).

**Options Considered:**
1. Implement all at once - Clean schema, single migration ← Rejected (high risk)
2. Stage 1: Communities only, Stage 2: Progressions ← Chosen
3. JSON static data first, then schema - Zero risk but not "real" data

**Decision:** Two-stage implementation:
- **Stage 1 (Session 181):** communities, community_members, community_resources tables + seed + pages + API
- **Stage 2 (Future):** progressions table + courses.progression_id migration

**Rationale:**
- Stage 1 is non-breaking (additive only)
- Stage 2 requires careful course migration planning
- Allows shipping community features immediately
- Isolates risk to Stage 2

**Consequences:**
- Completed Stage 1 this session (3 tables, 4 communities, 19 members, 11 resources)
- Courses not yet linked to communities (still via category)
- Stage 2 documented in RFC/CD-036/RFC.md for future work
