# Session Decisions - 2026-02-25

## 1. Creator Self-Certification via Existing Endpoint
**Type:** Implementation
**Topics:** astro, auth

**Trigger:** Creators need to teach their own courses during the "priming the pump" phase before S-Ts are available.

**Options Considered:**
1. New dedicated endpoint for creator self-certification
2. Extend existing POST `/api/me/courses/[id]/student-teachers` with conditional branch ← Chosen

**Decision:** Extended the existing POST handler with a 3-line conditional: when `body.user_id === creatorId`, skip the enrollment-completion check. The course ownership check already gates this to the actual creator.

**Rationale:** The existing endpoint already handles all S-T creation logic (duplicate checks, record creation, response formatting). A 10-line branch is simpler than a parallel code path with duplicated logic.

**Consequences:** Single code path for all S-T creation. The only behavioral difference is the enrollment check is skipped for self-certification.

## 2. Auto-Set `can_teach_courses` During Self-Certification
**Type:** Implementation
**Topics:** auth, d1

**Trigger:** After a creator self-certifies as S-T, the "Teaching" nav item needs to appear without admin intervention.

**Options Considered:**
1. Require admin to manually enable `can_teach_courses` flag
2. Auto-set `can_teach_courses = 1` during self-certification ← Chosen

**Decision:** After creating the S-T record, auto-update `users SET can_teach_courses = 1 WHERE id = ? AND can_teach_courses = 0`. The `AND can_teach_courses = 0` clause makes it a no-op for users who already have the flag.

**Rationale:** The intended workflow is frictionless: creator decides to teach → self-certifies → immediately sees Teaching nav. Requiring admin intervention defeats the purpose of self-service.

**Consequences:** Teaching nav appears immediately after self-certification. No admin action needed.

## 3. Payment Split: Creator Teaching Gets 85/15 (Not 70/15/15)
**Type:** Implementation
**Topics:** stripe

**Trigger:** When a creator is also the S-T, the payment split logic would set `instructorType = 'student_teacher'`, creating a confusing double-split where both `creator_royalty` and `student_teacher` shares go to the same person.

**Options Considered:**
1. Let the 70/15/15 split happen and reconcile on the dashboard ← Not chosen
2. Keep `instructorType = 'creator'` when S-T is the course creator ← Chosen

**Decision:** Compare `studentTeacher.user_id === course.creator_id`. If true, keep `instructorType = 'creator'` (85/15 split). Creator Dashboard already queries `recipient_type IN ('creator', 'creator_royalty')`, so earnings display cleanly.

**Rationale:** Clean single payment record. No confusing double-entries. Creator earnings show correctly on Creator Dashboard without special-case aggregation.

**Consequences:** 3-line change in `create-session.ts`. Creator teaching their own course sees all earnings on Creator Dashboard under the standard creator split.

## 4. "Course Creator" Badge: Purple, Matching Existing Pattern
**Type:** Implementation
**Topics:** astro

**Trigger:** Students need a way to identify when a teacher is also the course author in S-T listings.

**Options Considered:**
1. "Creator" badge (ambiguous in S-T context)
2. "Course Creator" badge ← Chosen
3. "Author" badge

**Decision:** Purple badge (`bg-purple-100 text-purple-800` / dark mode `bg-purple-900/30 text-purple-300`) with text "Course Creator". Full-size in CourseTabs, compact (`text-[10px]`) in CourseSTList grid cards.

**Rationale:** "Course Creator" is more informative than bare "Creator" in the S-T listing context. Purple matches the established Creator badge pattern used in `UserCard.tsx` and `UserProfileHeader.tsx`.

**Consequences:** Badge is computed server-side via `CASE WHEN` in the SSR loader query — no additional API call needed.
