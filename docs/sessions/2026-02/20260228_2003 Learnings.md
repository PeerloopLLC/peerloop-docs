# Session Learnings - 2026-02-28

## 1. Static Link Analysis of Astro/React Requires Import Following
**Topics:** astro, tooling

**Context:** Building a route adjacency scanner that extracts navigation links from `.astro` page files.

**Learning:** Just scanning the `.astro` page files misses the majority of navigation links. Links are often embedded in imported `.tsx` React components like `CourseTabs`, `StudentDashboard`, `CourseHero`, etc. Following imports 2 levels deep catches most navigation links without excessive noise.

**Pattern:** Parse `import X from '...'` statements, resolve aliases (`@components/`, `@layouts/`), try file extensions (`.tsx`, `.ts`, `.astro`, `/index.tsx`), then scan those files recursively with a depth limit.

## 2. Layout Infrastructure Must Be Excluded from Import Following
**Topics:** astro, tooling

**Context:** When the scanner followed all imports (including layouts), every page showed `/favicon.svg`, `/privacy`, `/terms`, `/cookies`, etc. â€” all coming from `BaseHead.astro` and `Footer.astro` via the layout chain.

**Learning:** Layout imports (`AppLayout` â†’ `Footer`, `BaseHead`, `AppNavbar`) inject links that are already represented by the shared navigation model. Following them duplicates and pollutes page-specific link data. The fix is to skip layout/infrastructure imports during page scanning and represent their links separately as "pseudo-pages."

## 3. Pseudo-Page Architecture for Shared Navigation Components
**Topics:** astro, docs-infra

**Context:** Shared components (Footer, AppNavbar, AdminNavbar, DiscoverSlidePanel, FeedSlidePanel, UserAccountDropdown) emit links but are not navigable destinations themselves.

**Learning:** Modeling shared components as "pseudo-pages" â€” FROM rows without TO columns in the adjacency matrix â€” cleanly separates shared navigation from page-specific navigation. This surfaces broken links in shared components (Footer has 15 broken targets out of 18 total links) that would otherwise be invisible. The `foot` and `nav` labels at intersections tell you *why* a page reaches a target.

## 4. Broken Link Surfacing Is More Valuable Than Suppression
**Topics:** tooling, docs-infra

**Context:** Initial scanner implementation suppressed footer/marketing links (`/privacy`, `/terms`, `/cookies`, `/about`, etc.) as "not yet implemented." Also suppressed `/404` redirects.

**Learning:** These are real broken links that a user clicking them will experience as Astro 404 pages. Suppressing them hides actionable information. The final scanner only excludes truly non-navigational paths (`/favicon.svg`, `/og-image.png` from `<head>` metadata) and surfaces everything else â€” including stale route references from pre-refactor code (`/dashboard/learning`, `/workspace`, `/profile/[handle]`).

## 5. Link Type Classification via URL Pattern Heuristics
**Topics:** tooling

**Context:** The scanner extracts `href=` attributes but can't see the surrounding DOM to know if a link is a button, card, tab, or breadcrumb.

**Learning:** Target URL patterns are a reasonable proxy for link type classification without AST parsing. Tab patterns (`/course/[slug]/feed`, `/teaching/sessions`), card patterns (`/course/[slug]`, `/@[handle]`), and button patterns (`/signup`, `/session/[id]`) can be matched with regex. This gets ~80% classification right. The remaining generic `link` types (39 out of ~200 total) are genuinely ambiguous in-page anchors.
