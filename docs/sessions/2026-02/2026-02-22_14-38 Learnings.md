# Session Learnings - 2026-02-22 (Session 252)

## 1. Resend Recipient Restriction Without Verified Domain
**Topics:** resend, deployment

**Context:** Tested the Resend API key from `.dev.vars` by sending a test email to `fgorrie@bio-software.com`. Resend returned a 403 error.

**Learning:** Without a verified sending domain in Resend, emails can only be sent to the Resend account owner's email address. The error message reveals the owner's email (in this case `fraser@frasergorrie.com`). Domain verification is purely a DNS proof-of-ownership check — it does not require a live website on that domain. The client can verify `peerloop.com` today by adding 3 DNS records, even though the site is still on `staging.peerloop.pages.dev`.

## 2. Resend API Key Scoping
**Topics:** resend

**Context:** Tried to query `GET /domains` on the Resend API to check domain verification status before sending a test email.

**Learning:** Resend API keys can be scoped to "sending only" — they'll return 401 with `restricted_api_key` error for non-send operations like listing domains. The key works fine for `POST /emails` but cannot query account metadata. This is a Resend dashboard setting when creating the key.

## 3. Stripe Hosted Onboarding Supports Branding
**Topics:** stripe

**Context:** Client asked about Stripe's embedded onboarding component vs our current hosted (redirect) approach. We noticed the onboarding screens looked generic during test mode.

**Learning:** Stripe's hosted Connect onboarding (Account Links) supports branding customization via **Stripe Dashboard → Settings → Branding**. You can set: logo/icon (appears on the left panel), brand color (the purple panel), accent color (buttons/links), and display name. The generic appearance was a combination of test mode banners and unconfigured branding, not a limitation of the hosted approach.

## 4. Domain Verification Unblocks Email Before Go-Live
**Topics:** resend, deployment

**Context:** Assumed Resend domain verification was a go-live task, since the production domain isn't decided yet.

**Learning:** The `from` address domain and the website domain are independent. You can send from `notifications@peerloop.com` while the site runs on `staging.peerloop.pages.dev`. Domain verification only requires DNS records (TXT + CNAME) — no web server, no HTTPS cert, no website. This means email testing can be unblocked immediately, not deferred to go-live.

## 5. Schema Tables for Onboarding Use Separate Table Pattern
**Topics:** d1, architecture

**Context:** Designed the member onboarding profile schema — needed to store user goals, referral source, and profession alongside the existing `users` table with 30+ columns.

**Learning:** Using a separate `member_profiles` table (keyed by `user_id` PK) rather than adding columns to `users` keeps concerns separated. The `users` table is read on every authenticated page load via `/api/me/full`, so keeping it lean matters. The onboarding data is joined with a LEFT JOIN only when needed, and the single `onboarding_completed_at` field is the only piece surfaced in `CurrentUser` for conditional menu visibility.

## 6. PeerloopTokenPayload Uses `.userId` Not `.sub`
**Topics:** auth, testing

**Context:** Created the `/api/me/onboarding-profile` endpoint using `authSession.sub` to get the user ID, following standard JWT convention. Tests failed.

**Learning:** Peerloop's `PeerloopTokenPayload` extends `JWTPayload` but adds a custom `userId` field. All existing endpoints use `authSession.userId`, not `authSession.sub`. While `sub` exists on the base `JWTPayload` type, it may not be populated. Always follow existing endpoint patterns rather than assuming standard JWT field names.
