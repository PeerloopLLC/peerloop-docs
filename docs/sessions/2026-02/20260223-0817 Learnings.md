# Session Learnings - 2026-02-23 (Session 261)

## 1. PLAN Drift: Outdated Route References
**Topics:** docs-infra, astro

**Context:** PLAN.md's CURRENTUSER.APP section referenced `/dash/*` routes (CDIS → `/dash/discover`, DCRS → `/dash/courses`, etc.) and a `DashLayout` component.

**Learning:** None of these exist. The actual routes are `/discover`, `/courses`, `/messages`, `/notifications`, `/learning`, `/teaching`, `/creating`. All use `AppLayout` with `AppNavbar` (which initializes CurrentUser). The PLAN had drifted from the codebase during the UI pivot to Twitter-style navigation, creating phantom work items.

**Pattern:** Regular PLAN audits against actual code prevent accumulating stale sections. When route architecture changes significantly, search PLAN.md for route references.

## 2. AdminNavbar Was Stateless — No Auth Awareness
**Topics:** auth, astro

**Context:** While auditing CURRENTUSER.ADMIN, discovered that `AdminNavbar.tsx` had zero CurrentUser integration — no `initializeCurrentUser()`, no `getNetworkState()`, no user identity display.

**Learning:** AdminNavbar was purely navigation-based (hardcoded menu items, path-based highlighting). This meant: (1) no session expiry detection — an admin with an expired JWT would see the navbar but get 401s on every API call, (2) no admin identity display — the navbar showed "Peerloop Admin" generically with no indication of who was logged in.

**Pattern:** When adding auth-aware components to a layout, check ALL layouts that need auth context, not just the primary one. AppLayout had full CurrentUser integration; AdminLayout was overlooked.

## 3. Platform vs Course Scope as a Documentation Gap
**Topics:** auth, docs-infra

**Context:** When creating the comprehensive ROLES.md, the user pointed out that the document didn't distinguish between platform-level roles (Admin, Visitor, base User) and course-scoped roles (Student *of* Course X, S-T *for* Course X).

**Learning:** The two-layer permission model (capability flags vs course relationships) is well-implemented in code (`canCreateCourses` vs `isCreatorFor(courseId)`) but wasn't clearly documented as an architectural concept. The Moderator role is a hybrid — currently a global flag (`can_moderate_courses`) but intended to eventually have per-course assignment too.

**Pattern:** When documenting role systems, always call out the scope dimension explicitly. "Student" means different things depending on whether you're asking "can this user take courses?" (platform) vs "is this user enrolled in *this* course?" (course-scoped).

## 4. Stripe Seed File Is Opt-In by Design
**Topics:** stripe, d1

**Context:** After adding real Stripe test-mode account IDs to `migrations-dev/0002_seed_stripe.sql`, the local D1 reset + seed workflow required an extra manual step (`npm run db:seed:stripe:local`).

**Learning:** The `db:setup:local` script intentionally chains reset → migrate → dev seed but excludes the Stripe seed. This allows testing both "fresh user with no Stripe" and "user with connected Stripe account" flows. The Stripe seed is opt-in because not every dev session needs Stripe accounts linked.
