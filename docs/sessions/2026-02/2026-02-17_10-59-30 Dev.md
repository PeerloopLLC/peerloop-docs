### Initial Fix: `<span>` with onClick

**Claude:** Replaced the inner `<a>` with a `<span role="link">` using `onClick` + `window.location.href` for navigation. Added `tabIndex={0}` and keyboard handler for accessibility. Fix resolved the hydration error.

### Stretched-Link Pattern Upgrade

**User:** Are there other concerns using span or other HTML with click handlers?

**Claude:** Identified 5 drawbacks of the `<span>` + `onClick` approach: no middle-click/Ctrl+click, no right-click context menu, no status bar preview, bypasses Astro View Transitions, and no crawlability. Proposed the "stretched-link" pattern instead — outer `<div>` with `position: relative`, primary `<a>` using `after:absolute after:inset-0`, secondary `<a>` with `relative z-10`. Applied to CourseCard.tsx, preserving all native link behaviors for both links.

### Codebase-Wide Navigation Audit

**User:** Are we using navigable HTML elements with onClick() events elsewhere that we should extend this technique to?

**Claude:** Launched an Explore agent to audit the full codebase. Found no other nested `<a>` patterns (CourseCard was the only one). Found 15 instances of `window.location.href` across the codebase. Categorized them into: intentional (auth boundaries, external URLs — 8 instances), improvable navigation (6 instances using `<div>` + `onClick` instead of `<a>` tags), and `window.location.reload()` calls (4 instances).
