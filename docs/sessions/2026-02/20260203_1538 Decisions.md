# Session 170 Decisions - 2026-02-03

## 1. User Account Menu: Slide Panel Instead of Dropdown
**Type:** Implementation
**Topics:** ui, navigation

**Trigger:** UserAccountDropdown initially rendered as a dropdown menu appearing above the trigger. User requested it match the other panel patterns.

**Options Considered:**
1. Dropdown menu (above trigger) - Original implementation
2. Slide-out panel anchored to bottom ← Chosen
3. Modal dialog

**Decision:** Convert UserAccountDropdown to a slide-out panel pattern, anchored at the bottom of the screen, consistent with FeedSlidePanel and DiscoverSlidePanel.

**Rationale:**
- Consistency with existing panel UI patterns
- Better mobile experience
- More room for content (profile links, settings, notifications)
- Unified panel state management in AppNavbar

**Consequences:**
- Component now uses `createPortal` to render outside navbar container
- State lifted to AppNavbar (`isUserPanelOpen`)
- Opening user panel closes other panels (mutually exclusive)

## 2. Custom Font Size: @theme Over @utility
**Type:** Implementation
**Topics:** tailwind

**Trigger:** Needed `text-xxs` (10px) for description text, smaller than Tailwind's `text-xs` (12px).

**Options Considered:**
1. Inline arbitrary value `text-[10px]`
2. `@utility text-xxs { ... }` - Explicit CSS utility ← Initially used as fallback
3. `@theme { --text-xxs: 0.625rem; }` ← Chosen

**Decision:** Use `@theme` CSS variable approach for custom font size.

**Rationale:**
- More idiomatic for Tailwind v4's CSS-first configuration
- Integrates with the design token system
- Allows related properties (line-height, letter-spacing) via `--text-xxs--line-height`
- `@utility` was only needed during dev because server wasn't restarted

**Consequences:**
- Custom font size defined in global.css `@theme` block
- Works like any built-in text size utility
- Requires dev server restart when first added

## 3. Navigation Refactoring: Role-Based Dashboard Links
**Type:** Implementation
**Topics:** navigation, ui

**Trigger:** Implementing tech-021 URL routing scheme required updating AppNavbar for new route patterns.

**Options Considered:**
1. Keep single "Workspace" link for all teaching/creating roles
2. Split into separate Learning/Teaching/Creating links ← Chosen
3. Dropdown menu for workspace options

**Decision:** Replace single "Workspace" link with three role-based links: Learning (`canTakeCourses`), Teaching (`canTeachCourses`), Creating (`canCreateCourses`).

**Rationale:**
- Clearer navigation for users with multiple roles
- Direct access without intermediate page
- Follows "Bare = My" routing principle from tech-021

**Consequences:**
- Users see only links for their active roles
- Learning visible to all students (most users)
- Teaching visible only to certified S-Ts
- Creating visible only to course creators

## 4. Public Profile Field Visibility Indicators
**Type:** Implementation
**Topics:** ui, settings, privacy

**Trigger:** User requested a way to indicate which settings fields appear on public profile cards.

**Options Considered:**
1. Separate "Public" and "Private" sections
2. Visual badge next to each public field ← Chosen
3. Tooltip explaining visibility on hover
4. Checkbox to control each field's visibility

**Decision:** Add a "Public" badge (eye icon + text) next to labels of fields that will appear on the user's public profile card.

**Rationale:**
- Non-intrusive visual indicator
- GDPR-friendly transparency
- Doesn't require interaction to understand
- Works well with existing form layout

**Consequences:**
- New `<PublicBadge />` component
- Applied to: name, handle, title, tagline, bio, teaching_philosophy, website, location, social links
- Not applied to: email, notification prefs, privacy settings
