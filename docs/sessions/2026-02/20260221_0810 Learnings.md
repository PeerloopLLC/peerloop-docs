# Session Learnings - 2026-02-21 (Session 236)

## 1. Astro Cloudflare Adapter Exposes `platformProxy` for Remote D1 Dev
**Topics:** astro, cloudflare, d1

**Context:** Needed to run the local Astro dev server against the remote staging D1 database so developers can reproduce bugs seen by staging users.

**Learning:** The `@astrojs/cloudflare` adapter accepts a `platformProxy` option that passes directly through to wrangler's `getPlatformProxy()`. This supports `environment` (to select a wrangler.toml environment like `preview`) and `remoteBindings: true` (to connect to actual remote Cloudflare services instead of local emulation). This means you can switch between local and remote D1 without a separate config file or build step.

**Pattern:**
```js
// astro.config.mjs
const platformProxy = process.env.USE_STAGING_DB
  ? { environment: 'preview', remoteBindings: true }
  : undefined;

adapter: cloudflare({ platformProxy })
```

## 2. Wrangler Has Two Auth Paths — getPlatformProxy vs CLI
**Topics:** cloudflare, d1, cc-workflow

**Context:** The `dev:staging` dev server connected to remote D1 successfully, but `wrangler d1 execute --remote` commands from Claude Code failed with "non-interactive environment" errors.

**Learning:** Wrangler has two distinct auth mechanisms:
- **`getPlatformProxy()`** (used by the Astro dev server): Reads cached OAuth tokens from `~/.wrangler/config/`. Works from any process, including non-interactive ones.
- **CLI commands** (`wrangler d1 execute --remote`, `wrangler d1 migrations apply --remote`): In non-interactive environments, requires `CLOUDFLARE_API_TOKEN` env var. Cannot trigger the browser OAuth flow.

Uncommenting `CLOUDFLARE_API_TOKEN` in `.dev.vars` unblocked all remote wrangler CLI commands from within Claude Code sessions.

## 3. CLOUDFLARE_ACCOUNT_ID Not Required for Wrangler Operations
**Topics:** cloudflare

**Context:** `.dev.vars` had `CLOUDFLARE_ACCOUNT_ID` commented out. Tested whether it was needed for remote D1 queries.

**Learning:** Wrangler resolves the account ID automatically from the API token. All remote operations (`d1 execute`, `d1 migrations apply`, `getPlatformProxy`) work with only `CLOUDFLARE_API_TOKEN` set. The account ID appeared in API paths (e.g., `/accounts/1dcd33...`) without being explicitly configured. Keep `CLOUDFLARE_ACCOUNT_ID` in `.dev.vars.example` as documentation, but it doesn't need to be uncommented.

## 4. Script Files Need Dedicated Documentation Beyond npm Wrappers
**Topics:** docs-infra, workflow

**Context:** While creating `SCRIPTS.md`, audited all 21 files in `scripts/` against existing documentation.

**Learning:** 13 of 21 script files had zero documentation — the audit scripts (`audit-api-coverage.mjs`, `audit-test-sufficiency.mjs`, `reconcile-planned-apis.mjs`), page tooling (`populate-page-metadata.ts`, `validate-page-spec.ts`), and integration tests (`run-feed-isolation-test.js`, `test-feed-isolation.sh`) were completely undiscoverable. The npm scripts that wrap some of them were documented in CLI-REFERENCE.md, but the 9 standalone scripts (no npm wrapper) had no documentation anywhere. A dedicated SCRIPTS.md bridges this gap by cataloging both npm scripts and raw script files in one place.
