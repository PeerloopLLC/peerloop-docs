# Session Decisions - 2026-02-16

## 1. Stay with JWT Auth, Defer Astro Sessions
**Type:** Strategic
**Topics:** auth, astro, cloudflare

**Trigger:** Dev server warning about Cloudflare KV SESSION binding led to evaluating whether to adopt Astro's built-in session management.

**Options Considered:**
1. Migrate to Astro Sessions (KV-backed, server-side state)
2. Keep custom JWT auth (stateless, cookie-based) ← Chosen
3. Hybrid approach (JWT for identity, KV for authorization checks)

**Decision:** Stay with custom JWT authentication. The system is built, tested across 169 API test files, and works. Astro Sessions evaluated and documented as a post-MVP option.

**Rationale:**
- JWT is already built and integrated across all endpoints
- Migration cost is high with no user-facing benefit
- The stale-role gap (up to 15 min) is mitigated by short token TTL and D1 re-checks on critical endpoints
- No infrastructure dependency (no KV cost or provisioning)
- Framework-agnostic — portable if we leave Astro

**Consequences:**
- Created `docs/tech/tech-027-auth-sessions.md`
- Added note to BEST-PRACTICES.md §7 Authentication
- Added DEPLOY-WARNINGS deferred block to PLAN.md
- Stale-role gap documented with mitigation strategies

## 2. Plain `<img>` + R2 for Images, Defer Cloudinary
**Type:** Implementation
**Topics:** cloudflare, astro

**Trigger:** Astro's sharp warning prompted discussion of how images are handled and whether optimization is needed.

**Options Considered:**
1. Cloudinary (best-in-class transforms, new vendor)
2. Cloudflare Image Resizing (native to CF, pull from R2)
3. Keep plain `<img>` tags with R2 storage ← Chosen

**Decision:** Keep plain `<img>` tags rendering R2 URLs with no optimization pipeline for MVP.

**Rationale:** Low image volume (~4 courses, <10 users). Adding an optimization pipeline adds vendor complexity with no measurable benefit at this scale.

**Consequences:**
- Astro image service set to `passthrough` via adapter config
- Created `docs/tech/tech-028-image-handling.md`
- Added IMAGE-OPTIMIZE deferred block to PLAN.md with re-evaluation triggers

## 3. Provision KV Namespace Despite Not Actively Using It
**Type:** Implementation
**Topics:** cloudflare

**Trigger:** SESSION KV binding warning from Astro adapter, plus discussion of KV's potential utility.

**Options Considered:**
1. Suppress the warning without provisioning KV
2. Provision KV namespace and make it available ← Chosen

**Decision:** Create KV namespaces (production + preview) and add bindings to `wrangler.toml`. KV is available for future use even though no application code uses `Astro.session` yet.

**Rationale:**
- Free tier is generous (100K reads/day)
- Opens up future use cases (feature flags, rate limiting, caching, tokens with TTL)
- Prevents deploy failures from missing binding
- Local emulation works automatically via Wrangler

**Consequences:**
- Created two namespaces: `SESSION` (prod) and `SESSION_preview` (preview)
- Added `[[kv_namespaces]]` to all three wrangler.toml environment sections
- Created `docs/tech/tech-029-cloudflare-kv.md`
- Created `src/pages/api/health/kv.ts` health endpoint
- Added KV-CONSISTENCY deferred block to PLAN.md

## 4. Use `imageService: 'passthrough'` on Adapter Constructor
**Type:** Implementation
**Topics:** astro, cloudflare

**Trigger:** Sharp warning persisted after setting `image.service` in Astro's top-level config.

**Options Considered:**
1. Set `image.service.entrypoint: 'no-op'` in Astro config (didn't work — adapter overrides)
2. Set `imageService: 'passthrough'` on `cloudflare()` constructor ← Chosen
3. Set `imageService: 'custom'` and keep our own config

**Decision:** Pass `imageService: 'passthrough'` to the `cloudflare()` adapter constructor.

**Rationale:** The adapter's `setImageConfig()` overrides Astro's top-level image config. The only way to control the image service AND suppress the warning is through the adapter's own `imageService` option. `passthrough` is the correct value for "no optimization."

**Consequences:**
- Sharp warning suppressed
- Images pass through unmodified (same behavior as before)
- Documented in `docs/tech/tech-016-astrojs.md` and `docs/tech/tech-028-image-handling.md`

## 5. Keep react-dom/server.edge Workaround Until @astrojs/cloudflare v13 Stable
**Type:** Implementation
**Topics:** astro, cloudflare, deployment

**Trigger:** Version check revealed the upstream fix (PR #12996) exists but hasn't shipped in the stable adapter line.

**Options Considered:**
1. Remove the workaround (would break production builds)
2. Upgrade to v13 beta (unstable, Astro 6 breaking changes)
3. Keep the workaround with documented removal criteria ← Chosen

**Decision:** Keep the `react-dom/server.edge` Vite alias workaround. Document that it's fragile (works by alias resolution order coincidence) and set clear removal criteria.

**Rationale:** The workaround works — verified by build output containing `.edge`. Removing it would reintroduce the MessageChannel error. Upgrading to Astro 6 beta introduces unknown breaking changes. The pragmatic path is to keep it and monitor for v13 stable release.

**Consequences:**
- Workaround retained in `astro.config.mjs`
- `docs/tech/tech-016-astrojs.md` updated with full analysis of why it's fragile and when to remove
- Removal criteria: `@astrojs/cloudflare` v13+ stable AND no `react-dom/server.browser` in adapter source
