# Session Learnings - 2026-02-01

## 1. Component Folder Reorganization Follows "Who + What" Mental Model
**Topics:** astro, workflow

**Context:** Reorganizing 34 component subfolders with confusing duplicates (creator/ vs creators/, dash/ vs dashboard/, teachers/ vs teaching/).

**Learning:** The most intuitive folder structure answers two questions: "Who uses this?" (admin, creator, student-teacher, visitor) and "What type?" (profiles vs studio/workspace, ui primitive vs feature). This eliminates the "which folder?" cognitive overhead.

**Pattern:**
```
src/components/
├── ui/                      # Design system primitives (what type)
├── layout/                  # App shell components (what type)
├── creators/
│   ├── profiles/            # Public-facing (what purpose)
│   └── studio/              # Authoring tools (what purpose)
├── student-teachers/
│   ├── profiles/            # Public-facing
│   └── workspace/           # Management tools
└── marketing/               # Visitor-facing
```

## 2. Git mv Preserves History During Large Reorganizations
**Topics:** workflow

**Context:** Moving 50+ component files across folders while preserving git history for blame/log.

**Learning:** Using `git mv` instead of manual move + add ensures git tracks the file as renamed rather than deleted + created. This preserves blame history and makes PR diffs cleaner (shows as rename with percentage similarity).

**Pattern:**
```bash
# Good - preserves history
git mv src/components/old/File.tsx src/components/new/File.tsx

# Bad - loses history
mv src/components/old/File.tsx src/components/new/File.tsx
git add .
```

## 3. Barrel Exports Require Three Levels of Updates
**Topics:** astro, workflow

**Context:** After moving components, TypeScript found broken imports even though files existed at new locations.

**Learning:** When reorganizing components, barrel exports need updating at three levels:
1. **Leaf level** - The moved folder's index.ts (e.g., `creators/profiles/index.ts`)
2. **Parent level** - The domain folder's index.ts (e.g., `creators/index.ts` re-exporting from subfolders)
3. **Consumer level** - All files importing from the barrel

Missing any level causes import failures that TypeScript catches.

## 4. Page JSON Files Contain Component Paths for Tooling
**Topics:** astro, workflow

**Context:** TypeScript passed but page JSON files still referenced old component paths in `tsxComponents` arrays.

**Learning:** Peerloop's `src/data/pages/**/*.json` files contain `metadata.tsxComponents` arrays listing component file paths. These are used by PageSpecView and tooling (like `npm run pages-map:write`). After component moves, these paths need updating or tooling/documentation will be stale.

**Pattern:**
```json
{
  "metadata": {
    "tsxComponents": [
      "src/components/creators/profiles/CreatorBrowse.tsx"  // Must match actual path
    ]
  }
}
```

## 5. Component Internal Imports Use Mixed Path Styles
**Topics:** astro, workflow

**Context:** After updating all `@components/` imports, TypeScript still found errors from relative imports within components.

**Learning:** Components use a mix of path styles:
- **Alias imports:** `@components/charts/AreaChart` (from tsconfig paths)
- **Relative imports:** `../teaching/AvailabilityQuickView` (for nearby files)
- **@/ imports:** `@/components/creators/CreatorCard` (another alias style)

All three styles need updating during reorganization. Grep patterns must account for all variations.

**Pattern:**
```bash
# Find all import variations
grep -r "from ['\"]@components/" src/
grep -r "from ['\"]@/components/" src/
grep -r "from ['\"]\.\./teaching/" src/components/
```

## 6. Subfolder Naming Convention: Profiles vs Studio vs Workspace
**Topics:** workflow

**Context:** Deciding how to name subfolders when merging creator/ + creators/ and teachers/ + teaching/.

**Learning:** Consistent naming helps discoverability:
- **profiles/** - Public-facing display components (cards, browse, profile pages)
- **studio/** - Content creation/authoring tools (editors, modals)
- **workspace/** - Task/management tools (availability, earnings, students)

This pattern scales: any role that has both "public face" and "private tools" can use profiles/ + studio/ or workspace/.
