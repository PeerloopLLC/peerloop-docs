# Session Decisions - 2026-02-19

## 1. Retire MBA-2017, Remove REST Fallback Entirely
**Type:** Strategic
**Topics:** cloudflare, d1, workflow

**Trigger:** MBA-2017 (MacBook Air 2017, macOS 12) is being replaced by a new Mac Mini M4 (24GB). Both machines now have full Cloudflare Workers/D1/R2 support.

**Options Considered:**
1. Rename MBA-2017 references to new machine name, keep REST fallback code dormant
2. Remove REST fallback entirely, delete all MBA-specific infrastructure ← Chosen
3. Keep fallback as opt-in "remote-only development" feature

**Decision:** Delete `d1-rest.ts`, `d1-rest-fallback.test.ts`, and all conditional logic (REST fallback branches in `getDB()`, `getEnv()`, `needsNodeAdapter()`, MBA skip logic in tests). Uninstall `@astrojs/node`.

**Rationale:** Dead code is a maintenance liability. The REST fallback was never needed in production — only for local dev on a machine that no longer exists. Keeping it "just in case" would mean maintaining 500+ lines of code with no active users. Both machines have identical capabilities, so no conditional logic is needed.

**Consequences:** ~500 lines of code deleted. `getDB()` reduced from 25 lines to 8. `getEnv()` reduced from 20 lines to 8. `astro.config.mjs` lost ~20 lines of adapter detection. Test helpers simplified (no more `isMBA2017()`, `skipOnMBA`). `@astrojs/node` package removed.

## 2. Machine Naming Convention: MacMiniM4-Pro and MacMiniM4
**Type:** Strategic
**Topics:** workflow

**Trigger:** With two Mac Minis, the old name "MacMini" is ambiguous. Need distinct, greppable names.

**Options Considered:**
1. `MacMini-64` / `MacMini-24` (RAM-based)
2. `MacMiniM4-Pro` / `MacMiniM4` (chip-based) ← Chosen
3. `Primary` / `Secondary` (role-based)
4. `Jamess-Mini` / `Livings-Mini` (hostname-based)

**Decision:** Use `MacMiniM4-Pro` and `MacMiniM4` as standardized machine names across all documentation, hooks, test helpers, and commit messages.

**Rationale:** Chip-based names are specific, unambiguous, and future-proof (if a third machine is added, its chip differentiates it). RAM-based names break if RAM is upgraded. Role-based names don't convey hardware. Hostname-based names are too informal for documentation.

**Consequences:** Updated machine detection hook (`detect-machine.sh`), 7 CC skill files, `CLAUDE.md`, `tech-013-devcomputers.md`, `DEVELOPMENT-GUIDE.md`, test helpers (`machine.ts`, `skip.ts`), and `DECISIONS.md`.

## 3. Hostname-Based Machine Detection Over macOS Version
**Type:** Implementation
**Topics:** testing, cloudflare

**Trigger:** Both Mac Minis run macOS 15+. The old detection (`major >= 13 ? 'MacMini' : 'MBA-2017'`) can't distinguish them.

**Options Considered:**
1. Continue using macOS version detection (both return same result)
2. Use hostname detection (`Jamess-Mac-mini` vs `Livings-Mac-mini`) ← Chosen
3. Use serial number or hardware UUID

**Decision:** Primary detection reads `~/.claude/.machine-name` cache file (written by startup hook). Fallback uses `hostname` command to match known patterns.

**Rationale:** Hostnames are explicitly set per machine and stable across OS upgrades. The cache file provides instant detection without spawning a subprocess. Serial numbers require `system_profiler` which is slower.

**Consequences:** `tests/helpers/machine.ts` rewritten to check cache file for new names, fallback to hostname. `detect-machine.sh` updated with new hostname patterns.
