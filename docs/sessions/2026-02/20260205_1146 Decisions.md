# Session Decisions - 2026-02-05

## 1. Remove `/teachers` Personal Route
**Type:** Implementation
**Topics:** astro, architecture

**Trigger:** Reviewing the routing doc's TODO for `teachers.astro` and questioning its usefulness.

**Options Considered:**
1. Keep `/teachers` - Shows S-Ts who have taught the user
2. Remove `/teachers` - Users navigate course-first via `/course/[slug]/teachers` ← Chosen

**Decision:** Remove `/teachers` from the routing plan. Students access teachers through course context, not as a standalone collection.

**Rationale:** The primary user journey is course-centric. A student thinks "I need help with Python 101" not "show me all my teachers." The `/course/[slug]/teachers` route already handles this. The `/learning` dashboard can show recent S-Ts inline if needed.

**Consequences:** Updated `tech-021-url-routing.md` to remove `/teachers` from personal routes, changed "Three levels of teachers" to "Two levels of teachers" with explanatory note.

## 2. Remove `/certificates` Personal Route
**Type:** Implementation
**Topics:** astro, architecture

**Trigger:** Applying the same analysis to `/certificates` - is it needed as a standalone page?

**Options Considered:**
1. Keep `/certificates` - Dedicated trophy case page
2. Put on profile `/@handle` - Public shareable
3. Keep on `/learning` dashboard - Already there ← Chosen

**Decision:** Remove `/certificates` from the routing plan. Certificates are already displayed on the `/learning` dashboard.

**Rationale:** Certificates are already visible on `/learning`. A dedicated page adds navigation surface without clear user benefit. If users want to share certificates, individual certificate URLs (`/verify/[id]`) are shareable.

**Consequences:** Updated `tech-021-url-routing.md`, removed from UserAccountDropdown slide panel.

## 3. Add `/feed` Aggregated Timeline
**Type:** Strategic
**Topics:** stream, architecture, astro

**Trigger:** Discussing the "My Feeds" slide panel and whether an aggregated view would be valuable.

**Options Considered:**
1. Individual feeds only - Users hop between communities/courses
2. Add `/feed` aggregated timeline - One place to see everything ← Chosen

**Decision:** Implement `/feed` as an aggregated timeline using Stream's `timeline:{userId}` feed. Add "Home Feed" as the top item in FeedSlidePanel.

**Rationale:** Modern UX expectation (Twitter, LinkedIn). Stream already has timeline aggregation built-in via follows. Reduces friction for users to catch up on activity across all their subscriptions.

**Consequences:**
- Created `/api/feeds/timeline.ts` endpoint
- Created `HomeFeed.tsx` component with filter tabs
- Created `/feed` page requiring auth
- Added "Home Feed" link at top of FeedSlidePanel
- Added `follow()`, `unfollow()`, `following()` methods to `stream.ts`
- Wired follow/unfollow into Stripe webhook and community join API

## 4. Auto-Join The Commons at Registration
**Type:** Strategic
**Topics:** stream, auth, architecture

**Trigger:** The `/feed` aggregated timeline would be empty for new users without follows. The Commons is the system community all users should see.

**Options Considered:**
1. Manual join - User discovers and joins The Commons themselves
2. Auto-join at registration - All new users start as members ← Chosen
3. Auto-follow without membership - Just Stream follow, not DB membership

**Decision:** Auto-join The Commons for all new users during registration, including creating the community_members record AND establishing the Stream follow.

**Rationale:** The Commons is the platform-wide community. New users should immediately see activity in their `/feed`. The `joined_via: 'registration'` field tracks how they joined.

**Consequences:**
- Created `src/lib/onboarding.ts` with `autoJoinTheCommons()` helper
- Updated email registration, Google OAuth, and GitHub OAuth callbacks
- New users immediately see The Commons posts in their Home Feed
