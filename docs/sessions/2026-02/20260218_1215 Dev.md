### Status Bug: "Pending" Badge with Green Checkmarks

**User:** Returned to Peerloop — shows "Setup Incomplete" (pending) but all three indicators green.

**Claude:** Diagnosed root cause: `connect-status.ts` returned `status` from the database (`pending` from account creation) while fetching live boolean flags from Stripe API. The `stripe listen` webhook forwarder wasn't running, so `account.updated` webhook never arrived. Fixed by making the endpoint derive status from Stripe's live data and sync the DB when drift detected. User refreshed — now shows "Connected" with green "active" badge.

### Stripe Webhook Infrastructure Audit

**User:** Need to determine what should be happening with the Stripe webhook. Is there only one webhook endpoint?

**Claude:** Conducted comprehensive audit across all environments:
- **Endpoint:** Single `POST /api/webhooks/stripe` handling 3 events (checkout.session.completed, charge.refunded, account.updated). Two missing: transfer.paid, payout.failed.
- **Local dev:** Works via `stripe listen` CLI but wasn't running.
- **Preview/staging:** Secrets in CF Dashboard but no webhook endpoint registered in Stripe.
- **Production:** Intentionally deferred per tech-026.
- **Risks found:** No idempotency keys on transfers (medium — double payouts possible), transfer.paid not handled (low), payout.failed not handled (medium).

Explained webhook architecture with diagram showing single endpoint, event routing via switch statement, and how Stripe CLI bridges local dev.
