### Debugging Slow AppNavbar Loading

**User:** Tested but only have home page. When hitting BACK from 404, AppNavbar takes 2-5 seconds to appear with skeleton. This was happening before View Transitions change.

**Claude:** Investigated the issue. Found that `isLoading` check was `authStatus === 'loading'` without considering existing user state. Fixed to `authStatus === 'loading' && !user` so cached user skips skeleton.

**User:** Still happening after fix.

**Claude:** Added debug logging to trace the issue. Discovered hydration mismatch error - server renders skeleton (no localStorage), client tries to render menu (has cached user). This causes React to throw error and re-render.

**User:** Shared console screenshot showing hydration mismatch error.

**Claude:** Implemented hydration-safe approach:
- Added `isHydrated` state starting as false
- Server and client both render skeleton initially (matching HTML)
- After mount, useEffect loads cache and sets `isHydrated = true`
- Subsequent render shows actual menu

**User:** Still skeleton for 3 seconds. Maybe should kill dev server ports?

**Claude:** Killed processes on ports 4321-4323 and restarted. Added more debug logging.

**User:** Still not working. Shared browser console showing correct flow but skeleton persists.

**Claude:** Tried `useLayoutEffect` instead of `useEffect` to run before browser paint. Still seeing 3-second delay.

**User:** Suggested using `client:only` to skip SSR.

**Claude:** Changed to `client:only="react"` - navbar area now blank instead of skeleton while JS loads.

**User:** Blank UI for same amount of time. Concerned about SEO with `client:only`. Need to think harder about this.

**Claude:** Analyzed the situation:
- The 3-second delay is JS download/parse/execute time, not React inefficiency
- No React optimization can fix this - JS must load first
- Dev mode is much slower than production
- Reverted `client:only` back to `client:load` for SEO
- Reverted `useLayoutEffect` back to `useEffect` for simplicity
