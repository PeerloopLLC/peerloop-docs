# Session Decisions - 2026-02-02

## 1. Permission vs State Separation for Role Flags
**Type:** Strategic
**Topics:** d1, auth

**Trigger:** Analyzing how to replace `is_creator` flag - initially planned simple replacement with `can_create_courses`, but discussion revealed deeper semantic issues.

**Options Considered:**
1. Replace `is_creator` with `can_create_courses` everywhere
2. Replace `is_creator` with derived check from `courses` table everywhere
3. Recognize two concepts and use appropriate check for each context ← Chosen

**Decision:** Role flags represent TWO distinct concepts that must be handled separately:
- **Permission** (`can_create_courses`): Admin-controlled capability, stored in users table
- **State** (`is_creator` derived): Historical fact, derived from `courses` table

**Rationale:**
- Enables "revoked creator" scenario (can't create new, can manage existing)
- Matches real-world semantics (permission can be revoked, but history can't be erased)
- Cleaner authorization logic (permission checks vs existence checks)

**Consequences:**
- More complex queries (subqueries for derived state)
- Better access control granularity
- Documentation added to DECISIONS.md
- Full implementation plan created

## 2. Derived State from Tables, Not Cached Flags
**Type:** Implementation
**Topics:** d1

**Trigger:** Deciding whether `is_creator` should be a cached flag or derived from `courses` table.

**Options Considered:**
1. Keep `is_creator` as cached flag, update when first course created ← Not chosen
2. Derive from `courses` table every time ← Chosen

**Decision:** Derive "is creator" state from `courses` table:
```sql
EXISTS (SELECT 1 FROM courses WHERE creator_id = ? AND deleted_at IS NULL)
```

**Rationale:**
- Single source of truth (no sync issues)
- Handles edge cases automatically (all courses deleted = no longer "is creator")
- Subquery performance is fine for single-user lookups

**Consequences:**
- Slightly more complex SQL
- No denormalization to maintain
- Always accurate

## 3. Dashboard Access: Permission OR State
**Type:** Implementation
**Topics:** auth, astro

**Trigger:** Determining what grants access to Creator Dashboard.

**Options Considered:**
1. Permission only (`can_create_courses = 1`)
2. State only (has courses)
3. Either permission OR state ← Chosen

**Decision:** Creator Dashboard access = `can_create_courses = 1 OR has_courses`

**Rationale:**
- Users with permission need access to create first course
- Revoked creators need access to manage existing courses
- Either condition is sufficient

**Consequences:**
- Access check pattern documented
- SQL pattern: `WHERE can_create_courses = 1 OR EXISTS (...)`

## 4. Create Course Button: Always Visible, Conditionally Enabled
**Type:** Implementation
**Topics:** astro

**Trigger:** Deciding visibility of "Create Course" button for users without permission.

**Options Considered:**
1. Hide button entirely if no permission
2. Show disabled button with tooltip ← Chosen

**Decision:** Always show the "Create Course" button, but disable it unless `can_create_courses = 1`.

**Rationale:**
- Consistent UI (button always in same place)
- Discoverability (users know the feature exists)
- Clear feedback (disabled state with tooltip explains why)

**Consequences:**
- UI pattern established for capability-gated features
- Tooltip should explain how to get permission

## 5. Same Pattern Applies to Student-Teachers
**Type:** Strategic
**Topics:** d1, auth

**Trigger:** Recognizing the permission vs state pattern applies beyond creators.

**Options Considered:**
1. Handle `is_student_teacher` differently
2. Apply same permission vs state pattern ← Chosen

**Decision:** Apply identical pattern:
- **Permission:** `can_teach_courses` (admin grants eligibility)
- **State:** Derived from `student_teachers` table (has active certification)

**Rationale:**
- Consistency across role types
- Same "revoked but still active" scenario applies
- Simpler mental model

**Consequences:**
- Implementation plan covers both role types
- SQL patterns documented for both
