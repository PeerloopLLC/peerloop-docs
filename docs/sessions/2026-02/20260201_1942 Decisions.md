# Session Decisions - 2026-02-01

## 1. Modal Login vs Full Page Login
**Type:** Implementation
**Topics:** auth, astro

**Trigger:** User wanted to offer sign-in without navigating away from the current page.

**Options Considered:**
1. Full page `/login` only (current state)
2. Modal login only ← Not chosen (need /login for direct links, password reset)
3. Hybrid: Modal + keep `/login` page ← Chosen

**Decision:** Implement modal login triggered from AppNavbar and protected actions, while keeping the existing `/login` page for direct navigation, password reset flows, and email links.

**Rationale:** Modal preserves user context (they stay on current page), but `/login` is still needed for:
- Direct links from emails
- Password reset flow entry point
- Bookmark-able login URL
- OAuth callback redirects

**Consequences:** Created reusable `LoginForm` component used by both modal and page. Modal triggers authentication without page navigation.

## 2. Registration in Modal vs Separate Page
**Type:** Implementation
**Topics:** auth

**Trigger:** Should the login modal also support registration?

**Options Considered:**
1. Tabbed modal with Login + Register ← Not chosen
2. Login modal only, link to `/register` page ← Chosen
3. Separate registration modal

**Decision:** Login modal only, with "Create Account" link navigating to `/register` page.

**Rationale:**
- Registration has more form fields and friction - modals work best for quick tasks
- Registration often triggers email verification, breaking the "stay in context" benefit
- New users registering are at journey start, not mid-action - full page with marketing context makes sense
- Simpler implementation: one form per modal

**Consequences:** Modal is focused and simple. Registration flow unchanged.

## 3. Global State vs React Context for Cross-Island Auth Modal
**Type:** Implementation
**Topics:** astro, auth

**Trigger:** Need to trigger login modal from any React island (AppNavbar, protected action buttons throughout the app).

**Options Considered:**
1. React Context provider wrapping all islands ← Not possible (Astro islands are separate React roots)
2. Global state on `window.__peerloop` with custom events ← Chosen
3. Event-based only (no callback capture) ← Not chosen (needed to execute action after login)

**Decision:** Use global state pattern (like existing `currentUser`) with custom events for modal state. Store pending action callback globally, execute after successful login.

**Rationale:** Astro's multi-island architecture means React Context can't be shared between components rendered with `client:load`. Global state with events is the established pattern in this codebase (`current-user.ts`).

**Consequences:**
- Created `src/lib/auth-modal.ts` for global modal state
- Any React island can call `openLoginModal(callback)`
- `LoginModalRenderer` component subscribes and renders modal
- Consistent with existing `currentUser` pattern

## 4. Forgot Password in Modal vs Separate Page
**Type:** Implementation
**Topics:** auth

**Trigger:** Should "Forgot Password" flow be handled in the modal?

**Options Considered:**
1. Multi-step modal flow ← Not chosen
2. Simple link to `/forgot-password` page ← Chosen

**Decision:** "Forgot Password" link in modal navigates to `/forgot-password` page.

**Rationale:**
- Password reset is a multi-page flow (enter email → check inbox → reset form)
- Rare, one-time operation - not worth modal complexity
- User has mentally committed to a side journey when clicking "forgot password"
- No pending action to preserve - they can't proceed until password is reset

**Consequences:** Minimal modal complexity. Existing password reset flow unchanged.
