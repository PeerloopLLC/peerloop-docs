# Session Learnings - 2026-02-25

## 1. Enrollment Table Uses `student_id`, Not `user_id`
**Topics:** d1, testing

**Context:** While implementing creator self-certification (WU1), the POST handler in `student-teachers.ts` had an enrollment check query referencing `enrollments.user_id`. The existing tests only tested the GET handler, so this bug was never caught.

**Learning:** The `enrollments` table uses `student_id` (not `user_id`) as the foreign key to `users`. The original query `SELECT user_id, status FROM enrollments WHERE course_id = ? AND user_id = ?` was a pre-existing bug — it would always fail with `SqliteError: no such column: user_id` when the code path was actually exercised against a real database. The fix: `SELECT student_id, status FROM enrollments WHERE course_id = ? AND student_id = ?`.

**Pattern:** When writing API handlers that reference the enrollments table, always use `student_id` (not `user_id`). Verify column names against `migrations/0001_schema.sql` when in doubt.

## 2. Two Separate Test Files Can Exist for the Same Component
**Topics:** testing

**Context:** After adding `teaching_courses` to the `CreatorDashboard` component, the per-file test run (`tests/pages/dashboard/CreatorDashboard.test.tsx`) passed, but the full suite revealed a second test file (`tests/components/dashboard/CreatorDashboard.test.tsx`) with its own mock data that also needed updating.

**Learning:** When adding a new required field to a component's data interface, search for ALL test files that mock that component's data — not just the first one found. Use `Glob` or `Grep` to find all test files referencing the component before assuming test coverage is complete. In this case, `grep "CreatorDashboard" tests/` would have revealed both files upfront.

**Pattern:**
```bash
# Before changing a component's interface, find ALL test files
grep -r "ComponentName" tests/ --include="*.test.*" -l
```

## 3. `CASE WHEN` in SQL for Computed Boolean Flags
**Topics:** d1, astro

**Context:** The SSR loader needed to expose whether each S-T is also the course creator, without an additional query or join.

**Learning:** A `CASE WHEN st.user_id = ? THEN 1 ELSE 0 END as is_course_creator` in the existing SELECT is the cleanest way to add a computed boolean flag to a query result. The parameter (`courseRow.creator_id`) was already available in scope, so no additional query was needed. Map it to a boolean in TypeScript with `Boolean(st.is_course_creator)`.

## 4. Extending `Promise.all` Queries is Safe and Clean
**Topics:** d1, astro

**Context:** The creator-dashboard API already had a `Promise.all` with 5 parallel queries. Adding a 6th query for `teaching_courses` followed the same pattern seamlessly.

**Learning:** When an API handler uses `Promise.all` for parallel data fetching, adding new data to the response is a simple, low-risk pattern: add the query to the array, add destructured variable, add to response object. The queries are independent and don't affect each other. This pattern scales well up to ~8-10 queries before readability suffers.
