# Session Learnings - 2026-02-23 (Session 260)

## 1. Wrangler Lockfile Doesn't Auto-Update with Caret Range
**Topics:** cloudflare, deployment

**Context:** Wrangler was at `^4.58.0` in package.json but `npm install` resolved to 4.61.0 from the lockfile, not 4.67.0 (latest in range).

**Learning:** `npm install` respects the lockfile even when a newer version satisfies the semver range. You must explicitly `npm install wrangler@latest` to update the lockfile resolution. The caret range only matters for fresh installs without a lockfile.

## 2. Cloudflare Pages Build Vars vs Runtime Bindings
**Topics:** cloudflare, deployment

**Context:** Checking whether the staging build log was missing environment variables — it only showed 3 vars (STRIPE_PUBLISHABLE_KEY, STREAM_API_KEY, STREAM_APP_ID).

**Learning:** Cloudflare Pages has three distinct layers for configuration:
1. **Build-time vars** — appear in build logs, baked into client JS (publishable keys only)
2. **Runtime bindings** — D1, R2, KV configured in wrangler.toml, injected via `Astro.locals.runtime.env`
3. **Secrets** — set in CF dashboard, never visible in logs, available at runtime

Only public/publishable keys should appear in build logs. Secrets and bindings are runtime-only.

## 3. Stripe Seed Is Opt-In, Not Part of Standard Setup
**Topics:** stripe, d1

**Context:** After running `db:setup:staging`, enrollment failed with "Course creator has not set up payment processing" because Guy Rymberg had NULL stripe_account_id.

**Learning:** The Stripe seed (`migrations-dev/0002_seed_stripe.sql`) is intentionally separate from `db:setup:local` and `db:setup:staging`. This is by design — it links dev users to real Stripe test-mode Express accounts, which aren't needed for all dev workflows. When testing checkout/enrollment flows, you must run the Stripe seed as an additional step. The quick reference table in `docs/tech/tech-024-migrations.md` now documents this.

## 4. Wrangler 4.65+ Auto-Injects CF_PAGES Env Vars in Local Dev
**Topics:** cloudflare, deployment

**Context:** Researching wrangler 4.58→4.67 changelog for upgrade decision.

**Learning:** Starting with wrangler 4.65.0, `wrangler pages dev` automatically injects `CF_PAGES`, `CF_PAGES_BRANCH`, `CF_PAGES_COMMIT_SHA`, and `CF_PAGES_URL` environment variables locally. This closes a parity gap where code checking `CF_PAGES === '1'` (like `src/lib/version.ts`) would behave differently locally vs deployed.

## 5. Wrangler 4.66+ Excludes .wrangler from Pages Uploads
**Topics:** cloudflare, deployment

**Context:** Researching wrangler changelog — discovered the `.wrangler` directory was previously being silently included in Pages uploads.

**Learning:** Prior to wrangler 4.66.0, the `.wrangler` directory (containing local D1 SQLite files, state, etc.) was accidentally included in Cloudflare Pages uploads. This is a correctness fix worth upgrading for — it prevents local dev state from leaking into deployments.
