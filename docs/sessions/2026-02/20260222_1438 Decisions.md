# Session Decisions - 2026-02-22 (Session 252)

## 1. Keep Stripe Hosted Onboarding (Reject Embedded Component)
**Type:** Strategic
**Topics:** stripe

**Trigger:** Client researched three Stripe onboarding approaches and asked about the embedded component option.

**Options Considered:**
1. Embedded Onboarding — Stripe React component renders in-app, user never leaves ← Considered
2. Stripe-Hosted Onboarding — Redirect to Stripe's page, user returns after ← Chosen (already implemented)
3. Full API Onboarding — Build custom UI, call Stripe APIs directly ← Already rejected

**Decision:** Keep the current Stripe-hosted (Account Links) approach. Do not switch to embedded onboarding.

**Rationale:** Stripe is a household name for people who earn online — showing the Stripe-branded page builds trust rather than diminishing it. The functional difference is cosmetic (same screens either way). The embedded approach would require adding `@stripe/connect-js` + `@stripe/react-connect-js` packages, a new Account Sessions endpoint, and theming work — for no material UX improvement.

**Consequences:** No code changes needed. Added Stripe branding configuration (logo, colors, display name) to the go-live checklist in PLAN.md.

## 2. Move Resend Domain Verification Before Go-Live
**Type:** Strategic
**Topics:** resend, deployment

**Trigger:** Discovered during API key testing that emails cannot be sent to real recipients without a verified domain, which blocks all email notification testing.

**Options Considered:**
1. Keep domain verification as a go-live task ← Original plan
2. Move it to an immediate client action ← Chosen

**Decision:** Resend domain verification is now a "Next Action" in PLAN.md, not a go-live task. The client should add DNS records for `peerloop.com` (or `mail.peerloop.com`) to unblock email testing immediately.

**Rationale:** Domain verification doesn't require a live website — it's purely DNS. Until it's done, email testing is completely blocked. Moving it forward unblocks moderator invites, creator application notifications, payment receipts, etc.

**Consequences:** PLAN.md updated with RESEND-DOMAIN as a Next Action section. Go-live section now references it instead of duplicating the steps.

## 3. Topics Table (Hybrid) for Onboarding Taxonomy
**Type:** Implementation
**Topics:** d1, architecture

**Trigger:** Member onboarding needs a way for users to select interests from categories and subcategories. Current categories are flat (15 top-level, no hierarchy).

**Options Considered:**
1. New subcategories table — formal parent/child hierarchy ← More schema than needed
2. Use existing course_tags — freeform strings organized under categories ← Too unstructured
3. Topics table (hybrid) — curated subtopics linked to categories ← Chosen

**Decision:** Created a `topics` table with ~55 curated subtopics (3-5 per category), linked to parent categories via FK. Used for onboarding interest selection and future course filtering.

**Rationale:** More structured than freeform tags (curated, admin-controlled) but lighter than a full subcategory system. Topics serve double duty: onboarding interest picker and future discover page filtering.

**Consequences:** 3 new tables added: `topics`, `member_profiles`, `user_topic_interests`. 55 topic seed rows. New API endpoints at `/api/topics` and `/api/me/onboarding-profile`.

## 4. Separate member_profiles Table (Not Users Columns)
**Type:** Implementation
**Topics:** d1, architecture

**Trigger:** Needed to store onboarding questionnaire answers (goal, referral source, profession, completion timestamp).

**Options Considered:**
1. Add columns to users table ← Users already has 30+ columns
2. Separate member_profiles table with user_id PK ← Chosen

**Decision:** Use a separate `member_profiles` table. Only the `onboarding_completed_at` field is surfaced in `CurrentUser` via LEFT JOIN on `/api/me/full`.

**Rationale:** Keeps the `users` table lean (it's read on every authenticated page load). Onboarding data is conceptually separate — a profile about interests, not identity. The single completion flag is the only piece that needs to be in every page load for conditional navbar rendering.

## 5. Per-Category Experience Level for Onboarding
**Type:** Implementation
**Topics:** architecture

**Trigger:** Needed to decide granularity for experience level in the onboarding questionnaire.

**Options Considered:**
1. One overall level (single beginner/intermediate/advanced) ← Simpler but less useful
2. Per-category level ← Chosen

**Decision:** Users pick experience level per category they select. Stored on `user_topic_interests` — all topics within a category inherit that category's experience level.

**Rationale:** A user might be advanced in Backend Development but beginner in AI & Prompt Engineering. Per-category granularity enables better course recommendations without making the form overly complex (only need one click per selected category, not per topic).

## 6. Recommendations on Discover Pages (Not Homepage)
**Type:** Implementation
**Topics:** architecture

**Trigger:** Needed to decide where personalized recommendations surface after onboarding.

**Options Considered:**
1. Homepage feed — "Recommended for you" on dashboard
2. Discover pages — sections on /discover/courses and /discover/communities ← Chosen
3. Both + dedicated "For You" page

**Decision:** Add "Recommended for You" sections to the existing discover pages only. No homepage changes or new pages.

**Rationale:** The discover pages are where users go to browse — putting recommendations there is contextual. The homepage already has its own curation. A dedicated page adds navigation complexity without clear benefit at the Genesis Cohort scale.
