# Session Learnings - 2026-02-25 (Session 288)

## 1. Availability Table Uses user_id, Not st_id
**Topics:** d1, auth

**Context:** The CURRENT-BLOCK-PLAN.md (designed in Session 287) assumed `st_id` and `course_id` columns existed on the `availability` table. Investigation revealed the actual schema uses `user_id` referencing `users(id)` with no `course_id`.

**Learning:** The Peerloop identity model is person-centric throughout: `availability`, `sessions.teacher_id`, `enrollments.student_teacher_id`, and `payment_splits.recipient_id` all use the person's `user_id`, not the `student_teachers.id` certification ID. The `student_teachers` table is a certification record, not an identity. The `st_id` naming in API responses is confusingly aliased from `users.id`, not `student_teachers.id`.

**Pattern:** Always check actual schema before implementing from a plan — design docs may reference intended schema that hasn't been built yet.

## 2. Confusing st_id Alias in API Queries
**Topics:** d1, astro

**Context:** While investigating the identity model, found that `/api/me/full.ts` aliases `users` table as `st` in joins, producing `st.id as st_id` which is actually the user's ID, not the ST certification ID.

**Learning:** The query `LEFT JOIN users st ON e.student_teacher_id = st.id` names the `users` table `st`, so `st.id` is a `users.id`. This means `st_id` in API responses = person ID, which is misleading. A future CERT-AUDIT block should rationalize this naming.

## 3. react-day-picker v9 Custom Grid vs Component Override
**Topics:** astro

**Context:** Initially planned to use react-day-picker v9's `components={{ Day: CustomDay }}` API to replace day cells. After implementing, switched to a fully custom grid instead.

**Learning:** For heavily customized calendars where every cell needs app-specific data (availability slots, overrides, booking counts), building a custom grid is cleaner than fighting the library's component override API. The library's `Day` component receives `DayProps` with `CalendarDay` and `Modifiers`, but threading custom data through requires React context or prop drilling that adds complexity. A plain `grid-cols-7` div with custom cells is simpler and gives full control. react-day-picker is still installed for potential future use (e.g., the student-facing booking calendar where simpler selection is needed).

## 4. Astro Supports File + Directory Routes for Same Path
**Topics:** astro

**Context:** Needed to add `/api/me/availability/overrides` alongside existing `/api/me/availability.ts`.

**Learning:** Astro handles `api/me/availability.ts` (for `/api/me/availability`) and `api/me/availability/overrides.ts` (for `/api/me/availability/overrides`) without conflict. Same pattern already used in the codebase: `certificates.ts` + `certificates/recommend.ts`.

## 5. Override Merge Strategy: Replace, Don't Layer
**Topics:** d1

**Context:** Designed the override merge logic for the student-facing availability API.

**Learning:** When overrides exist for a date, they fully replace recurring rules for that date — not merge alongside them. This is simpler and more predictable: if a teacher overrides Tuesday March 15 with 10am-12pm, students see 10am-12pm only, not the original 2-4pm recurring slot plus the override. The "blocked" override (is_available=false) produces zero slots for that date.
