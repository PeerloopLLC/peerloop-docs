# Session Decisions - 2026-02-22

## 1. Add Stripe Fields to CurrentUser as Cached Convenience Data
**Type:** Implementation
**Topics:** stripe, astro

**Trigger:** After completing Stripe Connect onboarding for a test user, asked whether the global CurrentUser should reflect Stripe account status.

**Options Considered:**
1. Keep Stripe status as a separate API call only (`GET /api/stripe/connect-status`)
2. Add Stripe status to CurrentUser as cached convenience fields ← Chosen
3. Add full Stripe account details (requirements, charges_enabled, etc.) to CurrentUser

**Decision:** Add `stripeAccountId`, `stripeStatus`, and `stripePayoutsEnabled` to CurrentUser as cached read-only fields. Also add `hasStripeConnected()` convenience method.

**Rationale:** CurrentUser is a read-only cache for UI convenience, not a source of truth. Adding lightweight Stripe fields lets any component instantly check "has this user set up payments?" without an extra API call. Full Stripe account details (requirements, disabled_reason) stay in the dedicated `/api/stripe/connect-status` endpoint — only the settings page needs that level of detail.

**Consequences:**
- `UserIdentity` interface, `CurrentUser` class, `UserRow`, and `/api/me/full` SQL query all updated
- `StripeConnectSettings.tsx` now calls `refreshCurrentUser()` on Stripe return
- Tests added to both `full.test.ts` and `StripeConnectSettings.test.tsx`

## 2. Document CurrentUser as Read-Only Cache Architectural Principle
**Type:** Strategic
**Topics:** astro, docs-infra

**Trigger:** Before adding Stripe fields, paused to audit whether CurrentUser was ever used as a source of truth. Audit confirmed clean separation — decided to document the principle explicitly.

**Options Considered:**
1. Leave it as implicit knowledge (risk of future violations)
2. Document in tech-020 only
3. Document across tech-020, BEST-PRACTICES.md, and API-USERS.md ← Chosen

**Decision:** Add explicit "CurrentUser is a read-only convenience cache, not a source of truth" documentation in three locations:
- `docs/tech/tech-020-state-management.md` — new "Architectural Principle" section at top
- `BEST-PRACTICES.md` — new "CurrentUser Global" subsection with do/don't patterns
- `docs/reference/API-USERS.md` — source-of-truth note on `/api/me/full` endpoint

**Rationale:** Three docs serve three audiences: deep architecture reference, quick coding patterns, and API contract. The principle was already followed in practice (audit confirmed) but never stated — making it explicit prevents future violations as the team grows.

**Consequences:** Three files updated with consistent messaging. API-USERS.md response schema also updated with the new Stripe fields and `onboardingCompletedAt`.

## 3. Refresh CurrentUser After Stripe Connect Return
**Type:** Implementation
**Topics:** stripe, astro

**Trigger:** Identified that `StripeConnectSettings.tsx` cleaned URL params on Stripe return but never refreshed the global user state.

**Options Considered:**
1. Rely on AppNavbar's window-focus refresh (existing behavior)
2. Add explicit `refreshCurrentUser()` call on Stripe return ← Chosen

**Decision:** Call `refreshCurrentUser()` in the `useEffect` that handles `?success=true` and `?refresh=true` query params, alongside the existing URL cleanup.

**Rationale:** The window-focus refresh is a safety net, not a guarantee — it only fires if the user switches tabs. A user who completes Stripe onboarding and stays on the settings page would have stale CurrentUser data. The explicit refresh eliminates this gap with one line of code and zero performance cost (fire-and-forget, parallel with `fetchStatus()`).

**Consequences:** Added `refreshCurrentUser` import and call. Added 3 tests: two asserting refresh IS called with query params, one asserting it is NOT called without them.
