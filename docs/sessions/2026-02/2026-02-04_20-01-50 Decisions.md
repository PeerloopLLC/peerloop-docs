## 2. No Creator Feeds - Use Communities Instead
**Type:** Strategic
**Topics:** stream, astro

**Trigger:** Discussion about routing for creator feeds vs community feeds vs course feeds.

**Options Considered:**
1. Keep `/creator/[handle]/feed` (IFED) as separate route
2. Make creator feeds use the same pattern as communities ← Chosen
3. Allow both patterns

**Decision:** Removed creator feed route. Creators who want announcement feeds should create a community (e.g., "Guy's Main Hall").

**Rationale:**
- Single feed concept: all feeds are community feeds or course feeds
- Avoids slippery slope (if creators have feeds, why not all users?)
- Communities provide built-in structure (members, resources, progressions)
- Creator branding: community can be named anything

**Consequences:**
- Removed `/creator/[handle]/feed.astro`
- Updated FeedSlidePanel to only show Communities + Course Feeds
- Communities without courses are explicitly valid

## 3. Bookmarkable Feed Tabs via Sub-Routes
**Type:** Implementation
**Topics:** astro

**Trigger:** User asked whether feeds can be both tabs within a page AND bookmarkable.

**Options Considered:**
1. Tabs as client state only (not bookmarkable)
2. Tabs via query params (`?tab=feed`)
3. Tabs via hash (`#feed`)
4. Tabs as sub-routes (`/feed`) ← Chosen

**Decision:** Feed tabs are implemented as sub-routes that render the same page with different active tab.

**Rationale:**
- Bookmarkable and shareable URLs
- Clean paths without query params
- Back/forward navigation works naturally
- Follows GitHub/Notion pattern

**Consequences:**
- Created CourseTabs component with URL-aware tabs
- `/course/[slug]/feed` renders course page with Feed tab active
- `/course/[slug]/discuss` now redirects to `/feed`

**Options Considered:**

**Consequences:**

## 5. Communities Without Courses Are Valid
**Type:** Strategic
**Topics:** d1, astro

**Trigger:** Discussion about creator "Main Hall" communities for announcements.

**Options Considered:**
1. Require all communities to have at least one course
2. Allow communities without courses ← Chosen
3. Create separate "announcement channel" concept

**Decision:** Communities are flexible containers. A community can have:
- Courses via progressions (typical)
- Just a feed and resources (announcement-style)
- Both

**Rationale:**
- Enables creator "Main Hall" pattern for announcements
- Simpler model: communities are the universal grouping mechanism
- No need for separate announcement/channel concept

**Consequences:**
- No schema changes needed (already flexible)
- Documented as valid use case
- Creators can create course-free communities for announcements
