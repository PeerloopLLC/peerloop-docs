# Session Decisions - 2026-02-18

## 1. Self-Healing API Endpoint Over Webhook Dependency for Stripe Status
**Type:** Implementation
**Topics:** stripe, auth

**Trigger:** After Stripe Connect onboarding, the Payment Settings page showed contradictory state — "pending" badge but all three status indicators green. Root cause: `connect-status` endpoint returned status from the database (stale) while fetching live flags from Stripe API.

**Options Considered:**
1. Require `stripe listen` to always be running for correct status ← Status quo
2. Make `connect-status` derive status from Stripe's live data and sync the database ← Chosen
3. Add a manual "refresh status" button to the UI

**Decision:** The `connect-status` endpoint now derives the correct status from Stripe's live API data (`active` when charges + payouts + details all enabled) and automatically syncs the database when it detects drift.

**Rationale:** Self-healing is more robust than depending on webhooks alone. Webhooks can be missed (listener not running, network issues, race conditions). The API endpoint is already making a Stripe API call for the boolean flags — deriving status from that data is essentially free. This pattern makes the system correct-by-default.

**Consequences:** `connect-status.ts` now writes to the database (was read-only before). Status is always accurate on page load regardless of webhook state. The `account.updated` webhook handler remains as an optimization for real-time updates.

## 2. Manual Bank Entry Path for Client Documentation
**Type:** Strategic
**Topics:** stripe, workflow

**Trigger:** During Stripe onboarding documentation, the bank account screen offered both bank login (Plaid) and manual entry (routing/account numbers).

**Options Considered:**
1. Document the bank login (Plaid) path — most common in production ← Not chosen
2. Document the manual entry path — shows the actual form fields ← Chosen
3. Document both paths equally

**Decision:** Used the manual bank entry path for the client-facing documentation guide, with a mention that bank login is also available.

**Rationale:** Manual entry shows the actual form fields (routing number, account number) that cause the most confusion for non-technical creators. The bank login path via Plaid is a third-party UI we can't control or reliably document. The guide should focus on the harder path; the bank login is self-explanatory.

**Consequences:** Created `docs/guides/stripe-onboarding-guide.md` focusing on manual entry with a note about the bank login alternative.

## 3. Skip Preview Webhook Endpoint, Rely on Local Dev for Payment Testing
**Type:** Strategic
**Topics:** stripe, cloudflare, deployment

**Trigger:** Discovered that `STRIPE_WEBHOOK_SECRET` exists in Cloudflare Preview environment vars but no corresponding webhook endpoint is registered in Stripe Dashboard. Cloudflare Pages Preview URLs are dynamic per deployment.

**Options Considered:**
1. Set up a stable staging domain alias and register it as a Stripe webhook endpoint
2. Skip Preview webhook setup, test payments locally only ← Chosen
3. Use a tunnel service (ngrok) for Preview testing

**Decision:** Do not configure a Stripe webhook endpoint for Preview. Payment testing will use local development with `stripe listen` CLI. Production will get its own webhook at go-live.

**Rationale:** Cloudflare Pages generates unique URLs per deployment (`<hash>.peerloop.pages.dev`), incompatible with Stripe's fixed webhook URL. Setting up a stable staging domain adds infrastructure complexity for limited benefit at this stage. Local testing with `stripe listen` provides full E2E coverage.

**Consequences:** Preview deployments cannot complete the full payment flow (checkout succeeds on Stripe but enrollment webhook never arrives). The `STRIPE_WEBHOOK_SECRET` in CF Preview is unused but harmless. If a stable staging domain is added later, the webhook can be configured then.

## 4. Handle New Stripe Events in Default Switch Block
**Type:** Implementation
**Topics:** stripe, testing

**Trigger:** Adding `transfer.paid` and `payout.failed` to the webhook switch statement caused TypeScript errors — these event names aren't in the Stripe SDK's literal type union for `event.type`.

**Options Considered:**
1. Use `@ts-ignore` on the case statements ← Not chosen
2. Cast `event.type` to string and handle in the `default` block ← Chosen
3. Extend Stripe types with declaration merging

**Decision:** Handle `transfer.paid` and `payout.failed` events in the `default` block using string comparison (`event.type as string`).

**Rationale:** Clean and explicit — no type suppression, no complex type extensions. The `default` block naturally handles events outside the SDK's known set. Adding a string comparison is clear about the intent: these events exist in Stripe but aren't in the TypeScript definitions.

**Consequences:** New event handlers work correctly at runtime. TypeScript is satisfied. Pattern is easy to extend for future events not in the SDK's type union.
