# Session Learnings - 2026-02-23

## 1. TEST-API.md and TEST-COVERAGE.md Can Drift Apart
**Topics:** testing, docs-infra

**Context:** Running `/q-sync-docs` after the MODERATION block completion revealed that TEST-API.md (detail table) and TEST-COVERAGE.md (summary) had inconsistent numbers: 190/191 vs 193/193.

**Learning:** When two documentation files track the same metric (API test counts), they can silently drift apart as updates target one but not the other. The discrepancy had three root causes: (1) Topics row was entirely missing from the TEST-API.md table, (2) Health showed 2 endpoints but has 3 (health/kv is untested), (3) Me row showed 45/46 but actual was 46/47. Multiple small omissions accumulated over sessions. Running `/q-sync-docs` periodically catches these before they compound.

## 2. Documentation Sync After Major Implementation Reveals Pre-Existing Gaps
**Topics:** docs-infra, workflow

**Context:** `/q-sync-docs` was run to update docs for the MODERATION block changes. Expected to find only the new fields (community_id, canManageModerators) and auth changes.

**Learning:** A documentation sync after a major feature finds not just the expected changes but also pre-existing discrepancies that accumulated silently. In this case: stale test counts, missing table rows, and outdated "(Future)" labels in ROLES.md that predated this session. The habit of running `/q-sync-docs` after major blocks serves double duty — it catches both fresh changes and accumulated drift.

## 3. ROLES.md Is a Hidden Dependency for Auth Changes
**Topics:** auth, docs-infra

**Context:** After updating API-ADMIN.md and DEVELOPMENT-GUIDE.md with the new two-tier moderation auth, the user asked whether a dedicated moderator document existed. ROLES.md had stale Tier 2 capabilities ("dismiss, warn, suspend, remove" when only dismiss and remove are allowed), "(Future)" labels on implemented endpoints, and a "(future)" annotation on `canModerateFor`.

**Learning:** When auth patterns change, the obvious doc targets are API reference files and the development guide. But `docs/reference/ROLES.md` is a comprehensive role reference that documents capabilities, restrictions, and endpoint access per role. It's a hidden dependency — not linked from `/q-sync-docs` change detection matrix — and should be checked whenever role permissions or moderation access changes.
