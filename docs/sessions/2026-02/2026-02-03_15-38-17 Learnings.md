# Session 170 Learnings - 2026-02-03

## 1. Tailwind v4 @theme Font Sizes Require Dev Server Restart
**Topics:** tailwind

**Context:** Added a custom `text-xxs` (10px) font size via `@theme { --text-xxs: 0.625rem; }` in Tailwind v4's CSS-first configuration. The class wasn't being recognized initially.

**Learning:** Theme tokens in Tailwind v4 are processed at build time, not during hot module replacement (HMR). When adding new theme variables:
- New `@theme` tokens require a dev server restart
- `@utility` definitions work immediately (they're raw CSS)
- Both approaches are valid, but `@theme` is more idiomatic for extending the design system

**Pattern:**
```css
@theme {
  --text-xxs: 0.625rem;
  --text-xxs--line-height: 0.875rem;
}
```
After adding, run: `Ctrl+C` → `npm run dev`

## 2. React Portal for Fixed Positioning Outside Parent Containers
**Topics:** astro, ui

**Context:** The UserAccountDropdown panel was rendering inside the navbar's `<aside>` container. Despite using `position: fixed`, it was constrained by the parent's boundaries and appeared inline.

**Learning:** Use React's `createPortal()` to render elements outside their parent's DOM hierarchy. This is essential for:
- Modal dialogs
- Tooltips and popovers
- Slide-out panels with fixed positioning
- Any element that needs to "escape" overflow:hidden or stacking context

**Pattern:**
```tsx
import { createPortal } from 'react-dom';

// Render panel to document.body, escaping parent container
{typeof document !== 'undefined' && createPortal(slidePanel, document.body)}
```

## 3. JavaScript Default Parameters Are Replaced, Not Merged
**Topics:** tailwind, ui

**Context:** `ChevronRightIcon` rendered at ~60px instead of 16px. The icon component has `className = 'h-4 w-4'` as default, but passing `className="text-secondary-400"` caused huge rendering.

**Learning:** When a component has default props via JavaScript defaults (`{ className = 'h-4 w-4' }`), passing ANY className replaces the entire default—it doesn't merge.

**Pattern:**
```tsx
// ❌ Bad - loses size classes
<ChevronRightIcon className="text-secondary-400" />

// ✅ Good - include all needed classes
<ChevronRightIcon className="h-4 w-4 text-secondary-400" />
```

## 4. Public Profile Field Indicators for User Transparency
**Topics:** ui, settings

**Context:** The settings/profile page has many fields, but users don't know which ones appear on their public profile card.

**Learning:** Adding a visual "Public" badge next to fields that will be publicly visible:
- Improves user trust and GDPR compliance
- Helps users make informed decisions about what to share
- Creates clear visual hierarchy between public and private settings

**Pattern:**
```tsx
function PublicBadge() {
  return (
    <span className="ml-2 inline-flex items-center gap-1 text-xxs font-medium text-primary-600 bg-primary-50 px-1.5 py-0.5 rounded">
      <EyeIcon className="w-3 h-3" />
      Public
    </span>
  );
}

// Usage
<label>Display Name {isPublic && <PublicBadge />}</label>
```
