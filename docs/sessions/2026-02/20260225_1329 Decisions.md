# Session Decisions - 2026-02-25 (Session 288)

## 1. Availability is Per-Person (user_id), Not Per-Course
**Type:** Strategic
**Topics:** d1, auth

**Trigger:** CURRENT-BLOCK-PLAN.md assumed `st_id` and `course_id` existed on `availability` table, but actual schema uses `user_id` with no course scoping.

**Options Considered:**
1. Match existing pattern — keep `user_id`, no `course_id` on availability ← Chosen
2. Evolve to per-course — add `course_id` to availability (breaking change)
3. Use `st_id` instead of `user_id` on new tables

**Decision:** Availability stays per-person using `user_id` (references `users(id)`). No `course_id` on availability or availability_overrides tables. A teacher can't teach two courses simultaneously — their TIME is personal, not per-course. The `teaching_active` toggle on `student_teachers` handles per-course opt-in/out.

**Rationale:** Per-person is simpler (one calendar to manage), avoids cross-course double-booking risk, and matches the existing codebase pattern where all scheduling tables use `user_id`. Per-course availability would require N separate calendars for teachers with multiple certifications — added complexity with no clear use case.

**Consequences:** All S-T-CALENDAR schema changes use `user_id`. CURRENT-BLOCK-PLAN.md schema section corrected. CERT-AUDIT added to PLAN.md deferred list for future `st_id` audit trail work.

## 3. Override Merge: Full Replacement, Not Layering
**Type:** Implementation
**Topics:** d1

**Trigger:** Needed to decide how overrides interact with recurring rules in the student-facing availability API.

**Options Considered:**
1. Overrides layer alongside recurring (both show) ← Not chosen
2. Overrides fully replace recurring for that date ← Chosen

**Decision:** When overrides exist for a date, they completely replace recurring rules for that date. Blocked overrides produce zero slots. Available overrides produce only the override's time range.

**Rationale:** Simpler mental model for teachers: "I overrode Tuesday March 15" means only the override times show, not a confusing mix of recurring + override. In the calendar UI, the `availability-utils.ts` merge logic reflects this: override slots replace non-overlapping recurring slots, and blocked overrides suppress everything.

**Consequences:** `buildMonthView()` in availability-utils.ts implements this. Student-facing API (`GET /api/student-teachers/[id]/availability`) checks `overridesByDate` first; if present, uses only override data.

## 4. Custom Calendar Grid Over react-day-picker Component Override
**Type:** Implementation
**Topics:** astro

**Trigger:** Block plan specified react-day-picker v9 with custom `Day` component replacement. After installing and reading the API, evaluated both approaches.

**Options Considered:**
1. Use react-day-picker's `components={{ Day: CustomDay }}` API
2. Build custom grid with plain divs ← Chosen

**Decision:** Built a custom 7-column CSS grid for the availability calendar instead of using react-day-picker's component override API. react-day-picker v9 remains installed for potential future use.

**Rationale:** Each cell needs app-specific data (availability slots, override state, booking counts, series-end flags) that would require complex prop threading through DayPicker's component system. A plain grid gives full control over cell rendering, click handling, and state management without fighting the library's abstractions.

**Consequences:** `AvailabilityCalendar.tsx` renders its own grid. `AvailabilityDay.tsx` (standalone component) was created then removed. react-day-picker (~22kB) installed but not imported in production bundle (tree-shaken).
