# Session Decisions - 2026-02-25

## 1. Month Calendar as Primary and Only Availability Interface
**Type:** Strategic
**Topics:** ui, architecture

**Trigger:** The existing weekly grid editor (`AvailabilityEditor.tsx`) only sets generic Mon-Sun patterns. S-Ts can't manage specific dates (vacations, holidays, extra availability). Needed to decide between keeping weekly as a tab alongside a new month view, or replacing it entirely.

**Options Considered:**
1. Keep weekly editor + add month view as second tab
2. Replace weekly editor with month calendar ŌåÉ Chosen
3. Keep weekly editor + add date-picker for exceptions only

**Decision:** The month calendar is the single interface. Recurring patterns are set within the same flow ŌĆö after marking a day available, prompt "Just this day" or "Every [Tuesday] for [N] weeks." No separate weekly tab.

**Rationale:** Two views for overlapping concepts creates confusion. The month calendar can do everything the weekly editor does (via the recurring prompt) plus handle date-specific overrides. One view, two interaction modes (ad hoc click, multi-select), recurring as an option within the flow.

**Consequences:** `AvailabilityEditor.tsx` will be replaced. `PUT /api/me/availability` will be refactored to accept `start_date` and `repeat_weeks`. The `availability` table gets two new columns.

## 2. react-day-picker v9 as Calendar Engine
**Type:** Implementation
**Topics:** astro, ui

**Trigger:** Month calendar needs: full custom cell rendering (6 different indicators per cell), non-contiguous multi-select, headless/Tailwind-native, lightweight bundle for Cloudflare Pages.

**Options Considered:**
1. react-day-picker v9 (~22 kB, full Day replacement, multi-select) ŌåÉ Chosen
2. FullCalendar (~50+ kB, partial cell rendering, no non-contiguous multi-select)
3. react-calendar (~49 kB, can only add content not replace cells, no multi-select)
4. react-big-calendar (~170 kB, wrapper-only customization, wrong tool)
5. @schedule-x (event-level only, can't customize day cells)

**Decision:** Use react-day-picker v9 as a "month grid engine" ŌĆö it handles calendar math, keyboard navigation, and accessibility while we render fully custom `Day` components.

**Rationale:** Only package that supports full component replacement for day cells AND built-in non-contiguous multi-select. Headless design works with Tailwind. 10.6M weekly downloads, actively maintained (last published Feb 2026). ~22 kB is the lightest option. The only gap (drag-select) is implementable via mouse events on the custom Day component.

**Consequences:** New dependency: `react-day-picker` + `date-fns`. Custom `AvailabilityDay` component will render time ranges, recurring badges, override marks, series-end indicators, and booked status.

## 3. Separate availability_overrides Table for Date-Specific Changes
**Type:** Implementation
**Topics:** d1, architecture

**Trigger:** Need to store both recurring availability rules (every Tuesday 2-4pm) and date-specific overrides (March 15 unavailable). Had to decide between a new table or flags on the existing `availability` table.

**Options Considered:**
1. Separate `availability_overrides` table ŌåÉ Chosen
2. Add `is_recurring` flag to existing `availability` table

**Decision:** Recurring rules stay in `availability` (keyed by `day_of_week`). Date-specific overrides go in a new `availability_overrides` table (keyed by `date`). Calendar rendering: expand recurring ŌåÆ overlay overrides ŌåÆ display merged.

**Rationale:** Different data shapes ŌĆö recurring has `day_of_week` (0-6), overrides have a specific `date`. Jamming both into one table requires nullable columns and `is_recurring` filtering on every query. Separate tables give clean queries and clear semantics.

**Consequences:** New `availability_overrides` table in `0001_schema.sql`. New API endpoints: `POST/DELETE /api/me/availability/overrides`. `GET /api/student-teachers/[id]/availability` updated to merge both tables.

## 4. Per-Course teaching_active Toggle for Creator-as-ST
**Type:** Implementation
**Topics:** d1, architecture

**Trigger:** Creators who self-certify as S-Ts may want to step back from teaching specific courses while keeping their S-T certification active. Need a mechanism to hide from booking without deactivating.

**Options Considered:**
1. Per-course toggle on `student_teachers` table ŌåÉ Chosen
2. Global toggle (separate user-level flag)

**Decision:** Add `teaching_active INTEGER DEFAULT 1` to the `student_teachers` table. Each course gets its own toggle.

**Rationale:** `student_teachers` is already one-row-per-course, so per-course is the natural granularity. A global toggle would actually be harder ŌĆö it would need separate storage for a concept that already maps to per-course rows. The "My Teaching" card on Creator Dashboard already shows per-course info, so each row just gets a toggle switch.

**Consequences:** Schema change to `student_teachers`. New `PATCH /api/me/student-teacher/:courseId/toggle` endpoint. Booking availability and `SessionBooking.tsx` teacher selection filter by `teaching_active = 1`.

## 5. Global /q-make-block-persistent Skill for Cross-Session Plan Persistence
**Type:** Strategic
**Topics:** cc-workflow, docs-infra

**Trigger:** Detailed block plans in PLAN.md get lost when sessions run out of tokens. Previously created `CURRENT-BLOCK-PLAN.md` manually for the BBB block. Needed a repeatable process.

**Options Considered:**
1. Create a global skill to automate it ŌåÉ Chosen
2. Continue doing it manually each time

**Decision:** Created `/q-make-block-persistent` as a global skill at `~/.claude/commands/`. Takes a block name argument, extracts from PLAN.md, writes to `CURRENT-BLOCK-PLAN.md` with a progress table, and updates PLAN.md status.

**Rationale:** The pattern has proven useful (BBB block, now S-T-CALENDAR) and will be needed for every future block. Automating it ensures consistent format and saves time. Global scope means it works across any project.

**Consequences:** New global skill available. One block at a time in `CURRENT-BLOCK-PLAN.md`. File deleted when block completes.
