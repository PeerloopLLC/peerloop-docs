# Session Learnings - 2026-02-05 (Session 193)

## 1. Cloudflare Secrets Dashboard Shows Identical Display for All Environments
**Topics:** cloudflare, deployment

**Context:** Comparing secrets across local `.dev.vars`, CF Preview, and CF Production to find gaps.

**Learning:** Cloudflare's environment variables UI shows "Value encrypted" for all secrets regardless of environment. You can't visually confirm that Preview and Production have *different* values — you have to trust they were set correctly. The audit must be done by secret *name*, not value. Keep a record of which environment uses which key tier (e.g., Stripe test keys in Preview, live keys in Production).

**Pattern:** Maintain a secrets audit table (like the one added to MEMORY.md this session) that tracks presence per environment.

## 2. OAuth Client ID/Secret Are App-Level Credentials, Not User Credentials
**Topics:** auth

**Context:** Clarifying what Google Client ID and Client Secret actually are and who creates them.

**Learning:** OAuth Client ID and Client Secret identify *your application* to the OAuth provider (Google, GitHub). They are not personal credentials. The Client ID is semi-public (visible in browser redirect URLs). The Client Secret is truly secret and only used server-side to exchange authorization codes for user profile data. Each OAuth provider requires a separate app registration, and each app registration produces its own ID/Secret pair.

## 3. MEMORY.md Should Be an Index, Not a Duplicate
**Topics:** workflow

**Context:** MEMORY.md had detailed Stream.io feed architecture info that duplicated content in `tech-002-stream.md`, and the two had drifted apart.

**Learning:** Claude Code's auto-memory file (`MEMORY.md`) works best as a quick-reference index with pointers to authoritative docs, not as a duplicate of those docs. When detailed info exists in a tech doc, MEMORY.md should store only the quick-lookup facts (e.g., App IDs, feed group names) and a pointer to the full doc. This prevents drift and keeps MEMORY.md within its 200-line budget.

**Pattern:** Structure MEMORY.md as:
```
## Quick References
- **Topic:** `docs/tech/tech-NNN-topic.md` (brief note)

## Topic Quick Facts
- Key fact 1
- Key fact 2
```

## 4. GitHub OAuth Apps Support Only One Callback URL Per App
**Topics:** auth, deployment

**Context:** Documenting OAuth setup instructions for Google and GitHub providers.

**Learning:** Google OAuth supports multiple authorized redirect URIs per client (localhost, preview, production all in one app). GitHub OAuth Apps only allow ONE callback URL per app. This means you may need separate GitHub OAuth Apps for production vs preview/local development, or use a dedicated staging domain. This is a meaningful difference when planning multi-environment OAuth.

## 5. Google Consent Screen Verification Is a Timeline Risk
**Topics:** auth, deployment

**Context:** Writing the tech-025 OAuth doc and noting caveats for the client.

**Learning:** Google requires a verification review of your OAuth consent screen when your app exceeds 100 users. This review can take 1-2 weeks. For Peerloop's Genesis Cohort targeting 60-80 students (plus creators and S-Ts), you'll approach this threshold quickly after launch. The client should start the verification process well before the launch date to avoid a blocker.
