# Session Decisions - 2026-02-05 (Session 192)

## 1. OAuth Post-Login Redirect Target: Homepage (/)
**Type:** Implementation
**Topics:** auth, astro

**Trigger:** OAuth callbacks (GitHub, Google) were redirecting to `/dashboard`, which no longer exists as a page route.

**Options Considered:**
1. `/` — Homepage, adapts based on auth state ← Chosen
2. `/learning` — Student dashboard
3. `/feed` — Personalized feed

**Decision:** OAuth callbacks redirect to `/` (homepage) after successful login.

**Rationale:** The homepage is the universal entry point that works for all user types (students, S-Ts, creators). Redirecting to `/learning` would be wrong for creators; `/feed` assumes the user wants social content. The homepage adapts its content based on authentication state.

**Consequences:** Updated `src/pages/api/auth/github/callback.ts` and `src/pages/api/auth/google/callback.ts`.

## 2. Add Teachers Tab to Course Page
**Type:** Implementation
**Topics:** astro, d1

**Trigger:** The `/course/[slug]/teachers` route was originally planned but nearly removed during doc cleanup. User identified it as a natural tab alongside About, Feed, and Curriculum.

**Options Considered:**
1. Keep S-T info embedded in About tab (status quo)
2. Create dedicated Teachers tab with richer cards ← Chosen
3. Rely on `/discover/teachers` for all S-T discovery

**Decision:** Add a public "Teachers" tab to the course page showing expanded S-T cards (bio, rating, students taught, certified date, "View Profile" / "Book Session" actions). Trim the About tab's S-T section to a compact summary with stacked avatars linking to the Teachers tab.

**Rationale:** Makes S-T discovery course-contextual rather than site-wide. Mirrors the natural student journey: learn about the course → see who teaches it → book a session. The About tab was getting long with the full S-T grid.

**Consequences:** New `teachers.astro` page, expanded `StudentTeacher` interface (added `handle`, `bio_short`, `rating`, `rating_count`, `certified_date`), richer SQL query joining `student_teachers` + `users`.

## 3. Add Resources Tab to Course Page (Client-Side Fetch)
**Type:** Implementation
**Topics:** astro, d1, cloudflare

**Trigger:** With Teachers as a tab, course resources (PDFs, videos, sample files) were the logical next tab — the infrastructure already existed but had no course-page UI.

**Options Considered:**
1. Server-side render resources in Astro page (like Teachers tab)
2. Client-side fetch from existing API ← Chosen

**Decision:** Resources tab fetches data client-side via `/api/courses/:id/resources` when the tab is activated, rather than server-rendering in the Astro page.

**Rationale:** The existing API already handles (a) enrollment gating, (b) R2 download URL generation, and (c) public/private filtering. Duplicating that logic server-side in every Astro page would be error-prone and violate DRY. Lazy-loading also keeps the initial page load fast — no API call unless the user clicks the Resources tab.

**Consequences:** Added `courseId` prop to `CourseTabs` component (needed for API call), `useCallback`/`useEffect` for lazy fetching, new `resources.astro` page.

## 4. Remove Migration Redirect Table (Pre-Production)
**Type:** Strategic
**Topics:** astro, workflow

**Trigger:** The routing doc contained a "Migration from Old Routes" table and code snippet for redirects from `/dashboard/*`, `/dash/*`, etc.

**Options Considered:**
1. Keep the migration table for future reference

**Decision:** Remove the entire Migration from Old Routes table and the aspirational redirect code snippet. These old routes (`/dash/*`, `/dashboard/*`) were never exposed externally.

**Consequences:** Removed ~30 lines from `tech-021-url-routing.md`. The redirect handling section now only documents actually-implemented redirects.

## 5. Course Page Tab Order: About → Teachers → Resources → Feed → Curriculum
**Type:** Implementation
**Topics:** astro

**Trigger:** With two new tabs added, needed to decide the ordering.

**Options Considered:**
1. About → Feed → Teachers → Resources → Curriculum
2. About → Teachers → Resources → Feed → Curriculum ← Chosen
3. About → Resources → Teachers → Feed → Curriculum

**Decision:** Tab order follows the student discovery journey: learn about the course → see who teaches it → preview materials → check community discussion → dive into curriculum (enrolled).

**Rationale:** Teachers and Resources are "what you get" information relevant to purchase decisions, so they come before social/community content (Feed). Curriculum is enrollment-gated and represents the deepest engagement, so it's last.

**Consequences:** `allTabs` array ordering in `CourseTabs.tsx`.
