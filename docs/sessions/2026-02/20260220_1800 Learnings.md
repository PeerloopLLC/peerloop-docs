# Session Learnings - 2026-02-20

## 1. Claude Code --add-dir Enables Cross-Repo Tool Access
**Topics:** workflow

**Context:** Needed to run Claude Code from a docs repo while editing files in a sibling code repo.

**Learning:** `claude --add-dir ../Peerloop` grants full Read/Edit/Bash tool access to the specified directory. Permission patterns in `settings.json` only govern the home project — the added directory gets its own access through the `--add-dir` mechanism. This means you don't need `/src/**` patterns in the docs repo's settings.json.

## 2. Symlinks Work Transparently for Node.js Build Tools
**Topics:** astro, workflow

**Context:** After migrating `docs/` and `research/` to a companion repo, the code repo's build scripts (`populate-page-metadata.ts`, `parse-all-pages.ts`) resolve paths via `path.resolve(ROOT, 'docs/...')`.

**Learning:** Creating symlinks (`Peerloop/docs → ../peerloop-docs/docs`) satisfies these `path.resolve` calls transparently. Node.js, Vite, and Astro all follow symlinks without configuration. Verified with a full build (0 errors) and full test suite (266 files, 4891 assertions, 0 failures).

---

## 3. git rm --cached Preserves Files on Disk
**Topics:** workflow

**Context:** Needed to remove 899 files from git tracking in the code repo without deleting them from disk (since we needed them present until symlinks were in place).

**Learning:** `git rm -r --cached <path>` removes files from git's index (staging area) without touching the working directory. This is the correct approach for a phased migration: (1) `git rm --cached` to untrack, (2) delete originals from disk, (3) create symlinks, (4) commit. The `.gitignore` entries prevent git from re-tracking the symlinks or any files that land there.

## 4. $CLAUDE_PROJECT_DIR Points to CC Home, Not Added Dirs
**Topics:** workflow

**Context:** Session hooks use `$CLAUDE_PROJECT_DIR` to locate files like `RESUME-STATE.md` and `SESSION-INDEX.md`.

**Learning:** `$CLAUDE_PROJECT_DIR` always resolves to the project where `.claude/` lives (the CC home). When running from `peerloop-docs/` with `--add-dir ../Peerloop`, hooks correctly reference docs repo files. The `check-env.sh` hook (which validates code repo dependencies) needed its path explicitly changed to `$CLAUDE_PROJECT_DIR/../Peerloop/scripts/check-env.sh`.

## 5. Symlink Bridge Pattern for Transitional Sessions
**Topics:** workflow

**Context:** After migrating root planning files (DECISIONS.md, PLAN.md, etc.) to the docs repo, skills running from the code repo couldn't find them.

**Learning:** Creating temporary symlinks (`ln -s ../peerloop-docs/DECISIONS.md DECISIONS.md`) in the code repo provides a seamless bridge. Since these paths are in `.gitignore`, the symlinks are invisible to git. This pattern could become permanent in `link-docs.sh` for any session that might be launched from the code repo.

---

## 6. Dual-Repo Architecture Reduces Code Repo by 69%
**Topics:** workflow

**Context:** The Peerloop code repo had grown to 2,012 files / 27.6 MB with documentation, session logs, and CC workflow artifacts mixed in with application code.

**Learning:** Separating docs/workflow into a companion repo reduced the code repo to 1,117 files / 8.6 MB — a 19 MB (69%) reduction. The bulk comes from `docs/sessions/` (542 session logs) and `research/` (94 spec files). These grow over time but have zero relevance to build/test/deploy, making them ideal candidates for separation.
