# Session Learnings - 2026-02-05

## 1. Wrangler Migration Tracking Can Fail While Data Succeeds
**Topics:** d1, cloudflare

**Context:** After resetting the local database and running `npm run db:migrate:local`, wrangler reported a FOREIGN KEY constraint error for the seed migration, but inspection showed all data was actually inserted correctly.

**Learning:** Wrangler's migration tracking (d1_migrations table) can fail independently from the actual SQL execution. The data may be applied successfully even when wrangler reports failure. When this happens:
1. Verify data with direct SQLite queries
2. Manually insert the migration record if needed
3. Don't assume the data didn't apply

**Pattern:**
```bash
# Check if data was actually applied
sqlite3 .wrangler/state/v3/d1/.../db.sqlite "SELECT COUNT(*) FROM users;"

# Manually fix migration tracking if needed
sqlite3 ... "INSERT INTO d1_migrations (id, name, applied_at) VALUES (2, '0002_seed.sql', datetime('now'));"
```

## 2. Stream Feed Group Names Must Match Exactly Between Code and Dashboard
**Topics:** stream

**Context:** The PROD Stream app had `timeline` (flat) and `timeline_aggregated` (aggregated), but code expected `timeline` to be aggregated type. Stream doesn't allow changing feed types after creation.

**Learning:** Feed group names and types in Stream Dashboard must exactly match what the code expects. There's no way to change a feed type after creation - you must delete and recreate. Always verify:
- Feed group name matches code constants
- Feed type (flat, aggregated, notification) matches expected behavior
- Both DEV and PROD environments have identical feed groups

## 3. Stream REST API Required for Cloudflare Workers (Not Node SDK)
**Topics:** stream, cloudflare

**Context:** Peerloop uses a custom Stream client in `src/lib/stream.ts` instead of the official Node SDK.

**Learning:** The official Stream Node SDK uses `http.Agent` which is incompatible with Cloudflare Workers' edge runtime. The codebase implements Stream's REST API directly using:
- JWT authentication with HS256 signing
- Base URL: `https://api.stream-io-api.com/api/v1.0`
- Manual request construction with proper auth headers

This pattern applies to any service with a Node SDK that uses non-web-standard APIs.

## 4. Feed Isolation is Automatic in Stream's Architecture
**Topics:** stream

**Context:** Needed to verify that posts to one community don't appear in another community's feed.

**Learning:** Stream's feed architecture provides automatic isolation through the `{feedGroup}:{feedId}` pattern. Each combination is a completely separate data store:
- `community:ai-tools` is isolated from `community:vibe-coding`
- `course:1` is isolated from `course:2`
- No cross-contamination is possible unless explicitly set up via follows

The integration test verified this works correctly for both community and course feeds.

## 5. Test Users Should Be Used Over Manual DB Manipulation
**Topics:** testing, d1

**Context:** When writing the feed isolation test, initially tried to add Brian as a community member directly, but the user preferred using existing seeded test data.

**Learning:** For integration tests, prefer using existing test users who already have proper relationships set up (memberships, enrollments) rather than manually manipulating the database. This:
- Keeps test data "organic" (could exist in real usage)
- Avoids FK constraint issues
- Makes tests more realistic
- Reduces test setup complexity

## 6. Environment Variables for Stream Require Per-Environment Setup in Cloudflare
**Topics:** stream, cloudflare, deployment

**Context:** Setting up Stream credentials for local, preview, and production environments.

**Learning:** Cloudflare Pages requires environment-specific secrets configuration:
1. **Local**: `.dev.vars` file
2. **Preview/Production**: Cloudflare Dashboard → Settings → Variables and Secrets

The Dashboard method is preferred over CLI for environment-specific secrets because `wrangler pages secret put` sets project-wide secrets, not per-environment.
