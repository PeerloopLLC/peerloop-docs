# Session Learnings - 2026-02-23

## 1. Separate Table Pattern for Dual-Role Relationships
**Topics:** d1, auth

**Context:** Designing the community_moderators table for Tier 2 moderators. Had to decide whether to add a `moderator` value to the `community_members.role` enum or create a separate table.

**Learning:** When a user needs to hold two orthogonal attributes simultaneously (membership role AND moderation authority), a separate table is the right pattern. The `community_members.role` enum (`creator`, `student_teacher`, `member`) describes what a user IS — adding `moderator` would force a choice between "member" and "moderator" when the answer is "both." This is the same pattern used by `student_teachers` (a user is both a `community_members` member AND has a `student_teachers` certification row).

**Pattern:** Separate relationship tables for orthogonal concerns:
- `community_members` → what you ARE (identity within community)
- `community_moderators` → what you CAN DO (authority within community)
- `student_teachers` → what you're CERTIFIED FOR (teaching credential per course)

## 2. Scope Inheritance via Existing Relational Chains
**Topics:** d1, auth

**Context:** Community Moderators need authority over course feeds within their community, but there's no direct `community_id` on courses. The relationship goes through progressions.

**Learning:** Rather than adding a denormalized `community_id` to courses or creating a separate moderator-course mapping table, the existing Community → Progression → Course chain provides the scope inheritance naturally. A single JOIN query resolves whether a community moderator has authority over a course feed: `community_moderators → progressions.community_id → courses.progression_id`. Adding a course to a community's progression automatically brings it under existing moderators' scope — zero manual assignment.

**Pattern:**
```sql
SELECT 1 FROM community_moderators cm
JOIN progressions p ON p.community_id = cm.community_id
JOIN courses c ON c.progression_id = p.id
WHERE cm.user_id = ? AND c.id = ? AND cm.is_active = 1
```

## 3. Creator Content Setup Gap — Communities and Progressions Are Read-Only
**Topics:** astro, d1

**Context:** Auditing the codebase to understand the Community → Progression → Course hierarchy, we discovered that only course creation has API + UI support.

**Learning:** Communities and progressions exist in the schema with `creator_id` FK columns, and read APIs are fully implemented (`GET /api/communities`, `GET /api/communities/[slug]/progressions`). But no create/edit/delete endpoints exist for either. All test communities and progressions are created via SQL seed files (`migrations-dev/0001_seed_dev.sql`). This means a Creator cannot set up their own teaching presence through the app — they can only create courses within pre-seeded progressions. The `CreateCourseModal` doesn't even ask for a `progression_id`.

This is a critical MVP gap: the Genesis cohort requires Creators to set up communities and learning paths.

## 4. BBB Implementation Is More Complete Than Expected
**Topics:** video

**Context:** Adding the BBB deferred block to PLAN.md required auditing what's already built.

**Learning:** The BBB integration has significantly more implementation than the PLAN.md and MVP-GOLIVE sections suggested:
- Full VideoProvider abstraction with BBB adapter (room create, join URL, SHA-1 auth, recordings)
- Session CRUD APIs with conflict detection
- Session join endpoint with auto-room creation and 15-min window
- SessionRoom UI with all states (early, joinable, joining, completed, cancelled)
- SessionBooking wizard (4-step: teacher → date → time → confirm)

The actual gap is narrower: no `/session/[id]` page route, no recording playback UI, no rescheduling, and no attendance tracking from webhooks. The components exist but aren't wired into a complete routable flow.

## 5. DB-SCHEMA.md Documentation Gaps Can Hide Implemented Features
**Topics:** docs-infra

**Context:** The `moderation_actions` and `user_warnings` tables exist in `0001_schema.sql` (lines 923-951) with full CREATE TABLE statements, indexes, and foreign keys. But they were completely absent from `research/DB-SCHEMA.md`.

**Learning:** DB-SCHEMA.md is the design document that developers consult when planning features. If a table isn't documented there, it's effectively invisible for planning purposes — even if it's fully defined in the SQL. This session filled the gap by adding both tables to DB-SCHEMA.md with field descriptions matching the actual SQL definitions. Going forward, any schema change to `0001_schema.sql` should be reflected in DB-SCHEMA.md in the same session.
