# Session Learnings - 2026-02-21

## 1. Additive Schema Changes Preserve Full Test Suite
**Topics:** d1, testing

**Context:** Implemented the creator application feature, which required adding a new `creator_applications` table and extending the `notifications` CHECK constraint with a new `'creator_application'` value.

**Learning:** Purely additive schema changes (new tables, appending to CHECK constraints, new type exports) cause zero test regressions. All 4891 tests across 266 files passed without modification. This is the ideal pattern for feature additions: extend, don't modify existing structures.

## 2. D1 batch() for Atomic Multi-Table Updates
**Topics:** d1, auth

**Context:** The creator application approve endpoint needs to update two tables atomically: set the application status to 'approved' AND set `users.can_create_courses = 1`. A partial update (application approved but permission not granted) would be a serious bug.

**Learning:** D1's `db.batch()` guarantees all statements succeed or all fail, making it the right tool for multi-table atomic operations. This avoids the need for explicit transactions (which D1 doesn't support in the traditional sense). The pattern is: `await db.batch([stmt1.bind(...), stmt2.bind(...)])`.

**Pattern:**
```typescript
await db.batch([
  db.prepare('UPDATE creator_applications SET status = ? WHERE id = ?').bind('approved', id),
  db.prepare('UPDATE users SET can_create_courses = 1 WHERE id = ?').bind(userId),
]);
```

## 3. Transactional vs Preference-Gated Emails
**Topics:** astro, workflow

**Context:** The creator application feature sends three types of emails: confirmation (on submit), approval, and denial. Needed to decide whether these should use `sendEmail()` (always sends) or `sendEmailToUser()` (checks user opt-in preferences).

**Learning:** Application lifecycle emails are transactional — they must always be sent regardless of user preferences, similar to password resets. The existing `EmailPreferenceType` union doesn't include a `'creator_application'` type, which is correct: transactional emails shouldn't be opt-outable. Use `sendEmail()` directly for these, and `sendEmailToUser()` only for notifications users might want to disable (session reminders, marketing, etc.).

## 4. excludeCapabilities Pattern for Inverse Menu Filtering
**Topics:** astro

**Context:** The AppNavbar needed to show "Become a Creator" only to users who are NOT already creators, while still showing "Creating" only to users who ARE creators. The existing `capabilities` filter shows items when the user HAS a capability; we needed the inverse.

**Learning:** Adding `excludeCapabilities` to the `MenuItem` interface creates a clean complement to `capabilities`. The filtering logic: `capabilities` = "show if user has ANY of these", `excludeCapabilities` = "hide if user has ANY of these". Extracting a shared `hasCap()` helper avoids duplicating the capability-checking switch statement.
