# Session Learnings - 2026-02-17

## 1. Nested `<a>` Tags Cause React Hydration Errors
**Topics:** astro, react

**Context:** The `/discover/courses` page was throwing hydration mismatch errors in the browser console. The CourseCard component wrapped the entire card in an `<a>` tag, with a second `<a>` tag inside for the creator profile link.

**Learning:** HTML does not allow `<a>` nested inside `<a>`. Browsers "fix" this by closing the outer `<a>` before opening the inner one, producing different DOM than React expects during SSR hydration. This manifests as `validateDOMNesting` warnings followed by a full hydration failure and client-side re-render.

## 2. The Stretched-Link Pattern Solves Nested Anchor Problems
**Topics:** astro, react, ui

**Context:** After identifying the nested `<a>` issue, we needed a pattern that makes the whole card clickable for the primary link while allowing a secondary link (creator name) inside.

**Learning:** The "stretched link" pattern (used by Bootstrap) is the correct solution:
1. Outer element becomes a `<div>` (not `<a>`) with `position: relative`
2. The primary link `<a>` uses `after:absolute after:inset-0` — its `::after` pseudo-element stretches over the entire card
3. Secondary links use `relative z-10` to sit above the stretched overlay

This preserves all native link behaviors for both links: right-click context menu, middle-click/Ctrl+click for new tab, status bar URL preview, and Astro View Transitions.

**Pattern:**
```tsx
<div className="group relative ...">
  <h3>
    <a href="/primary" className="after:absolute after:inset-0">Title</a>
  </h3>
  <a href="/secondary" className="relative z-10">Secondary link</a>
</div>
```

## 3. `<span role="link">` with onClick Loses Critical Link Behaviors
**Topics:** astro, react, ui

**Context:** The initial fix for the nested `<a>` problem was to change the inner `<a>` to a `<span>` with `onClick` → `window.location.href`. This worked but had significant drawbacks.

**Learning:** Replacing `<a>` with non-semantic elements + `onClick` loses:
- Middle-click / Ctrl+click (open in new tab)
- Right-click "Copy link address" / "Open in new tab"
- Status bar URL preview on hover
- Astro View Transitions (bypassed by `window.location.href`)
- Search engine crawlability (no `href` to discover)

The `role="link"` attribute helps screen readers but doesn't restore any of the browser-native behaviors. Always prefer real `<a>` tags with the stretched-link pattern over programmatic navigation workarounds.

## 4. `window.location.href` Bypasses Astro View Transitions
**Topics:** astro

**Context:** During the audit of navigation patterns, we found 15+ instances of `window.location.href` across the codebase. Astro's `ClientRouter` (View Transitions) is active on all layouts.

**Learning:** `window.location.href = url` triggers a full browser navigation, skipping Astro's View Transitions entirely. This means:
- No smooth page transition animation
- Full page reload (white flash)
- All React component state is destroyed and re-created

For internal navigation, use either real `<a href>` tags (preferred) or `navigate()` from `astro:transitions/client` (for post-API-call redirects). Reserve `window.location.href` for auth boundary crossings (login/logout) where clearing React state is intentional.

## 5. `window.location.href` Is Correct at Auth Boundaries
**Topics:** astro, auth

**Context:** When categorizing `window.location.href` instances, we identified that logout/login redirects should NOT use View Transitions.

**Learning:** After auth state changes (login, logout, session expiry), a full page load via `window.location.href` is intentionally correct. View Transitions would preserve stale React component state — the navbar might still show "logged in" after logout, or session-dependent data wouldn't refresh. The full reload ensures every component re-initializes with the new auth context.

## 6. `navigate()` from `astro:transitions/client` Works in React Islands
**Topics:** astro, react

**Context:** We needed programmatic navigation (after API calls) that respects View Transitions, from within React island components.

**Learning:** Astro's `navigate()` function can be imported and used inside React components loaded as islands: `import { navigate } from 'astro:transitions/client';`. Vite resolves this virtual module at build time. It does a client-side fetch of the new page's HTML and morphs the DOM via View Transitions — preserving scroll position and animating the transition. Also useful for soft-refreshing the current page after mutations: `navigate(window.location.pathname)`.

## 7. Fire-and-Forget Fetch for Non-Blocking Side Effects
**Topics:** react

**Context:** In NotificationsList, clicking a notification should mark it as read AND navigate. With the stretched-link pattern, the `<a>` tag handles navigation natively, but we also need to fire a PATCH request.

**Learning:** Using `.then()/.catch()` instead of `await` for the mark-as-read fetch allows the navigation to proceed immediately without waiting. The browser completes in-flight fetch requests even during page transitions (both full navigation and View Transitions). This pattern is appropriate when the side effect (mark as read) is non-critical and shouldn't block the user's primary intent (navigation).

## 8. Extracting useEffect Fetch Functions with useCallback
**Topics:** react

**Context:** AdminDashboard had `fetchDashboard` defined inside `useEffect`, making it inaccessible to retry/refresh buttons. We needed to call it from both the initial load effect and UI event handlers.

**Learning:** Extract the fetch function to a `useCallback` with an empty dependency array, then call it from both the `useEffect` and button `onClick` handlers. This avoids duplicating fetch logic and keeps the dependency chain clean:
```tsx
const fetchData = useCallback(async () => { ... }, []);
useEffect(() => { fetchData(); }, [fetchData]);
// Button: onClick={fetchData}
```
