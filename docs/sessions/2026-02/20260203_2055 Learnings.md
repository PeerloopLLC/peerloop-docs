# Session Learnings - 2026-02-03

## 1. Schema Column Names Don't Match Legacy Code
**Topics:** d1, astro

**Context:** When converting the CPRO (Creator Profile) page from LandingLayout to AppLayout, the original page referenced `is_creator` column which doesn't exist in the current schema.

**Learning:** The database schema uses permission-based columns (`can_create_courses`, `can_teach_courses`, `can_moderate_courses`) instead of role-based columns (`is_creator`, `is_student_teacher`, `is_moderator`). Legacy code in `_src/pages` may reference outdated column names.

**Pattern:**
```sql
-- Wrong (old schema):
SELECT * FROM users WHERE handle = ? AND is_creator = 1

-- Correct (current schema):
SELECT * FROM users WHERE handle = ? AND can_create_courses = 1
```

When mapping to component types that expect role booleans:
```typescript
is_creator: Boolean(creatorRow.can_create_courses),
is_student_teacher: Boolean(creatorRow.can_teach_courses),
is_moderator: Boolean(creatorRow.can_moderate_courses),
```

## 2. Student-Teacher Detection Requires Join Query
**Topics:** d1, astro

**Context:** Converting the STPR (Student-Teacher Profile) page revealed that `is_student_teacher` column doesn't exist. A user is an ST based on having active entries in the `student_teachers` table.

**Learning:** ST status is derived from the `student_teachers` table, not a column on `users`. Use an EXISTS subquery to find users who are active STs.

**Pattern:**
```sql
SELECT u.* FROM users u
WHERE u.handle = ?
AND EXISTS (
  SELECT 1 FROM student_teachers st
  WHERE st.user_id = u.id AND st.is_active = 1
)
```

## 3. User Table bio Column is Named bio_full
**Topics:** d1

**Context:** When assembling the ST profile object, `user.bio` was referenced but the actual column is `bio_full`.

**Learning:** The users table has two bio columns:
- `bio_short` - tagline/one-liner for cards
- `bio_full` - full biography for profile pages

Legacy code may reference `bio` which needs to be mapped to `bio_full`.

## 4. Astro Dynamic Routes Use Square Brackets
**Topics:** astro

**Context:** Clarification on Astro's file-based routing syntax.

**Learning:** Astro uses square brackets `[param]` for dynamic route segments, not curly braces `{param}`. Curly braces are just documentation notation.

- `[param]` - Dynamic segment: `/creator/[handle].astro` matches `/creator/brian`
- `[...slug]` - Rest/catch-all: `/docs/[...slug].astro` matches `/docs/a/b/c`
- `[[optional]]` - Optional parameter (Astro 4+)

## 5. API Routes vs Page Routes Are Different Namespaces
**Topics:** astro, workflow

**Context:** When updating course links from `/courses/` to `/course/`, we needed to be careful not to change API routes.

**Learning:** Page routes and API routes are separate concerns:
- **Page routes** can use singular nouns: `/course/[slug]`, `/creator/[handle]`
- **API routes** typically stay plural: `/api/courses/:slug`, `/api/creators/:handle`

When doing bulk link updates, exclude API routes (which start with `/api/`).
