# Session Decisions - 2026-02-11

## 1. Dark Sidebar Theme for Admin (bg-secondary-900)
**Type:** Implementation
**Topics:** astro, architecture

**Trigger:** Designing the AdminNavbar visual identity.

**Options Considered:**
1. White sidebar matching app side (`bg-white`)
2. Dark sidebar (`bg-secondary-900`) ← Chosen
3. Branded sidebar (primary color gradient)

**Decision:** Admin sidebar uses dark theme (`bg-secondary-900`) with light text, distinct from the app's white sidebar.

**Rationale:** The visual contrast immediately signals "admin mode" without reading text. Matches the existing AdminLayout design and follows the Stripe Dashboard / GitHub Admin pattern. The app side uses `bg-secondary-50` body with white sidebar; admin uses `bg-secondary-100` body with dark sidebar.

**Consequences:** AdminNavbar styles are entirely different from AppNavbar — no shared CSS classes for the sidebar chrome.

## 2. Full-Stack Build for /admin/moderators (UI + All APIs)
**Type:** Strategic
**Topics:** d1, architecture

**Trigger:** The moderators sidebar link had no page. User chose to build it when asked.

**Options Considered:**
1. UI + all APIs — fully working page ← Chosen
2. UI only with stub calls (APIs return 404, error state handles it)
3. UI + list API only (partial functionality)

**Decision:** Build the complete stack: 4 API endpoints + 2 React components + 1 Astro page.

**Rationale:** All the design inputs existed (schema, DB-API spec, user stories), and building everything at once avoids the cost of revisiting the same code twice. The moderator invite flow was already partially implemented (create endpoint + email template existed).

**Consequences:** 7 new files created. The /admin/moderators page is fully functional with list, invite, resend, revoke, and remove capabilities.

## 3. Single Dual-View Endpoint Over Separate Endpoints
**Type:** Implementation
**Topics:** d1, architecture

**Trigger:** Moderators page needs data from two different tables (users vs moderator_invites).

**Options Considered:**
1. Two separate endpoints: `/api/admin/moderators` and `/api/admin/moderator-invites`
2. Single endpoint with `view` query param ← Chosen

**Decision:** Single `GET /api/admin/moderators?view=moderators|invites` endpoint handles both views.

**Rationale:** Keeps the frontend simple — one fetch function, one pagination state, one set of loading/error handling. Stats are always returned regardless of view. The backend switches between SQL queries based on the view param.

**Consequences:** The React component has a tab switcher that resets pagination/filters when changing views, calling the same endpoint with different params.
