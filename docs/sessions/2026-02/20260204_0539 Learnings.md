# Session Learnings - 2026-02-04

## 1. Route Namespace Convention: Activity vs Resource
**Topics:** astro, workflow

**Context:** While converting multiple pages from legacy layouts to AppLayout, inconsistency emerged between `/creator/earnings` and `/creating/*` routes.

**Learning:** Establish clear namespace separation:
- **Activity namespaces** (`/creating/*`, `/teaching/*`, `/learning/*`): Dashboard and tools for users in that role
- **Resource namespaces** (`/creator/*`, `/teacher/*`, `/course/*`): Public-facing profile/detail pages

This creates intuitive routing where `/creating` is "what I'm doing" and `/creator/[handle]` is "viewing someone's profile".

**Pattern:**
```
/creating           → Creator Dashboard (my tools)
/creating/studio    → Course management (my tool)
/creating/earnings  → My royalty earnings (my data)

/creator/[handle]   → Public creator profile (viewing others)
```

## 2. Singular Resource Routes for Course Sub-Pages
**Topics:** astro, workflow

**Context:** Converting course-related pages (discuss, success, book) needed route decisions.

**Learning:** Use singular `/course/[slug]/*` for course sub-pages rather than plural `/courses/[slug]/*`. This treats each course as a resource with sub-pages:
- `/course/[slug]/discuss` - Discussion forum
- `/course/[slug]/success` - Enrollment confirmation
- `/course/[slug]/book` - Session booking

The plural `/courses` remains for the course listing/browse page.

## 3. Creator vs ST Earnings Are Separate Concerns
**Topics:** stripe, d1

**Context:** Discovered that the existing `EarningsDetail` component and `/api/me/st-earnings` endpoint only handle Student-Teacher earnings (70% share), not Creator royalties (15% share).

**Learning:** Creator and ST earnings require completely separate:
- **API endpoints**: `/api/me/st-earnings` vs `/api/me/creator-earnings`
- **Database queries**: Filter by `recipient_type = 'student_teacher'` vs `recipient_type = 'creator'`
- **Access checks**: ST check vs "has courses" check
- **UI components**: Different messaging, different percentage display

A user with both roles needs access to both pages.

**Pattern:**
```typescript
// ST earnings query
WHERE recipient_type = 'student_teacher'

// Creator earnings query
WHERE recipient_type = 'creator'
```

## 4. Page Code Naming Should Match Route Purpose
**Topics:** workflow

**Context:** The page code `CEAR` (Creator Earnings) was used for a page at `/dashboard/teaching/earnings` that actually showed ST earnings, not creator earnings.

**Learning:** Page codes should accurately reflect the page's purpose and user:
- `TEAR` = Teaching Earnings (ST's 70% share)
- `CEAR` = Creator Earnings (Creator's 15% royalty)

Misnamed codes cause confusion during navigation and link updates.

## 5. EarningsOverview Component is Reusable via Props
**Topics:** astro, workflow

**Context:** Both Creator and Teacher dashboards needed earnings summary sections.

**Learning:** The `EarningsOverview` component accepts a `detailsLink` prop, making it reusable across different dashboard contexts:
```tsx
<EarningsOverview earnings={earnings} detailsLink="/creating/earnings" />
<EarningsOverview earnings={earnings} detailsLink="/teaching/earnings" />
```

This pattern allows shared UI with role-specific navigation.

## 6. API Should Include User Handle for Profile Links
**Topics:** d1, auth

**Context:** The Teacher Dashboard had a "View Profile" link hardcoded to `/teachers/me`, but needed to use the universal `/@handle` format.

**Learning:** Dashboard APIs should include the user's `handle` in the response so components can construct proper profile URLs:
```typescript
// In API response
user: {
  name: user.name,
  handle: user.handle,  // Added for profile link
  is_available: !!user.is_available,
}

// In component
<a href={`/@${user.handle}`}>View Profile</a>
```

This enables consistent universal profile URLs across the app.
