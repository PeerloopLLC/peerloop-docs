# Session Learnings - 2026-02-01

## 1. Underscore-Prefixed Component Names Cause ESLint Parsing Errors in Astro
**Topics:** astro, workflow

**Context:** Attempted to rename `AppLayout.astro` to `_AppLayout.astro` to free up the `AppLayout` name for a new layout while keeping the old one accessible.

**Learning:** The Astro ESLint parser does not correctly handle component names starting with underscore (`_`). When using `<_AppLayout>` in Astro templates, ESLint reports "Unable to assign attributes when using <> Fragment shorthand syntax!" - a misleading error that actually indicates the parser is confused by the underscore prefix.

**Pattern:** When renaming layouts/components to "retire" them while freeing up the name:
- Use descriptive prefixes like `Legacy`, `Old`, or suffix like `Deprecated`
- Avoid underscore prefixes which cause parser issues
- Example: `AppLayout` → `LegacyAppLayout` (not `_AppLayout`)

## 2. Bulk Sed Replacements Can Introduce Double Semicolons
**Topics:** workflow

**Context:** Used sed to bulk-replace import statements across 30+ files when renaming a layout component.

**Learning:** When using sed to replace import statements that already end with semicolons, and the replacement pattern also includes a semicolon, you get double semicolons (`;;`). This passes TypeScript but looks incorrect and could cause issues in some parsers.

**Pattern:** After bulk sed replacements on import statements, run a cleanup pass:
```bash
# Fix double semicolons introduced by replacement
find src/pages -name "*.astro" -exec grep -l "COMPONENT" {} \; | xargs -I {} sed -i '' 's|";;|";|g' {}
```

Or ensure the original pattern captures the semicolon to avoid duplication.
