# Session Decisions - 2026-02-06

## 1. Replace Recharts with chart.js + react-chartjs-2
**Type:** Implementation
**Topics:** astro, deployment

**Trigger:** Staging deploy revealed `PieChart.qFX19o4l.js` at 392 KB (115 KB gzip) — the largest client bundle, nearly 2x the React runtime. This came from Recharts bundling large portions of D3 internally.

**Options Considered:**
1. chart.js + react-chartjs-2 — tree-shakeable, canvas-based, ~65 KB gzip ← Chosen
2. visx — low-level D3 primitives, SVG-based, requires 3x more code
3. Hand-rolled SVG — maximum control but no built-in animations/tooltips/responsive
4. Lazy-load Recharts — defers download but doesn't reduce total bytes

**Decision:** Replace Recharts with chart.js + react-chartjs-2 for all 3 chart types (area, bar, pie/donut).

**Rationale:** chart.js is tree-shakeable via its register pattern — only imported elements are bundled. Canvas-based rendering is actually better for analytics dashboards (faster re-renders, better performance with larger datasets). The project uses only 3 simple chart types, making the migration straightforward. The existing abstraction layer meant zero consumer changes.

**Consequences:**
- Chart bundle: 392 KB → 188.5 KB raw (52% reduction), 115 KB → 65.6 KB gzip (43% reduction)
- Rendering changed from SVG to canvas (no visual impact for analytics use case)
- Added canvas + ResizeObserver mocks to `vitest.setup.ts` for test compatibility
- Created centralized `chart-setup.ts` for chart.js component registration

## 2. Canvas + ResizeObserver Global Mocks in Test Setup
**Type:** Implementation
**Topics:** testing

**Trigger:** chart.js (canvas-based) needs `getContext('2d')` and `ResizeObserver`, neither of which jsdom implements. Tests that render chart components without mocking them would fail.

**Options Considered:**
1. Global mocks in `vitest.setup.ts` ← Chosen
2. Install `canvas` npm package (native C++ node-canvas dependency)
3. Mock chart components in each test file individually

**Decision:** Add global canvas and ResizeObserver mocks to `vitest.setup.ts`.

**Rationale:** Global mocks are simplest and avoid adding a native C++ dependency (`canvas` package). Per-test mocking would require updating every test file that renders charts (some don't mock them, like AdminAnalytics). The mocks are lightweight no-ops that let chart.js initialize without errors.

**Consequences:** All tests that render chart.js components work without per-file mocking. Tests don't actually verify canvas output (they test component behavior, not rendering), so no-op mocks are appropriate.
