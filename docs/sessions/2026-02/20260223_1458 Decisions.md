# Session Decisions - 2026-02-23

## 1. Two-Tier Moderation Access Pattern
**Type:** Implementation
**Topics:** auth, d1

**Trigger:** Community moderators (Tier 2) need access to moderation endpoints, but they lack the JWT `moderator` role. All 6 endpoints used `requireRole(['admin', 'moderator'])` which blocked them.

**Options Considered:**
1. Add a JWT `community_moderator` role on login — rejected (stale if moderator status changes)
2. Create a new `requireModerationAccess` helper that checks JWT roles then falls back to DB query ← Chosen
3. Add middleware that injects community moderator info into locals — over-engineered for the need

**Decision:** Created `requireModerationAccess(cookies, jwtSecret, db)` that returns `{ session, scope }` where scope is either `{ type: 'global' }` (admin/moderator) or `{ type: 'community', communityIds: [...] }`. The JWT check happens first (fast path), DB query only for non-admin/moderator users.

**Rationale:** DB-backed scope is always fresh (no stale JWT claims). The two-pass check (JWT first, then DB) means admins/moderators never hit the DB for auth. Returning a typed scope object lets each endpoint enforce its own rules.

**Consequences:** All 6 moderation endpoints refactored. Tier 2 moderators can now dismiss and remove community-scoped flags. Warn and suspend remain Tier 1+ only via `canPerformElevatedAction(scope)`.

## 2. Tier 2 Can Only Dismiss and Remove (Not Warn/Suspend)
**Type:** Strategic
**Topics:** auth

**Trigger:** Need to define what actions community moderators can perform.

**Options Considered:**
1. Full access to all moderation actions — too much power for community-level role
2. Dismiss and remove only ← Chosen
3. Dismiss only — too limited to be useful

**Decision:** Community moderators can dismiss flags and remove content within their community scope. Issuing warnings and suspending users requires global (Tier 1) moderator or admin access.

**Rationale:** Warnings and suspensions are platform-wide actions that affect a user across all communities. Community moderators should only have power within their scope. Dismiss + remove handles the 90% case (removing spam/inappropriate content).

**Consequences:** `warn.ts` and `suspend.ts` reject Tier 2 with 403 immediately after auth. `dismiss.ts` and `remove.ts` check `isFlagInScope` against the flag's `community_id`.

## 3. Out-of-Scope Flags Return 404 (Not 403)
**Type:** Implementation
**Topics:** auth

**Trigger:** When a community moderator tries to access a flag from another community, should we return 403 or 404?

**Options Considered:**
1. Return 403 "Forbidden" — leaks that the flag exists ← Rejected
2. Return 404 "Flag not found" ← Chosen

**Decision:** All scope checks return 404 when a flag's `community_id` doesn't match the moderator's communities.

**Rationale:** Prevents information leakage and enumeration. A moderator scoped to Community A shouldn't be able to determine which flag IDs exist in Community B by observing 403 vs 404 responses.

**Consequences:** Consistent 404 for both non-existent flags and out-of-scope flags across GET detail, PATCH, dismiss, and remove endpoints.

## 4. community_id on content_flags is Nullable with ON DELETE SET NULL
**Type:** Implementation
**Topics:** d1

**Trigger:** Adding `community_id` to `content_flags` to scope flags to communities.

**Options Considered:**
1. Non-nullable with default '' — loses SQL NULL semantics, breaks IN clauses
2. Nullable with ON DELETE CASCADE — would delete flag records if community deleted
3. Nullable with ON DELETE SET NULL ← Chosen

**Decision:** `community_id TEXT REFERENCES communities(id) ON DELETE SET NULL` — nullable because townhall and profile flags have no community. ON DELETE SET NULL preserves historical flag records if a community is deleted.

**Rationale:** Flags are audit records. If a community is deleted, we still want the flag history (now with null community_id, essentially "orphaned" to global scope). NULL community_id also naturally excludes these flags from Tier 2 moderator views (SQL IN clause excludes NULLs).

**Consequences:** Added `feed_group` alongside as a CHECK constraint column. Added index on `community_id`. All new columns nullable so existing data works without migration.
