# Session Learnings - 2026-02-27

## 1. Astro Island Hydration Order Creates Shared Global Race Conditions
**Topics:** astro, auth

**Context:** The login modal (rendered by `AuthModalRenderer` inside `AppNavbar`) was never appearing during E2E tests. Console showed `Cannot read properties of undefined (reading 'authError')`.

**Learning:** When multiple Astro islands share a `window.__peerloop` global, module initialization order matters. `auth-modal.ts` ran first and set `window.__peerloop = {}` (without `networkState`). When `current-user.ts`'s `ensureGlobal()` later checked `if (!window.__peerloop)`, it was truthy, so it skipped adding `networkState`. Result: `getNetworkState()` returned `undefined`, crashing AppNavbar hydration and preventing the login modal from rendering.

**Pattern:** When multiple modules share a global singleton, each module's initializer should ensure its required fields exist — not just check if the object exists:
```typescript
// BAD: only checks top-level existence
if (!window.__peerloop) { window.__peerloop = { networkState: {...} }; }

// GOOD: also ensures required sub-properties
if (!window.__peerloop) { window.__peerloop = { networkState: {...} }; }
if (!window.__peerloop.networkState) { window.__peerloop.networkState = {...}; }
```

## 2. Playwright Strict Mode Requires Precise Selectors
**Topics:** testing

**Context:** Many E2E tests failed with "strict mode violation" because selectors like `getByText('AI Tools Overview')` matched 3 elements (breadcrumb, heading, price card).

**Learning:** Playwright's strict mode (default) fails if a locator resolves to multiple elements. For pages with repeated text (course titles in breadcrumbs, headings, and cards), use the most specific locator:
- `getByRole('heading', { name: '...' })` instead of `getByText('...')`
- `{ exact: true }` to prevent substring matches
- `.first()` when multiple matches are acceptable
- `page.locator('a[href="..."]').filter({ hasText: '...' })` for link disambiguation

## 3. E2E Tests Must Track Layout Changes
**Topics:** testing, astro

**Context:** The existing `homepage.spec.ts` (5 tests) was completely broken — it referenced `getByRole('navigation').getByRole('link', { name: 'Peerloop' })` which assumed a top navigation bar. The app had since switched to a sidebar layout with `<aside>` containing the `<nav>`.

**Learning:** E2E tests are tightly coupled to page structure. When a major layout change happens (e.g., top-nav → sidebar), existing E2E tests silently break if not run regularly. The Playwright tests weren't part of the standard `npm test` workflow (only `npm run test:e2e`), so the breakage went undetected.

## 4. Login Modal Depends on Two-Island Coordination
**Topics:** astro, auth

**Context:** The `/login` page uses two separate Astro islands: `AutoOpenAuthModal` (calls `openLoginModal()`) and `AppNavbar` (contains `AuthModalRenderer` that renders the modal). Both must hydrate for the modal to appear.

**Learning:** The login flow requires: (1) AppNavbar hydrates and mounts AuthModalRenderer, (2) AutoOpenAuthModal hydrates and calls openLoginModal(), (3) AuthModalRenderer subscribes to state change and renders LoginModal. In E2E tests, a 15s timeout is sufficient for this chain, but any hydration error in either island silently prevents the modal from appearing — there's no visible error state on the page.
