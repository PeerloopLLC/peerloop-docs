# Session Learnings - 2026-02-02

## 1. Schema Flag Removal Requires Full Codebase Audit
**Topics:** d1, workflow

**Context:** Removed 5 columns from the users table schema (`bio`, `is_student`, `is_student_teacher`, `is_creator`, `is_moderator`) in Session 161. At the start of Session 162, needed to verify no code still referenced them.

**Learning:** Removing a database column from the schema is just step 1. The codebase will have references scattered across: types, mock data, API endpoints (SELECT/INSERT), components, SSR loaders, and tests. A grep-based audit is essential before the schema change can be considered complete.

**Pattern:** After schema column removal, run:
```bash
grep -r "column_name" src/ --include="*.ts" --include="*.tsx"
```
Then categorize by file type and create a task list.

## 2. bio vs bio_full vs bio_short: Context-Appropriate Field Selection
**Topics:** d1, astro

**Context:** The schema had `bio_short` (tagline, 150 chars) and `bio_full` (detailed, 2000 chars). Different UI contexts need different fields.

**Learning:** When you have short/long variants of a text field, the choice depends on UI context:
- **Cards, listings, search results:** Use `bio_short` (fits in tight spaces)
- **Profile pages, about sections:** Use `bio_full` (room for detail)
- **Content moderation snapshots:** Prefer `bio_full`, fallback to `bio_short`

**Pattern:**
```typescript
// In flags.ts for content snapshots
text: targetUser.bio_full || targetUser.bio_short || undefined
```

## 3. Permission vs State: Two Concepts in One Flag
**Topics:** d1, auth, workflow

**Context:** Analyzing how to replace `is_creator` flag, initially planned simple replacement with `can_create_courses`. Discussion revealed the flag actually represented two different concepts.

**Learning:** Role flags like `is_creator` often conflate two distinct concepts:

| Concept | Type | Question | Time Direction |
|---------|------|----------|----------------|
| **Permission** | Capability | "Can they do X?" | Future-looking |
| **State** | Historical | "Have they done X?" | Backward-looking |

These are **independently true**:
- User has permission but hasn't acted → `can_create_courses=1, has_courses=0`
- User acted but permission revoked → `can_create_courses=0, has_courses=1`

**Pattern:** Separate storage:
- Permission: `users.can_*` columns (admin-controlled)
- State: Derived from relationship tables (`courses`, `student_teachers`)

## 4. The Revoked Creator Scenario
**Topics:** auth, workflow

**Context:** Thinking through what happens when admin revokes a creator's permission.

**Learning:** When permission is revoked, historical state remains true. A "revoked creator" (permission=0, has_courses=1):
- Can't create NEW courses
- Should still access Creator Dashboard (to manage existing courses)
- Should still appear in "Browse Creators" (they have courses to show)
- Their existing courses remain valid in the catalog

This scenario is only possible when permission and state are tracked separately.

## 5. Warm-Up Migrations Validate Patterns
**Topics:** workflow

**Context:** Before tackling the complex `is_creator` removal, did the simpler `bio` → `bio_full` migration first.

**Learning:** Starting with a simpler migration (like `bio`) serves multiple purposes:
1. Validates the replacement pattern works
2. Identifies all the file types that need updating
3. Builds muscle memory for the larger migration
4. Proves TypeScript catches the remaining issues (tests only had errors after src/ was clean)

Run `npx tsc --noEmit` after source changes to verify - errors should only be in test files.
