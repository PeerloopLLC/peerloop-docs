# Session Decisions - 2026-02-06 (Session 204)

## 1. Environment Variable File Architecture
**Type:** Strategic
**Topics:** cloudflare, deployment

**Trigger:** Cognitive load from mixed secrets/non-secrets in `.dev.vars` and no clear reference for what to deploy to Cloudflare.

**Options Considered:**
1. Keep everything in `.dev.vars` only, set CF dashboard manually ← Original state
2. Move non-secrets to `wrangler.toml [vars]`, slim `.dev.vars` to secrets only ← Attempted, reverted
3. Non-secrets duplicated in both `.dev.vars` AND `wrangler.toml [vars]` ← Chosen

**Decision:** Non-secret env vars (publishable keys, app IDs, client IDs) exist in both `.dev.vars` (for local dev on both machines) and `wrangler.toml [vars]` (the only way to get them to CF runtime). Secrets exist in `.dev.vars` (local dev) and `.secrets.cloudflare` (reference for CF dashboard). The duplication is required by Cloudflare's architecture.

**Rationale:** CF Pages dashboard only allows encrypted secrets when `wrangler.toml` exists — non-secrets must come from `[vars]`. Meanwhile, `.dev.vars` must keep non-secrets for MBA-2017 compatibility (Node adapter doesn't read wrangler.toml) and to override production values with dev values locally.

**Consequences:**
- `wrangler.toml` has `[vars]` in three sections (top-level, production, preview)
- `.dev.vars` contains all env vars (secrets + non-secrets)
- `.secrets.cloudflare` contains only secrets for CF dashboard reference
- Non-secrets change rarely, so duplication maintenance is minimal

## 2. Secrets Reference File Named `.secrets.cloudflare`
**Type:** Implementation
**Topics:** cloudflare, deployment, workflow

**Trigger:** Need for a local reference file listing production secrets for Cloudflare deployment.

**Options Considered:**
1. `.prod.vars` — mirrors `.dev.vars` naming ← Initially chosen, renamed
2. `.secrets.cloudflare` — describes purpose explicitly ← Chosen
3. No reference file — rely on tech doc only

**Decision:** Use `.secrets.cloudflare` as the gitignored reference file containing only secrets (no non-secrets) organized by Production and Preview environments. Non-secrets were removed since they deploy automatically via `wrangler.toml`.

**Rationale:** The name `.secrets.cloudflare` is self-documenting — it says exactly what the file contains and what it's for. Removing non-secrets eliminates confusion about what needs manual dashboard setup vs what deploys via git.

**Consequences:** File is a complete checklist for CF dashboard secret setup.

## 3. All CF Secrets via Dashboard (Not CLI)
**Type:** Implementation
**Topics:** cloudflare, deployment

**Trigger:** `wrangler pages secret put` only supports production, no `--env` flag for preview.

**Options Considered:**
1. CLI for production + dashboard for preview ← Two workflows
2. Dashboard for everything ← Chosen
3. CLI for everything ← Not possible for preview

**Decision:** Manage all Cloudflare secrets (both Preview and Production) through the CF dashboard, not the CLI.

**Rationale:** `wrangler pages secret put` has no `--env` flag for Pages projects. Using one consistent workflow (dashboard) is simpler than remembering two different approaches. The CLI is noted as an alternative for production-only in the tech doc.

**Consequences:** One workflow for all secret management; `.secrets.cloudflare` organized by Production and Preview sections to match dashboard tabs.

## 4. Central Env Var Index Doc with Vendor Cross-References
**Type:** Strategic
**Topics:** cloudflare, deployment, workflow

**Trigger:** Env var information scattered across 6+ vendor tech docs with varying levels of detail.

**Options Considered:**
1. Centralize everything in one doc ← Would duplicate vendor code examples
2. Distribute everything to vendor docs ← Requires opening 6 files to answer "what do I need?"
3. Central index + vendor docs keep code examples ← Chosen

**Decision:** Created `tech-026-env-vars-secrets.md` as the master reference answering "what do I need, where does it go, what's secret?" Vendor docs (tech-002, tech-003, tech-004, tech-025) keep their SDK code examples and were augmented with consistent env var tables that cross-reference back to tech-026.

**Rationale:** The index pattern avoids duplication while reducing cognitive load. Tech-026 is the lookup table; vendor docs are the detail pages.

**Consequences:** Two-way cross-references between tech-026 and vendor docs. Vendor docs gained consistent "Environment Variables" sections.
