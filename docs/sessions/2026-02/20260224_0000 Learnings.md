# Session Learnings - 2026-02-24

## 1. encodeURIComponent Doesn't Encode `!` — BBB Expects `%21`
**Topics:** video, testing

**Context:** The BBB adapter's `buildQueryString` used `encodeURIComponent` for parameter values. During plan-mode analysis of Blindside Networks integration, the `!` encoding gap was identified.

**Learning:** `encodeURIComponent` follows RFC 3986 which classifies `!` as an unreserved character — so it's left as-is. However, BBB's checksum verification computes SHA1 on the URL-encoded query string and expects `!` to be `%21`. This causes checksum mismatches when meeting names, welcome messages, or any parameter contains `!`.

**Pattern:**
```typescript
// Fix: post-process encodeURIComponent output
const encoded = encodeURIComponent(String(v)).replace(/!/g, '%21');
```

## 2. Managed BBB Providers Already Include `/api/` in Their URL
**Topics:** video, cloudflare

**Context:** Blindside Networks' API URL is `https://peerloop.api.rna1.blindsidenetworks.com/bigbluebutton/api/`. Our `buildApiUrl` method stripped trailing `/` then appended `/api/`, producing `/api/api/`.

**Learning:** Self-hosted BBB URLs end at `/bigbluebutton` while managed providers like Blindside include `/api/` already. The URL normalization must handle both forms. Strip trailing `/api` or `/api/` before appending.

**Pattern:**
```typescript
const baseUrl = this.config.serverUrl.replace(/\/api\/?$/, '').replace(/\/$/, '');
return `${baseUrl}/api/${apiName}?${queryString}&checksum=${checksum}`;
```

## 3. Blindside Networks Blocks Iframe Embedding — Use `window.open()`
**Topics:** video, astro

**Context:** The original `SessionRoom.tsx` used `window.location.href = joinUrl` to navigate the user to the BBB room. This navigates the user completely away from PeerLoop.

**Learning:** Blindside's BBB deployment has X-Frame-Options headers that block iframe embedding. The correct pattern is `window.open(url, '_blank')` to open BBB in a new tab. This has the advantage of keeping the PeerLoop session page open — it can poll for session completion and transition to the rating view automatically when BBB closes.

**Pattern:**
```typescript
// Open BBB in new tab, keep session page open for polling
window.open(data.join_url, '_blank', 'noopener');
setViewState('in_session');
// Poll every 15s for session completion
```

## 4. Much of the BBB Block Was Already Implemented
**Topics:** video, workflow

**Context:** PLAN.md listed BBB as "pending (needs planning)" with many unchecked items. During implementation, systematic review revealed most components already existed.

**Learning:** Before writing new code, methodically reading every file in the plan's key-files list saves significant time. In this case: webhook handlers, attendance tracking, recording storage, booking wizard, S-T session history, and the book page all existed. The real gap was narrower than it appeared — primarily the session room page (`/session/[id].astro`) and the reschedule endpoint.

## 5. CURRENT-BLOCK-PLAN.md Pattern for Multi-Session Work
**Topics:** workflow, cc-workflow

**Context:** The BBB block is too large for a single session. We needed a persistent tracking file that survives `/compact` and context clears.

**Learning:** A `CURRENT-BLOCK-PLAN.md` at the docs repo root with checkboxes, key files, and a "next session" summary at the top works well. Each session updates checkboxes and adds a progress summary. When the block is complete, the file is deleted and PLAN.md is updated. This is lighter than RESUME-STATE.md (which is session-specific) and heavier than PLAN.md (which tracks all blocks).
