# Session Learnings - 2026-02-16

## 1. Renderless React Islands for Side Effects in Astro
**Topics:** astro, auth

**Context:** Needed to auto-open a modal when users navigate to `/login` or `/signup` from external links, while keeping AppLayout as the page shell.

**Learning:** In Astro's islands architecture, you can create "renderless" React components that return `null` but run side effects via `useEffect`. `AutoOpenAuthModal` renders nothing visually — it just calls `openLoginModal()` or `openSignupModal()` on mount. The cleanup function handles View Transitions back-button navigation by calling `closeAuthModal()` on unmount.

**Pattern:**
```tsx
// Renderless island — provides side effects, no visual output
export default function AutoOpenAuthModal({ mode }: Props) {
  useEffect(() => {
    if (mode === 'login') openLoginModal(onSuccess);
    else openSignupModal(onSuccess);
    return () => closeAuthModal(); // cleanup on unmount
  }, [mode]);
  return null;
}
```

## 2. State Machine > Boolean for Multi-Modal Systems
**Topics:** auth, astro

**Context:** The auth modal system originally used `isOpen: boolean` for a single login modal. Adding a signup modal required tracking *which* modal is open.

**Learning:** Replacing `isOpen: boolean` with `modalType: 'login' | 'signup' | null` makes illegal states unrepresentable — you can't accidentally have both modals open. The discriminated union also enables `switchToLogin()`/`switchToSignup()` functions that change the modal type while preserving `pendingAction`, which is critical for flows like "try to enroll → login modal → switch to signup → complete → enrollment fires."

## 3. Dual-Mode Components Create Dead Code When Architecture Changes
**Topics:** auth, workflow

**Context:** `LoginForm` and `SignupForm` both supported dual modes: page mode (redirect on success) and modal mode (`onSuccess` callback). After converting both `/login` and `/signup` to modal-based flows, the page-mode code paths became unreachable.

**Learning:** When an architecture change makes one mode of a dual-mode component obsolete, remove the dead code immediately rather than leaving it as a "fallback." The stranded page-mode code included `returnUrl` handling, `compact` conditionals, `getRedirectUrl()` with role-based routing, and page headers — all adding complexity to components that are now modal-only. Removing it simplified both components significantly (~120 lines removed total) and made `onSuccess` a required prop, making the contract clearer.

## 4. Password Reset Is Not a Good Modal Candidate
**Topics:** auth

**Context:** After converting login and signup to modals, we evaluated whether `/reset-password` should follow the same pattern.

**Learning:** Password reset has different UX characteristics that make it a poor modal candidate: (1) it has multi-state UI (form → "check your email" confirmation with large icon), (2) it's reached from two different contexts (login modal and settings page), (3) the future email callback link (`/reset-password?token=xyz`) must be a real page. Not every auth flow should be a modal — the right pattern depends on the flow's entry points, states, and future needs.

## 5. Console 401 Errors on /api/me/full Are Expected for Visitors
**Topics:** auth, astro

**Context:** Browser console showed `GET /api/me/full 401 (Unauthorized)` errors that appeared alarming but were actually normal.

**Learning:** In the stale-while-revalidate auth pattern, `initializeCurrentUser()` always calls `/api/me/full`. For unauthenticated visitors, this returns 401, which the code handles gracefully (setting `authStatus: 'visitor'`). Browsers log 4xx fetch responses as console errors regardless of whether the application handles them. This is expected behavior, not a bug. The same applies to window focus events triggering `refreshCurrentUser()`.
