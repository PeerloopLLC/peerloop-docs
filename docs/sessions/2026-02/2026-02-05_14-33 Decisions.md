# Session Decisions - 2026-02-05

## 1. Replace Instructor Feed with Community Feed
**Type:** Strategic
**Topics:** stream, architecture

**Trigger:** User noted that `instructor` feed had been removed from the codebase earlier and replaced with `community` feed, but old references remained.

**Options Considered:**
1. Keep both instructor and community feeds
2. Remove instructor feed entirely, use community feed ← Chosen
3. Merge instructor into community gradually

**Decision:** Remove all instructor feed code and use community feeds exclusively.

**Rationale:** The instructor feed was redundant - communities serve the same purpose of grouping users around topics/creators. Having two similar concepts adds confusion.

**Consequences:**
- Deleted 13+ files (components, API endpoints, tests, specs)
- Removed `instructor_feed_enabled` and `instructor_feed_created_at` from users schema
- Updated all page specs referencing IFED
- Stream Dashboard only needs 5 feed groups, not 6

## 2. Split Migration Seeds: Core (Production) vs Dev (Test Data)
**Type:** Strategic
**Topics:** d1, deployment, workflow

**Trigger:** Discussion about needing different migration paths for production (no test data) vs development (rich test data).

**Options Considered:**
1. Single seed file with conditional logic
2. Separate directories: migrations/ (production) and migrations-dev/ (dev) ← Chosen
3. Seed via application code instead of SQL
4. Environment variables to control seeding

**Decision:** Create two seed files in separate directories:
- `migrations/0002_seed_core.sql` - Essential production data
- `migrations-dev/0001_seed_dev.sql` - Test users, courses, etc.

**Rationale:**
- SQL conditionals don't work well in SQLite
- Complete separation prevents accidents
- Clear mental model: migrations/ = production-safe, migrations-dev/ = dev only
- npm scripts control which seeds apply

**Consequences:**
- New directory structure: `migrations/` and `migrations-dev/`
- New npm scripts: `db:setup:local`, `db:setup:local:clean`, etc.
- Production commands are safeguarded or blocked
- Created `docs/tech/tech-024-migrations.md` for documentation

## 3. DEV Stream App for Local/Preview, PROD for Production

**Topics:** stream, deployment

**Trigger:** User wanted to configure Stream.io credentials for different environments.

**Options Considered:**
1. Use PROD app for all environments
2. Use DEV app for local, PROD for staging and production
3. Use DEV app for local and preview, PROD for production only ← Chosen

## 4. Block Production Seed and Reset Commands
**Type:** Strategic
**Topics:** d1, deployment, workflow

**Trigger:** User emphasized "err on the side of safeguarding PRODUCTION rather than ease of use."

**Options Considered:**
1. Allow all commands with confirmation prompts
2. Block dangerous commands entirely ← Chosen
3. Require special flags for dangerous operations

**Decision:** Certain production commands are completely blocked:
- `db:seed:prod` - Errors immediately with explanation
- `db:reset:prod` - Errors immediately with explanation
- `db:migrate:prod` - Allowed but requires typing "production"

**Rationale:** Some operations should never happen on production (like applying dev seed). Blocking is safer than confirmation prompts which can be bypassed accidentally.

**Consequences:**
- Created `scripts/confirm-prod.js` for confirmation prompt
- npm scripts use `&& exit 1` for blocked commands
- Only schema migrations can be applied to production

## 5. Production Admin User with Placeholder Password
**Type:** Implementation
**Topics:** auth, d1

**Trigger:** Core seed needs an initial admin user for production access.

**Options Considered:**
1. No admin user in seed (create manually)
2. Admin user with same test password as dev
3. Admin user with different placeholder password ← Chosen

**Decision:** Create `usr-admin` (admin@peerloop.com) in core seed with a placeholder password hash, with documentation to change it after first login.

**Rationale:** Production needs initial access, but using the same dev password is risky. The placeholder approach balances convenience with a clear indication that the password should be changed.

**Consequences:**
- Core seed includes usr-admin with email admin@peerloop.com
- Comment in SQL reminds to change password
- Production will have one user after fresh install
