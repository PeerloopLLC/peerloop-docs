# Conv Learnings - 2026-06-30

## 1. Preserve-then-overlay is the correctness invariant for a persistent task file
**Topics:** cc-workflow

**Context:** Conv 351 built `/r-update-tasks` and flipped `/r-start` / `/r-end` / `/r-commit` onto the new git-tracked `CURRENT-TASKS.md` store. The file is the durable backlog SoT; TodoWrite (via `TaskList`) holds only what is in flight this conv (active-only model, DEC-350-2).

**Learning:** The refresh must treat the **file as the ordering authority** and overlay live statuses by `[CODE]` — never regenerate from `TaskList`. The load-bearing guard (spt's [TWAO] rule): do **NOT** drop a file row just because no `TaskList` entry matches its `[CODE]`. Under active-only TodoWrite, an unmatched backlog/Parked row is the *normal* case, not a deletion signal — a regenerate-from-TaskList would erase the entire backlog. Always full `Write`, never `Edit`, never reorder unless the user asks.

**Pattern:**
- Read file → parse H3 `### [CODE]` blocks (preserve Ordered list + every `Why` line + the `> ## ⏸️ PARKED` divider verbatim).
- `TaskList` → map `{CODE: status}`; overlay: `in_progress`→force `· 🔄 Active ·`, `completed`→move to Completed, everything else preserved.
- New `pending`/`in_progress` codes in neither section → append **above** the PARKED divider.
- Shared by `/r-update-tasks`, `/r-commit` Step 0, `/r-end` Step 5.

## 2. De-risk an atomic cutover: dry-run the write engine, then take first-live on the lower-stakes surface
**Topics:** cc-workflow

**Context:** The read path (active-only hydration) and write path (new `CURRENT-TASKS.md` format + narrative-only RESUME-STATE) are coupled — a half-flip (e.g. `/r-start` reading active-only while `/r-end` still emits the old RESUME-STATE-Remaining backlog) would silently lose the backlog. So all four skills had to flip together, which raised the stakes of getting the write engine right before it ran for real.

**Learning:** Validate a not-yet-live write engine **mechanically before flipping**: the new `/r-update-tasks` was dry-run via a mechanical splice against the live file (single insertion hunk, 14 rows preserved, 5 new above PARKED, zero loss) *before* B/C/D were flipped. Then its first genuine live write was sequenced onto the **lower-stakes** surface (`/r-commit` Step 0, verified 3 anchors / 15 rows / CT-* → Completed / banner auto-cleared / zero loss) rather than debuting at the full-scale `/r-end`.

**Caveat (only half-verified at conv end):** the **write** path is live-tested, but the **read** path (`/r-start` Step 7/7.5/7.6/8 against a narrative-only RESUME-STATE + `CURRENT-TASKS.md`) has *not* run yet — its first real test is next conv's `/r-start`. The cutover is intentionally only half-exercised when this conv closes.

## 3. A flat `pending` status must not downgrade hand-set priority symbols
**Topics:** cc-workflow

**Context:** Designing the overlay step that maps `TaskList` status onto the Ordered header symbol (`★ Next` / `🔄 Active` / `📋 Planned` / `⏸️ On hold`). The user hand-curates the Ordered list's priority symbols; `TaskList` only knows the flat states `pending` / `in_progress` / `completed`.

**Learning:** Only two `TaskList` states are *authoritative* over the file: `in_progress` (→ force `· 🔄 Active ·`, the one status TodoWrite knows for certain) and `completed` (→ move the row to Completed). A code-matched `pending` carries **no priority granularity**, so the engine **preserves the file's existing hand-set symbol verbatim** — never overwrites `★ Next` with a generic "pending." A naive "overlay every matched row" would clobber the user's curated ordering. The file is the authority for not-yet-started rows; TodoWrite only speaks for what's in flight or just done.
