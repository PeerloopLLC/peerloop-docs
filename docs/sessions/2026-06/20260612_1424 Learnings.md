# Conv Learnings - 2026-06-12

## 1. The promoted-lane UI consumer is entirely unbuilt — home-feed promo render must use the marketing path
**Topics:** stream, smart-feed, promotion

**Context:** The README framed entity-promo rendering as "the lane reads the fields → resolves the entity," but a code trace showed `getSmartFeed` never calls `getPromotedActivities`, and `GET /api/feeds/promoted` has no UI consumer at all.

**Learning:** For the home feed, entity-promo posts surface via the marketing **sample-post** query (any public-feed post), not the lane. The render-path work is detect `postKind==='entity_promo'` → resolve the anchor from `promoEntityId` (not the post's home feed) → FeedPost embed. The lane-on-feed-pages path is a separate, unbuilt feature.

## 2. `LIKE 'promo_%'` would swallow the gate-password key — escape the underscore
**Topics:** d1, promotion

**Context:** `loadPromotionConfig` mirrors `loadRailsConfig`'s `LIKE 'discovery_%'`. But `_` is a single-char wildcard, so `'promo_%'` matches `promotion_gate_password_hash` (the bcrypt hash) — which would `parseInt` to NaN.

**Learning:** Use `LIKE 'promo\_%' ESCAPE '\'` to force a literal underscore at position 6. The discovery prefix didn't need this (no `discoveryX…` collisions); promo does. Covered by a dedicated test.

**Pattern:**
```sql
WHERE key LIKE 'promo\_%' ESCAPE '\'
```

## 3. Keeping entity-promo as `kind='sample-post'` avoids the orchestrator silently dropping it
**Topics:** stream, smart-feed, promotion

**Context:** The plan warned entity-promo posts could be "silently dropped." The orchestrator filters injectables with `e.kind === 'sample-post'`; a new `kind='entity-promo'` would be excluded there unless every injection site is updated.

**Learning:** Keep `kind='sample-post'` and discriminate at RENDER via a `promoContext` field (checked before the discovery branch). The orchestrator never special-cases it → no drop risk; the render still picks the promoted-entity anchor.

---

## 4. React 19 types deprecate the bare `FormEvent` — match the codebase's `React.SyntheticEvent<HTMLFormElement>`
**Topics:** astro, testing

**Context:** `React.FormEvent` and an imported `FormEvent` both raised `ts(6385) 'FormEvent' is deprecated` (astro check hint). `npm run check` surfaces these even when `tsc` is silent.

**Learning:** The codebase form-submit convention (LoginForm) is `React.SyntheticEvent<HTMLFormElement>` — non-deprecated. Astro check catches deprecation hints that the build doesn't fail on, so it's worth keeping at 0 hints.
