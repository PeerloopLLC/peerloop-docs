# Conv Learnings - 2026-06-28

## 1. Persistent dev-D1 drift — schema edits don't reach a materialized local D1
**Topics:** d1, cloudflare, testing, cc-workflow

**Context:** First live submit returned 500 because `SELECT … r2_key` threw — the column existed in `0001_schema.sql` but not in the running local D1.

**Learning:** `migrations/0001_schema.sql` is `CREATE TABLE IF NOT EXISTS`, so editing it is a no-op against an already-materialized local D1. Production/staging get new columns because they apply the schema fresh; the long-lived local SQLite file does not. Any **live** verification of a schema change must first apply it to the dev DB — either a full `db:setup:local:dev` reset or a surgical, non-destructive `ALTER TABLE … ADD COLUMN` (exactly equivalent for nullable adds).

**Pattern:**
```bash
npx wrangler d1 execute peerloop-db --local --command "ALTER TABLE … ADD COLUMN …"
# then clean up any test rows afterward
```

## 2. Cross-realm `instanceof File` trap under jsdom
**Topics:** testing, astro

**Context:** Multipart endpoint tests returned 400 (file came back null) then 500 (undici parser assertion).

**Learning:** Under jsdom the handler's global `File`/`Blob` are jsdom's, but a file parsed by undici's `formData()` is an undici `File` — different constructor, so `instanceof File` is false for a genuine upload, and undici's own multipart parser asserts against globals jsdom has replaced. The resources upload endpoint sidesteps this by casting + truthiness, not `instanceof`.

**Pattern:**
```ts
// Duck-type uploaded files — realm-agnostic, identical behavior in the Workers runtime
f && typeof f !== 'string' && typeof (f as File).arrayBuffer === 'function' && (f as File).size > 0
```

## 3. Testing a multipart endpoint under a jsdom vitest env
**Topics:** testing

**Context:** jsdom's `Request.formData()` doesn't parse multipart bodies; building the request with undici conflicts with jsdom's global File/Blob; no test in the repo uses `@vitest-environment node`.

**Learning:** Rather than introduce a novel per-file test environment or fight cross-realm Blob validation, mock the exact request surface the handler uses — `headers.get('content-type')` + `formData().get(field)` — returning a plain object with `name`/`type`/`size`/`arrayBuffer` for the file. Because the handler duck-types the file, this is a faithful and more isolated unit test of the endpoint's logic.

## 4. DOM-verification catches a bug class unit tests can't
**Topics:** testing, cc-workflow

**Context:** The 6714-green unit suite passed *with* a cache-leak present; only switching sessions against the same cached URL in a real browser surfaced it.

**Learning:** `Cache-Control: private, max-age=3600` on a per-user authorization-gated resource is a shared-browser data-exposure risk (user B served user A's cached file without re-auth). Unit tests never involve a browser HTTP cache, so they can't see it. Verification is a different instrument, not a rubber stamp on green tests. For per-user private files use `private, no-store`.
