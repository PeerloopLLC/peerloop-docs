# Conv Learnings - 2026-06-22

## 1. Conformance status-token rule: map where a Matt token exists, keep honest-orphan where none does
**Topics:** matt-design, conformance, color-tokens

**Context:** The /learning playbook said "decorative sky → brand-purple", but AvailabilityCalendar (sky=available) and EarningsDetail (rich transaction-status palette) both use blue/colour as *semantic status*, not decoration. Applying brand blindly would have recolored a whole calendar purple and homogenized a money-status palette.

**Learning:** The clean general rule (user-decided this conv): **map a legacy hue to a Matt semantic token where a matching one exists** (red→`error-*`, amber→`warning-*`), and **keep it as an honest-orphan Tailwind hue where Matt has no matching token** (sky=available, green=saved/positive, the categorical stat-accent quartet). This is *why* the calendar blue and earnings palette were kept — Matt has no "available"/"success"/categorical tokens. Decorative-sky→brand still applies to genuinely *decorative* (non-semantic) sky tints.

**Pattern:**
Per colour occurrence ask:
1. Interactive control? → americana (`<Button>` / `text-primary-default`).
2. Decorative non-semantic tint? → brand.
3. Semantic status with a Matt token? → map (error/warning).
4. Semantic status with NO Matt token? → keep honest-orphan Tailwind hue.

## 2. Migrate kept-blue off the deprecated `primary-*` ramp to Tailwind `sky-*` (identical hex)
**Topics:** conformance, color-tokens

**Context:** AvailabilityCalendar's "available" blue was on the legacy `primary-*` ramp (being retired). Keeping the blue but leaving `primary-*` classes would (a) trip the `primary-[0-9]` leak grep and (b) break when the ramp is retired.

**Learning:** `--color-primary-50…600` in global.css ARE the Tailwind `sky` hex values (`#f0f9ff`=sky-50, `#0ea5e9`=sky-500). So `primary-N`→`sky-N` preserves the EXACT colour while dropping the deprecated-ramp dependency and clearing the leak grep. `sky-*` was unused in the codebase but Tailwind v4's default palette is intact (blue/green/amber/red/purple all resolve) — JIT generates `sky-*` on first source use. DOM-verified it resolves (not transparent).

## 3. DOM-truth substring selectors catch hover-variants — filter by `hover:` prefix
**Topics:** dom-truth, chrome-bridge, conformance

**Context:** The transparency-trap scan (`[class*="bg-brand-100"]` → check computed bg ≠ transparent) flagged an `<a>` as a "trap" on the overview tab.

**Learning:** A `[class*="bg-X"]` substring selector also matches `hover:bg-X` variants, which are correctly transparent at rest. Before counting a transparency hit as a real failure, confirm the matched class is NOT a `hover:`/`focus:`-prefixed variant. (Confirmed: the one hit was `hover:bg-brand-100`.) Same care applies to `border-primary-` matching `border-primary-default` (americana, not a `primary-N` leak).

## 4. Conformance breaks tests that assert on pre-conformance class selectors
**Topics:** conformance, testing

**Context:** `EarningsDetail.test.tsx` queried `.rounded-xl.border.p-4` (4 summary cards) and `.w-2.h-2.rounded-full` (3 status dots) — exactly the classes conformance changed to `rounded-12`/`p-16`/`w-8 h-8`. 3 tests failed (35 passed).

**Learning:** Class-coupled test selectors are brittle to conformance. The expected, legitimate fix is to update the selector to the conformed class (test *intent* — count 4 cards / 3 dots — is preserved), not to revert the component. Run the directly-affected component tests after each conform to catch these. (Precedent: /profile sweep updated `border-red-200→error-300` test assertions.)
