# Conv Learnings - 2026-06-14

## 1. Feed rewrite is a hybrid Stream/D1 architecture — seed coverage is split across two mechanisms
**Topics:** stream, d1, testing

**Context:** Assessing whether enough seed data exists to demo the feed rewrite (Convs 248–282).

**Learning:** Post *content* lives in Stream.io (seeded at runtime by `seed-feeds.mjs`, gated on Stream creds), while D1 holds the supporting substrate (feed_activities badge index, announcements, post_promotions). The static `0001_seed_dev.sql` looks thin for feeds because the bodies are runtime-seeded. Two D1-only FEED-U3 features (announcements, promotions) fell through BOTH paths and had zero seed — invisible until you trace each feature's read-path to its table.

## 2. post_promotions FK forces seed into seed-feeds.mjs, not SQL
**Topics:** d1, stream, testing

**Context:** Deciding where to seed post_promotions.

**Learning:** `post_promotions.source_activity_id → feed_activities(id)`, and feed_activities rows are written at runtime by `seed-feeds.mjs` (ids `fa-seed-NNN` = array index). A static SQL promotion row would dangle/FK-fail. Correct home is inside seed-feeds.mjs after Step 4, deriving the FK from `activity.index` — an inline `promoteTo` tag on each source post keeps the reference reorder-proof (the REACTIONS array's stale index comments show what hardcoded indices drift into).

**Pattern:** Tie a derived-FK seed row to its parent intrinsically (inline tag + `activity.index`) rather than by a hardcoded id duplicated in a side array.

## 3. wrangler d1 JSON output: alias the count or you parse meta.duration
**Topics:** d1, cloudflare

**Context:** Local D1 counts looked "dirty" (users=0) but FKs to users succeeded.

**Learning:** `wrangler d1 execute --command "SELECT COUNT(*) ..." | grep -oE '[0-9]+' | tail -1` grabs the trailing `meta.duration` (0/1), not the count. Always `SELECT COUNT(*) AS n`, pass `--json`, and parse the named `"n":` field. The "dirty DB" finding was a parser artifact, not real state.

## 4. db:setup:local:dev applies only 0001 — 0002/0003 are opt-in
**Topics:** d1, deployment, testing

**Context:** Choosing where the announcements seed must live to reach the default dev demo.

**Learning:** `db:setup:local:dev` = reset+migrate + `db:seed:local` (0001 only) + `db:seed:feeds:local`. The stripe (0002) and booking (0003) seeds are NOT in the default local chain (they ARE in the full `db:setup:staging:feeds` chain). So core demo data must live in 0001_seed_dev.sql or seed-feeds.mjs to reach a default local reseed.

---

## 5. seed-feeds.mjs --clean purges only D1, not Stream activities
**Topics:** stream, testing

**Context:** Browser render verification — a stale leftover "hi" Stream post appeared in the community feed.

**Learning:** `seed-feeds.mjs --clean` clears only D1 tables; Stream activities accumulate across reseeds (foreign_id-keyed so no duplication, but stale test posts persist). Could clutter a live demo — a Stream-side purge would be needed for a fully clean demo state.
