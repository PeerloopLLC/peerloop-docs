# Conv Decisions - 2026-06-11

## 1. Rails-backed cards vs standalone marketing queries
**Type:** Implementation
**Topics:** smart-feed, discovery-rails

**Trigger:** Phase-1 card/announcement source choice.

**Options Considered:**
1. Reuse the Discovery Rails blob ← Chosen
2. Build standalone de-personalized trending/new queries in `getMarketingCandidates`

**Decision:** `getMarketingCandidates` consumes an injected `DiscoveryRailsBlob` for cards; only the sample-post query is net-new.

**Rationale:** No duplicate aggregation (the redundancy `[RECO-UNIFY]` removes anyway); the blob is exactly the card set; dependency-injection keeps the builder unit-testable.

**Consequences:** Phase 3 must thread the real blob (KV-read + compute fallback) into `getSmartFeed`.

## 2. Tier-5 "The Commons anchor" retired as a dead premise
**Type:** Strategic
**Topics:** smart-feed, system-feed

**Trigger:** The Conv-258 always-full cascade lists a Commons anchor tier.

**Options Considered:**
1. Drop it — tiers 1–4 only ← Chosen
2. Designate a non-System always-on anchor feed (later sub-task)

**Decision:** Drop tier-5; no anchor sub-task. `getMarketingCandidates` excludes `system` feeds entirely.

**Rationale:** Tier-5's sole justification was "everyone's auto-joined"; SYS-RENAME (Conv 259) retired `autoJoinTheCommons` + made the System feed admin-only 7 convs ago, deleting that premise. Tiers 1–4 already draw globally from all public content, so always-full holds. Pulling Commons posts into a logged-out feed would leak admin-only content.

**Consequences:** Member-facing System content reaches users via Announcements (`[ADMIN-FEED-UI]` #30), not the marketing feed. Doc/code reconciled, not an override of a live decision.

## 3. Phase 2 keeps the endpoint gated until Phase 3
**Type:** Implementation
**Topics:** smart-feed, api

**Trigger:** Phase 2 reworks `getSmartFeed`'s internals + signature, but un-gating is Phase 3.

**Options Considered:**
1. Keep `/api/feeds/smart` 401-gated, pass `railsBlob=null`, filter cards ← Chosen
2. Fold Phase 3's un-gate + KV wiring into Phase 2

**Decision:** Endpoint stays gated; orchestrator becomes auth-aware (`userId: string | null`) but the endpoint passes a real session id, null blob, and filters `suggestion-card` items.

**Rationale:** No observable behavior change mid-build; fully test-driven; avoids pulling KV/caching forward (the reason Phase 1 was scoped alone).

**Consequences:** Phase 3 un-gates + wires the blob; Phase 4 stops filtering cards + adds render variants.
