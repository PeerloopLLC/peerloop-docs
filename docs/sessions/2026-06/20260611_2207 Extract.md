## Learnings

### 1. SmartFeed visitor-awareness via a server-baked response flag, not a client auth guess
**Topics:** smart-feed, astro, cc-workflow
**Context:** `SmartFeed` is a prop-less `client:load` island; making it visitor-aware (hide member-only tabs) needed the viewer's auth state without a first-paint `localStorage` guess.
**Learning:** Add the auth-dependent fact to the API payload (`viewerAuthenticated` on `/api/feeds/smart`) and let the dumb island read it — the same "server bakes auth into the response" pattern as Phase-6's `ctaUrl` branch. This is the project's standing defense against the `[NUDGE-CACHE-FLASH]` staleness class. The visitor branch stays cacheable because the flag is constant (`false`) for all visitors.

### 2. Astro → React component: pass `className`, not `class`
**Topics:** astro, d1
**Context:** Restyling `StickySignupBar.astro`, added a class to the React `<Button>`.
**Learning:** The React `Button` destructures `className` (excluding it from the `...anchorProps` spread). Passing `class=` from Astro leaks a second `class` attribute through the spread alongside `className={classes}` — the duplicate-attribute JSX gotcha. Always pass `className` to a React framework component from Astro.

### 3. "Cosmetic polish" can hide a cross-task scope fork
**Topics:** cc-workflow, feeds
**Context:** "Complete HOME-FEED-MERGE" looked like 3 paint jobs; remnant ① (FeedPost teaser restyle) actually needed a non-existent `CommunityAnchor` + candidate-meta enrichment and would regress the Phase-6 ctaUrl.
**Learning:** Investigate each "polish" item for hidden dependencies before committing to scope. The light path (restyle the existing component, preserve behavior) vs the full path (swap to the target component + its deps) is a real fork worth surfacing.
