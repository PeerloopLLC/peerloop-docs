# Conv Learnings - 2026-06-11

## 1. Two feed-delivery patterns already coexist; promotion needs the pull one
**Topics:** stream, smart-feed, architecture

**Context:** Deciding whether promotion must write to Stream.

**Learning:** The app reads feeds three ways — native `feed.get(group:id)` (push/direct), `timeline:userId` follow-fanout (push), and SmartFeed D1-select + `getActivitiesByIds` (pull). Promotion is cross-membership-boundary delivery, which feed-level access control exists to block, so it must use the PULL pattern (query-on-read), not push. Push (Stream fanout) only works in-boundary.

**Pattern:** Select candidates from D1 (`feed_activities`/`post_promotions`), fetch content by activity id (server token, feed-access-blind), render. Same path SmartFeed already uses.

## 2. SYS-RENAME structurally broke fanout to the System feed
**Topics:** stream, feeds, system-feed

**Context:** Why TO-targeting (option 2) fails for Course→…→System.

**Learning:** SYS-RENAME made the System/Commons feed admin-only AND retired `autoJoinTheCommons` (severed the follow graph). So a TO-targeted activity into the System feed reaches only admins — students neither read it nor follow it. The pull/reference model sidesteps this because the query (not Stream feed membership) decides who sees it.

---

## 3. Stream `to`-targets are updatable on existing activities; reactions are shared by activity id
**Topics:** stream

**Context:** Evaluating TO-target (option 2) vs copy (option 3).

**Learning:** Stream's `to` field copies an activity into additional feeds ("email CC", max 100); to-targets can be updated AFTER creation (select by id or foreign_id+time). Reactions attach to the activity id and propagate to all feeds where it appears. A separate `addActivity` (copy) gets a new id → split engagement. Removing the origin activity cascades to all to-targets. (Verified via getstream.io docs.)

---

## 4. Reaction/comment endpoints gate on auth only — a latent visitor-gating hole
**Topics:** auth, feeds, visitor-gating

**Context:** Mapping what an inbound promoted-to visitor can do in a source feed.

**Learning:** `POST /api/feeds/{course|community}/[slug]/{reactions,comments}` check `Authentication required` (401) but NOT membership/enrollment. A logged-in non-member can react/comment with just an activityId. The new-post path IS gated (`checkPostingRights` / membership); reactions/comments were left open. Promotion makes this load-bearing → folded into `[VISITOR-GATING]` #29.
