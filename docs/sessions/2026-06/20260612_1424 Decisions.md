# Conv Decisions - 2026-06-12

## 1. Promotion expiry is computed from a dial, not a stored `expires_at` column
**Type:** Implementation
**Topics:** promotion, schema, smart-feed

**Trigger:** U3a premise-check; the plan left it open as "`expires_at` or computed."

**Options Considered:**
1. Computed from `promo_active_duration_days` dial (lane filters on `created_at` window) ← Chosen
2. Add `post_promotions.expires_at`, populated `created_at + 14d`

**Decision:** Computed. The lane already filters by a `created_at` window; the schema's own comment defers columns to "avoid half-built columns." Add `expires_at` only when paid variable durations are real.

**Rationale:** No schema change, no backfill, matches existing lane behavior.

**Consequences:** `promoted.ts` defaults its window to the dial; cron purges on `promo_retention_days`.

## 2. Announcement data model deferred from U3a to U3c
**Type:** Strategic
**Topics:** promotion, feed, schema

**Trigger:** U3a premise-check revealed announcements already have substrate (System-feed activities + `notifications` 'system' + a deferred smart-feed Announcements lane) and a net-new table would have no done-test in U3a.

**Options Considered:**
1. Net-new `announcements` table now (per plan)
2. Reuse existing substrate
3. Defer the table to U3c ← Chosen

**Decision:** Defer. U3a ships only promotion lifecycle (real done-test); the announcement model is defined in U3c alongside its author+fan-out where real columns become knowable.

**Rationale:** Honors the plan's own "standalone done-test / no half-built artifacts" isolation principle.

**Consequences:** U3c now owns the announcement model + the admin surface.

## 3. A2 composer places the promo into the entity's own public feed (Option A)
**Type:** Strategic
**Topics:** promotion, smart-feed, feed

**Trigger:** A coherence gap — U3b's backbone surfaces entity-promo via the marketing query (public feeds only), but the README's A2 framing was lane-based (lane consumer unbuilt); the chain target may be the admin-only System feed.

**Options Considered:**
1. Author into a public feed the author controls; backbone surfaces it; record a post_promotions row too ← Chosen
2. Author into the chain target + build the unbuilt lane→home-feed consumer

**Decision:** Option A. Lowest-risk, coheres with the shipped backbone, no unbuilt-lane dependency.

**Rationale:** The whole point (cross-boundary reach to non-members) is already achieved by home discovery of a public-feed post.

**Consequences:** `createEntityPromo` posts into the entity's own feed; `post_promotions` from=to=entity feed (for U3c moderation + future lane, not home-feed visibility).

## 4. Composer mount deferred to U3d
**Type:** Implementation
**Topics:** promotion, ui

**Trigger:** The composer's natural home (a standing prompt in `/creating`+`/teaching`) is explicitly U3d's nudge layer.

**Decision:** Build the create endpoint + composer component now (unit-tested in isolation); wire the real workspace mount in U3d.

**Rationale:** Avoids pulling U3d's workspace-prompt work forward; component is verifiable via tests.

**Consequences:** `EntityPromoComposer.tsx` exists but is unmounted until U3d.
