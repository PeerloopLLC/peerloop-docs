# Conv Decisions - 2026-06-20

## 1. SessionBooking teacher ratings → 5-star fractional display (not compact ★ N.N)
**Type:** Implementation
**Topics:** astro, ui, route-sweep

**Trigger:** CC initially planned to skip SessionBooking (a `★ 4.5` badge ≠ a 5-star widget); user pushed to reconsider a 5-star presentation.

**Options Considered:**
1. Keep compact `★ 4.5` badge (StarRating interactive-only)
2. 5-star display + number (StarRating gains readOnly fractional-fill) ← Chosen

**Decision:** SessionBooking's two teacher-average spots render `★★★★½ 4.5` via a new `readOnly` fractional mode on StarRating.

**Rationale:** The spots live in the teacher-**selection-by-comparison** flow, the one display context where visual star-length aids scanning across options. It also makes SessionBooking a real StarRating consumer and genuinely earns the `/book` XCUT-BACKREF.

**Consequences:** StarRating carries an interactive mode AND a readOnly fractional-fill mode (half-star via width-clipped overlay) + `showValue`.

## 2. Extract StarRating now + unify the rating gold on `--star`
**Type:** Implementation
**Topics:** astro, ui, route-sweep, matt-provenance

**Trigger:** Sensor + sweep found the same hand-rolled ★-button widget in SessionCompletedView (main+sub) + CourseReviewModal (local StarRow) colouring stars 3 ways (`text-[#f5b800]`, `text-amber-400`, `text-star`) — Rule-of-Three met.

2. Extract now + adopt across all sites ← Chosen
3. Keep logged as watch

**Decision:** Built `ui/StarRating.tsx` and adopted it across SessionCompletedView, CourseReviewModal (surgical — its other legacy styling deferred to its own route sweep), and SessionBooking.

**Rationale:** Extraction collapses 3 divergent colourings into the one `--star` token in a single move; the cost (a `/book` back-pointer re-glance) is cheap. The stale "no token exists" comment in SessionCompletedView was retired (the token has existed since Conv 297).

**Consequences:** New registered primitive; star gold unified app-wide on `text-star`; CourseReviewModal's local `StarRow` removed (hover-label preserved via `onHoverChange`).

## 3. FeedPost reclassified as a non-primitive adapter (not registered)
**Type:** Implementation
**Topics:** testing, matt-provenance, cc-workflow

**Trigger:** prov:sweep "Provenance: UNMARKED but not in registry" drift on FeedPost; registering it caused 2 UNSTAMPED errors.

**Options Considered:**
1. Register FeedPost like the other primitives (caused UNSTAMPED — rejected)
2. Reclassify at source: drop the `Provenance: UNMARKED` note, document it's a non-rendering adapter ← Chosen

**Decision:** FeedPost stays unregistered + unstamped; its note now states it returns `<SocialPost>` directly (no DOM root), so SocialPost owns the provenance.

**Rationale:** A pass-through adapter has no DOM identity to stamp and can't collide with a Matt frame, so it isn't a registry collision candidate. The `UNMARKED` keyword was mis-applied.

**Consequences:** prov:sweep green (0 issues); the registry-vs-adapter distinction is now documented inline.
