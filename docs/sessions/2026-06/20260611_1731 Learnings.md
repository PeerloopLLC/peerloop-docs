# Conv Learnings - 2026-06-11

## 1. Put the visitor-vs-authed branch where auth is already known — the server
**Topics:** smart-feed, astro, auth

**Context:** HOME-FEED-MERGE Phase 6 needed feed discovery CTAs to differ by viewer auth (visitor → signup, authed → entity). The feed cards (`DiscoveryCard`/`SuggestionCard`) are prop-less dumb renderers; `SmartFeed` is a prop-less island.

**Learning:** Rather than thread `isAuthenticated` into the island or read the `localStorage` user-cache client-side (the [NUDGE-CACHE-FLASH] first-paint-staleness class), bake the branch into the server-built `ctaUrl`. The orchestrator already has `userId: string | null`. Cards stay dumb, caching stays valid (the visitor `ctaUrl` is identical for all visitors), and a whole class of first-paint cache bug is sidestepped.

**Pattern:** When a client component needs to branch on auth but the data it renders is already assembled server-side, branch in the server payload, not the client.

## 2. Check for existing machinery before building a "new" feature
**Topics:** signup, reuse, intent-preservation

**Context:** Phase 6 was framed as "build intent-preserving signup."

**Learning:** The signup `?redirect=` round-trip already existed end-to-end (`signup.astro` → `AutoOpenAuthModal` → `handleAuthSuccess`) and `EnrollButton` already consumed it. The actual deliverable was just making the feed CTAs *use* the proven pattern — a much smaller, lower-risk change that follows an established codebase precedent (so not a novel decision per §Critical Rule). An Explore-agent map of the signup flow surfaced this before any code was written.
