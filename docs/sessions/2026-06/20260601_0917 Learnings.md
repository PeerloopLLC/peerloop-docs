# Conv Learnings - 2026-06-01

## 1. Don't invent mechanisms when polishing a user's draft wording
**Topics:** cc-workflow, skills

**Context:** Turning the user's `/quit` draft into Step 9 text, I asserted CC auto-checks for updates on `/quit` — a mechanism I never verified. The user caught it: "You changed my wording and that is what your change implies."

**Learning:** When converting a user's draft into doc/skill text, preserve intent; do NOT add or assert behaviors/mechanisms you can't verify. If the wording implies an unsure factual claim, ask.

## 2. Verify the deterministic hook exists before claiming a downstream step consumes an artifact
**Topics:** cc-workflow, skills

**Context:** Claimed r-end would "read the scratch file from disk" before any mechanism pointed it there — the dependency lived only in chat, which `/compact` erodes.

**Learning:** A cross-step dependency isn't real until a deterministic hook (here: the r-end Step 2 glob) encodes it. Build the hook, then claim the consumption.

**Context:** User: "treat this insertion as you would any page port." The work inserted a ported composite into an existing page section, not a new route.

**Learning:** RTMIG/RTMIG-TIER (Tier classification + field-by-field legacy→new parity) apply to a section-level insertion the same as a full page. Did a parity table before declaring done.

## 6. prov:sweep is the registration gate for new @matt-inspired components
**Topics:** matt-provenance, testing

**Context:** `FeedsHubPanel` stamped `data-prov-name` but had no registry row → sweep errored UNTRACKED.

**Learning:** Any new `@matt-inspired` component must be added to `scripts/matt-inspired-registry.ts` (re-exported via the `prov-candidates.ts` shim) or prov:sweep fails. The sweep is the safety net that catches the omission.
