## 1. Merge /visitor into auth-aware /profile + noindex account/auth pages (PROF-MERGE)
**Type:** Strategic
**Topics:** astro, seo, architecture

**Trigger:** User asked to compare `/profile` vs `/visitor` "with an eye to consolidating," then added an SEO/bookmarkability lens.

**Options Considered:**
1. Keep `/visitor` as a standalone public logged-out twin of `/profile`
2. Fold `/visitor` into an auth-aware `/profile` + add a `noindex` mechanism ← Chosen

**Decision:** Delete `visitor.astro`; make bare `/profile` public and auth-aware (logged-out → Log in / Sign up + `<ThemeToggle>` entry surface; signed-in → account hub). Sidebar profile chip always links to `/profile`. Middleware: `/profile` leaves `PROTECTED_PREFIXES`; new `PROTECTED_SUBPATHS_ONLY = ['/profile']` gates sub-tabs while leaving the bare path public. New `noindex?: boolean` prop threaded `BaseHead`→`AppLayout`, applied to `/profile`, `/login`, `/signup`.

**Rationale:** Two lenses converged. **Addressability** — `/visitor` had one inbound (the Sidebar chip), zero redirects/deep-links, bounced authed users away, and its slug didn't name its purpose, so it never earned a standalone URL. **SEO** — the logged-out entry surface has no index value and the app had no robots/noindex mechanism at all (every public page indexable; `robots.txt` is `Allow: /`), so the merge added the `noindex` hook account/auth surfaces needed anyway.

**Consequences:** First robots/indexing control in the app (`noindex?` prop); new "bare path public, sub-paths gated" middleware pattern (`PROTECTED_SUBPATHS_ONLY`); `visitor.astro` deleted; route maps regenerated; PLATO + middleware tests updated. 5 gates GREEN (6728 tests). Code `7a74abd9`, docs `17d9c25`. Recorded in `docs/decisions/01-architecture.md` + `decision-log.md` + `url-routing.md`.

## 2. Keep the 3 public-profile routes — reject the /@handle + role-SubNav consolidation
**Type:** Strategic
**Topics:** astro, architecture

**Trigger:** User asked to "critically explore" collapsing `/@[handle]`, `/creator/[handle]`, and `/teacher/[handle]` into a single `/@[handle]` with a role-driven SubNav (mirroring the existing `[...tab]` triad pattern).

**Options Considered:**
1. Collapse the three routes into one `/@[handle]` + dynamic role SubNav (pattern-aligned, fixes hub SEO, no net URL increase)
2. Keep the three separate routes and reframe them ← Chosen

**Decision:** Keep `/@[handle]` + `/creator/[handle]` + `/teacher/[handle]` as three separate routes. Do not consolidate.

**Rationale:** Despite the attractions (pattern-aligned, SEO fix, no net URL increase), the objections outweighed them: consolidation would flatten the creator/teacher storefronts into tabs, name-collide with the `/teaching` + `/creating` role workspaces, mix audiences in one tab set, require a dynamic SubNav with sub-tab canonical-SEO problems, and impose a multi-conv migration cost. The user concluded "your criticism far outweighs the attractions."

**Consequences:** No route shape change; the funnel stays separated. Instead of consolidation, the spokes were reframed (see Decision 3). Captured inside the SPOKE-COMMERCE entry in `docs/decisions/01-architecture.md`.

## 3. Reframe /creator + /teacher as commercial entry surfaces; storefront deferred to Phase 2 (SPOKE-COMMERCE)
**Type:** Strategic
**Topics:** astro, architecture, requirements

**Trigger:** Having rejected consolidation, the user asked for "a reframing to the 'selling' aspects of /teacher and /creator." Risk: this could be read as a mandate to build Phase-2 storefronts.

**Options Considered:**
1. Build branded storefronts now (revenue dashboard, merchandising, analytics, showcases)
2. Lock the identity-vs-commerce framing + light MVP commercial polish only ← Chosen

**Decision:** Reframe the public-profile family along an **identity-vs-commerce** axis: `/@[handle]` = identity (WHO a person is); `/creator/[handle]` + `/teacher/[handle]` = commercial entry surfaces (WHERE a visitor transacts — buy courses / book a session). Light MVP polish only: the creator page gained a "View Courses" header CTA (it had none, vs the teacher page's existing "Book a Session") and its sections reorder so courses lead (About→Courses→Philosophy). Full storefronts remain **deferred to Phase 2**.

**Rationale:** The "deep profile view" (RG-PUBPROF) framing read as redundant with `/@handle`. Identity-vs-commerce justifies the separate routes and is the durable mental model. Grounded in the client creator-profiles doc, which scopes MVP as "basic public profile using same system… with Creator badge" and defers revenue dashboard/analytics/storefront to Phase 2 — probing it first prevented scope-creep.

**Consequences:** Docstrings on CreatorProfile/TeacherProfile/`/creator`/`/teacher`/`@[handle]` reframed; `url-routing.md` spoke rows + framing sentence updated; creator CTA + courses-lead reorder shipped (teacher unchanged). 5 gates GREEN (6728 tests). Code `8afccc60`, docs `8acea4b`. Recorded in `docs/decisions/01-architecture.md` + `decision-log.md`.
