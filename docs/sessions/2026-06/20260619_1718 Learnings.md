# Conv Learnings - 2026-06-19

## 1. File-level conformance ☑ ≠ sub-component conformed
**Topics:** matt-migration, testing, route-sweep

**Context:** The Conv-305 CDETAIL-CONF marked the course-detail hub conformant by snapping the top-level component *files* (CourseHeader.tsx, CreatorTab.astro…). The Conv-306 browser-verify found off-scale spacing still rendering.

**Learning:** A component file can be conformant while the SHARED sub-components/primitives it composes (SubNav, entity anchors, Button, IconLabelChip, SocialPost, FormField, AnalyticCount) are not. DOM-truth browser-verify is what catches this — a source-only check on the named files misses it. The "stale ☑" pattern recurs (SubNav last touched Conv 244; ResourcesTab `gap-[15px]`; TeachersTab in Conv 305). When declaring a route swept, the shared children must be checked too, not just the route's own files.

## 2. The 4px-snap transform must guard sub-4px optical nudges
**Topics:** spacing, tooling

**Context:** The deterministic perl snap (px→nearest scale step) snapped a sanctioned `py-[2px]`→`py-4` in CommunityAnchor.

**Learning:** Sub-4px values (1/2/3px — `py-[2px]`/`mt-[2px]` optical nudges, Conv-299 sanctioned keeps) must be EXCLUDED from the snap. Add `if ($n < 4 || $n > 64) keep` to the transform. The first batch had no sub-4px so the bug didn't surface until the shared-primitive batch — always grep the diff for `*-[123px]` removals after a blanket snap.

## 3. zsh does not word-split unquoted parameter expansions
**Topics:** tooling, shell

**Context:** `perl -pi -e '…' $FILES` (space-joined string) failed with "Can't open <all-files-joined>" — the Bash tool runs zsh, which (unlike bash) does NOT split `$FILES` on IFS.

**Learning:** For multi-file commands in this environment, use a zsh array (`files=( … )` + `"${files[@]}"`), not an unquoted string variable. Symptom: the whole list treated as one filename.

**Pattern:**
```zsh
files=( a b c ); perl -pi -e '…' "${files[@]}"
```

## 4. Modal.tsx had a latent backdrop-click bug
**Topics:** astro, components, modal

**Context:** A SecuritySettings test (backdrop-click-closes) failed after adopting Modal for DeleteAccountModal.

**Learning:** `Modal.tsx` put its click handler on the OUTER container (`onClick` + `e.target===e.currentTarget` guard) but rendered a full-bleed `bg-black/50` overlay CHILD on top. Clicking the visible overlay set `e.target=overlay≠container`, so `closeOnBackdropClick` (default true) never actually fired — the documented behavior was dead. Fix: put the close handler on the overlay div itself. Benefits all 6 Modal consumers (they all pass `closeOnBackdropClick` implicitly true).

## 5. The status-colour ramps (warning/success/error) already exist
**Topics:** palette, color-tokens

**Context:** Earlier notes (and my own statements) claimed "no Matt warning/success ramp" — used to frame PALETTE status as a foundation decision.

**Learning:** `tokens-semantic.css`/`tokens-primitives.css` define `--success-100/300/500` (Matt greens), `--error-100/300/500` (Matt tint + tuned), and `--warning-100/300/500` (#FEF3E2/#F59E0B/#B45309, minted amber, structurally parallel), all exposed as `bg-/text-/border-{success,warning,error}-*` via the bridge. So raw `text-/bg-{green,red,amber,yellow}-*` STATUS banners are a mechanical token swap — NOT a missing-token decision. Caveat: ~1565 raw status-hue utilities exist app-wide but most are legitimate role-tints (member directory, badges), not status — only genuine status banners swap.
