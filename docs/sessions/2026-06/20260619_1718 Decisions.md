## 3. Stripe-purple → tokenize (mint `--stripe-*`), not recolor or leave raw
**Type:** Implementation
**Topics:** palette, color-tokens, stripe

**Trigger:** "No stragglers" — the Stripe Connect CTA used raw `purple-600` (a documented honest-orphan since Conv 219).

**Options Considered:**
1. Recolor to app-brand blue (loses the Stripe cue)
2. Leave raw purple (a straggler)
3. Mint `--stripe-100/300/500` semantic tokens at the existing purple values ← Chosen

**Decision:** Mint a 3-step `--stripe-*` ramp at the existing purple-100/600/700 values; swap StripeConnect's icon box/logo/CTA to the tokens.

**Consequences:** No raw status/brand colour stragglers remain on the 4 routes.

## 4. Button gains a CC-owned `danger` variant beyond Matt's Color-collection
**Type:** Strategic
**Topics:** components, palette, design-system

**Trigger:** Destructive red buttons were hand-rolled inline (`bg-error-300 text-white`) in SecuritySettings/DeleteAccountModal; Matt's `<Button>` variant enum has no danger.

**Options Considered:**
1. Leave destructive buttons hand-rolled (conformant in-place but not the primitive)
2. Add a `danger` variant to Button.tsx ← Chosen

**Decision:** Add `danger` (text-white, `bg-error-300 hover:bg-error-500`, `border-error-300`) — explicitly CC-owned, extending Button beyond Matt's Color-collection variant set.

**Rationale:** Dedups the inline destructive buttons onto the primitive using the existing error ramp; additive (no regression to existing variants).

**Consequences:** Destructive buttons swap to `<Button variant="danger">`; precedent set that CC may extend the Button variant model with documented, token-backed variants.

## 5. Adopt the SHARED Modal by conforming it (not adopting it legacy)
**Type:** Strategic
**Topics:** components, modal, astro

**Trigger:** DeleteAccountModal (conformant, bespoke) was the Modal-primitive adoption target, but Modal/ConfirmModal/FormModal are all legacy-styled (`dark:`/`secondary-*`/`text-lg`/`p-4`).

**Options Considered:**
1. Adopt the existing Modal as-is (regresses DeleteAccountModal to legacy chrome)
2. Leave DeleteAccountModal bespoke
3. Conform Modal.tsx first, then adopt ← Chosen

**Decision:** Conform Modal.tsx (neutral tokens, scale spacing, fix the backdrop bug), then wrap DeleteAccountModal in it (red title in children, Cancel→`outlined`, Delete→`danger`).

**Rationale:** Adopting a legacy primitive would regress a conformant consumer; conforming the shared primitive fixes it for all 6 consumers + realizes `closeOnBackdropClick`.

**Consequences:** Modal now conformant + backdrop-close works; DeleteAccountModal uses the shared primitive; the other 5 Modal consumers get the conformance + behavior fix for free (functional change — backdrop now closes as the default intended).

## 6. SettingsSection — new React Card/Section primitive with body presets
**Type:** Implementation
**Topics:** components, design-system

**Trigger:** Three near-identical local `Section` functions across the settings islands; the `.astro` Card can't be used in React islands.

**Decision:** Build `components/settings/SettingsSection.tsx` (header card + `body` preset prop: `padded`/`spaced`/`divided` + `danger`); replace the 3 local Section fns.
