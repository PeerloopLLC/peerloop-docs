# Conv Decisions - 2026-06-24

## 1. Family-C micro token: one token, snap 9→10 (not two)
**Type:** Implementation
**Topics:** matt-design, typography

**Trigger:** AvailabilityCalendar's six sub-12px spots came in two sizes (9px, 10px).

**Options Considered:**
1. Two tokens (preserve 9 & 10) — pixel-faithful, two regime members for a 1px delta
2. One token, snap 9→10 ← Chosen

**Decision:** Mint a single `text-display-micro` at 10px (+ `-medium`); the four 9px badges snap up.

**Rationale:** Consistency-first (WS-GLYPHS precedent); the 1px enlargement is imperceptible; fewer regime members. No lower tier exists to snap toward, so a single mint is unambiguous (type does not borrow the spacing-axis "ties round up" rule).

**Consequences:** §9.2c Family C = one member (promoted pending→minted+applied); recorded in the migration ledger.

## 2. AvailabilityCalendar status colour: map red→error, amber→warning (sky kept)
**Type:** Implementation
**Topics:** matt-design, colour-tokens

**Trigger:** Raw red/amber status hues vs the sparse semantic ramp; the MyFeeds-style verified-keep precedent created ambiguity about whether to keep all status hues raw.

**Options Considered:**
1. Keep raw (MyFeeds-style verified-keep)
2. Map to error/warning (sky kept) ← Chosen

**Decision:** Map red→error-*, amber→warning-*; keep sky-* (no info/available token home).

**Rationale:** The file already used error/success tokens for banners; `warning-500` == `amber-700` exact; `error-*` is the house carmine; the "keep only what has no token home" rule justifies both mapping red/amber and keeping sky.

**Consequences:** Calendar status now matches the house semantic palette; recorded in the ledger RG-WORKSPACES studio row.

## 3. Raw-form-of-token (font-weight / text-[Npx] on an existing token) = ENFORCE
**Type:** Strategic
**Topics:** matt-design, conformance, process

**Trigger:** Phase-6 found the class pervasive on already-confirmed groups; the deep-verify had flagged it "debatable."

**Options Considered:**
1. Accept weight-on-token as conformant (relax the rule)
2. Defer the ruling
3. Enforce (normalize) ← Chosen

**Decision:** Count raw font-weight + `text-[Npx]`-equal-to-token as violations; normalize them to the token's weight variant / size token.

**Rationale:** It is the ledger's own stated rule; most sweeps already converted (the inconsistency was the bug); visually zero-change because every body token is lh 1.0.

**Consequences:** [WEIGHT-NORM] #30 normalized 11 spots across 8 shared/ui primitives this conv; [SHARED-COLOUR] #31 spun off for incidental colour residuals; recorded in ledger Open-decision #4.
