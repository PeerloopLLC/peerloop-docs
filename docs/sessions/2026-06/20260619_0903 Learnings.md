# Conv Learnings - 2026-06-19

## 1. Verify the sibling's actual implementation before claiming "matches precedent"
**Topics:** matt-design, route-sweep, process

**Context:** The StripeConnect disposition claimed its `[Npx]`/collapse-only spacing approach "matched the NotificationSettings precedent." Reading NotificationSettings' actual source during the SecuritySettings assessment revealed it used **bare Matt numerics** (`px-6→px-24`) AND normalized off-set numerics — diverging on both axes.

**Learning:** A precedent claim must be backed by reading the sibling's committed source *this conv*, not by memory of its general approach. The mis-statement was caught one tab later and required a retro-fix of the just-shipped tab. Cheap to fix because both render identically (DOM-verified) — but the convention divergence would have compounded across the remaining tabs.

## 2. Spacing-convention: bare Matt numerics + off-set normalization
**Topics:** matt-design, tailwind, spacing

**Context:** The NAV-RETROFIT bridge `@theme --spacing-*` makes the collapse set {4,8,12,16,20,24,32,40,48,64} render at literal-px (`gap-4`=4px, not 16px). Two ways to express intended px: arbitrary `[16px]` (bridge-independent) or bare Matt numeric `gap-16` (= 16px via the override).

**Learning:** The /profile sweep standardized on **bare Matt numerics + off-set normalized** (NotificationSettings precedent): rewrite every numeric spacing util as its intended-px Matt numeric (`gap-6`/24px → `gap-24`, `gap-1`/4px → `gap-4`). Sub-scale values with no Matt equivalent (`py-0.5`=2px, `px-1.5`=6px) are kept. Both styles render identically; bare-numeric is the chosen convention for consistency.

**Pattern:**
Tailwind `-N` → px = N×4 (off-set) or N-px (override set). Rewrite as `-{px}` when px ∈ Matt scale; keep otherwise.

## 3. Red danger → error-*; multi-state status colours have no Matt token
**Topics:** matt-design, palette, color

**Context:** SecuritySettings/ProfileSettings danger zones use red; StripeConnect uses a yellow/green/red status triad + green checks; ProfileSettings has amber (warning) + green (success) banners.

**Learning:** Red **danger/error** maps cleanly to `error-{100/300/500}` (Conv-301 account-chrome precedent: red-600→error-300, red-700/900→error-500, red-50→error-100, red-200→error-300). But **success/warning/info** status colours (green/amber/yellow) have NO Matt token ramps, so they stay raw Tailwind — converting only one breaks the semantic set. This is the status-colour gap, folded into [PALETTE-FDN] #29. Discriminator: single-purpose danger → error-*; multi-state status set → leave.

## 4. Tier-2 ripeness = primitive exists AND a sibling already consumes it
**Topics:** matt-provenance, primitives, route-sweep

**Context:** Each settings island's loading/error states were hand-rolled. SkeletonCard + ErrorRetryCard primitives exist and StripeConnect already used them.

**Learning:** A Tier-2 candidate is "ripe" to extract during a sweep when the target primitive already exists AND a sibling consumes it — then it's a verified drop-in. New-primitive candidates (Switch, Button danger/brand variants, Modal, Textarea) are deferred until the full consumer set is known. Completing all 6 /profile tabs crystallized that consumer set → [PROFILE-PRIM-SWEEP] is now actionable.

## 5. Conformance restyles break class-asserting tests
**Topics:** testing, route-sweep

**Context:** SecuritySettings.test asserted `border-red-200` on the Danger Zone; the conformant change to `border-error-300` failed it (waitFor timeout).

**Learning:** Tests that assert exact Tailwind classes must be updated when a conformance restyle changes the token. Update the assertion to the new token (intent — "danger styling" — preserved), not the component. ProfileSettings/StripeConnect tests had no class assertions and passed unchanged.
