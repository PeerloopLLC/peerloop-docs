# Conv Learnings - 2026-06-11

## 1. Green unit tests can't see which island a live route mounts
**Topics:** testing, matt-migration, feeds

**Context:** Step 3's endpoint test + the component edit were both green, but the Promote button never appeared on the live course feed.

**Learning:** The course feed was migrated to `MattCourseFeed` → `SocialPost` (POST-MATT, Conv 260); the legacy `CourseFeed.tsx`/`FeedActivityCard` edited is no longer the live course renderer. Unit tests exercise the endpoint and a component in isolation — neither knows the route's actual mount. Only driving the real route (DOM-truth browser check) revealed the gap. Same class as `feedback_read_legacy_source_before_conclusion`, applied to "which island is live." Before wiring UI into a component, confirm the live page's astro mount (`grep` the `[...tab].astro`).

**Pattern:**
When a feature touches a page that has a Matt redesign, check `src/pages/**/[...tab].astro` for the `Matt*` component it actually mounts, not the legacy sibling.

## 2. Model ① turns "promote twice" into a one-line constraint
**Topics:** promote-pipeline, schema, idempotency

**Context:** `post_promotions` had no uniqueness; re-clicking Promote would write a duplicate row that surfaces the post twice in the lane.

**Learning:** Because delivery Model ① stores a promotion as one D1 row (no Stream copy), idempotency is a pure-data concern: a `UNIQUE(source_activity_id, to_feed_type, to_feed_id)` index + a pre-insert SELECT early-return. Had promotion used the copy-model ③, a double-click would mint a second Stream activity with its own engagement counter — un-dedupable after the fact.

## 3. A passive `actions` slot keeps a display-only primitive pure
**Topics:** matt-primitives, component-design

**Context:** Needed an interactive Promote button inside the Matt `SocialPost`, which is intentionally display-only (non-interactive reaction pills).

**Learning:** Added an optional `actions?: ReactNode` slot that `SocialPost` merely renders in its footer — the interactivity lives in the child (`PromoteButton`) the parent passes. The primitive's "no interactivity" contract holds; existing callers (actions undefined) render byte-identically.
