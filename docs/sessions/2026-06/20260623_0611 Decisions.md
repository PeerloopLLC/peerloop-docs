## 1. Deep-verify the 3 ledger-missing groups rather than trust README prose or backfill blindly
**Type:** Strategic
**Topics:** matt-design, cc-workflow, docs-infra

**Trigger:** The RTMIG-4 reconciliation found 3 of the 11 "done" groups (RG-COMMS, RG-PUBPROF `/creator`, RG-WORKSPACES `/teaching*`+`/creating*`) marked Swept in the README but absent from the conformance ledger (the oracle). The README and ledger had silently diverged since ~Conv 317.

**Options Considered:**
1. Trust the README's "Swept" claims and backfill the ledger to match (cheap, but assumes prose SoT is correct)
2. Deep-verify each group's component tree against the 3-axis gate first, then decide ← Chosen
3. Treat all 3 as unswept and re-sweep from scratch (most expensive, ignores prior work)

**Decision:** Dispatched 4 parallel general-purpose agents to walk each group's route component trees and grep the 3-axis gate (Type/Spacing/Colour) with sanctioned-exception judgment, *before* any backfill.

**Rationale:** A SoT claim is a hypothesis until verified this conv (Baseline Verification discipline). Backfilling the ledger to match unverified README prose would have propagated any false "done" claim into the oracle, defeating its purpose.

**Consequences:** All 3 groups were found to have genuine forbidden-token residuals → NOT cleanly Swept. Findings preserved to `.scratch/2026-06-23-rtmig4-reconciliation-deepverify.md`; tracked as new task [RTMIG-RECON] #27.

## 3. Horizontal (shared-primitives-first) cleanup ordering over vertical group-by-group
**Type:** Strategic
**Topics:** matt-design, cc-workflow

**Trigger:** With all 3 groups confirmed to have residuals (and the confirmed-8 riding the same shared primitives), the user asked how to order the cleanup.

**Options Considered:**
1. Vertical — fix each group's full stack group-by-group (clean per-group done-tests, but re-touches shared primitives N times)
2. Horizontal — fix shared primitives first (root cause), then route-owned residuals per group ← Chosen

**Decision:** Horizontal ordering: Phase 1 functional bugs → Phase 2 interpretation ruling → Phase 3 shared primitives (UserAvatar, form/*, Modal, PromoteButton, charts) → Phase 4 route-owned per group → Phase 5 re-verify + ledger backfill + README re-sync → Phase 6 re-check the confirmed-8.

**Rationale:** Shared primitives are the highest-leverage root cause — conforming them once de-risks both the 3 confirmed-failing groups and the confirmed-8 that consume the same primitives, per the ledger granularity rule. Multi-conv scope was flagged.

**Consequences:** Phase 1 executed this conv (functional bugs + missed status-token map). Resume pointer is Phase 3 (shared primitives) in [RTMIG-RECON] #27.
