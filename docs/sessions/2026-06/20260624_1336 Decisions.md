## 4. [ADMIN-CONF-POLICY] — admin restyle policy
**Type:** Strategic
**Topics:** admin, conformance, design-system, astro

**Trigger:** RG-ADMIN is ~1470 legacy hits across 35 files (multi-conv); admin is a different UI archetype.

**Options Considered:**
1. Full Matt content-surface conformance (no relaxations)
2. Dense-operational-console with relaxations + a distinct admin identity ← Chosen

**Decision:** Admin = dense operational console. Relaxations A (density bias) / B (neutral-led, minimal brand) / C (lightweight inline controls in tables) / D (flat data containers). Type: 12px base (`text-body-small`) + 10px dense/meta (`text-display-micro`) + headings down one step. Visual identity: dark `neutral-900` sidebar, light-on-dark nav, "Admin" wordmark, `info`-blue accent (not brand-purple), role chip, shared page-header. Strict: drop all `dark:`, semantic-status colour, on-scale spacing.

**Rationale:** Admin is for ≤2 high-trust operators who want density/legibility; forcing the content aesthetic hurts usability. The 12-vs-14 type shift + dark sidebar give a deliberate "Admin" identity as operators cross from the light user app. Builds on RG-MOD precedent.

**Consequences:** Recorded to memory (`project_admin_conformance_policy.md`) + README RG-ADMIN row + task #3. Multi-conv sweep deferred to a future conv (start shell + AdminDashboard). No admin editing this conv (policy-setting only).
