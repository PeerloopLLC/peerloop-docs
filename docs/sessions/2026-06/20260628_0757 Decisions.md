# Conv Decisions - 2026-06-28

## 1. UserCard gets a generic `extraActions` slot, not follow-domain coupling
**Type:** Implementation
**Topics:** components, composition

**Trigger:** Gap A's Follow button needed to sit in the UserCard action row beside Message, but UserCard is shared (also renders in admin/member lists).

**Options Considered:**
1. Add `canFollow` + follow props directly to UserCard (couples shared component to follow domain)
2. Add a generic `extraActions?: ReactNode` slot; parent owns follow logic ← Chosen
3. Render the Follow button in PublicProfile, visually detached from the action row

**Decision:** Added `extraActions?: ReactNode` to UserCard; PublicProfile passes `<UserFollowButton>` only when `!isOwnProfile`.

**Rationale:** Keeps UserCard presentational (no follow coupling), surfaces Follow right beside Message, reusable for any future action. Composition over a domain-specific prop.

**Consequences:** UserCard stays dumb; follow logic lives in the self-contained UserFollowButton + PublicProfile.

## 2. Follow initial state via a new GET status endpoint (mirror can-message)
**Type:** Implementation
**Topics:** api

**Trigger:** Neither the user nor creator profile API returned `isFollowing`.

**Options Considered:**
1. Add `isFollowing` to `/api/users/[handle]` response (server-side join)
2. New `GET /api/users/[handle]/follow` → `{ isFollowing }`, mirroring `can-message` ← Chosen
3. Client computes nothing / always starts "not following"

**Decision:** Added the GET handler (visitors/self → `{ isFollowing: false }`, no 401), and a self-seeding UserFollowButton.

**Rationale:** Mirrors the established `can-message` status pattern; keeps the button self-contained; made the feature self-verifying (see Learning 1).

**Consequences:** Follow is a clean POST/GET/DELETE triad on one route.
