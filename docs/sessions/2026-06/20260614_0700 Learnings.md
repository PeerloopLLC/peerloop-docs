# Conv Learnings - 2026-06-14

## 1. Per-post promote ≠ entity promote — same word, different model
**Topics:** promotion, feeds

**Context:** The [U3D-POST] deferral note framed it as "entity-promo model + engagement floor," implying reuse of the entity PromoteNudge machinery.

**Learning:** Reading the code revealed it's the *per-post* model — `PromoteButton` + Stream `reaction_counts` — entirely distinct from the *entity* model (`PromoteNudge` + `promotable-entities` + student/member counts). They share only the verb "promote." This drove the decision to add a separate `promo_post_min_engagement` dial, because reusing `promo_nudge_min_engagement` would conflate "3 members" with "3 reactions" (same integer, different unit).

---

## 2. Centralize a system-level visual state inside the shared leaf component
**Topics:** components, design-system

**Context:** The hot Promote state had to look identical across two render paths that pass very different className strings (legacy slate pill vs Matt `text-[12px]` link).

**Learning:** Putting the `hot` styling *inside* `PromoteButton` (caller passes one boolean) means the highlight can never drift between surfaces. Cost: one subtle layout fix — `ml-auto` had to move from the button class to a wrapper span in `FeedActivityCard`, because hot mode discards the caller's className entirely.

---

## 3. Flip the config dial, not the data, to exercise a threshold in-browser
**Topics:** testing, browser-verify

**Context:** Per-post engagement lives in Stream (hard to seed reactions); the floor is a D1 dial.

**Learning:** Bracketing a fixed 0-engagement post with floor=0 then floor=5 exercises both hot/quiet branches in the browser without seeding Stream reactions — and as a bonus verifies the admin endpoint round-trip + the `loadPromotionConfig` default fallback. The threshold arithmetic itself is covered by `isPromoteHot` unit tests; the browser only needs to confirm the visual renders per branch.
