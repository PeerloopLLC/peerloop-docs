**Topics:** routing-addressability, matt-design-system

**Trigger:** Resuming the Conv 231 parked route-shape decision after the addressability audit.

**Options Considered:**
1. Separate addressable route ← Chosen

## 2. Hybrid — one component, two shells (user refinement of A)
**Type:** Strategic
**Topics:** matt-design-system, astro, routing

**Trigger:** User wanted a low-commitment browse surface in addition to the standalone page.

**Options Considered:**
1. Standalone page only
2. SubNav tab only

## 3. Both CTAs → Stripe (parallel chromes, not a two-step funnel)
**Type:** Implementation
**Topics:** stripe, matt-design-system

**Trigger:** Where each "Get Started Now" routes.

**Options Considered:**
1. Both → Stripe ← Chosen
2. Benefits → /precheckout → Stripe (two-step)
3. Benefits → /precheckout with a trimmed variant

**Decision:** Both surfaces' CTAs go straight to Stripe; the two routes are parallel entry chromes for the same content+action.

**Rationale:** Truest to "present both and pick"; keeps the single-component design intact; the no-commitment job is carried by being a tab at all.

**Consequences:** `/precheckout` and `/benefits` are functionally parallel; either can be removed without affecting the other's checkout path.

**Type:** Implementation
**Topics:** matt-design-system, stripe, testing
