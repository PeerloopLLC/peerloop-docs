# Conv Learnings - 2026-06-23

## 1. Sparse token ramps break mechanical 1:1 color-step swaps
**Topics:** matt-design, astro, cc-workflow

**Context:** CourseFeed's Conv-325 conform did `slate-400→neutral-400` / `slate-600→neutral-600`. The Matt neutral ramp is **sparse** (50/100/300/500/700/900 only), so `neutral-400`/`neutral-600` generate no CSS — the classes are inert and the elements render with no/inherited colour.

**Learning:** A mechanical `tailwind-N → token-N` swap silently breaks whenever N isn't in the sparse custom ramp. The result is invisible (absence of styling), so a per-state DOM glance often misses it. Map to the nearest *defined* step preserving light/dark hierarchy: `slate-400`(muted)→`neutral-500`, `slate-600`(darker)→`neutral-700`.

**Pattern:** A lint that rejects undefined token steps (the planned [SWEEP-SPACING-GREP] discipline, extended to colour) would catch this class at conform time.

## 2. Per-route DOM-verify structurally can't see shared-primitive or unrendered-branch residuals
**Topics:** matt-design, testing, cc-workflow

**Context:** The README claimed "0 forbidden tokens" (DOM-verified) for groups that a code-level grep then found to have many. The DOM checks ran on specific viewers/states (usually member view).

**Learning:** Visual per-route verification cannot see: (a) shared primitives the route mounts but the sweep didn't treat as "owned" (UserAvatar, form/Input, Modal, PromoteButton), (b) un-rendered branches (admin-only panels, error/empty states, specific tabs), (c) gate-interpretation cases (raw font-weight). A code-level forbidden-token cross-check complements DOM verification — neither alone is sufficient.

## 3. Two sources of truth silently diverged
**Topics:** matt-design, docs-infra, cc-workflow

**Context:** RTMIG-4 has two SoTs — the route-migration README (prose, DOM-verified per-state) and the conformance ledger (component-level oracle). The ledger stopped being updated ~Conv 317 once conformance started being recorded inline in README prose during the RG-WORKSPACES decomposition.

**Learning:** When a process has a prose SoT and a structured-oracle SoT, work recorded only in prose makes the oracle under-report — anyone trusting the oracle sees routes as unswept. The README's "Swept = client-showable" drifted ahead of "actually conformant." Re-merging them (ledger backfill + README correction) is Phase 5 of the cleanup.
