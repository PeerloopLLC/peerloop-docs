# Conv Learnings - 2026-06-07

## 1. `#` task ids are a per-conv slot index, not a backlog count
**Topics:** cc-workflow, docs-infra

**Context:** User saw Conv 245 reach id #29 but only 24 tasks carried into Conv 247 and asked how that reconciles.

**Learning:** TodoWrite ids increment and are never reused within a conv, so completed/dropped tasks leave numbering gaps. The pending count is always ≤ the highest id. `/r-start` renumbers the carried set into a fresh contiguous 1–N. Max-id ≠ count; the difference is mid-conv completions (Conv 245: 5 completions → 29 max, 24 pending, e.g. #28→#1, #29→#24).

## 2. The `*DONE*` live-sync marker is grep-safe by asterisk arity
**Topics:** cc-workflow

**Context:** [DONEGREP] asked whether /r-start's no-shrink reconciliation mis-handles `*DONE*`-prefixed rows.

**Learning:** The completion marker is single-asterisk (`*DONE*`) and lives in the meaning cell; codes are double-asterisk (`**CODE**`) in a separate cell. `\*\*…\*\*` never matches `*DONE*`, and rows key off the leading `| N |` cell which the marker doesn't touch. So done rows still count and still yield their code. The reconciliation is prose-driven (no enforced grep), so the safe patterns were pinned into `SKILL.md:368` to prevent a future single-asterisk regression.

**Pattern:**
- Row-count grep: `^\| *[0-9]+ ` (still counts `*DONE*` rows)
- Code-extract grep: `\*\*[A-Z0-9-]+\*\*` (never captures single-asterisk `*DONE*`)

## 3. Browser DOM reads must be in a separate tool call from the React-triggering click
**Topics:** testing, cc-workflow

**Context:** First mark-all-read test read `innerText` synchronously right after `.click()` and saw stale "2 unread".

**Learning:** React batches state updates and re-renders asynchronously; reading the DOM in the same JS execution as the click captures the pre-render state. Split click and assertion into separate `javascript_tool` calls so the render flushes between them. Also: the page REPL wrapper rejects top-level `await` — use promise-then stashing to `window` + a follow-up read.

## 4. Page-mounting trace beats provenance markers for legacy-vs-Matt component classification
**Topics:** cc-workflow, astro

**Context:** SELECT-AUDIT — guessed discover/courses components were Matt; provenance markers were all NONE (sub-page components don't carry page markers).

**Learning:** To classify whether a `.tsx` component is on a new-Matt vs legacy surface, trace which `.astro` page mounts it (`grep -rl Component src/pages`), not its own provenance comment. discover/MemberDirectory→old/discover/members, courses/MyCourses→old/courses revealed they're legacy despite "modern" dir names — and a separate root `members/MembersDirectory` already uses the primitive.
