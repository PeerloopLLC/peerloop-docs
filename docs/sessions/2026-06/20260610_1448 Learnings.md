# Conv Learnings - 2026-06-10

## 1. The cron Worker pattern already exists — Astro-SSR can't host scheduled()
**Topics:** cloudflare, deployment, infra

**Context:** DISCOVERY-RAILS needs a daily scheduled job. Assumed it might be net-new infra.

**Learning:** `workers/cron/` is a standalone Worker (separate from the main Astro Worker) with a `scheduled()` handler + cron triggers (`workers/cron/wrangler.toml`, prod 30min / staging 15min), currently running `runSessionCleanup`. The Astro `@astrojs/cloudflare` v13 adapter does NOT expose `workerEntryPoint`, so the main Worker can't cleanly add a `scheduled()` export — this standalone Worker IS the established pattern. New scheduled jobs extend it; they share D1/KV by referencing the same ids.

**Pattern:**
Cron worker imports lib via relative `../../../src/lib/...` and resolves `@lib` aliases through the root tsconfig (esbuild). `wrangler deploy --dry-run --config workers/cron/wrangler.toml` validates bundling without account access.

## 2. KV is effectively unused by app code (SESSION is adapter-managed)
**Topics:** cloudflare, infra

**Context:** Needed a KV namespace for the precomputed blob.

**Learning:** Only the `SESSION` KV namespace exists, and it's auto-managed by the Astro adapter; app code uses JWT cookies, not KV. So a new namespace (`DISCOVERY_CACHE`) had to be created via `wrangler kv namespace create` (a remote account op) and bound in BOTH the main `wrangler.toml` (reader: top-level + production + staging) and `workers/cron/wrangler.toml` (writer: production + staging) — same id per env so writer+reader share storage.

## 3. CLOUDFLARE_ENV selects the wrangler env without --env
**Topics:** deployment, cloudflare

**Context:** `deploy:staging` script is `CLOUDFLARE_ENV=staging … wrangler deploy` (no `--env`).

**Learning:** Wrangler honors the `CLOUDFLARE_ENV` env var to pick the `[env.X]` block, equivalent to `--env X`. So bare `wrangler deploy` under `CLOUDFLARE_ENV=staging` targets `peerloop-staging`, not the top-level prod worker. Verified by the deploy output (`Deployed peerloop-staging`).
