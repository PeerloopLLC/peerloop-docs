# Conv Learnings - 2026-06-26

## 1. File-aware grep: `-h` silently defeats a `/old` path filter
**Topics:** tooling, grep, sweep-verification

**Context:** Post-sweep verification used `grep -rhoE '…' | grep -v '/old/'` and reported residual on-scale spacing values (16×4, 32×2) that should already have been tokenized — alarming, looked like the sweep had missed sites.

**Learning:** `grep -h` strips the filename from output, so a downstream `grep -v '/old/'` has nothing to match — the `/old/` exclusion is INERT and the residual "hits" were entirely `/old` files. The authoritative file-aware grep (no `-h`, `-v '/old/'` operating on `path:line:content`) showed 0 residual. When excluding paths, never combine `-h`/`-o` value-extraction with a path-based `grep -v` in the same pipe. Extract values only AFTER the path filter has run on full `path:line` lines.

**Pattern:**
```bash
grep -rnE 'PAT' src | grep -v '/old/' | grep -oE 'PAT' | sed …   # filter paths first, extract second
```

## 2. A green-baseline claim carried forward is a hypothesis, not a fact (Rule 2, live)
**Topics:** testing, baseline-verification, regression-hiding

**Context:** The full-suite run during the spacing sweep surfaced 5 failures. RESUME-STATE Conv 336 claimed "all gates green throughout."

**Learning:** The 5 had been red since the RG-ADMIN conformance sweep (Convs 332–336) — admin styling was conformed but the className-asserting tests weren't updated, and the suite evidently wasn't run end-to-end in those convs. This is exactly the CLAUDE.md §Baseline Verification Rule-2 failure ("carry-forward claims hide regressions"): the gap existed for ~4 convs. `git stash` is the clean proof-of-pre-existence — stash the working change, run the failing tests on the clean tree; if they still fail, the change didn't cause them. Restoring green (6737/6737) makes the gate trustworthy again. Habit reinforcement: run the FULL suite (not just changed-file tests) at least once when a conformance sweep changes many components' classes, because className-asserting tests in OTHER files break silently.

**Pattern:**
```bash
git stash push   # set working change aside
# run just the suspect tests on the clean tree
git stash pop    # identical failures = pre-existing, not yours
```

## 3. "Deferred" ledger notes go stale — verify current source before assuming work remains
**Topics:** conformance, ledger-hygiene, scoping

**Context:** PALETTE-FDN #29 was picked expecting bounded work (conform the deferred profile status banners). Grepping the profile tabs found them ALREADY conformed to warning/success tokens.

**Learning:** The Conv-303 deferral ("amber/green banners LEFT → PALETTE-FDN #29") was completed in a later conv when warning/success tokens were minted, but the ledger note was never updated — so the task looked open. Before executing a "deferred" item, grep the actual source for current state; the deferral note may be stale. Re-scoping #29 app-wide also revealed it's not a bounded task (105 sites, route-entangled, mostly intentional keeps like medals/stars) — the bounded-looking task was a mirage on both ends.
