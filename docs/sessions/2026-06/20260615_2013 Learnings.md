# Conv Learnings - 2026-06-15

## 1. Investigate the existing shell before building a new one
**Topics:** layout, astro, refactor

**Context:** The LAYOUT-SG spec said to build a new `ContentShell` sibling of `ListingShell`.

**Learning:** Reading `AppLayout.astro` first revealed it ALREADY hosted the white card, grey bg, mobile pills, and a left `sub-nav` slot (from NAV-RETROFIT). The only missing piece was the `max-w-[1248px]` cap. Building ContentShell would have been redundant indirection. The "cap AppLayout" path fixed the jar app-wide with one container.

---

## 2. Measure the right DOM node — wrapper vs inner pill
**Topics:** dom-truth, browser-verification

**Context:** An early `getBoundingClientRect` on the bottom nav returned w=805 (full width), read as "ControlBar is full-width, deviates from Matt's centered pill."

**Learning:** That was the *fixed positioning wrapper* (`left-0 right-0`); the actual pill inside is `rounded-full ... justify-center`, content-sized and centered. Always measure the styled element, not its layout container. Reading the component source disambiguated it.

---

## 3. Verify doc claims against the design source, not the doc
**Topics:** figma, layout, docs

**Context:** Conv-288 `08-layout` §8.3.1 claimed page bg is grey `#f8fafc` "at every breakpoint."

**Learning:** Screenshotting Matt's populated mobile About frame (`1113:13756`) showed mobile content sits on WHITE — grey is the desktop shell *behind the card*. The doc claim (an extraction-time assumption) was wrong; current `bg-white lg:bg-[#f8fafc]` was already correct. Verifying against the frame prevented an erroneous app-wide bg change.

---

## 4. @matt-source markers in code give direct Figma node IDs to probe
**Topics:** figma, provenance

**Context:** Figma Remote MCP is link-based (user-as-navigator); enumerating the file isn't supported.

**Learning:** But `get_metadata`/`get_screenshot` are selection-free with explicit node IDs, and the `@matt-source <nodeId>` doc-comments in `.astro`/`.tsx` files are a ready index of node IDs. Grepping them (`480:10833` for course-detail) let me probe the desktop hero↔SubNav relationship directly without asking the user to paste links.
