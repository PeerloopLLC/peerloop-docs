## 1. Playwright E2E Deprioritized Pre-Launch; Browser Regression = PLATO API + Matt Sweep + Manual
**Type:** Strategic
**Topics:** testing

**Trigger:** User asked what value the Playwright tests provide, how they guard against shipping poor code/UI, and how much they overlap PLATO — then floated moving to PLATO browser mode "as it is customized to this app."

**Options Considered:**
1. Lean on PLATO API (gated) + Matt design sweep + manual review; freeze Playwright ← Chosen
2. Scope automating PLATO browser mode into a deterministic CI gate (scoping surfaced a decisive blocker — the `BrowserIntent` DSL is intentionally human-prose, so this reinvents Playwright)
3. Migrate + gate + maintain the 28 stale Playwright specs (multi-conv, unrealized value until green)

**Decision:** `[E2E-MIG]` **dropped**; the 28 Playwright specs (164 cases) are **frozen, not deleted** (kept as a journey reference). Pre-launch browser-regression coverage = **PLATO API mode** (gated/green in `npm test`) + **Matt design-conformance sweep** + **manual review**. An LLM-driven PLATO browser smoke-walker is a **post-launch** idea (`[BROWSER-SMOKE-2B]`, a periodic smoke-walk, NOT a deterministic gate). The PLATO half of `[E2E-GATE]` was **completed this conv** (3 static `Instance:` blocks added so their file-level `verify` runs in `npm test`; PLATO 10→13).

**Rationale:** Pre-launch (small team, 2 trusted users, fixed timeline/budget), backend correctness — already covered by the gated PLATO API suite — is the higher-value automated coverage. Playwright adds only *automated browser* regression whose value is unrealized until the stale specs are migrated AND gated AND kept green. Both automation paths for PLATO browser mode are poor gates (custom runner = Playwright-with-extra-steps; LLM runner = non-deterministic). Dropping Playwright does NOT lose visual/UX coverage — it never asserted that (Matt sweep + manual do).

**Consequences:** `e2e/README.md` (NEW) + `docs/reference/TEST-E2E.md` carry a FROZEN/not-a-gate banner; `npm run test:e2e` still runs but is expected-red and not a gate; `[BROWSER-SMOKE-2B]` deferred (#16). Revisit post-launch.

**Routing:** Already written to `docs/decisions/06-testing-ci.md` this conv (chunk entry). This agent completed the routing by appending the entry to `docs/decisions/decision-log.md` and adding the title to `docs/decisions/INDEX.md`.

## 2. Creator Homework-Grading Parity = a "Submissions" Tab in the Existing CourseEditor
**Type:** Implementation
**Topics:** testing, auth

**Trigger:** HW-GRADE-CREATOR — give the course creator the same homework-grading surface as teachers. RESUME-STATE assumed no per-course creator view existed and a new one was needed.

**Options Considered:**
1. New dedicated creator grading route/view
2. Add a "Submissions" tab to the existing `CourseEditor` (reached via `/creating/studio?course=<id>`), mounting the shared `HomeworkGradingPanel` ← Chosen

**Decision:** Add a **"Submissions" tab** to `CourseEditor` (after the authoring "Homework" tab) that mounts the shared self-contained `HomeworkGradingPanel` (one `courseId` prop), via cross-domain import — same reuse shape as the existing `CourseFeed`. `?tab=submissions` deep-links it. No backend changes (the list/submission/grade endpoints already authorize the creator).

**Rationale:** The creator's per-course surface already existed, so parity is a tab mounting a shared panel — not a new view. Follows the established cross-domain-import reuse pattern (CourseFeed), so it is not a novel architectural decision. 5 baseline gates GREEN (test 6723); DOM-verified live on `:4321` as creator Guy Rymberg (both creator-gated endpoints confirmed, read-only).

**Consequences:** `CourseEditor.tsx` gains the import + TabId union + validTabs + tab def + content block (code `21bd5e4b`).

## 3. SYNCGAP-FIX = Array + Membership Helper (Not a `[[ =~ ]]` Regex)
**Type:** Implementation
**Topics:** cc-workflow, testing

**Trigger:** Root-cause fix for `sync-gaps.sh`'s broken shared-basename guard (expanded `a|b|c` pipes never alternated in a `case` pattern — see Learnings #1).

**Options Considered:**
1. Rewrite the guard as a `[[ "$n" =~ ... ]]` regex match
2. `SHARED_BASENAMES` array + `is_shared_basename()` membership-loop helper ← Chosen

**Decision:** Replace the broken `case`+expanded-pipe guard with a `SHARED_BASENAMES` **array** + an `is_shared_basename()` membership helper, plus a `SYNC_GAPS_LIB_ONLY` sourcing hook so a calibration harness can exercise the helper directly.

**Rationale:** An exact-membership list is what the guard semantically wants; a regex re-introduces quoting/escaping fragility (the class of bug just fixed). The sourcing hook enables before/after calibration on the canonical case (per `feedback_heuristic_calibration`), and a permanent behavioral regression test was added (`test-drift-detection.sh` Test 7).

**Consequences:** `.claude/scripts/sync-gaps.sh` + `test-drift-detection.sh` updated; stale `SHARED_BASENAMES_PATTERN` comment fixed (docs `84cb6d3`).
