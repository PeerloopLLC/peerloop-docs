# Conv Decisions - 2026-06-24

## 1. UserEditModal adopts ui/Modal shell
**Type:** Implementation
**Topics:** matt-design, astro

**Trigger:** UserEditModal was a hand-rolled modal (fixed-inset backdrop + custom toggle-switch), diverging from locked sub-pattern (c) "admin modals → ui/Modal".

**Options Considered:**
1. Adopt ui/Modal shell, keep the custom toggle body ← Chosen
2. Token-conform the hand-rolled structure in place

**Decision:** Wrap in the conformed `ui/Modal` (drops bespoke backdrop/header), keep the custom toggle-switch body with conformed tokens.

**Rationale:** Matches sub-pattern (c) + the TopicsAdmin (route #4) precedent; FormModal doesn't fit (custom field logic).

**Consequences:** UserEditModal now renders inside ui/Modal (r12, backdrop, focus-trap, Escape); toggle indigo→info-500 + bridge-shrink fixed.

## 2. Adopt UserAvatar for admin avatar fallbacks + fix the primitive app-wide
**Type:** Strategic
**Topics:** matt-design, astro

**Trigger:** Inline indigo avatar fallbacks in UsersAdmin/UserDetailContent (and later EnrollmentsAdmin); then DOM-verify exposed UserAvatar itself was bridge-shrunk.

**Options Considered:**
1. Fix the UserAvatar primitive app-wide (resizes ~15 consumers, re-verify under XCUT-BACKREF) ← Chosen
2. Leave the primitive; conform admin avatars inline; log the bridge-fix separately
3. Surgical: bridge-safe sizes but pin existing consumers to today's px

**Decision:** Adopt UserAvatar in admin **and** fix its sizeClasses app-wide (durable).

**Rationale:** UserAvatar was genuinely bug-broken (latent since Conv 174); the user's adoption choice is only viable under the app-wide fix; 11 of the usages are `xs` (unchanged) and the rest were rendering absurdly tiny.

**Consequences:** All consumers now render intended avatar sizes; ~15 swept-page consumers need a layout re-verify, scoped into [XCUT-BACKREF]. `/creator/[handle]` xl spot-checked (96px, no overflow).

## 3. Force-Complete / positive admin actions map to Button `primary`
**Type:** Implementation
**Topics:** matt-design

**Trigger:** EnrollmentsAdmin's green "Force Complete" footer button — no `success` Button variant exists.

**Decision:** Positive primary admin actions → `Button variant="primary"` (americana-blue); graded destructive → `danger`/`warning`; neutral → `default`. (Full Refund→danger, Partial Refund→warning, Cancel→default.)

**Rationale:** Consistent with PayoutsAdmin (Process→primary) and the locked sub-pattern (a); the Button color-collection has no green/success variant.

**Consequences:** Green/orange/gray footer buttons normalize to the americana-blue + semantic-variant palette.
