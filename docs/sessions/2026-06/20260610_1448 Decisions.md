## 2. Trending = trailing-window join/enrollment velocity (not prior-window delta)
**Type:** Implementation
**Topics:** discovery-rails

**Trigger:** Spec says trending = "velocity (rolling-7d delta…, not absolute)".

**Options Considered:**
1. True prior-window delta (this7d − prev7d)
2. Trailing-window count of enrollments/joins ← Chosen

**Decision:** Count enrollments (`enrolled_at`) / community joins (`joined_at`) within the trailing window as the velocity signal; exclude zero-velocity.

**Rationale:** A true delta is noisier at Genesis scale (~60-80 users); trailing-window count is robust and still "a rate, not absolute" (the contrast vs popular). Documented as a future tuning refinement.

**Consequences:** `platform_stats` `discovery_trending_window_days` (default 7) tunes it.

## 3. Endpoint serving = KV-read with on-demand compute fallback (one endpoint)
**Type:** Implementation
**Topics:** discovery-rails, cloudflare

**Trigger:** Endpoint must work in dev before the KV namespace + cron exist.

**Decision:** `GET /api/discovery/rails` reads KV when the binding is present + blob version matches, else computes on demand from D1. KV access is a type-safe optional probe (`cfEnv.DISCOVERY_CACHE` cast through Record) that lights up when the binding is declared — no endpoint edit needed at Phase 3.

**Rationale:** Keeps the feature functional immediately; the precompute path is a transparent optimization. Version guard self-invalidates a stale blob on a version bump.

**Consequences:** Verified both paths live on staging (compute → kv after cron tick).
