# Conv Learnings - 2026-06-12

## 1. Crashed-conv recovery: restore RESUME-STATE.md rather than rely on the conv-tasks.md crash-survivor path
**Topics:** cc-workflow, r-start

**Context:** Conv 275 incremented the counter + deleted RESUME-STATE.md (its Step 7.6) but never ran /r-end, leaving the docs repo dirty with a staged `D RESUME-STATE.md` and `.conv-current`=275.

**Learning:** When a crashed conv's `/r-start` already deleted RESUME-STATE.md but the deletion is uncommitted, `git restore RESUME-STATE.md` recovers the full richer Conv-N end-state (Key Context + all 37 tasks) from HEAD — strictly better than committing the deletion and falling back to the leaner `.scratch/conv-tasks.md` crash-survivor restore. Both sources survived; HEAD's RESUME-STATE.md is the higher-fidelity one.

**Pattern:** dirty `D RESUME-STATE.md` + stale `.conv-current` + last commit `Conv N start` ⇒ crashed-mid-conv ⇒ `git restore` then run /r-start normally.

## 2. Stream-held post bodies shape what a D1 moderation list can show
**Topics:** stream, feeds, promotion

**Context:** Building the System-promotion moderation list; `feed_activities` stores only metadata (actor, type, stream id) — the post body lives in Stream under delivery model ①.

**Learning:** A pure-D1 query gives promoter/author/origin/when via JOINs, but a real content preview requires a per-row Stream fetch. The `promoted.ts` enrichment pattern (`getStreamClient` + `getActivitiesByIds` + best-effort try/catch, metadata-on-failure) is directly reusable, so a content preview is cheap to add without coupling the lib query to Stream.

**Pattern:** lib returns D1 rows (unit-testable); endpoint layers best-effort Stream content on top.
