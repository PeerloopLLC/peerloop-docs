# Conv Learnings - 2026-06-20

## 1. DOM-truth caught an invisible dead class (geometry-via-DOM payoff)
**Topics:** plato, dom-truth, tailwind

**Context:** Verifying SEGPILL's `tabs` variant on Home.

**Learning:** `getComputedStyle(...).borderTopLeftRadius === '0px'` on the Home tab exposed that `rounded-8` is a **no-op** in this Tailwind v4 setup (only `rounded-full` resolves, at 1.67e7px). Sharp corners *look* deliberate — a screenshot would have sailed past it. The class name has been lying since the row was written. Reinforces `feedback_dom_truth_over_screenshots` (geometry via DOM, not screenshots).

## 2. Token indirection hid a cross-cutting colour mismatch
**Topics:** matt, design-tokens, segpill

**Context:** The Tier-2 ledger predicted "3 sites hand-rolled identically," sourced from a class-string sensor.

**Learning:** `--Text-Primary` resolves to `var(--Primary-Default)` = `americana-blue`, same as `--Primary-Default`, while `--Text-Default` = `gray-base`. So a marker/string scan couldn't see that the `SegmentedPills` primitive (black active) disagreed with all 4 live consumers (blue active) on the one attribute that matters most — the active fill. Reading the *resolved tokens* is what surfaced it. Route-sweep PAUSE-before-extract earns its keep here.

## 3. MEMORY.md cap is entry-bound, not hook-bound
**Topics:** memory, cc-workflow

**Context:** Two consecutive levers (hook-trim, entry-merge) failed to clear the 80% SessionStart cap.

**Learning:** With ~79 genuinely-distinct load-bearing rules at ~260B each, the index sits at its content floor. Hook re-flattening (`/r-prune-memory`) took 90%→80.6% then flat-lined; entry-merging (`/r-coherence-check`) removed 5 files but barely moved bytes (combined hooks ≈ sum of what they replaced). Neither tool can create real headroom. The remaining levers are architectural: a tiered/2-file index (auto-load critical; grep-only below the 25KB line), raising the bar for what earns a memory, or accepting periodic prune as routine. **Open question for the user.**

---

## 4. A single `variant` enum beats orthogonal boolean props for bundled looks
**Topics:** react, component-design

**Context:** Three filter-row looks (wrap pills / segmented track / squared tabs) each differ on layout + shape + inactive treatment simultaneously.

**Learning:** Modeling them as one `variant: 'pills'|'segmented'|'tabs'` enum (each value bundling its combo) is a cleaner API than 3 orthogonal `layout`/`shape`/`tone` props, because only specific combinations are ever used. Follows the codebase's existing `Button`/`Select` variant pattern.
