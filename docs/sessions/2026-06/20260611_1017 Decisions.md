# Conv Decisions - 2026-06-11

## 1. Promotion delivery = Model 1 "reference + teaser lane" (D1 only, no Stream write)
**Type:** Strategic
**Topics:** stream, feeds, promotion

**Trigger:** User asked whether promotion needs to write to Stream at all.

**Options Considered:**
1. Copy (current `promote.ts`) — duplicate activity in target feed
2. TO-target — same activity, extra target feed (no copy) ← rejected (breaks at System; cross-boundary engagement leak)
3. Reference + teaser lane — D1 `post_promotions` row only, assembled at read time ← Chosen

**Decision:** Promoted posts live in D1 only; the original is never copied/re-homed in Stream. Every higher-feed appearance is assembled at read time (lane query → enrich by id → display-only teaser). `promote.ts` rewrites copy→reference; drop `post_promotions.target_activity_id`; lane joins `source_activity_id`.

**Rationale:** Posture A (act at source) + POST-MATT (display-only teaser) + System feed admin-only/unfollowed → pull is the only model that delivers cross-boundary uniformly at every level. Same pattern the Announcement model will reuse. Avoids duplication + engagement-split.

**Consequences:** Shipped copy-based `promote.ts` becomes a rewrite-to-simpler; lane injected at read surfaces (Home/community page/admin System view); a `canPromote` flag added to feed GET.

## 2. Inbound-visitor permission = posture A (read-only source feeds)
**Type:** Strategic
**Topics:** auth, visitor-gating, promotion

**Trigger:** "What can the promoted-to visitor do in the source feed?"

**Options Considered:**
1. A — read-only; join/enroll to participate ← Chosen
2. B — lightweight engage (react/comment allowed, only posting gated)
3. C — full member-gate (even viewing requires joining)

**Decision:** Non-members may view a source feed but not react/comment/post unless they join/enroll. Reacting/commenting count as participation → gated identically to posting.

**Rationale:** Cleanest conversion funnel; consistent with browse-vs-act model; resolves the 🔴 auth-only gap.

**Consequences:** `[VISITOR-GATING]` #29 owns it (PRIORITIZED) — close the gap across 6 endpoints via one `canParticipate` predicate + a "Join to participate" CTA + 403 tests.

## 3. Moderation = auto-live + post-hoc takedown
**Type:** Strategic
**Topics:** promotion, admin, moderation

**Trigger:** Admins need to see/remove System promotions that never reach the System Stream feed.

**Options Considered:**
1. A — auto-live + post-hoc takedown ← Chosen
2. B — admin approval queue (pending until approved)

**Decision:** Promotions go live on password-gate pass; admins moderate via the lane injected into their System feed view; takedown = delete the `post_promotions` row.

**Rationale:** Password gate is the sole anti-spam control (Conv 262); no approval-queue friction/build. Model 1 bonus: moderate-at-source cascades to all promoted appearances.

**Consequences:** Folds into `[ADMIN-FEED-UI]` #31.

## 5. "View Post" destination = feed-level (B-1), CTA relabeled "Visit Feed"
**Type:** Implementation
**Topics:** promotion, routing, addressability

**Options Considered:**

2. B-2 deep-link into feed (scroll/highlight)

## 6. Templates = entity-promo post content-type, author-direct-at-target (A2)
**Type:** Strategic
**Topics:** promotion, feeds, components

**Options Considered:**
1. A2-a author-direct-at-target (dedicated composer creates post + promotion in one gated action) ← Chosen
2. A2-b author-in-source-then-escalate (rejected — promo has no natural source conversation; circular origin)

**Decision:** A "promote-a-course/community" template = a feed post with Stream custom fields (`postKind:'entity_promo'`, `promoEntityType`, `promoEntityId`); display data resolved from D1 at render; renders a CourseAnchor/(new) CommunityAnchor in FeedPost's `embed` slot. Authored directly at a target via a composer. Promotion thus has two entry points (escalate-existing-post + author-new-promo) converging on `post_promotions` + the lane.

**Consequences:** New `CommunityAnchor` primitive needed; composer rides on `[HOME-FEED-MERGE]` #28; content-type+render+seed buildable independently.

## 7. Promote-nudges = PromoteNudge sibling, both surfaces, built last
**Type:** Implementation
**Topics:** promotion, nudges

**Decision:** A `PromoteNudge` sibling following ProgressionNudge's self-gating pattern. Per-post prompt (own post past engagement threshold, gated on a server `canPromote` flag) + workspace prompt in `/creating`+`/teaching` (client role-gated). Built LAST (encouragement layer atop the actions).

**Rationale:** Per-post can't be purely client-gated (canPromote is feed-specific/D1-backed → 403-after-click risk), so feed GET supplies `canPromote`. Anti-fatigue: per-post is rare (engagement threshold), workspace is one standing card.

## 8. Option labels must be typeable Latin — never symbols
**Type:** Implementation (process)
**Topics:** cc-workflow, docs-infra

**Trigger:** User flagged that 1/2/3-circled / Greek labels can't be typed back.

**Decision:** Option-question labels must be typeable Latin (`A/B/C`, `A1/A2` for nested) — never Greek/circled/emoji/math glyphs. Namespace nested questions with Latin compounds, not a different symbol set.
