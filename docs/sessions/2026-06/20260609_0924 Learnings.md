# Conv Learnings - 2026-06-09

## 1. EXISTS-filter vs scalar-projection collapse to one SQL fragment
**Topics:** d1, role-semantics, sql

**Context:** [RS-MOD-FRAG] had to centralize two `community_moderators` checks — one an `EXISTS (SELECT 1 ...)` filter (line 121), one a `(SELECT COUNT(*) > 0 ...)` projection (line 180).

**Learning:** In SQLite boolean *filter* position, `EXISTS (SELECT 1 …)` and `(SELECT COUNT(*) > 0 …)` reduce to the same 0/1, so a single scalar `COUNT(*)>0` fragment serves both filter and projection sites. This is why `roles.ts` can expose ONE `isXSubquery` per identity and have it work in `WHERE` / `roleClauses.push()` AND `… as is_x`.

**Pattern:**
```ts
export const isCommunityModeratorSubquery = (ref) =>
  `(SELECT COUNT(*) > 0 FROM community_moderators cm WHERE cm.user_id = ${ref} AND cm.is_active = 1)`
```

## 2. Astro role gates belong in middleware, not the page (blank-200 gotcha)
**Topics:** astro, auth, role-semantics

**Context:** /mod needed role-level authorization; the obvious move (page-frontmatter `requireModerationAccess`) collides with a known bug.

**Learning:** A `return Astro.redirect()` from inside a layout/nested component (AppLayout/AdminLayout) degrades to a blank 200 instead of a 302 (Conv-250 [ADMIN-REDIRECT-BLANK]). So any role gate on an AppLayout page MUST hoist to `middleware.ts` (runs before render). The codebase already resolved this for `/admin`; /mod mirrors it. Reusing `requireModerationAccess` (vs inline `roles.includes`) keeps the tiered global/community logic single-sourced.

## 3. Deterministic gate verification via dev-login + curl cookie jars
**Topics:** testing, auth, cc-workflow

**Context:** Needed to verify the /mod authorization gate across tiers without a heavy browser-login dance.

**Learning:** `POST /api/auth/dev-login {email}` (dev-only, PLATO's mechanism) issues a real session cookie. Curl with per-user `-c/-b` cookie jars gives deterministic HTTP-level gate verification (302 vs 200) — more reliable than eyeballing a browser. Reserve the browser for what curl can't see: React island *hydration* (DOM-truth: a rendered table with live data rules out a fatal hydration error; an error boundary would have replaced it).

## 4. conv-tasks.md needs a two-state thin marker; "stranded" is born, not decayed
**Topics:** cc-workflow, memory

**Context:** User probed whether the carry-over leaves enough signal that thin tasks are real.

**Learning:** "Is it real?" (strong: code + no-shrink reconciliation guard) is a separate axis from "do we know what it means?" (weak: the Thin marker). Splitting Thin into **Thin-Stub** (PLAN anchor verified — leave lazy) vs **stranded** (no def in PLAN/memory/git — decide keep/kill now) makes the at-risk subset visible. Empirically, 3 of 16 were already stranded — and tracing showed they were *born* as bare bundled codes, never described anywhere, not a decay over time.

## 5. Ordering the centralization first made the dependent feature a one-liner
**Topics:** role-semantics, refactor

**Context:** [RS-MOD-FRAG] (centralize community-mod SQL) ran before [MOD-NAV] (nav signal needing the same predicate).

**Learning:** Doing the fragment-centralization first meant the later nav signal consumed `isCommunityModeratorSubquery` instead of inlining a 3rd/4th copy of the subquery. Sequencing a DRY refactor ahead of its consumers converts would-be duplication into reuse for free.
