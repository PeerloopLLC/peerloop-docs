# Conv Learnings - 2026-06-26

## 1. Carried-forward "delete these" scope must be audited, not executed
**Topics:** cc-workflow, route-migration

**Context:** The OLD-PORTED-CLEANUP backlog note said "44 deletable copies + 4 dead components." Acting verbatim would have deleted 2 live components (FeedAllTab/FeedRoleTab, used by ExploreFeeds via `/old/discover/feeds.astro`) and risked nuking parked RG-PUBLIC pages that have no other copy. Re-vetting found 74 `/old` page files (not 44), only 12 with an exact root twin.

**Learning:** A backlog note phrased as a directive ("delete X") is really an *audit* request — re-verify importers/scope this conv before any deletion. The orphan-count from a `grep` looks identical whether a component is a true duplicate or a superset; only reading both implementations reveals the asymmetry.

## 2. Provenance-marker detection: first token in document order
**Topics:** docs-infra, provenance

**Context:** Marker names appear in graduation-history prose (`graduated @stand-in → @matt-inspired`, `Was @stand-in`) — often on the very marker line. Line-level `grep -l` and a `case` testing `*@stand-in*` before `*@matt-inspired*` both over-counted stand-ins.

**Learning:** The robust detector is the first marker **token** in document order, because the convention guarantees the real declaration is top-of-file. Worth baking into `prov-sweep.ts` ([PROV-SWEEP-MI]).

**Pattern:**
```bash
tok=$(grep -oE '@(stand-in|matt-source|matt-inspired)' "$f" | head -1)
```

## 3. Harvest boundary = which features compose with the data model
**Topics:** cc-workflow, components

**Context:** SessionHistory (flat table) vs TeacherSessionsList (course→student hierarchy). The harvest split cleanly: attendance + pre-group filters compose with the hierarchy; sortable columns + pagination are flat-table affordances with no clean meaning in a hierarchy.

**Learning:** When merging two components that solve the same surface differently, scope the port by *data-model compatibility*, not feature count. Features that fight the chosen model would mean partly un-choosing it.

---

## 4. Server-side filtering when the client only holds a windowed set
**Topics:** components, api

**Context:** TeacherSessionsList fetches only the most-recent 50 sessions. A client-side date-range filter would silently miss older sessions not in that window.

**Learning:** Filters over a windowed/paginated fetch must be applied server-side (pass params, re-fetch) — `/api/me/teacher-sessions` already supports `status`/`date_from`/`date_to`, so the port wired the filters into the fetch.
