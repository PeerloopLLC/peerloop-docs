# Conv Learnings - 2026-06-04

## 1. The flat CourseJourneyState already carried everything the two-tier cluster needed
**Topics:** astro, matt-design, ssr

**Context:** The [JOURNEY-LOOP] scope step 1 specified restructuring the journey state into nested `gates {…}` + `sessions {…}` objects.

**Learning:** `computeCourseJourney()` already returns `completedCount / scheduledCount / totalModules / bookableCount / nextSessionId / isComplete` — identical data, flat. Restructuring into nested objects would have churned the interface, `computeCourseJourney`, the NOT_ENROLLED default, and field reads in `[...tab].astro` (MySessionsTab) for zero behavioral gain. Kept it flat; the two-tier model lives in the *builder* (`buildCourseTabs`), which is where the gates-vs-cluster split actually renders. Deliberate, documented deviation from the literal step wording.

**Pattern:** When a plan step prescribes a data-shape refactor, check whether the existing shape already carries the data — prefer expressing the new concept at the consumer (builder/renderer) over a wide field-rename sweep.

## 2. Reuse the @matt-inspired ProgressBar inside an Astro container for the rail meter
**Topics:** astro, matt-design, primitives

**Context:** The Sessions cluster needs an "X/N" progress meter in `SubNav.astro` (an Astro file).

**Learning:** Astro renders a React `.tsx` component (`ProgressBar`, which uses no hooks — pure render) to static HTML with no client directive. Reusing it for the meter bar (vs hand-rolling a div) keeps the @matt-inspired primitive canonical and avoided adding markup. Per the scan-for-primitive-candidates rule, found `ProgressBar` + the existing `MySessionsTab` meter precedent before writing inline JSX.

## 3. `/book` dev-render gap is a pre-existing SessionBooking React-dup quirk, not a rail regression
**Topics:** astro, testing, cloudflare

**Context:** In `npm run dev`, `/book` intermittently dropped its whole layout (incl. the SubNav rail) after running `build`/`check` against the shared `node_modules/.vite`.

**Learning:** The dev log showed "more than one copy of React in the same app" / `useState` null in `SessionBooking`, `Sidebar`, `AuthModalRenderer` — all pre-existing islands, none in the change. Disambiguated by reloading `/course/[slug]` (which still rendered the full rail) → the error is specific to pages with the heavy `SessionBooking client:load` island, not global, not mine. The production `npm run build` SSR-rendered `/book` with zero errors. Resolves when the wizard leaves `@stand-in` in [CALENDAR2].

**Pattern:** When a dev-only SSR React-dup error drops a layout, reload a *lighter* route sharing the same shell to prove it's island-specific + environmental, and trust the production build as the real SSR gate.
