# Conv Learnings - 2026-06-07

## 1. conv-tasks.md no-shrink reconciliation has a cross-machine blind spot
**Topics:** cc-workflow, dual-repo

**Context:** `.scratch/conv-tasks.md` is gitignored/machine-local. On a machine that "skipped" intervening convs (here M4 skipped 247–248 done on M4Pro), the local companion is stale by MORE than one ledger, so /r-start Step 7.5's reconciliation (which reads only the latest RESUME-STATE Completed/Dropped) can't explain the shrink.

**Learning:** Git history is the authoritative cross-machine ledger. When the companion is multiple convs stale, reconcile dropped codes against `git log --grep` + prior RESUME-STATE ledgers via `git show <sha>:RESUME-STATE.md`, not just the current file. Even that has a gap: DOCGEN-SPEC left the carried backlog without ever being recorded as Completed OR Dropped — a task can silently evaporate. Recovery (re-add) beats halting.

## 2. "Pending" PLAN checklist items can be false-pending for ~50 convs
**Topics:** cc-workflow, route-migration

**Context:** MATT-EXEC-PG2's "5 remaining routes" included `/matt/` (dissolved Conv 197) and `/login` (ported Conv 207) — both done/dead but never struck from phase-5-pg2.md's checklist.

**Learning:** A checklist item asserting "pending" is a hypothesis until verified against the actual filesystem. Before tiering/porting work, audit the real `src/pages` tree rather than trusting a plan doc's open checkboxes. Two failure modes: false-pending (work done, checkbox not struck) and mis-bucketed (listed as Matt-design but really legacy-migration).
