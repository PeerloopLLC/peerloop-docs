# Conv Decisions - 2026-06-11

## 1. Extract the two-tier rails read into a shared lib module
**Type:** Implementation
**Topics:** smart-feed, discovery-rails, cloudflare

**Trigger:** Phase 3 needed the same KV-read + compute-fallback the `/api/discovery/rails` endpoint already had inline.

**Options Considered:**
1. Duplicate the two-tier logic in the smart endpoint
2. Extract `loadDiscoveryRailsBlob(db, kv?)` + `getDiscoveryRailsKV()` into `src/lib/discovery-rails/serve.ts`, consume from both ← Chosen

**Decision:** Shared module; KV passed as a param so the policy stays unit-testable; the rails endpoint refactored onto it (behavior-identical).

**Rationale:** One implementation, no drift; durable per §Solution Quality.

**Consequences:** `serve.ts` + barrel exports added; `rails.ts` shrank ~40 lines (inline logic deleted); behavior identical (`X-Discovery-Source` preserved). Both endpoints now consume the shared reader.

## 3. /feed visitor redirect → / (not /login) via middleware special-case
**Type:** Implementation
**Topics:** astro, middleware, smart-feed

**Trigger:** Phase 5 design Option B — a visitor who asked for "the feed" should land on the public feed, not a login wall.

**Options Considered:**
1. Drop /feed from `PROTECTED_EXACT` (would un-gate it for members too)
2. Special-case `pathname === '/feed'` in the no-session branch → redirect to `/`; every other route keeps login-with-returnUrl ← Chosen

**Decision:** Middleware special-case + mirrored page-level redirect in `feed.astro`.

**Rationale:** Keeps /feed auth-only for members (focused personal feed) while giving visitors the public feed (= Home).

**Consequences:** `middleware.ts` + `feed.astro` changed; middleware test updated (88 pass).
