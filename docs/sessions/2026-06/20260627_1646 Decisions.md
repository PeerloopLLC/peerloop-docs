## 1. PLATO-REVIVE data strategy → Option C (generate snapshots)
**Type:** Strategic
**Topics:** plato, testing

**Trigger:** Bucket 3's flagged-open decision: how the 4 snapshot-less browser instances get matching data.

**Options Considered:**
1. A — Re-pin instances to the dev seed (breaks their own `verify`, e.g. member-directory asserts COUNT=2 but dev seed has ~10; loses minimal-data edge cases)

**Decision:** Extend the established `flywheel` snapshot pattern to the snapshot-less instances.

**Rationale:** Zero instance/verify rewrites, no dev-seed pollution, faithful isolated data, verify queries pass as authored. It's the pattern already used by half the suite (`plato:restore`/`plato:snapshot:restore` exist), so it's finishing a half-applied pattern, not a novel decision.

**Consequences:** `snapshot: true` added to member-directory/activities/ecosystem; 3 `.sqlite` generated (gitignored). new-user-pair handled separately (Decision 2).

## 2. new-user-pair stays self-building (no snapshot)
**Type:** Implementation
**Topics:** plato, testing

**Trigger:** The decision-C follow-up: should the 4th snapshot-less instance also get a snapshot?

**Options Considered:**
1. Add `snapshot: true` (uniform with the other 7)
2. Keep self-building (no snapshot) ← Chosen

**Decision:** new-user-pair stays snapshot-less by design.

**Rationale:** Its walkthrough's *focus* is the register + onboarding flow itself; a restored snapshot would pre-create Alice & Bob and bypass exactly what the instance verifies. The snapshot pattern fits instances whose setup is condensed preamble to a feature-walk — new-user-pair is the inverse. The plan's "needs a `.scenario.ts`" premise was also wrong: its inline scenario already runs green as an instanceFile.

**Consequences:** 7/8 instances snapshotted; new-user-pair documented as a principled exception in PLAN + registry.
