# Conv Decisions - 2026-06-22

## 1. Course-level badges ‚Üí Matt semantic ramps (difficulty-as-status gradient)
**Type:** Strategic
**Topics:** matt-conformance, design-tokens, route-migration

**Trigger:** ProgressionCard's `getLevelBadge` used categorical hues (beginner=blue, intermediate=purple, advanced=red, default=gray). CC recommended keeping them honest-orphan per the locked playbook ("tokenless categorical/identity accent ‚Üí keep honest-orphan").

**Options Considered:**
1. Keep blue/purple/red honest-orphan (default‚Üíneutral) ‚Üê CC recommendation
2. Map onto Matt semantic ramps as a difficulty gradient ‚Üê Chosen (user)

**Decision:** Map difficulty onto semantic ramps: **beginner‚Üísuccess, intermediate‚Üíwarning, advanced‚Üíerror, unknown‚Üíneutral.**

**Rationale:** User chose to treat difficulty as an escalating status gradient rather than carry orphan hues. (Note: this is a deliberate override of the Conv-320 "identity/category ‚Üí keep honest-orphan" rule ‚Äî difficulty is treated as good‚Üíbad status, not an identity axis.)

**Consequences:** A course row showing both a "Beginner" level badge and a "Published" status badge now renders two success-green pills (level and status share the success ramp). Captured in the README swept-record. Sets a precedent worth applying consistently if course-level badges appear in other swept routes (XCUT-BACKREF territory).

## 3. Icon-ghost / compact-confirm / outlined-danger buttons kept raw-tokenized
**Type:** Implementation
**Topics:** matt-conformance, primitives

**Trigger:** ProgressionCard reorder/edit/delete icon buttons, the delete-confirm and danger-zone confirm strips, and the "Archive Community" outlined-danger trigger don't map cleanly onto a `<Button>` variant.

**Decision:** Adopt `<Button>` only for prominent standalone actions (Create/Save/Submit‚Üíprimary, Cancel‚Üíoutlined, Try-again‚Üídanger); keep icon-only ghost buttons, compact inline confirm-strip buttons, and the outlined-danger trigger as raw `<button>`s with Matt token colours.

**Rationale:** Consistent with the established "inline text-actions stay tokenized ‚Äî no primitive fits" precedent (CR-COMMUNITIES). The Button primitive has no outlined-danger variant, and Button-Small is too tall for the compact confirm strips.

**Consequences:** `<Button>` adoption is partial-by-design across these 4 files; not a conformance gap.
