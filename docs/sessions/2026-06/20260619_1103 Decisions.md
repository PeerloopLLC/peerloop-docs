## 2. Cross-cutting / shared-surface backward-pointer policy
**Type:** Strategic
**Topics:** rtmig-4, route-sweep, conformance

**Trigger:** "Swept = conform or consciously-allowed exception" leaves cross-cutting shared-surface residue that must be guaranteed-handled when a shared component changes after some routes are already swept.

**Options Considered:**
1. Fix-as-we-go, verify at next consuming route (user's first instinct — leaks orphans)
2. Verify-once-at-freeze ledger (CC's over-pitch — heavier)
3. User's model + backward-pointer ← Chosen

**Decision:** Sweep "done = client-showable"; shared components conform at first-touch (conformant-is-conformant); residual risk lands only on *unswept* routes. The one closed seam: any shared surface with **≥2 swept consumers** records its swept-route back-pointers, so a *later change* to that surface re-glances those swept routes. Rides the existing ledgers — no new tracker.

**Rationale:** Respects the user's demo-friendly model (risk on unswept routes is acceptable; swept routes stay conformant for client demos). The only leak — backward propagation from late changes to already-conformed shared surfaces — is closed by a cheap pointer rather than a heavier freeze-ledger.

**Consequences:** README § "Cross-cutting / shared-surface handling — the backward-pointer" added; tier2-ledger back-pointer convention; memory `feedback_route_sweep_pause_protocol` updated; `[XCUT-BACKREF]` #33 retro-seed task created.
