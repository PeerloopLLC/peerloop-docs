# Conv Learnings - 2026-06-29

## 1. Addressability-first test for route consolidation
**Topics:** astro, architecture

**Context:** Deciding whether `/visitor` deserved its own URL.

**Learning:** A route earns a standalone addressable URL only if something needs a *stable* link to it (deep-link, redirect target, `success_url`, notification, external/shareable link). `/visitor` had exactly ONE inbound (an in-app Sidebar chip click), zero redirects, and bounced authed users away — it failed the test and folded cleanly into an auth-aware `/profile`. Inbound-link count + redirect-target count are the concrete signals.

---

## 2. The app had no noindex/robots mechanism at all
**Topics:** seo, astro

**Context:** Account/auth surfaces shouldn't be indexed; public profiles should.

**Learning:** `BaseHead.astro` emitted title/description/OG/Twitter but NO `<meta name="robots">`, and `robots.txt` is `Allow: /` — so *every* public-reachable page was indexable by default. Added a `noindex?: boolean` prop threaded BaseHead→AppLayout. Now used to noindex account/auth surfaces (`/profile`, `/login`, `/signup`, private profiles, not-found) while public profiles stay indexable.

## 3. Ground route/scope decisions in the client requirements doc
**Topics:** project, requirements

**Context:** "Reframe /creator + /teacher to selling" risked building Phase-2 storefronts.

**Learning:** The client creator-profiles doc explicitly scoped MVP as "basic public profile using same system as Student/S-T profiles, with Creator badge" and deferred revenue dashboard/course-management/analytics to **Phase 2**. Probing it BEFORE building caught scope-creep — the reframe became "lock the identity-vs-commerce model + light CTA polish," not a storefront build.

---

## 4. Identity-vs-commerce resolves the hub-and-spoke "why"
**Topics:** astro, architecture

**Context:** The 3-URLs-per-person hub-and-spoke read as redundant/confusing.

**Learning:** `/@handle` = WHO a person is (identity); `/creator` + `/teacher` = WHERE a visitor transacts (commerce). Operational roles (admin/moderator) get consoles (`/admin`, `/mod`), never public profiles. This axis justifies the separate routes and is the durable mental model — the "deep profile view" framing was the confusing part.

---

## 5. One SSR loader as single source of truth for endpoint + page
**Topics:** astro, ssr, refactor

**Context:** The hub client-fetched `/api/users/:handle`; the spokes SSR'd via `fetch*ProfileData`.

**Learning:** Extracted `fetchPublicProfileData` into `lib/ssr/loaders/users.ts` and made BOTH the API endpoint (thin wrapper) and the SSR page call it. One query, one shape, no drift between the client-fetch and server-render paths. The endpoint's contract stayed identical (test 17/17), so the refactor was zero-risk to API consumers.

---

## 6. curl is the definitive SSR verification; browser DOM masks it
**Topics:** testing, ssr, browser

**Context:** Verifying the hub actually server-renders.

**Learning:** A `client:load` React island hydrates either way, so a browser DOM read can't tell SSR'd-content from client-filled. `curl` the raw HTML (no JS) is the real proof — confirmed `<title>{name}`, bio meta, no-robots, and `name×9 + About/Stats/Expertise` present in the server HTML. Also: the Chrome-bridge `javascript_tool` returns empty for async-IIFE bodies — use a synchronous last-expression read instead.
