# Conv Learnings - 2026-06-13

## 1. The Premise-Check Gate flips conclusions on "trivial" cleanups
**Topics:** process, schema, premise-check

**Context:** Two feed-group cleanups looked like rote one-liners but turned on premise verification, not code volume.

**Learning:** `[PROMOTE-IDEMP]`'s UNIQUE guard lived in a standalone `CREATE UNIQUE INDEX idx_post_promotions_unique` (line 1361), NOT inline in the table def — a table-only `awk` scan missed it and nearly produced a false "no constraint exists" alarm. `[SYS-NAMING]` was already done (folded into U1). Reading the actual source before acting flipped the conclusion in both cases. Verify constraints by grepping BOTH inline table defs AND standalone `CREATE [UNIQUE] INDEX` statements.

## 2. U3d's "per-post" was partly pre-built — premise correction
**Topics:** promotion, planning, scope

**Context:** The plan listed "per-post (server canPromote)" as Surface 2 of U3d.

**Learning:** Two distinct promotion models coexist: **post-escalation** (`PromoteButton` → `/api/feeds/promote`, push an existing post up the chain, PROMOTE-PIPELINE Conv 262, already on every eligible post) vs **entity-promotion** (`EntityPromoComposer` → `/api/feeds/promote-entity`, author a new promo, U3b Conv 274). The plan's "per-post (server canPromote)" literally names the former, which already existed. A real *nudge* (attention-drawing prompt) is distinct from the always-on action button.

## 3. PromoteNudge self-gates client-side without server flags
**Topics:** islands, current-user, promotion

**Context:** Considered whether the nudge needs per-post server `canPromote` checks.

**Learning:** `CurrentUser` already exposes everything the `canPromote` matrix checks (`isCreatorFor`/`isTeacherFor` for courses, `getCommunityMemberships` memberRole + `getTeachingCommunityIds` for communities), so a nudge can gate purely client-side like ProgressionNudge — no server-flag threading. The workspace card instead fetches `promotable-entities` (which already enforces the matrix server-side) and gates on the engagement floor.

## 4. Vite dep-optimizer churn produces false-negative DOM reads
**Topics:** browser-verify, vite, bridge

**Context:** First DOM read on a freshly-started dev server showed `islandCount: 0` (no islands hydrated at all).

**Learning:** A dev server's startup dep-optimizer reload (`astro-*.js does not exist / reloading`) can load a page mid-reoptimize so NO islands hydrate — a false negative. A hard reload after vite settles gives the real state. Reinforces `[BRIDGE-MEM]` settle-then-read and "verify against a settled dev server."
