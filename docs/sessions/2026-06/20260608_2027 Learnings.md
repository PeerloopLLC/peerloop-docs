# Conv Learnings - 2026-06-08

## 1. Unified dashboard aggregated at the wrong altitude (role, not entity)
**Topics:** ia, dashboard, role-semantics

**Context:** Re-assessing the combined `/dashboard` vs per-role dashboards.

**Learning:** Users navigate by ENTITY (a course, a session), not by role. The unified dashboard fragmented entities to aggregate roles (a course split across §MergedCourses + §MergedSchedule), and its "see all at once" promise died behind tabs for the deep surfaces. Cross-role simultaneity that genuinely matters is already solved ON the entity (course/community role tabs).

---

## 2. Three cross-role needs, each at its own altitude
**Topics:** ia, flywheel, nudges

**Context:** Deciding what the dashboard layer should own after entity tabs handle operational context.

**Learning:** (a) operational context → entity tabs (done); (b) cross-role triage → thin strip on `/`; (c) role progression → CTAs on the SOURCE role's surfaces. Progression nudges must live where the source role already is (course, home, teacher workspace), NEVER the target role's hub — so a never-/teaching student is still reached. This dissolves the "siloed hub is invisible to non-adopters" objection.

---

## 3. Performing vs oversight roles
**Topics:** ia, roles, admin

**Context:** Where do admin/moderator go when the dashboard deconstructs?

**Learning:** Performing roles (student/teacher/creator) DO the flywheel → deconstruct into entity-first workspaces. Oversight roles (admin/moderator) OVERSEE across domains → cross-cutting triage IS their job → keep a combined command center (`/admin` already is one; moderator → `/mod`). Connect domains via admin-only contextual deep-LINKS, not embedded oversight zones.

---

## 4. The hybrid `is_creator` gates are ACCESS gates, not identity
**Topics:** role-semantics, access-control, auth

**Context:** The blast-radius audit before committing to RS-HYBRID-FLIP.

**Learning:** `if (!is_creator) → 403` is an ACCESS gate → by the capability/identity rule it belongs on `canCreateCourses` (capability), NOT behavioral identity. The naive "flip hybrid→behavioral" would have 403'd every approved-but-0-course creator. Correct migration = split each site by PURPOSE: access→`canX`, identity/display/nudge→`isX`. Re-examining (the user's "C") caught a real regression cliff before any code changed.

---

## 5. session.roles JWT is permission-based — latent security trap
**Topics:** auth, role-semantics, security

**Context:** Audit Agent 3 mapped route gating.

**Learning:** `getUserRoles()` bakes a permission-based roles array into the JWT; `requireRole` only ever checks `['admin']`/`['moderator']` (assigned), so a creator/teacher identity change is security-neutral TODAY. Trap: a future `requireRole(['creator'])` would read the stale permission flag. Rule banked: never gate on `['creator']`/`['teacher']` until `getUserRoles` is behavioral.
