# Conv Learnings - 2026-06-23

## 1. Tokenizing raw font-weight requires per-element rendered-size determination
**Topics:** matt-design, conformance, typography

**Context:** RG-WORKSPACES "font-weight bulk" looked mechanical (58 hits), but most were standalone `font-medium`/`font-bold` with no type token on the element — size inherited from an ancestor row/container.

**Learning:** The correct target token (`text-body-small-medium` 12px vs `text-body-default-medium` 14px) depends on each element's ACTUAL rendered size, not the markup. Proven when EarningsDetail (14px `text-body-default` rows) and its twin CreatorEarningsDetail (12px `text-body-small` rows) needed DIFFERENT tokens for identical-looking `<th>` markup. A blind find-and-replace would have silently shrunk/grown half the table text.

**Pattern:**
Before folding a raw weight, read the element's own size class → else its row/container's size class. Only then pick the matching `-medium`/`-bold` variant.

## 2. UA `<th>{font-weight:bold}` defeats row-inherited weight tokens
**Topics:** css, conformance

**Context:** Considered dropping `font-medium` from table `<th>`s and setting `text-body-small-medium` on the `<tr>` (DRY).

**Learning:** A `<th>`'s UA-stylesheet `font-weight:bold` is a direct declaration that beats an inherited row token, so dropping the explicit weight would silently RE-BOLD every header. The weight-bearing token must sit directly on the element. (Also: this codebase's `body-*` weight variants hold size/lh constant — only weight differs — so direct-on-element folds are provably 0-change for medium; bold is 700→600.)

## 3. The display/glyph regime (§9.2c) doesn't cleanly cover off-scale inline avatars
**Topics:** matt-design, typography, primitives

**Context:** 5 inline avatar-initial fallbacks flagged as font-weight hits; tried to map to `text-display-avatar-*`.

**Learning:** The regime/UserAvatar keys glyph size to specific circle sizes (24→12, 32→14, 48→18, 64→24, 96→30px via default-Tailwind `h-6…h-24`), but the inline fallbacks use literal-px circles (32/36/40px) AND three different bg/text styles (gradient-white vs brand-100/brand-500 vs neutral). So "refactor all to UserAvatar" changes colour treatment + size, not just weight — a real visual change, Tier-2. Only avatars with an EXPLICIT current glyph size (14px) map cleanly to `text-display-avatar-sm` now.

## 4. In-code honest-orphan comments are first-class decisions a reconciliation must respect
**Topics:** matt-design, conformance, process

**Context:** The deep-verify flagged MyFeeds + SessionAnalytics colours as "Genuine" violations, but both carry in-code comments documenting them as deliberate honest-orphans (category accents / "no Matt notification scale" / quartet precedent).

**Learning:** A heuristic audit flag ("sibling cards use tokens, these don't") can conflict with a sound documented rationale. When the code already states a deliberate colour decision and "Colour keeps exceptions" applies, verify-and-keep — don't re-tokenize. Surface the conflict for a ruling rather than silently overriding.
