# Conv Decisions - 2026-06-14

## 1. Per-post promote highlight uses a new dedicated dial
**Type:** Implementation
**Topics:** promotion, feeds, admin

**Trigger:** [U3D-POST] needed an engagement floor to decide when a post is "hot" enough to highlight its Promote affordance.

**Options Considered:**
1. New dedicated dial `promo_post_min_engagement` ← Chosen
2. Reuse the entity dial `promo_nudge_min_engagement`
3. Hardcode a constant

**Decision:** Add a 4th `promo_%` dial `promo_post_min_engagement` (default 3, `stat-promo-004`), admin-tunable alongside the existing three.

**Rationale:** Post interactions (reactions + comments) and entity members are different units; one shared integer would be semantically muddy. The dial pattern is a cheap, established extension (config + seed + admin endpoint/UI).

**Consequences:** New platform_stats row; admin Promotion Settings page now edits 4 dials (grid → 2×2). 0 = always highlight.

## 2. Hot state is stateless "persist until promoted," elevated in-card
**Type:** Implementation
**Topics:** promotion, feeds, ui

**Trigger:** Needed to decide the persistence model and visual placement of the attention-drawing nudge.

**Options Considered:**
1. Persist-until-promoted (stateless) + elevate the in-card PromoteButton ← Chosen
2. One-time dismissable (localStorage) and/or a separate banner / home-feed rail

**Decision:** The highlight is computed every render from live engagement (no storage); it vanishes when the post is promoted (button flips to "Promoted") or cools below the floor. Visually it's the existing in-card `PromoteButton` swapped to an accented state, owned inside the component.

**Rationale:** Stateless sidesteps the localStorage cache-staleness bug class ([NUDGE-CACHE-FLASH]); in-card elevation reuses one component and auto-covers every feed; centralized styling can't drift across the two render paths.

**Consequences:** No new persistence/dismissal surface; `PromoteButton` gains a `hot` prop; `ml-auto` moved to a wrapper span in `FeedActivityCard`.
