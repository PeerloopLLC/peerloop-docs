# Conv Learnings - 2026-06-28

## 1. dev-login is blocked on staging — the [BRIDGE-MEM] verify pattern is dev-only
**Topics:** deployment, cloudflare, testing, cc-workflow

**Context:** Tried `POST /api/auth/dev-login {email}` (the standard password-free island-verify path) against staging to inspect the creator grading UI.

**Learning:** `dev-login.ts` guards on `if (!import.meta.env.DEV) return 404`. `import.meta.env.DEV` is true ONLY under `vite dev`; a staging deploy is a *production build* (just with staging bindings), so dev-login returns 404 on staging. The `[BRIDGE-MEM]` dev-login verification flow works on the local `:4321` dev server but NOT on staging — staging verification needs a real login (password), which CC cannot type (safety rule) → user-assisted handoff.

## 2. Stale login-state localStorage can break the staging login modal's render
**Topics:** deployment, testing

**Context:** The staging `/login` modal rendered once (prefilled a stale "Amanda Lee" identity) then stopped rendering on repeat visits.

**Learning:** The login island reads `peerloop_was_logged_in` + `peerloop_expired_identity` from localStorage to show an "expired session" prefill. Leftover values from a prior staging session corrupted the re-render (modal absent from DOM). Clearing those two keys + reload restored a clean modal. When DOM-verifying on a shared browser, stale client-side auth state is a real source of flakiness.

## 3. Pre-launch schema edits force a destructive staging-DB reset to converge
**Topics:** deployment, d1, migration

**Context:** Staging was 7 commits behind on `migrations/0001_schema.sql`; planning how to get the new tables/columns onto the staging D1.

**Learning:** Because pre-launch schema edits land directly in `0001_schema.sql` (not incremental migrations), and D1's tracker already counts `0001` as applied, `wrangler d1 migrations apply` will NOT re-apply the edits. The only way to converge is a destructive `db:reset:staging` → re-migrate → re-seed (`db:setup:staging:feeds`). This is the central gotcha of any large staging redeploy pre-launch.

## 4. PLATO seed is a separate, mutually-exclusive full-reset path
**Topics:** testing, deployment, d1

**Context:** Question of whether PLATO test data should be in the deploy seed.

**Learning:** `plato:seed:staging` (`scripts/plato-seed-staging.js`) exports a snapshot, **resets the staging D1 (drops all tables)**, and applies the dump — it REPLACES all staging data. It is mutually exclusive with `db:setup:staging:feeds`; you cannot have both demo data and the PLATO snapshot. PLATO API-mode tests run against the in-memory better-sqlite3 test DB, not staging, so PLATO data is not needed for `npm test` or the app. It belongs OUT of the deploy seed chain.

## 5. Homework attachment render path: r2_key → download_url → chip; file streams from R2
**Topics:** testing, cloudflare

**Context:** Seeding a file-upload homework submission so the grading UI shows a download link.

**Learning:** The submissions list endpoint sets `download_url = r2_key ? getSubmissionDownloadUrl(id) : null`; `HomeworkGradingPanel` renders the download chip iff `download_url` is truthy. The download endpoint `r2.get(r2_key)`s the actual file. So a seeded row with `r2_key` set makes the UI *render* the chip, but the download 404s until a matching object exists in the bucket (`wrangler r2 object put <bucket>/<key>`). `r2_key` format = `src/lib/r2.ts::generateSubmissionKey` = `courses/{courseId}/homework/{assignmentId}/submissions/{submissionId}/{sanitizedFilename}`.

## 6. Chrome bridge: native click + top-level-await + cookie-blocked responses
**Topics:** cc-workflow, testing

**Context:** Driving the staging browser verification.

**Learning:** (a) Re-confirmed `[BRIDGE-MEM]`: `element.click()` via `javascript_tool` reliably actions hydrated islands where coordinate `computer left_click` can no-op. (b) `javascript_tool` does NOT await an `(async () => {...})()` IIFE — it returned `{}`; use **top-level await** instead. (c) A `fetch()` whose response carries cookie/session data is returned as `[BLOCKED: Cookie/query string data]` by the extension — return only non-sensitive scalars (e.g. `res.status`) to confirm a request.
