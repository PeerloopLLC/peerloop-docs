# Conv Learnings - 2026-06-12

## 1. Stop hooks get the final message pre-extracted — no transcript parsing
**Topics:** cc-workflow

**Context:** Building the QLINT Stop hook; expected to parse `transcript_path` JSONL for the last assistant message.

**Learning:** The Stop-hook stdin JSON includes a `last_assistant_message` convenience field plus a `stop_hook_active` boolean loop-guard. Prefer the field; keep a transcript fallback (`.type=="assistant"`, non-sidechain, `.message.content[] | select(.type=="text") | .text`, last) for version resilience. Block via `{"decision":"block","reason":...}` (reason fed back to Claude); a hard 8-consecutive-block ceiling backstops loops.

**Pattern:** Fail OPEN — if both the field and fallback are empty, `exit 0`. An enforcement hook must never trap a turn it can't end.

## 2. Question-format enforcement is a typeability check, not a syntax check
**Topics:** cc-workflow

**Context:** User corrected the QLINT heuristic twice.

**Learning:** The rule exists because the user answers by *typing*. The gate fires when answering needs anything other than a single label or plain yes/no: an `or`-choice ANYWHERE in a soliciting message (not just the `?` sentence), OR a 3+ inline slash-list `(a/b/c)`. `(yes/no)` and "yes or no" are blessed. Only A)/B) labels exempt — a bare 👉 does not (a 👉'd compound-or question is still un-scannable). Precision guards: restrict the slash-list rule to inside parentheses (so file paths `src/lib/x.ts` don't trip it); strip code fences + double-quoted spans (so quoting the rule doesn't self-trip); require the last non-empty line to END with `?` for the no-👉 path (so statement turns stay silent).

## 3. Time-windowed seed data needs the relative-date freshness mechanism, not fixed dates
**Topics:** testing, d1

**Context:** 4/6 discovery rails were empty in dev.

**Learning:** The discovery rails compute `new` (created <30d) and `trending` (velocity <7d) signals. Seeded source rows with fixed historical dates never satisfy these windows. The seed already had a "TIMESTAMP FRESHNESS" section (post-INSERT `strftime('now','-N days')` UPDATEs) for exactly this; it just hadn't been applied to the rail-source tables (courses/communities/enrollments/community_members). Freshening must store ISO `T`-format (`strftime('%Y-%m-%dT%H:%M:%fZ',…)`) to match the JS `.toISOString()` cutoff the compute compares against (the SQLite-datetime pitfall).

**Pattern:** When freshening an entity's date, pick rows with NO conflicting dependents, or freshen the whole dependent sub-tree in step — else you invert a timeline (e.g. a completed-in-2024 enrollment predating a course created 6 days ago).
