# Conv Learnings - 2026-06-11

## 1. Source-feed handlers already encoded the participation rules the reaction/comment endpoints lacked
**Topics:** stream, auth, testing

**Context:** The 6 reaction/comment endpoints gated on `getSession` only (401), while the sibling `[slug].ts` GET/POST handlers already had the correct membership/enrollment/admin logic inline (community membership query, `checkPostingRights`, `isUserAdmin`).

**Learning:** The fix wasn't inventing a rule — it was extracting the existing-but-duplicated logic into one `canParticipate` predicate and pointing both the source handlers and the mutating endpoints at it. Single source of truth emerged as a byproduct of closing the gap, not as separate refactor work.

---

## 2. Adding a gate breaks happy-path tests that used non-existent fixtures
**Topics:** testing

**Context:** Existing reaction/comment tests posted to a fake `test-slug` with no community/membership row and expected 200. Once the gate landed they would 403.

**Learning:** A participation gate's test cost is two-sided: seed real participation for every happy-path actor AND add the deny cases. Placing the gate *after* input validation preserved the existing 401/400 tests untouched, limiting churn to the happy-path fixtures + new 403/404 cases.

**Pattern:** Block-scoped `{ let gateDb; ... }` insertion in comment endpoints avoids colliding with the handler's existing `db`/`user` locals when wiring a gate mid-handler.
