# Conv Learnings - 2026-06-09

## 1. Read the function body, not just the signature, before "fixing" a conflated field
**Topics:** role-semantics, refactoring

**Context:** `ContextActionsPanel.tsx` passed `is_creator: user.canCreateCourses` — flagged as "permission-as-identity" needing Decision A.

**Learning:** `getActionsForContext`'s body never reads `is_creator`/`is_teacher` (it gates on `entityData.isOwner`/`isAssignedST`). The fields were dead pass-throughs. A signature-only reading would have "fixed" the axis while changing nothing and preserving the smell; the durable fix was deletion. Type signatures lie about usage — grep the body before re-pointing a field.

## 2. Exact-string sweeps survive a near-identical look-alike via a structural delimiter
**Topics:** mechanical-refactor, sql

**Context:** `members/index.ts:175` (`is_creator`) and `:176` (`has_active_courses`) differ only by a trailing `AND c.is_active = 1`.

**Learning:** The creator identity string ends in `... IS NULL)` (closing paren); the look-alike has `... IS NULL AND c.is_active = 1)`. The paren immediately after `IS NULL` makes the exact-match miss the look-alike — so a global replace is safe *because* of the delimiter, not in spite of it. Always identify the look-alike and prove the match boundary excludes it before a pattern-global sweep.

**Pattern:**
Surface-before-sweep: for fuzzy-scope mechanical refactors, narrow the grep to the exact replaceable shape, surface a SWAP/LEAVE/DEFER disposition + 👉, then run an exact-string idempotent script with a count assertion.

## 3. Behavioral-in-JWT is a stale snapshot; permission-in-JWT is correct by construction
**Topics:** auth, role-semantics

**Context:** Considered flipping `getUserRoles` (JWT `session.roles`) to behavioral for axis purity during the Tier-3 fix.

**Learning:** The JWT is set at login. A behavioral role value (has-created-course) would freeze at login and go stale mid-session until re-login. Permission flags only change via admin action + re-login anyway, so permission-in-JWT has no staleness gap. Since `requireRole` gates only `['admin']`, there's zero gating benefit to flipping — so the rule "identity = behavioral" does NOT extend to the JWT. Axis purity has a legitimate exception at the auth-snapshot boundary.
