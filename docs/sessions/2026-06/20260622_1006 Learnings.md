# Conv Learnings - 2026-06-22

## 1. The bridge `--spacing-N` override is a FIXED set, not a global multiplier
**Topics:** matt, astro, spacing

**Context:** Conforming MyStudents (784 ln), several numeric utilities (`h-20`, `w-40`, `w-48`, `p-6`) needed restoring — the component was rendering at ~¼ intended spacing.

**Learning:** `tokens-tailwind-bridge.css` `@theme` overrides only `--spacing-{4,8,12,16,20,24,32,40,48,64}`. Values IN the set resolve to Matt px; values OUTSIDE fall through to Tailwind v4's default scale (the bridge-shrink trap — e.g. `w-40` rendered 40px instead of 160px, a status dropdown at ~¼ width). Conformance of a bridge-shrunk legacy component = `legacy-N → N×4` *landing on a set member* (`p-6`→`p-24`), and arbitrary `[Npx]` when `N×4` ∉ the set (`h-20`→`h-[80px]`, `w-40`→`w-[160px]`, `w-48`→`w-[192px]`). Radius set: `rounded-{2,4,6,8,12,16,24}`.

## 2. `neutral-400` is NOT a Matt token — silent Tailwind-default leak
**Topics:** matt, palette

**Context:** Used `text-neutral-400` in TeacherSessionsList; later found CourseCard:160 had the same.

**Learning:** The Matt neutral scale is `{50,100,300,500,700,900}` (no 400). The bridge does NOT clear Tailwind's default ramp (`@theme` adds steps, no `: initial`), so `neutral-400` silently resolves to Tailwind's default grey — an off-palette leak the standard `secondary-*`/`primary-[0-9]` leak grep misses. Map legacy `secondary-400` → `neutral-300` (per the /learning map), and add `neutral-(200|400|600|800)` to conformance leak greps.

## 3. Conforming one component audits the design rule
**Topics:** matt, process

**Context:** TCH-SESSIONS was "the easy tab" but surfaced that the Conv-319 status-token rule was written from an incomplete view of the token set.

**Learning:** Applying a design rule to a fresh component forces its edge cases into the light. The Conv-319 rule listed only `red→error`/`amber→warning` as examples, so green/blue silently fell to "honest-orphan" even though `success`/`info` ramps existed. The cheapest audit of a design rule is to apply it to one new surface.

## 4. Bound a "rule changed, re-check everything" task by swept-vs-unswept
**Topics:** matt, process

**Context:** The XCUT-BACKREF ripple could have ballooned into a codebase-wide green/blue hunt.

**Learning:** When a design rule changes, the re-application scope is ONLY the surfaces already conformed under the old rule. Unswept surfaces (and pending tabs like TeacherCourseView) get the corrected rule when their own sweep runs — they're out of ripple scope. Separating swept from unswept kept the ripple to 3 real occurrences instead of ~20 grep hits.

## 5. Map-vs-keep test for semantic colour tokens
**Topics:** matt, palette

**Context:** Repeatedly deciding whether a green/blue should map to a Matt semantic token or stay honest-orphan.

**Learning:** Test: *does the colour encode a state read as good/bad/in-between?* → MAP (completed→success, scheduled→info, failed→error). Is it identity/category/data-viz? → KEEP (chart series, categorical accent palettes whose set includes a tokenless member, a cohesive domain legend like the sky/red/amber calendar). Availability is a tokenless semantic (kept as sky), so availability-green also stays honest-orphan.

## 6. Chrome bridge REPL: top-level await is unavailable
**Topics:** plato, cc-workflow

**Context:** Dev-login + DOM-verify via `javascript_tool`.

**Learning:** The `javascript_tool` REPL harness rejects top-level `await` (and async IIFEs error too). Use the fire-and-stash pattern: kick off `fetch(...).then(r => { window.__x = result })` returning a marker, then read `window.__x` in a follow-up call. For client:load island DOM-verification, poll-until-rendered with `setTimeout` writing to a `window` global, then read it back.
