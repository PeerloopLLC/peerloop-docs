## 1. Full atomic cutover for the CURTASKS read/write flip
**Type:** Strategic
**Topics:** cc-workflow

**Trigger:** Conv 351 picked up `[CURTASKS]` Phases 2–3 — the skill cutover from the old RESUME-STATE-Remaining + machine-local `.scratch/conv-tasks.md` split to the single git-tracked `CURRENT-TASKS.md` store (architecture decided Conv 350, DEC-350-1/-2/-3).

**Options Considered:**
1. **Full atomic cutover** — build the new engine, dry-run it, then flip the read path and write path together in one commit. ← Chosen
2. Phased / partial flip — land one skill at a time across multiple convs.

**Decision:** Flip all four skills together in a single de-risked sequence: build A (`/r-update-tasks`) → dry-run against the live file → flip B (`/r-start` read path), C (`/r-end` Step 5 write path), D (`/r-commit` Step 0). The cutover goes live at this conv's `/r-end`.

**Rationale:** The read and write paths are coupled — a `/r-start` that hydrates active-only while `/r-end` still writes the old RESUME-STATE-Remaining backlog would silently drop the backlog. A partial flip has no safe intermediate state. The risk of a big-bang flip was bought down by dry-running the write engine mechanically first (zero row loss) and by sequencing the first live write onto the lower-stakes `/r-commit` Step 0 before the full-scale `/r-end`.

**Consequences:** 4 skills flipped in one docs commit (`217b8ad`, +275/−177); `RESUME-STATE.md` demoted to narrative-only (Branch line kept); `CURRENT-TASKS.md` got its first live refresh. Phases 4–5 (scripts/config + docs/memory cleanup) deferred per the user's scope pick. Read path remains unverified until next conv's first `/r-start`.

## 2. Three load-bearing H2 anchors — render in-progress inline, drop spt's "In progress" section
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** Porting spt's `CURRENT-TASKS.md` structure to peerloop. spt's model carries a fourth `## 🔴 In progress` H2 section.

**Options Considered:**
1. Mirror spt's 4 anchors (Ordered / Unordered / In progress / Completed).
2. **Collapse to 3 anchors** (🔥 Ordered / 📋 Unordered backlog / ✅ Completed this conv); surface in-progress via the Ordered header segment `· 🔄 Active ·`. ← Chosen

**Decision:** `CURRENT-TASKS.md` has exactly three real `## ` H2 anchors that the `/r-start` + `/r-end` + `/r-update-tasks` parsers key on; there is **no** In-progress section. A row that is `in_progress` shows it inline (`· 🔄 Active ·` in its Ordered header); a started backlog item keeps its place and is never auto-promoted into Ordered.

**Rationale:** In-progress state is conv-ephemeral — it lives in TodoWrite for the conv and resolves at `/r-end` (→ Completed, or stays backlog). A persistent In-progress *section* would duplicate TodoWrite and force auto-promotion/demotion logic on every refresh. An inline header glyph keeps the file's structure stable and the parser contract minimal (key on 3 anchors, not 4).

**Consequences:** Codified in `/r-update-tasks` Step 6 + the "Peerloop 3-anchor note" + the file's own format preamble. Verified live in the `/r-commit` Step 0 refresh ("3 anchors, 15 rows").

## 3. Pending preserves hand-set priority symbols (the overlay rule)
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** Defining how the preserve-then-overlay engine maps `TaskList` status onto the Ordered header symbol, given the user hand-curates `★ Next` / `🔄 Active` / `📋 Planned` / `⏸️ On hold`.

**Options Considered:**
1. Overlay every code-matched row's status from `TaskList` (maps the flat `pending` onto a generic symbol).
2. **Preserve the file's hand-set symbol for matched `pending` rows**; only `in_progress` and `completed` are authoritative. ← Chosen

**Decision:** On a code match, only `in_progress` (→ force `· 🔄 Active ·`) and `completed` (→ move to Completed) override the file. A matched `pending` **preserves the existing header symbol verbatim**; an unmatched row likewise preserves its symbol.

**Rationale:** `TaskList`'s `pending` carries no priority granularity, so trusting it would downgrade the user's curated `★ Next`/`📋 Planned`/`⏸️ On hold` ordering to a generic "pending." TodoWrite only speaks authoritatively for what is in flight; the file is the authority for not-yet-started rows.

**Consequences:** Encoded in `/r-update-tasks` Step 6 + the Rules section ("never downgrade a hand-set symbol to a generic one"). Together with the [TWAO] never-drop-unmatched guard, this makes the refresh safe to re-run any number of times without eroding hand-edits.
