# Conv Learnings - 2026-06-01

## 1. Stripe Checkout has no "fail" URL ‚Äî only success + cancel
**Topics:** stripe

**Context:** Asked what URL Stripe returns to on failure/timeout.

**Learning:** Stripe Checkout's redirect model has exactly two return URLs. Card declines/errors are handled **in-page** on Stripe's hosted page (user retries, never redirected). Session expiry shows an in-page expired notice. The only non-success redirect to the app is `cancel_url` (user clicks back). There is no third failure URL to design for ‚Äî failure UX lives on Stripe's side, not ours.

---

## 2. `create-session.ts` writes zero D1 state ‚Üí expired-webhook handler is a no-op
**Topics:** stripe, d1

**Context:** The `checkout.session.expired` handler's `FUTURE` comment claimed "clean up pending enrollment record, free up seat."

**Learning:** Checkout-session creation writes NOTHING to D1 (the `pendingEnrollmentId` is just a UUID in Stripe metadata; the enrollment row is created only on success via `createEnrollmentFromCheckout`), and courses have no capacity-limited seats. So an expired/abandoned checkout leaves no DB artifact to clean and no seat to free ‚Äî the handler would be dead code. The `pl_pending_sessions` hint is client-side localStorage, cleaned by `MyCourses.tsx`. Revisit only if pending-rows or seat-limiting are ever added.

## 3. Matt's success-frame metrics were mock ‚Äî bind to real curriculum data
**Topics:** matt-migration, design-translation

**Context:** Matt frame 579:16885 showed "Setting Up Claude Code ¬∑ 4 Modules ¬∑ 90 min."

**Learning:** Probing the seed showed the real first module is "Setting Up Claude Code" (title matches Matt) but **1 sub-module, 25 min** ‚Äî "4 Modules / 90 min" were design-mock numbers. Faithful port = bind title/description/duration to `course_curriculum` (first row by `module_order`), sub-module count via rows sharing `session_number` (per the 1:1 Session‚ÜîModule model), and let the figures be honest rather than copying Matt's placeholders.

## 4. `fetchCourseTabData` is the shared course loader ‚Äî reuse it for new course-family pages
**Topics:** astro, ssr

**Context:** Both the earnings wiring and the success page needed course/creator/curriculum/enrollment data.

**Learning:** `fetchCourseTabData` already returns course row + creator + curriculum + isEnrolled + enrollmentId. New course-family pages (precheckout, success) should reuse it and layer page-specific logic (self-heal, hasExpectations) on top rather than re-querying ‚Äî adding one field there (`teacherEarningsCents`) propagated to every consumer for free.

## 5. MattIcon supports per-icon viewBox; two-tone Material badges render via see-through counter
**Topics:** icons, matt-migration

**Context:** Added `verified` (48px) + `av-timer` (20px), neither on a 24-grid.

**Learning:** The MattIcon wrapper reads each SVG's own `viewBox`, so harvested icons keep native coordinate space (don't force 24√ó24). Mask `<rect>` fills stay `#D9D9D9`; only the visible path ‚Üí `currentColor`. The `verified` badge's inner check is an opposite-wound subpath, so with a single `currentColor` green fill the check renders as a see-through counter (showing the card bg) ‚Äî matching Matt's render without needing two colors.
