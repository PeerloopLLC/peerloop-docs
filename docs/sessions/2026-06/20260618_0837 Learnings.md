# Conv Learnings - 2026-06-18

## 1. ListingShell's flex model is wrong for a decorative whitespace panel
**Topics:** astro, layout, matt-design

**Context:** Client wanted Home's dead left gutter killed via a right-side light-blue panel that *grows* on wide screens.

**Learning:** `ListingShell` already ships the light-blue "coming soon" placeholder, but its flex model is fixed-panel + growing-list — the opposite of a decorative whitespace-absorber (fixed content + growing panel). The placeholder also renders LEFT per the CD-039 Q2 client decision. So "reuse the shared shell" was wrong; Home got a bespoke two-column layout instead. Reach for a bespoke layout when the intent (decorative absorber) differs from the component's purpose (fixed utility panel).

**Pattern:**
Bespoke: `flex lg:flex-row` (no justify-center) · feed `lg:w-[640px] lg:shrink-0` · `<aside class="hidden lg:block lg:flex-1">` light-blue. Growth bounded by AppLayout's 1248px content-card cap.

## 2. The type tokens are correct; the usage discipline was missing
**Topics:** matt-design, typography

**Context:** SmartFeed cards showed drifting font-size/line-height; first read was "foundation bug" (lh=font-size on `text-body-default`).

**Learning:** `tokens-typography.css` documents two deliberate leading regimes — dense (12–14px body + all headers, lh 1.0: captions/labels/headings) and prose (16–20px body, lh 1.5: paragraphs). `lh:1` on `text-body-default` is *correct by design*; the fault is consumers using a caption token (`text-body-default`, 166 uses, 99 on `<p>`) for paragraphs while the paragraph token (`text-body-medium`) had 3 uses. Always check whether a "token bug" is actually documented intent being misapplied — the fix is a written discipline, not a token-values change.

## 3. Matt's scale has no 14px paragraph token — a real gap
**Topics:** matt-design, typography

**Context:** The app's de-facto body size is 14px, but Matt offers 14px only as a caption (lh 1.0).

**Learning:** Minting `text-body-default-prose` (14px/lh-1.5) fills the gap as a CC-ownership extension (post-Conv-289) — preserves the app's 14px density, fixes leading, and leaves Matt's caption regime untouched. Chosen over upsizing all paragraphs to Matt's 16px (client-visible) or changing the caption token's lh (breaks ~67 legit label/dense uses).

## 4. Component-level ledger SoT beats route-level for shared components
**Topics:** cc-workflow, docs-infra

**Context:** User worried "light + defer" rots with no mechanism to finish; named PALETTE-FDN's `[FAC-RECOLOR]` colour deferral as a shortfall a checkbox would have caught.

**Learning:** Route-level tracking can't express "this shared piece is pending across N routes" — a deferred shared component (FeedActivityCard) is invisible. A component-level ledger tracks each component **once**, with routes cross-referenced; route completion is *derived* (a route is swept ⟺ all its components ☑). This is the mechanism both the colour and type sweeps were missing. The audit must be persisted as a committed artifact (not left in chat scrollback) or it evaporates.

## 5. Latent NAV-RETROFIT spacing bug — `p-4`/`h-4 w-4` render at 4px
**Topics:** matt-design, astro

**Context:** Migrating DiscoveryCard to the unified card spec.

**Learning:** The Conv-174 bridge override (`--spacing-N` = Matt literal px) means legacy `p-4` = 4px card padding and `h-4 w-4` = 4px icons (near-invisible), not the 16px the legacy code assumed. Components mix overridden classes (`p-4`=4px) with Tailwind-default ones (`gap-3`=12px) in one file. The unified-card migration surfaces these as real visual bugs, not just token hygiene.
