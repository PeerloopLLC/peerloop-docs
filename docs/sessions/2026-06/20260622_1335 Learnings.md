# Conv Learnings - 2026-06-22

## 1. Tailwind v4 spacing: only the 10 bridge-defined steps are literal-px; the rest fall back to the 0.25rem base
**Topics:** matt-design, astro, route-sweep

**Context:** Conforming bridge-shrunk legacy spacing during CR-ANALYTICS. The Matt bridge (`tokens-tailwind-bridge.css` `@theme`) defines `--spacing-{4,8,12,16,20,24,32,40,48,64}` = literal px (number = px). There is **no base `--spacing` override**, so Tailwind v4's default 0.25rem base applies to every OTHER numeric step.

**Learning:** A utility like `w-80` is NOT 80px — 80 isn't bridge-defined, so it computes `calc(0.25rem × 80)` = **320px**. Only the 10-set renders as literal px. For any out-of-set px value, use an arbitrary `[Npx]` utility (`h-[288px]`, `w-[80px]`, `w-[192px]`). The conformed TeacherCourseView used this precise convention; TeacherAnalytics's bare `w-80`/`w-96` skeleton bars are a latent bug (render 320/384px, not 80/96px).

**Pattern:**
legacy Tailwind class `N` → intended px = `N×4` → bare Matt utility iff `N×4 ∈ {4,8,12,16,20,24,32,40,48,64}`, else arbitrary `[<N×4>px]`.

## 2. The Matt `neutral` scale has gaps — only {50,100,300,500,700,900}; 200/400/600/800 silently fall back to Tailwind greys
**Topics:** matt-design, route-sweep

**Context:** Mapping `secondary-400` during CR-EARN assess; discovered CR-ANALYTICS had used `neutral-400` in 13 places.

**Learning:** The bridge defines `--color-neutral-{50,100,300,500,700,900}`. Tailwind v4 ships a built-in `neutral` palette, so `neutral-{200,400,600,800}` resolve to Tailwind's defaults — visually fine greys, but NOT Matt-canonical (and not what the design intends). Map `secondary-400`/`-600` to the nearest Matt step (`neutral-300`/`neutral-500`/`neutral-700`), never the off-scale step. The /teaching sweep's documented fix is `neutral-400→neutral-300`.

**Pattern:**
allowed neutral steps = {50,100,300,500,700,900}; a leak-grep for conformance should flag `neutral-(200|400|600|800)` as off-scale.

## 3. Vite SSR "multiple copies of React" on cold dev-server start crashes authed island-heavy routes
**Topics:** dev-environment, astro, testing

**Context:** A fresh `npm run dev` rendered an empty `<body>` on `/creating/analytics`; the dev log showed "Invalid hook call / Cannot read properties of null (reading 'useState')" in `CurrentUserInit`/`AuthModalRenderer`.

**Learning:** The chunk hashes revealed the cause — React optimized as `v=430bf14b` but react-dom_server as `v=47cfeada` (mismatched Vite optimizeDeps versions = "multiple copies of React" → null hooks → SSR render crash → empty body). Fix: `rm -rf node_modules/.vite` + restart. It frequently still errors on the FIRST authed-route compile and self-resolves on the 2nd load once deps settle. Environmental, unrelated to any app code change.

## 4. The persistent :4321 dev server's client user-init can wedge on a long-running instance
**Topics:** dev-environment, testing

**Context:** First CR-ANALYTICS verify stuck on `useCreatorGate`'s loading spinner forever despite a valid dev-login session (server-side rendered the creator nav). `getCurrentUser()` returned null because `initializeCurrentUser` (the `CurrentUserInit` headless island) never completed → `authStatus` stuck `'loading'`.

**Learning:** Clearing the stale `peerloop_user_cache` localStorage didn't unstick it; a fresh server process did. When DOM-verifying client-gated islands and the gate spins indefinitely, suspect a wedged long-running dev server, not the conformance.

## 5. Route-sweep scope must be assessed over the full render tree, not the orchestrator's line count
**Topics:** route-sweep, workflow

**Context:** CR-ANALYTICS was filed "378 ln / 18 leg / quickest" (orchestrator only); the real route was ~199 occ across 9 files (8 sub-components held the bulk).

**Learning:** A thin orchestrator that delegates to many sub-components drastically undercounts when measured by its own size. Before committing to a per-route scope, trace the import graph + grep tokens across every rendered child. (Also verify whether those children are shared — here all 8 were CreatorAnalytics-exclusive, so no ripple to conf-OUT AdminAnalytics.)
