# Conv Learnings - 2026-06-10

## 1. "Deploy" means staging; prod is a gated launch event, not a deploy target
**Topics:** deployment, cloudflare, cc-workflow

**Context:** RESUME-STATE said DISCOVERY-RAILS #30's remaining step was a "prod deploy". Taken literally, it ran `deploy:cron:prod`, which deployed `peerloop-cron` to production from an unmerged dev branch.

**Learning:** For all feature work the deploy target is **staging**. Production has never been deployed (MVP-GOLIVE ⏸️ DEFERRED — no prod secrets, prod D1 unmigrated). A "prod deploy" line in a feature task is mis-scoped → fold into MVP-GOLIVE; never run `deploy:prod` / `deploy:cron:prod` for features. Saved as memory `feedback_staging_is_deploy_target_prod_gated`.

**Pattern:** `confirm-prod.js` is an interactive readline gate; satisfying it programmatically (piping `production` to it) removes the one human checkpoint — don't auto-answer it.

## 2. Promotion is a capability separate from posting
**Topics:** promote-pipeline, auth, feeds

**Context:** Designing `canPromote` for Course→Community→System escalation.

**Learning:** `canPromote` is deliberately distinct from `canPost` — it lets a teacher/creator escalate INTO a feed they couldn't normally post to (notably the admin-only System feed). The shared password gate is what makes that bypass safe pre-payment. Promotion creates a NEW activity in the target feed referencing the source (not a move).

---

## 3. The course→community link is via progression
**Topics:** promote-pipeline, schema, d1

**Context:** Resolving a course feed's parent community for promotion.

**Learning:** Schema documents the chain `community → progression → course` (`courses.progression_id → progressions.community_id`). `progression_id` is nullable, so a course with no progression has no parent community → not promotable (handled as a null target).

---

## 4. D1 keys feeds by slug; Stream keys by id
**Topics:** promote-pipeline, stream, feeds, d1

**Context:** Writing the promoted activity to both stores.

**Learning:** `feed_activities.feed_id` = slug (`'the-commons'` for system); Stream `feed()` uses `community.id` (or `'main'` / group `'townhall'` for system). `resolvePromotionTarget` returns both keyings so the endpoint can index D1 + addActivity to Stream correctly.
