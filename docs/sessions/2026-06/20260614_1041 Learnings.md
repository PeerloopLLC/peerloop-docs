# Conv Learnings - 2026-06-14

## 1. Stream.io feed groups are declared in the dashboard, not created on write
**Topics:** stream, cloudflare, infra

**Context:** Renaming the system feed's Stream group `townhall`→`system` produced a runtime 500: `Stream API error: 400 - "system feed group does not exist"`. Re-seeding could not create the group.

**Learning:** Stream.io feed *groups* (e.g. `townhall`, `community`, `course`) are app-level configuration registered in the Stream dashboard. Code can only *use* an existing group; pointing at an unregistered group 400s with `FeedConfigException`. This is the one truly irreducible piece of "pre-production rename risk" — unlike a D1 slug (fully controlled via seed), the Stream group is external config. Resolution: keep the value `townhall` behind a named, documented constant (`SYSTEM_STREAM_GROUP`) so the name is gone from the working surface while the infra value stays valid.

**Pattern:**
When renaming any identifier that doubles as external-service config (Stream group, Stripe product id, R2 bucket), check whether the new value must be *provisioned* in the external service first — `db:seed` only covers what we own.

## 2. A single id↔slug substitution covers both when id = prefix + slug
**Topics:** testing, cc-workflow

**Context:** The system community's `id` is `comm-the-commons` and `slug` is `the-commons`. A naive rename would touch both separately.

**Learning:** Because `id` = `comm-` + `slug`, a single `s/the-commons/system/g` correctly produced `system` (slug) AND `comm-system` (id) AND `/community/system` (URL) in one pass. Recognizing the structural relationship between two identifiers collapses two substitutions into one and removes a class of inconsistency bug.

## 3. Verify what a token *means* before mass-renaming it
**Topics:** cc-workflow, testing

**Context:** 258 `townhall` hits looked like a huge blast radius. A blind rename would have been wrong.

**Learning:** `FeedsHub.tsx:53` is `allFeeds.find(f => f.type === 'system')` — the data model already used `type:'system'`; `townhall` survived only as local var-names and comments. Sampling the actual usage (quoted-literal vs identifier vs comment) before sed-ing let me split functional Stream-group literals (must change, but reverted) from cosmetic names (safely deferred). The quoted-vs-unquoted distinction (`'townhall'` = Stream group value; bare `townhall` = var name) was the reliable functional/cosmetic discriminator.
