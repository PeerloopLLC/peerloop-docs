# Conv Learnings - 2026-06-19

## 1. The `[Npx]`-only spacing grep undercounts legacy numeric-scale classes
**Topics:** css, matt-design, route-sweep

**Context:** Conv 301 flagged that an arbitrary-`[Npx]`-only grep missed the NotificationSettings spacing problems ([SWEEP-SPACING-GREP] #32). Applied the lesson this conv.

**Learning:** Two distinct failure modes hide behind numeric utilities: (a) **collapse bugs** — classes whose number is in the NAV-RETROFIT bridge's aliased set `{4,8,12,16,20,24,32,40,48,64}` render at Matt literal-px (`py-4`→4px, knob `h-4 w-4`→4px), a real visual bug; (b) **legacy-but-fine** — numbers NOT in the set (`px-6`, `space-y-6`) fall back to Tailwind's `calc(--spacing * n)` and render correctly (24px), only failing the no-legacy-scale conformance rule. A sizing audit must enumerate ALL numeric utilities and classify each, not just arbitrary `[Npx]`.

## 2. `--h4` is 16/500 but wrong for body labels (regime mismatch)
**Topics:** typography, matt-design

**Context:** Looking for an existing 16/500 token before minting.

**Learning:** `--h4` IS 16px/500 — but in the **header** regime (lh 1.0, ls 0) and semantically a heading. A form-control label needs the **body** regime (lh 1.5, ls -0.022em). Size+weight match is not enough; regime + semantics decide the token. This is why minting a body-family 16/500 (`text-body-medium-medium`) was correct over reusing `text-h4`.

## 3. Verify a newly-minted Tailwind v4 utility actually generates
**Topics:** css, testing, matt-design

**Context:** Minted `--body-medium-medium` and registered `--text-body-medium-medium` in the bridge.

**Learning:** Tailwind v4 auto-generates `text-<name>` from a `--text-<name>` namespace entry. A mis-registration silently no-ops (the class does nothing; text falls back to inherited). After minting, grep the built CSS (`dist/client/_astro/global.*.css`) for the generated rule and confirm its property wiring — don't assume registration worked. DOM-truth then confirms the cascade applies on the live page.

## 4. Component provenance is a separate axis from page provenance
**Topics:** matt-provenance

**Context:** Initially stamped `@matt-inspired` (a page marker) on the NotificationSettings subcomponent during the restyle.

**Learning:** The 3-marker convention (`@stand-in`/`@matt-source`/`@matt-inspired`) is for **pages**; page markers don't propagate to child components. Component provenance (`@matt-source` / `data-prov`) is a distinct axis owned by [PROV-STAMP-GAPS]. Reframed to a plain restyle doc-comment instead.

## 5. The project uses vitest, not jest
**Topics:** testing

**Context:** Ran `npx jest <test>` and got a Babel parse error (0 tests, "1 failed").

**Learning:** Raw `npx jest` pulls its own Babel resolver (wrong transform for the project's TS/TSX) and fails to parse. The project's test command is `vitest run`. Run a single file via `npx vitest run <path>` (or `npm test -- <path>`), which uses the configured transform + better-sqlite3 global setup.
