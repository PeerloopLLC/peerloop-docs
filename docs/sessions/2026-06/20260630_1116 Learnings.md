# Conv Learnings - 2026-06-30

## 1. The SoT's frozen-list, not the grep count, scopes a concept-retirement sweep
**Topics:** cc-workflow, docs-infra

**Context:** Retiring `conv-tasks.md` / the RESUME-STATE-as-task-store model app-wide. A raw grep returned hundreds of hits across TIMELINE.md, `docs/sessions/**`, and DOC-DECISIONS.md.

**Learning:** PLAN.md § CURTASKS line 112 explicitly enumerated the frozen set (`docs/sessions/**` + `docs/decisions/*` + historical records). Honoring that collapsed the raw hit-count down to a small live edit surface (skills/config/active-docs/memory). When a plan pre-declares what stays frozen, that list is the scoping authority — don't re-derive scope from grep volume.

---

## 2. Retire-and-replace beats rewrite-in-place when the FILENAME encodes the retired model
**Topics:** cc-workflow, docs-infra

**Context:** Two core memory files (`feedback_resume_state_as_todowrite_persistence`, `feedback_conv_tasks_live_sync`) described behaviors the cutover retired.

**Learning:** Their filenames themselves asserted the wrong model (RESUME-STATE *is* TodoWrite persistence; conv-tasks *live-sync*). Rewriting content under a contradictory name is worse than deleting both and creating one accurately-named file (`feedback_current_tasks_persistence`). Memory-discipline's "delete wrong memories" extends to the slug, not just the body.

## 3. A role-aware "wrong-model" grep catches references a literal grep misses
**Topics:** cc-workflow, docs-infra

**Context:** A plain `conv-tasks` grep found the obvious mentions, but the cutover also changed RESUME-STATE's *role* (task-store → narrative-only).

**Learning:** A second grep for `RESUME-STATE` co-occurring with `todowrite|transfer|remaining|ledger` surfaced 4 more incidental memory edits (alerts persisting to RESUME-STATE; `/r-start` RESUME-STATE→TodoWrite transfers; watch-tasks reloaded from RESUME-STATE) that named no retired file but still described the retired behavior. Sweep by the changed *concept's role*, not just its renamed *file*.

---

## 4. /r-start Step 2.5 self-update check false-positives on a no-op pull
**Topics:** cc-workflow

**Context:** The pull reported "Already up to date" yet Step 2.5's `HEAD@{1}` diff flagged the r-start SKILL.md as changed.

**Learning:** A no-op ff-only pull doesn't move HEAD, so `HEAD@{1}` points at whatever the last HEAD-moving op was — making the diff compare against a stale baseline and false-flag. The directive's "when in doubt, re-read the on-disk file" resolved it at the cost of one cheap Read; the body was current. Benign-by-design, not a bug.
