# Conv Decisions - 2026-06-16

## 1. Cross-project launch guard lives in the alias, symmetric for both projects
**Type:** Implementation
**Topics:** cc-workflow, dual-repo

**Trigger:** `spt` accidentally launched from a Peerloop terminal trampled a concurrent spt session (~30 min of coordinated cleanup).

**Options Considered:**
1. Guard in r-start
2. Guard in the `spt`/`peerloop` shell functions, symmetric ← Chosen
3. Minimal — guard only `spt`

**Decision:** Shared `_guard_launch` helper in `~/.zshrc`; `spt` refuses to launch inside Peerloop dirs and `peerloop` refuses inside spt dirs.

**Rationale:** The cwd signal only exists before the alias's `cd`; symmetric coverage handles the mirror-image footgun; hard-block with a "cd out to override" message.

**Consequences:** Machine-local (not in repo) → must be hand-applied on the M4; guard block saved to `.scratch/zshrc-launch-guard-for-M4.md` for that.

## 2. Keep the two Minis' shell configs isolated (no dotfiles sync)
**Type:** Strategic
**Topics:** dual-repo, cc-workflow

**Trigger:** `.zshrc` edits don't propagate; raised whether to sync dotfiles across the two Minis.

**Decision:** Leave the two Minis isolated for now; hand-apply machine-local changes.

**Rationale:** User preference; avoids introducing a dotfiles-sync mechanism.

**Consequences:** Machine-local shell changes need manual replication; in-repo changes (skills/scripts) still sync via git.

## 3. Concurrent-session lock: machine-local, PID+session-id ownership, prompt-on-stale
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** A second-terminal r-start trampling the active session (Conv 293 incident).

**Options Considered:**
1. cwd-based detection in r-start (impossible — see Learning 1)
2. Machine-local PID/session-id lock file ← Chosen
3. Committed/in-repo lock (rejected — PIDs meaningless cross-machine, git noise)

**Decision:** `active-session.lock` under `~/.claude/projects/<slug>/`; r-start Step 0 halts on ACTIVE, prompts on STALE, acquires on CLEAR; r-end Step 8 releases (owner-checked).

**Rationale:** PIDs are machine-meaningful and the Minis are isolated; self-heals via dead-PID detection; never false-blocks the `/clear` re-run (own-PID/own-session recognised).

**Consequences:** Guards same-machine concurrency (the actual incident); cross-machine simultaneity still caught only by the existing non-ff push-fail halt.
