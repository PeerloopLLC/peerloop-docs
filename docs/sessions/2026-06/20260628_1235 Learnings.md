# Conv Learnings - 2026-06-28

## 1. Bash `case "$x" in $VAR)` Does NOT Alternate When `$VAR` Expands to `a|b|c`
**Topics:** cc-workflow, testing

**Context:** `sync-gaps.sh`'s shared-basename guard used `case "$basename" in $SHARED_BASENAMES_PATTERN)` where `$SHARED_BASENAMES_PATTERN` expanded to a pipe-joined string like `a|b|c`. The guard silently never matched, so the basename-fallback masked undocumented test files (this is the root cause of the Conv-345 `download.test.ts` sync gap).

**Learning:** Bash parses `case`-pattern `|` alternation at **parse time — before** variable expansion. So pipes that arrive via an expanded variable are treated as **literal glob characters**, not alternation operators. `case "$x" in a|b|c)` works (literal pipes in source); `case "$x" in $VAR)` with `VAR="a|b|c"` matches the literal string `a|b|c`, never `a`/`b`/`c` individually. A guard built this way is silently always-false.

**Pattern:**
```bash
# BROKEN — expanded pipes are literal globs, never alternate
SHARED_BASENAMES_PATTERN="a|b|c"
case "$basename" in $SHARED_BASENAMES_PATTERN) ... ;; esac

# FIXED — array + membership helper
SHARED_BASENAMES=(a b c)
is_shared_basename() {
  local n=$1
  for s in "${SHARED_BASENAMES[@]}"; do [[ "$n" == "$s" ]] && return 0; done
  return 1
}
```
A `SYNC_GAPS_LIB_ONLY` sourcing hook lets a calibration harness exercise the helper directly. Prove a detection-guard fix on the canonical case before commit (BROKEN→masked, FIXED→flagged), per `feedback_heuristic_calibration` — done here as `test-drift-detection.sh` Test 7.

## 2. "Backend-Ready, UI-Only" Carry-Over Claims Must Be Verified Per-Actor Incl. the Enumeration Path; and Check Whether a Per-Entity Surface Already Exists
**Topics:** testing, workflow, auth

**Context:** RESUME-STATE carried HW-GRADE-CREATOR as "backend ready, no per-course creator view exists." Exploration showed both halves of the premise were slightly off: the list/submission/grade endpoints already authorize the course **creator** (endpoint-tested Conv 346), and a creator per-course surface **already existed** (`CourseEditor` via `/creating/studio?course=<id>`).

**Learning:** Treat a carried "backend-ready, UI-only" claim as a hypothesis, not a fact (mirrors §Baseline-Verification Rule 2). Verify it per-actor and include the **enumeration/list path**, not just the action endpoints — a creator who can grade per-assignment but has no list endpoint still has no usable surface. Equally, before assuming a new route/view is needed, search for an existing per-entity surface to mount onto; the parity work collapsed from "new view" to "new tab in CourseEditor" once the existing surface was found.

## 3. PLATO Browser-Mode `BrowserIntent` DSL Is Intentionally Human-Prose — Automating It as a Deterministic CI Gate Reinvents Playwright
**Topics:** testing

**Context:** Scoping "automate PLATO browser mode into a CI gate" (Option 2 of the testing-strategy pivot) surfaced a decisive blocker before any code was written.

**Learning:** PLATO's `BrowserIntent` (`pageAction` / `expect` as free text — `types.ts` + the Conv-074 BrowserIntent decision) is **deliberately** human-prose: an agent interprets it via the Chrome bridge. The app-customization that makes PLATO browser mode valuable lives in exactly this **non-automatable** prose layer. Two automation paths both fail as a gate: (a) re-authoring every walkthrough with machine selectors + programmatic assertions converges on "Playwright with extra steps" (you've rebuilt Playwright under a bespoke runner); (b) an LLM runner that interprets the prose is non-deterministic, so it is a smoke-walker, not a deterministic gate. The prose DSL is a feature, not a missing affordance — don't try to "finish" it into a gate.

## 4. Playwright E2E Asserts Presence/Text/Structure (Behavioral Regression) — NOT Visual/UX Quality
**Topics:** testing

**Context:** Evaluating what dropping the 28 stale Playwright specs would actually cost, to avoid silently losing a coverage category.

**Learning:** Playwright E2E is behavioral/integration regression coverage — it asserts element presence, text content, and DOM structure. It does **not** catch visual or UX-quality problems (layout, spacing, design-token correctness, "does this look right"). That coverage comes from the Matt design-conformance sweep + manual review. So deprioritizing Playwright loses *automated behavioral* coverage only; UX-quality coverage was never Playwright's job and is unaffected. Name the coverage category precisely before deciding a test layer is or isn't redundant.
