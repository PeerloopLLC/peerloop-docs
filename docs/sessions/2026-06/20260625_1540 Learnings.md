# Conv Learnings - 2026-06-25

## 1. Pre-computed /r-start counter is a snapshot — re-read after the pull
**Topics:** cc-workflow, dual-repo

**Context:** /r-start's pre-computed `CONV-COUNTER` said 333, but Step 2's pull brought in 2 convs from the other machine (MacMiniM4 ran 334 + 335), making the snapshot stale by 2.

**Learning:** When the other machine has run convs since this machine's last sync, the pre-computed counter (rendered at invocation, pre-pull) under-counts. After the ff-only pull, re-read `CONV-COUNTER` (and `RESUME-STATE.md`) from disk before incrementing — the snapshot is a hypothesis, the post-pull file is truth. The session lock acquired from the stale value must also be re-acquired at the corrected number.

**Pattern:** Treat every pre-computed `/r-start` value as pre-pull; cross-repo catch-up invalidates the counter, branch state, and RESUME-STATE snapshot together.

## 2. Chart/data-viz palettes are honest-orphans — only chrome tokenizes
**Topics:** matt-design, route-migration, conformance

**Context:** `/admin/analytics` was flagged as "the genuinely-different route" with a novel multi-series chart-palette decision (revenue split, funnel, topic distribution, progress buckets).

**Learning:** The "novel" concern dissolved once the already-conformed workspace charts were read: they keep chart series colours as explicit hex (`#2563eb`/`#059669`) with a documented "data-viz convention; not Matt-tokenized" comment, and legend swatches mirror the hex via `bg-[#hex]`. The conformance rule for chart components is: **tokenize the chrome (cards, headers, KPI values, status hues, skeletons), keep the data-viz palette as honest-orphan hex.** No palette invention needed — mirror the family.

**Pattern:** When conforming a charts/metrics surface, read a sibling already-conformed chart first; copy its hex-kept + commented convention rather than mapping series colours to semantic tokens.

## 3. "Zero backward-pointer" predicted analytics effort despite a different shape
**Topics:** route-migration, conformance

**Context:** `/admin/analytics` looked daunting (7 files, ~223 hits) but the chart primitives it composes were already conformed under RG-WORKSPACES.

**Learning:** The Conv-332 "zero backward-pointer" predictor (effort = does it first-render an unconformed shared primitive?) held even for a structurally-different route: the section wrappers had raw chrome, but every chart/table primitive + DateRangeSelector was already swept, so the work was bounded to the 7 wrapper files. Cross-group reuse (workspace analytics ↔ admin analytics) collapses backward-pointer cost.

## 4. App-wide footer links can 404 silently under route-404-honesty
**Topics:** route-migration, ux

**Context:** Conforming `Footer.astro` surfaced that its root marketing hrefs (`/privacy`, `/terms`, `/help`, …) all 404 because RG-PUBLIC pages live only in `/old`.

**Learning:** The route-404-honesty policy (unconverted → 404, no redirect layer) interacts with shared chrome: a footer rendered on every page can point at not-yet-ported routes and 404 sitewide. This is by-design while the group is deferred, but it's invisible until someone DOM-checks the footer. Worth recording as accepted-pending-redesign so it isn't re-discovered as a bug.
