# Conv Learnings - 2026-06-23

## 1. `.scratch/` is machine-local — only git-tracked artifacts cross machines
**Topics:** dual-repo, cc-workflow

**Context:** Conv 327's deep-verify plan (`.scratch/2026-06-23-rtmig4-reconciliation-deepverify.md`) and its resumable agent IDs were referenced by the synced RESUME-STATE but were absent on M4Pro.

**Learning:** `.scratch/`/`_scratch/` is gitignored + machine-local; plan files, agent IDs, and any scratch artifact do NOT propagate across machines. Only git-tracked files (RESUME-STATE, committed code/docs) cross. A RESUME-STATE that points at a `.scratch/` plan is a dangling cross-machine reference. Resumable agent IDs are doubly machine-local — the agent process lives on the spawning machine.

## 2. An unpushed conv's committed work is invisible despite a "clean" pull — verify hashes
**Topics:** dual-repo, cc-workflow

**Context:** Conv 328 ran on M4, shipped 4 commits, but never `/r-end`ed (API 500). M4Pro's `git pull` said "Already up to date" and the branch-check said MATCH, yet the work wasn't there.

**Learning:** "Already up to date" only means local == origin's *remote-tracking ref*; it says nothing about another machine's unpushed local commits. When a plan claims commits exist, verify with `git cat-file -t <hash>` / `git log <hash>` rather than trusting a clean pull. The symptom of stranded work is files in the un-migrated state while a doc claims they're done.

## 3. Salvaging a paused /r-end: plain `git push` of already-made commits is API-independent
**Topics:** cc-workflow, dual-repo

**Context:** M4's /r-end was stuck at the Step 4d checkpoint with a persistent Anthropic API 500; 4 conformance commits were already made.

**Learning:** Committed-but-unpushed work needs no Claude/API to rescue — a plain `git push` from a terminal sends it. Don't wait for a broken r-end to recover. **Ordering matters:** get the stranded machine's commits to origin BEFORE the live machine adds new commits on top, or the paused r-end can no longer fast-forward-push afterward (non-FF) and its docs close must be done manually.

## 4. Display-avatar regime: inherited-size glyphs render at the 16px base; snap to nearest tier
**Topics:** matt, astro

**Context:** #28's three 40px avatar initials had no explicit text-size (inherited). The body (`global.css`) sets only color, no font-size → 16px browser base. The §9.2c regime is 12/14/18/24/30 (xs/sm/md/lg/xl) — no 16px token.

**Learning:** An unsized glyph inherits the 16px base, which has no exact display-avatar token (sm=14 / md=18 gap). For a 40px circle (between UserAvatar's 32px→sm and 48px→md tiers), the call is consistency-driven: "ties round up" is **spacing-axis-only** (ledger §170 carve-out), so type doesn't borrow it — match the established precedent (Conv 328's 32/36px→sm) and snap to `sm`. `UserAvatar.tsx` sizeClasses is the canonical circle→token map.

**Pattern:** Inherited-size glyph (no explicit text-size) = a Tier-2 display-regime case; resolve the inherited px statically (body base = 16px), then snap to the nearest UserAvatar tier, consistency-first (type does not borrow the spacing "ties up" rule).
