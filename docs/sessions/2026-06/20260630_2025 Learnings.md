# Conv Learnings - 2026-06-30

## 1. Astro named slots need a direct child of the component
**Topics:** astro, layout

**Context:** Moving the community/course persistent headers above the SubNav.

**Learning:** A `slot="entity-header"` attribute is only projected when the element is a *direct child* of the component invocation — an element nested inside `<article>` (the default-slot content) is NOT hoisted. Had to physically relocate the header markup to be a sibling of `<slot name="sub-nav">`.

---

## 2. Constrain-container measurement is unreliable for responsive layout
**Topics:** css, testing, browser-automation

**Context:** Verifying the sub-466px overflow fix on a bridge stuck at a wide viewport.

**Learning:** Setting `main.style.width = '343px'` does NOT simulate a mobile viewport — media queries (`sm:`, `md:`, `min-[480px]:`) key off the actual viewport, so responsive classes stay in their desktop state and the measurement lies (image still 224px, nav still flex). Verify a responsive layout by **forcing the mobile-state classes** (image size, `display:grid`, `min-width:0`) via inline style at whatever viewport, then measure fit.

**Pattern:**
`el.style.width='96px'; nav.style.display='grid'; [...nav.children].forEach(c=>c.style.minWidth='0'); main.style.width='343px'; measure scrollWidth<=clientWidth`.

## 3. Flex/grid items overflow without `min-w-0`; fixed inline images are the usual culprit
**Topics:** css, layout

**Context:** The hero and nav horizontal overflow below ~466px.

**Learning:** `grid-cols-2` = `repeat(2, minmax(0,1fr))` still blows out if the grid items keep `min-width: auto` (their min-content). A flex text column beside a fixed-width image (`w-56` = 224px) with no `min-w-0` cannot shrink → horizontal spill. Fix = `min-w-0` on the shrinkable columns/items + a responsive (smaller) image on mobile.

## 4. This project's spacing scale is split (aliased vs standard)
**Topics:** css, tailwind

**Context:** Sizing the mobile hero image; reading `px-64`.

**Learning:** Utilities in the bridge's aliased set `{4,8,12,16,20,24,32,40,48,64}` render as literal px (so `px-64` = 64px each side), while others use standard Tailwind (`w-56` = 224px, measured). Don't assume the scale — measure, or use arbitrary values (`w-[96px]`) when a precise mobile size matters.

---

## 5. DOM-truth beats screenshot inference (reaffirmed)
**Topics:** browser-automation, plato

**Context:** "menu bar is much narrower than the screen width."

**Learning:** Measurement showed the nav container was full-width (hero and nav shared exact left/right edges); the "narrow" look was content-width tabs clustering left in a full-width bar, and the DevTools overlay's purple hatch was flexbox dead space. The container was never narrow — the fix was to make the tabs *fill*, not to widen a container.
