# Conv Decisions - 2026-06-28

## 1. Mount the grading UI as a Homework tab in TeacherCourseView (not a dedicated route)
**Type:** Implementation
**Topics:** routing, homework, ui

**Trigger:** A new grading surface needed a home; `routing-addressability-first` says decide addressability deliberately.

**Options Considered:**
1. Dedicated addressable route `/teaching/courses/[courseId]/homework`
2. New "Homework" tab inside `TeacherCourseView` (internal state) ← Chosen
3. A section within the existing Students tab

**Decision:** Add a Homework tab to `TeacherCourseView`, rendered like the Feed tab (`<HomeworkGradingPanel courseId={...} />`).

**Rationale:** `TeacherCourseView` tabs are internal React state (not URL-addressable), and the new tab follows that established pattern exactly. No notification deep-link or redirect target needs the grading surface yet, so addressability isn't required; consistency wins.

**Consequences:** Per-tab deep-linking is not available (matches every other tab). If a "new submission to grade" notification deep-link is later wanted, the tab would need promoting to an addressable sub-route.

## 2. Teacher-first scope; defer creator-side parity
**Type:** Strategic
**Topics:** homework, scope, roles

**Trigger:** Reviewer endpoints permit creator OR active teacher, but `TeacherCourseView` is teacher-only and no per-course creator view exists.

**Options Considered:**
1. Teacher-first, defer creator ← Chosen
2. Teacher + creator both this conv
3. Endpoint + minimal UI only

**Decision:** Build the teacher surface this conv; track creator parity as `[HW-GRADE-CREATOR]` (#16).

**Rationale:** Teacher-first is cleanly completable in one conv. Creator parity needs a NEW creator-side per-course surface (not a rebuild) — the shared endpoint already permits creator auth, so it's a thin later mount.

**Consequences:** Creators cannot yet grade via UI; follow-up tracked in PLAN.md + TodoWrite (#16).

## 3. New endpoint auth = reviewer predicate (creator OR active teacher), not the lighter resources.ts predicate
**Type:** Implementation
**Topics:** api, auth

**Trigger:** The `resources.ts` teaching endpoint uses teacher-cert-only (no creator, no `is_active`); the reviewer endpoints use creator OR active teacher.

**Decision:** The new assignment-list endpoint copies the reviewer predicate (`submissions/index.ts:75-85`).

**Rationale:** An assignment-list endpoint must be visible to exactly the set of people who can act on what it lists — otherwise a creator could grade but not see the list, or the list shows ungradeable rows. Align the predicate to the action it feeds.

**Consequences:** Creator + active-certified-teacher see the list; inactive teachers and students get 403 (tested).

---

## 4. Dedicated new endpoint, not an extension of the existing teaching-course payload
**Type:** Implementation
**Topics:** api, homework

**Trigger:** The assignment list could be folded into the existing `GET /api/teaching/courses/[courseId]` payload.

**Decision:** New `GET /api/teaching/courses/[courseId]/homework` mirroring `resources.ts`.

**Rationale:** Mirrors the established teaching sub-resource pattern (resources.ts), keeps the already-large course payload lean, and lets the grading tab lazy-fetch.

**Consequences:** One more endpoint to maintain; consistent with the existing structure.
