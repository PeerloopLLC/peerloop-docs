# Conv Learnings - 2026-06-22

## 1. Decompose a Large Opaque Unit by Its Mount Tree — Verification Composes Leaves-Before-Shell
**Topics:** matt-conformance, route-sweep, decomposition

**Context:** CR-STUDIO arrived as one ~4,726-line 🔴 [Opus] blocker flagged "needs sub-decomposition before conforming."

**Learning:** The component import/mount hierarchy *is* the cohesion-based decomposition. Splitting CreatorStudio→{CreateCourseModal, CourseEditor→{3 leaf editors}} into 5 units made each an independently conformable + DOM-verifiable slice. Critically, the *verification* composed: the 3 leaf editors (B/C/D) were verified with a leak-check **scoped** to their own subtree (excluding the still-legacy CourseEditor chrome), then the shell (E) was verified with an **unscoped** full-page sweep — at which point any residual leak was unambiguously the shell's own. Doing leaves-before-shell is what makes the final unscoped `leak = 0` a real end-to-end certification.

**Pattern:**
`root.querySelector('.<conformed-marker>')`-scoped leak count per leaf while the parent chrome is still legacy; whole-`document` unscoped leak count once the container/chrome unit lands.

## 2. When a Fast Local Signal Is Ambiguous, Verify Against the Authoritative Source (git)
**Topics:** memory, reconciliation, r-start

**Context:** r-start's no-shrink reconciliation flagged 7 codes dropped from `conv-tasks.md` (33→29). The `*DONE*`-prefix heuristic falsely reported even *known*-completed tasks (CR-APPLY etc.) as "not done."

**Learning:** The companion `conv-tasks.md` wasn't live-synced during Conv 321 (its header still read "Conv 321"), so its `*DONE*` markers were stale/absent. The reliable signals were (a) the RESUME-STATE `## Completed` ledger and (b) **git commit history** — which confirmed CR-ANALYTICS/CR-EARN were swept in Conv 322 (`e1a25942`/`9d7550ba`). Don't trust a derived local marker when the source of truth (git) is one grep away.

## 3. /r-start Ending on an Options Picker Is the Success Terminus, Not a Fault
**Topics:** cc-workflow, r-start

**Context:** User was unsettled that r-start finished by presenting selectable options, and inferred something had gone wrong (skipped r-end / unclear memory state).

**Learning:** r-start Step 8 (Recommended Action) ALWAYS ends by asking whether to start the resume task; the picker is the project's AskUserQuestion convention replacing inline yes/no. A genuine r-start problem surfaces as a ⚠️/⛔ HALT (dirty repos, pull failure, stale lock, data-loss memory diff). A clean picker = green light. Worth stating plainly when a user reads the picker as a warning signal. (Conv 324 cross-machine handoff: Conv 323 closed properly on MacMiniM4Pro, Conv 324 opened on MacMiniM4, memory synced 0-diff — no mistake, and the still-open CR-* tasks were correctly distinguished finished-drop from forgot-lose.)
