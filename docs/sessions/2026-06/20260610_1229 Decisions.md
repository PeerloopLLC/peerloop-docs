# Conv Decisions - 2026-06-10

## 1. SYS-RENAME boundary: C (mechanical rename) then A (admin-only lockdown), announcements deferred
**Type:** Implementation
**Topics:** smart-feed, access-control, feeds-redesign

**Trigger:** Client adopted the feeds model; SYS-RENAME is the foundation, but "member feed = System Announcements only" depends on an Announcement data model that doesn't exist.

**Options Considered:**
1. A — lockdown + defer announcement fan-out ← Chosen (for the lockdown half)
2. B — lockdown + minimal `is_announcement` column now
3. C — mechanical rename only ← Chosen first, then A

**Decision:** Do C (enum rename, no behavior change) first, then A (admin-only lockdown). Defer the Announcement data model + member/visitor fan-out to [ADMIN-FEED-UI] #33.

**Rationale:** Keeps each step small/verifiable; the announcement model genuinely belongs to ADMIN-FEED-UI; avoids a half-built announcement column.

**Consequences:** Interim — members get NO System broadcast until [ADMIN-FEED-UI] ships. Acceptable pre-launch. Cosmetic route/Stream/label rename split into [SYS-NAMING] #36.

## 2. Promotion launch gate = shared password (admin-managed via /admin/*), payment deferred
**Type:** Strategic
**Topics:** feeds-redesign, promotion, admin

**Trigger:** Client approved promotion policy: free at launch, gated by a password.

**Decision:** Promotion is free to everyone but gated behind a stored shared password, changeable only by Admins via an /admin/* interface. Stripe payment deferred to a later phase.

**Rationale:** Lightweight launch access-control (admins distribute/rotate the password to trusted promoters) without building payment first.

**Consequences:** Recorded on [PROMOTE-PIPELINE] #32 with 4 OPEN clarifications (global-vs-per-level, per-promotion-vs-session, storage+hashing, which levels gated) to resolve before building it. The /admin/* password UI folded into [ADMIN-FEED-UI] #33.
