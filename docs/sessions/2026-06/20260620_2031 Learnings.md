# Conv Learnings - 2026-06-20

## 1. Conv 312's "5 gates green" claim never ran the full suite
**Topics:** testing, baseline-verification

**Context:** Conv 313's full-suite run found 8 failures; stash-verify against Conv-312 tip `c53608a1` proved all 8 were already red at Conv 312's close.

**Learning:** Conv 312's RESUME-STATE "all 5 gates green" was based on tsc/check/lint/tailwind/build + the **targeted** `ModeratorQueue.test`, NOT the full suite — so 8 pre-existing failures hid behind a carry-forward baseline claim. Exactly the Conv 101→102 failure class (CLAUDE.md §Baseline Verification Rule 2). A "gates green" claim that omits the full `npm test` is not a real baseline.

**Pattern:** When a route-sweep tranche claims green, run (or explicitly scope-out) the full suite — a targeted test pass is not the suite.

## 2. vi.mock can't override a same-module internal call — mock the consumed export
**Topics:** testing

**Context:** The dismiss tests fail because `readDismissed()` internally calls `dismissalPersists()`, which returns false in DEV/test.

**Learning:** Mocking the `dismissalPersists` *export* would NOT change `readDismissed`'s internal call (ESM internal calls bind to the local, not the export). The fix is to mock the export the **component** imports — `readDismissed` itself — so the override actually intercepts the component's behavior.

**Pattern:**
```js
vi.mock('@lib/ephemeral-dismiss', async io => ({
  ...await io(),
  readDismissed: k => localStorage.getItem(k) === '1',
}))
```
To redirect behavior in a test, mock the export the component-under-test imports, not a helper that export calls internally.

## 3. ephemeral-dismiss makes dismiss tests environment-dependent
**Topics:** testing, cc-workflow

**Context:** OnboardingNudgeBanner / RecommendedCommunities / RecommendedCourses dismiss tests render the banner even when localStorage says dismissed.

**Learning:** Per Conv 292 (`project_ephemeral_dismiss_dev_staging`), `dismissalPersists()` returns false on DEV/localhost/staging by design, so `readDismissed()` always returns false there — including the vitest jsdom env (hostname `localhost`). Tests asserting "hide-on-dismiss" verify the **prod** persistence path and must opt into it (mock the persist path); the dev-ephemeral behavior is correct, not a bug.

## 4. Browser DOM-verify is invisible to a human watching Chrome
**Topics:** plato, cc-workflow

**Context:** User watched Chrome during a browser-verify and saw little/lagging activity.

**Learning:** DOM-truth verification (`getComputedStyle`/`querySelector`/`fetch` in page context) produces no on-screen motion — a whole "page" of checks runs with the window static. Combined with the MCP operating in its own tab, background dev-login fetch, and settle-wait + bridge latency, the visible blips lag the transcript. This is deliberate (DOM-truth > screenshots) but opaque to watch. If a user wants to follow along, drive the visible UI or capture screenshots/GIF (slower, less precise) — user here opted to keep the fast method.

## 5. Tailwind `hover:` utilities don't generate a plain base rule
**Topics:** testing, astro

**Context:** Probing `bg-warning-500` (plain) on an injected element returned transparent.

**Learning:** A variant using `hover:bg-warning-500` generates `.hover\:bg-warning-500:hover`, NOT a plain `.bg-warning-500` rule — so probing the non-hover class reads transparent. The base (`bg-warning-300`) resolved correctly. Probe the actual applied state, not the hover-only class, when DOM-verifying hover variants. Also: `document.styleSheets[].cssRules` can throw/return-empty under CORS on the dev CSS file — the inject-and-getComputedStyle probe is the reliable DOM truth, not a stylesheet scan.
