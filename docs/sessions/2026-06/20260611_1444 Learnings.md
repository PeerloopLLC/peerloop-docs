# Conv Learnings - 2026-06-11

## 1. EnrichedCandidate back-compat made Phase 4 a narrow add, not a client rewrite
**Topics:** smart-feed, astro

**Context:** `SmartFeed.tsx` was written for the Conv-017 response shape; the Phase-2 orchestrator changed it to the 3-kind `FeedItem` union (member-post / sample-post / suggestion-card).

**Learning:** `EnrichedCandidate` deliberately kept `activity` / `isDiscovery` / `discoveryContext`, so member-post and sample-post already rendered against the new endpoint. Only `suggestion-card` (no `.activity`, no `isDiscovery`) was unhandled — and it was filtered at the endpoint. So Phase 4 reduced to: add one render variant + branch on `kind`. Cards crash the post path if you unfilter them before branching, so the card branch must come first.

## 2. Matt `FeedPost` is display-only → it can't be the member-post renderer
**Topics:** smart-feed, matt, post-matt

**Context:** The design's "Full Phase 4" listed sample-post-with-CTA and POST-MATT had tagged `FeedPost` live-wiring as Phase 4.

**Learning:** `FeedPost`'s reaction/comment pills are non-interactive social proof. Wiring it for member posts would regress live reactions/comments — POST-MATT itself keeps interactive native-feed posts on `FeedActivityCard`. So `FeedPost` is for teaser/marketing posts, not member posts, and its restyle is cosmetic page-chrome correctly deferrable to a later pass.

## 3. `sticky` beats `fixed` for a content-column-aligned bar
**Topics:** astro, css

**Context:** A visitor sticky sign-up bar with `fixed inset-x-0` would collide with the mobile ControlBar (fixed-bottom, `lg:hidden`) and span under the desktop Sidebar.

**Learning:** `position: sticky` inside the main content column auto-aligns to the content width (no Sidebar overlap, no width-hardcoding) and rides the scroll. A responsive `bottom-[76px] lg:bottom-6` clears the mobile ControlBar without knowing its exact height.

**Pattern:**
```astro
<!-- StickySignupBar.astro — sticky, not fixed, to inherit the content column -->
<div class="sticky bottom-[76px] lg:bottom-6 ...">
```

## 4. HttpOnly session cookies can't be cleared from page JS — log out via the endpoint
**Topics:** auth, browser-testing, chrome-bridge

**Context:** Browser-verifying the visitor case; cleared `document.cookie` but the page still rendered authenticated.

**Learning:** The session cookie is HttpOnly, so `document.cookie` neither reads nor clears it (it shows empty while the cookie is still sent on requests). To force visitor state in the bridge, `POST /api/auth/logout` (then `/api/me/full` → 401 confirms); restore with `POST /api/auth/dev-login {email}`. A stale-page "looks authed" symptom is really "still has the HttpOnly session."

## 5. A stale Astro dev server masquerades as a code bug
**Topics:** astro, dev-server, browser-testing

**Context:** A pre-existing dev server on :4321 returned 500 `jsxDEV is not a function` on /feed (and /messages).

**Learning:** Adding new component files corrupted the running server's `.vite/deps_ssr` optimizer cache → every page-render 500'd identically while API routes stayed fine. Symptom looks like a per-route bug; cure is restart + `rm -rf node_modules/.vite`. Diagnose by checking whether an UNRELATED page (/messages) 500s with the same `jsxDEV` stack — if so, it's the server, not the route.
