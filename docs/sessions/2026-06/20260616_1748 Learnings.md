# Conv Learnings - 2026-06-16

## 1. A launcher alias's own `cd` destroys the cwd signal a guard needs
**Topics:** cc-workflow, dual-repo

**Context:** Deciding whether to guard the `spt` mis-launch at the alias or in r-start.

**Learning:** `spt`/`peerloop` run `cd <target-docs>` before launching Claude, so by the time any r-start executes, `$PWD` is the target repo — the "launched from the wrong folder" evidence is already gone. A cwd-based guard is only implementable *before* that `cd`, i.e. in the alias/function itself. r-start can never reconstruct it.

## 2. CC exposes session identity + reachable process PID to shell tools
**Topics:** cc-workflow

**Context:** Needed a way to distinguish "another live session" from "a stale, crashed one" for a concurrent-session lock.

**Learning:** `CLAUDE_CODE_SESSION_ID` is an env var in every Bash call, and walking up `$PPID` reaches the session's `claude` process. Combine them: ownership = same session-id OR same claude-PID; concurrency = a *different* PID that is alive and is a `claude` process; staleness = PID dead / not claude. This makes a lock self-healing across crashes and the `/clear` re-run flow without manual cleanup.

**Pattern:**
`kill -0 <pid>` + `ps -o comm= -p <pid>` matched against `*claude*`, with the lock's PID compared to the current session's own claude PID to avoid self-blocking.
