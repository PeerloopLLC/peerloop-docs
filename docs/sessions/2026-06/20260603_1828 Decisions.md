## 1. FEED-DETAIL re-scoped: single-page /feed port, not a [...tab] family
**Type:** Implementation
**Topics:** routing, astro

**Trigger:** Legacy `/feed` is a single SmartFeed page; no `/feed/[slug]` exists or is linked. ~7 components link to bare `/feed` (which 404'd — no root page).

**Options Considered:**
1. Build a `/feed/[slug]` multi-tab family (the task's literal wording)
2. Port `/old/feed` → root `/feed` as one Matt-shell page ← Chosen
3. Treat `/feed/[slug]` as a new design

**Decision:** Single-page `@matt-inspired` port mounting the existing prop-less `SmartFeed` island — `src/pages/feed.astro` (AppLayout + SectionTitle + breadcrumb + OnboardingNudgeBanner, `max-w-2xl`, `noNav`, getSession redirect on top of the existing PROTECTED_EXACT guard).

**Rationale:** Matches legacy reality and fixes the ~7 dead `/feed` links; the triad pattern is for multi-tab entity-detail, which a personal smart feed is not.

**Consequences:** New `src/pages/feed.astro`; ~7 links + middleware guard now resolve. Established a new pattern: single-page `@matt-inspired` legacy port (Matt shell + carried-forward island), distinct from the [...tab] triad. 5 gates green (6460/6460), browser + curl verified (unauth→302, authed→200 hydrating island).

## 2. "My Feed" nav placement + harvested sparkles icon
**Type:** Implementation
**Topics:** navigation, icons

**Trigger:** A bare "Feed" sidebar entry beside the existing "Feeds" reads as a typo-pair.

**Options Considered:**
1. "My Feed" near Home, keep "Feeds" in the discovery cluster ← Chosen
2. Bare "Feed" next to "Feeds"
3. Rename the plural

**Decision:** "My Feed"→/feed placed after Home (NAV + COLLAPSED_NAV); harvested Material `auto_awesome` → `sparkles` MattIcon (normalized to `fill="currentColor"`, provenance `'sparkles': source 'ours'`).

**Rationale:** The "My" prefix + position separate the personal-stream from the browse-directory; sparkles echoes SmartFeed branding; no existing icon fit and `stars-2` is the rating glyph (hardcoded #D9D9D9 fill). `matchHref`'s trailing-`/` guard prevents `/feed`↔`/feeds` cross-activation.

**Consequences:** Sidebar NAV + COLLAPSED_NAV +1 each; new `sparkles.svg` + provenance entry. Browser-verified active-state both ways (My Feed active on /feed with Feeds inactive, and inverse on /feeds).

## 3. COMM-TAG-FILTER = channels model + community_channels table
**Type:** Strategic
**Topics:** schema, stream, community

**Trigger:** "Real" tag filtering has no backing data — legacy chips were hardcoded `['general','announcements','help']`, `feed_activities` has no tag column, Stream posts carry no tag field, and the townhall API is limit/offset only. [COMM-TAG-FILTER] is a net-new feature, not a wiring task.

**Options Considered:**
1. Channels (per-community posting categories) ← Chosen
2. Reuse the 55-tag topic taxonomy (rejected — wrong purpose, unusable chip count)
- storage: fixed-list vs `community_channels` table ← table chosen (B2)

**Decision:** Model as channels stored in a new `community_channels` table (seed the Commons general/announcements/help). LOCKED Conv 238.

**Rationale:** Channels are the honest "town hall" model; the table generalizes per-community without a second migration. The topic taxonomy is for skill-matching, not feed organization.

**Consequences:** Build deferred to its own conv (schema→composer→API→UI→backfill); only a courtesy Matt check on the channel-filter framing remains. Design at `plan/comm-tag-filter/README.md`, cross-referenced in PLAN.md, conv-tasks #1.
