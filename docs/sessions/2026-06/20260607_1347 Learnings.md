# Conv Learnings - 2026-06-07

## 1. The no-shrink backstop must EYEBALL the file, not trust a one-line grep
**Topics:** cc-workflow, docs-infra

**Context:** `/r-start` Step 7.5 no-shrink backstop grepped `conv-tasks.md` for `*DONE*`, got 0, and a confident "never live-synced / benign" story was narrated on top — then the file was overwritten. The user had it open and saw many DONE items: the grep marker format didn't match (or the count was wrong), so the measurement was wrong and the narrative was a confabulation built on it.

**Learning:** The backstop exists precisely so a human-style LOOK happens at a shrink. When a guard trips, READ the file before asserting why and before any destructive overwrite. A grep count is evidence, not proof; a user's direct observation of the file outranks it. Never build a "benign" rationalization on an unverified number. (The pre-overwrite file was unrecoverable — no VS Code local-history dir, `.scratch/` gitignored — but substance was safe in git at `2868f18:RESUME-STATE.md`.)

## 2. Concurrent two-machine convs reconcile cleanly IF the later machine pulled first
**Topics:** dual-repo, cc-workflow

**Context:** M4Pro ran Conv 245 (uncommitted body) while M4 ran a full Conv 246 on top of 245-start. Because M4 pulled 245-start before working, origin stayed linear — no divergence, just "behind" (docs −2, code −1). Reconciliation = revert the 2 overlapping tracked files → `git pull --ff-only` → re-sync memory mirror→live → re-verify gates → /r-end the older number on top.

**Learning:** The danger in this pattern is NOT git (ff handles it) — it's (a) the memory mirror-clobber (live synced pre-other-conv; /r-end's live→mirror would delete the other machine's new memories unless you re-sync mirror→live after pulling) and (b) duplicated work on overlapping files. Always `git fetch` + inspect `HEAD..origin` BEFORE any /r-end when a cross-machine conv may have landed.

## 3. Reuse-first paid off on the messages port
**Topics:** matt-design, components

**Context:** MSG-PORT rebuilt 5 islands. Instead of re-inlining avatar/modal/search, the Matt versions delegate to canonical `UserIcon`, `Modal`, `SearchInput`; the styling-agnostic `../types` helpers are shared (no duplication).

**Learning:** A route-port's "rebuild new Matt components" rule does not mean rebuild EVERYTHING — shared helpers + canonical primitives cross the legacy/Matt boundary fine; only the styled surfaces need parallel rebuilds.
