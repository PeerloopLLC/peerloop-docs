# Conv Learnings - 2026-06-28

## 1. A read-path (GET status) endpoint makes a write-feature self-verifying
**Topics:** api, testing, verification

**Context:** Gap A's follow backend had only POST/DELETE. To seed the button's initial state, added `GET /api/users/[handle]/follow` mirroring `can-message`.

**Learning:** The GET endpoint wasn't just for rendering — it made the whole feature *self-verifying*. A single follow→unfollow click in the browser exercised the entire POST/GET/DELETE triad and proved end-to-end wiring **without leaving any persistent DB residue** (the unfollow restores state). Designing the read-path alongside the write-path paid off twice: render correctness + clean functional verification.

**Pattern:** For a toggle feature, a status GET + a UI round-trip (do→undo) is the cheapest high-signal DOM verification that doesn't mutate dev data.

## 2. A homework file-upload's hidden cost is authorization/serving, not the upload
**Topics:** r2, access-control, schema, scoping

**Context:** Gap C/c2 (submission file upload) looked like a reuse of the resources multipart pattern, but `homework_submissions` stores only `file_url` (no `r2_key`), and every R2 file in Peerloop is served via an id-based, access-controlled download route.

**Learning:** Wiring `uploadToR2` is the easy 20%. The real work is *serving*: Peerloop deliberately avoids public R2 URLs and gates every file behind a route that re-checks the requester. A submission is private student work, so c2 inherits a brand-new access-control question (student owner + creator + assigned teacher may read; nobody else) that public course resources don't have. That's why it's an architectural task (schema column + new authed route + access policy), not a mechanical wire-up — correctly deferred as [PLATO-GAP-C2].

## 3. Match the file's existing token tier when extending unported components
**Topics:** matt-provenance, styling

**Context:** `HomeworkTab.tsx` (student submission UI, the c2 surface) uses LEGACY Tailwind tokens (`secondary-*`, `primary-600`), not Matt-swept tokens.

**Learning:** New UI added to an unported component should match the surrounding legacy token style, not introduce Matt tokens — adding a feature is not retrofitting. (Captured for the c2 follow-up so it stays cohesive.) The Matt-styled surfaces (PublicProfile, CourseEditor, HomeworkEditor) correctly used the `Button` primitive + Matt tokens; the legacy HomeworkTab should not.
