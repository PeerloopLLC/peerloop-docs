# Conv Learnings - 2026-06-10

## 1. Self-gating islands are testable by identity alone — but race the stale user-cache
**Topics:** plato, testing, smart-feed, role-semantics

**Context:** Verifying ROLE-STUDIOS Phase-4 ProgressionNudge placements rendered for the right users.

**Learning:** Because `ProgressionNudge` self-gates on the current-user store, eligibility is a pure function of the user → D1 classification fully predicts the DOM. Switch users with `POST /api/auth/dev-login {email}`, hard-navigate, read the `[data-nudge]` marker. BUT a "break on first sighting" detector races `localStorage['peerloop_user_cache']` — the island first-paints from the *previous* user's cached classification, then corrects on the fresh `/api/auth/session`. Correct method = **settle-then-read** (~1.5–1.8s wait, then query once).

**Pattern:**
dev-login switch + hard navigate + settle-then-read on a `data-*` marker; confirm eligibility in D1 first. Memory: `reference_chrome_bridge_island_stale_cache`.

## 2. The existing Smart Feed already reconciles algo-ranking with time-cursor pagination
**Topics:** smart-feed, cloudflare

**Context:** Worried the new "high-value algo" member feed would break the planned Option-A time cursor.

**Learning:** `getSmartFeed` already does a time-windowed candidate fetch (`created_at < cursor`) plus algo-rank-within-window (`scoreCandidates`). So "algo" is the ranking lens within each window; time is the pagination axis — no conflict. Build-fix to bank: advance the cursor by the *oldest fetched* post, not the *last shown* (current code uses last-shown, a latent skip/dupe under reorder), plus a `(created_at, id)` tiebreaker.

## 3. Smart Feed levers are DB-tunable, not just constants
**Topics:** smart-feed

**Context:** Client wants the feed algorithm tunable via constants/enums.

**Learning:** `scoring.ts` `ScoringParams` = 7 weights + 6 structural dials, defaulted in `PARAM_DEFAULTS` and runtime-overridable from `platform_stats` (`key LIKE 'smart_feed_%'`) via `loadScoringParams(db)` — tunable without redeploy, with per-item score-breakdown explainability. New-model dials should extend this same pattern, not introduce a parallel config.

## 4. Matt's post frames collapse four task areas into one component
**Topics:** matt, components, feeds

**Context:** Probing the simple (477:8285) + complex (477:8203) post frames for [POST-MATT].

**Learning:** Both posts are one shell (header/body/reactions); the complex one adds an embedded entity card. That embedded course card (green "Learn More" CTA) is simultaneously the promote-a-course template, the feed sample-post/card variant, and a post attachment — i.e. post redesign + promote-templates + feed item-variants + Show-More are *one component with optional regions*. Build the post shell first as the foundation.
