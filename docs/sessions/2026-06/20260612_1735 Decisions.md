## 3. System-promotion moderation: system-only scope, admin-gated, no moderation_actions log
**Type:** Implementation
**Topics:** promotion, auth, moderation

**Trigger:** Building U3c ③.

**Decision:** (a) Scope the moderation list/remove to `to_feed_type='system'` only; (b) gate with `requireRole(['admin'])` (not the moderator-scoped `requireModerationAccess`); (c) write no `moderation_actions` audit row on removal.

**Rationale:** (a) the threat is non-admin content reaching the admin-only platform-wide System feed; (b) System promotions are a platform concern + the /admin page is admin-gated by middleware; (c) `moderation_actions.flag_id` is `NOT NULL → content_flags` — a promotion take-down isn't a flag resolution, so logging there would need a fake flag.

**Consequences:** `removeSystemPromotion` is scope-guarded so the admin path can never delete a community/course promotion; an audit-log enhancement is a possible follow-up.

## 4. Moderation tab placement via a thin ModerationPage wrapper
**Type:** Implementation
**Topics:** moderation, astro

**Trigger:** The plan says "wire the lane into admin/moderation.astro", but ModerationAdmin is a ~740-line content_flags monolith with no tab scaffold.

**Decision:** Introduce a thin `ModerationPage.tsx` tab shell ("Flagged Content" | "System Promotions") that mounts either component; `moderation.astro` mounts the shell; `ModerationAdmin.tsx` stays untouched.

**Rationale:** Satisfies the plan's placement while keeping ③ isolated (its own component + endpoints + done-test), avoiding coupling to the large component's churn.

**Consequences:** New `ModerationPage.tsx`; one-line change to `moderation.astro`.
