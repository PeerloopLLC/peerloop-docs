# Conv Decisions - 2026-06-13

## 1. PromoteNudge engagement gate = configurable dial
**Type:** Implementation
**Topics:** promotion, admin, config

**Trigger:** Plan mandated "decide engagement threshold at start"; no threshold concept existed.

**Options Considered:**
1. Capability-only (nudge whenever creator/teacher with a promotable entity)
2. Fixed hardcoded threshold
3. Configurable `platform_stats` dial ← Chosen

**Decision:** Add `promo_nudge_min_engagement` dial (default 3), admin-editable in the existing Promotion Settings page; gates *nudging*, not *promoting*; 0 = always nudge. Engagement = course `student_count` / community `member_count`.

**Rationale:** Follows the established promotion-dial pattern; admin tunability near-free; gating the prompt (not the action) keeps the manual composer fully open.

**Consequences:** New dial in config.ts + seed + admin endpoint + admin UI; `promotable-entities` extended with `engagementCount` + `minEngagement`.

## 2. U3d per-post nudge deferred — post-escalation already exists
**Type:** Strategic
**Topics:** promotion, scope, planning

**Trigger:** Premise-Check trace found `PromoteButton` already provides the per-post (server canPromote) affordance.

**Options Considered:**
1. Defer per-post; ship the workspace card ← Chosen
2. Declare existing PromoteButton satisfies it (no follow-up)
3. Build a new attention-drawing per-post nudge now

**Decision:** Ship the workspace card; defer the *attention-drawing* per-post nudge (entity-promo model) to a scoped follow-up `[U3D-POST]` pending a tighter spec.

**Rationale:** A second per-post control atop the always-on PromoteButton would be redundant clutter; a real nudge deserves its own spec.

**Consequences:** U3d card-half complete; `[U3D-POST]` carries the deferred work.
