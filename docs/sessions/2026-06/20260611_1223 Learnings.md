# Conv Learnings - 2026-06-11

## 1. Model ① reference-lane orphans the feed_activities lineage column
**Topics:** stream, d1, promotion

**Context:** The locked Build-sequence Step 1 named only `post_promotions.target_activity_id` for removal, but converting promotion from copy (③) to reference (①) means no target activity row is ever created.

**Learning:** Under Model ①, `feed_activities.promoted_from_activity_id` has no writer at all — it was only ever set on the copied target row. Implementing the design literally would have left a dead column + index + unused `indexFeedActivity` param landing in the same conv that killed them. Surfacing it as a decision (rather than silently keeping or dropping) got user approval to drop it cleanly.

**Pattern:** When a locked design enumerates schema drops, re-derive the *full* set the model orphans across tables — the design's list may be scoped to one table.

## 2. The canPromote flag must reflect actual promotability, not just role
**Topics:** promotion, auth, api

**Context:** The PromoteNudge per-post prompt gates on a server `canPromote` signal; the design explicitly calls out avoiding a 403-after-click (posture-A surprise anti-goal).

**Learning:** A pure role check is insufficient — `canPromote(role)` returns true for an admin even on the System feed (top of chain, no target) and for a course creator whose course has no progression (no parent community). The correct flag = role-allowed AND `resolvePromotionTarget !== null` AND not the System feed. Community always resolves to System so it's role-only; course needs the target probe.
