# Conv Decisions - 2026-06-11

## 1. Promote endpoint resolves posts by Stream activity id (not feed_activities.id)
**Type:** Implementation
**Topics:** promote-pipeline, feeds, api, stream

**Trigger:** The client holds only the Stream activity id everywhere (reactions/comments POST `activityId: activity.id`), but the Conv-262 promote endpoint keyed on `feed_activities.id`.

**Options Considered:**
1. Endpoint resolves by Stream id (`WHERE stream_activity_id = ?`) ← Chosen
2. Read path attaches `feedActivityId` to each activity; endpoint unchanged

**Decision:** Change the endpoint body to `{ streamActivityId, password }` and resolve the canonical row via `stream_activity_id` (UNIQUE-indexed). Internal FK / `recordPromotion` / lane stay on `feed_activities.id`.

**Rationale:** Consistent with the system-wide client identity model; scales free to every future promote surface (home feed, nudge) which also hold only Stream ids — option B would repeat the read-path join per surface.

**Consequences:** Endpoint body renamed; dead "no Stream activity" guard removed; UNIQUE index on `feed_activities.stream_activity_id` added (also hardens the lookup). 8 new endpoint tests.

## 2. Promotion is idempotent (UNIQUE index + early-return)
**Type:** Implementation
**Topics:** promote-pipeline, schema, d1

**Trigger:** `post_promotions` had no uniqueness; double-click / two promoters would duplicate.

**Decision:** `UNIQUE(source_activity_id, to_feed_type, to_feed_id)` + an endpoint pre-check returning the existing promotion (`{ alreadyPromoted: true }`, 200) before the password gate.

**Rationale:** Building the action durably; the lane would otherwise surface a post twice. Pre-check before the gate so a re-clicker isn't re-challenged for a no-op.

**Consequences:** New schema index; endpoint early-return path; idempotency test.

## 3. Extract a shared PromoteButton across both live renderers
**Type:** Implementation
**Topics:** feeds, component-design, matt-migration

**Trigger:** Two live feed renderers (`FeedActivityCard` community / `MattCourseFeed`→`SocialPost` course) both need the button.

**Options Considered:**
1. Duplicate the modal+fetch logic in each
2. Extract `src/components/feed/PromoteButton.tsx`, themed per surface via `className` ← Chosen

**Decision:** One shared component owns the password modal + idempotent POST + "Promoted" flip.

**Rationale:** DRY — the password/idempotency logic has one home; can't drift between surfaces. The browser-found second renderer forced (and justified) the extraction.

**Consequences:** New `PromoteButton.tsx`; `FeedActivityCard` refactored onto it; `SocialPost` gained an `actions` slot.
