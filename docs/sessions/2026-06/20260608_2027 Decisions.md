**Type:** Strategic
**Topics:** ia, dashboard, flywheel, nudges

**Trigger:** `UnifiedDashboard` is built but orphaned (`/old/dashboard`); the sidebar has no role/dashboard entry. **Decision:** Deconstruct the unified dashboard. Three cross-role needs map to three surfaces: operational context → entity tabs (already done, course/community role tabs); cross-role triage → thin strip on `/`; role progression → CTAs on the SOURCE role's surfaces. Progression nudges elevated from nice-to-have to MUST-HAVE. Wrote the full ROLE-STUDIOS decision + 7-phase build plan to PLAN.md; realigned route-migration README cluster 0/2/3/4.

**Rationale:** Users navigate by entity, not role; the combined dashboard fragmented entities to aggregate roles, and its "see all at once" promise died behind tabs for deep surfaces. Deconstruct wins every flywheel transition because conversions anchor to entity events / source-role surfaces. Source-role nudges reach a never-/teaching student that a siloed target hub never would.

**Consequences:** s/t/c role-aware dashboards merge into `/learning` `/teaching` `/creating`; nudge layer is a must-have build target; in-app conversion nudges found almost entirely absent today (only `CourseDetail.tsx:109`). ## 3. ROLE-SEMANTICS two-axis rule: capability `canX` vs identity `isX`
**Type:** Implementation
**Topics:** auth, role-semantics, access-control

**Trigger:** Canonical home `CurrentUser` exists but there are no account-level `isRole` getters and ~20 `/api/me` inline recomputes. B's first deliverable surfaced as a product decision (permission vs behavioral "is a creator"), load-bearing for nudge gating.

**Options Considered:**
1. Single role definition (permission-based or behavioral)
2. Two-axis split — capability `canX` (permission, gates ACTIONS) / identity `isX` (behavioral, gates NAV/display/nudges) ← Chosen

**Decision:** Split role checks into two axes. Implemented canonical `get isCreator/isTeacher/isStudent/isModerator` getters on `current-user.ts` (behavioral; moderator/admin assigned) + canonical `isCreatorSubquery`/`isTeacherSubquery` SQL fragment builders on `roles.ts` (pure strings, client-bundle-safe). Migrated 6 pure-behavioral inline SQL sites (7 instances). Documented the rule in PLAN.

**Rationale:** `canX` answers "may this user do X?" (permission); `isX` answers "has this user done X?" (behavioral). Conflating them breaks on the new-creator (permission=1, courses=0) and revoked-creator (permission=0, courses>0) edges. The split is the foundation for correctly gating nudges and navigation.

**Consequences:** New canonical getters + SQL fragments; 15 tests (12 identity + roles-sql). `roles.ts userRoles()` found to be a 3rd permission-based definition site. Blast radius flagged as ~15+ sites across TWO definitions → tasks #24 [RS-HYBRID-FLIP] + #25 [RS-SQL-SWEEP].

## 4. RS blast-radius — split each hybrid site by PURPOSE (Decision A)
**Type:** Implementation
**Topics:** auth, role-semantics, access-control, security

**Trigger:** User: "C. This is crucial work" — re-examine the blast radius before committing the flip. 3 parallel Explore audits (server SQL, client CurrentUser, roles.ts/display/middleware).

**Options Considered:**
1. Naive flip: hybrid `is_creator` → behavioral everywhere
2. Decision A — split each site by PURPOSE: access gates → `canX`, identity/display/nudge → `isX` ← Chosen

**Decision:** Split by purpose. The 3 hybrid gates are ACCESS gates → belong on capability (`canCreateCourses`), not behavioral identity. Recorded the full audit + Decision A + security rule + the revoke edge in PLAN; corrected #24 framing.

**Rationale:** The naive flip would have 403'd every approved-but-0-course creator (`if (!is_creator) → 403` is an access gate). Verified the revoke edge: `can_create_courses` IS admin-revocable and creation requires it. Banked security rule: never gate on `requireRole(['creator'])`/`['teacher']` until `getUserRoles` is behavioral — today's JWT roles are permission-based, and `requireRole` only checks `['admin']`/`['moderator']`, so the change is security-neutral now but a future creator/teacher gate would read the stale flag.

**Consequences:** #24 [RS-HYBRID-FLIP] reframed to split-by-purpose (not flip); accepted default for the revoke edge (admin-revoked ex-creator with courses loses `/creating` access, badge still shows creator) — revisit only if orphaned-course management bites.
