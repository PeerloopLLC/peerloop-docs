# Conv Decisions - 2026-06-12

## 1. Announcement fan-out architecture = A+B (feed-pin read-time + optional notify)
**Type:** Strategic
**Topics:** stream, architecture

**Trigger:** U3c④ platform announcements; no announcements table or lane exists today (both deferred in code); CLAUDE.md §Critical Rule novel decision.

**Options Considered:**
1. A — read-time feed lane (shared row, mirrors promotion model ①)
2. B — per-user notification rows (write-time fan-out, free is_read/dismiss, bell surface)
3. A+B — feed by default + optional "also notify" for urgent ones ← Chosen
4. C — dedicated announcements table + its own read path

**Decision:** A+B. Every announcement renders in the home smart feed at read time (read-time fan-out, zero write amplification, reuses the promotion substrate); an optional `notify` flag additionally fans out a per-user 'system' notification for time-sensitive announcements (e.g. maintenance) that must be seen even if the user never opens the feed.

**Rationale:** The 4 example announcements showed ①③④ are feed-native (live in the feed, the "50% of learning" surface) while ② (maintenance) wants the bell. A+B covers both; pure A can't guarantee an urgent message is seen, pure B buries feed-native posts in the inbox, C is the most net-new code for no reuse benefit.

**Consequences:** Reuses promotion model ① read-assembly philosophy; the feed is the default surface; notify is opt-in per announcement.

## 2. D1-only announcement storage (no Stream activity)
**Type:** Implementation
**Topics:** stream, d1

**Trigger:** Option A reuses "the lane" — but an announcement is authored fresh (unlike a promotion, which points at an existing Stream post).

**Options Considered:**
1. D1-only — text in the `announcements` table, lane renders straight from it ← Chosen
2. Stream-backed — create a System-feed Stream activity (postKind='announcement') for native reactions/comments

**Decision:** D1-only. The announcement lives entirely in the `announcements` table; no Stream write, no per-read Stream fetch.

**Rationale:** A one-way broadcast needs no native reactions/comments and references no pre-existing post, so a Stream activity would only hold text D1 can store directly. Simpler, one fewer write, no per-read Stream round-trip.

**Consequences:** The earlier "A vs C" framing conflated "own table" with "own read path" — the real reuse is the lane assembly, not Stream. A small `announcements` pointer-style table is consistent with A.

## 3. Optional admin-set `active_until` (the one stored-expiry divergence)
**Type:** Implementation
**Topics:** d1, architecture

**Trigger:** The promotion system computes lifecycle from `created_at` + a dial (no `expires_at` column, deliberate). But the maintenance example has a real-world end-time a dial can't express.

**Options Considered:**
1. Optional `active_until` (NULL → dial fallback) ← Chosen
2. Dial-only (strict promotion-pattern purity, no stored end-time)

**Decision:** Optional admin-set `active_until`; when NULL, falls back to an `announcement_active_duration_days` dial.

**Rationale:** A maintenance window has a precise end the dial can't model; a contained, justified divergence from the "no expires_at" stance. All active-window comparisons stay string-vs-string on the canonical ISO strftime format (SQLite Datetime Rule); the dial path is inverted to `created_at > now + (-N days)` so the stored timestamp is never re-parsed.

**Consequences:** `active_until TEXT` column on `announcements`; retention purge guards against purging a still-active (future `active_until`) row.
