# Session Learnings - 2026-03-05

## 1. Tech Docs Accumulate Untracked Features
**Topics:** docs-infra, workflow

**Context:** Scanned all 31 tech docs for actionable items (TODOs, MISSING, planned features) and compared against PLAN.md. Found 4 blocks worth of development work documented in tech docs but invisible to planning.

**Learning:** Tech docs naturally accumulate "Future Features" and "Planned" sections that drift out of sync with PLAN.md. Without periodic sweeps, features fall through the cracks. The fix is two-fold: (1) a CLAUDE.md rule requiring all features to be tracked in PLAN.md with cross-references in tech docs, and (2) occasional scans like this one to catch drift.

**Pattern:** Use parallel agents to scan batches of tech docs, then verify findings against PLAN.md before creating new blocks. Filter out: client questions, decision rationale, completed items, and on-hold items awaiting client input.

## 2. Integration Tests and E2E Tests Catch Different Failure Modes
**Topics:** testing

**Context:** User asked whether existing integration tests (notification-lifecycle, message-lifecycle) made E2E browser tests redundant. Compared what each layer catches.

**Learning:** Integration tests (calling API handlers directly) catch data logic bugs cheaply and fast (milliseconds). E2E tests (Playwright browser) catch UI wiring failures: React hydration, CSS visibility, polling behavior, click handlers, auth cookies. A correct API is worthless if the component doesn't call it or ignores the response. Both layers have value — integration tests are the base of the pyramid (many, cheap), E2E tests are the top (few, expensive, but catch assembly failures).

## 3. Month View vs Week/Day View Are Architecturally Different
**Topics:** astro, testing

**Context:** Audited all calendar components in the codebase. Found 3 independent custom month grids but no week or day views. react-big-calendar is installed but never imported.

**Learning:** Month view is a grid of day cells with badges — relatively simple layout. Week/Day views require a vertical time axis (e.g., 6am-10pm) where items are absolutely positioned blocks based on start/end time, with overlap handling and scroll-to-now. These are fundamentally different rendering problems, not just zoom levels on the same component. The base calendar component must handle both layout modes.

## 4. Branching Workflow Tests: Shared Setup + Diverging Decision Points
**Topics:** testing

**Context:** Discussed testing patterns for multi-step flows (booking, completion, payment). Current integration tests cover single-flow lifecycles but real user workflows are trees with decision-point branches.

**Learning:** The highest-value test branches are those where the decision changes downstream state (different DB records, different statuses, different errors) — not those where input content varies but the flow is identical. A shared setup helper (e.g., `setupCompletedSession(db)`) that gets to the decision point lets you test 5 branches for roughly the cost of 1.5 full tests. The expensive part is building the world; the branches are cheap once you're at the decision point.
