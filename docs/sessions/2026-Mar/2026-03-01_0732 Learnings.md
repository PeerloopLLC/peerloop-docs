# Session Learnings - 2026-03-01 (Session 319)

## 1. Resend Domain Verification Gives Misleading Error Messages
**Topics:** resend, email

**Context:** Creator application approval triggered an email send via Resend SDK. The `from` address was correctly formatted as `Peerloop <noreply@peerloop.com>` but Resend returned a 422 error: "Invalid `from` field. The email address needs to follow the `email@example.com` or `Name <email@example.com>` format."

**Learning:** The error is misleading — it's actually a domain verification failure, not a format issue. Resend verifies a sending *subdomain* (e.g., `send.peerloop.com`), not the root domain. Emails must use `@send.peerloop.com` as the from address. The sandbox fallback `onboarding@resend.dev` works for development without any verified domain.

**Pattern:** Documented in `docs/tech/tech-004-resend.md` under "Caveat: Unverified Domain Errors" section.

## 2. Permission vs State: Two Meanings of "Is Creator"
**Topics:** auth, architecture

**Context:** After admin approved a creator application (setting `can_create_courses = 1`), the user was blocked from `/creating` with "Creator access required." Traced to the `/api/me/creator-dashboard` endpoint which checked for existing courses via subquery, not the permission flag.

**Learning:** The codebase had two independent patterns for "is this user a creator":
- **Permission** (`can_create_courses` flag) — set by admin, used by POST endpoints
- **State** (course count subquery) — derived from data, used by GET endpoints

This created a chicken-and-egg bug for new creators. The fix required auditing all 15 creator access gates and establishing clear rules for when to use each. View/manage actions should check permission OR state. Create actions should check permission only.

## 3. Centralized Email Function Can Be Bypassed
**Topics:** resend, architecture

**Context:** While fixing the email `from` address, discovered that two moderator endpoints (`invite.ts`, `resend.ts`) called `resend.emails.send()` directly, bypassing the centralized `sendEmail()` function in `src/lib/email.ts`.

**Learning:** When a centralized function exists (like `sendEmail()`), new code can still bypass it by importing the underlying library directly. All three locations had the same hardcoded `from` address, but if they'd diverged, the bug would have been harder to find. Worth scanning for pattern violations when fixing centralized utilities.

## 4. Three Decision Document Types Serve Different Purposes
**Topics:** docs-infra

**Context:** Session established platform behavior policies that didn't fit cleanly into DECISIONS.md (architectural) or PLAYBOOK.md (docs-repo). Created POLICIES.md.

**Learning:** The three docs now serve distinct, non-overlapping purposes:
- **POLICIES.md** — How the platform *behaves* (access rules, business logic). Prescriptive: if code contradicts a policy, the code is the bug.
- **DECISIONS.md** — How we *build* the platform (architecture, tech choices)
- **PLAYBOOK.md** — How the *docs repo* works (workflow, conventions)
