# Session Decisions - 2026-03-05

## 1. Session Cancellation Policy: Always Allowed, Late Requires Reason
**Type:** Strategic
**Topics:** booking, policy

**Trigger:** CURRENT-BLOCK-PLAN item #4 (US-S014) was blocked on undefined cancellation business rules.

**Options Considered:**
1. Hard block — no cancellation within 24h of session start
2. Always allowed, no conditions (raw CD-033 interpretation)
3. Always allowed, but < 24h requires a reason + S-T notification <- Chosen

**Decision:** Cancellation is always permitted (per CD-033: "bail at anytime"). Late cancellations (< 24h before session start) require a reason, which is sent to the S-T as an in-app notification. The `is_late_cancel` flag is saved for admin visibility.

**Rationale:** Hard blocks would contradict the client directive. No conditions at all would leave S-Ts blind-sided. The middle ground respects the student's freedom while giving S-Ts accountability and context.

**Consequences:** `cancelled_at`, `is_late_cancel`, `session_cancelled` notification type added to schema. DELETE handler enforces reason requirement for late cancels.

## 2. Reschedule Limit: Max 2 Per Session
**Type:** Implementation
**Topics:** booking, policy

**Trigger:** CURRENT-BLOCK-PLAN item #5 (US-S013) — no limit existed on rescheduling.

**Options Considered:**
1. No limit — unlimited reschedules
2. Max 2 reschedules per session <- Chosen
3. Max 1 reschedule per session

**Decision:** Sessions can be rescheduled at most 2 times. On the 3rd attempt, the API returns 422 with guidance to cancel and rebook.

**Rationale:** Unlimited reschedules create scheduling chaos for S-Ts. A limit of 2 is generous enough for legitimate conflicts (timezone mistakes, emergencies) while preventing abuse. Cancel-and-rebook resets the count, which is appropriate since it creates a new session record.

**Consequences:** `reschedule_count` column added to sessions table. PATCH handler checks count before allowing reschedule. `can_reschedule` flag added to GET session detail response.

## 3. Mark Complete Gated on Session Completion via Existing Sessions Endpoint
**Type:** Implementation
**Topics:** astro, api-design

**Trigger:** CURRENT-BLOCK-PLAN item #3 — students could self-report module completion without attending the tutoring session.

**Options Considered:**
1. New dedicated API endpoint for module-session status
2. Server-side prop from Astro page (resolveModuleAssignments in frontmatter) <- Not chosen (CourseLearning is client:load)
3. Reuse existing `GET /api/sessions?role=student` with client-side filtering <- Chosen

**Decision:** Fetch the student's sessions list from the existing endpoint, build a `Map<module_id, session_info>` filtered by course slug, and pass session status to ModuleContent as props to gate the Mark Complete button.

**Rationale:** No new API infrastructure needed. The sessions endpoint already returns module assignment data. Client-side filtering by course slug is trivial. The CourseLearning component is a React island that already makes parallel API calls on mount — one more fetch has negligible overhead.

**Consequences:** Three new props on ModuleContent: `sessionStatus`, `sessionDate`, `courseSlug`. Button shows contextual hints: "Book a session first" (with link), "Session scheduled for [date]", or enabled when session completed.
