# Session Decisions - 2026-03-05

## 1. Dedicated Count Endpoint vs Reusing Conversation List
**Type:** Implementation
**Topics:** messaging, d1

**Trigger:** Messages badge in AppNavbar needs total unread count. The existing `GET /api/conversations` endpoint returns per-conversation unread counts but also fetches participants, last messages, and other data the badge doesn't need.

**Options Considered:**
1. Fetch `GET /api/conversations` and sum `unread_count` client-side
2. Create dedicated `GET /api/me/messages/count` endpoint with single SQL query ← Chosen

**Decision:** Created `/api/me/messages/count` with a correlated subquery that computes the total in one database round-trip.

**Rationale:** Option 1 would transfer unnecessary data (participants, messages) and do N+1 queries on every 60s poll. Option 2 mirrors the existing notification pattern (`/api/me/notifications/count`) and uses a single efficient query.

**Consequences:** New endpoint created. AppNavbar polls it every 60s alongside the notification count endpoint.

## 2. Messages Badge: Mirror Notification Pattern Exactly
**Type:** Implementation
**Topics:** messaging, astro

**Trigger:** Messages had no badge in AppNavbar while Notifications got one in Session 338. User requested parity.

**Options Considered:**
1. Different styling/placement for messages badge (e.g., different color, different cap)
2. Same red pill, same 99+ cap, same 60s polling as notifications ← Chosen

**Decision:** Messages badge uses identical visual style, overflow cap (99+), and polling interval (60s) as the notification badge.

**Rationale:** Consistency reduces cognitive load. Both are "unread content" indicators — they should look and behave the same. The sidebar has enough space for 99+ (vs AppHeader's tighter 9+ cap).

**Consequences:** Both badges render identically in the AppNavbar menu items.

## 3. Mark-All-Read Communication via CustomEvent
**Type:** Implementation
**Topics:** messaging, astro

**Trigger:** "Mark all read" button in ConversationList (child) needs to trigger conversation refresh in Messages (parent). Three patterns available.

**Options Considered:**
1. Prop-drill an `onMarkAllRead` callback from Messages through ConversationList
2. Lift filter state and mark-all-read to Messages component
3. Dispatch `CustomEvent` from child, listen in parent ← Chosen

**Decision:** Use `window.dispatchEvent(new CustomEvent('messages:mark-all-read'))` pattern.

**Rationale:** Avoids adding more props to ConversationList. The event is a one-shot signal ("data changed, please refresh") rather than shared state. This is the same pattern used by Astro's `astro:page-load` events already in the codebase.

**Consequences:** Messages.tsx adds a `useEffect` event listener. ConversationList dispatches after the API call succeeds.

## 5. ConversationList Filter Tabs Match Notifications Visual Pattern
**Type:** Implementation
**Topics:** messaging, astro

**Trigger:** Messages had no filtering while Notifications had All/Unread tabs. User requested parity.

**Options Considered:**
1. Different filter UI (dropdown, segmented control)
2. Same pill-button tabs as NotificationsList ← Chosen
3. No filtering, just sort by unread first

**Decision:** Added All/Unread filter tabs using the same visual pattern as NotificationsList: pill buttons in a `bg-secondary-100` container with white active state.

**Rationale:** Visual consistency across the app. Users learn one filter pattern, it works everywhere. The Messages version uses slightly more compact sizing (`text-xs`, `py-1.5`) since it shares header space with the conversation list.

**Consequences:** ConversationList now manages local `filter` state and derives `filteredConversations`. Added "All caught up!" empty state for unread filter.
