# Session Decisions - 2026-03-05 (Session 338)

## 1. Messaging Relationship Policy
**Type:** Strategic
**Topics:** messaging, auth

**Trigger:** Investigation revealed that any authenticated user could message any other user, contradicting user stories (US-S016, US-S017, US-S018) that define specific messaging relationships.

**Options Considered:**
1. Keep open messaging (status quo) -- simple but contradicts design intent, enables abuse
2. Relationship-gated messaging with shared `canMessage()` function <- Chosen
3. Whitelist-based approach (users explicitly approve who can message them) -- too complex for MVP

**Decision:** Direct messaging requires authentication plus a platform relationship. 11 specific sender->recipient rules defined in POLICIES.md section 4. Student-to-student messaging blocked for MVP (US-S017, P2).

**Rationale:** Matches user story intent. Prevents spam/harassment vectors as platform scales. Genesis Cohort is small enough that the relationship graph covers all legitimate messaging needs.

**Consequences:** POLICIES.md section 4 created. Three API endpoints need relationship validation (Phase 1 implementation).

## 2. Two-Layer Enforcement Model
**Type:** Implementation
**Topics:** messaging, auth

**Trigger:** Need to decide how to enforce messaging restrictions across both UI and API.

**Options Considered:**
1. UI-only enforcement (hide buttons, filter search) -- UX polish only, not secure
2. API-only enforcement (let UI show buttons, reject at API) -- secure but bad UX
3. Dual-layer: UX visibility (Layer 1) + API gates (Layer 2, authoritative) <- Chosen

**Decision:** Layer 1 (UX) controls button visibility and search filtering. Layer 2 (API) is the authoritative security boundary on three endpoints. If layers disagree, Layer 2 wins.

**Rationale:** Defense in depth. UX layer prevents wasted clicks and provides good experience. API layer is the actual security boundary that cannot be bypassed by URL manipulation or direct API calls.

**Consequences:** All existing "Message" buttons on profile pages need conditional rendering (Phase 1). API gates are the priority security work.

## 3. Three-Phase Implementation Priority
**Type:** Implementation
**Topics:** messaging

**Trigger:** 28 surfaces audited, need to prioritize which get messaging controls first.

**Options Considered:**
1. Implement everything at once -- too large, blocks other work
2. Phase 1: security gates, Phase 2: inherently-valid UX, Phase 3: conditional UX <- Chosen
3. Start with UX buttons, add API gates later -- wrong order (security first)

**Decision:** Phase 1 gates existing entry points (3 API endpoints + profile page conditional rendering). Phase 2 adds message buttons to inherently-valid surfaces (session screens, dashboard widgets, admin panels). Phase 3 adds conditional buttons to relationship-dependent surfaces (course pages, community members).

**Rationale:** Security first (Phase 1 closes the open-DM gap), then expand UX on safe surfaces (Phase 2 where relationship is guaranteed by context), then the harder conditional work (Phase 3 requires per-pair relationship checks).

**Consequences:** Phased rollout documented in tech-018-messaging.md.

## 4. Policy/Implementation Document Split
**Type:** Strategic
**Topics:** messaging, docs-infra

**Trigger:** Comprehensive messaging policy + surface catalog too large for a single document.

**Options Considered:**
1. Everything in POLICIES.md -- too much implementation detail for a policy doc
2. Everything in tech-018 -- buries the rules in implementation noise
3. Rules in POLICIES.md, implementation catalog in tech-018 <- Chosen

**Decision:** POLICIES.md section 4 contains the relationship rules, enforcement principles, and behavioral policies. tech-018-messaging.md contains the 28-surface catalog, phased rollout plan, URL normalization, and `canMessage()` pseudocode.

**Rationale:** Follows the existing pattern where POLICIES.md defines "what the code SHOULD do" and tech docs define "how to implement it." Cross-referenced via `See:` links in both directions.

**Consequences:** POLICIES.md section 4 created. tech-018-messaging.md updated with new "Messaging Access Control (Session 338)" section.
