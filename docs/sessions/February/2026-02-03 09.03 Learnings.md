# Session Learnings - 2026-02-03

## 1. Permission vs State Architecture Pattern
**Topics:** d1, auth, architecture

**Context:** While auditing for deprecated "roles" usage, discovered the Session 163 migration that established the "Permission vs State" architecture.

**Learning:** The codebase distinguishes between two types of role-related data:
- **Permissions** (capabilities): Admin-controlled flags stored on the user (`can_create_courses`, `can_teach_courses`, `can_take_courses`, `can_moderate_courses`)
- **State** (derived): Computed at query time from relationship tables (`is_creator` from `courses` table, `is_student_teacher` from `student_teachers` table)

This eliminates data staleness - state is always current because it's derived from the source of truth (relationship tables), not cached on the user record.

**Pattern:**
```sql
-- Derive is_creator state via subquery
SELECT * FROM users u
WHERE EXISTS (SELECT 1 FROM courses c WHERE c.creator_id = u.id AND c.deleted_at IS NULL)

-- Derive is_student_teacher state via subquery
SELECT * FROM users u
WHERE EXISTS (SELECT 1 FROM student_teachers st WHERE st.user_id = u.id AND st.is_active = 1)
```

## 2. EXISTS Subqueries for State Derivation
**Topics:** d1, performance

**Context:** Needed to replace deprecated `WHERE is_creator = 1` column references with derived state checks.

**Learning:** `EXISTS (SELECT 1 FROM ...)` is preferred over JOINs for existence checks because:
1. **Semantic clarity** - Asks "does at least one row exist?" directly
2. **No duplicates** - JOINs can produce duplicate rows when there are multiple matches
3. **Optimizer efficiency** - SQLite often stops at the first match
4. **Consistency** - Aligns with the Permission vs State architecture

**Pattern:**
```sql
-- Good: EXISTS for existence check
WHERE EXISTS (SELECT 1 FROM courses c WHERE c.creator_id = u.id AND c.deleted_at IS NULL)

-- Avoid: JOIN can cause duplicates and is semantically less clear for existence checks
JOIN courses c ON c.creator_id = u.id WHERE c.deleted_at IS NULL
```

## 3. Contextual Roles vs Global Roles in UI
**Topics:** auth, astro

**Context:** Investigated whether role badges were incorrectly showing a single role when users could have multiple roles.

**Learning:** The codebase correctly distinguishes between two role display contexts:

1. **Global Roles** (profile pages, admin panel, leaderboard): Shows ALL applicable roles - a user can display Creator, S-T, and Student badges simultaneously

2. **Contextual Roles** (course feeds): Shows the user's role **for that specific course** - uses a priority hierarchy:
   - Creator (of this course)
   - Admin
   - Student-Teacher (certified for this course)
   - Graduate (completed this course)
   - Student (enrolled in this course)

This is correct design - showing "Creator" on a course feed where you're enrolled as a student would be misleading.

**Pattern:**
```typescript
// Course feed contextual role (priority-based, single role)
if (userId === creatorId) return { role: 'creator' };
if (user?.is_admin) return { role: 'admin' };
if (isSTForThisCourse) return { role: 'student-teacher' };
if (completed) return { role: 'graduate' };
if (enrolled) return { role: 'student' };

// Profile page global roles (all applicable)
const roles = [];
if (is_admin) roles.push('Admin');
if (is_creator) roles.push('Creator');  // derived from having courses
if (is_student_teacher) roles.push('S-T');  // derived from active ST records
```

## 4. Removing Redundant State Updates
**Topics:** d1, architecture

**Context:** Found two endpoints (`approve.ts`, `revoke.ts`) still trying to UPDATE a `is_student_teacher` column that no longer exists.

**Learning:** When state is derived from relationship tables, you don't need to manually update user flags. The act of inserting/updating the relationship table row IS the state change.

For certificate approval:
- **Before:** Insert ST record → Update `users.is_student_teacher = 1`
- **After:** Insert ST record (state is automatically derived)

For certificate revocation:
- **Before:** Deactivate ST record → Check if other active STs → Maybe update `users.is_student_teacher = 0`
- **After:** Deactivate ST record (state is automatically derived)

This simplifies code and eliminates bugs where the flag and table get out of sync.
