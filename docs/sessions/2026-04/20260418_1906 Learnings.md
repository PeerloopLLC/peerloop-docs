# Conv Learnings - 2026-04-18

## 1. Decision Records vs Reference Docs Must Be Separate
**Topics:** docs-infra, auth, drift-patterns

**Context:** All 4 Opus audits this conv exposed the same failure mode. `auth-libraries.md` mixed decision rationale (why jose/bcryptjs/arctic were chosen) with code examples (how to use them). The rationale is stable; the examples drift. `DB-API.md §Authentication` wrote specs for endpoints that never shipped. `plato.md Step Catalog` table fell behind the source-of-truth `tests/plato/steps/*.ts` files.

**Learning:** Docs drift when they try to do two jobs at once. A doc should be **either** a decision record (frozen-ish history of *why*) **or** a living reference (auto-regeneration-friendly *how*). Mixing them guarantees the how rots while the why survives, producing misleading composite docs.

**Pattern:** For decision records: include library/pattern choices + rationale + comparison matrix; point to code for implementation. For reference docs: cite endpoint-file paths, use generators where possible, keep prose minimal. The `route-api-map.generated.ts` → `route-api-map.md` pattern is a good model; DB-API.md could do similar.

## 2. Session Files Are Historical Records — Don't Edit Them
**Topics:** docs-infra, sessions, audit-methodology

**Context:** DEVCOMP-REVIEW task implied reviewing 59 (actually 114) session files for machine-specific drift. The temptation was to edit stale session notes. But session files are immutable records of what was done; `devcomputers.md` is the living synthesis. Once a machine-specific gotcha is captured in devcomputers.md, its origin in the session file is historical context, not drift.

**Learning:** "Periodic review" of session files should ask "has new info accumulated *in sessions* that hasn't flowed *out to* living docs?" — not "are the sessions still accurate?" Sessions don't go stale; they age out of relevance but stay truthful about their moment.

**Pattern:** When doing retrospective audits, separate (a) records (immutable) from (b) synthesis targets (mutable). Only edit (b). Delete stale tasks that accumulated over multiple RESUME-STATE snapshots.

## 3. `/r-start` Workflow Mechanics vs Actual Work Commits
**Topics:** cc-workflow, dual-repo

**Context:** After finishing fetchCourseDetailData deletion, `/r-commit2` produced two commits: code (actual work) and docs (RESUME-STATE.md deletion from /r-start). Initial framing gave them different Blocks (`RA-SSR` vs `(misc)`) because the docs commit was purely workflow mechanics, not content work. User pushed back — from the day-level view, both commits were "post-Conv-130 cleanup."

**Learning:** Workflow-mechanic commits (RESUME-STATE.md deletion, package-lock.json sync, conv-counter bumps) can reasonably be attributed to the same Block as the conv's real work when the user views them as a unified cleanup activity. "Strict provenance" framing (docs commit is mechanically unrelated) wasn't wrong, but was too narrow for the timecard's purpose.

**Pattern:** When `/r-commit2` produces paired code + docs commits and only one has substantive work, default to giving the docs commit the same `Block:` and `Block-summary:` unless the docs change is genuinely standalone.

## 4. Sandbox-Safe Paths When Spawning Subagents
**Topics:** cc-workflow

**Context:** (Pattern reminder from /r-end2 spec.) Subagent sandboxes block `/tmp` writes — the prune manifest must live at `$CLAUDE_PROJECT_DIR/.extract-manifest.txt` (gitignored in docs repo root) or the learn-decide agent silently fails to record consumed lines.

**Learning:** When adding cross-agent coordination files, use docs-repo-relative paths, never `/tmp`.
