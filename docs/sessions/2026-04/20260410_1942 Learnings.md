# Conv Learnings - 2026-04-10

## 1. Existing helpers can sit dead for months without adoption sweeps
**Topics:** testing, test-infrastructure

**Context:** `tests/api/helpers/api-test-helper.ts` contained a full set of type-safe JSON response helpers (`json<T>`, `expectSuccess<T>`, `expectError`, `expectJSONResponse<T>`, `getResponseJSON<T>`, `expectRedirect`) that were all re-exported from `@api-helpers/index.ts`. Only `json<T>` was used (by 4 integration tests); the other 5 were dead.

**Learning:** When investing in type-safe helpers, adoption must be enforced by a sweep at creation time (or by a lint/grep gate). Otherwise the helpers sit dead while the old pattern persists. The 198-file `.json() as any` debt in Conv 102 existed because the helpers were added but not swept.

**Pattern:** After adding a helper meant to replace a pattern: (a) add a grep-based reminder or (b) do the sweep immediately or (c) add an eslint rule. Leaving it to organic adoption leads to a partial-adoption mess.

## 2. ts-morph codemods beat parallel subagents when the pattern is uniform
**Topics:** testing, tooling, skills

**Context:** Original sweep estimate was ~30 files via parallel subagents. Re-grep revealed 198 files with 1,587 sites. The options were (A) 4–6 parallel subagents or (B) a ts-morph codemod.

**Learning:** For uniform mechanical patterns (≥90% same shape), a codemod is faster, more consistent, and produces a bisectable single commit. Subagents produce N slightly-different style variations and are harder to verify. The codemod wrote + ran + verified in under 1 hour; subagents would have been 2–4 hours.

**Pattern:** Decision rule: if the scope is >50 files AND the pattern is >90% uniform, write a ts-morph codemod. If the scope is <50 files OR the pattern requires semantic judgment per file, use subagents.

## 3. Non-optional `any` fields preserve `as any` ergonomics while catching top-level typos
**Topics:** testing, typescript

**Context:** When migrating `const body = await response.json() as any` to `json<T>`, had to choose the shape of T: optional `{ field?: unknown }`, optional `{ field?: any }`, required `{ field: any }`, or `Record<string, any>`.

**Learning:** Required fields with `any` values are the sweet spot:
- `body.nonExistent` → TypeScript error (catches top-level typos)
- `body.stats.find(...)` → works (any propagates to nested)
- `body.data[0]` → works (any allows index access)
- Optional (`?:`) fields produce `any | undefined` which breaks chained access under strict null checks
- `unknown` values break nested access entirely

**Pattern:** For mechanical test-shape inference: use `{ field1: any; field2: any; ... }` where fields are the set of top-level property accesses found in the enclosing scope.

## 4. Time-fragile test patterns fail only at certain wall-clock times
**Topics:** testing, timezone, availability

**Context:** The 5 failing session-creation tests used `new Date(Date.now() + Nh)` for scheduled_start and `start + 1h` for end. They passed whenever `Date.now()` fell in early UTC hours (California morning/noon) and failed when late UTC (California afternoon/evening).

**Learning:** Any test that constructs a date from `Date.now()` and then adds hours to build an end time can cross midnight, exposing latent day-boundary bugs in code under test. `isSlotWithinAvailability` checks only the `startDate`'s day, so a midnight-crossing session fails against any single-day availability window. The test was fragile; the app behavior was also suboptimal, but the immediate failure was the test's fault.

**Pattern:** Replace `new Date(Date.now() + Nh)` with `futureAt(daysFromNow, utcHour=12)` which pins the hour to noon UTC so +1h never crosses midnight:
```typescript
function futureAt(daysFromNow: number, utcHour = 12, utcMinute = 0): Date {
  const base = new Date(Date.now() + daysFromNow * 24 * 60 * 60 * 1000);
  return new Date(Date.UTC(
    base.getUTCFullYear(), base.getUTCMonth(), base.getUTCDate(),
    utcHour, utcMinute, 0
  ));
}
```

## 5. Baseline claims in RESUME-STATE.md are hypotheses until re-verified
**Topics:** skills, cc-workflow, session-discipline

**Context:** Conv 101's RESUME-STATE.md claimed "tests: 6399/6399 passing (366 test files)". Conv 102 ran the full suite for the first time and found 5 failing tests. The 5 failures had been lurking for an unknown number of conversations because no one ever re-ran the suite to verify.

**Learning:** When RESUME-STATE numbers are copy-forwarded without re-verification, regressions hide in plain sight. Every claimed baseline must come from a command that actually ran in the current conv — not inherited from the previous conv's RESUME-STATE. This rule is now encoded in `feedback_verify_baselines_in_conv.md` [MR].

**Pattern:** At `/r-end` time, the update-plan or docs agents must either (a) cite a verification command from the current conv's transcript, or (b) mark the baseline explicitly: `(unchanged from Conv N, not re-verified this conv)`.

## 6. SKILL.md `!` backtick commands fail if the script exits non-zero
**Topics:** skills, cc-workflow, tooling

**Context:** `/r-end` started failing with `Shell command failed for pattern "!\`...resume-state-check.sh\`": EXISTS — saved on:`. The script was doing `grep 'Saved:'` on a file whose format had changed and no longer contained `Saved:`. grep exited 1 (no match), which the harness treated as a command failure, blocking SKILL.md from loading.

**Learning:** `!` backtick expressions in SKILL.md are pre-computed context, not validation. They must always exit 0 regardless of what they find. Any script that uses grep, find, test, or similar commands that exit non-zero on "no match" must explicitly `|| true` the failing command or `exit 0` at the end.

**Pattern:** Every script invoked from a SKILL.md `!` backtick should end with `exit 0` as a safety net. Add a comment: `# This is an informational probe, not a validation — must always exit 0`.
