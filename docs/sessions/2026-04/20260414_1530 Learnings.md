# Conv Learnings - 2026-04-14

## 1. CF Workers + Static Assets: subrequests bypass `run_worker_first`
**Topics:** cloudflare, deployment, astro

**Context:** Conv 114 migrated from Pages → Workers. SSR pages doing `await fetch(new URL('/api/…', Astro.url.origin))` started returning text/plain 404 — but ONLY for SSR self-subrequests; external curls to the same endpoint returned 200. Wrangler tail showed no Worker invocation for the subrequest path.

**Learning:** `run_worker_first` (boolean or path-array) in `[assets]` configures CF's external-edge routing but has zero effect on Worker-internal subrequests. CF treats `fetch(self-origin)` from inside a Worker as a request that goes to the Assets layer — and that layer 404s without falling through to the Worker, even with `run_worker_first = true`. Verified empirically by deploying both forms and inspecting tail.

**Pattern:** Don't try to fix SSR self-fetch with assets routing config. Either (a) use a `[[services]]` self-binding and call `env.SELF.fetch(url)` to bypass the assets layer, or (b) refactor SSR to import data-loader functions directly (the chosen approach for Peerloop).

## 2. CF Pages → Workers regression surfaces only in deployed Workers
**Topics:** cloudflare, astro, testing, deployment

**Context:** Migration from Pages to Workers (Conv 114) introduced the SSR self-fetch bug, but it didn't show up in `npm run dev` (Vite + miniflare) or in the unit test suite. Only the deployed staging Worker exhibited it.

**Learning:** Pages had a single-layer routing model — every request hit the SSR Function, including subrequests. Workers + Static Assets has a two-layer model where Assets sit in front of the Worker. Dev (miniflare + Vite) doesn't emulate the two-layer routing accurately enough to catch this. Unit tests call API handlers as functions and never exercise SSR fetch paths.

**Pattern:** After any Cloudflare adapter migration, do a mandatory live-staging smoke pass on every `.astro` page that does HTTP-style data loading. `dev:staging` would catch this if it actually used remote bindings (see [DS]).

## 3. SSR self-fetch is an antipattern even when it works
**Topics:** astro, testing

**Context:** The 8 community/discover pages used `await fetch(new URL('/api/…', Astro.url.origin), { headers: { Cookie: Astro.request.headers.get('Cookie') || '' } })` — a recognisable pattern that copies cookies forward and round-trips through HTTP. The rest of the codebase (loaders/about, courses, creators, home, etc.) already used direct DB-loader functions.

**Learning:** Even ignoring the CF Workers regression, SSR self-fetch is wasteful: ~2× SSR latency, double JSON serialise/parse, manual cookie forwarding, no shared types between caller and callee, and silent failure modes (non-OK → fetchError, opaque to debugging). The codebase already had the "right" pattern — these 8 pages were just an inconsistency.

**Pattern:** SSR pages should call typed loader functions directly (`fetchXxxData(db, params)`). API endpoints that need to be reachable from the browser should be thin wrappers around the same loaders. Never round-trip through HTTP from inside a Worker.

## 4. Wrangler 4.x requires User:Memberships:Read scope
**Topics:** cloudflare, deployment

**Context:** Old `CLOUDFLARE_API_TOKEN` worked fine pre-Conv-114. After PACKAGE-UPDATES bumped wrangler 4.54 → 4.81.1, every D1 remote command failed with API code 9106 on `/memberships`.

**Learning:** Newer wrangler versions probe `/memberships` to resolve account context for D1/R2/KV remote operations. If the token lacks `User → Memberships → Read`, every remote command fails with a misleading "Authentication failed" error. `wrangler whoami` still works because it doesn't need that scope.

**Pattern:** When generating CF API tokens for wrangler use, always include `User → Memberships → Read` and `User → User Details → Read`. If the token grants access to >1 account, also set `CLOUDFLARE_ACCOUNT_ID` so wrangler can pick non-interactively.

## 5. Account API Tokens cannot grant User-scope permissions
**Topics:** cloudflare, deployment

**Context:** Tried to add `User → Memberships → Read` to the existing token via the dashboard API Tokens page (`/account/.../api-tokens`). The scope dropdown in the policy editor only offered Account-resource scopes (Entire Account, All Domains, Specified Domains, R2 Buckets) — no User option. Same in the Create Token flow on the same page.

**Learning:** Account API Tokens are restricted to account-level permissions by design. To grant User-scope perms (Memberships, User Details), you must use User API Tokens at `/profile/api-tokens` instead.

**Pattern:** For wrangler tokens that need user-scope, generate from `https://dash.cloudflare.com/profile/api-tokens`, not from the account-scoped tokens page.

## 6. Wrangler reads `.env` from cwd implicitly
**Topics:** cloudflare, deployment

**Context:** `CLOUDFLARE_API_TOKEN` was not set in the user's shell or `~/.zshrc`, yet `wrangler whoami` reported it was reading the token from env. Found it in `~/projects/Peerloop/.dev.vars` with `.env` symlinked to it.

**Learning:** Wrangler auto-loads `.env` from the project directory at command time, picking up `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID` for authentication (in addition to the `.dev.vars` parsing it does for `wrangler dev`).

**Pattern:** When debugging wrangler auth: check `.env` / `.dev.vars` in the project root before assuming env-shell config. Useful for keeping per-project tokens separated from global env.

## 7. vi.mock module-internal calls don't catch internal callers
**Topics:** testing

**Context:** Refactored API handlers to use `getCurrentUserId(cookies, jwtSecret)` from `@/lib/auth`. 7 tests broke. Tests mock `vi.mock('@/lib/auth', { ...actual, getSession: vi.fn() })` — only `getSession` is overridden. `getCurrentUserId` from `actual` calls `getSession` via its OWN module scope, not via the import path the test mocked, so the mock never fires.

**Learning:** `vi.mock` replaces the module exports for IMPORTERS, but not for module-internal callers. If `module-A` exports `outer()` and `inner()`, and `outer()` calls `inner()` directly, mocking `module-A.inner` from a test won't change what `outer()` sees — `outer()` still calls the real `inner()`. Workarounds: (a) call `inner` exclusively from outside the module, or (b) use `vi.spyOn(moduleA, 'inner')` to patch the module export object, or (c) just call the most-mocked function directly from the consumer.

**Pattern:** When a function depends on auth/db/external service calls, prefer calling the lowest-level mocked primitive directly from the consumer (in this case: API handlers call `getSession` directly, not via `getCurrentUserId`). Keeps test mocks honest.
