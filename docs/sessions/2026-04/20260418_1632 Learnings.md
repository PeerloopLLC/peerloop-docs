# Conv Learnings - 2026-04-18

## 1. Availability Window Misalignment: SessionBooking vs CourseAvailabilityPreview
**Topics:** booking, availability, frontend

**Context:** `SessionBooking.tsx` hard-coded a 28-day fetch window; `availability-summary.ts` defaulted to 30 days. Both surfaces describe the same concept to the user but used different numbers.

**Learning:** When two surfaces (booking wizard + course card) describe the same platform concept (availability horizon), they must read from the same source or use the same constant. Divergence causes user confusion without any obvious error.

## 2. Schema-Aware SQL Lint: Template-Literal Block Scanning
**Topics:** tooling, sql, codecheck, schema

**Context:** Conv 117 introduced a regression where `deleted_at IS NULL` was used in a query against `communities` (which uses `is_archived`). No existing check caught it.

**Learning:** Effective SQL lint for column-name mistakes requires parsing SQL within its template-literal context — not just grepping for the column name globally. A file might legitimately reference both `communities` and `deleted_at` for different tables; only co-occurrence within the same SQL block is a violation.

**Pattern:**
```js
content.matchAll(/\`([^\`]*(?:SELECT|INSERT|UPDATE|DELETE|WHERE|JOIN)[^\`]*)\`/gis)
```
Captures SQL template literals for per-block analysis.

## 3. Soft-Delete Filter Must Be Explicit
**Topics:** auth, enrollment, download

**Context:** `download.ts` enrollment gate checked `status != 'cancelled'` but did not filter `deleted_at IS NULL`. Soft-deleted enrollments with non-cancelled status silently passed the gate.

**Learning:** Soft-delete columns must be explicitly filtered wherever an entity's "active" state is asserted. A `status` check alone is insufficient when the table also supports soft-deletion.
