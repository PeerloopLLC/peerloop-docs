# Conv Learnings - 2026-04-10

## 1. Direction-filter rule validated in-session by running hardened skill on itself
**Topics:** cc-workflow, docs-infra

**Context:** Right after hardening w-sync-skills with the DIRECTION block in [SD], the [SS6] run of w-sync-skills against itself hit the exact ambiguity the rule was designed to catch. Local had the DIRECTION block that source lacked (a local-only customization). The rule correctly classified it as "preserve, not port" instead of treating it as a gap.

**Learning:** Hardened rules should be tested on the same conversation where they're added if the ordering permits. The meta-test ([SS6] running on the hardened w-sync-skills skill) both validated the rule and would have surfaced any regression — cheap verification.

**Pattern:** When adding a guardrail, look for a way to exercise it in the same session. If the skill is self-applicable, run it on itself.

## 2. Direction filter surfaced signal, not just suppressed noise
**Topics:** cc-workflow

**Context:** The direction filter didn't just suppress false positives (local-only additions) — it also surfaced genuine SOURCE→LOCAL gaps cleanly in [SS2] (Doc Changes extraction), [SS5] (Schema + Cross-Document audits), and [SS6] (Step 5b). The `Source has: … / Local has: …` phrasing forced explicit comparison rather than vague "source is different" narratives.

**Learning:** Direction-filter rules don't suppress signal; they separate signal from noise. Four of seven skills had real gaps that were cleanly ported because the phrasing required comparing in the right direction.

## 3. PLAN.md status markers can lie silently for convs without a cross-check
**Topics:** plan-management, cc-workflow

**Context:** PACKAGE-UPDATES showed "🔥 IN PROGRESS (Conv 096)" with a branch field `jfg-package-updates` for three convs. The branch never existed. The block was in the "ACTIVE" table. The reality was zero code changes — only a planning commit.

**Learning:** "IN PROGRESS" markers can lie silently because nothing cross-checks them against git state. Future convs inherit the lie and either waste cycles trying to "continue" work that was never started or (worse) treat the audit as done-enough.

**Pattern:** Status markers should be verifiable. If a block claims "IN PROGRESS" with a branch name, `git branch --list {name}` should return the branch. This could be a /r-end sanity check or a /w-codecheck audit.

## 4. Generation/extraction contract must be symmetric across all readers
**Topics:** cc-workflow, docs-infra

**Context:** Conv 098 added `Doc:` tag extraction to r-timecard-day but not to r-timecard. Writer-side commits (r-commit, r-end) emitted `Doc:` lines that r-timecard silently discarded. Discovered during [SS2] skill-sync comparison.

**Learning:** Writer tags must be matched by every reader, not just one. r-commit/r-end (writer) → r-timecard-day, r-timecard, w-git-history (readers). Any new tag added to one side requires all other sides to consume it or content disappears from bill-side output.

**Pattern:** When adding a tag to a writer, list all readers and confirm each consumes it. When adding a tag to one reader, check whether peer readers need the same update.
