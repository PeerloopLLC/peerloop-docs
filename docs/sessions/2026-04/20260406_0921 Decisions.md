# Conv Decisions - 2026-04-06

## 1. Structured Tag System for Commit Messages
**Type:** Strategic
**Topics:** commits, timecards, cc-workflow

**Trigger:** Timecard Git History was verbose and useful categories were buried in generic bullets

**Options Considered:**
1. YAML metadata block in commits — too structured, requires parser
2. Bracketed tags `[API]` — hard to grep (bracket escaping), inconsistent with existing patterns
3. Prefixed lines `API:`, `Page:`, `Role:`, `Infra:` ← Chosen

**Decision:** Use prefixed single-line tags in commit bodies, matching the existing `User-facing:` / `Admin-facing:` pattern

**Rationale:** Simple prefix matching on git log output, naturally optional, backward-compatible (old commits just produce empty sections that get omitted)

**Consequences:** Updated 4 skills (r-commit, r-end, r-timecard-day, r-timecard). Timecards now have 4 new optional sections.

> **Insight:** Commit messages as structured data sources follow the Unix philosophy — keep the storage format simple and greppable, push complexity into the extraction tool. This avoids the parser-maintenance burden of YAML/JSON in commit bodies while remaining human-readable.

## 2. Replace Verbose Git History with Compact Table
**Type:** Implementation
**Topics:** timecards, readability, cc-workflow

**Trigger:** User found full commit body dumps in Git History not as useful as originally thought

**Options Considered:**
1. Keep verbose format — too much noise
2. Remove Git History entirely — loses commit reference
3. Compact table (Time, Conv, Repo, Hash, Machine, Block) ← Chosen

**Decision:** Replace per-commit verbose blocks with a single chronologically-sorted table interleaving both repos

**Rationale:** All important categorical detail now lives in structured tag sections (API/Page/Role/Infra); Git History only needs to serve as a commit reference, not a content dump

**Consequences:** Updated r-timecard-day and r-timecard templates. Date line added above table for standalone readability.
