# Conv Learnings - 2026-04-01

## 1. Client-Server Contract Mismatches Are STUMBLE-Only Findings
**Topics:** testing, plato, stumble-audit

**Context:** PLATO proved POST /api/me/onboarding-profile saves correctly and GET returns the data. But the OnboardingProfile component crashed on revisit because it expected `topicInterests` while the API returns `tags`.

**Learning:** PLATO tests the API layer (correct data flows, DB state). STUMBLE tests the UI layer (can the component render that data). Contract mismatches between API response shapes and component type expectations only surface when the browser actually renders. This validates the PLATO+STUMBLE pairing — they're complementary, not redundant.

## 2. Astro Modal State Persists Across Page Navigation
**Topics:** astro, react, auth

**Context:** Clicking "Forgot password?" navigated to /reset-password but the login modal stayed open on top, blocking the reset form.

**Learning:** In Astro's multi-island architecture, global state stored on `window.__peerloop` persists across full-page navigations. The modal island on the new page reads stale "open" state and re-renders. Fix: clear global state before navigation (closeAuthModal() in onClick alongside href).

## 3. BaseHead.astro Appends "| Peerloop" — Pages Must Not Include It
**Topics:** astro, ui

**Context:** login.astro, signup.astro, reset-password.astro, and onboarding.astro all passed titles ending in "| Peerloop" or "| PeerLoop", producing double suffixes like "Log In | Peerloop | Peerloop".

**Learning:** BaseHead.astro:30 appends `| Peerloop` to every title. Page title props must contain only the page-specific part (e.g., "Log In" not "Log In | Peerloop"). Additionally, ~70 pages use "PeerLoop" (capital L) which will also produce double suffixes — needs bulk fix.

## 4. PLATO Instance System: Thin Orchestration Over Existing Infrastructure
**Topics:** plato, testing, architecture

**Context:** Built the complete instance system (types, runner, reporter, step, personas, instance file, tests) — all working first try.

**Learning:** The instance system worked immediately because it's a thin orchestration layer. The `when` guard adds 3 lines to the runner loop. `executeInstanceFile()` swaps persona data per-instance and delegates to existing `executeScenario()`. No changes to step execution, value resolution, or mock management. The existing architecture (actorBindings, runtimeOverrides) was already designed for pluggable data — instance is the next logical layer up.
