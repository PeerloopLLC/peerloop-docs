# Conv Decisions - 2026-04-15

## 1. Hard-kill `member_role='teacher'` in `0001_schema.sql` (no upgrade migration)
**Type:** Strategic
**Topics:** d1, deployment

**Trigger:** Conv 119 audit identified `community_members.member_role='teacher'` as dead-code drift; needed kill plan before doing any UI work.

**Options Considered:**
1. Hard kill in `0001_schema.sql` ← Chosen (after pre-release context)
2. Soft deprecate (leave CHECK wide, stop seeding/reading 'teacher')
3. Hybrid: narrow `0001_schema.sql` + add `0003_narrow_member_role.sql` upgrade migration for staging/prod

**Decision:** Edit `0001_schema.sql:222` directly to narrow the CHECK to `('creator','member')`. Reset local + staging + remote D1 after the kill lands. No `0003_*.sql` migration.

**Rationale:** User confirmed pre-release status. The project's own DB tooling (`db:reset:staging`, `db:reset:remote`) already supports clean schema mutations. Hybrid would have added a migration file that becomes irrelevant the moment we ship.

**Consequences:** Pre-release schema mutability is now a documented decision pattern — defer migration ceremony until release-candidate.

## 2. Re-derive teaching communities server-side in `/api/me/full` (Q2c)
**Type:** Implementation
**Topics:** astro, code-quality, currentuser

**Trigger:** Need to replace 3 React sites that filtered `memberships.filter(m => m.memberRole === 'teacher')`.

**Options Considered:**
1. Add `getTeachingCommunityIds()` getter on CurrentUser computed from existing `teacherCertifications` map (would need course→community lookup data, not currently in CurrentUser)
2. Inline the join at each of 3 React call sites
3. Server-side: add `teachingCommunityIds: string[]` to `/api/me/full` response ← Chosen

**Decision:** Server-side derivation. New `fetchTeachingCommunityIds()` joins `teacher_certifications → courses → progressions` once at API boundary; CurrentUser exposes via `getTeachingCommunityIds()` and `isTeachingIn(communityId)`.

**Rationale:** Single source of truth, hides the join complexity, parallels existing `communityModeratedCourseIds` pattern. CurrentUser already has the parallel `Set<string>` shape for moderation-derived course IDs — keeps the pattern consistent.

**Consequences:** New field is optional on `MeFullResponse` for backward compat with cached payloads (constructor uses `?? []` fallback). `getFeeds()` updated to also push `'teacher'` role from this set, preserving the per-community role annotation behavior previously driven by `member_role`.

## 3. Extract permissions to `src/lib/permissions.ts` helper (Q3b)
**Type:** Implementation
**Topics:** astro, code-quality

**Trigger:** 6 Astro pages had identical inline `canUploadResources={membership?.role === 'creator' || membership?.role === 'teacher'}` — and the missing `isAdmin` branch was the [UI-ADMIN-GATE] bug from Conv 119.

**Options Considered:**
1. Inline fix all 6 pages: `membership?.role === 'creator' || currentUser?.isAdmin`
2. Extract `canUploadCommunityResources(membership, isAdmin)` to `src/lib/permissions.ts` ← Chosen

**Decision:** New helper file with pure function over already-fetched DTOs. Extended `fetchCommunityDetailData` SSR loader to expose `isAdmin: boolean` (refactored existing admin query so it now feeds both `canManageModerators` and `isAdmin`).

**Rationale:** 6 sites is past the duplication threshold. Pure helper is trivially testable (5 unit tests, including a defensive case for the retired `'teacher'` value never granting upload rights). Loader extension is a minimal additive change.

**Consequences:** New `src/lib/permissions.ts` module is the future home for similar gating logic. The pattern of "loader exposes raw flags + helper composes them" keeps Astro pages declarative and JSX-friendly.
