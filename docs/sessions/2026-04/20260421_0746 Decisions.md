# Conv Decisions - 2026-04-21

## 1. [CT] — Option 2 (new `detectOrphanedParticipants` function) over tightening grace or widening reconcile
**Type:** Implementation
**Topics:** cloudflare, video, d1, testing

**Trigger:** Phase B Cron timeout requirement — empty-room auto-complete requires both participants fire `participant_left`, leaving one-sided crashes stuck until the 1-hour DB backstop.

**Options Considered:**
1. Quick — tighten `detectStaleInProgress` grace from +1h to +15m
2. Durable — new `detectOrphanedParticipants()` function wired before `detectStaleInProgress` ← Chosen
3. Middle — widen `reconcileBBBSessions` time window to include mid-session checks

**Decision:** Add `detectOrphanedParticipants` as a new exported function in `src/lib/booking.ts`, called from `runSessionCleanup` before `detectStaleInProgress`. Uses BBB as source of truth, attendance-aware (narrower scope than reconcile).

**Rationale:** Matches existing pattern (BBB-authoritative detection as a dedicated function alongside `reconcileBBBSessions`). Doesn't change grace semantics of either existing path. Produces cleaner attendance records as a side effect (force-closes orphaned rows). Avoids false-positives on legitimate long sessions (Option 1) and on mid-session API stalls (Option 3).

**Consequences:** New function + new summary field `orphaned_completed` + new endpoint response key. `detectStaleInProgress` at +1h remains the final DB-only backstop when BBB is unreachable. 4-stage narrowing cascade in `runSessionCleanup`: noShows → orphaned → stale → reconcile.

## 2. [IO] — Partial unique index + `INSERT OR IGNORE` over `SELECT EXISTS` check
**Type:** Implementation
**Topics:** d1

**Trigger:** [IO] task description listed both options explicitly; pre-launch so schema edit to `0001_schema.sql` is the cheap path.

**Options Considered:**
1. `SELECT EXISTS` guard in webhook handler before INSERT
2. Partial unique index on `(session_id, user_id) WHERE left_at IS NULL` + `INSERT OR IGNORE` ← Chosen

**Decision:** Add partial unique index, use `INSERT OR IGNORE`. Allows legitimate rejoins; duplicate opens are DB-level atomic no-ops.

**Rationale:** `SELECT EXISTS` has a small race window between check and insert (two simultaneous webhook deliveries could both pass the check). Partial index pushes the invariant down to the DB — atomic, race-free. Since schema is pre-launch, edit `migrations/0001_schema.sql` directly per CLAUDE.md.

**Consequences:** Duplicate `participant_joined` deliveries silently produce one row. Legitimate join → leave → rejoin still produces 2 rows (new test proves it). No test regressions.

## 3. [DF] — Centralize `started_at` backfill in `completeSession`, not in the webhook handler
**Type:** Implementation
**Topics:** d1

**Trigger:** [DF] task description said "in src/pages/api/webhooks/bbb.ts OR completeSession()." Analysis showed 4 of 5 `completeSession` callers have no duration data.

**Options Considered:**
1. Backfill in `handleRoomEnded` only (webhook caller), leave other paths unchanged
2. Backfill in `completeSession`, with duration as optional parameter ← Chosen

**Decision:** Extend `completeSession(db, sessionId, endedAt?, durationSeconds?)`. Compute `inferredStartedAt = (endedAt − durationSeconds)` if duration provided, else `scheduled_start`. UPDATE uses `COALESCE(started_at, ?)` to preserve existing values.

**Rationale:** Any path that completes a session with null `started_at` has the same invariant violation (duration-dependent reads break). Centralizing the fallback means cron paths (`detectStaleInProgress`, `reconcileBBBSessions`, admin endpoint) automatically get a sensible default (`scheduled_start`) without per-caller plumbing. Webhook path keeps its authoritative fallback via the new parameter.

**Consequences:** One function signature change; one UPDATE change. 2 new tests prove the backfill and the preservation semantics. No other callers need modification.
