# Conv Decisions - 2026-04-03

## 1. FormModal over InputModal for prompt() replacement
**Type:** Implementation
**Topics:** react, admin-ui, modals

**Trigger:** 23 prompt() calls across 6 files needed replacement; 3 options presented

**Options Considered:**
1. Single-field InputModal (mirrors ConfirmModal pattern)
2. Multi-field FormModal ← Chosen
3. Both InputModal + FormModal

**Decision:** Build one FormModal component that handles all prompt() use cases via a `fields` array

**Rationale:** Multi-prompt sequences (suspend: duration→reason→notes) and constrained choices (resolution type with 7 options) are terrible UX as sequential popups or free-text. A single form modal with dropdowns + textareas handles all 23 calls with one component.

**Consequences:** Created `src/components/ui/FormModal.tsx`. 6 files modified, 2 test files rewritten. Test count 6363→6361 (2 invalid-duration tests removed — structurally impossible with select dropdown). Zero prompt() calls remain in src/.

## 3. Bidirectional admin↔member links: dual-link pattern
**Type:** Strategic
**Topics:** admin-ui, navigation, architecture

**Trigger:** User corrected initial proposal that framed admin-to-admin as replacing admin-to-member

**Options Considered:**
1. Replace memberUrlFor with adminUrlFor (rejected)
2. Add adminUrlFor alongside memberUrlFor ← Chosen

**Decision:** Each entity reference in admin detail panels shows two links: admin context (adminUrlFor, primary) + member context (memberUrlFor, secondary). Never remove existing member links.

**Rationale:** Admin-to-member links are the admin's window into what members experience + ADMIN-INTEL overlay access. Admin-to-admin links provide entity-in-context-of-like-entities view. Both directions serve distinct, complementary purposes.

**Consequences:** PLAN.md .MENU §D and .UI F4 updated with dual-link pattern, CRITICAL callout, design principle added.
