# Conv Learnings - 2026-04-15

## 1. PLATO cross-step state requires per-step discovery GET
**Topics:** plato, testing

**Context:** First attempt at the new `upload-community-resources` step referenced `$context.createCommunity.communitySlug` from the prior step in the chain. Test failed: "No context for action 'createCommunity'. Has it executed yet in this run?" — the runner's context is per-step, not per-scenario.

**Learning:** PLATO steps are isolated context units within a scenario. To use state from a prior step (e.g., a slug created earlier), the new step must do its own discovery GET (e.g., `GET /api/me/communities` then `findBy(name,$persona.X).slug`). This is the established pattern in `add-modules.step.ts` (which discovers `courseId` via `loadCourses`).

**Pattern:** First visit/action in a downstream step should be a discovery GET that re-derives the IDs/slugs it needs from persona-stable fields (name, email, handle).

## 2. Audit findings calibrate against actual call graph, not just file presence
**Topics:** audit, dead-code

**Context:** The Conv 123 ROLE-AUDIT report (§A.2) flagged `UserProfile.tsx`'s `/api/me/profile` and `/api/me/stats` fetches as migration candidates with the caveat "verify self-view conditional before migrating." Verification revealed `UserProfile.tsx` itself has zero callers across `src/`, `.astro`, and dynamic imports — the file is dead code.

**Learning:** Audit categorizations based on file content alone (e.g., "this component fetches role state") miss the upstream question of whether the file is even reachable. Adding a one-line `grep -rn "ComponentName"` reachability check before classifying migration candidates would have surfaced the dead-code finding in the audit itself, not as a downstream discovery.

**Pattern:** Audit reports of UI components should classify each as REACHABLE / DEAD before recommending migration work. Dead components → delete; reachable → migrate.

## 3. `.catch(() => null)` masks phantom-URL bugs indefinitely
**Topics:** error-handling, testing

**Context:** During [RA-API], discovered `/api/me/stats` endpoint never existed. `UserProfile.tsx` was hitting a phantom URL whose 404 was absorbed by `await fetch('/api/me/stats').catch(() => null)`. The component shipped with a permanently-broken stats fetch that nobody noticed because the catch always produced `null` and the UI degraded gracefully.

**Learning:** `.catch(() => null)` on an `await fetch()` masks both network failures AND missing endpoints. Both surface as "stats unavailable" with no log, no telemetry, no test failure. Combined with a component that's never mounted (Learning #2), the bug was undetectable without an explicit endpoint-existence audit.

**Pattern:** When swallowing fetch errors, log at minimum the URL and status; consider distinguishing "endpoint 404" (programming bug) from "network/server error" (transient). Alternatively, fail loud in dev and degrade silent only in prod.
