# Conv Learnings - 2026-04-06

## 1. Availability Validation Must Be Seeded in All Booking Tests
**Topics:** testing, plato, booking

**Context:** Adding server-side availability window validation to `POST /api/sessions` broke 19 tests across 2 files because existing test setups had no availability rows for teachers.

**Learning:** Any test that books a session (directly or via PLATO scenario chains) must seed availability data for the teacher. When adding server-side validation, audit ALL test paths that touch the validated endpoint — not just the direct unit test file.

**Pattern:** Add all-day/all-week UTC availability in test beforeAll for simplicity: `INSERT INTO availability (id, user_id, day_of_week, start_time, end_time, timezone, is_recurring) VALUES (?, ?, ?, '00:00', '23:59', 'UTC', 1)` for days 0-6.

---

## 2. PLATO Scenario Chains Must Match API Validation Requirements
**Topics:** plato, testing

**Context:** The `set-availability` step existed but wasn't in the flywheel or ecosystem scenario chains. The `complete-course` step books sessions via `POST /api/sessions` — which now requires availability.

**Learning:** PLATO scenario chains must be kept in sync with API validation requirements. When adding server-side validation, check whether PLATO steps that call that endpoint have prerequisite steps in their chains.

---

## 3. PLATO Step Data Should Be Test-Friendly
**Topics:** plato, testing

**Context:** The original `set-availability` step created 3 slots on specific days (Mon/Wed/Fri) in America/New_York timezone. Session times are generated from `Date.now()` and could land on any day — causing intermittent failures.

**Learning:** PLATO steps that seed data for other steps should use maximally permissive values (all days, wide time windows, UTC) to avoid calendar-dependent test flakiness.
