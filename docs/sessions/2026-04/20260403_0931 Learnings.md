# Conv Learnings - 2026-04-03

## 1. FormModal replaces both prompt() and ConfirmModal+prompt combos
**Topics:** react, admin-ui, modals

**Context:** 23 prompt() calls across 6 admin/moderation files with 6 distinct patterns (optional text, required text, constrained choice, numeric, pre-filled, multi-sequence)

**Learning:** A multi-field FormModal with `fields` array (text/textarea/select/number/email) handles all prompt() use cases in one component. ConfirmModal calls that had embedded prompt() in their onConfirm callback become FormModal calls directly — the confirmation message becomes the FormModal description. Constrained choices that were free-text prompt() with validation become `<select>` dropdowns, making invalid input structurally impossible.

**Pattern:** `useState<FormModalState | null>` + `<FormModal state={formState} onClose={() => setFormState(null)} />` — same callback-in-state pattern as ConfirmModal.

## 2. Test element selection in pages with shared primitives
**Topics:** testing, react, admin-ui

**Context:** ModeratorQueue tests failed because `getByRole('combobox')` and `getAllByRole('textbox')` matched page elements (search bar) in addition to FormModal fields

**Learning:** When testing modals/overlays on pages that have shared filter/search primitives, use specific selectors: `getByPlaceholderText` for known placeholder text, `getAllByRole` with index selection (FormModal elements are appended after page elements so typically last), or scope queries within the modal container. Avoid `getByRole` for common roles (textbox, combobox) on complex pages.

## 3. Admin↔Member bidirectional boundary crossing is a design principle
**Topics:** admin-ui, navigation, architecture

**Context:** Initial analysis framed admin-to-admin links as replacing admin-to-member links. User corrected: both directions are intentional.

**Learning:** The admin system's `memberUrlFor` links are not a deficiency to fix — they're a deliberate design that keeps admins "in touch" with the member experience and provides ADMIN-INTEL overlays for in-context decisions. Admin-to-admin links (`adminUrlFor`) are additive. Each entity reference in detail panels should offer both directions: admin context (primary) + member context (secondary). The infrastructure (`admin-links.ts`) already supports both; only the wiring is missing.

## 4. Admin system risk model: decision-data and action-execution
**Topics:** testing, admin-ui, architecture

**Context:** User framed the admin as "2 max users, often used" — regressions are the primary risk since admins develop usage patterns and don't test breadth

**Learning:** For low-user/high-trust systems, test priority should be by decision consequence, not user traffic. A refund endpoint used monthly matters more than a dashboard endpoint hit daily because the refund is irreversible. Decision-data tests (what the admin sees) are as important as action-execution tests (what the admin does) because wrong data → wrong decisions silently.
