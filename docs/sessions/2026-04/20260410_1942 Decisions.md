## 3. Non-optional `any` field shapes in migrated response types
**Type:** Implementation
**Topics:** testing, typescript

**Trigger:** Deciding what T to use in `json<T>(response)` at each migrated call site

**Options Considered:**
1. Optional `{ field?: unknown }` — safest typing but breaks nested access under strict null checks
2. Optional `{ field?: any }` — breaks chained access
3. Required `{ field: any }` — catches top-level typos, preserves nested ergonomics ← **Chosen**
4. `Record<string, any>` — no typo detection

**Decision:** Required fields, `any` values, sorted alphabetically for codemod determinism

**Rationale:** Best balance of durability (catches top-level typos) and migration feasibility (preserves all existing nested access like `body.stats.find(...)`, `body.data[0]`).

**Consequences:** All 1,585 codemod-migrated sites use this shape. Does not catch nested typos (out of scope for mechanical migration).
