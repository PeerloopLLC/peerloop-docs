## 4. Platform-stats env-marker: stub row + chained UPDATE in migrate scripts
**Type:** Implementation
**Topics:** d1, deployment, infrastructure

**Trigger:** `migrations/0002_seed_core.sql` runs uniformly across local/staging/prod — cannot encode a per-env value. But `/api/debug/db-env` needs a discriminator.

**Options Considered:**
1. Per-environment SQL files (separate 0003_{env}.sql files)
2. Wrangler CLI command in each deploy pipeline (out-of-repo)
3. Stub row in seed + chained UPDATE in npm scripts ← Chosen

**Decision:** Seed `('stat-007', 'environment', 'unknown', ...)` in 0002; chain `wrangler d1 execute … UPDATE … WHERE key='environment'` after each `db:migrate:{local,staging,prod}` script.

**Rationale:** Idempotent (UPDATE is a no-op if value is correct), single source of truth for the marker row, deploy logic lives in package.json alongside migrate commands. No new files, no pipeline changes, no out-of-repo state.

**Consequences:** Running `npm run db:setup:local` (or `:staging` / migrate-direct) now self-stamps the env marker. Production inherits via `db:migrate:prod` with `confirm-prod.js` gate.
