# Conv Decisions - 2026-04-13

## 1. Deployment Target: Cloudflare Workers (not Pages)
**Type:** Strategic
**Topics:** cloudflare, deployment

**Trigger:** Conv 113's postbuild patch produced successful builds but broken deploys (SSR 404s). Adapter 13 simply doesn't emit Pages-compatible output.

**Options Considered:**
1. Extend postbuild patch to restructure output — brittle, undocumented territory, fights the adapter
2. Downgrade adapter to last Pages-supporting version — reverts deliberate Conv 104-113 upgrade work
3. Migrate to CF Workers with Static Assets ← **Chosen**

**Decision:** Peerloop ships as a single CF Worker with Static Assets. Staging = `peerloop-staging` Worker; production = `peerloop` Worker (not yet deployed).

**Rationale:** Works with the adapter's intended output shape. Durable — won't break on future adapter updates. Same account-level D1/R2/KV resources bind identically. Recorded in `docs/DECISIONS.md`.

**Consequences:** No more Git-push auto-deploy (until GH Actions ships — deferred). `wrangler.toml` reshape. Feature branches no longer get free preview URLs. Secrets need re-upload. Worker names (not branches) are the deployment unit.

## 2. Manual `wrangler deploy` Now, GitHub Actions Later
**Type:** Strategic
**Topics:** cloudflare, deployment, cc-workflow

**Trigger:** Deploy method question during Plan mode.

**Options Considered:**
1. GitHub Actions workflow from day one — most durable, most upfront work
2. Manual `wrangler deploy` forever — no CI, every deploy needs a dev
3. Hybrid: CLI now, GH Actions in a follow-up task ← **Chosen**

**Decision:** Ship CF-WORKERS with manual deploy commands (`npm run deploy:staging`/`deploy:prod`), track GH Actions as a follow-up in DEPLOYMENT.GHACTIONS block.

**Rationale:** Unblocks the migration fastest. GH Actions setup is a separate concern (secrets, workflow YAML, branch-protection rules) that shouldn't gate the primary migration.

**Consequences:** Every deploy currently needs a developer at a terminal with wrangler auth. DEPLOYMENT.GHACTIONS sub-block added to PLAN.md.

## 5. `preview_urls = false` for Production, `true` for Staging
**Type:** Implementation
**Topics:** cloudflare, deployment

**Trigger:** Wrangler warning about preview_urls not being explicitly set.

**Options Considered:**
1. `false` globally — simplest; loses preview URLs for staging
2. `false` for prod, `true` for staging ← **Chosen**
3. Leave as-is (implicit `true` everywhere) — unsafe for prod

**Decision:** Production env will never expose per-version preview URLs; staging gets them for testing deploys before they become primary.

**Rationale:** Per-version preview URLs are guessable and carry full bindings. On prod, that's a security liability — on staging with test data, it's a useful affordance. Explicit `true` on staging silences the warning too.

**Consequences:** Each deploy produces predictable behavior per environment. Recorded in `docs/reference/cloudflare.md`.

## 6. Disconnect Old Pages GitHub Integration (Client Action)
**Type:** Implementation
**Topics:** cloudflare, dual-repo, cc-workflow

**Trigger:** After staging works on Workers, pushes to staging still trigger CF Pages builds that serve broken SSR.

**Options Considered:**
1. Leave Pages auto-deploy alone — wastes CI, confuses users
2. Disconnect Pages ↔ GitHub integration ← **Chosen**
3. Delete the Pages project entirely — DANGEROUS, prod still on Pages

**Decision:** Client (org-admin on PeerloopLLC) will disconnect the Pages ↔ GitHub integration via CF Dashboard OR GitHub org settings. Existing deployed content stays live.

**Rationale:** Severs the auto-rebuild trigger without deleting the production fallback. Pages project itself deleted only after production cutover.

**Consequences:** DEPLOYMENT.PAGES-DISCONNECT sub-block added to PLAN.md as client-blocked. TaskCreate #13.
