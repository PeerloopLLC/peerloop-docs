# Conv Decisions - 2026-04-18

## 1. DOC-SYNC-STRATEGY registry schema shape — pattern-based with metadata
**Type:** Strategic
**Topics:** docs-infra, cc-workflow

**Trigger:** Needed a schema that all three drift tools can read without forcing per-file registration.

**Options Considered:**
1. Per-file explicit registry — list every doc by path with explicit category
2. Tree-level convention only — classify by directory/naming convention, no config
3. Pattern-based groups with explicit overrides ← Chosen

**Decision:** `docsRegistry.groups[]` with `id`, `category`, `pattern`/`patterns`/`docs`, `sourceOfTruth`, `checks`, `consumers`. Groups use globs for scale; explicit `docs` array for mixed-ownership groups; `notPattern` for excluding sub-patterns like `_`-prefix.

**Rationale:** Per-file registry creates friction for every doc addition (200+ files, growing). Tree-level only is too coarse — `_COMPONENTS.md` needs driftCheck even though its siblings are archival. Pattern-based + overrides captures both shape and exceptions.

**Consequences:** `config.json` gained a 138-line `docsRegistry` section. Phase 2 tools will consume it; the schema fields (`checks`, `consumers`) currently serve as documentation until Phase 2 wires them up.

## 4. Fix stale `config.json` paths inline with docsRegistry merge
**Type:** Implementation
**Topics:** docs-infra, cc-workflow

**Trigger:** Stale `paths.vendorDocs` / `paths.architectureDocs` discovered while auditing config.json consumers.

**Options Considered:**
1. Leave alone — no consumers, no runtime error
2. Remove both entries entirely
3. Correct to current folders inline with docsRegistry merge ← Chosen

**Decision:** Update `paths.vendorDocs: "docs/vendors/"` → `"docs/reference/"` and `paths.architectureDocs: "docs/architecture/"` → `"docs/as-designed/"`.

## 5. Strategy doc placement: `docs/as-designed/`, not `docs/reference/`
**Type:** Implementation
**Topics:** docs-infra

**Trigger:** Needed a home for `doc-sync-strategy.md`.

**Options Considered:**
1. `docs/reference/doc-sync-strategy.md` — alongside vendor/CLI docs
2. `docs/as-designed/doc-sync-strategy.md` ← Chosen
3. Root level (like `DECISIONS.md`)

**Decision:** Place in `docs/as-designed/` next to `skills-system.md`.

**Rationale:** It's a system-design doc describing how a subsystem (docs sync) should work. `as-designed/` is the canonical home for such docs per CLAUDE.md's documentation map. `reference/` is for usage-oriented material; root is for project-wide governance ledgers.

**Consequences:** Pattern established — future strategy/system docs for `.claude/` subsystems belong in `docs/as-designed/`, not `docs/reference/`.
