# Conv Learnings - 2026-04-03

## 1. Callback-in-State Pattern for Multiple Confirm Dialogs
**Topics:** react, ui-patterns, components

**Context:** Files like PayoutsAdmin had 5+ different confirm() calls (create, process, retry, cancel, batch). Needed a pattern that avoids N separate state variables.

**Learning:** A single `useState<ConfirmState | null>` where the state object includes an `onConfirm` async callback handles unlimited confirm dialogs per component. The action-specific context lives in the closure, the modal manages its own loading/error state, and auto-closes on success.

**Pattern:**
```tsx
const [confirmState, setConfirmState] = useState<ConfirmState | null>(null);
// Trigger: setConfirmState({ title, message, confirmLabel, variant, onConfirm: async () => {...} });
// JSX: <ConfirmModal state={confirmState} onClose={() => setConfirmState(null)} />
```

## 2. DOM-Based Toast Survives React Re-renders and Astro Island Remounts
**Topics:** react, astro, ui-patterns

**Context:** CourseEditor.tsx originally defined showToast inline because React state-based banners were lost on Astro island remounts.

**Learning:** Creating a toast via `document.createElement` with a fixed ID (`app-toast`) and `document.body.appendChild` works across all React component boundaries and Astro island lifecycles. The previous toast is removed before creating a new one (dedup by ID).

## 3. Test Migration: window.confirm/alert Spies to Modal DOM Interaction
**Topics:** testing, vitest, react-testing-library

**Context:** Replacing confirm()/alert() broke 19 tests that spied on window.confirm/window.alert.

**Learning:** Tests must now: (1) click the action button, (2) find the ConfirmModal by its title/message text in the DOM, (3) click the confirm button by role/name, (4) await the async action. For toast assertions, check `document.getElementById('app-toast')?.textContent`. Critical: use `vi.resetAllMocks()` not `vi.clearAllMocks()` to prevent cross-test contamination from `mockReturnValueOnce` queues.
