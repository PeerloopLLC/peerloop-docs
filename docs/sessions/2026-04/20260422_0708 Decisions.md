## 1. /w-sync-skills divergence gate — two independent signals (Jaccard + line-diff ratio)
**Type:** Implementation
**Topics:** cc-workflow, docs-infra

**Trigger:** Conv 140 `r-end` port attempt produced a long per-finding port plan when the right answer was "evolve independently." Memory `feedback_skill_sync_same_name_divergence.md` documented this; Conv 147 [SY] work added the gate to the skill.

**Options Considered:**
1. Single-signal: header-Jaccard < 0.70 — simplest; fails on `r-end` (Jaccard 1.00, headers identical, content forked 30%)
2. Single-signal: line-diff-ratio > 0.30 — catches `r-end` but redundant for cases where header-adds/removes are obvious (w-codecheck, w-sync-docs)
3. Two-signal OR — Jaccard<0.70 OR diff_ratio>0.30 ← Chosen — each catches a distinct class; OR-combination keeps false-negatives low
4. Two-signal AND — rejected; would require BOTH structural AND content divergence, missing `r-end` again

**Decision:** Two-signal OR gate in `/w-sync-skills` Step 3 (step 4 of the per-skill iteration). Thresholds: Jaccard < 0.70 (structural), diff_ratio > 0.30 (content). Either fires → DIVERGED → skip per-finding enumeration → present structured signal values + "evolve independently" recommendation.

**Rationale:** Calibration across all 13 peerloop-docs ↔ spt-docs exact-match skills showed the two signals are orthogonal. `r-end` (Jaccard 1.00, diff 30%) and `w-sync-docs` (Jaccard 0.30, diff 116%) are both legitimately DIVERGED but for different reasons. Single-signal designs each miss one class. Thresholds chosen from the empirical data: the memory's "~30%" language maps to Jaccard < 0.70 for structural and diff_ratio > 0.30 for content.

**Consequences:** `/w-sync-skills` now short-circuits on structurally-forked or content-forked skills with a DIVERGED status (signal values included in the presentation). The old per-finding enumeration runs only when both signals stay below threshold. Calibration data embedded in the skill so future readers understand the threshold's grounding.

## 3. Drift-fix vs Restructure classification before queuing (docs workflow convention)
**Type:** Strategic
**Topics:** docs-infra, cc-workflow

**Trigger:** Conv 140 queued "14 cosmetic drift items" for TEST-COVERAGE.md; Conv 146 sanctioned a mid-execution "Other API" restructure (10 new H3s). Two different work classes landed under one `[TC]` task; scope was accurate only in hindsight. Conv 146 §Uncategorized captured the observation.

**Options Considered:**
1. Batch drift-fix and restructure under one task — status quo, silently misrepresents scope
2. Classify at queue time with distinct task subjects ← Chosen

**Decision:** When assembling a drift-audit queue (manual or via `/w-sync-docs`), classify each finding as drift-fix (values only) vs restructure (heading + structure). Queue with distinct task subjects. Update `/w-sync-docs` SKILL.md with a "Classify Findings for Queuing" subsection.

**Rationale:** Drift-fix and restructure have different review surfaces, risk profiles, and downstream effects (restructure changes anchors; cross-references in other docs may go stale). Batching them silently misrepresents scope and mis-sizes review effort.

**Consequences:** New decision entry in DOC-DECISIONS.md §"Drift-Fix vs Restructure: Classify Before Queuing" (Conv 147). `/w-sync-docs` SKILL.md has new subsection between Report Format and Fix Mode. Cross-links to Conv 146 siblings "Avoid Catch-All 'Other'/'Misc' Sections" and "Preserve Audit-Trail Observations." Last Updated header on DOC-DECISIONS.md bumped.
