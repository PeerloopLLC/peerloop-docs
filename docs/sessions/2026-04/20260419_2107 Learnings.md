# Conv Learnings - 2026-04-19

## 1. Pre-computed context in skills can lie — verify before building on it
**Topics:** cc-workflow, skills

**Context:** `/w-sync-skills` pre-computed context stated "(no skillSync config)" — but `config.json` actually had a full `skillSync.sources[]` section with 17 replacement mappings. Building on that lie would have meant running the whole skill without any terminology normalization, surfacing hundreds of false-positive differences.

**Learning:** When a skill's pre-computed context reports "none" or "(no X config)", spot-check the underlying file once before accepting the claim. Pre-computed context is static at skill-load time; if the generator script has a bug or the config path changed, the context becomes misleading without any warning.

**Pattern:** In any skill with conditional pre-computed context (e.g., "X enabled" / "no X config"), the generator should echo the actual lookup key it used so a human can verify. For `w-sync-skills`, that would mean the pre-computed context saying "queried `config.skillSync.sources[]` → found N entries" rather than "(no skillSync config)".

## 2. Skills that "port" across projects accumulate same-name divergence over time
**Topics:** cc-workflow, dual-repo, skills

**Context:** Tried to port `r-end` improvements from `spt-docs` → `peerloop-docs`. Both skills started from a similar template, but months of independent evolution means: LOCAL uses ACTIVE/ON-HOLD/DEFERRED tables + strip-on-complete + `(Conv NNN)` attributions in PLAN.md; SOURCE uses inline 🔥 emoji markers + checkoff-in-place + no Conv-N attributions. LOCAL promoted shared scripts (`sync-gaps.sh`, `tech-doc-sweep.sh`) to `.claude/scripts/`; SOURCE keeps them skill-scoped. LOCAL accumulated Peerloop-specific Change Detection Matrix rows (access control, route mappings, page specifications, policies), API Route Mapping table, CLI Detail Files table, expanded checklists (10 vs 7).

**Learning:** Past ~3 months of independent evolution, same-named skills should be treated as sibling-but-distinct rather than mergeable. Merge discussions force philosophical alignment decisions (e.g., "should PLAN.md have Conv-N attributions?") that are appropriate for a standalone conversation, not a tool-driven port.

**Pattern:** `/w-sync-skills` should detect structural divergence (>30% by body length, new/missing sections, conflicting philosophies) and recommend "evolve independently" for that specific skill instead of opening a merge discussion. Source-only skills (new capabilities not yet in LOCAL) remain straightforward to adopt.

## 3. Claude Code `!` backtick expansions don't share state across calls
**Topics:** cc-workflow, skills

**Context:** When wiring per-agent model selection, considered using one `!` backtick to compute the value and reference it by name in three subsequent lines. Doesn't work — each `!` is a separate shell invocation, no variable passes between them.

**Learning:** Each `!` backtick in SKILL.md resolves independently at skill-load time to a literal string. There's no "shell session" that persists variables between backticks. For values that need to appear in multiple places, either (a) write one backtick that prints all values in a single output block, or (b) accept the redundancy of N near-identical backticks.

**Pattern:** For compact rendering of multiple related values, prefer (a): one Node expression that logs a multi-line formatted output:
```
!`node -e "const m=require('...').rEnd.agentModels;console.log('  learn-decide: '+m.learnDecide+'\n  update-plan:  '+m.updatePlan+'\n  docs: '+m.docs)"`
```
For values that need to render right at the point-of-use (e.g., inline in a heading), accept the N-backtick redundancy — visibility at point-of-use outweighs DRY.

## 4. User-confirmed sub-decisions survive later topic-level declarations
**Topics:** cc-workflow, feedback

**Context:** User said "I want the Opus reassessment step definitely" early in a discussion. Later, after a more general statement "I am going to evolve the same-named skills independently", I read that as revoking the Opus port too. It didn't — the later statement was about the HARD RULES discussion, not the already-confirmed Opus item.

**Learning:** When a user makes a topic-level declaration after having confirmed a specific sub-item, do NOT assume the declaration overrides the prior confirmation unless they explicitly say so. The safer read is: confirmations stand unless revoked by name. When ambiguous, ask before revoking.

**Pattern:** Maintain a running list of user-confirmed items during a discussion. Before applying a topic-level pivot, replay that list against the pivot to check: "User confirmed X early. Does this pivot revoke X?" If unclear, ask.
