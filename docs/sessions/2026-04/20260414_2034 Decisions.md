# Conv Decisions - 2026-04-14

## 1. Use base Modal primitive over FormModal
**Type:** Implementation
**Topics:** astro, ui

**Trigger:** Phase 5 needs a file input, multi-step logic (MIME sensing, type dropdown appearing after file pick), and mode toggle.

**Options Considered:**
1. Extend `FormModal.tsx` to support file type — invasive, leaks file-specific concerns into a generic primitive
2. Build a purpose-built `AddCommunityResourceModal.tsx` using the base `Modal.tsx` primitive ← Chosen
3. Inline the form into CommunityTabs (matches `ResourcesEditor.tsx` pattern) — clutters CommunityTabs further

**Decision:** Dedicated modal component. Separate file, reuses Modal's focus trap / Esc / scroll lock.

**Rationale:** FormModal's value is generic field rendering — not worth polluting with a file-input special case. A purpose-built component keeps the shared primitive clean and gives the modal room to grow (validation messages, upload progress, preview thumbnails) without leaking into other callers.

**Consequences:** One more component file. `FormModal` stays generic. The modal is reusable within community UIs if moderator/teacher upload is later allowed.

## 2. Refresh on success via `window.location.reload()` instead of client-side list mutation
**Type:** Implementation
**Topics:** astro, ui, ssr

**Trigger:** CommunityTabs receives `resources` as a prop from the SSR loader, which denormalizes `uploadedBy.{name,handle,role}`. On client-side append, reconstructing that shape is awkward.

**Options Considered:**
1. Reload page after success ← Chosen
2. Append returned resource to local state — requires either a secondary fetch to get uploadedBy, or inlining the current user's profile as a prop
3. Mutate local state with a placeholder uploadedBy — flicker until next navigation

**Decision:** `window.location.reload()` in the onSuccess handler.

**Rationale:** Upload is a low-frequency action (creator-only, a few per session). Reload guarantees the exact shape from SSR with zero client-side denorm logic. Keeps the modal simple.

**Consequences:** Minor UX cost (full reload vs. instant append). If usage frequency grows, revisit with a smaller state mutation pattern.

## 3. Fix `deleted_at → is_archived` rather than add `deleted_at` to communities
**Type:** Implementation
**Topics:** d1

**Trigger:** Conv 117 endpoint used a non-existent column, causing all POST/PUT/DELETE to return 500.

**Options Considered:**
1. Add `deleted_at TEXT NULL` to communities schema to match sibling tables
2. Fix the endpoint SQL to use the existing `is_archived = 0` guard ← Chosen
3. Remove the soft-delete guard entirely (no archive filter)

**Decision:** Match the established convention — communities uses `is_archived`, so the endpoint should too.

**Rationale:** 10+ other community endpoints already use `is_archived = 0`. Changing the schema mid-block for consistency would force migrations and risk more drift. The established convention is correct; the new endpoint was the outlier.

**Consequences:** Two one-line SQL fixes. Community soft-delete convention is now consistently applied across all endpoints.
