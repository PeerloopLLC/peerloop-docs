# Conv Learnings - 2026-04-21

## 1. Stripe's partial unique indexes need predicate-matching guards
**Topics:** stripe, d1, schema-design

**Context:** [VD] duplicate-purchase guard needed to prevent `SQLITE_CONSTRAINT_UNIQUE` from `idx_enrollments_one_active ON enrollments(student_id, course_id) WHERE status IN ('enrolled', 'in_progress')`. A naive guard `WHERE student_id=? AND course_id=?` would over-fire and reject legitimate re-enrollments after completion/cancellation.

**Learning:** When designing a pre-INSERT guard against a partial unique index, the guard's WHERE clause must mirror the index's predicate exactly. In SQLite: `WHERE status IN ('enrolled', 'in_progress')` must appear in both the index definition and the guard's SELECT.

**Pattern:**
```ts
// Matches the partial index predicate exactly
const activeEnrollment = await queryFirst<{ id: string }>(
  db,
  `SELECT id FROM enrollments
   WHERE student_id = ? AND course_id = ? AND status IN ('enrolled', 'in_progress')`,
  [student_id, course_id]
);
```

## 2. CF Worker fire-and-forget DB writes race termination
**Topics:** cloudflare, d1, webhooks, concurrency

**Context:** [VW] wrapped `webhook_log` INSERT in `locals.cfContext.waitUntil(...)`. The existing `.run().catch(...)` pattern LOOKED safe — error handler attached — but CF Workers aggressively terminate request contexts, especially on short-path handlers (e.g., Stripe default-case). The JS microtask schedules the `.run()`, but the underlying D1 HTTP call gets cancelled when the context dies.

**Learning:** On CF Workers, any DB write that should survive after the response is returned MUST be wrapped in `ctx.waitUntil(...)`. The `.catch()` pattern alone doesn't protect against runtime cancellation — it only handles *logical* failures. `waitUntil` tells the runtime "keep the execution context alive until this Promise settles."

**Pattern:**
```ts
locals.cfContext.waitUntil(
  db.prepare(SQL).bind(...).run().catch((err) => console.warn('insert failed:', err))
);
```

## 3. `ADMIN_ALERT`-prefixed warnings are a cheap alerting hook
**Topics:** observability, logging

**Context:** [VD] needed a distinctive log signal for duplicate-purchase attempts without adding any new infrastructure. Chose to prefix the warning with the literal string `ADMIN_ALERT duplicate_enrollment_attempt:`.

**Learning:** A consistent, greppable prefix on `console.warn` is the cheapest "alerting hook" that composes with *any* future log pipeline. No dependency on Sentry, PostHog, Grafana, or any future tooling. The prefix is stable, scannable in `wrangler tail`, and convertible to a concrete alerting rule whenever observability gets wired up.

## 4. Stripe account IDs are embedded in pk_test / sk_test prefixes
**Topics:** stripe

**Context:** [VA] audit. When staging deploy emitted `pk_test_51SkSfYRu7i9fxxy08...` in the bindings list, the `/api/admin/stripe-mode` endpoint returned `account_id: acct_1SkSfYRu7i9fxxy0`. Noticed the substring `1SkSfYRu7i9fxxy0` appears in both.

**Learning:** Stripe embeds the platform account identifier as a substring of every API key prefix (roughly `sk_test_5{account_id}...` shape). This is a quick sanity-check mechanism: if you have a pk_test visible in CI logs or wrangler output, you can already identify which account the key belongs to without fetching anything from Stripe. It doesn't distinguish Sandbox from Test mode on its own (both use `sk_test_`), but matches across keys within an environment should agree on account_id — a secret-slot mismatch will show different account substrings.

## 5. Same-file `{}` → real-stub test fixture: side-effect budget
**Topics:** testing, infrastructure

**Context:** [VW] required `cfContext` to actually have a `waitUntil` method. Two sites held `cfContext: {} as ExecutionContext`: `tests/api/helpers/api-test-helper.ts:218` (covers all API tests) and `tests/middleware.test.ts:46` (middleware-only test). Changed only the first; left the second untouched.

**Learning:** When a test helper stub needs to grow a real shape, assess blast radius before amending every site. The middleware test doesn't exercise any code path that calls `waitUntil`, so amending it would be scope-drift. Narrow the change to the actual consumer — the file sitting upstream of 367 API test suites. 6410/6410 passing confirmed zero regression from the surgical approach.

## 6. Stripe CLI pairing code is a human-comparable MITM defense
**Topics:** stripe, security, CLI

**Context:** User's `stripe login` flow surfaced an "Allow / Deny" modal asking to verify a pairing code. The CLI had printed the code in a separate terminal before opening the browser; the user's terminal had timed out.

**Learning:** `stripe login`'s out-of-band pairing flow is a deliberate anti-MITM mechanism. An attacker running their own `stripe login` against your Stripe account would reach the same modal on your browser but their pairing code would differ. The right response when the CLI session has timed out and the pairing code is unrecoverable: always Deny. Allowing without verification defeats the entire mechanism's purpose.

## 7. Stripe has unified Test mode into the Sandboxes listing UI
**Topics:** stripe, documentation

**Context:** During [VA] verification, user's screenshot showed the Stripe Dashboard's Sandboxes page listing BOTH the named Sandbox workbench AND the legacy Test-mode entry side-by-side with a banner: "Test mode is now part of sandboxes, so you can manage all of your test environments in one place."

**Learning:** Stripe has recently (2025/2026) merged Test mode into the Sandboxes UI as a peer of named Sandboxes. Account IDs remain distinct (Test-mode `acct_1SkSfMRyHGcVUhoO` vs named-sandbox `acct_1SkSfYRu7i9fxxy0`) — the isolation at the account level is unchanged — but the Dashboard navigation has shifted. Peerloop's `stripe.md §Stripe Mode Discipline` still treats these as two distinct workbenches with different UI locations; a one-paragraph note should be added to reflect the current Dashboard state.

## 8. `stripe listen` is a tunnel, not a test tool
**Topics:** stripe, webhooks, dev-environment

**Context:** User asked how the Stripe CLI helps during ad-hoc manual testing. Clarifying conversation revealed this was a conceptual gap in how we've documented the CLI.

**Learning:** `stripe listen` solves a **network physics** problem (Stripe's servers can't reach `localhost`), not a testing problem. Every real manual test on Dev implicitly depends on it, even when the developer isn't thinking about "using the CLI." Its role should be introduced in Tier A walkthroughs as "the WebSocket bridge between Stripe's event queue and your Mac," distinct from `stripe trigger` (synthetic event generator) and `scripts/trigger-webhook.sh` (local bypass harness).

## 9. Secret-suffix regex-based leak triage is safer than reading files
**Topics:** security, diagnostics

**Context:** During [VL], needed to check whether any repo file contained the leaked `sk_test_...PP6iSq` key's full value. Using `cat` or unbounded `grep` would have re-leaked the secret if it existed. Used `grep -cE "sk_test_[A-Za-z0-9_]{30,}PP6iSq"` — length-bounded pattern returns only a count.

**Learning:** When auditing leak blast radius, prefer count-only searches over content-returning ones. A length-bounded regex discriminates "full-key occurrence" from "suffix-stub reference" without exposing either. Codified in the [FD] memory extension.
