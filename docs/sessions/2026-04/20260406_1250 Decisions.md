## 2. Server-Side Availability Validation on Booking
**Type:** Implementation
**Topics:** booking, availability, validation

**Trigger:** STUMBLE-AUDIT.INVENTORY identified that `POST /api/sessions` had no validation that booking times fall within teacher availability.

**Options Considered:**
1. Document-only — write tests capturing current behavior
2. Fix + test — add validation and tests ← Chosen
3. Skip — defer entirely

**Decision:** Added `isSlotWithinAvailability()` utility to check recurring rules, day-of-week, and overrides. Returns 400 with descriptive reason when booking is outside availability.

**Rationale:** Server-side validation prevents impossible bookings. The utility function is reusable and mirrors the existing `countAvailableSlots()` algorithm.

**Consequences:** 3 new tests. All PLATO scenarios and integration tests updated with availability seed data. Set-availability step changed to all-day/all-week UTC for test reliability.
