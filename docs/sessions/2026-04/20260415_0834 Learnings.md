# Conv Learnings - 2026-04-15

## 1. Conv 119 audit overstated UI scope ~7×
**Topics:** code-quality, audit-discipline

**Context:** RESUME-STATE.md from Conv 119 said "20+ UI/API sites" needed updates for COMMUNITY-TEACHER-KILL. Real scope was 3 React sites + 6 Astro gates (one inline expression repeated 6×).

**Learning:** The 20+ count came from `grep` for `member_role` (76 hits, 29 files), but most occurrences were unrelated reads/SQL fragments that survive a CHECK narrowing untouched. Always re-grep with the precise behavioral pattern (`memberRole === 'teacher'`, `member_role.{0,10}=.{0,5}'teacher'`) before estimating effort, not the loose schema/field name.

**Pattern:** For role-removal audits, the real scope = sites that *write* + sites that *read with equality on the value*. Schema-name greps inflate by ~10×.

## 2. Astro pages and React components consumed different DTO shapes
**Topics:** astro, dto-design

**Context:** The 3 React sites read `m.memberRole === 'teacher'` (camelCase from `/api/me/full`). The 6 Astro pages used `membership?.role === 'teacher'` (the SSR loader's flat `role` field, not nested `memberRole`). Same domain concept, different field names depending on entry point.

**Learning:** When two entry points (REST vs SSR loader) describe the same resource, divergent property names (`memberRole` vs `role`) make grep-based audits miss half the sites. Worth flagging as a longer-term consistency target — but for the immediate kill, just remember to grep both shapes.

**Pattern:** Future role audits: grep both `memberRole === '<value>'` and `\.role\s*===\s*'<value>'`.

## 3. The teaching-community signal lives behind a 3-table join
**Topics:** d1, schema-design

**Context:** RESUME-STATE.md said "teacher_certifications JOIN courses ON community_id" but courses have no `community_id` field. Verified via grep on the schema.

**Learning:** The real chain is `teacher_certifications.course_id → courses.progression_id → progressions.community_id`. A teacher cert for a non-progression course (e.g., legacy/standalone) yields no teaching community — that's a real edge case but it matches the platform's own grouping semantics (unstructured courses don't belong to a community feed).

**Pattern:** When verifying a JOIN claim, always check the schema's foreign-key columns directly — RESUME-STATE memos compress these down and lose the intermediate table.

## 4. Pre-release schema mutability collapses CHECK-narrowing risk
**Topics:** d1, deployment, decisions

**Context:** Q1 originally posed 3 options for narrowing the schema CHECK: hard kill in `0001_schema.sql`, soft deprecate, or hybrid (kill + add `0003_*.sql` upgrade migration for staging/prod). User pointed out we're still pre-release — schema can be edited freely and propagated via reset.

**Learning:** Migration ceremony exists to handle deployed-data risk. In pre-release, the project's `db:reset:staging` and `db:reset:remote` tooling already supports clean schema edits, so the "rigorous" hybrid option becomes unnecessary overhead. Keep the question alive but default to the simpler path until a release-candidate flag flips.

**Pattern:** Decision-tree pattern for schema edits: ask "is this pre-release?" first — collapses Q1 entirely for CHECK narrowings, ENUM extensions, and column drops. Re-evaluate post-release.

## 5. Schema CHECK as exhaustive seed-data validator
**Topics:** d1, testing

**Context:** Ran `npm run db:reset:local && npm run db:setup:local:dev` after the kill to verify no `'teacher'` rows slipped through. The CHECK constraint accepted every insert.

**Learning:** A narrowing CHECK constraint is the cheapest exhaustive validator for "did I update every seed/test row?" — if any insert violates it, the migration fails loud. Better than grep because it catches dynamically-constructed inserts (e.g., the failing test that built `'teacher'` from string concatenation).

**Pattern:** Whenever narrowing a CHECK, follow with a fresh `db:reset:local` and the equivalent for staging — both will fail loud if any insert survives the kill in seeds, fixtures, or test setup.

## 6. Targeted tests miss test-only seed drift
**Topics:** testing, ci-discipline

**Context:** Targeted vitest run (5 affected files) passed clean, but the full test suite caught one failure: `tests/api/me/communities/[slug]/members.test.ts` inserts `'teacher'` directly into `community_members` without touching any of the renamed source files. Targeted runs filter by import dependency; tests that own their seed data aren't reached.

**Pattern:** For schema/CHECK changes, the full test suite is the only complete coverage check — targeted runs are insufficient. Schedule the ~3-min full run as part of any CHECK-narrowing change.
