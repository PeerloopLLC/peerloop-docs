# Conv Decisions - 2026-04-02

## 1. DOM-Based Toast Over React State for Save Feedback
**Type:** Implementation
**Topics:** react, ui-feedback, astro

**Trigger:** React state-based toast never rendered due to component remount after `setCourse()`

**Options Considered:**
1. React state toast in parent component — Failed (state lost on remount)
2. React portal to document.body
3. DOM manipulation via `showToast()` function ← Chosen

**Decision:** Use a standalone `showToast()` function that manipulates the DOM directly, outside React's lifecycle.

**Rationale:** Simpler than portals, immune to component remounts, works across all tabs/sub-components in CourseEditor.

**Consequences:** Toast styling is inline CSS (not Tailwind), but consistent with the design. All save handlers across CourseEditor tabs now use `showToast()`.

> **Insight:** In hybrid frameworks like Astro where React components live inside islands, React's state lifecycle is not guaranteed to be stable — the island itself may remount. Escape hatches to the DOM are sometimes the most robust solution for transient UI that must survive framework-level re-renders.

## 2. Screenshot Convention: macOS screencapture with Region Targeting
**Type:** Implementation
**Topics:** stumble-audit, dev-workflow

**Trigger:** User wanted reviewable screenshots from browser walkthroughs

**Options Considered:**
1. Chrome MCP `computer` screenshot (inline to conversation only)
2. macOS `screencapture -x` full screen (20MB Retina)
3. macOS `screencapture -x -R{bounds}` region capture ← Chosen

**Decision:** Use `screencapture -x -R{x},{y},{w},{h}` targeting Chrome window bounds. Save to `/tmp/stumble-screenshots/{instance}/`.

**Rationale:** 525KB vs 20MB, captures only Chrome window, Retina-sharp, no base64 piping needed.

**Consequences:** Convention saved to memory. User says "with screenshots" to enable, default is off.

## 3. Pre/Post Segment Split for Fix-and-Verify
**Type:** Strategic
**Topics:** plato, stumble-audit, testing-workflow

**Trigger:** Need to re-test fixes in isolation without re-walking entire flows

**Options Considered:**
1. Per-step snapshots (snapshot at every step)
2. Pre/Post segment split with single snapshot ← Chosen
3. Full re-run from scratch

**Decision:** Split instance at the failure point: Pre-segment (steps before) runs in API mode with `snapshot: true`, Post-segment (failure onward) is walked in browser after restoring the snapshot.

**Rationale:** One snapshot, not N. Simpler, uses existing infrastructure. Pre-segment builds DB state fast (API mode), Post-segment verifies fix visually (browser mode).

**Consequences:** Documented in `docs/as-designed/stumble-workflow.md`. Segment files are one-off by default but may be promoted to permanent scenarios.
