# Conv Learnings - 2026-04-14

## 1. Dependency drift detection already has a robust mechanism
**Topics:** cc-workflow, dev-env

**Context:** User asked about automating `npm install` after SessionStart. Initially imagined a new hash/timestamp compare, then found `scripts/check-env.sh:136-159` already does this using sha256 + a `postinstall` hook that writes `node_modules/.package-lock-hash`. Detection is content-based (not mtime-based), robust to touch/checkout timestamp churn.

**Learning:** Before proposing a new detection mechanism, search existing scripts — the content-hash approach in `check-env.sh` is reusable and should be what any new automation piggybacks on.

**Pattern:**
`sha256(package-lock.json) != cat(node_modules/.package-lock-hash)` → drift. Hash is refreshed by a `postinstall` npm script after every install, so "correct" state = "hash matches what was installed last."

## 2. Skill step placement matters for change-tracking windows
**Topics:** cc-workflow, dual-repo

**Context:** User's phrasing "after the Conv is set up and tracking changes" pointed at an invariant: side effects from startup flow should be attributable to a conv, not to pre-conv limbo. Placing `npm install` before the counter commit would mean lockfile mods land in an untracked window (no conv number yet).

**Learning:** For any `/r-start` side-effect (install, codegen, migration sync, etc.), correct placement is AFTER Step 5 (counter push). Subsequent `/r-commit` or `/r-end` then captures the artifact under the current conv.
