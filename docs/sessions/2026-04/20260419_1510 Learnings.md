# Conv Learnings - 2026-04-19

## 1. Stripe Webhook Stream-Unfollow Needs Course-Follow Guard
**Topics:** stripe, stream, course-follows

**Context:** When enrollment is refunded, `webhooks/stripe.ts` removes the user's Stream timeline follow for the course. But if the user also has a `course_follows` row (explicit non-enrollment follow), removing the Stream follow would break their course feed subscription silently.

**Learning:** Anytime enrollment-driven Stream follows are removed, check whether the user has an independent `course_follows` entry before calling `timeline.unfollow`. The follow mechanism has two independent sources and both must be checked before unfollowing.

**Pattern:** `const stillFollowing = await queryFirst(db, 'SELECT id FROM course_follows WHERE user_id = ? AND course_id = ?', [...])` before `timeline.unfollow(...)`.

## 2. ExploreCourseHero Is the Right Injection Point for Discover-Only UI
**Topics:** astro, components, architecture

**Context:** `CourseHero` is used in 8+ places (enrolled pages, `CourseDetail`, `ExploreCourseHero`). Adding follow button to `CourseHero` would have required plumbing optional props across all callers.

**Learning:** `ExploreCourseHero` is a discover-specific wrapper specifically for this purpose — adding discover-context UI here keeps enrolled pages untouched and localises the feature to its correct context.

## 3. SSR isFollowing Pattern
**Topics:** astro, d1, ssr

**Context:** The discover page already had a `Promise.all` with an enrollment check. Adding the follow state check follows the same conditional-query pattern.

**Learning:** Pattern: add `followRow` to the `Promise.all` destructure, add `userId ? queryFirst(...course_follows...) : Promise.resolve(null)` as the last entry, then `isFollowing = !!followRow` after the await. Keeps the SSR data fetch as a single parallel batch.
