# Conv Learnings - 2026-04-21

## 1. @astrojs/cloudflare 13 does not expose workerEntryPoint
**Topics:** astro, cloudflare, deployment, architecture

**Context:** PLAN.md:693 flagged "Investigate Astro + CF adapter dual exports (fetch + scheduled)" as unresolved. Phase B required adding a scheduled cron handler.

**Learning:** `@astrojs/cloudflare@^13.1.8`'s public `Options` type (in `node_modules/@astrojs/cloudflare/dist/index.d.ts`) exposes only `auxiliaryWorkers | configPath | inspectorPort | persistState | remoteBindings` from `@cloudflare/vite-plugin`'s PluginConfig, plus Astro-specific fields. It does NOT expose `workerEntryPoint`. Therefore, adding `scheduled()` to the SAME Worker that Astro builds is not cleanly supported.

**Pattern:** For Astro+CF projects that need cron handlers, deploy a **separate standalone Worker** rather than trying to augment Astro's Worker entry. Share bindings (D1/R2/KV) via binding IDs, share logic via shared `src/lib/*.ts` modules that both Workers import.

## 2. CF account subdomain is discoverable via API
**Topics:** cloudflare, deployment, tooling

**Context:** The `.workers.dev` URL for a Worker is `<worker-name>.<account-subdomain>.workers.dev` but the subdomain is separate from account name/email and not shown in `wrangler whoami`.

**Learning:** Query the CF API at `/accounts/{account_id}/workers/subdomain` with a `CLOUDFLARE_API_TOKEN` header — returns `{"subdomain": "..."}`. For this project: subdomain is `brian-1dc` and staging URL is `https://peerloop-staging.brian-1dc.workers.dev`.

**Pattern:**
```bash
TOKEN=$(grep '^CLOUDFLARE_API_TOKEN=' .dev.vars | cut -d= -f2-)
curl -s -H "Authorization: Bearer $TOKEN" \
  "https://api.cloudflare.com/client/v4/accounts/$ACCOUNT_ID/workers/subdomain" | jq -r '.result.subdomain'
```

## 3. wrangler tail double-suffix pitfall
**Topics:** cloudflare, wrangler, tooling

**Context:** Attempted `wrangler tail peerloop-cron-staging --env staging --config workers/cron/wrangler.toml` → API error "This Worker does not exist on your account" with URL containing `/scripts/peerloop-cron-staging-staging/tails`.

**Learning:** When the `[env.staging]` block in wrangler.toml already sets `name = "peerloop-cron-staging"`, passing `--env staging` AND the explicit worker name causes wrangler to **append `-staging` again**, yielding `peerloop-cron-staging-staging`.

**Pattern:** Use one or the other, not both: either `wrangler tail --env staging --config <path>` (lets env name lookup work) OR `wrangler tail peerloop-cron-staging --config <path>` (explicit name, no env).

## 4. Stripe CLI `stripe trigger` is localhost-only
**Topics:** stripe, testing, staging

**Context:** Extending the existing webhook harness to also target staging for Stripe events.

**Learning:** `stripe trigger` forwards events to whatever `stripe listen --forward-to <url>` is running, which is typically `localhost:4321`. It does NOT post to remote registered webhook endpoints. To fire Stripe events at a registered staging/prod endpoint, either (a) use `stripe events resend --webhook-endpoint <id>` (needs a pre-existing event in history), or (b) construct + sign events directly using `STRIPE_WEBHOOK_SECRET` and POST via curl or a helper script.

**Pattern:** For staging harness testing, build a direct-sign POST helper — Stripe signature is `t=<timestamp>,v1=<hmac-sha256(secret, "<ts>.<payload>")>`. `stripe.webhooks.generateTestHeaderString()` from the Node SDK produces this string if preferred over raw crypto.

## 5. Shared orchestration modules for endpoints + cron workers
**Topics:** architecture, cloudflare, testing

**Context:** Refactoring admin cleanup endpoint so the same logic runs from both HTTP handler and cron Worker.

**Learning:** Extract the orchestration into a pure `src/lib/*.ts` module accepting dependencies via parameters (db, videoProvider). The HTTP endpoint reads auth + bindings then delegates; the cron worker reads its env then delegates. Both paths test the same code. `runSessionCleanup(db, bbb)` in `src/lib/cleanup.ts` is the template.

**Pattern:** Endpoint becomes ~50 lines of auth + adapter. Shared module is ~100 lines of actual work. Cron worker is ~40 lines of scheduled handler. Standalone Worker's tsconfig must `include` the shared files + `src/env.d.ts` so Astro namespace augmentations resolve.
