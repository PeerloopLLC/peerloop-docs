## 1. Leave audit-trail observations intact when marking fixes
**Type:** Implementation
**Topics:** docs-infra

**Trigger:** When updating `webhook-miss-resilience.md` to mark [VD]/[VW]/[VA] as closed by Conv 145, the Conv 144 "Stripe live observations" section (lines 127-129) contained 🔴 framings for the same items — dated as discovery records.

**Options Considered:**
1. Rewrite the observations to reflect current state (erase historical 🔴 framing)
2. Leave observations as dated historical discovery records; mark closure only in forward-looking "Stripe gaps — revised" bullets ← Chosen
3. Delete the observations entirely since the items are closed

**Decision:** Option 2 — preserve the Conv 144 observations (🔴 framing + discovery context) intact; flip closure markers only in the forward-looking "Stripe gaps — revised" section.

**Rationale:** The discovery record explains *why* each fix was prioritized — erasing the 🔴 framing would leave future readers wondering "was this ever a real concern?" The closure marker belongs in the "current state" view, not by rewriting the discovery view.

**Consequences:** Doc now shows a two-phase lifecycle for each fix: 🔴 at Conv N (discovery) → ✅ at Conv N+1 (closure). Audit trail preserved. Small prose cost (two places per fix) — acceptable.

## 2. Full restructure of "Other API" section (not just heading rename)
**Type:** Strategic
**Topics:** docs-infra

**Trigger:** While completing [TC] surfaced that `### Other API — tests/api/ top-level (23 files)` heading was misleading + contained a duplicate row. User asked to address.

**Options Considered:**
1. Rename heading only — cheap, restores accuracy to the heading text (~2 min)

3. Hybrid — split multi-file subdirs only; keep 1-file subdirs in a renamed catch-all

**Decision:** Option 2 — full restructure with 5 alphabetical insertions + 1 replacement.

**Rationale:** Per CLAUDE.md "default to durable" — the doc's established pattern is H3-per-subdir. Option 1 restored heading accuracy but left the structural inconsistency; Option 3 still had a catch-all with asymmetric rules. Option 2 gives parity: each subdir is greppable + anchor-navigable, counts stay honest per-subdir (not just in aggregate), and future adds have an obvious home (the subdir's H3).

**Consequences:** 10 new H3 sections added (Certificates, Debug, Recommendations, Resources, Reviews, Stories, Stream, Stripe, Submissions, Topics) + 1 renamed (Other API → Top-Level). Duplicate topics row eliminated automatically. The doc's API section is now structurally uniform — every subdir has the same treatment. Future drift will be contained to individual H3s, not hidden in a catch-all.

## 3. Cross-link lessons → spec asymmetrically (not bidirectional)
**Type:** Implementation
**Topics:** docs-infra, cc-workflow

**Trigger:** After writing [EG] ESLint v10 gotcha in PACKAGE-UPDATES notes, considered whether to also add a reverse cross-link from `DEVELOPMENT-GUIDE.md §"ESLint Configuration"` back to the PACKAGE-UPDATES entry.

**Options Considered:**
1. Bidirectional cross-links (both directions)
2. Only lessons → spec (unidirectional, skip spec → lessons) ← Chosen
3. No cross-link (treat the two as fully independent)

**Decision:** Option 2 — cross-link from PACKAGE-UPDATES ESLint v10 entry → DEVELOPMENT-GUIDE.md spec only.

**Rationale:** Spec doc reader is usually configuring fresh (low probability of needing historical gotcha context). Lessons reader is usually debugging / planning an upgrade (high probability of wanting to find the config spec). Asymmetric cross-linking matches asymmetric demand.

**Consequences:** Each doc maintainable in isolation — spec can be rewritten as ESLint evolves without stale back-references to old upgrade lessons. Slight cost: a future upgrade reader starting in DEVELOPMENT-GUIDE.md won't see the PACKAGE-UPDATES entry unless they grep.
