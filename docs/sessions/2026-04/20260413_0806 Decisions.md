# Conv Decisions - 2026-04-13

## 1. Simplify AppNavbar — Remove My* and Role-Specific Items
**Type:** Strategic
**Topics:** ui, navigation, ux

**Trigger:** Client reviewed AppNavbar and approved testing without My Feeds, My Courses, My Communities, Teaching, Creating menu items.

**Options Considered:**
1. Keep all items — current state
2. Comment out items for testing ← Chosen
3. Permanently remove items

**Decision:** Comment out 5 nav items (feeds, courses, communities, teaching, creating) for client testing. These are accessible via /discover/* and unified /dashboard.

**Rationale:** Client-driven. The unified Dashboard replaces role-specific nav entries. My* items are redundant with Discover sub-pages.

**Consequences:** Simplified nav for all roles. Items can be re-enabled if client decides against.

## 2. Open Member-to-Member Messaging
**Type:** Strategic
**Topics:** messaging, access-control, policy

**Trigger:** Client wants any member to be able to message any other member, reversing the relationship-based restriction from Session 338.

**Options Considered:**
1. Keep relationship-based messaging — current state
2. Open all member-to-member messaging, keep admin rules ← Chosen
3. Open messaging with rate limiting

**Decision:** Any authenticated member can message any other non-deleted member. Admin rules unchanged (anyone can message admin as support channel, admin can message anyone).

**Rationale:** Client-approved. Genesis Cohort is small enough that abuse is manageable through admin oversight.

**Consequences:** `messaging.ts` simplified dramatically (125 lines removed). POLICIES.md section 4 rewritten. Student-to-student messaging now allowed. Relationship-ending behavior (403 on deleted users) still applies.

> **Insight:** Relationship-gated messaging was architecturally sound but premature for a Genesis Cohort of 60-80 users. The infrastructure cost (3 queries, 6 EXISTS clauses, 28 surface audits) outweighed the abuse prevention benefit at this scale. The three-function architecture (`canMessage`, `getMessageableFlags`, `messageableContactsSQL`) survived the policy change intact — only internal rules simplified — validating the separation of function signatures from policy logic.
