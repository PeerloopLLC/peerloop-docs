## Learnings

### 1. Adapter 13's legacy-access proxy throws with migration guidance
**Topics:** cloudflare, astro, migration
**Context:** Inspecting `@astrojs/cloudflare@13.1.8`'s `dist/utils/handler.js` revealed that the adapter wraps every request's `locals` object with a non-enumerable `runtime` getter proxy. Each sub-field (`.env`, `.cf`, `.caches`, `.ctx`) is its own getter that **throws** with a descriptive migration message — e.g., `'Astro.locals.runtime.env has been removed in Astro v6. Use import { env } from "cloudflare:workers" instead.'`
**Learning:** Cloudflare adapter maintainers chose a maximum-signal migration path. There is no "it silently works but warns" fallback — any code path still touching `locals.runtime.env` will crash at the first request with a clear pointer to the replacement. This is the opposite of the usual deprecation warning pattern, and it makes silent breakage structurally impossible. For future major-version bumps, check if the new version uses throwing proxies — they are a strong indicator that the field is gone, not just renamed.

### 2. `Cloudflare.Env` is the declaration-merge target, not the bare `Env`
**Topics:** typescript, cloudflare, types
**Context:** My first `src/env.d.ts` rewrite declared `module 'cloudflare:workers' { export const env: Env }` at the top level, with `interface Env { DB: D1Database; ... }` also at the top level. Tsc complained `Property 'DB' does not exist on type 'Env'` even though `Env` clearly had the field. Investigation: `@cloudflare/workers-types/latest/index.d.ts` at line 12943 declares `declare namespace Cloudflare { interface Env {} }` with a comment explicitly naming this the project augmentation point, and at line 13126 declares `declare module 'cloudflare:workers' { export const env: Cloudflare.Env }`. My local module declaration was being shadowed by the workers-types one, so `cfEnv` resolved to `Cloudflare.Env` (empty interface), not my project `Env`.
**Learning:** The canonical Cloudflare + TypeScript project pattern is to augment `Cloudflare.Env` by re-declaring the namespace, not to write a bare `interface Env`. The bare `Env` works for third-party signatures like `ExportedHandler<Env>` (which resolves via global lookup), but for the `cloudflare:workers` virtual module specifically, you must target `Cloudflare.Env`. Keeping `interface Env extends Cloudflare.Env {}` as a top-level alias preserves compatibility with any code that still references the bare name.
**Pattern:**
```ts
declare namespace Cloudflare {
  interface Env {
    DB: D1Database;
    STORAGE: R2Bucket;
    JWT_SECRET: string;
    // ...
  }
}
// Backward-compat alias for bare-Env references
interface Env extends Cloudflare.Env {}
```

### 3. Vitest alias-to-mock-file is the durable pattern for virtual modules
**Topics:** testing, vitest, virtual-modules
**Context:** `cloudflare:workers` is a virtual module provided by `@cloudflare/vite-plugin`, which is registered in `astro.config.mjs` but **not** in `vitest.config.ts`. Tests that import from it have no resolution path by default. Three options considered: alias-to-mock-file, `vi.mock('cloudflare:workers')` in `vitest.setup.ts`, and dynamic import with try/catch in the helpers.
**Learning:** Vitest's `resolve.alias` was already being used for two other virtual modules (`astro:transitions/client`, `astro:middleware`), each pointing at a small mock file under `tests/helpers/`. Matching that pattern for `cloudflare:workers` is the least surprising option — single explicit resolution point, no vi.mock hoisting quirks, no async helper init. Matching an existing pattern in the project is almost always more durable than introducing a new one.

### 4. Centralization investment pays off at the next major version boundary (blast radius 5)
**Topics:** refactor, architecture, durability
**Context:** Conv 100 Phase 2-prep centralized all Cloudflare env access through 3 helpers (`getEnv`, `getDB`, `getR2`) and migrated ~95 files off direct `locals.runtime?.env?.*` access. Adapter 13's breaking change — removing `locals.runtime.env` entirely — landed in Phase 2a. A pre-centralization codebase would have had a ~100-file sweep; post-centralization, only 5 files touched `locals.runtime` directly (the 3 helpers + 2 test fixture files with inline mock shapes).
**Learning:** Investing in a centralization refactor _before_ consuming a breaking major version produces a dramatic multiplier. The ~95-file churn in Conv 100 paid off instantly in Conv 101, reducing Phase 2a's mechanical component to nearly zero. The durability principle (*favour durable decisions over faster options*) is directly validated by this outcome — the "simpler" Conv 100 path would have been to bump the adapter without centralizing, and we would now be paying it back with a 100-file sweep bundled into an already-risky major-version commit.

### 5. React 19 `server.edge` workaround is fixed upstream in Astro 6
**Topics:** react, astro, vite
**Context:** `astro.config.mjs` had a `vite.resolve.alias` entry mapping `react-dom/server` → `react-dom/server.edge` in PROD builds only, to avoid a `MessageChannel from node:worker_threads` polyfill error (Astro issue #12824). Conv 100 flagged this as a candidate for removal during Phase 2a. Removing it and running `npm run build` → built cleanly in 7.54s with zero warnings or errors.
**Learning:** Upstream Astro #12824 is fixed in the combination of Astro 6 + `@vitejs/plugin-react` 5 + React 19. The workaround is no longer needed. More broadly: during major-version upgrades, try removing every downstream workaround that references an upstream issue number — many of them have quietly become unnecessary. Isolating the removal in the same commit as the version bump makes bisection trivial if the workaround turns out to still be needed.

### 6. Adapter 13 `platformProxy` is gone; split into `remoteBindings` + `CLOUDFLARE_ENV`
**Topics:** cloudflare, config, migration
**Context:** `astro.config.mjs` had `platformProxy: { environment: 'preview', remoteBindings: true }` passed to `cloudflare(...)` when `USE_STAGING_DB` was set, to power `npm run dev:staging`. Adapter 13's `Options` interface picks only `'auxiliaryWorkers' | 'configPath' | 'inspectorPort' | 'persistState' | 'remoteBindings'` from `PluginConfig` — no `platformProxy`, no `environment`. Inside `@cloudflare/vite-plugin/dist/index.mjs` line 7200: `getCloudflareEnv = getEnvironmentVariableFactory({ variableName: "CLOUDFLARE_ENV" })` confirms the plugin reads `CLOUDFLARE_ENV` to select the wrangler.toml environment.
**Learning:** The old `platformProxy` option is replaced by two orthogonal mechanisms:
- `remoteBindings: true` is now a top-level option on the adapter, forwarded directly to `@cloudflare/vite-plugin`.
- `environment: 'preview'` has no adapter-level equivalent; set `CLOUDFLARE_ENV=preview` at the CLI (e.g. in the npm script) and the plugin will pick the corresponding wrangler.toml `[env.preview]` section.
**Pattern:**
```js
// astro.config.mjs
adapter: cloudflare({
  imageService: 'passthrough',
  remoteBindings: process.env.USE_STAGING_DB ? true : undefined,
}),
```
```json
// package.json
"dev:staging": "USE_STAGING_DB=1 CLOUDFLARE_ENV=preview astro dev"
```

## Decisions

### 1. Keep `locals` parameter on helpers (Fork 2 = X)
**Type:** Implementation
**Topics:** refactor, architecture, testing
**Trigger:** After switching helpers to `import { env } from 'cloudflare:workers'`, the `locals` parameter is nominally dead — env access no longer depends on it in production. But the `__testEnv` injection path *does* use `locals` (`locals.__testEnv?.[key]`), so the parameter has a narrow valid purpose.
**Options Considered:**
1. **Y** — Drop `_locals` from all helpers now; sweep ~130 call sites as part of Phase 2a
2. **X** — Keep the parameter for Phase 2a; defer the sweep to a dedicated follow-up commit ← Chosen
**Decision:** Keep the parameter throughout Phase 2a. Schedule a follow-up mechanical sweep as a separate commit on the same branch.
**Rationale:** Phase 2a commit stays tight and reviewable (29 files for the refactor core, not 150+). Bisection is clean — the adapter bump and the signature sweep live in different commits. The `__testEnv` path means the parameter is not dead weight, just narrowly used. Durability principle still served because follow-up cleanup has a better blame history when isolated.
**Consequences:** 39+ tests using `createAPIContext({ db: ... })` need no changes. A new follow-up task (see §Progress → New Subtasks) tracks the eventual `_locals` drop.

### 2. Test path via vitest alias + mock file (Fork 1 = A)
**Type:** Implementation
**Topics:** testing, vitest, virtual-modules
**Trigger:** `cloudflare:workers` is a virtual module provided by `@cloudflare/vite-plugin`, which vitest does not load. Without intervention, `import { env } from 'cloudflare:workers'` in the helpers cannot resolve in tests.
**Options Considered:**
1. **A** — `vitest.config.ts` alias → `tests/helpers/mock-cloudflare-workers.ts` (a small file exporting a Proxy over `process.env`) ← Chosen
2. **B** — `vi.mock('cloudflare:workers', () => ({ env: ... }))` in `vitest.setup.ts`
3. **C** — Dynamic `import('cloudflare:workers').catch(() => null)` inside helpers, cached at module load
**Decision:** Option A — alias + dedicated mock file.
**Rationale:** Matches the existing virtual-module pattern already used in `vitest.config.ts` for `astro:transitions/client` and `astro:middleware`. No `vi.mock` hoisting quirks. Helpers stay synchronous (no async init plumbing). Single explicit resolution point that a newcomer can grep for. Zero impact on production code paths.
**Consequences:** One new file (`tests/helpers/mock-cloudflare-workers.ts`, ~25 lines). One new alias entry in `vitest.config.ts`. No changes to helper signatures or existing tests that use `vi.stubEnv` or `vi.mock` patterns.

### 3. Use `__testEnv` as the test-only injection slot on `App.Locals`
**Type:** Implementation
**Topics:** testing, typescript, architecture
**Trigger:** `createAPIContext` previously built `locals.runtime.env = { JWT_SECRET, DB, ... }` to inject per-test env values; helpers read them back. Adapter 13's throwing proxy breaks that pattern in production, but tests still need per-context injection (39+ tests use `{ db: ... }`, 10 use `env: { STORAGE: ... }`).
**Options Considered:**
1. Migrate all 39+ tests to `vi.stubEnv` + `vi.mock('@/lib/db')` scopes
2. Global `vi.mock('@/lib/db')` + `vi.mock('@/lib/r2')` in `vitest.setup.ts` reading from a module-scoped test DB ref
3. Add `__testEnv?: Record<string, unknown>` to `App.Locals`; helpers check it before falling through to `cloudflare:workers` ← Chosen
**Decision:** Option 3 — `__testEnv` as a dedicated test-only slot on `App.Locals`, checked first by each helper.
**Rationale:** Preserves the `createAPIContext({ db, env })` ergonomics that 39+ tests depend on — no mechanical test sweep needed. The test-awareness added to helpers is minimal: a single `locals?.__testEnv?.[key]` check at the top of each function, with zero cost in production (where `__testEnv` is always undefined). Alternatives 1 and 2 would have introduced significant global state or per-test boilerplate.
**Consequences:** Three helpers, api-test-helper, middleware.test.ts mock, and 2 r2 mocks need updates (total: ~6 targeted edits plus 17 mechanical cast renames). A new ESLint rule (task [LE]) should eventually forbid application code from setting `__testEnv` to prevent misuse.

### 4. Keep `vite.ssr.external` as-is rather than migrate to Vite 7 Environments API
**Type:** Implementation
**Topics:** vite, config
**Trigger:** Vite 7 introduces a new per-environment config shape (`environments.ssr.resolve.external`) that replaces the top-level `ssr.external`. The new shape is marked `@experimental` in Vite's types.
**Options Considered:**
1. Migrate to `environments.ssr.resolve.external` during Phase 2a
2. Keep `vite.ssr.external` as-is until Vite stabilizes the new API ← Chosen
**Decision:** Leave `vite.ssr.external: ['stream', 'crypto', 'path', 'node:buffer', 'node:stream']` unchanged.
**Rationale:** Vite 7's `SSROptions` interface still declares `external?: string[] | true` — the old path is not deprecated, just joined by a new experimental alternative. Phase 2a's build succeeded with the old shape. Migrating to an experimental API that may still shift would be the opposite of durable — we'd likely need to migrate again when Vite marks the new API stable, plus debug any edge cases the experimental version carries.
**Consequences:** Config remains as it was. Revisit when Vite marks the new API stable.
