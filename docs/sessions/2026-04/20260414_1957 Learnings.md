# Conv Learnings - 2026-04-14

## 1. CF Workers schema evolution: edit-in-place is still the right call pre-launch
**Topics:** d1, deployment

**Context:** Client flagged community file uploads as "must fix now." `community_resources` had a single `url` column while `session_resources` had a proper r2_key/external_url split. Pre-launch, so a direct edit to `0001_schema.sql` (rather than a 0003_*.sql migration) aligned with existing project convention in CLAUDE.md §Database Migrations.

**Learning:** When a client-reported bug exposes a schema design gap pre-launch, fix the schema properly (direct edit) rather than working around it in application code. The PR is larger but the codebase stays clean and future-you has fewer migration layers to unwind.

**Pattern:** Before settling on a schema workaround, check whether the asymmetry between similar tables (here, `community_resources` vs `session_resources`) is itself the bug.

## 2. Pre-compute cross-concern URL fields in SSR loaders
**Topics:** astro, testing

**Context:** The Resources tab had `href={resource.url}` — trusting the DB column to be a clickable URL. For file-backed resources this required the URL to be a literal path, not an R2 key. The fix added a `downloadUrl` field at the SSR loader: `r2_key ? '/api/community-resources/{id}/download' : external_url`. The component now renders `href={resource.downloadUrl}` without knowing whether it's a link or a stream.

**Learning:** When a UI component must derive a URL from multiple DB columns under conditional logic, pre-compute the final URL at the loader/serializer layer. Components stay dumb, the loader stays the single source of URL-shape truth, and tests can assert on one field instead of three.

**Pattern:**
```ts
downloadUrl: r.r2_key
  ? `/api/community-resources/${r.id}/download`
  : (r.external_url ?? ''),
```

## 3. Three-tier auth checks: order by cost
**Topics:** auth

**Context:** The new community-resources download endpoint authorizes creator → admin → community member. Written naively, all three DB queries fire. Written correctly (ordered cheapest-first), the common path (regular member) hits 2 queries; uncommon paths hit 1 or 3.

**Learning:** For short-circuiting auth chains, order by query cost ascending. Creator check is a single-field comparison against a value already in hand (`community.creator_id`). Admin check is an `is_admin` lookup on users (single-row, indexed PK). Membership is most expensive (`community_members` two-column lookup). Short-circuit after each.

## 4. PLATO in-process architecture blocks remote Worker testing
**Topics:** testing, cloudflare, deployment

**Context:** User wanted PLATO against staging to reproduce 404s. Investigation revealed PLATO imports Astro handler functions directly and calls them with an in-memory SQLite — no HTTP layer. "Point PLATO at staging" is not a config flip; it requires adding an HTTP adapter. Browser mode is design-only.

**Learning:** PLATO is the wrong tool for "is this endpoint working on the live Worker?" For live-endpoint debugging, use Chrome MCP or curl directly. Use PLATO when you want in-process reproducibility with fast iteration and isolated DB state.

## 5. "Unbounded matches existing pattern" > "restrict the bounds correctly"
**Topics:** astro

**Context:** A buggy prev-month calendar guard could have been fixed by computing the comparison at month-level granularity. User chose to remove the guard entirely because the next-button was already unbounded and the teacher's `AvailabilityCalendar` was also unbounded. Removing the one outlier made all three consistent.

**Learning:** When a bug forces you to look at a guard's logic, also check nearby related code. If the guard is an outlier, removing it to match surrounding patterns may be simpler and more correct than "fixing" it. Flag the design question as a follow-up rather than inventing a constraint.
