# Conv Decisions - 2026-04-15

## 1. Keep status quo: do NOT embed isAdmin in JWT claims ([RA-JWT])
**Type:** Strategic
**Topics:** auth, jwt, security

**Trigger:** [RA-JWT] TodoWrite task #10 flagged as pending decision from Conv 123 [RA-ADM] audit.

**Options Considered:**
1. Naive embed in both access and refresh tokens — fastest but 7-day staleness window
2. Access-token-only embed + access-token rotation on refresh fallback — preserves 15-min bound, ~2-3 hr engineering
3. Revocation blacklist in KV/D1 — adds the cache hit it was trying to eliminate
4. **Keep status quo** (use `isUserAdmin` helper, 1 DB round-trip per gate) ← **Chosen**

**Decision:** `isAdmin` stays out of `PeerloopTokenPayload`. The 6 current gate sites keep calling `isUserAdmin(db, userId)`.

**Rationale:** The refresh-token-as-auth fallback widens staleness to 7 days (not 15 min as the audit framed), which is incompatible with instant admin revocation for security-sensitive gates like payouts (DEPLOYMENT). Performance savings (~6 × 10ms) are negligible at MVP + 10x scale. The helper layer already captures the DRY ergonomics.

**Consequences:** Application decision recorded in `docs/DECISIONS.md` §4 with full 4-option analysis, revisit triggers, and reference to pre-commit session.ts:91 TODO ("could refresh access token here"). Spawned `[RA-SSR-LOADER]` for the one site missed by Conv 123.

> **Note:** Already appended to `docs/DECISIONS.md` §4 during the conv — not re-appended here.
