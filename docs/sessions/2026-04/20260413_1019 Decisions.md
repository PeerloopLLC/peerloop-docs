# Conv Decisions - 2026-04-13

## 1. Consolidate Three Discover Pages into Unified Member Directory
**Type:** Strategic
**Topics:** discover, ux, navigation

**Trigger:** User identified missing Members link and broken search on /discover/creators and /discover/teachers

**Options Considered:**
1. Fix search on existing pages individually
2. Consolidate into single /discover/members page with server-side search ← Chosen
3. Keep pages but wire them to API endpoints

**Decision:** Replace /discover/teachers, /discover/creators, /discover/students with a single /discover/members page featuring server-side search, multi-role filtering, and Load More UX

**Rationale:** Simplifies navigation (3 links → 1), eliminates client-side search cap problem, reduces code (~2350 lines removed, ~450 added), establishes single mental model for member discovery

**Consequences:** Old URLs now 301-redirect; slideout menu simplified; admin-only /discover/members now public (admin extras shown inline)

## 2. Student = Enrollment-Based, Not Capability-Based
**Type:** Implementation
**Topics:** roles, schema, member-directory

**Trigger:** Defining what "Student" means in the member directory filter

**Options Considered:**
1. `can_take_courses = 1` (almost everyone has this capability)
2. Has at least 1 enrollment ← Chosen
3. Has completed at least 1 course

**Decision:** "Student" in the member directory means the user has >= 1 enrollment (enrolled, in_progress, or completed). Members with no enrollments and no other roles get the "Monitoring" badge.

**Rationale:** `can_take_courses = 1` is meaningless as a filter (nearly all users have it). Enrollment-based definition reflects actual engagement.

**Consequences:** API uses EXISTS subquery on enrollments table; "Monitoring" is client-derived (no roles = monitoring)

## 3. Open /discover/members to All Users, Admin Extras Inline
**Type:** Implementation
**Topics:** discover, access-control, admin

**Trigger:** Previously /discover/members was admin-only; consolidation needed a public page

**Options Considered:**
1. Single route, role-aware rendering (Option A) ← Chosen
2. Separate routes (/discover/members public + /admin/members management)
3. Shared component, two routes

**Decision:** Single /discover/members route for everyone. Admins see extra fields (email, status, last active) inline. Management actions stay at /admin/users.

**Rationale:** /admin/users already exists for management. Discovery page doesn't need management actions. Single route avoids code duplication.

**Consequences:** Admin gate removed from members.astro; API returns adminExtras only when caller is admin/mod; privacy_public respected for non-privileged callers

## 4. Dimmed Creator Badge for Creators Without Active Courses
**Type:** Implementation
**Topics:** ux, roles, badges

**Trigger:** Creators who have the capability but haven't published courses need visual differentiation

**Options Considered:**
1. Always show full Creator badge (badges reflect capability)
2. Only show if they have active courses (badges reflect output)
3. Show dimmed badge (opacity-50) ← Chosen

**Decision:** Creator badge renders at 50% opacity when `hasActiveCourses` is false. Tooltip shows "Creator (no active courses)". Click-through to profile explains why.

**Rationale:** Dimming preserves purple color identity while signaling incomplete state. Avoids extra UI complexity (tooltips explaining why badge is absent).

**Consequences:** API returns `hasActiveCourses` boolean; MemberRoleBadge component supports `dimmed` prop; getMemberBadges() sets dimmed flag
