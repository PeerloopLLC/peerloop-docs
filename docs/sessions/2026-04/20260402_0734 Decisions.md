# Conv Decisions - 2026-04-02

## 1. Defer Segments to PLATO-ON-STEROIDS Block
**Type:** Strategic
**Topics:** plato, architecture

**Trigger:** User questioned whether segments add complexity to a working system

**Options Considered:**
1. Implement segments now (7 tasks from Conv 072 design)
2. Defer to PLATO-ON-STEROIDS block — all envisioned scenarios work with existing primitives ← Chosen
3. Implement partial (types only, no runner changes)

**Decision:** Defer all segment implementation. Current primitives (StepRef + actorBindings + sequential instances) are sufficient for all envisioned scenarios.

**Rationale:** Multi-student, post-enrollment, restartability, and step group reuse all work without segments at current scale (4 scenarios, 2 instances). Segments are DX convenience, not capability unlock.

**Consequences:** Created PLATO-ON-STEROIDS deferred block (#41) capturing the full vision: composable data, segments, DB snapshots, automated agent walkthroughs. Removed 7 segment tasks from STUMBLE-AUDIT.

> **Insight:** When a new abstraction layer doesn't unlock capabilities that existing primitives can't achieve, it's DX debt disguised as architecture. Defer until scale forces the issue.

## 2. PLATO Terminology: Modes Replace STUMBLE as System Name
**Type:** Strategic
**Topics:** plato, terminology, glossary

**Trigger:** User confused by overloaded "STUMBLE" term — used as system name and project block

**Options Considered:**
1. Keep existing terms (PLATO run / STUMBLE walkthrough)
2. PLATO = system, API mode / Browser mode = execution modes ← Chosen
3. Rename STUMBLE to something else

**Decision:** PLATO is one system with two modes. "Run" = one execution of an instance. STUMBLE-AUDIT = PLAN.md project block only.

**Rationale:** Eliminates "is this a PLATO thing or a STUMBLE thing?" confusion. WalkthroughCheckpoint already lives in PLATO's type system, confirming it's architecturally one system.

**Consequences:** Updated GLOSSARY.md §2 (9 new terms), plato.md §Execution Modes, memory file. Shorthand: "API-run flywheel" / "browser-run flywheel".

## 3. Snapshot Bridge: Always-Regenerate via Combined Command
**Type:** Implementation
**Topics:** plato, devex, snapshots

**Trigger:** Need to bridge in-memory test DB → real D1 for browser-runs

**Options Considered:**
1. Persistent snapshots with metadata sidecars for staleness detection
2. Timestamp-based staleness checks
3. Always regenerate — `npm run plato:restore` = API-run + restore in one command ← Chosen

**Decision:** No caching. Every restore regenerates the snapshot from current code.

**Rationale:** API-run takes ~400ms — faster than thinking about staleness. Eliminates entire class of bugs (stale schema, stale persona data, stale steps).

**Consequences:** Two scripts (`plato-restore.js`, `plato-restore-snapshot.js`), one npm script, gitignored snapshots directory.

**Topics:** plato, naming

**Trigger:** User concerned about lineage visibility of derived scenarios

**Options Considered:**
1. Generic names (post-enrollment)
