# Conv Learnings - 2026-04-21

## 1. Stripe SDK Crypto Provider Selection by Runtime
**Topics:** stripe, cloudflare, testing

**Context:** Every Stripe webhook delivered to the CF Workers staging Worker since Conv 114's migration had been silently HTTP 400'd. Local `astro dev` (Node) masked the bug entirely — unit tests passed, manual local testing passed.

**Learning:** Stripe SDK's `stripe.webhooks.constructEvent()` dispatches to `NodeCryptoProvider` (sync HMAC via Node `crypto`) in Node runtimes and `SubtleCryptoProvider` (async-only WebCrypto) on CF Workers/browser. Calling the sync form on Workers throws `"SubtleCryptoProvider cannot be used in a synchronous context"` every single time — 100% failure rate. Use `constructEventAsync()` + `await` everywhere to be runtime-agnostic.

**Pattern:**
```ts
// Works on both Node (tests, local dev) and Workers (staging, prod)
export async function constructWebhookEvent(
  stripe: Stripe, payload: string, signature: string, webhookSecret: string
): Promise<Stripe.Event> {
  return stripe.webhooks.constructEventAsync(payload, signature, webhookSecret);
}
```

## 2. Live-Verification Catches Runtime-Specific Bugs That Code Review + Unit Tests Miss
**Topics:** testing, cloudflare, stripe

**Context:** [VS] scope was "run 7 scenarios against staging to verify the miss-resilience doc's predicted behavior." The actual value was finding three runtime-specific production bugs ([VH] constructEventAsync, [VD] UNIQUE collision, [VW] fire-and-forget race) that unit tests + code inspection had not surfaced.

**Learning:** Staging live-verification is not optional QA theater — it's the only place bugs that depend on actual runtime behavior (Workers vs Node, real D1 constraint enforcement, async termination semantics) become observable. Code-inspection-only verification would have shipped all three bugs to production.

## 3. `wrangler tail --env staging` Is the Definitive Debugging Tool for Staging Worker Failures
**Topics:** cloudflare, testing

**Context:** Spent ~30 min assuming a secret-mismatch when HTTP 400 persisted after careful rotation. `stripe trigger` in the CLI was firing to Test mode (wrong Stripe mode for Sandbox endpoint) and had zero diagnostic value — no entries anywhere.

**Learning:** When a Worker returns an opaque 4xx/5xx, `wrangler tail --env <env>` streams the Worker's `console.error` output in real time — frequently the exact error message, not just the HTTP code. Should be the first diagnostic step, not a late-stage fallback.

## 4. Stripe Sandbox Is a Third Mode (2024+), Distinct from Test Mode
**Topics:** stripe, docs-infra

**Context:** The Peerloop project's assumed mental model was "Live vs Test mode" — a two-axis framing from pre-2024 Stripe. Debugging blew up when `stripe trigger` fired Test-mode events that never reached the Sandbox-registered endpoint, despite both modes using `sk_test_` / `pk_test_` prefixes that look identical in CLI output.

**Learning:** Stripe released Sandboxes in 2024 as fully isolated test environments parallel to (not replacing) Test mode. They have disjoint API keys, webhook endpoints, customers, charges, events — a key issued in one mode is unusable in another. Only the Dashboard's Workbench banner (`Alpha Peer LLC sandbox` vs plain test-data header) and the webhook endpoint URLs visually distinguish them. Project decision recorded: **Local=Test, Staging=Sandbox, Prod=Live** (never mix).

## 5. Fire-and-Forget DB Writes Race Worker Termination in Short-Path Handlers
**Topics:** cloudflare, testing

**Context:** S7 (`transfer.reversed`) returned HTTP 200 but no `webhook_log` row landed. S1-S6 all logged correctly because their handlers did real async DB work that kept the Worker alive long enough for the unawaited `webhook_log` INSERT to complete.

**Learning:** On CF Workers, `promise.then(...).catch(...)` without `await` or `ctx.waitUntil(...)` is racing Worker termination. For handlers that return a Response quickly (default-case loggers, early-returns), the write can be cut short. Pattern: always wrap forensics/telemetry writes in `ctx.waitUntil(promise)` so the Worker runtime guarantees completion.

## 6. Secret Rotation UX Pitfall: "Expiry" Applies to the OLD Secret, Not the New One
**Topics:** stripe, security

**Context:** User hit the Stripe Dashboard's roll-secret modal and was confused whether the expiration dropdown was setting when the OLD key dies or when the NEW key becomes invalid.

**Learning:** Stripe (and many other services with graceful key rotation) puts expiry on the OLD secret. The new secret is immediately active and permanent until the next rotation. The grace period serves multi-instance deploys so both keys verify during rollout. Single-Worker environments should always pick the shortest/immediate expiry option. Same principle applies to Stripe API keys (`sk_test_`, `sk_live_`) when rolling those.
