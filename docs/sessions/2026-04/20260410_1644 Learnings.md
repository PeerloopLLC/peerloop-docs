# Conv Learnings - 2026-04-10

## 1. TypeScript 6 ecosystem lag — force-fitting peer versions is anti-durable
**Topics:** typescript, package-updates, ecosystem

**Context:** Phase 2 attempted to bump TS 5.9.3 → 6.0.2 alongside Astro 6. Install succeeded with 12+ peer-dep warnings because `@astrojs/check@0.9.8`, `@typescript-eslint/*@8.58.1`, and even `tsconfck` (vendored inside Astro 6 itself) all pin `typescript@^5.0.0`. TS 6.0.2 shipped ~early April 2026 — the ecosystem literally had not had time to catch up.

**Learning:** "Latest-published" is not the same as "latest-usable". A major version that breaks peer deps across the primary meta-framework's own vendored modules is not a durable upgrade — it's a silence-warnings-and-hope upgrade. The TS 6 release blog itself calls it a "bridge release" toward TS 7's native rewrite. Let one ecosystem cycle pass; split the phase, defer the bump, revisit when `npm ls typescript` shows zero "invalid peer" lines.

**Pattern:** Before accepting a major version bump, run `npm ls <package>` after install and check for `invalid` markers. If the framework you're using bundles a version-pinned internal dep (like Astro vendoring `tsconfck`), that's a hard blocker — you cannot upgrade faster than the framework allows.

## 2. Abstraction-centralization as a prep step decouples pattern change from version bump
**Topics:** refactoring, cloudflare, astro, architecture

**Context:** `@astrojs/cloudflare` v13 removes `locals.runtime.env.*` in favor of `import { env } from 'cloudflare:workers'`. The naive approach would be to bump the adapter and rewrite 101 files in one commit. Problem: two coupled changes in one diff means any failure could be either the version bump or the pattern change.

**Learning:** If a version bump requires a pattern change across many call sites, first centralize the pattern behind helpers on the OLD version. The centralization is independently valuable (consistency, dedup, testability) and makes the version bump a mechanical change to a small number of helper files. Two atomic commits, two verification points, surgical rollback if anything breaks.

**Pattern:** "Preparatory refactor" commits — they're never wasted work even if the version bump is later abandoned, because the centralization stands on its own merit.

## 3. Subagent instructions must anticipate pattern-hiding disguises
**Topics:** subagents, refactoring, delegation

**Context:** Dispatched a subagent to sweep ~95 files converting direct `locals.runtime?.env?.X` reads to helper calls. The prompt was detailed (six named patterns, explicit file exclusions, pilot files, verification instructions). Subagent reported "0 files flagged, 100% clean" — but an independent grep found 5 files the subagent had completely missed. All 5 used `locals.runtime?.env as { BBB_URL?: string; BBB_SECRET?: string }` — an **inline type cast** was hiding the runtime access from the subagent's pattern matcher.

**Learning:** Subagent grep-driven sweeps only find patterns you named. Patterns that use syntactic disguises (type casts, chained property access, wrapped expressions) will be invisible unless explicitly enumerated in the instructions. After any subagent sweep, the main context must re-grep for the target pattern with a broader regex and verify claimed completeness.

**Pattern:** Post-sweep verification step: `grep -rn "<baseline pattern>" src/ | grep -v "<expected exclusions>"` should return zero lines. If not, the remainder are either edge cases the subagent should have flagged, or cases it failed to recognize.

## 4. Test mocks of library modules must expose all imported symbols, not just the used ones
**Topics:** testing, vitest, refactoring

**Context:** After centralizing env access through `@lib/r2` helpers, two test files broke. Their `vi.mock('@/lib/r2', () => ({ ... }))` stubs exposed only the symbols the endpoint imported **at the time the mock was written**. The refactor added new imports (`getR2Optional`) that the mock didn't cover, so the partial mock returned `undefined` for the new import. Calling `undefined(locals)` threw a TypeError caught by the outer try/catch → wrong status code → test failure.

**Learning:** Partial module mocks are a time bomb. The test "works" until someone adds an import from the mocked module, at which point the runtime error is silent (caught by the endpoint's own error handling) and only manifests as a status-code assertion failure far from the root cause. Prefer `importOriginal` pattern (vitest supports it) OR list every exported symbol in the mock stub with a reasonable default. When refactoring a file that's mocked in tests, grep the tests for `vi.mock('@/lib/<mocked>')` and update mock surfaces.

**Pattern:**
```ts
vi.mock('@/lib/r2', async (importOriginal) => ({
  ...(await importOriginal()),
  // override only what you need
  replicateRecordingToR2: (...args) => mockReplicateRecording(...args),
}));
```

## 5. `baseUrl` without paths in tsconfig needs `./` prefix
**Topics:** typescript, tsconfig

**Context:** Removed `"baseUrl": "."` from `tsconfig.json` (deprecated in TS 6, harmless on TS 5 but cleaner to drop ahead of time). tsc immediately errored with `TS5090: Non-relative paths are not allowed when 'baseUrl' is not set` across every `paths` entry.

**Learning:** When `baseUrl` is unset, TypeScript requires all `paths` values to start with an explicit relative prefix (`./` or `../`). On older TS versions, `["src/*"]` was implicitly interpreted as relative to `baseUrl` (defaulted to the tsconfig dir when unset). The newer TS versions don't infer — they require the prefix.

**Pattern:** When removing `baseUrl`, prefix every paths value: `"@lib/*": ["./src/lib/*"]`. Mechanical change, but tsc fails loudly if you miss one.
