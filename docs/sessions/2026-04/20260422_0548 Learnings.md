# Conv Learnings - 2026-04-22

## 1. Doc-drift audit memos go stale between identification and execution
**Topics:** docs-infra, cc-workflow

**Context:** Conv 140's `/w-sync-docs` audit produced an exact 14-item drift list with specific X→Y deltas (e.g., "Admin 65→67"). By Conv 146 (7 days later), counts had shifted — Admin was now 68, API total 231. Blindly applying the Conv 140 deltas would have introduced a new drift.

**Learning:** For drift-audit execution, treat stored deltas as *scope* (what to fix) but re-verify the *values* against current ground truth at execution time. Audit and execution are temporally decoupled; the audit identifies which fields need reconciling, not what they should be reconciled to.

**Pattern:** Before applying each fix from an old audit memo, run a ground-truth command (`find`, `grep`, `wc -l`) to recompute the target value. Use the audit's list-of-fields as the work scope, not its list-of-values.

## 2. Catch-all "Other"/"Misc" sections in reference docs breed drift
**Topics:** docs-infra

**Context:** TEST-COVERAGE.md's `### Other API — tests/api/ top-level (23 files)` section name was misleading for ≥1 year — the rows actually included 15 files from 10 subdirs (certificates, debug, recommendations, resources, reviews, stories, stream, stripe, submissions, topics) plus 8 true root files. A duplicate `topics/index.test.ts` row went undetected in the flat table. The per-subdir H3 pattern used for every *other* subdir (admin, auth, ...) kept those sections accurate; the "Other" exception was drift-vulnerable.

**Learning:** Catch-all sections invite low-visibility drift because the heading's imprecision legitimizes imprecise contents. When a reference doc has a clear structural pattern (one H3 per unit), "Other" exceptions should be broken into proper sub-units as soon as they grow beyond a handful of entries. The structural discipline is cheaper over the long run than the catch-all's "convenience".

**Pattern:** For reference docs with per-X subsections, avoid "Other X" sections. If they exist, split as soon as the catch-all exceeds ~3-5 items. Cost: a few H3 headers. Benefit: each item becomes grep/anchor-navigable + counts stay honest.

## 3. Spec and lessons-learned belong in separate doc locations
**Topics:** docs-infra, cc-workflow

**Context:** For the ESLint v10 unknown-rule gotcha, Conv 143 had put the spec portion in `DEVELOPMENT-GUIDE.md §"ESLint Configuration"`. Conv 146's `[EG]` task asked for the *lessons* portion in `PACKAGE-UPDATES notes` — a seemingly redundant ask, but actually a different audience.

**Learning:** Spec doc (DEVELOPMENT-GUIDE.md) answers "how do I configure this now" — live content, subject to rewrite when the config is restructured. Lessons/gotchas (PACKAGE-UPDATES notes) answers "what bit us during the last upgrade" — frozen history attached to the upgrade event. The split means the next upgrade review can find the gotcha intact even after `eslint.config.js` has been rewritten three times.

**Pattern:** Cross-link asymmetrically — lessons → spec (high value, spec is stable anchor), skip the reverse (spec readers are usually configuring fresh, not debugging upgrades). Keeps both docs maintainable in isolation.
