# Conv Learnings - 2026-04-13

## 1. Astro 6 Adapter 13 Output Layout is Workers-only — CF Pages Can't Execute It
**Topics:** astro, cloudflare, deployment

**Context:** Conv 113 patched the adapter's generated wrangler.json to pass Pages validation; that worked for builds but produced static-only deployments (no SSR code served).

**Learning:** `@astrojs/cloudflare@13` emits `dist/server/entry.mjs` (Workers entrypoint) + `dist/client/` (static) + `dist/server/wrangler.json` (Workers-format config). CF Pages only recognizes a `functions/` directory OR a `_worker.js` at the asset root — neither of which the adapter produces. The "successful deploy" log line `"No functions dir at /functions found. Skipping"` is the smoking gun: Pages silently serves static assets with no request handler, so every dynamic route 404s. Postbuild patching can fix config validation but cannot restructure the output shape.

**Pattern:** When an adapter migration involves a platform-specific output layout change, the only correct fix is to migrate the platform. Config patching delays the reckoning.

## 2. The `@cloudflare/vite-plugin` Validates `main` at Config-Resolution Time
**Topics:** astro, cloudflare

**Context:** Initial plan put `main = "dist/server/entry.mjs"` and `[assets]` at the root `wrangler.toml`. Build failed with "main field doesn't point to an existing file" — before the build had run.

**Learning:** The vite plugin reads wrangler.toml during Astro's sync phase (well before the build output exists). Keeping `main`/`[assets]` out of the root `wrangler.toml` and letting the adapter emit them into `dist/server/wrangler.json` avoids the chicken-and-egg failure. The generated file is what wrangler actually reads at deploy time via `.wrangler/deploy/config.json` redirection.

**Pattern:** Root wrangler.toml for Astro 6 + adapter 13 = bindings, vars, and env overrides only. No `main`, no `[assets]`.

## 3. Environments are a Build-Time Concern with This Adapter, Not a Deploy-Time One
**Topics:** cloudflare, deployment, astro

**Context:** `wrangler deploy --env staging` silently deployed to the wrong Worker (`peerloop` with prod bindings instead of `peerloop-staging`) because the adapter's generated wrangler.json is flat — no env sections left to select.

**Learning:** `@cloudflare/vite-plugin` reads `CLOUDFLARE_ENV` at BUILD time and flattens the chosen `[env.<name>]` overrides from wrangler.toml into `dist/server/wrangler.json`. At deploy time, wrangler follows `.wrangler/deploy/config.json` to that already-flattened file — `--env staging` is a no-op. The deploy scripts must prefix `CLOUDFLARE_ENV=staging` (selects bindings) AND `ENVIRONMENT=staging` (bakes into Vite `define`).

**Pattern:** `"deploy:staging": "CLOUDFLARE_ENV=staging ENVIRONMENT=staging npm run build && wrangler deploy"`.

## 4. `legacy_env: true` + `--env staging` Double-Suffixes the Worker Name
**Topics:** cloudflare, deployment

**Context:** Ran `wrangler secret bulk /tmp/secrets.json --name peerloop-staging --env staging` → created Worker `peerloop-staging-staging` (not `peerloop-staging`) and uploaded secrets to it.

**Learning:** With `legacy_env: true` (set by the adapter in the generated config), `--env staging` appends `-staging` to the worker name. Combined with `--name peerloop-staging`, the result is `peerloop-staging-staging`. For secret operations (and any non-deploy wrangler command), use `--name peerloop-staging` alone, no `--env`. For deploy, use neither — the CLOUDFLARE_ENV at build time handles it.

**Pattern:** Secrets: `wrangler secret bulk FILE --name peerloop-staging`. Deploy: `CLOUDFLARE_ENV=staging ... wrangler deploy` (no `--env`).

## 5. Build-Time Constants via Vite `define` Beat Runtime Env Detection
**Topics:** astro, testing

**Context:** `src/lib/version.ts::getEnvironment()` had gnarly `CF_PAGES`/`CF_PAGES_BRANCH`/hostname detection that ran on both server and client. Replaced with `__ENVIRONMENT__` build-time constant.

**Learning:** Vite constant-folds `getEnvironment()` down to a literal `return "STG"` when the ternary chain is statically resolvable. Zero runtime cost, zero hostname drift risk, zero footgun surface. The tradeoff — you must build fresh per environment, can't promote a staging build to prod — is actually a safety feature.

**Pattern:** Declare in `astro.config.mjs` define + `src/env.d.ts`. Read in source as `ENVIRONMENT` constant. Set `process.env.ENVIRONMENT` in deploy scripts.

## 6. Workers Has No Git Integration by Default — Explicit is the New Implicit
**Topics:** cloudflare, deployment, cc-workflow

**Context:** User asked how branches relate to Workers after the migration.

**Learning:** Pages' "branch = deployment, 1:1 auto" model is gone. Workers' deployment unit is the **Worker name** (`peerloop-staging`, `peerloop`), not the branch. Code is whatever the last `wrangler deploy` uploaded. Feature branches no longer get free preview URLs. GitHub Actions can restore the "branch triggers deploy" UX but as explicit CI config, not platform magic.

**Pattern:** Document the mental model shift alongside the technical migration — it's not obvious to teammates coming from a Pages/Vercel-style workflow.

## 7. Pages ↔ Workers Secrets Don't Carry Over
**Topics:** cloudflare, deployment

**Context:** User tried to log in on the new staging Worker → "JWT secret not configured".

**Learning:** CF Pages conflated "env vars" and "secrets" in one dashboard panel. Workers enforces a strict split: `[vars]` in wrangler.toml are public (committed), secrets are encrypted, set out-of-band via `wrangler secret put/bulk`. When migrating, secrets must be re-uploaded to each new Worker.

**Pattern:** Use `wrangler secret bulk /tmp/secrets.json --name <worker>` with a JSON extracted from `.dev.vars`. Documented in `docs/reference/cloudflare.md` §Secrets.
