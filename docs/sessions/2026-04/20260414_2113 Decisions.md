# Conv Decisions - 2026-04-14

## 1. Kill `community_members.member_role='teacher'` entirely
**Type:** Strategic
**Topics:** schema, roles, community_members, terminology

**Trigger:** Discovered the value is unreachable from application code (no INSERT/UPDATE site writes it) despite 20+ UI/API sites consuming it.

**Options Considered:**
1. Kill it entirely — drop from schema + all consumers ← Chosen
2. Auto-maintain it via trigger/sync from `teacher_certifications`
3. Redefine as an explicit curator role (needs product input)

**Decision:** Option 1 (kill). Filed as task #28 COMMUNITY-TEACHER-KILL.

**Rationale:** The value carries no current meaning and has accumulated UI misrepresentation (e.g., "teachingCount" badges always zero in prod). Auto-maintenance (option 2) adds a sync surface for a concept that's already course-scoped via `teacher_certifications`. A redefinition (option 3) would need product input and is not on the near-term roadmap.

**Consequences:** 20+ sites change. `CommunityTeachingTab`, "Teaching" filter pill, and teachingCount badge must be re-derived from `teacher_certifications JOIN courses ON community_id` (per decision 2 below). Schema CHECK narrows from 3 values to 2. Dev seed rows update.

## 2. "Communities where I'm teaching" — re-derive rather than drop
**Type:** Strategic
**Topics:** discover-ui, teacher-certifications

**Trigger:** Killing `member_role='teacher'` kills the data source for this feature.

**Options Considered:**
1. Kill the feature entirely — drop tab/pill/badge
2. Re-derive from `teacher_certifications JOIN courses ON community_id` — preserve UX ← Chosen
3. Defer — leave tab hidden until re-derived

**Decision:** Option 2 (re-derive).

**Rationale:** The feature is user-visible and purposeful — teachers want to see which communities their certified courses live in. The data exists in `teacher_certifications` already; this is a computation change, not a feature scope change.

**Consequences:** New query/loader needed. `CurrentUser` already has `teacherCertifications` map with course metadata — could add a derived getter like `getTeachingCommunityIds()`.

## 4. Test POST coverage via JSON-link path only for Phase 7
**Type:** Implementation
**Topics:** testing, scope

**Trigger:** Multipart file-upload tests require R2 mocking + File/FormData construction overhead.

**Options Considered:**
1. Test both JSON and multipart paths
2. Test JSON only — auth gate is identical across both code paths ← Chosen

**Decision:** Option 2 for this conv.

**Rationale:** `resolveAndAuthorize` is the same function for both paths. Auth-matrix coverage is fully satisfied by JSON. Multipart-specific plumbing (file-type allowlist, size limits, R2 upload call, key generation) would be additional happy-path coverage worth adding later but not required to gate the auth behavior.

**Consequences:** ~50 tests landed. Multipart upload path remains untested — noted for future follow-up.
