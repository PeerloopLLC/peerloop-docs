# Conv Decisions - 2026-04-02

## 1. BrowserIntent replaces WalkthroughCheckpoint
**Type:** Strategic
**Topics:** plato, testing, architecture

**Trigger:** PLATO browser-runs were using prose WalkthroughCheckpoints that "teleported" to pages via address bar instead of navigating like real users.

**Options Considered:**
1. Keep prose WalkthroughCheckpoints (status quo)
2. Fully structured navigation AND page actions (mini-Playwright DSL) ← Too complex
3. Structured navigation + prose page actions (BrowserIntent) ← Chosen

**Decision:** Replace WalkthroughCheckpoint with BrowserIntent: `navigate: { via, clicks[] }` for deterministic navigation, `pageAction: string` for prose instructions, `coversStepActions` for API-run cross-reference.

**Rationale:** Navigation is mechanical and testable. Page actions are too varied to structure. The hybrid captures the deterministic contract (fail-fast on missing UI elements) without over-engineering.

**Consequences:** New types in types.ts. All existing instances migrated. Navigation helper created for authoring-time use.

## 2. Route↔API Map as self-maintaining pipeline
**Type:** Implementation
**Topics:** plato, infrastructure, docs

**Trigger:** Need to know which UI pages call which API endpoints and how to navigate there.

**Options Considered:**
1. Documentation-only Markdown reference ← Easy but not consumable by code
2. TypeScript lookup only ← Code-consumable but not human-readable
3. Both, generated from same scanner script ← Chosen

**Decision:** Single scanner script (`route-api-map.mjs`) generates both TypeScript lookup and Markdown reference. Wired into `/r-end` docs agent for auto-regeneration.

**Rationale:** Code needs the data (PLATO helpers). Humans need the reference (scenario authoring). One source of truth prevents drift.

**Consequences:** `npm run route-api-map` regenerates both. Docs agent detects API/component changes and triggers re-run.

## 3. Diagnostic instances are ephemeral, not permanent
**Type:** Strategic
**Topics:** plato, testing

**Trigger:** `flywheel-to-enrollment` was a diagnostic segment created to isolate Conv 073 bugs, now fixed.

**Options Considered:**
1. Keep all instances permanently ← Accumulates stale diagnostic artifacts
2. Delete diagnostic instances when bugs are fixed ← Chosen
3. Rename with `diag-` prefix to distinguish ← Middle ground, still accumulates

**Decision:** Diagnostic segments are deleted when their bugs are fixed. If the journey is independently useful, promote to a named scenario.

**Rationale:** PLATO-REGISTRY should track permanent scenarios, not ephemeral debug probes. Git history preserves the diagnostic if needed later.

**Consequences:** Deleted `flywheel-to-enrollment.instance.ts`, removed from index and test file. 8 → 7 PLATO tests.

## 4. Navigation rules: same-page first, then AppNavbar
**Type:** Implementation
**Topics:** plato, navigation

**Trigger:** Multiple paths exist to reach most pages. Need deterministic rule for PLATO browser-runs.

**Options Considered:**
1. Always start from AppNavbar ← Misses same-page shortcuts
2. Same-page link first, AppNavbar fallback ← Chosen (user's a/b rules)
3. Shortest BFS path regardless of source ← Could pick non-obvious paths

**Decision:** Rule (a): If target route has a link/button on the CURRENT page → `via: 'same-page'`. Rule (b): If not → start from AppNavbar, follow BFS shortest path.

**Rationale:** Users naturally click links on the current page before going to the sidebar. This matches real user behavior.

**Consequences:** `suggestNavigation()` helper implements these rules. `outboundLinks` added to RouteInfo to support rule (a) checks.
