# Conv Learnings - 2026-04-11

## 1. `tsc --noEmit` does NOT scan `.astro` files — baseline gap
**Topics:** testing, astro, cc-workflow

**Context:** Running /w-codecheck for the first time this conv revealed 10 type errors in `.astro` files that `tsc --noEmit` had never caught. Conv 100–103 all claimed "tsc 0 errors, clean baselines" based on the triple of tsc/test/build, but were incomplete.

**Learning:** The Peerloop baseline check set must be five gates, not three: `tsc --noEmit`, `npm run check` (astro), `npm run lint`, `npm test`, `npm run build`. Astro provides its own TypeScript diagnostics for `.astro` files via the Astro Language Server; tsc only looks at `.ts`/`.tsx`. This gap existed at every layer: local baselines, CI, /r-end/r-commit skills. All five gates must be green before claiming "clean baselines" in session docs, RESUME-STATE, PR descriptions, or memory entries.

## 2. Stripe SDK major bumps are usually only a one-line apiVersion change
**Topics:** stripe, testing

**Context:** Upgrading `stripe@^20.1.0` → `^22.0.1` produced exactly one tsc error: `apiVersion: '2026-02-25.clover'` not assignable to `'2026-03-25.dahlia'`. Everything else — `new Stripe(...)`, `Stripe.Event`, `Stripe.Checkout.Session`, `.accounts.create`, `.transfers.create`, `.webhooks.constructEvent` — stayed identical.

**Learning:** Stripe SDK majors pin the `apiVersion` literal type to exactly one version (the one the SDK ships with). Upgrading the SDK mechanically forces bumping the `apiVersion` string to match. Because we centralize the pin in `src/lib/stripe.ts` (Conv 100 directive), the upgrade surface is one line. However, bumping `apiVersion` is a BEHAVIORAL change, not just a type change — the server returns different wire shapes. Always audit Stripe's changelog for the specific delta (here: `.clover` → `.dahlia` had BC to checkout UI mode enum + Capabilities API risk requirements, both safely handled by our code).

## 3. ESLint `argsIgnorePattern` does NOT apply to tuple destructure bindings
**Topics:** testing, cc-workflow

**Context:** While cleaning up [LD], hit a case where `const [actionLoading, setActionLoading] = useState(false)` had `setActionLoading` as the unused half. Prefixing with `_setActionLoading` alone didn't silence the lint rule because `argsIgnorePattern` only applies to function arguments, not tuple destructure variables.

**Learning:** `@typescript-eslint/no-unused-vars` has three separate patterns: `argsIgnorePattern` (function args), `varsIgnorePattern` (regular variables), `destructuredArrayIgnorePattern` (tuple/array destructure bindings). Extended our config to include all three keyed on `^_`. This is also the pattern React's `useState` produces, so relevant for any half-wired state scaffolding pattern.

## 4. TypeScript `noUnusedLocals` vs ESLint — parallel checks with different ignore semantics
**Topics:** testing, astro

**Context:** `r2-recording.test.ts:142` had `const { prepare, _bind, _run } = createMockDB()` — `_run` was flagged as unused despite the `_` prefix. ESLint config respects `_` for these (post-Conv 104 change) but `tsc --noEmit` has its own `noUnusedLocals` check that's not configurable the same way.

**Learning:** There are TWO parallel unused-checking systems:
- ESLint `@typescript-eslint/no-unused-vars` (configurable ignore patterns, scoped by `src/`)
- TypeScript native `noUnusedLocals` / `noUnusedParameters` (stricter for locals, softer for params — underscore prefix works for params only)

Astro check surfaces the TS native check across `.astro` AND `tests/` directories that ESLint doesn't scan. So TS's stricter locals check is the effective gate for tests. Cleanest fix for tests is to actually remove the binding (skip-comma or full delete), not suppress with underscore.

## 5. Fix type definitions honestly rather than cast at call sites — duplicate types are a symptom
**Topics:** astro, testing

**Context:** The 8× `CourseTag` type mismatch had THREE definitions: `lib/db/types.ts` (junction row `{course_id, tag_id}`), `lib/mock-data.ts` (display `{tag_id, name}`), `components/courses/course-tabs/types.ts` (display `{tag_id, name}`). The .astro pages imported `CourseTag` from `@lib/db/types` intending the display shape but getting the junction shape.

**Learning:** When a type name is used for two semantically different things, the fix is rarely "cast at the call site" — it's "rename the wrong one and consolidate the right one." We renamed `CourseTag` → `CourseTagRow` (junction), made `CourseTag` the canonical display shape in `lib/db/types.ts`, and had `mock-data.ts` + `course-tabs/types.ts` re-export from there. **Zero .astro file edits needed** because they were already importing by name from the right module — the rename fixed them en masse. Contrast: if we'd cast at each site, we'd have 8 casts to maintain and the duplication would rot further.

## 6. Runtime narrower than type annotation lie — `initialTab?: TabId` accepted any string
**Topics:** astro, testing

**Context:** While fixing [AC], discovered `CourseTabs.tsx:94` uses `useState<string>(initialTab)` — the runtime accepts any string. But the type declared `initialTab?: TabId` (6 universal tab names). Role-specific IDs like `'certificate'`, `'studio'` passed through `extraTabs` wouldn't type-check as `initialTab`, even though runtime handled them fine.

**Learning:** When a TS type is narrower than the runtime, the type is the bug. The narrow type passed tsc for months because `.astro` pages (where the mismatch happened) weren't scanned. Widened to `TabId | (string & {})` — the `(string & {})` idiom keeps autocomplete for the literal union while accepting any string. This is a pattern worth remembering for React props that flow into open-string state.

## 7. Dead dependencies survive refactors unless the cleanup explicitly deletes them
**Topics:** cc-workflow

**Context:** `zod` was added in commit `17f6db9` (2026-01-08) for the PageSpec validation system. Session 307 deleted the entire PageSpec system in commit `424ef2c` (149 files, 39,964 lines) — but the zod dep in `package.json` was never removed. Discovered during Conv 104 [P3] survey.

**Learning:** Large refactor-deletes should include a `package.json` audit step. Dead deps are invisible because `npm install` succeeds and tests pass. Added [ZU] follow-up for zod specifically; could generalize to a `/w-codecheck` extension that greps `node_modules` for direct deps with zero source imports.
