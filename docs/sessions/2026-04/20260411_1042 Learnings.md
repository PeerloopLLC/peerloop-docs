# Conv Learnings - 2026-04-11

## 1. Generated-artifact edits don't survive — only source edits do
**Topics:** cc-workflow, skills

**Context:** Earlier in the conv I edited `.timecard.md` directly to add a Branch column as a demo, then later had to re-discover that the same change needed to go into `.claude/skills/r-timecard-day/SKILL.md` to persist. The user explicitly flagged this.

**Learning:** When a skill generates an output file (`.timecard.md`, `Extract.md`, etc.), fixing the output is cosmetic — the next run will overwrite it. Persistent behavior changes must go into the SKILL.md itself.

**Pattern:** Rule of thumb: "if the change should outlive one run, edit the recipe, not the baked good." Same distinction as source code vs build output.

## 2. Git log without a branch argument is a silent billing bug
**Topics:** cc-workflow, skills, git

**Context:** `/r-timecard-day` ran `git log --since... --until...` on the code repo, reading from HEAD (`jfg-dev-9`) only. Today's 4 code commits lived exclusively on `jfg-dev-10up` (the Astro 6 upgrade branch). The skill returned zero code commits and would have produced a timecard missing 4 of 10 commits. I only caught it because Conv 100's docs commit message textually referenced the code hashes by short-SHA.

**Learning:** For reporting skills (as opposed to commit-creation skills), HEAD-only `git log` is a billing-accuracy bug whenever long-lived branches exist. The bug is silent by default — a missing-commit timecard still looks valid.

**Pattern:**
- **Reporting skills** → scan all local branches (`git for-each-ref refs/heads/` or `--branches`), de-dup by hash.
- **Commit-creation skills** (`/r-commit`, `/q-commit`) → HEAD-only is correct by design.

Same plumbing, opposite correct answer depending on intent.

## 3. Date-based queries warrant a prompt; unique-ID queries don't
**Topics:** cc-workflow, skills, ux

**Context:** Both `/r-timecard-day` (date-based) and `/r-timecard` Mode 2 (conv-based) had the same HEAD-only bug. But the fix UX is different:
- `/r-timecard-day` uses **per-branch iteration + 👉👉👉 prompt** because date windows are inherently ambiguous (a user can have multiple branches with legitimate in-window commits — not all of which belong in the bill).
- `/r-timecard` conv mode uses **`--branches` union + silent per-commit resolve** because conv numbers are uniquely identifying (one conv's work is unambiguous — just find the commits wherever they are).

**Learning:** When deciding whether to prompt, ask: is the query inherently unambiguous? If yes, silent fix with union+dedup. If no, prompt with sensible default (include-all).

**Pattern:** Same underlying fix (`--branches` or `for-each-ref` + dedup), different wrapper UX based on query semantics.

## 4. `git log --branches` is the clean "all local heads" flag
**Topics:** git

**Context:** Used `git log --branches` in `/r-timecard` conv mode fix — cleaner than `--all` (which includes `origin/*` remote noise and tags) and more readable than `git log $(git for-each-ref --format='%(refname)' refs/heads/)`. Didn't use it in `/r-timecard-day` because that skill needs per-branch commit *counts* for the detect-and-prompt display — iteration was required.

**Learning:** `--branches` = all local refs/heads/*, no remotes, no tags. Use it when you need union-across-branches without the explicit iteration.

**Pattern:** `git log --branches --grep=...` = canonical "find commits matching X on any local branch."

## 5. RESUME-STATE.md is tracked, not gitignored
**Topics:** cc-workflow, dual-repo

**Context:** I confidently told the user earlier in the conv that RESUME-STATE.md was gitignored and could be edited freely. When /r-start halted on a dirty repo check, `git status` showed ` M RESUME-STATE.md` (tracked + modified). Conv 102's commit message had explicitly said "RESUME-STATE.md comprehensively rewritten with 24 pending tasks carried forward" — confirming it IS committed to the repo.

**Learning:** RESUME-STATE.md is part of the committed repo history, not scratch state. Any edit to it is a real uncommitted change. However, it's also *consumable* by /r-start Step 7 which reads it and deletes it.

**Pattern:** The dirty-repo guard in /r-start correctly halts on RESUME-STATE.md modifications, but the user can safely choose "proceed anyway" because the deletion in Step 7 is the intended end-state.

## 6. /r-start Step 7 doesn't dedupe against pre-existing TodoWrite items
**Topics:** cc-workflow, skills

**Context:** In the no-clear /r-start path (Conv 103), TodoWrite already contained 2 tasks from earlier in the same conversation (the [BD] and [HA] tasks I'd just created). When Step 7 transferred the 26 items from RESUME-STATE.md's TodoWrite Items list, the last two (tasks #26 [BD] and #27 [HA]) were duplicates of the pre-existing ones. I manually deleted the pre-existing ones before running the bulk TaskCreate.

**Learning:** /r-start Step 7 assumes TodoWrite is empty because it normally runs after `/clear`. In no-clear paths this assumption breaks and manual dedup is needed.

**Pattern:** Not adding a task for this — it's a minor 3rd-order skill improvement, lower priority than the skill bugs already in the queue. Documented here for future awareness.
