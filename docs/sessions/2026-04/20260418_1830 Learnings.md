# Conv Learnings - 2026-04-18

## 1. jsdom FormData Does Not Preserve File.name Through Serialization
**Topics:** testing, vitest, jsdom, formdata

**Context:** Writing multipart upload tests with `vi.env: 'jsdom'`. Test used `new File(['content'], 'filename.pdf')` appended to FormData, then `new Request(url, { body: formData })`. The endpoint's `request.formData()` returned a Blob with `.name === 'blob'` instead of the original filename.

**Learning:** jsdom's FormData serialization drops the `filename=` parameter from Content-Disposition even with the 3-arg `append(key, file, name)` form. Node.js native FormData handles this correctly, but the test environment's jsdom FormData does not. Workaround: construct the multipart body manually as a `Uint8Array` with explicit boundary and Content-Disposition headers, then set Content-Type header directly on the Request.

**Pattern:**
```typescript
async function createMultipartRequest(fields: Record<string, string | File>): Promise<Request> {
  const boundary = `TestBoundary${Math.random().toString(36).slice(2)}`;
  const encoder = new TextEncoder();
  const chunks: Uint8Array[] = [];
  for (const [key, value] of Object.entries(fields)) {
    if (value instanceof File) {
      chunks.push(encoder.encode(`--${boundary}\r\nContent-Disposition: form-data; name="${key}"; filename="${value.name}"\r\nContent-Type: ${value.type}\r\n\r\n`));
      chunks.push(new Uint8Array(await value.arrayBuffer()));
      chunks.push(encoder.encode('\r\n'));
    } else {
      chunks.push(encoder.encode(`--${boundary}\r\nContent-Disposition: form-data; name="${key}"\r\n\r\n${value}\r\n`));
    }
  }
  chunks.push(encoder.encode(`--${boundary}--\r\n`));
  const body = new Uint8Array(chunks.reduce((n, c) => n + c.length, 0));
  let offset = 0; for (const c of chunks) { body.set(c, offset); offset += c.length; }
  return new Request(url, { method: 'POST', body, headers: { 'Content-Type': `multipart/form-data; boundary=${boundary}` } });
}
```

## 2. vi.mock Factory Form Requires Explicit Enumeration of All Exports
**Topics:** testing, vitest, mocking

**Context:** `decline.ts` was updated to import `createNotification` from `@/lib/notifications`. The integration test used the factory form `vi.mock('@/lib/notifications', () => ({ ... }))` listing only the functions it needed. When the endpoint called the newly-imported `createNotification`, Vitest threw "No 'createNotification' export is defined on the mock."

**Learning:** The `vi.mock(module, () => ({ ... }))` factory form produces a mock with only the explicitly-listed exports — it does NOT fall through to the real module. Adding any new import to a module under test requires updating all factory-form mocks in tests that cover that code path. The `importOriginal` form (`vi.mock(m, async (importOriginal) => { const actual = await importOriginal(); return { ...actual, ... }; })`) avoids this — it spreads real exports and only overrides what you list.

## 3. fetchCourseDetailData Name Collision Prevented Silent Bug
**Topics:** astro, ssr, refactoring

**Context:** Adding `fetchCourseTabData` to `src/lib/ssr/loaders/courses.ts`. Discovered an existing `fetchCourseDetailData` function with a completely different return type (returns a mock-data `Course` object with no enrollment context, not the real DB data needed by the tab pages).

**Learning:** Always grep for function names before creating new ones in loader files — the existing function had a similar name and could have been accidentally overwritten or confused. Using a distinctive name (`fetchCourseTabData` vs `fetchCourseDetailData`) makes the distinction clear in call sites.
