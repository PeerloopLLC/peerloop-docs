# Conv Learnings - 2026-04-03

## 1. Two Parallel Certification Paths with No Unified Visibility
**Topics:** certification, architecture, admin

**Context:** Walking the certification flow revealed that the Creator "Certify" button and the Teacher "Recommend" → Admin "Approve" path use different database tables

**Learning:** The creator path writes directly to `teacher_certifications` (no `certificates` record), while the recommend/approve path creates `certificates` records then syncs to `teacher_certifications`. The admin Certificate Management page only shows `certificates` table data, so creator-certified teachers are invisible to admins.

**Pattern:** When two paths lead to the same outcome (teacher certification), they should share a data model or at minimum have unified admin visibility.

---

## 2. Dashboard Attention Items Must Link to Actionable Destinations
**Topics:** ux, dashboard, navigation

**Context:** The unified dashboard shows "Certification recommendation: 1 student awaiting recommendation" but links to `/teaching/students` which has no recommend action

**Learning:** Attention items that tell users to act but link to pages without the relevant action create frustrating dead-ends. The correct destination would be the CourseEditor Teachers tab (`/creating/studio?course=X` → Teachers tab), but that's deep in the creator workspace, not the teacher workspace.

---

## 3. ConfirmModal Consistency for Significant Actions
**Topics:** ux, confirmation, certification

**Context:** The Certify button was a one-click action granting teaching privileges and earning capability, while the less impactful Cancel Session button already had a ConfirmModal

**Learning:** Actions that grant capabilities (teach, earn, access) deserve confirmation modals just as much as destructive actions. The pattern is already established in the codebase — Revoke uses ConfirmModal with danger variant; Certify now uses it with default variant.

---

## 4. Admin Page Auth Guard Gap
**Topics:** auth, admin, security

**Context:** Navigating to `/admin/certificates` as non-admin Mara Chen showed the full admin sidebar layout before the API returned "Access denied"

**Learning:** Admin pages render the admin layout for any authenticated user — the auth check happens at the API level, not the page level. The page-level guard should redirect non-admins before rendering the admin sidebar.

---

## 5. Denormalized Counters Require Sync Verification
**Topics:** database, communities, denormalization

**Context:** The Commons header showed "2 members" but the Members tab listed 3 (Admin, Mara, Alex)

**Learning:** `communities.member_count` is a denormalized counter updated via API triggers. When members are added through seed data or direct DB manipulation (e.g., system community auto-join during registration), the counter may not be updated. This is a data issue, not a code bug — but it affects the user experience.
