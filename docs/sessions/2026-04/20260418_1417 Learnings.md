# Conv Learnings - 2026-04-18

## 1. Per-H4 predicate engine replaces tier cascade
**Topics:** cc-workflow, skill-r-timecard-day2, skill-r-end2, tooling

**Context:** Conv 126 built `classifyItem()` as a 5-tier first-match-wins cascade routing each bullet to exactly one H4. Legacy commits with mixed concerns (e.g. a bullet mentioning both an API path and a doc file) got routed to whichever tier hit first and lost the other appearance.

**Learning:** Replacing "classification" (1-of-N routing) with "inclusion" (N-of-N, each H4 independently) is the right model. Each H4 owns an `include` predicate evaluated independently over every bullet; a bullet appears in every H4 whose predicate matches. Block Progress already did this for multi-Block commits â€” the refactor extends the same semantic to every bullet-rollup H4.

**Pattern:** `evalPredicate(pred, item, ctx) â†’ boolean` called O(bullets Ă— H4s) times. Predicate DSL closed set: `src`, `matchesRegex`, `textContainsAny`, `startsWithAny`, `docsMentionGt`/`Eq`/`Gte`, `testRelated`/`notTestRelated`, `isRoutine`, `commitFileMatchesPrefix`, `allCommitFilesUnder`, `flag`, `fallthrough`, `anyOf`/`allOf`. String values like `"reroute.apiMethodRe"` resolve to config field references at eval time.

## 2. Fallthrough as a combinator, not a tier
**Topics:** cc-workflow, skill-r-timecard-day2, tooling

**Context:** Work Effort is both the "home for workEffort-src bullets" AND the "catch-all for bullets that matched nothing". A single predicate couldn't express this without either being too greedy (eat everything) or too restrictive (only unmatched bullets).

**Learning:** Use a 2-pass engine. Pass 1 evaluates all H4 predicates; `{fallthrough: true}` deliberately returns false in Pass 1 so it never claims bullets during the independent-matching phase. Pass 2 looks at bullets that matched nothing in Pass 1 and routes them to the first H4 whose `include` (recursively) contains `fallthrough: true`. This lets Work Effort use `{anyOf: [{src: "workEffort"}, {fallthrough: true}]}` â€” workEffort-src bullets match via the first arm in Pass 1, and truly-unclaimed bullets get Work Effort via Pass 2.

**Pattern:** `predicateHasFallthrough(pred)` walks nested `anyOf`/`allOf` arrays looking for `fallthrough: true`. The engine finds the first H4 section whose predicate recursively contains a fallthrough marker and uses it as the safety net.

## 3. Named H5/H6 strategies decouple config from script
**Topics:** cc-workflow, skill-r-timecard-day2, tooling

**Context:** Different H4s need wildly different H5 grouping (doc filenames, infra labels, code path prefixes, em-dash splits, first-word, tag-based). Making each H5 strategy configurable via regex/template would bloat config; hard-coding them blocks portability.

**Learning:** Name the strategies in script (lookup tables `H5_STRATEGIES` / `H6_STRATEGIES`), reference by name in config per H4. Adding a new strategy is a one-time script edit; swapping which strategy an H4 uses is config-only; tuning a strategy's patterns (e.g. `dbBulletPrefixWords`) is config-only because all 8 strategies read from the existing `reroute.*` pattern library.

**Pattern:** 8 H5 strategies (`emDashOrFirstWord`, `firstWord`, `infraGroupLabels`, `docFilename`, `codePrefix`, `dbBulletPrefix`, `testingPath`, `tagOrUntagged`) + 1 H6 strategy (`testingSubdir`) cover all current H4 sections. `docFilename` is the only strategy that legitimately multiplies a single bullet into multiple H5 entries (one per doc mentioned).

## 4. v2 commit format hints `src`, doesn't bypass predicates
**Topics:** cc-workflow, skill-r-commit2, skill-r-end2, skill-r-timecard-day2

**Context:** Originally considered a v2 fast-path that skipped the predicate engine entirely when `Format: v2` was detected, reading bullets directly from H3 sections. That creates two code paths to maintain and muddles v1/v2 parity.

**Learning:** v2 H3 headers merely set each bullet's `src` metadata via titleâ†’id lookup. Predicates read `src` like any other attribute (`{src: "api"}`). v1 commits set `src` from the `API:` / `Doc:` / etc. tag lines. Same engine, same predicates, just richer metadata on v2. This means multi-H4 replication works identically for both formats â€” and a v2 bullet under `### Work Effort` mentioning `/api/foo` and `API-REFERENCE.md` still renders in Work Effort + API Changes + Doc Changes because the predicates catch the API path + doc mention regardless of which H3 the author placed it under.
