# Conv Decisions - 2026-04-06

## 1. PLATO Seed Data is Local-Dev Only
**Type:** Strategic
**Topics:** testing, deployment, plato

**Trigger:** User stated PLATO seed data is incompatible with SQL seed data and cannot be merged

**Options Considered:**
1. Merge PLATO and SQL seed data into compatible format
2. Keep PLATO local-only, staging uses SQL seeds ← Chosen
3. Have separate staging environments for each seed type

**Decision:** PLATO seed data is exclusively for local development. Staging uses only the SQL seed chain (core → dev → stripe → booking → feeds).

**Rationale:** PLATO and SQL seeds use different entity IDs and are incompatible. Staging should match production data shapes.

**Consequences:** Documented in migrations.md. `plato:seed:staging` script exists but should not be used.

## 2. parseBody Utility for Consistent JSON Error Handling
**Type:** Implementation
**Topics:** api, error-handling

**Trigger:** Audit found ~35 unguarded `request.json()` calls returning 500 on malformed JSON

**Options Considered:**
1. Add individual try/catch to each call site
2. Create shared `parseBody<T>()` utility ← Chosen
3. Use middleware/validation library (zod)

**Decision:** Created `src/lib/request.ts` with `parseBody<T>()` that throws `Response(400)` on parse failure.

**Rationale:** 35 call sites with identical pattern — a utility is appropriate. Throws Response which existing catch blocks handle via `instanceof Response` check.

**Consequences:** ~34 files updated. One exception: `reset-password.ts` intentionally avoids it for security.

> **Insight:** The `instanceof Response` pattern for propagating typed HTTP errors through catch blocks is elegant — it lets utilities throw the exact response the client should receive, while generic error handlers still produce 500s. This avoids the "error code enum" anti-pattern.

## 3. Dashboard Error Suppression: Log Don't Fail
**Type:** Implementation
**Topics:** api, observability

**Trigger:** Audit found dashboard endpoints using `.catch(() => [])` silently hiding DB failures

**Options Considered:**
1. Remove `.catch()` — let dashboard fail with 500 on any query failure
2. Add `console.error` logging but keep resilient pattern ← Chosen
3. Return partial success with warnings array in response

**Decision:** Keep the resilient `.catch(() => default)` pattern but add `console.error` so failures are visible in logs.

**Rationale:** Dashboard should render even if one of 12 queries fails. Showing partial data is better UX than a 500 error page. But failures must be visible for debugging.

**Consequences:** 16 catch handlers across 3 files now log errors.
