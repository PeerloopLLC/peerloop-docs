## 1. Use SSR loaders (Path B), not service binding (Path A)
**Type:** Strategic
**Topics:** cloudflare, astro

**Trigger:** [SF] discovered. CF Workers SSR self-fetch hits Assets layer ŌåÆ 404. Two viable paths: (A) add `[[services]] binding = "SELF"` and use `env.SELF.fetch()` to bypass assets, or (B) refactor SSR to call data-loader functions directly.

**Options Considered:**

2. Path B ŌĆö refactor to direct loader calls (~3-4 hr, eliminates round-trip) ŌåÉ Chosen

**Decision:** Path B.

**Rationale:** Per CLAUDE.md ┬¦Solution Quality: lean durable. Path B eliminates HTTP round-trip (~2├Ś SSR latency saved), removes cookie-forwarding hack, gives end-to-end type safety, ensures API and SSR can't drift, and survives any future CF routing model changes. The codebase already had the loaders pattern ŌĆö these 8 pages were just inconsistent.

**Consequences:** ~750 LOC of duplicated query logic deleted. New `src/lib/ssr/loaders/communities.ts` (~570 lines, 3 loader functions). API handlers reduced from 161/283/206 ŌåÆ ~50 lines each. All callers (SSR pages + API endpoints) share one implementation.

## 2. Extend existing SSR loaders convention (B1), not parallel services module (B2)
**Type:** Implementation
**Topics:** astro

**Trigger:** Discovered pre-existing `src/lib/ssr/loaders/` with loaders for about, courses, creators, home, static, teachers, verify ŌĆö but no `communities.ts`. Used `SSRDataError` (throws) with codes DB_UNAVAILABLE/QUERY_FAILED/NOT_FOUND/INVALID_PARAMS. Decision: follow that convention or introduce a new `src/lib/services/` module returning `{ ok, data } | { ok: false, status, error }` outcome unions.

**Options Considered:**
1. B1 ŌĆö extend existing loaders (add `communities.ts`, extend SSRDataError with FORBIDDEN/UNAUTHORIZED) ŌåÉ Chosen
2. B2 ŌĆö new `src/lib/services/` with outcome unions (parallel pattern)

**Decision:** B1 ŌĆö extend the existing convention.

**Rationale:** Per CLAUDE.md ┬¦Solution Quality + memory rule "preserve existing patterns where they work". Codebase consistency wins. Throwing `SSRDataError` with a typed code maps cleanly to both page-redirect logic (catch + check `e.code`) and HTTP-response logic (catch + use `e.httpStatus`). No second pattern to remember.

**Consequences:** Extended `SSRDataError` codes from 4 to 6 (added UNAUTHORIZED, FORBIDDEN). All SSR loaders now share one error model.

## 5. Switch API handlers from `getCurrentUserId` to `getSession` to preserve test mocks
**Type:** Implementation
**Topics:** auth, testing

**Trigger:** 7 test failures after API refactor. Tests mock `getSession`; my code used `getCurrentUserId` which internally calls real `getSession`, bypassing the mock.

**Options Considered:**
1. Update tests to mock `getCurrentUserId` instead
2. Switch handlers to call `getSession` directly ŌåÉ Chosen
3. Use `vi.spyOn` to patch module-internal calls

**Decision:** Option 2.

**Rationale:** The test mocking pattern is the established convention across the codebase. Changing the convention would require updating an unknown number of other test files, increasing the blast radius of a refactor that should be invisible to tests. Calling `getSession` directly matches the pre-refactor behaviour exactly.

**Consequences:** API handlers lost ~3 lines of brevity but tests pass green. Pages still use `getCurrentUserId` (no test coverage there, so no mock contract to preserve).
