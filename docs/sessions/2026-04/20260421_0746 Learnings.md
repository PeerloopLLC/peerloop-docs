# Conv Learnings - 2026-04-21

## 1. Partial unique indexes are the correct tool for "at most one open row" invariants
**Topics:** d1, testing

**Context:** [IO] — needed to guard `INSERT session_attendance` against duplicate `participant_joined` webhook deliveries without blocking legitimate rejoins (a user who joins → leaves → rejoins needs 2 rows).

**Learning:** Plain `UNIQUE(session_id, user_id)` would forbid rejoins. A **partial** unique index (`... WHERE left_at IS NULL`) applies the constraint only to *open* rows; once a row is closed (`left_at` set), it falls out of the index and a new open row is allowed. Combined with `INSERT OR IGNORE`, duplicate-open webhook deliveries become silent, atomic no-ops — race-free even with simultaneous deliveries.

**Pattern:**
```sql
CREATE UNIQUE INDEX idx_X_open_unique ON table(k1, k2) WHERE status_col IS NULL;
-- combined with:
INSERT OR IGNORE INTO table ...
```
SQLite supports the `WHERE` predicate on CREATE INDEX natively.

## 2. `COALESCE(col, ?)` in UPDATE is the clean pattern for backfill-without-overwrite
**Topics:** d1, testing

**Context:** [DF] — `completeSession` needs to populate `started_at` when it's null (dropped meeting-started webhook) but must never overwrite an existing value from earlier `room_started`.

**Learning:** `UPDATE ... SET started_at = COALESCE(started_at, ?)` in a single statement is clearer and atomic vs. read-modify-write in JS. The bound value acts as the fallback, not the override. Extending this to any nullable column makes "set if null" a one-line change. Avoids the classic RMW race window entirely.

## 3. Strict narrowing cascade beats parallel cron detectors
**Topics:** cloudflare, video, d1

**Context:** [CT] — 4 detection passes now run in `runSessionCleanup`: noShows → orphaned (NEW) → staleInProgress → reconcile.

**Learning:** Ordering them so each pass sees a strictly smaller candidate set eliminates double-work. Because `detectOrphanedParticipants` completes orphan sessions in-place, the subsequent `reconcileBBBSessions` sweep finds them as `status = 'completed'` and skips the BBB API call that would have been duplicated. The 15-min cron interval (not the 1-hour DB-backstop grace) now bounds real-world latency for one-sided-crash detection, because BBB-authoritative passes fire at `scheduled_end + 0`.

## 4. BBB webhook payload duration enables a meaningful backfill
**Topics:** video

**Context:** [DF] — three of five `completeSession` callers have no duration data; only the webhook does.

**Learning:** The webhook's `durationSeconds` field is the single authoritative signal for "how long did this session actually last." Threading it through as an optional parameter gives the webhook path a tight backfill (`ended_at − duration`) while other paths degrade gracefully to `scheduled_start`. Design principle: a fallback chain where the caller's best data wins, with a conservative default, scales cleanly to new callers.
