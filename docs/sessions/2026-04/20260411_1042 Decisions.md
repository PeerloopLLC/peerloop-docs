# Conv Decisions - 2026-04-11

## 1. Detect-and-prompt for date-based, silent union for conv-based
**Type:** Implementation
**Topics:** cc-workflow, skills, git

**Trigger:** Same HEAD-only bug surfaced in both /r-timecard-day (date-based) and /r-timecard (conv-based), but the right UX differed.

**Options Considered:**
1. Identical fix in both (detect-and-prompt everywhere)
2. Identical fix in both (silent `--all` everywhere)
3. ← Chosen: Different fixes based on query semantics — prompt for date, silent for conv

**Decision:** /r-timecard-day scans `refs/heads/*`, counts in-window commits per branch, displays a list, and prompts the user with 👉👉👉 if any non-HEAD branches have in-window commits (default include-all). /r-timecard conv mode uses `git log --branches --grep=...` + per-commit `git branch --contains` + silent de-dup. /r-commit unchanged (HEAD-only is correct for commit creation). /r-timecard count mode unchanged (HEAD-only is correct for "last N on current branch").

**Rationale:** Date queries are inherently ambiguous — a user can have multiple branches with legitimate in-window work, and some may be exploration that shouldn't bill. Prompting gives the user a decision point. Conv queries are unambiguous — one conv's work is a single entity identified by a unique number, so union-and-find is always correct.

**Consequences:**
- Today's Apr 10 timecard rerun would now produce the correct 10-commit output automatically.
- Future `/r-timecard conv=100` calls while HEAD is on `main` will still find the `jfg-dev-10up` commits.
- /r-timecard Mode 1 (cNdN) remains a latent footgun if user forgets which branch they're on — deliberately not fixed because the semantic is correct.

## 2. Proceed past /r-start dirty-repo guard when dirty file is RESUME-STATE.md
**Type:** Implementation
**Topics:** cc-workflow, dual-repo

**Trigger:** /r-start halted on ` M RESUME-STATE.md` after I'd appended 2 tasks as insurance. Normally the guard exists to prevent losing uncommitted work.

**Options Considered:**
1. Commit first via /r-commit — creates a commit that Step 7 immediately orphans by deleting the same file
2. ← Chosen: Proceed anyway — Step 7 consumes RESUME-STATE.md by design; the deletion is the intended end-state
3. Stash, run /r-start, pop — pointless because Step 7 deletes the file
4. Discard — loses the appended tasks

**Decision:** When the only dirty file is RESUME-STATE.md and its dirtiness is an edit (not an unintended change), proceed past the guard. Step 7's transfer-and-delete is the canonical consumption path.

**Rationale:** The dirty-repo guard exists to prevent losing uncommitted work. In this specific case the "uncommitted work" IS the thing /r-start is supposed to consume. Committing first would add noise to git history (a commit that adds lines that a subsequent commit deletes).

**Consequences:** RESUME-STATE.md is now deleted, and the deletion is staged as part of the Conv 103 /r-end commit. The two tasks safely transferred into TodoWrite and persist via this conv's pending-task snapshot.

## 3. Leave /r-timecard Mode 1 (cNdN) HEAD-only
**Type:** Implementation
**Topics:** cc-workflow, skills

**Trigger:** Auditing /r-timecard for the same HEAD-only blind spot as /r-timecard-day, noticed Mode 1 also reads HEAD only.

**Options Considered:**
1. Fix Mode 1 the same way as Mode 2 (scan branches, add detect-and-prompt)
2. ← Chosen: Leave Mode 1 alone; the HEAD-only semantic is correct

**Decision:** Mode 1 (`cNdN`) stays HEAD-only. Only Mode 2 (`conv=NNN`) gets the branch-scan fix.

**Rationale:** `cNdN` means "last N commits on the branch I'm on." If the user just committed, HEAD is correct. If they switched branches before running the skill, that's a user error, not a skill bug. Fixing this would redefine the semantic and likely surprise users who depend on the current behavior.

**Consequences:** Latent footgun: user on wrong branch runs `/r-timecard c3d2`, gets 3 unrelated commits silently. User should own this trade-off — not me deciding on their behalf. Explicitly documented in the rules bullet for the Branch column ("count mode is HEAD-only by design").
