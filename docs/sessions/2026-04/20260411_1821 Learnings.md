# Conv Learnings - 2026-04-11

## 1. "Half-wired" can mean "superseded", not "missing"
**Topics:** react, testing, refactoring

**Context:** The Conv 104 resume-state flagged two half-wired features as fix-or-defer for Conv 105. On close reading both turned out to be legacy state from a previous UX era (FormModal lockout, showToast migration), not missing UI. The correct fix was deletion, not building.

**Learning:** When you see an underscored `_var` next to a setter that's never called, the question isn't "how do we wire this up?" — it's "what replaced this, and why wasn't it cleaned up?" Grep for JSX usage of the raw variable name before assuming UI work is needed.

**Pattern:** For each half-wired state pair, run these three checks before deciding to build: (1) `Grep \{varName\}` to confirm zero JSX consumers; (2) trace siblings to see if a newer mechanism (toast, modal, notification) provides the same feedback; (3) check for a backdrop / blocking element that makes the original guard redundant.

## 2. Silent-failure sites hide inside "dead code" removals
**Topics:** react, testing, refactoring

**Context:** While removing `_error` / `setError(null)` pairs in TeachersTab, 2 call sites turned out to be real error captures (`setError(err.message)` in fetchData and handleToggleActive) where the state was never rendered — a pre-existing silent failure. Pure deletion would have preserved the silent failure under cover of "just cleaning up".

**Learning:** Before deleting any `setX(err...)` call, check whether `{x}` is actually rendered anywhere. If it isn't, the code was already broken — the removal has to include a replacement feedback path (here: showToast) or the bug persists invisibly.

**Pattern:** As a /w-codecheck idea: detect "error-captured-never-rendered" — `setError(...)` in a catch block where the corresponding state variable has zero JSX consumers. Listed for [HD2] + [OD] as a future codecheck lint.
