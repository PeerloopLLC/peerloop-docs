# Conv Decisions - 2026-04-10

## 1. Keep `locals` parameter on helpers (Fork 2 = X)
**Type:** Implementation
**Topics:** refactor, architecture, testing

**Trigger:** After switching helpers to `import { env } from 'cloudflare:workers'`, the `locals` parameter is nominally dead for production env access, but the `__testEnv` injection path still uses it.

**Options Considered:**
1. **Y** — Drop `_locals` from all helpers now; sweep ~130 call sites as part of Phase 2a
2. **X** — Keep the parameter for Phase 2a; defer the sweep to a dedicated follow-up commit ← Chosen

**Decision:** Keep the parameter throughout Phase 2a. Schedule a follow-up mechanical sweep as a separate commit on the same branch.

**Rationale:** Phase 2a commit stays tight and reviewable (29 files, not 150+). Bisection is clean — the adapter bump and the signature sweep live in different commits. The `__testEnv` path means the parameter is narrowly used, not dead weight. Durability still served because follow-up cleanup has better blame history when isolated.

**Consequences:** 39+ tests using `createAPIContext({ db: ... })` need no changes. A new follow-up task tracks the eventual `_locals` drop.

## 2. Test path via vitest alias + mock file (Fork 1 = A)
**Type:** Implementation
**Topics:** testing, vitest, virtual-modules

**Trigger:** `cloudflare:workers` is a virtual module provided by `@cloudflare/vite-plugin`, which vitest does not load. Without intervention, `import { env } from 'cloudflare:workers'` in the helpers cannot resolve in tests.

**Options Considered:**
1. **A** — `vitest.config.ts` alias → `tests/helpers/mock-cloudflare-workers.ts` (Proxy over `process.env`) ← Chosen
2. **B** — `vi.mock('cloudflare:workers', () => ({ env: ... }))` in `vitest.setup.ts`
3. **C** — Dynamic `import('cloudflare:workers').catch(() => null)` inside helpers, cached at module load

**Decision:** Option A — alias + dedicated mock file.

**Rationale:** Matches the existing virtual-module pattern already used in `vitest.config.ts` for `astro:transitions/client` and `astro:middleware`. No `vi.mock` hoisting quirks. Helpers stay synchronous. Single explicit resolution point a newcomer can grep for. Zero impact on production code paths.

**Consequences:** One new file (`tests/helpers/mock-cloudflare-workers.ts`, ~25 lines). One new alias entry in `vitest.config.ts`. No changes to helper signatures.

## 3. Use `__testEnv` as the test-only injection slot on `App.Locals`
**Type:** Implementation
**Topics:** testing, typescript, architecture

**Trigger:** `createAPIContext` previously built `locals.runtime.env = { JWT_SECRET, DB, ... }` to inject per-test env values. Adapter 13's throwing proxy breaks that pattern in production, but tests still need per-context injection (39+ tests use `{ db: ... }`, 10 use `env: { STORAGE: ... }`).

**Options Considered:**
1. Migrate all 39+ tests to `vi.stubEnv` + `vi.mock('@/lib/db')` scopes
2. Global `vi.mock('@/lib/db')` + `vi.mock('@/lib/r2')` in `vitest.setup.ts` reading from a module-scoped test DB ref
3. Add `__testEnv?: Record<string, unknown>` to `App.Locals`; helpers check it before falling through to `cloudflare:workers` ← Chosen

**Decision:** Option 3 — `__testEnv` as a dedicated test-only slot on `App.Locals`, checked first by each helper.

**Rationale:** Preserves the `createAPIContext({ db, env })` ergonomics that 39+ tests depend on — no mechanical test sweep needed. The test-awareness added to helpers is minimal: a single `locals?.__testEnv?.[key]` check at the top of each function, with zero cost in production. Alternatives 1 and 2 would have introduced significant global state or per-test boilerplate.

**Consequences:** Three helpers, api-test-helper, middleware.test.ts mock, and 2 r2 mocks updated. A new ESLint rule (task [LE]) should eventually forbid application code from setting `__testEnv`.

## 4. Keep `vite.ssr.external` as-is rather than migrate to Vite 7 Environments API
**Type:** Implementation
**Topics:** vite, config

**Trigger:** Vite 7 introduces a new per-environment config shape (`environments.ssr.resolve.external`) marked `@experimental` in Vite's types.

**Options Considered:**
1. Migrate to `environments.ssr.resolve.external` during Phase 2a
2. Keep `vite.ssr.external` as-is until Vite stabilizes the new API ← Chosen

**Decision:** Leave `vite.ssr.external: ['stream', 'crypto', 'path', 'node:buffer', 'node:stream']` unchanged.

**Rationale:** Vite 7's `SSROptions` interface still declares `external?: string[] | true` — the old path is not deprecated, just joined by a new experimental alternative. Phase 2a's build succeeded with the old shape. Migrating to an experimental API that may still shift is the opposite of durable.

**Consequences:** Config remains as it was. Revisit when Vite marks the new API stable.
