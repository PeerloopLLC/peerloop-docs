## 2. Separate "Active" vs "Proposed" Endpoints in DB-API.md §Communities
**Type:** Implementation
**Topics:** docs-infra

**Trigger:** DSA audit — existing §Communities had proposed endpoints interleaved with real ones, marked `*(proposed — not implemented)*` inline. Hard to scan.

**Options Considered:**
1. Keep interleaved inline markers
2. Separate into two subsections: Active (implemented) and Proposed (Block 2+) ← **Chosen**

**Decision:** Option 2. §Communities now has all 15 implemented endpoints in the main body, and `### Proposed (Block 2+ — not yet implemented)` as a final subsection listing 3 aspirational ones (`/feed`, `/posts`, `/invite`).

**Rationale:** Readers typically scan for what exists now; the interleaved layout forced them to re-read the `*(proposed)*` tag on every entry to filter. Separating the proposed into a trailing subsection makes "what's live" the default view.

**Consequences:** §Communities now renders as 15 endpoints followed by a clearly-bounded proposal section. Pattern could be applied to other DB-API.md sections that carry aspirational entries.
