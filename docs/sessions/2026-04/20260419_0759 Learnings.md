# Conv Learnings - 2026-04-19

## 1. bash `${var%%|*}` truncates at literal `|` — backslash escapes are cosmetic
**Topics:** cc-workflow, docs-infra

**Context:** Migrating `tech-doc-sweep.sh` rule format from bash array to JSON-driven. The original rule entries looked like `"webhook.*bbb\|video\|bigblue\|plugnmeet|bigbluebutton bbb video plugnmeet"` with a single `|` meant to be the field separator and `\|` meant to be the grep-E alternation.

**Learning:** Bash parameter-expansion patterns (`%%`, `##`) use GLOB matching, not regex. In glob patterns, `\|` is two literal characters — the backslash has no special meaning at the glob layer and the `|` still matches as an ordinary character. `${rule%%|*}` therefore truncates at the FIRST `|` — the one inside `\|video`, not the "intended" field separator `|bigbluebutton...`. The backslashes were cosmetic noise that never affected what `%%|*` matched.

**Pattern:** When migrating pipe-delimited bash configuration to JSON, either (a) verify empirically what the old shell parser was actually doing, or (b) switch the reader to a format with an unambiguous separator (tab, NUL). Don't assume the original author's intent matches the shell's behavior. The full-alternation "bug" had been silently suppressing drift signal for 60+ convs.

## 2. Migration surfaces dormant drift as a side effect — this is a feature, not a regression
**Topics:** cc-workflow, docs-infra

**Context:** Strategy doc §4 Phase 2 promised "same audits, same reports — no behavioral change." The bug fix in tech-doc-sweep.sh surfaced 9 previously-suppressed doc flags from the same HEAD~5 diff.

**Learning:** When a migration corrects a latent bug, the "no behavioral change" contract has to give way. The right framing is: the tool's documented intent (full alternation matching in the JSON `codePattern`) was already correct; the old bash runtime was quietly under-enforcing it. Migration realigns runtime to spec. Downstream: expect a one-time bump in flag volume on first post-migration run; triage is NOT a new `/r-end` task, it's the expected debt being paid.

## 3. Dual-repo script consolidation makes orphans visible
**Topics:** cc-workflow, dual-repo, docs-infra

**Context:** `r-end/scripts/` and `r-end2/scripts/` each had copies of `sync-gaps.sh` + `tech-doc-sweep.sh` + `route-mapping.txt`. The `r-end2` SKILL.md referenced `r-end/scripts/` paths, making the `r-end2/scripts/` copies orphans — nobody called them, and they silently drifted from the authoritative `r-end/scripts/` copies (in this case they happened to be identical, but nothing enforced that).

**Learning:** When two sibling skills share generated infrastructure, the choice is (a) explicit shim (one skill's scripts delegate to the other's), or (b) hoist to a shared parent location. Leaving duplicated copies in place is the worst of both worlds — maintainers can't tell which copy is authoritative, and fixes land in one place while the other rots. The `.claude/scripts/` consolidation makes this discoverable: `find .claude -name '*.sh'` now shows one canonical copy per tool.
