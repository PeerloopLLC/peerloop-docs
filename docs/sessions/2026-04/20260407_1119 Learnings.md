# Conv Learnings - 2026-04-07

## 1. Dev Server Stability During Browser Testing
**Topics:** testing, dev-environment

**Context:** Dev server crashed twice during Chrome MCP browser walkthrough — once when login was attempted, once when page was reloaded.

**Learning:** When launching the dev server from Claude Code for browser testing sessions, use `nohup npm run dev > /tmp/peerloop-dev.log 2>&1 &` instead of the Bash tool's `run_in_background` parameter, which may terminate the process during cleanup.

**Pattern:** `nohup npm run dev > /tmp/peerloop-dev.log 2>&1 &`

---

## 2. Chrome MCP Element Interaction Reliability
**Topics:** testing, chrome-mcp

**Context:** Coordinate-based clicks on the "Sign in" link in the modal missed the target. Using `read_page` to get element refs and clicking by ref was more reliable.

**Learning:** For interactive elements (buttons, links, form fields), prefer `read_page` with `filter: "interactive"` to get refs, then click/input by ref. Reserve coordinate-based clicks for visual elements without semantic refs.

---

## 3. Chrome MCP and Playwright Are Complementary
**Topics:** testing, architecture

**Context:** User asked for comparison of Chrome MCP vs Playwright testing approaches.

**Learning:** They fill different quadrants: Playwright = repeatable+scripted (CI regression), Chrome MCP = one-shot+exploratory (ad-hoc QA, demos). Neither replaces the other. The interesting gap is "repeatable+exploratory" — potentially filled by using MCP to generate Playwright tests.
