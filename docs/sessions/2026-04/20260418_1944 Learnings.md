# Conv Learnings - 2026-04-18

## 1. config.json has hidden path drift
**Topics:** docs-infra, cc-workflow

**Context:** While investigating whether config.json's doc whitelists could become a shared registry, discovered `paths.vendorDocs: "docs/vendors/"` and `paths.architectureDocs: "docs/architecture/"` pointing at folders that don't exist. Grep found zero consumers. `DOCS-REORG-MAP.md` documents the first half of the folder reshuffle (Session 362: `docs/tech/` → `docs/vendors/`) but not the later merge (`docs/vendors/` → `docs/reference/`). The config was never updated.

**Learning:** Check config.json path fields against disk before relying on them. Config paths can go stale without any runtime error — only `paths.claudeOffload` has an active consumer (`r-prune-claude`); the rest are aspirational documentation.

**Pattern:** Any future tool that consumes a `paths.*` entry should either (a) assert the folder exists at startup, or (b) be documented as canonical so future reorgs update it.

## 2. Three drift tools, three hardcoded lists — classic "registry needed" smell
**Topics:** docs-infra, cc-workflow

**Context:** DOC-SYNC-STRATEGY Phase 1 investigation found `sync-gaps.sh`, `tech-doc-sweep.sh`, and `/w-sync-docs` each maintaining independent hardcoded lists of docs-of-interest — shared basenames (sync-gaps), keyword rules (tech-doc-sweep), audit sections (w-sync-docs). Overlap on API, CLI, and test docs without cross-check. Adding a new tech vendor or doc family means editing three places.

**Learning:** When three sibling tools each carry "the same kind of list" internally, that's the signal to extract a registry. The migration target is `.claude/config.json.docsRegistry` — pattern-based groups with `category` / `sourceOfTruth` / `checks` / `consumers` fields.

**Pattern:** Phase-gated migration: author strategy + schema first (Phase 1), migrate tools 1:1 preserving behavior (Phase 2), then add new capabilities (Phase 3). Don't mix strategy and implementation in the same conv — the strategy work alone is substantial enough.

## 3. Load-bearing legacy docs hide in plain sight
**Topics:** docs-infra

**Context:** The retirement audit of 9 `_`-prefix legacy docs was straightforward except for `_COMPONENTS.md` (1,761 lines). Quick `grep` across `.claude/skills/` surfaced 2 active references in `/w-add-client-note` SKILL.md — it's actively queried when processing client change requests. CLAUDE.md line 499 also points to it for "UI components" research.

**Learning:** Before proposing any doc for retirement, grep across `.claude/skills/**` (not just `docs/**`) and the top-level CLAUDE.md. Skills are invisible consumers that don't show up in casual content searches.

**Pattern:** `Grep --path=.claude/skills {filename}` and `Grep {filename} CLAUDE.md` should be part of every retirement audit.

## 4. "Or"-tail sentences read as yes/no
**Topics:** cc-workflow

**Context:** Asked "Want to review the strategy doc, or would you rather handle the retirement decisions first?" User replied "yes" — ambiguous because the long "or"-tail structure reads as a yes/no question. User corrected: "These long sentences with 'or' far inside look like Yes/no."

**Learning:** Present A/B or A/B/C options as a numbered list on separate lines, then end with a short pointing question that references the numbers. Never embed the second option at the tail of a long sentence.

**Pattern:** Saved as `feedback_option_phrasing.md` memory. Example correction in the memory file.
