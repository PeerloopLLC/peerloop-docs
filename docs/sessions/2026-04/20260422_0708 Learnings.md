# Conv Learnings - 2026-04-22

## 1. Verify new heuristics on their canonical motivating case before shipping
**Topics:** cc-workflow, docs-infra

**Context:** For the `/w-sync-skills` divergence gate, an initial single-signal heuristic — Jaccard similarity on H2/H3 section headers, threshold < 0.70 → DIVERGED — looked clean and matched the feedback memo's "~30% structural divergence" language. Before committing, the heuristic was run on the exact case cited in the memo — `r-end` (Conv 140 DIVERGED). Result: Jaccard = 1.000. The canonical DIVERGED case would have been classified as "portable" by the heuristic.

**Learning:** Almost shipped a heuristic that would have failed on the *one case it was explicitly designed to catch*. Headers can stay stable while content inside headers forks — `r-end` had identical section skeletons in both repos but 30% of lines differed. The memo hadn't distinguished structural-fork from content-fork; the initial design conflated them.

**Pattern:** When designing a threshold or gate from qualitative guidance ("~30% divergence"), run it against the specific case(s) cited in the guidance *before* committing the design. If the heuristic fails on its motivating case, the threshold is wrong OR the signal is incomplete. In this instance the signal was incomplete — needed a second signal (post-normalization line-diff ratio). Calibration data across all 13 exact-match skills made the two-signal design empirically grounded rather than first-principles.

## 2. Triage IS the durable product when volume exceeds one-conv capacity
**Topics:** cc-workflow, docs-infra

**Context:** [LE-TRIAGE] presented 31 warnings across 26 files, estimated 3-4 hours of work if fixes landed cleanly. User's options were A (fix Cat 1+2 now, 5 warnings), B (fix Cat 1+2+3 now, 19 warnings), C (commit the triage doc only, no fixes). User chose C.

**Learning:** The durable move was classifying the 31 warnings by risk (7 categories), file, and fix pattern — then staging execution as 5 phases with natural commit boundaries. The resulting PLAN.md block carries enough structure that a future conv can pick up Phase 1 cold and execute without re-deriving any analysis. Compare to Option B: 19 fixes in one conv would generate a diff no reviewer would audit line-by-line, and the remaining 12 harder warnings would still need the triage doc anyway — effectively doing the analysis twice.

**Pattern:** When a cleanup/refactor task exceeds one-conv capacity (>~5-10 non-trivial changes in unrelated files), treat the triage doc as the primary deliverable. Structure it risk-sorted, with per-category fix patterns, file lists, phase boundaries, and an explicit baseline assertion. The fixes themselves become a series of follow-on convs that each commit one phase. The triage earns its keep many times over via reduced re-analysis and clean reviewable phase diffs.

## 3. Compound "or"-tail questions read as yes/no even when numbered in prose
**Topics:** cc-workflow

**Context:** This conv, compound A/B/C questions were asked of the user twice ([PC]+[SY] transition: "Continue now, save for its own conv, or stop here?") and each time the user replied "yes" meaning option 1. The existing `feedback_option_phrasing.md` (Conv 132 origin) said "use numbered list instead of or-tail" but the recurrence was written as "Continue with [LE-TRIAGE], save for its own conv, or stop here?" — numbered mentally but or-tail on the page.

**Learning:** The existing rule was insufficiently specific. User prescribed A)/B)/C) labeled format — the visual label is what signals "this is not a yes/no question" at a glance. Numbered-in-prose ("1, 2, or 3?") still reads as yes/no because the "or" triggers the short-circuit read. The user has had to correct this at least twice (Conv 132, Conv 147).

**Pattern:** For any question where the answer set is larger than {yes, no} — including 2-option binary choices that aren't literal yes/no — use A)/B)/C) labels, one per line, with a short 👉 pointing question referencing the letters. Never write "X, Y, or Z?" sentences, regardless of numbering. Rule strengthened in `memory/feedback_option_phrasing.md` with explicit wrong-form examples including numbered-in-prose.
