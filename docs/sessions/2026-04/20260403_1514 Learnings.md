# Conv Learnings - 2026-04-03

## 1. Composable Battle-Tested Primitives Produce Reliable First-Run Compositions
**Topics:** plato, testing, architecture

**Context:** The `seed-dev` scenario was written speculatively with 53 API steps + 48 SqlTopUp + 44 verifications but had never been executed.

**Learning:** It passed first try because it composes the same step primitives that were battle-tested through 16 STUMBLE-AUDIT conversations (Conv 067-082). The accumulation model means each step builds on the previous one's DB state through real API calls, so consistency is guaranteed by construction. Composable, well-tested primitives produce reliable compositions even on first run.

## 2. Password Validation Drift Creates Silent Seed Failures
**Topics:** auth, seed-data, validation

**Context:** Seed SQL said "dev123" but `dev123` fails the app's own password validation (8+ chars, uppercase required). Hashes were generated for `dev123` before validation existed.

**Learning:** When validation rules change, seed data that bypasses the API (raw SQL INSERT) can silently become invalid. PLATO seed avoids this because it goes through the register endpoint, which enforces current validation. This is a concrete example of why API-driven seeding is more durable than SQL INSERT seeding.

## 3. PLATO Snapshot to Remote D1 via sqlite3 .dump
**Topics:** plato, d1, staging, deployment

**Context:** PLATO snapshots are local `.sqlite` files. Remote D1 requires SQL via `wrangler d1 execute --file`.

**Learning:** `sqlite3 .dump` exports the snapshot as clean SQL (~146KB for the full seed). After filtering PRAGMA/transaction wrappers, this can be applied directly to remote D1 via wrangler. The dump includes CREATE TABLE statements, so it needs a clean DB (reset first, no separate migration step). Must mark migrations as applied in `d1_migrations` afterward so wrangler doesn't try to re-apply schema.

## 4. testNamePattern Specificity Prevents Accidental Double Execution
**Topics:** testing, vitest, plato

**Context:** `--testNamePattern="seed-dev"` matched both the standalone scenario test and the dynamic instance test, causing the scenario to run twice.

**Learning:** Vitest `--testNamePattern` matches against the full concatenated describe+it name. Using a more specific pattern (`"run seed-dev"` instead of `"seed-dev"`) targets only the intended test block. When a scenario has both a standalone test and an instance-based test, the pattern must distinguish between them.

---

## 5. Layout-Level Auth Guards Are Secure by Default
**Topics:** auth, astro, admin

**Context:** 13 admin pages had no auth check — non-admin users saw the admin layout before API calls rejected.

**Learning:** Putting the auth guard in `AdminLayout.astro` (the shared layout) creates a single enforcement point. Any future admin page using this layout gets admin-only access automatically. Three-layer check: no JWT secret -> login, no session -> login with redirect, session but not admin -> redirect to homepage.
