## 3. Split adapter 13 bump: centralize env access first, bump second (Option A)
**Type:** Strategic
**Topics:** refactoring, cloudflare, astro, durability

**Trigger:** `@astrojs/cloudflare` v13 removes `locals.runtime.env.*` entirely. 101 files access this directly. Bumping the adapter without a plan would require simultaneous change across 100+ files.

**Options Considered:**
1. Option A — Centralize env access through helpers on Astro 5 first (independently valuable), then bump adapter in a small follow-up commit ← Chosen
2. Option B — Bulldoze: bump + refactor in one massive commit
3. Option C — Defer Astro 6 entirely, schedule dedicated ASTRO-6-MIGRATION block

**Decision:** Option A. The centralization refactor is valuable on its own merit (consistency, dedup, single source of truth for env access). Once complete, the adapter bump is a small change to 4-5 helper files. Two small atomic commits > one massive commit.

**Rationale:** Per the durability principle, each step is independently verifiable and reversible. If Astro 6 turns out to be unstable later, the centralization stays as net-positive. Option B entangles the version bump with the pattern change so any failure is ambiguous. Option C is valid but postpones a refactor that's beneficial regardless.

**Consequences:** Committed `5284708` — 106 files changed (5 helper files, 2 config files, 1 types file, ~95 call sites, 2 test files). Baseline preserved: tsc 18 errors, tests 6399/6399. Next conv picks up the actual adapter 13 bump as a small follow-up.

> **Insight:** "Preparatory refactor" is a high-leverage pattern. When facing any breaking upstream change that touches many call sites, first ask: "Can I reshape my code to absorb this change in one place?" If yes, that refactor is worth doing even if you later abandon the upgrade.

## 4. `/r-commit` mid-conv is acceptable when commits are strategic snapshots
**Type:** Implementation
**Topics:** cc-workflow, conv-lifecycle

**Trigger:** Memory said "Always use /r-end to commit, never /r-commit directly". But user wanted to snapshot Phase 1 without ending the conversation to continue into Phase 2.

**Options Considered:**
1. Run `/r-end` now, lose context, restart for Phase 2
2. Run `/r-commit` as a strategic mid-conv snapshot ← Chosen

**Decision:** Memory rule applies to isolated commits. For strategic snapshots within a multi-phase work block, `/r-commit` is appropriate because the conversation continues into the next phase. User stated: "the commit is strategic not isolated in affect. The goal is a complete upgrade of packages and we'll do that commit with /r-end when we get there."

**Rationale:** The `feedback_always_r_end.md` memory exists to prevent overlapped session docs. A mid-conv `/r-commit` doesn't create session docs — it only commits code. The end-of-conv `/r-end` still owns session doc creation. The rules don't conflict.

**Consequences:** Three commits this conv: `cbaf250` + `0c7e4c4` (Phase 1, via `/r-commit`), `5284708` (centralization, direct commit), then the `/r-end` commit with session docs at the end. Memory rule remains valid for the default case; this is a recognized exception.
