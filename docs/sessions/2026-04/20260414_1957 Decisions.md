# Conv Decisions - 2026-04-14

## 1. COMMUNITY-RESOURCES schema editing strategy — direct edit
**Type:** Implementation
**Topics:** d1

**Trigger:** Client bug CB3 exposed schema asymmetry between `community_resources` and `session_resources`. Needed to add r2_key/external_url/size_bytes/mime_type.

**Options Considered:**
1. Add a new `0003_community_resources_r2.sql` migration (safer if already in prod)
2. Edit `migrations/0001_schema.sql` directly ← Chosen
3. Keep single `url` column with convention-based interpretation

**Decision:** Edit `0001_schema.sql` directly. Align `type` CHECK values with session_resources (`document/image/audio/video_link/other`). Replace `url NOT NULL` + `file_size` with `r2_key` + `external_url` + `size_bytes` + `mime_type` nullable columns.

**Rationale:** Pre-launch project (per CLAUDE.md). Direct edit keeps migration history clean. Schema asymmetry was itself the bug; removing it removes the need for workaround code.

**Consequences:** Local D1 reset + migrate cleanly. All 6 SSR Astro callers + CommunityTabs component + tests updated for new type enum. Staging will need a reset+migrate on next deploy.

## 2. COMMUNITY-RESOURCES upload authorization — creator + platform admin only
**Type:** Implementation
**Topics:** auth

**Trigger:** MVP scoping — who can upload community resources?

**Options Considered:**
1. Creator + Moderators + Teachers (anyone who can post)
2. Creator + Moderators
3. Creator + Platform Admin only ← Chosen
4. Any community member

**Decision:** Uploads restricted to the community's creator (`communities.creator_id === userId`) OR platform admin (`users.is_admin = 1`). Moderators, teachers, regular members cannot upload resources in MVP.

**Rationale:** Resources are reference material defining the community. Moderators moderate content; they don't author it. Starting strict is reversible; starting loose is hard to tighten.

**Consequences:** Narrower capability matrix. Moderator-upload capability can be added later if client requests. Community-members dropdown has a `canUploadResources` prop — it resolves true only for creator/admin.

## 3. COMMUNITY-RESOURCES download authorization — any authenticated member
**Type:** Implementation
**Topics:** auth

**Trigger:** "Must-be-a-community-member-to-see-community-files" rule from user.

**Options Considered:**
1. Public community → anyone (even unauthenticated); private → members
2. Any authenticated community member, public/private irrelevant ← Chosen
3. Same as uploads (creator + admin only) — too restrictive

**Decision:** Download requires authentication + membership. Creator and platform admin always have access regardless of membership row. Public vs private community flag does NOT affect download auth — to download, you must join.

**Rationale:** Matches user's stated gate ("must be a community member"). Public/private is about discoverability and posting, not resource access.

**Consequences:** Anonymous visitors browsing a public community's Resources tab will see the list but get 401 on download clicks. They'll need to join first; the "join" conversion becomes the gate.

## 4. CB2 calendar back-nav — remove guard entirely (unbounded)
**Type:** Implementation
**Topics:** astro

**Trigger:** Client reported "calendar won't go back past May, it is currently April." Bug: prev-month guard compared day+time.

**Options Considered:**
1. Fix the guard to compare month-starts (keep "can't navigate to past months" invariant)
2. Fix the guard AND add `disabled` state on boundary (better UX)
3. Remove the guard entirely — unbounded back-nav ← Chosen
4. Hybrid

**Decision:** Removed the `if (prev >= new Date())` guard. Back-nav is now unbounded, matching the existing unbounded next-button and teacher's `AvailabilityCalendar`. Added `aria-label` to both buttons as a free accessibility fix.

**Rationale:** Three calendars in the codebase; two were already unbounded. The constraint was the outlier, not correct behavior. The per-day cells already `disabled` on `isPast`, which is the real business rule — no-op navigation was never the right UX.

**Consequences:** Users can browse arbitrary past months. Cells still block past-date selection. Two follow-ups filed: `[BKC-NEXT]` (should next have an upper bound?) and `[BKC-FETCH]` (4-week fetch horizon UX gap when paging forward).
