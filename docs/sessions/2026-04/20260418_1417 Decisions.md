# Conv Decisions - 2026-04-18

## 1. Keep `/r-end` + `/r-commit` + `/r-timecard-day` alongside v2 forks
**Type:** Strategic
**Topics:** cc-workflow, skill-r-end2, skill-r-commit2, tooling

**Trigger:** Need for v2 commit format couldn't retrofit history; whether to replace or coexist was open.

**Options Considered:**
1. Replace v1 skills (simpler, fewer skills)
2. Coexist with pure-additive v2 forks ← Chosen
3. Add v2 as a mode flag on the original skills

**Decision:** Fork `/r-end` → `/r-end2` and `/r-commit` → `/r-commit2` as separate skills. Keep v1 skills untouched.

**Rationale:** Zero risk to existing workflow; user adopts v2 on their own timeline; rollback is trivial (stop using v2 skills). The refactored `timecard-day.js` handles both formats through the same predicate engine, so no timecard-side branching.

**Consequences:** Four conv-lifecycle skills instead of two. Can retire v1 later after v2 proven stable.

## 2. `Format: v2` trailer as the detection marker, not a date cutoff
**Type:** Implementation
**Topics:** cc-workflow, skill-r-end2, skill-r-commit2, skill-r-timecard-day2

**Trigger:** Parser needs to distinguish v1 from v2 commits.

**Options Considered:**
1. Date cutoff (all commits after 2026-04-18 are v2)
2. `Format: v2` trailer present/absent ← Chosen
3. Detect v2 via presence of `### ` H3 headers in body

**Decision:** Explicit `Format: v2` trailer emitted by `/r-commit2` and `/r-end2`; absent on v1 commits.

**Rationale:** Date cutoffs drift across timezones and break on backfilled commits. Heuristic detection via `###` presence is fragile (some commits might legitimately contain them in bullet text). A dedicated trailer is cheap, unambiguous, and version-extensible (future `Format: v3` needs no new logic).

**Consequences:** v1 commits go through the predicate engine's text heuristics; v2 commits set `src` precisely from H3 headers and flow through the same engine. One engine, two metadata sources.

## 3. `h4Sections[].title` is the literal header string in both commit body and timecard
**Type:** Implementation
**Topics:** cc-workflow, skill-r-commit2, skill-r-end2, skill-r-timecard-day2

**Trigger:** V2 commits need to emit `### Section Name` headers that the parser can map back to H4 IDs.

**Options Considered:**
1. Separate commit-header strings vs timecard-title strings in config
2. Use the H4 `id` as the commit-body header (e.g. `### userFacing`)
3. Use `title` as the literal string in both ← Chosen

**Decision:** The `title` field in each `h4Sections[]` entry IS the literal `### Title` string emitted in v2 commit bodies AND the `#### Title` string rendered in the timecard. One string, two uses.

**Rationale:** Portability — a sibling project can rename "DB Changes" → "Schema Changes" in one config field and have both the commit authors and the parser pick it up. No duplication, no sync risk.

**Consequences:** Renaming a section requires a single config edit; commit authors write whatever is in `title`; the parser reverse-lookups title→id when parsing v2 bodies.

## 4. Workflow fallthrough lives in a combinator, evaluated in a 2nd pass
**Type:** Implementation
**Topics:** cc-workflow, skill-r-timecard-day2

**Trigger:** A bullet with `src=workEffort` that also matches API + Doc predicates should appear in ALL three H4s, not just Work Effort. Initial Work Effort predicate `{fallthrough: true}` excluded such bullets.

**Options Considered:**
1. Work Effort matches everything greedily (every bullet appears in Work Effort)
2. Work Effort only fallthrough (bullets matching elsewhere skip Work Effort)
3. Work Effort union: `{anyOf: [{src: "workEffort"}, {fallthrough: true}]}` ← Chosen

**Decision:** Work Effort's include uses an anyOf union. Engine detects fallthrough recursively inside `anyOf`/`allOf` (new `predicateHasFallthrough` helper). Pass 1 evaluates predicates (fallthrough returns false); Pass 2 routes unmatched bullets to whichever H4 has `fallthrough: true` nested.

**Rationale:** Preserves the user's "a bullet shows up in every matching H4" rule AND guarantees no bullet is invisible. Single config pattern covers both semantics.

**Consequences:** Engine is 2-pass. `predicateHasFallthrough` is recursive because fallthrough can nest inside arbitrarily deep `anyOf`/`allOf` trees (unlikely in practice but correct).
