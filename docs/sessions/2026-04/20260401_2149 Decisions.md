# Conv Decisions - 2026-04-01

## 1. Publishing Checklist: Tag-Based Instead of Topic-Based
**Type:** Implementation
**Topics:** tag-taxonomy, publish-gate

**Trigger:** STUMBLE found "Topic selected" always unchecked despite topic being set

**Options Considered:**
1. Check course has tags with a topic_id (`course.tags.some(t => t.topic_id)`)
2. Check course has at least one tag assigned (`course.tags.length > 0`) ← Chosen
3. Remove the topic/tag requirement from publishing checklist

**Decision:** Changed publishing requirement from "Topic selected" to "At least one tag assigned" — checks `course.tags.length > 0` on frontend, `COUNT(*) FROM course_tags` on backend.

**Rationale:** Post TAG-TAXONOMY, courses are discoverable via tag:tag matching between course tags and user interests. A course needs tags to be findable. Whether those tags have topic associations is a data quality concern, not a publishing gate.

**Consequences:** Frontend `CourseDetail` interface updated to remove stale `topic_id`/`topic_name` fields, type `tags` correctly as objects. Backend publish API queries `course_tags` count. All PLATO personas updated with `courseTags`.

## 2. Enrollment: Teacher Defaults to Course Creator
**Type:** Implementation
**Topics:** enrollment, self-healing

**Trigger:** STUMBLE found `assigned_teacher_id` null after self-healing enrollment

**Options Considered:**
1. Fix checkout session creation to always include teacher in metadata
2. Fix enrollment creation to default to course creator when no teacher specified ← Chosen
3. Leave null and assign teacher on first session invite acceptance

**Decision:** In `createEnrollmentFromCheckout()`, if `assigned_teacher_id` is null/empty in Stripe metadata, look up the course creator and assign them as the default teacher.

**Rationale:** For creator-taught courses (the most common case), the creator IS the teacher. The checkout metadata may omit `assigned_teacher_id` when no explicit teacher selection happened. Defaulting to creator is safe for both creator-taught and peer-teacher courses (peer-teacher courses require explicit teacher selection during checkout, so metadata will have the value).

**Consequences:** Enrollment now always has a teacher assigned. Also added `student_count` increment (`UPDATE courses SET student_count = student_count + 1`) since it was never wired up.

## 3. PLATO Segment Design: Composability + Restartability
**Type:** Strategic
**Topics:** plato, architecture, testing

**Trigger:** Conv 071 identified need for composable STUMBLE segments; Conv 072 discussion merged composability with restartability

**Options Considered:**
1. Segments as organizational sugar only (runner flattens, no runtime effect)
2. Segments as snapshot boundaries + composition units ← Chosen
3. Full sub-scenario model with isolation and explicit input/output contracts

**Decision:** Segments are named step groups (2-3 steps) with goals, transparent context flow, and DB snapshots at boundaries. New types: `PlatoSegment`, `SegmentRef` as `ChainEntry` variant. Future consideration: `$flow.state` (Node-RED `msg` pattern) for self-aware segments.

**Rationale:** Merges two goals — composability (reuse segments across scenarios) and restartability (restore pre-failure snapshot, re-run segment). Transparent context avoids input/output contract overhead. Snapshots solve partial-write problem (restore clean state rather than re-running failed step).

**Consequences:** Full design written to `docs/as-designed/plato.md`. Implementation deferred — 7 TODO items in PLAN.md. Existing steps, scenarios, instances, personas unchanged.
