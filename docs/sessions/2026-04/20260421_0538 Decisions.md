**Rationale:** Data-first approach separates "what are the gaps" from "how do we close them" — the report becomes a reusable artifact that drives both the immediate BBB fix block and a future Stripe fix block.

## 2. Cron Worker architecture: Option B — separate standalone Worker
**Type:** Strategic
**Topics:** cloudflare, astro, deployment, architecture

**Trigger:** `@astrojs/cloudflare 13` doesn't expose `workerEntryPoint`. Adding `scheduled()` to the main Astro Worker is not cleanly supported.

**Options Considered:**
1. Auxiliary Worker via Astro adapter's `auxiliaryWorkers` option — CF-native but couples to adapter internals
2. ← **Separate standalone Worker project** (`peerloop-cron` / `peerloop-cron-staging`) with its own wrangler config, shared bindings via IDs
3. External GitHub Actions cron hitting `/api/admin/sessions/cleanup` — no Worker changes but couples to GHA reliability + auth-token secret
4. Post-build esbuild wrapper on Astro's output — hacky, fragile to adapter updates

**Decision:** Separate standalone Worker at `workers/cron/` with own wrangler.toml.

**Rationale:** Cleanest separation from Astro internals; most portable (same approach works for future Stripe polling cron); easiest to test/deploy independently; cost is one additional deploy step which aligns with existing DEPLOYMENT block work.

**Consequences:** Created `workers/cron/` directory structure. Added `deploy:cron:*` npm scripts. `peerloop-cron-staging` deployed and firing every 15min on staging. First run (2026-04-21T09:30:35Z) recovered 1 real missed `recording_ready` webhook.

## 3. Staging cron cadence: 15min; prod: 30min
**Type:** Implementation
**Topics:** cloudflare, deployment, cost

**Trigger:** Configuring `[triggers.crons]` for the standalone Worker.

**Options Considered:**
1. Every 5 min on both (fastest feedback, highest cost)
2. ← 15 min staging / 30 min prod (faster staging iteration, balanced prod)
3. 30 min on both (uniform, minimal cost)

**Decision:** `*/15 * * * *` for staging, `*/30 * * * *` for production.

**Rationale:** Staging benefits from faster feedback during verification; production balances freshness of no-show/recording healing against D1 + BBB API call cost. Most BBB sessions complete their formal lifecycle within 30min of `scheduled_end`, so 30min captures most missed webhooks quickly enough for UX.

**Consequences:** Set in `workers/cron/wrangler.toml` per environment. Adjustable per env if needed.

## 4. Staging verification file format: per-environment dev.vars
**Type:** Implementation
**Topics:** tooling, staging-verification

**Trigger:** Extending `trigger-webhook.sh` to target staging required access to staging-side secrets + URL.

**Options Considered:**
1. Single `.dev.vars` with both local + staging secrets namespaced (risk of wrong-environment selection)
2. ← Separate `.dev.vars.staging` (gitignored) with its own template `.dev.vars.staging.example`
3. Shell env vars only (ephemeral, harder to run ad-hoc)

**Decision:** Separate `.dev.vars.staging` file, gitignored, with template. Harness reads it when `ENV_TARGET=staging`.

**Rationale:** Mirrors existing `.dev.vars` / `.dev.vars.example` pattern. Explicit file = explicit environment. Template keeps onboarding fast while never leaking real secrets.

**Consequences:** Added `.dev.vars.staging` to `.gitignore`. Created `.dev.vars.staging.example`. Harness `ENV_TARGET` switch reads the correct file automatically.
