# Conv Decisions - 2026-04-19

## 1. Wire /r-end agent models via config.json, not inline SKILL.md annotations
**Type:** Implementation
**Topics:** cc-workflow, skills

**Trigger:** User requested per-aspect model selection for /r-end. Two established patterns exist — SOURCE (spt-docs) uses inline `**Agent N: name** (model: X)` annotations; Peerloop's config.json already has skill-specific sections (`rTimecardDay`, `docsRegistry`).

**Options Considered:**
1. Inline SKILL.md annotation only
2. Central `config.rEnd.agentModels` only
3. Both (config stores the value, SKILL.md heading reads it via `!` backtick) ← Chosen

**Decision:** Store model selection in `config.json → rEnd.agentModels`; SKILL.md resolves via `!` backticks in Pre-computed Context and inline in Agent N headings.

**Rationale:** Matches Peerloop's established pattern (config-driven skill behavior). Keeps model selection editable without SKILL.md edits. Inline backticks in headings preserve point-of-use visibility for the dispatching context. Graceful fallback (`|| echo "(inherit)"`) when config missing.

**Consequences:** Future model changes are a one-line config edit. Deleting a field from config causes the expression to print `(inherit)`, and the Step 3 dispatch instruction tells the runtime to omit the `model` parameter in that case — agent falls back to main-context model.
