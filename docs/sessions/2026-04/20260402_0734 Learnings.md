# Conv Learnings - 2026-04-02

## 1. PLATO Snapshot Bridge Pattern
**Topics:** plato, testing, browser-mode

**Context:** API-run uses in-memory better-sqlite3 but browser-run needs real D1. The gap prevented browser-walking post-enrollment checkpoints.

**Learning:** `better-sqlite3.serialize()` dumps the entire in-memory DB to a Buffer. Copy that to wrangler's D1 SQLite file and the dev server serves the exact state the API-run produced. The round-trip is ~1 second.

**Pattern:** `snapshot: true` on PlatoInstanceFile → auto-save after successful API-run → `npm run plato:restore -- <name>` regenerates + restores in one command.

---

## 2. Always-Regenerate Beats Cache Management for Fast Operations
**Topics:** plato, devex

**Context:** Snapshot files could become stale when schema, steps, or personas change. Considered metadata sidecars and timestamp checks.

**Learning:** When regeneration costs less than thinking about staleness (~400ms), don't cache. The combined `plato:restore` command always regenerates, making staleness impossible. Add cache management only when the operation becomes expensive.

---

## 3. BBB Webhook Has No Signature Validation
**Topics:** bbb, webhooks, security

**Context:** Needed to trigger session completion during browser-walk without a real BBB server.

**Learning:** The BBB webhook handler at `/api/webhooks/bbb` accepts any POST with `{ event: "meeting-ended", meeting_id: "<session_id>" }` — no signature verification. This enabled the hybrid browser-walk but is a security gap for production.

---

## 4. Flywheel Split Point Selection
**Topics:** plato, testing, stripe, bbb

**Context:** Needed to bridge API mode (mocked externals) and browser mode (real UI) for the flywheel.

**Learning:** The split point should be chosen to put mocked external services on the API-run side of the boundary. The flywheel-to-enrollment snapshot absorbs Stripe (mocked in API-run, baked into snapshot). BBB stays on the browser-run side but only the webhook trigger is needed, not the video room. The split point determines what the browser-run can and can't test.

---

## 5. Dashboard Module Stats Use Different Source Than Enrollment Progress
**Topics:** dashboard, enrollment, sessions

**Context:** After completing all 3 sessions via BBB webhook, enrollment.progress_percent = 100 and status = 'completed', but dashboard showed 0 modules completed and 0%.

**Learning:** The dashboard stats query likely reads from a different source (possibly `module_progress` table) than the enrollment's `progress_percent` field. The webhook completion path (`completeSession → checkEnrollmentCompletion`) updates enrollment status but may not populate whatever the dashboard stats query uses.
