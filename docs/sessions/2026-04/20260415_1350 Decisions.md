## 4. Helper coverage: 3 narrow helpers, not one composite
**Type:** Implementation
**Topics:** auth, refactoring, helpers

**Trigger:** Sites had 3 different query shapes; tempting to unify under one `getUserPermissions(db, userId)` returning everything.

**Options Considered:**
1. Single mega-helper returning `{isAdmin, canModerateCourses, isAdminByList?}` ŌĆö overfetches
2. 3 narrow helpers per query shape ŌåÉ Chosen
3. Keep everything inline

**Decision:** 3 helpers: `isUserAdmin` (boolean), `getUserPermissionFlags` (two-flag object), `getAllAdminUserIds` (array).

**Rationale:** Each helper's signature matches exactly one query shape. No ambiguity at the call site about what's being fetched. Moderator endpoints stay inline because their query is a superset (extra fields + deletion filter) ŌĆö forcing through a helper would split their single-query validation into two round-trips.

**Consequences:** 9 sites migrated, 3 moderator sites intentionally inline. `@lib/auth` barrel now exports 3 new functions.
