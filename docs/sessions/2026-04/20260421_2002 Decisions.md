## 1. Endpoint-based diagnostic over one-off script for [VA]
**Type:** Implementation
**Topics:** stripe, observability, cc-workflow

**Trigger:** [VA] needed to confirm staging Worker's `STRIPE_SECRET_KEY` is Sandbox-scoped. Options: ad-hoc script invoked once, or permanent diagnostic endpoint.

**Options Considered:**
1. One-off Node script that reads the staging secret via Wrangler and hits Stripe API — simple, single-use
2. Permanent admin-gated `/api/admin/stripe-mode` endpoint ← Chosen
3. Manual Dashboard-only comparison — no artifact, no reproducibility

**Decision:** Permanent endpoint. The audit is not a one-time concern — every future secret rotation, every pre-go-live check, every mode-drift investigation benefits from having a standing diagnostic.

**Rationale:** Per CLAUDE.md default-to-durable, the permanent endpoint's incremental cost (60 LOC + 4 tests) is amortized over every future audit. It also provides an authoritative mode check independent of Dashboard UI changes (the Sandboxes-merge UI shift would have required updating a one-off script's instructions).

**Consequences:** `src/pages/api/admin/stripe-mode.ts` exists as a permanent tool. Future audits run it via browser-login + navigate, not script invocation. Any Stripe Mode Discipline verification (including prod go-live) uses this path.

## 2. Broaden feedback_no_paste_tokens_in_chat.md to cover Claude diagnostic leaks
**Type:** Strategic
**Topics:** cc-workflow, security, documentation

**Trigger:** [FD] — Conv 144's two near-miss leaks (whsec partial, sk_test full) were both Claude-initiated via `od -c` and `stripe config --list`, not user-paste. Existing memory was silent on this surface.

**Options Considered:**
1. Add a separate sibling memory file — two rules, two files
2. Broaden the existing memory to cover both surfaces in one document ← Chosen
3. Leave as-is and rely on ad-hoc vigilance

**Decision:** Broaden. Renamed the memory to "Never write secrets or tokens into chat — user-paste OR Claude diagnostics". Section 1 preserves Conv 113 rule verbatim; Section 2 adds Claude-initiated patterns with unsafe-list, safer-alternatives table, and redactor one-liners.

**Rationale:** Both surfaces share the same outcome (secret in transcript = burned). The unified structure also makes the shared "Why" section serve as the motivation anchor for both surfaces.

**Consequences:** Future convs have a clear pre-flight heuristic before running any diagnostic: "does this command's output need to contain a secret, or just a presence/identifier check?" Almost always the latter. Also includes a redactor library (sed one-liners for Stripe, JWT, CF hex, JSON) that's reusable.

## 3. STRIPE-E2E-DEV as a deferred block with explicit value chain
**Type:** Strategic
**Topics:** stripe, testing, planning

**Trigger:** User asked for a go-live-grade confidence strategy around Stripe integration. The gap between unit tests + one-shot staging live-verification + harness replay vs. "real browser in real user flow" has no current coverage.

**Options Considered:**
1. Start implementing the walkthrough immediately in this conv
2. Defer as a scoped block with enough detail for future Plan Mode to execute from ← Chosen
3. Add a smaller single-tier task ("write a Dev testing guide") without the value analysis

**Decision:** Defer as a comprehensive block with 4 tiers (A paper / B scripted / C Playwright / D Claude-MCP-driven), 11-scenario matrix, Dev → Staging → Live value chain, and 7 open questions for Plan Mode.

**Rationale:** Conv 144 proved integration bugs exist and can hide for months (8 weeks for `constructEventAsync`). The user explicitly wants go-live-grade bullet-proofing, which requires a bigger frame than a single testing guide. Deferring with full context (rather than implementing in a rushed slice) lets Plan Mode pick the right tier intentionally rather than by default. The block's "What Dev E2E testing buys us" section makes the *why* explicit so tier selection hinges on goals (team enablement / regression prevention / ad-hoc rehearsal) rather than effort alone.

**Consequences:** `PLAN.md` has a new DEFERRED row #20 and a ~160-line block. Claude-MCP browser-automation of Stripe Checkout is now on the roadmap (Tier D), with the iframe-handling question flagged as the decisive unknown. The block is a durable artifact — next Plan Mode session starts from a shared framing rather than reinventing the scope.
