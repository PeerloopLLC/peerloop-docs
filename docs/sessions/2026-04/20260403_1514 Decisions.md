# Conv Decisions - 2026-04-03

## 1. Standardize Dev Password to Password1
**Type:** Implementation
**Topics:** auth, seed-data, testing

**Trigger:** Conv 082 discovered seed hashes didn't match any known password; had to manually set `Password1` to proceed with walkthroughs.

**Options Considered:**
1. Fix hashes to match `dev123` — fails validation (too short, no uppercase)
2. Use `Password1` everywhere <- Chosen
3. Use `Dev12345` (PLATO's original choice) — would require changing already-fixed Conv 082 DB state

**Decision:** Standardize all dev/test passwords to `Password1` across SQL seeds, mock-data, E2E tests, PLATO personas, and documentation.

**Rationale:** Meets all validation rules (8+ chars, uppercase, lowercase, number), matches what was already set in the running DB during Conv 082, simple and memorable.

**Consequences:** 19 files updated across both repos. All tests pass. PLATO personas aligned.

## 2. Admin Auth Guard in Layout, Not Pages
**Type:** Implementation
**Topics:** auth, astro, admin, security

**Trigger:** STUMBLE-AUDIT Conv 082 found admin pages render admin layout for non-admin users.

**Options Considered:**
1. Guard in each of 13 admin pages — fragile, easy to miss on new pages
2. Guard in AdminLayout.astro <- Chosen — single enforcement point
3. Middleware-based guard — Astro middleware would work but is heavier

**Decision:** Auth guard in `AdminLayout.astro` using `getSession()` + role check + `Astro.redirect()`.

**Rationale:** Secure by default — new admin pages automatically protected. Three-layer redirect (no secret, no session, wrong role) provides appropriate UX for each case.

**Consequences:** All 13 admin pages protected. `TODO: guard` comment replaced with implementation.

## 3. PLATO Seed as Default for Staging Testers
**Type:** Strategic
**Topics:** plato, seed-data, staging, testing

**Trigger:** Preparing staging deployment for 3 core testers; needed to decide which seed data to provide.

**Options Considered:**
1. SQL seed (`db:setup:staging:dev`) — works today but data may have consistency issues
2. PLATO seed (`plato:seed:staging`) <- Chosen — richer, API-validated data
3. Hybrid (PLATO base + SQL overlays) — incompatible due to UUID vs hardcoded ID mismatch

**Decision:** Use PLATO seed for staging testers. Built `plato-seed-staging.js` script and `plato:seed:staging` npm command.

**Rationale:** Testers need to explore the platform across roles, see accurate dashboards, and understand what's possible. PLATO data is richer (14 sessions vs fewer, 3 reviews, 10 notifications, accurate counters) and inherently consistent. The things PLATO can't do (real Stripe checkout, Stream feeds) require external service configuration that isn't set up for staging anyway.

**Consequences:** New staging seed path created. Old `db:setup:staging:dev` still works as fallback. Two parallel seed paths documented but not yet unified.

## 4. PLATO seed-dev Instance with Snapshot
**Type:** Implementation
**Topics:** plato, seed-data, infrastructure

**Trigger:** seed-dev scenario passed validation; needed a way to produce a restorable snapshot.

**Options Considered:**
1. Always run via standalone scenario test — no snapshot produced
2. Create instance file with `snapshot: true` <- Chosen

**Decision:** Created `seed-dev.instance.ts` with snapshot support + `plato:seed` npm script that runs API -> snapshot -> restore in one command.

**Rationale:** The instance mechanism already handles snapshot generation. Adding an instance file is trivial and enables the full restore pipeline.

**Consequences:** `npm run plato:seed` (local) and `npm run plato:seed:staging` (remote) both work. Snapshot is 1324KB.
