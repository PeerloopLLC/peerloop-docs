# Conv Decisions - 2026-04-02

## 1. URL-Embedded HMAC Token for BBB Webhook Authentication
**Type:** Implementation
**Topics:** security, bbb, webhooks

**Trigger:** Task #1 (carried from Conv 074) — BBB webhook endpoint had no signature validation

**Options Considered:**
1. URL-embedded HMAC token — append `?token=HMAC(sessionId, BBB_SECRET)` at room creation ← Chosen
2. Payload-based HMAC with custom header — BBB doesn't forward custom headers back, not viable
3. URL token + timing window — adds complexity without real benefit since session IDs are unique

**Decision:** Use HMAC-SHA256(sessionId, BBB_SECRET) as hex string appended to webhook URL. Verify by recomputing from roomId in parsed event.

**Rationale:** Only viable approach given BBB's `meta_endCallbackUrl` has no built-in auth. CD-038 confirmed analytics callback uses JWT (already secured), but endCallbackUrl is a bare URL callback. Constant-time comparison prevents timing attacks.

**Consequences:** New `src/lib/webhook-auth.ts` utility. `join.ts` generates token. `bbb.ts` verifies. PLATO api-runner auto-injects tokens. 4 new auth tests added.

> **Insight:** When a third-party service doesn't support webhook signing, URL-embedded HMAC tokens bound to a resource ID are a practical alternative. The token proves the callback was registered by your system (only your server knows the secret) and is scoped to a specific resource (preventing replay against different resources). This is the same pattern used by some payment processors for return URLs.
