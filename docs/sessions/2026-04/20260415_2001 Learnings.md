# Conv Learnings - 2026-04-15

## 1. Refresh-token-as-auth fallback fundamentally changes JWT-claim-staleness calculus
**Topics:** auth, jwt, security

**Context:** [RA-JWT] decision brief — the audit report framed "embedding isAdmin in JWT" as a "15-minute staleness" tradeoff. That framing is wrong.

**Learning:** `getSession()` at `src/lib/auth/session.ts:88-94` treats a valid refresh token as a valid auth credential (not just a key for minting new access tokens) when the access token is missing or expired. Therefore any claim embedded in the refresh token is trusted for the full 7-day TTL, not 15 min. Any future decision about per-request JWT claims must engage with this fallback — not just the access-token TTL.

**Pattern:** When reasoning about JWT claim freshness, the bound is `max(access_ttl, refresh_ttl_when_fallback_used)`. The fallback's existence quietly promotes refresh-token claims to first-class auth, which changes every "how stale can this be?" question.

## 2. "Where else is this pattern?" — audit scope blindspots in SSR loaders
**Topics:** audits, role-audit, ssr, methodology

**Context:** During [RA-JWT] brief prep, grep'd `is_admin` and found `src/lib/ssr/loaders/communities.ts:471-476` doing a raw `SELECT is_admin FROM users WHERE id = ?` — exactly the pattern Conv 123 [RA-ADM] migrated 9 sites away from. Conv 123 missed it.

**Learning:** Conv 123's audit was scoped implicitly to API handlers (`src/pages/api/**`) and didn't sweep `src/lib/ssr/`. When designing audits that enumerate call sites, ask "what directory structure am I implicitly excluding?" — loaders, middleware, and background jobs are frequent blindspots.

**Pattern:** After finishing a pattern-migration audit, run one more grep `<pattern>` across ALL of `src/` (not just where the audit looked) and spot-check any hits in unexpected directories. 5 minutes of grep saves spawning follow-up tasks weeks later.

## 3. Field-deletion refactors can cascade into query-plan improvements
**Topics:** refactoring, sql, performance

**Context:** [CTR] was framed as a type-cleanup micro-task: "drop an unused `role` field from a shared type." Tracing the chain revealed the field was sourced via a `LEFT JOIN community_members cm ON cr.community_id = cm.community_id AND cr.uploaded_by_user_id = cm.user_id` — a JOIN that existed *only* to populate the unused field.

**Learning:** Deletion refactors should trace *up* the data pipeline to the query layer before stopping. An "unused field" downstream may be load-bearing for an otherwise-invisible JOIN, SELECT column, or LEFT-JOIN cost. Cascade here: 6 Astro pass-through removals → 1 SSR loader transformation → 1 type definition → 1 type declaration in component → 1 SELECT column → 1 JOIN eliminated.

**Pattern:** For any "drop unused field" task touching a shared type, grep the SQL layer for the SELECT column alias, then inspect the JOIN graph of that query to find whether the join itself is now orphaned.

## 4. tech-doc-sweep.sh auth matcher is a structural noise source, not a drift signal
**Topics:** tooling, docs, tech-doc-sweep

**Context:** [ADR] investigation of 4 flagged auth docs.

**Learning:** The sweep rule `"auth|auth"` (line 32 of `tech-doc-sweep.sh`) matches any file path containing "auth" against any doc filename containing "auth". Any commit touching `src/lib/auth/*` or `src/pages/api/auth/**` will flag all 4 auth docs regardless of what actually changed. This is coarse by design (script explicitly positions itself as a reminder, not a gate) but means future /r-end reviewers should pre-judge those 4 flags.

**Pattern:** Recorded in DOC-DECISIONS.md §5 as "Tech Doc Sweep: Auth-Doc False Positives Are Expected" with a dismissal table mapping code-path → doc-to-actually-review.

## 5. Ghost tasks from prior-conv session extracts
**Topics:** tasks, memory, task-hygiene

**Context:** The `[TS]` ("Re-baseline tech-doc-sweep.sh hash state") task from Conv 104-era Extract files surfaced during the [ADR] option 3 analysis.

**Learning:** The committed `tech-doc-sweep.sh` has no hash-state or baseline mechanism — it's stateless. `[TS]` was written against a planned-or-earlier implementation that never shipped. Tasks inherited from old session extracts can be "ghost tasks" referencing features that don't exist; before acting on them, grep the current codebase to verify the feature they reference is actually present.

**Pattern:** CLAUDE.md "Before recommending from memory" rule formalized this. This conv demonstrated the rule in action — grepping the script revealed the mismatch, avoiding a silent-fail "re-baseline" of a nonexistent state file.
