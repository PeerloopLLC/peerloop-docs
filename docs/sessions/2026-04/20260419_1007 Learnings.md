# Conv Learnings - 2026-04-19

## 1. Trust-but-verify on sub-agent architectural claims
**Topics:** cc-workflow

**Context:** Conv 135 Group A Explore agent reported `feeds.md` as a REAL GAP, claiming Conv 130's `fetchCourseTabData` consolidation "moved feed data to SSR." A 30-second verification — reading `fetchCourseTabData`'s actual return shape and the diff to `feed.astro` — proved the claim wrong. The loader returns course metadata (objectives, curriculum, teachers, testimonials), not feed activities. The `/api/feeds/course/[slug]` endpoint still exists; Stream-backed feed loading is unchanged.

**Learning:** When a sub-agent reports a REAL GAP — especially one that implies an architectural change — verify the specific claim before acting. Pattern-matching agents conflate "file X refactored to use Y" with "behavior Y of file X changed." Cost of verification: 1-2 tool calls. Cost of acting on a false claim: unnecessary doc rewrites + confidence erosion.

**Pattern:** Before applying any "REAL GAP" sub-agent verdict on an architectural claim, read either (a) the returned shape of the cited loader/function, or (b) the actual diff of the triggering code change. Only proceed with the fix after the claim survives verification.

## 2. Pipeline-symmetry gaps hide until two halves are tested together
**Topics:** cc-workflow

**Context:** `[RS]` patched `/r-start` to assign mnemonic codes on RESUME-STATE → TodoWrite transfer. While writing the edit, realized `/r-end` writes RESUME-STATE and its format isn't necessarily code-preserving. Reader and writer live in separate skills — independent tests of each would pass (reader assigns codes for code-less items; writer produces `{subject}`-faithful output for TodoWrite Items section). But the `## Remaining` section is prose-curated by dev-log extraction, and that is what `/r-start` actually reads. Without an explicit preservation instruction in the writer, codes could drop silently during the merge step.

**Learning:** When a convention transits a serialized form between two skills, BOTH halves need to enforce it. Patching one side creates a false-safe condition — the contract looks upheld until a real cross-conv test surfaces the drop. The `[RS]` → `[RE]` pair illustrates: testing the pipeline end-to-end requires at least 2 real conv boundaries, which won't happen until at least Conv 136 brings tasks carrying `[XX]` codes through `/r-end` → RESUME-STATE → `/r-start`.

**Pattern:** When establishing a new cross-skill convention, patch every pipeline stage (producer + transit + consumer) in one go, even if only one seems to need it. Flag a "convention end-to-end test" task for the first natural conv boundary that exercises it.

## 3. 89% first-run FP rate is a strong design signal, not just noise
**Topics:** docs-infra

**Context:** The DOC-SYNC-STRATEGY SessionStart hook's first real run (Conv 135 `/r-start`) produced 9 flags, of which 1 was a real drift gap (`session-booking.md`) and 8 were false positives. That's ~11% precision. The immediate response was to add a prose-based dismissal entry to DOC-DECISIONS.md — a band-aid that scales linearly in prose volume as new FP classes emerge.

**Learning:** A first-run FP rate above ~70% means the matcher's recall-vs-precision tradeoff isn't in an acceptable zone. Prose dismissals are a stopgap; the structural fix is tightening the matcher (what `[DT]` plans). Worse: the prose pattern treats FP as "cost of doing business" instead of "bug surface" — over time, a fresh CC instance must read more prose to know what to dismiss, which is exactly the doc-rot the strategy is supposed to prevent.

**Pattern:** Track FP rate as a first-class metric in drift-detection tooling (like test coverage or type-check warnings). Set an explicit threshold (e.g., <20% FP on 3 consecutive batches) as the signal that matcher precision is adequate. Phase 4 exit criteria already reflects this.
