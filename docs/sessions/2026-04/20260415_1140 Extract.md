### #13 [DS] dev:staging adapter 13 regression (investigation)

**Claude:** Read `astro.config.mjs` fully ‚Äî found the adapter-13 migration comment explaining `remoteBindings` is now forwarded directly (was `platformProxy: { environment, remoteBindings }`). Checked package.json for `dev:staging` script: `USE_STAGING_DB=1 CLOUDFLARE_ENV=staging astro dev`.

Reproduced the bug. Ran `dev:staging` for ~20s, `curl -s /` ‚Üí HTTP 200 returned, but server logs showed:
```
TypeError: Cannot read properties of null (reading 'useState')
    at exports.useState (node_modules/.vite/deps_ssr/chunk-BPGYRLWZ.js:959:35)
    at useCurrentUser (src/lib/current-user.ts:1225:27)
    at OnboardingNudgeBanner (src/components/onboarding/OnboardingNudgeBanner.tsx:44:23)
    at renderWithHooks (react-dom_server.js:3973:24)
```

Same pattern on every React island using `useCurrentUser` / `useAuthStatus` (both hooks in `src/lib/current-user.ts`).

Ran plain `npm run dev` as control ‚Äî HTTP 200, clean render, zero errors. Differential confirmed: regression specific to `dev:staging`.

Read `src/lib/current-user.ts:1215-1255` ‚Äî `useCurrentUser` and `useAuthStatus` are standard React hooks with `useState`/`useEffect`. Nothing unusual.

Root cause (hypothesis): two copies of React in the SSR graph. The `deps_ssr/chunk-BPGYRLWZ.js` pre-bundled chunk holds one React; `deps_ssr/react-dom_server.js` imports a second. Dispatchers are not shared ‚Üí `useState` resolves to `null` when react-dom/server's `renderWithHooks` runs.

Trigger pivot: `USE_STAGING_DB=1` ‚Üí `remoteBindings: true` in astro.config.mjs:49. This activates a branch inside `@astrojs/cloudflare` 13 that mutates vite config mid-session and regenerates a mismatched SSR dep cache.

Marked #13 completed with full root-cause description. Spawned #20 [DS-FIX] for the one-line attempted fix.
