# Conv Learnings - 2026-04-02

## 1. Astro url Context vs request.url in Tests
**Topics:** testing, astro

**Context:** BBB webhook handler destructured `url` from Astro context to read query params, but tests failed because `createAPIContext` hardcodes `url` to `http://localhost:4321/api/test`

**Learning:** In Astro API handlers, prefer `new URL(request.url)` over the `url` context parameter when reading URL properties. The `request.url` is always set correctly in both production and test environments. The Astro `url` context may not match the request URL in test harnesses.

**Pattern:** `const requestUrl = new URL(request.url); const token = requestUrl.searchParams.get('token');`

## 2. URL-Embedded HMAC for Webhook Endpoints Without Built-in Auth
**Topics:** security, bbb, webhooks

**Context:** BBB's `meta_endCallbackUrl` has no signature mechanism (unlike the analytics callback which uses JWT). Needed to authenticate webhook callbacks.

**Learning:** When the webhook caller (BBB) doesn't support signing or custom headers, embed an HMAC token in the callback URL at registration time. The token binds the URL to a specific resource ID (roomId), so a leaked URL can't be reused for a different resource. Use constant-time comparison to prevent timing attacks.

## 3. PLATO Browser Mode Executes via Chrome MCP Bridge
**Topics:** plato, testing, browser

**Context:** Incorrectly suggested BrowserIntents could be translated to Playwright and framed chrome MCP as "ad-hoc debugging"

**Learning:** PLATO browser-runs execute through the /chrome MCP bridge, not Playwright. The BrowserIntent design is intentionally split: NavClick navigation is deterministic (fail-fast), pageAction is prose (instructional, executed interactively). This split is by design — pageAction is too varied to automate. E2E (Playwright) is a separate test layer.
