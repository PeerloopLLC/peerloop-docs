# Conv Learnings - 2026-04-21

## 1. ESLint v10 is strict about unknown rules in disable directives
**Topics:** eslint, package-updates, lint

**Context:** Conv 142 baseline surfaced `Definition for rule 'react-hooks/exhaustive-deps' was not found` as a new lint error on `MemberDirectory.tsx:141`. The disable comment had existed without issue on earlier ESLint versions.

**Learning:** ESLint v10 (installed Convs 104-114) treats `// eslint-disable-next-line <unknown-rule>` as a hard error, whereas earlier versions silently ignored or warned. This is a breaking-change surface that can turn dormant copy-paste bugs into baseline regressions long after the original error was introduced.

**Pattern:** After any eslint major-version bump, scan for unknown-rule disable comments (`grep -r "eslint-disable.*/" src | check against registered plugins`). Pre-existing "fine" configs may contain invalid directives that only fail under the new version.

## 2. Hook-deps rule had never been running on this codebase
**Topics:** lint, react, code-quality

**Context:** During [LE] root-cause diagnosis, grep showed 127 React files use hook deps with 405 hook occurrences — but exactly one file (`MemberDirectory.tsx`) had a `react-hooks/exhaustive-deps` disable comment.

**Learning:** The presence of a disable comment does NOT imply a rule is actually being enforced. This codebase had been flying blind on stale-closure bugs for its entire lifetime because `eslint-plugin-react-hooks` was never installed, only referenced. Always cross-check: a disable comment for rule X implies rule X should be configured in the eslint config — if it isn't, the comment is a lie.

**Pattern:** Before accepting "the linter approves this file," verify the rules you expect are actually registered. `npm run lint -- --print-config <file>` enumerates the effective config for a specific file.

## 3. JS template strings + bash double-quotes corrupt signed payloads
**Topics:** shell, crypto, testing, harness

**Context:** Building [VH]'s Stripe SDK verification, my first attempt embedded the JSON payload directly into a bash command via JS backtick interpolation. The payload contained nested double-quotes, which bash re-parsed at the shell level — corrupting the byte-exact signing input and producing a mismatched HMAC.

**Learning:** Any time you HMAC a JSON string passed through a shell layer, the signing input must not traverse any quote-interpreted context. Use stdin pipes, heredocs, or a wrapper script that receives the payload through `cat` — never interpolate JSON into a double-quoted shell command line.

**Pattern:** For HMAC-over-JSON testing, the canonical recipe is: write the payload to a temp file or pipe via stdin, then `printf '%s' "$(cat)" | openssl dgst ...`. The Node child_process `spawnSync('bash', [wrapper])` with `input: PAYLOAD` pattern was what worked.
