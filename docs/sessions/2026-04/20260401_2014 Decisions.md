# Conv Decisions - 2026-04-01

## 1. Level Per Topic, Not Per Tag
**Type:** Implementation
**Topics:** onboarding, schema, smart-feed

**Trigger:** TopicPicker had experienceLevel per tag but no DB backing

**Options Considered:**
1. Remove the dropdown entirely (if feature not needed)
2. Store level per tag in `user_tags` ← Chosen (denormalized but practical)
3. New `user_topic_levels` table (normalized but extra join)

**Decision:** Store level per tag in `user_tags` column, but set all tags within a topic to the same level (per-topic conceptually, per-tag in storage).

**Rationale:** Makes future Smart Feed join trivial (`user_tags.tag_id = course_tags.tag_id` + compare levels). The denormalization is controlled — TopicPicker enforces same level for all tags in a topic.

**Consequences:** Added `level TEXT NOT NULL DEFAULT 'beginner'` to `user_tags`. Created deferred block LEVEL-MATCH (#40) for Smart Feed integration.

## 2. Composable STUMBLE Segments Over Monolithic Instances
**Type:** Strategic
**Topics:** stumble, plato, testing-strategy

**Trigger:** Flywheel STUMBLE showed natural service boundary at checkpoint 10 (Stripe) and inability to test checkpoints 11-14 (BBB)

**Options Considered:**
1. Keep flywheel as single 14-checkpoint instance
2. Split flywheel into pre/post-enrollment instances
3. Build from small composable segments (2-3 steps each) ← Chosen

**Decision:** Design instances from chainable segments. Segments align with service boundaries. Failed segment can be re-run independently.

**Rationale:** Failure isolation (re-run from failed segment, not from beginning), incremental coverage, service boundary alignment, reuse across instances.

**Consequences:** Added design direction to PLAN.md and plato.md. TODO: refactor flywheel into segments, create post-enrollment instance.
