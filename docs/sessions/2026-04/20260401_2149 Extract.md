### Composable Segments Design Discussion

**User:** "very good. YOu mentions that the flywheel instance chains. What does it chain?"

**Claude:** Showed the flywheel scenario's `chain` array — 11 ordered StepRef entries executed sequentially. Explained the relationship between instance (walkthrough checkpoints), scenario (chain), and steps.

**User:** "I am unclear hoe a 14 step flywheel that fails at step 9, can be restarted at the beginning of step 9 with the needed data alreadu in the db?"

**Claude:** Proposed DB snapshots at segment boundaries. Explained per-step snapshots vs per-segment snapshots. Introduced the concept of SQLite file copy for restartability.

**User:** "Steps are needed. They are atomic and independant of the data they use. Scenarios was our term for a bunch of steps run in sequence, still independant of data. Then we tacked on data to each scenarion and we got scenarion instances. Now we are looking to add composability so that a 14 step flywheel can be made into (just as an example) 7 2-step scenarios that we then string together into a sequence. And then we tack on data. So it may be we have to break waya from our current (though elegant) structure for this to work..."

**Claude:** Refined the design — segments are reusable step groups (with goals), not mini-scenarios. Showed the hierarchy: Step → Segment → Scenario → Instance. Discussed context flow (transparent vs boundaries).

**User:** "My previous (to thjis) discussion was about restarting a failed scenarion instance, the I notoced you no longer has composability in a way that I understood that, and so I explained my idea. But now we no longer have restarting at segment X. SOmehow we have to merge these two goals..."

**Claude:** Merged both goals: segments serve as both composition units AND snapshot boundaries. DB snapshot before each segment enables restartability; segment references in scenario chains enable composability. Proposed minimal type additions (PlatoSegment, SegmentRef as new ChainEntry variant).

**User:** "Let's hold off on decided between steps and segments for now. Let's deal with Using a segment created in the flywheelinside a new scenario. Would that make sense?"

**Claude:** Showed concrete reuse example — `setup-creator` and `create-offering` segments shared between flywheel and a new `enrollment-focus` scenario.

**User:** "The validation would need to attach data to become an instance"

**Claude:** Confirmed — validation proof requires the full stack: segment → scenario → instance → pass. Two different instances sharing the same segment, both passing verify blocks.

**User:** "yes write it up for future"

**Claude:** Wrote full design section in `docs/as-designed/plato.md` (~150 lines). Updated PLAN.md with implementation TODO checklist. Added design decisions to the decisions table.

**User:** "I'd like you to add this consideartion yto your delibreation documents. Node-Red offeres a series of nodes that you connect into a sequence..."

**Claude:** Added Node-RED `msg` pattern as design inspiration — mapped `msg` properties to PLATO's `$context`, identified the `$flow.state` concept as a future addition for flow-wide shared state.

**User:** "I use it in node-red to rewire flows, bypas nodes, etc and still each node knows what to expect and how complete it is"

**Claude:** Expanded the design to capture `$flow.state` as a control plane (not just data plane) — carrying skip directives, mode flags, and completion tracking. Key insight: segments become self-aware (decide their own behavior from flow state) rather than runner-directed.
