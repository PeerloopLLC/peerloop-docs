# Conv Learnings - 2026-04-01

## 1. TAG-TAXONOMY Refactoring Long Tail
**Topics:** tag-taxonomy, refactoring, publish-gate

**Context:** Four bugs traced back to the TAG-TAXONOMY refactor (Conv 048-054) which removed `courses.topic_id` and replaced it with `course_tags → tags → topics`. Several downstream consumers were never updated.

**Learning:** Large renames/refactors have a long tail of downstream references that only surface during manual testing (STUMBLE). The publishing checklist, course editor interface, and PLATO personas all still referenced the old `topic_id` field. Grep for removed columns/fields after a schema refactor isn't enough — you also need to check runtime behavior where the field was consumed but produced a falsy value rather than an error.

**Pattern:** After removing a DB column, grep for the column name across all frontend interfaces and test data (personas, seed files) — not just the backend code.

## 2. student_count Never Wired Up
**Topics:** enrollment, schema-gap

**Context:** The `courses.student_count` column existed in the schema with `DEFAULT 0` but no code ever incremented it. Discovered during STUMBLE walkthrough when the enrollment card showed 0 students for an enrolled course.

**Learning:** Schema columns with defaults can silently remain unwired — they don't cause errors, they just stay at their default value. Discovery requires runtime observation (STUMBLE) or a systematic schema-to-code audit.

## 3. Node-RED msg Pattern for Flow State
**Topics:** plato, architecture, node-red

**Context:** User described Node-RED's `msg` object pattern where a single object travels through all nodes, serving as both data plane and control plane.

**Learning:** The `msg` pattern makes nodes self-aware — each node decides its own behavior based on flow state rather than being directed by an orchestrator. This enables rewiring, bypassing, and partial re-execution without changing the runner. Applied to PLATO as `$flow.state`: a reserved shared context that carries completion tracking, skip directives, and mode flags. Key distinction from current `$context`: `$context` is namespaced per-step (prevents collisions), while `$flow.state` is flow-wide (enables cross-cutting concerns).
