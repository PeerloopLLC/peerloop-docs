# Conv Learnings - 2026-04-11

## 1. RESUME-STATE.md can disagree with on-disk git state
**Topics:** cc-workflow, dual-repo

**Context:** Conv 105's RESUME-STATE.md said 3 [HW] files were "uncommitted pending user decision," but git log showed commit `3e15f8a` already on `jfg-dev-10up` (stamped "Conv 105: [HW] remove dead half-wired state..."). The edits were committed before state was saved, and the save captured stale pre-commit context.

**Learning:** Before acting on a RESUME-STATE.md claim about "uncommitted edits," verify against `git status` and `git log --oneline`. State files describe the author's intent at save time; git is the source of truth. This is especially important for /r-start Step 7 task-transfer, because an "uncommitted" item carried forward as a todo can be obsolete before it's read.

**Pattern:** Whenever RESUME-STATE.md references uncommitted files, early in the next conv run a quick `git -C REPO status --short` + `git log --oneline -5` check on the named files/branch. Resolve the contradiction before proceeding.

## 2. Commit already on the target branch = automatically "bundled" into a future PR
**Topics:** dual-repo, cc-workflow

**Context:** User asked to "bundle [HW] into [P6]" assuming the [HW] edits were uncommitted and would need to be amended into a [P6] commit. Actually the [HW] commit was already on `jfg-dev-10up`, so the PR (once opened) will include it by default — no action needed.

**Learning:** "Bundle X into PR Y" has two distinct meanings: (a) **compose a single commit** combining both, or (b) **ensure both ride on the same PR**. When the commits are already on the same branch that the PR will be opened from, (b) is automatic. Surface this explicitly so the user doesn't think a manual merge/squash is required.

## 3. Forward-looking guidance in a dated decision record needs a rationale refresh when the environment moves
**Topics:** docs-infra, docs-discipline

**Context:** The "Stay on Node 22" decision (2026-02-16) cited "Astro 5.x only lists Node 18/20/22 as supported." The decision itself was still correct, but after PACKAGE-UPDATES Phase 2a the codebase moved to Astro 6.1.5, so the rationale's premise was stale even though the conclusion held.

**Learning:** Decision records that cite the *current environment* as rationale need a refresh (not a rewrite) when the environment changes. The format: preserve original date + decision, append an "Updated YYYY-MM-DD" note explaining what changed and why the decision still stands. This preserves historical intent while keeping the forward-looking guidance correct.

**Pattern:** Add an `**{YYYY-MM-DD} update:**` block below the Rationale rather than editing the original Rationale text in place.
