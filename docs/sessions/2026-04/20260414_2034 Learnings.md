# Conv Learnings - 2026-04-14

## 1. macOS Chrome automation ‚Äî File-input filling via DataTransfer
**Topics:** testing, browser-automation

**Context:** Phase 5 modal uses a hidden `<input type=file>` triggered via a "Choose file" proxy button. Chrome MCP has no native file-picker support.

**Learning:** Use page-JS to construct a File from a string/Blob, wrap in DataTransfer, assign `.files` to the input, and dispatch a synthetic `change` event with `bubbles:true`. React's onChange picks it up and state updates (auto-title, size preview, type dropdown all fire).

**Pattern:**
```js
const file = new File([content], 'name.txt', {type:'text/plain'});
const dt = new DataTransfer();
dt.items.add(file);
fileInput.files = dt.files;
fileInput.dispatchEvent(new Event('change', {bubbles:true}));
```
Reliable way to exercise a file-input flow without native file-picker dialogs.

## 2. Fast path for QA of multi-path endpoints
**Topics:** testing, workflow

**Context:** Phase 5 has 4+ variants (link-JSON, file-multipart √ó ui-via-modal, ui-via-direct-fetch) √ó (persist, render, download).

**Learning:** For bulk QA, bypass the UI entirely and use page-console `fetch` + `new File()` + `FormData`. Indistinguishable to the backend from real modal submissions. Reserve full UI automation for one representative per feature (catches React-integration bugs: change events, state updates, focus handling). Cut Phase 5 verification from ~10 minutes of modal clicking to <2 minutes of scripted POSTs.

## 3. SSR `downloadUrl` pre-compute decouples client from storage backend
**Topics:** astro, ssr, architecture

**Context:** `CommunityTabs` renders `<a href={resource.downloadUrl}>` blindly ‚Äî no branching on r2_key vs external_url.

**Learning:** Pre-computing `downloadUrl` in the SSR loader (`src/lib/ssr/loaders/communities.ts` ‚Üí `downloadUrl: r.r2_key ? '/api/community-resources/{id}/download' : r.external_url`) lets the client render resources without knowing whether they live in R2 or are external. Verified today: uploaded file resource rendered with `href="/api/community-resources/cres-3927168e/download"` pointing at the R2-backed download endpoint, exactly as an external_url would render an outbound link ‚Äî same component, same code path. Pattern should extend to `session_resources` (noted in Phase 9 docs task).

## 4. Static analysis misses SQL column-name drift
**Topics:** testing, d1

**Context:** The Conv 117 `resolveAndAuthorize` used `communities.deleted_at IS NULL`, but communities' schema has no `deleted_at` (it uses `is_archived`). `tsc --noEmit` and `astro check` both reported 0 errors. Runtime 500 only surfaced when actually hitting the endpoint.

**Learning:** Raw SQL strings are invisible to TypeScript. A missing column only surfaces at runtime as a D1 error. Mitigations: Phase 7 auth-matrix tests would have caught this pre-push; `/w-codecheck` could grep for `deleted_at IS NULL` on tables that don't have the column (schema-aware SQL lint). Worth considering as a codecheck enhancement.

## 5. Schema-pattern drift from copy-paste
**Topics:** d1, code-review

**Context:** Conv 117's community resources endpoints appear to have been modeled on an analog (session_resources or similar) that uses `deleted_at`. Communities use `is_archived` ‚Äî a different soft-delete convention.

**Learning:** Tables in this schema split into two soft-delete conventions: `users`, `courses`, `enrollments`, `progressions` use `deleted_at TEXT NULL`; `communities` uses `is_archived INTEGER DEFAULT 0`. When writing a new endpoint that queries a parent table, always grep an existing endpoint that queries *the same parent* rather than trusting an endpoint for a similar-feeling concept. The "10+ places use `is_archived = 0` on communities" grep would have surfaced this immediately.
