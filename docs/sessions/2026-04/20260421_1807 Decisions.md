# Conv Decisions - 2026-04-21

## 1. Stripe Mode Discipline: Local=Test, Staging=Sandbox, Prod=Live
**Type:** Strategic
**Topics:** stripe, docs-infra, cloudflare

**Trigger:** Secret mismatch debugging exposed that Peerloop had no documented rule for which Stripe mode serves which deployment tier; Stripe CLI auth was in Test mode while the staging webhook endpoint was in the Sandbox.

**Options Considered:**
1. Collapse to Test mode only (simple, but CI/parallel testing collides in shared Test-mode data)
2. Collapse to Sandbox only (requires migrating local dev + CLI + `stripe listen` config — full session of work)
3. Keep both with strict role split — local=Test, staging=Sandbox, prod=Live ← **Chosen**

**Decision:** Peerloop uses three fully isolated Stripe environments: Test mode for local dev, Sandbox for staging, Live for production. Hard rule: every `.dev.vars*` file and every `wrangler secret put` invocation is single-mode. Never cross `sk_test_` values from Test mode with Sandbox `whsec_` secrets (or vice versa) — signature verification, API lookups, and webhook routing break silently when modes split.

**Rationale:** Stripe Sandboxes (2024+) are disjoint from Test mode: separate API keys, webhook endpoints, customers, charges, events. Since Peerloop already has the Sandbox provisioned (Dashboard workbench `Alpha Peer LLC sandbox`) and the staging Worker secret is Sandbox-scoped, the durable answer is to codify and document this division rather than collapse it. Option 3 gives each environment clean isolation while requiring no active migration of local dev — existing `.dev.vars` + `stripe listen` flow stays as-is.

**Consequences:**
- New DECISIONS.md §8 entry explicitly enumerating the rule + Conv 144 incident rationale
- `docs/reference/stripe.md` gains §Stripe Mode Discipline (CRITICAL) sidebar + rewrites of §Staging, §Environment Variables, §Connected Accounts (now per-Mode AND per-Env)
- Two follow-up tasks: [VA] audit staging Worker STRIPE_SECRET_KEY is Sandbox-mode (not Test); [VL] rotate leaked Test-mode `sk_test_` (hygiene — does not affect Sandbox or Live)
- Future go-live secret provisioning will follow the same rule: prod gets `sk_live_`/`whsec_` from the Live workbench, never cross-contaminated

## 2. Fix `constructEvent` → `constructEventAsync` Signature Across Stripe Webhook Path
**Type:** Implementation
**Topics:** stripe, cloudflare

**Trigger:** `wrangler tail` exposed that every Stripe webhook to staging Workers was silently HTTP 400'd since Conv 114's migration. Root cause: sync `stripe.webhooks.constructEvent()` dispatches to async-only SubtleCryptoProvider on Workers runtime.

**Options Considered:**
1. Patch at the boundary only (Workers-specific fallback) — leaves Node test path working sync, Workers uses async. More code, more surface area.
2. Switch the wrapper to async everywhere — one-line signature change + `await` at caller. Works identically on Node (tests) and Workers. ← **Chosen**
3. Register a custom crypto provider at init time — forces NodeCryptoProvider on Workers via polyfill. Unreliable (Workers lacks full `crypto` Node API) and fights the SDK's runtime detection.

**Decision:** `src/lib/stripe.ts` `constructWebhookEvent` becomes `async`, delegates to `stripe.webhooks.constructEventAsync()`. Caller at `src/pages/api/webhooks/stripe.ts:64` awaits the call. Comment in wrapper explains the Workers/Node provider split so future-devs don't "optimize" back to sync.

**Rationale:** The async variant is backward-compatible with Node runtimes (`await` on a resolved-sync value is a no-op), so one path works for both test and prod. Test mocks returning sync values pass through `await` cleanly — no test changes needed. Minimal surface, zero runtime cost, clean semantics.

**Consequences:**
- 2-line code fix (plus comment block) deployed to staging Worker as version `254fa8e9`
- 6409/6409 tests pass (including 17/17 stripe webhook tests)
- Unblocks [VS] scenarios S1–S7 live-verification
- Entry added to `docs/as-designed/webhook-miss-resilience.md` §Stripe live observations so future readers understand the history
- Prod Worker has the SAME bug — deploying this fix to prod is a pre-requisite for real Stripe webhooks landing at go-live (flagged as part of PROD sub-block in DEPLOYMENT.PROD)

## 3. Harness Seed-ID Env Overrides (`STUDENT_ID`, `COURSE_ID`, ...)
**Type:** Implementation
**Topics:** testing, stripe

**Trigger:** S1's default target (David + n8n) collided with seed `enr-david-n8n` UNIQUE constraint — harness hard-coded `STUDENT_ID="usr-david-rodriguez"` with no override mechanism. Blocked staging verification for any seed-populated (student, course) pair.

**Options Considered:**
1. Use `stripe-direct-raw` escape hatch with a custom JSON body (works but loses the canonical signing/formatting path)
2. Add env-var fallback to the 6 seed constants at the top of `trigger-webhook.sh` ← **Chosen**
3. Delete the colliding seed row before firing (hacky, data-destructive, breaks subsequent scenarios)

**Decision:** Six constants at the top of `scripts/trigger-webhook.sh` (`SESSION_ID`, `TEACHER_ID`, `STUDENT_ID`, `COURSE_ID`, `CREATOR_ID`, `TEACHER_CERT_ID`) now use `${VAR:-default}` so env-var presence overrides the default. Backward-compatible: existing invocations with no override still get David/n8n/Marcus.

**Rationale:** Three-line change that generalizes the harness for any future [VS]-style staging run, any non-default seed, and cross-student scenarios (e.g., Admin-review staging tests targeting other users). No behavioral change for local dev flows.

**Consequences:**
- Harness now testable against seed rows for Amanda, Sarah, Jennifer, Guy, etc., without needing `stripe-direct-raw`
- Doc update in `webhook-miss-resilience.md` notes the harness improvement and cites it as part of [VS] delivery
