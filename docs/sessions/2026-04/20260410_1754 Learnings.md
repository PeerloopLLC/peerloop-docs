# Conv Learnings - 2026-04-10

## 1. Adapter 13's legacy-access proxy throws with migration guidance
**Topics:** cloudflare, astro, migration

**Context:** `@astrojs/cloudflare@13.1.8`'s `dist/utils/handler.js` wraps every request's `locals` with a non-enumerable `runtime` getter-proxy. Each sub-field (`.env`, `.cf`, `.caches`, `.ctx`) is its own getter that **throws** with a descriptive migration message â€” e.g., `'Astro.locals.runtime.env has been removed in Astro v6. Use import { env } from "cloudflare:workers" instead.'`

**Learning:** Cloudflare adapter maintainers chose a maximum-signal migration path â€” no silent warning fallback. Any code still touching `locals.runtime.env` crashes at the first request with a clear pointer to the replacement. For future major-version bumps, throwing proxies are a strong signal the field is gone, not just renamed.

## 2. `Cloudflare.Env` is the declaration-merge target, not the bare `Env`
**Topics:** typescript, cloudflare, types

**Context:** First `src/env.d.ts` rewrite declared `module 'cloudflare:workers' { export const env: Env }` at the top level with a top-level `interface Env`. Tsc complained `Property 'DB' does not exist on type 'Env'`. Investigation: `@cloudflare/workers-types/latest/index.d.ts` line 12943 declares `declare namespace Cloudflare { interface Env {} }` as the project augmentation point, and line 13126 declares `declare module 'cloudflare:workers' { export const env: Cloudflare.Env }`. The local module declaration was being shadowed, so `cfEnv` resolved to the empty `Cloudflare.Env` interface.

**Learning:** The canonical Cloudflare + TypeScript project pattern is to augment `Cloudflare.Env` by re-declaring the namespace, not to write a bare `interface Env`. Bare `Env` still works for third-party generics like `ExportedHandler<Env>`, but the `cloudflare:workers` virtual module specifically requires `Cloudflare.Env`. Keep `interface Env extends Cloudflare.Env {}` as a top-level alias for backward compatibility.

**Pattern:**
```ts
declare namespace Cloudflare {
  interface Env {
    DB: D1Database;
    STORAGE: R2Bucket;
    JWT_SECRET: string;
    // ...
  }
}
// Backward-compat alias for bare-Env references
interface Env extends Cloudflare.Env {}
```

## 3. Vitest alias-to-mock-file is the durable pattern for virtual modules
**Topics:** testing, vitest, virtual-modules

**Context:** `cloudflare:workers` is a virtual module provided by `@cloudflare/vite-plugin`, registered in `astro.config.mjs` but not in `vitest.config.ts`. Three options considered: alias-to-mock-file, `vi.mock('cloudflare:workers')` in setup, and dynamic import with try/catch in helpers.

**Learning:** Vitest's `resolve.alias` was already being used for two other virtual modules (`astro:transitions/client`, `astro:middleware`), each pointing at a small mock file under `tests/helpers/`. Matching that pattern for `cloudflare:workers` is least-surprising â€” single explicit resolution point, no `vi.mock` hoisting quirks, no async helper init. Matching an existing project pattern is almost always more durable than introducing a new one.

## 4. Centralization investment pays off at the next major-version boundary (blast radius 5)
**Topics:** refactor, architecture, durability

**Context:** Conv 100 Phase 2-prep centralized all Cloudflare env access through 3 helpers (`getEnv`, `getDB`, `getR2`) and migrated ~95 files off direct `locals.runtime?.env?.*` access. Adapter 13's breaking change â€” removing `locals.runtime.env` entirely â€” landed in Phase 2a. A pre-centralization codebase would have had a ~100-file sweep; post-centralization, only 5 files touched `locals.runtime` directly.

**Learning:** Investing in a centralization refactor _before_ consuming a breaking major version produces a dramatic multiplier. The ~95-file churn in Conv 100 reduced Phase 2a's mechanical component to nearly zero. The durability principle is directly validated â€” the "simpler" Conv 100 path would have been to bump the adapter without centralizing, paying it back with a 100-file sweep bundled into an already-risky major-version commit.

## 5. React 19 `server.edge` workaround is fixed upstream in Astro 6
**Topics:** react, astro, vite

**Context:** `astro.config.mjs` had a `vite.resolve.alias` entry mapping `react-dom/server` â†’ `react-dom/server.edge` in PROD builds only, to avoid a `MessageChannel from node:worker_threads` polyfill error (Astro issue #12824). Removing it and running `npm run build` â†’ built cleanly in 7.54s with zero warnings.

**Learning:** Upstream Astro #12824 is fixed in Astro 6 + `@vitejs/plugin-react` 5 + React 19. More broadly: during major-version upgrades, try removing every downstream workaround that references an upstream issue number â€” many have quietly become unnecessary. Isolating the removal in the same commit as the version bump makes bisection trivial.

## 6. Adapter 13 `platformProxy` is gone; split into `remoteBindings` + `CLOUDFLARE_ENV`
**Topics:** cloudflare, config, migration

**Context:** `astro.config.mjs` had `platformProxy: { environment: 'preview', remoteBindings: true }` to power `npm run dev:staging`. Adapter 13's `Options` interface picks only `'auxiliaryWorkers' | 'configPath' | 'inspectorPort' | 'persistState' | 'remoteBindings'` â€” no `platformProxy`. `@cloudflare/vite-plugin/dist/index.mjs` line 7200 confirms the plugin reads `CLOUDFLARE_ENV` to select the wrangler.toml environment.

**Learning:** The old `platformProxy` option is replaced by two orthogonal mechanisms:
- `remoteBindings: true` is now a top-level option on the adapter, forwarded to `@cloudflare/vite-plugin`.
- `environment: 'preview'` has no adapter-level equivalent; set `CLOUDFLARE_ENV=preview` at the CLI (e.g., in the npm script) and the plugin picks the wrangler.toml `[env.preview]` section.

**Pattern:**
```js
// astro.config.mjs
adapter: cloudflare({
  imageService: 'passthrough',
  remoteBindings: process.env.USE_STAGING_DB ? true : undefined,
}),
```
```json
// package.json
"dev:staging": "USE_STAGING_DB=1 CLOUDFLARE_ENV=preview astro dev"
```
