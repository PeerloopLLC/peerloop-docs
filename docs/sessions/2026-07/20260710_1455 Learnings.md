# Conv Learnings - 2026-07-10

## 1. Browser walks produce state equivalent to API producers — dual-producer design validated end-to-end
**Topics:** plato, testing

**Context:** PLATO-SEQ's waypoint architecture posits that a browser walk through pure-UI intents produces the same state a deterministic API producer builds via mocks. This conv is the first end-to-end proof.

**Learning:** All 3 browser segments' captured waypoints were **row-identical to their API producers**: B1 `wp-published`==`flywheel-pre-9` (14 tables), B3 `wp-booked`==`flywheel-pre-14` (8 tables), B4 `wp-certified`==`flywheel` (9 tables). The equivalence holds at every cut point, structurally closing `[FLYWHEEL-WALK-GAP]`.

**Pattern:** Validate a captured waypoint by comparing `sqlite3 COUNT(*)` across the key tables of the browser `.sqlite` vs its API-producer `.sqlite`; a per-table diff flag catches divergence immediately.

## 2. Fully browser-produced waypoints carry the test machine's timezone, not the persona's
**Topics:** plato, testing, timezone

**Context:** The genesis personas deliberately use a cross-boundary tz pair (creator LA, student Tokyo) for `[TZ-TESTS]`, so a booked session renders at different local times per viewer.

**Learning:** Browser *registration* auto-detects the machine's zone client-side (here `America/Toronto`) — the signup UI exposes no tz field. So a browser walk from `wp-fresh` (B1) lands both users in the machine zone, silently collapsing the cross-boundary difference `[TZ-TESTS]` depends on. Waypoints *restored* from an API snapshot (B3/B4) already carry correct persona tzs. Fix for browser-produced waypoints: reconcile `users.timezone` to the persona spec after registration, before capturing. Documented in the snapshots README so the next B1 walker isn't blind.

## 3. Hybrid walk fidelity — real UI for interactive gates, app-endpoint fetch for bulk/repeat
**Topics:** plato, testing, chrome-bridge

**Context:** The course About tab has ~12 fields incl. nested arrays + a thumbnail with no working browser file-upload; the curriculum has 3 modules × 6 fields.

**Learning:** Driving every field via computer-use is slow and fragile. The effective split: drive the genuinely interactive gates via real UI (create-course modal, one add-module form, Publish button, /signup, the booking calendar/slot/confirm flow, the Certify modal) to prove those surfaces work; submit bulk/repeat payloads via the app's own authed fetch endpoints (the same ones the PLATO steps call). This exercises real app code + auth + page rendering while staying reliable — the `[BRIDGE-UPLOAD]` precedent generalized.

## 4. Chrome-bridge interaction gotchas reconfirmed live
**Topics:** chrome-bridge, plato

**Context:** Driving the booking flow surfaced several known-but-worth-reconfirming behaviors.

**Learning:** (a) `[BRIDGE-MEM]` blank-body on the *first client-side* navigation to an authed island route (clicking "Book Session") — a hard reload fixes it. (b) A coordinate `left_click` on the "Continue" button didn't register; a `find` + ref-click did — prefer ref-clicks for buttons that don't respond to coordinate clicks. (c) `[PUB-CHECKLIST-STALE]` fires whenever course detail is written outside the client form (fetch) — the Publishing checklist stays stale until a full reload.

## 5. `plato:snapshot:restore` races the port release after `kill`
**Topics:** plato, tooling

**Context:** B4 needed to restore `flywheel-pre-15` right after stopping the dev server.

**Learning:** The restore script re-checks port 4321 and aborts if still bound; a `kill` + `sleep 2` was not always enough for the socket to release, so the first restore aborted. A `lsof -ti:4321 | xargs kill -9` + re-check + retry succeeded. When scripting stop→restore, verify the port is actually free (not just that `kill` returned) before invoking restore.
