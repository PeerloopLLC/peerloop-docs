# Conv Learnings - 2026-07-16

## 1. A guard built on a file the workflow deletes is a permanent green light
**Topics:** cc-workflow, dual-repo, tooling

**Context:** `[CBG]` asked for a commit-time branch guard. The intuitive implementation — call the existing `conv-branch-check.sh` from `/r-commit` — looked like a 2-line wiring job.

**Learning:** `conv-branch-check.sh` derives its expected branch from `RESUME-STATE.md`, but `/r-start` **Step 7.6 deletes that file** and `/r-end` Step 5 only regenerates it at the very end. For the entire middle of a conv — exactly when `/r-commit` runs — it returns `NO-RESUME-STATE` and skips silently. Wiring it in would have produced a guard reporting "clean" on every commit forever: **worse than no guard, because it looks like coverage**. Verified live before building. The second, deeper problem: it answers the wrong question — RESUME-STATE describes the *previous* conv, while the Conv-371 failure was drift *within* a conv.

**Pattern:** Before reusing an existing checker in a new lifecycle position, **run it in that position and read the verdict**. A checker's inputs have a lifetime; a guard is only as good as its input's availability at the moment it fires.

## 2. "Printed but not verified" is a recurring bug class in this repo
**Topics:** cc-workflow, tooling

**Context:** `/r-end` has **always** printed the code branch in its precomputed block. Conv 371 still committed to the client's `brian-July-7`.

**Learning:** Conv 297 (`[RSTART-DIFFGATE]`) was the same shape — SessionStart printed the branch, nothing cross-checked it, and the first symptom was a colour-token lookup failing mid-sweep. Information that is technically on screen but buried in output nobody reads **is information you don't have**. This is the same root cause as `[CHATSEP]`: the fix isn't printing more, it's making the print load-bearing (a HALT) or removing the noise that buries it.

**Pattern:** When adding diagnostics, ask "what consumes this?" If the answer is "the user, visually, in a wall of output", it will be missed. Either gate on it or don't bother.

## 3. Claude cannot observe the user's UI — config is not the screen
**Topics:** cc-workflow

**Context:** The user repeatedly asked "are you in focus mode?" / "are you now in fullscreen?".

**Learning:** Claude has **no visibility into the renderer, panes, or mode state**. Those questions were only answerable because (a) the command's stdout landed in the transcript, and (b) the choice is persisted to `settings.json`, which is readable. `tui: fullscreen` proves *the setting is set* — it does not prove the screen changed. Reading config and reporting it as UI state is a category error. Screenshots are the only real channel, and even then: **Conv 395 misattributed CC's grey autocomplete suggestion in the input line to the user**, putting words in their mouth from their own screenshot.

**Pattern:** State the epistemic basis explicitly ("I'm reading your config, not your screen"). In screenshots, the input line may contain CC-generated suggestions awaiting TAB — never attribute it to the user.

## 4. The tool's own docs corrupt the tool's own parser
**Topics:** tooling

**Context:** `chat-replay.sh -n 2` returned one exchange instead of two.

**Learning:** The marker detector matched a bare `▸`, so Claude's own prose *discussing* `▸ YOU` / `▸ CLAUDE` counted as conversation headers. Any message about the tool corrupted the tool's paging — including the script's own doc comments. Fixed by anchoring to the full line-start ANSI escape.

**Pattern:** When a text-processing tool's markers are also things people will *write about*, anchor the match structurally (line-start, full escape sequence), never on the bare token.

## 5. Empirical testing killed the plan's headline lever
**Topics:** cc-workflow

**Context:** `[CHATSEP]` opened with four levers; `/focus` was the recommended first move and appeared to win outright (a whole turn's tools → one line).

**Learning:** `/focus` requires the fullscreen renderer, and fullscreen uses the **alternate screen buffer** — which destroys native terminal scrollback *by design*, the same as `vim`. That's a **deal-breaker**, not a preference, and it was only discovered by actually switching. This should have been flagged *before* recommending the switch: it's inherent to how fullscreen TUIs work, not an obscure bug. The eventual winner (`verbose: false` in the classic renderer) was a one-line change that had been sitting untested the whole time, masked because fullscreen was adopted immediately after setting it.

**Pattern:** When two changes are bundled, the cheaper one may be doing the work. Isolate before concluding — and name a change's *inherent* costs when recommending it, not just its benefits.

## 6. After removing 92% of the noise, the remaining noise was Claude
**Topics:** cc-workflow

**Context:** With `verbose: false` live, the screenshot showed **2 lines of tool output and 25 lines of Claude prose**.

**Learning:** The user's complaint was "tool output buries what's meant for me". Solved — and what's left burying it is Claude's own replies. Measured: median ~450 B, spikes of 2–3 KB. There is **no reliable internal gauge**: CLAUDE.md §Explanatory Style already mandates matching reply length to the question, and it was overshot anyway, repeatedly, while actively discussing verbosity. Note the trap: a byte-ceiling rule is the same class of self-check that just failed, and the same class as the **retired QLINT Stop-hook** (Conv 273 chose structural prevention over post-hoc detection).

**Pattern:** Self-policing rules fail where structure succeeds. `AskUserQuestion` fixed malformed questions by making them *impossible*, not by reminding Claude. Deferred by the user.
