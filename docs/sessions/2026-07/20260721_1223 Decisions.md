# Conv Decisions - 2026-07-21

## 1. Fix the task-tools leak by hardening the launcher + restarting VS Code
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** Root-caused `CLAUDE_CODE_CHILD_SESSION=1` (+ `AI_AGENT=…agent`) leaking from VS Code's frozen env into every `peerloop`→`claude` launch, misclassifying each session as a child and withholding the todo/task tools.

**Options Considered:**
1. Harden launcher (`env -u`) + user restarts VS Code ← Chosen
2. Harden launcher only (recurs if VS Code relaunched from a claude shell)
3. VS Code restart only (no durable guard)
4. Explain only, do nothing

**Decision:** Edit `~/.zshrc` `peerloop()` (and, on a follow-up, `spt()`) to strip `CLAUDE_CODE_CHILD_SESSION` + `AI_AGENT` via `env -u … claude …` before launching; user Cmd-Q + relaunches VS Code cleanly after this conv.

**Rationale:** The `env -u` guard is durable and idempotent — it protects regardless of VS Code's env state, so it survives re-pollution. The VS Code restart is what actually clears the current 3-day-old pollution (a running process's env is frozen at launch). Both together = immediate unblock + no recurrence.

**Consequences:** Next `peerloop`/`spt` launch starts as a clean top-level session with the todo tools restored. This conv's session stays child-flagged (tools register at startup). `~/.zshrc` is a home dotfile — not committed by r-end.
