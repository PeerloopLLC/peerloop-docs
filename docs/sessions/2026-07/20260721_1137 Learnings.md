# Conv Learnings - 2026-07-21

## 1. Astro 7 runs `astro dev` as a detached background daemon
**Topics:** astro, cc-workflow

**Context:** During the v7 dev SSR smoke, `npm run dev` returned exit-0 immediately with a banner ("Dev server running at http://localhost:4321 (pid …)  Stop: astro dev stop / Status: astro dev status / Logs: astro dev logs").

**Learning:** Astro 7 spawns the dev server as a detached daemon; the launching `npm run dev` process exits once it's up. Stop it with `npx astro dev stop` (plain `astro` isn't on PATH). Per-request logs go to `astro dev logs`, NOT the launcher stdout — so the authoritative SSR-health signal in a curl smoke is the **response bodies** (full render + no error markers), not the launcher log.

## 2. The Astro-7 dev daemon inherits env vars from its launching npm script
**Topics:** astro, testing

**Context:** `dev:staging` sets `USE_STAGING_DB=1 CLOUDFLARE_ENV=staging` inline; the concern was whether the detached daemon would still see them (astro.config.mjs reads them at config-load).

**Learning:** The daemon (verified via `ps eww <pid>`) inherited both env vars, so `remoteBindings` engaged. Decisive confirmation it read remote staging D1: the same route rendered a different byte size against staging vs local (267 KB vs 314 KB) — a cheap, reliable "is the remote binding actually live" oracle without needing a known fixture.

## 3. `npm update <pkg>` stays conservative when the package doubles as a peer dependency
**Topics:** cloudflare, cc-workflow

**Context:** `npm outdated wrangler` wanted 4.94→4.112 (in-range under `^4.94.0`), but `npm update wrangler` was a no-op.

**Learning:** Our root `wrangler` also satisfies `@astrojs/cloudflare`'s `peer wrangler@^4.83.0`, so npm held it at 4.94 rather than risk the peer graph. The explicit `npm install wrangler@4.112.0` then surfaced the real blocker: wrangler 4.112 `peerOptional`-requires `@cloudflare/workers-types@^5` while we're on v4 — so the "clean lockfile bump" actually drags in a CF-types major. Lesson: when `npm update` silently no-ops a wanted bump, an explicit install reveals the true peer conflict.

## 4. PLATO local-D1 restore fails on a stray empty `.sqlite` left by a prior `dev:staging`
**Topics:** testing, cc-workflow

**Context:** `npm run plato:snapshot:restore -- member-directory` errored "Multiple .sqlite files found in D1 directory... Expected exactly one."

**Learning:** A `dev:staging` run (CLOUDFLARE_ENV=staging) creates a **separate** local D1 file for the staging binding config — empty (~4 KB), because it actually uses remote D1 — alongside the real populated local DB (~1.4 MB). The restore script expects exactly one. Fix: identify the empty placeholder by size and move it aside; restore then targets the single real local DB. (The default `npm run dev` re-uses the populated hash file.)

## 5. AAP root cause is unchanged by the v7 Rust-compiler swap; production is clean
**Topics:** astro

**Context:** Re-scoped `[AAP]` (dev-mode ClientRouter absolute `<script src>`) on the upgraded stack.

**Learning:** Root cause on v7 is `node_modules/astro/dist/vite-plugin-astro/compile.js:22` — `SUFFIX += import "${compileProps.filename}?astro&type=script..."` with `compileProps.filename` absolute. The Rust-compiler swap changed `.astro`→JS compilation but not this vite-plugin dev-mode script-import wiring, so the leak persists identically. Strictly dev-only: `dist/client/` (browser bundle) is leak-free; the `/Users/…` strings in `dist/server/*.mjs` are SSR-internal manifest metadata (rootDir/outDir/.bin/astro), never browser-served. Tell: the `type=style` sub-import is relative, only `type=script` is absolute.

## 6. Frozen Playwright specs are the wrong tool for post-upgrade real-browser validation
**Topics:** testing, astro

**Context:** User asked whether Playwright is a better test than PLATO for post-v7 Astro gotchas.

**Learning:** Architecturally a real-browser automated tool is the ideal shape for SSR→hydration→ClientRouter gotchas (PLATO API-mode never renders; can't see them). But Peerloop's 28 Playwright specs are frozen since Conv 347 and stale against every redesign since — running them yields selector/seed drift-noise, not migration signal. The right post-upgrade real-browser check is a PLATO browser-run on a *current* instance, backed by the already-green jsdom SSR/component/pages tests.
