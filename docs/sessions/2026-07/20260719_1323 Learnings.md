# Conv Learnings - 2026-07-19

## 1. A shared code repo silently breaks every branch-walking tool
**Topics:** billing, tooling, dual-repo, client-collaboration

**Context:** The client commits to `Peerloop` from his own Claude Code instances using the same `Conv NNN:` convention, with colliding numbers. `timecard-day.js` enumerated local branches with no filter.

**Learning:** Repo *sharing* is a property that invalidates an assumption every conv-archaeology tool makes — that `Conv NNN:` means *our* conv N. The contamination was invisible because his commit messages are well-formed and follow our convention; nothing looked wrong. Detection required running the tool and diffing with/without his branch, not reading it.

**Pattern:** Allowlist (`^jfg-dev`), never denylist — a new client branch appears without warning. Gate at the single enumeration choke point (`discoverCandidateBranches()`), and scope the filter to the shared repo only; the docs repo holds the heartbeats and filtering it would zero every timecard.

## 2. Zero billing delta was luck, not safety
**Topics:** billing, verification

**Context:** Comparing timecards with and without `brian-July-7` produced identical totals (765/665/830/845 min) on all four contaminated days.

**Learning:** The totals matched only because his commits fell *inside* windows our own commits already bounded. Day windows are computed from first/last commit timestamps, so a single client commit outside our window would have extended it and inflated a billable day. A "no impact measured" result can be a property of the sample rather than of the system — worth distinguishing before treating an issue as benign.

## 3. Correct local decisions can collide across long-lived branches
**Topics:** merge, orphan-detection, route-migration

**Context:** Conv 392 `[ORPHAN-PURGE]` deleted `CourseTabs.tsx` (389 lines) as confirmed-dead after `/old` routing retired; the client forked at Conv 370, before the purge, and built 8+ commits of tab architecture on it.

**Learning:** Both determinations were right on their own branch — nothing routed to it on ours, it was the centerpiece on his. Git reports this as a modify/delete conflict, but no per-hunk resolution answers it; it is a design question about which architecture survives. Orphan-purge work on a branch that a collaborator forked from earlier creates this class of collision systematically, not incidentally.

## 4. Duplication inside a capped file costs more than verbosity
**Topics:** memory, cc-workflow

**Context:** `/r-prune-memory` targets bloated pointer lines, but the largest single win was the 5-line intro blockquote restating `CLAUDE.md §Memory`.

**Learning:** 910 bytes of the 25 KB SessionStart budget were spent restating a section that loads on every turn anyway. Pointer trimming has a hard floor (~190 B of fixed prefix + retained hook), so it yields ~190 B per line; removing duplication has no floor. Check for content duplicated from an always-loaded file *before* trimming summaries.

## 5. Impact and intent are separable inputs
**Topics:** client-collaboration, process

**Context:** Argued for running the 3-question gate before the client conversation, since gates 1–3 measure impact and only "why" needs him.

**Learning:** The argument was sound on information grounds and incomplete on human grounds — the user pointed out that reading an impact map first primes toward skepticism before hearing the upside. When sequencing analysis against a conversation, receptiveness is a real constraint, not a soft one.
