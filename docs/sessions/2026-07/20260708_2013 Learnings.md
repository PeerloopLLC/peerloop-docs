# Conv Learnings - 2026-07-08

## 1. On a UTC CI host, timezone boundary tests provide zero protection
**Topics:** testing, cloudflare, deployment

**Context:** TZ-TESTS added boundary tests that catch host-local date math by asserting UTC-correct values; they were flip-verified on the Toronto dev host. Then CI turned out to run on `ubuntu-latest` (UTC).

**Learning:** On a UTC host, `getUTC*()` and the local `get*()` accessors return identical results, so a `getUTC*`â†’`get*` regression passes every boundary test in CI. Local (non-UTC) dev is the *only* place those tests bite â€” unless CI deliberately runs under a non-UTC zone. A hostile-TZ CI lane (`TZ=Pacific/Kiritimati`, +14, max divergence) is what makes CI actually enforce the UTC contract.

**Pattern:** GitHub Actions matrix `timezone: ['UTC', 'Pacific/Kiritimati']` + `env: TZ: ${{ matrix.timezone }}` on the test job â€” UTC leg = prod parity, hostile leg = host-local-bug detector.

## 2. The "flip check" is the acceptance test for a timezone-hardening test
**Topics:** testing

**Context:** The whole TZ-MODEL suite stayed green through Phase 3 despite the fixes being unprotected, because no assertion was tz-sensitive.

**Learning:** A test only protects a UTC-date fix if temporarily reverting the fix (`getUTC*`â†’`get*`, `Date.UTC`â†’`new Date`) makes it go **red** on the dev host. A test that stays green when the fix is reverted gives false confidence â€” the exact trap being fixed. Every P1 test this conv was flip-verified before being accepted.

## 3. `fillMissingDates`-style bucketers are best protected by lint, not runtime tests
**Topics:** testing

**Context:** 7 analytics endpoints have near-identical date-bucket generators using `setUTCDate`/`setUTCMonth`.

**Learning:** Their date keys derive from `toISOString()` (always UTC) over an epoch-based range, so the `setUTC*` fix is defensive hygiene with negligible observable divergence â€” a flip-protective runtime test would be contrived and low-value. The exact regression (`setUTCDate`â†’`setDate`, `getMonth` without UTC) IS caught statically by `lint:tz` (verified: flipping a bucketer â†’ guard fails). Lint is the right protection layer for hygiene; reserve runtime tests for real arithmetic logic (e.g. `getPeriodDates` last-month year rollover).

## 4. PLATO was timezone-blind by construction
**Topics:** testing, video

**Context:** Assessing whether PLATO/browser tests would catch a tz display bug.

**Learning:** PLATO's runtime-created users had no timezone (â†’ UTC fallback), so every session-touching browser walk exercised only the fallback path â€” never real per-viewer zone conversion. Seeding cross-boundary zones on the genesis flywheel actors (creator `America/Los_Angeles`, student `Asia/Tokyo`) turns existing walks into real tz tests at near-zero cost. No PLATO/E2E assertion pins a rendered local time, so a display regression still needs the manual checklist (or an automated browser-tz test, deferred).

## 5. In a dual-repo project, `.claude/` hooks live in the docs repo â€” scope negative checks to `$CLAUDE_PROJECT_DIR`
**Topics:** cc-workflow, dual-repo

**Context:** The TZ-TESTS review agent reported `lint:tz` was enforced via a Claude-Code PreToolUse hook (`.claude/hooks/pre-commit-lint-tz.sh`). While hardening TZ-LINT-CI I "verified" it didn't exist â€” but I ran the `ls .claude/hooks/` / `grep .claude/settings.json` checks from the **code** repo cwd (`~/projects/Peerloop`), where they're absent. The r-end docs agent caught it: the hook actually **exists and is registered in the docs repo** (`$CLAUDE_PROJECT_DIR/.claude/hooks/pre-commit-lint-tz.sh` + docs-repo `.claude/settings.json`). The agent was right; my negative conclusion had a repo-scoping error.

**Learning:** A "it doesn't exist" conclusion is only as good as the path you checked. In this dual-repo setup, ALL Claude-Code config (`.claude/hooks`, `.claude/settings.json`, `$CLAUDE_PROJECT_DIR`) lives in **peerloop-docs**, not the code repo â€” so a check for a hook/setting must target the docs repo explicitly. Negative findings warrant the same source-of-truth rigor as positive ones; verify against the correct repo before recording "absent". (The TZ-LINT-CI work stayed valid regardless â€” CI gates human/IDE/PR paths the CC hook can't, plus the hostile-TZ test matrix â€” but the recorded *rationale* was wrong until corrected here.)

## 6. Deferring a scope-extension after an audit beats blanket-applying it
**Topics:** testing, cc-workflow

**Context:** TZ-LINT-CI's task text said "extend the scan to src/components, .astro, src/emails, workers."

**Learning:** A dry-run count first showed emails/workers = 0 violations (safe) but components = ~60 (client-side local-tz, a different concern) and .astro = 6 (need per-line triage). Blanket-applying per the task text would have turned `lint:tz` red on ~66 mostly-legitimate lines. Surfacing the counts before editing let the guard stay green and split the real audit into its own task.
