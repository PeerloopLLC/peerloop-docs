# Conv Learnings - 2026-07-10

## 1. PLATO `completeSession()` is order-independent by construction
**Topics:** testing, plato, booking

**Context:** Designing `complete-sessions` to complete 3 discovered sessions, worried that completing them in `GET`-returned DESC order (vs the fused step's ASC order) would mis-assign frozen `module_id`s.

**Learning:** `src/lib/booking.ts completeSession()` skips `module_id` when earlier scheduled sessions still exist (out-of-order completion), then `backfillModuleIds()` assigns them once earlier sessions complete. So completing in ANY order lands the identical final state (all 3 with correct module_ids, enrollment completed on the last). This is what makes a discover-then-complete API bridge robust — no ordering constraint.

---

## 2. PLATO Model B — no cross-step context; the DB is the only shared state
**Topics:** testing, plato

**Context:** `complete-sessions` needs the 3 booked session IDs that `book-sessions` produced.

**Learning:** `api-runner.ts` clears `this.context` at the start of every `executeStep` ("Model B eliminates cross-step carry state. The DB IS the state."). A step cannot read a prior step's `provides`; it must re-discover from the DB via a GET. This is also exactly what the restore-then-bridge flow needs (after a snapshot restore there's no in-memory context at all), so the discovery pattern is doubly correct.

## 3. [BRIDGE-MEM] reinforced — fetch-based auth change needs storage-clear + hard reload
**Topics:** plato, browser-run, chrome-bridge

**Context:** After granting Mara's creator role via `PATCH` + `dev-login` (all via `fetch`), `/creating` still showed "Creator Access Required".

**Learning:** DB grant persisted (`can_create_courses=1`) but the client island held a stale auth cache. Fix: `localStorage.clear(); sessionStorage.clear(); location.reload()` — then the island re-reads auth and the studio unblocks. Also: a stale **httpOnly** JWT session (invisible to `document.cookie`) from a prior dev-login makes `signup.astro` redirect `/signup`→`/`; clear it with `POST /api/auth/logout` before registering a fresh user.

## 4. Producing + verifying a waypoint snapshot without touching local D1
**Topics:** testing, plato

**Context:** Wanted to generate/verify the 4 waypoint snapshots without overwriting local D1.

**Learning:** `plato:restore` = API-run (regenerates the `.sqlite` from current code + runs its verify assertions) THEN restores into local D1. The API-run half alone — `PLATO_INSTANCE=<name> npx vitest run tests/plato/api/plato-scenarios.api.test.ts --testNamePattern="run <name>"` (the dynamic instance runner) — produces the snapshot (in-memory better-sqlite3 → serialize) and asserts every action + verification, WITHOUT touching local wrangler D1. Restore only when a browser walk needs the state live.

---

## 5. `plato:split` partitions the walkthrough by `coversStepActions`
**Topics:** plato, tooling

**Context:** Splitting the flywheel after decomposing complete-course into two steps + two intents.

**Learning:** `plato-split.js` routes each walkthrough intent to the pre/post segment by checking whether its `coversStepActions` step-names fall in the pre-step set. So a decomposed intent MUST reference the right step (`book-sessions.*` vs `complete-sessions.*`) for the segment partition to be clean. Confirmed: pre-14 got the booking intent, post-14 got the completion intent.
