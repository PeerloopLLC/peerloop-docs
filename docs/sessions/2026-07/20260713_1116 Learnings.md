# Conv Learnings - 2026-07-13

## 1. Orphaned .ts modules survive migration like orphaned components — but the .tsx detector misses them
**Topics:** orphan-detection, dead-code, tooling, testing

**Context:** Conv 392's Category-A deletion left stray dead `.ts` utils/types/barrels; the `[ORPHAN-DETECT]` script only reports `.tsx`/`.astro`.

**Learning:** The detector's `reachable` BFS already traverses all `.ts` — it just filters the *report* to components. A `.ts` variant is cheap to derive, but ONLY safe when scoped to `src/components/**`: component-dir `.ts` is reachable only via routes, same as components. `src/lib/**` has additional entry points (`middleware.ts`, `workers/**` importing `src/lib/**`, config, tests) that a pages-rooted reachability sweep can't see — running the sweep there would false-positive on every worker/middleware-only lib file.

**Pattern:** Reachability-from-routes is authoritative for `src/components/**` (both `.tsx` and `.ts`); it is NOT authoritative for `src/lib/**`.

## 2. "live importers: NONE" ≠ safe-to-delete — parked orphans are still type-checked
**Topics:** dead-code, tsc, deletion-safety, testing

**Context:** Several dead barrels were imported by *other* orphans (parked Category-B marketing pages, the creators barrel-chain).

**Learning:** A file with zero *reachable* importers can still be imported by a parked/unreachable orphan that Vite tree-shakes at build but `tsc` still compiles. Deleting it dangles that orphan → tsc breaks. Must check ALL importers (reachable + orphan + test), not just live ones. The 🟡-DANGLES classifier caught 7 such files and kept them with the parked set.

**Pattern:** Deletion-safety = "zero importers of ANY kind," not "zero route-reachable importers." tsc is the final arbiter regardless.

## 3. A "needs-a-look" orphan can be a wiring gap (missing page), not dead code
**Topics:** orphan-detection, wiring-gaps, moderator-invite

**Context:** `invite/ModeratorInvite` was route-unreachable like the other 3 Category-C orphans, but its backend (invite email + token APIs) was fully live.

**Learning:** Route-unreachability alone doesn't tell you whether a component is *dead* or *waiting for a page that was never built*. The tell: trace what SHOULD reach it. The admin invite email links to `/invite/mod/{token}`; no page existed there → the component is the missing piece of a shipped-but-broken feature, and the fix is to WIRE it, not delete it. On staging (RESEND live) this was a real 404'd invite link.

**Pattern:** For an orphan with live adjacent infra (an API, an email, a link target), grep the link/redirect target for a page route before concluding "dead."

## 4. Prior closure-analysis KEEP rationales decay — re-verify importers at delete time
**Topics:** dead-code, decision-decay

**Context:** Conv 392 explicitly *kept* `courses/course-tabs/types.ts` because "live community tabs import it." By Conv 393 that importer was gone (zero importers), so the keep-rationale had silently expired.

**Learning:** A "keep because X imports it" decision is a hypothesis pinned to a snapshot; when X later changes, the rationale expires without notice. Re-run the importer check at deletion time rather than trusting a prior conv's closure note.

---

## 5. Tooling-data .ts reads as "dead" to an import-graph detector
**Topics:** orphan-detection, false-positives, tooling

**Context:** `icons/icon-provenance.ts` has zero code importers but is consumed by `scripts/prov-sweep.ts` (read as a file, `npm run prov:sweep`) and is the documented source-of-truth (`_INDEX.md`, decision docs).

**Learning:** An import-graph reachability detector flags data/config files that tooling reads-as-a-file (not import) as unreachable. These are legitimate keeps — the `.ts`-detector equivalent of a `KNOWN_ORPHANS` allowlist entry.
