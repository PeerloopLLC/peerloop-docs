# Conv Decisions - 2026-07-20

## 1. Adopt eslint-plugin-jsx-a11y at `warn` (upstream 6.10.2 + `overrides` peer pin), not the fork or Oxlint
**Type:** Strategic
**Topics:** eslint, astro, testing, tooling

**Trigger:** [A11Y] (from Conv-397 RDOC: "zero a11y linting anywhere") — pick a permanent a11y linter on ESLint 10; the obvious pick peer-caps at eslint ^9.

**Options Considered:**
1. Upstream `jsx-a11y@6.10.2` + `overrides` peer pin ← Chosen
2. es-tooling fork `eslint-plugin-jsx-a11y-x` (native eslint-10, but drops `.astro` coverage + young 0.x + ESM-redesigned API)
3. Oxlint (separate Rust linter, no eslint-peer coupling, but a whole parallel tool)
4. Hold for upstream (#1079/#1081 stalled ~5mo, no ETA)

**Decision:** Upstream + a one-line self-healing `overrides` peer pin, `recommended` set, all at `warn`, covering `.tsx` (100 findings) + `.astro` (0, free via astro plugin).

**Rationale:** Only option covering both file types with native eslint-plugin-astro integration; canonical package + huge userbase; the override is trivial and self-heals (delete when upstream ships eslint ^10). Mirrors the react-hooks posture we already run. The research confirmed the apparent "clean fix" (fork) silently drops `.astro`.

**Consequences:** `eslint-plugin-jsx-a11y` devDep + `overrides` in package.json; jsx-a11y wired into eslint.config.js at warn; lint gate stays green (100 warnings, 0 errors); full 5-gate baseline verified. Committed (code f561cdae, docs c26c6b5). Decision routed to `docs/decisions/06-testing-ci.md § A11Y`. The 100 warnings become the [A11Y] triage backlog; deferred: escalate recommended→strict, whether to ever gate at error, drop the override when upstream ships.
