## 1. HW-SUBMIT-UI placement — enrolled-only Explore tab (Option A)
**Type:** Implementation
**Topics:** ui-ux, courses, navigation

**Trigger:** The Conv-383 regression: `HomeworkTab` (student submission form) was mounted by no live page.

**Options Considered:**
1. Enrolled-only Explore tab → `/course/[slug]/homework` ← Chosen
2. Enrolled Sessions-cluster peer (alongside My Sessions/Book/Join)
3. Fold homework into the existing Modules tab (assignments have `module_id`)

**Decision:** Add "Homework" to the Explore strip only when enrolled; bounce non-enrolled like `/sessions`.

**Rationale:** `HomeworkTab` is already a complete standalone tab UI; a graded obligation deserves top-strip discoverability; "My Sessions" already proves enrolled-gated course tabs work.

**Consequences:** First enrolled-only entry in a strip that was previously pure-public browse; `buildCourseExploreTabs` gained an `isEnrolled` param.

## 3. CAF compute approach — exact-lazy batch (Option 2)
**Type:** Strategic
**Topics:** availability, courses, performance

**Trigger:** "A user won't mind waiting for an answer."

**Options Considered:**
1. Cheap proxy JOIN (eager, approximate — over-counts fully-booked teachers)
2. Exact, lazy batch — SSR untouched, computed on toggle, boolean map, v1 no cache ← Chosen
3. Denormalized precompute (new table + cron + invalidation, multi-conv)

**Decision:** Option 2 — exact per-course availability computed on demand via a batch endpoint, cached in the catalog island.

**Rationale:** Latency tolerance removes Option 2's only downside, and reframes the others: accuracy is the thing worth waiting for (kills the proxy); the precompute exists to avoid a wait the user accepts (de-justified). Respects the codebase's deliberate "defer availability" decision (no SSR hot-path regression). Simplification: drop the KV cache from v1.

**Consequences:** New public batch endpoint; the availability filter is the one browse filter that needs a round-trip (loading state), but it composes with the client-side filters via a boolean map.

## 4. CAF window key — reuse `availability_window_days` → 14
**Type:** Implementation
**Topics:** platform-stats, config, availability

**Trigger:** Need an admin-configurable window; a key with the same meaning already exists.

**Options Considered:**
1. Reuse `availability_window_days`, default 30/28 → 14 (shared with the detail-page preview) ← Chosen
2. Separate `browse_availability_window_days` (detail preview stays 30)

**Decision:** Reuse the existing key; one "availability window" concept.

**Rationale:** Simplest, most consistent — what you filter by equals what you see on the detail page.

**Consequences:** The course-detail availability preview also moves 30→14. The seed change only affects fresh DBs — existing staging keeps 30 until an admin sets it via the new `/admin/availability-settings` UI (no SQL needed).
