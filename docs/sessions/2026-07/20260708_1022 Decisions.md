# Conv Decisions - 2026-07-08

## 1. Phase-0 tz NULL semantics — nullable column + UTC-labelled fallback + opportunistic login-capture
**Type:** Implementation
**Topics:** timezone, database, d1, auth

**Trigger:** [TZ-MODEL] Phase 0 start — the Conv-371 Model-A decision deferred the "existing-user backfill default / NULL semantics" sub-decision to here. Key scoping fact established: **production is undeployed**, so "existing users" = seed/test only; the choice mainly governs the display/email *fallback*, not a real backfill.

**Options Considered:**
1. `NOT NULL DEFAULT 'America/New_York'` (or UTC) — every row always has a value, no null-handling
2. **Nullable `TEXT` + UTC-labelled fallback on render + opportunistic login-capture** ← Chosen

**Decision:** `users.timezone TEXT` is **nullable**; a NULL renders as UTC (labelled) via the shared formatters; `captureTimezoneIfMissing()` opportunistically backfills a user's tz from their browser at next login.

**Rationale:** A `NOT NULL DEFAULT` would silently stamp a *wrong* zone (ET/UTC) onto real users and make "never captured" indistinguishable from "genuinely UTC". Nullable keeps "unknown" honest, the UTC-labelled fallback is safe/deterministic, and login-capture converges real users to their true zone without a migration. Cheap because prod is undeployed (backfill scope = seed rows only).

**Consequences:** Seed rows backfilled explicitly (all ET; Guy Rymberg → `Asia/Jerusalem` fixture). Shared `formatSessionTime`/`formatSessionDate` must handle `null → UTC-labelled`. `captureTimezoneIfMissing()` added to `current-user.ts` login path.

## 2. Phase-0 capture UX — store raw detected IANA + reuse the 12-entry COMMON_TIMEZONES picker
**Type:** Implementation
**Topics:** timezone, ux, astro

**Trigger:** [TZ-MODEL] Phase 0 second deferred sub-decision — how to capture tz at signup and let users edit it in Settings.

**Options Considered:**
1. Constrain capture to a fixed whitelist (drop anything off the 12-entry list)
2. **Store the raw detected IANA zone; reuse the existing 12-entry `COMMON_TIMEZONES` picker, injecting an out-of-list detected zone as an extra option** ← Chosen

**Decision:** Signup detects `Intl…resolvedOptions().timeZone` and stores it raw (validated by `isValidTimezone()`); Settings reuses the existing 12-entry picker and injects the user's detected zone when it isn't already in the list.

**Rationale:** Detection already yields a precise IANA zone — discarding it to force a whitelist loses fidelity. Reusing the established picker (same pattern as the profile-field/nav_layout precedent) avoids new UI; injecting the detected zone keeps exotic zones editable without a 400-entry dropdown.

**Consequences:** `SignupForm` → `register.ts` capture path; `me/profile.ts` GET/PATCH whitelist+validation; `ProfileSettings` `TimezoneSelect`; `isValidTimezone()` ICU validator added to `timezone.ts`.

## 3. Phase-1 canonical display-threading pattern — middleware `Astro.locals.userTimezone` + shared helpers + `useUserTimezone()` hook
**Type:** Strategic
**Topics:** timezone, astro, ssr, architecture

**Trigger:** [TZ-MODEL] Phase 1 needs the viewer's stored tz on ~40 display surfaces split between SSR `.astro` and client `.tsx` islands, without the UTC→local hydration flash.

**Options Considered:**
1. `useCurrentUser` in each leaf island — flashes UTC→local after hydration; SSR surfaces unserved
2. Reuse `CourseHeader`'s existing `data-session-time` browser-local rewrite script — REJECTED (it's Model B; would disagree with Phase-2 emails)
3. **Middleware `resolveUserContext` → `Astro.locals.userTimezone` (following the `resolveNavLayout` precedent) + shared `formatSessionTime`/`formatSessionDate` + `useUserTimezone()` hook** ← Chosen

**Decision:** Extend the existing per-user middleware query to also `SELECT timezone`, publish `Astro.locals.userTimezone` for SSR surfaces, and a `useUserTimezone()` hook (composing hydration-safe `useCurrentUser`) for islands. Both call shared `formatSessionTime`/`formatSessionDate` (tz `null` → UTC-labelled). This is the pattern replicated across all ~40 sites.

**Rationale:** `resolveNavLayout` (Conv 356) is the exact precedent — one round-trip, flash-free, server-resolved. The browser-local `data-session-time` reuse was rejected because "follow the established pattern" must be checked against the *decision* (Model A: stored tz everywhere incl. email), not merely "a pattern exists" — browser-local SSR would disagree with Phase-2 recipient-local emails.

**Consequences:** `middleware.ts` + `env.d.ts` (`App.Locals.userTimezone`); `formatSessionTime`/`formatSessionDate` + `useUserTimezone()` added; +8 unit tests. DOM-verified end-to-end on the student slice (`Asia/Tokyo` stored tz rendered `7:00 PM` for a `10:00Z` session). Teacher slice reused the proven mechanism (not re-DOM-verified).

## 4. Phase-1 slice decomposition — cohesive vertical slices, one slice per commit checkpoint
**Type:** Strategic
**Topics:** timezone, planning, cc-workflow

**Trigger:** ~40 display sites is too large for one conv; needed a decomposition that both fits per-conv scope and produces verifiable checkpoints.

**Options Considered:**
1. Convert all ~40 sites in one pass, commit once
2. **Decompose into cohesive role/domain slices (Foundation → Student → Teacher → Booking → Admin → Messages → misc stamps), one slice per commit checkpoint** ← Chosen

**Decision:** Foundation first (middleware + helpers + hook + tests), then Student, Teacher, Booking, Admin/mod, Messages, and misc date-only stamps — decomposed by which role actually sees the surface, each slice a standalone commit. This conv shipped Foundation + Student + Teacher.

**Rationale:** Vertical role-slices are individually verifiable (the Student slice was DOM-verified end-to-end before moving on) and map to who sees each surface — which is also what disambiguates out-of-slice code (e.g. `AllSessionsTabContent` serves creator/admin → Admin slice, not Student). One-slice-per-commit gives clean checkpoints and keeps per-conv scope bounded.

**Consequences:** Three commits this conv (`da83e7d9` Phase 0, `2f662f1d` Foundation+Student, `b1a17045` Teacher). Remaining slices (Booking mixed-zone pairs + calendar day-cell, Admin/mod, Messages day-bucketing, misc stamps) queued in `plan/tz-model/README.md`; resolved the `⚠️ TZ-AUDIT #10` deferral marker in the Teacher slice.
