## 2. Adopt Model A — store `users.timezone`, render viewer-local everywhere
**Type:** Strategic
**Topics:** timezone, database, architecture, ux

**Trigger:** [TZ-AUDIT] meta-finding — no per-user tz model is the root cause of the recurring TZ-display pain.

**Options Considered:**
1. **A — store `users.timezone` (IANA), render app + email in each viewer's stored tz** ← Chosen
2. B — browser-local everywhere, labeled (no schema; app-only; emails stay UTC)
3. C — UTC everywhere, labeled (simplest/deterministic; users self-convert)

**Decision:** Model A. Add `users.timezone`; render app and email in the viewer's stored tz.

**Rationale:** A scheduling product must show each user *their* local time, including the booking-confirmation email (only A can localize emails); pre-launch is the cheapest time to add the column. Multi-conv scope accepted.

**Consequences:** New phased block **[TZ-MODEL]** (Phase 0 schema+capture+backfill → 1 display ~40 sites → 2 email → 3 cleanup). Two novel sub-decisions deferred to Phase 0 start: existing-user backfill default, capture UX. Interim B3 "UTC" email label will be replaced by recipient-local in Phase 2.

> **Insight:** When a bug class survives repeated site-by-site sweeps, the missing piece is usually a model/invariant, not more patched sites. Adding one authoritative timezone-per-viewer collapses three divergent render zones into one — and it's the only shape that reaches non-browser surfaces (email), which client-side conversion can never touch.
