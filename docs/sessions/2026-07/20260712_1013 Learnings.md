# Conv Learnings - 2026-07-12

## 1. D1 caps compound SELECT terms below 70
**Topics:** cloudflare, d1, testing

**Context:** Building a full staging-vs-`0001` column diff via a 70-way `UNION ALL` of `pragma_table_info('t')` calls.

**Learning:** D1 rejects it with `too many terms in compound SELECT: SQLITE_ERROR [code: 7500]`. Split into batches (≤35 terms) — but note the batched version also timed out on wrangler round-trip latency. Prefer per-table literal-arg probes or dump `sqlite_master.sql` and diff locally.

---

## 2. `wrangler d1 execute` / `wrangler tail` write results to stderr
**Topics:** cloudflare, tooling

**Context:** Repeated "0 staging columns" results despite valid queries.

**Learning:** wrangler writes command *results* to stderr, so `2>/dev/null | grep` silently drops them. Always capture with `2>&1`. Also: the pragma **table-valued** function with a *column* argument (`FROM sqlite_master m, pragma_table_info(m.name)`) is NOT supported on D1 — only *literal* args work.

---

## 3. `timeout` is not available on macOS
**Topics:** tooling, dual-repo

**Context:** Two `wrangler tail` captures produced only 1 line and no cron logs.

**Learning:** macOS has no `timeout` binary (it's `gtimeout` from coreutils). `( timeout N npx wrangler tail ... )` dies immediately with "command not found." Use a background process + explicit `kill` to bound a tail's duration, or install coreutils.

---

## 4. The staging RESEND_API_KEY is a send-only restricted key
**Topics:** resend, deployment

**Context:** Trying to read Resend delivery status (`last_event`) to machine-confirm inbox delivery.

**Learning:** The key sends fine (POST /emails → 200 + id) but returns 401 `restricted_api_key` on GET /emails/:id. So delivery-event readback isn't possible via the API — rely on an inbox check or the Resend dashboard. A `200 + message id` still proves the send passed key + verified-domain + recipient gates.

---

## 5. A deployed scheduled Worker has no on-demand trigger
**Topics:** cloudflare, deployment, testing

**Context:** Verifying the cron fires reminders on staging.

**Learning:** The cron worker exposes only `scheduled()` (no `fetch` handler), and `wrangler dev --test-scheduled` is local-only. To verify a deployed cron, arrange a due row and wait for the next `*/15` tick, observing a DB effect (here `reminder_24h_sent_at`). `dev-login` is also blocked on staging (gated on `import.meta.env.DEV`, false in a build).
