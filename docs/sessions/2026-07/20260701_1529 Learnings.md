# Conv Learnings - 2026-07-01

## 1. Astro.locals threads cleanly from a layout to its slotted children
**Topics:** astro, ssr

**Context:** [LAYOUT-MODE] Phase A needed one per-user layout value to reach both `AppLayout` (wrapper direction) and `SubNav` (its own orientation), but `SubNav` is rendered by the PAGE into `AppLayout`'s `sub-nav` slot — AppLayout can't prop-drill to it.

**Learning:** Setting `Astro.locals.x` in `AppLayout`'s frontmatter is visible to a slotted child component that reads `Astro.locals.x` — AppLayout's frontmatter runs before its `<slot>` renders the child. Verified empirically (temp-flipping the value flipped the rendered orientation of both). Use a safe fallback (`Astro.locals.x ?? CONSTANT`) so any non-AppLayout render path still works.

**Pattern:**
```
AppLayout: Astro.locals.navLayout = resolved;
SubNav:    const layout = Astro.locals.navLayout ?? SUBNAV_LAYOUT;
```

## 2. A section-INDEX tab (bare-base href) stays active on sibling routes once longer matches are removed
**Topics:** navigation, astro

**Context:** After [SNAV-TOP] Phase 2a moved the journey out of SubNav, the Explore SubNav had no item matching `/course/[slug]/sessions`. "About" (href = the bare `/course/[slug]`) then won the prefix-match by default on every sub-route.

**Learning:** Prefix-matching active-state needs an `exact` opt-out for the index tab whose href is a prefix of all siblings. In the old zoned rail a longer journey item always out-matched it, masking the bug; removing those items exposed it.

**Pattern:** `exact?: boolean` on the SubNav item; matcher: `item.exact ? path === href : matchHref(href, path)`.

## 3. `overflow-x: auto` forces `overflow-y` to compute as `auto` (CSS spec)
**Topics:** css

**Context:** The `CourseJourneyStepper` `<ol>` had `overflow-x-auto`; a 1px sub-pixel content overflow (scrollH 121 > clientH 120) rendered a spurious vertical scrollbar.

**Learning:** Per spec, a non-`visible` overflow on one axis makes a computed-`visible` other axis behave as `auto`. Pin the unwanted axis: `overflow-y: hidden`.

## 4. The shared SubNav top strip was non-responsive at md+ (overflow, not wrap)
**Topics:** css, navigation

**Context:** 7 Explore tabs (~962px) exceeded the ~948px course content card; `md:flex md:flex-row` had no wrap and `overflow-x: visible`, so they spilled even at desktop width.

**Learning:** The top strip needs `md:flex-wrap` (chosen over horizontal-scroll: wrap keeps all 7 discoverability-important tabs visible). Mobile keeps the equal-width 2-col grid.

## 5. ListingShell's "right-panel" slot actually renders on the LEFT
**Topics:** navigation, layout

**Context:** Assessing the client's "kill the left panel on listing pages" request.

**Learning:** `/courses`, `/members`, `/communities` use `ListingShell`, whose filter slot is named `right-panel` but renders as a 320px LEFT aside (`order-1`, before the listing column). So the listing pages DO have left panels, and the client's listing request is NOT yet done — it's a distinct workstream ([LAYOUT-MODE] Phase D) with net-new "filters on top" UI.

## 6. Twitter's IA, mapped precisely, supports a rail for RICH sub-navs
**Topics:** design, navigation

**Context:** The client cites Twitter as the north star for "no left panels."

**Learning:** Twitter's left rail is its MAIN menu (= Peerloop's `Sidebar`); its sub-menus are top single-row horizontal-scroll and only 2–4 items. Peerloop's section sub-nav is 7 tabs + a 4-step journey with hierarchy — richer than any Twitter sub-nav — so "be like Twitter" works for THIN sub-navs and breaks on RICH ones. That is the non-arbitrary justification for the desktop-rail reserve.

## 7. This machine's Chrome bridge doesn't apply window resizes to the page viewport
**Topics:** plato, testing

**Context:** `resize_window` to 900px, but `window.innerWidth` stayed 1302.

**Learning:** Resize verification of responsive behavior is unreliable on this bridge (consistent with the Conv 354 note). DOM-measure (`scrollWidth`/`clientWidth`/`getComputedStyle`) at the current width instead of trusting a resized viewport.
