# Conv Learnings - 2026-07-09

## 1. A PreToolUse hook can be "registered" yet completely inert
**Topics:** cc-workflow, testing

**Context:** [TZ-HOOK-CHECK] — `pre-commit-lint-tz.sh` was registered, matched correctly, ran the right lint, and computed the right `deny` — but never blocked anything for ~3 months (since Conv 091).

**Learning:** A PreToolUse permission decision is honored ONLY if the emitted JSON includes `hookSpecificOutput.hookEventName:"PreToolUse"`. Omit it (or emit malformed JSON) and exit 0 → the decision is silently ignored and the tool proceeds. "Registered + matches + runs" is NOT "enforces" — the emit shape is a separate, invisible failure point. Verify a guard by observing a real block, not by reading its code.

**Pattern:**
Build deny/ask JSON with `jq -nc` (mirrors `guard-dangerous-bash.sh`); never heredoc multi-line text into a JSON string:
```bash
jq -nc --arg r "$MSG" '{hookSpecificOutput:{hookEventName:"PreToolUse",permissionDecision:"deny",permissionDecisionReason:$r}}'
```

## 2. A dead guard hides its own downstream bugs
**Topics:** cc-workflow

**Context:** The hook's case-insensitive matcher (`Peerloop.*commit`) also matched `peerloop-docs` commits — but this was invisible for months because the hook never blocked anything. Fixing the `hookEventName` emit is what *exposed* the matcher over-scope.

**Learning:** When a guard is non-functional, bugs in its *other* logic (matcher scope, etc.) produce no observable effect and accumulate silently. Re-enabling a long-dead guard can surface a second layer of latent bugs — audit the whole guard, not just the fix.

## 3. Line-scoped grep lies about multi-line calls — verify at the byte level
**Topics:** testing, cc-workflow

**Context:** [TZ-LINT-SCAN2] — a `grep -v timeZone` (line-scoped) twice overstated "real bugs" because `toLocale*` calls put `timeZone` on an adjacent line or in a variable (`timeOpts`). Also mis-read the fixed hook's output as "invalid JSON" when it was actually the session hook intercepting my own diagnostic command.

**Learning:** For anything spanning multiple lines (options objects, template literals, hook I/O), a per-line grep is unreliable. Confirm with multi-line context (`-A3`), `od -c` on real bytes, or by reading the surrounding source before asserting a finding. Own and correct overstatements promptly.

## 4. The tz-lint FAIL patterns are server-oriented; client code is a different concern
**Topics:** testing, timezone

**Context:** Extending `lint-timezone.sh`'s `new Date()`/local-accessor patterns to `src/components` would flag ~65 hits, ~all benign (calendar UI in browser-local time, tz-safe absolute-instant comparisons `Date > Date`, DB-timestamp writes).

**Learning:** `getNow()`/UTC-math enforcement is for the UTC Worker. Client components legitimately use browser-local time. The genuine client-tz risk is a *display* one — raw `.toLocaleDateString/TimeString()` without a `timeZone`, bypassing the viewer-tz formatters — which is a different pattern the FAIL scan doesn't detect. A line-based lint is too blunt to distinguish them (it can't see a `timeZone` in a variable), so it would be pure exemption-noise. Declining a scan extension can be the correct, documented outcome.

## 5. Testing-library getByText matches an element's full text
**Topics:** testing

**Context:** `getByText('9:00 AM - 10:00 AM')` failed even though the string rendered — it's a bare text node inside a `<p>` that also holds the date + bullet.

**Learning:** `getByText`/`getAllByText` match an ELEMENT whose (normalized) own-text equals the matcher. For a substring inside a larger element, use `{ exact: false }` or a regex. (Also: `toLocaleTimeString('en-US', …)` renders a REGULAR space before AM/PM in this Node/ICU build, not U+202F — verified by char codes.)
