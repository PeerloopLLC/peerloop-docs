# Conv Learnings - 2026-07-21

## 1. "Task tools down" = session misclassified as a child session via a leaked env var
**Topics:** cc-workflow, dual-repo

**Context:** `TodoWrite` + `TaskCreate/List/Update/Get` had been unavailable across whole convs (noted in RESUME-STATE 402), degrading task tracking to hand-editing `CURRENT-TASKS.md`.

**Learning:** Claude Code decides child-session status at startup via `Lgi = Boolean(process.env.CLAUDE_CODE_CHILD_SESSION)`. A child session doesn't own the todo list (the parent does), so the todo/task-management tools are withheld while process-control ones (`TaskOutput`/`TaskStop`) remain. These are built-in tools (no `mcp__` prefix — so "No matching deferred tools found" is a `ToolSearch` error, not the tool being an absent MCP), present with identical counts in every 2.1.x binary — so the symptom is never a version removal or a settings gate (permission gating makes a tool *prompt*, not disappear). The var leaked in from **VS Code's frozen process environment**: VS Code was launched (`code`/`open -a`) from inside a Claude Code agent Bash context, inherited `CLAUDE_CODE_CHILD_SESSION=1` + `AI_AGENT=…agent`, and injects that env into every integrated terminal — persisting for VS Code's whole lifetime (a running process's env is frozen at launch).

**Pattern:**
Diagnostic method that nailed it by elimination:
- grep all installed binaries for the tool-name strings → rules out version removal
- grep settings for tool gates → rules out a permission/deny gate
- check `$TMUX` + `tmux show-environment -g CLAUDE_CODE_CHILD_SESSION`
- grep shell profiles for the var
- check VS Code `terminal.integrated.env`
- walk `ps -o pid=,ppid=,command=` ancestry for a real parent `claude`

`AI_AGENT=claude-code_*_agent` is a fingerprint — set *only* on a CC agent spawn. Fix: `env -u CLAUDE_CODE_CHILD_SESSION -u AI_AGENT claude …` in the launcher (durable, next-launch) + fully quit/relaunch VS Code from the Dock (clears the current pollution). Never launch VS Code from inside a Claude Code Bash call.

## 2. A PLATO `expect` mismatch on a designed fallback = update the spec, not the component
**Topics:** testing

**Context:** `/members` rendered a "Monitoring" badge where the Conv-343 PLATO spec expected none; surfaced during Conv-402's Astro-7 browser smoke.

**Learning:** The "Monitoring" badge is a deliberate `getMemberBadges` fallback for role-less members (`src/components/discover/role-utils.ts:110–113`), first-class in the `MemberRole` type since Conv 111 (`9a81450a`) — not a regression, and Astro 7/vite 8 provably can't affect computed badge text. Per the frozen-legacy-spec rule (`feedback_plato_expect_is_legacy_spec`), when intentional current UI diverges from a PLATO `expect`, the spec is the stale artifact and gets corrected; the component is left alone.
