# Conv Learnings - 2026-07-13

## 1. Verify a "redundant, safe-to-delete" assumption before deleting — a cheap overlap check saved 83 KB of unique history
**Topics:** docs-infra, plan-management

**Context:** During PLAN-XTRACT, the two biggest inline PLAN.md cells (ROUTE-MIGRATION 58 KB, HOME-FEED-MERGE 25 KB) belonged to DONE blocks that already had 100 KB+ per-block READMEs, so they *looked* like redundant history safe to delete.

**Learning:** A 20-phrase overlap sample against the READMEs returned only 1/20 and 0/20 hits — the inline cells were a *unique conv-by-conv status log* the READMEs never captured (the READMEs are plan/design detail; the cells are running status). The correct move was move-not-delete (append to README, then slim the cell). Running the check *before* slimming is what prevented silent loss of unique planning history.

**Pattern:** Before treating content as redundant, sample distinctive phrases and grep the supposed source-of-truth; only "delete" when overlap is high, otherwise "move."

## 2. Editing `0001_schema.sql` does NOT reach an already-migrated remote D1 — `db:migrate:staging` no-ops
**Topics:** d1, cloudflare, deployment, migrations

**Context:** FEEDBACK-DEPLOY needed 3 new columns (added to `0001_schema.sql` per the pre-launch convention) to reach the live staging D1.

**Learning:** `wrangler d1 migrations apply` only runs migration files not yet recorded in the DB's migration-tracking table. An *edit* to the already-applied `0001_schema.sql` creates no new file, so `db:migrate:staging` silently does nothing. The columns reach an existing remote DB only via (a) a full reseed (`db:setup:staging:*`, destructive) or (b) a manual `ALTER TABLE ADD COLUMN` (surgical, non-destructive). Conv 388 [SESSION-REMIND-DEPLOY] used reseed; this conv used ALTER.

**Pattern:** For any pre-launch `0001_schema.sql` column addition that must reach a deployed environment:
```
wrangler d1 execute DB --env <env> --remote --command "ALTER TABLE ... ADD COLUMN ..."
```
(preserves data) or reseed (wipes it).

## 3. Per-participant dedup needs per-party stamp columns, not one-per-row
**Topics:** d1, testing

**Context:** Session-reminders dedups with one stamp per session per band (both parties notified together, symmetric content). Feedback ratings are per-participant (`session_assessments` has one row per `assessor_id`; student and teacher rate independently).

**Learning:** A single `feedback_reminder_sent_at` couldn't express "student nudged but teacher wasn't." Solution: two columns (`feedback_reminder_student_sent_at` / `_teacher_sent_at`), each stamped only when that party is nudged, with an `EXISTS(session_assessments …)` guard so a party who already rated is never nudged. Stamp semantics stay clean ("reminder attempted"); rated-but-unstamped parties are re-evaluated each tick but the 72h window bounds the cost.

## 4. Line-number matching is unsafe across edits to the same file — match by content
**Topics:** cc-workflow

**Context:** During PLAN-XTRACT, a line-number-based awk misfired after the first splice shifted line numbers (it printed HOME-FEED-MERGE content when asked for LAYOUT-SG's row).

**Learning:** When a script edits a file across multiple passes, address rows by a stable content key (block-name prefix), never by fixed line number. The extraction script matched `| BLOCK |` prefixes throughout, so it stayed correct despite line drift.
