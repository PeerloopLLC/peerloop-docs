# Conv Learnings - 2026-07-01

## 1. A per-user SSR-computed layout preference cannot be applied by a client-only toggle
**Topics:** astro, ssr, react-islands

**Context:** The nav placement (top vs rail) is resolved at SSR in `AppLayout` (`Astro.locals.navLayout`) so there's no first-paint flash and it's cross-device/account-scoped. A React island flipping local state can't re-run the server render.

**Learning:** The `LayoutToggle` island persists via `PATCH /api/me/profile` then does a **hard `window.location.reload()`** — the reload is what re-runs SSR so the whole shell re-orients. Contrast `ThemeToggle` (localStorage, client-only, no server round-trip). SSR-driven preferences need a reload (or a full nav), not just a state flip.

**Pattern:** SSR-computed per-user preference toggle = optimistic `PATCH /api/me/profile` → `window.location.reload()`. Use only when the preference is resolved server-side; a client-only theme-style toggle stays localStorage + state flip.

## 2. Two-vocabulary enum with a single translation point keeps DB/design and internal code aligned
**Topics:** schema, naming, layout-mode

**Context:** The approved design named the setting `'top' | 'rail'`, but the Conv-355 internal plumbing (`SubNavLayout`, `Astro.locals.navLayout`) is `'top' | 'left'` (placement).

**Learning:** Store the user-facing `'top' | 'rail'` in DB/API (matches the design + reads well to a human) and translate `'rail' → 'left'` **once** in `AppLayout` when publishing to `Astro.locals`. Everything downstream stays on the untouched internal vocabulary. One ternary is a trivial cost; it avoided churning just-shipped Conv-355 code and honored the approved design without escalation.

## 3. Deleting inert code safely = audit every caller of its public surface first
**Topics:** refactor, subnav, layout-mode

**Context:** [SNAV-CLEAN] removed the zoned/cluster SubNav machinery, dead since [SNAV-TOP] Phase 2a moved the course journey into `CourseJourneyStepper`.

**Learning:** Before deleting, swept for `zone:`/`kind:'cluster'`/`done:`/`disabled:`/`zoneLabels` across all SubNav callers **and** the tab builders, plus tests. Key nuance: `done`/`disabled` still appear in `_course-tabs.ts` but on the `CourseJourney`/`CourseSessionAction` types (feeding the stepper), NOT SubNav items — reading the type ownership, not just grepping the token, is what confirmed safety.
