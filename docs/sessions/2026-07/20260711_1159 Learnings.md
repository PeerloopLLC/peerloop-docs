# Conv Learnings - 2026-07-11

## 1. The PLATO dependency graph was latent, not missing — assembly, not new schema
**Topics:** plato, testing, architecture

**Context:** The user asked for a "dependence tree and a known place to look for the results of prior runs." The instinct is to add new metadata.

**Learning:** The edges already existed as strings — `restoreFrom` (depends-on), `snapshot`/`capturesTo` (produces). A grep showed exactly 4 declared consumption edges. The gap was that nothing *read them all* into a graph, validated them, or tracked freshness. The fix derives the DAG from existing fields (no new `dependsOn`/`produces` schema) — single source of truth, no drift risk. The one genuinely absent thing was the *lineage/staleness* fingerprint.

**Pattern:** Before adding schema, grep whether the relationship is already encoded and merely un-assembled.

## 2. Provenance sidecar, not in-DB table — keep the artifact byte-clean for its own validators
**Topics:** plato, testing

**Context:** Phase 4a needed per-waypoint "when did this last run + from what code." The literal proposal was a `_plato_provenance` table inside each `.sqlite`.

**Learning:** The project validates snapshots with a *manual per-table row-identity diff* (Conv 383–384, `notifications` is already an excluded footgun). An in-DB provenance table would be a second permanent thing to remember to exclude. A gitignored JSON **sidecar** (`<wp>.sqlite.prov.json`) keeps the `.sqlite` byte-clean, is readable from both the JS capture script and the TS CLI with no sqlite round-trip, and can never leak into a row COUNT.

**Pattern:** When adding metadata to an artifact that has its own byte-level validators, prefer a sidecar over mutating the artifact.

## 3. Transitive-closure source hash keyed off the scenario chain
**Topics:** plato, testing

**Context:** Staleness must be trustworthy across infrequent runs — "did the code that produces this waypoint change?"

**Learning:** Hash the producer instance + its scenario(s) + **every step in the chain** + persona + shared schema, unioned with each `restoreFrom` parent's hash. Because `flywheel-pre-12`'s chain runs `enroll-student` and `flywheel-pre-9`'s (steps 1–8) does not, editing `enroll-student` marks pre-12/14/15/flywheel/session-invite STALE but leaves pre-9 FRESH. Use the runtime scenario object to enumerate *which* files, then hash file *contents* (robust to formatting).

**Pattern:** For "is this derived artifact stale," fingerprint the transitive closure of source files that determine it, not just the top-level producer.

## 4. API producers full-replay → `--from-waypoint` means regenerate-downstream, not restore+continue
**Topics:** plato, testing

**Context:** The Conv-072 "restartability" design framed restart as "restore a snapshot and continue."

**Learning:** The API producers (`flywheel-pre-N`) each replay the full step prefix from the core seed in-memory (sub-second) — they do NOT restore a parent snapshot. So on the API side, restart is just "re-run the subset"; `--from-waypoint X` = "regenerate X + everything that transitively depends on it." True restore-and-continue is the *browser* side (the deferred agent walker). The runner is really `make` for waypoints (staleness-aware regeneration), not a debugger-style restart.

---

## 5. tsx for the graph CLI — real objects beat the text-parse precedent
**Topics:** plato, cc-workflow

**Context:** Existing PLATO tooling (`plato-split.js`) reads `.ts` files as text and regex-parses them.

**Learning:** `tsx` is already a dep, and correctness of the graph is the whole point. Running the CLI under `tsx` and importing the real instance/scenario objects via the loaders is robust to formatting and gives typed edges/chains for free — the better of two available mechanisms, without contradicting the text-parse precedent (which is a code-*generator*, a different job).
