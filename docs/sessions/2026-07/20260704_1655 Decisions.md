# Conv Decisions - 2026-07-04

## 1. [PROF500] Fix `/profile` staging 500 via full reseed + code guard (Fix B), not a surgical ALTER
**Type:** Implementation
**Topics:** d1, deployment, testing

**Trigger:** `/profile` returned 500 on staging only. Root cause: `profile/[...tab].astro` runs an **unguarded** `SELECT ... nav_layout FROM users`, and the staging `users` table was missing `nav_layout` — the column was added to base `migrations/0001_schema.sql` (Conv 356) but `0001` is edited in place, so an un-reset staging DB never received it. Guarded readers (`middleware.ts`, `AppLayout.astro`) degraded silently; the unguarded one 500'd.

**Options Considered:**
1. Surgical `ALTER TABLE users ADD nav_layout` — data-only, minimal, fixes just this column
2. Full reset + reseed via `db:setup:staging:booking` (+ feeds) — clears ALL latent drift ← Chosen (data)
3. Code guard (Fix B) only — wrap the query in try/catch, leave the drift ← Chosen as parallel defense

**Decision:** Do **both**: reset + reseed staging from the current `0001` (`db:setup:staging:booking`, then `db:seed:feeds:staging` since the booking chain skips feeds), **and** apply Fix B (wrap the profile account query in try/catch → falls to the existing "couldn't load account details" card). Staging data is disposable seed data (all `@example.com` accounts, fabricated `created_at`), so the destructive reseed is safe.

**Rationale:** Reseed is strictly better than a surgical `ALTER` when the data is disposable — it clears *all* in-place-`0001` drift at once, not just `nav_layout`. Fix B is defense-in-depth so any *future* column drift degrades silently like the already-guarded readers instead of 500ing.

**Consequences:** Staging D1 reset + reseeded to `:feeds` level; profile query guarded; `nav_layout` verified present, `/profile`→200, gated sub-tabs →302 (correct). Deployed staging `a5ea3a28`; phone-confirmed by user. 5 gates green (6751).

## 2. [VBAR] Replace the persistent StickySignupBar with dismissable in-feed CTA cards + a persistent Sidebar Sign up/Log in (Option A)
**Type:** Strategic
**Topics:** ui, navigation, feed

**Trigger:** Logged-out visitors saw an undismissable "Join Peerloop…" overlay (`StickySignupBar.astro`) on every home return — poor UX. The bar was intentionally persistent (Conv 258/267/270) precisely because the Matt shell has **no** visitor auth affordance (neither Sidebar nor HeaderBar offered login/signup).

**Options Considered:**
1. **A — In-feed dismissable CTA cards + persistent chrome sign-up** ← Chosen
2. B — In-feed cards only
3. C — Recurring dismissable sticky bar
4. D — Top banner

**Decision:** Option A. A new dismissable `SignupCtaCard.tsx` is interleaved by SmartFeed every 4 real items for visitors (`withSignupCta`, ephemeral-dismiss), **plus** a persistent Sign up button + Log in link added to the `Sidebar.tsx` bottom cluster (expanded/collapsed/drawer), replacing the old generic "Visitor" row. `StickySignupBar.astro` is deleted. The chrome form was confirmed as a "Sign up row + Log in link".

**Rationale:** Splits the three jobs the old bar conflated — an *always-reachable* auth path (now persistent chrome, covering desktop and the shared-`Sidebar` drawer) vs a *periodic conversion nudge* (now a dismissable in-feed card). This fixes the "no way to dismiss" complaint without losing conversion, and fills the real gap (no visitor auth affordance in the Matt shell). SmartFeed already interleaves dismissable non-post cards, so the CTA card is cheap.

**Consequences:** New `SignupCtaCard.tsx`; `SmartFeed.tsx` gains `withSignupCta`; `Sidebar.tsx` visitor affordance; `index.astro` drops StickySignupBar; `StickySignupBar.astro` deleted. +6 tests, 5 gates green (6757). Deployed staging `0a73df0e` (visitor home: Sign up ×2 + Log in ×2, StickySignupBar 0×). **Supersedes** the Conv 258/267/270 "one loud persistent conversion bar" design.

## 3. [THEME-CS] Dark-mode toggle parked "coming soon"; `nav_layout` toggle stays logged-in-only
**Type:** Strategic
**Topics:** ui, astro, navigation

**Trigger:** After VBAR replaced the "Visitor → /profile" row with Sign up/Log in, visitors lost their only nav path to the `/profile`-only controls. Audit found the sole light/dark control is `ThemeToggle` (only in `/profile`); the `.dark` mechanism exists but is legacy (Matt pages use tokens), so dark mode is effectively dropped. `LayoutToggle` (`nav_layout`) is signed-in-only + server-persisted (visitors never had it).

**Options Considered (dark mode):**
1. Restore a visitor nav path to the theme toggle
2. Remove `ThemeToggle` entirely
3. **Mark it "coming soon" — a disabled logged-in bonus** ← Chosen

**Options Considered (`nav_layout`):**
1. Give visitors access to the layout toggle
2. **Leave it logged-in-only, unchanged** ← Chosen

**Decision:** Leave `nav_layout` and the rest of `Sidebar.tsx` untouched (the logged-in-only rationale accepted). Add a `comingSoon` prop to `ThemeToggle.tsx` — disabled switch + "Coming soon" badge + "Dark mode is on the way", with the mount effect and toggle no-op'd so it never flips `.dark` behind a disabled control — and pass it at both `/profile` usages. Fix the now-stale Sidebar comment.

**Rationale:** User confirmed light/dark was never a real cross-app affordance (built for `/admin`, consciously dropped in the Matt-inspired porting), so parking it beats restoring a path to a broken feature or deleting the component outright. `nav_layout` is a legitimately signed-in server-persisted setting, so visitors correctly never had it — keeping `Sidebar.tsx` as-is and clean.

**Consequences:** `ThemeToggle.tsx` gains the `comingSoon` prop pattern (reusable for parking incomplete controls); `profile/[...tab].astro` passes it twice; stale Sidebar comment fixed; +2 tests, 5 gates green (6759). Deployed staging `95b5887a`.
