# Conv Learnings - 2026-07-22

## 1. Author allowlist is the only timecard filter that survives a history merge
**Topics:** timecard, git, tooling

**Context:** [TC-MERGE-TZ] — a history-preserving merge of a client branch into a `^jfg-dev` branch defeats the branch allowlist because `git log <branch>` walks full ancestry; regenerating any past day's timecard would silently include client commits.

**Learning:** Filter at the commit-extraction choke point on authorship (`%ae` + config allowlist regex), not reachability. Verified: post-merge filtered run was byte-identical (billing AND rendered markdown) to the pre-merge baseline. Allowlist over denylist — a new client email would leak under a denylist, same argument as the script's existing branch-allowlist comment.

**Pattern:** capture `%ae` in the `git log --format` stream; `.filter((c) => allowRe.test(c.authorEmail))` where commits enter the pipeline; config default `'.'` (allow-all) for backward compatibility.

## 2. Astro 7 `astro dev` daemonizes; Astro 6 stays foreground
**Topics:** astro, dev-workflow

**Context:** Launching the side-by-side servers: our checkout (Astro 7.1.3) vs the pivot worktree (Astro 6.3.7).

**Learning:** Astro 7's `astro dev` detaches into a daemon (prints pid, exits the foreground npm process — a background Bash task "completes" while the server lives on; stop via `astro dev stop`, status via `astro dev status`). Astro 6 runs foreground. Teardown must be daemon-aware: `astro dev stop`, never a port-based kill ([DEVSRV-KILL]).

## 3. Wrangler and the astro dev daemon contend for the same miniflare D1 store
**Topics:** cloudflare, d1, dev-workflow

**Context:** Diagnosing the :4321 login failures: `wrangler d1 execute --local` failed with "The Workers runtime failed to start" while the astro-7 daemon was running.

**Learning:** Stop the dev server before any local wrangler D1 operation (seeding included). For read-only diagnosis while the daemon runs, query the miniflare sqlite directly: `sqlite3 -readonly .wrangler/state/v3/d1/miniflare-D1DatabaseObject/<hash>.sqlite "SELECT …"` (the `file:…?mode=ro` URI form fails in the sqlite3 CLI; the `-readonly` flag works).

## 4. Git worktrees don't carry gitignored env files
**Topics:** git, dev-workflow, worktree

**Context:** `npm run db:setup:local:dev` in the fresh reference worktree failed ENOENT on `.dev.vars`.

**Learning:** A worktree checks out only tracked files — `.dev.vars`/`.env` must be copied in (`cp` + recreate the `.env → .dev.vars` symlink; never cat secrets to chat). Conversely, `.wrangler/` state is per-directory, so a worktree gets its own isolated local D1 — a feature for reference environments.

## 5. Substring classifiers on prose fields false-positive — anchor on sigils
**Topics:** tooling, task-board

**Context:** `current-tasks-check.sh` classified State lines via `index($0,"parked")` — the annotation "un-parked Conv 407" tripped a false STATE-MISMATCH.

**Learning:** Classify structured-markdown state by the emoji/sigil (`⏸️`), not an English substring that can appear in free prose on the same line. Same [EDITSAFE] class as ambiguous edit anchors. Test fixtures already used the emoji, so the fix was fixture-compatible (29/29 green).

## 6. Vite cold-start dep optimization can eat a login navigation
**Topics:** astro, vite, debugging

**Context:** :4341 login "flickers, no modal"; console showed `/api/me/full` 401.

**Learning:** On first page visit, vite may optimize newly-discovered deps and force-reload — cancelling in-flight navigations (reads as a flicker). And `/api/me/full` 401 is the app's *normal* logged-out probe — an effect of not being logged in, not the cause of the failure. Check the dev-server log timeline before trusting a browser console symptom; verify the auth loop server-side (`dev-login` + cookie-jar `me/full`) to split browser vs server causes.

## 7. Fable-5 calibration: decision reasoning is the deliverable, never compressible
**Topics:** cc-workflow, fable5

**Context:** First Fable 5 conv. Three user-flagged misses (silent branch retarget; shallow grep-patches of a premise change; choices offered without pre-explained reasoning). Focus mode was constant across 406 prior convs — the model was the variable.

**Learning:** The user's working model: reasoning is the **decision gate** — they read the findings/reasoning to decide whether the options deserve consideration at all, may challenge the reasoning, and only then pick. Under focus mode only the final message is visible, so every decision-bearing message must carry findings → interpretation → why each option exists → recommendation, in that message. Outcome-first compression must never be applied to the reasoning behind an offered choice. Codified in `memory/feedback_assess_ask_before_acting.md`; written rules acknowledged as probabilistic — the structural remedy chosen was the focus-off trial.

## 8. Client-branch names lie; measure tips
**Topics:** git, client-collaboration

**Context:** "brian-July-7" — the user asked for "the last commit of brian-July-7"; its origin tip is dated 07-20 11:29 (`8a1e677f`), and the machine-local copy of the branch was 28 commits stale.

**Learning:** Branch names are labels, not dates. Resolve "latest X" against origin and quote the tip SHA + timestamp back before acting. The user had also created `brian-July-20` as a deliberate exploration sandbox at the agreed pivot — branch topology can encode a human agreement (a pivot point) that containment math ("strict descendant, nothing lost") does not override.
