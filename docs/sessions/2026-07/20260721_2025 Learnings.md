# Conv Learnings - 2026-07-21

## 1. Decompiling the CC binary reveals undocumented tool gates the changelog and docs hide
**Topics:** cc-workflow, dual-repo

**Context:** The Task subsystem vanished; the docs and a research agent gave a confounded/incorrect picture (the agent conflated the subagent "Task tool" `mode` deprecation with the task-*list* tools).

**Learning:** `strings -a ~/.local/share/claude/versions/<v>` + grepping each tool's `isEnabled(){…}` is the authoritative way to determine why a built-in tool is present/absent. It found `uZ()` reading remote config `tengu_vellum_ash` against the model ID — a gate that appears in **no** changelog entry (`vellum` = 0 hits). `TaskCreate/List/Update/Get` are `isEnabled(){ return uH() && !uZ() }` and `TodoWrite` is `!uH() && !uZ()`; because `uZ()` sits on **both** branches, a true `uZ()` suppresses Task* AND TodoWrite simultaneously — a state no local setting can produce. Comparing the same string across adjacent versions (214/215/216 identical) rules out a version bump; `~/.claude/tasks/` mtimes date the outage precisely.

**Pattern:**
To diagnose a missing built-in:
1. Decompile + grep `isEnabled` (`strings -a` the version bundle).
2. Diff the string across adjacent versions.
3. Date the outage via on-disk artifacts (`~/.claude/tasks/`, session JSONLs).
4. Read the local cached changelog `~/.claude/cache/changelog.md`, not memory.

## 2. Self-falsify a proposed test before asking the user to run it
**Topics:** cc-workflow

**Context:** Proposed `CLAUDE_CODE_DISABLE_1M_CONTEXT=1` as a cheap way to test the model-substring hypothesis; the user approved it.

**Learning:** Tracing `mi()`/`Tb()`/`kD()` showed the var short-circuits `Tb()` so the `[1m]` suffix is never stripped from `Di()` — so `Di()` returns `claude-opus-4-8[1m]` either way and the test could not work. Verifying the mechanism *after* approval but *before* execution caught it and voided the proposal (valid substitute: pick the non-1M Opus 4.8 via `/model` so the model string carries no `[1m]` suffix). Better to burn model tokens self-checking than to send the user through a void relaunch.

## 3. Write-through preserves the discipline forcing-function that a batched refresh loses
**Topics:** cc-workflow, docs-infra

**Context:** The detach removed `TaskCreate`. The risk was losing the "an alert must become a tracked task" guarantee.

**Learning:** A tool call and a file edit are equally verifiable — both either happened or didn't. The failure mode is a *promise* ("I'll log it at /r-end"). So editing `CURRENT-TASKS.md` directly the moment a task changes keeps the forcing-function intact; the r-end "CRITICAL: call TaskCreate" rule had already gone unenforceable at Conv 403 precisely because it named a tool that could silently vanish.

## 4. A validator is only trustworthy with a self-test that proves it catches each failure
**Topics:** testing, docs-infra

**Context:** The first `current-tasks-check.sh` claimed (via the r-update-tasks spec) to check slugs + State agreement but only checked code-membership.

**Learning:** Writing fixtures — a good one plus one deliberately-broken variant per failure class — surfaced the gap and forced the checker to actually implement the missing checks. A checker without a self-test is trusted on faith.
