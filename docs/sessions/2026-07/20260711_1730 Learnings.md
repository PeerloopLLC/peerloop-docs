# Conv Learnings - 2026-07-11

## 1. Teacher availability is not stored — it's an expensive per-teacher expansion; batch it as a boolean map for lazy filtering
**Topics:** availability, courses, performance

**Context:** Adding a "teacher available within N days" filter to the client-side-filtered `/courses` catalog.

**Learning:** Availability = expand recurring `availability` rules × certified teachers × booked `sessions` × `availability_overrides`, per teacher (`countAvailableSlots` = 3 queries + JS slot expansion each). There is no cheap single-JOIN answer and no stored flag; the codebase deliberately defers it to lazy client fetches (`EnrollButton`, `CourseAvailabilityPreview`). The clean integration for a browse filter is a batch endpoint returning a **boolean map** `{courseId: hasAvailability}` (not a pre-filtered list), so availability composes with the other in-memory filters (level/topic/search) instead of fighting them.

**Pattern:**
`POST /api/courses/availability-batch {ids} → {windowDays, availability}`; short-circuit per course at the first teacher with `totalSlots>0`.

## 2. The `availability_window_days` config key already existed — reuse over reinvention
**Topics:** platform-stats, config

**Context:** The request asked for an "admin-configurable number of days (default 14)".

**Learning:** `platform_stats.availability_window_days` was already seeded (30) and read by `availability-summary.ts` (inline `|| '28'`). Rather than mint a new key, we reused it, changed the default 30/28 → 14, and centralized reads in a new `lib/availability-config.ts` loader (mirrors `lib/promotion/config.ts`). Reusing the key means the course-detail availability preview and the new browse filter share ONE window ("what you filter by = what you see").

## 3. `innerText` reflects CSS `text-transform` — a DOM-verification gotcha
**Topics:** browser-verification, testing

**Context:** Verifying the admin page showed "14 days"; a regex on `document.body.innerText` for "Look-ahead window" failed.

**Learning:** `innerText` returns *rendered* text, so a CSS `uppercase` class makes it "LOOK-AHEAD WINDOW" — a case-sensitive match misses it. Use `textContent` (unaffected by `text-transform`) for exact text assertions, or case-insensitive matching. (`getBoundingClientRect`/`getComputedStyle` for layout; `textContent` for text.)

## 4. Matt-token restyle idioms (course surfaces)
**Topics:** matt-design-system, styling

**Context:** Restyling `HomeworkTab.tsx` off the legacy `primary-*`/`secondary-*` palette.

**Learning:** Established idioms from the live codebase: course button = `bg-course-primary text-white hover:opacity-90 transition-opacity`; cards/borders/text = the PALETTE-FDN neutral ramp (`border-neutral-300`, `text-neutral-500/700`, `bg-neutral-50/100`) + `text-text-default`; status = `success/warning/error/info`-100 bg / -500 text / -300 border; form controls = `rounded-8 border border-neutral-300 focus:border-primary-default focus:ring-2 focus:ring-primary-default`; label = `text-body-default-bold text-neutral-700`. NB the course-entity primary is **green** (`rgb(50,125,0)`), not blue.

## 5. `*/` inside a JS block comment closes it — even in prose
**Topics:** javascript, gotcha

**Context:** A header comment describing the restyle wrote `primary-*/secondary-*`.

**Learning:** The `*/` sequence terminates a `/* */` block comment anywhere it appears, including inside descriptive prose. Writing "primary/secondary" (no `*`) or spacing the glob (`primary-* / secondary-*`) avoids the parse error. tsc/eslint catch it immediately.
