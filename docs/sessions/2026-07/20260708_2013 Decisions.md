# Conv Decisions - 2026-07-08

## 1. Session-reminder window semantics — partition bands + lead-time-neutral copy
**Type:** Implementation
**Topics:** testing, video

**Trigger:** Building `sendSessionReminders`; no existing reminder job to follow; the 24h/1h template copy could be wrong for late-booked sessions.

**Options Considered:**
1. Partition bands `(now+1h, now+24h]` advance / `(now, now+1h]` imminent + reword the "24h" copy to be lead-time-neutral ← **Chosen**
2. Tight 24h band (only ~20–24h out, keeps "tomorrow" accurate) — short-lead sessions get only the 1h reminder
3. Wide band, keep "Session Tomorrow" copy — accept occasional inaccurate "tomorrow"

**Decision:** Option 1 — bands partition the whole ≤24h future so every scheduled session gets ≥1 reminder; dedup-stamped per slot for at-most-once; template reworded ("Upcoming Session", never "tomorrow").

**Rationale:** Maximizes coverage (even late bookings) and the copy is always accurate; the exact date/time is shown in the info box regardless.

**Consequences:** `SessionReminderEmail.tsx` copy changed; two dedup columns added; window bounds computed in JS as ISO strings (no `datetime()`).

## 3. Protect the TZ fixes with flip-verified tests; lint-protect the bucketers
**Type:** Strategic
**Topics:** testing

**Trigger:** Pre-GO-LIVE review found the TZ-MODEL fixes had no tz-sensitive test.

**Options Considered:**
1. Runtime tests for every Phase-3 site (incl. 7 analytics bucketers)
2. Flip-verified runtime tests for the high-value sites (earnings/expiry/cleanup/validation/capture) + rely on `lint:tz` for the bucketers ← **Chosen**

**Decision:** Option 2. Each runtime test flip-verified; bucketers left to the static guard.

**Rationale:** Bucketer date keys derive from `toISOString` (negligible divergence) — a runtime test there would be contrived and give false confidence. `lint:tz` catches their regression deterministically (verified).

**Consequences:** +20 tests; 6 files exported private helpers for unit testing; bucketers covered by lint only.

## 4. Enforce the tz guard via CI with a hostile-TZ matrix; exclude client-side scan
**Type:** Implementation
**Topics:** testing, cloudflare, deployment

**Trigger:** Hardening [TZ-LINT-CI]; discovered CI runs on UTC and `lint:tz` was ungated in CI. (A CC PreToolUse hook — docs-repo `.claude/hooks/pre-commit-lint-tz.sh` — already gates CC-initiated code-repo commits, but not human/IDE/PR/CI paths, and `lint:tz` was RED on baseline anyway. A mid-work "no hook exists" claim was a repo-scoping error, corrected at r-end: the hook lives in the docs repo, not the code repo I checked.)

**Options Considered:**
1. Just add `lint:tz` to CI (but UTC CI can't catch host-local date bugs)
2. `lint:tz` in CI **+** hostile-TZ test matrix `[UTC, Pacific/Kiritimati]` + extend scan to server dirs only ← **Chosen**

**Decision:** Option 2. Wire `lint:tz` into the lint job; matrix the test job over UTC + Pacific/Kiritimati; extend the guard scan to `src/emails` + `workers` (0 violations); exclude `src/components` (~60, client-side) + `.astro` (needs triage) → [TZ-LINT-SCAN2].

**Rationale:** The hostile-TZ leg is the only thing that makes UTC CI enforce the UTC contract. Components run in the browser's local tz — a different concern that would flood the guard red.

**Consequences:** `ci.yml` + `lint-timezone.sh` changed; verified the full suite passes under +14; [TZ-LINT-SCAN2] filed.
