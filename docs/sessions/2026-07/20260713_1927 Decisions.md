## 2. FEEDBACK-NUDGE — email-only (no in-app notification)
**Type:** Strategic
**Topics:** d1, cc-workflow

**Trigger:** The task note said "+ notif type," but exploration found a `feedback_reminder` notif type needs a `notifications` table CHECK-constraint rebuild (SQLite can't ALTER a CHECK).

**Options Considered:**
1. Email-only (no notif type) ← chosen
2. Full parity — add notif type via table-rebuild migration
3. Reuse the existing `session_completed` notif type

**Decision:** Email-only — the pref-gated reminder email with a direct "Rate Your Session" link. No `notifications.ts` change.

**Rationale:** A rating nudge doesn't strongly need an in-app bell; the table-rebuild cost isn't justified.

**Consequences:** Simpler build; the in-app notif originally listed is dropped.

## 4. FEEDBACK-DEPLOY — manual ALTER (not reseed) to apply staging schema
**Type:** Implementation
**Topics:** d1, deployment

**Trigger:** Staging D1 has data (the client uses it); the change is purely additive (3 columns).

**Options Considered:**
1. Manual ALTER (non-destructive) ← chosen
2. Reseed `db:setup:staging:booking` (destructive, matches Conv 388)
3. Don't deploy

**Decision:** Manual ALTER — ran exactly the 3 `ALTER TABLE ADD COLUMN` statements on staging D1, preserving all staging data; redeployed the cron worker.

**Rationale:** No technical reason to wipe staging for an additive column change. Contrasts with Conv 363 (PROF500), which preferred reseed when staging data was *disposable seed*.

**Consequences:** Staging schema is "0001 + manual ALTER" (minor drift from a clean rebuild); data preserved; cron redeployed (`deploy:cron:staging`, `37e506d5`).
