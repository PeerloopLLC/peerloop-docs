# Conv Learnings - 2026-07-10

## 1. Row-identity validation forces lineage-exact producers — reuse fails on setup tables
**Topics:** plato, testing

**Context:** activities uses genesis personas + the same course as the flywheel, so `flywheel-pre-12` (wp-enrolled) is state-*sufficient* to drive activities' booking UI. It's tempting to reuse it as the restore base.

**Learning:** But the user's chosen validation bar is row-identity (browser-captured end-state == API producer, all tables). `flywheel-pre-12`'s lineage differs from activities' own scenario (community-resources rows, availability slot count, step order), so a capture restored from it would differ from `activities.sqlite` on setup-derived tables even if the activity tables match. Row-identity therefore requires a **lineage-exact** producer per journey (`activities-pre-11`, `ecosystem-pre-12`) — a prefix of that journey's own scenario — not a convenient existing snapshot.

**Pattern:** When a browser segment must be validated row-identical to an API producer, its `restoreFrom` waypoint must be produced by a prefix of the SAME scenario, not a different scenario that happens to reach an equivalent state.

## 2. `add-teacher-cert` is not a cut — only the first self-cert crosses Stripe
**Topics:** plato, stripe

**Context:** Annotating ecosystem's cut points. ecosystem self-certs the creator for course-0 (CUT-1) and later runs `add-teacher-cert` for course-1.

**Learning:** Reading the step files (not inferring): `self-certify-creator` has a `connectStripe` action (`POST /api/stripe/connect`, stripe-connect mock) = the CUT-1 boundary, plus a pure-DB `selfCertify`. `add-teacher-cert` has ONLY `selfCertify` — no Stripe — because the account is already connected. So a creator's SECOND-course certification is pure-UI/browser-drivable; only the FIRST self-cert is a cut. Confirms the `[EMP]` guard's value — the boundary set can't be inferred from step names.

## 3. `book-complete-session` is half pure-UI, half cut
**Topics:** plato, bbb

**Context:** activities embeds `book-complete-session` mid-walk.

**Learning:** Its `bookSession` action (`POST /api/sessions`, no mock) is pure-UI/browser-drivable; its `completeSession` action (`POST /api/webhooks/bbb`, bbb mock) is CUT-3. So a browser walk books the session in the UI and applies the completion as an inline API bridge — the same book/complete split the flywheel got in Phase 2.

## 4. Restore-from-waypoint sidesteps the tz-reconciliation caveat
**Topics:** plato, timezone, testing

**Context:** Conv 381's flywheel B1 registered users fresh in-browser, so their tz auto-detected to the machine's zone and had to be reconciled to the persona spec (Mara=LA, Alex=Tokyo) before capture.

**Learning:** The activities walk restores `activities-pre-11` (which already carries persona tzs from its API run) and registers NO users — so no reconciliation is needed. The `[TZ-TESTS]` cross-boundary rendering ("Teacher: America/New_York / You: Asia/Tokyo", slots shown as 10–11 PM Tokyo) worked out of the box. A restore-from-waypoint segment that doesn't register users inherits correct tzs for free.

## 5. Chrome bridge: `computer` clicks use screenshot pixels, DOM uses CSS pixels
**Topics:** chrome-bridge, plato

**Context:** Coordinate-based `computer` left_click calls on the booking calendar/slots kept missing.

**Learning:** The screenshot is 1242px wide but the CSS viewport is 1177px (~1.055× device-scale), and `getBoundingClientRect` returns CSS pixels — so DOM-derived coordinates don't map 1:1 to `computer` click coordinates. Fix: click by element `ref` (from `find`/`read_page`), which resolves position server-side, instead of passing raw coordinates. Reliable for React handlers too.

**Pattern:** In Chrome-bridge walks, prefer `find` → click-by-`ref` over coordinate math whenever device-scale ≠ 1.
