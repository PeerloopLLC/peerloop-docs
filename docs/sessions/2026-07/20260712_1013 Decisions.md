# Conv Decisions - 2026-07-12

## 1. Apply buffer_minutes end-to-end now, default 15
**Type:** Implementation
**Topics:** availability, booking

**Trigger:** `buffer_minutes` was persisted nowhere AND applied nowhere; a persisted-but-unapplied buffer is a UI promise the booking flow doesn't keep (`PUT /api/me/availability` validated 0–60 but the INSERT dropped it; `GET` hardcoded 15; all 3 per-teacher conflict checks used raw slot adjacency).

**Options Considered:**
1. Persistence only (defer application)
2. Apply now, default 15 ← Chosen
3. Apply now, opt-in (default 0)

**Decision:** Wire the buffer into all 3 per-teacher conflict checks with symmetric-gap semantics (a slot within `buffer` min of a booked session is blocked), keep column DEFAULT 15. Layer 1 (persistence): `availability.buffer_minutes INTEGER NOT NULL DEFAULT 15` (denormalized per-row like `timezone`); PUT INSERTs it, GET reads `availability[0]?.buffer_minutes ?? 15`. Layer 2 (application): `POST /api/sessions` double-book gate expands the conflict window by buffer, `teachers/[id]/availability.ts` greys buffered slots, `lib/availability.ts countAvailableSlots` applies it.

**Rationale:** Completes the feature; pre-launch (~2 users) so the retroactive "15-min buffer blocks the adjacent on-the-hour slot" change is negligible.

**Consequences:** Booking availability now leaves a gap; +7 tests (persist/round-trip/validation + 409-within-buffer + read-only greying) lock it in. 5 gates green (suite 6883). Filed `[AVAIL-BUFFER-DEPLOY]`.

## 2. Full staging reset+reseed over surgical ALTER
**Type:** Strategic
**Topics:** deployment, d1

**Trigger:** "make it all work on staging" — staging was behind on multiple schema edits (`availability.buffer_minutes` AND both `sessions.reminder_*` columns); a clean full drift diff proved painful on D1 (compound-SELECT cap, wrangler-stderr, no macOS `timeout`).

**Options Considered:**
1. Reset + reseed (`db:setup:staging:feeds`) ← Chosen
2. Surgical ALTERs (finish drift enumeration, add missing cols)
3. Check with client first

**Decision:** Reset + reseed staging with the `feeds` level, then deploy. (Reaffirms the existing Conv-363 [PROF500] deployment decision — "prefer a full reseed over a surgical ALTER" — already banked in `08-deployment-infra.md`; not re-routed to avoid a duplicate chunk entry.)

**Rationale:** Guarantees staging schema == current `0001` (no piecemeal drift hunting). Staging data was seed-scale + disposable. User explicitly authorized the wipe on the client's `brian-1dc` Cloudflare account.

**Consequences:** Staging schema fully current (deploy version `170a0b1a`, code `81f57e17`); `[SESSION-REMIND-DEPLOY]` item (2) resolved as a side effect since the reseed applied the reminder columns.
