# Conv Learnings - 2026-07-08

## 1. Middleware is the flash-free channel for per-user SSR values (tz followed nav_layout)
**Topics:** timezone, astro, ssr, middleware

**Context:** Phase 1 needed the viewer's stored tz on ~40 display surfaces, split between SSR `.astro` and client `.tsx` islands. The naive island approach (`useCurrentUser` in the leaf) flashes UTC→local after hydration.

**Learning:** `resolveNavLayout` (Conv 356) already queries a per-user pref and stashes `Astro.locals.navLayout` — the exact precedent. Extending the SAME query row to `SELECT nav_layout, timezone` and setting `Astro.locals.userTimezone` gives every SSR surface the stored tz for free (one round-trip, flash-free, Model-A-consistent). This turned a "novel architecture" into composition of an established pattern.

**Pattern:** SSR surface → `Astro.locals.userTimezone`; client island → `useUserTimezone()` (composes hydration-safe `useCurrentUser`: null on SSR/first-render → UTC, upgrades after mount). Both call shared `formatSessionTime`/`formatSessionDate` (tz `null` → UTC-labelled on times).

## 2. Rejected the `data-session-time` browser-local upgrade — it's Model B
**Topics:** timezone, ssr

**Context:** `CourseHeader` has an existing `data-session-time` client script that rewrites SSR times to *browser*-local.

**Learning:** Under Model A (stored tz everywhere, incl. Phase-2 emails), browser-local SSR would DISAGREE with emails when the two differ. Reusing the "established" script would have been wrong. The rule "follow the established pattern" must be checked against the *decision* (Model A), not just against "some pattern exists."

## 3. DOM-verify a tz change with a zone whose offset differs from BOTH browser and UTC
**Topics:** timezone, testing, dom-verification

**Context:** The logged-in dev user's browser was `America/Toronto` — same offset as the seeded `America/New_York`, so an ET render was indistinguishable from browser-local. First DOM read showed "6:00 AM" and proved nothing.

**Learning:** To prove the *stored* tz is used (not browser, not UTC), temporarily PATCH the stored tz to an exotic zone (`Asia/Tokyo`, UTC+9) via the Phase-0 endpoint, clear `peerloop_user_cache`, hard-nav, and read the DOM: `10:00Z` → **7:00 PM** is unambiguous (Toronto 6 AM / UTC 10 AM are both excluded). This also exercised the Phase-0 PATCH + `/api/me/full` + hook chain together. Always restore the mutated seed value.

**Pattern:** `PATCH /api/me/profile {timezone:'Asia/Tokyo'}` → `localStorage.removeItem('peerloop_user_cache')` → hard nav → read DOM → restore.

## 4. The hook goes in the sub-component that renders, not the top island
**Topics:** react, timezone

**Context:** `TeacherUpcomingSessions` maps to a `<SessionCard>` sub-component; `MyCourses` renders an `EnrollmentRow`; `TeacherSessionsList` a `SessionRow`. Adding `useUserTimezone()` to the top component left it out of scope at the render site (tsc: "Cannot find name 'userTz'").

**Learning:** For components that delegate rendering to a sub-component, the hook must live in the sub-component. tsc catches the scope error reliably, so it's low-risk — but worth checking the render site's enclosing function first. For files with many call sites sharing a `start` var (SessionsTabContent, TeacherSessionsList), moving the formatters to closures over `userTz` inside the sub-component keeps call sites unchanged.

## 5. Live-status check before converting: CourseTabs is discover-preview-only
**Topics:** timezone, route-migration

**Context:** `SessionsTabContent`/`AllSessionsTabContent` are mounted by `CourseTabs`, which `MySessionsTab.astro` explicitly calls the "legacy" surface it replaced.

**Learning:** Only actual `<CourseTabs` mount is `ExploreCourseTabs` (discover preview) — the Matt `/course/[slug]/sessions` route uses `MySessionsTab.astro`. So `SessionsTabContent` is a live-but-secondary student surface (converted), while `AllSessionsTabContent` serves creator/admin/moderator tabs → belongs to the **Admin slice**, not Student. Decompose by which role actually sees the surface, and verify the mount chain before converting possibly-dead or out-of-slice code.
