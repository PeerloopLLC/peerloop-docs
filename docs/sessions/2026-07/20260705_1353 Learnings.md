# Conv Learnings - 2026-07-05

## 1. Vite optimizeDeps esbuild scanner mis-extracts literal `<script>` from `.astro` frontmatter comments
**Topics:** astro, cloudflare, testing

**Context:** After clearing `node_modules/.vite` and cold-starting `astro dev`, the log showed `Failed to run dependency scan. Skipping dependency pre-bundling` with esbuild parse errors `Expected ";" but found "wire"` / `"→"`.

**Learning:** The Conv-177 `optimizeDeps.entries` glob (`src/**/*.astro`) makes esbuild scan `.astro` files. When a **frontmatter comment** contains the literal token `<script>` (even inside a `/** */` JSDoc or `//` comment), the astro scanner mis-extracts it as an inline `<script id=0>` and parses the trailing prose as JS → the whole dependency scan fails and pre-bundling is silently SKIPPED on every cold start. A `→` (or other non-JS char) in that mis-extracted region compounds it. Net effect: the pre-scan hardening is silently defeated on cold starts, degrading to the crash-prone lazy-discovery state.

**Pattern:** Keep literal `<script>`/`<style>` tokens and non-ASCII arrows OUT of `.astro` frontmatter comments; write "inline script" / `->` instead. Detection: `rm -rf node_modules/.vite && npm run dev` and grep the log for `Skipping dependency pre-bundling`.

## 2. Antipattern — gating success on a second request that swallows its own failures
**Topics:** auth, astro, cloudflare

**Context:** AUTHBUG root cause. `LoginForm` did `markAsLoggedIn(); await refreshCurrentUser(); onSuccess();` — but `refreshCurrentUser()` never throws (its 401 branch and catch both `return` after setting `authStatus`), so `onSuccess()` (modal close) ran unconditionally even when the follow-up `/api/me/full` failed.

**Learning:** When step B (a fragile network call) determines whether step A (the user-visible success) is real, B must signal success/failure to A, and A must gate on it. A "fire and forget, never rejects" helper used in a success path silently converts transient failures into permanent wrong-state. The fix pattern: (a) make the helper return an explicit success flag, (b) gate the visible success on it, (c) on failure fall back to a path that re-establishes true state (here: reload, since auth cookies are already set) rather than assuming success.

## 3. Peerloop client-auth architecture + the two never-cleared bits
**Topics:** auth

**Context:** Mapping the auth surface for AUTHBUG.

**Learning:** Client auth = a hand-rolled singleton on `window.__peerloop.currentUser` + `networkState.authStatus` in `src/lib/current-user.ts`, stale-while-revalidate against `GET /api/me/full`, `isVisitor() === (getCurrentUser() === null)`. Three localStorage keys: `peerloop_user_cache`, `peerloop_was_logged_in`, `peerloop_expired_identity`. Two latent bugs found: (1) **no** logout handler called `clearCurrentUser()` — all three just hit `/api/auth/logout` + redirect, leaving the cache to be cleared only by the next page's 401; (2) `peerloop_was_logged_in` was **never** cleared by any reachable path (every `clearCurrentUser` caller passed `explicit=false`), so the app could never reach a clean `visitor` state and always classified a post-login 401 as `session_expired`. `clearCurrentUser(true)` on explicit logout is the intended-but-unwired semantics (its own docstring says so).

**Pattern:** `resolve.dedupe`-style "two independent defenses" thinking applies to auth too — pre-bundling/no-store (avoid the failure) + authoritative success gate (survive the failure).

## 4. `resolve.dedupe` complements `optimizeDeps` for the Vite multiple-React SSR crash
**Topics:** astro, cloudflare

**Context:** VITE-DEDUP.

**Learning:** The dev-server "two React copies → invalid hook / useState-of-null" crash has two attack surfaces: the **discovery** path (Vite re-optimizes deps mid-session, reloads React) — closed by `optimizeDeps.entries` pre-scan (Conv 177); and the **module-identity** path (a stale/duplicated `.vite` cache hands back two React instances) — closed by `resolve.dedupe: ['react','react-dom']`. They're complementary, not redundant. With the scan-failure bug (Learning 1) defeating pre-bundling on cold start, `resolve.dedupe` was empirically the load-bearing guard.
