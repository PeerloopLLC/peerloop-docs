### AUTHBUG — intermittent lost-login (signed in but stays visitor)
**User:** "I want you to investigate intermittent problems the client has reported (and I too have seen, but not found the cause of) where logging out and then logging in as a different user (might happend for same user), upon pressing "sign in", the modal diappears and the signin was lost. We are still in visitor mode. It doesnt happen often and it seems to be related to the cache or local storgae as I sustect it only happens when you do not log in immediately after you log out"
**Claude:** Fanned out 3 read-only explorer agents (client modal/submit, client auth-state/localStorage, server endpoints). All converged. Root cause (high conf): `LoginForm` gates success on a second fragile request — `refreshCurrentUser()` → `GET /api/me/full` — that NEVER throws; a transient 401 nulls the user, sets `session_expired`, and closes the modal anyway → silent visitor mode. Amplifiers: no logout handler clears client auth cache; `peerloop_was_logged_in` is never cleared by any reachable path (all callers pass `explicit=false`). Server EXONERATED — no CSRF/nonce/KV-session/rate-limiter; login only needs `{email,password}`, re-mints cookies + returns 200 regardless of delay. Delay's role: raises odds the single post-login `/api/me/full` fails (HTTP-cache replay of a stale GET most likely — endpoint sent no `Cache-Control`; also cookie-propagation / `secure:true` on http / Worker cold start). Presented layered fix; asked how to proceed.

**User:** (picked) "Implement the durable fix"
**Claude:** Read the code to resolve the flagged seeding design-choice: login returns a *partial snake_case* user; `/api/me/full` returns the rich camelCase `MeFullResponse` that `currentUser` holds — so seeding from the login response is a shape mismatch. Chose reload-fallback-from-cookies instead. Implemented across 10 files:
- **FIX-CORE:** `refreshCurrentUser()` → `Promise<boolean>` (true only on authenticated 200); both `/api/me/full` fetches get `cache:'no-store'`. `LoginForm`/`SignupForm` only close the modal when actually authed; on failure re-navigate (reload / home / onboarding) to re-establish from the just-set cookies.
- **FIX-LOGOUT:** all 3 explicit-logout handlers (`Header.tsx`, `SecuritySettings.tsx` via `clearCurrentUser(true)`; `profile/[...tab].astro` via inline `removeItem` to avoid bundling the module) clear cached client auth state.
- **FIX-CACHE:** `Cache-Control: no-store` on every `/api/me/full` response.
- **FIX-SMARTFEED:** `SmartFeed` refetches on login-success so inline login flips visitor→member chrome without reload.
- **Tests:** updated `LoginForm`/`SignupForm` mocks to the new boolean contract + added a failure-path regression test each (onSuccess must NOT fire when refresh fails).
All 5 gates green (tsc 0, astro check 0/0/0, lint clean, 6761 tests, build Complete) + live-verified `/api/me/full` 401 now carries `cache-control: no-store`.
