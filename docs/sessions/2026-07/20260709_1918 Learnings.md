# Conv Learnings - 2026-07-09

## 1. jsdom render tests under a hostile-TZ CI leg catch the "dropped timeZone" bug class without a browser
**Topics:** testing, timezone, cc-workflow

**Context:** [TZ-BROWSER-AUTO] was framed as "only a real browser with a controllable tz catches a display bug → Playwright". That premise was half-wrong.

**Learning:** Because the viewer-tz display model formats to the user's **stored** IANA zone (passed as an explicit `{ timeZone }` to `Intl`), NOT the browser's zone, a real browser's `timezoneId` is not what exposes the regression. The regression is a component **dropping** its explicit `timeZone` and falling back to *host*-local formatting. That is exposed by a hostile **host** TZ — which jsdom + the existing CI matrix `['UTC','Pacific/Kiritimati']` already provide. So a jsdom component-render test (render with a mocked `useUserTimezone`, assert the viewer-zone wall-clock) gives durable, gated, deterministic coverage of the exact bug class with no browser.

**Pattern:** Mock `useUserTimezone` per-test (`vi.mock('@lib/current-user', importOriginal spread + override)`), render with a fixed UTC instant, assert the viewer-zone wall-clock + the `" UTC"` null-fallback. Correct code is host-tz-independent (passes explicit `timeZone`), so the test is green on any host but red when the wiring regresses to host-local.

## 2. Single-shot flip-verification across many tests: break the shared formatter + `-t` filter
**Topics:** testing, cc-workflow

**Context:** The project's flip-verification discipline requires every added test to go red when the fix is reverted. Editing 6 source sites to prove 12 tests would be tedious.

**Learning:** Because all 6 islands route through the same `formatSessionTime`, a **single** temporary edit (make it ignore the `tz` arg → host-local) turns every viewer-tz assertion red under `Pacific/Kiritimati`. Combined with `vitest run <file> -t "TZ-BROWSER-AUTO"`, the run isolates exactly the new tests (skips the rest), so one edit flip-verifies all 12 at once. Revert the one function.

**Pattern:** Prove tz-sensitivity of N tests via one shared-helper flip + a describe-name `-t` filter, rather than N per-component source edits.

## 3. vitest v4 treats multiple positional filter args as AND, not OR
**Topics:** testing

**Context:** `npx vitest run fileA fileB … fileF` printed "No test files found" even though each file ran fine individually.

**Learning:** vitest v4 intersects multiple positional filters against a single path (AND semantics), so no single file matches all 6 substrings. Run files one at a time in a loop (or use a single common substring / `-t` name filter) when you need several specific files.

## 4. RTL getByText mechanics for split/combined time nodes
**Topics:** testing

**Context:** Time ranges render as multiple adjacent text nodes (`{start} - {end}`), and modern Node Intl emits a narrow-no-break space (U+202F) before AM/PM.

**Learning:** RTL's default `getNodeText` concatenates an element's **direct** text-node children, so a `<p>` holding `{start} - {end}` matches `getByText('9:00 AM - 10:00 AM')`. RTL's whitespace normalizer collapses U+202F to a regular space, so regular-space assertions match. For nodes with extra punctuation (en-dash `–`, middot `·`), a **regex matcher** (`/9:00 AM.*10:00 AM/`) is robust and avoids reproducing exact glyphs. Ancestor `<div>`s with element children have empty `getNodeText`, so they don't collide.

## 5. CI triggers scope which jobs actually run — corrected an over-stated FYI
**Topics:** testing, deployment, cc-workflow

**Context:** A surfaced FYI claimed the CI e2e job is "red on every push."

**Learning:** `ci.yml` triggers on `push`/`pull_request` to **main** only (no `paths` filter), and dev work lives on `jfg-dev-*` branches → the e2e job is largely **dormant**, not red-on-every-push. Verify a CI job's actual trigger scope (and, where possible, run history) before asserting its impact. `gh` was unauthenticated here, so run-history confirmation wasn't possible — state that limitation rather than infer.
