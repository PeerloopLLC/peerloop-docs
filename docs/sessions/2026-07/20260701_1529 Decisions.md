# Conv Decisions - 2026-07-01

## 1. Course journey top menu = Direction C (stepper in the hero)
**Type:** Strategic
**Topics:** navigation, design

**Trigger:** [SNAV-TOP] Phase 2 — move the course enrollment journey off the left rail onto a top menu.

**Options Considered:**
1. A — two stacked rows (Explore tabs + Journey stepper)
2. B — journey folded into a "My Journey" dropdown
3. C — Journey as a horizontal progress stepper in the course hero; tab bar = pure Explore ← Chosen

**Decision:** Direction C.

**Rationale:** Best funnel visibility; keeps the tab bar identical to the Phase-1 flat pages; cleanly separates "where am I in the funnel" (progress stepper) from "what section am I viewing" (tabs).

**Consequences:** New `CourseJourneyStepper` + `CourseSessionsActions` components; `_course-tabs.ts` split into 3 builders; all 4 course pages rewired. Browser-verified, 5 gates green (test 6731). Committed `e55e49d0`.

## 3. SubNav responsiveness = keep mobile grid + `md:flex-wrap`
**Type:** Implementation
**Topics:** css, navigation

**Trigger:** DOM-measured the SubNav top strip — 7 Explore tabs (scrollWidth 962 > clientWidth 948) overflow the content card even at full desktop width; `/community/[slug]`'s header wraps but the SubNav strip lacked `flex-wrap`.

**Options Considered:**

**Consequences:** One-word fix to the shared `SubNav.astro`; tabs wrap to 2 rows (948=948, no overflow) on course + community. Committed `74e90342`.

## 4. [LAYOUT-MODE] architecture = one model, orientation-aware presentation, one per-user setting
**Type:** Strategic
**Topics:** navigation, astro, design

**Trigger:** Client wants left panels gone site-wide (default); user wants the desktop rail as a reserve — without maintaining two copies of content.

**Options Considered:**
1. Two hand-maintained renderings (revive old zoned rail for ON) — cancels [SNAV-CLEAN], two divergent systems
2. Lightweight hybrid (Explore tabs left, journey stays a stepper) — loses the Journey indentation the user cares about
3. ONE model + orientation-aware presentation components + ONE per-user setting ← Chosen

**Decision:** Option 3. Content authored once; each component renders it top or rail via layout branches (the pattern `SubNav` already uses). The Journey's rail form is a NEW vertical mode of `CourseJourneyStepper`, not the old code.

**Rationale:** No duplicated content; no old-code revival → **[SNAV-CLEAN] still proceeds**; restores the Journey's indentation from the single model.

**Consequences:** Captured in `plan/layout-mode/README.md`; 4 phases (A setting infra / B SubNav orientation + SNAV-CLEAN / C journey vertical mode / D listing filters top-vs-rail). Committed `91aec49`.

## 6. Phase A threading = AppLayout resolves + Astro.locals (not middleware)
**Type:** Implementation
**Topics:** astro, ssr

**Trigger:** [LAYOUT-MODE] Phase A backbone — where to resolve the per-user layout value and how to deliver it to `AppLayout` + slotted `SubNav`.

**Options Considered:**
1. Middleware resolves the user's `nav_layout` into locals — robust but adds a DB query per request (incl. public pages)
2. AppLayout resolves it (reusing its existing per-page user query) + publishes on `Astro.locals` ← Chosen

**Decision:** Option 2 — reuses AppLayout's existing user query (near-zero extra cost); the course/community detail pages are public (middleware short-circuits them anyway). Verified the slot-order/locals path holds.

**Consequences:** `navLayout` added to `App.Locals` (env.d.ts); AppLayout resolves + publishes + drives `subNavOnLeft`; SubNav reads `Astro.locals.navLayout ?? SUBNAV_LAYOUT`. Committed `4f0486ac` + docs `38a4c47`.
