# Conv Learnings - 2026-07-23

## 1. "Cannot reproduce in the bridge Chrome" is evidence of stale client state, not a refuted bug report
**Topics:** cc-workflow, testing, dual-repo

**Context:** A reported `/profile` → `/@sarah-miller` redirect was unreproducible everywhere the bridge Chrome looked (curl 200, real + synthetic click → /profile, pivot identical, no such redirect in `src/`). It turned out to be a browser-cached HTTP 301 from a retired route, live only in the user's Brave.

**Learning:** The bridge Chrome profile is near-fresh by design; the user's Brave carries ~400 convs of localhost history. So for any state-rooted bug, "I can't reproduce it" is nearly guaranteed and points AT the user's browser, not against their report. Default hypothesis when a report doesn't reproduce = stale client state in Brave.

**Pattern:** Diagnostic for a navigation that lands where no current code points: `git log --format='=== %h %cs %s' -S"redirect('<target>', 301)" --name-only -- src/` — finds retired redirects by content, across renames. Confirm via DevTools Network `301 (from disk cache)`; fix via Clear site data (Cmd+Shift+R often won't evict a cached redirect). No server-side remedy exists for already-poisoned clients. Design rule: never 301 an app route that might later become a real page — use 302 or a 301 with explicit Cache-Control.

## 2. Modules ≠ Sessions in the schema — a session-first course IA cannot be built naively
**Topics:** astro, d1, data-model

**Context:** MERGE-BRIAN mechanism 2 proposed merging our Modules + My Sessions into a single session-first tab. The user challenged: modules are public/student-independent, sessions require enrolment.

**Learning:** Confirmed in `migrations/0001_schema.sql`: `course_curriculum` has no enrollment reference (exists with zero students, public); `sessions.enrollment_id` is NOT NULL (no session pre-enrolment); and the link is late-bound — `sessions.module_id` is nullable, "frozen on completion, NULL while scheduled". So a booked-but-incomplete session has NO module and can't render `Session N · <module>`. `homework_assignments.module_id` is also nullable (course-level OR module-level). Consequence: a sessions-first list must invent pseudo-rows for visitors and can't label in-progress sessions — the defensible synthesis is curriculum-first with a session overlay.

**Pattern:** Before adopting an IA that treats two entities as one, check the schema for (a) NOT NULL foreign keys that gate existence, and (b) nullable join columns that are late-bound. Both break naive 1:1 rendering.

## 3. Side-by-side comparison across a fork requires diffing the seeds, not just the code
**Topics:** testing, dual-repo, cc-workflow

**Context:** The MERGE-BRIAN review assumed both local D1s held identical datasets ("dev-level seeded, identical"). They don't — `diff migrations-dev/0001_seed_dev.sql` ours vs the worktree = 52 differing lines (his: accent_color/logo_url, Guy availability, a completion certificate row; ours: [TZ-MODEL] timezone seeding).

**Learning:** When comparing two branches' UI side-by-side, state-dependent differences (journey completion, session counts, certificate/diploma state, availability) may be seed-data artifacts, not UI-logic differences. His "2 of 2 · Certificate" vs our "1 of 2 · Diploma" was partly the certificate seed row, not his code.

**Pattern:** Diff the seed files first; attribute a state difference to code only after ruling out the seed rows behind it. Structural/layout comparisons stay valid regardless.

## 4. A disposition walk needs its process contract stated explicitly up front
**Topics:** cc-workflow

**Context:** After mechanism 1's ADAPT was recorded, the user was confused that the hero on :4321 hadn't changed — they expected each disposition to build immediately.

**Learning:** "Record all decisions, then build, then check" is a legitimate workflow but it is NOT self-evident; a disposition that changes nothing visible reads as a no-op. State the collect→build→check contract before the first decision and confirm it. (User confirmed this order and it's now written into the plan README.)
