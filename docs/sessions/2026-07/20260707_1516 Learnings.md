# Conv Learnings - 2026-07-07

## 1. `localToUTC` single-offset-sample is wrong across DST transitions
**Topics:** timezone, scheduling, date-handling

**Context:** [TZ-AUDIT] availability-deep agent flagged, then reproduced empirically, that `src/lib/timezone.ts`'s `localToUTC` stored the wrong UTC instant for wall-times near a DST change.

**Learning:** Converting "wall time in tz → UTC" needs the offset sampled at the **target** instant, not the wall-as-UTC guess. Since `u = wall − offset(u)` and `offset` changes across DST, a single sample is ±1h off within the transition's offset-window. Non-DST days were always correct, which is why the bug hid — and why the existing tests (all on Mar-20/Jan-15/Jul-15) never caught it. Fix = fixpoint: sample offset, apply, re-sample at the result, re-apply (converges in 2 passes for standard 1h DST).

**Pattern:**
```
let utc = wall − tzOffsetMsAt(wall); utc = wall − tzOffsetMsAt(utc);
```
where `tzOffsetMsAt` reads the tz wall-clock at a given UTC instant via `toLocaleString('sv-SE',{timeZone})`.

## 2. The recurring "TZ feels wrong" is one root cause: no per-user timezone model
**Topics:** timezone, architecture, ux

**Context:** Prior sweeps patched individual `toLocale` sites but the pain kept recurring; the audit found why.

**Learning:** `users` has no `timezone` column, so three surfaces render in three zones (times = browser-local, dates = teacher-local/UTC, emails = UTC) that disagree across a UTC-day boundary. Patching sites individually can't fix it — only committing to ONE zone-per-viewer (a model decision) does. The code already had a `⚠️ TZ-AUDIT #10` marker deferring exactly this.

**Pattern:** When a bug class keeps recurring despite sweeps, look for a missing model/invariant, not more sites to patch.

## 3. Multi-agent read-only audit + parent verification
**Topics:** cc-workflow, audit

**Context:** ~297 `new Date()` + 103 `toLocale` + 94 get/set-field sites — too much for one linear pass.

**Learning:** Fan out N read-only agents by *surface* (each with the same contract rubric + required `file:line` anchors), then the parent verifies a sample of each agent's claims against source before trusting (guards against confabulation). Cross-agent agreement (C+E both flagged the earnings `getPeriodDates` pattern) raises confidence; empirical repro (node) settles the highest-stakes claim (localToUTC).

## 4. Branch can drift mid-conv; verify before committing
**Topics:** cc-workflow, dual-repo, git

**Context:** `/r-start` confirmed `jfg-dev-14`, but the code repo was checked out to `brian-July-7` externally mid-conv; `/r-commit` landed the commit there.

**Learning:** The `[RSTART-DIFFGATE]` branch check only runs at `/r-start`; nothing re-checks at commit time. A mid-conv external checkout goes unnoticed until the commit reports an unexpected branch. Low cost to recover when caught (ff + reset), but worth a glance at `git branch --show-current` before committing when a conv runs long.
