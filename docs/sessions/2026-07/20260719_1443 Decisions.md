# Conv Decisions - 2026-07-19

## 1. React Doctor Is an Occasional External Audit, Not Toolbelt Tooling
**Type:** Strategic
**Topics:** testing, tooling, lint

**Trigger:** User asked for an assessment of the package's usefulness in our codebase.

**Options Considered:**
1. Adopt as standing tooling — CI gate + agent skill + git hook
2. Occasional external audit, run pre-go-live; do not add to the toolbelt ← **Chosen**
3. Reject entirely

**Decision:** Run `npx react-doctor@latest` occasionally as an external audit — explicitly pre-go-live — and do not adopt it into the toolchain. `install` and `ci install` left unexecuted.

**Rationale:** The disqualifier for standing tooling is **structural, not quality**: it analyzed 0 of our 90 `.astro` files, so a gate built on it could stay green while a route regressed — it lacks the property a gate needs. Secondary: `0.x` shipping ~4 versions/day suits deterministic tooling poorly, and `@latest` pins nothing. But as a one-time `npx` audit the cost is genuinely **zero** — no dependency, no lockfile change, nothing committed — so occasional runs are pure upside, and pre-launch is the natural cadence for a11y and dead-code drift.

**Consequences:** Decision entry in `docs/decisions/06-testing-ci.md`; reproducible procedure as `MVP-GOLIVE.CODE-AUDIT` in `plan/mvp-golive/README.md` + PLAN.md block-3 row; `VERNACULAR.md` row; baseline report preserved at `.scratch/rdoc-report-conv397.json`. Three tasks spun out: `[A11Y]`, `[RHOOKS]`, `[RDFIX]`.

> **Insight:** A scanner's value as *audit* and its value as *gate* are separate properties. An audit is judged on yield per run; a gate is judged on whether green is trustworthy. A tool blind to a whole layer of the codebase can score well on the first and be permanently disqualified on the second.

## 2. Run the Evaluation In Place on `jfg-dev-14` — No Branch
**Type:** Implementation
**Topics:** git, dual-repo, evaluation-method

**Trigger:** User asked "Do we need a new branch for this?"

**Options Considered:**
1. In place on `jfg-dev-14`, clean tree as the debris detector ← **Chosen**
2. Isolated worktree
3. Cut `jfg-dev-15` up front

**Decision:** Run Phases 1-3 in place; branch only at Phase 4 if the verdict were adopt.

**Rationale:** Phases 1-2 are read-only and Phase 3's only write was one throwaway line; the clean tree makes `git status` a *detector* for anything the tool drops. A worktree isolates accidental writes only — `npx` runs downloaded code as us either way — so it buys tidiness, not safety. Minting a `jfg-dev-NN` snapshot branch for work intended to be discarded inverts that convention (those branches are permanent point-in-time snapshots). **Constraint recorded:** any future branch here must match `^jfg-dev` or the Conv-396 timecard allowlist silently drops the work.

**Consequences:** Code repo ended the conv untouched and clean at `2d148da7`; zero code commits this conv.

## 3. Keep the Evaluation Run Local — Both Privacy Flags
**Type:** Implementation
**Topics:** tooling, security, privacy

**Trigger:** The CLI defaults to Sentry telemetry and a Socket.dev supply-chain scan, both network-transmitting.

**Options Considered:**
1. Accept defaults (telemetry + supply-chain both on)
2. `--no-telemetry` only
3. `--no-telemetry` **and** `--no-supply-chain` ← **Chosen**

**Decision:** Ran with both flags; the supply-chain scan was called out to the user as a separate opt-in rather than allowed to ride along.

**Rationale:** Telemetry reports rule names + counts and project shape to Sentry; the supply-chain scan transmits the dependency manifest to Socket.dev. Neither is high-risk (package names are public), but sending content to an external service is a publish, and it deserves an explicit nod rather than a silent default — particularly on a repo shared with the client.

**Consequences:** Both flags recorded in the `MVP-GOLIVE.CODE-AUDIT` procedure so future runs reproduce the same posture.

## 4. Decompose the Findings by Coverage Gap, Not by Tool
**Type:** Implementation
**Topics:** task-management, lint

**Trigger:** User asked for tasks covering "the categories that were found that we currently did not".

**Options Considered:**
1. One `[RDOC-FOLLOWUP]` task holding all 712 findings
2. Three tasks split by the coverage gap each represents ← **Chosen**

**Decision:** `[A11Y]` (no a11y linting anywhere — 154 findings), `[RHOOKS]` (15 unenabled rules we already own — 108 findings), `[RDFIX]` (confirmed defects, false positives stripped).

**Rationale:** Each is a vertical slice with its own standalone done-test and its own real decision — `[A11Y]`'s is *which permanent linter*, `[RHOOKS]`'s is *severity landing strategy*, `[RDFIX]`'s is *delete-safety*. A single task would have buried three unrelated decisions behind one checkbox. Verified false positives were listed explicitly as *do not action* inside `[RDFIX]` so a future conv doesn't re-litigate them.

**Consequences:** Three backlog rows in `CURRENT-TASKS.md`, each self-sufficient (TodoWrite is active-only and wiped each conv, so the detail must live in the file).
