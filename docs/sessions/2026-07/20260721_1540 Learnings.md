# Conv Learnings - 2026-07-21

## 1. A diagnostic probe run inside the Bash tool cannot observe the session's own environment
**Topics:** cc-workflow, debugging

**Context:** Conv 403 diagnosed missing task tools by running `echo $CLAUDE_CODE_CHILD_SESSION` from the Bash tool, saw `1`, and concluded VS Code was leaking a child-session marker. Conv 404 repeated the same probe at `/r-start` and reached the same wrong conclusion — twice.

**Learning:** Claude Code **injects** `CLAUDE_CODE_CHILD_SESSION=1` and `AI_AGENT=claude-code_*_agent` into the Bash tool's subprocesses to mark its own child shells. That probe therefore prints `1` in *every healthy session* — it has no discriminating power and can only ever produce a false positive. The session's real environment must be read from the `claude` process itself.

**Pattern:**
```bash
ps -wwwE -p $(pgrep -n claude) | tr ' ' '\n' | grep -E '^VAR='
```
Cross-check the parent shell and terminal host the same way. Generalized: *the Bash tool's environment is not the session's environment.*

## 2. Check the vendor's default-off list before theorizing about a leak
**Topics:** cc-workflow, external-source-of-truth

**Context:** Two convs were spent theorizing about env contamination for a tool that had simply been turned off by default upstream.

**Learning:** `TodoWrite` has been disabled by default since CC **v2.1.142**, superseded by the `Task*` tools (`code.claude.com/docs/en/tools-reference.md`, `…/agent-sdk/todo-tracking.md`). A single docs check would have pre-empted the entire Conv-403 investigation. This is the `[VDF]` guard (`feedback_external_source_of_truth_first`) applied to the harness itself, not just to app vendors — `claude-code-guide` is the fast path.

**Pattern:** Missing built-in tool → check the official tool-reference for a default/flag **first**; only then investigate the environment.

## 3. Reading every call site before extracting a primitive changes the primitive
**Topics:** architecture, a11y

**Context:** A shared primitive was proposed and approved on the strength of "13 sites, real reuse". Reading all 13 first showed they were 4 distinct a11y families.

**Learning:** 7–8 of the 13 were dismiss backdrops, where the proposed `role="button"` + `tabIndex` fix would have made accessibility **worse** — seven focusable decorative divs in the tab order. The reuse count that justified the extraction only ever supported 4 sites. Aggregate lint counts describe symptoms, not shared structure; grouping by rule name is not grouping by contract.

**Pattern:** Before extracting from N lint hits, read all N sites and cluster by *required contract*, not by the rule that flagged them. Cohesion split (`feedback_decompose_by_cohesion_not_pseudo_isolation`) applies to primitives too.

## 4. The correct idiom was already in the codebase, unflagged
**Topics:** a11y, architecture

**Context:** Searching for how to fix 7 backdrops.

**Learning:** `Modal.tsx:92` already had the right pattern — `aria-hidden="true"` + `onClick`, no `tabIndex`, no role — which is precisely why the linter flags `Modal.tsx:83` but not `:92`. `aria-hidden` removes the element from the accessibility tree, satisfying both `click-events-have-key-events` and `no-static-element-interactions` without making anything focusable. The extraction became *formalizing an existing correct instance* rather than inventing a pattern.

**Pattern:** When a linter flags N instances, look for the instance it *doesn't* flag — that's usually the reference implementation.

## 5. A batch reachability check must cover the batch
**Topics:** cc-workflow, orphan-detection

**Context:** `[ORPHAN-DETECT]` reachability was verified for `HomeworkEditor` before editing, then not re-applied to the 7 backdrop sites picked up afterward. One of them, `TestimonialsBrowse`, is a known Cat-B orphan.

**Learning:** The check was treated as a per-task gate rather than a per-file gate, so it silently lapsed the moment the file set grew mid-task. Two of the 28 cleared warnings are on code no user reaches — the honest figure is 26.

**Pattern:** Run the orphan detector against the whole diff (`git status --short` ∩ orphan list) *before* reporting results, not just against the first file touched.

## 6. Superseding a wrong decision needs the correction above the claim
**Topics:** docs-infra

**Context:** `DOC-DECISIONS.md` §3's title *was* the false conclusion ("Launchers Must Strip …"), so a footnote would have been read after the damage.

**Learning:** For append-only decision logs, the correction belongs in the heading and in a blockquote *preceding* the original text, with the original retained below. Archival session docs (`docs/sessions/**`) stay untouched by policy; the live index carries the warning that they contain the falsified version.
