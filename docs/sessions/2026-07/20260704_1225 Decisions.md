# Conv Decisions - 2026-07-04

## 1. Mobile ControlBar → 5 icons (add Members)
**Type:** Implementation
**Topics:** navigation, mobile

**Trigger:** MOBNAV Conv-361 ⬜ follow-up — Communities↔Members ambiguity in the bottom bar.

**Options Considered:**
1. Keep Communities only
2. Swap Communities → Members
3. Add both — Home·Courses·Communities·Members·Messages ← Chosen

**Decision:** Widen the bottom pill to 5 bare shortcuts, adding Members (`student-teacher` icon) alongside Communities.

**Rationale:** User chose "both"; the ~314px pill still fits any viewport ≥314px.

**Consequences:** `ControlBar.tsx` + test updated; layout doc §8.6.4 records 5 icons.

## 3. Mobile contextual up-nav = compact up-chevron, deep flows only, above the hero
**Type:** Strategic
**Topics:** navigation, ux, mobile

**Trigger:** Breadcrumbs (the "up" affordance) are desktop-only (`hidden lg:block`), leaving mobile deep pages with no in-app up-navigation.

**Options Considered:**
1. Surface the full breadcrumb trail on mobile
2. Compact up-chevron (single `‹ parent`) ← Chosen (form)
3. `history.back()` button ← rejected as fragile

**Decision:** Add a `lg:hidden` compact `‹ {parent}` chevron (`MobileUpNav`) to the **deep-flow pages only**, rendered **above the hero** via a new AppLayout `mobile-upnav` slot; target the deterministic immediate-parent href (never `history.back()`).

**Rationale:** Up-target is deterministic and thumb-friendly; a chevron doesn't crowd the header pill; top-level pages skip it (their "up" is Home, already a ControlBar shortcut).

**Consequences:** New `MobileUpNav.astro` + AppLayout slot + 7 wired pages + 8 tests. Layout doc §8.6.4 updated.

## 4. Journey-page hero conformity (SHERO + BHERO)
**Type:** Implementation
**Topics:** courses, layout, ux

**Trigger:** Tapping the "Payment" journey step relocated the hero to the page bottom (`/success`), and `/book` had no hero at all — inconsistent with the `[...tab]` journey destinations.

**Options Considered:**
1. Leave the bespoke per-page layouts
2. Make every journey destination render the hero at the top via `entity-header` ← Chosen

**Decision:** Move `CourseHeader` into the `entity-header` slot (top) on `/success` (removing the bottom `variant="enrolled"` card), and add a `CourseHeader` to `/book` (which had none) fed by the shared `fetchCourseTabData` loader.

**Rationale:** All four journey destinations (benefits/sessions/success/book) should present the hero identically; the bottom card's "back to course" role is now covered by the MOBUP up-chevron. `isEnrolled={true}` on both (enrollment guaranteed/verified).

**Consequences:** `success.astro` + `book.astro` edited; all journey destinations now hero-at-top. DOM-verified.
