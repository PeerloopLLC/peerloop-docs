# Conv Learnings - 2026-07-20

## 1. grep-based dead-code detection has structural blind spots; a module-graph oracle closes them
**Topics:** dead-code, tooling, refactoring

**Context:** The user pushed back on deleting anything while the "is it in use" check had known holes.

**Learning:** `git grep` importer analysis mis-classifies dead code in five distinct ways: (1) same-named local variables inflate the hit count (`hasRole` had 90+ hits, all local `const hasRole`), (2) relative-path imports get missed by a path-shaped pattern (`from './styles'` invisible to a `emails/styles` grep — nearly read a live 17-importer file as dead), (3) barrel re-exports look like usage but are passthroughs, (4) internal/namespace self-use isn't a `{ named }` import, (5) `.astro` markup is invisible to a JS-only text search. A production build (or a module-graph tool) resolves all of these because it follows the real graph from entry points and tree-shakes.

**Pattern:** Rule of thumb — **absent from the build = dead (bundler-grade); present = keep.** For deletion decisions, prefer a module-graph oracle over grep; if only grep is available, re-check every path shape a module could be imported under before trusting an "unused" verdict.

## 2. knip parses `.astro` and is the productized form of the hand-rolled orphan detector
**Topics:** knip, tooling, astro, dead-code

**Context:** Chose knip as the durable reachability oracle over a one-off build check.

**Learning:** `knip@6` auto-detects Astro/vitest/playwright plugins from `package.json`, compiles `.astro` frontmatter (closing grep's one structural hole), and reports unused files + exports + deps. Configured for this repo it needed only two gap entries — the second worker (`workers/cron/src/index.ts`) and `scripts/**` — to produce a trustworthy first run: it confirmed all 9 RDFIX candidates, independently reproduced the 14 parked `[ORPHAN-BACKLOG]` Cat-B files, and kept `styles.ts` live. It is effectively `codecheck-orphan-components.mjs` generalized (that detector is scoped to `src/components/**`; knip covers all of `src/` + `.astro`).

**Pattern:** knip's file/export reachability was solid on first run; its *dependency* analysis had false positives (`zod`/`tailwindcss`/`react-day-picker` used via CSS `@plugin`/runtime, not JS import) — trust the file/export sections before the dependency section, and tune the latter before gating on it.

## 3. Side effects inside a setState updater can double-fire — mirror-ref + compute-outside is the fix
**Topics:** react, hooks, bug-fix

**Context:** The `SessionRoom.tsx:126` `no-impure-state-updater` finding.

**Learning:** Calling a *different* setter (`setTimeUntilJoin`) inside a `setState(prev => …)` updater is impure: React may invoke updater callbacks more than once (Strict Mode, concurrent rendering), so the side effect can repeat or observe inconsistent external state. The updater form was originally chosen to read the *latest* `viewState` inside a stable `setInterval` closure — the correct way to keep that freshness without the impurity is a `useRef` mirror of the state, read inside the interval, with all computation + setters moved outside any updater.

## 4. Deleting one export can orphan another — check the local dependency chain before removing
**Topics:** dead-code, refactoring

**Context:** Removing `MEMBER_ROLE_COLORS` orphaned `MONITORING_COLORS` (its only consumer) and the `MemberRole` type import.

**Learning:** A "dead" export may be the sole consumer of *other* exports or imports in the same file. Removing it cascades: the import becomes unused (must be cleaned to keep tsc/lint green — that's in-scope follow-through), and any export it solely consumed becomes newly dead (out of an approved narrow scope — leave it, and let the next detector run catch it rather than silently expanding the deletion set).
