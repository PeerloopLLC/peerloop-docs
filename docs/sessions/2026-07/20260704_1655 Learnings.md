# Conv Learnings - 2026-07-04

## 1. Staging schema drifts from base 0001 because reset isn't automatic
**Topics:** d1, deployment, testing

**Context:** `/profile` 500'd on staging only; every other page worked.

**Learning:** Because `migrations/0001_schema.sql` is edited **in place** pre-launch (not shipped as incremental `000N` migrations), any column added to it since a staging DB was last reset is **missing** on that staging DB. The symptom is narrow: only an **unguarded** reader of the new column 500s; guarded readers (try/catch) degrade silently, so most of the app looks fine. A full `db:setup:staging:*` reset+reseed rebuilds from the current `0001` and clears **all** such drift at once ÔÇö strictly better than a surgical `ALTER` when the staging data is disposable seed data.

**Pattern:** Probe drift with `wrangler d1 execute DB --env staging --remote --command "SELECT sql FROM sqlite_master WHERE name='users'"` (or `PRAGMA table_info`).

## 2. The staging seed chain differs from the local seed chain ÔÇö `:booking` skips feeds
**Topics:** d1, deployment

**Context:** After `db:setup:staging:booking`, staging Home/Discover feed was empty.

**Learning:** `db:setup:staging:dev` = reset + migrate + `db:seed:staging` only ÔÇö it does **NOT** bundle feeds. Feeds are seeded on staging only by the top-level `db:setup:staging:feeds` (or a direct `db:seed:feeds:staging`). This is UNLIKE the **local** chain, where `db:setup:local:dev` DOES include `db:seed:feeds:local`. Don't assume the staging `:level` chain mirrors the local one.

## 3. `node --check` misses browser-global collisions
**Topics:** testing, astro

**Context:** A published Artifact threw a console error and every button was dead; `node --check` on the extracted script reported clean.

**Learning:** `const chrome = ...` at script top-level is a **parse error in the browser** ("Identifier 'chrome' has already been declared") because `chrome` is a predefined browser global (`window.chrome`) ÔÇö but Node has no such global, so `node --check` passes. For browser JS, a real parse/runtime check requires a browser (served the file locally, drove it in Chrome, read the console). Avoid `chrome`, `name`, `top`, `status`, `open`, etc. as top-level `const`/`let` names.

## 4. python `http.server` serves without charset Ôćĺ mojibake (test artifact, not a bug)
**Topics:** testing

**Context:** Local-served mockup showed `├óÔéČ"` for `ÔÇö`, `├âÔÇö` for `├Ś`.

**Learning:** `python3 -m http.server` sends no `charset=utf-8`, so Chrome guesses Latin-1 and multi-byte UTF-8 renders as mojibake. This does NOT reproduce in the Artifact wrapper (which serves proper UTF-8) ÔÇö so garbled text in that local harness is an artifact of the harness, not the page.

## 5. SmartFeed is a mixed-kind feed ÔÇö interleaving new card kinds is cheap
**Topics:** feed, ui

**Context:** Needed to add periodic in-feed conversion cards for visitors.

**Learning:** `SmartFeedItem = PostFeedItem | SuggestionCardFeedItem | AnnouncementFeedItem` ÔÇö the feed already interleaves non-post cards (suggestion cards, announcements), each with dismiss support. A new visitor CTA is a render-time interleave over the mapped nodes (`withSignupCta` helper), no API change, gated on `viewerAuthenticated === false` (server-baked into the `/api/feeds/smart` payload).

## 6. The Matt shell has no visitor auth affordance; HeaderBar is mobile-only
**Topics:** navigation, ui

**Context:** Deciding where a persistent visitor "Sign up" lives.

**Learning:** `HeaderBar.astro` is `lg:hidden` (mobile/tablet only); at desktop the chrome is the **Sidebar** + an inline breadcrumb (page-owned `header-bar` slot). Neither `Sidebar.tsx` nor `HeaderBar` had ANY visitor login/signup ÔÇö visitors only got a generic "Visitor Ôćĺ /profile" row. `NavDrawer` renders the same `Sidebar.tsx` (`variant="drawer"`), so a visitor affordance added once in Sidebar covers desktop **and** mobile.

## 7. Dark mode was effectively dropped in the Matt porting
**Topics:** ui, astro

**Context:** Confirming the light/dark provision across the codebase.

**Learning:** The only theme control is `ThemeToggle`, rendered only in `/profile`. The `.dark` mechanism still exists (`@custom-variant dark`, 392 `dark:` utilities across 48 files) but those are legacy; Matt-inspired pages use design tokens, not `dark:` variants, so toggling produces an inconsistent half-dark UI. `dev-login` is `import.meta.env.DEV`-gated, so it's disabled on staging/prod builds (can't script an authed staging session).
