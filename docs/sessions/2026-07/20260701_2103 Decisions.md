# Conv Decisions - 2026-07-01

## 1. Phase C: relocate the rail-mode journey via a `CourseRail` wrapper (not inline per-page slotting)
**Type:** Implementation
**Topics:** astro, layout-mode, ui-ux

**Trigger:** Phase C carried an open architecture fork — how the 4 course pages should wire the rail-mode journey relocation (the vertical journey + nested Sessions cluster moving into a left rail column).

**Options Considered:**
1. Inline per-page slotting — each of the 4 course pages assembles the rail column in its own `sub-nav` slot.
2. A dedicated `CourseRail.astro` wrapper that owns the rail-column assembly ← Chosen (user).

**Decision:** Build a new `CourseRail.astro` that owns the course `sub-nav` slot: top mode = SubNav byte-identical passthrough; rail mode = a 196px flex column (SubNav + vertical journey + nested Sessions cluster, `display:contents` below `lg`). The 4 course pages swap `SubNav` → `CourseRail`.

**Rationale:** DRY — a single home for the rail-column assembly, versus repeating the same slot wiring across 4 pages. User pick.

**Consequences:** New `CourseRail.astro`; `orientation` prop added to `CourseJourneyStepper` + `CourseSessionsActions`; rail Sessions cluster nests via a `sessions-cluster` slot; 4 course pages rewired. DOM-verified both modes; 5 gates green (test 6732). Commit `1fb23ac2` (code) / `2c6f188` (docs).

## 2. Phase D top-bar: Option B — minimal bar + inline Filters collapse (not all-controls-inline)
**Type:** Implementation
**Topics:** astro, layout-mode, ui-ux

**Trigger:** In top (non-rail) mode the listing-page filter islands need a horizontal presentation; the rich Courses filter set risked crowding a single inline bar.

**Options Considered:**
1. Option A — all controls inline in the top bar.
2. Option B — a minimal bar (search + Sort inline + a Filters toggle that collapses the attribute controls / role pills) ← Chosen (user).

**Decision:** Top mode renders a minimal bar: search + Sort inline, plus a Filters collapse holding the attribute controls / role pills (Communities has none). Search + Filters + Sort sit in a `shrink-0` cluster. Rail mode keeps the original vertical stack.

**Rationale:** Matches the client's top/minimal navigation intent and avoids crowding the rich Courses filters into one row. User pick. (CC self-decided the mechanics: `navLayout` drives it, single-column in top, inline-above-catalog placement.)

**Consequences:** `ListingShell` navLayout branch (rail = 320px left aside; top = single centered column); `orientation` prop on the 3 filter islands; 3 pages place inline (top) vs right-panel slot (rail). DOM-verified courses + communities (+ members top); 5 gates green (test 6732). Commit `51daa758` (code) / `e3a20e8` (docs).

## 3. Resolve per-user `navLayout` in middleware (`resolveNavLayout`), not layout/page frontmatter
**Type:** Implementation
**Topics:** astro, layout-mode, architecture

**Trigger:** Phase D hit a rail double-render (inline top island + empty left aside). Root cause: Astro evaluates a page's slot-content expressions eagerly, before the wrapping `AppLayout`'s frontmatter runs — so the `Astro.locals.navLayout` the layout writes is not reliably readable in the page's own template, and the page's read diverged from `ListingShell`'s.

**Options Considered:**
1. A per-page frontmatter helper each page calls before slotting.
2. Have `ListingShell` position everything so no page reads `navLayout`.
3. Move `navLayout` resolution to middleware, which runs before page frontmatter ← Chosen.

**Decision:** New `resolveNavLayout` in `src/middleware.ts` resolves the per-user nav layout onto `Astro.locals` for page requests before page frontmatter runs. `AppLayout` prefers the middleware-set `Astro.locals.navLayout` (fallback to its own resolution).

**Rationale:** This is the durable fix for the eager-slot-eval limitation — resolving in middleware guarantees the value is present before any page slot expression evaluates. It also strengthens the Phase A–C reads (which relied on the layout-published value) by giving them a single, earlier, reliable source. It reverses the Conv-355 Phase-A note that resolution was "chosen over a middleware query per request."

**Consequences:** `src/middleware.ts` +`resolveNavLayout`; `AppLayout` prefers `Astro.locals.navLayout`. The eager-slot-eval double-render resolved; both modes DOM-verified; 5 gates green. Recorded in `docs/decisions/05-ui-ux-components.md` (LAYOUT-MODE entry) + decision-log. Commit `51daa758` (code).
