# Conv Learnings - 2026-07-11

## 1. CUT-3 completion fires against a live dev server via the BBB meeting-ended trigger
**Topics:** testing, plato, video, booking, browser-run

**Context:** The Phase-3 activities walk needed to complete a browser-booked session, but the completion is a checksum-verified BBB webhook — the OPEN "how to fire CUT-3 on `npm run dev`" blocker carried from Conv 382.

**Learning:** `SESSION_ID=<booked-id> ./scripts/trigger-webhook.sh bbb-meeting-ended` is the mechanism. The script sends `{event:"meeting-ended"}` with an HMAC-SHA256 token over the session id (`generate_bbb_token`); `src/lib/video/bbb.ts:543` maps `meeting-ended`→`room_ended`; the handler looks up the session by `id=roomId`, and `completeSession()` (`booking.ts:93`) accepts a session in `status='scheduled'` (not just `in_progress`) — it backfills `started_at` from `durationSeconds` (5400). So a single trigger completes a freshly-booked, never-joined session: freezes `module_id`, writes a `module_progress` row. No empty-room/attendance flow needed. This resolves the blocker and is the reusable CUT-3 bridge for the ecosystem walk.

**Pattern:**
```bash
SESSION_ID=$(sqlite3 "$DB" "SELECT id FROM sessions WHERE status='scheduled' LIMIT 1") ./scripts/trigger-webhook.sh bbb-meeting-ended
```

## 2. The student homework-SUBMISSION UI is orphaned in the Matt route-flip
**Topics:** plato, matt-migration, regression, homework, astro

**Context:** The walk's "submit homework" intent (planned as pure-UI/browser) had no reachable UI. Checked `buildCourseTabs` (`_course-tabs.ts`), `StudentDashboard`, and `/learning/[...tab].astro`.

**Learning:** `src/components/learning/HomeworkTab.tsx` (where Conv 345 built the file-upload submission form) is only reachable via `components/courses/LearnTab.tsx` → `CourseTabs.tsx`, and **CourseTabs is mounted by no live page**. The current Matt course page `src/pages/course/[slug]/[...tab].astro` (Conv 166 route-flip) exposes About/Course Feed/Meet the Creator/Teachers/Reviews/Resources/Modules (+ session funnel) — no Learn/Homework tab. So an enrolled student cannot submit homework through any UI; the creator side (create homework via `/creating/studio`) works. Classic Matt DISC-DROP — a legacy feature enhanced (Conv 345) after it was already orphaned by the Conv-166 migration. Tracked as `[HW-SUBMIT-UI]`.

## 3. Reusable browser-walk mechanics (Chrome bridge)
**Topics:** plato, browser-run, chrome-bridge

**Context:** Driving the full activities journey through the Chrome bridge as two actors.

**Learning:**
- (a) Actor-switch = `POST /api/auth/dev-login {email}` + `localStorage.clear();sessionStorage.clear()` + hard nav ([BRIDGE-MEM]).
- (b) **Click by `ref` from `find`, not raw coords** — the `computer` tool uses screenshot px (1242w) vs CSS px (1177w, ~1.055 scale).
- (c) **Islands hydrate late**: a first ref-click on a freshly-rendered island button (Follow, New Message, editor tabs) can silently no-op; re-click (or click after a settle) works.
- (d) `plato:capture` is WAL-safe with the server up; restore needs it down.
- (e) SQL bridges applied via `sqlite3` directly to the live miniflare D1 are seen by the running server on its next read.

## 4. Row-identity validation is per-table COUNT, not exact column values
**Topics:** plato, testing

**Context:** The browser walk picks arbitrary booking slots/timestamps, so `sessions.scheduled_start` etc. differ from the API producer.

**Learning:** The Conv-381 "row-identical" bar is per-table row COUNT across all app tables (70 here), not column-level equality — so slot choice, generated ids, and timestamps are free. All 70 tables matched and all 8 `activities` verify assertions passed, reproving the dual-producer claim for the activities journey.
