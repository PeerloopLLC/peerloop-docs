# Conv Learnings - 2026-07-01

## 1. Astro evaluates page slot-content expressions EAGERLY — before a wrapping layout's frontmatter runs
**Topics:** astro, architecture

**Context:** Phase D relocated the listing-page filters to a right-panel slot in rail mode, with the page reading `Astro.locals.navLayout` (written by the wrapping `AppLayout`) to decide inline-vs-slot placement. This produced a rail double-render: the inline top island AND an empty left aside both appeared.

**Learning:** A page's `<slot>`-content expressions are evaluated **eagerly**, before the wrapping layout's frontmatter executes. So a value the layout writes to `Astro.locals` is NOT reliably readable in the page's own template/frontmatter — the page read and `ListingShell`'s read of the same `navLayout` diverged, and both branches rendered. The durable fix is to resolve per-request state in **middleware** (which runs before page frontmatter), not in the layout.

**Pattern:** `src/middleware.ts` gained `resolveNavLayout`, which resolves the per-user nav layout onto `Astro.locals` for page requests before page frontmatter runs. `AppLayout` prefers the middleware-set `Astro.locals.navLayout` (falling back to its own resolution). Any state a page must read in its slot expressions should be middleware-resolved, not layout-written.

## 2. Orientation-aware component via a responsive split — no page needs to read `navLayout`
**Topics:** astro, navigation

**Context:** Phase C needed the course journey to render either as a top band (default) or a vertical rail column, off one authored model, without every consuming page branching on the layout setting.

**Learning:** Rendering both orientations and letting breakpoint utilities pick — a `top` instance (mobile-visible, `lg:hidden` in rail mode) and a `rail` instance (`hidden lg:flex`) — means exactly one shows per breakpoint, so no page needs to read `navLayout` for that component. This mirrors the pattern `SubNav` already uses, keeping content authored once.

**Pattern:** `CourseJourneyStepper.astro` / `CourseSessionsActions.astro` take an `orientation: 'top' | 'rail'` prop; the top band gains `lg:hidden` in rail mode while the rail column is `hidden lg:flex`. `CourseRail.astro` assembles the rail column.

## 3. A mixed-DOM state that survives a HARD reload is a real SSR bug, not a ClientRouter artifact
**Topics:** astro, testing

**Context:** The Phase D rail double-render (inline top island + empty left aside coexisting) initially looked like a ClientRouter view-transition persistence artifact — the kind of stale-DOM carryover that view transitions can produce.

**Learning:** The mismatch **persisted through a hard reload**, which proves it is a real SSR bug (here, the eager-slot-eval issue), not a transition-persistence artifact. Diagnostic rule: hard-reload first to rule out ClientRouter transition persistence — a mismatch that survives the hard reload is genuinely produced by SSR and must be root-caused there.

## 4. `flex-wrap` + `flex-1` with a large `min-width` can push a sibling control to a new row
**Topics:** astro, tailwind

**Context:** Building the Phase D minimal top bar (search + Sort + a Filters toggle), the right-side controls dropped to a second row instead of sitting on the search row.

**Learning:** With `flex-wrap`, a `flex-1` element carrying a large `min-width` consumes enough width that a sibling control wraps to the next line. Grouping the right-side controls into a single `shrink-0` cluster keeps them on the search row.

**Pattern:** In the filters islands' top-bar branch, search + Filters + Sort live in a `shrink-0` cluster so they stay together on the search row.
