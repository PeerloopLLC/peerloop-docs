# Conv Learnings - 2026-07-19

## 1. A Tool's Engine Docs Do Not Establish Its Coverage — Measure `analyzedFiles`
**Topics:** tooling, astro, evaluation-method

**Context:** The oxlint docs state that framework files (`.vue`, `.svelte`, `.astro`) are linted "by linting only their `<script>` blocks", which set the expectation that react-doctor would give our 90 `.astro` files partial coverage.

**Learning:** It gave them **zero**. The JSON report's `analyzedFiles` array contained 0 `.astro` entries out of 1268 files, and `framework` came back `"unknown"` on an Astro project. The engine's documented capability said nothing about whether this consumer invokes it. **Always measure coverage from the tool's own output before trusting a capability claim** — and prefer `--json` precisely because it exposes coverage data the human-readable output never shows.

**Pattern:**
```bash
node -e "const r=require('./report.json'); const ext=f=>f.match(/\.[a-z]+$/i)?.[0]; const by={}; r.projects[0].analyzedFiles.forEach(f=>by[ext(f)]=(by[ext(f)]||0)+1); console.log(by)"
```

## 2. A Bundled Dependency Does Not Mean Its Rules Run
**Topics:** tooling, lint, evaluation-method

**Context:** `react-doctor`'s `package.json` lists `eslint-plugin-react-hooks: ^7.1.1` — the *exact* version we already ship. The natural inference was that adopting react-doctor would subsume the hooks rules, making the two tools redundant.

**Learning:** Exactly backwards. react-doctor's 712 findings included **zero** `react-hooks/*` rules; its own 413-rule catalogue has no equivalent (the sole "react-hooks" string is `preact-no-react-hooks-import`). The dependency is used internally, not surfaced. **Check the fired-rule set and the tool's own rule catalogue — never infer capability from the dependency list.** Had we skipped Phase 3, we would have adopted react-doctor believing it covered the hooks rules and silently kept 91 `set-state-in-effect` findings invisible.

## 3. "What Do We Already Own But Not Run?" Is a Cheap, High-Yield Audit Question
**Topics:** lint, tooling, technical-debt

**Context:** Building the control for Phase 3 required enumerating what our installed `eslint-plugin-react-hooks` actually exposes.

**Learning:** It exposes **29 rules**; `recommended-latest` enables 17; `eslint.config.js` enables **2**. Fifteen React Compiler rules — `set-state-in-effect`, `purity`, `immutability`, `static-components`, `preserve-manual-memoization` and more — have been sitting in `node_modules` unused, and produced 108 findings on first run. **This question is independent of any new tool and costs one command to answer.** Worth asking of every plugin we ship, not just when a vendor prompts it.

---

## 4. Enabling Lint Rules Is a Baseline-Gate Decision, Not a Config Tweak
**Topics:** lint, testing, baseline-gates

**Context:** The obvious follow-up to Learning 3 is "just turn them on."

**Learning:** 105 of the 108 control findings are `error` severity, and `npm run lint` is one of the five baseline gates — so flipping `recommended-latest` on turns the gate **red immediately** and blocks *all* baseline verification until 105 findings are triaged. The severity distribution has to be measured before enabling, and the landing pattern is `warn`-first. Our own history already encodes this: `[LE-TRIAGE]` (Convs 147-149) did exactly that for `exhaustive-deps` — 31 warnings, set to `warn`, triaged incrementally — and the rationale is still a comment in `eslint.config.js`.

## 5. Read the Code at Sampled Sites — Finding Counts Are Not Findings
**Topics:** evaluation-method, lint

**Context:** 712 findings is an impressive-sounding number that means nothing on its own.

**Learning:** Reading actual code at sampled sites moved the verdict materially in **both** directions. Downward: 1 of the 2 `error`-severity hits was a false positive (`MyStudents.tsx` — the cleanup exists in a separate unmount `useEffect` using the captured-ref pattern the rule can't follow), and `workers/cron/src/index.ts` was flagged unused when it's `main` in its own `wrangler.toml`. Upward: 3 of 3 sampled a11y warnings were genuine, and the dead-code pass found real hits (`src/emails/*`, dead `db/types.ts` exports) that our own detector **structurally cannot see** because it's scoped to `src/components/**`. Neither correction was visible from the summary counts.

## 6. Branch Naming Is Now Load-Bearing for Billing
**Topics:** dual-repo, timecard, git

**Context:** Deciding whether the evaluation needed its own branch.

**Learning:** Conv 396's `codeBranchAllowPattern: "^jfg-dev"` is a **whitelist**, not a blacklist. Any branch not matching `^jfg-dev` is invisible to every timecard surface — so a descriptively-named branch (`rdoc-eval`, `react-doctor-assess`) would zero the work's billable minutes with no error and no warning. Future branches must be `jfg-dev-NN`. This is a new constraint that did not exist before Conv 396 and is not self-evident from the branch-naming convention alone.
