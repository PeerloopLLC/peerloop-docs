## 2. Bucket-3 Hardening: Make Server Date Math Explicit UTC
**Type:** Implementation
**Topics:** testing, cloudflare, timezone

**Trigger:** Phase 3(b) required hardening host-tz-dependent server date math (earnings period boundaries, analytics bucketing loops, moderation expiry, cleanup date stamps) across 13 files. Before committing, verified whether the test env is UTC — which would make the change a cosmetic no-op.

**Options Considered:**
1. Leave the bare `Date` constructor / local `getDate`/`setDate` math (host-tz-dependent) — assumed a no-op if tests run in UTC
2. Rewrite all server date math to explicit `getUTC*` / `Date.UTC` / `setUTC*` ← Chosen

**Decision:** Convert all host-tz-dependent server date math to explicit UTC: earnings `getPeriodDates` ×2, 7 analytics bucketing loops, 3 moderation expiry helpers, and `lib/cleanup.ts` notification date stamps (`{timeZone:'UTC'}`).

**Rationale:** The vitest host is `America/Toronto` (UTC−4), NOT UTC, and the setup does not force `TZ=UTC` (see Learnings #1). So the prior bare-`Date` math produced host-local (Toronto) boundaries in the test env that diverged from the UTC Worker — this is a real behavioural fix that aligns the test env with production, not a no-op. Explicit UTC accessors are the only form that is both test-deterministic and production-correct.

**Consequences:** 13 files changed (`5db13be6`); establishes UTC-explicit server date math as the standard for any stored/compared date arithmetic (routed to `docs/decisions/06-testing-ci.md`). 5 gates green (tsc · lint · astro 0/0/0 · full suite 6784✓ · build).
