# Conv Learnings - 2026-07-10

## 1. PLATO has exactly 4 external-service cut points; everything else is browser-drivable
**Topics:** testing, plato, stripe, video

**Context:** A pure browser-run of the flywheel dead-ended at enrollment. A mapping agent read all 25 steps + `MockRegistry`.

**Learning:** `MockRegistry.resetAll()` applies an `all-silent` default before every step (notifications, email, Stream feeds, BBB `createMeeting`/`getJoinUrl` always stubbed). So a step is only a HARD boundary when it hits one of 4 endpoints:
- **CUT‑1** `self-certify-creator.connectStripe` → `POST /api/stripe/connect` (Stripe Connect onboarding, sets `stripe_account_id`)
- **CUT‑2a** `enroll-student.createCheckout` → `POST /api/checkout/create-session` (off-site Stripe Checkout)
- **CUT‑2b** `enroll-student.stripeWebhook` → `POST /api/webhooks/stripe` (signature-verified — *this* creates the `enrollments` row)
- **CUT‑3** `complete-course`/`book-complete-session` completion → `POST /api/webhooks/bbb` (checksum BBB `room_ended`)

Everything else — incl. the teacher-cert row (only the Stripe part of self-cert is a boundary), set-availability, session **booking** (`POST /api/sessions` inserts `scheduled`, no BBB at booking) — is pure-UI. Stream on community-create is best-effort/degradable, not a hard cut.

## 2. The snapshot bridge was one-directional; `plato:capture` closes it WAL-safely
**Topics:** testing, plato, cloudflare

**Context:** The waypoint design needs browser-runs to persist their end-state; no such path existed.

**Learning:** Snapshots flowed only API-run in-memory better-sqlite3 → `.serialize()` → `snapshots/<name>.sqlite` (written by the vitest wrapper `runInstanceFile()` at `plato-scenarios.api.test.ts:400`, only on a green run, only for `snapshot:true` instance files). No reverse. Built `plato:capture` = live wrangler D1 → `snapshots/<name>.sqlite` via **`sqlite3 ".backup"`** (SQLite online-backup) — consistent even while the dev server holds the file, so **no server-stop** (unlike restore, which replaces the file and requires the server down). `sqlite3` CLI is already a repo dep (`plato-seed-staging.js`). Verified: captured a 1.37 MB snapshot with :4321 up, `PRAGMA integrity_check` = ok.

**Pattern:**
```
execFileSync('sqlite3', [livePath, ".backup '"+destPath+"'"])
```
Produces a single consolidated `.sqlite`, no `-wal`/`-shm` sidecars, exactly the shape restore expects.

## 3. Behavioral-role model: sidebar "Creating"/"Teaching" gated by ≥1 course/cert, not the capability flag
**Topics:** roles, navigation, testing

**Context:** After the admin granted `can_create_courses=1`, the sidebar still showed only "Learning"; "Creating" appeared only after Mara created a course.

**Learning:** `Sidebar.tsx` gates the "Creating" nav on `user?.isCreator` = "behavioral Creator (≥1 created course)" (isCreatorSubquery, ROLE-STUDIOS Conv 256), NOT the `can_create_courses` capability. The capability gates the `/creating` **route** (reachable by direct nav once granted); the sidebar item is behavioral. The flywheel walkthrough's intent-3 expect ("sidebar now shows Creating") was a pre-behavioral-role legacy assumption. Confirmed live: nav flipped at course-create (intent 5), and the badge went Student→Creator then.

## 4. `plato:split` mints arbitrary waypoints; the split-cleanup lifecycle promotes/deletes
**Topics:** testing, plato

**Context:** Needed to produce the wp-creator-ready waypoint.

**Learning:** `plato:split -- <instance> --at <step|N>` partitions the wrapped scenario: **Pre = steps 1..N-1 (gets `snapshot:true`)**, **Post = steps N..end**, auto-registers both. Any interior index 2..N works → any cut point becomes a waypoint. `plato:restore -- <pre>` API-runs it (regenerates the snapshot from current code) + restores. `plato:split-cleanup --keep <pre> --delete <post>` promotes the pre's inline scenario to a real `.scenario.ts` and removes the post. Used it to create `flywheel-pre-11` (steps 1–10 through self-cert + availability = wp-creator-ready).

## 5. Restoring an API-produced waypoint unblocks the browser past a boundary — proven
**Topics:** testing, plato, stripe

**Context:** The enrollment dead-end ("no teachers available") was the motivating failure.

**Learning:** API-running `flywheel-pre-11` (steps 1–10, Stripe Connect **mocked**) produced a DB with Mara self-certified (`can_teach=1`, `stripe_account_id` set, 1 cert, 5 availability slots) and Alex registered but not enrolled. Restoring it into local D1 made the course Benefits page render **enrollable** ("Get Started Now") for Alex — the exact blocker resolved. This is the whole thesis of the waypoint model, demonstrated end-to-end.

## 6. Browser-run gotchas: stale-auth redirect + mid-swap blank body
**Topics:** testing, plato, chrome-bridge

**Context:** Two transient DOM traps during the walk.

**Learning:**
- (a) A leftover httpOnly auth cookie for a now-deleted user (after a DB reset) redirects `/signup`→`/` while the sidebar still shows logged-out — clear it with `POST /api/auth/logout` + `localStorage.clear()` before walking.
- (b) The first MCP `navigate()` to some authed island routes (`/creating/*`) can land a **fully blank body** (server HTML is complete, but an Astro ClientRouter/view-transition swap caught the DOM mid-replace) — assert `document.body.innerText.trim().length > 0` after settle, else re-navigate/reload.

Both folded into `reference_chrome_bridge_island_stale_cache` [BRIDGE-MEM].

## 7. `file_upload` MCP tool broke for filesystem paths (point-in-time)
**Topics:** chrome-bridge, tooling

**Context:** Uploading a course thumbnail to satisfy the publish gate.

**Learning:** `mcp__claude-in-chrome__file_upload` rejected a real path ("MCP controller must read the file and pass its contents via the `files` parameter"), but the exposed schema only had `paths` — so no working browser file-upload this session. Fallback that worked: drive the app's own endpoint via `javascript_tool` fetch (`PUT /api/me/courses/[id]/thumbnail` with `{thumbnail_url}` external URL). Re-test on a newer Chrome-in-Claude build. Tracked [BRIDGE-UPLOAD].
