### Course-completion certification (the initial ask → CERT-AUTO)

**User:** "I want to address the lack of a means too certify a student as completing the course and generating the certificate in this conv" (via /r-start recommended-action picker)

**Claude:** Investigated the existing certificate machinery (schema + endpoints + a delegated mapping agent). Found: `certificates` table exists (`type IN ('completion','mastery','teaching')`) but the only writers were admin-manual-create and the teacher recommend→admin-approve flow; `certificate_url` was **never** generated anywhere; `/certificates/[id]` had no page route (email + notification linked to it — broken; only `/verify/[id]` existed). Course completion (`enrollments/[id]/progress.ts`) flipped enrollment→completed but issued nothing. Surfaced the gap + two design forks (issuance model; artifact format).

**User:** "You bring up an excellent point. A certifiicate of completion can be automatic. A certificate of mastery (i..e. able to teach) is a different beast and needs a separate UI"

**Claude:** Locked Decision 1 = auto-issue completion cert on completion; mastery/teaching (judgment tiers) deferred to a separate future UI ([CERT-MASTERY-UI]). Stated inline: auto-issued cert attributed to the course Creator (per docs "Creator grants"). Re-asked only the artifact-format decision.

**User:** (picker) "Rendered HTML page (Rec.)"

**Claude:** Built [CERT-AUTO]: `src/lib/certificates.ts::issueCompletionCertificate` (idempotent vs `UNIQUE(user,course,type)`, `type='completion' status='issued'` attributed to Creator, sets `certificate_url='/certificates/{id}'`, fires `notifyCertificateEarned` + `CertificateIssuedEmail`) wired into `progress.ts` + `force-complete.ts`; new `/certificates/[id].astro` printable page; canonical `Certificate` type in `db/types.ts`; +3 tests. 5 gates green (6886); DOM-verified. Committed (code c1dfeb75, docs 66a0ab1).

### Where does it show up? (visibility investigation → exposed a real gap)

**User:** "Does the granting of a completion certificate show up in certifications in /admin and in the /teaching /learning and is registered to the course so it shows (somewhere) number of course completions?"

**Claude:** Delegated a 4-surface trace. Answer: /admin ✅ (Certificates view), /learning ✅ (My Certificates), /teaching ⚠️ event-only (teacher sees the completion status but not the cert), per-course completions count = admin-only + enrollment-backed. **🔴 Exposed a real gap:** the completion cert fired on only 2 of 3 completion paths — the **session-completion path** (`booking.ts checkEnrollmentCompletion` → `triggerPostSessionActions`, likely the primary path in a tutoring product) issued NO cert and was the ONLY path bumping `user_stats.courses_completed`. So cert and completion-stats were on mutually-exclusive paths. Proposed unifying via a single `onEnrollmentCompleted` hook.

### The conceptual reframe (→ DIPLOMA)

**User:** "The app, before last conv had only one notion of certification. Now we have 2: completed a course certification and ready to teach certification. I think the app formerly conflated the two"

**Claude:** Affirmed with code evidence — before Conv 389 the single certificate pipeline WAS the teaching pipeline (recommend→approve required a certified teacher; the completion email bundled an apply-to-teach CTA). Laid out the clean two-concept model (Course Completion = factual/auto vs Teaching Readiness = judgment/human) + a 4-item de-conflation plan.

**User:** "I think the deconflation is a worthy goal. The original certification was for mastery (ability to teach). So I introduced the problem by allowing a completion event to create a completion certificate. Maybe we should remove the certificate label for completion and use a different name. If we did, would that leave the rest of the certificate apparatus intact and correctly be mastery certificates" … "Yes. reshape that way. Diploma for completion. Drop Mastery"

**Claude:** Verified the apparatus is coherent as teach-readiness (only `teaching` has behavior — upserts `teacher_certifications`; `mastery` was decorative). Executed the full reshape: `certificates.type` CHECK → `('teaching')` (retired completion/mastery); course completion → **Diploma** (no table — derived from the completed enrollment + new `enrollments.diploma_awarded_at`; `/diploma/[enrollmentId]` page + loader; `DiplomaEmail`); single idempotent `onEnrollmentCompleted` hook (`src/lib/completion.ts`) wired into all 3 completion paths; deleted `certificates.ts`; reworded the cert email/page to teaching. Fixed seed + PLATO fixtures + ~9 cert test files (delegated the mechanical fixture conversion). 5 gates green (6886); both pages DOM-verified against a reseeded local D1 (0 completion certs). GLOSSARY/resend docs + memory updated. Checkpointed before the /learning UI.
