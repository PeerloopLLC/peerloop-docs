# Conv Learnings - 2026-07-02

## 1. Tailwind v4 dev-JIT source-order trap breaks stacked responsive offsets
**Topics:** astro, css, tailwind

**Context:** Stacking `top-[80px] sm:top-[88px] lg:top-[16px]` on a sticky bar, the `lg` override lost to `sm` at desktop width — the bar docked at 88px instead of 16px.

**Learning:** In Tailwind v4 dev (JIT), the compiled source order of responsive utilities for the same property is not guaranteed to follow breakpoint order (arbitrary values sort after named; discovery order can leak in), so an override at a larger breakpoint can be beaten by a smaller one. Do NOT rely on min-width cascade for conflicting responsive values.

**Pattern:** Use **mutually-exclusive range variants** so no two rules ever both match: `max-lg:top-[88px] lg:top-[16px]` (max-lg = <1025, lg = ≥1025). Order-independent by construction. Also relevant: named spacing like `top-80` ≠ 80px under the NAV-RETROFIT bridge (only {4,8,12,16,20,24,32,40,48,64} are aliased to literal px), so 80/88 must be arbitrary `[88px]`.

## 2. A sticky bar with a non-zero top offset needs an opaque "riser" to occlude the gap above it
**Topics:** css, layout

**Context:** Cards/entity headers scrolling up showed through the strip *above* the pinned toolbar (the region between the pin line and viewport top / fixed header).

**Learning:** The bar's own bg only covers the bar; the gap above the pin line is uncovered. Cover it with a `::before` pseudo that extends the bg upward.

**Pattern:** `before:absolute before:inset-x-0 before:bottom-full before:h-[Npx] before:bg-white before:content-['']` on the sticky element (which is its own positioned containing block). Size N to the at-rest gap-above so it stays invisible when unpinned: 20px on the listing pages (below a 24px section flex-gap), 16px on the detail pages (AppLayout's `gap-16` above the SubNav strip). Absolute pseudos are out-of-flow, so this is safe even on a `grid`/`flex` container.

## 3. z-index ties between a sticky bar and content's own z-10 elements resolve by DOM order
**Topics:** css, layout

**Context:** On `/courses`, the course card's author link (`relative z-10`, elevated to beat the card's stretched-link overlay) painted OVER the pinned toolbar; community/member cards didn't (they don't elevate their author).

**Learning:** A `position:relative` element with no z-index is NOT a stacking context, so a descendant's `z-10` leaks to the root and competes globally. Two `z-10`s tie; the one later in the DOM (the catalog card, below the toolbar) wins.

**Pattern:** Add `isolate` (`isolation:isolate`) to the offending component's root — it becomes a stacking context, scoping its internal z-values so the whole component sits below the sticky bar. Click/overlay behaviour inside the card is unchanged. This is the durable fix (root-cause) vs bumping the bar's z (which muddies the 10=sticky/20=popover/30=fixed scale).

## 4. The claude-in-chrome harness blocks programmatic scroll, but the `computer` tool's wheel scroll works
**Topics:** testing, browser, plato

**Context:** `window.scrollTo` / `document.documentElement.scrollTop` / `scrollIntoView` all failed to move the page (scrollY stayed 0; scrollIntoView froze the renderer for 45s), so I couldn't demonstrate the pinned state via `javascript_tool`.

**Learning:** Programmatic scroll is blocked/reset in the injected JS context, but `mcp__claude-in-chrome__computer` with `action:'scroll'` dispatches REAL wheel events that actually scroll. Use it to verify sticky/pinned scroll states (and to screenshot them).

**Pattern:** For scroll-state verification: `computer` scroll (real wheel) + screenshot. For static geometry: `javascript_tool` computed styles / getBoundingClientRect. Don't waste turns fighting programmatic scroll.

## 5. Reaching auth-gated islands in a fresh MCP browser tab
**Topics:** testing, browser, auth

**Context:** `/profile`'s SubNav only renders for authenticated users; a fresh MCP tab is a separate, logged-out Chrome session.

**Learning:** `POST /api/auth/dev-login` with `{email}` (dev-only, no password) sets the session cookie in the tab; then hard-nav to the target page. Seeded emails live in `migrations-dev/` (e.g. `guy-rymberg@example.com`, `sarah.miller@example.com`).

**Pattern:** fetch dev-login from page context → re-navigate → verify. Session dies with the tab (no cleanup needed).
