# Conv Learnings - 2026-07-20

## 1. jsx-a11y's ESLint-10 peer gap is a stale declaration, not a real incompatibility — and the "clean fix" fork silently drops `.astro`
**Topics:** eslint, astro, testing, tooling

**Context:** [A11Y] needed a permanent a11y linter on ESLint 10; `eslint-plugin-jsx-a11y@6.10.2` peer-caps at eslint ^9.

**Learning:** The plugin loads and runs all 34 rules on ESLint 10 (empirically verified — scanned 326 `.tsx` + 90 `.astro`, caught bad markup); the peer range just lags (upstream #1075 stalled). The es-tooling fork `eslint-plugin-jsx-a11y-x` *does* declare eslint ^10 but, tested via npm alias, resolves **0 rules through eslint-plugin-astro** (astro hard-codes upstream's shape + export API) → it silently removes `.astro` coverage. Precedent for accepting the lag: `eslint-plugin-react-hooks@7.1.1` already ships eslint ^10 and we run it green. `[EMP]`/`[VDF]` win: alias-testing the fork before recommending it is what caught the coverage loss.

## 2. npm `overrides` resolves a stale peer range without poisoning every install — but rejects `"//"` comment keys
**Topics:** cc-workflow, tooling, eslint

**Context:** Making `npm install`/`npm ci` clean on both machines despite the eslint ^9 peer cap.

**Learning:** `"overrides": { "eslint-plugin-jsx-a11y": { "eslint": "$eslint" } }` forces the plugin's eslint peer to the root version, so `npm install` succeeds with no flag (verified + reverted) — cleaner than `.npmrc legacy-peer-deps=true`, which weakens peer detection project-wide. npm rejects a `"//"` documentation key inside `overrides` ("Override without name: //") — document the rationale elsewhere (the decision doc), not inline.

## 3. eslint-plugin-astro's jsx-a11y config registers `jsx-a11y` in an UNSCOPED object → ESLint-10 plugin-redefine collision
**Topics:** eslint, astro

**Context:** Wiring jsx-a11y for both `.tsx` and `.astro` in one flat config.

**Learning:** `eslintPluginAstro.configs['flat/jsx-a11y-recommended']` includes one **unscoped** object (`files` absent) that registers the `jsx-a11y` plugin. Under ESLint 10 that collides with a `.tsx` block's own `jsx-a11y` registration ("Cannot redefine plugin jsx-a11y"). Fix: `.map()` that one object to `files: ['**/*.astro']` so the two registrations never apply to the same file.

**Pattern:**
```js
const astroA11y = eslintPluginAstro.configs['flat/jsx-a11y-recommended'].map(c =>
  c.plugins && c.plugins['jsx-a11y']
    ? { ...c, files: ['**/*.astro'], rules: asWarn(c.rules) }
    : c
);
```

## 4. Our `.astro` files carry 0 a11y findings; Astro's dev-toolbar Audit covers them at runtime
**Topics:** astro, testing

**Context:** Whether static `.astro` a11y linting is load-bearing.

**Learning:** All 90 `.astro` files produced 0 a11y findings under the recommended set — they're structural shells; every interactive element (forms, buttons, labels) lives in `.tsx` islands. Astro's dev-toolbar **Audit** app already runs axe-based a11y checks on rendered `.astro` in dev. So static `.astro` a11y linting is future-proofing, not a present gap — which is why losing it to the fork was survivable but still not worth it (it comes free with upstream).

## 5. Throwaway `npm install` experiments pollute resolution state and can break a later `npm ci`
**Topics:** cc-workflow, tooling

**Context:** Multiple `--no-save`/modified-manifest installs during the investigation.

**Learning:** After throwaway installs, `npm ci` failed "package.json and package-lock.json out of sync — Missing @emnapi/core@1.11.1" even though the committed lockfile was byte-identical to HEAD (it installed fine at conv start). Cause: the experiments pulled newer transitive optional wasm pins (knip's oxc-parser → @emnapi) into npm's resolution/cache. Reconcile with `npm install` (repopulates node_modules) then `git restore package-lock.json` (keep committed) — node_modules stays functional, repo stays pristine. Better next time: use a throwaway worktree for dep experiments.
