# Conv Learnings - 2026-07-03

## 1. CC AskUserQuestion AFK auto-proceed is `CLAUDE_AFK_TIMEOUT_MS`
**Topics:** cc-workflow

**Context:** The user hit the 60s "proceed using your best judgment" timeout message and wanted it gone.

**Learning:** New CC feature (v2.1.198+): an unanswered `AskUserQuestion` auto-continues after `CLAUDE_AFK_TIMEOUT_MS` (default 60000). There is **no boolean disable**; `0` fires *immediately* (a trap). Neutralize by setting it huge (`2147483647` = max int32, ~24.8 days). Companion `CLAUDE_AFK_COUNTDOWN_MS` (default 20000) controls the on-screen warning. Settable via the `env` block in `settings.json` (same channel as the already-working `CLAUDE_CODE_DISABLE_MOUSE`); env is read at session start, so it takes effect next launch.

## 2. Reuse the real component in a drawer = single source of truth
**Topics:** navigation, astro-islands, matt-design

**Context:** Mobile/tablet couldn't hold the desktop Sidebar's ~13 role-aware destinations.

**Learning:** Rather than a bespoke mobile nav list (drift risk), render the **actual `Sidebar` in a `variant="drawer"` mode** inside the overlay. Role gating, Workspaces, Profile all come free and can never drift from desktop. Cost = a small variant branch (close-X vs collapse; full-width; always-expanded) and two `<Sidebar>` instances (desktop in-flow + drawer), same `user`/`currentPath`.

**Pattern:** Add a `variant` prop that swaps only the chrome (aside classes + the header control), keeping the nav body identical.

## 3. Body-level overlay to escape a transformed ancestor's containing block
**Topics:** css, layout

**Context:** `HeaderBar` uses `sm:-translate-x-1/2` at tablet; a `position: fixed` overlay nested inside it would be positioned relative to the header, not the viewport.

**Learning:** A transformed ancestor becomes the containing block for `position: fixed` descendants. Mount full-screen overlays (NavDrawer) at **AppLayout body level**, decoupled from the trigger via a global `CustomEvent` (`nav:open`), mirroring the existing filter-island idiom (`courses:filterchange`).

---

## 4. Test raw window events via `fireEvent(window, …)`, not `window.dispatchEvent`
**Topics:** testing, react

**Context:** A NavDrawer test dispatched `nav:open` with raw `window.dispatchEvent` and the panel never opened.

**Learning:** A React `setState` fired from a raw `dispatchEvent` handler isn't wrapped in `act()`, so the re-render isn't flushed before the assertion (and silently made two "close" tests pass trivially by never opening). Use RTL `fireEvent(window, new CustomEvent('nav:open'))`, which wraps in `act`.

---

## 5. Chrome window won't shrink below the desktop breakpoint via `resize_window`
**Topics:** browser-automation, plato

**Context:** DOM-verifying the mobile nav needed a <1024px viewport.

**Learning:** `mcp__claude-in-chrome__resize_window` reported success but `window.innerWidth` stayed 1482 (OS/Chrome clamp), so the page always rendered the desktop shell. Workaround: verify viewport-independent facts + drive the interaction logic programmatically (click the hamburger, assert the drawer class flips + content) instead of relying on true-mobile rendering. True-mobile *visibility* (pure responsive CSS) stays unverified in-browser.

**Topics:** navigation

**Context:** Discovered while scoping [MOBNAV].
