# Conv Decisions - 2026-07-05

## 1. AUTHBUG fix: reload-fallback from cookies, not seed-from-login-response
**Type:** Implementation
**Topics:** auth

**Trigger:** FIX-CORE needs the client to end up authenticated after a successful login POST even if `/api/me/full` transiently fails. Flagged the seeding approach for user input.

**Options Considered:**
1. Seed `currentUser` directly from the login endpoint's returned `user` object
2. Reload-fallback: on failed post-login refresh, re-navigate so `initializeCurrentUser()` re-establishes from the auth cookies ← Chosen
3. Retry `/api/me/full` N times inline

**Decision:** On a failed post-login refresh, re-navigate (reload / home from `/login|/signup` / `/onboarding` for signup) rather than seeding state from the login response.

**Rationale:** Login returns a partial snake_case user; `currentUser` holds the rich camelCase `MeFullResponse` — seeding is a shape mismatch that would require a lossy adapter. The auth cookies are already set on the 200, so a fresh load via `initializeCurrentUser()` (now `no-store`) re-establishes true state with zero shape risk. Failure path is rare, so a reload's cost is acceptable.

**Consequences:** `refreshCurrentUser()` gained a `boolean` return; Login/Signup branch on it; no shape adapter needed.

## 2. Implement the full durable fix (all 5 layers), not the minimal patch
**Type:** Strategic
**Topics:** auth

**Trigger:** User asked to "implement the durable fix" after root-cause diagnosis.

**Options Considered:**
1. Minimal: only gate the modal-close in LoginForm
2. Durable: FIX-CORE + FIX-LOGOUT + FIX-CACHE + FIX-SIGNUP + FIX-SMARTFEED ← Chosen

**Decision:** Ship all five layers.

**Rationale:** The confirmed defect (silent discard) plus the delay-trigger (cache replay) plus the different-user amplifier (stale cache / never-cleared `was_logged_in`) each needed a distinct layer; the fix is trigger-agnostic only with all of them. FIX-SMARTFEED addresses the residual "still in visitor mode" on inline home-page logins.

**Consequences:** 10 files changed; two independent defenses against the crash; manual `.vite` nuke and the silent-login-loss both retired.

## 3. Clear client auth state on all three explicit-logout handlers (inline in the .astro)
**Type:** Implementation
**Topics:** auth, astro

**Trigger:** FIX-LOGOUT; the `profile/[...tab].astro` logout is a vanilla inline script, not React.

**Options Considered:**
1. Import `clearCurrentUser` into all three (incl. the .astro script)
2. `clearCurrentUser(true)` in the two React handlers; inline `localStorage.removeItem` for the three keys in the .astro ← Chosen

**Decision:** React handlers call `clearCurrentUser(true)`; the .astro inline script removes the three keys directly.

**Rationale:** Importing `current-user.ts` into the profile page's inline `<script>` would bundle the whole module (CurrentUser class, polling, etc.) into a separate script graph — heavy. Inlining three stable-literal `removeItem` calls (with a comment cross-referencing `clearCurrentUser`) is lighter and equally correct.

**Consequences:** Small, documented key-name duplication in one .astro file.
