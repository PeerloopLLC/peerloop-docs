## 1. Waypoint-sequenced API+browser test architecture
**Type:** Strategic
**Topics:** testing, plato

**Trigger:** A pure browser flywheel walk dead-ends at each external-service boundary the API-run mocks but the live dev server can't.

**Options Considered:**
1. Push browser through real Stripe/BBB test flows (heavy, external, brittle)
2. Chain API-runs (produce state past boundaries via snapshots) + browser-runs (verify pure-UI segments) via **waypoints** ← Chosen
3. Give up on browser-mode for those legs

**Decision:** Adopt the waypoint model — API-runs produce snapshots at each of the 4 cut points; browser segments (B1 creator-setup, B2 availability, B3 post-enroll booking, B4 completion+certification) each restore the nearest waypoint and walk pure-UI only. `session-invite` already works this way (it's the template).

**Rationale:** Matches the already-written `plato.md` principle "put mocked external services on the API side"; makes runs build on each other deterministically; structurally eliminates the flywheel/ecosystem walkthrough precondition gaps.

**Consequences:** Phase 1 foundation built + committed; `plato.md` + PLAN.md updated; new `PLATO-SEQ` block owns Phases 2–4.
