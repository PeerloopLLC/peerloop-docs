# Conv Learnings - 2026-07-08

## 1. Booking-calendar off-by-one was a THREE-zone collision, not just a time-format bug
**Topics:** timezone, booking, calendar

**Context:** `SessionBooking.tsx` rendered slot times in browser-local, grouped calendar cells by the teacher-local `slot.date`, and built the grid with browser-local `Date` math — three zones that disagree across a UTC-day boundary.

**Learning:** Fixing only the times leaves the off-by-one: the fix must **regroup the calendar by the viewer-tz day of each slot's UTC `start_time`** (via `dateKeyInTz`), and rebuild the grid as tz-independent **civil** math (UTC-constructed `Date.UTC(y,m,1)`), so the day a slot files under and the time it shows never disagree.

**Pattern:** For a bare civil `YYYY-MM-DD` selectedDate, `formatDateUTC` is *correct* (a bare date formatted in UTC shows that date) — the date-formatter was never the bug; the grouping key was.

---

## 2. `dateKeyInTz` — the viewer-tz day-key that anchors calendar grouping
**Topics:** timezone, helpers

**Context:** Needed a machine key (not a display string) to group instants by "the day the viewer would call it."

**Learning:** `new Intl.DateTimeFormat('en-US', {timeZone, year,month,day}).formatToParts(d)` reassembled into `YYYY-MM-DD` is locale-independent (never relies on a locale that happens to emit ISO order). Its zero-padded output matches the calendar cell-key format, so grouping key and grid cell agree by construction.

**Pattern:** `dateKeyInTz(utcIso, tz)` in `src/lib/timezone.ts`; null tz → UTC.

---

## 3. Emails need an explicit zone label; in-app display drops it
**Topics:** timezone, email, phase-2

**Context:** Phase 1 in-app surfaces render bare local time ("9:00 AM") because the viewer's context is implicit. An email is read async / out of context.

**Learning:** For emails, always show a short zone label via `timeZoneName:'short'` ("9:00 AM EST"); a null tz then renders "9:00 AM UTC" — one code path covers both the localized and tz-unknown cases. `formatRecipientSession(iso, tz)` encapsulates this.

---

## 4. Per-recipient email formatting requires fetching each party's tz
**Topics:** timezone, email, sql

**Context:** Both senders computed one UTC string and shared it between teacher + student via `...emailProps`.

**Learning:** Add `t.timezone`/`st.timezone` to the existing detail JOINs and format **per recipient** — each email/notification gets the session time in *its* recipient's zone. The `notify*` helpers already take a pre-formatted string per call, so per-recipient means passing each call its own string (no signature change).

---

## 5. Pure date-only stamps should stay UTC-stable, not localize
**Topics:** timezone, policy, date-stamps

**Context:** The final Phase-1 slice's flagged sites mixed session dates with pure calendar milestones (`certified_date`, review date, member-since).

**Learning:** A date-only display has no wall-clock for a viewer to misread, so Model A's payoff (correct meeting time) doesn't apply; and localizing a midnight-stored calendar value to a behind-UTC viewer *introduces* a ±1-day off-by-one. Keep them `formatDateUTC` (matches the dominant convention). Only values carrying a time-of-day localize. `formatSessionDate(iso, null, opts)` gives UTC-stable output when a custom format (month-year) must be preserved.
