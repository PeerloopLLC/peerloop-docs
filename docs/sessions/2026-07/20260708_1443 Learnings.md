# Conv Learnings - 2026-07-08

## 1. The vitest host runs in America/Toronto, not UTC
**Topics:** testing, timezone, cloudflare

**Context:** TZ-MODEL Phase 3(b) hardening replaced host-local `Date` math with explicit UTC across 13 files. Before committing, checked whether the test env is UTC (which would make the change a no-op) or not.

**Learning:** `vitest.config.ts` / `vitest.setup.ts` / `vitest.global-setup.ts` do **not** set `process.env.TZ`, so the suite runs in the machine's local zone (`America/Toronto`, UTC−4, confirmed via `Intl.DateTimeFormat().resolvedOptions().timeZone`). This means any server code using the bare `Date` constructor or local `getDate`/`setDate`/`getMonth` behaves differently in tests than on the UTC Cloudflare Worker. Explicit `getUTC*` / `Date.UTC` / `setUTC*` is the only way to make server date math both test-deterministic AND production-correct. The full suite stayed green (6784✓) only because no existing test pinned the exact boundaries — so the divergence was latent, not caught.

**Pattern:**
For any server date arithmetic whose result is stored or compared, use the UTC accessors — never the local-zone equivalents. The Worker is UTC; the test host is not.

```ts
// ❌ host-tz-dependent — Toronto boundaries in tests, UTC on the Worker
const start = new Date(now.getFullYear(), now.getMonth(), 1);
cursor.setDate(cursor.getDate() + 1);

// ✅ host-tz-independent — UTC everywhere
const start = new Date(Date.UTC(now.getUTCFullYear(), now.getUTCMonth(), 1));
cursor.setUTCDate(cursor.getUTCDate() + 1);
```
