## 1. Solve tool-output noise with `verbose: false`, not `/focus`
**Type:** Implementation
**Topics:** cc-workflow, docs-infra

**Trigger:** User: "I never look at the tooling output and have to sift through them looking for the text meant for me."

**Options Considered:**
1. `/focus` ‚ÄĒ built-in, collapses a whole turn's tools to one line. **Rejected: requires fullscreen, which destroys native scrollback.**
2. `verbose: false` in the classic renderer ‚Üź **Chosen**
3. Two-pane live tailer as CC infrastructure ‚ÄĒ rejected; user supplies their own terminal
4. `suppressOutput` / output styles / statusline ‚ÄĒ investigated, none suppress tool results

**Decision:** `verbose: false` in **project** `.claude/settings.json`; `tui` stays `"default"`.

**Rationale:** Collapses tool output to `ctrl+o to expand` stubs **while keeping native scrollback**, which the user declared non-negotiable. Live without a restart. Project-scoped so it is git-tracked and reaches MacMiniM4; global settings restored byte-identical to their pre-conv backup so nothing lingers outside this project.

**Consequences:** `/focus` and fullscreen are off the table permanently (recorded in memory so they aren't re-proposed). The `!` backtick 44% is untouched ‚ÄĒ user rated it secondary.

> **Insight:** The measurement reframed the problem: skill `!` backtick precomputed context (44%) outweighed tool results (27%) and tool calls (19%). The loudest complaint pointed at the second-largest source ‚ÄĒ measure before optimizing.

## 3. `[CBG]` guards on `.conv-branch`, not RESUME-STATE ‚ÄĒ and HALTs
**Type:** Implementation
**Topics:** cc-workflow, dual-repo

**Trigger:** Building `[CBG]`; discovered the existing checker returns `NO-RESUME-STATE` mid-conv.

**Options Considered:**
1. Reuse `conv-branch-check.sh` at commit time ‚ÄĒ **rejected: permanent green light** (verified live)
2. New `.conv-branch` sentinel, ephemeral sibling of `.conv-current` ‚Üź **Chosen**

**Decision:** `/r-start` Step 4 records the branch it already validates; `/r-commit` Step 0.5 + `/r-end` Step 0.7 compare and **HALT + AskUserQuestion** on mismatch. Removed at `/r-end` Step 8.

**Rationale:** Follows the established `.conv-current` pattern (not novel). Catches **mid-conv** drift ‚ÄĒ the actual Conv 371 failure ‚ÄĒ which no RESUME-STATE-based check can see. HALT over warn because the bug class *is* unread warnings.

**Consequences:** `/r-commit` loses some autonomy on mismatch (user's explicit choice). `NO-CONV-BRANCH` stays advisory so pre-[CBG] convs are never blocked.

## 4. `CLAUDE_CODE_DISABLE_MOUSE=1` is a guard, not an obstacle
**Type:** Strategic
**Topics:** cc-workflow

**Trigger:** Claude flagged ūüĒī that the mouse setting would break fullscreen's click-to-expand.

**Options Considered:**
1. Re-enable the mouse to regain click-to-expand ‚ÄĒ **rejected once the rationale was known**
2. Keep the guard, lose click-to-expand, use `Ctrl+O` ‚Üź **Chosen**

**Decision:** Never propose re-enabling. Recorded as `[MOUSE-GUARD]` with an explicit prohibition.

**Rationale:** A stray click (user thought they were foregrounding VS Code; it already was) landed on the `AskUserQuestion` picker and **selected an answer they never gave**. That's accidental consent ‚ÄĒ the click-analogue of `[AFK-CFG]`'s timeout auto-proceed. Both are ways the picker records a decision the user never made.

**Consequences:** Memory written + cross-linked both ways. Corollary recorded: if a picker selection ever looks accidental or contradicts what the user just said, **re-ask** ‚ÄĒ a click is not more authoritative than the conversation.

> **Insight:** A setting that looks like an obstacle may be a scar. Before proposing to remove a constraint, ask what incident put it there ‚ÄĒ the user's rationale reversed the recommendation entirely.
