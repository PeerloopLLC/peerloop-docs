# Conv Decisions - 2026-07-01

## 1. `nav_layout` stored as user-facing `'top' | 'rail'`, translated to internal `'left'` in AppLayout
**Type:** Implementation
**Topics:** schema, layout-mode, astro

**Trigger:** Adding the per-user column needed a value vocabulary; the design says `'rail'` but internal code says `'left'`.

**Options Considered:**
1. Store `'top' | 'left'` (reuse `SubNavLayout` exactly, zero translation) — but `'left'` leaks internal placement into DB/API and diverges from the approved design wording.
2. Store `'top' | 'rail'`, translate `'rail'→'left'` once in AppLayout ← Chosen.
3. Rename internal `'left'`→`'rail'` everywhere (5+ files) — churns just-shipped Conv-355 code.

**Decision:** Store `'top' | 'rail'`; map to internal `'top' | 'left'` at the single AppLayout boundary.

**Rationale:** Honors the approved README design (not a novel divergence → no escalation needed), keeps the Conv-355 internal plumbing untouched, and costs one ternary. User-facing toggle labels are "Top bar" / "Side rail".

**Consequences:** DB/API vocabulary is `'top'|'rail'`; `Astro.locals.navLayout`/`SubNavLayout` stay `'top'|'left'`.

## 2. Reuse `PATCH /api/me/profile` for the nav_layout persist rather than a new endpoint
**Type:** Implementation
**Topics:** api, layout-mode

**Trigger:** The toggle needs to persist a per-user preference.

**Options Considered:**
1. New dedicated endpoint — extra surface for a single field.
2. Extend the existing profile-field whitelist path ← Chosen.

**Decision:** Add `nav_layout` to `PROFILE_FIELDS` + a `'top'|'rail'` validation case in the existing `PATCH/GET /api/me/profile` (same path as the email/marketing prefs), which already bumps `data_version`.

**Rationale:** `nav_layout` is exactly analogous to `marketing_opt_out`/`email_*`; the established preference-persistence pattern already handles validation, dynamic UPDATE, and version bump.

**Consequences:** No new endpoint; the toggle island PATCHes the same route the settings forms use.
