# Conv Learnings - 2026-07-12

## 1. Orphaned page-level components survive route migrations with every gate green
**Topics:** astro, testing, cc-workflow

**Context:** [DIPLOMA-WALK] — walking the 9 Conv-391 diploma surfaces live found 4 rendered by NO route (LearnTab completion card, discover `CompletedTabContent`, `role-utils` tab label, `CourseHero`). Root cause: Conv 339 retired `/old` routing by deleting the *pages* but leaving the *components*.

**Learning:** When a route migration deletes pages but keeps their components, those components become unreachable yet all five gates stay green — `tsc`/`lint`/`astro check`/`build` only see errors reachable from an entrypoint, and unit tests keep passing because they import the components DIRECTLY, bypassing routing entirely. So a component can be "tested and green" while being mounted nowhere. Corollary: Conv 391's [DIPLOMA-UI-GAPS] "all user-visible certificate wording purged" baseline was partly made against unreachable code — a grep-driven copy sweep edited 4 dead surfaces. Reachability, not compilation or test-pass, is the property that catches this class.

## 2. Closed-orphan-set deletion is safe; tsc is the precise dangling-import arbiter
**Topics:** testing, astro

**Context:** [ORPHAN-PURGE]/[ORPHAN-BACKLOG] — deleting 20 then 74 orphaned files (components + tests) with `tsc` between family batches.

**Learning:** Reachability is transitive, so by definition no reachable (live) file imports an orphan. Deleting the *full* orphan set can therefore only dangle three things: (a) tests that imported the orphan, (b) dead barrel/index re-exports, (c) other orphans. `tsc --noEmit` after each batch is the precise arbiter — it flags every dangling import exactly, so "green tsc after delete" proves zero live breakage. Across both passes every dangler was a test or a dead barrel; zero live code broke.

## 3. The reachability detector is component-only — dead `.ts` utils slip through
**Topics:** cc-workflow, testing

**Context:** [ORPHAN-DETECT] — the new `codecheck-orphan-components.mjs` BFS flags `src/components/**` `.tsx`/`.astro` unreachable from `src/pages/**`.

**Learning:** The detector scans component files, not plain `.ts` utility modules. Dead helper `.ts` files (e.g. `discover/community-role-utils.ts`, `discover/types.ts`, `course-tabs/types.ts`) that were only imported by now-deleted orphaned components won't be flagged even after their consumers vanish — they need a separate `.ts` reachability sweep. Know the detector's scope before reading "0 component orphans" as "0 dead code."

## 4. grep import pre-checks false-positive — verify real imports or trust tsc
**Topics:** cc-workflow, testing

**Context:** Triaging orphans, using a loose grep for imports of a component by basename.

**Learning:** A loose `/Basename'` grep pattern matches (a) same-basename files in *different* directories (two `types.ts`, two `index.ts`) and (b) doc-comment mentions of the name — both false "this is still imported" positives. Confirm actual path-qualified `import ... from` statements, or just delete-and-let-tsc-decide; don't gate a deletion on a bare-basename grep.

## 5. Admin force-complete is a clean live notification-fixture generator
**Topics:** d1, testing, notifications

**Context:** [DIPLOMA-WALK] — needed the student + teacher completion notifications to appear live to walk the notification surfaces.

**Learning:** `/diploma/[id]` gates on the enrollment `status='completed'` (NOT on `diploma_awarded_at`, which it uses as the awarded-date but falls back from to `completed_at`). Admin force-complete runs `onEnrollmentCompleted`, which sets `diploma_awarded_at` AND fires both the student "you completed" and teacher "your student completed" notifications through the real code path. So force-completing an enrollment (`enr-david-n8n`) is a one-click way to generate genuine, live notification fixtures for surface-walking — no hand-inserted rows.
