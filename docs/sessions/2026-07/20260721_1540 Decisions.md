## 2. Two behavioral primitives, split by a11y contract — not one shared primitive
**Type:** Strategic
**Topics:** architecture, a11y

**Trigger:** The approved "one shared primitive for ~13 sites" plan collapsed on reading the call sites.

**Options Considered:**
1. One `ClickableRow` for all 13 — would put 7 decorative backdrops in the tab order
2. `ModalBackdrop` + `ClickableRow`, split by contract ← Chosen
3. No primitives; 13 one-off `role="presentation"` / keyboard-handler edits — 13 copies of the same handler, free to drift

**Decision:** Build two primitives with deliberately different contracts, and document in each that it must not absorb the other's cases.

**Rationale:** The two families need opposite things: a backdrop must be *removed* from the a11y tree and never focusable; a row must be *added* to it as a button. One component cannot honour both. `ModalBackdrop` additionally centralizes 5 divergent overlay styles.

**Consequences:** 16 interactive warnings cleared across 8 sites; both primitives carry explicit "do not widen" notes; `AvailabilityCalendar:654`, `AdminDataTable:137`, and `Modal.tsx:83` left as documented one-offs.

## 3. Behavioral primitives are neither `data-prov`-stamped nor registry-registered
**Type:** Implementation
**Topics:** architecture, docs-infra

**Trigger:** `matt-inspired-registry.ts` entries require `figmaMatchNames`, and all three `data-prov` values assert design provenance.

**Options Considered:**
1. Stamp `matt-inspired` + invent `figmaMatchNames` — fabricates provenance
2. Leave unstamped + unregistered, verify `prov:sweep` stays green ← Chosen
3. Extend `matt-provenance.md §12e` with a formal behavioral tier first — larger scope, unprompted

**Decision:** Ship both unstamped and unregistered; verify empirically that `prov:sweep` raises nothing for them.

**Rationale:** These carry behavior, not design, and have no Figma counterpart — a stamp would be a false claim in a scheme whose whole value is trustworthy provenance. Verified rather than asserted.

**Consequences:** `prov:sweep` raises nothing for either. If a behavioral tier is ever wanted, §12e is the extension point (noted in both files and in `[A11Y]`).

## 4. Revert the Conv-403 `~/.zshrc` guards rather than keep them as harmless no-ops
**Type:** Implementation
**Topics:** cc-workflow

**Trigger:** User asked directly after being told the guards were harmless.

**Options Considered:**
1. Keep them — zero runtime cost
2. Revert both launchers and remove the false-cause comments ← Chosen (user's call)

**Decision:** Revert; back up to `~/.zshrc.bak-conv404` first; strip the explanatory comments too.

**Rationale:** The comments were the greater liability — they documented a mechanism that does not exist, so a future reader would infer the leak was real. Reverting code without the comments would have left the misinformation.

**Consequences:** Both launchers back to plain `claude …`; `zsh -n` clean; verified in a fresh login shell; all three records updated from "no-ops in place" to "reverted / decision fully withdrawn". Recorded in `DOC-DECISIONS.md` §3 as the withdrawal blockquote on the Conv-403 entry rather than as a new entry.
