# Conv Decisions - 2026-07-10

## 1. Walk fidelity — Hybrid
**Type:** Implementation
**Topics:** plato, testing

**Trigger:** How faithfully to drive the UI for the browser re-walks (create-course, curriculum, publish, register, book).

**Options Considered:**
1. Full UI drive — click every field on every page
2. Hybrid — real UI for interactive gates, app-endpoint fetch for bulk/repeat ← Chosen
3. State-first — screenshot each page, submit all payloads via fetch

**Decision:** Hybrid.

**Rationale:** Proves each interactive surface renders + works while staying reliable; full-UI is slow/fragile on array fields + the un-uploadable thumbnail, and state-first barely tests the UI.

**Consequences:** Drove create-course modal, one add-module form, Publish gate, /signup, the full booking calendar/slot/confirm flow, and the Certify modal via real UI; bulk detail + repeat modules/sessions via fetch.

## 2. Reconcile browser-detected timezone to the persona spec before capturing wp-published
**Type:** Implementation
**Topics:** plato, testing, timezone

**Trigger:** Browser registration produced `America/Toronto` for both users vs the persona's cross-boundary LA/Tokyo pair.

**Options Considered:**
1. Reconcile to persona tzs (Mara→LA, Alex→Tokyo) then capture ← Chosen
2. Capture browser-honest as-is (both Toronto) + document the divergence

**Decision:** Reconcile then capture.

**Rationale:** The persona tzs are the spec; tz is settable state the UI simply doesn't expose (the API producer injects it), so aligning isn't fabricating UI behavior — it's restoring the intended waypoint. Preserves `[TZ-TESTS]`; the user has known low confidence in tz handling.

**Consequences:** `wp-published` is a tz-faithful drop-in for `flywheel-pre-9`. Caveat documented in the snapshots README for future B1 walkers.
