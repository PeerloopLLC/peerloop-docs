# Conv Decisions - 2026-07-12

## 1. Remove the admin certificate type filter entirely (vs. trim to one option)
**Type:** Implementation
**Topics:** ui, certificates, admin

**Trigger:** `CertificatesAdmin` had a type-filter dropdown with completion/mastery/teaching options; retiring the first two would leave a single-value ("All Types" + "Teaching") filter.

**Options Considered:**
1. Trim options to teaching-only (keep the filter scaffold for future teach-tiers)
2. Remove the type filter entirely (state + query param + deps + config) ← Chosen

**Decision:** Removed the whole filter — state, the `type=` query param, both useEffect deps, and the filter config entry.

**Rationale:** A single-option filter is a no-op control that only confuses; removing dead UI is the durable cleanup (§Solution Quality). Re-adding a filter if future teach-tiers arrive is trivial. Not novel (follows the retirement decision).

**Consequences:** Cleaner admin filter bar; the `type` param is simply omitted (endpoint returns all, which are all teaching).

## 2. Thread email env as a trailing `CompletionEmailEnv` opts param (vs. options-object refactor)
**Type:** Implementation
**Topics:** api, completion, refactoring

**Trigger:** The Diploma email needed env threaded through `completeSession` (which already has 3 trailing positional optionals) and the cron reconcile chain.

**Options Considered:**
1. Trailing positional `email?: CompletionEmailEnv` following the established `onEnrollmentCompleted(db, id, opts)` pattern ← Chosen
2. Convert `completeSession`'s trailing params to a single options object (cleaner call sites, but rewrites the delicate bbb.ts waitUntil-ordered positional calls + test call sites)

**Decision:** Trailing positional opts, accepting a couple of `undefined, undefined, undefined, email` placeholders at the two clean call sites.

**Rationale:** Follows the exact existing pattern (not novel), minimal blast radius (doesn't touch the Conv-384 PSA-WAITUNTIL bbb.ts ordering), and eslint's unused-var check structurally guarantees full threading.

**Consequences:** 9 functions / 6 files threaded; env-less callers still award the Diploma + in-app notification, only the email is key-gated.

## 3. CERT-MASTERY-UI scope: ship A+B (both surfaces), defer C (user decision)
**Type:** Strategic
**Topics:** ui, certificates, scope

**Trigger:** `[CERT-MASTERY-UI]` is a 3-part feature (A recommend UI, B cleanup, C row-shape centralization); multi-conv carve-out required a scope check-in.

**Options Considered:** A only / **A+B ← Chosen** / A+B+C. Surface breadth: MyStudents only / **both surfaces ← Chosen**.

**Decision:** Ship A (recommend UI on both `MyStudents` + `TeacherCourseView`) + B (cleanup incl. the mislabel bug); defer C (row-shape refactor, high-churn/low-visibility) → `[CERT-ROWSHAPE]`.

**Rationale:** A is the feature, B is cheap and fixes a live bug + is thematically tied; C is pure refactor with the most risk and least user value. User chose both surfaces up front.

**Consequences:** Feature shipped + DOM-verified; `[CERT-ROWSHAPE]` + `[CERT-GOALS-DOC]` filed to backlog.
