# Conv Decisions - 2026-07-21

## 1. Hard-detach the project from the server-gated Task subsystem
**Type:** Strategic
**Topics:** cc-workflow, docs-infra

**Trigger:** `TaskCreate`/`TaskList`/`TaskUpdate`/`TodoWrite` are gated OFF by an undocumented server-side switch (`tengu_vellum_ash`) that could flip at any time; the project's task-tracking skills silently depended on them.

**Options Considered:**
1. Tolerant fallback (skills prefer TaskList when present, else write-through)
2. Hard removal — write-through to CURRENT-TASKS.md is the only mechanism ← Chosen (user)
3. Leave it and keep chasing the gate

**Decision:** Rewrite all task tracking to write-through `CURRENT-TASKS.md` directly; strip every `Task*` reference from skills, CLAUDE.md, the hook, and memory.

**Rationale:** The tools held nothing the file didn't; a file edit is as verifiable as the tool call. The user wanted a clean break, not a dual-path.

**Consequences:** ~15 files rewritten; the Task subsystem is no longer a dependency; the `/model` non-1M probe remains the one untested way the tools could return (recorded in `[TASK-TOOLS-VERIFY]`).

---

## 2. Redesign CURRENT-TASKS.md as TOC + stable bodies
**Type:** Implementation
**Topics:** docs-infra, cc-workflow

**Trigger:** The user found the em-dash-prose + Ordered/Unordered format hard to read and disliked that tasks physically move as their state changes.

**Options Considered:**
1. Keep the 3-anchor Ordered/Unordered/Completed format
2. TOC (`🎯 Now` / `⏸️ Parked`) driving order + alphabetical stable `## Tasks` bodies + `✅ Done this conv` ← Chosen

**Decision:** Ordering lives in two tables of contents that link to anchors; task bodies live alphabetically and never move — only a TOC line + a `State:` bullet change. Status is kept off the heading so anchors never churn.

**Rationale:** Kills the churn of moving H3 blocks between sections; makes reprioritise/start/park zero-movement edits; bullets replace prose walls.

**Consequences:** All parsers (r-start Step 7.5/8, r-end Step 5, r-commit Step 0, r-update-tasks) re-pointed to the four new anchors in the same batch; the new deterministic checker keys on them.

---

## 3. Wire the full test suite into the commit-boundary skills
**Type:** Implementation
**Topics:** testing, cc-workflow

**Trigger:** The detach + format are only durable if a regression (a Task tool creeping back, an anchor reverting) is caught automatically.

**Options Considered:**
1. Leave the suites manual
2. Run the full `test-task-board.sh` at r-commit Step 0 and r-end Step 5 ← Chosen (user)

**Decision:** Both boundary skills run the full suite (~0.9s); r-update-tasks stays on the fast checker for mid-conv tidies. Each branches pass / live-board ISSUES (fix inline) / test-suite fail (🔴 tooling regression).

**Rationale:** 0.9s is negligible; the gate that commits the board now also verifies its structure, the validator's correctness, the workflow, and the detach invariants.

**Consequences:** The detach is self-defending — a red suite blocks a commit.
