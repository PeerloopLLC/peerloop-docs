# Conv Learnings - 2026-07-04

## 1. Chrome freezes CSS-transition timelines in hidden/background tabs
**Topics:** browser-testing, plato, cc-workflow

**Context:** During MOBUP verification, the NavDrawer appeared "frozen" — open class applied + body-scroll-locked, but the panel stayed at `translate:-100%` with `getAnimations()` reporting a `CSSTransition` stuck `playState:"running"`, `currentTime:0`. Even an inline `translate:0` was ignored (a running transition's value overrides inline styles).

**Learning:** When the MCP browser tab is not the foreground tab, `document.visibilityState === "hidden"` and Chrome **freezes CSS-transition timelines at frame 0** — transitions never advance, pinned at their start value. This is NOT an app bug. Confirmed with a fresh control `<div>` whose plain `transition:transform 200ms` was also pinned at `currentTime:0`. The transition's *end state* still resolves correctly (proved by setting `transition:none` → value jumps to target), so the DOM contract is verifiable even when the animation isn't filmable.

**Pattern:** Before diagnosing a "stuck transition" in the MCP browser, check `document.hidden` / `getAnimations()[0].currentTime`. If hidden, verify the transition's END STATE (via `transition:none` or class-toggle + computed final value), not the animated interim.

## 2. "Up" navigation is deterministic; `history.back()` is fragile
**Topics:** navigation, ux, url-routing

**Context:** User weighing a mobile back affordance: reuse the breadcrumb trail vs a dedicated back control, and asked "do you always know where the URL should go?"

**Learning:** **Up (hierarchical parent)** is a fixed property of the route — the breadcrumb's second-to-last item — knowable at render time regardless of how the user arrived, so it never dead-ends. **Back (`history.back()`)** assumes a safe previous in-app entry, which breaks on deep-link / push-notification / fresh-tab / external-referrer entry. So a single "back" control is only non-fragile when implemented as an *up*-link to the parent href, never as `history.back()`.

**Pattern:** For an in-app back/up affordance, target the deterministic parent href (breadcrumb's immediate parent), guarded by a component test asserting no `history.back`/`history.go`.

## 3. Tailwind v4 `transition-transform` includes the `translate` property
**Topics:** astro, tailwind, css

**Context:** Investigating the drawer slide (before the hidden-tab root cause). `getComputedStyle(aside).transitionProperty` on a `transition-transform` element read `"transform, translate, scale, rotate"`.

**Learning:** In Tailwind **v4**, `transition-transform` expands to transition all four individual transform properties (`transform, translate, scale, rotate`), not just `transform` as in v3. So drawers that slide via the `translate-x-*` utilities (which use the CSS `translate` property in v4) DO get transitioned by `transition-transform`. (Not the cause of the drawer issue here — that was the hidden tab — but a real v3→v4 behavior change.)

## 4. success.astro `data` can be stale for a just-self-healed enrollment
**Topics:** ssr, enrollment, courses

**Context:** Copying the standard `CourseHeader` into the success page; deciding the `isEnrolled` prop value.

**Learning:** `success.astro` loads `data` via `fetchCourseTabData` *before* it self-heals a missing enrollment row (it recomputes a `railJourney` afterward for exactly this reason). So `data.isEnrolled` (and `data.journey` completion flags) can read stale-false on the confirmation page. On a page whose semantics guarantee enrollment (the "Enrollment Confirmed" render), pass `isEnrolled={true}` rather than the loader value.

## 5. The OS window clamp doesn't block below-`lg` verification (only below ~472px)
**Topics:** browser-testing, responsive

**Context:** Repeated across MOBNAV/MOBUP: `resize_window` to 400 lands at ~472–520px (OS min), and sometimes snaps back to a wide value.

**Learning:** The machine's OS window minimum (~472px) blocks a literal phone-width (375/360/320) render, BUT 472–520px is still well below the `lg` breakpoint (1025px), so the mobile chrome (ControlBar, HeaderBar, MobileUpNav, `lg:hidden` elements) DOES render and is DOM-verifiable. Content-sized elements (e.g. the 314px ControlBar pill) can then be bounded arithmetically for true phone widths. The literal phone-width visual remains the user's real-device eyeball.
