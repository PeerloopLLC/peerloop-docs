# Conv Learnings - 2026-07-11

## 1. PLATO browser-walk row-identity excludes producer-mocked tables
**Topics:** plato, testing

**Context:** The ecosystem/activities browser walks run on the live dev server; the API-producer oracle runs in vitest with `MockRegistry`.

**Learning:** A live browser walk can never be row-identical to a mocked API producer on tables written by mocked services. `notifications` is **irreducible** — the producer silences every `notify*` call (0 rows), the live server writes them for real — so it must be **excluded** from the per-table COUNT diff. This is the correct validation methodology (Option A), not a defect to bridge away.

**Pattern:** Per-table COUNT diff excluding `notifications`; both walks landed at 69/70.

## 2. Conv-383 "activities 70/70 identical" was inaccurate — verify assertions ≠ full row-identity
**Topics:** plato, testing

**Context:** Re-diffing `activities-walk.sqlite` vs `activities.sqlite`.

**Learning:** The 8 scenario `verify` assertions are targeted COUNT checks that never touch `notifications` or `user_stats`, so "all assertions pass" does NOT imply "all 70 tables row-identical." The Conv-383 capture actually diverged on those two tables. Always run a real full per-table diff to claim row-identity — don't infer it from assertions passing.

## 3. Fire-and-forget promises are dropped on the CF Worker — use ctx.waitUntil
**Topics:** cloudflare, video, testing

**Context:** `completeSession`'s `triggerPostSessionActions` (user_stats + completion notifications) ran as a floating promise (`booking.ts:171`).

**Learning:** On a Worker (miniflare + production), a floating promise is dropped when the request terminates — so `user_stats` never updated and completion notifications never sent after a real BBB `room_ended` webhook. vitest (node, no Worker lifecycle) drains it, which is why the oracle had the rows and hid the bug. The Stripe webhook already guards its insert with `locals.cfContext.waitUntil` for exactly this reason.

**Pattern:** Optional `waitUntil` param threaded from the webhook handler; `await` fallback for non-Worker/batch callers.

## 4. CUT-2 Stripe enroll bridge for a live browser walk
**Topics:** plato, stripe

**Context:** `ecosystem-pre-12` stops before enrollment, so 3 enrollments had to be bridged.

**Learning:** Fire a direct-signed `checkout.session.completed` via `trigger-webhook.sh stripe-direct-raw` with metadata `pending_enrollment_id`/`student_id`/`course_id`/`creator_id`/`instructor_type:creator`/`teacher_certification_id`/`assigned_teacher_id` and **no `payment_intent`** (matches the producer's mock → 0 transactions/payment_splits). The webhook is self-contained — `create-session` writes no D1 state, so it isn't needed.
