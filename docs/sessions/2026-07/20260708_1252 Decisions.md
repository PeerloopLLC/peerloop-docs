# Conv Decisions - 2026-07-08

## 1. Booking calendar regrouped entirely in viewer-tz (Model A)
**Type:** Implementation
**Topics:** timezone, booking

**Trigger:** Booking slice — the mixed-zone calendar off-by-one.

**Options Considered:**
1. Fix only the times (leaves the off-by-one) ← rejected
2. Regroup the whole calendar (grouping + grid + today + month-nav) in viewer-tz ← Chosen

**Decision:** Derive each slot's viewer-tz day from its UTC `start_time` via `dateKeyInTz`; build the grid as civil UTC math anchored on the viewer-tz day.

**Rationale:** Durable/correct — the plan explicitly scoped the off-by-one into this slice.

**Consequences:** Dropped browser-local `currentMonth`/`parseLocalDate`; added `stepMonth`/`ymdParts` civil helpers.

## 2. Admin/mod slice scope — A+B now, defer C
**Type:** Strategic
**Topics:** timezone, admin, scope

**Trigger:** The flagged admin bucket mixed session times with created-at/activity stamps; the plan separates Admin/mod from a later misc-stamps slice.

**Options Considered:**
1. Group A only (true session surfaces)
2. A + B (everything with a time-of-day) now, defer C (pure date-only) ← Chosen
3. All three groups now

**Decision:** Localize everything that shows a time-of-day now; defer pure date-only stamps to the misc-stamps slice where the date-only policy is decided once.

**Rationale:** Group B shows a wall-clock (admin should see it in their zone); Group C is the genuine judgment call, best decided in one place.

**Consequences:** 6 admin files localized; Group-C stamps (AdminMemberSummary, AdminCourseTab/CommunityTab, Announcements created-date, 2 MemberCards) deferred.

## 3. Pure date-only stamps stay UTC-stable (misc-stamps policy)
**Type:** Strategic
**Topics:** timezone, policy, date-stamps

**Trigger:** Final Phase-1 slice — how to render cert/review/member-since/last-active/flag/earnings dates.

**Options Considered:**
1. UTC-stable via `formatDateUTC` ← Chosen
2. Viewer-tz via `formatSessionDate(…, userTz)`
3. Split by field nature

**Decision:** Pure date-only stamps render UTC-stable; only time-of-day values localize.

**Rationale:** Fixes the browser-local bug, matches the dominant convention, needs no userTz threading, and avoids introducing a ±1-day off-by-one on midnight-stored calendar milestones (`certified_date`).

**Consequences:** ~13 stamps → `formatDateUTC`/`formatSessionDate(…,null,…)`; 2 `nextSessionAt` session-date chips → viewer-tz (they're session dates, not stamps).

## 4. Phase 2 email/notification tz treatment
**Type:** Implementation
**Topics:** timezone, email, notifications, phase-2

**Trigger:** Rendering session emails in the recipient's tz.

**Options Considered:**
1. Zone label: local+short-zone ("9:00 AM EST") ← Chosen · vs bare local ("9:00 AM")
2. Scope: emails + in-app notification strings ← Chosen · vs emails only

**Decision:** Emails show an explicit short zone label; localize the in-app notification strings too (not just emails).

**Rationale:** Emails are read out of context → the zone must be shown; localizing notifications too keeps email + notification center + app all consistent.

**Consequences:** New `formatRecipientSession` helper; both senders format per-recipient across emails AND notifications; interim UTC labels removed.
