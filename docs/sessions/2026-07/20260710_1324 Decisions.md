# Conv Decisions - 2026-07-10

## 1. Decompose `complete-course` into `book-sessions` + `complete-sessions`
**Type:** Implementation (architectural — shared step catalog)
**Topics:** plato, testing

**Trigger:** The flywheel's `complete-course` fuses browser-drivable booking with the CUT-3 BBB completion webhook, so a pure browser segment can't produce `wp-booked` and hand it to an API bridge.

**Options Considered:**
1. Keep `complete-course` fused (coarse chain; booking UI not browser-verified)
2. Decompose into flywheel-scoped `book-sessions` + `complete-sessions`, leaving the shared step untouched for ecosystem/seed-dev ← Chosen
3. Decompose the shared step everywhere (higher blast radius — touches seed-dev/ecosystem)

**Decision:** Option 2 — add two flywheel-scoped steps; flywheel grows to 15 steps; shared `complete-course` unchanged.

**Rationale:** Matches the waypoint plan's segment model (browser books → `wp-booked`; API completes → `wp-completed`), gives session-booking real browser coverage, and contains blast radius (seed-dev seeds the dev DB — must not be disturbed).

**Consequences:** New steps + `wp-booked`/`wp-completed` producers; `complete-sessions` re-discovers session IDs from the DB (Model B); completion is order-independent.

## 2. Mechanical `flywheel-pre-N` producer naming with `wp-*` semantic mapping
**Type:** Implementation
**Topics:** plato, naming

**Trigger:** Needed names for the 4 new waypoint producer instances.

**Options Considered:**
1. Semantic instance names (`wp-published.instance.ts` etc.)
2. Mechanical `flywheel-pre-N` from `plato:split`, annotated as `wp-*` in the manifest ← Chosen

**Decision:** Follow the Conv-379 `flywheel-pre-11` precedent — `plato:split`-generated `flywheel-pre-9/12/14/15`, with the `wp-*` meaning carried in `snapshots/README.md` + `plato.md`.

**Rationale:** Established pattern; avoids a novel naming convention; the split tooling produces these names deterministically.

**Consequences:** Manifest maps `flywheel-pre-9→wp-published`, `-12→wp-enrolled`, `-14→wp-booked`, `-15→wp-completed`, `flywheel→wp-certified`.

## 3. Drop the `apiBridge` intent field (use omission, not a marker)
**Type:** Implementation
**Topics:** plato

**Trigger:** Wanted to mark the completion intent as a non-browser API bridge in `flywheel.instance.ts`.

**Options Considered:**
1. Add `apiBridge?: boolean` to `BrowserIntent`
2. Omit the field; convey the bridge nature via prose + `complete-sessions.*` coversStepActions ← Chosen

**Decision:** Removed the invented field.

**Rationale:** The established pattern for CUT boundaries is omission (e.g. `flywheel-pre-11` simply drops the self-certify intent); no consumer for a marker exists until the Phase-4 segment runner. Adding a type field now would be speculative.
