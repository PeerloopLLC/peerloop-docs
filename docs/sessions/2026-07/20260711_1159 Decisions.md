## 1. Phase 4 sequenced as 4a (foundation) → 4b (runner) → 4c (agent walker, deferred)
**Type:** Strategic
**Topics:** plato, testing

**Trigger:** "Start Phase 4" — but Phase 4's shape (runner vs orchestrator vs agent walker) was undetermined and partly multi-conv.

**Options Considered:**
1. API-side chain runner only
2. Live-D1 orchestrator (multi-conv)
3. Agent-driven browser walker (multi-conv, speculative) ← deferred as 4c
4. Dependency-graph + registry + provenance foundation first ← Chosen (user-directed)

**Decision:** Build the dependency-graph + registry + provenance **foundation** (4a), then the make-for-waypoints runner (4b) on top; the agent browser walker becomes 4c, gated post-launch (`[BROWSER-SMOKE-2B]`).

**Rationale:** The graph is the substrate every runner variant reads; building it first turns the runner from "hardcoded chain" into a dependency-aware `make`. The user explicitly reframed Phase 4 around the dependency tree.

**Consequences:** Both 4a and 4b shipped this conv; PLATO-SEQ's active work is complete, 4c parked post-launch.

## 2. Three dials — committed static graph + live-read per-machine run-state + transitive-closure staleness
**Type:** Implementation
**Topics:** plato, testing

**Trigger:** "PLATO runs are infrequent — we need to know how recently the graph was updated AND when each run last ran."

**Options Considered:**
1. Cache run-state into the committed manifest (would lie on the other machine — snapshots are gitignored/per-machine)
2. Committed static graph + live-read local run-state + transitive-closure hash ← Chosen
3. Coarse producer-file-only hashing (misses shared-step edits)

**Decision:** Two clocks. Clock 1 = committed `manifest.generated.json` (`generatedAt`/`gitRev`/`graphSourceHash`) + `plato:graph check`. Clock 2 = live-read gitignored provenance sidecars + `plato:graph status`. Staleness = transitive-closure `sourceHash` compare.

**Rationale:** Run-state is inherently per-machine (local gitignored snapshots), so it must not be committed; the static graph is shared and committed. Transitive hashing is the only way shared-step edits invalidate the right downstream waypoints.

**Consequences:** The manifest is committed; sidecars are `.gitignore`d (`*.prov.json` line added).

## 3. Provenance as gitignored JSON sidecar, not an in-DB table (flagged deviation)
**Type:** Implementation
**Topics:** plato, testing

**Trigger:** Integration check found the manual row-identity diff would gain a permanent excluded table.

**Options Considered:**
1. `_plato_provenance` table inside each `.sqlite` (self-describing, but a row-identity footgun)
2. Gitignored JSON sidecar `<wp>.sqlite.prov.json` ← Chosen

**Decision:** Sidecar. Flagged to the user as a deviation from the literal "stamp into the .sqlite" proposal; accepted.

**Rationale:** Keeps the `.sqlite` byte-clean for the Conv-383/384 row-identity workflow; readable from JS + TS with no sqlite round-trip.

**Consequences:** `waypoint-provenance.ts` writes `.prov.json`; both produce paths (API snapshot-save + `plato:capture`) stamp it.
