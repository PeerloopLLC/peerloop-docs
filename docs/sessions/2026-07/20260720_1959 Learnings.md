# Conv Learnings - 2026-07-20

## 1. `useSyncExternalStore` is the right migration for a client-store hydration hook — when the store already has the pieces
**Topics:** react, astro, testing

**Context:** `useCurrentUser`/`useAuthStatus` used `useState(null/'loading')` + a mount `useEffect` that hydrates from a global cache and subscribes — tripping `react-hooks/set-state-in-effect`.

**Learning:** The migration is clean IFF: (a) the store's subscribe fn is already `(listener) => unsubscribe` (React passes a no-arg `onStoreChange`, assignable to a `(value) => void` listener); (b) `getSnapshot` returns a referentially-stable value — here `getCurrentUser()` returns the cached `window.__peerloop.currentUser`, kept stable by `setCurrentUser`'s id+dataVersion dedup guard, and `authStatus` is a primitive; (c) `getServerSnapshot` returns the exact prior SSR init (`null`/`'loading'`) so hydration still matches. Verify getSnapshot stability BEFORE writing — an unstable snapshot infinite-loops.

**Pattern:**
```ts
return useSyncExternalStore(subscribeToUserChange, getCurrentUser, () => null);
```

## 2. `set-state-in-effect` is a poor fit for an Astro-islands SSR + fetch-on-mount codebase
**Topics:** react, astro, testing

**Context:** Adopting react-hooks recommended-latest surfaced 91→95 `set-state-in-effect` findings, dwarfing everything else.

**Learning:** ~90 of 95 are correct-for-architecture: ~49 idiomatic fetch-on-mount (`useEffect(() => fetchX(), [])`) and ~25 deliberate SSR-hydration-safety (client-only reads of `localStorage`/`window.location.hash`/`document`/user-cache intentionally deferred to an effect so SSR + the first client render both produce the server value). Rewriting these to lazy-`useState` init or render-time compute reintroduces hydration mismatches / SSR crashes (`window` undefined) — the rule cannot see SSR constraints. Treat as an accepted baseline at warn (like `exhaustive-deps`/[LE-TRIAGE]); chase only new/egregious cases.

## 3. Reordering a loader above its `useEffect` can UNMASK a latent `set-state-in-effect`
**Topics:** react, testing

**Context:** The `immutability` "cannot access variable before it is declared" fired on `useEffect(() => fetchX())` where `fetchX` was declared below. Reordering (loader above the effect) cleared it.

**Learning:** The `immutability` error had been *masking* the compiler's `set-state-in-effect` analysis of the same effect. After the reorder, the effect analyzes cleanly and the latent `set-state-in-effect` surfaces (net warning count unchanged for those files). Expect a bucket-shift, not a count drop, when fixing this class — and it's still worth doing (the forward-reference smell is genuinely resolved).

## 4. An inline component used in two spots → a plain JSX element value, not a hoisted component
**Topics:** react

**Context:** `TestimonialsBrowse`'s `FilterContent` (inline `const FilterContent = () => (...)` used as `<FilterContent/>` in desktop aside + mobile drawer) tripped `static-components` and closed over ~9 filter values.

**Learning:** Hoisting to module scope would need threading all 9 as props. Simpler and better: `const filterContent = (...)` (a plain element value) referenced as `{filterContent}` in both places. It's not a component definition, so `static-components` doesn't fire; needs zero props; and it FIXES the remount (React reconciles the element instead of unmounting a new component type each render). Same element description reused in two sibling subtrees is valid React.

## 5. react-hooks@7.1.1 needs no `overrides` pin (unlike jsx-a11y)
**Topics:** testing, cc-workflow

**Context:** `[A11Y]` (Conv 399) needed a package.json `overrides` peer pin because jsx-a11y@6.10.2 caps at eslint ^9.

**Learning:** react-hooks@7.1.1 already ships the `eslint ^10` peer, so adopting its full recommended-latest set required only an `eslint.config.js` change — no dependency change, no pin. The `asWarn` helper (built for [A11Y]) generalized perfectly to downgrade the whole rule-set to warn while preserving options and `off` states.
