The Peerloop settings architecture (Conv 209 three-tier restructure → ongoing) **favors local control**: the project-repo `.claude/settings.json` (checked in, shared across machines) plus the gitignored per-machine `.claude/settings.local.json` are the authoritative tiers. The **global** `~/.claude/settings.json` is being **deprecated / cleaned separately** — exclude it from settings-prune analysis.

**Pruning `settings.local.json`:** compare each local allow entry against the **project `settings.json` only** (not global). Drop entries that are exact duplicates, prefix-subsumed by a broader project rule (e.g. `Bash(git add:*)` ⊂ `Bash(git:*)`), stale (renamed/removed skills, one-off dated commands, compound-command artifacts like `for dir:*`/`done`), or now-promoted. Keep only genuinely M4-unique, still-relevant grants. Allows merge as a **union**; **deny always wins** over allow regardless of tier, so local allows can't actually *conflict* with project denies — pruning is about redundancy, not conflict.

**Promoting broad grants into project `settings.json`:** when a useful broad grant is promoted from local → project, collapse the narrower project rules it subsumes. **Keep the project file portable** — use project-relative single-leading-slash paths (`Read(/**)`, `Read(/.claude/**)`), NEVER filesystem-absolute `Read(//Users/<name>/...)`. The two dev machines have **different `$HOME`** (M4 = `/Users/livingroom`, M4Pro = `/Users/jamesfraser` per [[project_preflip_worktree_reference]]), so an absolute home path silently fails on the other machine.

**Conv 212 [SETTINGS-PRUNE]** (M4): pruned 137 redundant/stale local entries, then promoted **32 grants** to project `settings.json` across two rounds — 5 broaders (`tee:*`, `node:*`, `npx tsx:*`, `python3:*`, `Read(/**)`, collapsing 12 narrower rules) + 27 portable tools/WebFetch/Figma-MCP/skills. End state: project `settings.json` 80 → 99; `settings.local.json` 173 → **4** machine-local allows + `outputStyle`. The 4 kept local: `Bash(bash:*)`, `Bash(osascript:*)`, `Bash(brew install:*)` (higher-power — deliberately NOT shared by default) + `Bash(~/skills-canon/tools/drift-report.sh:*)` (skills-canon unconfirmed on M4Pro). See [[feedback_msi_sync_user_checkpoint]] for the memory-mirror sync that carries cross-machine state.

**Conv 212 [SETTINGS-GUARD]** — safety lens, NOT a removal lens. The tools are all operationally needed; the goal is **guarding irreversible/external execution that could run autonomously**. Mechanism = keep broad `allow` + escalate dangerous *invocations* via two paths, because Claude Code permission patterns are **prefix-anchored** (can't see a mid-command flag like `wrangler … --remote`):
1. **`permissions.ask` tier** (21 entries) — prompt-first for prefix-catchable dangers: `wrangler deploy:*`, `npm run deploy:*`, `npm run db:{reset,migrate,setup,seed,studio}:staging:*` + `:prod:*` (staging hits **remote shared D1** with no confirm-prod guard — prod scripts already gate behind `node scripts/confirm-prod.js` / hard-block, staging did NOT), `gh pr merge:*`, `gh release delete:*`, `rm -rf:*`.
2. **`permissions.deny`** += `gh repo delete:*` (never autonomous; joins the git force/reset/clean cliff).
3. **PreToolUse hook** `.claude/hooks/guard-dangerous-bash.sh` (registered in the existing Bash matcher) — regex-scans each command, returns `permissionDecision:"ask"` with a custom reason for mid-command patterns the lists can't catch: `wrangler --remote`, `wrangler --env prod`, `curl -X POST/PUT/DELETE/PATCH|--data`, destructive SQL (`DROP/TRUNCATE/DELETE FROM`), `git push --force`. Smoke-tested (local wrangler / GET curl / `npm run build` / plain `git push` pass silently). Permission + hook changes need a **session restart** to take effect → [GUARD-VERIFY].

**Conv 212 decision — Bash allow list KEPT AT FULL BREADTH.** Re-examined every Bash `allow` entry for trimming (advantage of an allow = prompt-avoidance × frequency; rare entries add standing surface for little benefit). Identified 15 low-frequency trim candidates (Tier 1 rare exec/mutate: `perl`, `npx kill-port`, `stripe trigger`, `trigger-webhook list`, `claude mcp`, `code`, `open`, `sips`; Tier 2 rare read-only: `npm ls/outdated/view/audit/ci`, `lsof`, `brew list`). **User chose C = keep all** — convenience worth it, and dangerous slices are already fenced by the ask/hook layer so broad allows carry low residual risk. Do NOT re-propose trimming the allow list unless the user reopens it.
