# CD-037: Instant Session Booking ("Book Now")

**Date:** 2026-03-17
**Source:** Client discussion (Conv 004)
**Priority:** High

---

## Client Request

The client wants students and teachers to be able to start a session immediately rather than only booking future time slots. The teacher must initiate/authorize the instant booking — the student cannot unilaterally start a session.

### Key Requirements

1. **Teacher-initiated authorization** — the teacher must agree before any "Book Now" option is available to the student.
2. **No overbooking** — a 2-module course cannot get a 3rd session. Session count invariant must hold.
3. **Module assignment** — the instant session must be assigned the correct next module automatically.
4. **Two entry points:**
   - **New booking:** When the student has unbooled modules, they can "Book Now" instead of picking a time slot — but only if the teacher has authorized it.
   - **Reschedule to Now:** When all modules are already booked, the student can reschedule a future session to "now" — but only if the teacher has authorized it.
5. **Smooth coordination** — avoid messy code-exchange or manual list management. The teacher should be able to offer an instant session naturally, presumably from a message thread or dashboard.

### Client Quotes

> "The ability of a student and a teacher to book a session to start just as soon as the student has paid (if it's the first module) or to change/book a session right away rather than pick a time slot."

> "The Book Now should only occur at the allowance of the Teacher. They must agree to it before any Book Now or Reschedule to Now can be shown."

> "There must be a smoother way without so much coordination, but initiated by the teacher (presumably from some message sent to the student)."

---

## Agreed Design: Session Invite Model

After discussion, the following design was agreed:

### Concept

The teacher sends a **Session Invite** — a short-lived, enrollment-scoped authorization that enables the student to book or reschedule a session to start immediately. The invite is delivered via the existing notification system.

### Session Invite Record

```
session_invites
├── id TEXT PRIMARY KEY
├── teacher_id TEXT NOT NULL → users(id)
├── enrollment_id TEXT NOT NULL → enrollments(id)
├── status TEXT ('pending' | 'accepted' | 'expired' | 'declined')
├── created_at TEXT
├── expires_at TEXT              -- created_at + 30 minutes
├── accepted_session_id TEXT     -- FK → sessions(id), set on accept
└── notification_id TEXT         -- FK → notifications(id), for cleanup
```

New notification type: `session_invite` (added to CHECK constraint on `notifications.type`).

### Flow: Teacher Initiates

1. Teacher clicks **"Offer Session Now"** from teaching dashboard, student list, or message thread.
2. System validates: does this enrollment have a bookable module OR a future scheduled session to reschedule?
3. Creates `session_invites` row: `status='pending'`, `expires_at=now+30min`.
4. Sends notification to student:
   - Title: "*[Teacher Name]* is ready for a session now"
   - Body: "Your teacher wants to start *[Module Title]*. Accept within 30 minutes."
   - `action_url`: `/course/[slug]/book?invite=[invite_id]`
   - `action_label`: "Accept & Join"
   - `type`: `session_invite`

### Flow: Student Accepts

1. Student clicks notification action button → lands on booking page with `?invite=[id]`.
2. System validates: invite is pending, not expired, correct enrollment.
3. **If `bookable_count > 0`:** Shows one-click "Start Session Now" confirmation.
4. **If `bookable_count === 0`:** Auto-selects the next upcoming scheduled session and shows "This will reschedule your session on [date] to now. Confirm?"
5. On confirm:
   - If reschedule: cancels next upcoming session, books new at `now+5min`.
   - If new booking: books at `now+5min`, `end = now+65min`.
   - Updates invite: `status='accepted'`, `accepted_session_id` set.
6. Redirects to `/session/[id]` (session room).

### Flow: Teacher Notified

When student accepts, teacher receives notification:
- Title: "*[Student Name]* accepted — session starting now"
- `action_url`: `/session/[new_session_id]`
- `action_label`: "Join Session"

### Expiry

No background job needed. When the student clicks an expired invite, the system checks `expires_at < now` and shows: "This invite has expired. Ask your teacher to send a new one."

### Reschedule Target Selection

When all modules are booked and the student accepts an invite, the system **auto-selects the next upcoming scheduled session** — no manual picker. Rationale: the student is doing "the next module" now instead of waiting for its scheduled time. Sequential learning order is preserved.

### Existing System Compatibility

- **Positional module assignment** (`booking.ts`): Works unchanged. A session at `now+5min` sorts correctly by `scheduled_start ASC`. Module reflow handles the reordering naturally.
- **Overbooking guard** (`getBookingEligibility`): Runs at accept time. If another session was booked between invite and accept, the guard catches it.
- **Conflict checks** (`POST /api/sessions`): Teacher and student conflict checks still run. If either party has a conflicting session, the accept fails gracefully.
- **Join window** (`join.ts`): With `scheduled_start = now+5min`, the 15-minute-early join window is already open. Both parties can join immediately.
- **Notifications**: Existing `action_url` + `action_label` infrastructure handles the invite delivery. New `session_invite` type gets its own icon/color in `NotificationsList.tsx`.
